## False-positive traps (226-RESEARCH.md §Orphan Scan) — handle before flagging

1. `indexes.ts` `ADDITIONAL_INDEXES` raw-SQL references PG table names — count as LEGITIMATE references for orphan purposes. Tables observed in scout: `coinflip_leaderboard`, `affiliate_earnings`, `baf_flip_totals`, `jackpot_distributions`, `level_transitions`, `degenerette_bets`, `decimator_claims`, `vault_deposits`, `vault_claims`.
2. `views.ts` raw-SQL JOINs reference tables (e.g. `player_winnings`, `token_balances`, `coinflip_results`) — LEGITIMATE references.
3. String matches inside block comments (`/** ... */`) or `//` comments — NOT a usage. Filter with `rg -v` or by inspecting match context.
4. A handler that imports via barrel but never references the binding still counts as importing — scan binding USE SITES, not just import lines.
5. Enum exports (`gamePhaseEnum` from `game-state.ts`) and constant exports from the barrel are NOT tables — exclude from the orphan-table universe.
6. `indexes.ts` itself is NOT in `drizzle.config.ts` per Gotcha #7; it IS a legitimate orphan-scan SOURCE per D-226-03 — its raw-SQL references count as code usage of the target table names.

## Algorithm (226-RESEARCH.md §Orphan Scan)

1. **Build table universe:** from `src/db/schema/index.ts` re-exports + `views.ts` `pgMaterializedView` names. Each entry = `{jsBinding, pgName, sourceFile:line}`. Exclude enums + non-table constants.
2. **Schema → code:** for each `(jsBinding, pgName)`:
   - Run pattern (4) — JS binding usage across handlers / indexer / routes / views.ts / indexes.ts.
   - Run pattern (3) — raw-SQL pg-name references in the same surfaces.
   - Run pattern (1) or (2) — import resolution confirming the binding is actually imported somewhere.
   - Total references = sum of usage sites (excluding comment-only matches). Zero → orphan finding.
3. **Code → schema:** for every `from '.../db/schema/<file>.js'` import, confirm `<file>.ts` exists and the named binding(s) resolve to an exported table/view in `src/db/schema/index.ts` (or directly in `<file>.ts`). Imported-but-not-exported = finding.
4. **Raw-SQL table-name check:** grep raw SQL strings anywhere in code for table names not in the universe = finding (missing-in-schema table name referenced by code).

```markdown
- **Resolution:** {INFO-ACCEPTED (legitimate unused) | RESOLVED-CODE (remove orphan OR wire up missing reference OR add missing table)}
- **Kind:** {schema-orphan | code-missing-in-schema}
- **Details:** {one-to-three sentences — for schema-orphan: pg-name, js-binding, zero-hit grep evidence. for code-missing: referring site, table-name referenced, no matching schema export.}
```

## Severity rules

- Default: **INFO** (orphan tables and stale references are documentation/cleanup issues, not bugs).
- Promote to **LOW** only when a code-side missing-in-schema reference would cause a runtime query failure (e.g. handler does `db.select().from(table)` where `table` is an unresolved import — build fails OR runtime SELECT fails against Postgres).
