<threat_model>
## Trust boundary
A player acting BETWEEN the VRF request and its fulfillment (the freeze window). The untrusted input is
any in-window player-controllable call that could steer an as-yet-unknown random outcome.

## Threat register

| Threat ID | Category | Component | Disposition | Mitigation (what the invariant asserts) |
|-----------|----------|-----------|-------------|------------------------------------------|
| T-381-02-01 | Tampering (RNG steering) | a player mutating a VRF-DERIVED seed slot (rngWordByDay / lootboxRngWordByIndex / lootbox index) inside the window | mitigate | the invariant snapshots each at request and asserts byte-equality at unlock |
| T-381-02-02 | Tampering (non-VRF in-window read) | a player mutating a NON-VRF slot the consumption reads alongside the word (ticket buffer / dailyIdx / index cursor) | mitigate | the enumerated set INCLUDES these non-seed slots ([[feedback_rng_window_storage_read_freshness]]) |
| T-381-02-VAC | Tampering (vacuous green) | the invariant passes because the window is never actually opened or no in-window action is attempted | mitigate | a ghost counter records windows-opened + in-window-actions-attempted; the property is only meaningful when both > 0 |

Severity is blocked on the property being un-fuzzable / vacuously green: if the handler never opens the
window or every in-window action reverts before touching state, the freeze assertion is vacuous → the
plan FAILS acceptance until windows-opened > 0 AND in-window-actions-attempted > 0.
</threat_model>
