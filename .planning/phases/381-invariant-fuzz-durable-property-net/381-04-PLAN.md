enqueued for the permissionless auto-open, never held un-enqueued so the owner can time the open to a

<threat_model>
## Trust boundary
A box owner who is the ONLY party able to open a box (manual openLootBox is operator-gated). If a box is
persisted but NOT enqueued for the permissionless openBoxes() auto-opener, the owner can hold it closed
and time the open to a favorable live level/boon state — defeating the lootbox-resolution-timing
by-design ruling for that box class (the WHALE-01 finding).

## Threat register

| Threat ID | Category | Component | Disposition | Mitigation (what the invariant asserts) |
|-----------|----------|-----------|-------------|------------------------------------------|
| T-381-04-01 | Elevation (timing edge) | a box-creating path that persists lootboxEth/presaleBoxEth (base!=0) but skips the boxPlayers[index] enqueue | mitigate | the invariant asserts every tracked (index,owner) with base!=0 is in boxPlayers[index] until opened |
| T-381-04-02 | (false positive on resolved boxes) | an already-opened box (base==0) flagged as unenqueued | mitigate | the invariant only checks base!=0 entries; opened boxes (base==0) are excluded |
| T-381-04-VAC | (vacuous green) | the campaign creates 0 boxes, so the invariant is trivially true | mitigate | a ghost counter of boxes-created PER PATH; the property is meaningful only when boxes are created across ≥2 paths |

Severity is blocked on the property being un-fuzzable / vacuously green: if BoxCreationHandler creates no
persisted boxes (every creation reverts) the invariant is vacuous → the plan FAILS acceptance until
boxes-created > 0 across at least the mint-lootbox and a pass path.
</threat_model>
