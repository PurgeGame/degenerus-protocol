## Section 1: Change Summary

Phase 201 (commit 47004abc) removed the erroneous `creditFlip(ContractAddresses.SDGNRS, burnieToCredit)` side effect from all 3 redemption resolution paths in `DegenerusGameAdvanceModule.sol`. The creditFlip was phantom inflation: it added BURNIE-denominated coinflip position to sDGNRS during resolution without any corresponding debit, meaning sDGNRS's claimable coinflip pool grew by `burnieToCredit` every resolution with no backing source. Additionally, the now-unnecessary return value from `resolveRedemptionPeriod` was removed from both the implementation and interface, and 4 stale comments in `StakedDegenerusStonk.sol` were corrected.

**Files changed (3):**

## Section 4: BURNIE Supply Conservation Proof

Tracing the complete BURNIE lifecycle for a redemption claim:

### Phase A: Reservation (_submitGamblingClaimFrom, line 741)

When a player burns sDGNRS via gambling burn:
```

pendingRedemptionBurnieBase += burnieOwed;       // line 781 (of StakedDegenerusStonk)
```

Where `burnieOwed = (totalBurnie * amount) / supplyBefore` (line 768). `totalBurnie` is computed as `coin.balanceOf(address(this)) + coinflip.previewClaimCoinflips(address(this)) - pendingRedemptionBurnie` (lines 765-767), correctly excluding already-reserved amounts.

**Supply impact:** ZERO. This is pure accounting reservation. No tokens minted, transferred, or burned.

### Phase B: Resolution (resolveRedemptionPeriod, line 574)

When the game advances and resolves the pending period:
```

pendingRedemptionBurnie -= pendingRedemptionBurnieBase;                // line 589
pendingRedemptionBurnieBase = 0;                                       // line 590
```

The rolled amount `burnieToCredit` is emitted in `RedemptionResolved` (line 598) for off-chain indexing only. The reservation is released by subtracting `pendingRedemptionBurnieBase` from `pendingRedemptionBurnie`.

**PRE-FIX:** After the above, `creditFlip(ContractAddresses.SDGNRS, burnieToCredit)` was called. This invoked `_addDailyFlip(SDGNRS, burnieToCredit, 0, false, false)` in BurnieCoinflip, increasing sDGNRS's coinflip daily stake by `burnieToCredit`. This was phantom inflation: no corresponding debit existed. The BURNIE position grew for free.

**POST-FIX:** No creditFlip call. The reservation release is the only state change. `burnieToCredit` exists only as a local variable for the event.

**Supply impact:** ZERO. No tokens minted, no creditFlip, no position inflation.

**Path 1 (sufficient balance):** Transfers existing BURNIE from sDGNRS to redeemer. Supply unchanged (pure transfer).

**Path 2 (insufficient balance):** `claimCoinflipsForRedemption(sDGNRS, remaining)` (BurnieCoinflip line 358) calls `_claimCoinflipsAmount(sDGNRS, remaining, true)` (line 389). This:
1. Processes sDGNRS's pending coinflip claims (`_claimCoinflipsInternal`, line 395)
2. Computes `stored = claimableStored + mintable` (line 396)

**Supply impact of Path 2:** Exactly `toClaim` new BURNIE minted. This is the ONLY source of new BURNIE supply in the redemption flow.

### Conservation Assertion

```
                      = exactly the shortfall between sDGNRS's BURNIE balance and the owed amount
```

Pre-fix: `creditFlip(SDGNRS, burnieToCredit)` inflated sDGNRS's `claimableStored` via `_addDailyFlip` without any corresponding debit. This phantom position would eventually be minted via `mintForGame` during a claim, creating BURNIE not backed by any real accounting source. ## Section 6: Pool Conservation Proof

### Coinflip pool conservation

**Invariant:** For each player, `claimableStored + pending day stakes = total credited - total claimed`.

**Credits (additions to position):**
- `creditFlip(player, amount)` -> `_addDailyFlip(player, amount, 0, false, false)` adds to the player's daily stake for the current coinflip day
- `creditFlipBatch(players, amounts)` -> same, batched
- Daily coinflip processing: `processCoinflipPayouts` resolves day results, moving stakes to `claimableStored` based on win/loss

**Debits (reductions from position):**
- `_claimCoinflipsAmount(player, amount, mintTokens)` subtracts from `claimableStored`
- Called via: `claimCoinflips`, `claimCoinflipsForRedemption`, `claimCoinflipsFromBurnie`, `consumeCoinflipsForBurn`

**Pre-fix violation:** `creditFlip(SDGNRS, burnieToCredit)` during resolution added to sDGNRS's daily stake with no corresponding reservation or backing source. This inflated credits without a matching economic event. Conservation was violated because the credit had no origin in any game action or prize.

**Post-fix:** sDGNRS's credits come only from the 1/20 prize pool (line 781), which is a legitimate game mechanic funded by the consolidated prize pool. All debits are backed by actual credits. Conservation holds: `total_credited - total_claimed >= 0` at all times.

### pendingRedemptionBurnie conservation

**Invariant:** `pendingRedemptionBurnie >= 0` and represents the sum of all unresolved redemption BURNIE reservations.

**Credits (increase reservation):**
- `_submitGamblingClaimFrom` (line 780): `pendingRedemptionBurnie += burnieOwed`
- `_submitGamblingClaimFrom` (line 781): `pendingRedemptionBurnieBase += burnieOwed`

**Debits (release reservation):**
- `resolveRedemptionPeriod` (line 589): `pendingRedemptionBurnie -= pendingRedemptionBurnieBase`
- `resolveRedemptionPeriod` (line 590): `pendingRedemptionBurnieBase = 0`

**Flow:** Multiple `_submitGamblingClaimFrom` calls within a period accumulate into both `pendingRedemptionBurnie` and `pendingRedemptionBurnieBase`. On resolution, `pendingRedemptionBurnieBase` is subtracted from `pendingRedemptionBurnie` and zeroed. This correctly releases exactly the base amount for the resolved period. If new submissions arrive for a new period, they start accumulating into a fresh `pendingRedemptionBurnieBase`.

**Used by view functions:**
- `previewBurn` (line 714): `totalBurnie = burnieBal + claimableBurnie - pendingRedemptionBurnie` -- excludes reserved amounts from new burn calculations
- `burnieReserve` (line 725): same formula -- returns BURNIE backing net of reservations

**Conservation holds:** Both reservation and release are symmetric. The reservation segregates BURNIE to prevent double-counting in new burns. The release makes the amount available again. No BURNIE tokens are created or destroyed in this process.
