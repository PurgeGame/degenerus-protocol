## Revision 2 — Correctness refinements discovered during Plan 02 test authoring

While writing the Plan 02 forge invariant + binding tests, three concurrent issues were surfaced and corrected in a second contract commit on top of 432fb8f9. The final production code reflects the Revision 2 shape; Revision 1's shape is preserved below for audit trail.

### R2.a — Queue-length gate (liveness fix)

**Problem (Revision 1):** The pre-finalize block fired whenever `!ticketsFullyProcessed`, regardless of queue contents. On fresh deploy (`LR_INDEX = 0`) the `idx = LR_INDEX - 1` underflowed to `type(uint48).max`, `lootboxRngWordByIndex[maxUint48] == 0`, and `rngWordCurrent == 0` → `revert NotTimeYet()`. Fresh-deploy `advanceGame` could never complete. Additionally, if ticket queue were ever non-empty with `LR_INDEX = 0` (invariant violation), the drain would still reach `processTicketBatch:690` with the underflowed index and feed `entropy = 0` into trait generation — exactly the bug the fix is designed to prevent.

**Revision 2 fix:** Gate lives INSIDE the `if (ticketQueue[preRk].length > 0)` body. If the queue has no tickets to drain, the gate never fires. The invariant "read-slot non-empty ⇒ `LR_INDEX > 0`" (tickets reach the read slot only via `_swapTicketSlot`, which runs inside `_swapAndFreeze` only after `_finalizeRngRequest` bumps `LR_INDEX`) makes the underflow unreachable.

### R2.b — Nudged word for lootbox slot (consistency with game-over path)

**Problem (Revision 1):** The Revision 1 gate wrote `rngWordCurrent` (RAW VRF word) via `_finalizeLootboxRng(cw)`. In pre-fix code, `rngGate` at L1114 wrote `currentWord` AFTER `_applyDailyRng` applied nudges, so the slot held the NUDGED word. Similarly, `_gameOverEntropy` at L1176 writes the NUDGED word after `_applyDailyRng`. Revision 1 made the normal daily path write RAW while game-over and mid-day-fallback paths wrote NUDGED — creating a player-influenceable inconsistency (a player can influence which code path fires via 120-day liveness stall into game-over vs normal daily advance).

**Revision 2 fix:** Gate writes `rngWordCurrent + totalFlipReversals` (unchecked, matching `_applyDailyRng` at L1713). Normal daily path, game-over path, and rngGate's L1114 fallback all write the same nudged word. `totalFlipReversals` is frozen within the advance tx (`rngLockedFlag = true` between `_requestRng` and `_unlockRng` blocks `reverseFlip`), so the nudge count my gate reads equals the nudge count `_applyDailyRng` reads — the two paths compute identical nudged words, and `_finalizeLootboxRng`'s idempotent short-circuit at L1135 prevents double-writes.

### R2.c — do-while integration (structural cleanup)

**Problem (Revision 1):** The pre-drain block used a custom `emit Advance(STAGE_TICKETS_WORKING, lvl) + coinflip.creditFlip(...) + return` sequence when ticket work remained. This duplicated the post-do-while emit+creditFlip code at L460-465 and didn't match the established `stage = STAGE_*; break;` pattern used by every other stage in the do-while loop.

**Revision 2 fix:** Pre-drain block moves INTO the do-while as its first step (BEFORE `rngGate`). On partial drain, `stage = STAGE_TICKETS_WORKING; break;` exits the loop; the common post-loop emit+creditFlip at L460 handles the event + bounty payment. Net code is shorter and matches existing pattern.

### Final (Revision 2) shape

```solidity
uint8 stage;
do {
    // --- Daily drain gate: ensure read slot is fully processed before RNG ---
    if (!ticketsFullyProcessed) {
        uint24 preRk = _tqReadKey(purchaseLevel);
        if (ticketQueue[preRk].length > 0) {
            uint48 preIdx = uint48(_lrRead(LR_INDEX_SHIFT, LR_INDEX_MASK)) - 1;
            if (lootboxRngWordByIndex[preIdx] == 0) {
                uint256 cw = rngWordCurrent;
                if (cw == 0) revert NotTimeYet();
                unchecked {
                    cw += totalFlipReversals;
                }
                _finalizeLootboxRng(cw);
            }
            (bool preWorked, bool preFinished) = _runProcessTicketBatch(purchaseLevel);
            if (preWorked || !preFinished) {
                stage = STAGE_TICKETS_WORKING;
                break;
            }
        }
        ticketsFullyProcessed = true;
    }
    // ... rngGate + in-cycle drain etc. (unchanged) ...
} while (false);
emit Advance(stage, lvl);
coinflip.creditFlip(caller, ...);
```

## Theoretical Worst-Case Gas

Derived analytically BEFORE empirical measurement, per `feedback_gas_worst_case.md`.

| Path | Gates added | Storage ops | Other | Total overhead |
|---|---|---|---|---|
| Hot (slot already populated) | 1 SLOAD + 1 JUMPI | 0 SSTORE | — | ~2100 (cold) / ~100 (warm) |
| Cold (slot zero, rngWordCurrent non-zero) | 2 SLOAD + 1 SSTORE + 1 LOG3 | 1 SSTORE 0→nonzero (22100) | LOG3 ~2700 | net ≈ SLOAD only (SSTORE + LOG subsume rngGate's later write) |
| Stall (slot zero, rngWordCurrent zero) | 1 SLOAD + revert | 0 | revert | gas consumed up to revert; bounty not paid |

**Derivation:**

- The storage slot for `LR_INDEX` (packed in the `_lrRead(LR_INDEX_SHIFT, LR_INDEX_MASK)` SLOAD) is already touched earlier in the function at L191 (mid-day guard). On the daily-drain flow (`LR_MID_DAY != 0` check miss), that SLOAD did NOT execute, so LR_INDEX is cold on first post-fix access. EIP-2929 prices a cold SLOAD at 2100 gas; a warm SLOAD at 100.
- `lootboxRngWordByIndex[idx]` is a mapping slot (separate access-list entry from any packed state). On the daily-drain path, this is the first read of this slot in the tx — cold SLOAD, 2100 gas.
- `rngWordCurrent` is a plain state variable. Cold on first access in the cold-path branch. SLOAD ~2100 gas.
- `_finalizeLootboxRng(cw)` writes `lootboxRngWordByIndex[index] = rngWord`. The slot is zero pre-write; EIP-2200 SSTORE 0→nonzero costs 22100 gas. It then emits `LootboxRngApplied(index, rngWord, vrfRequestId)` — LOG3 (3 topics + data) ≈ base 1500 + 3*375 topics + data ≈ ~2700.
- **Cold-path net cost vs pre-fix:** In pre-fix code the SSTORE + LOG3 were paid later in the same tx inside rngGate's `_finalizeLootboxRng` call. The fix moves that work earlier in the tx without duplicating it (the idempotent short-circuit `if (lootboxRngWordByIndex[index] != 0) return;` at L1129 makes rngGate's later call a no-op). Net new cost is the 2 SLOADs only.
- **Stall-path:** revert consumes gas up to the revert opcode; caller is charged. Identical liveness profile to the existing L194 / L219 reverts (per CONTEXT.md D-02 and §Code Context "Caller-retry liveness"). Bounty is NOT paid.
- **Hot-path:** slot is already non-zero (the common post-VRF case once rngGate has run at least once in the tx's sequence). Only 1 new SLOAD + 1 JUMPI. Cold SLOAD ≈ 2100 gas; warm ≈ 100 gas.

**+5% budget derivation per SPEC §Constraints:** The happy-path advanceGame call cost varies by test scenario. From the pre-fix VRFCore.t.sol gas report (8638 calls across 22 tests exercising VRF fulfillment): min=16,684, avg=931,139, median=522,921 on the module-level advanceGame entry. A +2100 gas overhead on the happy path is +0.4% of avg and +12.6% of min. On any realistic production call (deploy + daily drain + RNG request, ≥ ~150k gas), +2100 is comfortably within +5%.

## Revision 3 — Game-over best-effort drain + liveness-triggered ticket block

Committed as `749192cd` on top of Revision 2 (`d09e93ec`) after user approval. Surfaces two pre-existing defects that were independent of the D-01 fix but discovered while Plan 02's forge tests were being written against game-over scenarios.

### R3.a — Game-over path skipped the drain

**Problem:** `_handleGameOverPath` at `contracts/modules/DegenerusGameAdvanceModule.sol:503` took its early-return branch without ever running the pre-drain or the in-cycle drain. Any tickets queued at game-over time were never processed through `processTicketBatch` → `_raritySymbolBatch`, so they had no traits and were excluded from `runTerminalJackpot`'s trait-bucket share. Funds released but queued-ticket holders forfeited eligibility. The defect predates Plan 01 — Revision 1 and Revision 2 each changed the daily-drain gate shape but left game-over skip-drain untouched.

**Fix:** best-effort dual-round drain inside `_handleGameOverPath` AFTER `_gameOverEntropy` populates the lootbox slot, BEFORE `handleGameOverDrain` runs the terminal jackpot:

- **Round 1:** drain the current read slot via low-level `delegatecall` to `processTicketBatch`. Partial result (`finished=false`) returns `(true, STAGE_TICKETS_WORKING)` so the caller retries. Catastrophic revert (OOG on huge queue) is swallowed and falls through to `handleGameOverDrain` — funds release is never blocked.
- **Round 1 finished:** `_swapTicketSlot` moves write→read so the previously-write buffer's tickets become drainable.
- **Round 2:** drains the new read slot via another `delegatecall`. Same partial/fail semantics as round 1.
- Both rounds read `slot[LR_INDEX-1]`, which `_gameOverEntropy` populated with `raw_vrf + totalFlipReversals` (nudged). See R3.d "Known issue: non-determinism" below.

**Signature change:** `_handleGameOverPath` return type widens from `bool shouldReturn` to `(bool shouldReturn, uint8 stage)`. The caller at `advanceGame:L178` dispatches the emitted stage and pays the advance bounty on `STAGE_TICKETS_WORKING` (matches the mid-day `STAGE_TICKETS_WORKING` precedent). `STAGE_GAMEOVER` continues to pay no bounty (pre-existing).

### R3.b — Terminal jackpot manipulation risk via post-VRF ticket insertion

**Problem:** once liveness triggers, the game-over VRF word is fulfilled at a specific block. Between the VRF callback tx and the next advance that runs `runTerminalJackpot`, the `finalWord` is on-chain and readable. An attacker who knows `finalWord` can pre-compute which `queueIdx`/`player` combinations would produce winning traits. Adding tickets to the queue after that point (via purchase, lootbox resolution, whale pass claim, affiliate reward, vault perpetual) would let the attacker asymmetrically insert only tickets they know would hit winning buckets.

**Fix:** `_livenessTriggered()` helper added to `DegenerusGameStorage` (mirrors the `_handleGameOverPath` liveness condition). Injected as a `revert E()` guard into `_queueTickets`, `_queueTicketsScaled`, and `_queueTicketRange` — the three internal helpers every ticket-queue push routes through. Once liveness triggers, NO path can add tickets to the queue. `_handleGameOverPath` also refactored to use the shared helper (removes inline duplication).

Coverage verified by grep: every `ticketQueue[...].push(...)` site in `contracts/` is reached via one of the three helpers.

### R3.c — Stage dispatch at caller

**Problem:** original caller pattern at `advanceGame:L178-181` hard-coded `emit Advance(STAGE_GAMEOVER, lvl); return;` on game-over return. With R3.a introducing a partial-drain case (`STAGE_TICKETS_WORKING`), the caller needed to dispatch the stage.

**Fix:** caller at `advanceGame:L178` destructures the `(shouldReturn, stage)` tuple, emits the returned stage, and conditionally pays the advance bounty on `STAGE_TICKETS_WORKING` — consistent with the mid-day `STAGE_TICKETS_WORKING` path's bounty payment at `advanceGame:L208-213`.

### R3.d — Known issues (non-blocking)

1. **Shared game-over entropy across buffers.** Both drain rounds read the same `slot[LR_INDEX-1]` populated by `_gameOverEntropy`. Tickets from the read buffer and the post-swap (previously-write) buffer share one entropy source. In normal flow, each buffer would draw from its own RNG slot. This is a deliberate compromise for game-over emergency: avoids a second VRF request and a multi-tx wait. Not manipulable (both entropies are VRF-derived). Worth documenting as a "small non-determinism" for future audit traceability.

2. **Intended-RNG substitution for pre-liveness tickets.** Tickets purchased under one RNG flow (normal daily or mid-day) that remained in the read buffer at game-over trigger time will be drained using the **game-over** entropy rather than the RNG the player originally bound to. The original RNG word may still exist in a different slot but has no readers in the game-over drain path. Again not manipulable — both RNGs are VRF-derived, state-driven, fair — but an edge-case determinism note.

3. **Comprehensive game-over test phase.** The Plan 02 forge tests verify the binding/ordering invariants and the single best-effort drain path. A full end-to-end game-over regression test (timing + VRF choreography + drain completeness + terminal jackpot eligibility audit for tickets across both buffers and lootbox-origin tickets) is tracked as a follow-up phase deliverable — see `.planning/notes/gameover-thorough-test.md` for the scope outline.
