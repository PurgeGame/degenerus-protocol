# Phase 232.1: RNG-Index Ticket Drain Ordering Enforcement — Specification

**Created:** 2026-04-17
**Ambiguity score:** 0.103 (gate: ≤ 0.20)
**Requirements:** 6 locked

## Goal

Every ticket queued at lootbox RNG index X is fully resolved using the correct non-zero `lootboxRngWordByIndex[X]` before any bucket-swap advances `LR_INDEX` to X+1 and requests a new VRF — under all reachable code paths.

## Background

**Current contract state (bug):** `DegenerusGameAdvanceModule.sol:185-262` runs two drain paths that call `_runProcessTicketBatch(purchaseLevel)` — the mid-day path at L202 (guarded by `LR_MID_DAY != 0` + L194 revert on unset RNG) and the daily-drain path at L242-262 (**unguarded**). Both fire before `rngGate` at L268. `rngGate` is what eventually calls `_applyDailyRng` → `_storeLootboxRng` (L1128-1130), which writes the fulfilled VRF word to `lootboxRngWordByIndex[LR_INDEX - 1]`.

`_runProcessTicketBatch` delegatecalls to `DegenerusGameMintModule.processTicketBatch` (L666-723), which at L690 reads `lootboxRngWordByIndex[uint48(_lrRead(LR_INDEX_SHIFT, LR_INDEX_MASK)) - 1]` with no zero-guard and passes it through `_processOneTicketEntry` → `_raritySymbolBatch` (L541-591) as the `entropyWord` argument.

**Temporal window where bug manifests:** `LR_INDEX` is bumped at VRF *request* time inside `_swapAndFreeze` (L1025-1030), but `lootboxRngWordByIndex[X]` is only written at VRF *fulfillment* time (via `rngGate` in a later block). Between those two events — which span VRF confirmation latency (seconds to minutes, longer during stalls) — `LR_INDEX == X+1` but `lootboxRngWordByIndex[X] == 0`. The daily-drain path at L242-262 reads this unpopulated slot and feeds `entropy = 0` into trait generation.

**Sim evidence:** 25L/100P turbo sim against the real compiled contract (not a re-impl) shows 136 batches / 3439 traits at L5 alone minted with `entropy = 0` — approximately half of L5's 6884 samples. With `entropy = 0`, the trait seed collapses to `keccak256(abi.encode(baseKey, 0, groupIdx))`, making trait RNG a deterministic function of on-chain-visible inputs (level, queueIdx, playerAddr, groupIdx). Per-quadrant category distribution shows Q0 cat-0 at 39.1% (design 13.3%) and Q0 cat-7 at 0.3% (design 10.7%), with 6 trait IDs (56-61) completely unreachable at L5.

**User's articulated invariant (locked 2026-04-17):** "Tickets are queued → an RNG comes in that should resolve those tickets (mid-day only if ticket threshold is met; daily always triggers bucket-swap) → use that RNG to resolve those tickets → only then may another bucket-swap happen and a new RNG be requested. Every path must clear the read-slot queue before bucket-swap, under all circumstances."

**Derived constraint:** Temporary reverts during a VRF stall are acceptable (caller retries on next block); permanent reverts that break game progression are not. The fix must be structural, not a revert-forever trap.

## Requirements

1. **Normal end-of-day drain-before-swap**: The normal daily `_swapAndFreeze` path at `DegenerusGameAdvanceModule.sol:1002-1036` must not bump `LR_INDEX` or request a new VRF while the current read slot has undrained tickets.
   - Current: `_swapAndFreeze` can fire even when `ticketQueue[_tqReadKey(purchaseLevel)].length > 0` with unprocessed entries. `LR_INDEX` advances; subsequent drain-before-rngGate at L242-262 reads `lootboxRngWordByIndex[LR_INDEX-1] == 0`.
   - Target: Bucket-swap at end-of-day is gated on the read slot being empty for the level(s) in scope; if not, the current-cycle drain must complete (or advanceGame must return with a stage signal instructing retry) before any LR_INDEX bump.
   - Acceptance: Forge invariant test reproducing the window (VRF fulfilled but read slot undrained) demonstrates that no `_swapAndFreeze` call advances `LR_INDEX` while any read-slot ticket remains; AND no `_raritySymbolBatch` invocation observes `entropyWord == 0`.

2. **Mid-day drain-before-swap**: The mid-day `_swapTicketSlot` path at `DegenerusGameAdvanceModule.sol:1007-1009` (threshold-triggered swap inside `_swapAndFreeze`) must honor the same drain-first invariant.
   - Current: Mid-day swap fires when `ticketQueue[wk].length > 0 && ticketsFullyProcessed`. The existing L194 guard (`if (word == 0) revert NotTimeYet()`) covers the read side when `LR_MID_DAY != 0`, but does not prevent the bucket-swap from bumping `LR_INDEX` while prior tickets remain.
   - Target: The mid-day path produces the same end-state invariant as end-of-day: prior read-slot tickets fully drained with their correct `lootboxRngWordByIndex[X]` before `LR_INDEX` advances to X+1.
   - Acceptance: Forge test covering a mid-day threshold-cross scenario shows zero `entropyWord == 0` observations in `_raritySymbolBatch` events across the mid-day → end-of-day transition window.

3. **Game-over / terminal drain-before-swap**: The game-over path (`GameOverModule.handleGameOverDrain`, `_gameOverEntropy`) must not allow `_swapAndFreeze` or equivalent LR_INDEX-advancing operation while undrained tickets reference a prior index.
   - Current: Game-over is an asymmetric stage-sequence relative to normal advance. Any reachable path in `DegenerusGameGameOverModule.sol` that bumps `LR_INDEX` must be verified against the invariant; any that draws from `lootboxRngWordByIndex[X]` must be verified to read a populated slot.
   - Target: Game-over paths either (a) complete all remaining ticket drain using the last fulfilled RNG before any further swap/request, or (b) do not advance LR_INDEX at all post-game-over.
   - Acceptance: Forge test simulating game-over triggered while tickets are frozen at a recent index shows zero `entropyWord == 0` observations in terminal/game-over drain paths.

4. **processTicketBatch entropy consumption**: `DegenerusGameMintModule.processTicketBatch:690` must read the correct non-zero RNG for the tickets it is draining — no silent consumption of `entropyWord = 0`.
   - Current: Line 690 reads `lootboxRngWordByIndex[uint48(_lrRead(LR_INDEX_SHIFT, LR_INDEX_MASK)) - 1]` with no guard; passes result to `_processOneTicketEntry` → `_raritySymbolBatch` regardless of value.
   - Target: Either (a) the structural ordering fix above makes `entropy = 0` at this site unreachable by construction, OR (b) the entropy read is sourced differently (e.g. from `rngWordCurrent` or passed as a parameter) such that the drain consumes the exact RNG word bound to the tickets being drained. No defensive revert guard is added — if the structural fix is correct, any guard here would be dead code (per user feedback rule: "remove unreachable safety caps; don't waste gas on dead branches").
   - Acceptance: Source audit confirms every reachable path into `processTicketBatch` supplies a non-zero `entropy` for the tickets being drained. No new `revert RngNotReady()` is introduced at this site unless the structural fix proves impossible.

5. **Ticket↔RNG binding consistency**: For every ticket frozen at lootbox RNG index X, the entropy value consumed by `_raritySymbolBatch` must equal `lootboxRngWordByIndex[X]` (or the equivalent source of RNG_X at consume time) and must be non-zero.
   - Current: No contract-level or test-level assertion exists proving bind-time → consume-time RNG identity. The sim surfaced a violation empirically.
   - Target: A forge invariant test verifies the binding across at least the normal, mid-day, and game-over drain paths.
   - Acceptance: The binding-consistency test passes on the fixed contract and fails on the pre-fix contract when run against the same sim-reproduction scenario.

6. **Sim-replay regression**: The 25L/100P turbo sim that produced the original bug evidence must show zero `_raritySymbolBatch` invocations with `entropyWord == 0`, and no regression in downstream trait-distribution statistics beyond what the pure-keccak trait algorithm predicts.
   - Current: Sim shows 136 batches / 3439 traits at L5 with `entropy = 0`; Q0 cat 0 inflated to 39.1%, Q0 cat 7 collapsed to 0.3%, 6 zero-hit trait IDs at L5 alone.
   - Target: Post-fix sim shows zero `entropy = 0` events and per-quadrant category distribution within ±1 percentage point of the weighted design distribution (cat 0–3 ≈ 13.3%, cat 4–6 ≈ 12.0%, cat 7 ≈ 10.7%).
   - Acceptance: Sim re-run post-fix produces zero `entropy = 0` batches; SQL aggregation of trait counts at L5 shows zero zero-hit trait IDs and max/min ratio consistent with Poisson variance (≤ 10× for ~6800 samples across 256 trait IDs).

## Boundaries

**In scope:**
- Structural re-ordering or gating of `_swapAndFreeze` / `_swapTicketSlot` paths in `DegenerusGameAdvanceModule.sol` to enforce drain-before-swap invariant under normal end-of-day, mid-day-threshold, and game-over/terminal paths.
- Any edits in `DegenerusGameMintModule.sol` necessary to source the correct non-zero RNG at `processTicketBatch:690` (e.g. parameter passthrough instead of storage read, if the structural fix requires it).
- Forge tests: invariant assertion test, binding-consistency test, sim-replay regression test.
- Audit trace of `processFutureTicketBatch` (DegenerusGameMintModule.sol:385) documenting that its `entropy` parameter always arrives non-zero on every reachable caller path; no code change to that function.

**Out of scope:**
- **Phase-transition path** (`DegenerusGameAdvanceModule.sol:283-316`, `phaseTransitionActive` branch) — deferred to Phase 235 TRNX-01 audit (already in roadmap). Phase 232.1 does not modify phase-transition stage logic; if the audit surface that Phase 235 will cover overlaps with 232.1's invariant, Phase 235 is responsible for reconciling them.
- **Trait-RNG algorithm fix** (LCG vs keccak-slice) — separate concern exposed by the same sim. Phase 232.1 only ensures the correct RNG reaches `_raritySymbolBatch`; the downstream question of whether that RNG is stretched via LCG or sliced via keccak is a distinct phase.
- **VRF-stall recovery mechanism** — gap-day backfill at `DegenerusGameAdvanceModule.sol:1689` (`_backfillOrphanedLootboxIndices`) remains unchanged; Phase 232.1 accepts that during a VRF stall the drain path reverts (caller retries on later advanceGame) as long as the revert is temporary (stall-bounded, not architectural).
- **processFutureTicketBatch code change** — audit only; verify all callers pass non-zero entropy; no `revert RngNotReady()` added (per user: "audit, don't fix").
- **Defensive drain-site revert guard** at `processTicketBatch:690` — not added; user selected "only structural" in Round 2 (matches `feedback_no_dead_guards.md` rule).
- **Sim infrastructure changes** — the sim is surfacing a real contract bug, not a sim defect; no changes to sim code or mock VRF coordinator.
- **Contract modifications to `contracts/` directory require explicit per-diff user approval** before commit (per `feedback_no_contract_commits.md`). Phase 232.1 plan/execute will produce diffs for user review — no autonomous commits to `contracts/` or `test/`.

## Constraints

- **No permanent reverts**: Any new revert introduced must be temporary (retry-able on subsequent blocks) or structural (removable by upstream state progressing). A drain path that reverts and cannot unstick on any caller retry is not acceptable.
- **Preserve existing mid-day guard semantics**: The `LR_MID_DAY` flag at `DegenerusGameAdvanceModule.sol:1009` and the L194 revert must not be weakened; Phase 232.1 extends coverage, does not regress it.
- **No new external caller surface**: Phase 232.1 does not add any new external entry points, only re-orders or gates existing internal stage transitions.
- **Gas budget**: Changes to advanceGame stage ordering must not exceed +5% gas on the core happy-path (normal end-of-day advance with tickets to drain).
- **Contract-edit rule**: All `contracts/*.sol` modifications require explicit user approval before commit. `ContractAddresses.sol` is exempt per `feedback_contractaddresses_policy.md`; every other contract file in `contracts/` is not.
- **Audit milestone context**: v29.0 is a READ-only audit milestone. Phase 232.1 is a scope expansion to a FIX phase (approved by user at phase-insertion time, recorded in STATE.md:80-82). Finding this bug counts toward v29.0's C4A-readiness goal; fixing it advances the same goal.
