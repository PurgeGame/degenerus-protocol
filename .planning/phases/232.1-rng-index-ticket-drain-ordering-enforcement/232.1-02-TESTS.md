## Instrumentation Strategy

**Option A — Event-capture via `TraitsGenerated`** (selected)

The `TraitsGenerated` event is already declared at `contracts/storage/DegenerusGameStorage.sol:479-486`:

```solidity
event TraitsGenerated(
    address indexed player,
    uint24 indexed level,
    uint32 queueIdx,
    uint32 startIndex,
    uint32 count,
    uint256 entropy
);
```

It emits from INSIDE `processTicketBatch` and `_processOneTicketEntry` in `contracts/modules/DegenerusGameMintModule.sol:470` and `:800`, IMMEDIATELY after the `_raritySymbolBatch(player, baseKey, processed, take, entropy)` call. The `entropy` field in position 5 (last data field, bytes 96-128 of the event data payload) contains the exact `entropyWord` passed to `_raritySymbolBatch`.

**Step-1 grep verification confirmed the signature is unchanged:**

```bash
$ grep -n "event TraitsGenerated" contracts/storage/DegenerusGameStorage.sol
479:    event TraitsGenerated(

$ grep -cn "emit TraitsGenerated" contracts/modules/DegenerusGameMintModule.sol
2
```

Both confirmations pass — event exists at L479 with the documented 6-field signature; >=2 emit sites (L470 + L800) present. Option A is live.

**Option B — Storage probe via `vm.load`** (not used)

Would read `lootboxRngWordByIndex[X]` directly from storage to verify the "bound word" invariant. Option A is strictly stronger: event-capture sees the ACTUAL `entropy` argument passed to `_raritySymbolBatch`, whereas storage probing only verifies the slot content (which could differ from the consumed entropy if there was a stale-read bug). Option B remains a documented fallback if a future change removes or renames the `TraitsGenerated` event — executor must halt and consult before switching.

## Mid-Day Cross-Day Edge Reproduction

`testBindingConsistencyMidDayCrossDay` reproduces SPEC §Background "mid-day → cross-day edge" scenario:

1. Day 1: purchase + `_completeDayWithLogs` cleanly (establishes a baseline day).
2. `vm.warp(block.timestamp + 1 days)` to day 2 boundary.
3. Day 2: purchase + `_completeDayWithLogs` cleanly.
4. `vm.warp(block.timestamp + 1 days)` to day 3 boundary.
5. Day 3: `_purchase(400, 1 ether)` — includes a non-zero lootbox amount (`lootboxWei=1 ether`), which sets `LR_PENDING_ETH` and may trigger mid-day lootbox RNG eligibility per `requestLootboxRng`'s threshold.
6. Day 3: `_purchase(400, 0)` — queue more tickets for subsequent drain.
7. `_completeDayWithLogs(vrfWord)` — advance through the day, fulfill VRF, advance loop.
8. Assertion: captured `TraitsGenerated.entropy` equals `lootboxRngWordByIndex[LR_INDEX - 1]` AND is non-zero.

On pre-fix, step 7's drain at `L242-262` reads `lootboxRngWordByIndex[LR_INDEX - 1] == 0` during the race window (daily VRF callback wrote `rngWordCurrent` but `rngGate` hasn't finalized the slot yet), passing `entropy=0` into `_raritySymbolBatch`. The test's `_capturedEntropies` helper captures this via the `TraitsGenerated` event and asserts `entropy != 0` — which fails with `mid-day cross: entropy zero` on pre-fix.

On post-fix, the drain gate populates `lootboxRngWordByIndex[LR_INDEX - 1] = rngWordCurrent + totalFlipReversals` lazily before the drain reads it, so `entropy` is the nudged word and non-zero. Test passes.

## Game-Over Path Isolation Rationale (W-3)

The game-over test asserts ZERO `TraitsGenerated` events fire across the game-over transition (triggered by warping past `DEPLOY_IDLE_TIMEOUT_DAYS = 365` without purchase, at level 0). The test loops up to 20 `advanceGame` calls + VRF fulfillments to drive the transition to completion (`gameOver = true`).

**Why ZERO `TraitsGenerated` is the correct signal:**

`TraitsGenerated` emits ONLY from inside `processTicketBatch` and `_processOneTicketEntry` at `contracts/modules/DegenerusGameMintModule.sol:470` and `:800` — the daily-drain consumers. Zero emits proves the daily-drain branch body never ran.

**Why an LR_INDEX-advance ghost counter would be WRONG:**

`_handleGameOverPath` at `DegenerusGameAdvanceModule.sol:178-181` takes an EARLY RETURN at the top of `advanceGame`, BEFORE reaching the do-while loop that contains the pre-finalize gate and `rngGate` (which is the only site that bumps `LR_INDEX`). So during a game-over advance, `LR_INDEX` does NOT advance at all. A ghost counter keyed on "did LR_INDEX advance during this tx?" would report 0 — satisfying an assertion "LR_INDEX should not advance during game-over advance" vacuously for the wrong reason (the gate check never ran, but that's because `_handleGameOverPath` short-circuited above it, not because the isolation property itself held). The correct signal is "did the daily-drain body actually execute" — which `TraitsGenerated` directly reports.

**Why the test also asserts `game.gameOver() == true`:**

Without this assertion, a test run where the game-over transition never completes (e.g., VRF never fulfills, or the liveness threshold wasn't crossed) would trivially report zero `TraitsGenerated` events — again, vacuously satisfying the isolation claim. The `assertTrue(game.gameOver())` ensures the test only passes when it has ACTUALLY driven the game into the game-over branch, making the zero-emits claim meaningful.
