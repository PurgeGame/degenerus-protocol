Call tree:
- vote reads: currentLevel, levelProposalStart[level], levelProposalCount[level], hasVoted[level][voter][proposalId], sdgnrs.balanceOf(voter), levelVaultOwner[level], vault.isVaultOwner(voter), levelSdgnrsSnapshot[level], proposals[proposalId].approveWeight/rejectWeight

Attack angles:

a) **Flash-loan sDGNRS for vote weight:** sdgnrs.balanceOf(voter) is read at vote time (line 416), not from a snapshot. Attacker flash-loans sDGNRS, votes with inflated weight, returns sDGNRS. This IS the classic flash-loan governance attack. INVESTIGATE: Is sDGNRS flash-loanable or atomically transferable in a way that allows borrow-vote-return in one transaction?

b) **Vault owner double-weight (lines 418-421):** Vault owner gets their sDGNRS balance PLUS 5% of snapshot. Can someone become vault owner mid-level just for voting? The check on line 418 allows it: `voter == levelVaultOwner[level] || (levelVaultOwner[level] == address(0) && vault.isVaultOwner(voter))`. If no vault owner has acted yet (levelVaultOwner is address(0)), the first person who is vault owner at vote time gets locked in. But if vault ownership changed between propose and vote, the new owner gets the bonus. Is this intended?

c) **Vote on every proposal independently:** A single voter can vote on ALL proposals in a level (different proposalIds). This means a whale can approve their preferred proposal AND reject all competitors. Is this intended? It gives disproportionate power to large holders. Document as INFO — this is by design per the contract comments.

d) **Zero-weight vote prevention (line 422):** `if (weight == 0) revert InsufficientStake()`. This prevents non-sDGNRS holders from voting. SAFE.

Total possible approve weight = total sDGNRS supply + 5% vault bonus = ~1.05e30. Well under uint128 max. SAFE.

f) **Proposal range validation (line 410):** `proposalId < start || proposalId >= start + count`. The `start + count` could overflow if proposalCount is near uint48 max. But count is uint8 (max 255) and start is uint48 — no overflow risk. SAFE.

**3. resolveLevel(uint24 level) (lines 443-498) — HIGH PRIORITY**

- resolveLevel reads: currentLevel, levelResolved[level], levelProposalCount[level], levelProposalStart[level], proposals[start+i].approveWeight/rejectWeight, balanceOf[address(this)], proposals[bestId].recipient

Timing doesn't change the outcome unless votes are still incoming. An attacker could call resolveLevel to "close" voting before opposition votes arrive. This is front-running the resolution. INVESTIGATE: Is there any time-lock or minimum voting period? Scanning the code — NO. resolveLevel can be called at any time after level starts (after at least one proposal exists, since it checks levelProposalCount). This means: (1) Proposer proposes, (2) Proposer's ally immediately calls resolveLevel, (3) Proposal wins with zero opposition because no one had time to vote. This is a real vector.

   BUT WAIT: resolveLevel is called by the game contract during level transitions (per D-07, Phase 124 wiring). In practice, it's called by DegenerusGameAdvanceModule._finalizeRngRequest at `charityResolve.resolveLevel(lvl - 1)`. But there's no `onlyGame` modifier — anyone CAN call it externally. The question is whether external calling is harmful.

   If someone front-runs the game's resolveLevel call: the game's call would then revert with LevelAlreadyResolved (line 445). The game's _finalizeRngRequest has the resolveLevel call but does it revert-on-revert? Check: if the call is a regular CALL (not try/catch), the game transaction reverts entirely, blocking day advancement. THIS COULD BE A GRIEFING VECTOR — an attacker calls resolveLevel before the game does, but the game contract reverts. Actually wait — re-read the code. resolveLevel increments currentLevel (line 449). If an attacker calls resolveLevel for the current level, the game's subsequent call would be for the SAME level number, which is now resolved, causing revert. But the game passes `lvl - 1` which is the level that just ended. If the attacker already resolved it, the game's call reverts.

   Actually, check more carefully. The game calls `charityResolve.resolveLevel(lvl - 1)` where lvl is the NEW level. So it's resolving the PREVIOUS level. If an attacker already resolved it, the game gets LevelAlreadyResolved revert. Is this caught? Check DegenerusGameAdvanceModule._finalizeRngRequest for try/catch around the resolveLevel call. This is CRITICAL — if uncaught, it bricks day advancement.

Starting from 0, after all levels: recipient can have at most the initial supply (1T * 1e18 = 1e30). uint256 max = 1.15e77. SAFE.
