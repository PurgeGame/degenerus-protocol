<!-- From research: BurnieCoin supply architecture -->
Supply struct: {totalSupply: uint128, vaultAllowance: uint128}
Initial: {totalSupply: 0, vaultAllowance: 2_000_000 ether}

Mint paths (increase totalSupply):
  _mint(to, amount)        -- if to != VAULT: totalSupply += amount
  _mint(VAULT, amount)     -- vaultAllowance += amount (NOT totalSupply)
  vaultMintTo(to, amount)  -- vaultAllowance -= amount, totalSupply += amount
  mintForCoinflip(to, amount) -- calls _mint
  mintForGame(to, amount)     -- calls _mint
  creditCoin(player, amount)  -- calls _mint

Burn paths (decrease totalSupply):
  _burn(from, amount)        -- if from != VAULT: totalSupply -= amount
  _burn(VAULT, amount)       -- vaultAllowance -= amount (NOT totalSupply)
  burnForCoinflip(from, amount) -- calls _burn
  burnCoin(target, amount)      -- calls _burn

Special:
  vaultEscrow(amount)  -- vaultAllowance += amount (totalSupply unchanged)
</interfaces>

**Part A: Core _mint and _burn routing**

1. Read BurnieCoin.sol _mint() function
2. Verify the conditional: if `to == VAULT`, increment vaultAllowance. Else, increment totalSupply.
3. Conservation check: in both branches, does supplyIncUncirculated() increase by `amount`?
   - Non-VAULT: totalSupply += amount, vaultAllowance unchanged -> total increases by amount. CORRECT.
   - VAULT: totalSupply unchanged, vaultAllowance += amount -> total increases by amount. CORRECT.

4. Read _burn() function
5. Verify the conditional: if `from == VAULT`, decrement vaultAllowance. Else, decrement totalSupply.
6. Conservation check: in both branches, does supplyIncUncirculated() decrease by `amount`?
   - Non-VAULT: totalSupply -= amount -> total decreases by amount. CORRECT.
   - VAULT: vaultAllowance -= amount -> total decreases by amount. CORRECT.

**Part B: vaultMintTo path**

1. Read vaultMintTo() function
2. Verify: vaultAllowance -= amount AND totalSupply += amount (or calls _mint with appropriate routing)
3. Conservation check: supplyIncUncirculated() should remain UNCHANGED (vaultAllowance down, totalSupply up by same amount)
4. Overflow check: can totalSupply + amount overflow uint128? At initial vaultAllowance = 2M ether, totalSupply can grow to at most ~2M ether before allowance is exhausted. uint128 max is ~3.4e38, 2M ether = 2e24. Safe.

**Part C: vaultEscrow path**

1. Read vaultEscrow() function
2. Verify: vaultAllowance += amount with NO totalSupply change
3. This INCREASES supplyIncUncirculated(). Document when this is called and why (game deposits to vault reserve).
4. Check: who can call vaultEscrow? Only the GAME contract? If so, is this access-controlled?
5. Conservation check: this is an EXPANSION of total supply, not conservation. Document as intentional.

**Part D: External caller paths**

Trace each external mint/burn function to verify it delegates to _mint/_burn correctly:
