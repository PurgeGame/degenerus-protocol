**VULN-01: Weighted Vulnerability Ranking**

Step 1: Define criteria and weights BEFORE scoring any function:

| Criterion | Weight | 1 (Low) | 5 (Medium) | 10 (High) |
|-----------|--------|---------|------------|-----------|
| Value moved | 40% | No value transfer | Indirect value (pool accounting) | Direct ETH/token transfer |
| Complexity | 20% | <50 LOC, no branches | 50-200 LOC, moderate branching | >200 LOC, deep nesting, delegatecall |
| External interactions | 15% | No external calls | 1-2 known contracts | 3+ calls, or calls to external protocols (Lido, VRF) |
| Prior coverage depth | 15% | Deep multi-phase audit | Single-phase review | Shallow or no prior review |
| Novelty of code | 10% | Unchanged for 6+ months | Modified in last month | Newly written in last month |

**VULN-02: Deep Adversarial Audit of Top 10**

For each of the top 10 ranked functions, perform a dedicated adversarial audit:

Per-function audit structure:
```
### Rank [N]: [function_name] in [contract]
**Score:** [total] (Value: [X], Complexity: [X], Interactions: [X], Coverage: [X], Novelty: [X])
**Prior coverage:** [which phases reviewed this, with what depth]

**Adversarial hypothesis:**
[Think like a C4A warden: what exploit would I submit? What is the most plausible attack vector?]

**Attack trace:**
1. [Precondition: what state must exist?]
2. [Action: what does the attacker call?]
3. [Mechanism: how does the exploit work?]
4. [Extraction: what value is gained?]

**Defense analysis:**
[For each step in the attack trace, identify the defense that prevents it]
[Cite specific file:line for each defense]

**Cross-system interaction:**
[Does this function interact with systems audited in Phases 26-27 in a way that was not tested?]
[Reference edge case findings from Plan 04 if relevant]

**Verdict:** PASS (with defense summary) | FINDING-severity (with exploitation details)
```

CRITICAL: For each function, construct the MOST PLAUSIBLE exploit hypothesis a C4A warden would submit. Do not just verify known properties -- think adversarially:
- Can an attacker manipulate inputs to reach an unexpected code path?
- Can an attacker use flash loans to change preconditions?
- Can an attacker exploit cross-function state inconsistency?
- Can an attacker use reentrancy through any external call?
- Can an attacker profit from rounding, overflow, or timing?

### Statistical Overview

- Score distribution: min, max, mean, median
- Number of top-10 functions from each contract
- Correlation: do high-value functions tend to have high coverage? (they should)

- New findings severity distribution: [C/H/M/L/I]
- Functions confirmed PASS after adversarial audit: [count]
