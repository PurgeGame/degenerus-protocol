   - When a coinflip wins, can the player auto-rebuy into the next day?
   - If the next day's RNG is already committed, can the player see the outcome before deciding?
   - What is the exact sequence: VRF callback -> state update -> auto-rebuy -> day advance?

3. Construct extraction scenario:
```
State: Day N coinflip resolved, VRF word for day N+1 already committed
       Alice has auto-rebuy enabled and just won day N flip

Question: Can Alice selectively auto-rebuy only into winning days?
```

4. Check defenses:

   - Is there an RNG lock during VRF callback processing?
   - Can the player cancel auto-rebuy between VRF callback and day advance?

**EDGE-07: Rounding Accumulation Analysis**

1. Identify ALL BPS division sites in the protocol:

   - Each site truncates (Solidity default floor division)
   - List every site with file:line and the formula

2. For each division site, calculate worst-case rounding loss:
   - Minimum: 0 wei (when division is exact)
   - Maximum: divisor - 1 wei per operation
   - For BPS divisions: max 9999 wei per operation (~0.00001 ETH)

3. Accumulation analysis:
   - How many times does a single user interaction trigger BPS divisions?
   - What is the maximum number of divisions in a single advanceGame call?
   - Can an attacker trigger many small operations to accumulate rounding?

4. Concrete worst-case example:
```
Scenario: 1000 purchase transactions at minimum amount
Each transaction: BPS split loses up to 9999 wei
Total rounding loss: 1000 * 9999 = ~0.01 ETH
Over 100 days: 100 * 0.01 = 1 ETH
Question: Where does the "lost" rounding go? Retained in pool? Lost to ether?
```

5. Determine: Is the rounding always in favor of the protocol (truncation means less distributed)? Or can it favor the user?

6. Cross-reference with INV-01: Does rounding accumulation threaten the claimablePool solvency invariant?
