**Current state:** 24,380B deployed bytecode, 196 bytes from the 24,576B limit. Optimizer: viaIR=true, runs=50 (already aggressive).

**Analysis approach (per D-01: best effort, per D-02: within JackpotModule only):**

1. **Error string analysis:** Search for all `require()` and `revert` statements with string messages. Each character in a string literal costs ~1 byte of bytecode. Identify strings that could be shortened without losing debugging value. Also check for custom errors (which are cheaper than string reverts) that could replace string reverts.

2. **Dead code analysis:** Search for any private/internal function in JackpotModule that is never called. With viaIR optimizer at runs=50, the compiler should already remove dead code, but check for functions that are called but whose return values are ignored or whose effects are no-ops under current constants.

3. **Redundant check analysis:** Look for:
   - Guards that can never trigger given the constant values (e.g., `if (count > MAX_BUCKET_WINNERS)` where count is already capped by scaleTraitBucketCountsWithCap)
   - Double-checking conditions already enforced by callers
   - Branching on conditions that are compile-time constant

4. **Struct/memory optimization:** Check for memory arrays or structs that could be replaced with stack variables. Memory allocation costs bytecode (MLOAD/MSTORE sequences).

5. **Library inlining bloat:** JackpotBucketLib functions are `internal` so they're inlined. Check if any function is called from multiple sites with identical arguments -- the compiler inlines each call site separately.

6. **NatSpec and metadata:** These don't affect deployed bytecode, skip.

7. **Existing patterns that are already optimal:** Document what can't be improved (e.g., unchecked blocks, tight loops, packed storage reads).

For each finding, estimate bytecode savings (rough: count removed/simplified instructions * ~1 byte per instruction).
