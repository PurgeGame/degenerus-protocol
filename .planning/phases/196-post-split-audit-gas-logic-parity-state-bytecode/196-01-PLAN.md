**Theoretical worst case for _processDailyEth (daily two-call split):**
- Pool >= 200 ETH -> max scale 6.36x -> bucket counts 159/95/50/1 = 305 total
- All 305 winners are unique addresses with autorebuy enabled
- Each winner's autorebuy triggers _processAutoRebuy which does: _calcAutoRebuy + _queueTickets + pool writes
- Final jackpot day (isFinalPhysicalDay=true, jackpotCounter + cs >= JACKPOT_LEVEL_CAP) means 100% of pool distributed
- Call 1 processes largest(159) + solo(1) = 160 winners
- Call 2 processes mid(95) + small(50) = 145 winners
- Each winner: _randTraitTicket (SSTORE for ticket consumption) + _handleSoloBucketWinner or _payNormalBucket (SSTORE for claimable/autorebuy) + event emission

**The test cannot practically construct 305 unique players who ALL hold trait-matching tickets for ALL 4 traits.** Trait matching depends on VRF entropy selecting specific trait IDs, and players must hold burned tickets with those traits. The existing system burns tickets into `traitBurnTicket[level]` which is a mapping(uint8 traitId => ...). Winners are selected from holders of the specific winning trait.

**Practical approach:** Use as many unique signers as possible (305), have them all buy large quantities of tickets across multiple levels so trait distribution is broad. The VRF word determines which traits win, and the test can't control that. But with 305 players each buying tickets, most buckets will have plenty of holders.

**NOTE:** If 305 unique autorebuy players is impractical in test setup time (each needs a purchase TX = 305 TXs), batch them. The test may take several minutes. Set `this.timeout(1_200_000)` (20 min) for the worst-case section.
