## D-11 rngLocked Invariant (Verbatim User-Locked Statement)

> "During the rngLocked window (VRF request → fulfillment), across the newly-packed housekeeping step: (a) NO far-future ticket queue write may occur, AND (b) NO write may land in the active (read-side) buffer. Writes to the write-side buffer at the current level ARE PERMITTED — they drain next round with the next VRF word. rngLocked is NOT a blanket ticket-queueing block."

This is the load-bearing invariant for TRNX-01. Every path walk below verifies clauses (a) and (b) hold AND verifies that writes to the current-level write buffer remain correctly permitted during the rngLocked window. The audit models `rngLockedFlag` (the storage slot at `DegenerusGameStorage.sol:279`) as a guard that fires ONLY on the two enumerated surfaces — far-future queue writes (enforced at `DegenerusGameStorage.sol:570 / 602 / 658`) and read-side buffer writes (prevented structurally by the single-origin caller pattern; see Buffer-Swap Site Citation below). Write-side buffer writes during the rngLocked window are permitted by design and drain on the next VRF word.

## Buffer-Swap Site Citation (per D-12)

**Swap fires at RNG REQUEST TIME (not fulfillment).**

**Concrete site — daily-RNG buffer swap:**

```solidity
// contracts/modules/DegenerusGameAdvanceModule.sol:282-294
            // RNG: use existing word or request new one
            bool bonusFlip = (inJackpot && jackpotCounter == 0) || lvl == 0;
            (uint256 rngWord, uint32 gapDays) = rngGate(
                ts,
                day,
                purchaseLevel,
                lastPurchase,
                bonusFlip
            );
            psd += gapDays;
            if (rngWord == 1) {
                _swapAndFreeze(purchaseLevel);   // <-- BUFFER SWAP FIRES HERE AT RNG REQUEST TIME
                stage = STAGE_RNG_REQUESTED;
                break;
            }
```

**Why this is RNG request time (not fulfillment):** `rngGate` at AdvanceModule:1133-1199 returns the sentinel value `1` specifically on the VRF-request path. Three reachable sub-paths in `rngGate` return `1`:

- **Fresh request (AdvanceModule:1197):** `_requestRng(isTicketJackpotDay, lvl); return (1, 0);` — invokes `_requestRng → vrfCoordinator.requestRandomWords(...) → _finalizeRngRequest`, which sets `rngLockedFlag = true` at AdvanceModule:1579 and `vrfRequestId = requestId` at AdvanceModule:1576.
- **Timeout retry (AdvanceModule:1190):** `_requestRng(isTicketJackpotDay, lvl); return (1, 0);` — same code path, different precondition (12h stall on a prior request).
- **Waiting-for-VRF (AdvanceModule:1193):** `revert RngNotReady();` — does NOT return 1; the caller never reaches the `_swapAndFreeze` line.

Only the two `return (1, 0)` sub-paths reach `_swapAndFreeze` at AdvanceModule:292. Both are inside the VRF request tx (the request is already sent before `rngGate` returns). The fulfillment (`rawFulfillRandomWords` at AdvanceModule:1690) is a SEPARATE transaction originated by the Chainlink VRF coordinator; it does NOT invoke `_swapAndFreeze`. Therefore the swap is strictly at RNG request time, never at fulfillment.

**Read/write buffer flip semantics (per `_swapAndFreeze` at `contracts/storage/DegenerusGameStorage.sol:768-774` + `_swapTicketSlot` at `DegenerusGameStorage.sol:758-763`):**

```solidity
// contracts/storage/DegenerusGameStorage.sol:758-774
    function _swapTicketSlot(uint24 purchaseLevel) internal {
        uint24 rk = _tqReadKey(purchaseLevel);
        if (ticketQueue[rk].length != 0) revert E();      // Pre-swap guard: read slot MUST be drained
        ticketWriteSlot = !ticketWriteSlot;                // Flip the write-slot bit (was read → now write; was write → now read)
        ticketsFullyProcessed = false;                     // New read slot has tickets; mark not-yet-drained
    }

    function _swapAndFreeze(uint24 purchaseLevel) internal {
        _swapTicketSlot(purchaseLevel);
        if (!prizePoolFrozen) {
            prizePoolFrozen = true;
            prizePoolPendingPacked = 0;
        }
    }
```

Pre-swap: read buffer = buffer-A (empty — drained by the pre-finalize gate at AdvanceModule:257-278 + the in-cycle `_runProcessTicketBatch` at AdvanceModule:350), write buffer = buffer-B (accumulating current-level tickets). Swap fires at line 292 (`_swapAndFreeze`). Post-swap: read buffer = buffer-B (previously written tickets now ready to drain against the pending VRF word, next round), write buffer = buffer-A (empty, freshly initialized; receives NEW writes at current level during the rngLocked window).

**Per D-11 clause:** writes to the new write buffer (buffer-A) at the current level are PERMITTED during the rngLocked window — these writes land via `_queueTickets` / `_queueTicketsScaled` / `_queueTicketRange` which compute `_tqWriteKey(targetLevel)` at `DegenerusGameStorage.sol:573 / 605 / 659` (write key = the freshly-swapped slot bit). Writes to the new read buffer (buffer-B) would be BLOCKED because all queueing code paths write to `_tqWriteKey` (write key), not `_tqReadKey` (read key); structurally impossible to write to the read slot. Writes to far-future (`_tqFarFutureKey`) are guarded by the `isFarFuture && rngLockedFlag && !rngBypass` revert at `DegenerusGameStorage.sol:570 / 602 / 658`.

**The `rngBypass=true` exception for `_processPhaseTransition`:** `_queueTickets` at AdvanceModule:1480-1491 passes `rngBypass=true` for the vault perpetual tickets (`targetLevel = purchaseLevel + 99`, always far-future). This exception is SAFE because the caller is `_processPhaseTransition`, which is only reachable from the `advanceGame` phase-transition branch (AdvanceModule:307) — a same-tx origin, not an external attacker. The far-future write honors D-11 clause (a) by being structurally unreachable from external `_queueTickets` callers under `rngLockedFlag = true` (external purchase paths in MintModule go through `_queueTickets` / `_queueTicketsScaled` with `rngBypass=false`, triggering the revert).

**Other swap sites (cross-referenced, non-primary per D-12):**

- `DegenerusGameAdvanceModule.sol:595` — `_swapTicketSlot(lvl + 1)` inside `_handleGameOverPath` between Round 1 and Round 2 of the best-effort game-over drain. This swap does NOT fire at a VRF request; it fires after Round 1 fully drains and before Round 2 runs. No new VRF word is introduced — both rounds reuse the game-over entropy `_gameOverEntropy` populated earlier (see `232.1-01-FIX.md` §Revision 3 R3.d "Known issue: non-determinism"). No rngLocked interaction.
- `DegenerusGameAdvanceModule.sol:1082` — `_swapTicketSlot(purchaseLevel_)` inside `requestLootboxRng` just before the mid-day VRF `vrfCoordinator.requestRandomWords` call at AdvanceModule:1088. Therefore this swap is irrelevant to TRNX-01's rngLocked invariant (it fires only when daily rngLocked is FALSE). Included here for completeness; not load-bearing.
