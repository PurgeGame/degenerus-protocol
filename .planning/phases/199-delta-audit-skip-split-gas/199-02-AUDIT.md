## Section B: Non-Daily Caller Parity (D-05)

**Audit question:** Does the unified `_processDailyEth(SPLIT_NONE, isJackpotPhase=false)` produce identical results to the deleted `_distributeJackpotEth` + `_processOneBucket`?

### Behavioral Comparison

| Aspect | Old (`_distributeJackpotEth`) | New (`_processDailyEth(SPLIT_NONE, false)`) | Match? |
|--------|-------------------------------|---------------------------------------------|--------|
| Early exit on ethPool==0 | `if (ethPool == 0) return 0` | `if (ethPool == 0) return 0` (line 1194-1196) | YES |
| Unit calculation | `priceForLevel(lvl+1) >> 2` | `priceForLevel(lvl+1) >> 2` (line 1198) | YES |
| Solo bucket index | `soloBucketIndex(entropy)` | `soloBucketIndex(entropy)` (line 1199) | YES |
| Share computation | `bucketShares(ethPool, shareBps, bucketCounts, remainderIdx, unit)` | Same call (lines 1200-1202) | YES |
| Bucket iteration order | Sequential: `0, 1, 2, 3` | Largest-first: `bucketOrderLargestFirst` (lines 1204-1206) | **DIFFERENT** |
| Entropy chain | `entropyStep(state ^ (traitIdx << 64) ^ share)` via `_resolveTraitWinners` | `entropyStep(entropyState ^ (traitIdx << 64) ^ share)` inline (lines 1233-1235) | YES (same formula) |
| Winner selection | `_randTraitTicket(traitBurnTicket[lvl], ..., uint8(200 + traitIdx))` via `_resolveTraitWinners` | `_randTraitTicket(traitBurnTicket[lvl], ..., uint8(200 + traitIdx))` (lines 1243-1249) | YES |
| Per-winner payout | `share / totalCount` via `_resolveTraitWinners` | `share / totalCount` (line 1252) | YES |
| Payout dispatch | `_addClaimableEth` + `JackpotEthWin` via `_resolveTraitWinners` | `_payNormalBucket` -> `_addClaimableEth` + `JackpotEthWin` (lines 1273-1277) | YES |

| claimablePool update | `ctx.liabilityDelta` summed, written once | `liabilityDelta` summed, written once (lines 1281-1283) | YES |
| MAX_BUCKET_WINNERS cap | Applied in `_resolveTraitWinners` | Applied inline (line 1238) | YES |

### Analysis of Differences

**1. Bucket iteration order (DIFFERENT -- but economically equivalent):**

The old `_distributeJackpotEth` iterated `0, 1, 2, 3` (sequential). The new function iterates via `bucketOrderLargestFirst`. Since SPLIT_NONE processes ALL 4 buckets regardless of order, and since:
- Each bucket's `share` is pre-computed in the `shares[4]` array (line 1200)
- Each bucket's `count` is pre-computed in `bucketCounts[4]`
- `perWinner = share / totalCount` is per-bucket independent
- The entropy chain XORs `traitIdx` into the derivation (`entropyState ^ (uint256(traitIdx) << 64) ^ share`), so each bucket gets deterministic entropy regardless of iteration order

The only effect of different iteration order is that the cumulative `entropyState` differs across buckets. Since entropy is XORed with the bucket-specific `traitIdx` and `share`, the final entropy state after all 4 buckets will differ between old and new. However, this entropy state is NOT used after the loop -- it's discarded. Each bucket's winner selection is determined by its own `traitIdx` and `share`, making the economic outcome identical.

**Wait -- the entropy IS cumulative.** Each bucket's entropy input is the OUTPUT of the previous bucket's `entropyStep`. So iteration order DOES affect winner selection within buckets 2, 3, 4 (the ones that receive entropy already transformed by prior buckets). This means the new function selects DIFFERENT winners for buckets processed later in the sequence.

However: The economic VALUE each winner receives is identical (same shares, same counts). The TOTAL payout is identical. Only WHICH addresses are selected differs. This is a design-accepted change: the unified function trades sequential iteration for largest-first iteration, which was already the pattern for the daily path. Since winner selection is pseudo-random anyway, this is not an economic parity concern.

**2. Whale pass removal for non-daily callers (BEHAVIORAL CHANGE -- INTENTIONAL):**

The old `_distributeJackpotEth` routed through `_resolveTraitWinners` which did NOT have whale pass logic. The old `_processDailyEth` DID have whale pass via `_handleSoloBucketWinner`. Since `_distributeJackpotEth` was only called by non-daily paths (early-burn, terminal), whale pass was NEVER awarded for non-daily callers in the old code either.

In the new code, `isJackpotPhase=false` skips the whale pass guard at line 1255. This is FUNCTIONALLY IDENTICAL to the old behavior.

To verify: the old `_resolveTraitWinners` (lines 1312-1392) processes ALL buckets identically -- no special solo bucket handling. The old `_processOneBucket` called `_resolveTraitWinners` for every bucket including the solo bucket. So the old code gave the solo bucket winner 100% ETH (no whale pass) in non-daily context. The new code does the same via the `else` branch at line 1271 (`_payNormalBucket`).

CONFIRMED: `_payNormalBucket` (lines 1456-1478) and `_resolveTraitWinners` (lines 1312-1392) have identical per-winner logic: `_addClaimableEth(w, perWinner, entropy)` + `JackpotEthWin` emission.

### Verdict: CLEAN

Economic outcomes are identical for non-daily callers. Winner selection differs due to iteration order change (largest-first vs sequential), which is design-accepted. Whale pass behavior is preserved (never awarded for non-daily callers in either old or new code). Payout computation, entropy derivation formula, and claimable accounting are all identical.

| B-01 | INFO | Iteration order changed from sequential (0-3) to largest-first. Winner selection differs but economic totals are identical. Design-accepted. |

## Section C: Skip-Split Parity (D-06)

**Audit question:** Does SPLIT_NONE with N total winners produce identical economic outcomes to SPLIT_CALL1 (call1 winners) + SPLIT_CALL2 (call2 winners) when N = call1 + call2?

### Trace: SPLIT_NONE Path

When `totalWinners <= JACKPOT_MAX_WINNERS (160)`, `payDailyJackpot` sets `splitMode = SPLIT_NONE` (line 448).

In `_processDailyEth`:
1. **SPLIT_CALL2 guard skipped** (line 1190: `splitMode == SPLIT_CALL2` is false)
2. **ethPool used directly** (passed from caller, not from storage)
3. **call1Bucket mask NOT built** (line 1210: `splitMode != SPLIT_NONE` is false, so the `if` block is skipped entirely)
4. **Loop skip logic inactive**: Since `call1Bucket` is all-false:
   - Line 1226: `splitMode == SPLIT_CALL1 && !call1Bucket[traitIdx]` -- `SPLIT_CALL1` is false, so always false
   - Line 1227: `splitMode == SPLIT_CALL2 && call1Bucket[traitIdx]` -- `SPLIT_CALL2` is false, so always false
   - All 4 buckets process in the single call
5. **resumeEthPool NOT written** (line 1286: `splitMode == SPLIT_CALL1` is false)

### Trace: SPLIT_CALL1 + SPLIT_CALL2 Path

When `totalWinners > 160`, `payDailyJackpot` sets `splitMode = SPLIT_CALL1` (line 449).

**Call 1 (SPLIT_CALL1):**
1. `ethPool` = dailyEthBudget (passed from caller)
2. `call1Bucket` mask built: `order[0]` (largest) + `remainderIdx` (solo) -- or `order[0]` + `order[1]` if they overlap
3. Loop processes only buckets where `call1Bucket[traitIdx]` is true (2 buckets)
4. `resumeEthPool = uint128(ethPool)` written at line 1287

**Call 2 (SPLIT_CALL2):** (separate advanceGame call, new VRF word)
1. `ethPool = uint256(resumeEthPool)` read from storage (line 1191)
2. `resumeEthPool = 0` cleared (line 1192)
3. `call1Bucket` mask rebuilt identically (same entropy derivation)
4. Loop processes only buckets where `call1Bucket[traitIdx]` is false (2 buckets)
5. `resumeEthPool` NOT written (line 1286: `splitMode == SPLIT_CALL1` is false for CALL2)

### Economic Equivalence Analysis

| Aspect | SPLIT_NONE | SPLIT_CALL1 + SPLIT_CALL2 | Equivalent? |
|--------|------------|---------------------------|-------------|
| ethPool amount | Same budget | Same budget (stored/restored via resumeEthPool) | YES |
| Share computation | `bucketShares(ethPool, ...)` once | Same computation in both calls (same ethPool value) | YES |
| Per-winner payout | `share / totalCount` per bucket | Same formula per bucket | YES |
| Total payout | Sum of all 4 buckets | Sum of call1 buckets + sum of call2 buckets | YES |
| Bucket ordering | `bucketOrderLargestFirst` | Same ordering in both calls | YES |
| claimablePool update | One atomic write | Two writes (one per call) | YES (sum identical) |
| **Entropy chain** | **Continuous across all 4 buckets from single VRF word** | **Call 1: from VRF word 1. Call 2: from VRF word 2 (different advanceGame call)** | **NO -- different VRF words** |
| **Winner selection** | **Determined by single entropy chain** | **Call 1 winners from word 1, call 2 winners from word 2** | **NO -- different winners selected** |

### Entropy Divergence (Design-Accepted)

The skip-split path processes all 4 buckets with a single VRF word's entropy chain. The two-call split uses VRF word 1 for call 1 (largest + solo) and VRF word 2 for call 2 (mid buckets). Since different VRF words produce different entropy chains, the mid-bucket winners will be different people in skip-split vs two-call split.

This is design-accepted per Phase 198 CONTEXT D-01: the skip-split is an optimization that avoids a second advanceGame call + VRF request. The economic TOTALS are identical (same ETH distributed, same share percentages, same bucket sizes). Only the specific winners differ, which is inherent to using different random seeds.

### Storage Safety

- **SPLIT_NONE never writes resumeEthPool** (line 1286 guard confirmed)
- **SPLIT_NONE never reads resumeEthPool** (line 1190 guard: only SPLIT_CALL2 reads it)
- No stale value risk: if resumeEthPool was zero before, it stays zero

### Verdict: CLEAN

SPLIT_NONE is economically equivalent to SPLIT_CALL1+SPLIT_CALL2. Total payouts are identical. Entropy divergence (different winners) is design-accepted. Storage lifecycle is correct -- no stale resumeEthPool possible.

| ID | Severity | Finding |
|----|----------|---------|
| C-01 | INFO | Skip-split selects different winners than two-call split due to single VRF word vs two. Economic totals identical. Design-accepted per Phase 198 D-01. |

## Section D: resumeEthPool Lifecycle (D-07)

**Audit question:** Is `resumeEthPool` never written on SPLIT_NONE, never stale across levels?

### Storage Declaration

```solidity
// DegenerusGameStorage.sol line 968
uint128 internal resumeEthPool;
```

### Path Analysis

**SPLIT_NONE:**
- Write: Line 1286 guard `if (splitMode == SPLIT_CALL1)` -- NOT entered. No write. VERIFIED.
- Read: Line 1190 guard `if (splitMode == SPLIT_CALL2)` -- NOT entered. No read. VERIFIED.
- Result: `resumeEthPool` is completely untouched. If it was 0 before, it stays 0.

**SPLIT_CALL1 (call 1 of split):**
- Read: Line 1190 guard NOT entered (SPLIT_CALL2 check fails for CALL1).
- Write: Line 1287: `resumeEthPool = uint128(ethPool)`. Stores the full ethPool value.
- Value written: `uint128(ethPool)` where `ethPool` = `dailyEthBudget`. This is the ORIGINAL pool, not `paidEth`. CORRECT -- call 2 needs the original pool to recompute share allocations.

**SPLIT_CALL2 (call 2 of split):**
- Read: Line 1191: `ethPool = uint256(resumeEthPool)`. Reads stored value.
- Clear: Line 1192: `resumeEthPool = 0`. Clears immediately after read.
- Order: Read-then-clear on consecutive lines. No window for reentrancy (no external calls between lines 1191-1192). SAFE.
- The read value is used to recompute `shares`, `bucketCounts` (passed from caller who also reads from `resumeEthPool` at line 1140), and `unit`.

### Cross-Level Safety

**Question:** Can `resumeEthPool` carry a stale nonzero value from a prior level into a new level?

**Analysis:** Level transitions occur when `jackpotCounter >= JACKPOT_LEVEL_CAP` (checked in AdvanceModule line 424). The jackpot phase ends via `_endPhase()` (line 425). For resumeEthPool to be stale:

1. SPLIT_CALL1 writes `resumeEthPool` (day N, level L)
2. Before SPLIT_CALL2 reads it, level transitions

This CANNOT happen because:
- AdvanceModule checks `resumeEthPool != 0` at line 414 BEFORE checking `dailyJackpotCoinTicketsPending` (line 422) and before running a fresh daily (line 436)
- If `resumeEthPool != 0`, it routes to `payDailyJackpot(true, lvl, rngWord)` which calls `_resumeDailyEth` (SPLIT_CALL2), clearing it
- The stage machine CANNOT advance to coin/ticket phase or a new daily until `resumeEthPool` is cleared
- Level transition only happens after `jackpotCounter >= JACKPOT_LEVEL_CAP`, which requires completing all 5 days of dailies, each requiring resume completion first

**Question:** Can `resumeEthPool` carry a stale value from a prior level if SPLIT_NONE was used for all 5 days?

**Analysis:** If SPLIT_NONE fires for every daily, `resumeEthPool` is never written. If it was 0 at level start, it stays 0. The AdvanceModule check `resumeEthPool != 0` (line 414) naturally skips the resume stage. No stale data.

**Question:** Can a revert leave `resumeEthPool` stale?

**Analysis:** All JackpotModule functions are called via `delegatecall` from the main game contract. If a delegatecall reverts, ALL storage writes from that call are reverted (EVM guarantee). So if SPLIT_CALL1 writes `resumeEthPool` and then the transaction reverts, the write is undone. No stale data possible from reverts.

### Verdict: CLEAN

`resumeEthPool` lifecycle is correct in all 3 split modes. No stale data possible across levels or from reverts. Read-before-clear order is safe.
