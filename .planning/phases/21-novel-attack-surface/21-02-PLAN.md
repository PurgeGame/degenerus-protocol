**Section: `## NOVEL-03: Griefing Vector Enumeration`**

Enumerate ALL griefing vectors on new sDGNRS/DGNRS entry points. For EACH vector, provide: Description, Attack Cost (gas + capital), Victim Impact, Duration, Verdict (GRIEFABLE/NEGLIGIBLE/BLOCKED), Mitigation Status.

**Griefing Vector 1: Dust Burn Spam**
- Attack: Call DGNRS.burn(1) or sDGNRS.burn(1) repeatedly with minimum amounts.
- Cost: Gas per burn tx (~100k-200k gas for burn with 5+ external calls). At $0.10/tx, 1000 spam burns = $100.
- Impact: Each dust burn reduces totalSupply by 1 wei, claims ~0 reserves (rounding to zero for dust amounts). No impact on other users' burn values. No state bloat (same storage slots updated).
- Trace: StakedDegenerusStonk.sol line ~384: `if (amount == 0) revert ZeroAmount()` -- burn(0) is blocked. burn(1) proceeds but claims effectively nothing due to integer division rounding down.
- Verdict: NEGLIGIBLE -- attacker wastes gas for no impact.

**Griefing Vector 2: Gas Limit Attack on Burn**
- Attack: Call DGNRS.burn() with exactly enough gas to execute the DGNRS._burn but not enough for the full sDGNRS.burn() call chain.
- Trace: DegenerusStonk.sol burn() makes: _burn (cheap), stonk.burn() (expensive: reads, writes, up to 3 external calls), then forwards ETH/stETH/BURNIE.
- Impact: If gas runs out mid-execution, EVM reverts the ENTIRE transaction atomically. No partial execution possible. All state changes revert.
- Verdict: BLOCKED -- EVM atomicity prevents partial burn execution.

**Griefing Vector 3: Block Stuffing Before Burns**
- Attack: Fill blocks with high-gas txs to delay other users' burns (e.g., before gameOver to prevent burning at pre-gameOver prices).
- Cost: Extremely expensive (full block gas limit * N blocks).
- Impact: Temporary delay of burns. Does not change burn payout values since reserves don't change.
- Verdict: NEGLIGIBLE -- cost vastly exceeds benefit, standard blockchain DoS.

**Griefing Vector 4: Approval Griefing (front-run approve with transferFrom)**
- Attack: Monitor pending DGNRS approve() calls, front-run with transferFrom() to spend old allowance, then victim's approve sets new allowance, attacker spends that too.
- Trace: DegenerusStonk.sol uses standard ERC20 approve(). This is the known ERC20 approval race condition (not specific to DGNRS).
- Impact: Only affects users who change non-zero allowances. Standard mitigation: use increaseAllowance/decreaseAllowance or set to 0 first.
- Verdict: KNOWN -- standard ERC20 behavior, documented in DELTA-L-01 context.

**Griefing Vector 5: Pool Exhaustion Racing**
- Attack: Front-run legitimate pool transfers with own claims to exhaust pool balances.
- Trace: All pool transfers are `onlyGame` -- only the game contract can call transferFromPool. Users cannot directly race pool transfers.
- Verdict: BLOCKED -- onlyGame restriction prevents public access to pool operations.

After all vectors, add a **Griefing Summary Table**: | Vector | Cost | Impact | Verdict |
