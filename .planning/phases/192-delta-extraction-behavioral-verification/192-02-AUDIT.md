## Section 1: Intentional Change -- Whale Pass Daily Path Restriction (D-07)

### 1.1: Old Behavior Trace

Reconstructed from `git show 93c05869^:contracts/modules/DegenerusGameJackpotModule.sol`.

**Old `_resolveTraitWinners` (line 1372)** contained inline solo-bucket detection:

```solidity
// Special handling for solo bucket (winnerCount == 1)
bool isSoloBucket = (winnerCount == 1);
// ...
for (uint256 i; i < len; ) {
    address w = winners[i];
    if (w != address(0)) {
        if (isSoloBucket) {
            (uint256 claimDelta, uint256 paid, uint256 wpSpent, uint256 newEntropy)
                = _processSoloBucketWinner(w, perWinner, entropyState);
            // ... emit JackpotTicketWinner(..., AWARD_ETH) for paid
            totalWhalePassSpent += wpSpent;
            // ...
        } else {
            // Normal bucket: pay full amount as ETH
            uint256 claimableDelta = _addClaimableEth(w, perWinner, entropyState);
            // ... emit JackpotTicketWinner(..., AWARD_ETH)
        }
    }
}
```

When `winnerCount == 1`, `_processSoloBucketWinner` applied the 75/25 ETH/whale-pass split. When `winnerCount > 1`, all winners received straight ETH.

**Callers of `_resolveTraitWinners` in the old code:**

1. **`_processOneBucket` (line 1335)** -- called from `_distributeJackpotEth` (line 1302), which was the shared entry point for:
   - **`runTerminalJackpot` (line 253)** -- x00 terminal jackpot path
   - **`_executeJackpot` -> `_runJackpotEthFlow` (line 1146)** -- early-burn path (from `payDailyJackpot` non-daily branch at line 492)
2. **`payDailyJackpot` daily path** -- did NOT call `_resolveTraitWinners` for ETH; it used `_processDailyEth` (line 1187) which had its OWN inline loop. However, `_processDailyEth` did NOT have solo-bucket detection -- it paid ALL winners straight ETH with `_addClaimableEth`.

**Key finding in old code:** The solo-bucket whale pass split lived in `_resolveTraitWinners`, which was called by `_distributeJackpotEth`. The callers of `_distributeJackpotEth` were:
- `runTerminalJackpot` (external, for x00 levels)
- `_runJackpotEthFlow` -> `_executeJackpot` (early-burn path)

The daily path (`_processDailyEth`) did NOT go through `_resolveTraitWinners` and therefore did NOT get the whale pass split in the old code either. Wait -- this needs careful re-examination.

Looking more carefully at the old `_processDailyEth` (line 1187 in the old code): it had its own inline loop that called `_addClaimableEth` directly for ALL winners -- no solo-bucket detection, no whale pass split. The whale pass split only existed in `_resolveTraitWinners` via `_processSoloBucketWinner`.

**Corrected old path analysis:**

- `_processDailyEth` (daily jackpot ETH): Straight ETH for ALL winners, no whale pass split
- `_resolveTraitWinners` (called from `_distributeJackpotEth`): Whale pass split for solo buckets

`_distributeJackpotEth` was called by:
- `runTerminalJackpot` -> `_distributeJackpotEth`
- `payDailyJackpot` (non-daily/early-burn) -> `_executeJackpot` -> `_runJackpotEthFlow` -> `_distributeJackpotEth`

So in the OLD code, whale pass was awarded on: terminal path and early-burn path (when solo bucket occurred in `_resolveTraitWinners`), but NOT on the daily jackpot path (which used `_processDailyEth`).

### 1.2: New Behavior Trace

Current code (post-commits 93c05869 + 520249a2):

**`_resolveTraitWinners` (line 1317):** NO solo-bucket detection. All winners get straight ETH via `_addClaimableEth` + `JackpotEthWin`. The `isSoloBucket` variable and `_processSoloBucketWinner` call are completely removed from this function.

**`_processDailyEth` (line 1128):** Now has the `isFinalDay` parameter (added in 93c05869). Contains `traitIdx == remainderIdx` check (line 1193) to detect the solo bucket. When detected:
- Calls `_handleSoloBucketWinner` (line 1426) which calls `_processSoloBucketWinner` for the 75/25 split + whale pass + optional DGNRS reward on final day

**Early-burn path:** `payDailyJackpot` (non-daily) -> `_executeJackpot` -> `_runJackpotEthFlow` -> `_distributeJackpotEth` -> `_processOneBucket` -> `_resolveTraitWinners` -> straight ETH (no whale pass)

**Terminal path:** `runTerminalJackpot` -> `_distributeJackpotEth` -> `_processOneBucket` -> `_resolveTraitWinners` -> straight ETH (no whale pass)

**Early-bird lootbox path:** `_runEarlyBirdLootboxJackpot` (line 626) -> `_randTraitTicket` directly, awards tickets not ETH; whale pass was never relevant here.

**BAF path:** `runBafJackpot` (line 2375) -> its own winner selection, not through `_resolveTraitWinners`; unaffected.

### 1.3: Path Enumeration

| Path | Old: Whale Pass? | New: Whale Pass? | Change |
|------|------------------|------------------|--------|
| Daily normal bucket | No (straight ETH in `_processDailyEth`) | No (straight ETH via `_payNormalBucket`) | Unchanged |
| Daily solo bucket | No (straight ETH in old `_processDailyEth`) | Yes (75/25 split via `_handleSoloBucketWinner`) | **INTENTIONAL: ADDED** |
| Early-burn solo bucket | Yes (via `_resolveTraitWinners` -> `_processSoloBucketWinner`) | No (straight ETH in `_resolveTraitWinners`) | **INTENTIONAL: REMOVED** |
| Terminal solo bucket | Yes (via `_resolveTraitWinners` -> `_processSoloBucketWinner`) | No (straight ETH in `_resolveTraitWinners`) | **INTENTIONAL: REMOVED** |
| Early-bird lootbox | N/A (tickets only) | N/A (tickets only) | Unchanged |
| BAF | N/A (own winner selection) | N/A (own winner selection) | Unchanged |

**Summary of path changes:**
- Whale pass split was MOVED from the early-burn/terminal paths (via `_resolveTraitWinners`) to the daily jackpot path (via `_processDailyEth` -> `_handleSoloBucketWinner`).
- The daily path previously did NOT have whale pass split (old `_processDailyEth` paid all winners straight ETH). Now it DOES for solo buckets.
- The early-burn and terminal paths previously DID have whale pass split (via `_resolveTraitWinners`). Now they DO NOT.

### 1.4: Correctness Argument

The whale pass path change is **INTENTIONAL-CORRECT**:

1. **Whale pass is a daily reward mechanic.** Moving the whale pass split to the daily-only path (`_processDailyEth`) and removing it from early-burn/terminal paths aligns whale pass awards with the daily jackpot lifecycle.

2. **No value loss on removed paths.** Early-burn and terminal paths now pay 100% ETH instead of 75% ETH + 25% whale pass conversion. Players on these paths receive the full ETH value directly rather than having 25% converted to whale passes with ETH locked in `futurePrizePool`.

3. **`_processSoloBucketWinner` is unchanged.** The 75/25 split logic itself (lines 1511-1554) is identical in old and new code:
   - `quarterAmount = perWinner >> 2`
   - `whalePassCount = quarterAmount / HALF_WHALE_PASS_PRICE`
   - If count > 0: `ethAmount = perWinner - whalePassCost`, credit ETH, award whale passes
   - If count == 0: pay full amount as ETH
   The function is the same; only its call site moved from `_resolveTraitWinners` to `_handleSoloBucketWinner` (called from `_processDailyEth`).

4. **Daily path now includes solo-bucket handling it previously lacked.** Old `_processDailyEth` treated all buckets identically (straight ETH). New `_processDailyEth` detects solo buckets via `traitIdx == remainderIdx` and applies the 75/25 split, which is a more intentional placement of the reward mechanic.

**Verdict: INTENTIONAL-CORRECT**

## Section 2: Intentional Change -- DGNRS Solo Reward Fold (D-08)

### 2.1: Old Behavior Trace

Reconstructed from `git show 93c05869^`.

**Old flow (3 components):**

1. **`awardFinalDayDgnrsReward` in JackpotModule (line 620, external):**
   ```solidity
   function awardFinalDayDgnrsReward(uint24 lvl, uint256 rngWord) external {
       uint256 dgnrsPool = dgnrs.poolBalance(IStakedDegenerusStonk.Pool.Reward);
       uint256 reward = (dgnrsPool * FINAL_DAY_DGNRS_BPS) / 10_000;
       if (reward == 0) return;
       // Re-derive solo bucket from stored Day 5 winning traits
       uint256 entropy = rngWord ^ (uint256(lvl) << 192);
       uint8 soloIdx = JackpotBucketLib.soloBucketIndex(entropy);
       uint32 packed = lastDailyJackpotWinningTraits;
       uint8[4] memory traitIds = JackpotBucketLib.unpackWinningTraits(packed);
       // Pick winner using salt 254
       address[] memory winners = _randTraitTicket(
           traitBurnTicket[lvl], entropy, traitIds[soloIdx], 1, 254
       );
       if (winners.length > 0 && winners[0] != address(0)) {
           dgnrs.transferFromPool(IStakedDegenerusStonk.Pool.Reward, winners[0], reward);
           emit JackpotTicketWinner(winners[0], lvl, traitIds[soloIdx], reward, 0, AWARD_DGNRS);
       }
   }
   ```

2. **`_awardFinalDayDgnrsReward` in AdvanceModule (line 798, private wrapper):**
   ```solidity
   function _awardFinalDayDgnrsReward(uint24 lvl, uint256 rngWord) private {
       // delegatecall to JackpotModule.awardFinalDayDgnrsReward(lvl, rngWord)
   }
   ```

3. **Call site in AdvanceModule (line 414):** Called AFTER `payDailyJackpot`:
   ```solidity
   _awardFinalDayDgnrsReward(lvl, rngWord);
   ```

**Old entropy path:**
- The old function called `_randTraitTicket` (address-only version) with salt `254`
- Entropy derivation: `keccak256(abi.encode(rngWord ^ (uint256(lvl) << 192), traitIds[soloIdx], 254, 0))`
- This is a DIFFERENT salt from the daily ETH path which uses `uint8(200 + traitIdx)` (salts 200-203)
- The old code re-derived the solo bucket index and re-picked a winner from the same trait bucket but with different randomness
- Result: the DGNRS reward winner could be a DIFFERENT player than the ETH winner

### 2.2: New Behavior Trace

Current `_processDailyEth` (line 1128) with `_handleSoloBucketWinner` (line 1426):

```solidity
// In _processDailyEth, when traitIdx == remainderIdx:
(uint256 claimDelta, uint256 paidDelta, uint256 newEntropy) = _handleSoloBucketWinner(
    w, lvl, traitIds[traitIdx], ticketIndexes[0], perWinner, entropyState, isFinalDay
);
```

In `_handleSoloBucketWinner` (line 1426):
```solidity
// After processing the 75/25 ETH/whale-pass split...
if (isFinalDay) {
    uint256 dgnrsPool = dgnrs.poolBalance(IStakedDegenerusStonk.Pool.Reward);
    uint256 reward = (dgnrsPool * FINAL_DAY_DGNRS_BPS) / 10_000;
    if (reward != 0) {
        dgnrs.transferFromPool(IStakedDegenerusStonk.Pool.Reward, w, reward);
        emit JackpotDgnrsWin(w, reward);
    }
}
```

The DGNRS reward is given to address `w` -- the SAME address that won the solo bucket ETH. No re-pick occurs.

### 2.3: Correctness Proof

**1. Amount equivalence:**

| Component | Old Code | New Code | Match? |
|-----------|----------|----------|--------|
| Pool source | `dgnrs.poolBalance(IStakedDegenerusStonk.Pool.Reward)` | `dgnrs.poolBalance(IStakedDegenerusStonk.Pool.Reward)` | YES |
| Formula | `(dgnrsPool * FINAL_DAY_DGNRS_BPS) / 10_000` | `(dgnrsPool * FINAL_DAY_DGNRS_BPS) / 10_000` | YES |
| `FINAL_DAY_DGNRS_BPS` | `100` (line 167 old) | `100` (line 186 new) | YES |
| Transfer call | `dgnrs.transferFromPool(Pool.Reward, winners[0], reward)` | `dgnrs.transferFromPool(Pool.Reward, w, reward)` | YES (same function, same pool, same amount) |

The amount computation is character-for-character identical.

**2. Winner change (INTENTIONAL):**

- **Old:** `_randTraitTicket(traitBurnTicket[lvl], entropy, traitIds[soloIdx], 1, 254)` -- re-picked a winner from the solo bucket using salt 254. This could select a DIFFERENT holder than the one who won the daily ETH jackpot.
- **New:** `w` is the winner address already selected during `_processDailyEth` using salt `uint8(200 + traitIdx)`. The DGNRS reward goes to the SAME player who won the ETH solo bucket.

This is the **INTENTIONAL fix**: the old behavior where a different salt picked a potentially different winner from the same solo bucket was a design inconsistency. The solo bucket has exactly one winner -- giving the DGNRS reward to that same winner is the intended behavior. Re-picking with a different salt on the same bucket with 1 winner usually yields the same winner anyway (same `effectiveLen`, different hash modulus), but could differ if the bucket population changed between the two calls (which shouldn't happen within the same transaction, but the unnecessary re-derivation was still incorrect in principle).

**3. Timing:**

- **Old:** Called AFTER `payDailyJackpot` completed, via delegatecall from AdvanceModule. The DGNRS transfer happened as a separate operation.
- **New:** Called INLINE during `_processDailyEth`, within `_handleSoloBucketWinner`. The DGNRS transfer happens during the daily jackpot processing.

This timing change does not affect correctness because:
- The DGNRS reward pool (`IStakedDegenerusStonk.Pool.Reward`) is independent of the ETH prize pools
- The `dgnrs.poolBalance()` read and `dgnrs.transferFromPool()` write are self-contained -- they do not interact with any ETH pool state
- No other operation between the old and new call points modifies the DGNRS reward pool

**4. Event:**

- **Old:** `emit JackpotTicketWinner(winners[0], lvl, traitIds[soloIdx], reward, 0, AWARD_DGNRS)`
- **New:** `emit JackpotDgnrsWin(w, reward)`

The new event is a specialized DGNRS event. Fields `lvl` and `traitId` are dropped but are recoverable from the co-emitted `JackpotEthWin` event for the same winner `w` in the same transaction (documented in 192-01-AUDIT.md Section 5, emission E2).

### 2.4: Entropy Path Analysis

**Old entropy consumption:** The old `awardFinalDayDgnrsReward` called `_randTraitTicket` with salt 254, producing:
```
keccak256(abi.encode(rngWord ^ (uint256(lvl) << 192), traitIds[soloIdx], 254, 0)) % effectiveLen
```

**New entropy consumption:** No new `_randTraitTicket` call. The DGNRS reward uses the winner `w` already selected by `_processDailyEth`'s call to `_randTraitTicket` with salt `uint8(200 + traitIdx)`.

**Impact of removed entropy derivation:**

1. The salt-254 derivation was a terminal operation in the old flow -- `awardFinalDayDgnrsReward` was the LAST operation called from AdvanceModule for the daily jackpot. No downstream operations depended on this entropy derivation or consumed sequential salts after it.

2. Removing a keccak256 consumption does not make any remaining consumptions less random. The VRF word is the same regardless. Each salt produces an independent derivation: `keccak256(abi.encode(word, trait, salt, index))` -- removing salt 254 has no effect on salts 200-203 or any other derivation.

3. The `entropyState` variable in `_processDailyEth` (line 1155) is NOT affected by this change because the old entropy derivation happened in a completely separate function (`awardFinalDayDgnrsReward`) that did not feed entropy back to `_processDailyEth`.

4. The `_handleSoloBucketWinner` function sets `newEntropy = newEnt` from `_processSoloBucketWinner`, which returns `entropy` unchanged (line 1529: `newEntropy = entropy`). This is the SAME behavior as the old code's `_processSoloBucketWinner`.

**Verdict: INTENTIONAL-CORRECT** -- The entropy path change is a non-issue. The removed salt-254 derivation was terminal, independent, and produced a potentially different (and unintended) winner. The new code correctly gives the reward to the already-selected solo bucket winner.
