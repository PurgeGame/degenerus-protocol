## TRIG-01: Liveness Guard Conditions

**Requirement:** Liveness guard fires at exactly 365d (L0) and 120d (L1+), with safety abort when nextPool >= levelPrizePool, and no other trigger conditions exist.

### Trace

**1. Constants (AdvanceModule L108-109)**

```solidity
uint48 private constant DEPLOY_IDLE_TIMEOUT_DAYS = 365; // L108
uint48 private constant GAMEOVER_RNG_FALLBACK_DELAY = 3 days; // L109
```

- `DEPLOY_IDLE_TIMEOUT_DAYS` is 365. OK.
- These are `private constant` -- immutable, not overridable. OK.

**2. Liveness check (AdvanceModule L481-483)**

```solidity
bool livenessTriggered = (lvl == 0 &&
    currentDay - psd > DEPLOY_IDLE_TIMEOUT_DAYS) ||
    (lvl != 0 && currentDay - psd > 120);
```

- L0 branch: fires when `currentDay - psd > 365` (i.e., day 366+). Uses `DEPLOY_IDLE_TIMEOUT_DAYS` constant. OK.
- L1+ branch: fires when `currentDay - psd > 120` (i.e., day 121+). Hardcoded 120. OK.
- `psd` is `purchaseStartDay`, passed from `advanceGame()` at L162. This records when the current level's purchase phase began. Each level advance resets it, so the 120-day window restarts per level. OK.
- `currentDay` is `day` (L480), which is `_simulatedDayIndexAt(ts)` from L159. OK.
- Strict inequality (`>`) means the guard fires on day 366 (L0) or 121 (L1+), not on the boundary day itself. OK.
- No other conditions in this function trigger gameover. The only early-returns before L485 are the function parameters themselves. OK.

**3. Safety abort (AdvanceModule L502)**

```solidity
if (lvl != 0 && _getNextPrizePool() >= levelPrizePool[lvl]) {
    return false;
}
```

- Only applies at L1+ (`lvl != 0`). At L0, there is no `levelPrizePool[0]` target to meet, so the abort is correctly skipped. OK.
- Checks whether `nextPrizePool` already meets the level target. If so, gameover is aborted -- the game can still advance normally. OK.
- This runs AFTER the `gameOver` check at L490 (which routes to `handleFinalSweep` if already game-over). So the safety abort only prevents initial gameover activation, not post-gameover sweep. OK.

**4. No other gameover trigger paths**

- `gameOver = true` appears exactly ONCE in the entire codebase: `DegenerusGameGameOverModule.sol` L136, inside `handleGameOverDrain`. OK.
- `handleGameOverDrain.selector` is referenced exactly ONCE: AdvanceModule L519, inside `_handleGameOverPath`. OK.
- No other function sets `gameOver = true` or calls `handleGameOverDrain`. The only path to gameover is through `_handleGameOverPath`. OK.

**5. Entry guard (AdvanceModule L178)**

```solidity
if (!inJackpot && !lastPurchase && _handleGameOverPath(day, lvl, psd)) {
```

- `!inJackpot`: gameover cannot fire during jackpot phase (`jackpotPhaseFlag == true`). OK.
- `!lastPurchase`: gameover cannot fire on the last-purchase day (when `lastPurchaseDay == true` and not in jackpot). OK.
- These guards prevent gameover from interrupting active jackpot distribution or the final purchase day. OK.

### Findings

| # | Class | Description |
|---|-------|-------------|
| 1 | OK | DEPLOY_IDLE_TIMEOUT_DAYS = 365 (L108), L1+ uses hardcoded 120 (L483) |
| 2 | OK | Safety abort at L502 prevents gameover when target already met |
| 3 | OK | `gameOver = true` only set at GameOverModule L136 via `handleGameOverDrain`, called only from `_handleGameOverPath` L517-522 |
| 4 | OK | Entry guard at L178 blocks gameover during jackpot phase and last-purchase day |

### Verdict: PASS

All liveness guard conditions verified. Exactly two trigger conditions (365d L0, 120d L1+), one safety abort (nextPool >= target), one entry point. No alternative gameover paths exist.

## TRIG-02: _gameOverEntropy Three Paths

**Requirement:** `_gameOverEntropy` resolves correctly on all three paths: VRF word already available, 3-day fallback to historical VRF, and VRF pending (request or timer start).

### Trace

**Path A: VRF word already stored (AdvanceModule L1089)**

```solidity
if (rngWordByDay[day] != 0) return rngWordByDay[day];
```

- If a previous call already stored the word for this day (via `_applyDailyRng` on a prior advanceGame call), return it immediately. OK.
- This is the fast-path for when VRF delivered between the request call and the current call. OK.

**Path B: rngWordCurrent available (AdvanceModule L1091-1118)**

```solidity
uint256 currentWord = rngWordCurrent;
if (currentWord != 0 && rngRequestTime != 0) {
    currentWord = _applyDailyRng(day, currentWord);  // L1093
    ...
    return currentWord;  // L1118
}
```

- Condition: VRF callback has delivered `rngWordCurrent != 0` AND a request was made (`rngRequestTime != 0`). OK.
- L1093: `_applyDailyRng(day, currentWord)` stores into `rngWordByDay[day]` at L1626. Confirmed by trace:

```solidity
// _applyDailyRng at L1613-1628:
rngWordCurrent = finalWord;       // L1625
rngWordByDay[day] = finalWord;    // L1626
```

- L1094-1100: Coinflip payouts processed (only if `lvl != 0`). Normal side effect. OK.
- L1101-1116: Redemption resolution -- calculates `redemptionRoll` from word bits, resolves pending gambling burn redemptions. `flipDay = day + 1` sets the flip target one day ahead. OK.
- L1117: `_finalizeLootboxRng(currentWord)` -- settles any pending lootbox entropy. OK.
- Returns the processed `currentWord`. This is the resolved RNG word. OK.

**Path C: 3-day fallback (AdvanceModule L1121-1153)**

```solidity

    uint48 elapsed = ts - rngRequestTime;              // L1122
    if (elapsed >= GAMEOVER_RNG_FALLBACK_DELAY) {       // L1123

```

- Enters when `rngRequestTime != 0` but `rngWordCurrent == 0` (VRF callback hasn't arrived). OK.
- L1123: `GAMEOVER_RNG_FALLBACK_DELAY = 3 days` (L109). Uses `>=` so fires at exactly 3 days elapsed. OK.
- L1125: `_getHistoricalRngFallback(day)` collects up to 5 historical VRF words from `rngWordByDay[1..30]`, hashes them with `currentDay` and `block.prevrandao`. Returns `word` (guaranteed non-zero via L1200: `if (word == 0) word = 1`). OK.
- L1126: `_applyDailyRng(day, fallbackWord)` stores fallback word into `rngWordByDay[day]`. Same storage path as Path B. OK.
- L1127-1149: Same coinflip + redemption + lootbox side effects as Path B. OK.
- L1153: If elapsed < 3 days, reverts `RngNotReady()`. This prevents gameover from proceeding until either VRF delivers or 3 days elapse. OK.

**Path D: No request pending (AdvanceModule L1156-1163)**

```solidity
if (_tryRequestRng(isTicketJackpotDay, lvl)) {
    return 1;     // L1157
}
// VRF request failed
rngWordCurrent = 0;       // L1161
rngRequestTime = ts;      // L1162
return 0;                 // L1163
```

- L1156: `_tryRequestRng` attempts a VRF request via `vrfCoordinator.requestRandomWords` in a try/catch (L1405-1419). On success, calls `_finalizeRngRequest` and returns `true`. On failure (VRF subscription issue, gas), returns `false`. OK.
- L1157: Returns 1 (signal: "VRF request sent, retry on next call"). OK.
- L1161-1162: On VRF failure, sets `rngWordCurrent = 0` and `rngRequestTime = ts` to start the 3-day fallback timer. OK.
- L1163: Returns 0 (signal: "waiting on fallback timer"). OK.

**Caller handling (AdvanceModule L507-514)**

```solidity
if (rngWordByDay[day] == 0) {
    uint256 rngWord = _gameOverEntropy(ts, day, lvl, lastPurchaseDay);
    if (rngWord == 1 || rngWord == 0) return true;

```

- L507: Only calls `_gameOverEntropy` if word not already stored. Prevents redundant processing. OK.
- L514: When return is 0 or 1 (pending VRF or fallback timer), returns `true` to exit `advanceGame()` early. No drain happens. OK.
- When return is the actual word (Path A, B, or C), execution falls through to L517 which delegatecalls `handleGameOverDrain`. The word is already stored in `rngWordByDay[day]` at this point. OK.

### _getHistoricalRngFallback detail (AdvanceModule L1177-1201)

```solidity
uint48 searchLimit = currentDay > 30 ? 30 : currentDay;    // L1182
for (uint48 searchDay = 1; searchDay < searchLimit; ) {     // L1183

- L1200: Guarantees non-zero return. This is important because `_applyDailyRng` stores it, and `handleGameOverDrain` checks `rngWord == 0`. OK.
- NOTE: At L0 with 365 days of inactivity, there may be zero historical VRF words (no advances ever completed). In that case `combined == 0`, and the fallback degrades to `keccak256(0, currentDay, prevrandao)`. The NatSpec at L1173 acknowledges this: "only at level 0, so 1-bit validator bias is irrelevant." OK.

### Verdict: PASS

All four entropy paths verified. VRF word resolution, 3-day fallback, and pending-request handling are correct. Caller properly gates drain on confirmed word availability.

## TRIG-03: RNG Word Lifecycle

**Requirement:** RNG word requested by gameover is stored via `_applyDailyRng` into `rngWordByDay[day]`, consumed once by `handleGameOverDrain`, and unlocked by `_unlockRng` so no other consumer reuses it.

### Trace

**1. Request (AdvanceModule L1401-1419)**

```solidity
function _tryRequestRng(bool isTicketJackpotDay, uint24 lvl) private returns (bool requested) {
    try vrfCoordinator.requestRandomWords(...) returns (uint256 id) {
        _finalizeRngRequest(isTicketJackpotDay, lvl, id);  // L1417
        requested = true;                                   // L1418
    } catch {}
}
```

- VRF request goes to Chainlink coordinator. On success, `_finalizeRngRequest` records the request ID and sets `rngLockedFlag = true`, `rngRequestTime`. OK.
- On failure (catch), `requested = false` is returned. Caller (Path D) then sets `rngRequestTime = ts` to start fallback timer. OK.

**2. VRF Callback -> rngWordCurrent**

- Chainlink VRF coordinator calls back with the random word, which gets stored in `rngWordCurrent`. This happens asynchronously between advanceGame calls. OK.

**3. Storage via _applyDailyRng (AdvanceModule L1613-1628)**

```solidity
function _applyDailyRng(uint48 day, uint256 rawWord) private returns (uint256 finalWord) {
    uint256 nudges = totalFlipReversals;
    finalWord = rawWord;
    if (nudges != 0) {
        unchecked { finalWord += nudges; }
        totalFlipReversals = 0;
    }
    rngWordCurrent = finalWord;    // L1625
    rngWordByDay[day] = finalWord; // L1626
    lastVrfProcessedTimestamp = uint48(block.timestamp);
    emit DailyRngApplied(day, rawWord, nudges, finalWord);
}
```

- Applies nudges (flip reversals) to the raw word. OK.
- Stores finalized word in both `rngWordCurrent` (L1625) and `rngWordByDay[day]` (L1626). OK.
- The word stored in `rngWordByDay[day]` is the one consumed by `handleGameOverDrain`. OK.

**4. Consumption by handleGameOverDrain (GameOverModule L96-98)**

```solidity
if (preRefundAvailable != 0) {
    rngWord = rngWordByDay[day];    // L97
    if (rngWord == 0) revert E();   // L98
}
```

- Reads `rngWordByDay[day]` and reverts if zero (defense-in-depth). OK.
- The word is passed to `runTerminalDecimatorJackpot(decPool, lvl, rngWord)` at L162 and `runTerminalJackpot(remaining, lvl + 1, rngWord)` at L175. OK.
- `rngWordByDay[day]` is NOT zeroed by `handleGameOverDrain`. The word persists in storage. OK.

**5. Cleanup via _unlockRng (AdvanceModule L1513-1520)**

```solidity
function _unlockRng(uint48 day) private {
    dailyIdx = day;
    rngLockedFlag = false;
    rngWordCurrent = 0;    // L1516
    vrfRequestId = 0;      // L1517
    rngRequestTime = 0;    // L1518
    _unfreezePool();
}
```

- Called at AdvanceModule L524, AFTER `handleGameOverDrain` completes. OK.
- Clears `rngWordCurrent` (L1516), `vrfRequestId` (L1517), `rngRequestTime` (L1518). OK.
- Does NOT clear `rngWordByDay[day]`. The consumed word remains in the mapping. OK.
- Sets `dailyIdx = day` and `rngLockedFlag = false`. This advances the day index and unlocks RNG for future use. OK.

**6. Exclusivity -- no other consumer reuses gameover's rngWordByDay[day]**

The gameover flow is terminal:
- `handleGameOverDrain` sets `gameOver = true` (L136) and `gameOverFinalJackpotPaid = true` (L143).
- After `_unlockRng` at L524, `_handleGameOverPath` returns `true` at L525.
- On next `advanceGame()` call, `_handleGameOverPath` sees `gameOver == true` (L490) and routes to `handleFinalSweep` (L492-498). No further RNG consumption occurs.
- `handleFinalSweep` does not read `rngWordByDay`. OK.
- The `gameOverFinalJackpotPaid` guard at L80 prevents `handleGameOverDrain` from executing a second time. OK.

Normal advanceGame paths cannot consume the gameover day's word because:
- `_unlockRng` sets `dailyIdx = day` (L1514). The next normal advance would be on a future day.
- But with `gameOver == true`, the L178 guard short-circuits to the gameover path. Normal advance logic never runs again. OK.

### Findings

| # | Class | Description |
|---|-------|-------------|
| 1 | OK | VRF request via `_tryRequestRng` (L1401), callback stores `rngWordCurrent` |
| 2 | OK | `_applyDailyRng` stores finalized word in `rngWordByDay[day]` (L1626) |
| 3 | OK | `handleGameOverDrain` reads `rngWordByDay[day]` (L97), passes to jackpot functions |
| 4 | OK | `_unlockRng` clears transient state but preserves `rngWordByDay[day]` (L1513-1520) |
| 5 | OK | Post-gameover, `gameOverFinalJackpotPaid` (L80) and `gameOver` (L490) prevent any reuse |
| 6 | OK | Normal advance paths cannot run post-gameover due to `gameOver == true` routing at L490 |

### Verdict: PASS

RNG word lifecycle is clean: requested -> stored in `rngWordByDay[day]` -> consumed once by drain -> transient state cleared by `_unlockRng` -> no reuse possible due to terminal state latching.

## DRNA-01: Fund Split Correctness

**Requirement:** Drain splits funds correctly: 10% to runTerminalDecimatorJackpot, 90% + decimator refund to runTerminalJackpot, remainder to _sendToVault.

### Trace

**1. Available computation (GameOverModule L150-157)**

```solidity
uint256 available = totalFunds > claimablePool ? totalFunds - claimablePool : 0;  // L150
if (available == 0) return;  // L152
uint256 remaining = available;  // L157
```

- `totalFunds` computed at L86: `address(this).balance + steth.balanceOf(address(this))`. OK.
- `claimablePool` may have grown from deity refunds (L131). The recalculation at L150 uses the updated `claimablePool`. OK.
- `remaining` starts at `available`. OK.

**2. 10% Terminal Decimator (GameOverModule L160-169)**

```solidity
uint256 decPool = remaining / 10;  // L160
if (decPool != 0) {
    uint256 decRefund = IDegenerusGame(address(this))
        .runTerminalDecimatorJackpot(decPool, lvl, rngWord);  // L162
    uint256 decSpend = decPool - decRefund;  // L163
    if (decSpend != 0) {
        claimablePool += decSpend;  // L165
    }
    remaining -= decPool;   // L167
    remaining += decRefund; // L168
}
```

- L160: `decPool = remaining / 10` -- exactly 10% (integer division, truncated). OK.
- L162: Self-call to `runTerminalDecimatorJackpot`. The DegenerusGame.sol wrapper at L1149-1170 delegates to DecimatorModule.

  **Self-call guard:** DegenerusGame.sol L1154: `if (msg.sender != address(this)) revert E()`. This prevents external callers from invoking `runTerminalDecimatorJackpot` directly. But `handleGameOverDrain` runs via delegatecall, so `address(this)` is the DegenerusGame proxy, and the self-call `IDegenerusGame(address(this)).runTerminalDecimatorJackpot(...)` passes because `msg.sender` == the proxy calling itself. OK.

- L162 return value: `runTerminalDecimatorJackpot` returns `returnAmountWei`:
  - 0 if winners exist (DecimatorModule L770: all of `poolWei` allocated to claims)
  - `poolWei` if no winners or double-resolution (DecimatorModule L732, L759)
  So `decRefund` is either 0 (winners found, full pool committed) or `decPool` (no winners, full refund). OK.

- L163: `decSpend = decPool - decRefund`. This is the amount actually committed to decimator winners. OK.
  - Winners: `decSpend = decPool - 0 = decPool`. All 10% committed.
  - No winners: `decSpend = decPool - decPool = 0`. Nothing committed.

- L165: `claimablePool += decSpend`. Only the committed amount is added. NOTE: The actual per-player crediting happens inside DecimatorModule when players call `claimTerminalDecimatorJackpot()` -- the module stores a snapshot (poolWei, totalBurn) and pro-rata distribution happens on claim. The `claimablePool += decSpend` here pre-reserves the liability. OK.

- L167-168: `remaining -= decPool; remaining += decRefund`. Net effect: `remaining = available - decPool + decRefund = available - decSpend`. When no winners: `remaining = available`. When winners: `remaining = available - decPool`. OK.

**3. 90% (+ refund) Terminal Jackpot (GameOverModule L173-180)**

```solidity
if (remaining != 0) {
    uint256 termPaid = IDegenerusGame(address(this))
        .runTerminalJackpot(remaining, lvl + 1, rngWord);  // L174-175
    remaining -= termPaid;  // L176
    if (remaining != 0) {
        _sendToVault(remaining, stBal);  // L178
    }
}
```

- L174-175: `runTerminalJackpot(remaining, lvl + 1, rngWord)`. Self-call to JackpotModule via DegenerusGame.sol L1187-1206. Same self-call guard at L1192. OK.
- `remaining` at this point equals `available - decSpend` (which is 90% + refund when winners exist, or 100% when no winners). OK.
- `lvl + 1`: Terminal jackpot draws winners from the next level's ticket holders. OK.
- `runTerminalJackpot` return value `paidWei`: total ETH distributed to winners via `_processDailyEth` -> `_payNormalBucket` -> `_addClaimableEth`. OK.

  **claimablePool update in runTerminalJackpot:** JackpotModule L1286: `claimablePool += liabilityDelta`. Since `gameOver == true` at this point (set at L136 before distribution), `_addClaimableEth` at L777 skips auto-rebuy and returns `claimableDelta == weiAmount`. Therefore `liabilityDelta == paidWei`. So `claimablePool` grows by exactly `paidWei` inside the function. The NatSpec at L1182 confirms: "Updates claimablePool internally -- callers must NOT double-count." OK.

- L176: `remaining -= termPaid`. Remaining is whatever couldn't be distributed (dust from integer division, or missing ticket holders). OK.
- L178: If remainder exists, `_sendToVault(remaining, stBal)` sends it to the three vault destinations. OK.

**4. Accounting Identity**

At drain completion:
```
available = decSpend + termPaid + vault_remainder
```

Where:
- `decSpend` = added to `claimablePool` at L165 (for decimator pro-rata claims)
- `termPaid` = added to `claimablePool` inside `runTerminalJackpot` at JackpotModule L1286
- `vault_remainder` = sent to vault/sDGNRS/GNRUS at L178

Verification:
- `remaining` starts at `available` (L157)
- After decimator: `remaining = available - decSpend` (net of L167-168)
- After jackpot: `remaining = available - decSpend - termPaid` (L176)
- If `remaining != 0`: sent to vault (L178)
- Total: `decSpend + termPaid + remaining = available`. Identity holds. OK.

### Findings

| # | Class | Description |
|---|-------|-------------|
| 1 | OK | 10% decimator allocation via `remaining / 10` (L160), refund returned to pool |
| 2 | OK | Self-call guard (`msg.sender != address(this)`) at DegenerusGame.sol L1154 and L1192 prevents external invocation |
| 3 | OK | `decSpend = decPool - decRefund` correctly captures committed amount (L163) |
| 4 | OK | `remaining` accounting tracks: `available - decSpend - termPaid = vault_remainder` |
| 5 | OK | `runTerminalJackpot` updates `claimablePool` internally (JackpotModule L1286); caller does not double-count |
| 6 | NOTE | When all ticket holders are missing (no winners sampled), `termPaid` could be 0 and the entire `remaining` goes to vault. This is correct behavior -- funds are preserved, not lost. |

### Verdict: PASS

Fund split accounting is correct. The identity `available = decSpend + termPaid + vault_remainder` holds in all cases. No funds are lost or double-counted.

## DRNA-02: Deity Pass Refund Math

**Requirement:** Deity pass refunds pay exactly 20 ETH * purchasedCount per owner, FIFO order, capped by preRefundAvailable budget, with claimablePool updated correctly.

### Trace

**1. Level guard (GameOverModule L104)**

```solidity
if (lvl < 10) {
```

- Refunds only apply at levels 0-9. At level 10+, the deity pass refund block is entirely skipped. OK.
- NOTE: This means if gameover fires at L10+, deity pass holders receive no refund. This is by design -- "early gameover" refunds only apply when the game ends before level 10. OK.

**2. Refund constant (GameOverModule L49, L105)**

```solidity
uint256 private constant DEITY_PASS_EARLY_GAMEOVER_REFUND = 20 ether;  // L49
uint256 refundPerPass = DEITY_PASS_EARLY_GAMEOVER_REFUND;  // L105
```

- 20 ETH per pass. `private constant` -- immutable. OK.

**3. Budget initialization (GameOverModule L107)**

```solidity
uint256 budget = preRefundAvailable;
```

- `preRefundAvailable` computed at L90: `totalFunds > claimablePool ? totalFunds - claimablePool : 0`. This is the pre-refund available, before any deity refunds reduce it. OK.
- Budget caps total refunds to available funds. No refund can exceed what the contract actually holds minus existing obligations. OK.

**4. FIFO refund loop (GameOverModule L109-129)**

```solidity
for (uint256 i; i < ownerCount; ) {
    address owner = deityPassOwners[i];  // L110
    uint16 purchasedCount = deityPassPurchasedCount[owner];  // L111
    if (purchasedCount != 0) {
        uint256 refund = refundPerPass * uint256(purchasedCount);  // L113
        if (refund > budget) {
            refund = budget;  // L115
        }
        if (refund != 0) {
            unchecked {
                claimableWinnings[owner] += refund;  // L119
                totalRefunded += refund;  // L120
                budget -= refund;  // L121
            }
        }
        if (budget == 0) break;  // L124
    }
    unchecked { ++i; }
}
```

- L110: Iterates `deityPassOwners` array in insertion order (FIFO). OK.
- L111: `deityPassPurchasedCount[owner]` records how many deity passes this owner has purchased. OK.
- L113: `refund = 20 ETH * purchasedCount`. Correct per-owner refund amount. OK.
- L114-116: Caps refund to remaining budget. Prevents overspend. OK.
- L119: Credits `claimableWinnings[owner]`. Player can claim via `claimWinnings()`. OK.
- L120: Tracks `totalRefunded` for `claimablePool` update. OK.
- L121: `budget -= refund` in unchecked block. Safe because L114-116 already capped `refund <= budget`. No underflow possible. OK.
- L124: Early break when budget exhausted. Remaining owners get nothing. OK.

**5. claimablePool update (GameOverModule L130-132)**

```solidity
if (totalRefunded != 0) {
    claimablePool += totalRefunded;  // L131
}
```

- `claimablePool` grows by exactly the sum of all refunds. OK.
- The `totalRefunded != 0` check avoids a useless SSTORE. OK.

**6. Double-refund prevention**

- `deityPassPurchasedCount` is NOT zeroed after refund. This means the mapping still holds the purchase count.
- However, `gameOverFinalJackpotPaid` at L80 prevents the entire `handleGameOverDrain` function from executing a second time. On any subsequent call, L80 returns immediately.

**2. _sendToVault (GameOverModule L217-225)**

    uint256 thirdShare = amount / 3;                        // L218
    uint256 gnrusAmount = amount - thirdShare - thirdShare; // L219

```

- L219: `gnrusAmount = amount - thirdShare - thirdShare` -- 34% (gets the remainder). This ensures `thirdShare + thirdShare + gnrusAmount = amount` exactly. No dust lost. OK.

**3. _sendStethFirst (GameOverModule L232-247)**

```solidity
function _sendStethFirst(address to, uint256 amount, uint256 stethBal) private returns (uint256) {
    if (amount == 0) return stethBal;           // L233
    if (amount <= stethBal) {                    // L234
        if (!steth.transfer(to, amount)) revert E();  // L235
        return stethBal - amount;
    }
    if (stethBal != 0) {                         // L238
        if (!steth.transfer(to, stethBal)) revert E();  // L239
    }
    uint256 ethAmount = amount - stethBal;       // L241
    if (ethAmount != 0) {                         // L242
        (bool ok, ) = payable(to).call{value: ethAmount}("");  // L243
        if (!ok) revert E();                      // L244
    }
    return 0;
}
```

- Sends stETH first, ETH as fallback. OK.
- L233: Handles `amount == 0` gracefully. OK.
- L235, L239: stETH transfer failures revert. Hard revert, not silent. OK.
- L243-244: ETH transfer failures revert. OK.
- NOTE: Hard reverts here mean a stuck stETH transfer or ETH-rejecting recipient would block the entire gameover drain. This is acknowledged in the NatSpec at L211-213: "a stuck stETH transfer would block game-over processing until the transfer succeeds." The recipients (SDGNRS, VAULT, GNRUS) are protocol-controlled contracts designed to accept transfers. OK.

**6. Edge case: stBal stale value (GameOverModule L85 vs L178)**

// ... later ...

This would block gameover drain. However, stETH negative rebases are extremely rare (require Ethereum validator slashing) and the amount would be dust-level. NOTE classification.
- NOTE: This is a known stETH property. The stale `stBal` snapshot is a practical trade-off to avoid multiple `balanceOf` calls. The 30-day `handleFinalSweep` provides a safety net for any micro-mismatch.
