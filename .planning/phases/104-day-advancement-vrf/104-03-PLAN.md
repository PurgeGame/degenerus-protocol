**Part 1: Review every VULNERABLE and INVESTIGATE finding**

For each finding in ATTACK-REPORT.md that has a verdict of VULNERABLE or INVESTIGATE:

1. **Read the cited code.** Go to the exact lines. Read them.

2. **Trace the execution path.** If the finding claims "X can happen," trace whether X can actually happen. Check:
   - Can the attacker actually reach this code path? What modifiers/requires block them?
   - Does the claimed state actually exist at this point in execution?
   - Are there guards the Mad Genius missed that prevent the attack?
   - For cached-local-vs-storage findings: does the parent actually WRITE BACK the stale local? (If it only reads the local but never writes it back to storage, there is no overwrite.)
   - For RNG manipulation findings: is the RNG word actually unknown at the time inputs were committed? (Per project memory: trace BACKWARD from each consumer.)
   - For cross-module findings: does the parent actually use the stale local AFTER the delegatecall returns? (If the local is not used again, there is no bug.)

3. **Classify the finding:**

```
### [FINDING-ID]: [Title]
**Mad Genius Verdict:** VULNERABLE / INVESTIGATE
**Skeptic Verdict:** CONFIRMED / FALSE POSITIVE / DOWNGRADE TO INFO

**Analysis:** [Your precise technical analysis. Cite exact line numbers from the source.]

**If CONFIRMED:**
- Severity: CRITICAL / HIGH / MEDIUM / LOW / INFO
- Justification: [Why this severity? What is the real-world impact? How much ETH is at risk?]
- Recommendation: [How to fix]

**If FALSE POSITIVE:**
- Reason: [EXACT reason -- which guard prevents it, which precondition is impossible]
- Cite: [The specific line(s) that prevent the attack]

**If DOWNGRADE TO INFO:**
- Original concern: [What the Mad Genius flagged]
- Why downgrade: [Why it is informational, not exploitable]
```

4. **If you CANNOT fully disprove a finding, it stays flagged.** You only dismiss findings you can PROVE are false positives.
