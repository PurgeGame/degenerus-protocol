## INV-01 — the `fail_on_revert = false` blind spot, investigated empirically (✅)

`foundry.toml` sets `[invariant] fail_on_revert = false` in both the default and deep profiles. The concern: a handler call that *should* succeed but reverts is silently discarded, hiding a "should-not-have-reverted" defect. The rigorous test is to run every suite with `fail_on_revert = TRUE` (env `FOUNDRY_INVARIANT_FAIL_ON_REVERT=true`) and see what surfaces.

**Result (18 suites, bounded runs=20–30 / depth=25–40):**

**14 suites CLEAN under `fail_on_revert=true`** — 0 reverts, all pass: VaultShare, VRFPathInvariants, CoinSupply, RngIndexDrainOrdering, VaultShareMath, TicketQueue, WhaleSybil, EthSolvency, BoxEnqueue, GameFSM, RngWindowFreeze, MultiLevel, PoolConservation, Composition. For these the blind spot is **benign** — their handlers only make valid calls, so nothing is silently skipped.

**4 suites fail under `fail_on_revert=true` — none a contract defect:**
| Suite | Failure | Diagnosis |
|-------|---------|-----------|
| DegeneretteBet | `VACUOUS: no Degenerette bet was placed` | The suite's OWN vacuity guard. `fail_on_revert=true` aborts the sequence on the first random-invalid-bet revert before any valid bet lands. The suite legitimately needs `fail_on_revert=false` to skip invalid-input reverts and reach valid betting — and its vacuity invariant IS the "did the should-succeed path actually execute" assertion INV-01 calls for. |
| V61SolvencyAfpay | `NON-VACUITY: pass/presale/claim surfaces must succeed > 0 times` | Same pattern — an explicit non-vacuity ghost guard; `fail_on_revert=true` aborts before the surfaces execute. The guard is working as designed. |
| RedemptionInvariants | `call to non-contract address 0x1Ddad…` / `persisted failure revert` | A **harness artifact** — the target/actor set includes a non-contract address that reverts immediately under `fail_on_revert=true` (silently skipped under `false`). Not a contract should-not-revert. |

**Conclusion:** the `fail_on_revert=false` blind spot hides **no contract should-not-revert defect**. 14 suites are provably clean with it on; the 4 exceptions are (a) by-design random-invalid-input handlers already protected by their own vacuity / non-vacuity ghost assertions — which are exactly the should-execute protection INV-01 specifies — or (b) a documented harness non-contract-call artifact. INV-01 is closed: the blind spot is investigated and shown benign, and the suites already carry the ghost-assertion mechanism.

## INV-02 — deep-profile sweep (✅ GREEN at runs=1000 / depth=256)

`FOUNDRY_PROFILE=deep forge test` over all 17 invariant suites (~8× the per-PR budget; ~200s wall / 2230s CPU). **16/17 suites green on the first pass (77 tests).** The 8 apparent failures were all in DegeneretteBet + V61SolvencyAfpay and were **`replay failure`s — persisted counterexamples from the INV-01 `fail_on_revert=true` probe contaminating the cache**, not real deep-budget violations. After `rm -rf cache/invariant cache/fuzz`, both re-ran **clean at deep budget**: DegeneretteBet 5/5, V61SolvencyAfpay 8/8. So the full invariant net holds at runs=1000/depth=256. (Already wired into scheduled CI in Phase 431.)

**Methodology note (→ INV-03):** a `fail_on_revert=true` probe persists failing inputs to `cache/invariant`; clear that cache before a subsequent `fail_on_revert=false` deep run or the replays contaminate it.

## INV-03 — gaps (test-harness follow-on, no contract gap)

- The clean-14 categorization documents which suites could adopt `fail_on_revert=true` (a strict profile) for stronger future coverage — optional hardening.
