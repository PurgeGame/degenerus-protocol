<bug_summary>
## Known Bug: runRewardJackpots in EndgameModule

**Root cause (three-leg pattern):**
1. READ-LOCAL: `uint256 futurePoolLocal = _getFuturePrizePool()` at line 169 — snapshots `prizePoolsPacked` (high 128 bits) into a local.
2. NESTED-WRITE: `_runBafJackpot` calls `_addClaimableEth` for each winner. When auto-rebuy is enabled, `_addClaimableEth` calls `_setFuturePrizePool(_getFuturePrizePool() + calc.ethSpent)` — writing directly to `prizePoolsPacked` in storage.
3. STALE-WRITEBACK: At line 235, `_setFuturePrizePool(futurePoolLocal)` overwrites the storage value with the stale local, erasing auto-rebuy's contribution.

**Result:** Every auto-rebuy contribution to `futurePrizePool` during a BAF jackpot is silently lost. The ETH disappears from accounting — solvency invariant is broken.

**Storage slot:** `prizePoolsPacked` (uint256, EVM slot 3+ region). High 128 bits = `futurePrizePool`, low 128 bits = `nextPrizePool`.

## Pattern to Apply to Every Function

A function is a CANDIDATE if:
- It assigns a storage variable to a local (`uint256 x = storageVar` or `uint256 x = _getStorageVar()`)
- AND it calls any function (internal, external, delegatecall) that could write to the same storage slot

A CANDIDATE is VULNERABLE if:
- The nested call CAN reach a write to the same storage slot (trace every path including auto-rebuy, jackpot distribution, any delegation)
- AND the local is written back to storage AFTER the nested call returns

A CANDIDATE is SAFE if:
- The nested call CANNOT reach a write to the cached slot (prove it: trace all call paths to confirm no write)
- OR the local is ONLY read (never written back to storage) — read-only caches cannot cause stale overwrite
- OR the write-back is guarded by a re-read of storage (delta reconciliation already present)

## Key Storage Slots to Watch

| Slot | Variable | Accessor Functions |
|------|----------|--------------------|
| prizePoolsPacked high-128 | futurePrizePool | _getFuturePrizePool() / _setFuturePrizePool(val) |
| prizePoolsPacked low-128 | nextPrizePool | _getNextPrizePool() / _setNextPrizePool(val) |
| EVM slot 2 | currentPrizePool | direct read/write (uint256) |
| Various | claimablePool | direct read/write (uint256) |

## Auto-Rebuy Write Surface (_addClaimableEth in both EndgameModule and JackpotModule)

When auto-rebuy triggers, `_addClaimableEth` / `_processAutoRebuy` writes to:
- `_setFuturePrizePool(...)` — when `calc.toFuture == true` (75% probability)
- `_setNextPrizePool(...)` — when `calc.toFuture == false` (25% probability)

Neither write touches `currentPrizePool` or `claimablePool`.

Any function that caches `futurePrizePool` or `nextPrizePool` and then calls `_addClaimableEth` (directly or indirectly) is a VULNERABLE candidate.
</bug_summary>

Only relevant if Step 2 confirms a nested write. If the local is only READ (used for arithmetic output but never passed to a setter), it cannot cause stale overwrite.

STEP 4 — Assign verdict.
- VULNERABLE: Step 1 + Step 2 + Step 3 all TRUE
- SAFE: Any step is FALSE (with specific reason)
