## Section 2: Death Clock Equivalence (D-01, D-02)

### Algebraic Foundation

Old arithmetic: `ts - levelStartTime > 120 * 86400` (seconds).
New arithmetic: `currentDay - purchaseStartDay > 120` (day-indices).

Since `currentDay = GameTimeLib.currentDayIndexAt(ts)` and `purchaseStartDay` was set via `GameTimeLib.currentDayIndex()` at level transition (AdvanceModule line 312: `purchaseStartDay = day`), both use the same day-index formula. The comparison `currentDay - psd > 120` fires when the elapsed day count is `>= 121`.

Old code fired when `> 10,368,000 seconds` elapsed, i.e., `>= 120 days + 1 second`. At whole-day granularity (game only advances once per day via `dailyIdx`), these are equivalent because `advanceGame` is called at most once per day boundary.

### Level != 0 Paths

Current code (AdvanceModule line 479-481):
```solidity
bool livenessTriggered = (lvl == 0 &&
    currentDay - psd > DEPLOY_IDLE_TIMEOUT_DAYS) ||
    (lvl != 0 && currentDay - psd > 120);
```

| Path | Level | purchaseStartDay | currentDay | Elapsed Days | `currentDay - psd > 120` | Old: `ts - lst > 120 days` | Match? |
|------|-------|------------------|------------|--------------|--------------------------|---------------------------|--------|
| Normal advance (5->6) | 6 | 100 | 221 | 121 | `221 - 100 = 121 > 120` = TRUE | `(100+121)*86400 - 100*86400 = 121*86400 > 120*86400` = TRUE | YES |
| x10 skip (10->20) | 20 | 200 | 320 | 120 (boundary) | `320 - 200 = 120 > 120` = FALSE | `120*86400 > 120*86400` = FALSE | YES |
| x100 skip (100->200) | 200 | 500 | 619 | 119 (no trigger) | `619 - 500 = 119 > 120` = FALSE | `119*86400 > 120*86400` = FALSE | YES |
| Gap backfill (lvl 3) | 3 | 50 | 176 | 126 (but psd shifted) | See gap example below | See gap example below | YES |

**Gap backfill worked example:**
- Initial: `purchaseStartDay = 50`, VRF stall produces `gapCount = 5`
- Line 1055: `purchaseStartDay += 5` => `purchaseStartDay = 55`
- Now: `currentDay = 176`, `176 - 55 = 121 > 120` = TRUE (triggers)
- Without gap shift: `176 - 50 = 126 > 120` = TRUE (would have triggered 5 days earlier)
- Old equivalent: `levelStartTime += 5 * 86400` shifted deadline by exactly 5 days
- The gap extension correctly extends the deadline by the stall duration

### Level 0 Path

Current code (AdvanceModule line 479-480):
```solidity
(lvl == 0 && currentDay - psd > DEPLOY_IDLE_TIMEOUT_DAYS)
```

Where `DEPLOY_IDLE_TIMEOUT_DAYS = 365` (AdvanceModule line 110).

| Path | Level | purchaseStartDay | currentDay | Elapsed Days | `currentDay - psd > 365` | Old: `ts - lst > 365 days` | Match? |
|------|-------|------------------|------------|--------------|--------------------------|---------------------------|--------|
| Level 0 deploy (trigger) | 0 | 1 | 367 | 366 | `367 - 1 = 366 > 365` = TRUE | `366*86400 > 365*86400` = TRUE | YES |
| Level 0 deploy (no trigger) | 0 | 1 | 366 | 365 | `366 - 1 = 365 > 365` = FALSE | `365*86400 > 365*86400` = FALSE | YES |

### Edge Case: Pool Crosses Target on Day 120

Scenario: `purchaseStartDay = 100`, `currentDay = 220`, `lvl = 5`.
- Elapsed = `220 - 100 = 120`
- `currentDay - psd > 120` => `120 > 120` = FALSE. Liveness does NOT trigger.
- The game continues to the safety guard at line 500: `if (lvl != 0 && _getNextPrizePool() >= levelPrizePool[lvl]) return false;`
- This is a second defense but is only reached IF liveness triggers -- since liveness does NOT trigger at exactly 120, pool crossing target on day 120 proceeds normally through the purchase phase.
- **Strict `>` (not `>=`) ensures day 120 is the last safe operating day.**

**VERDICT: EQUIVALENT** -- Death clock produces identical trigger at day 121+ boundary for normal advance, x10 skip, x100 skip, and gap backfill. Level 0 uses 365-day timeout identically.

## Section 3: Future Take Curve Equivalence (D-01)

### Function Signature

Current code (AdvanceModule line 1256-1288):
```solidity
function _nextToFutureBps(uint48 elapsed, uint24 lvl) internal pure returns (uint16)
```

Where `elapsed` is a day-count (not seconds). Constants:
- `NEXT_TO_FUTURE_BPS_FAST = 3000` (30%)
- `NEXT_TO_FUTURE_BPS_MIN = 1300` (13%)
- `NEXT_TO_FUTURE_BPS_DAY_STEP = 14` (0.14%)

### Threshold Mapping

| Segment | Old (seconds) | New (days) | Division | Equivalent? |
|---------|--------------|------------|----------|-------------|
| Fast period | `elapsed <= 1 days` | `elapsed <= 1` | `86400s / 86400 = 1d` | YES |
| Ramp-down | `elapsed <= 14 days` | `elapsed <= 14` | `14*86400 / 86400 = 14d` | YES |
| Ramp-up | `elapsed <= 28 days` | `elapsed <= 28` | `28*86400 / 86400 = 28d` | YES |
| Linear growth | `elapsed > 28 days` | `elapsed > 28` | N/A | YES |

### Integer Division Equivalence

Old ramp-down: `(delta * elapsedAfterDay) / (13 * 86400)` where `elapsedAfterDay` was in seconds.
New ramp-down: `(delta * elapsedAfterDay) / 13` where `elapsedAfterDay` is in days.

Since `elapsedAfterDay_seconds = elapsedAfterDay_days * 86400`:
```
(delta * elapsedAfterDay_days * 86400) / (13 * 86400) = (delta * elapsedAfterDay_days) / 13
```
The `86400` factors cancel, producing identical integer division results at day boundaries.

Same logic for ramp-up: `(delta * elapsedAfterMin) / (14 * 86400)` => `(delta * elapsedAfterMin) / 14`.

### Future Take Caller Elapsed

Current code (AdvanceModule lines 627-628):
```solidity

uint48 elapsed = day > start ? day - start : 0;
```

Old equivalent: `start = levelStartTime + 7 days; elapsed = ts > start ? ts - start : 0;`

Since `psd` is the day-index of `levelStartTime` and `day` is the day-index of `ts`, `psd + 7` in day-space equals `levelStartTime + 7*86400` in timestamp-space at day boundaries.

- `elapsed = 110 - 107 = 3`

## Section 4: Distress Mode Equivalence (D-03)

### Intentional Behavioral Change

Current code (DegenerusGameStorage.sol lines 533-542):
```solidity
function _isDistressMode() internal view returns (bool) {
    if (gameOver) return false;
    uint48 psd = purchaseStartDay;
    uint48 currentDay = _simulatedDayIndex();
    if (level == 0) {
        return currentDay >= psd + _DEPLOY_IDLE_TIMEOUT_DAYS;
    }
    return currentDay >= psd + 120;
}
```

Old code: `ts + 6 hours > lst + 120 days` -- triggered within last ~6 hours of the window.
New code: `currentDay >= psd + 120` -- triggers on the entire final day.

### Worked Example

`purchaseStartDay = 100`, `currentDay = 220`, level = 5:
- New: `220 >= 100 + 120 = 220 >= 220` = TRUE. Distress active all of day 220.
- Old: only triggered when `ts > lst + 120 days - 6 hours`, i.e., within the last 6 hours of the 120-day window.
- New triggers up to ~24 hours earlier (from the start of day 220 vs. 6 hours before the deadline).

### Liveness vs Distress Interaction

Distress mode fires on day 120 (`>=`), while liveness fires on day 121+ (`>`). This means:
- Day 119: No distress, no liveness.
- Day 120: Distress active. Liveness NOT triggered (120 > 120 is false). Players get distress benefits.
- Day 121: Distress active. Liveness triggers, game proceeds to gameover path.

**VERDICT: KNOWN/ACCEPTABLE -- distress mode boundary widened from ~6 hours to up to ~24 hours on boundary day. This is conservative: distress activates earlier, favoring players over protocol. The one-day warning before liveness fires is an improvement.**

## Section 5: Gap Extension Equivalence

### Current Code

AdvanceModule lines 1046-1056:
```solidity
uint48 gapCount = day - idx - 1;
_backfillGapDays(currentWord, idx + 1, day, bonusFlip);
_backfillOrphanedLootboxIndices(currentWord);
purchaseStartDay += gapCount;
gapDays = gapCount;
```

Old equivalent: `levelStartTime += gapCount * 1 days` (adds seconds to timestamp anchor).

### Algebraic Proof

Old: adding `gapCount * 86400` to a timestamp-based anchor shifts the deadline by `gapCount` days.
New: adding `gapCount` to a day-index anchor shifts the deadline by `gapCount` days.

Since `deadline_old = levelStartTime + 120 * 86400` and `deadline_new = purchaseStartDay + 120`:
- Old shift: `(levelStartTime + gapCount*86400) + 120*86400 = levelStartTime + (gapCount+120)*86400`
- New shift: `(purchaseStartDay + gapCount) + 120 = purchaseStartDay + gapCount + 120`
- These represent the same day-boundary deadline.

### Worked Example

`gapCount = 5`, `purchaseStartDay = 50`:
- Before: deadline = `50 + 120 = day 170`
- After gap shift: `purchaseStartDay = 55`, deadline = `55 + 120 = day 175`
- Deadline shifted by exactly 5 days (the stall duration).
- Old: `levelStartTime += 5*86400`, deadline shifts from `lst + 120 days` to `(lst + 5 days) + 120 days` = same 5-day extension.

**VERDICT: EQUIVALENT** -- Gap extension `purchaseStartDay += gapCount` is algebraically identical to old `levelStartTime += gapCount * 1 days` at day granularity.

## Section 6: JackpotModule isEthDay Verification (D-05)

### Current Code

JackpotModule line 466:
```solidity

```

### Old Code (from 188-03 SUMMARY)

13-line manual computation:
```solidity
// Old: (levelStartTime - JACKPOT_RESET_TIME) / 1 days - DEPLOY_DAY_BOUNDARY + 1
// This is equivalent to GameTimeLib.currentDayIndexAt(levelStartTime)
// which is exactly what purchaseStartDay stores.
```

### Equivalence Proof

1. `GameTimeLib.currentDayIndexAt(ts)` computes: `(ts - JACKPOT_RESET_TIME) / 86400 - DEPLOY_DAY_BOUNDARY + 1`
2. `purchaseStartDay` is set at level transition via `purchaseStartDay = day` (AdvanceModule line 312) where `day = _simulatedDayIndexAt(ts)` (line 161), which calls `GameTimeLib.currentDayIndexAt(ts)`.
3. Therefore `purchaseStartDay` stores exactly `GameTimeLib.currentDayIndexAt(levelTransitionTimestamp)`.
4. The old manual computation performed the same arithmetic inline -- it was a hand-written version of `GameTimeLib.currentDayIndexAt(levelStartTime)`.

Since `purchaseStartDay` stores the same day-index value the old code computed from `levelStartTime`, the comparison `questDay > purchaseStartDay` is semantically identical to `questDay > _simulatedDayIndexAt(levelStartTime)`.

### Sentinel Check Removal

Old code had a `type(uint48).max` sentinel check for uninitialized `levelStartTime`. This is safe to remove because:
- `purchaseStartDay` is always initialized in the constructor (DegenerusGame.sol line 206: `purchaseStartDay = currentDay`)
- At level transition, `purchaseStartDay = day` (AdvanceModule line 312) always sets a valid day index
- There is no code path where `purchaseStartDay` can be `type(uint48).max` or zero

### `lvl > 1` Guard

Unchanged between old and new code. The guard ensures ETH drip only activates after level 1 (not on the deploy level or first transition).

**VERDICT: EQUIVALENT** -- `questDay > purchaseStartDay && lvl > 1` produces identical behavior to the old 13-line manual timestamp-to-day conversion. Sentinel check removal is safe.

## Section 8: Edge Cases from CONTEXT.md Specifics

### 8.1: Pool Crosses Target on Day 120

**Scenario:** `purchaseStartDay = 100`, `currentDay = 220`, `level = 5`. Pool crosses target threshold on this day.

**Analysis of `_handleGameOverPath` (AdvanceModule line 479-481):**
```solidity
bool livenessTriggered = (lvl == 0 &&
    currentDay - psd > DEPLOY_IDLE_TIMEOUT_DAYS) ||
    (lvl != 0 && currentDay - psd > 120);
```

- `220 - 100 = 120`, `120 > 120` = FALSE.
- Liveness does NOT trigger on day 120. The function returns `false`.
- `advanceGame` continues normally to the purchase-phase logic.
- The target-met check at line 348-351 proceeds, `lastPurchaseDay` is set, and the game transitions to jackpot phase.

**The safety guard at line 500 is never reached** because liveness did not trigger:
```solidity
if (lvl != 0 && _getNextPrizePool() >= levelPrizePool[lvl]) {
    return false;
}
```
This guard is a defense-in-depth for the case where liveness triggers but the pool has actually met target. Since `>` is strict, day 120 never reaches this path.

**Day 121:** If the pool had NOT met target by day 121, `221 - 100 = 121 > 120` = TRUE, liveness triggers. The safety guard at line 500 would then check if pool met target in the interim, preventing false game-over.

### 8.2: Underflow Guard in `_evaluateGameOverAndTarget`

Current code (AdvanceModule line 1764):
```solidity
uint256 daysRemaining = psd + 120 > day ? psd + 120 - day : 0;
```

**Normal case:** `psd = 100`, `day = 200`:
- `100 + 120 = 220 > 200` = TRUE
- `daysRemaining = 220 - 200 = 20`

**Boundary case:** `psd = 100`, `day = 220`:
- `100 + 120 = 220 > 220` = FALSE
- `daysRemaining = 0` (ternary takes the 0 branch)

**Past-deadline case:** `psd = 100`, `day = 250`:
- `100 + 120 = 220 > 250` = FALSE
- `daysRemaining = 0` (no underflow, ternary guards it)

**Calling context:** `_evaluateGameOverAndTarget` is called in two places:
1. Level transition (line 315): `psd = day` (just set), so `day + 120 > day` is always true. Returns `daysRemaining = 120`.
2. Purchase phase (line 349): `psd` from level start, `day` is current. The ternary guard handles all cases.

At both call sites, `psd` is either current-day or a previous day, and `day` is current. The guard correctly prevents underflow when the deadline has passed.

**VERDICT: EQUIVALENT** -- Underflow guard handles `psd + 120 <= currentDay` correctly, returning 0 instead of underflowing. The strict `>` in liveness ensures day 120 is safe for normal operation.
