### Arithmetic Proof

Let the following symbols hold:

- F0 = futurePrizePool at function entry = baseFuturePool = initial futurePoolLocal
- B = BAF pool budget deducted from futurePoolLocal (0 if BAF does not fire)
- netSpend_BAF = actual ETH spent from B by _runBafJackpot (may be less than B if refunds/lootbox recycling occurs)
- D = Decimator pool budget deducted from futurePoolLocal (0 if Decimator does not fire)
- netSpend_DEC = actual ETH spent from D (may be less than D if returnWei > 0)
- R = total auto-rebuy contributions written directly to futurePrizePool storage during the call (sum of all calc.ethSpent where toFuture==true)

What futurePoolLocal holds at the write-back site (after all jackpot branches):

```
futurePoolLocal = F0
  - B                         (BAF budget pulled)
  + (B - netSpend_BAF)        (BAF refund/lootbox recycled back)
  - D                         (Decimator budget pulled)
  + (D - netSpend_DEC)        (Decimator refund recycled back)
= F0 - netSpend_BAF - netSpend_DEC
= F0 - totalJackpotSpend
```

What futurePrizePool storage holds at the write-back site:

Auto-rebuy wrote R to storage independently of futurePoolLocal. Storage was NOT decremented by runRewardJackpots for the jackpot deductions (those deductions live only in futurePoolLocal).

Therefore: storage = F0 + R

What rebuyDelta computes:

```
rebuyDelta = _getFuturePrizePool() - baseFuturePool
           = (F0 + R) - F0
           = R
```

What the fixed write-back writes:

```
_setFuturePrizePool(futurePoolLocal + rebuyDelta)
= _setFuturePrizePool((F0 - totalJackpotSpend) + R)
```

Correct final futurePrizePool must equal F0 - totalJackpotSpend + R.
The fix writes exactly that. QED.

### Path-by-Path Verification

**Path 1: Zero-rebuy (no player has auto-rebuy enabled, or no players won)**
- R = 0
- Storage at write-back: F0
- rebuyDelta = 0
- Write-back: futurePoolLocal + 0 = F0 - totalJackpotSpend
- Identical to pre-fix behavior. No regression.

**Path 2: Single auto-rebuy contribution**
- R = r1
- Storage at write-back: F0 + r1
- rebuyDelta = r1
- Write-back: (F0 - totalJackpotSpend) + r1
- Pre-fix: wrote F0 - totalJackpotSpend, losing r1. Fixed.

**Path 3: Multiple auto-rebuy contributions**
- R = r1 + r2 + ... + rN
- Storage at write-back: F0 + R
- rebuyDelta = R
- Write-back: (F0 - totalJackpotSpend) + R
- All contributions preserved. Pre-fix lost all of R. Fixed.

**Path 4: Level 100 (BAF + Decimator both fire)**
- B and D both non-zero; auto-rebuy may fire during either branch
- R = sum of all auto-rebuy contributions across both calls
- rebuyDelta captures the full sum regardless of which call generated it
- Analysis identical to Path 3.

### Underflow Safety

Could `_getFuturePrizePool() - baseFuturePool` underflow?

The only writes to futurePrizePool during runRewardJackpots are:
1. Auto-rebuy: `_setFuturePrizePool(_getFuturePrizePool() + calc.ethSpent)` -- always additive
2. No other path writes to futurePrizePool inside runRewardJackpots (Phase 100 scan confirmed)

Therefore `_getFuturePrizePool() >= baseFuturePool` is an invariant at the write-back site.
No underflow is possible. Solidity 0.8+ checked arithmetic would revert if violated (safety net).
