**Section 1: ETH Entry Points**
Enumerate every way ETH enters the protocol. For each entry point:
- Contract and function (with visibility and payable modifier)
- What triggers it (user action, automated, callback)
- Where the ETH goes initially (which pool/variable)
- Any splits that happen on entry (e.g., mint splits to multiple pools)

Search ALL contracts for: `payable` functions, `receive()`, `fallback()`, and any function that reads `msg.value`. Format as:

**Section 2: Internal ETH Movements**
Enumerate every internal pool-to-pool transfer within the protocol. These are state variable changes that move ETH between accounting pools WITHOUT actually sending ETH externally.

Extract from audit "ETH Mutation Path Map" sections and verify against source. Key pools to track:
- `nextPrizePool`, `currentPrizePool`, `futurePrizePool`
- `levelPrizePool[N]`
- `claimablePool`, `claimableWinnings[player]`
- `bafPool`, `decimatorPool`
- `lootboxPresaleMintEth`
- `affiliatePool`, `affiliateCreditPending`
- Vault pools (stETH, BURNIE backing)
- Stonk reserves

Format as:
| # | Source Pool | Destination Pool | Trigger | Function | Direction |
|---|-----------|-----------------|---------|----------|-----------|

**Section 3: ETH Exit Points**
Enumerate every way ETH leaves the protocol. Search for:
- `.call{value:` patterns (low-level ETH sends)
- `.transfer(` patterns
- `steth.submit{value:` (ETH to stETH conversion)
- Any external payable call with value

For each exit:
| # | Contract | Function | Recipient | Source Pool | Trigger |
|---|----------|----------|-----------|-------------|---------|

5. **Yield flow**: excess ETH -> stETH -> vault yield -> vault claims
6. **Affiliate flow**: mint affiliate cuts -> affiliate claims

Each diagram should show the pool names and the conditions that trigger movement.
