**Step 3 -- Create resume entry point (JackpotModule):**

Add a new external function `resumeDailyJackpotEth(uint24 lvl, uint256 randWord)` in JackpotModule. This function:

1. Reads `resumeEthPool` (must be non-zero -- the AdvanceModule check guarantees this)
2. Determines which path is being resumed:
   - Daily path: `dailyJackpotCoinTicketsPending == true` (set by payDailyJackpot daily path before the split)
   - Early-burn path: `dailyJackpotCoinTicketsPending == false`
3. For daily path resume:
   - Reconstruct the same parameters that payDailyJackpot built:
     - `entropyDaily = randWord ^ (uint256(lvl) << 192)`
     - `traitIdsDaily` from `lastDailyJackpotWinningTraits`
     - `bucketCountsDaily` from `bucketCountsForPoolCap(resumeEthPool, entropyDaily, DAILY_ETH_MAX_WINNERS, DAILY_JACKPOT_SCALE_MAX_BPS)`
     - `shareBpsDaily` from share packing (same logic as payDailyJackpot lines 450-454)
     - `isFinalPhysicalDay` from `jackpotCounter + counterStep >= JACKPOT_LEVEL_CAP`
   - Call `_processDailyEth(lvl, uint256(resumeEthPool), entropyDaily, traitIdsDaily, shareBpsDaily, bucketCountsDaily, isFinalPhysicalDay, true)` -- note `isResume=true`
   - Handle pool deductions for the resume paidEth (deduct from currentPrizePool or futurePrizePool as appropriate)
   - The _processDailyEth call with isResume=true clears resumeEthPool internally

4. For early-burn path resume:
   - Reconstruct parameters: `entropy = randWord ^ (uint256(lvl) << 192)`, winning traits from `lastDailyJackpotWinningTraits`
   - Call `_executeJackpot(JackpotParams{...}, true)` with isResume=true
   - Handle pool deductions for resume paidEth
   - resumeEthPool cleared internally

**Important edge case -- isFinalPhysicalDay reconstruction:**
The `isFinalPhysicalDay` flag is needed for daily path resume to pass to `_processDailyEth`. It can be recomputed from `jackpotCounter` + `counterStep`. The counterStep is packed in `dailyTicketBudgetsPacked` (set during call 1's payDailyJackpot). Call `_unpackDailyTicketBudgets(dailyTicketBudgetsPacked)` to get counterStep, then check `jackpotCounter + counterStep >= JACKPOT_LEVEL_CAP`.

**Important edge case -- pool deduction for daily final day:**
On the final physical day, lines 465-474 deduct full `dailyEthBudget` from currentPrizePool and refund unpaid to futurePrizePool. With the split, call 1 should NOT do the full deduction. Options:
- Option A: Call 1 deducts only its paidEth1. Call 2 (resume) deducts paidEth2 and handles the unpaid refund.
- Option B: Defer ALL deduction to call 2. Call 1 stores ethPool but touches no pools. Call 2 deducts totalPaid (paidEth1 + paidEth2) and refunds remainder.

Present both options to user. Option A is simpler (each call self-contained) but requires tracking paidEth1 across calls. Option B requires no extra storage but delays deductions.

Given D-07 says "No paidEthSoFar needed", Option A with per-call deduction (call 1 deducts its own from the pool, call 2 deducts its own) is preferred. For non-final days, each call independently `_setCurrentPrizePool(_getCurrentPrizePool() - paidEthN)`. For final day, call 1 deducts paidEth1 from pool. Call 2 deducts the remaining ethPool-paidEth1-paidEth2 as unpaid to futurePrizePool plus paidEth2 from current pool. But this requires knowing the original ethPool to compute unpaid -- which IS stored in resumeEthPool. So call 2 can compute: `unpaid = originalEthPool - paidEth1AlreadyDeducted - paidEth2`. Actually simpler: just store the original budget and let call 2 do `_setCurrentPrizePool(currentPool - paidEth2)` for non-final, and for final `unpaidEth = originalEthPool - totalPaidSoFar - paidEth2` using stored value.

Present the cleanest approach to the user and get approval.
