**Step 3 -- Split _processDailyEth (DegenerusGameJackpotModule.sol):**

Add a `bool isResume` parameter to `_processDailyEth`. When `isResume=false` (call 1):
- Compute `order = JackpotBucketLib.bucketOrderLargestFirst(bucketCounts)`
- Compute `remainderIdx = JackpotBucketLib.soloBucketIndex(entropy)`
- Compute `shares = JackpotBucketLib.bucketShares(...)`
- Iterate only `order[0]` (largest bucket) and the solo bucket (`remainderIdx`)
  - If `order[0] == remainderIdx`, they are the same bucket; iterate just that one, then iterate `order[1]` as well to keep call 1 with 2 buckets
  - Otherwise iterate `order[0]` and `remainderIdx` (2 buckets)
- After processing, store `resumeEthPool = uint128(ethPool)` to signal resume pending
- Return paidEth for call 1

When `isResume=true` (call 2):
- Read `ethPool` from `resumeEthPool` storage (cast back to uint256)
- Recompute `order`, `remainderIdx`, `shares` identically from the same entropy + stored ethPool
- Iterate the remaining 2 buckets (those NOT processed in call 1)
- Clear `resumeEthPool = 0`
- Return paidEth for call 2

The key logic for determining which buckets are "call 1" vs "call 2":
```
order = bucketOrderLargestFirst(bucketCounts)
remainderIdx = soloBucketIndex(entropy)

// Call 1 processes: order[0] (largest) + remainderIdx (solo)
// Call 2 processes: the other 2 from order[1..3] that are not remainderIdx

// Edge case: if order[0] == remainderIdx (largest IS the solo bucket),
// call 1 processes order[0] + order[1], call 2 processes order[2] + order[3]
```

Build a `bool[4] memory processedInCall1` mask and iterate accordingly.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| advanceGame caller -> stage machine | External caller triggers jackpot processing; stage machine controls flow |
| call 1 -> call 2 (resumeEthPool) | Inter-call state stored in storage; must be tamper-resistant |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-195-01 | Tampering | resumeEthPool between calls | mitigate | resumeEthPool is an `internal` storage variable, only writable by the contract's own code via delegatecall. No external setter exists. Stage machine prevents re-entry into call 1 while resume is pending. |
| T-195-02 | Spoofing | Calling resume path without a pending call 1 | mitigate | AdvanceModule stage routing checks `resumeEthPool != 0` before entering STAGE_JACKPOT_ETH_RESUME. If zero, stage is skipped. |
| T-195-03 | Denial of Service | Attacker skips call 2, leaving resumeEthPool non-zero permanently | accept | advanceGame is permissionless (anyone can call it). Stage machine will route to resume stage on next call. No permanent lockout possible. |
| T-195-04 | Information Disclosure | resumeEthPool leaks pool size | accept | ethPool is already observable via prize pool getters. No new information exposed. |
| T-195-05 | Repudiation | Inconsistent bucket computation between call 1 and call 2 | mitigate | Both calls recompute shares from the same deterministic inputs (stored ethPool + on-chain RNG word + entropy). bucketShares and bucketOrderLargestFirst are pure functions. Identical inputs = identical outputs. |
</threat_model>
