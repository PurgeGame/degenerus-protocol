### A.1.1 The two new sibling producers (RARE-01 — frozen producers untouched)

The foil tickets are generated by a **new sibling** of the Degenerette producer, cloned
structurally from `packedTraitsDegenerette` / `_degTrait` (`DegenerusTraitUtils.sol:201-223`):

- `traitFromWordFoil(uint64 rnd, uint256 multBps) internal pure returns (uint8)` —
  the per-quadrant 6-bit `[CCC][SSS]` producer: **color** from the tapered ladder of §A.1.4
  (the multiplier `multBps` enters here, `10000 = 1×`), **symbol** uniform `1/8` from
  `rnd >> 32 & 7` (byte-identical to `_degTrait`'s symbol stage). Quadrant bits added by the caller.
- `packedTraitsFoil(uint256 rand, uint256 multBps) internal pure returns (uint32)` —
  the 4-quadrant packer: calls `traitFromWordFoil` on each of the four 64-bit lanes
  (`uint64(rand)`, `>>64`, `>>128`, `>>192`), OR's in the quadrant identifiers
  (`| 64`, `| 128`, `| 192`), and packs four bytes into a `uint32` in the
  **identical `[QQ][CCC][SSS]`-per-byte layout** as `packedTraitsFromSeed` /
  `packedTraitsDegenerette` (`return uint32(traitA) | (traitB << 8) | (traitC << 16) | (traitD << 24)`).

**ONLY the color stage changes** vs the Degenerette sibling. The symbol stage and the pack
layout are copied verbatim. **RARE-01: the frozen producers `weightedColorBucket`,
`traitFromWord`, and `packedTraitsFromSeed` (`DegenerusTraitUtils.sol:115`, `:143`, `:169`)
are NOT edited, retyped, or moved** — the foil path is purely additive (a new sibling),
identical in spirit to how `packedTraitsDegenerette` already coexists beside the frozen
heavy-tail producer.

`multBps` is the `foilBoostBps` output (§A.2), **frozen at buy** from `cachedScore` and passed
in at resolve — never live-read (RARE-03; see §A.2.4).

### A.1.3 The taper rule (RARE-04 — tapered, NOT flat ×M)

The boost is **linear in the rare-rank**: the rarest tier (gold) takes the full multiplier `M`,
each less-rare tier tapers toward `1×`. With `M = multBps / 10000`:

```
boost(c) = 1 + (M − 1) · w_c      with rare-rank weights w_c ∈ [0,1]
  w_7 = 5/5   (gold = full ×M)
  w_6 = 4/5
  w_5 = 3/5
  w_4 = 2/5
  w_3 = 1/5   (least-rare boosted tier; still > 1× for any M > 1)
  w_0 = w_1 = w_2 = 0   (the three 25% commons — the SOLE funding sink; never boosted)
```

Per-tier probability:

```
p_c       = (base[c] / 256) · (1 + (M−1)·w_c)        for c ∈ {3,4,5,6,7}
p_0 = p_1 = p_2 = (192/256 − Δ) / 3                  where Δ = Σ_{c∈{3..7}} p_c − Σ_{c∈{3..7}} base[c]/256
```

This satisfies every D-03 acceptance constraint (all re-verified exactly in V1):

- **`p_gold = (2/256)·M` exactly** — the LOCKED invariant. `w_7 = 5/5 = 1 ⇒ p_7 = (2/256)·M`.
  At `M = 6 ⇒ 12/256 = 4.6875% ≈ 4.7%` (the spec's gold target). **FOIL-05.**
- **Monotone-decreasing boost down the tail** — `w_7 > w_6 > w_5 > w_4 > w_3 > 0`, so
  `boost(7) > … > boost(3) > 1` for all `M > 1` (factors `[6,5,4,3,2]` at M=6,
  `[2,1.8,1.6,1.4,1.2]` at M=2). **RARE-04.**
- **Commons 0/1/2 the sole funding source** — only the three 25% commons shrink; tiers 3–7 only grow.
- **Symbol uniform 1/8 — out of scope, unchanged** (the boost is color-tier ONLY; symbol stage
  copied verbatim from `_degTrait`). **RARE-02 / D-03.**

### A.1.4 Resolve mechanism — the `/15360` integer cutoff ladder (build-ready, exact)

256-resolution is too coarse for the taper (at M=2 the commons are `55.867/256` and tiers 3–6 are
`38.4 / 22.4 / 12.8 / 10.8`, none integral). The build-ready resolve computes cutoffs in a
`/15360` super-ladder (`15360 = 256 × 60`; the `×60` clears both the `/5` taper denominator and
the 3-way common split), then compares a scaled draw against the running sum — style-identical to
`weightedColorBucket`'s `if (scaled < …) return …` chain.

Per rare tier `c ∈ {3,4,5,6,7}` with `w5 = {3:1, 4:2, 5:3, 6:4, 7:5}` (the `w_c · 5` integers):

```
width15360[c] = base[c]·60 · (50000 + (multBps − 10000)·w5[c]) / 50000
rare          = Σ_{c∈{3,4,5,6,7}} width15360[c]
rem           = 15360 − rare
commons[0..2] = rem / 3, with the (rem mod 3) residual handed to the FIRST common (color 0)
```

Gold lands exactly on `width15360[7] = base[7]·60·M = 2·60·M = 120·M`
(i.e. exactly `baseline×M`, the `p_gold` invariant on the ladder).

**The IMPL ladder, in directive form:** compute the five `width15360[c]` for c=3..7 by the formula
above; sum to `rare`; set `rem = 15360 − rare`; the three commons each get `rem / 3`, and color 0
additionally absorbs `rem mod 3`. A draw scaled into `[0, 15360)` walks the running cutoff chain
in the order `[0, 1, 2, 3, 4, 5, 6, 7]` (or any fixed order whose widths sum to 15360) and returns
the first tier whose cumulative cutoff it falls under — identical control flow to `weightedColorBucket`.
