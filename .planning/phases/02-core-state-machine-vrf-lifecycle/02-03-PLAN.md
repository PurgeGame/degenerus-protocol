VRF request sites:
  1. _requestRng() in AdvanceModule — daily RNG request
     - Sets rngLockedFlag via _finalizeRngRequest()
     - Uses VRF_REQUEST_CONFIRMATIONS = 10

  2. requestLootboxRng() in AdvanceModule — mid-day lootbox request
     - Does NOT set rngLockedFlag
     - Uses VRF_MIDDAY_CONFIRMATIONS = 3
     - Checks: rngLockedFlag == false, rngRequestTime == 0, daily RNG consumed, LINK >= 40
     - Sets vrfRequestId, rngRequestTime directly (not via _finalizeRngRequest)

  3. _tryRequestRng() in AdvanceModule — game-over RNG request
     - Uses _finalizeRngRequest() (same as daily)

VRF callback:
  rawFulfillRandomWords() — single callback for both daily and lootbox
  Dispatch: rngLockedFlag determines which path (daily vs lootbox)

Single vrfRequestId slot:
  Only ONE pending VRF request at a time. This means:
  - Daily request blocks lootbox requests (rngLockedFlag check)
  - Lootbox request blocks daily requests (rngRequestTime check in rngGate)
  - No concurrent requests possible by design

Chainlink VRF V2.5 8-point checklist (from research):
  1. RequestId matching
  2. Block confirmations
  3. No re-requesting/cancellation
  4. Close input windows
  5. Callback must not revert
  6. Use VRFConsumerBaseV2Plus
  7. Avoid ERC-4337 wallets
  8. Maintain adequate funding

2. Trace the requestId lifecycle for daily RNG:
   a. _requestRng() calls vrfCoordinator.requestRandomWords() → returns id
   b. _finalizeRngRequest() sets vrfRequestId = id
   c. rawFulfillRandomWords() checks requestId != vrfRequestId → silent return on mismatch
   d. _unlockRng() clears vrfRequestId = 0
   Question: Is there any window where vrfRequestId has the wrong value?

3. Trace the requestId lifecycle for lootbox RNG:
   a. requestLootboxRng() calls vrfCoordinator.requestRandomWords() → returns id
   b. Directly sets vrfRequestId = id
   c. _reserveLootboxRngIndex(id) maps id → lootbox index
   d. rawFulfillRandomWords() checks requestId != vrfRequestId → silent return on mismatch
   e. On match with rngLockedFlag=false: reads lootboxRngRequestIndexById[requestId] → index
   f. Clears vrfRequestId = 0, rngRequestTime = 0
   Question: What if a stale daily fulfillment arrives after vrfRequestId was overwritten by a lootbox request?

4. Trace the 18h retry path:
   a. rngGate() detects elapsed >= 18h with rngWordCurrent == 0
   b. Calls _requestRng() again → new VRF request with new requestId
   c. _finalizeRngRequest() overwrites vrfRequestId with new id
   d. Old fulfillment (if it arrives) will have wrong requestId → silent return
   Question: Does the retry properly handle lootbox index remapping? Research says "retry remaps lootbox index" — verify this.

**Part A: 8-Point VRF Security Checklist**

Evaluate each point against the codebase. For each, provide:
- PASS / FAIL / DEVIATION / N-A verdict
- Code reference (file:line)
- Reasoning

1. **RequestId matching:**
   - Verify requestId == vrfRequestId check in rawFulfillRandomWords
   - Verify mismatch handling (silent return, not revert)
   - Assess: is silent return the right choice? (Yes — prevents coordinator retries from being wasted)

2. **Block confirmations:**
   - Document VRF_REQUEST_CONFIRMATIONS = 10 for daily
   - Document VRF_MIDDAY_CONFIRMATIONS = 3 for lootbox
   - Assess whether 3 confirmations for lootbox is sufficient (lower-value outcome, acceptable)

3. **No re-requesting/cancellation:**
   - The 18h timeout re-requests the VRF — this is technically a deviation from the guideline
   - The old requestId is overwritten, so old fulfillments are silently ignored
   - Assess whether this creates any risk: can a validator withhold fulfillment for 18h to force a re-request, then fulfill the original to cause a mismatch? (No — requestId is overwritten)

4. **Close input windows:**
   - rngLockedFlag blocks: nudges, purchases, lootbox opens, config changes
   - List all operations blocked during lock (from the 13 check sites)
   - Are any state-changing operations NOT blocked during the VRF wait?

5. **Callback must not revert:**
   - Reference the revert path analysis from Task 1 / Plan 02-02
   - Summarize: only the coordinator check can revert, and it's unreachable in normal operation

6. **Use VRFConsumerBaseV2Plus:**
   - The protocol does NOT inherit VRFConsumerBaseV2Plus — deliberate deviation for delegatecall compatibility
   - Document what VRFConsumerBaseV2Plus provides that the custom implementation must replicate:
     a. Coordinator address validation (implemented: msg.sender check)
     b. Request tracking (implemented: vrfRequestId)
     c. Callback routing (implemented: custom rawFulfillRandomWords)
   - Assess whether the custom implementation is equivalent in security terms

7. **Avoid ERC-4337 wallets:**
   - N/A — no ERC-4337 integration

8. **Maintain adequate funding:**
   - Lootbox RNG checks MIN_LINK_FOR_LOOTBOX_RNG (40 LINK)
   - Daily RNG does NOT check LINK balance — reverts if request fails (intentional: halts game)
   - Assess whether the daily RNG approach is safe (Yes — halting is the correct behavior when underfunded)

**Part B: 18h Retry Timeout Deep Dive (RNG-07)**

1. Read the 18h timeout handling in rngGate():
   - What is the exact condition checked?
   - What happens on retry? (New request, new requestId, old request abandoned)

2. Assess validator abuse scenario:
   - A validator who is also a player could withhold VRF fulfillment for 18h to trigger a re-request
   - On re-request, the validator sees both the old and new VRF words
   - Can they choose which to apply? (No — old requestId is overwritten, old fulfillment silently ignored)
   - Can they delay the NEW fulfillment to get another re-request? (Yes, but same analysis applies)
   - Is there any advantage to forcing a re-request? (The new random word is independent of the old one)

3. Verdict for RNG-07: Can the 18h timeout be abused by a validator?
