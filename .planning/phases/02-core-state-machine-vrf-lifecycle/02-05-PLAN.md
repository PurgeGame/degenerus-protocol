Enumerate all possible stuck states in the Degenerus game FSM and trace every recovery mechanism. Verify that each recovery path is reachable when needed and cannot be triggered prematurely by a malicious actor. Confirm that no game state exists from which there is no exit.

Purpose: FSM-02 requires proof that every game state is exitable. RNG-06 requires confirmation that the RNG lock specifically cannot be permanently stuck. Together these requirements ensure the game's liveness guarantee — the protocol can always make progress or gracefully terminate, regardless of VRF failures, validator behavior, or attacker actions.

Stuck-state recovery mechanisms (from research):

1. **18h VRF retry (most common recovery):**
   Trigger: rngRequestTime != 0 && rngWordCurrent == 0 && elapsed >= 18h
   Action: rngGate() calls _requestRng() again with new requestId
   Location: AdvanceModule rngGate() around line 667-669
   Anti-premature: elapsed < 18h → revert RngNotReady

2. **3-day emergency coordinator rotation (admin recovery):**
   Trigger: _threeDayRngGap(day) returns true (3 consecutive days without RNG word)
   Action: updateVrfCoordinatorAndSub() rotates coordinator AND clears rngLockedFlag
   Location: AdvanceModule line ~1148
   Anti-premature: _threeDayRngGap checks rngWordByDay[day-2] == 0 (guard against underflow at day < 2)
   Access: Requires msg.sender == ADMIN (DegenerusAdmin contract)

3. **Game-over VRF fallback (3-day delay):**
   Trigger: Game-over path needs RNG but VRF not available for 3+ days
   Action: _gameOverEntropy() uses keccak256(historicalVRFWord, currentDay) instead of live VRF
   Location: AdvanceModule _gameOverEntropy()
   Anti-premature: GAMEOVER_RNG_FALLBACK_DELAY = 3 days must elapse
   Fallback failure: If no historical VRF words exist → revert E() (catastrophic)

4. **30-day final sweep (post-game-over):**
   Trigger: block.timestamp >= gameOverTime + 30 days
   Action: handleFinalSweep() sweeps remaining funds to designated recipient
   Location: GameOverModule line ~228-243
   Anti-premature: Timestamp comparison

5. **Liveness timeout (game-over trigger itself):**
   Trigger: Level 0: 912 days idle; Level 1+: 365 days since levelStartTime
   Action: _handleGameOverPath() initiates game-over sequence
   Location: AdvanceModule _handleGameOverPath()
   Anti-premature: Timestamp comparison against levelStartTime

Time constants (from MEMORY.md):
  912 days: pre-game timeout (level 0)
  365 days: post-game inactivity
  18 hours: VRF retry timeout
  3 days: emergency stall
  30 days: final sweep

**Part A: Premature-Trigger Resistance Analysis**

For each recovery mechanism, assess whether a malicious actor can trigger it before the legitimate timeout:

1. **18h VRF retry:**
   - Can an attacker call advanceGame() before 18h and get a new VRF request?
   - The guard is: if elapsed < 18h, revert RngNotReady
   - Can block.timestamp be manipulated? (Validators can shift by seconds, not hours)
   - Can rngRequestTime be manipulated? (Storage variable, not user-controllable)
   - Verdict: Premature trigger possible or impossible?

2. **3-day emergency coordinator rotation:**
   - Can an attacker call updateVrfCoordinatorAndSub() without ADMIN access?
   - Trace the access control: DegenerusGame → DegenerusAdmin → who can call?
   - _threeDayRngGap checks: what if an attacker fills rngWordByDay[day-2] to prevent the gap detection?
   - Can the gap be artificially created? (Would require 3 days of no RNG — attacker cannot prevent VRF fulfillment unless they control the validator network)
   - Verdict: Premature trigger possible or impossible?

3. **Game-over VRF fallback (3-day):**
   - The GAMEOVER_RNG_FALLBACK_DELAY requires 3 days of VRF failure during game-over
   - Can this be shortened? (Constant, not configurable)
   - Can an attacker prevent VRF fulfillment for 3 days to force the fallback? (Validator attack — assess cost and feasibility)
   - If the fallback IS triggered, is the keccak256(historicalWord, day) output manipulable? (No — historical word is already recorded)
   - Verdict: Premature trigger possible or impossible?

4. **30-day final sweep:**
   - Can an attacker trigger handleFinalSweep() before 30 days?
   - The guard: block.timestamp >= gameOverTime + 30 days
   - gameOverTime is set in handleGameOverDrain() — can it be set to a past value? (Uses block.timestamp at call time — cannot be past)
   - Verdict: Premature trigger possible or impossible?

5. **Liveness timeout (game-over trigger):**
   - Can an attacker trigger the 912-day or 365-day timeout early?
   - levelStartTime is set during phase transitions — can it be manipulated?
   - What resets levelStartTime? (Each new level starts a fresh 365-day countdown)
   - Could a validator manipulate block.timestamp enough to trigger? (Would need months of manipulation — impossible)
   - Verdict: Premature trigger possible or impossible?

**Part B: Completeness Check**

Systematically verify: is there ANY game state with NO recovery path?

1. Create a matrix:
   | State | Stuck Condition | Recovery | Access | Timeout | Verdict |

2. For each state, confirm at least one recovery path exists.

3. Special case: What if the game is at level 0 with no players, no VRF words, and _handleGameOverPath is reached?
   - _gameOverEntropy() needs historical VRF words — but if no RNG ever recorded, it reverts
   - Is this reachable? Only after 912 days of inactivity from deployment
   - Assess: would any VRF word have been generated in 912 days? (Yes — daily advanceGame would have generated hundreds)
   - Edge case: game deployed but advanceGame() never called for 912 days → revert E() with no recovery

   - updateVrfCoordinatorAndSub requires 3-day gap and ADMIN (emergency path)
   - If ADMIN key is lost AND VRF never fulfills → locked forever? (Assess this scenario)
