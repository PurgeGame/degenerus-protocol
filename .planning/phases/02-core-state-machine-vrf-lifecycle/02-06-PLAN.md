EntropyLib implementation (contracts/libraries/EntropyLib.sol, 24 lines):
```solidity
function entropyStep(uint256 state) internal pure returns (uint256) {
    unchecked {
        state ^= state << 7;
        state ^= state >> 9;
        state ^= state << 8;
    }
    return state;
}
```

Known properties:
  - Shift constants (7, 9, 8) are NON-STANDARD for xorshift
  - Standard xorshift64 uses (13, 7, 17) or (12, 25, 27)
  - Standard xorshift128 uses (11, 8, 19) or (23, 17, 26)
  - No published analysis exists for (7, 9, 8) on 256-bit state
  - Zero is a fixed point (XOR of 0 with any shift of 0 = 0)
  - VRF words are mapped from 0 to 1, so zero-state is unreachable

Consumer modules that call entropyStep:
  - AdvanceModule — jackpot winner selection
  - JackpotModule — jackpot distribution
  - LootboxModule — lootbox outcome determination
  - GameOverModule — game-over trait generation
  - DegeneretteModule — degenerette bet outcomes
  - Other modules — perk odds, trait selection, etc.

The RNG derivation chain:
  VRF coordinator → rawFulfillRandomWords → rngWordCurrent
  → advanceGame → _applyDailyRng (add nudges) → rngWordByDay[day]
  → consumers read rngWordByDay[day] as seed
  → repeated entropyStep() calls derive multiple values
  → entropy % range for modular reduction

block.timestamp usage (from research — timing/gating only):
  - levelStartTime comparison for liveness timeout
  - rngRequestTime for 18h timeout
  - boon expiry checks
  - VRF fallback delay (3 days)
  NEVER used as randomness

blockhash/prevrandao/difficulty (from research):
  - NOT used anywhere in codebase
  - Only mentioned in comments as comparison

**Part A: XOR-Shift Analysis**

1. **Read EntropyLib.sol** completely (24 lines). Document the exact implementation.

2. **Analyze the shift constants (7, 9, 8) on uint256:**

   a. Fixed points: The only fixed point of xorshift is 0 (XOR of 0 << n = 0 for any n).
      Confirm: if state = 0, then state ^= 0 << 7 = 0, state ^= 0 >> 9 = 0, state ^= 0 << 8 = 0.
      Result: state = 0 is a fixed point. This is universal for all xorshift variants.
      Mitigation: VRF word 0 → 1 mapping in rawFulfillRandomWords prevents zero-seed entry.

   b. Near-fixed-point behavior: What about state = 1? Compute:
      step(1) = 1 ^ (1 << 7) ^ ((1 ^ (1 << 7)) >> 9) ^ ((result) << 8)
      Manually trace the XOR operations for state = 1 to verify non-trivial output.

   c. Period analysis: For standard xorshift with properly chosen constants, the period is 2^n - 1 where n is the state width. For non-standard constants, the period may be shorter.
      - For 256-bit state with shifts (7, 9, 8), the theoretical maximum period is 2^256 - 1.
      - Without a polynomial irreducibility proof specific to these constants, we cannot guarantee full period.
      - HOWEVER: the protocol never runs entropyStep more than ~20-30 times per VRF word (one per random selection needed). A short cycle would need to be < 30 to affect outputs.
      - Assessment: Even a poor xorshift variant with period > 2^64 would be sufficient for < 30 iterations.

   d. Output distribution: Standard xorshift is NOT equidistributed (some bit patterns more likely than others). However, after modular reduction (entropy % range), the bias is negligible for small ranges.
      - For typical ranges (< 10000), the bias from xorshift is astronomically small compared to VRF seed uniformity.

   e. Comparison to standard constants: Why might (7, 9, 8) be worse than standard (13, 7, 17)?
      - Standard constants are chosen to maximize the number of non-zero terms in the characteristic polynomial of the linear transformation.
      - Non-standard constants may have lower rank, producing more correlated outputs.
      - For security-seeded usage (VRF seed), this matters less than for standalone PRNG usage.

3. **Assess exploitability:**
   Given that:
   - The seed is VRF (cryptographically random)
   - entropyStep is called < 30 times per seed
   - The attacker does not choose the seed (only VRF coordinator provides it)
   - Nudges add a small integer to the seed (max ~10-20 in practice due to exponential cost)

   Can an attacker predict or bias the output of entropyStep?
   - To predict: would need to know the VRF word before it's applied (blocked by rngLockedFlag — covered by RNG-01/RNG-08)
   - To bias: would need to influence the seed (only via nudges, which are limited and applied after VRF)
   - To exploit correlations: would need the xorshift to have exploitable structure AND know the seed

   - Whether the entropy variable is properly threaded (each call uses previous output as input)

   - Verify no call site reuses the SAME entropy state for multiple selections (would produce identical results)
   - Verify no call site passes a constant or user-controlled value instead of the VRF-derived state

**Part B: End-to-End RNG Derivation Chain**

Trace the complete chain from VRF to consumer:

1. **VRF acquisition:**
   vrfCoordinator.requestRandomWords() → requestId → rawFulfillRandomWords(requestId, words) → rngWordCurrent = words[0]

2. **Nudge application:**
   advanceGame() → _applyDailyRng(day, rngWordCurrent) → finalWord = rngWordCurrent + totalFlipReversals → rngWordByDay[day] = finalWord

3. **Consumer access:**
   Each module reads rngWordByDay[dailyIdx] as the seed. How?
   - Does each module read the storage variable directly? (Yes — delegatecall shares storage)
   - Is the seed passed as a parameter? (Check both patterns)

4. **Entropy derivation:**
   seed → entropyStep(seed) → entropy1 → entropyStep(entropy1) → entropy2 → ... → entropyN
   Each entropyN % range produces one random selection.

5. **Independence assessment:**
   - Different days use different VRF words → independent seeds
   - Within a day, all random selections derive from the SAME seed via sequential entropyStep
   - This means within-day selections are deterministically related (by design — reproducible outcomes)
   - An attacker who knows the seed can compute all outcomes for that day
   - Is this a problem? Only if the attacker can see the seed before outcomes are applied (covered by rngLockedFlag)
