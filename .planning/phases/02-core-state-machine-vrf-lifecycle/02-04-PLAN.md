  PURCHASE → PURCHASE_LAST_DAY → JACKPOT → JACKPOT (x4) → PHASE_TRANSITION → PURCHASE

Level increment timing (research pitfall #6):
  Level increments in _finalizeRngRequest() BEFORE VRF fulfillment.

  purchaseLevel handles this: (lastPurchase && rngLockedFlag) ? lvl : lvl + 1

Liveness timeouts:
  Level 0: 912 days (2.5 years)

**Part A: Multi-Step Game-Over Sequence (FSM-03)**

The game-over process is NOT a single transaction. It requires multiple advanceGame() calls:

1. **Call 1: Liveness timeout detected**
   - _handleGameOverPath() detects timeout
   - What happens next? Does it request VRF? Does it set gameOver immediately?
   - Read _handleGameOverPath() carefully — trace every branch

2. **Call 2 (if VRF needed): VRF request submitted**
   - _gameOverEntropy() acquires RNG
   - What if VRF fails? (3-day fallback using historical VRF word)
   - What state is the game in between Call 1 and Call 2?

3. **Call 3: VRF fulfilled, game-over finalized**
   - handleGameOverDrain() distributes funds
   - gameOver = true, gameOverTime = block.timestamp
   - What state changes happen? Trace all SSTOREs

4. **Call 4 (after 30 days): Final sweep**
   - handleFinalSweep() sweeps remaining funds
   - Where do swept funds go?
   - What if there are unclaimed player balances?

For each intermediate state, document:
- What state variables are set/unset
- What operations are legal in this state (can players still purchase? claim?)
- What operations are illegal
- What happens if advanceGame() is called before the state is ready
