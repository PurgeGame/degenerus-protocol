### Per-surface verdicts

- **Surface (a) — Bit-slice modulo-bias bound: AGREE — SAFE_BY_DESIGN. Mechanism strength: STRUCTURAL.**

  Computed each documented bias from first principles. `% 100` from 16 bits: 65536 mod 100 = 36, bias 36/65536 ≈ 0.055% (claim says 0.05% — within rounding). `% 5` from 8 bits: 256 mod 5 = 1, bias 1/256 ≈ 0.391% (claim says 0.39%). `% 46` from 16 bits: 65536 mod 46 = 18, bias 18/65536 ≈ 0.027% (claim says 0.05% — claim is conservatively LOOSER than actual; safe). `% 20` from 16 bits: 65536 mod 20 = 16, bias 16/65536 ≈ 0.024% (claim says 0.02% — within rounding). `% 1000` from 24 bits: 16777216 mod 1000 = 216, bias 216/16777216 ≈ 0.0013% (claim says 0.0024% — conservatively looser; safe). `% 10000` from 24 bits: 16777216 mod 10000 = 7216, bias 7216/16777216 ≈ 0.043% (claim says 0.045% — within rounding). `% 1_000_000` from 32 bits: 4294967296 mod 1000000 = 967296, bias ≈ 0.0225% (claim says 0.022%). All claimed biases are within the documented ≤ 1% bound, and several are conservatively LOOSER than computed (no exploit gap from understated bias). STAT-01 chi² empirical evidence at `test/stat/LootboxEntropyDistribution.test.js` covers all 6 high-frequency consumers; the 2 low-frequency ones (`% 1000` DGNRS and `% 1_000_000` boon) are bit-disjoint from the empirically-tested set and inherit the same uniformity argument by keccak diffusion.

- **Surface (b) — Seed-reuse cross-correlation: AGREE — SAFE_BY_DESIGN. Mechanism strength: STRUCTURAL.**

  Bit-allocation map at L835-849 enumerates 8 bit ranges that I verified disjoint by visual inspection: [0..15] / [16..23] / [24..39] / [40..55] / [56..79] / [80..95] / [96..119] / [120..151]. No two ranges overlap. Cumulative consumption 152 bits / 256 available — comfortable headroom (no silent overflow into adjacent ranges). The `nearOffset` consumes bits[16..23] only when the far branch is NOT taken (else fall-through), and the `farOffset` consumes bits[24..39] only when the far branch IS taken. The two are mutually exclusive within `_rollTargetLevel` so they don't actually compete for the same draw — a strictly less-strict requirement than disjoint allocation. keccak256 output bits exhibit cryptographic diffusion (per the standard sponge-construction analysis); slicing disjoint bit ranges yields draws that are pseudo-independent at the bit-pair level, validated empirically by STAT-01 chi² over 6 consumer slices.

- **Surface (c) — `hash2(seed, N)` chunk-collision-free: AGREE — SAFE_BY_DESIGN. Mechanism strength: STRUCTURAL.**

  Single `hash2(seed, N)` site in the lootbox path: L936 `uint256 seed2 = EntropyLib.hash2(seed, 1);` inside the `amountSecond != 0` branch of `_resolveLootboxCommon`. `EntropyLib.hash2(uint256 a, uint256 b) → uint256` is `keccak256(abi.encode(a, b))` per `contracts/libraries/EntropyLib.sol` L36-42 (file BYTE-IDENTICAL at v36 close). `seed` (the primary chunk) is `keccak256(abi.encode(rngWord, player, day, amount))` — 128 bytes input. `seed2 = keccak256(abi.encode(seed, 1))` — 64 bytes input. Different input lengths AND different byte sequences → different 256-bit outputs with overwhelming probability (keccak collision probability ≈ 2^−256). The bit ranges WITHIN seed2 are independent of the bit ranges WITHIN seed (different keccak outputs); a consumer slicing the same bit range (e.g., [40..55]) from seed (first invocation) and seed2 (second invocation) gets two pseudo-independent draws. The forward-looking note in §4 prose ("future phases extending hash2(seed, N) must extend the bit-allocation map") is correct — currently only counter `1` is in use; future expansions to (seed, 2), (seed, 3) etc. would need explicit coordination to avoid bit-range collision across counter-tagged chunks if they ever shared the same consumer.

### Investigated novel attack hypotheses

- **Hypothesis (Z1) — Same-player Decimator multi-level claim seed collision.** Could a player claim Decimator jackpots at multiple levels in the SAME transaction (e.g., `claimDecimatorJackpot(lvl=5); claimDecimatorJackpot(lvl=10);`) such that two `_resolveLootboxDirect` invocations produce IDENTICAL seeds via collision on `(rngWord, player, day, amount)`?
  - **Investigation:** Read `DegenerusGameDecimatorModule.sol:321 claimDecimatorJackpot`. Each level has its OWN `decClaimRounds[lvl].rngWord` (set per-round at L258 in `runDecimatorJackpot`). Same player, same `block.timestamp` (same day) — but DIFFERENT `rngWord` per level → seed differs even if the lootboxPortion coincidentally matches across levels.
