# Phase 483 / Plan 01: FF-Salvage Entry-Granularity (gated contract diff #5 — BEHAVIOR change)

## Objective

Close the LAST entry-granularity gap in the protocol. Far-future (FF) salvage is the **sole** surface
that still operates at whole-ticket granularity while purchase / redeem are already entry-granular
(ledger §10.3, granularity scan `wf_da866c28`). Make FF-salvage entry-granular via the **5-site coupled
change**, and rename the FF-salvage surface `sellFarFutureTickets → sellFarFutureEntries` (+ the preview,
internal helper, and Vault/IGamePlayer/Game wrappers in lockstep).

This is a **genuine value-behavior change** (unlike 480/481/482 which were value-neutral): after the
change a seller can salvage SUB-whole-ticket amounts (e.g. 2 entries = half a whole ticket), and the
floor that previously forced a whole-ticket minimum now floors at ONE ENTRY. At whole-ticket-aligned
points (entries a multiple of 4) every value is **byte-identical** to the pre-change whole-ticket code —
proven by an explicit no-regression test.

## Unit algebra — why all 5 sites MUST move together (no 4× mis-value)

The input array `quantities[i]` changes meaning: **whole tickets → entries** (4 entries = 1 whole ticket).
`priceForLevel(L)` is the price of one WHOLE ticket; one entry's face = `priceForLevel(L) / 4`.

| symbol | OLD (whole-ticket input W) | NEW (entry input E) |
|--------|----------------------------|---------------------|
| faceWei (one line) | `priceForLevel(L) * W` | `priceForLevel(L) * E / 4` |
| entries debited / queued | `W * 4` | `E` |
| budget | `faceWei * frac(d) * jitter / 1e8` | same formula (faceWei now per-entry-scaled) |

Consistency at the aligned point `E = 4W` (same physical amount):
- NEW faceWei = `priceForLevel(L) * 4W / 4 = priceForLevel(L) * W` = OLD faceWei ✔
- NEW budget = OLD budget ✔
- NEW entries debited = `4W` = OLD entries debited ✔  →  byte-identical, no regression.

**The 4× trap:** changing ONLY site 1 (`entries = quantities[i]` instead of `* 4`) while leaving site 2
(`faceWei = priceForLevel(L) * n`) would value `E` entries as `E` WHOLE tickets → the buyer overpays /
the seller is under-debited by exactly 4×. Site 2's `/ 4` is the coupled valuation half that prevents it.

## The 5 coupled sites (ledger §10.3)

1. **`DegenerusGameMintModule.sol` ~1162** — `uint32 entries = uint32(quantities[i]) * 4;` →
   `uint32 entries = uint32(quantities[i]);` (input now IS entries; debit/queue use it directly).
2. **`DegenerusGameMintStreakUtils.sol` ~196** — `uint256 faceWei = PriceLookupLib.priceForLevel(L) * n;`
   → `... * n / 4;` (coupled valuation half: per-entry face = whole-ticket price / 4).
3. **`DegenerusGameMintStreakUtils.sol` ~207** — ticket-leg floor
   `if (ticketWei < oneTicketWei) ticketWei = oneTicketWei;` → floor at one ENTRY:
   `if (ticketWei < oneTicketWei / 4) ticketWei = oneTicketWei / 4;` (the preview-safety clamp
   `ticketWei > totalBudget` below it is unchanged — still underflow-safe).
4. **`DegenerusGameMintModule.sol` ~1134** — too-small-bundle revert
   `if (totalBudget < oneTicketWei) revert E();` → relax to one entry price:
   `if (totalBudget < oneTicketWei / 4) revert E();`.
5. **NatSpec / comments + rename.** `oneTicketWei` STAYS named (it is the whole-ticket price per §10.6
   KEEP; the `/4` derives one entry inline). Rename across contracts + interfaces + Vault lockstep:
   - `sellFarFutureTickets → sellFarFutureEntries` (Game external, MintModule impl, IDegenerusGameModules,
     IGamePlayer in Vault, Vault forwarder call).
   - `previewSellFarFutureTickets → previewSellFarFutureEntries` (Game external, MintModule impl,
     IDegenerusGame, IDegenerusGameModules). Game forwards `msg.data` → the module selector must match;
     both rename so the selector stays self-consistent.
   - `_removeFarFutureTickets → _removeFarFutureEntries` (MintModule internal; decl + 1 call).
   - `gameSellFarFutureTickets → gameSellFarFutureEntries` (Vault owner wrapper; only referenced inside
     DegenerusVault.sol — no interface decl, no test, no off-chain caller).
   - `quantities` param NatSpec: "Whole far tickets to sell" → "Entries to sell (4 entries = 1 whole
     ticket)"; "too small to deliver even 1 whole ticket" → "even 1 entry"; the `@return totalFaceWei
     Sum of priceForLevel(L) * n` docs gain the `/4` per-entry note; "floored at 1 whole ticket" →
     "floored at 1 entry". (Comments describe what IS — no phase IDs, no history.)

**Granularity-agnostic (NO change — verified):**
- `_purchaseFor` mint-qty (`qty = ticketWei * 4 * QTY_SCALE / oneTicketWei`) — a ticketWei↔scaled-units
  ratio; with ticketWei floored at `oneTicketWei/4` it yields `qty = QTY_SCALE = 1 entry` (entry-granular
  purchase path already supported).
- `_quoteFarFutureFlipSplit` (`targetFlip = targetEth * PRICE_COIN_UNIT / priceWei`) — an ETH↔FLIP price
  ratio, independent of granularity.
- `_removeFarFutureEntries` / `_queueEntries` already take `entries` directly.
- The far-future jackpot samplers, `_tqFarFutureKey`, `TICKET_FAR_FUTURE_BIT` / `TICKET_SLOT_BIT` flag
  constants — KEEP (480 ruling); membership `<=> packed != 0` is preserved.
