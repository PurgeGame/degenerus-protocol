**D-256-CANCEL-QUEUED-01 verdict (recorded inline per plan-body requirement):**

The branch at `contracts/GNRUS.sol:382-391` (cancellation path: `currentSlate[slot] == 0 AND pendingEditSet[slot] == 1 AND recipient == 0`) is **structurally unreachable** under current invariants:

- `pendingEditSet` bit `i` is SET only inside the queue branch (L405): `pendingEditSet = pendingEditSet | slotMask`. The queue branch fires only when `current != address(0)` (L399).
- `pendingEditSet` bit `i` is CLEARED in three places: (a) the cancellation path itself L388, (b) the start-of-pickCharity flush L630 (`pendingEditSet = 0`), (c) the cancellation path again.
- Bit `i` cannot be set without `currentSlate[i]` being non-zero at the moment of the set.
- After flush at L612-630, `pendingEditSet = 0` AND `currentSlate[i]` reflects the flushed value (which may be `address(0)` if a remove was applied).
- Therefore the state `(currentSlate[slot] == 0 AND pendingEditSet[slot] == 1)` cannot be reached from any sequence of external calls.

**Disposition:** Defensive guard, no test path. Plan 03 does NOT add a test driving this branch (would require contract surgery to construct). Phase 257 audit will cite this verdict and grep-prove unreachability as a SAFE row in AUDIT-02.

**D-256-GAS-01 worst-case derivation (recorded inline per `feedback_gas_worst_case.md` "derive theoretical worst case FIRST"):**

`pickCharity` worst case (full slate via 20 instant-apply pre-fill + 20 votes one-per-slot, all weighted; pendingEditSet pre-flush is empty in this scenario — the gas test exercises the winner-loop hot path without paying flush cost, which is the realistic per-level transition shape):

- Winner loop, 20 iterations, each: SLOAD `slotApproveWeight[level][i]` cold (2,100) + arith + branch (~100): 44,000

**Theoretical ceiling: ≈ 130,000 gas (no-flush path, winner-only) — but the test SHALL include a flush load to exercise the dual hot path.**
