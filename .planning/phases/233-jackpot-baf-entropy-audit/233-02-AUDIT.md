## D-07 No-Re-Use-Bias Sub-Section

**Problem:** Within a single `advanceGame` transaction, the same `rngWord` local flows to IM-10 (FF-promotion call at line 315-318), IM-11 (near-future prep at line 337 → internal fan-out via `_prepareFutureTickets`), and IM-12 (post-transition FF at line 407). Does re-use of the same seed across multiple consumers introduce bias?

**Per-consumer salt-space analysis:**

Each consumer eventually invokes one of two keccak formulations inside `MintModule.processFutureTicketBatch`:

1. `_rollRemainder(entropy, baseKey, rem)` at lines 443 and 489 — uses `EntropyLib.hash2(entropy, rollSalt)` at line 652, where `rollSalt == baseKey` (from the `_rollRemainder` call signature `(uint256 entropy, uint256 rollSalt, uint8 rem)`).
2. `_raritySymbolBatch(player, baseKey, processed, take, entropy)` at line 469 — uses `keccak256(abi.encode(baseKey, entropyWord, groupIdx))` at line 567-569 for the initial seed per LCG group; the LCG then steps deterministically for 16 iterations (lines 570-578).

**Critical domain-separating input: `baseKey`** computed at MintModule line 423-425:
```
uint256 baseKey = (uint256(lvl) << 224) |
    (idx << 192) |
    (uint256(uint160(player)) << 32);
```

- **Bits 255-232:** `lvl` (24 bits)
- **Bits 223-192:** `idx` (the queue index within the level)
- **Bits 191-32:** `player` address (160 bits)
- **Bits 31-0:** zero

**Cross-call bias analysis (IM-10 vs IM-11 vs IM-12 in the same advanceGame tx):**
- IM-10 FF-promotion call uses `ffLevel` (= purchaseLevel + 4 per line 303). Different bits 255-232 of baseKey → different keccak preimage.
- IM-11 near-future prep uses `inJackpot ? lvl : purchaseLevel` per line 338. Inside `_prepareFutureTickets` at lines 1412-1413, `startLevel = lvl + 1; endLevel = lvl + 4;` — the internal fan-out iterates over four distinct level values (lvl+1..lvl+4). Each iteration produces a distinct baseKey → distinct keccak preimage.
- IM-12 post-transition FF uses `nextLevel = purchaseLevel + 1` per line 402. Different again.
- Within a single call, each queue index `idx` (queue position) further separates per-ticket: bits 223-192 of baseKey change per player entry.
- Each distinct player address in the queue further separates: bits 191-32 of baseKey change per ticket-holder.

Therefore even though the `entropy` input to `hash2(entropy, rollSalt)` and `keccak256(abi.encode(baseKey, entropyWord, groupIdx))` is identical across IM-10/11/12 within the same advanceGame transaction, the other keccak inputs (baseKey composed from lvl + idx + player) are distinct at every (call-site × queue-index × player) triple. Keccak-256 collision resistance guarantees disjoint outputs for distinct preimages — no bias from shared `entropy`.

**Cross-consumer bias within a single call (same player, same idx):**
- `_rollRemainder` at line 443 (pre-roll): `hash2(entropy, baseKey)` → `rollEntropy % TICKET_SCALE < rem`
- `_rollRemainder` at line 489 (post-roll): same `entropy`, same `baseKey`, SAME `rem` possibly differing → if the two calls at lines 443 and 489 both fire on the same ticket entry with the same `baseKey` and same `entropy`, `hash2(entropy, baseKey)` produces the SAME value → `rollEntropy % TICKET_SCALE` evaluates to the same result → idempotent with respect to the original roll outcome.

Examining the code flow: the line 443 path is gated by `if (owed == 0 && rem != 0)` at line 429-442 — fires when the queue entry has only a fractional remainder. The line 489 path is gated by `if (remainingOwed == 0 && rem != 0)` — fires after main `_raritySymbolBatch` has consumed all whole tickets. In practice the two call sites CANNOT both fire on the same queue entry during the same `processFutureTicketBatch` invocation (the line 443 path returns-continues at line 450 or sets `rem = 0; owed = 1` before proceeding past line 468; if it takes the continue path, line 489 is unreachable for that entry; if it sets rem=0 before reaching line 489, the `rem != 0` guard at line 488 fails).

But even hypothetically: if both did fire with the same inputs, they'd compute the same hash and make the same probabilistic decision — no bias, just redundancy. No finding.

- `_raritySymbolBatch` at line 469 uses a DIFFERENT keccak composition: `keccak256(abi.encode(baseKey, entropyWord, groupIdx))` with the 3-argument form. This is cryptographically independent from `hash2(entropy, baseKey)` (different preimage length: 96 bytes vs 64 bytes) — keccak-256 collision resistance guarantees disjoint outputs. No bias between `_rollRemainder` output and `_raritySymbolBatch` output sharing the same `entropy`.

**Conclusion:** No re-use bias. Domain separation is achieved by (a) baseKey lvl/idx/player composition making cross-call preimages distinct, (b) keccak formulation differences (2-input hash2 vs 3-input abi.encode) making cross-consumer preimages disjoint-by-length.

**Cross-formulation collision analysis (addendum page 2 re-verification):**

- `hash2(a, b)` hashes a 64-byte preimage `[bytes32 a || bytes32 b]`.

- Preimage length differs (64 vs 96) → preimage bit pattern differs → keccak-256 collision resistance (2^128 bits security against collisions; 2^256 against preimage) guarantees disjoint outputs.
