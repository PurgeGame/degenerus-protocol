## High-Risk Patterns Analyzed

### Sentinel Type-Domain Separation

`uint16 private constant BAF_TRAIT_SENTINEL = 420;` declared at line 136 carries three structural guarantees that prevent any on-chain collision with a real trait ID:

1. **Type width:** `uint16` is strictly wider than `uint8`. `type(uint8).max = 255 < 420 <= type(uint16).max = 65535`. Any attempt to assign 420 to a `uint8` variable is rejected at compile time (Solidity 0.8.x implicit narrowing is forbidden) or truncates at runtime if an explicit cast is used — either way 420 cannot appear as a `uint8` value.
2. **Domain separation:** Real trait IDs are 0-255 by the existing `uint8[4] memory traits` bucket convention throughout `JackpotBucketLib` (`getRandomTraits`, `packWinningTraits`, `unpackWinningTraits`). 420 is above the real-trait ceiling by construction. The event field was widened to `uint16` solely to carry the sentinel — no other uint16 values are assigned to it at this time.
3. **Visibility:** `private` ensures the symbol is not exposed in the `IDegenerusGameJackpotModule` interface; external callers have no access to `BAF_TRAIT_SENTINEL` and would have to hard-code the literal 420 to reference it. This keeps the sentinel an internal implementation detail rather than a protocol-level value.

### Emit Argument Correctness at All Four BAF Sites

Each of the four emit sites at lines 2002 / 2014 / 2034 / 2038 passes `BAF_TRAIT_SENTINEL` (the compile-time symbol, not a numeric literal `420`) as the third argument. The symbol resolves to `uint16 420` at compile time; no intermediate variable mutation occurs between the constant declaration and the emit site. Each emit sits on a straight-line control-flow path after its respective branch guard:

- Line 2002: inside `if (amount >= largeWinnerThreshold)` (line 1990), after `_addClaimableEth` returns (lines 1996-2000). Emit fires unconditionally after the `claimableDelta += cd` state update.
- Line 2014: inside the inner `if (lootboxPortion <= LOOTBOX_CLAIM_THRESHOLD)` branch (line 2006), after `_awardJackpotTickets` returns new rngWord (lines 2008-2013). Emit fires unconditionally within the small-lootbox path.
- Line 2034: inside `else if (i % 2 == 0)` small-winner even-idx branch (line 2026), after `_addClaimableEth` returns (lines 2028-2032). Emit fires unconditionally after `claimableDelta += cd`.
- Line 2038: inside the `else` small-winner odd-idx branch (line 2035), after `_awardJackpotTickets` returns new rngWord (line 2037). Emit fires unconditionally.

No conditional `traitId` assignment exists anywhere in `runBafJackpot`; the only possible value at all four sites is `BAF_TRAIT_SENTINEL`. Pre-fix diff (via `git show 104b5d42 -- contracts/modules/DegenerusGameJackpotModule.sol`) shows the four sites were previously literal `0` in the same argument position — a pure `0 -> BAF_TRAIT_SENTINEL` substitution with no surrounding state-mutation addition/removal and no CEI reordering.

### Domain-Collision Sweep — winningTraitsPacked / bonusTraitsPacked

Both identifiers are always populated from `_rollWinningTraits(randWord, bool)` at line 1865-1875, which returns a `uint32 packed` value produced by `JackpotBucketLib.packWinningTraits(uint8[4] memory traits)`. The input array is always declared `uint8[4]`, so the packed value is 4 × 8 bits of pure `uint8` data. No code path injects a `uint16` or wider value into these identifiers.

Downstream unpacking via `JackpotBucketLib.unpackWinningTraits(uint32 packed) -> uint8[4]` narrows back to `uint8` at every consumer site (grep hits at lines 278, 444-445, 893-894, 1082-1083, 1722-1723). Every per-element trait read produces a `uint8` value; 420 is unrepresentable by the output type.

The BAF sentinel 420 thus CANNOT appear in either identifier by type construction. The sentinel is injected directly into event emit-argument slots (widened to `uint16` at the event decl) and never touches storage or the trait-bucket packed format.

### Domain-Collision Sweep — On-Chain Readers of Event traitId Field

`emit` statements produce log topics and data bytes that are consumed by off-chain observers (indexers, UI subgraphs, The Graph) — they are never read back by EVM execution within the same transaction or any subsequent transaction. Events in Solidity do not create a re-readable on-chain value for the EVM.

Grep across `contracts/` for any pattern that could inadvertently read an event field yields zero matches: there is no `abi.decode(event_data, ...)` usage, no pre-computed event-topic memory reads (e.g. no `EXTCODECOPY`-based log scraping — not that it would be possible within the same block anyway), and no state variable that stores emitted `traitId` values.

Therefore the 420 sentinel in `JackpotEthWin.traitId` / `JackpotTicketWin.traitId` is observationally invisible to every on-chain branch.

### Event-Signature Hash Change (Off-Chain Only)

Solidity ABI event-signature hash is computed as `keccak256(canonical_signature_string)` where the canonical signature includes the event name and the canonical param types. Widening `uint8 traitId -> uint16 traitId` changes the canonical type string from `uint8` to `uint16`, which changes the keccak hash (the value of `topic0` for non-anonymous events).

Off-chain ABI consumers (indexers, UI, subgraphs) must regenerate their ABI to decode post-fix events. The v28.0 Phase 227 precedent (DCM-02 DecimatorClaimed / TerminalDecimatorClaimed indexer-compat OBSERVATION) established that such event-decl drifts are indexer-compat OBSERVATIONS rather than on-chain findings.

EVM topic encoding itself is 32-byte left-padded regardless of declared width (per `abi.encode` rules for indexed parameters), so on-chain emission of the log record is byte-identical to what the pre-fix decl would have produced for the same argument value. Only the off-chain decoding side sees the difference.

Per the 104b5d42 commit message ("pre-launch so no live indexers"), no production decoders are affected. The observation is recorded as two SAFE-INFO Finding Candidate: Y rows (one per widened event decl) for Phase 236 FIND-01 / FIND-02 to route either into the on-chain FINDINGS-v29.0.md as INFO or into KNOWN-ISSUES.md as a design decision — the routing decision belongs to Phase 236.
