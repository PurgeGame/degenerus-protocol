## Summary

Phase 319.1 adds an `address fundingSource` field to the `Sub` record so one funded wallet (S) can cover BOTH the ETH `_poolOf` auto-buy draw AND the BURNIE `burnForKeeper` charge for a player M's several subscriber addresses. Default `fundingSource == address(0)` resolves to self, making every untouched path byte-identical to the shipped 317 behavior. The field does not fit the 19 free bytes of the single-slot `Sub` standalone (a 20-byte address needs 20 bytes); it requires collapsing the two standalone bools `drainGameCreditFirst` + `useTickets` into the existing `flags` byte to free a 20-byte run. The result is still ONE slot, so this is a layout *break* (slot offsets shift) but not a slot-*count* growth — and pre-launch redeploy-fresh makes the break free.

The contract is small (789 lines, single file, no inheritance, no external libraries), the routing surface is exactly three points (one ETH draw + two `burnForKeeper` sites), and the authorization is reuse of the existing game `operatorApprovals[owner][operator]` mapping. The whole phase is a single batched USER-APPROVED `contracts/AfKing.sol` diff plus a test-mirror sync (the Foundry tests pack/unpack `Sub` offsets via `vm.store` with hardcoded byte offsets — those constants drift when the bools collapse into `flags`).

**The single most load-bearing landmine:** `AfKing.sol:610` ALREADY performs a per-draw operator-approval check (`isOperatorApproved(player, address(this))`, PlayerSkipped reason 5). The OPENE-04 spec says the cross-account funding auth is checked "at subscribe + renewal ONLY (never per-draw)". The planner MUST reconcile the NEW cross-account check (`isOperatorApproved(fundingSource, M)`) against this EXISTING per-draw self-keeper check — they are different relationships (subscriber→keeper vs. source→subscriber) and the new one must not be naively folded into line 610 or it inherits the per-draw cost the spec forbids.

**Primary recommendation:** Pack `drainGameCreditFirst` + `useTickets` (+ keep `windowPaid`) into `flags` as bit constants, append `address fundingSource` to fill offsets 13..32 of the single `Sub` slot, route the three spend points through a `_resolveFundingSource(player)` helper (returns `player` when `fundingSource == 0`), and gate the cross-account case on `isOperatorApproved(fundingSource, player)` at subscribe + the day-31 renewal branch only. Sync the test-mirror offset constants. Document the BURNIE blast-radius caveat in the `fundingSource` setter NatSpec.

## OPENE-01 — `Sub.fundingSource` field + repack (the storage byte math)

### Current `Sub` struct (VERIFIED `AfKing.sol:80-88`, doc at `:69-79`)

```solidity
struct Sub {
    uint8  dailyQuantity;        // offset 0,  1 byte  — 0 = paused/never-subscribed
    bool   drainGameCreditFirst; // offset 1,  1 byte
    bool   useTickets;           // offset 2,  1 byte  — default false = lootbox mode
    uint32 lastSweptDay;         // offset 3,  4 bytes (3..6)
    uint32 paidThroughDay;       // offset 7,  4 bytes (7..10)
    uint8  reinvestPct;          // offset 11, 1 byte
    uint8  flags;                // offset 12, 1 byte  — bit 0 = FLAG_WINDOW_PAID
}
```

### Byte math (VERIFIED against the field widths above) [VERIFIED: source struct]

- **Used today:** offsets 0..12 = **13 bytes**.
- **Free today:** offsets 13..31 = **19 bytes**.
- **An `address` is 20 bytes.** 20 > 19 → **does NOT fit standalone.** (This is the CORRECTION the ROADMAP/CRANK-plan flag: the earlier "fits the free bytes, no new slot" note was wrong.) [VERIFIED: `PLAN-CRANK-DO-WORK-INCENTIVE.md:170` correction note]
- **Collapse the two standalone bools** `drainGameCreditFirst` (offset 1) + `useTickets` (offset 2) into the existing `flags` byte (offset 12). That removes 2 used bytes → **used becomes 11 bytes, free becomes 21 bytes** ≥ 20. The 20-byte `address fundingSource` now fits in ONE slot. [VERIFIED: arithmetic on source widths]

### Proposed repacked layout (one slot, slot-count unchanged)

```
offset 0   uint8   dailyQuantity
offset 1   uint32  lastSweptDay        (1..4)
offset 5   uint32  paidThroughDay      (5..8)
offset 9   uint8   reinvestPct
offset 10  uint8   flags               (bit0 windowPaid, bit1 drainGameCreditFirst, bit2 useTickets)
offset 11  uint8   (free pad, 1 byte)
offset 12  address fundingSource       (12..31, 20 bytes)
```
Exact offsets are Claude's discretion (the compiler packs declaration-ordered fields); the only HARD constraint is `address fundingSource` lands in the same slot 1 and the slot *count* stays at 5 (slots 0-4 as documented at `AfKing.sol:189-213`). [ASSUMED — exact offset assignment is one valid packing; the planner should re-derive via `forge inspect` and not hardcode offsets in this doc]

### `flags` bit constants

Today: `FLAG_WINDOW_PAID = 1` (`AfKing.sol:238`). Add two more bit constants (e.g. `FLAG_DRAIN_FIRST = 2`, `FLAG_USE_TICKETS = 4`), mirroring the existing `&= ~FLAG` / `|= FLAG` idiom already used at `:392`, `:398`, `:581`, `:593`, `:603`, `:682`. Every read of `s.drainGameCreditFirst` / `s.useTickets` becomes a `(s.flags & FLAG_x) != 0` test; every write becomes a masked set/clear. [VERIFIED: existing flag idiom at the cited lines]

**Read/write sites that must convert from struct-field to flag-bit** (grep-verified):
- `drainGameCreditFirst`: read `:646` (`if (!sub.drainGameCreditFirst)`), write `:381` (subscribe), `:433` (`setDrainGameCreditFirst`); echoed in events at `:387,418,426,434,442,451`.
- `useTickets`: read `:634` (`if (!sub.useTickets && ...)`), write `:382` (subscribe), `:441` (`setMode`); echoed in events at `:387,418,426,434,442,451`.

**Event signature note:** `SubscriptionUpdated(address,uint8,bool,bool,uint8)` (`AfKing.sol:159-165`) passes `drainGameCreditFirst` + `useTickets` as `bool` ABI params. Packing them into `flags` storage does NOT require changing the event ABI — the emit sites can pass `(s.flags & FLAG_DRAIN_FIRST) != 0` as the bool argument. Keeping the event ABI stable avoids churning the test event-decoders (`AfKingFundingWaterfall.t.sol:55-57` etc.). [VERIFIED: event def + emit sites]

### The setter (mirror the existing `NotSubscribed`-gated mutators)

The four existing in-place setters are the template — all share the identical guard `if (_subscriberIndex[msg.sender] == 0) revert NotSubscribed();` then mutate `_subOf[msg.sender]` then emit:
- `setDailyQuantity(uint8)` `AfKing.sol:410-427`
- `setDrainGameCreditFirst(bool)` `:430-435`
- `setMode(bool)` `:438-443`
- `setReinvestPct(uint8)` `:446-452`

The new `setFundingSource(address source)` MUST mirror this exactly: `NotSubscribed` guard, write `_subOf[msg.sender].fundingSource = source`, emit. **Caller-scoped:** only `msg.sender` can set their own `fundingSource` (matching every other setter), so M cannot set someone else's source. [VERIFIED: setter pattern]

> NOTE: setting `fundingSource = S` is M declaring "draw from S." Whether that declaration is *honored* depends on `isOperatorApproved(S, M)` checked at subscribe/renewal (OPENE-04). The setter itself need not check approval (the sweep does) — but the planner may choose to check at the setter too for fail-fast UX. The spec's "checked at subscribe + renewal ONLY" is about the *per-draw* prohibition, not a ban on a setter-time check. Flag as a design choice for the planner.

## OPENE-02 — ETH draw routes to `_poolOf[fundingSource]`

### The ETH draw + debit (VERIFIED `AfKing.sol:643-695`)

The funding-skip check and the debit are the two `_poolOf[player]` touches in the sweep:
- **Read (the skip gate):** `:672` `if (_poolOf[player] < msgValue) {` — inside the InsufficientPool two-tier skip-kill block `:665-689`.
- **Debit (CEI):** `:693-695` `unchecked { _poolOf[player] -= msgValue; }`.

Both read the SAME `_poolOf` slot 0 keyed on `player`. Routing both to `_poolOf[fundingSource]` (resolved to `player` when `fundingSource == 0`) is **gas-neutral**: it is the same single mapping SLOAD/SSTORE on a different key — no additional storage read is introduced. [VERIFIED: both touches are single-key `_poolOf` ops]

The resolved source must be computed ONCE per player iteration (e.g. `address src = sub.fundingSource == address(0) ? player : sub.fundingSource;`) and used at BOTH the `:672` read and the `:694` debit so they stay consistent. [VERIFIED: the two sites are 22 lines apart, same iteration]

### The Vault/sDGNRS exemption MUST stay subscriber-keyed (the spoof landmine)

The two-tier skip-kill exemption is at `AfKing.sol:672-689`:
```solidity
if (_poolOf[player] < msgValue) {
    if (player == ContractAddresses.VAULT || player == ContractAddresses.SDGNRS) {  // :673
        emit PlayerSkipped(player, 3);  // transient skip, stays in set
        ... continue;
    }
    // else: NORMAL sub CANCELLED via swap-pop
    sub.dailyQuantity = 0; ... _removeFromSet(player); ...
}
```
The exemption condition at `:673` keys on `player` (the iterated SUBSCRIBER identity, un-spoofable — it is `_subscribers[cursor]`). **This MUST NOT change to key on `fundingSource`.** If it keyed on the source, a NORMAL sub could set `fundingSource = VAULT` (or sDGNRS) and inherit the never-cancel exemption, defeating the two-tier skip-kill that protects the active set from fragile claimable-only subs. The pool *read* `:672` routes to the source; the *exemption branch* `:673` stays on `player`. These are deliberately different keys. [VERIFIED: `:672` reads `_poolOf[player]`, `:673` compares `player` to the pinned constants]

> This is the exact surface the Phase 320 terminal audit is charged to attack ("can the `fundingSource` redirect spoof the Vault/sDGNRS subscriber-identity skip-kill exemption?", ROADMAP:211). The answer must be "no — exemption keys on `player`."

### `msgValue == 0` (claimable-only) edge

When the waterfall picks `Claimable` (`:651-653`), `msgValue == 0`, so `_poolOf[src] < 0` is false and the debit subtracts 0 — the source pool is untouched. Routing to the source is harmless in this path (no pool spend). [VERIFIED: waterfall `:644-663`, `msgValue=0` in the Claimable branch]

> Note: `drainGameCreditFirst` and `reinvestPct` read `claimableWinningsOf(player)` (`:625,:650`) — the CLAIMABLE side stays keyed on the SUBSCRIBER (M), not the source. Only the ETH `_poolOf` side moves to the source. The spec scopes OPEN-E to "the ETH `_poolOf` auto-buy draw" + the two BURNIE sites — claimable is the subscriber's own winnings and is correctly left subscriber-keyed. [VERIFIED: claimable reads use `player`]

## OPENE-04 — Operator-approval authorization (the hardest part)

### The approval mapping + direction (VERIFIED `DegenerusGame.sol:426-441`, `DegenerusGameStorage.sol:912`)
```solidity
// DegenerusGameStorage.sol:912
mapping(address => mapping(address => bool)) internal operatorApprovals;

// DegenerusGame.sol:426
function setOperatorApproval(address operator, bool approved) external {
    if (operator == address(0)) revert E();
    operatorApprovals[msg.sender][operator] = approved;  // msg.sender = owner, grants operator
}
// DegenerusGame.sol:436
function isOperatorApproved(address owner, address operator) external view returns (bool) {
    return operatorApprovals[owner][operator];  // operatorApprovals[owner][operator]
}
```
**Direction confirmed: `operatorApprovals[OWNER][OPERATOR]`.** The owner (money-holder) calls `setOperatorApproval(operator, true)` to grant. So for OPEN-E: **source S** (the `_poolOf[S]` / BURNIE-holder) calls `setOperatorApproval(M, true)`; the keeper checks `isOperatorApproved(S, M)` = `operatorApprovals[S][M]`. M cannot unilaterally point at S. [VERIFIED: mapping + both fns; matches `PLAN-CRANK-DO-WORK-INCENTIVE.md:170` "money-holder grants to the spender"]

### What is "M"? (resolve before planning)
M is the SUBSCRIBER (the iterated `player` / the `subscriber` in `subscribe`). S is `fundingSource`. The check honors "subscriber M may draw from source S" iff S approved M. This is the natural reading of OPENE-04 ("source S calls `setOperatorApproval(M, true)`; keeper requires `isOperatorApproved(S, M)` to honor `fundingSource = S`"). [VERIFIED: requirement text + mapping direction]

### THE LANDMINE — the EXISTING per-draw operator check at `AfKing.sol:610`
```solidity
// (3) NotApproved — fires EVERY sweep iteration (per-draw):
if (!IGame(ContractAddresses.GAME).isOperatorApproved(player, address(this))) {  // :610
    emit PlayerSkipped(player, 5);  // skip reason 5
    ... continue;
}
```
This is a DIFFERENT relationship: it checks `isOperatorApproved(player, address(this))` = "did the SUBSCRIBER (player/M) approve the KEEPER CONTRACT (`address(this)`)?". It is the keeper's own draw authorization, and it ALREADY runs per-draw (every sweep iteration). [VERIFIED: `:610`]

**Reconciliation the planner MUST handle:**
1. The NEW OPEN-E check is `isOperatorApproved(fundingSource, player)` = "did S approve M?" — a SEPARATE pair from `(player, address(this))`.
2. OPENE-04 says the NEW check is at subscribe + renewal ONLY, never per-draw. So the planner must NOT add the `(fundingSource, player)` read into the per-draw loop body (e.g. not next to `:610`). It belongs in `subscribe()` (alongside the existing SUB-02 check at `:358-364`) and in the day-31 renewal branch (`:577-606`, which fires only when the window lapses — NOT every sweep).
3. The existing `:610` check is unrelated to OPEN-E and should be LEFT AS-IS (do not "optimize" it away — it is a security-floor guard per GAS-05; the CRANK plan's "dropped per-draw operator check" intent was NOT applied to the shipped 317 contract). Removing or altering `:610` is out of scope for 319.1.
4. **Subtle:** when `fundingSource == 0` (self), there is NO cross-account relationship and the OPEN-E approval check must be SKIPPED entirely (S == M, no third party) — otherwise self-funding subs would need to self-approve, breaking the default-self behavior-identity. The check is `if (fundingSource != 0 && !isOperatorApproved(fundingSource, M)) { /* refuse to honor the source */ }`. [VERIFIED: default-self requirement + mapping semantics]

### Where the checks land (subscribe + renewal only)
- **At `subscribe()`** (`:348-400`): after resolving `subscriber` and BEFORE honoring a non-zero `fundingSource`, verify `isOperatorApproved(fundingSource, subscriber)`. On failure, the planner chooses: revert, or fall back to self-funding (`fundingSource` effectively ignored). The spec says "on revoke the draw reverts to self / auto-pauses" — for subscribe the natural choice is revert (consistent with `NotApproved` at `:362`) OR store-but-don't-honor. Flag as a design decision.
- **At the day-31 renewal branch** (`:577-606`): when the window lapses and the sub renews, re-verify `isOperatorApproved(fundingSource, player)`. On revoke: per spec, the source is no longer honored → fall back to self (draw `_poolOf[player]` / burn `player`) OR auto-pause. This bounds post-revoke drain to ≤1 renewal window (the source can be drawn at most until the next renewal re-checks). [VERIFIED: renewal branch location; the "≤1 window" bound follows from checking only at renewal]

### Revoke → drain bound (the security property)
Because the source approval is checked only at subscribe + each renewal (every `WINDOW_DAYS`=30 days), after S revokes `setOperatorApproval(M, false)`, M can still draw S's pool for the daily auto-buys until the NEXT renewal (≤30 days / ≤1 window), at which point the renewal check fails and the source falls back to self / auto-pauses. This is the explicitly-accepted tradeoff (per-draw checks would tighten it to ≤1 day but cost ~2.6k gas/draw). [VERIFIED: requirement rationale `PLAN-CRANK-DO-WORK-INCENTIVE.md:170`]

> A tighter variant the planner could consider: per-draw the ETH pool spend is naturally bounded by S's pool balance and M's sub config (cost × qty), and S can drain their own pool via `withdraw()` at any time to cut M off immediately. So the "≤1 renewal window" bound is the *approval-state* bound; the *economic* bound (S empties the pool) is immediate. Worth noting in the caveat. [VERIFIED: `withdraw()` at `:307-320` lets S pull their pool anytime]

### The BURNIE blast-radius caveat (MUST be documented)
The same `operatorApprovals[S][M]` approval that authorizes drawing S's ETH `_poolOf` ALSO authorizes `burnForKeeper(S, cost)` — which burns S's GENERAL-WALLET BURNIE balance + pending coinflip (not a pre-funded escrow). This is SHARPER than the ETH-pool case (where S only risks the ETH they deposited into `_poolOf[S]`): the BURNIE charge reaches S's entire liquid BURNIE position. The caveat must appear in the `setFundingSource` / `subscribe` NatSpec. The named alternative (NOT built this phase) is a dedicated `allowBurnieFunding[S][M]` opt-in that separates ETH-pool authorization from BURNIE-burn authorization. [VERIFIED: requirement OPENE-04 + `burnForKeeper` NatSpec `:42-48`]

## Common Pitfalls

### Pitfall 1: Spoofing the Vault/sDGNRS exemption via `fundingSource`
**What goes wrong:** Keying the InsufficientPool exemption (`:673`) on the resolved source instead of `player` lets any NORMAL sub set `fundingSource = VAULT` to inherit the never-cancel exemption.
**Why it happens:** The pool READ at `:672` legitimately moves to the source; a careless edit moves the exemption COMPARISON at `:673` with it.
**How to avoid:** Read `_poolOf[src]` at `:672` but compare `player` (NOT `src`) to `ContractAddresses.VAULT/SDGNRS` at `:673`. The exemption keys on the un-spoofable iterated subscriber.
**Warning signs:** A grep showing `fundingSource`/`src` inside the `:673` comparison; a test where a normal sub with `fundingSource=VAULT` survives a funding skip.

### Pitfall 2: Adding the OPEN-E approval check per-draw
**What goes wrong:** Folding `isOperatorApproved(fundingSource, player)` into the sweep loop body (near `:610`) re-introduces a per-draw cross-contract STATICCALL the spec explicitly forbids, and tightens the bound in a way that diverges from the locked "≤1 renewal window" design.
**Why it happens:** `:610` already has a per-draw operator check, so it looks like the natural home.
**How to avoid:** Place the OPEN-E check in `subscribe()` and the day-31 renewal branch (`:577-606`) ONLY. Leave `:610` (a different relationship: subscriber→keeper) untouched.
**Warning signs:** Two `isOperatorApproved` calls in the per-iteration loop body; gas regression on the per-draw hot path.

### Pitfall 3: Breaking default-self equivalence
**What goes wrong:** Requiring an approval check even when `fundingSource == 0` makes self-funding subs need a self-approval, breaking behavior-identity with pre-OPEN-E.
**Why it happens:** Forgetting the `fundingSource != 0` short-circuit.
**How to avoid:** Every resolve + check is gated `fundingSource == 0 ? self-path : source-path`. When 0, NO approval read happens and the drawn identity equals today's exact identity.
**Warning signs:** A self-subscribe (player == msg.sender, no fundingSource) reverting `NotApproved`; the Vault/sDGNRS self-subscribes (SUB-09) failing.

### Pitfall 4: Test-mirror offset drift
**What goes wrong:** `vm.store`-based tests read/write the wrong `Sub` bytes after the repack, silently passing while testing nothing.
**Why it happens:** `OFF_LASTSWEPT=3`/`OFF_PAIDTHROUGH=7`/`OFF_FLAGS=12` are hardcoded for the OLD layout.
**How to avoid:** Re-derive from post-repack `forge inspect AfKing storage-layout` and update `AfKingFundingWaterfall.t.sol:42-49` (and audit the other AfKing test files for raw offsets).
**Warning signs:** Funding-waterfall tests passing with no change to assertions despite a layout shift; `forge inspect` offsets not matching the test constants.

### Pitfall 5: Event ABI churn
**What goes wrong:** Changing `SubscriptionUpdated`'s `bool drainGameCreditFirst, bool useTickets` params to packed-uint forces every test event-decoder to change.
**Why it happens:** Over-eager "the bools are gone from storage so the event should change too."
**How to avoid:** Keep the event ABI; pass `(s.flags & FLAG_x) != 0` as the bool arg. Storage packing is independent of event signature.
**Warning signs:** `SubscriptionUpdated` signature edited; test decoders failing to match the topic.

## Open Questions for Planner

1. **What identity is "M" in `isOperatorApproved(S, M)`?** Research concludes M = the subscriber (`player`/`subscriber`), because (a) the mapping is `[owner][operator]` with S the owner, and (b) the natural "S lets subscriber M draw" reading. But confirm: at `subscribe(player, ...)` the *caller* (`msg.sender`) may differ from `player` (the SUB-02 operator path at `:359-364`). Is the OPEN-E spender M the subscriber `player`, or the `msg.sender` who called subscribe? The locked text says "honor `fundingSource = S`" which is a property of the subscriber record → M = subscriber. Recommend: M = subscriber identity (consistent across subscribe + sweep where `msg.sender` is the cranker, not M).

2. **Subscribe-time approval failure: revert or store-but-ignore?** OPENE-04's "reverts to self / auto-pauses" is explicitly about the sweep/renewal path. For `subscribe()` with a non-zero `fundingSource` lacking approval: revert `NotApproved` (fail-fast, mirrors `:362`) vs. accept the subscription but ignore the unapproved source (draw self until approved). Recommend: revert at subscribe for clarity, since the user is actively configuring; fall-back-to-self at renewal (where the user isn't present). Planner decides.

3. **Should `setFundingSource` itself check approval, or only subscribe/renewal?** The spec says the per-draw prohibition; it does not forbid a setter-time check. A setter-time check is fail-fast UX but adds a STATICCALL to the setter (called rarely, so cheap). Recommend: check at subscribe/renewal (the spend points) and OPTIONALLY at the setter for UX; do NOT check per-draw. Planner decides whether the setter validates.

3b. **Does a standalone `setFundingSource` setter need an emit?** Every existing setter emits `SubscriptionUpdated` (which does not carry `fundingSource`). Either extend the event with a `fundingSource` field (ABI churn → test decoder updates) or add a small dedicated `FundingSourceUpdated(player, source)` event. Recommend a dedicated event to avoid churning `SubscriptionUpdated`'s ABI. Planner decides.

4. **Renewal-revoke behavior: fall-back-to-self vs. auto-pause?** OPENE-04 says "reverts to self / auto-pauses" (either). Fall-back-to-self keeps the sub alive drawing the subscriber's own pool/BURNIE; auto-pause cancels it. Recommend fall-back-to-self (less disruptive; the subscriber's own funds still flow) — but if the subscriber has no self-funding, that degrades to the existing InsufficientPool / shortfall path anyway. Planner picks the explicit branch.

5. **`forge inspect` re-derivation cadence:** confirm the planner runs `forge inspect AfKing storage-layout` AFTER the repack and feeds the exact offsets into the test-mirror update (don't trust this doc's illustrative offsets). This mirrors the 317-06 combined-inspect workflow.
