# Phase 319.1: Code Review Report

**Reviewed:** 2026-05-24
**Depth:** standard
**Files Reviewed:** 8
**Status:** issues_found

## Summary

Reviewed the OPEN-E "shared funding source" addition to the AfKing subscription
keeper (`contracts/AfKing.sol`) plus the two trivial `subscribe()` call-site
ripples in `DegenerusVault.sol` and `StakedDegenerusStonk.sol`, against the 5
test files that exercise the change. The diff under review is commit `42140ceb`
and its test follow-ups.

I went hard at the four security-critical surfaces called out in the phase
context — CEI integrity, resolved-source routing accidentally bypassing the two
landmines, flag-bit masking / `windowPaid` clobbering, and unauthorized spend of
another account's ETH/BURNIE. **All four are sound.** The two landmines hold:
(A) the VAULT/SDGNRS never-cancel exemption keys on the un-spoofable `player`
(AfKing.sol:725), not the resolved `src`; (B) the per-draw
`isOperatorApproved(player, address(this))` gate (AfKing.sol:656) and the new
cross-account `isOperatorApproved(fundingSource, subscriber)` read (AfKing.sol:396)
are correctly partitioned — the new read fires at `subscribe()` ONLY and is absent
from both the per-draw loop and the day-31 renewal branch. The funding-skip check
(`_poolOf[src] < msgValue`, line 724) and the CEI debit (`_poolOf[src] -= msgValue`,
line 746) both key on the same resolved `src`, so the unchecked debit cannot
underflow. The storage repack (31 bytes, single slot) is correct and the
flag-bit `|=` / `&= ~` masking preserves `FLAG_WINDOW_PAID` across the new
`FLAG_DRAIN_FIRST` / `FLAG_USE_TICKETS` writes (verified each mutator touches only
its own bit). The cross-account auth direction is correct (money-holder approves
spender), giving no privilege-escalation path. The Vault/sDGNRS call sites pass
`address(0)` (self-fund) and their interface declarations match the new 6-arg
signature verbatim.

No Critical findings. The two Warnings concern an observability gap on the new
cross-account binding and stale offset documentation inside security-critical test
helpers. The accepted-risk items (revoke-does-not-stop-active-sub; BURNIE
blast-radius) are documented in NatSpec and are NOT flagged as bugs per the phase
brief; they are noted under Info for completeness.

## Warnings

### WR-01: No event records the cross-account funding-source binding

**File:** `contracts/AfKing.sol:425` (and the `SubscriptionUpdated` event decl at :157-163)
**Issue:** `subscribe()` writes `s.fundingSource = fundingSource` (line 422) and then
emits `SubscriptionUpdated(subscriber, dailyQuantity, drainGameCreditFirst,
useTickets, reinvestPct)` — which does NOT include `fundingSource`. When a subscriber
M binds a cross-account source S (`subscribe(..., fundingSource = S)`), there is no
log topic carrying S. The keeper then proceeds to draw S's `_poolOf` ETH and burn
S's wallet BURNIE on every sweep day. Because the design is subscribe-time-auth-only
(a later `setOperatorApproval(M, false)` by S does NOT stop the active sub, per the
NatSpec at :362-370), S's only on-chain signal that it has become a standing funding
source for M is the absence of one — S must either inspect `subscriptionOf(M)` for
every M it ever approved, or watch `KeeperBurn` / pool-debit side effects after the
fact. For a real-money contract where the approval also authorizes burning S's
general-wallet BURNIE (the documented blast-radius), the binding event is the cheap
monitoring primitive that lets a source detect and respond to (by defunding) an
unexpected standing draw. This is a robustness/auditability gap, not an exploit.
**Fix:** Add `fundingSource` to the canonical subscription event (or emit a dedicated
one-shot binding event) so off-chain monitors and the source itself can react:
```solidity
// extend the event:
event SubscriptionUpdated(
    address indexed player,
    uint8 dailyQuantity,
    bool drainGameCreditFirst,
    bool useTickets,
    uint8 reinvestPct,
    address fundingSource   // NEW — index it if monitors filter by source
);
// emit with s.fundingSource at subscribe() (and in the setX mutators, which
// already re-emit the full state). Note: contracts are frozen at deploy, so if a
// new event is undesirable, at minimum document that off-chain monitoring of a
// source's standing bindings requires polling subscriptionOf() per approved sub.
```

### WR-02: Stale byte-offset documentation in security-critical test helper docstrings

**File:** `test/fuzz/AfKingConcurrency.t.sol:442` and `:455`; `test/fuzz/AfKingSubscription.t.sol:430`; `test/gas/SweepPerPlayerWorstCaseGas.t.sol:382`
**Issue:** Several test-helper docstrings still describe the PRE-OPEN-E packed
layout while the executable code uses the correctly re-derived constants. Concretely:
- `AfKingConcurrency._lastSweptDayOf` (:442) says "bytes 3..6" — the live offset is
  bytes 1..4 (`OFF_LASTSWEPT = 1`).
- `AfKingConcurrency._setWindow` (:455) says "paidThroughDay (bytes 7..10) and the
  windowPaid bit (byte 12)" — the live offsets are bytes 5..8 (`OFF_PAIDTHROUGH = 5`)
  and byte 10 (`OFF_FLAGS = 10`).
- `AfKingSubscription._forceRenewalDue` (:430) says "Clear lastSweptDay (bytes 3..6)
  and paidThroughDay (bytes 7..10)".
- `SweepPerPlayerWorstCaseGas._lastSweptDayOf` (:382) says "bytes 3..6".

These are comments only — every cited helper computes its mask/shift from the
`OFF_*` constants (which the file headers correctly re-derived to the post-repack
layout), so the tests are CORRECT and would fail loudly if the offsets were wrong.
But these are the slot-surgery helpers (`vm.store` into the packed `Sub` slot) that
the whole AfKing test suite's correctness rests on; a future maintainer who trusts
the docstring over the constant (e.g. hand-writing a new `vm.store` to "byte 12")
would silently clobber the wrong field. In a suite whose explicit job is to prove
the repack is byte-faithful, stale offset prose is a maintainability trap.
**Fix:** Update the four docstrings to the live offsets (lastSweptDay bytes 1..4;
paidThroughDay bytes 5..8; flags byte 10; fundingSource bytes 11..30), matching the
already-correct `OFF_*` constant comments at the top of each file. Example for
`AfKingConcurrency._setWindow`:
```solidity
/// @dev Pin `who`'s window: write paidThroughDay (bytes 5..8) and the windowPaid
///      bit (byte 10) of the packed Sub slot.
```

## Info

### IN-01: Funding-skip kill and day-31 lapse leave a stale `fundingSource` in the paused record

**File:** `contracts/AfKing.sol:733-736` (InsufficientPool kill) and `:638-640` (day-31 shortfall auto-pause)
**Issue:** Both auto-pause paths zero `dailyQuantity` and clear `FLAG_WINDOW_PAID`
but do NOT clear `fundingSource`; the `setDailyQuantity(0)` preserve-paid-window
branch (:457-459) likewise keeps the stale source. The record is removed from the
iterable set, so it is never re-processed by a sweep, and the ONLY way to re-enter
the set is `subscribe()`, which unconditionally overwrites `fundingSource` (:422)
after re-running the cross-account auth check (:393-399). So the stale value is
unreachable and harmless — there is no path that re-activates a paused record
without a fresh `subscribe()`. Flagged only as a note: the residue is dead data,
not an exploitable spoof (the LANDMINE-A test at
`AfKingFundingWaterfall.t.sol:416` confirms a killed spoofer does not inherit the
exemption). No fix required; if a future change ever re-activates a paused record
in place (rather than via `subscribe()`), the stale source would need re-validation.

### IN-02: Accepted-risk items present and documented (no action)

**File:** `contracts/AfKing.sol:362-370` (NatSpec)
**Issue:** Two by-design risks are present and explicitly documented, per the phase
brief these are NOT bugs: (1) revoke-does-not-stop-active-sub — the day-31 renewal
and per-draw paths trust the stored `fundingSource` and never re-check
`isOperatorApproved`, so a source that revokes the subscriber's game approval still
gets drawn/burned until it defunds (verified by
`AfKingSubscription.testRevokeDoesNotStopActiveSubButDefundDoes`); (2) BURNIE
blast-radius — the operator approval also authorizes the keeper to burn the source's
general-wallet BURNIE and pending coinflip stake (not just a pre-escrowed amount),
sharper than the ETH-pool escrow. Both are called out in NatSpec with the deferred
tighter alternative (`allowBurnieFunding[S][M]`) named. Recorded for the audit trail
only; no change requested.
**Fix:** None — documented design decision.

### IN-03: `drainGameCreditFirst` claimable read keys on `player`, ETH top-up on `src` (correct, non-obvious)

**File:** `contracts/AfKing.sol:702` vs `:746`
**Issue:** In the drain-first waterfall the claimable balance is read from
`claimableWinningsOf(player)` (:702) while the ETH top-up in Combined/DirectEth
mode debits `_poolOf[src]` (:746). On a cross-account sub this means the player's
OWN protocol claimable is spent first, and only the ETH shortfall comes from the
source's pool. This is the intended and correct split — claimable winnings are a
per-player protocol-internal credit consumed inside `batchPurchase` -> `_purchaseFor`
for `buyer == player` (verified in `DegenerusGame.batchPurchase` :1704 /
`_batchPurchaseUnit` :1729), and the source's operator approval authorizes the keeper
to fund the player's buy. The mode/`msgValue` selection and the `_poolOf[src]` skip
guard remain internally consistent. Recorded because the `player`-vs-`src` asymmetry
on adjacent lines is easy to misread as a routing bug; it is not.
**Fix:** None — behavior is correct. Optionally add a one-line inline comment at :702
noting the deliberate `player` (claimable) vs `src` (ETH) split for future readers.
