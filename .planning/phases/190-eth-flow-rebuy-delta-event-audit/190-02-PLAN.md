**DELTA-01: Auto-rebuy storage write safely overwritten by _setPrizePools**

Trace the exact write chain:
1. _consolidatePoolsAndRewardJackpots loads memFuture from storage (line 625: `uint256 memFuture = _getFuturePrizePool()`)
2. BAF self-call: runBafJackpot -> _addClaimableEth -> _processAutoRebuy -> _setFuturePrizePool(_getFuturePrizePool() + calc.ethSpent) at line 839
   This writes to `prizePoolsPacked` STORAGE during execution.
3. Control returns to _consolidatePoolsAndRewardJackpots. memFuture (which was loaded BEFORE the self-call) is used for all subsequent math.
4. At line 788: `_setPrizePools(uint128(memNext), uint128(memFuture))` overwrites prizePoolsPacked with the final memFuture value.

The key question: Does the old code's `memFuture += _getFuturePrizePool() - storageBaseFuture` (the removed line) produce a DIFFERENT final memFuture than the new code's simple `memFuture -= claimed`?

OLD code after BAF:
  memFuture = memFuture_pre_baf - bafPoolWei + (bafPoolWei - netSpend) + lootboxToFuture
  // Then at the removed reconciliation line:
  memFuture += _getFuturePrizePool() - storageBaseFuture
  // _getFuturePrizePool() reads storage which now includes auto-rebuy writes
  // storageBaseFuture was the snapshot before BAF
  // So this adds the delta = (auto-rebuy writes during BAF)

NEW code after BAF:
  memFuture = memFuture_pre_baf - claimed
  // No reconciliation. The auto-rebuy writes to storage are overwritten by _setPrizePools.

Show that OLD memFuture at _setPrizePools = NEW memFuture at _setPrizePools.

From Plan 01's algebra: old memFuture change from BAF = -claimableDelta (same as new).
The ONLY difference is the old code ALSO added the rebuy delta. But the new code does NOT add the rebuy delta. This means:
  OLD final memFuture = initial - claimableDelta + rebuyDelta
  NEW final memFuture = initial - claimableDelta

These are DIFFERENT if rebuyDelta != 0. The commit message says the old pattern "double-counts auto-rebuy writes" -- explain WHY the old rebuy delta addition was the bug (or was redundant because the new simplified deduction already accounts for it).

The critical insight: In the OLD code, the multi-step deduction was:
  memFuture -= bafPoolWei (deduct full pool)
  memFuture += (bafPoolWei - netSpend) (add back refund)
  memFuture += lootboxToFuture (add back lootbox)
This nets to: memFuture -= netSpend - lootboxToFuture = memFuture -= (claimableDelta + lootboxToFuture) - lootboxToFuture = memFuture -= claimableDelta

BUT in the old code, netSpend = poolWei - refund. And the ETH that went to auto-rebuy tickets was counted in claimableDelta? No -- in the old runBafJackpot, claimableDelta only accumulated _addClaimableEth return values. For auto-rebuy, _addClaimableEth returns calc.reserved (take-profit), NOT weiAmount. So the ticket conversion ETH (calc.ethSpent) was NOT in claimableDelta.

Where did the ticket ETH go in the old accounting? It was part of netSpend (spent from pool) but NOT in lootboxToFuture (which only tracked _awardJackpotTickets calls, not auto-rebuy). So the old deduction over-deducted by calc.ethSpent. The rebuy delta reconciliation line added it back.

In the NEW code: claimableDelta only includes calc.reserved (take-profit). The ticket ETH stays in memFuture because memFuture is only reduced by claimableDelta. But the ticket ETH was written to storage by _processAutoRebuy, effectively double-counting it (once in memFuture, once in storage). The _setPrizePools overwrite resolves this by clobbering the storage write with memFuture.

Document this complete trace showing the new code is correct because:
1. memFuture -= claimed deducts only the take-profit ETH
2. Ticket ETH remains in memFuture (correct -- it is still in the pool)
3. The storage write by _processAutoRebuy is overwritten by _setPrizePools
4. Net result: memFuture at _setPrizePools = initial - claimed = initial - take-profit (correct)
