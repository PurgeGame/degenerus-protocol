Formally trace the CEI (Checks-Effects-Interactions) order for the two primary reentrancy surfaces in the protocol: (1) `claimWinnings()` and its `_payoutWithStethFallback`/`_payoutWithEthFallback` helpers, and (2) the LINK ERC-677 `onTokenTransfer` callback in DegenerusAdmin and the `adminStakeEthForStEth()` stETH submission path. Produce ACCT-04 and ACCT-05 verdicts.

Purpose: CEI violations are the most commonly exploited class in Code4rena game audits. If `claimWinnings()` transfers ETH before zeroing state, a re-entering attacker can drain the contract. The LINK `onTokenTransfer` has a confirmed formal CEI violation (state write after external call) that must be assessed for exploitability.

<!-- CEI reference from research — verify against actual source -->

claimWinnings() CEI structure (DegenerusGame.sol ~1468-1483):
```
CHECKS:  amount = claimableWinnings[player]; if (amount <= 1) revert
EFFECTS: claimableWinnings[player] = 1 (sentinel)
EFFECTS: claimablePool -= payout
INTERACT: _payoutWithStethFallback(player) OR _payoutWithEthFallback(player)
```
Verify: _payoutWithStethFallback and _payoutWithEthFallback must NOT write to state before ETH transfer.

LINK onTokenTransfer (DegenerusAdmin.sol ~589-638):
```
CHECK:   msg.sender == LINK_TOKEN
INTERACT: linkToken.transferAndCall(coordinator, ...)   ← external call
EFFECT:  coin.creditLinkReward()                       ← state write AFTER external — formal CEI violation
```
Assess: Is the Chainlink coordinator trusted enough that re-entry during transferAndCall cannot exploit the post-call creditLinkReward state?

adminStakeEthForStEth() (DegenerusGame.sol ~1873-1888):
```
CHECKS:  msg.sender == ADMIN; amount > 0; ethBal >= amount; ethBal > claimablePool
INTERACT: steth.submit{value: amount}()   ← external call LAST
```
No state writes after steth.submit() — appears compliant. Verify.
