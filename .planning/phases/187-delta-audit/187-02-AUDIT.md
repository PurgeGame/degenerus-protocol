### 1b. Self-Call Guard on JackpotModule.runBafJackpot

**File:** `contracts/modules/DegenerusGameJackpotModule.sol` line 2487

```solidity
function runBafJackpot(uint256 poolWei, uint24 lvl, uint256 rngWord) external returns (...) {
    if (msg.sender != address(this)) revert E();
```

The guard is the FIRST statement in the function body (line 2487, immediately after the opening brace at 2486).

**Delegatecall context analysis:**
- AdvanceModule calls `IDegenerusGame(address(this)).runBafJackpot(...)` -- this is a CALL to the proxy
- Game.sol receives the call where `msg.sender == address(this)` (self-call)
- Game.sol guard passes, then delegatecalls to JackpotModule
- In the delegatecall context, `msg.sender` is preserved from the outer CALL -- still `address(this)`
- JackpotModule guard passes because `msg.sender == address(this)`

For an external EOA or contract calling runBafJackpot on the proxy:
- Game.sol receives the call where `msg.sender == external_caller != address(this)` -- reverts with E()
- Even if somehow reaching JackpotModule directly, `msg.sender` would not be `address(this)` -- reverts

**Verdict: PASS** -- guard correctly rejects unauthorized external callers

## Quest Entropy Audit

**File:** `contracts/modules/DegenerusGameAdvanceModule.sol` lines 399-401

**Old code:** `uint256(keccak256(abi.encodePacked(rngWordByDay[day], "LEVEL_QUEST")))`
**New code:** `uint256(keccak256(abi.encodePacked(rngWord, "LEVEL_QUEST")))`

**Equivalence proof:**

1. `rngWord` is assigned by `rngGate(ts, day, purchaseLevel, lastPurchase, bonusFlip)` at line 267.
2. `rngGate` returns `rngWordByDay[day]` when it already exists (line 1040).
3. When processing a fresh VRF word, `rngGate` writes `rngWordByDay[day] = finalWord` (via `_processVrf` at line 1696) and returns `finalWord`.
4. In both cases, the return value equals `rngWordByDay[day]` at the point of return.
5. No other code path modifies `rngWordByDay[day]` between `rngGate` and the quest entropy call at line 399.

Therefore `rngWord == rngWordByDay[day]` at the quest entropy call site. The keccak256 hash produces identical output.

**Gas savings:** 1 cold SLOAD (~2,100 gas) eliminated by using the already-in-memory value.

**Verdict: PASS** -- semantically identical; pure gas optimization

## Daily Jackpot Final-Day Change

**File:** `contracts/modules/DegenerusGameJackpotModule.sol` lines 450-458

**Old code (single path):**
```solidity
_setCurrentPrizePool(_getCurrentPrizePool() - paidDailyEth);
```

**New code (branched):**
```solidity
if (isFinalPhysicalDay_) {
    uint256 unpaidDailyEth = dailyEthBudget - paidDailyEth;
    _setCurrentPrizePool(_getCurrentPrizePool() - dailyEthBudget);
    if (unpaidDailyEth != 0) {
        _setFuturePrizePool(_getFuturePrizePool() + unpaidDailyEth);
    }
} else {
    _setCurrentPrizePool(_getCurrentPrizePool() - paidDailyEth);
}
```

**Analysis:**

This is a **bugfix** for the final physical day of a jackpot phase. On the final day, if not all daily ETH budget was distributed to winners (unpaid portion), the old code only deducted `paidDailyEth` from currentPool. The unpaid portion remained stranded in currentPool, where it would be consolidated into the next level's pool without being tracked as future-pool ETH.

New behavior: `currentPool -= dailyEthBudget` (full deduction) and `futurePool += unpaidDailyEth` (explicitly routes unpaid ETH to future pool where it participates in future economics).

**Pool accounting verification:**

- `dailyEthBudget = paidDailyEth + unpaidDailyEth` (by definition)
- Old: `currentPool -= paidDailyEth`. Net change: -paidDailyEth from currentPool.
- New: `currentPool -= dailyEthBudget`, `futurePool += unpaidDailyEth`. Net change: -dailyEthBudget from current, +unpaidDailyEth to future = -(paidDailyEth + unpaidDailyEth) + unpaidDailyEth = -paidDailyEth net across both pools.

The total ETH leaving the tracked pools equals `paidDailyEth` in both cases (the actual ETH distributed to winners). No ETH is created or destroyed.

**Non-final days:** `_setCurrentPrizePool(_getCurrentPrizePool() - paidDailyEth)` -- unchanged behavior.

**Verdict: PASS** -- correct bugfix that properly routes unpaid final-day ETH to futurePool; pool accounting conserved

### 6a. _evaluateGameOverPossible -> _evaluateGameOverAndTarget

**Old:** Two separate operations at the purchase-phase daily check (line 343 area):
1. `if (gameOverPossible) { _evaluateGameOverPossible(lvl, purchaseLevel); }` -- re-evaluates flag
2. `if (_getNextPrizePool() >= levelPrizePool[purchaseLevel - 1])` -- separate target check

**New:** Combined into single operation:
```solidity
bool targetMet = gameOverPossible
    ? _evaluateGameOverAndTarget(lvl, purchaseLevel)
    : _getNextPrizePool() >= levelPrizePool[purchaseLevel - 1];
```

The combined function `_evaluateGameOverAndTarget` (lines 1752-1769) reads `nextPool` and `target` once, returns `targetMet`, and updates `gameOverPossible`. When `gameOverPossible` is false, the ternary skips the function call entirely and just does the target check (identical to old behavior).

**Analysis of `_evaluateGameOverAndTarget`:**
- `targetMet = nextPool >= target` -- same condition as old separate check
- `if (lvl < 10 || targetMet) { gameOverPossible = false; return targetMet; }` -- clears flag when target met (same as old) or below level 10 (same as old)
- Deficit/drip projection path unchanged from old `_evaluateGameOverPossible`

**Turbo path (line 173):** `if (gameOverPossible) gameOverPossible = false` -- old was unconditional `gameOverPossible = false`. The conditional avoids a cold SSTORE when flag is already false. Behaviorally identical: setting false to false is a no-op.

**Verdict: PASS** -- identical behavior with gas optimization (shared SLOADs, avoided SSTORE)

## distributeYieldSurplus Visibility Change

**Old:** `function _distributeYieldSurplus(uint256 rngWord) private` inside JackpotModule
**New:** `function distributeYieldSurplus(uint256 rngWord) external` in JackpotModule

**Access control assessment:**

1. **No self-call guard** on `distributeYieldSurplus` (unlike `runBafJackpot`). This is correct because it is NOT called via the self-call pattern. It is called via delegatecall from AdvanceModule's private `_distributeYieldSurplus` wrapper (line 591-601), which uses `ContractAddresses.GAME_JACKPOT_MODULE.delegatecall(...)`.

2. **Can an external caller reach this on the proxy?** DegenerusGame.sol does NOT have:
   - A `distributeYieldSurplus` function definition
   - A generic `fallback()` function that routes unknown selectors

   The only fallback-like function is `receive() external payable` (line 2553), which only handles plain ETH transfers. Unknown function selectors will revert with "function selector not found" at the proxy level.

3. **Direct calls to the implementation contract?** Calling `distributeYieldSurplus` directly on the JackpotModule implementation contract would execute against the implementation's own storage (not the proxy's). Since the implementation contract holds no real ETH or state, this is harmless.

**Verdict: PASS** -- `external` visibility is safe because (a) the proxy has no fallback routing unknown selectors to JackpotModule, and (b) direct implementation calls operate on dead storage
