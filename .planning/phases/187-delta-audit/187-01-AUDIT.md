## Section 1: Operation Order Analysis

### OLD Call Sequence (pre-Phase 186)

```
1. levelPrizePool[purchaseLevel] = _getNextPrizePool()
2. _applyTimeBasedFutureTake(ts, purchaseLevel, rngWord)
   - Reads: nextPool, futurePool, levelStartTime from STORAGE
   - Writes: _setNextPrizePool(), _setFuturePrizePool(), yieldAccumulator (3 SSTOREs)
3. _consolidatePrizePools(purchaseLevel, rngWord)     [delegatecall -> JackpotModule]
   a. x00: yield dump 50% yieldAccumulator -> futurePool (SSTORE)
   b. Merge next -> current (2 SSTOREs)
   c. x00: keep roll future -> current (2 SSTOREs)
   d. Coinflip credit (from _getCurrentPrizePool() -- SLOAD)
   e. _distributeYieldSurplus(rngWord) -- reads obligations from storage
4. _runRewardJackpots(lvl, rngWord)                   [delegatecall -> JackpotModule]
   a. Read futurePool from STORAGE into futurePoolLocal
   b. BAF jackpot (x10): draw from baseFuturePool snapshot
   c. Decimator x100: draw from baseFuturePool snapshot
   d. Decimator x5: draw from running futurePoolLocal
   e. Rebuy delta reconciliation (re-read storage)
   f. SSTORE: futurePool, claimablePool
5. _drawDownFuturePrizePool(lvl)                      [in AdvanceModule]
   - x00: skip. Non-x00: 15% future -> next (2 SSTOREs)
```

**Total intermediate SSTOREs in old flow:** 7-11 depending on path (time take: 3, consolidation: 2-5, reward jackpots: 2, drawdown: 0-2).

### NEW Call Sequence (Phase 186)

```
1. levelPrizePool[purchaseLevel] = _getNextPrizePool()
2. _distributeYieldSurplus(rngWord)                   [delegatecall -> JackpotModule]
   - Reads obligations from STORAGE (unchanged values)
3. _consolidatePoolsAndRewardJackpots(lvl, purchaseLevel, ts, rngWord)  [private, inline]
   a. Read all pools into memory: memFuture, memCurrent, memNext, memYieldAcc
   b. Time-based future take (in memory -- was separate _applyTimeBasedFutureTake)
   c. x00: yield dump 50% memYieldAcc -> memFuture (in memory)
   d. Snapshot: storageBaseFuture = _getFuturePrizePool(), baseMemFuture = memFuture
   e. BAF jackpot (x10): draw from baseMemFuture snapshot
   f. Decimator x100: draw from baseMemFuture snapshot
   g. Decimator x5: draw from running memFuture
   h. Rebuy delta: memFuture += _getFuturePrizePool() - storageBaseFuture
   i. x00: keep roll memFuture -> memCurrent (in memory)
   j. Merge memNext -> memCurrent (in memory)
   k. Coinflip credit (from memCurrent)
   l. Future->next drawdown 15% non-x00 (in memory)
   m. SSTORE batch: _setPrizePools, currentPrizePool, yieldAccumulator, claimablePool
```

**Total intermediate SSTOREs in new flow:** 1 batch (4-5 slot writes at end) + any auto-rebuy writes during jackpot execution.

### Ordering Changes

**Change 1: Yield surplus moved BEFORE pool consolidation**

- **Old:** Ran inside `_consolidatePrizePools` AFTER merge, keep roll, and coinflip credit.
- **New:** Runs at step 2, before `_consolidatePoolsAndRewardJackpots`.
- **Safety:** The obligations check in `distributeYieldSurplus` computes `getCurrentPrizePool + getNextPrizePool + claimablePool + getFuturePrizePool + yieldAccumulator`. At step 2, no pool values have been written to storage yet (only `levelPrizePool[purchaseLevel]` was set, which is not part of the obligations sum). The obligations sum equals the sum of all tracked pool balances. Pool consolidation moves ETH between pools but does not change the total. Therefore the surplus calculation produces the same result regardless of when it runs relative to consolidation. **PASS -- ordering is safe.**

**Change 2: Time-based future take batched into memory instead of separate SSTOREs**

- **Old:** `_applyTimeBasedFutureTake` wrote 3 SSTOREs: `_setNextPrizePool`, `_setFuturePrizePool`, `yieldAccumulator +=`.
- **New:** Mutates `memNext`, `memFuture`, `memYieldAcc` in memory. Written once at end.
- **Safety:** Same arithmetic, different write timing. Since all subsequent operations in the new code read from memory variables (not storage), the deferred SSTORE has no effect on computation. **PASS -- pure batching optimization.**

**Change 3: Keep roll now runs AFTER jackpots instead of BEFORE**

- **Old:** In `consolidatePrizePools`: yield dump -> merge next->current -> keep roll -> coinflip. Then `runRewardJackpots`: jackpots draw from futurePool (post-keep-roll).
- **New:** yield dump -> jackpots -> rebuy delta -> keep roll -> merge -> coinflip.
- **Safety analysis:** This is an **intentional behavioral change** (per D-01). In the old flow, keep roll reduced futurePool before jackpots, meaning jackpots drew from a smaller pool. In the new flow, jackpots draw from the full post-yield-dump futurePool (before keep roll reduces it). The jackpot allocation uses `baseMemFuture` snapshot (taken after yield dump, before jackpots), which is the post-yield-dump value. Keep roll then operates on what remains after jackpots. This means:
  - Jackpot sizes are slightly LARGER (drawn from pre-keep-roll pool)
  - Keep roll operates on the post-jackpot remainder
  - Net ETH conservation is maintained: every wei removed from memFuture by jackpots is accounted for in claimableDelta or returned via refund/lootbox
- **See check 3f for detailed analysis. ACCEPTED behavioral change.**

**Change 4: Merge next->current moved AFTER keep roll**

- **Old:** Merge happened BEFORE keep roll in consolidatePrizePools.
- **New:** Merge happens AFTER keep roll (line 785 vs line 763).
- **Safety:** In the old code, merge happened first, then keep roll moved ETH from future->current. The merge and keep roll are independent operations -- merge adds next to current (not future), keep roll moves future to current. The order doesn't matter because they affect different source variables (next vs future). **PASS -- operations are independent.**

**Change 5: Coinflip credit uses memCurrent instead of _getCurrentPrizePool()**

- **See check 3a for detailed analysis.**

**Change 6: decWindowOpen relocation**

- **See check 3i for detailed analysis.**

**Change 7: x100 level detection parameter change**

- **Old:** `_consolidatePrizePools(purchaseLevel, ...)` used `purchaseLevel` for x100 checks (yield dump, keep roll).
- **Old:** `_runRewardJackpots(lvl, ...)` used `lvl` for jackpot trigger checks (BAF x10, Decimator x100/x5).
- **New:** `_consolidatePoolsAndRewardJackpots(lvl, purchaseLevel, ...)` uses `lvl` for ALL level-based checks (yield dump, keep roll, BAF, Decimator).
- **Impact:** The yield dump and keep roll now trigger one level later:
  - Old: fires when `purchaseLevel % 100 == 0` (at transition from level 99->100, 199->200, etc.)
  - New: fires when `lvl % 100 == 0` (at transition from level 100->101, 200->201, etc.)
  - BAF/Decimator triggers are UNCHANGED (both old and new use `lvl`)
- **See finding F-187-01 for detailed analysis.**

### 3f. x00 Keep Roll Timing

**Old order:**
1. consolidatePrizePools: yield dump -> merge -> keep roll -> coinflip -> yield surplus
2. runRewardJackpots: BAF -> Decimator (draw from post-keep-roll futurePool)

**New order:**
1. yield dump -> BAF -> Decimator (draw from pre-keep-roll memFuture) -> rebuy delta -> keep roll -> merge -> coinflip

**Behavioral difference:** In the old flow, keep roll REDUCED futurePool before jackpots drew from it. In the new flow, jackpots draw from the FULL post-yield-dump pool, then keep roll operates on what remains.

**Example with numbers:**
- Suppose memFuture = 100 ETH after yield dump.
- Old flow: keep roll 50% -> futurePool = 50 ETH, currentPool += 50. BAF draws 10% of 50 = 5 ETH.
- New flow: BAF draws 10% of 100 = 10 ETH. After BAF, memFuture ~ 90 ETH. Keep roll 50% -> memFuture = 45, memCurrent += 45.

So jackpots are ~2x larger in the new flow (drawn from full pool vs half pool). This is a real economic change.

**Is this intentional?** Per D-01 in 187-CONTEXT.md: "The audit must prove the new order is sound, not that outputs are byte-identical." Per the plan check 3f: "State whether this ordering change is intentional and whether it changes pool economics."

**Assessment:** This is an intentional design change. The new ordering is logically sound:
- All jackpot draws are bounded by fixed percentages of the pool
- The keep roll operates on the remainder, so total conservation holds
- The net effect is that more ETH flows to jackpot winners and less to current pool from keep roll
- No ETH is created or destroyed

However, this changes the economic model: jackpot sizes at x100 levels increase. At x100, BAF draws 20% and Decimator draws 30% of baseFuture. With 100 ETH: old flow (post-keep-roll) gives 10 + 15 = 25 ETH to jackpots. New flow: 20 + 30 = 50 ETH to jackpots. The difference is significant.
