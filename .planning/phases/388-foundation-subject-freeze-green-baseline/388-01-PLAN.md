<objective>
Re-derive the AUTHORITATIVE storage layout for the byte-frozen subject `a8b702a7` via
`forge inspect <C> storageLayout` for the four contracts the post-v62 packing phase reshuffled
(DegenerusGame tail, StakedDegenerusStonk, BurnieCoinflip, DegenerusAdmin), and reconcile every
slot-hardcoded forge harness poke against it (FND-02). The packing-phase shifts are REGION-DEPENDENT,
not a uniform -1: the DegenerusGame tail shifted up ~4 slots (BASE max 63 → SUBJ max 59) from the
levelDgnrsPacked / deityBoonPacked / lootboxEvCapPacked / bingoFirsts / DecClaimRound consolidations;
StakedDegenerusStonk shifted -3 (the slot-0 totalSupply/pending pack + poolBalances uint128[5]);
BurnieCoinflip repacked slots 0/1 + a new bool @slot4-off23; DegenerusAdmin folded votes into
voterRecords. A slot-layout change keeps the compile green but silently reads/writes the WRONG field
at RUNTIME on a stale-slot harness — so recalibrate from the INSPECTED layout, not from constants.

Purpose: the slot-hardcoded harnesses are the regression oracle the sweeps (389+) reproduce findings
against. A stale-slot poke that writes the wrong field could MASK a real packing bug. The full forge
suite already runs green at the subject, so this is verify-confirm-and-record (authoritative layout +
reconciliation ledger + the canary), not a mass repair — but it is the load-bearing first foundation
step, because the green baseline (Plan 03) is only trustworthy once the slot pokes are proven to hit
the right fields under the new layout.

Output: an authoritative a8b702a7 layout key for the four reshuffled contracts + a per-harness
slot-poke reconciliation ledger + the StorageFoundation canary confirmed against the new tail packs.
</objective>
