**Slot 0 repack (per SLOT-01):**
The current slot 0 layout is 32/32 bytes with purchaseStartDay (6 bytes) + dailyIdx (6 bytes) + rngRequestTime (6 bytes) + level (3 bytes) + 11 flags (11 bytes) = 32 bytes.

After Task 1, purchaseStartDay and dailyIdx are 4 bytes each, freeing 4 bytes. The new slot 0 should be 30/32 bytes:
- purchaseStartDay (uint32, 4 bytes)
- dailyIdx (uint32, 4 bytes)
- rngRequestTime (uint48, 6 bytes) — UNCHANGED
- level (uint24, 3 bytes) — UNCHANGED
- jackpotPhaseFlag through gameOverPossible (11 bools/uint8s, 11 bytes) — UNCHANGED
- ticketWriteSlot (bool, 1 byte) — MOVED from slot 1
- prizePoolFrozen (bool, 1 byte) — MOVED from slot 1

To achieve this, MOVE the `ticketWriteSlot` and `prizePoolFrozen` declarations from the "EVM SLOT 1" section to AFTER `gameOverPossible` in the "EVM SLOT 0" section. They must be declared in sequence with the other slot 0 variables — no intervening non-storage declarations allowed.

The variable declaration order in Solidity determines slot packing, so the order must be:
1. purchaseStartDay (uint32)
2. dailyIdx (uint32)
3. rngRequestTime (uint48)
4. level (uint24)
5. jackpotPhaseFlag (bool)
6. jackpotCounter (uint8)
7. lastPurchaseDay (bool)
8. decWindowOpen (bool)
9. rngLockedFlag (bool)

16. ticketWriteSlot (bool) — MOVED HERE
17. prizePoolFrozen (bool) — MOVED HERE
