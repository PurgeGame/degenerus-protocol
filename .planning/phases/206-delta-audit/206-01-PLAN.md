**Step 4: Classify each change category for GameOverModule.sol.**

The diff has these change categories (executor must verify each independently):

A. **NatSpec updates (lines ~75-79 in diff):** Two doc comment lines changed. "returns early" -> "reverts". Added "funds exist but RNG word unavailable" to reverts doc. These are COMMENT-ONLY.

B. **New preRefundAvailable computation (added ~11 lines):** Computes `preRefundAvailable = totalFunds > claimablePool ? totalFunds - claimablePool : 0` before any side effects. This is the same formula that was previously used for `budget` (old line: `uint256 budget = totalFunds > claimablePool ? totalFunds - claimablePool : 0`). Verify the formula is identical. This is OK -- intentional restructuring.

C. **New RNG gate block (added ~6 lines):** Declares `uint256 rngWord`, checks `if (preRefundAvailable != 0) { rngWord = rngWordByDay[day]; if (rngWord == 0) revert E(); }`. This replaces the old `uint256 rngWord = rngWordByDay[day]; if (rngWord == 0) return;` that was 30+ lines later. The OLD code returned silently AFTER side effects. The NEW code reverts BEFORE side effects. This is the INTENTIONAL behavioral change per Phase 203 plan. Classify as OK.

D. **Deity refund budget change:** Old: `uint256 budget = totalFunds > claimablePool ? totalFunds - claimablePool : 0;` New: `uint256 budget = preRefundAvailable;` These are equivalent -- `preRefundAvailable` is exactly `totalFunds > claimablePool ? totalFunds - claimablePool : 0`. OK.

E. **Removed old `available` computation and moved it after refunds:** Old code computed `available` before latch/burns. New code computes `available` after refunds (because `claimablePool` may have grown). The result is the same when no refunds happen. When refunds DO happen, the new code correctly deducts the refund amount from distributable funds (they were already added to claimablePool). Verify this is mathematically equivalent to the old behavior where `budget` was deducted separately. OK -- this is the intentional fix.

F. **Latch/burn/zeroing reordering:** Old code had `available == 0` early return with its own `gameOverFinalJackpotPaid/pool zeroing` block, then the RNG check, then a SECOND `gameOverFinalJackpotPaid/pool zeroing` block. New code has a SINGLE `gameOverFinalJackpotPaid/pool zeroing` block that always executes (before the `available == 0` return). This eliminates code duplication. Verify no logic difference. OK.

G. **Removed old RNG check block:** The old `uint256 rngWord = rngWordByDay[day]; if (rngWord == 0) return;` is deleted. Replaced by the new RNG gate block above. OK.

H. **Comment changes within function body:** Several comments trimmed or reworded. Verify none mask logic changes. COMMENT-ONLY.

I. **Removed trailing blank line before closing brace.** WHITESPACE.
