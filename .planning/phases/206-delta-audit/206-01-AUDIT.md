#### Hunk 3: New RNG gate block (added after preRefundAvailable)

```diff
+        // RNG gate: when distributable funds exist, require RNG word.
+        // Defense-in-depth -- caller (_handleGameOverPath) already guarantees
+        // rngWordByDay[day] != 0 before calling, so this revert should never fire.
+        uint256 rngWord;
+        if (preRefundAvailable != 0) {
+            rngWord = rngWordByDay[day];
+            if (rngWord == 0) revert E();
+        }
```

**Verification against old code:** The OLD code had two separate RNG-related blocks:
1. Old line 143: `uint256 rngWord = rngWordByDay[day];` followed by `if (rngWord == 0) return;` -- this was AFTER `gameOver = true`, `gameOverTime` write, `burnAtGameOver()` calls, and the `available == 0` early-return block.
2. The old `return` silently exited after irreversible side effects had already executed.

The NEW code:
- Declares `rngWord` as uninitialized (will be 0 if `preRefundAvailable == 0`, which is fine since the distribution path is never reached when `available == 0`).
- Only reads `rngWordByDay[day]` when `preRefundAvailable != 0` (funds to distribute exist).
- Reverts with `E()` instead of returning silently.
- Executes BEFORE any side effects (deity refunds, latch, burns, pool zeroing).

**Behavioral change:** When `preRefundAvailable > 0` and `rngWord == 0`, OLD code: returns after executing side effects. NEW code: reverts before any side effects. This is the **intentional fix** -- the entire purpose of Phase 203.

When `preRefundAvailable == 0`, the RNG gate is skipped entirely. The old code would have read `rngWord` anyway (at old line 143) but returned at the `available == 0` check (old line 139) before using it. So skipping the read is functionally equivalent.

**Verdict: OK** -- This is the primary intentional change. Revert-vs-return swap, moved above all side effects.

#### Hunk 7: Eliminated duplicated available==0 early-return block

OLD code (lines 136-145):
```diff
-        if (available == 0) {
-            gameOverFinalJackpotPaid = true;
-            _setNextPrizePool(0);
-            _setFuturePrizePool(0);
-            _setCurrentPrizePool(0);
-            yieldAccumulator = 0;
-            return;
-        }
-
-        // Get RNG word for jackpot selection (includes VRF fallback after 3 days)
-        uint256 rngWord = rngWordByDay[day];
-        if (rngWord == 0) return; // RNG not ready yet — don't latch, allow retry
-
         gameOverFinalJackpotPaid = true;
         _setNextPrizePool(0);
         _setFuturePrizePool(0);
         _setCurrentPrizePool(0);
         yieldAccumulator = 0;
+
+        // Recalculate available after refunds (claimablePool may have grown)
+        uint256 available = totalFunds > claimablePool ? totalFunds - claimablePool : 0;
+
+        if (available == 0) return;
```

**Detailed analysis:**

1. **Removed `if (available == 0) { ... return; }` block:** The OLD code had a separate early-return block that set `gameOverFinalJackpotPaid`, zeroed pools, and returned. The NEW code unconditionally executes `gameOverFinalJackpotPaid = true` and pool zeroing, THEN checks `if (available == 0) return`. This is **logically equivalent**: in both old and new, when `available == 0`, the pools are zeroed and `gameOverFinalJackpotPaid` is set. The new code simply eliminates the duplicated pool-zeroing block.

2. **Removed old RNG check block:** `uint256 rngWord = rngWordByDay[day]; if (rngWord == 0) return;` -- this is the old RNG check that was AFTER side effects. Replaced by the new RNG gate in Hunk 3 (BEFORE side effects). The old comment "RNG not ready yet -- don't latch, allow retry" was incorrect: by this point, `gameOver = true` and `burnAtGameOver()` had already fired, so "retry" was impossible. The new code fixes this by reverting before any side effects.

3. **Moved `available` computation:** Now computed AFTER `gameOverFinalJackpotPaid` and pool zeroing, but since pool zeroing sets `nextPrizePool/futurePrizePool/currentPrizePool/yieldAccumulator = 0` (which don't affect `totalFunds` or `claimablePool`), the computation result is identical.

**Verification of `available` equivalence:**
- OLD: `available = totalFunds > claimablePool ? totalFunds - claimablePool : 0` (old line 120, after deity refunds, before pool zeroing)
- NEW: `available = totalFunds > claimablePool ? totalFunds - claimablePool : 0` (new line 150, after deity refunds AND pool zeroing)
- `totalFunds` = `ethBal + stBal` -- unchanged by deity refunds (refunds only add to `claimableWinnings` mapping, not transfer ETH)
- `claimablePool` -- same in both: includes deity refunds from `claimablePool += totalRefunded`
- Pool zeroing (`_setNextPrizePool(0)` etc.) does NOT modify `totalFunds` or `claimablePool`
- Therefore: `available` is **identical** in both positions.

**Verdict: OK** -- Eliminates code duplication. Single pool-zeroing block. Moved RNG check to Hunk 3. `available` computed at equivalent point. No behavioral difference.
