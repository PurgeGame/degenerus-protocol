From contracts/libraries/GameTimeLib.sol (TIME-01, TIME-02):
- currentDayIndexAt(uint48 ts) internal pure returns (uint48):
    uint48 currentDayBoundary = uint48((ts - JACKPOT_RESET_TIME) / 1 days);
    return currentDayBoundary - ContractAddresses.DEPLOY_DAY_BOUNDARY + 1;
- JACKPOT_RESET_TIME = 82620 (= 22*3600 + 57*60 = 22:57:00 UTC in seconds-since-midnight)
- 1 days = 86400 seconds
- ±900s validator drift = ±1.04% of 86400s day period
- Day boundary: ts is shifted to 22:57 UTC — a tx at 22:56:30 is day N; at 22:57:30 is day N+1

Temporal math for ±900s boundary case:
- A tx at 22:57:00 UTC (ts = boundary) lands on day N+1
- Validator delays by 900s: tx at 22:56:00 (ts = boundary - 60s) would be day N, but pushed to 22:57:00 = day N+1
- Validator pulls forward by 900s: tx at 22:57:30 (ts = boundary + 30s) = day N+1, pushed to 22:56:30 = day N

From contracts/DegenerusGame.sol advanceGame() (TIME-01):
- advanceGame() is gated — requires sufficient time elapsed since last advance (game day progression)
- Daily jackpot allocation happens inside advanceGame() when the game advances from day N to N+1
- The game day counter is derived from currentDayIndexAt(block.timestamp)
- Key question: can a validator cause advanceGame() to be callable TWICE for the same real calendar day by manipulating timestamp?
