## IXNR-01: Claims Window (Drain-to-Sweep)

**Requirement:** Claims work between drain and sweep (claimablePool accessible), and all claim paths blocked after finalSwept.

### Trace

**1. `_claimWinningsInternal` (DegenerusGame.sol L1366-1382)**

```solidity
function _claimWinningsInternal(address player, bool stethFirst) private {
    if (finalSwept) revert E();            // L1367
    uint256 amount = claimableWinnings[player];
    if (amount <= 1) revert E();           // L1369
    uint256 payout;
    unchecked {
        claimableWinnings[player] = 1;     // L1372 — sentinel
        payout = amount - 1;               // L1373
    }
    claimablePool -= payout;               // L1375 — CEI
    emit WinningsClaimed(player, msg.sender, payout);
    if (stethFirst) {
        _payoutWithEthFallback(player, payout);
    } else {
        _payoutWithStethFallback(player, payout);
    }
}
```

- L1367: `if (finalSwept) revert E()` is the ONLY gate. There is NO `gameOver` check. When `gameOver==true` and `finalSwept==false`, claims proceed normally. When `finalSwept==true`, claims revert. OK.
- L1368-1373: Amount check, sentinel write, payout computation. All independent of gameOver state. OK.
- L1375: `claimablePool -= payout`. Between drain and sweep, `claimablePool` reflects the sum of all credited amounts. `handleFinalSweep` (GameOverModule L193-194) sets `finalSwept = true` then `claimablePool = 0` atomically in the same transaction. So `claimablePool` is only zeroed AFTER `finalSwept` blocks further claims. No underflow risk during the claim window. OK.

**2. Public entry points: `claimWinnings` and `claimWinningsStethFirst` (L1354-1363)**

```solidity
function claimWinnings(address player) external {       // L1354
    player = _resolvePlayer(player);
    _claimWinningsInternal(player, false);               // L1356
}

function claimWinningsStethFirst() external {            // L1361
    if (msg.sender != ContractAddresses.VAULT) revert E();
    _claimWinningsInternal(msg.sender, true);            // L1363
```

- Both route directly to `_claimWinningsInternal`. No additional gameOver check that would block legitimate claims during the window. OK.
- `claimWinningsStethFirst` restricted to VAULT (sDGNRS contract). The vault calls this during `_deterministicBurnFrom` (StakedDegenerusStonk L534: `game.claimWinnings(address(0))`) to top up its ETH balance before paying out burn proceeds. This path must remain open during the claim window. OK.

**3. Atomicity: `claimWinnings` vs `handleFinalSweep`**

`handleFinalSweep` (GameOverModule L188-208) runs atomically within a single transaction. It sets `finalSwept = true` at L193 before any external calls (VRF shutdown at L197, fund transfers at L207). Ethereum transactions are sequential within a block. A `claimWinnings` call in the same block either executes entirely before or entirely after `handleFinalSweep`. No race condition possible. OK.

**4. `claimAffiliateDgnrs` (DegenerusGame.sol L1393-1428)**

No `finalSwept` check, no `gameOver` check. Distributes DGNRS tokens via `dgnrs.transferFromPool(Pool.Affiliate, ...)` at L1413-1418. Does not touch `claimablePool` or ETH. Affiliate rewards are token-denominated and drawn from a separate sDGNRS pool. No interaction with the sweep mechanism. OK.

- L315-317: When `gameOver==true`, credits directly to `claimableWinnings` via `_creditClaimable` (PayoutUtils L33-39). No lootbox split. OK.
- `_creditClaimable` writes `claimableWinnings[beneficiary] += weiAmount` but does NOT increment `claimablePool`. This is correct because the decimator pool amount was already added to `claimablePool` during level transition at AdvanceModule L793-794 (`claimablePool += claimableDelta`), where `claimableDelta` includes the decimator `spend` (L735, L745). ## IXNR-02: Post-Gameover Auto-Rebuy Bypass

**Requirement:** `_addClaimableEth` skips auto-rebuy when gameOver==true, crediting winnings directly instead of converting to tickets.

### Trace

**1. JackpotModule `_addClaimableEth` (L765-789)**

```solidity
function _addClaimableEth(
    address beneficiary, uint256 weiAmount, uint256 entropy
) private returns (uint256 claimableDelta, uint24 rebuyLevel, uint32 rebuyTickets) {
    if (weiAmount == 0) return (0, 0, 0);           // L773
    if (!gameOver) {                                  // L777
        AutoRebuyState memory state = autoRebuyState[beneficiary];
        if (state.autoRebuyEnabled) {
            return _processAutoRebuy(beneficiary, weiAmount, entropy, state);
        }
    }
    _creditClaimable(beneficiary, weiAmount);         // L787
    return (weiAmount, 0, 0);                         // L788
}
```

- L777: `if (!gameOver)` — when `gameOver==true`, the entire auto-rebuy block (L777-783) is skipped. Auto-rebuy never fires post-gameover. OK.
- L787: Falls through to `_creditClaimable(beneficiary, weiAmount)` — credits directly to `claimableWinnings[beneficiary]`. OK.
- Return value `(weiAmount, 0, 0)`: `rebuyLevel=0` and `rebuyTickets=0` confirm no ticket minting. OK.

**2. `_creditClaimable` (PayoutUtils L33-39)**

```solidity
function _creditClaimable(address beneficiary, uint256 weiAmount) internal {
    if (weiAmount == 0) return;
    unchecked {
        claimableWinnings[beneficiary] += weiAmount;  // L36
    }
    emit PlayerCredited(beneficiary, beneficiary, weiAmount);
}
```

- Updates `claimableWinnings` only. No gameOver dependency inside the function. OK.
- The caller (`_addClaimableEth` at L787) is the only path in JackpotModule that calls `_creditClaimable` in this context. The `claimablePool` increment is handled by the caller of `_addClaimableEth` (via the returned `claimableDelta`). OK.

**3. DegeneretteModule `_addClaimableEth` (L1090-1094)**

```solidity
function _addClaimableEth(address beneficiary, uint256 weiAmount) private {
    if (weiAmount == 0) return;                       // L1091
    claimablePool += weiAmount;                       // L1092
    _creditClaimable(beneficiary, weiAmount);          // L1093
}
```

- Different function signature (no `entropy` parameter), no auto-rebuy logic. Always credits directly. OK.
- This is consistent: degenerette payouts never auto-rebuy (degenerettes are a side-game, not the main ticket system). OK.

**4. Exhaustive `_processAutoRebuy` caller check**

`_processAutoRebuy` is called exactly ONCE in the entire codebase: JackpotModule L781, inside the `if (!gameOver)` block. No other path can trigger auto-rebuy. OK.

### Findings

| # | Class | Description |
|---|-------|-------------|
| 1 | OK | JackpotModule `_addClaimableEth` L777: `if (!gameOver)` gates auto-rebuy. When gameOver, credits directly via `_creditClaimable`. |
| 2 | OK | `_processAutoRebuy` called only from L781, inside the gameOver-gated block. No bypass path. |
| 3 | OK | DegeneretteModule `_addClaimableEth` (L1090) is a separate function with no auto-rebuy logic. Always credits directly. |
| 4 | OK | `_creditClaimable` has no gameOver dependency. Pure accounting. |

### Verdict: PASS

Auto-rebuy is gated by `if (!gameOver)` at JackpotModule L777. When gameOver==true, all winnings credit directly to `claimableWinnings` without ticket conversion. Single entry point (`_processAutoRebuy` at L781) confirmed. DegeneretteModule has independent `_addClaimableEth` with no auto-rebuy path.

**3. `_deterministicBurnFrom` (StakedDegenerusStonk L516-558)**

```solidity
function _deterministicBurnFrom(address beneficiary, address burnFrom, uint256 amount) private returns (uint256 ethOut, uint256 stethOut) {
    uint256 bal = balanceOf[burnFrom];
    if (amount == 0 || amount > bal) revert Insufficient();
    uint256 supplyBefore = totalSupply;

    uint256 ethBal = address(this).balance;           // L521
    uint256 stethBal = steth.balanceOf(address(this));// L522
    uint256 claimableEth = _claimableWinnings();      // L523
    uint256 totalMoney = ethBal + stethBal + claimableEth - pendingRedemptionEthValue; // L524
    uint256 totalValueOwed = (totalMoney * amount) / supplyBefore; // L525

    unchecked {
        balanceOf[burnFrom] = bal - amount;           // L528
        totalSupply -= amount;                        // L529
    }
    emit Transfer(burnFrom, address(0), amount);

    if (totalValueOwed > ethBal && claimableEth != 0) {
        game.claimWinnings(address(0));               // L534
        ethBal = address(this).balance;
        stethBal = steth.balanceOf(address(this));
    }

    if (totalValueOwed <= ethBal) {
        ethOut = totalValueOwed;                      // L540
    } else {
        ethOut = ethBal;                              // L542
        stethOut = totalValueOwed - ethOut;           // L543
        if (stethOut > stethBal) revert Insufficient();
    }
    // ... transfer ETH and stETH ...
    emit Burn(beneficiary, amount, ethOut, stethOut, 0);
}
```

- L521-525: Pro-rata share computation. `totalMoney = ethBal + stethBal + claimableEth - pendingRedemptionEthValue`. Deducts pending gambling burns (pre-gameover submissions). Pure arithmetic — no RNG. OK.
- L523: `_claimableWinnings()` (L846-850) reads `game.claimableWinningsOf(address(this))` with sentinel-1 adjustment. View call, no RNG. OK.
- L534: `game.claimWinnings(address(0))` — calls back to DegenerusGame to claim sDGNRS's accrued winnings. This works because `finalSwept` is false during the claim window. No RNG involvement. OK.
- L525: `totalValueOwed = (totalMoney * amount) / supplyBefore` — pro-rata, deterministic. No RNG. OK.
- No coinflip, no roll, no VRF request, no `_submitGamblingClaim`. The deterministic path is entirely arithmetic. OK.

**11. Degenerette operations (`placeDegeneretteBet`, `resolveDegeneretteBets`)**

`placeDegeneretteBet` (DegenerusGame.sol L721-745) delegates to DegeneretteModule `placeDegeneretteBet` (L346-362) -> `_placeDegeneretteBet` (L384-414) -> `_placeDegeneretteBetCore` (L416-432).

```solidity
function _placeDegeneretteBetCore(...) private returns (uint256 totalBet) {
    if (ticketCount == 0 || ticketCount > MAX_SPINS_PER_BET) revert InvalidBet();
    if (amountPerTicket == 0) revert InvalidBet();
    uint48 index = lootboxRngIndex;
    if (index == 0) revert E();                                // L429
    if (lootboxRngWordByIndex[index] != 0) revert RngNotReady(); // L430
```

No explicit `gameOver` check. However, implicitly blocked by RNG lifecycle:

- L430: `lootboxRngWordByIndex[index] != 0` reverts with `RngNotReady` once the last pending VRF word is fulfilled. Post-gameover, no new indices are created (no `advanceGame` calls), so once the final word is filled, all future bet attempts revert.
- There is a narrow window between `gameOver=true` and VRF fulfillment where the last index still has `word==0`. During this window, a bet could technically be placed. The bet would accept ETH (via payable) or BURNIE (burned) or claimable (debited from `claimableWinnings`). Resolution would require `lootboxRngWordByIndex[betIndex] != 0`, which happens only after the last VRF fulfills. Bets placed in this window ARE resolvable once the word arrives, so funds are not permanently stuck.
- After VRF shutdown (`handleFinalSweep`), the VRF subscription is cancelled. No new words can arrive. Any bet placed after the last word fulfillment but before shutdown would be permanently stuck.

`resolveDegeneretteBets` (DegenerusGame.sol L750-764): Delegates to `resolveBets` (DegeneretteModule L368-377). No gameOver check, but resolution only works when `lootboxRngWordByIndex[betIndex] != 0`.

CONCERN: No explicit `gameOver` check on degenerette bet placement. Implicit blocking via RNG lifecycle means there is a narrow window post-gameover where bets can be placed. These bets are either resolvable (if placed before last VRF fill) or permanently stuck (if placed after). An explicit `gameOver` revert at `_placeDegeneretteBetCore` entry would be cleaner.
