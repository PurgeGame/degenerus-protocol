## SWEP-01: 30-Day Delay Enforcement

**Requirement:** `handleFinalSweep` enforces a 30-day delay from `gameOverTime` with a non-manipulable timestamp comparison, and the `finalSwept` flag prevents double-sweep.

### Trace

**1. Entry point (GameOverModule L188-191)**

```solidity
function handleFinalSweep() external {                      // L188
    if (gameOverTime == 0) return;                           // L189
    if (block.timestamp < uint256(gameOverTime) + 30 days)   // L190
        return;
    if (finalSwept) return;                                  // L191
```

- L189: `gameOverTime == 0` guard. Returns (not reverts) when game not over. This is correct for a delegatecall target -- the caller (`_handleGameOverPath` at AdvanceModule L492-497) routes here when `gameOver == true`, but a no-op return is safe. OK.
- L190: `block.timestamp < uint256(gameOverTime) + 30 days`. `gameOverTime` is `uint48`, upcast to `uint256` for safe arithmetic. `30 days` = 2,592,000 seconds. `block.timestamp` is validator-influenceable by ~15s on Ethereum (irrelevant for a 30-day window). No overflow risk: `uint48` max = ~8.9 million years, so `+ 30 days` cannot overflow `uint256`. OK.
- L191: `finalSwept` guard. Returns silently if already swept. OK.

**2. Single-write verification for `gameOverTime`**

Grep of all assignments to `gameOverTime` across the contracts directory:

```
contracts/modules/DegenerusGameGameOverModule.sol:137:  gameOverTime = uint48(block.timestamp);
```

Single write at L137 inside `handleGameOverDrain`, which is gated by `gameOverFinalJackpotPaid` (L80). The `gameOver` latch at L136 ensures `handleGameOverDrain` runs at most once. No other setter exists anywhere in the codebase. OK.

Read-sites:
- `GameOverModule L189`: guard (this function)
- `GameOverModule L190`: delay check (this function)
- `DegenerusGame L2093`: view function `gameOverTimestamp()`
- `DegenerusStonk L306`: reads via `game.gameOverTimestamp()` for redemption timing

No site modifies `gameOverTime` after the initial latch. OK.

**3. Single-write verification for `finalSwept`**

Grep of all assignments to `finalSwept` across the contracts directory:

```
contracts/modules/DegenerusGameGameOverModule.sol:193:  finalSwept = true;
```

Single write at L193, set BEFORE any external calls (VRF shutdown at L197, fund transfers at L207). `finalSwept` is never reset to `false` anywhere. OK.

Read-sites (DegenerusGame.sol):
- `L1367`: `if (finalSwept) revert E();` -- blocks claims after sweep
- `L2082`: `if (finalSwept) return 0;` -- `claimablePoolView` returns 0
- `L2088`: `return finalSwept;` -- view function
- `L2266`: `if (finalSwept) return 0;` -- total owed view returns 0
- `L2280`: `if (finalSwept) return 0;` -- another view

All reads are consistent: `finalSwept` is a one-way latch that blocks further claims. OK.

**4. Bypass analysis**

Can any path bypass the 30-day delay?
- The three guards (L189, L190, L191) are sequential; all three must pass for execution to continue.
- `gameOverTime` is latched once and never modified.
- `block.timestamp` cannot be backdated sufficiently (15s validator drift vs 30-day window).
- `finalSwept` is never reset.
- No other function calls `_sendToVault` except `handleGameOverDrain` L178 (which has its own guards and only runs once due to `gameOverFinalJackpotPaid`).

No bypass path exists. OK.

**5. Atomicity with revert rollback**

L193 sets `finalSwept = true` BEFORE external calls. If `_sendToVault` at L207 reverts (e.g., stETH transfer failure), the entire transaction reverts, rolling back `finalSwept = true`. This means the sweep can be retried after the blocker resolves. This is correct behavior -- the state flag only persists on successful completion. OK.

### Findings

| # | Class | Description |
|---|-------|-------------|
| 1 | OK | `gameOverTime` written exactly once at GameOverModule L137, never modified after |
| 2 | OK | `finalSwept` written exactly once at GameOverModule L193, set before external calls, never reset |
| 3 | OK | 30-day delay uses `uint256` upcast, no overflow risk, validator drift irrelevant (15s vs 30d) |
| 4 | OK | Three sequential guards (L189-191) are comprehensive: game-not-over, too-early, already-swept |
| 5 | OK | Revert rollback preserves retry capability -- `finalSwept` only persists on success |

### Verdict: PASS

All three guards are comprehensive. `gameOverTime` and `finalSwept` are one-way latches with single-write semantics. The 30-day timestamp comparison is non-manipulable. No bypass path exists.

## SWEP-02: claimablePool Forfeiture and Fund Split (33/33/34)

**Requirement:** `handleFinalSweep` zeroes `claimablePool` before transfers and splits funds 33/33/34 to sDGNRS/vault/GNRUS via `_sendToVault`.

### Trace

**1. claimablePool forfeiture (GameOverModule L193-194)**

```solidity
finalSwept = true;                                           // L193
claimablePool = 0;                                           // L194
```

- L194: `claimablePool = 0` unconditionally zeroes all unclaimed winnings. This happens BEFORE fund balance reads at L199-201. After this line, no player can claim (claim paths check `finalSwept` at DegenerusGame L1367 and revert). OK.
- Order: `finalSwept = true` (L193) then `claimablePool = 0` (L194). Both state changes happen before any external calls. OK.

**2. Fund balance capture (GameOverModule L199-201)**

```solidity
uint256 ethBal = address(this).balance;                      // L199
uint256 stBal = steth.balanceOf(address(this));              // L200
uint256 totalFunds = ethBal + stBal;                         // L201
```

- L199: Native ETH balance of Game contract. OK.
- L200: stETH (Lido) balance of Game contract. This is an external call to the stETH token contract. OK.
- L201: Sum of both. Note: stETH is rebasing, so `balanceOf` may differ by 1 wei from the amount that can be transferred (Lido's 1-wei rounding). This is a known stETH behavior, not a bug in this contract. OK.
- These reads happen AFTER `claimablePool = 0`, so `totalFunds` captures the full contract balance including what was previously reserved for claims. OK.

**3. Zero-balance early return (GameOverModule L203-205)**

```solidity
emit FinalSwept(totalFunds);                                 // L203
if (totalFunds == 0) return;                                 // L205
```

- L203: Event emitted even if `totalFunds == 0`. This records that the sweep executed. OK.
- L205: If no funds, return after setting `finalSwept = true` and `claimablePool = 0`. State is correctly latched even with zero balance. OK.

**4. Fund split in `_sendToVault` (GameOverModule L217-225)**

```solidity
function _sendToVault(uint256 amount, uint256 stethBal) private {  // L217
    uint256 thirdShare = amount / 3;                                // L218
    uint256 gnrusAmount = amount - thirdShare - thirdShare;         // L219
    stethBal = _sendStethFirst(ContractAddresses.SDGNRS,            // L222
        thirdShare, stethBal);
    stethBal = _sendStethFirst(ContractAddresses.VAULT,             // L223
        thirdShare, stethBal);
    _sendStethFirst(ContractAddresses.GNRUS,                        // L224
        gnrusAmount, stethBal);
}
```

- L218: `thirdShare = amount / 3` -- integer division floors. For `amount = 100`: `thirdShare = 33`. OK.
- L219: `gnrusAmount = amount - thirdShare - thirdShare` -- remainder goes to GNRUS. For `amount = 100`: `gnrusAmount = 100 - 33 - 33 = 34`. For `amount = 1`: `thirdShare = 0`, `gnrusAmount = 1`. For `amount = 2`: `thirdShare = 0`, `gnrusAmount = 2`. All correct -- GNRUS always gets the remainder, no dust is lost. OK.
- L222: sDGNRS receives `thirdShare` (33%). OK.
- L223: Vault receives `thirdShare` (33%). OK.
- L224: GNRUS receives `gnrusAmount` (34% + any rounding remainder). OK.
- Total sent: `thirdShare + thirdShare + gnrusAmount = thirdShare + thirdShare + (amount - 2*thirdShare) = amount`. No funds lost. OK.

**5. Rounding verification**

The split `amount/3 + amount/3 + (amount - 2*(amount/3))` is algebraically exact for all positive integers:
- `amount % 3 == 0`: all three shares equal (`amount/3` each)
- `amount % 3 == 1`: sDGNRS and vault get `floor(amount/3)`, GNRUS gets `floor(amount/3) + 1`
- `amount % 3 == 2`: sDGNRS and vault get `floor(amount/3)`, GNRUS gets `floor(amount/3) + 2`

No dust is ever lost. GNRUS absorbs all rounding remainder. OK.

### Findings

| # | Class | Description |
|---|-------|-------------|
| 1 | OK | `claimablePool = 0` at L194 forfeits all unclaimed winnings before fund reads |
| 2 | OK | `totalFunds = ethBal + stBal` captures full contract balance (ETH + stETH) |
| 3 | OK | Zero-balance path (L205) still latches state correctly |
| 4 | OK | Split is exactly 33/33/34 with GNRUS absorbing rounding remainder, zero dust |
| 5 | NOTE | stETH `balanceOf` may differ from transferable amount by 1 wei (Lido rebasing). Not a contract bug -- inherent stETH behavior. At most 1 wei could be stuck. |

### Verdict: PASS

`claimablePool` is zeroed before fund capture. The 33/33/34 split is algebraically exact with no dust loss. All funds are accounted for.

## SWEP-03: stETH-First Transfer and Hard-Revert

**Requirement:** `_sendStethFirst` transfers stETH preferentially before ETH fallback, and all transfer failures hard-revert (no silent failure).

### Trace

**1. Zero-amount guard (GameOverModule L232-233)**

```solidity
function _sendStethFirst(address to, uint256 amount,               // L232
    uint256 stethBal) private returns (uint256) {
    if (amount == 0) return stethBal;                               // L233
```

- L233: Skips zero-amount transfers. Returns `stethBal` unchanged. OK.

**2. Pure stETH path (GameOverModule L234-236)**

```solidity
    if (amount <= stethBal) {                                       // L234
        if (!steth.transfer(to, amount)) revert E();                // L235
        return stethBal - amount;                                   // L236
    }
```

- L234: When enough stETH to cover the full amount. OK.
- L235: `steth.transfer` return value checked. Reverts with `E()` on `false`. stETH's `transfer` can also revert internally (e.g., insufficient balance from rebasing) -- that propagates as a revert too. OK.
- L236: Returns updated stETH balance. Underflow impossible since `amount <= stethBal` was checked at L234. OK.

**3. Mixed stETH + ETH path (GameOverModule L238-246)**

```solidity
    if (stethBal != 0) {                                            // L238
        if (!steth.transfer(to, stethBal)) revert E();              // L239
    }
    uint256 ethAmount = amount - stethBal;                          // L241
    if (ethAmount != 0) {                                           // L242
        (bool ok, ) = payable(to).call{value: ethAmount}("");       // L243
        if (!ok) revert E();                                        // L244
    }
    return 0;                                                       // L246
```

- L238-239: If any stETH remains, send ALL of it first. Transfer checked, reverts on failure. OK.
- L241: `ethAmount = amount - stethBal`. Since we're past the `amount <= stethBal` check (L234), we know `amount > stethBal`, so `ethAmount > 0` (unless `stethBal == amount`, but that's handled by the L234 branch). Actually: if `amount > stethBal`, then `ethAmount = amount - stethBal > 0`. OK.
- L242-244: ETH transfer via low-level `call`. Return value checked. Reverts on failure. OK.
- L246: Returns 0 -- all stETH has been consumed. OK.

**4. Hard-revert verification**

Every transfer failure path reverts:
- L235: stETH transfer returns false -> `revert E()`
- L239: stETH transfer returns false -> `revert E()`
- L244: ETH `.call` returns false -> `revert E()`
- stETH's own internal revert -> propagates as revert

No try/catch wraps any transfer in `_sendStethFirst`. No silent failure path exists. OK.

**5. stethBal tracking across sequential calls in `_sendToVault`**

```solidity
stethBal = _sendStethFirst(SDGNRS, thirdShare, stethBal);          // L222
stethBal = _sendStethFirst(VAULT, thirdShare, stethBal);           // L223
_sendStethFirst(GNRUS, gnrusAmount, stethBal);                     // L224
```

The return value of each call is the updated remaining stETH balance, which is passed to the next call. This creates a pipeline:

**Example:** `totalFunds = 100`, `stethBal = 50`, `thirdShare = 33`, `gnrusAmount = 34`.

- Call 1 (sDGNRS, 33): `33 <= 50`, pure stETH. Returns `50 - 33 = 17`.
- Call 2 (vault, 33): `33 > 17`, mixed. Sends 17 stETH + 16 ETH. Returns `0`.
- Call 3 (GNRUS, 34): `34 > 0`, pure ETH. Sends 34 ETH. Returns `0`.

Total stETH sent: `33 + 17 = 50`. Total ETH sent: `16 + 34 = 50`. Total: `100`. Correct. OK.

**Example 2:** `totalFunds = 100`, `stethBal = 100` (all stETH).

- Call 1 (sDGNRS, 33): `33 <= 100`, pure stETH. Returns `67`.
- Call 2 (vault, 33): `33 <= 67`, pure stETH. Returns `34`.
- Call 3 (GNRUS, 34): `34 <= 34`, pure stETH. Returns `0`.

Total stETH sent: `33 + 33 + 34 = 100`. Total ETH sent: `0`. Correct. OK.

**Example 3:** `totalFunds = 100`, `stethBal = 0` (all ETH).

- Call 1 (sDGNRS, 33): `33 > 0`, `stethBal == 0` skips stETH. Sends 33 ETH. Returns `0`.
- Call 2 (vault, 33): same, sends 33 ETH. Returns `0`.
- Call 3 (GNRUS, 34): same, sends 34 ETH. Returns `0`.

Total ETH sent: `33 + 33 + 34 = 100`. Correct. OK.

**6. Reentrancy consideration**

ETH transfers at L243 use low-level `.call`, which forwards gas and could trigger a receive/fallback in the recipient. Recipients are `sDGNRS`, `VAULT`, and `GNRUS` -- all protocol-controlled contracts (set as immutable constants in `ContractAddresses.sol`). Even if a recipient's fallback re-entered `handleFinalSweep`, the `finalSwept = true` guard at L191 would cause an immediate return. OK.

### Findings

| # | Class | Description |
|---|-------|-------------|
| 1 | OK | stETH transferred preferentially before ETH fallback in all cases |
| 2 | OK | All three transfer failure paths (L235, L239, L244) hard-revert with `E()` |
| 3 | OK | No try/catch wraps transfers -- no silent failure |
| 4 | OK | `stethBal` return-value pipeline correctly tracks remaining stETH across sequential calls |
| 5 | OK | Reentrancy via ETH `.call` is blocked by `finalSwept` guard and recipients are protocol-controlled |
| 6 | NOTE | stETH 1-wei rounding (Lido rebasing) may cause last transfer to attempt 1 wei more than available; stETH's internal check would revert. Retry after rebase resolves this. Theoretical edge case, not a bug. |

### Verdict: PASS

stETH is transferred preferentially. All failure paths hard-revert. The stethBal pipeline correctly tracks balances across sequential transfers. No silent failure or reentrancy path exists.

## SWEP-04: VRF Shutdown and LINK Recovery

**Requirement:** VRF shutdown calls `admin.shutdownVrf()` with fire-and-forget try/catch. `shutdownVrf` cancels VRF subscription and sweeps LINK to vault.

### Trace

**1. Fire-and-forget call (GameOverModule L197)**

```solidity
try admin.shutdownVrf() {} catch {}                                  // L197
```

- `try ... catch {}`: failure is silently caught. VRF shutdown failure does NOT block the sweep. OK.
- `admin` is `IDegenerusAdminShutdown(ContractAddresses.ADMIN)` -- immutable constant pointing to the Admin contract. OK.

**2. msg.sender context verification**

`handleFinalSweep` is called via delegatecall from Game (AdvanceModule L492-496):

```solidity
(ok, data) = ContractAddresses.GAME_GAMEOVER_MODULE.delegatecall(   // L492
    abi.encodeWithSelector(
        IDegenerusGameGameOverModule.handleFinalSweep.selector       // L494
    )
);
```

In delegatecall context, `handleFinalSweep` executes in Game's storage and with Game's `address(this)`. When it makes the external call `admin.shutdownVrf()` at L197, `msg.sender` in `shutdownVrf` is `address(this)` of the delegatecall context = Game's address (`ContractAddresses.GAME = 0x3381cD18e2Fb4dB236BF0525938AB6E43Db0440f`).

`shutdownVrf` checks at Admin L945:

```solidity
if (msg.sender != ContractAddresses.GAME) revert NotAuthorized();    // L945
```

Since `msg.sender == ContractAddresses.GAME`, the auth check passes. OK.

**3. Subscription cancellation (Admin L946-954)**

```solidity
uint256 subId = subscriptionId;                                      // L946
if (subId == 0) return;                                              // L947
subscriptionId = 0;                                                  // L949
address target = ContractAddresses.VAULT;                            // L950
try IVRFCoordinatorV2_5Owner(coordinator)                            // L952
    .cancelSubscription(subId, target) {
    emit SubscriptionCancelled(subId, target);                       // L953
} catch {}                                                           // L954
```

- L946-947: Reads `subscriptionId`. If zero (already cancelled or never set), returns as no-op. OK.
- L949: Zeroes `subscriptionId` BEFORE the cancellation call. This prevents reentrancy: if `cancelSubscription` somehow re-entered `shutdownVrf`, the `subId == 0` check at L947 would cause an immediate return. OK.
- L950: Target for LINK refund is `ContractAddresses.VAULT`. OK.
- L952: `cancelSubscription(subId, target)` -- Chainlink VRF coordinator function. Cancels subscription and sends any LINK balance held by the coordinator for this subscription to `target` (vault). OK.
- L953-954: Success emits `SubscriptionCancelled`. Failure is silently caught. OK.

**4. Direct LINK sweep (Admin L956-964)**

```solidity
uint256 bal = linkToken.balanceOf(address(this));                    // L956
if (bal != 0) {                                                      // L957
    try linkToken.transfer(target, bal) returns (bool ok) {          // L958
        if (ok) {                                                    // L959
            emit SubscriptionShutdown(subId, target, bal);           // L960
            return;                                                  // L961
        }
    } catch {}                                                       // L963
}
emit SubscriptionShutdown(subId, target, 0);                         // L966
```

- L956-957: Checks if Admin holds any LINK directly (not from coordinator, but from prior donations or funding). If zero, skip. OK.
- L958: Try to transfer LINK to vault. OK.
- L959-961: If transfer returns `true`, emit success event with amount and return. OK.
- L963: If transfer reverts, catch silently. OK.
- L966: Fallback emit: always emits `SubscriptionShutdown` with amount=0 if LINK transfer failed or balance was zero. This ensures event emission in all paths. OK.

**5. LINK recovery completeness**

Two sources of LINK:
1. **Coordinator-held**: Refunded to vault via `cancelSubscription(subId, target)` at L952. The coordinator sends LINK directly to `target` (vault).
2. **Admin-held**: Swept to vault via `linkToken.transfer(target, bal)` at L958.

Both sources target `ContractAddresses.VAULT`. OK.

**6. Failure mode analysis**

If `cancelSubscription` fails (coordinator reverts):
- Caught by try/catch at L954. Coordinator LINK is NOT recovered in this case.
- But the subscription is already invalidated (`subscriptionId = 0` at L949), so no future VRF requests can be made.
- LINK held by the coordinator for the cancelled sub may be lost if the coordinator's cancellation fails. However, Chainlink's V2.5 coordinator is a trusted external contract, and cancellation failure is an extreme edge case (e.g., coordinator upgraded or paused).

If `linkToken.transfer` fails:
- Caught by try/catch at L963. Admin-held LINK is NOT recovered.
- `SubscriptionShutdown` event still emits with amount=0, signaling the failure.
- Since `subscriptionId = 0`, the function cannot be called again (no-op at L947). However, LINK could be recovered via other admin functions if needed.

Both failure modes are acceptable: they are fire-and-forget, non-fatal to the sweep, and involve extreme edge cases with trusted external contracts. OK.

**7. Idempotency**

After first call: `subscriptionId = 0`. Any subsequent call returns at L947 (no-op). Cannot double-cancel or double-sweep LINK. OK.

### Findings

| # | Class | Description |
|---|-------|-------------|
| 1 | OK | Fire-and-forget try/catch at GameOverModule L197 ensures VRF failure does not block sweep |
| 2 | OK | `msg.sender` in `shutdownVrf` is Game's address (delegatecall context), passes auth check at Admin L945 |
| 3 | OK | `subscriptionId` zeroed at L949 before cancellation call, prevents reentrancy and double-cancel |
| 4 | OK | LINK recovered from two sources: coordinator refund (L952) and admin balance (L956-958), both to vault |
| 5 | OK | Fallback event emission at L966 ensures auditability even on failure |
| 6 | NOTE | If coordinator's `cancelSubscription` reverts, coordinator-held LINK is not recovered. Extreme edge case with trusted Chainlink infrastructure. |
| 7 | OK | Idempotent: `subscriptionId = 0` after first call, subsequent calls are no-ops at L947 |

### Verdict: PASS

VRF shutdown is fire-and-forget with comprehensive try/catch. `shutdownVrf` correctly zeroes state before external calls, recovers LINK from both sources (coordinator + admin balance), and is idempotent. The delegatecall context correctly resolves `msg.sender` to Game's address for auth.
