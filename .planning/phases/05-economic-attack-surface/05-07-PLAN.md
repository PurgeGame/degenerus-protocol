<objective>
Audit afKing mode transitions for double-spend or double-credit windows. AfKing mode enables enhanced auto-rebuy behavior with different recycling rates (1.6% vs 1%). The concern is whether toggling afKing mode can create windows where a player receives both the afKing-enhanced and standard-mode benefits, or where pending coinflip state is handled inconsistently across the mode transition.

Purpose: ECON-07 requires proving that afKing mode transitions do not create windows for double-spend or double-credit. The three-layer protection (settleFlipModeChange, rngLockedFlag, 5-level lock) should prevent exploitation, but each layer needs verification.

Output: `05-07-FINDINGS-afking-mode-transitions.md` containing the transition audit and ECON-07 verdict.
</objective>

    Activation (enabled=true):

   - afKingMode=true enables new behavior
   - If there is a reentrant callback from settleFlipModeChange, could it observe the old mode while executing settlement logic?

**2. Deactivation Path Trace**

Read `setAfKingMode` with `enabled=false`:

   - settleFlipModeChange before mode change

**3. External Deactivation Hooks**

Read `deactivateAfKingMode` (line ~1726):
a. Who can call this? Expected: coinflip or coin contracts.
b. Does it call settleFlipModeChange? Or does it directly set afKingMode=false?
c. If it does NOT call settleFlipModeChange: is there a risk that pending state is not settled?

Read `syncAfKingLazyPass` (line ~1738):
a. This checks if the lazy pass is still active. If not, revokes afKing.
b. Does it settle pending state?
c. Can an expired lazy pass create an inconsistency?

**4. settleFlipModeChange Analysis**

Read BurnieCoinflip.settleFlipModeChange(player) (line 221):
a. What state does it settle? Expected: any pending coinflip bets or auto-rebuy state.
b. Does it make external calls? If so, to what? Is there reentrancy risk?
c. Is the settlement idempotent? Can calling it twice cause issues?
d. Does it correctly handle the case where there is no pending state?

**5. Double-Spend Scenarios**

Enumerate potential double-spend scenarios:

a. **Scenario 1: Toggle afKing to get enhanced recycling then immediately toggle back**
   - Prevented by: 5-level lock. After activation, player cannot deactivate for 5 levels.
   - Verify: is the lock strictly enforced? Can the player bypass it?

b. **Scenario 2: Activate afKing while coinflip is in-flight**
   - Prevented by: rngLockedFlag check. Cannot activate during VRF processing.
   - But: is rngLockedFlag checked at the RIGHT point? If rngLocked is checked early but a VRF callback arrives mid-execution...
   - This should be impossible in EVM (atomic transactions), but verify.

c. **Scenario 3: Pending auto-rebuy state from pre-afKing processed at afKing rates**
   - Prevented by: settleFlipModeChange called before mode change.
   - Verify: all pending state is settled BEFORE the flag changes.

d. **Scenario 4: deactivateAfKingMode called externally bypassing settleFlipModeChange**
   - If deactivateAfKingMode does NOT call settleFlipModeChange:
   - Pending afKing-rate auto-rebuys might be processed AFTER deactivation at the wrong rate.
   - Or: pending state at standard rate might be processed at afKing rate.

e. **Scenario 5: Lazy pass expires mid-level during afKing**
   - syncAfKingLazyPass revokes afKing if pass expired.
   - Does it settle pending state? If not, there could be a brief window.

**6. Auto-Rebuy Rate Consistency**

a. Where is the 1.6% vs 1% rate applied? In BurnieCoinflip when processing auto-rebuys?
b. Does BurnieCoinflip read afKingMode from DegenerusGame at the time of processing, or is the rate cached at bet time?
c. If read at processing time: no double-credit risk (always uses current mode).
d. If cached at bet time: mode change could create inconsistency.

**7. ECON-07 Verdict**

- Is it possible or prevented?
- Which protection layer (settleFlipModeChange, rngLockedFlag, 5-level lock) provides the defense?
- Are there any gaps in the three-layer protection?
