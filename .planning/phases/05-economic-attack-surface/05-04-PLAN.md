Analyze the MEV attack surface on ticket purchase price escalation at phase/level boundaries. Standard MEV attacks (sandwich, frontrunning) rely on price being affected by transaction ordering. Since ticket pricing is a pure function of game level (PriceLookupLib), and level transitions require a multi-day VRF cycle, this analysis determines whether any profitable sandwich or reorder strategy exists.

Purpose: ECON-04 requires proving that MEV/sandwich attacks on ticket purchase price escalation cannot extract value at phase boundaries. The key insight is that prices are deterministic from level state, not from transaction ordering within a block.

**1. Price Determinism Verification**

a. Read PriceLookupLib.sol (all 47 lines). Confirm it is a `pure` function with no state reads (no `sload`, no storage variable access).
b. Read MintModule.purchase() to identify which price is used for ticket cost calculation. Research notes there are TWO pricing systems:
   - PriceLookupLib.priceForLevel(level): pure, deterministic
   - `price` storage variable: set in AdvanceModule during advanceGame()
   Which one is used for `costWei` calculation in purchase()? This is critical -- if `price` is the storage variable, then advanceGame() changes the price, creating a sandwich window around advanceGame calls.
c. Document the price source with exact line numbers.

**2. Sandwich Attack on advanceGame()**

a. If purchase() uses the `price` storage variable (updated by advanceGame):
   - Scenario: Block proposer sees pending advanceGame() tx that will increase `price`.
   - Attack: Buy tickets BEFORE advanceGame (at old, lower price), then those tickets are in the prize pool at the new (higher) level.
   - Profit condition: tickets bought at old price compete for jackpot funded by new-price tickets.

b. Analyze whether this cross-level arbitrage is profitable:
   - Tickets bought at level N are in level N's queue. When level N+1 starts, do level N tickets compete for level N+1's pool?
   - Or are they settled at level N's transition and level N+1 starts fresh?
   - Trace the level transition in AdvanceModule to determine how tickets are handled.

c. If purchase() uses PriceLookupLib directly (pure function of level):
   - No sandwich is possible because the price changes atomically with the level.
   - Buying before advanceGame pays level N price, buying after pays level N+1 price.
   - No arbitrage: you pay the correct price for whichever level you buy at.

**3. Transaction Ordering Within a Level**

a. Within a single level, all tickets are priced the same (step-function pricing, not AMM curve).
b. Does the order of ticket purchases within a level matter?
   - No slippage: each ticket costs the same regardless of volume.
   - No price impact: buying 100 tickets does not change the price for subsequent buyers.
c. Can a frontrunner extract value by buying before another user within the same level?
   - Only if there is a limited resource (e.g., last-ticket-wins for jackpot). Analyze if ticket ordering affects jackpot probability.
   - Tickets are added to a queue. Does position in the queue affect payout probability?

**4. advanceGame Timing Control**

a. advanceGame() is permissionless -- anyone can call it.
b. The day-index gate prevents multiple calls per day.
c. Can a block proposer delay advanceGame by not including it in their block?
   - Yes, but only for ~12 seconds (one block). The next proposer can include it.
   - Any user can submit advanceGame at the next block.
   - Delay does not change the outcome -- it only delays when it happens.
d. Can a block proposer include their own purchase BEFORE advanceGame in the same block?
   - Yes. They buy at level N price, then advanceGame transitions to N+1.
   - But: tickets bought at level N are queued for level N. They compete at level N's prize pool.
   - No cross-level arbitrage unless ticket queuing allows level N tickets to participate in level N+1.

**5. Lootbox MEV**

a. Lootbox opening uses VRF randomness. No MEV on the randomness itself (Phase 2 confirmed).
b. Can a block proposer see the VRF fulfillment and front-run lootbox opening?
   - VRF fulfillment calls rawFulfillRandomWords which stores the word. Opening a lootbox later uses that stored word combined with entropy stepping.
   - The player does not choose which word is used -- it is determined by the contract state.
c. No meaningful lootbox MEV vector.

**6. Whale Bundle / Deity Pass Purchase MEV**

a. These have fixed prices (not level-dependent in the sandwich sense). No MEV on pricing.
b. The only MEV concern: deity pass has escalating price based on `deityPassOwners.length`. Frontrunning another buyer means the frontrunner gets a cheaper pass.
   - But there are only 32 deity pass symbols. The pool of potential sandwich targets is tiny.
   - And the price difference between consecutive passes (T(k) - T(k-1) = k ETH) makes frontrunning costly.

**7. ECON-04 Verdict**

Synthesize all MEV vectors:
- Price determinism from PriceLookupLib eliminates standard AMM-style sandwich
- Level transitions are multi-day VRF-gated events, not atomic
- Transaction ordering within a level has no price impact
- Cross-level ticket arbitrage analysis
- Deity pass frontrunning bounded by tiny pool and escalating costs

ECON-04 Verdict: PASS/FAIL with specific reasoning per vector.
