**GOV-09: Proposal Expiry Analysis**

1. Trace expiry logic (lines 449-453):
   - Triggered during `vote()` call on an expired proposal
   - Condition: `block.timestamp - uint256(p.createdAt) >= PROPOSAL_LIFETIME` (168h)
   - State transition: `p.state = ProposalState.Expired`
   - Counter decrement: `unchecked { activeProposalCount--; }`
   - Then: `revert ProposalExpired()` -- transaction reverts, so the voter gets gas refund on remaining gas

2. Adversarial check -- passive expiry:
   - Expiry is LAZY -- only triggered when someone tries to vote. A proposal can sit expired without being cleaned up.
   - Impact: `activeProposalCount` remains inflated until someone votes on the expired proposal or `_voidAllActive` runs.
   - This means `anyProposalActive()` could return true even after all proposals have technically expired. The death clock stays paused even though governance is effectively dead.
   - Severity: LOW -- someone just needs to call `vote()` on the expired proposal to trigger cleanup. Or a new proposal + execution would trigger `_voidAllActive`.

3. Adversarial check -- expiry vs threshold returning 0:
   - `threshold()` returns 0 at elapsed >= 168h. But vote() reverts before reaching the threshold check (line 449 runs before line 485). So threshold=0 is unreachable via vote(). Consistent.
   - `canExecute()` checks `>= PROPOSAL_LIFETIME` at line 542 and returns false. Consistent.

4. Cite existing expiry test from VRFGovernance.test.js.

**VOTE-01: sDGNRS Supply Frozen During VRF Stall (MOST CRITICAL)**

This is the foundational invariant. If sDGNRS balances can change during governance, vote arithmetic is broken.

1. Exhaustively enumerate ALL sDGNRS balance-mutation paths. Read the StakedDegenerusStonk contract (find it at contracts/StakedDegenerusStonk.sol or similar) and list every function that changes a balance:

   a) **Minting** (increasing totalSupply):
      - `advanceGame` -> various pool distributions mint sDGNRS
      - During VRF stall: `advanceGame` requires a VRF word. No VRF word = no advance = no minting. BLOCKED.

   b) **unwrapTo** (DegenerusStonk -> StakedDegenerusStonk):
      - `DegenerusStonk.unwrapTo()` calls `stonk.wrapperTransferTo(recipient, amount)` which moves sDGNRS from DGNRS contract to recipient
      - During VRF stall (>20h): unwrapTo has stall guard `if (block.timestamp - lastVrfProcessed() > 20 hours) revert`. BLOCKED.
      - BUT: At exactly 20h, `20h > 20h` is FALSE, so unwrapTo is ALLOWED. This is the 1-second window analyzed in WAR-04. Note but defer.

   c) **burn** (DegenerusStonk.burn -> burns sDGNRS backing):
      - Burns DGNRS tokens, which triggers proportional burn of sDGNRS reserves
      - During VRF stall: burn is NOT blocked by stall guard. Burning DGNRS reduces sDGNRS.totalSupply and sDGNRS.balanceOf(DGNRS).
      - circulatingSupply = totalSupply - balanceOf(SDGNRS) - balanceOf(DGNRS). If both totalSupply and balanceOf(DGNRS) decrease by the same amount, circulatingSupply is UNCHANGED. Verify the proportionality.
      - Impact on individual voter balances: burn does NOT change any voter's sDGNRS balance (it burns from the DGNRS contract's held sDGNRS, not from voter wallets). Voters' sDGNRS balances are UNCHANGED by burn.

   d) **Transfers** (sDGNRS is soulbound):
      - Standard ERC20 transfers are overridden to revert (soulbound). Verify the transfer/transferFrom override.
      - No peer-to-peer sDGNRS movement possible. BLOCKED.

   e) **Any other path?**
      - Check for admin mint functions, emergency functions, or other paths that modify sDGNRS balances
      - grep for all functions in StakedDegenerusStonk that call `_mint`, `_burn`, `_transfer`, or modify `_balances`

2. Conclusion: During VRF stall (>20h), sDGNRS individual balances are frozen because:
   - No minting (no advanceGame without VRF)
   - No unwrapTo (stall guard)
   - No transfers (soulbound)
   - burn only affects DGNRS contract's sDGNRS, not voter wallets

**VOTE-02: circulatingSnapshot Immutability**

1. `circulatingSnapshot` is stored in `Proposal` struct at creation (line 424: `p.circulatingSnapshot = circulatingSupply()`).
2. Search for ALL writes to `circulatingSnapshot` after creation:
   - `p.circulatingSnapshot` is a struct field in storage. Only written in `propose()`.
   - In `vote()`, `_executeSwap()`, `_voidAllActive()` -- only `p.state`, `p.approveWeight`, `p.rejectWeight` are modified. Never `p.circulatingSnapshot`.
   - Verify by grepping: `circulatingSnapshot` should appear only in `propose()` (write) and in `vote()`/`canExecute()` (read).
3. Adversarial check: Can a malicious proposal trick the snapshot?
   - The snapshot is taken from `circulatingSupply()` at creation time. An attacker could burn sDGNRS from DGNRS wrapper BEFORE proposing to lower the snapshot, then the threshold comparison uses a lower denominator. But burning reduces both numerator (total supply) and the DGNRS balance (excluded), so circulatingSupply is unchanged. Verify this mathematically.

**VOTE-03: uint8 activeProposalCount Overflow Analysis**

1. `activeProposalCount` is `uint8` (max 255). `unchecked { activeProposalCount++; }` wraps 255 -> 0.
2. Impact if overflow to 0:
   - `anyProposalActive()` returns `activeProposalCount > 0` -> returns FALSE
   - Death clock in `_handleGameOverPath()` calls `anyProposalActive()`. If FALSE, death clock is NOT paused.
   - Result: Game could enter game-over while 256+ proposals are Active. Governance is active but the game dies.
3. Exploitability analysis:
   - Requires 256 `propose()` calls during a single VRF stall
   - Admin path: each call needs DGVE >50.1% and stall >= 20h. Cost: ~100k gas per propose * 256 = ~25.6M gas (~8 transactions at 3M gas each). FEASIBLE if admin is attacker.
   - Community path: each call needs 0.5% sDGNRS and stall >= 7d. Single community member could call propose 256 times. FEASIBLE.
   - But: each proposal has a different proposalId, different coordinator/keyHash. Attacker must supply 256 coordinator addresses and keyHashes. Trivially possible (deploy 256 dummy contracts or use same address 256 times).
4. Severity: LOW-MEDIUM. Requires sustained VRF failure AND motivated attacker with DGVE or 0.5% sDGNRS. The primary harm is unpausing the death clock, but the death clock has its own timeout (120 days at non-zero level). In practice, a VRF stall long enough for 256 proposals likely already has other problems.
5. Mitigation suggestion: Change `uint8` to `uint16` or add a `require(activeProposalCount < 255)` check. Note as KNOWN-ISSUE if verdict is not FAIL.
