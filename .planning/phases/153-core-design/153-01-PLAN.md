## Document Structure

### 1. Eligibility (per D-01, D-02, D-03 — addresses ELIG-01, ELIG-02)

Specify the exact boolean expression:

```
eligible = (
    (levelStreak >= 5)  // mintPacked_[player] bits 48-71
    OR
    (deityPassCount[player] > 0)  // separate mapping in DegenerusGameStorage
    OR
    (frozenUntilLevel > 0 AND whaleBundleType != 0)  // mintPacked_ bits 128-151 and 152-153
)
AND
(
    levelUnits >= 4  // mintPacked_[player] bits 228-243
    // BUT only valid when unitsLevel == current level (mintPacked_ bits 104-127)
)
```

For each storage read:
- Name the exact mapping/variable
- Specify the bit positions and mask operations
- Count the SLOADs (mintPacked_ = 1 SLOAD, deityPassCount = 1 SLOAD = 2 SLOADs worst case, 1 SLOAD best case if mintPacked_ short-circuits)
- Note: levelUnits is ONLY valid when unitsLevel matches current game level. If unitsLevel != level, the player has not minted this level, so levelUnits = 0 effectively.

Estimate gas cost: 2 cold SLOADs worst case = ~4200 gas (2100 each), 1 cold SLOAD best case = ~2100 gas.

### 2. Global Quest Roll Mechanism (per D-04, D-05, D-06 — addresses MECH-01)

Specify:
- **When:** During advanceGame level transition, specifically inside `_processPhaseTransition(purchaseLevel)` BEFORE `phaseTransitionActive = false`. This is when the new level's purchase phase begins. The VRF word (`rngWord`) is already available at this point (rngGate returned it earlier in the same advanceGame call).
- **VRF entropy source:** Use `keccak256(abi.encodePacked(rngWordByDay[day], "LEVEL_QUEST"))` to derive independent entropy from the daily VRF word. This follows the existing pattern where "full" consumers use keccak mixing (see VRF bit allocation map comment in AdvanceModule lines 790-808).
- **Quest type selection:** Reuse the exact same weight table from `_bonusQuestType` in DegenerusQuests.sol:
  - MINT_BURNIE: weight 10
  - MINT_ETH: weight 1
  - FLIP: weight 4
  - AFFILIATE: weight 1
  - RESERVED (type 4): weight 0 (skipped)
  - DECIMATOR: weight 4 (when decWindowOpen && eligible level, else 0)
  - LOOTBOX: weight 3
  - DEGENERETTE_ETH: weight 1
  - DEGENERETTE_BURNIE: weight 1
  - Total eligible weight (no decimator): 21. With decimator: 25.
  - Selection: `entropy % totalWeight`, scan weights in order.
- **Storage:** Quest type (uint8) and level (uint24) stored in a global mapping. Specify the exact mapping signature and packing.

### 4. Per-Player Progress Tracking (per D-09, D-10, D-13 — addresses MECH-03)

Design the storage following `questPlayerState` pattern:

```solidity
struct LevelQuestPlayerState {
    uint24  questLevel;      // Level this progress belongs to (invalidation key)
    uint128 progress;        // Accumulated progress toward target
    bool    completed;       // Once-per-level completion flag
}

mapping(address => LevelQuestPlayerState) internal levelQuestPlayerState;
```

**Level-boundary invalidation (per D-10):**
- When a handler reads `levelQuestPlayerState[player]`, check `questLevel != currentLevel`.
- If mismatch: progress resets to 0, completed resets to false, questLevel updates to currentLevel.
- This is simpler than daily quest version-based invalidation because levels are global and monotonic (no mid-level re-rolls).
- Document: use level-based check (not version counter) because level quests cannot re-roll mid-level.

**Why not version-based:** Daily quests use version counters because quests can change within a day (edge case). Level quests roll once per level and never change. Level comparison is cheaper (1 fewer storage read) and simpler.

**Packing opportunity:** `questLevel` (24 bits) + `progress` (128 bits) + `completed` (1 bit) = 153 bits. Fits in a single uint256. Specify the packed layout:
```
Bits 0-23:   questLevel (uint24)
Bits 24-151: progress (uint128)
Bit 152:     completed (bool)
Bits 153-255: unused (available for future use)
```
This means `levelQuestPlayerState` can be a `mapping(address => uint256)` for single SLOAD/SSTORE.

### 5. Global Per-Level Quest Storage (per D-06, D-14 — addresses STOR-01)

```solidity
mapping(uint24 => uint8) internal levelQuestType;
```

Store only the quest TYPE per level. The target is derived from the type at evaluation time (same as daily quests use `_questTargetValue`). This saves storage vs storing the target.

Rationale for not storing target: targets are deterministic from type + mintPrice. Storing the target would require an extra SSTORE at roll time and wouldn't save any SLOAD at evaluation time (still need mintPrice for ETH-based types).

SLOAD count: 1 cold SLOAD to read quest type for a level.

### 6. Completion Flow (per D-11, D-12 — addresses MECH-04)

Specify the exact completion sequence:
1. Handler detects `progress >= target`
2. Check `completed == false` in packed state (once-per-level guard per D-11)
3. Set `completed = true` in packed state
4. Call `coinflip.creditFlip(player, 800 ether)` (per D-11: 800 BURNIE payout)
5. Emit a `LevelQuestCompleted(player, level, questType, reward)` event

**No slot system (per D-12):** Unlike daily quests which have slot 0 and slot 1 with completion masks, level quests have exactly ONE quest. No completion mask needed -- just the single `completed` bool.

**No streak interaction:** Level quest completion does NOT affect daily quest streak. Systems are completely independent (per D-09).

**creditFlip details:** `coinflip.creditFlip(player, 800 ether)` credits 800 BURNIE (800e18 base units) as flip stake to the player's next coinflip day. The `onlyFlipCreditors` modifier must include whatever contract calls this. If called from DegenerusQuests, the quest contract must be in the flip creditors whitelist. If called from DegenerusGame via delegatecall module, it inherits the game address.

### 7. Storage Layout Summary (addresses STOR-01, STOR-02)

**New storage slots required:**

| Variable | Type | Slot Impact | Location |
|---|---|---|---|
| `levelQuestType` | `mapping(uint24 => uint8)` | 1 new mapping slot | DegenerusQuests.sol (if level quests live there) OR DegenerusGameStorage (if via module) |
| `levelQuestPlayerState` | `mapping(address => uint256)` | 1 new mapping slot | Same as above |

**SLOAD budget per operation:**

| Operation | SLOADs | Detail |
|---|---|---|
| Eligibility check | 1-2 | mintPacked_ (1) + deityPassCount (0-1, only if mintPacked_ doesn't short-circuit) |
| Quest roll (level transition) | 1 | rngWordByDay[day] (already loaded), levelQuestType write = 1 SSTORE |
| Progress handler (hot path) | 2 | levelQuestType[level] (1) + levelQuestPlayerState[player] (1) |
| Completion check + write | 1 | levelQuestPlayerState[player] (already loaded from progress check, 1 SSTORE to update) |

**SSTORE budget:**
- Roll: 1 SSTORE (levelQuestType[level])
- Progress update: 1 SSTORE (levelQuestPlayerState[player])
- Completion: 1 SSTORE (same packed word, already updated in progress path)

**Collision analysis:**
- Both new mappings occupy their own keccak256-derived slot spaces
- No collision with existing storage (mappings are isolated by definition)
- If placed in DegenerusQuests.sol: append after existing `questVersionCounter` -- new slots, no reorder
- If placed in DegenerusGameStorage.sol: append at end of storage layout -- new slots, no reorder
