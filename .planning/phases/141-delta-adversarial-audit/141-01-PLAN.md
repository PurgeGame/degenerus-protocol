**Backfill cap safety proof (DELTA-03):**

Analyze: if a VRF stall exceeds 120 days, `_backfillGapDays` processes the OLDEST 120 days and skips the rest. What happens to the skipped days?

1. Trace the caller (_handleDayRng line 804-807): after _backfillGapDays returns, dailyIdx is NOT updated inside _backfillGapDays — it is updated later in _applyDailyRng (the normal daily path). So after a >120 day stall:
   - Days [idx+1, idx+120] get backfilled (cap applied)
   - Day `day` (current) gets processed by _applyDailyRng
   - Days [idx+121, day-1] are NOT backfilled in this call

2. Check: does dailyIdx advance past the gap? Read _applyDailyRng to see if it sets dailyIdx = day. If yes, then on the NEXT advanceGame call, the gap detection `day > idx + 1` would NOT catch the missed days [idx+121, old_day-1] because dailyIdx jumped ahead.

3. Read _applyDailyRng carefully. Find where dailyIdx is set. If dailyIdx = day after processing, the skipped gap days [121..day-1] would have rngWordByDay[gapDay] == 0 permanently. Trace what happens when rngWordByDay is 0 for a day that had active coinflips:
   - processCoinflipPayouts was never called for those days
   - Players with coinflips on those days: can they still claim? Or are payouts lost?
   - Check BurnieCoinflip: does it have a fallback for unresolved days, or do claims revert?

4. Evaluate realistic scenarios: Is a >120 day VRF stall plausible? The VRF stall recovery already extends levelStartTime by gap duration (line 815), preventing game-over timeout. A 120-day stall would require Chainlink VRF to be completely down for 4 months. Document the probability assessment.

5. Check: could an attacker deliberately cause a >120 day gap? The gap is between the last successful VRF fulfillment and the next advanceGame call. VRF requests are permissionless (anyone can call advanceGame). The gap grows only if NO advanceGame call succeeds for 120+ days AND VRF is stalled. This requires sustained VRF outage, not just attacker inaction.

6. If skipped days DO lose coinflip payouts: classify severity. Quantify worst case (maximum pending coinflip value at day 121+). Note this only matters if VRF is down 120+ days continuously. If the risk is INFO-level (extreme edge case with Chainlink failure), document as such. If it could affect real users under plausible conditions, flag as finding.
