the implementing contract does. Any divergence between interface NatSpec and actual behavior is a
finding, even if the implementing contract's own comments are accurate.

  <action>
Read all 5 library files in full. For every comment (single-line `//`, block `/* */`, and
NatSpec `///` / `/** */`), verify that the comment accurately describes the code.

Focus areas for BitPackingLib (103 lines):

1. **Bit layout table in header** — The library header contains a detailed bit layout map.
   Verify each row of the table:
   - Bit range is arithmetically correct given the constant value (e.g., LEVEL_COUNT_SHIFT=24,
     24-bit mask → range [24-47]; AFFILIATE_BONUS_LEVEL_SHIFT=185, 24-bit mask → [185-208])
   - Description matches what the field is actually used for
   - Width annotation (24 bits, 32 bits, etc.) matches the mask constant used with that shift
   - Unused ranges ([154-159], [215-227]) are confirmed unused (no constants defined in those positions)
   - [244-255] "(reserved)" — verify no constants fall in that range

2. **Individual constant `@notice` comments** — Each constant (LAST_LEVEL_SHIFT, LEVEL_COUNT_SHIFT,
   etc.) has a `@notice` describing bit position and range. Verify that the description matches
   the constant's numeric value and the header table. For example, if a constant says
   "bits 185-208" in its notice but the table says "bits 185-208" and the value is 185 with
   a 24-bit mask, that is consistent. If any of the three diverge, that is a finding.

3. **`setPacked` function NatSpec** — Verify `@param` names match actual parameter names in the
   function signature. Verify the formula comment matches the actual implementation:
   `(data & ~(mask << shift)) | ((value & mask) << shift)`

Focus areas for JackpotBucketLib (307 lines):

1. **Bucket definition comments** — Any comment describing jackpot bucket thresholds, sizes,
   payout percentages, or tier names. Verify against actual constants or struct values in the file.

2. **Function NatSpec** — Every `@notice`, `@param`, `@return`, `@dev`. Verify accuracy.
   Pay special attention to `@return` descriptions on functions returning multiple values.

3. **Inline math comments** — Any `// e.g.`, `// 25%`, `// threshold:`, `// bps` comments.
   Verify the stated percentages or values match actual constants.

Focus areas for PriceLookupLib (47 lines):

1. **Price source comments** — PriceLookupLib replaced storage-based price reads in v14.0.
   Verify that any comment describing where prices come from (passed as argument, oracle call,
   constant, storage read) accurately reflects the current implementation. If the old mechanism
   was a storage SLOAD and the new one is argument-passing or an external oracle call, any
   comment still describing the old mechanism is a discrepancy.

2. **Function NatSpec** — Every `@notice`, `@param`, `@return`. Verify parameter names and
   return value descriptions match the actual function signature and behavior.

Focus areas for EntropyLib (24 lines) and GameTimeLib (35 lines):

1. **Function NatSpec** — These are small utility libraries. Verify every `@notice`, `@param`,
   `@return` tag matches the actual function signature and behavior.

2. **Inline comments** — Any inline math or logic comments (bit operations, time constants).
   Verify accuracy.

For each discrepancy found, record:
- Severity: LOW (misleads about bit field ranges, price source, bucket thresholds, or function
  behavior) or INFO (minor imprecision, stale wording, typo)
- Location: library filename + line N (or line range N-M)
- What the comment says: exact quote or close paraphrase
- What the code actually does: plain-English correct description

Create `.planning/phases/177-infrastructure-libraries-misc-comment-sweep/177-03-FINDINGS.md` with
