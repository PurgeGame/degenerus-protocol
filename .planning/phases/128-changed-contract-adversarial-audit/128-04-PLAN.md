**STEP 1 — MAD GENIUS (Attacker)**

For each function, apply the full Mad Genius methodology:

1. Read the function line-by-line
2. Build complete call tree
3. Map every storage write
4. Attack from every angle per ULTIMATE-AUDIT-DESIGN.md
5. Explicit BAF-class cached-local-vs-storage check (AUDIT-04)

**Special focus areas for the "every address is an affiliate" change:**

**defaultCode (#1) — Key New Function:**
- How is the deterministic code computed from an address? (likely keccak256 or truncation)
- Can two different addresses produce the same default code? (collision analysis)
- Can a malicious address be crafted to produce a specific default code that matches an existing custom code?

**_resolveCodeOwner (#2) — Key New Function:**
- When a code is looked up, how does it decide between custom code owner vs default code owner?
- Priority: does custom always override default? What if someone creates a custom code that matches another address's default code?

**payAffiliate (#5) — ETH Flow:**
- Does the default code system change how ETH flows to affiliates?
- Can an attacker set up default referral codes to redirect affiliate fees?
- Can the default code system be used for self-referral (referring yourself to capture fees)?
- What happens if a player is referred by a default code and the code owner hasn't claimed their code — where does the ETH go?

**createAffiliateCode (#3) — Code Ownership:**
- Can someone create a custom code that hijacks another address's default code?
- What happens if address A's default code is "0xABC..." and someone creates custom code "0xABC..."?
- Is there a race condition between default code usage and custom code creation?

**referPlayer (#4) — Referral Manipulation:**
- Can a player switch referrers by using different default codes?
- Can a player refer themselves using their own default code?
- What happens if referPlayer is called with the player's own default code?

6. Verdict per function: VULNERABLE / INVESTIGATE / SAFE
