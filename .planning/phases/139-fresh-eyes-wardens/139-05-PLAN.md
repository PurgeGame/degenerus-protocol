    Act as an adversarial C4A warden specializing in composition attacks and cross-domain exploit chains. You have ZERO knowledge of any prior audit — approach the codebase with completely fresh eyes. Read the C4A README and KNOWN-ISSUES first.

    Your unique value: You look for attacks that span multiple domains and contracts simultaneously. A single-domain warden might see each piece as safe individually, but the combination could be exploitable.

    Systematically audit every composition attack surface:

    **1. Delegatecall Module Seam Analysis**
    DegenerusGame uses delegatecall to route to ~12 modules, all sharing DegenerusGameStorage:
    - Map which storage slots each module reads and writes
    - Identify slots written by Module A that are read by Module B during the same transaction
    - Check for state corruption: can Module A leave storage in a state that causes Module B to behave unexpectedly?
    - Check for reentrancy via module seams: can an external call in Module A re-enter via Module B?
    - Verify the delegatecall routing cannot be manipulated (function selector collisions, fallback behavior)

    **2. Cross-Domain Attack Sequences**
    Test these specific multi-step attack patterns:

    RNG + Money:
    - Can a player manipulate RNG outcome to guarantee winning a jackpot?
    - Can VRF fulfillment timing be exploited to extract value?
    - Can lootbox RNG manipulation yield disproportionate rewards?

    Admin + Gas:
    - Can admin set parameters that make advanceGame exceed gas limit?
    - Can admin configuration changes brick the protocol?
    - Can a governance proposal + gas manipulation prevent legitimate day advancement?

    RNG + Admin:
    - Can admin influence which VRF coordinator is used to control RNG?
    - Can governance voting be timed around VRF fulfillment for advantage?

    Money + Gas:
    - Can many small transactions accumulate state that makes payouts fail?
    - Can ticket queue growth prevent day advancement?

    Money + Admin:
    - Can admin drain funds via parameter manipulation?
    - Can admin redirect affiliate fees to themselves?

    **3. State Transition Attack Chains**
    - Map critical state transitions (game start -> active -> endgame -> gameover)
    - Check each transition for exploitable race conditions
    - Can a player force a premature state transition?
    - Can state transitions be blocked or reversed?

    **4. Flash Loan and Sandwich Attacks**
    - Can flash loans affect governance votes (sDGNRS voting power)?
    - Can sandwich attacks extract value from game transactions?
    - Can MEV bots extract value from advanceGame or jackpot distributions?

    **5. Reentrancy Across Contract Boundaries**
    - Map all external calls across all contracts
    - For each external call: what state has been modified, what state hasn't yet?
    - Can a reentrant call via one contract exploit state in another?
    - Check CEI (Checks-Effects-Interactions) pattern compliance across module boundaries

    **6. Token Interaction Chains**
    - DGNRS -> sDGNRS -> governance -> admin action: full chain assessment
    - BURNIE -> coinflip -> ETH payout: cross-token value extraction
    - GNRUS -> burn -> ETH/stETH redemption: composition with vault state
    - Can interleaving token operations create inconsistent state?

    Write results with structure:
    - Executive Summary
    - Methodology (composition attack approach)
    - Module Seam Map (table: Module A -> shared state -> Module B -> risk level)
    - Cross-Domain Attack Matrix (table: domain combo -> attack tested -> result)
    - Findings (each with: title, severity, multi-step attack description, PoC with full attack sequence)
    - SAFE Proofs (each with: attack chain tested, cross-contract trace, why it fails)
    - Attack Surface Inventory (every composition surface checked with disposition)
  </action>
