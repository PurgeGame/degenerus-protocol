    Then systematically audit every RNG-related attack surface:

    **1. VRF Commitment Window Analysis**
    For EVERY VRF request site (advanceGame in AdvanceModule, coinflip in BurnieCoinflip, lootbox in LootboxModule):
    - Trace BACKWARD from each RNG consumer to verify the random word was unknown at the time inputs were committed
    - Identify what player-controllable state can change between VRF request and fulfillment
    - Check if any mutable state read during fulfillment was settable after the request
    - For each window: produce a PoC demonstrating manipulation OR a SAFE proof with the exact code path trace showing why manipulation is impossible

    **2. Request-to-Fulfillment Path Integrity**
    For each VRF integration:
    - Verify requestId mapping and fulfillment routing
    - Check for replay, front-running, or reordering attacks
    - Verify the contract handles VRF callback failures/reverts
    - Check stall resilience (what happens if VRF never responds)
    - Trace the rngLocked guard and its scope

    **3. RNG Consumer Cross-Contract Tracing**
    For every function that consumes an RNG word (EntropyLib consumers, direct randomWord users):
    - Trace the word from VRF fulfillment through storage to consumption
    - Verify no path allows the same word to be consumed twice
    - Verify no path allows a stale/predictable word to be used
    - Check all bit-extraction patterns in EntropyLib for bias

    **4. Cross-Domain Findings**
    If you discover money, gas, admin, or composition issues while tracing RNG paths, document them in a separate "Cross-Domain Findings" section.

    - Executive Summary (1-2 paragraphs)

    - SAFE Proofs (each with: attack surface, trace, conclusion)

    - Attack Surface Inventory (table of every surface checked with disposition)
