## Section 2 — F-03 / SOLV-01 fix-variant LOCK

**Decision: VARIANT (a) — return the BAF whale-pass `remainder` from
`_queueWhalePassClaimCore` and fold it into the BAF caller's `claimableDelta`, so the existing
single `memFuture -= claimed` / `claimablePool += claimableDelta` accounting debits
`futurePrizePool` for it.** Variant (b) (push the remainder back to `futurePrizePool` via the
`_processSoloBucketWinner` pattern) is **REJECTED**.

### Call sites variant (a) touches (AS-FOUND)

1. `DegenerusGamePayoutUtils.sol:45` — change `_queueWhalePassClaimCore` from `internal` (no
   return) to `internal returns (uint256 remainderCredited)`; return `remainder` (0 when the
   `remainder != 0` branch is not taken). Keep the inline `claimablePool += uint128(remainder)`
   at `:58` **removed** in the BAF caller's accounting? No — see the rationale: the cleanest
   shape returns `remainder` AND drops the inline `claimablePool += uint128(remainder)` at `:58`,
   letting the caller's single `claimablePool += uint128(claimableDelta)` at
   `AdvanceModule:972` cover it (mirrors `_addClaimableEth`, which credits via `_creditClaimable`
   and returns the wei for the caller to fold — it does NOT bump `claimablePool` itself).
2. `DegenerusGameJackpotModule.sol:1949` (BAF large-lootbox leg) — capture the return:
   `claimableDelta += _queueWhalePassClaimCore(winner, lootboxPortion);` so `runBafJackpot`'s
   returned `claimableDelta` (`:1903`) now includes the remainder.
3. `DegenerusGameJackpotModule.sol:2001` (the second/solo `_queueWhalePassClaimCore` caller) —
   this caller must ALSO fold the returned remainder into whatever delta it reconciles, OR (if
   its surrounding accounting already debits the source pool for the full `amount`) keep the
   inline `claimablePool` bump local to that path. The IMPL author confirms which by reading the
   `:2001` caller's pool math; the SPEC flags it as a required reconciliation point so the
   return-shape change does not silently drop the inline bump on that path.
4. `DegenerusGameAdvanceModule.sol:902-903`/`:972` — unchanged in shape; `claimed` (the
   `runBafJackpot` return) now includes the remainder, so `memFuture -= claimed` debits
   `futurePrizePool` for it and `claimablePool += uint128(claimableDelta)` credits it exactly
   once. No new statement.

### Rationale (grounded in the frozen bodies)

- **Why (a) correctly debits `futurePrizePool` without disturbing the lootbox leg that adds
  nothing to `claimableDelta`.** Today `runBafJackpot` (`JackpotModule.sol:1901-1909`) returns
  `claimableDelta`, accumulated ONLY from `_addClaimableEth` (`:730-740` — credits via
  `_creditClaimable`, returns `weiAmount`, **no inline `claimablePool` bump**). The large-lootbox
  whale-pass leg calls `_queueWhalePassClaimCore` (`:1949`) which today bumps `claimablePool`
  inline AND returns nothing → its `remainder` is INVISIBLE to the caller's `memFuture -= claimed`
  (`AdvanceModule:902`). Making `_queueWhalePassClaimCore` return `remainder` and folding it into
  `claimableDelta` makes the whale-pass leg behave EXACTLY like `_addClaimableEth`: the caller's
  one `memFuture -= claimed` now subtracts the remainder from `futurePrizePool`, and the one
  `claimablePool += uint128(claimableDelta)` at `:972` credits it — the same wei leaves
  `futurePrizePool` and enters `claimablePool` atomically. The `fullHalfPasses` leg is untouched
  (whale-pass claims are backed by the pool draw itself, not by `claimablePool`), so the leg that
  "adds nothing to `claimableDelta`" stays adding nothing.

- **Why (b) is rejected (the decisive structural fact).** Variant (b) would have
  `_queueWhalePassClaimCore` (or the JackpotModule leg) call
  `_setFuturePrizePool(_getFuturePrizePool() + remainder)` to keep the remainder backing in
  `futurePrizePool` (the `_processSoloBucketWinner` shape, `JackpotModule.sol:1382`). But in the
  BAF path, `futurePrizePool` is a **stale cached local** `memFuture` (read once at
  `AdvanceModule:801`, mutated only in the local, written back with a SINGLE
  `_setPrizePools(uint128(memNext), uint128(memFuture))` at `:968`). A mid-loop storage write via
  `_setFuturePrizePool` inside `runBafJackpot` would be **silently CLOBBERED** by the `:968`
  writeback of the stale `memFuture`. `_processSoloBucketWinner` can use the push-back shape
  precisely because it runs OUTSIDE that cached-`memFuture` window (it reads/writes
  `_getFuturePrizePool()` live); the BAF leg cannot. Variant (b) is therefore not just
  inferior — it is **incorrect in this caller** without also threading the remainder back out to
  the AdvanceModule and adjusting `memFuture` there, at which point it collapses into variant (a)
  with extra indirection. Variant (a) reuses the existing, already-correct single writeback.

- **Why `_processSoloBucketWinner` is NOT a counter-precedent.** There the whale pass is granted
  as **full passes only** (`whalePassClaims[winner] += whalePassCount`, no remainder credited to
  `claimableWinnings`); the kept `whalePassCost` in `futurePrizePool` backs the FUTURE pass
  claims, not a withdrawable ETH credit. In `_queueWhalePassClaimCore` the `remainder` IS a
  withdrawable `claimableWinnings` ETH obligation (`:56`), so the correct backing is to MOVE wei
  from `futurePrizePool` into `claimablePool` — which is exactly variant (a).
