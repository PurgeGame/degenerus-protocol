As an external auditor on the council sweep for the Degenerus Protocol at commit `c4d48008`, I have completed the investigation into the specified VRF and loop properties.

### 1. Bounded-loop re-verification (LOOP-02)

I have verified that the following closed loops remain strictly bounded under worst-case conditions:

*   **`_backfillGapDays` (`AdvanceModule:1843`):** Explicitly capped to 120 iterations via a numeric check: `if (endDay - startDay > 120) endDay = startDay + 120;`. This ensures the loop stays within the 16.7M gas ceiling even if the VRF stalls for months.
*   **`_backfillOrphanedLootboxIndices` (`AdvanceModule:1866`):** Verified bounded to ~1 iteration. As per the structural invariant, `LR_INDEX` only increments when `rngRequestTime == 0`. Once a request is in flight, further increments are blocked. Thus, the index is at most 1 ahead of the last filled slot, and the loop breaks immediately upon hitting a filled index (`if (lootboxRngWordByIndex[i] != 0) break;`).
*   **Deity-refund (`GameOverModule:109`):** Bounded by `deityPassOwners.length`. The `deityPassOwners` array is strictly capped at `DEITY_PASS_MAX_TOTAL = 32` via a check in the `purchaseDeityPass` path.
*   **Subscriber stage (`GameAfkingModule:1144`):** Bounded by both `weightBudget` (caller-supplied gas limit) and the `SUBSCRIBER_CAP = 1000` (physical limit of the `_subscribers` array). Each iteration consumes at least 1 weight unit, ensuring termination.

### 2. New unbounded loop (LOOP-03)

No new unbounded iterations were found in the `c4d48008` codebase. All recent additions (v61/v62) involving iteration (e.g., in `JackpotModule` for coin distributions or `MintModule` for batch processing) employ either hard constants (like `DAILY_COIN_MAX_WINNERS = 50` or `BAF_SCATTER_ROUNDS = 50`) or the `writesBudget` / `weightBudget` pattern to ensure gas-bounded execution.

### 3. VRF window freshness (VRF-01)

**PROPERTY:** RNG-consumer determinism (the v45 "North Star").
**REACHABLE SEQUENCE:**
1. `advanceGame` triggers a daily VRF request. `rngLockedFlag` is set to `true`.
2. A player (or VRF-node attacker) sees the VRF fulfillment in the mempool or anticipates the result.
3. The player calls `purchase` for near-future tickets (`targetLevel <= level + 5`).
4. `DegenerusGameStorage._queueTickets` (L596) allows the purchase because `isFarFuture` is false.
5. The player's tickets are added to `ticketQueue[targetLevel]` *after* the entropy was committed but *before* it is fulfilled/consumed.
6. The same `advanceGame` transaction (or a subsequent one) fulfills the VRF and immediately calls `payDailyJackpot` or `_payDailyCoinJackpot`.
7. The draw consumes the non-frozen `ticketQueue`, allowing the attacker to "stuff" the queue with winning tickets.

**CONSUMER:** `payDailyJackpot` / `_payDailyCoinJackpot` in `DegenerusGameJackpotModule.sol`.
**BOUND/FRESHNESS FAILURE:** `DegenerusGameStorage.sol` lines 595-596 (`isFarFuture` exception).
**SEVERITY:** **DOMINANT**. This is a direct violation of the protocol's core RNG-determinism invariant, which states that all consumer inputs must be committed at the time entropy is requested. By allowing near-future ticket purchases during the `rngLockedFlag` window, the protocol leaves the daily and coin jackpots vulnerable to front-running manipulation by anyone capable of seeing the VRF word before it lands on-chain.

### 4. Carried candidates

*   **FC1 (Mid-day stall):** **SAFE.** A pending mid-day request (`LR_MID_DAY != 0`) does not stall the daily `advanceGame`. If a daily request is issued, it overwrites the shared `vrfRequestId`. The mid-day index is then correctly backfilled by `_backfillOrphanedLootboxIndices` once the daily word arrives, ensuring no stranding of lootbox results.
*   **FC5 (Entropy binding):** **ENFORCED.** Although the `TraitsGenerated` event is slimmed, the trait derivation in `_raritySymbolBatch` remains strictly bound to the VRF `entropy` and the `baseKey` (containing `player`, `lvl`, and `owed` count). The binding is enforced on-chain; the event change is only an observability reduction.
*   **FC6 (Coordinator swap backfill):** **SAFE.** If a coordinator swap occurs during a mid-day stall, the days are not permanently stranded. While the mid-day fulfillment doesn't trigger a backfill, the subsequent daily `advanceGame` will detect the gap (`day > dailyIdx + 1`) and trigger `_backfillGapDays` via the normal `rngGate` path.

### Summary
The protocol's gas-bounded loop logic is robust across all investigated paths. However, the **RNG-freeze property is broken for the near-future ticket queue**, creating a reachable exploit for the daily and coin jackpots during the VRF fulfillment window.
