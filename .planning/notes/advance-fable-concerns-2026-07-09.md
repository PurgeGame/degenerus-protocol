===== CONCERN REVIEW #3 =====
(a) Definite inconsistencies

1. **Lines 731 and 746 — the game-over ticket drain keys the queue at `lvl + 1` unconditionally, but the pending queue lives at `lvl` whenever game-over is entered via the VRF-death deadman.**
   The normal advance path derives the drain key conditionally: line 213 `purchaseLevel = (lastPurchase && locked) ? lvl : lvl + 1` and line 502 `_runProcessTicketBatch(inJackpot ? lvl : purchaseLevel)`. `level` is pre-incremented at the last-purchase daily RNG request (`level = lvl` in `_finalizeRngRequest`, line ~1835), and jackpot-phase purchases queue at `targetLevel = cachedLevel` (MintModule line 1967). So during a locked last-purchase day and throughout the jackpot phase, queued tickets sit at key `lvl`, not `lvl + 1`.
   Inducing input: the daily RNG request is sent on the last purchase day (or any jackpot day) — line 437 `_swapAndFreeze()` runs at request time, leaving `ticketsFullyProcessed = false` with that day's tickets in the read slot at key `lvl` — then VRF never fulfills, `_VRF_DEADMAN_DAYS` (120) pass, and `advanceGame` enters `_handleGameOverPath` via the line-216/686 deadman gate (the exact entry the comments at 684–685 describe). `_gameOverEntropy` commits the historical fallback and falls through to the drain block: `processTicketBatch(lvl + 1)` (line 731) reads an empty queue and returns `finished = true`; `_swapTicketSlot()` (line 745) then executes with a **non-empty read slot**, the state its own header (lines 1732–1736) documents as unreachable; line 746 checks `ticketQueue[_tqReadKey(lvl + 1)]` — also empty or holding only post-request stragglers — and latches `ticketsFullyProcessed = true` (line 747) while both slots still hold undrained entries at key `lvl`. Result: `ticketsFullyProcessed` diverges from its "all queued tickets drained" meaning, and every ticket queued at `lvl` (the target-met final purchase day is typically the level's largest cohort) is excluded from `handleGameOverDrain`'s terminal distribution, contradicting the block's stated purpose at lines 712–714 ("every purchased ticket is eligible"). Note the same unconditional `+1` convention appears out-of-range in `handleGameOverDrain`'s `runTerminalJackpot(remaining, lvl + 1, rngWord)` (GameOverModule line ~199), which the same pre-increment shifts to `L + 2`; that leg deserves a follow-up review of the JackpotModule.

2. **Line 725 — `if (!ticketsFullyProcessed)` skips the entire drain block on a stale `true`, stranding the write-slot queue when game-over is reached without the stall day's daily request having passed through `rngGate`.**
   `ticketsFullyProcessed` is only cleared by `_swapTicketSlot`, which on the daily path runs at request time (line 437, `rngWord == 1` branch). Two reachable sequences arrive at game-over with the flag still `true` from the last sealed day while the write slot holds every ticket purchased since that seal:
   (i) full abandonment — tickets are bought after day D seals, `advanceGame` is never called again; 120 days later the liveness/deadman gate opens game-over, `_gameOverEntropy` issues a fresh request via `_tryRequestRng` (line 1424) **without any slot swap**, the word fulfills, the next call commits it and falls to line 725, which is `false`-branched;
   (ii) the coordinator reverts at request time so no request is ever sent (`rngRequestTime` set in `_gameOverEntropy`'s failure tail, line ~1429), and after `GAMEOVER_RNG_FALLBACK_DELAY` the fallback word commits — again with no swap ever having run.
   In both, the write-slot queue at `_tqWriteKey(lvl + 1)` is never swapped into read position, so those buyers' entries never materialize before `handleGameOverDrain` runs, and the terminal jackpot distributes without them — including tickets bought before any liveness window opened (the storage-side `_queueEntries` ruling covers only tickets queued *during* the timeout window). Contrast: the mid-day request path handles exactly this state explicitly at lines 1190–1194 (`if (ticketQueue[wk].length > 0 && ticketsFullyProcessed) _swapTicketSlot()`); the game-over path has no equivalent.

Holds because `GAME_MINT_MODULE` is a trusted constant module returning canonical bools (and is code-bearing, so the empty-returndata `< 64` swallow branch is never hit by an empty-code target).

Holds because every level's `levelPrizePool` is set non-zero at the phase transition before that level's purchase phase — an invariant maintained in `_endPhase`/transition code outside this range.
- **Lines 725–750 (swap-then-latch discipline)**: setting `ticketsFullyProcessed = true` after the swap is sound only because purchase paths gate on the same `_livenessTriggered()` predicate (comment, lines 663–667), so no new tickets can enter the write slot mid-drain and be silently orphaned after the latch.
