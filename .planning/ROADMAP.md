### v4.3 prizePoolsPacked Batching Optimization (Phases 99-102)

**Milestone Goal:** Eliminate ~320 redundant SSTOREs per daily jackpot by batching prizePoolsPacked writes -- accumulate deltas in memory, write once at end of loop.

- [x] **Phase 99: Callsite Audit** - 1 plan, 2 requirements (completed 2026-03-25)
- [ ] **Phase 100: Batching Implementation** - Refactor _processAutoRebuy to return deltas; update all callers to accumulate and batch-write
- [ ] **Phase 101: Equivalence + Gas Verification** - Prove batched writes produce identical state; confirm gas savings and ceiling safety
- [ ] **Phase 102: Comment Cleanup** - NatSpec and inline comments accurate for all modified functions

## Phase Details

### Phase 99: Callsite Audit
**Goal**: Every callsite of _processAutoRebuy and every prize pool storage write within auto-rebuy paths is fully inventoried with current behavior documented, providing a complete map before any code changes
**Depends on**: Phase 98 (v4.2 complete)
**Requirements**: CALL-01, CALL-02
**Success Criteria** (what must be TRUE):
  1. Every callsite of _processAutoRebuy is listed with its caller context (daily ETH, earlybird, daily coin, etc.) and the number of loop iterations at each site
  2. Every _setFuturePrizePool and _setNextPrizePool call within auto-rebuy paths is mapped with the packed slot being written and the write frequency per jackpot execution
  3. The total SSTORE count per daily ETH jackpot execution is quantified from the callsite map (baseline for gas comparison)
**Plans**: TBD

### Phase 100: Batching Implementation
**Goal**: _processAutoRebuy returns pool deltas instead of writing storage directly, and every caller accumulates deltas in memory then writes once at the end
**Depends on**: Phase 99
**Requirements**: BATCH-01, BATCH-02, BATCH-03
**Success Criteria** (what must be TRUE):
  1. _processAutoRebuy no longer calls _setFuturePrizePool or _setNextPrizePool -- it returns delta values to the caller
  2. _processDailyEth accumulates deltas across its full winner loop and performs a single prizePoolsPacked write after the loop completes
  3. All non-daily-ETH callers of _processAutoRebuy (earlybird, daily coin, any others found in Phase 99) handle the returned deltas and write accumulated values to storage
  4. The protocol compiles cleanly with no new warnings
**Plans**: TBD

### Phase 101: Equivalence + Gas Verification
**Goal**: The batched implementation is proven to produce identical final prizePoolsPacked state as the original sequential writes, all existing tests pass, and gas savings are confirmed
**Depends on**: Phase 100
**Requirements**: EQUIV-01, EQUIV-02, EQUIV-03, GAS-01, GAS-02
**Success Criteria** (what must be TRUE):
  1. A formal equivalence argument proves that accumulating deltas in memory and writing once produces the same final prizePoolsPacked value as N sequential SSTORE writes (commutativity + associativity of addition over the delta set)
  2. All existing Hardhat tests pass with zero regressions (same pass/fail counts as v4.2 baseline)
  3. All existing Foundry tests pass with zero regressions
  4. Gas profiling shows approximately 1.6M gas savings on the daily ETH jackpot worst-case path compared to v4.2 baseline
  5. Gas ceiling re-profiled for all 3 daily jackpot stages -- all still SAFE with updated headroom percentages documented
**Plans**: TBD

### Phase 102: Comment Cleanup
**Goal**: All modified functions have accurate NatSpec and inline comments reflecting the new batching pattern
**Depends on**: Phase 101
**Requirements**: CMT-01
**Success Criteria** (what must be TRUE):
  1. Every modified function has NatSpec that accurately describes the delta-return / batch-write pattern
  2. No stale comments referencing the old sequential-write behavior remain in modified files
**Plans**: TBD
