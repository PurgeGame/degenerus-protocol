**Section 1: VRF Architecture Overview**

Fulfillment via `rawFulfillRandomWords` callback storing word in `rngWordCurrent`, then consumed in advanceGame loop via `_finalizeRngRequest` which writes `rngWordByDay[day]`. Note that the VRF word is unknown to all parties until the Chainlink callback executes.

**Section 2: Path-by-Path Commitment Window Traces (new/modified paths only, per D-01)**

For EACH of the following paths, trace BACKWARD per D-03/D-04/D-05:

**Path A: rollLevelQuest entropy (v13.0 NEW)**
- Consumer: `DegenerusQuests.rollLevelQuest(entropy)` at line ~1777
- Entropy derivation: `uint256 questEntropy = uint256(keccak256(abi.encodePacked(rngWordByDay[day], "LEVEL_QUEST")))` in AdvanceModule at line ~382
- Backward trace: rollLevelQuest <- advanceGame phaseTransitionActive block <- _finalizeRngRequest writes rngWordByDay[day] <- rawFulfillRandomWords callback stores rngWordCurrent <- vrfCoordinator delivers random word
- Commitment window: Quest type selection inputs are: (1) the entropy word (VRF-derived, unknown before callback), (2) type(uint8).max as primaryType (constant), (3) decAllowed (derived from decWindowOpen flag which is set in advanceGame BEFORE VRF request). Player cannot manipulate the entropy after VRF request because keccak256 preimage includes rngWordByDay which only exists after VRF fulfillment. The "LEVEL_QUEST" tag provides domain separation from daily quest entropy.
- Player-controllable state between request and fulfillment: NONE. The level transition block in advanceGame runs atomically within a single transaction after rngWordCurrent is available. No external calls between entropy derivation and rollLevelQuest call. The phaseTransitionActive block (line ~370-387) executes sequentially: _processPhaseTransition, _drawDownFuturePrizePool, rollLevelQuest. No player action can interleave.
- Verdict: SAFE

**Path B: rollDailyQuest entropy (v13.0 MODIFIED -- access control changed from onlyCoin to onlyGame)**
- Consumer: `DegenerusQuests.rollDailyQuest(day, entropy)` at line ~332
- Entropy source: `rngWord` passed directly from advanceGame loop at line ~258
- Backward trace: rollDailyQuest <- advanceGame daily processing <- rngWord from _finalizeRngRequest/gap backfill <- rawFulfillRandomWords
- Commitment window: entropy is the raw VRF word (or keccak256-derived backfill for gap days). For the current day: rngWord comes from rngWordCurrent stored by VRF callback. For gap days: rngWord is keccak256(combined, w) where w is a prior rngWordByDay value. In both cases the entropy is VRF-derived and unknown before the fulfillment callback.
- Player-controllable state: rollDailyQuest is idempotent per day (line 334: `if (quests[0].day == day) return`). The daily quest type selection uses `(entropy >> 128) | (entropy << 128)` as bonusEntropy -- a deterministic bit-shuffle of the VRF word. No player input affects type selection. The only player-controllable state is whether advanceGame is called at all (but quest types are deterministic from the VRF word, calling earlier or later doesn't change the outcome for a given day).
- Verdict: SAFE

**Path C: _bonusQuestType entropy consumption (v13.0 MODIFIED -- added sentinel 0 skip)**
- Consumer: `DegenerusQuests._bonusQuestType(entropy, primaryType, decAllowed)` at line ~1463
- Called from: rollDailyQuest (with bonusEntropy) and rollLevelQuest (with questEntropy)
- Backward trace: Same as paths A and B above -- entropy flows through from VRF word
- Commitment window: The function is `private pure` -- it has no state access, no external calls. It consumes entropy deterministically: `entropy % totalWeight` selects from weight table, then `entropy /= 10` for next iteration if candidate is skipped. All inputs are either VRF-derived (entropy) or constants/flags (primaryType, decAllowed). No player manipulation vector.
- v13.0 change impact: Adding `candidate == 0` skip does not affect commitment window -- it only changes which quest types are in the selection pool. The entropy source is unchanged.
- Verdict: SAFE

**Path D: payAffiliate PRNG (v13.0 MODIFIED -- winner-takes-all mod-20 roll)**
- Consumer: `DegenerusAffiliate.payAffiliate` mod-20 roll for 75/20/5 distribution
- Entropy source: `keccak256(AFFILIATE_ROLL_TAG, dayIndex, sender, storedCode) % 20` -- this is NOT VRF-derived. It is a deterministic PRNG using on-chain state.
- Backward trace: The entropy comes from `keccak256(abi.encodePacked(AFFILIATE_ROLL_TAG, dayIndex, sender, storedCode))`. All inputs are on-chain: AFFILIATE_ROLL_TAG is a constant, dayIndex is the current game day, sender is msg.sender, storedCode is the player's stored referral code.
- Known tradeoff: This is a deliberate design choice documented in Phase 165-03 findings. A miner or sophisticated actor could predict the roll outcome for a given transaction. However, the economic value at stake per roll is small (affiliate commission split among 3 tiers) and the PRNG provides sufficient fairness for the use case. Not a VRF consumer -- no commitment window applies.
- No-referrer path: Uses same PRNG pattern for VAULT/DGNRS 50/50 flip. Same known tradeoff applies.
- Verdict: KNOWN TRADEOFF (not VRF, documented as acceptable)

**Path E: clearLevelQuest (v13.0 NEW)**
- Consumer: `DegenerusQuests.clearLevelQuest()` at line ~1773
- Entropy: NONE -- this function zeros levelQuestType. It does not consume entropy.
- Relevance: Called in advanceGame BEFORE VRF request (line ordering: clearLevelQuest before _requestRng). This ensures no quest progress accumulates during the gap between VRF request and fulfillment. When rollLevelQuest fires after fulfillment, it overwrites the zeroed type.
- Verdict: SAFE (no entropy consumed, ordering is correct)
