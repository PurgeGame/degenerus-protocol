SPEC §3.1 tally rule (LOCKED):
- mult <= NEUTRAL: store score+1 only on first deposit; NO cap draw; adjustedPortion stays 0.
- mult > NEUTRAL (bonus): remaining = used >= CAP ? 0 : CAP - used; add = min(deposit, remaining);
  lootboxEvBenefitUsedByLevel[player][lvl] += add; adjustedPortion += add (RMW the packed word).
- First deposit writes score+1 AND baseLevel+1; subsequent deposits accumulate adjustedPortion only.
- Subsequent-deposit multiplier is FROZEN from the first-deposit score snapshot (read score+1 back
  from the packed word; do NOT recompute from current activity).

CAP-KEY LEVEL (LOCKED — pre-derived against live source, do NOT re-derive):
- The shared cap accumulator `lootboxEvBenefitUsedByLevel[player][lvl]` MUST be keyed at deposit
  time by the SAME game level the open/resolution path uses. The resolver computes
  `currentLevel = level + 1` and passes it as the cap key:
  `_applyEvMultiplierWithCap(player, currentLevel, ...)` (LootboxModule resolveLootboxDirect ~675,
  resolveRedemptionLootbox ~711; openLootBox ~539 also uses `currentLevel = level + 1`).
- Mint deposit cap key = `cachedLevel + 1` (Mint snapshots the game level into `cachedLevel`;
  `cachedLevel + 1` == the resolver's `currentLevel`).
- Whale deposit cap key = `level + 1` (Whale reads the global game-level storage var `level`
  directly at the deposit block; `level + 1` == the resolver's `currentLevel`). This is
  EXPLICITLY NOT the `level + 2` value Whale writes into the packed baseLevel field — `level + 2`
  is the DIV-1 packed-field SENTINEL offset (a different value), not the cap level.
- NOTE: deposit-vs-open game-level drift behavior (if `level` advances between deposit and open)
  is the LOCKED SPEC §3.1 design — the accumulator is keyed at ALLOCATION time and is proven
  order-independent at Phase 311 TST-01. Do NOT "fix" drift in this plan.
