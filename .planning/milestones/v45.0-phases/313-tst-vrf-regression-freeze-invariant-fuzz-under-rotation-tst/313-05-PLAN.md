## Context — why these tests fail and the scope discipline

The Phase 312 fix replaced `updateVrfCoordinatorAndSub`'s blanket RNG-state reset + `LR_MID_DAY=0` clear with a preserve+re-issue branch. A baseline classification (running the suite at the pre-fix parent `41546f16` vs post-fix HEAD) identified exactly 17 tests that PASSED pre-fix and FAIL post-fix because they encode the OLD behavior:

**Class A — assert the DELETED blanket-reset (need new assertions):**
- `test_coordinatorSwapResetsAllVrfState` (VRFStallEdgeCases.t.sol) — asserts rngLocked()==false, vrfRequestId==0, rngRequestTime==0, rngWordCurrent==0 after swap (the deleted reset).
- `test_tryRequestRngGuardBranches` (VRFStallEdgeCases.t.sol) — asserts "rngLocked cleared after swap".
- `test_coordinatorSwap_clearsRngLocked` (VRFCore.t.sol) — asserts "rngLocked should be false after swap".
- `test_coordinatorSwapClearsMidDayPending` (VRFStallEdgeCases.t.sol) — asserts midDayTicketRngPending (LR_MID_DAY) cleared by swap.

**Class B — `_doCoordinatorSwap`/`_resumeAfterSwap` helper leaves the re-issued request unfulfilled, so advanceGame reverts RngNotReady():**
- VRFStallEdgeCases.t.sol: test_zeroSeedAtGameStart, test_zeroSeedUnreachableAfterSwap, test_manipulationWindowIdenticalToDaily, test_gapDayPositionsPreCommitted, test_gapDaysSkipResolveRedemptionPeriod, test_gapBackfillSingleDayGap, test_gapBackfillZeroGuard, test_gapBackfillGas30Days, test_gapBackfillGas120Days.
- StallResilience.t.sol: test_stallSwapResume, test_coinflipClaimsAcrossGapDays, test_lootboxOpenAfterOrphanedIndexBackfill.
- VRFPathCoverage.t.sol: test_gapBackfillEntropyUnique_fuzz.

**STRICT SCOPE GUARD:** Migrate ONLY these 17 fix-induced regressions. Many OTHER tests in the suite fail on BOTH the pre-fix baseline `41546f16` AND post-fix HEAD (affiliate test_orderIndependence/test_revertDoubleClaim `E()`, solvency-violation tests, arithmetic-overflow panics, `InvalidBet()` degenerette tests, `testFreezeUnfreezeRoundTrip`, the `index should increment by 1: 0 != 1` tests, etc.) — these are PRE-EXISTING suite debt unrelated to the VRF fix. DO NOT touch them. The plan 313-06 verification asserts NO NEW failures beyond the pre-fix baseline, not a fully-green suite.

The migration is mechanical: the new `updateVrfCoordinatorAndSub` PRESERVES the lock/flag and RE-ISSUES the in-flight request on the new coordinator. So (Class A) assertions must flip from "reset to zero" to "preserved + re-issued", and (Class B) the shared swap+resume helpers must FULFIL the already-re-issued request on the new coordinator (it exists immediately after the swap — `newVRF.lastRequestId()` is non-zero) before advancing, instead of triggering a fresh request.

Purpose: restore the suite to its pre-fix pass/fail baseline (no NEW fix-induced failures) so SC-5 holds. Output: the 4 migrated test files.
