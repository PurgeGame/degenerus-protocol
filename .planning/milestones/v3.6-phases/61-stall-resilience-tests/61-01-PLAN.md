**test_lootboxOpenAfterOrphanedIndexBackfill()** (TEST-03):
```
// Setup buyer with enough ETH for lootbox purchases
address buyer = makeAddr("lootBuyer");
vm.deal(buyer, 200 ether);

// Day 1: purchase with lootbox amount
vm.prank(buyer);
game.purchase{value: 1.01 ether}(buyer, 400, 1 ether, bytes32(0), MintPaymentKind.DirectEth);

// Complete day 1 (VRF request reserves lootbox index 1, fulfillment writes word for index 1)
_completeDay(0xL007B001);

// Record the current lootbox index (should be 2 now -- index 1 was used for day 1)
uint48 preStallIndex = game.lootboxRngIndexView();

// Warp to day 2, trigger VRF request (reserves lootbox index = preStallIndex)
vm.warp(block.timestamp + 1 days);

// Purchase with lootbox amount BEFORE advanceGame so lootboxEth[preStallIndex][buyer] has value
vm.prank(buyer);
game.purchase{value: 1.01 ether}(buyer, 400, 1 ether, bytes32(0), MintPaymentKind.DirectEth);

game.advanceGame(); // triggers VRF request, reserves lootbox index preStallIndex
assertTrue(game.rngLocked(), "Day 2 VRF pending");

// The orphaned index is the one just reserved by the day 2 VRF request
uint48 orphanedIndex = game.lootboxRngIndexView() - 1;

// Verify no RNG word yet for orphaned index
assertEq(game.lootboxRngWord(orphanedIndex), 0, "Orphaned index has no RNG word before swap");

// Stall + swap (warp 3 days, then coordinator swap which backfills orphaned index)
MockVRFCoordinator newVRF = _stallAndSwap(3);

// After swap: orphaned index should have a fallback word from updateVrfCoordinatorAndSub
assertTrue(game.lootboxRngWord(orphanedIndex) != 0, "Orphaned index backfilled after swap");

// Resume to drive game forward
_resumeAfterSwap(newVRF, 0xL007CAFE);

// Verify openLootBox does not revert for the orphaned index
// openLootBox will revert with E() if lootboxEth[index][buyer]==0, or RngNotReady if word==0.
// We purchased with lootbox amount targeting this index, and word is now set.
game.openLootBox(buyer, orphanedIndex);  // should not revert
```

Note on lootbox index timing: `_reserveLootboxRngIndex` is called from `_finalizeRngRequest` inside `_requestRng`. It runs when advanceGame triggers a VRF request. So the index is reserved at request time, not fulfillment time. The lootbox purchase before advanceGame puts ETH into `lootboxEth[currentIndex][buyer]` where currentIndex = `lootboxRngIndex` at purchase time. After advanceGame reserves that index, `lootboxRngIndex` increments. So the buyer's lootbox entry is at the orphaned index.

However, if lootboxRngIndex was already at preStallIndex before the day 2 advanceGame, then purchases made before advanceGame write to lootboxEth[preStallIndex][buyer]. When advanceGame calls _requestRng -> _finalizeRngRequest -> _reserveLootboxRngIndex, it maps the request ID to preStallIndex and increments to preStallIndex+1. This means the orphaned index IS preStallIndex, and the buyer's lootbox entry IS at preStallIndex. This is correct.

If the exact index tracking proves difficult due to _finalizeRngRequest's retry logic (the day 2 request may be fresh or a retry depending on whether the day 1 VRF was a stale cross-day request), use `game.lootboxRngIndexView() - 1` AFTER advanceGame to capture the just-reserved index. Alternative: record lootboxRngIndexView BEFORE advanceGame and verify that index matches.
