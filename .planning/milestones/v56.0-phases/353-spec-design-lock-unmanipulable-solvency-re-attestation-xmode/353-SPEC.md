## AFF-01 — Affiliate Deterministic-Split PULL Non-Gameability (LOCKED — AMENDED 2026-06-01: flat-7% deterministic-split PULL, supersedes the XMODEL roll-unification)

**Requirement OWNED by Phase 353.** AFF-01 locks that the v56 afking affiliate distribution is non-gameable by settle-timing. **The mechanism is now a flat-7% deterministic-split PULL** (the prior "roll-unification" XMODEL lock is SUPERSEDED — there is no roll at all, so the entire seed/settle-timing/free-option surface is REMOVED). Three load-bearing facts, all anchor-attested above, ground the lock:

1. **There is NO roll and NO seed.** The accrued affiliate base is split by a **fixed deterministic 75/20/5 proportion** (affiliate `A` 75% / upline1 20% / upline2 5%) computed once at claim — NOT a `keccak % 20` winner-takes-all roll. Removing the roll **deletes the entire seed/settle-timing manipulability surface**: there is no `currentDayIndex()` input, no window-boundary-day seed, no "favorable roll" to wait for, and (per the XMODEL C1/C2 finding) **no two-distribution "free option" can exist** because there is exactly ONE deterministic distribution path. The live roll (`keccak256(AFFILIATE_ROLL_TAG, currentDayIndex(), sender, storedCode) % 20`, `DegenerusAffiliate.sol:558-567`) is a push-model gas trick (pay ONE winner per buy instead of three); under a PULL model each recipient lazily claims their own batched share, so "pay all three" is no longer 3× per-buy and the roll's reason-to-exist disappears.

2. **The buyer can NEVER receive the affiliate base** — `A ≠ sender` is GUARANTEED (self-referral resolves to `VAULT` in the referral layer), so the **75% leg never skips** to the buyer; the ONLY skip is the rare mutual-referral *cycle* where an upline `U1`/`U2` equals the sub (the sub is also their affiliate's upline), which drops (not redirects) that leg, preserving the live buyer-never-wins semantic (`contracts/DegenerusAffiliate.sol:579`). So `sender` (the afking subscriber / their funding wallet) has **zero EV** from the split regardless of timing.

3. **Redistribution is intra-upline-chain-only** — the split routes value among `A` (75%), `U1 = _referrerAddress(A)` (20%), `U2 = _referrerAddress(U1)` (5%). Manipulation only moves value among the buyer's own affiliate chain; it never creates or destroys protocol value. The accepted intra-chain model (`DegenerusAffiliate.sol:556-557` "PRNG is known — accepted design tradeoff") is preserved (and is now even simpler — no PRNG at all).

**Mechanism LOCKED (flat-7% deterministic-split PULL):**

- **Accrue (per-buy, in the STAGE):** accrue a **flat 7% BURNIE-equivalent affiliate base** into the sub's running `affiliateBase` — `base = _ethToBurnie(ethSpent) × 7 / 100` (FLAT 7% of the ETH spent valued in BURNIE at the ETH-equivalent rate; `_ethToBurnie` = the v55 helper). **NO activity taper** on afking (flat 7%, zero branches; the AFF-02 taper applies to MANUAL buys only), **NO kickback** for afking (the gross 7% accrues; the affiliate's `kickbackPct` rebate-to-buyer is honored only on the live manual path). One warm SSTORE per buy: `affiliateBase += base`. `affiliateBase` IS a **running unclaimed balance** — buys add, claim zeroes; no window, no day marker, no roll, no cross-contract, no leaderboard write on the hot path. The accrued base is the **uplines' money**, captured immutably at each buy's rate — a sub mutation (`setDailyQuantity` / funding change / unsub) does NOT settle or forfeit it; it persists in the slot for the uplines to PULL.

- **Settle by PULL — `claim(address[] subs)`:** permissionless, SAME-AFFILIATE batch. Read the direct affiliate `A` from `subs[0]`; `require` every other sub resolves to the same `A` (mixed arrays revert) → the upline chain `A / U1 / U2` is read ONCE. Walk the array accumulating **ONE running total `sumB`**: for each sub, `B = affiliateBase[sub]`; if `B == 0` skip; **read-and-zero `affiliateBase[sub]` ATOMICALLY per iteration** (NEVER pre-load bases to a memory array — that reopens the duplicate-sub double-credit); `sumB += B`; accumulate `skipU1 += B`/`skipU2 += B` only for the rare cycle where `U1 == sub`/`U2 == sub` (near-always 0; `A ≠ sub` guaranteed so the 75% leg never skips). After the walk, split + credit **ONCE for the whole batch**: `u1 = (sumB − skipU1)×20/100`, `u2 = (sumB − skipU2)×5/100`, `aShare = sumB − u1 − u2` (A never skips; remainder→A; `Σ ≤ sumB`, never over-mints) → `pendingClaim[A] += aShare`, `pendingClaim[U1] += u1`, `pendingClaim[U2] += u2`. Leaderboard once at the **claim level**: `affiliateCoinEarned[lvl][A] += sumB` + one `_updateTopAffiliate(A,…)` (`:509` analog; USER-accepted claim-time distortion). **noReferrer subs** form their own group, split deterministically **50/50 VAULT / DGNRS** (`pendingClaim[VAULT] += ΣB/2`, `pendingClaim[DGNRS] += ΣB/2`, remainder to VAULT) — EV-matching the live no-referrer `:539-554` VAULT/DGNRS 50/50 WITHOUT entropy, preserving the sDGNRS-pool funding leg.

This grants nothing: (a) a **real** upline is set-once / `REF_CODE_LOCKED` → cannot be swapped retroactively; (b) an **unset** upline resolves to `VAULT`, and pointing it at one's own wallet is a choice available at any time with the identical outcome (the affiliate captures the same 20%/5% regardless of WHEN they set the sybil upline) — there is no timing edge.
