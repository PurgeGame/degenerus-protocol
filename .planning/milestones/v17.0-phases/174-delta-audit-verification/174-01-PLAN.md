**VRFY-03: Cache Correctness Proof**

Prove the cache produces identical results to the uncached computation for all reachable inputs. Analyze the three execution paths:

**Path A — Cache hit (cachedLevel == currLevel):**
The cache stores `(lvl, affiliateBonusPointsBest(lvl, player))` at the moment `recordMintData` runs with `!sameLevel`. At read time, `currLevel` is the contract's `level` state variable. Cache hit means `cachedLevel == level`. Since `recordMintData` is called with `lvl` = the level being purchased (which equals `level` at that point), and `affiliateBonusPointsBest` is a deterministic function of (level, player), the cached value equals `affiliateBonusPointsBest(level, player)`.

Prove the value cannot go stale: `affiliateBonusPointsBest` looks back over the last 5 levels of affiliate volume. Once a level closes, its affiliate volume is frozen (no more mints at that level). So for a given `(level, player)`, the return value is immutable once cached. The cache only hits when `cachedLevel == level`, meaning the game hasn't advanced. If the game advances to `level + 1`, the cache misses (falls through to Path B).

**Path B — Cache miss (cachedLevel != currLevel):**
Falls through to `affiliate.affiliateBonusPointsBest(currLevel, player)` — identical to the pre-change code path. No behavioral change.

**Path C — Cache never written (fresh player or no level transition):**
`AFFILIATE_BONUS_LEVEL_SHIFT` bits default to 0. `cachedLevel` = 0, `currLevel` >= 1 (level starts at 1). Cache misses, falls through to live computation. Identical to pre-change behavior.

Also verify edge cases:
- `affPoints` from `affiliateBonusPointsBest` is capped at `AFFILIATE_BONUS_MAX` (50). MASK_6 max = 63. So 50 fits in 6 bits without truncation.
- The `setPacked` with MASK_6 masks to 6 bits. If somehow `affPoints > 63` (impossible due to cap at 50), it would truncate. But cap is 50, so no truncation.
- Level values: 24-bit MASK_24 max = 16,777,215. Game levels won't exceed this.
