## Implementation Decisions

### TST-DPNERF-04 EV Regression Methodology
- **D-295-EV-METHODOLOGY-01:** **Hybrid: 1,000 JS-replay + small-N (16-32) production cross-attestation.** JS-reference impl drives the 1,000 iterations across both ETH + BURNIE-path arithmetic shapes (deterministic, ~second-scale runtime); 16-32 production-path replays via natural jackpot resolution (`payDailyJackpot` / `payDailyJackpotCoinAndTickets` for ETH; `payDailyCoinJackpot` for BURNIE) cross-attest JS‚ÜîEVM bit-identity by counting deity wins in the returned `winners[]` across the 25-winner draws (ETH) or single-winner pull (BURNIE) and asserting JS-replay total matches EVM-observed total. The deity sentinel for ETH `_randTraitTicket` is `winners[i] == deity` paired with `ticketIndexes[i] == type(uint256).max` per the L1755-L1757 sentinel-pair invariant; for BURNIE `_awardDailyCoinToTraitWinners` it's `JackpotBurnieWin(winner, lvlPrime, trait, amount, ticketIdx)` event emission with `winner == deity` paired with `ticketIdx == type(uint256).max` per the L1888-L1893 sentinel-pair invariant. **Why:** mirrors Phase 293 D-293-INVOKE-01 hybrid methodology ‚Äî pure JS-replay weakest evidence link to runtime EVM behavior; pure production-path test runtime would be tens of minutes to hours per 1,000 trials with worst-case bucket-setup cost per iteration; hybrid is fast + audit-grade. **How to apply:** Plan-phase implements `test/helpers/randTraitTicketRef.mjs` as a pure-function bit-mirror of BOTH the `_randTraitTicket` body (L1707-L1763, 25-winner ETH draw) AND the `_awardDailyCoinToTraitWinners` inline-duplicate gold-tier block (L1867-L1874, single-winner-per-pull BURNIE draw). Helper exposes 2 named exports OR a shared lower-level `goldTierVirtualCount(trait, len, deityPresent)` primitive plus 2 path-specific draw wrappers ‚Äî planner's discretion at file-shape time. The 1,000 iterations split across paths is planner-chosen (e.g., 500 ETH + 500 BURNIE, or 750 ETH + 250 BURNIE proportional to production call frequency); document the split + EV math in test JSDoc. Cross-attestation N (16-32) is planner-chosen within that band ‚Äî pick the smallest N that fits a per-iteration deity-win-count chi¬≤ goodness-of-fit at p > 0.05 against the analytical 1/(len+1) per-draw expectation.

### TST-DPNERF-03 BURNIE Path Coverage Technique
- **D-295-BURNIE-PATH-01:** **Natural production-path BURNIE invocation via `payDailyCoinJackpot` ‚Üí `_awardDailyCoinToTraitWinners`.** Set up a BURNIE coin-jackpot scenario: mint deity-pass NFT to seed the `deityBySymbol[fullSymId]` mapping; populate near-future `ticketQueue` at appropriate levels via `placeFutureTickets` or equivalent; fund coin budget via the natural `_calcDailyCoinBudget(lvl) > 0` precondition; invoke `payDailyCoinJackpot(lvl, randWord, minLevel, maxLevel)` against worst-case-seeded gold-tier holder buckets; assert `virtualCount == 1` semantics via the resulting `JackpotBurnieWin` event sequence ‚Äî deity is selected with probability `1/(len+1)` per pull (vs the v41 baseline of `max(len/50, 2)/(len + max(len/50, 2))`). **Why:** D-42N-PATH-COVERAGE-01 + D-294-CALLER-UNIFORM-01 lock path-uniform behavior structurally (the BURNIE path inlines the same gold-tier branch logic at L1867-L1874 ‚Äî NOT a caller of `_randTraitTicket`, an inline duplicate per `feedback_verify_call_graph_against_source.md` Phase 294 BURNIE gap precedent), so pure JS-replay would suffice evidentially; but the roadmap success criterion 3 EXPLICITLY mandates "fixture replicates the gold-tile scenario via `payDailyCoinJackpot` ‚Üí `_awardDailyCoinToTraitWinners` ‚Üí `_randTraitTicket`" ‚Äî i.e., natural-production-flow invocation is the deliverable shape. Natural-flow attestation provides direct evidence for the BURNIE half of D-42N-PATH-COVERAGE-01 without requiring the structural-argument-only fallback. **How to apply:** Plan-phase implements one production-path BURNIE invocation as the primary TST-DPNERF-03 vehicle (NOT the JS-replay alternative); the JS-replay impl for BURNIE arithmetic is still implemented as the test-helper component of D-295-EV-METHODOLOGY-01's TST-DPNERF-04 1,000-iteration hybrid, but TST-DPNERF-03 itself uses the natural-production-flow shape.
