<threat_model>
## Trust boundary
The property guards the solvency spine: ETH/afking accounting across the pass + presale-box + claim
surfaces. The "untrusted input" is any randomized action sequence mixing pass buys, afking spends, and
cashouts.

## Threat register (the invariant-violation class this guards)

| Threat ID | Category | Component | Disposition | Mitigation (what the invariant asserts) |
|-----------|----------|-----------|-------------|------------------------------------------|
| T-381-01-01 | Tampering (unbacked credit) | a pass-buy / presale-box / claim spend path that drops its paired `claimablePool` move | mitigate | `invariant_v61PoolEqualsSumOfHalves` over the WIDENED tracked set fails the instant claimablePool != Σ(halves) |
| T-381-01-02 | Information disclosure (insolvency) | claimablePool grows past ETH+stETH backing | mitigate | `invariant_v61PoolNeverExceedsBacking` over the same set |
| T-381-01-VAC | Tampering (vacuous green) | the invariant could pass because no spend path is reached | mitigate | handler call-counter ghosts + a falsifiability scenario (seeded dropped-pairing) prove non-vacuity; balances created ONLY via real paired entrypoints (never vm.store'd into existence) |

Severity is blocked on the property being un-fuzzable or vacuously green: if `SolvencyActionHandler`
reverts every action (0 successful pass/presale/claim calls) the property is vacuous → the plan FAILS its
acceptance until the success counters are non-zero.
</threat_model>
