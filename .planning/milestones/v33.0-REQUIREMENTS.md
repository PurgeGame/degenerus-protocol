# Requirements: Degenerus Protocol — Charity Allowlist Governance (v33.0)

**Defined:** 2026-05-05
**Core Value:** Every finding a C4A warden could submit is identified and either fixed or documented as known before the audit begins.

**Goal:** Replace the open `propose(address)` / approve-reject vote flow in `GNRUS.sol` with a vault-owner-curated allowlist that voters select from directly — eliminating the collusion path where any sDGNRS coalition could route the per-level GNRUS distribution to an address they control.

**Audit baseline:** v32.0 HEAD `acd88512` (closure signal `MILESTONE_V32_AT_HEAD_acd88512`).

**Mixed-shape milestone:** Phases 254-256 modify `contracts/GNRUS.sol` + add tests under `test/governance/` and `test/integration/`; Phase 257 delta-audits the result; Phase 258 patches the result post-closure (Phase 257 independent re-run surfaced a queue-branch redirect bug) + re-audits.

**Write policy:** READ-only LIFTED for v33.0 (consistent with v32.0 audit posture). All `contracts/` + `test/` changes require explicit per-commit user approval per `feedback_no_contract_commits.md`. Phase 258 uses the batched approval pattern per `feedback_batch_contract_approval.md`.

**Deliverable:** `audit/FINDINGS-v33.0.md` (single canonical 9-section deliverable, FINAL READ-only at HEAD `4ce3703d`, with regression appendix re-verifying v32.0 closure signal `MILESTONE_V32_AT_HEAD_acd88512` non-widening).

**Accepted RNG exceptions** (NEGATIVE-scope at v33; charity governance does not touch any RNG-consuming path):

1. EXC-01 — Non-VRF entropy for affiliate winner roll (KNOWN-ISSUES.md). NEGATIVE-scope.
2. EXC-02 — Gameover prevrandao fallback (`_getHistoricalRngFallback`; KNOWN-ISSUES.md). NEGATIVE-scope.
3. EXC-03 — Gameover RNG substitution for mid-cycle write-buffer tickets / F-29-04 class (KNOWN-ISSUES.md). NEGATIVE-scope.
4. EXC-04 — EntropyLib XOR-shift PRNG (KNOWN-ISSUES.md). NEGATIVE-scope.

- [x] **ALW-03**: Two-branch write semantics work as designed — instant-apply branch writes directly to current slate when `currentSlate[slot] == address(0)`; queue branch writes to pending when `currentSlate[slot] != address(0)`; pending overwrite for the same slot replaces (no separate cancel API); `CapExceeded` fires when post-flush active count would exceed 20 from either branch. **Phase 254 deviation:** `RecipientIsContract` revert path removed (D-256-CONTRACT-RECIPIENT-01) — vault-owner curation is the trust boundary; contract recipients accepted by design. — COMPLETE in Phase 254 Plan 254-02 (commit `469d7fc1`).

### RES — pickCharity Resolution (Phase 255, 4 REQs)

- [x] **RES-01**: `pickCharity(uint24 level)` atomically applies every queued edit to the current slate; emits per-edit `CharityFlushed` events for indexer reconstructability; flush itself does not revert on bad pending state (admin-side validation in `setCharity` makes the queue trustable). **Phase 258 FIX-01 amendment:** flush block reordered to execute AFTER the distribution payout instead of before it, so queued edits applied during level L take effect for level L+1, not for L itself. ### FIX — Post-Closure Patch (Phase 258, 2 REQs)

- [x] **FIX-01**: `contracts/GNRUS.sol` `pickCharity(level)` body restructured so flush block executes AFTER the distribution payout. Skip-paths A/B/C fall through to flush instead of returning early so queued edits apply to L+1 even on a skipped level. Closes the queue-branch vote-redirect mechanism surfaced by the Phase 257 independent re-run (where queued edits applied during level L would take effect in time for level L's vote tallying). Verified at GNRUS.sol:632-650 (winner phase against PRE-FLUSH slate; strict `>` tie-break) → L652-659 (paid branch: distribution math) → L662-681 (flush block, post-payout) → L685 (single `LevelSkipped` emit in terminal `if(!paid)` branch). — COMPLETE in Phase 258 Plan 258-01 (commit `636f60ea`).
- [x] **FIX-02**: `contracts/GNRUS.sol` adds `address public lastWinningRecipient` storage slot (GNRUS.sol:196); `pickCharity` writes `lastWinningRecipient = recipient` after payout (in distribution-paid path only — skip paths leave the value unchanged so a skipped level does not reset the block). Adds `error PreviousWinnerNotVotable()` declaration (GNRUS.sol:99); `vote(uint8 slot)` reverts `PreviousWinnerNotVotable()` if `currentSlate[slot] == lastWinningRecipient` (GNRUS.sol:580 — after existing slot-active check L575, before already-voted check L585). Prevents consecutive wins by the same recipient. Three new test cases added in `test/governance/CharityAllowlist.test.js` covering: (a) charity that won level L cannot be voted for at level L+1; (b) if vault-owner queue-replaces the winning slot's recipient between level L payout and L+1 vote opening, the new recipient is votable at L+1; (c) if level L is skipped, `lastWinningRecipient` retains its prior value so the L-1 winner remains blocked at L+1. — COMPLETE in Phase 258 Plan 258-01 (commits `636f60ea` + `4ce3703d`).
