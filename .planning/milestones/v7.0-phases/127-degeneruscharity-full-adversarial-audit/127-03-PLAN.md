## PART A: Game Hook Analysis

**1. handleGameOver() (DegenerusCharity.sol lines 331-343)**

Call tree:
- handleGameOver reads: finalized, balanceOf[address(this)], totalSupply
- handleGameOver writes: finalized, balanceOf[address(this)], totalSupply
- handleGameOver calls: nothing external

Storage writes:
- finalized = true (line 333)
- balanceOf[address(this)] = 0 (line 337)
- totalSupply -= unallocated (line 338, unchecked)

Attack angles:

a) **Access control (line 331, onlyGame modifier):** Only ContractAddresses.GAME can call. Verify the modifier on line 235-238: `if (msg.sender != ContractAddresses.GAME) revert Unauthorized()`. This is a direct address comparison — not spoofable. SAFE.

b) **Double-call protection (lines 332-333):** `if (finalized) revert AlreadyFinalized()` and `finalized = true`. Cannot be called twice. SAFE.

c) **Unchecked supply decrement (line 338):** `unchecked { totalSupply -= unallocated; }`. Is totalSupply >= unallocated? totalSupply = sum of all balances. unallocated = balanceOf[address(this)]. Since balanceOf[address(this)] <= totalSupply (it's one component), this is safe. SAFE.

d) **GameOverFinalized event (line 342):** Emits ethClaimed=0, stethClaimed=0. This is correct — handleGameOver only burns unallocated GNRUS, it doesn't claim ETH/stETH. The game contract pushes ETH/stETH to this contract during gameover processing (via handleGameOverDrain). SAFE.

e) **Reentrancy:** handleGameOver makes NO external calls. It only modifies storage and emits events. No reentrancy risk. SAFE.

**CRITICAL — Path A handleGameOver removal:**
Per PLAN-RECONCILIATION.md (Phase 124 DRIFT item): handleGameOver was removed from the no-funds early return path (Path A: `if (available == 0)`). The plan specified handleGameOver in BOTH terminal paths. The final state has it in Path B only.

Analyze: If the game ends with zero available funds (Path A), handleGameOver is NOT called. This means:
- finalized is never set to true
- Unallocated GNRUS is never burned
- The GNRUS contract retains its unallocated tokens forever

Is this safe? Consider:
- Can someone later call handleGameOver directly? No — only the game contract can, and handleGameOverDrain is presumably called once at game over.
- Does unallocated GNRUS matter if the contract has zero ETH/stETH? GNRUS holders can still burn for 0 ETH. The unallocated tokens are worthless if the contract is empty.
- But: the contract might accumulate ETH/stETH from yield surplus distributions (Phase 124 wiring) that arrived BEFORE gameover. If the game enters gameover with zero ETH in the game contract but the charity already holds ETH from prior distributions, those funds are still redeemable by GNRUS holders. The unallocated GNRUS remaining in address(this) would dilute the denominator for burn redemption (totalSupply includes unallocated tokens). This means existing GNRUS holders get LESS per burn than they should.

VERDICT for Path A removal: INVESTIGATE — Potential economic impact on GNRUS holders if gameover triggers with zero game funds but non-zero charity balance. The unallocated GNRUS dilutes the burn redemption ratio.
