This plan has two parts: (A) Cross-contract integration seam analysis, and (B) Storage layout verification + consolidated Taskmaster coverage.

**PART A — INTEGRATION SEAM ANALYSIS (per D-07)**

Analyze 5 cross-contract seams scoped to v6.0 changes only. For each seam, trace the end-to-end flow across contract boundaries and verify correctness at each handoff point.

**Seam 1: Fund split arithmetic end-to-end**
- Trace: `handleGameOverDrain()` in GameOverModule computes 33/33/34 split
- Sends to: DegenerusStonk (via DGNRS address), DegenerusVault (via _sendToVault), DegenerusCharity (via _sendStethFirst or direct)
- Verify: Each recipient's receive/fallback/handler correctly processes the incoming ETH+stETH
- Verify: No ETH is stranded between the three sends (what if one reverts?)
- Verify: handleFinalSweep follows the same 3-way pattern consistently
- Attack angle: Can an attacker make one of the three sends revert to grief the other two?

**Seam 2: Yield surplus redistribution**
- Trace: `_distributeYieldSurplus(uint256)` in JackpotModule splits 46% into 23% charity + 23% accumulator
- Sends to: charity receiver address
- Verify: The charity address is correctly configured (how is it set? can it be changed?)
- Verify: 23% arithmetic is correct (no rounding loss vs old 46% accumulator)
- Attack angle: Can yield surplus be redirected by manipulating the charity address?

**Seam 3: yearSweep timing vs gameOver state**
- Trace: `DegenerusStonk.yearSweep()` reads `gameOverTimestamp()` from `DegenerusGame`
- Verify: The timestamp read is consistent — can gameOverTimestamp change after being set?
- Verify: yearSweep cannot be called before gameOver actually fires
- Verify: yearSweep interacts correctly with handleGameOverDrain (called first) and handleFinalSweep
- Attack angle: Can yearSweep be called between handleGameOverDrain and handleFinalSweep to double-drain?

**Seam 4: claimWinningsStethFirst access control impact**
- Trace: `DegenerusGame.claimWinningsStethFirst()` now VAULT-only (was VAULT+SDGNRS)
- Verify: StakedDegenerusStonk (SDGNRS) can still claim its ETH winnings through the normal path
- Verify: No ETH/stETH stranded because SDGNRS can't call stETH-first path
- Attack angle: Is there any scenario where SDGNRS MUST use stETH-first and now can't?

**Seam 5: resolveLevel call path**
- Trace: `AdvanceModule._finalizeRngRequest()` calls `charityResolve.resolveLevel(lvl - 1)` at level transition
- Also: LootboxModule may trigger level-related calls
- Cross-reference: Phase 127 GOV-01 finding about permissionless resolveLevel desync
- Verify: The resolveLevel call from game modules is safe even if GOV-01 applies (or doesn't)
- Verify: If resolveLevel reverts, does it block advanceGame? (it should not)
- Attack angle: Can resolveLevel be weaponized to stall day advancement?

For each seam: VERDICT: SAFE / VULNERABLE / INVESTIGATE with supporting evidence.
