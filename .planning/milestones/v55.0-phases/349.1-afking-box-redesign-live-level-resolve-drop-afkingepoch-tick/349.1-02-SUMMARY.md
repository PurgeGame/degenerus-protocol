# Phase 349.1 Plan 02: STAGE — No-Orphan Guard + Ticket/Lootbox Split Summary

**The process STAGE gains a NO-ORPHAN SKIP guard as its first per-sub branch (dominating re-stamp / cancel-reclaim / pass-evict / funding-kill), drops the dead `_afkingEpoch` write + `epochIndex` param, and the P2 ticket/lootbox split routes ticket subs to `purchaseWith` (real tickets) and lootbox subs to a box stamp — plus the USER-requested 500 active-sub cap.**

- **Task 1 — NO-ORPHAN guard:** `if (sub.lastOpenedDay < sub.lastAutoBoughtDay) { ++cursor; ++processed; continue; }` is the FIRST per-sub branch (before cancel-reclaim), so a pending-box sub is left fully untouched + in-set this cycle. Dominates all four orphan paths. SKIP (LOCKED §3), no double-charge (debit is downstream).
- **Task 2 — drop epoch:** signature is now `(uint32 processDay, uint256 maxCount)`; the `_afkingEpoch[processDay]` write block is gone; `mp` + `currentLevel` reads kept; doc-comments rewritten to the no-epoch / pre-RNG-freeze model.
- **Task 3 — P2 split:** step-(6) debits `afkingFunding[src]` + `claimablePool` (fail-loud) for both modes, then branches on `isTicket`: ticket → `purchaseWith(player, amount, 0, bytes32("DGNRS"), payKind, ethValue)` via `GAME_MINT_MODULE` delegatecall (+ `_revertDelegate`), no box; lootbox → stamp `scorePlus1` + `amount`. Added the `IDegenerusGameMintModule` import.
- **USER addition — 500-sub cap:** `SUBSCRIBER_CAP` 65535 → 500 (enforcement unchanged: `_addToSet` new-insert-only, re-subscribe exempt); comments updated in module + storage; recorded in DESIGN §6/§8.

## Decisions Made
- **Ticket sub marker invariant (clarification of plan intent):** the plan's behavior says "the open-leg never sees a ticket sub's stamp." The mechanism: a ticket sub sets BOTH `lastOpenedDay = processDay` AND `lastAutoBoughtDay = processDay` (equal ⇒ not box-pending). Without setting `lastOpenedDay`, the no-orphan guard / `_afkingBoxReady` would treat the ticket sub as having a pending box and `_openAfkingBox` could fire on a stale `sub.amount`. This is required for correctness and fully consistent with the plan.
