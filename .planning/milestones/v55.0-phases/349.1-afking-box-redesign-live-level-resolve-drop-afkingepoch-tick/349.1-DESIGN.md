## 0. The one-sentence freeze story

> **The box's RNG word is pinned to the FROZEN stamp day (`rngWordByDay[lastAutoBoughtDay]`), so it
> was unknown when the box was committed. Everything else — the target level, the EV-cap key — reads
> LIVE at open.**

Because afking boxes are **auto-opened** (permissionless, BURNIE-bounty-driven), the player cannot
control open timing, so the level/day freezes that `openLootBox` needs (a human opens their *own*
box) are unnecessary here. The afking box follows the **`resolveLootboxDirect` pattern** (auto-resolve,
live level — `DegenerusGameLootboxModule.sol:763/:769`), NOT the `openLootBox` pattern (manual,
frozen baseLevel — `:535`).

---

## 1. Box model — DROP `_afkingEpoch` entirely

The held diff stores `_afkingEpoch[day] = (index, baseLevelPlus1)`. Both halves go away:

- **`baseLevelPlus1` → live level.** Roll the target from the LIVE `currentLevel` at open
  (`_rollTargetLevel(currentLevel, seed)`), exactly like `resolveLootboxDirect:769`. No stored floor.
- **`index` → day-keyed word.** Source the word from `rngWordByDay[sub.lastAutoBoughtDay]` (an
  existing map). The lootbox `LR_INDEX` is NOT 1:1 with `dailyIdx` (mid-day `requestLootboxRng`
  bumps `LR_INDEX` independently — `AdvanceModule.sol:1170`), so it cannot be derived from the day;
  but the **day-keyed word is the same committed word** in the common case (`rngGate` writes both maps
  from one `currentWord` — `AdvanceModule.sol:1273`+`:1294`), and once we declare the day-keyed word
  *canonical*, the multi-day-stall divergence is **moot** (the box just uses day-D's word, always
  committed after the stamp → freeze-safe in every case, stall or not).

**Result:** `_afkingEpoch` / `_packAfkingEpoch` / `_unpackAfkingEpoch` are DELETED. The open needs
only `(player, amount, lastAutoBoughtDay, scorePlus1)` — all in the `Sub` — plus `rngWordByDay[day]`.

`resolveAfkingBox` collapses to ≈ `resolveLootboxDirect` with TWO deviations:
1. the word is `rngWordByDay[lastAutoBoughtDay]` (not a caller-passed word), and
2. the **frozen stamp day** is passed into the seed `keccak256(rngWord, player, day, amount)` (NOT the
   live `_simulatedDayIndex()`) — the day MUST stay frozen in the seed or a self-keepering player
   could grind the seed by open-timing. Everything else (live `currentLevel`, EV-cap, ticket math)
   is identical to `resolveLootboxDirect`.

## 2. Freeze proof — why live-level is SAFE here (structural, not "couldn't find one")

A box outcome can only be gamed through two channels. Both are structurally closed:

1. **Picking the word** — CLOSED. The word is `rngWordByDay[lastAutoBoughtDay]`, and
   `lastAutoBoughtDay` is frozen at stamp (subscribe never writes it — see §4; re-stamp is guarded —
   see §3). The word for day D is requested *after* the day-D STAGE stamps (the STAGE runs pre-RNG),
   so it was unknown at stamp. The v45 VRF-freeze invariant ("no manipulation of the word") HOLDS.
2. **Picking when to open** — CLOSED by the permissionless bounty. `_autoOpen` is permissionless and
   pays the BURNIE open-bounty, so opening any ready box is a dominant strategy for every third-party
   keeper. **A player has no mechanism to hold their own box unopened** — the moment the word commits,
   a bounty hunter opens it at the current-ish level. No ability to delay → no ability to time the
   level. (And it's -EV anyway: higher level = fewer tickets at constant face — `tickets ≈
   amount/priceForLevel(targetLevel)`, `:1174` — and the BURNIE/boon/face value is level-invariant, so
   a rational player's own incentive is to open EARLY, which the bounty already makes happen.)

**Structural coupling:** level-advance and box-open are driven by the SAME permissionless `doWork`
router, so "level rising while my box sits unopened" cannot occur in a normally-keepered game.

**Residual (sweep note, NOT a blocker):** the only way to recover timing control is to **monopolize
all keepering** (open nobody's boxes, self-fund the gas to advance the game, then open your own at a
higher level) — expensive, self-defeating (-EV), and it collapses the instant a second keeper exists.
Accepted by design (USER, the author: the level/day freezes were defensive against player-timed
opens, which auto-open removes). **`force-open` (§3) would close even this**; `skip` leaves it
theoretically open. Re-attest at 352 TERMINAL.

## 3. NO-ORPHAN INVARIANT (the load-bearing correctness rule) ⚠

**Problem:** a box is STAMPED at process (day D) but OPENED later. The box exists ONLY as
`(Sub stamp + lastAutoBoughtDay)` — there is no cold ledger (`GameAfkingModule.sol:712`). So ANY
mutation of the `Sub` record, OR any removal from the iterable `_subscribers` set, between stamp and
open **orphans** the box (the player was debited at stamp — `:754` — and gets nothing). **There are
FOUR orphan paths, not one** (verified against the held diff):

| # | Path | Site | Effect |
|---|------|------|--------|
| 1 | daily **re-stamp** | STAGE step 6, `:742-763` | overwrites `amount` + `lastAutoBoughtDay` → box unreachable |
| 2 | cancel-tombstone **reclaim** | STAGE step 0, `:602-604` | `delete _subOf` + `_removeFromSet` → destroyed |
| 3 | pass **eviction** | STAGE step 2, `:637-638` | `_removeFromSet` → unreachable by `_autoOpen` |
| 4 | funding **skip-kill** | STAGE step 5, `:703-704` | `_removeFromSet` → unreachable by `_autoOpen` |

`_removeFromSet` (`:394`) pops from `_subscribers` — the array `_autoOpen` walks (`:892`) — so even
when `_subOf` survives (paths 3/4), the box becomes **unreachable**. A re-stamp-only guard misses
paths 2–4.

### The fix — ONE guard at the TOP of the STAGE loop

```
// (−1) NO-ORPHAN: a sub with a pending unopened box is left ENTIRELY untouched this cycle
//      — no reclaim, no evict, no funding-kill, no re-stamp. It stays in-set (reachable),
//      _autoOpen opens it, and a LATER cycle processes it (now boxless).
if (sub.lastOpenedDay < sub.lastAutoBoughtDay) { ++cursor; ++processed; continue; }
```

Placed **before** the cancel-reclaim (`:602`), so it dominates all four paths. Properties:

- **Pending ⟺ `lastOpenedDay < lastAutoBoughtDay`.** After `_autoOpen` opens day-D's box,
  `lastOpenedDay = D`, the predicate is false, and the next cycle processes the sub normally.
- **The sub stays in `_subscribers` while pending** — this is *why* `_autoOpen` can always reach it.
  (Removing it while pending is the bug; the guard's whole job is to NOT remove/mutate while pending.)
- **`_autoOpen` ignores `dailyQuantity`** (`_afkingBoxReady` checks only `lastOpenedDay <
  lastAutoBoughtDay` + word-ready), so a cancelled/evicted sub's *paid-for* box still opens before the
  later-cycle reclaim. Correct: they paid, they get it.
- **Cannot starve:** `_autoOpen` wraps its cursor and re-walks (`:888-890`); a pending box is always
  reachable. Worst case under keeper congestion = a *preserved* (not orphaned) box that opens when any
  keeper runs `_autoOpen`. (If NO keeper runs, the game isn't advancing either — same liveness floor.)
- **No double-charge:** a skipped sub is not debited (the debit `:754` is downstream of the guard).
- Composes with the existing same-day idempotency skip (`lastAutoBoughtDay >= processDay`, `:614`),
  which still handles the chunked-same-day case.

### Why SKIP (not force-open) — the chosen guard

`skip` = defer ALL sub processing while a box is pending (the box opens via `_autoOpen`).
`force-open` = open the pending box inside the STAGE, then proceed.

**SKIP chosen** — it keeps the heavy open OUT of the 16.7M gas-critical advance chain, is one
top-of-loop guard, and the BURNIE bounty keeps opens prompt so it rarely actually skips a buy.
**force-open** is the alternative iff "never miss a buy even in a stalled/monopolized game" is wanted
— it also closes the §2 degenerate sole-keeper edge (opens at the pre-rise level inside the advance)
and is freeze-safe (lootbox tickets always queue to the future write-slot, so no mid-advance
double-process), at the cost of loading the advance path (chunk-safe via a smaller `SUB_STAGE_BATCH`).
**Decision: SKIP** (USER-decided 2026-05-31; flip to force-open only if "never miss a buy" is judged to outweigh liveness isolation — see rejected alternatives).

### Accepted tradeoffs + rejected alternatives (USER-decided 2026-05-31)

- **A sub with an unopened prior box does NOT buy that day** (skipped until its box opens) —
  USER-ACCEPTED. This is the self-throttling property: buy-rate can never outrun open-rate, so the
  backlog is structurally bounded at ≤1 pending box/sub. Near-zero in practice (the BURNIE bounty opens
  boxes same-day → the next day's STAGE sees no pending box and buys normally); it only bites under
  genuine open-congestion (large set + quiet keepers). **The cost of a skip is MORE than a forgone
  box** (USER correction): a skipped day is not charged (the debit `:754` is downstream of the guard),
  BUT on a streak-relevant day the missed auto-buy **breaks the player's mint streak** — the
  per-consecutive-level activity-score streak that is the ENTIRE POINT of afking mode (up to −50%
  activity score on reset). USER-ACCEPTED anyway, because the trigger — a box left unopened for a full
  ~24h — is **economically implausible**: the day-D word commits at that day's unlock (near the daily
  reset), and the BURNIE open-bounty draws a keeper to open the box the same day, long before day D+1's
  STAGE, so the skip ~never fires. The player also retains the **manual-buy escape hatch** (they can
  always buy a box directly to preserve the streak if they ever observe congestion). The sub resumes
  auto-buying once its box opens. (This is the one place SKIP is strictly worse than force-open — which
  would always buy, preserving the streak — but force-open's liveness coupling [rejected below] is the
  worse trade.)
- **A pending box is openable after an ARBITRARY `_autoOpen` delay** — confirmed durable: the stamp
  survives (skip guard), the word `rngWordByDay[day]` is immutable (write-once, day-keyed), the sub
  stays in `_subscribers` (reachable), and `_autoOpen` wraps + re-walks (`:888-890`). So `_autoOpen`
  not running for a day (or more) only DELAYS the box — it opens at the live level whenever any keeper
  next runs `_autoOpen`. It is never lost. (Only total keeper abandonment never opens it — but then the
  game isn't advancing either, so there's nothing to race.)
- **REJECTED — global advance-gate** ("block `dayIdx` / the STAGE until all prior-day boxes are
  opened"). Rejected for LIVENESS ISOLATION: it inverts the dependency (the game heartbeat — level
  progression, jackpots, the 120-day game-over clock in `_unlockRng` — would be hostage to afking-open
  throughput) and amplifies any open-failure into a full-game brick (`_autoOpen` propagates reverts,
  no per-box valve — `:901`). Both the gate AND force-open put the open INTO the advance path → both can
  brick; **SKIP is the ONLY liveness-isolated option** (the advance never opens a box; opens live solely
  in the separate `_autoOpen` category). The box's durability (above) makes the gate unnecessary — a box
  can only be delayed, never lost. The safe lever for minimizing lag is open-bounty calibration (350
  GAS), which keeps the advance isolated.

## 4. `Sub` layout (1 slot, lighter)

`lastOpenedIndex(48b)` → `lastOpenedDay(32b)`; `amount(wei)` → `quantity` (P1, §6). Open-guard +
no-orphan predicate both key on `lastOpenedDay` vs `lastAutoBoughtDay`.

```
config (56b):  dailyQuantity(8) + validThroughLevel(32) + reinvestPct(8) + flags(8)
per-sub (112b): scorePlus1(16) + amount(96)          // amount = frozen wei spend (see §6 note)
markers (64b):  lastAutoBoughtDay(32) + lastOpenedDay(32)
```
≈ 232/256 bits. `subscribe` upsert writes ONLY config (never the stamp/markers — `:313-333`), so a
re-subscribe never touches a pending box; `subscribe` is also rngLock-blocked (`:255`). Both confirmed
safe — no `subscribe` guard needed beyond what exists.

## 6. Remaining scope (folded from the old P1–P5)

- **P1 — whole-ticket quantity.** Store `quantity` (whole tickets), not wei. `reinvestQty =
  (claimable*pct/100)/mp` floors to whole tickets. NOTE: the box seed still needs the frozen wei
  `amount` (= `quantity * frozen mp`); since the open now reads LIVE level/price, the **frozen mp**
  must travel with the box. Options: keep a 96-bit `amount`(wei) stamp in the `Sub` (simplest, §4
  shows it), OR store `quantity` + derive `amount = quantity * mp_at_stamp` — but mp_at_stamp must be
  frozen, so the wei `amount` stamp is the clean choice. **Keep the frozen wei `amount` in the Sub**;
  `quantity` is the buy-side input, `amount` the frozen box spend. (This is the one place the "drop
  everything frozen" instinct stops — the SPEND is per-sub and must stay frozen for the seed; only
  the LEVEL went live.)
- **P2 — ticket/lootbox split (CONFIRMED BUG).** `_resolveBuy` returns `isTicket` and sets
  `amount = effectiveQty*400` (entry-units, `:488`), but `_openAfkingBox` always feeds `amount` to
  `resolveAfkingBox` as a WEI lootbox spend (`:842`) → garbage micro-box, no tickets queued. Fix:
  route ticket-mode subs to queue whole tickets at the frozen level (the existing ticket-queue
  mechanic, no box) and lootbox-mode subs to the box open. Homogeneous handling so the open never
  mis-sizes a ticket sub.
- **mintBurnie rename** — `doWork()` → `mintBurnie()` (the permissionless router; ~6 contract sites
  incl. `IDegenerusGameModules` + `DegenerusGame` stub/selector + 156 test refs). Cosmetic; names the
  reward, not the action (router still runs autoBuy→advance→autoOpen).
- **P5 dead-code cleanup** (gated on P2) — remove the AF_KING-gated `batchPurchase` + `BatchBuy`
  struct (`DegenerusGame.sol` ~1886-1955, zero callers) + the dead `AF_KING` entry in
  `BurnieCoinflip.onlyFlipCreditors` (then `ContractAddresses.AF_KING` unused).
- **SUBSCRIBER_CAP 65535 → 500 (USER addition 2026-05-31).** Cap the active subscriber set at 500.
  The advance chain walks `_subscribers` in the process + open passes, so the cap bounds the
  per-cycle work the *protocol* pays for; demand past 500 is not the protocol's burden — those
  users arrange their own off-chain keepering. Enforced exactly where the cap already lived —
  `_addToSet`'s NEW-insert-only branch (`_subscribers.length >= SUBSCRIBER_CAP` →
  `SubscriberCapReached`); a re-subscribe of an existing member is already-in-set (no growth) so it
  is exempt. 500 ≪ uint16, so the `_subCursor`/`_subOpenCursor` headroom is unaffected (no cursor
  aliasing). One-line constant retune + comment updates in `GameAfkingModule.sol` (+ the storage
  cursor doc-comment in `DegenerusGameStorage.sol`); zero behavioral change beyond the lower ceiling.
- **`subscribe` msg.value → resolved funding bucket (USER decision 2026-05-31).** Previously
  `subscribe`'s `msg.value` always credited `afkingFunding[subscriber]` (the player), but an
  operator-funded sub's draws debit `afkingFunding[fundingSource]` (the funder) — so ETH attached to
  an operator-funded subscribe landed in the player's bucket (unused for this sub, and
  operator-unrecoverable). Fixed: `msg.value` now credits the SAME bucket the draws debit — the
  non-self `fundingSource` (already OPENE-04-approved just above, so the funder consented to fund this
  subscriber ⇒ no forced gift to a non-consenting funder) for an operator-funded sub, else the
  subscriber. `claimablePool` moves in tandem (solvency invariant preserved). Self-funded subs are
  byte-unchanged. Edited `GameAfkingModule.subscribe` + the `DegenerusGame.subscribe` stub doc-comment.
- **Remove the `requestLootboxRng` `!subsFullyProcessed` guard (USER decision 2026-05-31).** Its
  stated rationale (FREEZE-02c: block a mid-day `LR_INDEX` advance so a sub stamped to that index
  doesn't bind a committed word) is VOID under 349.1 — subs no longer bind to `LR_INDEX`, and
  `requestLootboxRng` never writes `rngWordByDay[day]` (only `rngGate` does, at `:1910`/`:1862`), so
  it cannot affect the day-keyed afking word. It was also redundant with the `rngWordByDay[currentDay]
  != 0` check right below it (the daily word can't be set until the day's STAGE completed). Removed the
  guard + its stale comment; `subsFullyProcessed` stays as the STAGE's internal chunk flag. The lootbox
  RNG flow is still gated by the reroll-block + the daily-word check.
- **Decouple the afking STAGE per-day reset from `_swapTicketSlot` (USER decision 2026-05-31).** The
  reset (`subsFullyProcessed = false; _subCursor = 0;`) lived in `_swapTicketSlot`, which also fires
  mid-day (`:1135` lootbox-VRF) and on the ticket round-swap (`:617`) — spurious afking resets (harmless,
  but a coupling smell). Moved to a FORWARD-LOOKING day-gated reset at the STAGE entry in advanceGame:
  `if (_afkingResetDay != day) { _afkingResetDay = day; subsFullyProcessed = false; _subCursor = 0; }`
  before the `if (!subsFullyProcessed)` block. Fires exactly once per new day (before that day's STAGE
  drains), idempotent across chunked re-entries, on every RNG path — and `subsFullyProcessed` now stays
  true after a day's STAGE completes (meaning "afking done for that day") until the next day flips it.
  New `uint32 _afkingResetDay` packs into the existing cursor slot (16+16+32=64 bits). A `dayIdx`-based
  gate was rejected: `dailyIdx` updates late (in `_unlockRng`, often a separate VRF-callback tx), so
  `day > dailyIdx` would re-fire the reset within the day.
- **Remove the dead `lootboxSkip` / `LOOTBOX_MIN` mechanism (USER decision 2026-05-31).** Provably
  unreachable under 349.1: (a) `cost = mp × effectiveQty`, `effectiveQty ≥ dailyQuantity ≥ 1`, and
  `priceForLevel`'s global floor is 0.01 ETH = `LOOTBOX_MIN`, so `cost ≥ LOOTBOX_MIN` always →
  `cost < LOOTBOX_MIN` can never be true; (b) even if it could, the 349.1 lootbox path stamps a box →
  `resolveAfkingBox` (no `LOOTBOX_MIN` floor), so the MintModule:1057 `< LOOTBOX_MIN` revert it guarded
  is unreachable (only the buy path hits it, which lootbox subs no longer take). Removed the
  `LOOTBOX_MIN` constant, the `cost < LOOTBOX_MIN` branch + the `lootboxSkip` return in `_resolveBuy`
  (6→5 returns), the STAGE step-(4) skip, and the PlayerSkipped reason-4 / doc refs — all in
  GameAfkingModule.sol. forge build clean.
