**StakedDegenerusStonk.sol (DELTA-01) -- analyze these security dimensions:**

1. **Reentrancy in burn()** (lines ~379-441): Trace every external call in the burn function:
   - `game.claimWinnings(address(0))` (~line 405) -- verify CEI: state changes (balanceOf, totalSupply at lines 398-401) happen BEFORE this call
   - `coinflip.claimCoinflips(address(0), remainingBurnie)` (~line 426) -- verify no callback into sDGNRS
   - `coin.transfer(player, burnieOut)` -- BURNIE ERC20 transfer
   - `steth.transfer(player, stethOut)` -- stETH ERC20 transfer
   - `player.call{value: ethOut}("")` -- ETH transfer (LAST, CEI compliant?)
   - For each: document whether state is fully committed before the call, and whether the callee can re-enter any sDGNRS function

2. **Access control**: Check every function with `onlyGame` modifier, `onlyWrapper` modifier, and any other access restriction:
   - `transferFromPool` -- onlyGame
   - `transferBetweenPools` -- onlyGame
   - `burnRemainingPools` -- onlyGame
   - `wrapperTransferTo` -- onlyWrapper (only DGNRS contract)
   - `depositSteth` -- onlyGame
   - `receive()` -- onlyGame
   - `burn()` -- public (any holder) -- verify this is intentional
   - `previewBurn()` -- view (safe)
   - Document: who can call each, is the restriction correct, any bypass?

3. **Reserve accounting**: Verify the pool balance tracking:
   - Constructor pool BPS allocation: Creator 2000, Whale 1000, Affiliate 3500, Lootbox remainder, Reward 500, Earlybird 1000 -- verify they sum correctly and dust goes to Lootbox
   - `transferFromPool`: verify `poolBalances[pool]` decrement matches `balanceOf[address(this)]` decrement (lines ~320-327). Verify the capped-to-available logic (lines ~321-322)
   - `transferBetweenPools`: verify zero-sum between pools, `balanceOf[address(this)]` unchanged
   - `burnRemainingPools`: verify it burns `balanceOf[address(this)]` which should equal sum of all poolBalances

4. **Unchecked arithmetic safety**: For each `unchecked` block, verify the mathematical proof that overflow/underflow is impossible:
   - `balanceOf[player] = bal - amount` after `if (amount > bal) revert` -- safe
   - `balanceOf[to] += amount` in transferFromPool/wrapperTransferTo -- safe because totalSupply < uint256.max
   - Document each unchecked block with its safety proof

**DegenerusStonk.sol (DELTA-02) -- analyze these security dimensions:**

1. **ERC20 compliance**: Verify standard ERC20 behavior for transfer, transferFrom, approve, allowance:
   - Does `transfer(to, amount)` correctly deduct from sender and credit to recipient?
   - Does `transferFrom` correctly check and deduct allowance?
   - Does `approve` set allowance correctly?
   - Any edge cases with zero address, zero amount, max uint256 allowance?

2. **Burn delegation (burn-through)**: Trace the full burn-through path:
   - `DGNRS.burn(amount)` at line ~125-165
   - Verify: `_burn(msg.sender, amount)` reduces DGNRS balance/supply BEFORE calling `stonk.burn(amount)`
   - Verify: sDGNRS sends ETH/stETH/BURNIE to the DGNRS contract (msg.sender from sDGNRS perspective)
   - Verify: DGNRS forwards BURNIE via `coin.transfer(msg.sender, burnieOut)`
   - Verify: DGNRS forwards stETH via `steth.transfer(msg.sender, stethOut)`
   - Verify: DGNRS forwards ETH via `msg.sender.call{value: ethOut}("")`
   - Check: Can reentrancy occur during the forwarding phase? DGNRS balance already burned, so re-entering burn would fail (insufficient balance)

3. **unwrapTo auth**: Verify creator-only access:
   - Only `CREATOR` can call `unwrapTo(recipient, amount)` (line ~168)
   - Verify: `_burn(msg.sender, amount)` deducts from creator's DGNRS balance
   - Verify: `stonk.wrapperTransferTo(to, amount)` moves sDGNRS from DGNRS balance to recipient
   - Check: Can unwrapTo be used to drain more sDGNRS than the creator's DGNRS balance? (No -- _burn reverts first)

4. **Constructor correctness**: Verify DGNRS constructor:
   - Reads `stonk.balanceOf(address(this))` -- this is the sDGNRS balance held by the DGNRS contract address
   - Assigns `balanceOf[CREATOR] = deposited` and `totalSupply = deposited`
   - Reverts if `deposited == 0` (deploy ordering protection)
   - Verify no way to construct DGNRS before sDGNRS

5. **receive() function**: `receive() external payable {}` at line ~79
   - Accepts ETH from sDGNRS during burn-through
   - Check: Can anyone send stray ETH to DGNRS? (Yes -- but it benefits all future burners proportionally, not an exploit)

**Cross-contract supply invariant (DELTA-03) -- formal analysis:**

Prove or disprove the following invariant:
```
sDGNRS.balanceOf[DGNRS_ADDRESS] >= DGNRS.totalSupply   (always)
```

Trace every code path that modifies either side:
- **sDGNRS.balanceOf[DGNRS_ADDRESS] decreases**: only via sDGNRS.burn() (called by DGNRS.burn()) and sDGNRS.wrapperTransferTo() (called by DGNRS.unwrapTo())
- **DGNRS.totalSupply decreases**: only via DGNRS._burn() (called in burn() and unwrapTo())
- For burn(): both sides decrease by the same amount (DGNRS._burn reduces totalSupply by amount, sDGNRS.burn reduces balanceOf[DGNRS] by amount)
- For unwrapTo(): both sides decrease by the same amount (DGNRS._burn reduces totalSupply by amount, wrapperTransferTo reduces balanceOf[DGNRS] by amount)
- Prove: no path increases DGNRS.totalSupply without increasing sDGNRS.balanceOf[DGNRS]
- Prove: initial state is correct (constructor sets both to the same value)
