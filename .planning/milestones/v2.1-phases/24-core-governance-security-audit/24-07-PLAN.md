**WAR-01: Compromised Admin Key Scenario**

Attacker profile: Holds >50.1% DGVE (vault governance token). Goal: swap VRF coordinator to a malicious one controlled by attacker.

Attack path:
1. Wait for (or cause) VRF stall > 20h
2. Call `propose()` with malicious coordinator address (admin path, no stake required beyond DGVE)
3. Need to get proposal approved: `approveWeight * BPS >= threshold * circulatingSnapshot AND approveWeight > rejectWeight`

Defense analysis:
- Admin CANNOT vote (vote requires sDGNRS, not DGVE). The admin path only grants propose access, not vote power.
- Approval requires sDGNRS holders to vote. If community detects malicious proposal, they can vote Reject.
- Threshold decay gives community 7 days to respond (starts at 60%, decays to 5% at day 6, 0% at day 7)
- At threshold 0 (day 7+): ANY single approve vote would execute if approve > reject. If community is absent and attacker holds ANY sDGNRS, they could approve.
- Key question: Does the attacker need sDGNRS to self-approve? YES -- `vote()` requires `sDGNRS.balanceOf(msg.sender) > 0`.

Feasibility: MEDIUM -- requires compromised DGVE key + some sDGNRS + inattentive community for 7 days during VRF stall. The 7-day decay is the defense window.

Severity: Rate based on what a malicious coordinator can do (control RNG output -> manipulate game outcomes). This is the original M-02 vector but now gated by governance.

Cross-reference: Coercion.test.js and NationState.test.js for existing PoC analysis.

**WAR-02: Colluding Voter Cartel at Low Threshold**

Attacker profile: Small group controlling >= 5% of circulating sDGNRS (day 6 threshold). Goal: approve a malicious proposal at low threshold.

Attack path:
1. VRF stalls for 7+ days (community can propose at 7d)
2. Any community member with 0.5% sDGNRS proposes (or admin proposes at 20h)
3. Wait 6 days (144h): threshold drops to 500 BPS (5%)
4. Cartel votes Approve with combined weight >= 5% of circulatingSnapshot
5. If no Reject votes, proposal executes

Defense analysis:
- Requires 5% sDGNRS holdings -- in a protocol with real users, this may be concentrated or dispersed
- Threshold at day 6 is 5% (500 BPS). Need `approveWeight * 10000 >= 500 * circulatingSnapshot` => `approveWeight >= 5% of circulating`
- Also need `approveWeight > rejectWeight` -- any single reject voter blocks if reject >= approve
- Community defense: A single large holder voting Reject can block if their weight >= cartel's approve weight

Feasibility: Depends on sDGNRS distribution. In a protocol with a few large holders (whales), 5% concentration is likely. In a widely distributed protocol, harder.

Severity: MEDIUM -- same RNG control vector, but requires multi-day VRF stall AND voter inattention AND concentration.

**WAR-03: VRF Oscillation Attack**

Attacker profile: Controls VRF coordinator (or can cause intermittent VRF failures). Goal: create permanent DoS by cycling stall -> governance -> recovery -> invalidation -> repeat.

Attack path:
1. VRF stalls (attacker stops responding)
2. After 20h, governance activates (proposals can be created)
3. Attacker resumes VRF service
4. `vote()` stall re-check (line 441): `block.timestamp - lastVrf < ADMIN_STALL_THRESHOLD` is now TRUE (lastVrf was just updated by _applyDailyRng). All votes revert with NotStalled.
5. Active proposals cannot be voted on, but remain Active (not expired yet)
6. `activeProposalCount > 0` so death clock remains paused
7. Attacker stops VRF again
8. After 20h, governance reactivates. Old proposals are still Active and can be voted on again.
9. Repeat

Analysis:
- Each oscillation cycle: ~20h stall -> brief recovery -> stall again. Minimum cycle ~24h (20h stall + some recovery time).
- During recovery, `_applyDailyRng` updates `lastVrfProcessedTimestamp`, which auto-invalidates governance (votes revert). This is the INTENDED auto-cancellation mechanism.
- Old proposals survive oscillation -- they are not killed by VRF recovery, just temporarily unvoteable.
- Proposals expire after 168h regardless. So oscillation delays governance but doesn't prevent it forever.
- After 7 days, proposals expire (via lazy cleanup in vote()) and `activeProposalCount` decreases.
- New proposals can be created each time the stall resumes.

DoS assessment: The oscillation delays but cannot permanently prevent governance. After each proposal expires (7d), new ones can be created. The attacker must maintain oscillation indefinitely to prevent any proposal from completing.

Feasibility: LOW-MEDIUM -- requires ongoing VRF coordinator control (which is exactly what governance is trying to fix). Circular dependency.

Severity: LOW -- governance is degraded but not defeated. The death clock pause via `anyProposalActive` protects the game during oscillation.

**WAR-04: Creator unwrapTo Timing Attack**

Attacker profile: Protocol creator (holds DGNRS tokens). Goal: increase voting power by converting DGNRS to sDGNRS at the governance activation boundary.

Attack path:
1. VRF stalls at t=0
2. At t = lastVrf + 72000 seconds (exactly 20h): unwrapTo guard `> 20 hours` is FALSE (72000 > 72000 is false). unwrapTo is ALLOWED.
3. Creator calls `unwrapTo(recipient, amount)` -- burns DGNRS, sends sDGNRS to recipient
4. At t = lastVrf + 72001 seconds: governance is active (stall >= 20h). Recipient can now vote with the sDGNRS received at t=72000.

Analysis:
- The recipient receives sDGNRS at the EXACT second governance activates
- This is a 1-second window (only at t=72000, by t=72001 unwrapTo is blocked)
- Practical exploitability: Creator must submit transaction at exact second. In practice, with Ethereum's 12s block time, the creator has a ~12s window where the block containing t=72000 allows unwrapTo.
- Impact: Recipient gains vote weight they wouldn't otherwise have. Amount limited by creator's DGNRS balance.
- BUT: sDGNRS is sent to `recipient`, not `msg.sender`. Creator cannot directly increase their own vote weight unless they are also the recipient. The function signature is `unwrapTo(address recipient, uint256 amount)` with `msg.sender == CREATOR` check.
- If creator IS the recipient: they gain sDGNRS vote weight. But they already hold DGNRS which they could have burned for ETH -- the conversion is not free.
- circulatingSupply impact: unwrapTo moves sDGNRS from DGNRS contract to recipient. circulatingSupply = totalSupply - balanceOf(SDGNRS) - balanceOf(DGNRS). If balanceOf(DGNRS) decreases by X, circulatingSupply increases by X. So the snapshot taken by subsequent proposals would be larger, diluting the attacker's weight relative to the snapshot. Self-correcting.

Feasibility: LOW -- requires exact timing, limited by DGNRS balance, self-correcting via snapshot
Severity: Informational -- documented boundary behavior, not practically exploitable

**WAR-05: Post-Execute Governance Loop**

Attacker profile: Any governance participant. Goal: immediately re-propose after a swap execution to create perpetual governance cycles.

Attack path:
1. Proposal 1 is executed via `_executeSwap(1)`
2. `_voidAllActive(1)` runs: all other Active proposals are killed, `activeProposalCount = 0`
3. `lastVrfProcessedTimestamp` is NOT reset by `updateVrfCoordinatorAndSub` (XCON-01 confirmed this)
4. Stall condition: `block.timestamp - lastVrfProcessedTimestamp >= ADMIN_STALL_THRESHOLD` is still TRUE because lastVrfProcessedTimestamp was never updated
5. New `propose()` call: stall check passes, new proposal created
6. `activeProposalCount` increments to 1, death clock is re-paused

Analysis:
- After a governance-executed swap, the new coordinator must successfully process a VRF word (calling `_applyDailyRng`) to update `lastVrfProcessedTimestamp` and end the stall condition.
- Until that happens, governance remains active and new proposals can be created.
- Is this a problem? It could be:
  - If new coordinator is slow: death clock gap exists between `_voidAllActive` (count=0, death clock unpaused) and the new proposal being created (count=1, death clock re-paused). This gap is within the same transaction if propose is called immediately. But if the new coordinator never works, the stall continues indefinitely.
  - If used for griefing: Admin with DGVE could repeatedly propose after each execution, cycling through coordinators. But each execution requires majority sDGNRS approval -- this is governance working as intended.

Conclusion: Post-execute governance loop is INTENTIONAL behavior. The stall condition correctly persists until RNG actually works. The only concern is the brief moment after `_voidAllActive` sets count=0 and before a new proposal is created -- but this gap only matters if `advanceGame` is called in that exact window.

Feasibility: N/A (by design)
Severity: Informational -- document the design choice. Not a vulnerability.

**WAR-06: Admin Spam-Propose Gas Griefing**

Attacker profile: Admin (>50.1% DGVE). Goal: bloat proposalCount to increase `_voidAllActive` gas cost for legitimate executions.

Attack path:
1. VRF stalls > 20h
2. Admin calls `propose()` repeatedly with different coordinator addresses
3. `proposalCount` increments each time
4. When a legitimate proposal is eventually executed, `_voidAllActive` iterates from 1 to proposalCount

Cost analysis:
- Each `propose()` call: ~100k gas (storage writes for Proposal struct, mapping updates, event emission)
- 256 proposals: ~25.6M gas total for proposals (~8-9 transactions at 3M gas each)
- `_voidAllActive` with 256 proposals: 256 iterations * (~2100 gas cold SLOAD for proposals[i].state + conditional write) = ~538K gas for reads + ~20K gas per Active -> Killed write. If all 256 are Active: ~538K + 256*20K = ~5.6M gas. Within single transaction limits but expensive.
- With 1000 proposals (if somehow created): ~21.5M gas for _voidAllActive. Approaching block gas limit. Could cause legitimate execution to fail.

Defense:
- Creating proposals costs the attacker gas (not free griefing)
- Each proposal requires passing stall check and access control
- uint8 overflow at 256 is already documented in VOTE-03
- No per-proposer cooldown exists -- this is a genuine minor weakness

Feasibility: MEDIUM for admin (has DGVE), LOW for community (needs 0.5% sDGNRS per call)
Severity: LOW -- primarily a gas cost issue. Does not affect correctness. Attacker pays more gas to spam than the victim pays for _voidAllActive overhead.

Mitigation suggestion: Add `require(activeProposalCount < 10)` or similar cap per proposer. Note as informational finding.
