```solidity
function _voidAllActive(uint256 exceptId) internal {
    uint256 count = proposalCount;
    for (uint256 i = 1; i <= count; i++) {
        if (i == exceptId) continue;
        if (proposals[i].state == ProposalState.Active) {
            proposals[i].state = ProposalState.Killed;
            emit ProposalKilled(i);
        }
    }
    activeProposalCount = 0;  // Hard-set to 0, not decremented
}
```

**Step 3: Reentrancy analysis -- malicious coordinator at call #2 (createSubscription):**
The malicious coordinator receives a call at line 582. It could reenter DegenerusAdmin.

Reentry paths to analyze:
a) `vote()` on sibling proposal 2:
   - Stall check passes (stall still active)
   - Proposal 2 is still Active (p.state not yet changed by _voidAllActive, which runs at line 618)
   - If proposal 2 has enough approve weight, `_executeSwap(2)` is called
   - Inside `_executeSwap(2)`:
     - `p.state = Executed` for proposal 2
     - `coordinator` was already set to `newCoordinator` at line 581 of the outer call
     - `oldSub = subscriptionId` -- this was NOT yet updated (still old subscription). Wait -- line 581 sets coordinator BEFORE line 582 call, but subscriptionId is set at line 584 AFTER the call returns. So during reentrancy at line 582, `subscriptionId` still has the OLD value.
     - The inner `_executeSwap(2)` would try to cancel the OLD subscription (already cancelled by outer call at line 571)
     - Then create ANOTHER subscription on newCoordinator (the inner call's newCoordinator could be different if proposal 2 has a different coordinator)
     - Then `_voidAllActive(2)` runs -- kills all Active proposals INCLUDING proposal 1, which is already Executed. `proposals[1].state == Executed`, so the `if Active` check skips it. Good.
     - But `activeProposalCount = 0` is set. Then when outer `_voidAllActive(1)` runs at line 618, it sets `activeProposalCount = 0` again. No harm from double-set.
   - The REAL concern: two coordinator swaps execute. The Game contract gets `updateVrfCoordinatorAndSub` called twice -- the second call overwrites the first. Final state depends on which proposal's coordinator wins (the inner one, since it runs last before outer continues).
   - After inner returns, outer `_executeSwap(1)` continues at line 584: sets `subscriptionId = newSubId` (from the OUTER createSubscription call, which returned before reentrancy... wait, the reentrancy happens DURING createSubscription). Actually, `createSubscription()` is the call that triggers reentrancy. If the malicious coordinator reenters, the return value `newSubId` has not yet been assigned. The reentrant call completes, then control returns to the outer `createSubscription()` which returns its value. The outer `subscriptionId` would be overwritten with the outer newSubId, potentially clobbering the inner call's subscription setup.

b) `propose()` reentry:
   - Could create more proposals during execution. `activeProposalCount++` would increment. Later `_voidAllActive` would set it to 0. Net effect: new proposals created during execution would be voided.

c) `shutdownVrf()` reentry:
   - Requires `msg.sender == GAME`. Malicious coordinator cannot satisfy this.

**Step 4: Severity assessment of reentrancy:**
- The reentrancy IS theoretically possible via malicious coordinator
- The state guard (`p.state = Executed` at line 560) prevents re-execution of the SAME proposal
- But a SIBLING proposal could be triggered via `vote()` reentry
- Practical exploitability: The attacker must have a proposal that is Active AND has enough approve weight to trigger execution AND the malicious coordinator returns from `createSubscription` while reentering. The coordinator is user-supplied in `propose()`, so an attacker could propose a malicious coordinator.
- Impact: Dual execution could corrupt VRF coordinator state (two subscriptions created, Game configured with wrong one)
- Mitigating factor: After all reentrancy resolves, `_voidAllActive` runs and cleans up. But the VRF state in the Game contract may already be inconsistent.

The attacker would need to control enough sDGNRS votes AND propose a malicious coordinator that other voters approve. If the attacker controls the votes, they already control governance -- the reentrancy is not a privilege escalation, it's redundant with the primary threat model (compromised governance).

**GOV-08: _voidAllActive Correctness Analysis**

1. Trace `_voidAllActive(exceptId)` (lines 622-632):
   - Iterates `i = 1` to `proposalCount` (inclusive)

2. Adversarial check -- off-by-one:
   - Loop starts at `i = 1` (first proposalId is 1, from `++proposalCount` in propose). Correct.
   - Loop condition `i <= count` where `count = proposalCount`. This includes the last proposal. Correct.

3. Adversarial check -- non-Active proposals in the loop:
   - Proposals can be Expired, Killed, or Executed before `_voidAllActive` runs
   - The `if (proposals[i].state == ProposalState.Active)` check skips these. Correct.
   - The executed proposal is also skipped via `exceptId`. But its state was already set to Executed at line 560. Double-guarded.

4. Adversarial check -- `activeProposalCount = 0` instead of decrementing:
   - This hard-set is CORRECT because `_voidAllActive` voids ALL active proposals (except the executed one, which is already Executed). After this function, zero proposals should be Active. Setting to 0 is cleaner than counting decrements.
   - CONCERN: What if `_executeSwap` was reentered (GOV-07) and `_voidAllActive` was called twice? First call sets to 0, second call sets to 0 again. No harm -- idempotent for this field.
