**GOV-04: Threshold Decay Verification**

1. Verify `threshold()` function (lines 524-534) matches spec:
   - 0-24h: 6000 (60%)
   - 24h-48h: 5000 (50%)
   - 48h-72h: 4000 (40%)
   - 72h-96h: 3000 (30%)
   - 96h-120h: 2000 (20%)
   - 120h-144h: 1000 (10%)
   - 144h-168h: 500 (5%)
   - 168h+: 0 (0%)

2. Adversarial check -- boundary conditions:
   - At exactly 24h (86400 seconds): `elapsed >= 24 hours` is TRUE, returns 5000. But the earlier checks (`>= 168`, `>= 144`, etc.) are FALSE. So the function falls through to `>= 24 hours`. Correct.
   - At 23h59m59s: `elapsed >= 24 hours` is FALSE, returns 6000. Correct.
   - At exactly 168h: `elapsed >= 168 hours` is TRUE, returns 0. This means expired proposals have 0 threshold. But the expiry check in `vote()` (line 449) uses `>= PROPOSAL_LIFETIME` (168h), which reverts. So a 168h-old proposal cannot be voted on. The threshold returning 0 at 168h is unreachable via `vote()` -- but `canExecute()` view checks this too. Verify `canExecute()` also checks lifetime.
   - At 167h59m59s: threshold = 500 (5%). Last possible vote window.

3. Adversarial check -- if-else ordering:
   The function checks from largest to smallest. If order were reversed, a 168h proposal would match `>= 24 hours` and return 5000 instead of 0. Current order is correct.

4. Cite existing threshold decay tests from VRFGovernance.test.js.

**GOV-05: Execute Condition Verification**

1. Analyze: `p.approveWeight * BPS >= uint256(t) * p.circulatingSnapshot`
   - `approveWeight` is uint256, `BPS` is uint16 (10000). Product: `approveWeight * 10000`. Max approveWeight = sDGNRS totalSupply. sDGNRS is an ERC20 with 18 decimals. If totalSupply is 1 billion tokens = 1e27. Then `1e27 * 10000 = 1e31`. Well within uint256 (max ~1.15e77). NO OVERFLOW.
   - `uint256(t)` is at most 6000. `circulatingSnapshot` is at most totalSupply = 1e27 (realistic max). Product: `6000 * 1e27 = 6e30`. Well within uint256. NO OVERFLOW.
   - Truncation: No division involved, all multiplications. No truncation risk.

2. Adversarial check -- boundary at threshold = 0:
   If threshold = 0 (day 7+): `approveWeight * 10000 >= 0 * circulatingSnapshot` is always TRUE (if approveWeight > 0). Combined with `approveWeight > rejectWeight`, any single approve vote would execute. This is the intended decay behavior -- 0% threshold at 7 days means any nonzero approval wins.

3. Adversarial check -- when `circulatingSnapshot == 0`:
   If somehow `circulatingSnapshot` was 0 in the proposal, `uint256(t) * 0 = 0`, and any approve weight would meet the threshold. But `propose()` checks `circ == 0` for community path. Admin path does NOT check circ. If admin proposes when circ is 0, the snapshot is 0, and threshold is automatically met by any weight. Since no one has sDGNRS when circ=0, no one can vote... but the admin who proposed doesn't need to vote -- someone else must. This edge case should be documented.

4. Verify `approveWeight > p.rejectWeight` is strict greater-than (not >=). Ties go to neither side. Correct -- prevents 50/50 deadlock.

**GOV-06: Kill Condition Verification**

1. Analyze symmetry with execute:
   - Execute: `approveWeight * BPS >= threshold * circ AND approveWeight > rejectWeight`
   - Kill: `rejectWeight > approveWeight AND rejectWeight * BPS >= threshold * circ`
   - These are symmetric. Same overflow analysis applies (no overflow).
   - Note: condition order is swapped (reject > approve checked first in kill, approve > reject checked second in execute). Both are conjunction (AND), order doesn't matter for correctness.

2. Adversarial check -- can both conditions be true simultaneously?
   If `approveWeight > rejectWeight` AND `rejectWeight > approveWeight` -- impossible. Mutual exclusion guaranteed. Cannot simultaneously execute and kill.

3. Adversarial check -- neither condition met:
   If `approveWeight == rejectWeight` (tie), neither execute nor kill triggers. Proposal remains Active. This is correct -- tie goes to status quo (no swap).

4. Note: Kill does NOT call `_executeSwap`, it just sets state to Killed and decrements `activeProposalCount`. Verify the decrement: `unchecked { activeProposalCount--; }`. Same uint8 overflow concern as VOTE-03.
