**Part 1: stETH Rebase Mechanics**
Document how Lido stETH rebasing works:
- stETH `balanceOf()` returns proportional share of total staked ETH pool
- Rebases daily when Lido oracle reports (~12:00 UTC)
- Positive rebase (normal): balances increase by ~0.007% daily (~2.5% APR / 365)
- Negative rebase (slashing): balances decrease (rare, unpredictable)
- 1-2 wei rounding per share conversion (already documented as I-20 / DELTA-I-03)

**Part 2: Rebase Impact on burn() Payout**

Trace the burn calculation at StakedDegenerusStonk.sol lines ~385-396:
```
stethBal = steth.balanceOf(address(this))    // line ~387
totalMoney = ethBal + stethBal + claimableEth  // line ~389
totalValueOwed = (totalMoney * amount) / supplyBefore  // line ~390
```

Calculate concrete numbers:
- Assume sDGNRS holds 100 ETH in stETH (typical magnitude)
- Daily rebase at 2.5% APR: 100 * 0.025/365 = 0.00685 ETH = ~$17 at $2500/ETH
- If a user burns 1% of supply, their share of the rebase: 0.01 * 0.00685 ETH = 0.0000685 ETH = ~$0.17
- Cost to time the burn (gas): ~$0.10-0.50 per tx depending on gas price
- Conclusion: For a 1% holder, the extractable value from timing burns around the rebase is $0.17 -- barely covering gas costs. For larger holders, the value scales linearly but remains small relative to the total burn payout.

**Part 3: previewBurn vs burn Discrepancy**

Analyze whether calling `previewBurn()` before a rebase and executing `burn()` after creates issues:
- `previewBurn()` (if it exists -- check StakedDegenerusStonk.sol) reads the same live balances as `burn()`.
- Between calling previewBurn and executing burn, a rebase changes stethBal.
- This means the actual payout differs from the preview. Document the maximum discrepancy: ~0.007% daily.
- Is this exploitable? No -- the user gets the ACTUAL reserves at burn time, which is the correct behavior. The preview is informational only.
- Note: DELTA-I-03 already documented this as "by design."

**Part 4: Branch Condition Flipping**

The burn function has a critical branch at StakedDegenerusStonk.sol line ~410:
```
if (totalValueOwed <= ethBal) {
    // Pure ETH payout path
} else {
    // Mixed ETH + stETH payout path
}
```

Can a stETH rebase flip this branch?
- `totalValueOwed` includes stETH in its calculation (via totalMoney).
- If stETH increases (positive rebase), totalValueOwed increases, making the `>` branch MORE likely.
- If stETH decreases (negative rebase/slashing), totalValueOwed decreases, making the `<=` branch MORE likely.
- The branch determines payout COMPOSITION (pure ETH vs mixed), not total VALUE. Both paths pay `totalValueOwed` worth of assets.
- Can flipping the branch create an exploit? Analyze: in the `>` path, stETH is sent directly. In the `<=` path, all ETH. The user gets the same total value either way (subject to 1-2 wei rounding).
- Verdict: Branch flipping changes payout composition but not value. No exploit.

**Part 5: stETH Slashing Scenario**

Analyze the rare but catastrophic scenario: Lido validator slashing reduces stETH balances significantly.
- If stETH balance drops by X%, every burn payout's stETH portion drops by X%.
- This is not an attack -- it's protocol-inherent risk from holding stETH.
- Can an attacker trigger slashing? No -- Lido validator operations are external to this protocol.
- Can an attacker front-run a known slashing event to burn before the price drop? If a slashing event is known in the mempool (unlikely -- slashing is on the beacon chain, not mempool), an attacker could burn to avoid the loss. But this is standard DeFi awareness, not a protocol vulnerability.
- Verdict: KNOWN RISK (stETH inherent), not protocol vulnerability.

**Part 2: Race Condition Analysis**

Analyze 5 specific race conditions:

**Race 1: User burn vs handleGameOverDrain (same block)**
- Scenario: User burn tx and gameOver tx both in same block.
- EVM behavior: Transactions in a block execute SEQUENTIALLY. Order determined by miner/block builder.
- Case A (burn first): User burns at pre-gameOver prices. totalSupply and reserves both reduce proportionally. Then gameOver executes, burning remaining pool tokens. Remaining holders get higher per-token value. FAIR.
- Case B (gameOver first): gameOver sets flag, burns pool tokens (reducing totalSupply without reducing reserves). Then user burns at POST-gameOver prices, getting higher per-token value. FAIR -- same result as any burn after gameOver.
- Verdict: No race condition -- both orderings produce correct results. The user either burns before or after gameOver, both valid.

**Race 2: User burn during GAME_OVER_PENDING_RNG (State 1)**
- Scenario: gameOver=true but gameOverFinalJackpotPaid=false (RNG not ready at line 126).
- State: gameOver flag is set, but burnRemainingPools() has NOT been called yet (that happens at line 163, after jackpot payment).
- Question: Can users burn in this state? Check burn() in StakedDegenerusStonk.sol -- does it check gameOver? Trace the burn function. If burn() has no gameOver check, users can burn during this pending state.
- Impact: In State 1, pool tokens are NOT yet burned. totalSupply includes pool tokens. User burns get proportional share based on CURRENT totalSupply (including pool tokens). This means their per-token value is LOWER than it will be after burnRemainingPools executes.
- Is this exploitable? An informed attacker would WAIT for State 2 (after burnRemainingPools) to burn, getting higher per-token value. Uninformed users might burn during State 1, getting less per-token value.
- Verdict: INFORMATIONAL -- users who burn during the pending RNG window get a lower per-token value than those who wait. This is not an exploit (they get exactly their proportional share of current reserves), but it creates an information asymmetry.

**Race 3: Multiple concurrent burns (same block)**
- Scenario: Two users submit burn txs in the same block.
- EVM: Sequential execution. First burn reads totalSupply=S, burns amount=A, gets (A/S)*M. Second burn reads totalSupply=S-A, burns amount=B, gets (B/(S-A))*(M - A/S*M).
- Math proof: Second burner gets (B/S)*M = exactly their proportional share of original reserves. Show the algebra: (B/(S-A)) * (M*(S-A)/S) = (B/S)*M. The order doesn't matter.
- Verdict: SAFE -- proportional formula guarantees each burner gets exactly their fair share regardless of ordering.

**Race 4: handleFinalSweep claimablePool zeroing**
- Scenario: User has unclaimed winnings (claimablePool > 0). handleFinalSweep at line 177 sets claimablePool = 0, forfeiting all unclaimed winnings.
- Impact on burns: sDGNRS.burn() reads `_claimableWinnings()` which includes claimablePool. After sweep, claimableWinnings = 0, but sDGNRS receives 50% of swept funds as a deposit. Per-token burn value changes composition (less claimable, more ETH) but the total should be analyzed.
- Trace line 188: `_sendToVault(totalFunds, stBal)`. This sends funds to vault and sDGNRS. The sDGNRS portion increases reserves. Net effect on burn value: reserves increase by the sDGNRS deposit, minus the forfeited claimable amount. Calculate whether this is net positive or negative for remaining burners.
- Verdict: Expected behavior -- 30-day unclaim window is documented. Remaining holders benefit from the sDGNRS deposit portion.

**Race 5: burnRemainingPools per-token value jump**
- Scenario: burnRemainingPools() at line ~359-367 reduces totalSupply by burning pool tokens but does NOT reduce reserves (ETH/stETH stay in contract).
- Impact: Each remaining token's burn value = totalMoney / totalSupply. After pool burn, totalSupply decreases but totalMoney stays the same. Per-token value increases.
- Can users front-run this? To front-run burnRemainingPools, they would need to buy DGNRS BEFORE the gameOver tx and burn AFTER burnRemainingPools. But buying DGNRS before gameOver is just normal market behavior -- they don't know gameOver is coming.
- More precisely: if gameOver tx is visible in mempool, an MEV bot could: flash-buy DGNRS on DEX (if listed), wait for gameOver + burnRemainingPools in same tx, then burn. But this requires DGNRS to be DEX-listed AND the value increase to exceed gas + slippage costs.
- Quantify: If pool tokens are X% of totalSupply, per-token value increases by X/(1-X) percent. If pools are 80% of supply, value increases by 400%. If pools are 20%, value increases by 25%. The actual pool token percentage depends on game progression.
- Verdict: KNOWN BEHAVIOR -- the per-token value increase is intentional (remaining holders benefit from unclaimed pool tokens). MEV extraction requires DEX liquidity (out of scope for protocol itself).
