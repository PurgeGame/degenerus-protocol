**Invariant 1: sDGNRS Supply Conservation**
- Statement: `sDGNRS.totalSupply == sum(sDGNRS.balanceOf[addr]) for all addr` at all times.
- Proof approach: Enumerate EVERY function that modifies `totalSupply` or `balanceOf` in StakedDegenerusStonk.sol:
  - `_mint(to, amount)`: totalSupply += amount, balanceOf[to] += amount. Sum preserved.
  - `burn(amount)`: totalSupply -= amount, balanceOf[msg.sender] -= amount. Sum preserved.
  - `burnRemainingPools()` at line ~359-367: totalSupply -= bal, balanceOf[address(this)] = 0 (where bal WAS balanceOf[address(this)]). Sum preserved.
  - `transfer(to, amount)`: balanceOf[from] -= amount, balanceOf[to] += amount. totalSupply unchanged. Sum preserved.
  - `transferFrom(from, to, amount)`: same as transfer. Sum preserved.
  - `wrapperTransferTo(to, amount)`: balanceOf[msg.sender] -= amount, balanceOf[to] += amount. totalSupply unchanged. Sum preserved.
  - `transferFromPool(pool, to, amount)`: balanceOf[address(this)] -= amount, balanceOf[to] += amount, poolBalances[pool] -= amount. totalSupply unchanged. Sum preserved.
  - `transferBetweenPools(from, to, amount)`: poolBalances[from] -= amount, poolBalances[to] += amount. balanceOf unchanged. Sum preserved.
- Verdict: INVARIANT HOLDS -- every path that modifies totalSupply correspondingly modifies balanceOf, and every path that moves balanceOf between addresses sums to zero.

**Invariant 2: Cross-Contract Supply Invariant**
- Statement: `sDGNRS.balanceOf[DGNRS_contract] >= DGNRS.totalSupply` at all times.
- Proof approach: Build on Phase 19 proof (v2.0-delta-core-contracts.md, Supply Invariant section). Enumerate all 6 modification paths:
  1. Constructor: sDGNRS mints to DGNRS address, DGNRS.totalSupply set to same amount. Equality holds.
  2. DGNRS.burn(amount): DGNRS.totalSupply -= amount (at DegenerusStonk.sol line ~158), sDGNRS.balanceOf[DGNRS] -= amount (at StakedDegenerusStonk.sol line ~401). Both decrease by same amount. Invariant preserved.
  3. DGNRS.unwrapTo(recipient, amount): DGNRS.totalSupply -= amount (line ~178), sDGNRS.balanceOf[DGNRS] -= amount (via wrapperTransferTo). Both decrease by same amount. Invariant preserved.
  4. DGNRS.transfer: DGNRS.totalSupply unchanged, sDGNRS.balanceOf[DGNRS] unchanged. Invariant preserved.
  5. sDGNRS operations that don't involve DGNRS contract: No effect on either side. Preserved.
  6. Can sDGNRS.balanceOf[DGNRS] increase without DGNRS.totalSupply increasing? Only via `transferFromPool(pool, DGNRS_addr, amount)` -- but this is onlyGame and the game would never transfer pool tokens to the DGNRS contract address (pool transfers go to player addresses). Even if it did, it would INCREASE the gap, STRENGTHENING the invariant.
- Verdict: INVARIANT HOLDS across all paths.

**Invariant 3: Backing Solvency**
- Statement: For any burn of `amount` tokens from total supply `S`, the proportional payout `(totalMoney * amount) / S` can always be fulfilled by actual reserves.
- Proof approach:
  - `totalMoney = ethBal + stethBal + claimableEth` at line ~389.
  - Payout: `totalValueOwed = (totalMoney * amount) / S`.
  - After claimWinnings (if needed), ethBal is re-read. The function then pays from ETH first, stETH for remainder.
  - Key: Does the function EVER promise more than it has?
  - Trace line ~410-430: if `totalValueOwed <= ethBal`, pays pure ETH. OK -- ethBal available.
  - If `totalValueOwed > ethBal`, calls claimWinnings to convert claimableEth to actual ETH. Then: `stethOut = totalValueOwed - ethBal` (line ~413). Then checks `if (stethOut > stethBal) revert Insufficient()` at line ~415.
  - The `Insufficient()` revert means: if stETH rounding causes a 1-2 wei discrepancy, the BURN REVERTS rather than paying out more than available. This is a SAFETY check that PREVENTS insolvency.
  - Edge case: Can `ethBal + stethBal < totalValueOwed` after claimWinnings? Only if claimableEth was overestimated (game owes more than it can pay). Trace `_claimableWinnings()` to verify it returns conservative estimates.
- Verdict: INVARIANT HOLDS -- the Insufficient() revert is the backstop. The protocol never pays out more than it has. Worst case: a burn reverts, which is safe.

**Invariant 4: Pool Balance Consistency (pre-gameOver)**
- Statement: `sDGNRS.balanceOf[address(this)] == sum(poolBalances[i] for all pools)` before gameOver.
- Reference Phase 19 proof and confirm no new paths invalidate it.
- After gameOver: burnRemainingPools sets balanceOf[address(this)] = 0 and totalSupply -= bal. poolBalances become stale but unreachable (all pool operations revert when gameOver == true or when pool balance is 0). Already documented in DELTA-I-01.
- Verdict: INVARIANT HOLDS (pre-gameOver), STALE BUT SAFE (post-gameOver).

End with an **Invariant Summary Table**: | Invariant | Statement | Holds? | Backstop | Evidence |

**Part 2: Escalation Path Analysis**

Analyze 4 potential escalation vectors:

**Escalation 1: Delegatecall to sDGNRS**
- Can any contract delegatecall into sDGNRS to bypass access control?
- Check: sDGNRS has no fallback with delegatecall. It's not a proxy. No delegatecall target.
- Trace: DegenerusGame.sol uses delegatecall to modules (DegenerusGameAdvanceModule, etc.). These modules execute in the game's storage context. When a module calls sDGNRS.transferFromPool(), the `msg.sender` is the game contract (because delegatecall preserves msg.sender of the original call to the game, which then delegatecalls to module, but the module's external call to sDGNRS has msg.sender = game contract address).
- Actually: Verify this carefully. In delegatecall chain: User -> Game.advance() -> delegatecall to AdvanceModule.advance() -> sDGNRS.transferFromPool(). In this case, msg.sender for the sDGNRS call is the Game contract (because the module code runs in Game's context, and external calls from delegated code use the delegating contract's address as msg.sender). Confirm this is correct.
- Verdict: No escalation -- delegatecall modules call sDGNRS as the game contract, which is the authorized caller.

**Escalation 2: Proxy Upgrade**
- sDGNRS is not upgradeable (no proxy pattern). Verify: no UUPS, no TransparentProxy, no diamond pattern.
- Check constructor of StakedDegenerusStonk.sol -- it's a plain contract, not behind a proxy.
- Verdict: No escalation -- contracts are not upgradeable.

**Escalation 3: CREATE2/Selfdestruct Address Collision**
- Could an attacker deploy a contract at the same address as the game contract after it selfdestructs?
- Post-Cancun (EIP-6780): selfdestruct only clears state in creation tx. The game contract will not selfdestruct. Even if it did, CREATE2 would need the exact same salt and initcode.
- Verdict: Theoretical only -- game contract has no selfdestruct, making this impossible.

**Escalation 4: msg.sender Spoofing via tx.origin**
- Check: Does sDGNRS use tx.origin anywhere? If it uses msg.sender for all access control, tx.origin spoofing is irrelevant.
- Trace: Search StakedDegenerusStonk.sol for "tx.origin" -- should find 0 occurrences. All access control is msg.sender-based.
- Verdict: No escalation -- msg.sender-only access control.

**Part 3: Worst-Case Privileged Actions**

For each privileged address, document the worst-case damage:
- GAME: Can drain all pool balances via transferFromPool. But game is a trusted contract deployed by the protocol. If game is compromised, the entire protocol is compromised regardless of sDGNRS.
- DGNRS: Can move sDGNRS from its own balance to any recipient via wrapperTransferTo. Limited to sDGNRS.balanceOf[DGNRS] (the wrapper's backing). Cannot create new tokens.
- CREATOR: Can unwrap DGNRS to sDGNRS for any recipient. Limited to CREATOR's DGNRS balance. Cannot access other users' tokens.

End with **Privilege Assessment Summary**: Overall verdict on whether any privilege escalation paths exist. Expected conclusion: NO escalation paths. All access control is immutable, msg.sender-based, and cannot be spoofed or bypassed.
