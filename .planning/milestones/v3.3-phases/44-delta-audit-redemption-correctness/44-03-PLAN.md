# Phase 44 Accounting, Solvency, and Interaction Audit

## Accounting Reconciliation (DELTA-01)

### ETH Accounting: pendingRedemptionEthValue

Trace EVERY line in StakedDegenerusStonk.sol that reads or writes `pendingRedemptionEthValue`. For each mutation site, document:
- Exact line number
- Direction (increment/decrement)
- Formula
- Context (which function, when called)

Expected mutation sites (verify against actual code):
1. **Submit (increment):** `_submitGamblingClaimFrom` -- `pendingRedemptionEthValue += ethValueOwed` (segregation at submit)
2. **Resolve (adjustment):** `resolveRedemptionPeriod` -- `pendingRedemptionEthValue = pendingRedemptionEthValue - pendingRedemptionEthBase + rolledEth` (adjustment to rolled amount)
3. **Claim (decrement):** `claimRedemption` -- `pendingRedemptionEthValue -= ethPayout`

For each mutation, verify:
- Does the increment at submit correctly represent the player's proportional share?
- Does the adjustment at resolve correctly convert from base to rolled value?
- Does the decrement at claim correctly remove the player's individual rolled payout?

### Rounding Analysis

Construct a numerical example with worst-case parameters:
- N = 100 players each burning 1 sDGNRS
- totalMoney = 10 ETH (10^19 wei)
- roll = 175 (maximum)

Calculate:
1. Per-player ethValueOwed = (10^19 * 1) / 100 = 10^17 wei each
2. pendingRedemptionEthBase = sum = 10^19 exactly (no rounding -- all divisions are exact in this case)
3. rolledEth = (10^19 * 175) / 100 = 1.75 * 10^19 wei
4. pendingRedemptionEthValue after resolve = old - base + rolledEth

Now per-player claim:
5. ethPayout = (10^17 * 175) / 100 = 1.75 * 10^17 wei (exact)
6. sum of all ethPayouts = 100 * 1.75 * 10^17 = 1.75 * 10^19 (matches rolledEth)

Now construct a case where rounding DOES occur:
- N = 3 players, amounts = [7, 11, 13] sDGNRS (total 31)
- totalMoney = 10^18 wei
- roll = 137

Calculate each step showing integer division truncation. Show that `sum(individual ethPayouts)` may differ from `rolledEth` by up to N-1 wei. Document the direction of dust accumulation (does pendingRedemptionEthValue increase or decrease over many periods?).

Verdict: Is the rounding safe? What is the maximum dust accumulation rate per period?

## Segregation Solvency Proof (CORR-02)

### Invariant Statement

Prove: At every point in execution, `pendingRedemptionEthValue <= (address(this).balance + steth.balanceOf(this) + game.claimableWinnings())`.

For each lifecycle phase, show the balance sheet:

**After Submit:**
- pendingRedemptionEthValue increased by ethValueOwed
- But no ETH left the contract (burn only destroys tokens, doesn't move ETH)
- Holdings unchanged, reservation increased
- SAFE if: ethValueOwed <= proportional share of holdings (it is, by construction: ethValueOwed = totalMoney * amount / supply where totalMoney = holdings - existing reservations)

**After Resolve:**
- pendingRedemptionEthValue adjusted: `old - base + rolledEth`
- If roll > 100: pendingRedemptionEthValue increases (player gets more than base)
- If roll < 100: pendingRedemptionEthValue decreases (player gets less than base)
- Worst case: roll = 175, pendingRedemptionEthValue increases by 0.75 * base
- SAFE if: even at max roll, total segregated value <= total holdings
- Prove: At submit, `ethValueOwed = totalMoney * amount / supply` where `totalMoney = holdings - pendingRedemptionEthValue`. So `pendingRedemptionEthValue + ethValueOwed = pendingRedemptionEthValue + (holdings - pendingRedemptionEthValue) * amount/supply`. For single-player max burn (amount = supply/2 from 50% cap): new reservation = old_reservation + (holdings - old_reservation) * 0.5. After resolve with roll=175: reservation = 1.75 * new_base. Calculate whether this exceeds holdings.
- CRITICAL: What happens with MULTIPLE periods? Period 1 resolves at 175%, then period 2 submits claiming proportional share of (holdings - period1_reservation). Then period 2 resolves at 175%. Can cumulative reservations exceed holdings?

**After Claim:**
- pendingRedemptionEthValue decreased by ethPayout
- ETH sent to player via _payEth
- Both reservation and holdings decrease by approximately same amount
- SAFE (reservation can only decrease, holdings decrease by payment)

### BURNIE Solvency

Verify: When _payBurnie is called, sDGNRS has sufficient BURNIE balance.
1. Trace: BURNIE credited to sDGNRS via `coin.creditFlip(SDGNRS, burnieToCredit)` at resolution
2. Trace: `claimCoinflipsForRedemption(sDGNRS, remaining)` mints additional BURNIE to sDGNRS at claim time
3. Edge case: Can another code path drain sDGNRS's BURNIE balance between resolution and claim? Search for all paths that reduce sDGNRS's BURNIE balance:
   - Deterministic burns call `_payBurnie` which transfers from sDGNRS balance
   - Any other transfer calls?
4. Document whether race condition exists between concurrent claims

### Known Issues from Findings

Reference the CP-08 finding from Plan 01: if `_deterministicBurnFrom` doesn't subtract `pendingRedemptionEthValue`, post-gameOver deterministic burns include reserved ETH in their proportional share. This creates a solvency violation where total promised payouts exceed holdings. Quantify the maximum solvency gap.
