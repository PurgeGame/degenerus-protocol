## Period State Machine Proof (CORR-04)

### Monotonicity Proof

Prove that `redemptionPeriodIndex` only advances (never decreases or revisits a previous period):

1. Find every line in StakedDegenerusStonk.sol that writes to `redemptionPeriodIndex`. There should be exactly ONE write site (inside `_submitGamblingClaimFrom`).
2. Document the guard: `if (redemptionPeriodIndex != currentDay)` -- the new period index is `currentDayView()` which is derived from `GameTimeLib.currentDayIndexAt(block.timestamp)`.
3. Prove monotonicity: `currentDayIndexAt` is `(block.timestamp - JACKPOT_RESET_TIME) / 1 days - DEPLOY_DAY_BOUNDARY + 1`. Since `block.timestamp` only increases, `currentDayIndexAt` only increases. Therefore `redemptionPeriodIndex` can only advance to higher values.
4. Document the edge case: multiple burns in the same period do NOT update `redemptionPeriodIndex` (guarded by the `!=` check).

### Resolution Ordering Proof

Prove that `resolveRedemptionPeriod()` is called at most once per period:

1. Find the call site: `rngGate()` in DegenerusGameAdvanceModule.sol.
2. Document the guard: `sdgnrs.hasPendingRedemptions()` -- what does this check? It reads `pendingRedemptionEthBase != 0 || pendingRedemptionBurnieBase != 0`.
3. After resolution, `pendingRedemptionEthBase` and `pendingRedemptionBurnieBase` are both set to 0. Therefore `hasPendingRedemptions()` returns false, preventing double-resolution.
4. Confirm: `redemptionPeriods[periodIndex].roll` is set once and never overwritten (no other write site).

### 50% Supply Cap Enforcement

Prove the 50% cap is correctly enforced:

1. Find the cap check in `_submitGamblingClaimFrom`: `if (redemptionPeriodBurned > redemptionPeriodSupplySnapshot / 2) revert ExceedsRedemptionCap()`
2. Document when `redemptionPeriodSupplySnapshot` is set: on the first burn of a new period, `redemptionPeriodSupplySnapshot = totalSupply()`.
3. Prove the cap is per-period: when a new period starts, `redemptionPeriodBurned` resets to 0 and `redemptionPeriodSupplySnapshot` captures current supply.
4. Edge case analysis: Can an attacker manipulate `totalSupply()` between the snapshot and subsequent burns? Since sDGNRS has no mint function (only burns reduce supply), the snapshot is the highest supply within the period. Subsequent burns can only reduce supply below the snapshot. The cap is therefore conservative (allows burning up to 50% of the supply at period start, not 50% of current supply).
5. Document: `redemptionPeriodBurned` increments by `amount` AFTER the `_burn` call (or before? check exact ordering). Note: the increment is checked AFTER increment via `>` not `>=`.

## Supply Invariant Proof (CORR-05)

### burnWrapped() Dual Burn Trace

Prove that `burnWrapped(amount)` correctly burns equal amounts of both sDGNRS and DGNRS:

1. Trace `burnWrapped(uint256 amount)` in StakedDegenerusStonk.sol:

2. Jump to `DegenerusStonk.burnForSdgnrs(address player, uint256 amount)`:

   - Step 2: Calls `stonk.burn(amount)` -- calls back into sDGNRS.burn()

3. Back in `StakedDegenerusStonk.burn(uint256 amount)`:
   - The gambling path calls `_submitGamblingClaim(msg.sender, amount)` which calls `_burn(msg.sender, amount)` -- burns `amount` sDGNRS. BUT msg.sender here is DGNRS, not the player. So sDGNRS is burned from DGNRS's balance, not the player's.
