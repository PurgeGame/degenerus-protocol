**CP-08 (DELTA-03): Deterministic Burn Double-Spend**
1. Read `_deterministicBurnFrom` in StakedDegenerusStonk.sol -- find the line where `totalMoney` is computed. Confirm whether `pendingRedemptionEthValue` is subtracted.

3. Read `_submitGamblingClaimFrom` in StakedDegenerusStonk.sol -- find the line where `totalMoney` is computed. Confirm it subtracts `pendingRedemptionEthValue`.
4. Compare all three. If `_deterministicBurnFrom` does NOT subtract but the other two DO, verdict = CONFIRMED HIGH.
5. Also check `totalBurnie` computation in all three -- same pattern expected for `pendingRedemptionBurnie`.
6. Recommended fix: Add `- pendingRedemptionEthValue` and `- pendingRedemptionBurnie` to `_deterministicBurnFrom`'s `totalMoney` and `totalBurnie` computations.

**CP-06 (DELTA-04): Stuck Claims at Game-Over**
1. Read `rngGate()` in DegenerusGameAdvanceModule.sol -- find lines that check `sdgnrs.hasPendingRedemptions()` and call `sdgnrs.resolveRedemptionPeriod()`. Note exact line numbers.
2. Read `_gameOverEntropy()` in DegenerusGameAdvanceModule.sol -- search for any call to `sdgnrs.resolveRedemptionPeriod()` or `sdgnrs.hasPendingRedemptions()`.
3. If `_gameOverEntropy()` does NOT call `resolveRedemptionPeriod()`, verdict = CONFIRMED HIGH. Players who burned sDGNRS in the last active period before game-over have permanently lost tokens with no claim path.
4. Recommended fix: Add redemption resolution block to `_gameOverEntropy()` mirroring the `rngGate()` pattern (check hasPendingRedemptions, compute roll from currentWord, call resolveRedemptionPeriod, credit BURNIE).

**Seam-1 (DELTA-05): DGNRS.burn() Fund Trap**
1. Read `DegenerusStonk.burn()` -- confirm it calls `stonk.burn(amount)` where `stonk` = sDGNRS contract.
2. Note that when DegenerusStonk calls `stonk.burn()`, `msg.sender` inside sDGNRS = address(DegenerusStonk), NOT the original player.
3. Read `StakedDegenerusStonk.burn()` -- confirm that when `!game.gameOver()`, it calls `_submitGamblingClaim(msg.sender, amount)` recording the claim under `msg.sender`.
4. If `msg.sender` = DegenerusStonk address and DegenerusStonk has no `claimRedemption()` function, verdict = CONFIRMED HIGH. The gambling claim is orphaned -- tokens burned, no claim possible.
5. Check: Does the deterministic burn path (`game.gameOver() == true`) also have this issue? It sends assets to msg.sender = DGNRS, which then forwards to the player. If deterministic path works but gambling path doesn't, document this asymmetry.
6. Recommended fix: Three options -- (a) revert `DGNRS.burn()` during active game (simplest; DGNRS holders use `burnWrapped()` instead), (b) add `burnFor(address beneficiary, uint256 amount)` to sDGNRS called by DGNRS with the original player address, (c) modify DGNRS.burn() to call `burnWrapped()` path instead. Document tradeoffs of each.

**CP-07 (DELTA-07): Coinflip Resolution Stuck-Claim**
1. Read `claimRedemption()` in StakedDegenerusStonk.sol -- find the line that reads `coinflip.getCoinflipDayResult(period.flipDay)`. Note that `rewardPercent` and `flipWon` must be resolved (non-zero) for the claim to proceed.

4. Check: does `_gameOverEntropy()` call `coinflip.processCoinflipPayouts()` for all remaining days? If yes, the coinflip for day N+2 would be resolved at game-over. Read the _gameOverEntropy loop carefully -- does it iterate through remaining days and call processCoinflipPayouts for each?

6. Recommended fix: Two options -- (a) split claim into ETH-only and BURNIE-only paths so ETH can be claimed independently, (b) add emergency resolution function that allows claiming after game-over with a default flipWon=false.
