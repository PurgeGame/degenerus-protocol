**Section 4: Bank-Run Scenario Analysis**

Model the worst-case bank-run scenario where many players burn simultaneously near the supply cap. Analyze the following sub-scenarios:

**4.1: Maximum Single-Period Burn**
- Setup: All holders decide to burn in the same period
- Constraint: 50% supply cap limits burns to `totalSupply / 2` per period
- Model: N players each burn their full balance, up to the cap
- Outcome: `redemptionPeriodBurned` reaches `redemptionPeriodSupplySnapshot / 2`, then subsequent burns revert with `Insufficient()` (cite exact line)
- Key insight: The 50% cap is a circuit breaker -- no more than half of supply can be redeemed in a single period

**4.2: Sequential Period Exhaustion (Multi-Period Bank Run)**
- Setup: Players burn 50% of supply in period 1, 50% of remaining in period 2, etc.
- Model:
  - Period 1: Supply = S, burned = S/2, remaining = S/2, new snapshot = S/2
  - Period 2: Supply = S/2, burned = S/4, remaining = S/4, new snapshot = S/4
  - Period N: Supply = S/2^N, burned = S/2^(N+1)
  - After K periods: totalSupply = S/2^K (geometric decay, never reaches zero)
- Key insight: Supply asymptotically approaches zero but never reaches it. Each period's cap is half of the CURRENT supply, not the original.

**4.3: Multi-Period Cumulative Reservation vs Holdings**
- Use the Phase 44 contraction mapping proof: `P_new = 0.125 * P_old + 0.875 * H`
  - P = pendingRedemptionEthValue (reserved ETH)
  - H = address(this).balance + steth.balanceOf(this) + claimableWinnings() (total holdings)
- Verify this holds under the bank-run scenario:
  - After period 1: P_1 = 0.125 * 0 + 0.875 * H_0 = 0.875 * H_0 (worst case: 87.5% reserved)
  - Wait -- this is wrong. The 50% cap means at most 50% of supply burns. Each burner gets proportional share. So P_1 = sum of ethValueOwed for all burners = sum((H * amount_i) / S) where sum(amount_i) <= S/2. So P_1 <= H * (S/2) / S = H/2 = 50% of holdings. This is LESS than the contraction mapping suggests.
  - Actually re-derive properly: cite the exact formula from StakedDegenerusStonk.sol where ethValueOwed is computed, using `totalMoney = address(this).balance + steth.balanceOf(this) + claimableWinnings() - pendingRedemptionEthValue - pendingRedemptionBurnie` (cite exact line for totalMoney computation). The key is that totalMoney EXCLUDES already-reserved funds.
  - P_after = P_before + sum((totalMoney * amount_i) / supplyBefore) for all burns in the period
  - With 50% cap: sum(amount_i) = supplyBefore / 2 at max
  - P_after = P_before + totalMoney * (supplyBefore/2) / supplyBefore = P_before + totalMoney / 2
  - totalMoney = H - P_before (gross holdings minus existing reservations)
  - P_after = P_before + (H - P_before) / 2 = P_before/2 + H/2
  - Show: P_after < H iff P_before < H (always true since P starts at 0 and H is total holdings)
  - Conclusion: The solvency invariant (P < H) is maintained at every step

**4.4: Claim Phase During Bank Run**
- After burns are resolved, players claim. Claims reduce P by ethValueOwed and reduce H by ethPayout.
- If roll > 100: player gets more than ethValueOwed (up to 1.75x). Does this break solvency?
- Show: ethPayout = ethValueOwed * roll / 100. For roll = 175 (max): ethPayout = 1.75 * ethValueOwed.
- But the AGGREGATE across all claimants has E[roll/100] = 1.0. With 50% cap and many claimants, the law of large numbers applies. Some get more, some get less, but total outflow averages to total reservation.
- Edge case: small number of claimants with max rolls. Show: even if ALL claimants get roll=175, the total payout = 1.75 * sum(ethValueOwed) = 1.75 * P. If 1.75 * P > H, the contract cannot pay. But P <= H/2 (from 4.3), so 1.75 * P <= 1.75 * H/2 = 0.875 * H < H. The contract remains solvent even in the absolute worst case (all max rolls).
