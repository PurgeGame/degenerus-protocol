**Section 1: Cross-Contract Composability Attack Catalog**

**Mandatory attack sequences to test (minimum -- add more if analysis reveals additional paths):**

1. **Burn-then-claim-in-same-tx:** Can a contract call `sDGNRS.burn()` and then `sDGNRS.claimRedemption()` in the same transaction? Trace the state machine -- burn creates a PENDING claim, claim requires RESOLVED status (which requires `resolveRedemptionPeriod` from AdvanceModule). Outcome should be SAFE (claim reverts with UnresolvedClaim).

2. **Flash-loan-burn:** Can a flash-loaned sDGNRS be burned for gambling claim? Trace: flash loan sDGNRS -> burn -> claim. The burn reduces balance, but the claim requires resolution in a future transaction. The flash loan must be repaid in the same tx. Outcome should be SAFE (claim is deferred to future tx).

3. **Re-enter-via-_payEth:** When `_payEth` sends ETH to player via `player.call{value:}`, can the player's fallback re-enter any sDGNRS function? Trace the CEI pattern: claim data is deleted BEFORE the ETH transfer. Re-entry would find no claim to exploit.

4. **Re-enter-via-stETH-fallback:** When `_payEth` falls back to `stETH.transfer(player, amount)`, can the stETH transfer trigger a callback that re-enters sDGNRS? Trace: stETH uses `_transfer` which is a simple balance update (ERC20), no callback hooks. Outcome should be SAFE.

5. **Delegatecall-module-redemption-bypass:** Can a delegatecall module in DegenerusGame access sDGNRS storage directly via the shared storage layout? Trace: sDGNRS is a separate contract, not part of the delegatecall chain. Modules execute in DegenerusGame's context, which has no sDGNRS storage. Outcome should be SAFE.

6. **Multiple-burn-before-resolution:** Can a player call `burn()` multiple times before `advanceGame` resolves the period? Trace: second `burn()` calls `_submitGamblingClaimFrom`, which checks `claims[player].periodIndex != 0` (UnresolvedClaim revert at line ~720). Outcome should be SAFE.

7. **AdvanceModule-redemption-skip:** Can `advanceGame` be called in a way that skips `resolveRedemptionPeriod`? Trace: `rngGate` calls `sDGNRS.resolveRedemptionPeriod` only when `sDGNRS.hasPendingRedemptions()` returns true. If no pending redemptions, the call is correctly skipped. Check: what if a burn happens between VRF request and fulfillment? The `rngLocked` guard prevents burns during VRF resolution window.

8. **BurnieCoinflip-direct-call:** Can anyone call `claimCoinflipsForRedemption` directly (bypassing sDGNRS)? Trace: guard at line 348 checks `msg.sender != ContractAddresses.SDGNRS`. Direct external call reverts. Outcome should be SAFE.

9. **burnForSdgnrs-direct-call:** Can anyone call `burnForSdgnrs` directly (bypassing sDGNRS)? Trace: guard at line 238 checks `msg.sender != ContractAddresses.SDGNRS`. Direct external call reverts. Outcome should be SAFE.

10. **Cross-period-claim-manipulation:** Can a player submit a burn in period N, wait for resolution, then manipulate period N+1 to get a better roll for their period N claim? Trace: the roll is determined at resolution time by `resolveRedemptionPeriod` using the VRF word for the advancing day. The player's claim is locked to the roll computed at resolution. They cannot re-resolve or change it.
