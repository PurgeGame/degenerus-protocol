**D-256-CANCEL-QUEUED-01 verdict (recorded inline per plan-body requirement):**

The branch at `contracts/GNRUS.sol:382-391` (cancellation path: `currentSlate[slot] == 0 AND pendingEditSet[slot] == 1 AND recipient == 0`) is **structurally unreachable** under current invariants:

- `pendingEditSet` bit `i` is SET only inside the queue branch (L405): `pendingEditSet = pendingEditSet | slotMask`. The queue branch fires only when `current != address(0)` (L399 `else` of L380).
- `pendingEditSet` bit `i` is CLEARED in two places: (a) the cancellation path itself L388, (b) the start-of-pickCharity flush L630 (`pendingEditSet = 0`).
- Bit `i` cannot be set without `currentSlate[i]` being non-zero at the moment of the set.
- After flush at L612-630, `pendingEditSet = 0` AND `currentSlate[i]` reflects the flushed value (which may be `address(0)` if a remove was applied — but `pendingEditSet` is also zeroed, so the bit is no longer set).
- Therefore the state `(currentSlate[slot] == 0 AND pendingEditSet[slot] == 1)` cannot be reached from any sequence of external calls.

**Disposition:** Defensive guard, no test path. Plan 03a does NOT add a test driving this branch (would require contract surgery to construct). Phase 257 audit will cite this verdict and grep-prove unreachability as a SAFE row in AUDIT-02.

**CapExceeded structural unreachability verdict (Blocker #2 resolution — parallel to D-256-CANCEL-QUEUED-01):**

Per checker-requested re-verification of `_futureBitmapAfter` at `contracts/GNRUS.sol:416-444` traced against the post-Phase-255 HEAD:

- `future` is initialized to `currentActiveBitmap` (uint32). `currentActiveBitmap` is only ever modified via `currentActiveBitmap | slotMask` (L397) where `slotMask = uint32(1) << slot` and `slot < MAX_ACTIVE_SLOTS = 20` is enforced at L371. So `currentActiveBitmap` only has bits 0-19 settable; bits 20-31 are structurally always 0.
- `pSet = pendingEditSet | slotMask`. `pendingEditSet` is only modified via the queue branch L405 with the same `slotMask` constraint, so `pendingEditSet` also only has bits 0-19 settable.
- The loop body at L427-443 iterates `i = 0..19` (`i < MAX_ACTIVE_SLOTS = 20`). For each iteration:
  - If bit `i` is set in `pSet`: either set bit `i` in `future` (via `future | mask_i`) or clear bit `i` in `future` (via `future & ~mask_i`).
  - If bit `i` is NOT set in `pSet`: `future` bit `i` is unchanged.
- For bits NOT in pSet, bit `i` of `future` retains its value from `currentActiveBitmap` (which is 0 for any bit `i >= 20` and either 0 or 1 for `i < 20`).
- For bits in pSet (i.e., `i < 20`), bit `i` of `future` is either set or cleared.
- **Therefore future bits 20-31 are always 0**, and only bits 0-19 can be set.
- **Maximum `_popcount32(future)` is 20.** The check `_popcount32(future) > MAX_ACTIVE_SLOTS` (20) at L394 + L402 can never be true from any external call sequence.

**Disposition:** CapExceeded is defensively guarded but mathematically unreachable from external calls. Plan 03a does NOT add a positive CapExceeded test (would require `hardhat_setStorageAt` to forge `currentActiveBitmap` or `pendingEditSet` to a corrupt state with bits 20+ set, which would test artificial state, not real reachability — explicitly forbidden by the orchestrator's Blocker #2 instructions).

The plan DOES add a 20-slot fill smoke test (instant-apply slots 0-19 succeeds) and verifies `activeCount() == 20` after. The plan also adds inline documentation citing this verdict for Phase 257 AUDIT-02 SAFE-row use.
