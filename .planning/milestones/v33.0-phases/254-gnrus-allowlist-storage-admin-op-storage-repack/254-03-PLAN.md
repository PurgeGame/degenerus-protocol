### Signatures

```solidity
function getCharity(uint8 slot) external view returns (address);
function getActiveSlots() external view returns (uint8[] memory slots, address[] memory recipients);
function getPendingEdits() external view returns (uint8[] memory slots, address[] memory recipients);
function activeCount() external view returns (uint8);
function activeCountAfterFlush() external view returns (uint8);
```

### Behaviors

**`getCharity(uint8 slot)`**:
- Bounds-check `slot >= MAX_ACTIVE_SLOTS` → revert `InvalidSlot()` (consistent with setCharity revert).
- Return `currentSlate[slot]` (may be `address(0)` for empty slots).

**`getActiveSlots()`**:
- Length = `_popcount32(currentActiveBitmap)`.
- Iterate slots 0..MAX_ACTIVE_SLOTS-1; for each set bit in `currentActiveBitmap`, push `(slotIndex, currentSlate[slotIndex])` into the paired arrays.
- Empty slate → `(uint8[](0), address[](0))`.

**`getPendingEdits()`**:
- Length = `_popcount32(pendingEditSet)`.
- Iterate slots 0..MAX_ACTIVE_SLOTS-1; for each set bit in `pendingEditSet`, push `(slotIndex, pendingEdit[slotIndex])` into the paired arrays.
- `recipients[i] == address(0)` is meaningful — signals queued-remove (per D-254-PENDING-01 sentinel semantics).

**`activeCount()`**:
- Return `_popcount32(currentActiveBitmap)`.
- Single SLOAD (hot-pack slot warm if called in same tx as setCharity).

**`activeCountAfterFlush()`**:
- Compose future bitmap from current ± all pending edits applied as if flush ran.
- For each set bit in `pendingEditSet`: if `pendingEdit[slot] == 0` clear future bit, else set future bit.
- Return `_popcount32(future)`.

### Visibility / state mutability

- `getCharity`, `activeCount`, `activeCountAfterFlush`: `external view`
- `getActiveSlots`, `getPendingEdits`: `external view` (memory-only allocation; gas scales with active count bounded by 20)

### Why paired arrays (D-254-VIEW-01)

- Single round-trip from indexer/frontend (no per-slot `getCharity` followup).
- Sparse `uint8[]`-only-and-call-`getCharity`-per-slot rejected for round-trip cost.
- Raw bitmap exposure rejected — paired arrays ARE the enumerator per ALW-04 ("getActiveSlots() returns (uint8[]) (or equivalent enumerator)").

### Memory allocation cost (paired arrays at length 20)

- Two `uint8[20]` + `address[20]` dynamic arrays: 20 × 32 bytes × 2 = 1280 bytes.
- Solidity dynamic-array allocation: 1 length word + 20 elements per array.
- Memory expansion gas (~3 gas per word) negligible at this size.

### Invariants worth measuring (Phase 256 task)

- popcount monotonic w.r.t. add (instant-apply or queue-add): `activeCountAfterFlush` increases by exactly 1.
- popcount monotonic w.r.t. queue-remove on filled slot: `activeCountAfterFlush` decreases by exactly 1.
- popcount stable w.r.t. queue-replace on filled slot (recipient nonzero → nonzero): unchanged.
- `activeCount()` always equals `_popcount32(currentActiveBitmap)` (trivially true by construction).

```solidity
/// @notice Enumerate all active charity slots as paired arrays
/// @return slots The slot indices that hold a non-zero recipient (length = activeCount())
/// @return recipients The recipients at those slots (recipients[i] is the recipient at slots[i])
function getActiveSlots() external view returns (uint8[] memory slots, address[] memory recipients) {
    uint32 bitmap = currentActiveBitmap;
    uint8  len    = _popcount32(bitmap);
    slots      = new uint8[](len);
    recipients = new address[](len);
    uint8 j;
    for (uint8 i; i < MAX_ACTIVE_SLOTS;) {
        if ((bitmap & (uint32(1) << i)) != 0) {
            slots[j]      = i;
            recipients[j] = currentSlate[i];
            unchecked { ++j; }
        }
        unchecked { ++i; }
    }
}
```

**6. Implement `_flushedBitmap()` internal pure (or view) helper.** Composes the would-be future bitmap from `currentActiveBitmap` ± all pending edits applied as-if flush ran (no proposed write — distinct from `_futureBitmapAfter` which models a proposed setCharity write):

```solidity
/// @dev Compose the future currentActiveBitmap as if all pending edits flushed.
///      For each set bit in pendingEditSet: pending value zero  → clear bit (pending-remove);
///                                            pending value !zero → set bit (pending-add/replace).
/// @return future The post-flush currentActiveBitmap
function _flushedBitmap() private view returns (uint32 future) {
    future = currentActiveBitmap;
    uint32 pSet = pendingEditSet;
    for (uint8 i; i < MAX_ACTIVE_SLOTS;) {
        uint32 mask_i = uint32(1) << i;
        if ((pSet & mask_i) != 0) {
            if (pendingEdit[i] == address(0)) {
                future = future & ~mask_i;
            } else {
                future = future | mask_i;
            }
        }
        unchecked { ++i; }
    }
}
```

Position this helper directly under `_futureBitmapAfter` from Plan 02 — keeps both bitmap-composition helpers co-located.
