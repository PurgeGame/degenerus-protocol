**Section 1: Ticket Queue Write-Buffer Vulnerability (TQ-01)**

Write a complete exploitation scenario with these subsections:

1.1 **Vulnerability Summary** -- One paragraph: _awardFarFutureCoinJackpot at JackpotModule:2544 reads ticketQueue[_tqWriteKey(candidate)] instead of ticketQueue[_tqReadKey(candidate)]. This means the function reads from the active write buffer that purchase() can modify, not the frozen read buffer. Contrast with processTicketBatch (JackpotModule:1891) which correctly uses _tqReadKey.

1.2 **Phase 69 Verdict Correction** -- Note that the Phase 69 verdict at lines 1416-1423 of this document states "ticketQueue[readKey] is the far-future winner pool" (line 1422). This is incorrect for _awardFarFutureCoinJackpot which uses _tqWriteKey. State the corrected verdict: "ticketQueue double-buffer is SAFE for processTicketBatch (uses _tqReadKey) but VULNERABLE for _awardFarFutureCoinJackpot (uses _tqWriteKey)."

1.3 **Preconditions** -- List all 4 preconditions from research: (a) jackpotPhaseFlag = true, (b) rngLockedFlag = true (daily VRF requested), (c) rngWordCurrent visible on-chain after rawFulfillRandomWords, (d) payDailyJackpotCoinAndTickets not yet called.

1.4 **Attack Sequence** -- Step-by-step with contract line references:
- Step 1: Monitor chain for rawFulfillRandomWords tx
- Step 2: Read rngWordCurrent via eth_getStorageAt(gameProxy, slot 4)
- Step 3: Precompute entropy derivation (quote the exact code from JackpotModule:2529-2548 showing entropy = rngWord ^ (uint256(lvl) << 192) ^ uint256(FAR_FUTURE_COIN_TAG), the entropyStep loop, candidate = lvl + 5 + uint24(entropy % 95), and idx = (entropy >> 32) % queue.length)
- Step 4: Call purchase() to add tickets at target far-future level(s). Trace: purchase() -> _purchaseFor() (MintModule:619, NO rngLockedFlag check) -> _queueTickets() -> pushes to ticketQueue[_tqWriteKey(targetLevel)]
- Step 5: When advanceGame() resumes, payDailyJackpotCoinAndTickets -> _awardFarFutureCoinJackpot reads attacker's freshly-added ticket from write buffer

1.5 **Outcome Manipulation** -- Attacker guarantees selection as one of up to FAR_FUTURE_COIN_SAMPLES (10) winners, receiving share of FAR_FUTURE_COIN_BPS (2500 = 25%) of daily BURNIE jackpot budget.

**Section 2: Fix Identification and Verification (TQ-02)**

2.1 **Fix Option A: Change _tqWriteKey to _tqReadKey** -- Analyze by tracing the far-future ticket lifecycle:
- When a player purchases at level L, tickets go to ticketQueue[_tqWriteKey(L)].
- _swapTicketSlot flips ticketWriteSlot, making old write key become the new read key.
- For far-future levels (candidate = lvl + 5 to lvl + 99), trace: does the write buffer for these levels EVER get swapped to become the read buffer? Check if _swapTicketSlot is a GLOBAL swap (flips ticketWriteSlot which affects ALL level keys) or per-level.
- Read _swapTicketSlot at DegenerusGameStorage:713-718 to verify: it flips ticketWriteSlot (a single global bit), so ALL levels swap simultaneously. This means Fix A DOES work -- far-future level read buffers contain data from pre-swap writes.
- Document: "Fix A is correct. _swapTicketSlot is a global swap, not per-level. When the buffer swaps at _swapAndFreeze time, ALL levels' write buffers become read buffers simultaneously."

2.2 **Fix Option B: Add rngLockedFlag to _purchaseFor** -- Document tradeoffs: blocks all purchases during commitment window. May be overly broad (blocks legitimate purchases that don't affect outcomes).

2.3 **Recommended Fix** -- Fix Option A (change _tqWriteKey to _tqReadKey at JackpotModule:2544). One-line change, minimal impact, aligns with processTicketBatch pattern.

2.4 **Fix verification** -- After applying Fix A, the function reads from the frozen read buffer. New purchases during commitment window write to the write buffer, which is invisible to _awardFarFutureCoinJackpot. The same double-buffer protection that makes processTicketBatch safe now also protects _awardFarFutureCoinJackpot.
