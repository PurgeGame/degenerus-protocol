### Cross-Day Carry-Over Analysis (DAYRNG-03)

This section proves that day N pending state cannot leak into or contaminate day N+1 RNG outcomes. "Contamination" is defined precisely: day N's RNG OUTCOME (specific random selections made) influencing day N+1's RNG WORD or OUTCOME SELECTION mechanism. State that persists across days as game context (prize pool sizes, ticket arrays, counter positions) is NOT contamination -- it is the game state the RNG operates ON.

**Part 1: _unlockRng Day Boundary Reset**

Document the 5 state resets performed by `_unlockRng(day)` at AdvanceModule line 1409-1415 (verify exact lines from contract source):

| Variable | Reset Value | Purpose | Why This Prevents Contamination |
|----------|------------|---------|-------------------------------|
| dailyIdx | day (current) | Advances monotonic day counter | Next advanceGame detects new day via `if (day == dailyIdx)` check; prevents reprocessing same day |
| rngLockedFlag | false | Releases daily lock | Allows new reverseFlip/burns; day N+1 starts with clean lock state |
| rngWordCurrent | 0 | Clears consumed VRF word | Day N's word cannot leak forward; day N+1 must obtain fresh VRF word |
| vrfRequestId | 0 | Clears VRF request state | Day N+1's request gets fresh ID from coordinator; no stale ID matching |
| rngRequestTime | 0 | Clears request timestamp | Day N+1's timeout logic starts fresh |

For each variable, cite the exact line in _unlockRng where the reset occurs.

**Part 2: rngWordByDay Immutability Proof**

Prove that `rngWordByDay[day]` is write-once:
1. Written by `_applyDailyRng` at AdvanceModule line ~1533 (verify): `rngWordByDay[day] = finalWord`
2. Protected by `rngGate` early-return at AdvanceModule line ~776 (verify): `if (rngWordByDay[day] != 0) return rngWordByDay[day]`

4. Grep the contract for `rngWordByDay\[` or `rngWordByDay [` to confirm no other write paths exist.
5. Conclusion: once written, a day's RNG word is immutable. Day N+1's processing cannot read or modify day N's archived word in any way that affects day N+1's outcome.

**Part 3: totalFlipReversals Consumed-and-Cleared**

Prove that day N nudges do not carry over:
1. `_applyDailyRng` reads `totalFlipReversals` at AdvanceModule line ~1524 (verify)
2. Adds nudges to rawWord: `finalWord = rawWord + nudges` (unchecked)
3. Clears: `totalFlipReversals = 0` at AdvanceModule line ~1528 (verify)
4. Day N+1 starts with totalFlipReversals = 0
5. New nudges can only accumulate after _unlockRng sets rngLockedFlag=false (reverseFlip requires rngLockedFlag==false -- verify at AdvanceModule line ~1420)
6. Any nudges accumulated for day N+1 belong to day N+1 and will be applied to day N+1's VRF word

**Part 4: Key-Based Isolation Mechanisms**

Document 4 key-based isolation mechanisms that prevent cross-day data leakage:

**4a. Coinflip epoch keying:**
- `processCoinflipPayouts` receives `epoch` parameter (= day index) from AdvanceModule line ~796 (verify)
- `coinflipDayResult[epoch]` stores result keyed by day
- `coinflipBalance[epoch][player]` stores bets keyed by day
- Day N's result written to `coinflipDayResult[dayN]` cannot affect day N+1's resolution which reads `coinflipDayResult[dayN+1]`
- Cite exact lines in BurnieCoinflip.sol

**4b. Lootbox index keying:**
- `lootboxRngIndex` incremented at `_finalizeRngRequest` (verify exact line in AdvanceModule)
- Day N's VRF word stored at `lootboxRngWordByIndex[indexN]`
- New purchases after day N's RNG request target index N+1
- `_finalizeLootboxRng` stores word at the current index (verify line)
- Day N's lootbox resolution uses `lootboxRngWordByIndex[indexN]`, day N+1 uses a different index

**4c. Redemption period isolation:**
- `resolveRedemptionPeriod` stores results keyed by `redemptionPeriodIndex` (verify in StakedDegenerusStonk)
- Period increments at resolution boundaries
- Day N resolution writes to `redemptionPeriods[periodN]`, cannot overwrite day N+1's period

**4d. dailyIdx gating:**
- `advanceGame` checks `if (day == dailyIdx)` to distinguish same-day (mid-day ticket drain) from new-day (full RNG cycle)
- `dailyIdx` only advances at `_unlockRng` after ALL processing complete
- This prevents any re-entry or double-processing of the same day's RNG

**Part 5: Gap Day Handling**

Prove that VRF stalls (gap days) do not introduce contamination:
1. `_backfillGapDays(currentWord, idx+1, day, bonusFlip)` at AdvanceModule line ~1462 (verify)
2. For each gap day: `keccak256(abi.encodePacked(vrfWord, gapDay))` produces a deterministic derived word (verify the exact line and encoding)
3. Each gap day's derived word is stored to `rngWordByDay[gapDay]` (verify)
4. The derivation uses the FRESH VRF word (unknown at request time) and the gap day index (deterministic from game state)
5. No player-controllable input enters the derivation -- the VRF word is externally sourced, the gap day index is determined by elapsed time
6. `_backfillOrphanedLootboxIndices(currentWord)` handles lootbox indices that were incremented but never got a VRF word due to the stall (verify function behavior)
7. Conclusion: gap days get unique derived words that cannot be predicted or manipulated, and do not contaminate the fresh day's processing
