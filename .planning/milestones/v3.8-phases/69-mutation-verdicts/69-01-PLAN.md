1. **depositCoinflip / coinflipBalance analysis (Research Pitfall 3):** depositCoinflip() has NO rngLockedFlag guard. coinflipBalance appears in Cat 1 backward trace. However, `_targetFlipDay()` returns `currentDayView() + 1`, so deposits target TOMORROW's coinflip, not today's. The current day's balance was committed in prior transactions. Verify this by reading BurnieCoinflip.sol lines 1060-1062. Also verify: can `currentDayView()` change between requestLootboxRng and its fulfillment? Read AdvanceModule.sol to confirm that mid-day VRF is only requested AFTER daily RNG processing is complete (requestLootboxRng checks `rngLockedFlag == false` at line 675, meaning daily processing finished). If day cannot change during mid-day window, coinflipBalance is SAFE by temporal separation.

2. **bountyOwedTo analysis (Research Open Q2):** depositCoinflip can set bountyOwedTo if deposit breaks biggest-flip record. BUT the `recordFlip` function within `_depositCoinflip` has the guard `if (!game.rngLocked())` at BurnieCoinflip:645, meaning it only runs when rngLockedFlag is FALSE. During daily window (rngLockedFlag TRUE), the guard blocks it. During mid-day window (rngLockedFlag FALSE), bountyOwedTo CAN be set. However, bountyOwedTo is consumed by processCoinflipPayouts (Cat 1) to determine who gets the bounty. The bounty recipient is NOT determined by the VRF word -- it is determined by who made the biggest flip. The VRF word determines win/loss, not bounty ownership. Verdict should be SAFE because bountyOwedTo does not influence VRF-DEPENDENT outcome computation (it is a side-effect, not an entropy input).

3. **prizePoolsPacked mid-day analysis (Research Pitfall 5):** purchase() always writes to the future pool share of prizePoolsPacked (10% via PURCHASE_TO_FUTURE_BPS), even during both commitment windows. This is permissionless and unguarded. Check whether future pool value is consumed by any VRF-dependent computation during the current commitment window. It feeds into _applyTimeBasedFutureTake and _consolidatePrizePools, but both run DURING advanceGame (after VRF fulfillment), not during a pending VRF request. The mid-day VRF fulfillment (rawFulfillRandomWords when !rngLockedFlag) ONLY writes lootboxRngWordByIndex -- it does NOT read prizePoolsPacked. Verdict should be SAFE because the future pool mutation cannot influence the VRF word itself.

4. **claimableWinnings/claimablePool analysis (Research Open Q3):** These are modified by claimWinnings(), purchase(), placeFullTicketBets(). Check all 7 backward-trace categories. claimableWinnings appears as a WRITE destination in forward trace (payDailyJackpot credits winners) and is not an input to outcome computation. The Degenerette path caps payouts at 10% of currentPrizePool, not claimablePool. Verdict should be SAFE.

5. **currentPrizePool mid-day analysis:** During mid-day window, prizePoolFrozen is FALSE. purchase() can write directly to currentPrizePool. currentPrizePool appears in Cat 2 (ETH Jackpot), Cat 6 (Prize Pool Consolidation), and Cat 7 (Degenerette payout cap). However, during mid-day VRF fulfillment, rawFulfillRandomWords does NOT read currentPrizePool -- it only writes lootboxRngWordByIndex. currentPrizePool is consumed by payDailyJackpot (which runs during advanceGame, not during mid-day fulfillment) and by _resolveFullTicketBet (which runs when a player opens a lootbox, after VRF fulfillment is complete). The currentPrizePool mutation during mid-day window does not influence the VRF word being stored. The later consumption at resolution time is a separate question for Phase 70/71. Verdict should be SAFE for the commitment window (the mid-day VRF word itself is not influenced by currentPrizePool).

6. **advanceGame-internal writes (Research Pitfall 4):** Variables like rngWordByDay, traitBurnTicket, jackpotPhaseFlag, level, price, etc. are ONLY written by advanceGame's own VRF processing pipeline. These are the processing itself, not external mutations. A player calling advanceGame between request and fulfillment triggers a DIFFERENT code path (no VRF word available yet, so rngGate reverts). These are SAFE by game-internal-only classification.
