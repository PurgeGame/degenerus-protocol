For EACH of the 7 attack sequences below, write a structured analysis with these exact fields:
- **Attack Name**: descriptive name
- **Attacker Goal**: what the attacker is trying to achieve
- **Preconditions**: what must be true for the attack to be attempted
- **Attack Steps**: numbered sequence of transactions with specific function calls
- **Target State**: which storage variable(s) the attack tries to manipulate
- **Defense Mechanism**: which protection stops the attack (with line reference)
- **Verdict**: SAFE or VULNERABLE
- **Feasibility**: Not feasible / Feasible but harmless / Exploitable
- **C4A Severity**: Critical / High / Medium / Low / Informational / N/A

Model these 7 attacks (derive all line references from contract source, not from research):

**Attack 1: Deposit-During-Lock**
- Sequence: Attacker sees VRF request tx in mempool -> deposits BURNIE via depositCoinflip() -> VRF fulfilled -> advanceGame resolves coinflips
- Goal: Place bet on a coinflip whose outcome the attacker knows (or can predict)
- Target: coinflipBalance
- Expected defense: _targetFlipDay() = currentDayView() + 1 means deposit targets TOMORROW, resolution processes TODAY
- Verify: read _targetFlipDay() implementation and confirm day+1 offset
- Also verify: during gap-day backfill, can _targetFlipDay() return a gap day? (No: _targetFlipDay uses real-time currentDayView which is ahead of historical gap days)

**Attack 2: Auto-Rebuy Toggle Extraction**
- Sequence: Attacker has auto-rebuy ON with accumulated carry -> sees VRF word on-chain (rawFulfillRandomWords emits event or attacker reads rngWordCurrent) -> toggles auto-rebuy OFF via setCoinflipAutoRebuy(false) to extract carry value -> avoids future loss
- Goal: Extract autoRebuyCarry value after knowing whether today's flip is a win or loss
- Target: playerState.autoRebuyCarry
- Expected defense: setCoinflipAutoRebuy guarded by rngLockedFlag (line 706)
- Verify: rngLockedFlag is true between VRF fulfillment and advanceGame completion
- Also check: both setCoinflipAutoRebuy and setCoinflipAutoRebuyTakeProfit are guarded

**Attack 3: Bounty-Arm During Mid-Day Window**
- Sequence: During mid-day (rngLocked=false), attacker makes record-breaking flip via depositCoinflip() -> sets bountyOwedTo to attacker address -> next daily resolution pays bounty to attacker on win
- Goal: Become bounty recipient to capture bounty payout
- Target: bountyOwedTo, biggestFlipEver (read from coinflipTopByDay)
- Expected defense: bountyOwedTo is outcome-INDEPENDENT (determines WHO gets bounty, not WHETHER coinflip wins)
- Verify: bountyOwedTo does not appear in processCoinflipPayouts outcome computation (lines 784-816), only in bounty payout distribution
- Note: the bounty record path in depositCoinflip checks !game.rngLocked() at line 645 for the biggestFlipEver update; confirm this

**Attack 4: BAF Credit Front-Running**
- Sequence: Attacker deposits large amount at a BAF-relevant level (every 10th level) -> triggers _claimCoinflipsInternal auto-claim -> records BAF credit via jackpots.recordBafFlip() -> BAF jackpot resolves with known VRF word
- Goal: Front-run BAF jackpot draw with inflated BAF credit
- Target: BAF leaderboard (JackpotModule via recordBafFlip)
- Expected defense: _coinflipLockedDuringTransition() blocks deposits at BAF levels when rngLocked
- Verify: read _coinflipLockedDuringTransition implementation (BurnieCoinflip:1000-1013), confirm it checks level % 10 == 0 AND lastPurchaseDay AND rngLocked
- Also verify: the sDGNRS auto-claim in processCoinflipPayouts at line 861 uses ContractAddresses.SDGNRS which triggers the BAF exclusion at line 556

**Attack 5: Claim-Timing Manipulation**
- Sequence: Player has multiple unresolved days -> sees VRF word -> selectively claims specific days to optimize (skip losses, claim wins)
- Goal: Cherry-pick winning days and avoid claiming losing days
- Target: playerState.lastClaim cursor
- Expected defense: _claimCoinflipsInternal iterates sequentially from lastClaim+1 to flipsClaimableDay -- no selective skip
- Verify: read the claim loop in _claimCoinflipsInternal and confirm it processes ALL days in sequence with no skip mechanism
- Also verify: there is no function to claim a SPECIFIC day index (only sequential)

**Attack 6: Cross-Contract Boon Consumption**
- Sequence: Player deposits coinflip during daily VRF window -> deposit triggers boon consumption (consumeCoinflipBoon) -> boon doubles the stake -> larger stake on tomorrow's flip
- Goal: Use boon-boosted stake to maximize winnings on a day whose VRF word is known
- Target: boonPacked (DegenerusGameStorage slot 107) and coinflipBalance
- Expected defense: boon affects TOMORROW's stake (day+1 keying), the VRF word resolving TODAY is for a different day
- Verify: the boon consumption path is inside _addDailyFlip which writes to coinflipBalance[_targetFlipDay()] = day+1, not today
- Note: even if attacker knows tomorrow's VRF word (impossible -- not yet requested), the boon consumption itself is fine because it increases future stake, not current-day stake

**Attack 7: Game-Over Predictable Fallback**
- Sequence: Game over triggered -> VRF stalls for 3+ days -> fallback word derived from rngWordByDay[1] (on-chain, readable) -> attacker deposits for the game-over day -> resolution uses predictable word
- Goal: Bet on a coinflip with a known outcome using the predictable fallback word
- Target: coinflipBalance for the game-over resolution day
- Expected defense: requires verifying whether deposits are blocked during game-over
- This is the key analysis: trace depositCoinflip during game-over state, check for gameOver guards. Use the resolution from Plan 01 Section 2.4 if it found a guard. If NOT guarded:
  - Assess: can a player deposit AFTER game-over is set but before _gameOverEntropy runs?
  - If yes: the fallback word from rngWordByDay[1] is known, so the player knows win/loss in advance
  - If yes but deposits target day+1: does the fallback resolve the day+1 or the game-over day?
  - Provide C4A severity assessment based on findings

For each attack, cite specific line numbers from the contract source to support the verdict. Do NOT copy line numbers from the research -- re-derive from source.
