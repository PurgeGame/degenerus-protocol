**Section 3: Mutation Surface Catalog (CW-03)**

For EVERY variable that appears in the Forward-Trace Catalog (Section 1) or Backward-Trace Catalog (Section 2), systematically identify every external/public function that can write to it. This includes both direct writes and indirect writes through internal function calls.

**Methodology:**

For each cataloged variable:
1. Search ALL contracts (not just the module that reads it) for assignments to that variable. Remember: all delegatecall modules share DegenerusGameStorage, so a variable written by MintModule is the same storage slot as one read by JackpotModule.
2. For each write found, trace backward to the nearest external/public function entry point.
3. Record the call-graph depth:
   - **D0 (direct):** External function directly writes the variable (e.g., `function foo() external { myVar = x; }`)
   - **D1 (one hop):** External -> internal that writes (e.g., `function foo() external { _bar(); }` where `_bar` writes)
   - **D2 (two hops):** External -> internal -> internal that writes
   - **D3+ (deep):** Three or more hops
4. Record the access control: is the external function callable by anyone (permissionless), only by specific roles (admin/owner), or only by the game contract?

**Output table per variable:**

```
| Variable | External Function | Module/Contract | Call Path | Depth | Access Control | Notes |
```

**Special attention items:**

1. **Ticket queue double-buffer:** Document the `_swapAndFreeze` commitment boundary explicitly. The `ticketWriteSlot` controls which buffer receives new purchases. When `_swapAndFreeze` is called at RNG request time, it swaps the write slot. New purchases after the swap go to the NEW write slot, but processing reads the OLD slot. Document: which functions call `_swapAndFreeze`, when in the lifecycle, and what happens if `purchase()` is called between swap and fulfillment.

2. **`reverseFlip()` during commitment window:** This function increments `totalFlipReversals`, which directly affects coinflip outcome via nudge computation. If callable between VRF request and fulfillment, this is a commitment window violation. Document its access control and timing constraints.

3. **`requestLootboxRng()` side effects:** This function may swap the ticket buffer (`_swapAndFreeze`) and write `midDayTicketRngPending`. Document exactly what state changes it makes beyond the VRF request.

4. **`purchase()` / `purchaseCoin()` / `purchaseLazyPass()` during commitment window:** These write to `ticketQueue`, `ticketsOwedPacked`, prize pools. Document whether the double-buffer protects against all of these.
