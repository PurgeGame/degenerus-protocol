**Test 1 -- testEdge01NoDoubleCount_FFThenWriteKey (EDGE-01):**
Scenario: player deposits to FF key at low level, same player deposits to write key at higher level, processing drains both independently.
- buyer = address(0xBEEF)
- harness.setLevel(5)
- harness.queueTickets(buyer, 15, 3) -- level=5, 15>11, goes to _tqFarFutureKey(15)

- Assert: getTicketsOwedPacked(tqFarFutureKey(15), buyer) has owed=3 (packed >> 8 == 3)
- harness.setLevel(10) -- now 15 <= 10+6=16, near-future
- harness.queueTickets(buyer, 15, 5) -- goes to _tqWriteKey(15)
- Assert: getQueueLength(tqFarFutureKey(15)) == 1 (unchanged)
- Assert: getQueueLength(tqWriteKey(15)) == 1 (new entry)
- Assert: getTicketsOwedPacked(tqWriteKey(15), buyer) has owed=5 (packed >> 8 == 5)
- Assert: getTicketsOwedPacked(tqFarFutureKey(15), buyer) has owed=3 (unchanged)
Key assertion: the two key spaces are independent; depositing to write key did NOT modify FF key's owed count. Each deposit is tracked separately under its own key.

- Assert: getQueueLength(tqFarFutureKey(15)) == 0 (deleted)

**Test 3 -- testEdge02RoutingPreventsNewFFDeposits (EDGE-02):**

Key assertion: once level >= L-6, the isFarFuture condition (targetLevel > level + 6) is permanently false for level L. New deposits can never reach the FF key for already-near-future levels.

## Summary

## EDGE-01: No Double-Counting Between FF Key and Write Buffer

### Claim
A ticket deposited into the FF key for level L is correctly handled when level L later
enters the +2 to +6 near-future window. No double-counting occurs between FF key and
write buffer.

### Proof

**Structural Fact 1: Key spaces are disjoint.**
[Reference DegenerusGameStorage.sol lines 716-731. Three key spaces:
Slot0 = lvl (bits 22-23 clear), FF = lvl | 0x400000 (bit 22), Slot1 = lvl | 0x800000 (bit 23).
Phase 74 fuzz tests prove non-collision for all valid levels.]

**Structural Fact 2: Routing is deterministic at deposit time.**
[Reference DegenerusGameStorage.sol line 544: `bool isFarFuture = targetLevel > level + 6`.
At deposit time, the ticket goes to exactly ONE key space. The check uses the CURRENT
level at time of deposit, not the level at time of processing.]

**Structural Fact 3: ticketsOwedPacked is keyed on wk (the encoded key), not raw level.**
[Reference DegenerusGameStorage.sol line 547: `uint40 packed = ticketsOwedPacked[wk][buyer]`.
A player's owed count at ticketsOwedPacked[_tqFarFutureKey(L)][player] is completely
independent of ticketsOwedPacked[_tqWriteKey(L)][player].]

**Structural Fact 4: processFutureTicketBatch drains queues sequentially, not merged.**
[Reference DegenerusGameMintModule.sol lines 302-303:
`bool inFarFuture = (ticketLevel == (lvl | TICKET_FAR_FUTURE_BIT));`
`uint24 rk = inFarFuture ? _tqFarFutureKey(lvl) : _tqReadKey(lvl);`
Each queue is processed against its own ticketsOwedPacked[rk] entries. No cross-contamination.]

### Scenario Trace
[Include the full scenario trace from RESEARCH.md: player deposits at T1 (level=5, target=15,
goes to FF key), deposits again at T2 (level=10, target=15, goes to write key), processing at

correct behavior, not double-counting.]

### Verdict: SAFE

## EDGE-02: No Re-Processing After FF Key Drain

### Claim
Once processFutureTicketBatch has fully drained the FF key for a level, subsequent
deposits to that same level's FF key start a fresh queue that does not re-process
already-minted entries.

### Proof

**Structural Fact 1: Monotonic level progression prevents new FF deposits to drained levels.**
[Reference DegenerusGameAdvanceModule.sol line 1340: `level = lvl` where lvl = purchaseLevel
= level + 1. Level only increases. Once level >= L-6 (required for L to be in processing
window), the routing check `targetLevel > level + 6` is permanently false for level L.
New deposits go to _tqWriteKey(L), not _tqFarFutureKey(L).]

**Structural Fact 2: `delete ticketQueue[rk]` creates a clean slate.**
[Reference DegenerusGameMintModule.sol line 438. Solidity `delete` on dynamic array sets
length to 0 and clears all elements. Any subsequent push starts at index 0 fresh.]

**Structural Fact 3: ticketsOwedPacked is zeroed per-player during processing.**
[Reference DegenerusGameMintModule.sol lines 351-431. Every player processed until
remainingOwed == 0. At completion: ticketsOwedPacked[rk][player] = 0 (newPacked = 0
when remainingOwed == 0 && rem == 0, written at line 420). Queue deletion only occurs
when idx >= total, meaning all players fully processed.]

**Structural Fact 4: Even hypothetically, new deposits start fresh.**
[In _queueTickets (DegenerusGameStorage.sol line 550-551):
`if (owed == 0 && rem == 0) { ticketQueue[wk].push(buyer); }`
After processing zeros ticketsOwedPacked and delete clears queue, a hypothetical new
deposit would find packed == 0, push a fresh entry, and set new owed count. No old entries
exist to re-process.]

### Monotonic Level Argument
[Formal argument: FF key for level L drained requires currentLevel >= L-6.
Since level only increases, for any future deposit to level L:
isFarFuture = (L > level + 6) = (L > (>=L-6) + 6) = (L > >=L) = false.
Therefore new deposits to level L always go to _tqWriteKey(L), never _tqFarFutureKey(L).
The drained FF queue can never receive new entries.]

### Verdict: SAFE
