## Methodology

Four EBD-03 attack vectors from `231-CONTEXT.md` D-08 applied end-to-end across the combined state machine:

1. **(a) No double-spend across the (purchase finalize) ‚Üí (jackpot-phase run) handoff.** The pool balance `_finalizeEarlybird` dumps into Lootbox must not be consumed twice, and no post-finalize purchase-side `_awardEarlybirdDgnrs` call may allocate out of the now-dumped pool.
2. **(b) No orphaned reserves when `_finalizeEarlybird` fires at `EARLYBIRD_END_LEVEL`.** The dump must transfer the FULL remaining pool balance (not a partial fraction), and the sentinel must gate every downstream purchase-side call so no late allocation attempts leave residual DGNRS trapped in the Earlybird pool.
3. **(c) No missed emissions across normal / skip-split / gameover transitions in `DegenerusGameAdvanceModule.advanceGame`.** Every reachable path must either fire `_finalizeEarlybird` exactly once at the correct level or correctly skip (game-over before `EARLYBIRD_END_LEVEL` is designed to leave the pool intact; Phase 236 REG-01 owns whether the unallocated pool in that path is novel or pre-existing behavior).
4. **(d) Cross-commit invariant: pool dumped by `_finalizeEarlybird` equals pool consumed by `_runEarlyBirdLootboxJackpot`.** The Lootbox-pool-side balance credited at level-transition must land cleanly in the jackpot-phase `futurePrizePool ‚Üí totalBudget` arithmetic feeding `_runEarlyBirdLootboxJackpot` without drift. Algebraic sum-before = sum-after across every pool SSTORE is DEFERRED to Phase 235 CONS-01 per D-07.

Path enumeration covers every reachable transition path through `DegenerusGameAdvanceModule.advanceGame` at HEAD: normal level progression, skip-split / phase-transition (from 2471f8e7 packing), game-over before `EARLYBIRD_END_LEVEL`, and game-over at or after `EARLYBIRD_END_LEVEL`. Verdict vocabulary is locked to `PASS | FAIL | DEFER`. Column set locked per D-02 adapted for state-machine tracing: `Path | Attack Vector Considered | Verdict (PASS/FAIL/DEFER) | Evidence | Owning SHA`.

231-01 and 231-02 verdicts are treated strictly as regression anchors per D-03. This audit re-verifies the cross-commit hand-off independently by reading HEAD source at each hop; no prior-phase verdict is carried forward as pre-approval.

### Path C: Game-Over Before `EARLYBIRD_END_LEVEL`

**Entry:** `advanceGame` at a level `< EARLYBIRD_END_LEVEL` (i.e. `level ‚àà {0, 1, 2}`) where the liveness guard triggers (`currentDay - psd > DEPLOY_IDLE_TIMEOUT_DAYS` if lvl=0, else `> 120`) OR the gameover condition is met. 6. `_finalizeEarlybird` therefore NEVER fires on game-over-before-EBD-END paths. The Earlybird DGNRS pool remains in the Earlybird slot of the StakedDegenerusStonk contract ‚Äî unallocated.

**Is the undumped Earlybird pool an orphaned reserve?** Per `231-CONTEXT.md` Specific Ideas bullet 3 and D-08 EBD-03 (c), this behavior is documented as possibly pre-existing (the OLD pre-f20a2b5e code finalized the pool INSIDE `_awardEarlybirdDgnrs` itself, which also would not fire at game-over-before-EBD-END because there would be no fresh purchase to trigger it in a dead game). The f20a2b5e refactor preserves this behavior ‚Äî it does NOT widen the orphaned-pool surface. The DGNRS in the Earlybird pool can still be consumed by any subsequent purchase-side `_awardEarlybirdDgnrs` call (if the game is still alive and has pending purchases up to the EARLYBIRD_TARGET_ETH cap). In a true dead-game terminal state, the DGNRS stays trapped ‚Äî but this is a PRE-EXISTING design property, not a delta-introduced regression. **Hand-off:** Phase 236 REG-01 regression sweep should cross-check this against v25.0/v26.0/v27.0 to confirm no prior finding covered this surface.

**Verdict:** PASS on all four attack vectors (see Per-Path Verdict Block rows for Path C below). (a) no double-spend ‚Äî hook never fires, no pool movement. (b) no orphaned reserves introduced by the delta ‚Äî f20a2b5e preserves the pre-existing "dead-game terminal trapped DGNRS" design property (the OLD code would also not have finalized on this path); whether the pre-existing trapped-DGNRS surface is a novel concern is deferred as a regression cross-check to Phase 236 REG-01 via the Findings-Candidate hand-off, not as a row-level verdict DEFER. (c) no missed emission ‚Äî correctly skipped. (d) cross-commit invariant ‚Äî not exercised (no dump, no consume).

**Verdict:** PASS on all four attack vectors. The sentinel flipped in Path A makes it mathematically impossible to dump the pool twice, and the game-over hop does not re-enter the hook anyway. No orphaned reserves beyond what Path A already handled (the pool was dumped to Lootbox in Path A; downstream consumption by `_runEarlyBirdLootboxJackpot` may or may not have happened depending on whether the game reached the first jackpot-phase day ‚Äî but in either case, the Lootbox pool retains the DGNRS and the SDGNRS contract has its own burn-remaining-pools terminal cleanup hook).

## High-Risk Patterns Analyzed

### Cross-Commit Invariant (`_finalizeEarlybird` dump vs `_runEarlyBirdLootboxJackpot` consumption)

The most load-bearing invariant of EBD-03 is clarified by the HEAD source reading:

- **`_finalizeEarlybird`** (contracts/modules/DegenerusGameAdvanceModule.sol:1582-1595) moves DGNRS tokens from `IStakedDegenerusStonk.Pool.Earlybird` to `IStakedDegenerusStonk.Pool.Lootbox`. This is a DGNRS-side accounting transfer inside the StakedDegenerusStonk external contract, affecting the `poolBalances[Earlybird]` and `poolBalances[Lootbox]` slots of that external contract's storage.
- **`_runEarlyBirdLootboxJackpot`** (contracts/modules/DegenerusGameJackpotModule.sol:664-712) reads and mutates `futurePrizePool` (ETH accounting, line 665 `_getFuturePrizePool()`, line 668 `_setFuturePrizePool(...)`) and `nextPrizePool` (line 711 `_setNextPrizePool(...)`). These are ETH-denominated slot-based accumulators inside `DegenerusGameStorage`. The function does NOT read from `IStakedDegenerusStonk.Pool.Lootbox` directly.

**Consequence:** The "cross-commit invariant" is NOT a direct balance flow from the DGNRS Earlybird pool into the ETH futurePrizePool arithmetic. The two sides operate on orthogonal storage namespaces. The invariant EBD-03 must actually verify is TEMPORAL + CAUSAL:

1. **Temporal:** `_finalizeEarlybird` must fire BEFORE the first `_runEarlyBirdLootboxJackpot` call of the game. Verified: `_finalizeEarlybird` runs at the `lvl == EARLYBIRD_END_LEVEL = 3` RNG-request-time transition (Path A, hop 4). `_runEarlyBirdLootboxJackpot` runs on the FIRST jackpot-phase day (`isEarlyBirdDay = (jackpotCounter == 0)` at line 362 of JackpotModule) which, by state-machine topology, can only occur AFTER the level has transitioned to 3 AND the purchase phase has handed off to jackpot phase. The level-transition in `_finalizeRngRequest:1514` sets `level = 3` in the SAME tx as `_finalizeEarlybird`, and the `jackpotPhaseFlag = true` SSTORE at line 418 happens in a LATER tx (after rngGate returns a non-1 word, after `_consolidatePoolsAndRewardJackpots` runs at line 402). So `_runEarlyBirdLootboxJackpot` cannot run in the same tx as the finalize ‚Äî there is always at least one tx of separation.
2. **Causal:** The DGNRS moved from Earlybird to Lootbox ends up usable by downstream Lootbox consumers (BURNIE lootbox opening, decimator payouts, etc.). Whether `_runEarlyBirdLootboxJackpot` specifically consumes the dumped DGNRS is NOT the EBD-03 invariant ‚Äî the function operates on ETH-pool arithmetic, not DGNRS-pool arithmetic. The two systems are connected only at the SDGNRS contract's level-transition hooks and the game's external-call boundary.

**What could go wrong (and doesn't):**

- If `_finalizeEarlybird` failed to fire (missed emission) ‚Üí DGNRS trapped in Earlybird pool forever. Verified not possible: Path A always fires the hook at `lvl == 3`, sentinel is ONE-SHOT-AFTER-FIRST-FIRE semantics enforced by line 1583 + line 1584.
- If `_finalizeEarlybird` fired twice ‚Üí double transfer. Verified not possible: sentinel flips from `(0 or real-balance) ‚Üí type(uint256).max` at line 1584; every subsequent call short-circuits at line 1583.
- If a purchase-side `_awardEarlybirdDgnrs` call ran AFTER the dump and drained the Earlybird pool further ‚Üí draining a no-longer-existent pool. Verified not possible: `dgnrs.poolBalance(Earlybird)` after the dump is 0, and the external `transferFromPool` target caps to `available` defensively (external DGNRS contract ‚Äî out of EBD-03 scope for deep audit; 231-01's Methodology paragraph verified the external callee has no reentrancy/callback surface). Additionally, the sentinel check at contracts/storage/DegenerusGameStorage.sol:1011 no-ops `_awardEarlybirdDgnrs` entirely post-flip ‚Äî even attempting the external call is avoided.

**Algebraic closure hand-off:** The full sum-before = sum-after proof across ALL pool SSTOREs from the moment `_finalizeEarlybird` fires through the end of the first jackpot-phase day (covering DGNRS pools + ETH pools + the `_consolidatePoolsAndRewardJackpots` transition) is deferred to Phase 235 CONS-01 per D-07. ### Sentinel One-Shot Semantics

`earlybirdDgnrsPoolStart` at contracts/storage/DegenerusGameStorage.sol:978 is the single cross-path guard preventing BOTH:

1. **Double-dump from `_finalizeEarlybird`** ‚Äî line 1583 early-return if `poolStart == type(uint256).max`; line 1584 flips. CEI-compliant: flip BEFORE the external call, so any reentrant attempt from the DGNRS contract (which doesn't reenter ‚Äî the external `transferBetweenPools` / `transferFromPool` callees are pure state writes followed by events, no callbacks, no ERC777 hooks ‚Äî confirmed in 231-01 Methodology paragraph regression anchor) would hit the guard.
2. **Post-finalize purchase-side double-allocation from `_awardEarlybirdDgnrs`** ‚Äî line 1011 of DegenerusGameStorage.sol also tests `poolStart == type(uint256).max` and returns early. Any in-flight purchase that lands AFTER the `_finalizeEarlybird` SSTORE sees the sentinel and no-ops.

The sentinel's dual role is load-bearing for EBD-03: without it, a buyer executing a purchase in the same block as (but in an earlier tx than) the advanceGame call that triggers `_finalizeEarlybird` could still be awarded DGNRS from a pool that's about to be dumped ‚Äî an accounting inconsistency. With the sentinel flipped as the FIRST state mutation inside `_finalizeEarlybird` (line 1584, before any external call), this window is tx-atomic: either `_awardEarlybirdDgnrs` ran BEFORE the finalize (and the buyer got their DGNRS, reducing the pool balance that the finalize will dump), OR `_awardEarlybirdDgnrs` ran AFTER the finalize (and the buyer got zero DGNRS per the sentinel guard ‚Äî which is correct, since the pool has been moved). There is no race condition; transaction serialization + the sentinel together form a total ordering.
