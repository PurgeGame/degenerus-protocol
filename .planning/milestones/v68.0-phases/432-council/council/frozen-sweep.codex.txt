1. Mid-day lootbox RNG latch release: no regression found.

The callback writes the reserved word to `LR_INDEX - 1` and clears only request metadata, not the latch: `contracts/modules/DegenerusGameAdvanceModule.sol:1848`.

Both drain paths require the word before clearing the latch. Same-day drain reverts if the word is missing and clears only after tickets finish: `contracts/modules/DegenerusGameAdvanceModule.sol:224`, `contracts/modules/DegenerusGameAdvanceModule.sol:238`. New-day drain similarly requires either the mid-day word or a daily `rngWordCurrent`, then clears only after the read queue is fully processed: `contracts/modules/DegenerusGameAdvanceModule.sol:273`, `contracts/modules/DegenerusGameAdvanceModule.sol:299`.

Strongest hypothesis tested: day rollover while mid-day VRF is in-flight. The source defeats it because the new-day path hits `RngNotReady` before the clear when no word exists: `contracts/modules/DegenerusGameAdvanceModule.sol:280`. If a daily request supersedes the mid-day request, the old callback is ignored by request id, and the daily word is bound once to `LR_INDEX - 1`; duplicate finalization is blocked by `_finalizeLootboxRng`: `contracts/modules/DegenerusGameAdvanceModule.sol:1287`.

2. Direct-call guards on payable delegatecall-only helpers: no regression found.

The four named helpers all guard `address(this) == GAME`: `contracts/modules/DegenerusGameBoonModule.sol:68`, `contracts/modules/DegenerusGameBoonModule.sol:128`, `contracts/modules/DegenerusGameBoonModule.sol:294`, `contracts/modules/DegenerusGameLootboxModule.sol:874`. Under every legitimate nested delegatecall path, `address(this)` remains the Game address, so those paths are admitted. A direct external call to the module address fails the guard.

I also checked analogous payable nested helpers in the degenerette box-resolution path; they already have the same guard: `contracts/modules/DegenerusGameDegeneretteModule.sol:1292`, `contracts/modules/DegenerusGameDegeneretteModule.sol:1347`, `contracts/modules/DegenerusGameDegeneretteModule.sol:1402`.

Strongest hypothesis tested: a legitimate module-to-module delegatecall where `address(this)` becomes the module address. Solidity `delegatecall` preserves the original storage/address context, and the source uses delegatecall through the Game stubs, so I found no legitimate path where `address(this) != GAME`.
