The VRF Coordinator Rotation (VRFSWAP) logic and its interaction with the protocol's "spinal column" (RNG fulfillment and liveness) were reviewed against the provided claims and hotspots.

### VRFSWAP-01 — Rotation holds every freeze-relevant variable consistent
**Verdict: REFUTED**
**Severity: LOW (INFO)**
Reasoning: `updateVrfCoordinatorAndSub` (`DegenerusGameAdvanceModule.sol:1761`) correctly identifies in-flight requests using `LR_MID_DAY` and `rngLockedFlag`.
- **Atomicity:** Since the module is called via `delegatecall` from the HUB, storage updates to the coordinator configuration (Slots 31/32/33) and the re-issuance of `vrfRequestId` (Slot 4) are atomic within the same transaction.
- **Nudge Preservation:** `totalFlipReversals` (Slot 5) is intentionally preserved to ensure players do not lose the value of burned FLIP during a stall.
- **Liveness Reset:** Re-requesting a word resets `rngRequestTime` to `block.timestamp`. While this technically delays the 12h/14d recovery timers, it is the correct behavior for an honest admin to allow the new coordinator a full window to fulfill. For the `_gameOverEntropy` fallback path (where `rngLockedFlag` is false), the timer is **not** reset, ensuring rotation cannot be used to maliciously delay terminal liquidation.

### VRFSWAP-02 — Mid-day / mid-request / stalled / while-locked rotation cannot brick or corrupt the binding
**Verdict: REFUTED**
**Severity: LOW (INFO)**
Reasoning: The system is robust against rotations at any timing.
- **Mid-day:** If a rotation occurs while a mid-day request is in-flight, it re-requests using `VRF_MIDDAY_CONFIRMATIONS`. The word will land in the reserved `lootboxRngWordByIndex[LR_INDEX - 1]` slot because `LR_INDEX` was already advanced at the original request and is not touched by rotation.
- **Locked/Stalled:** If the daily RNG is locked, rotation re-requests the daily word. If the word was already delivered (`rngWordCurrent != 0`) but not processed, rotation skips re-requesting and preserves the word, allowing the next `advanceGame` to proceed.
- **Binding:** The `requestId` binding is refreshed upon rotation, and `LR_INDEX` preservation ensures that even if a mid-day request is converted to a daily fulfill (via `rngGate` timeout), the word correctly backfills the pending lootbox index via `_finalizeLootboxRng`.

### VRFSWAP-03 — rawFulfillRandomWords requestId/coordinator validation correct across a rotation
**Verdict: REFUTED**
**Severity: LOW (INFO)**
Reasoning: `rawFulfillRandomWords` (`DegenerusGameAdvanceModule.sol:1834`) implements two strict gates:
1. `if (msg.sender != address(vrfCoordinator)) revert E();` — This uses the current state variable. After rotation, only the **new** coordinator can fulfill.
2. `if (requestId != vrfRequestId || rngWordCurrent != 0) return;` — Overwriting `vrfRequestId` during rotation ensures that stale callbacks from the old coordinator (even if they bypassed the `msg.sender` check) are ignored. The `rngWordCurrent != 0` check prevents double-fulfillment if a word was delivered just as the rotation tx was being mined.

### Hotspot Analysis

#### Slot 5 co-residence (`totalFlipReversals | lastVrfProcessedTimestamp`)
**Verdict: SAFE**
The packing of `uint64 totalFlipReversals` and `uint48 lastVrfProcessedTimestamp` is handled correctly by the Solidity 0.8.34 compiler. `reverseFlip` (HUB:1826) and `_applyDailyRng` (Module:1927, 1931) perform masked writes that preserve the co-resident field. Since `totalFlipReversals` is bounded by the compounding nudge cost (exceeds FLIP supply within ~100 nudges), it will never overflow its 64-bit allocation to corrupt the timestamp.

#### `_gameOverEntropy` (~:1337)Fallback
**Verdict: SAFE**
The fallback path correctly pre-subtracts `totalFlipReversals` (`Module:1337`) before calling `_applyDailyRng`. This ensures that even if a committer purchased nudges during a VRF stall (while `rngLockedFlag` was false), those nudges are cancelled out, making the terminal distribution dependent only on the verifiable historical word.

#### Grace-threshold / 12h-14d recovery timers
**Verdict: SAFE**
As analyzed in VRFSWAP-01, the rotation logic is "emergency-aware". It resets the 12h daily retry timer to give the new coordinator a chance but **preserves** the 14d game-over timer if the lock isn't held, preventing governance from "resetting the death clock" during a terminal stall.

### New Vectors
- **Conversion to Daily:** A stalled mid-day request where `LR_MID_DAY == 0` (no ticket swap) will not be re-requested by rotation or `retryLootboxRng`. However, it will naturally resolve via the next daily `rngGate` (which sees `rngRequestTime != 0` and eventually retries a daily request, whose word then backfills the index). This is a safe fallback.

**Final Audit State: REFUTED. The VRF Coordinator Rotation logic is sound and maintains liveness and consistency across all analyzed states.**
