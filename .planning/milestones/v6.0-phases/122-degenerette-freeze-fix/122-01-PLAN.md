**The fix (per D-01, D-02, D-03):**

Replace the current L685 `if (prizePoolFrozen) revert E();` block with a freeze-aware branching structure. When `prizePoolFrozen` is true:

1. Read pending pools: `(uint128 pNext, uint128 pFuture) = _getPendingPools();`
2. Compute ethPortion and lootboxPortion identically to the unfrozen path (25/75 split, 10% cap against pFuture instead of live pool).
3. Check solvency: `if (pFuture < ethPortion) revert E();` -- revert if pending accumulator cannot cover (per D-02).
4. Debit pending: `_setPendingPools(pNext, pFuture - uint128(ethPortion));`
5. Credit player: `_addClaimableEth(player, ethPortion);`
6. Resolve lootbox: `_resolveLootboxDirect(player, lootboxPortion, rngWord);` if nonzero.

When `prizePoolFrozen` is false: the existing code path is UNCHANGED (read _getFuturePrizePool, debit, _setFuturePrizePool, _addClaimableEth, _resolveLootboxDirect).

**Critical detail per D-05:** The frozen path reads/writes ONLY `_getPendingPools`/`_setPendingPools`. It does NOT touch `_getFuturePrizePool`/`_setFuturePrizePool`. The live futurePrizePool snapshot that advanceGame/runRewardJackpots operates on remains untouched.
