**RNG-02 section must contain these subsections:**

1. **BAF Guard Location and Conditions** -- Document the BAF guard at BurnieCoinflip.sol lines 542-563 (verify exact lines in current code). List all 7 conditions that must ALL be true for the guard to revert:
   ```
   1. winningBafCredit != 0     (player has winning BAF flips to record)
   2. player != SDGNRS          (not the sDGNRS contract address)
   3. !inJackpotPhase           (game in purchase phase, not jackpot)
   4. !over                     (game not over)
   5. lastPurchaseDay_          (purchase target met, transitioning)
   6. rngLocked_                (VRF request currently in flight)
   7. purchaseLevel_ % 10 == 0  (current level is a BAF resolution level)
   ```

2. **BAF Guard Bypass Enumeration** -- For each of the 7 conditions, document:
   - What makes it false (legitimate bypass path)
   - Whether an attacker can force it false to bypass the guard
   - Whether bypassing this condition actually creates a security issue

   Key legitimate bypasses to document:
   - Condition 1 false: No wins to record -- no BAF credit, guard not needed
   - Condition 3 false: Jackpot phase -- BAF already resolved for this round
   - Condition 6 false: RNG not locked -- no pending VRF word, nothing to front-run
   - Condition 7 false: Not a BAF level (e.g. level 13) -- no BAF resolution at this level

3. **sDGNRS BAF Ineligibility Proof** -- Prove sDGNRS cannot participate in BAF at two layers:
   - **Caller-side (BurnieCoinflip):** Line 542 `player != ContractAddresses.SDGNRS` skips entire BAF section. Find this exact line and document it.
   - **Callee-side (DegenerusJackpots):** `recordBafFlip` function returns early for sDGNRS address. Find this at DegenerusJackpots line ~175 and document the exact early-return condition.
   - **processCoinflipPayouts path:** Verify that `processCoinflipPayouts` calls `_claimCoinflipsInternal(ContractAddresses.SDGNRS, false)` during active RNG lock (called from advanceGame). The sDGNRS skip at line 542 prevents the BAF guard from reverting on this path. Document exact line of the processCoinflipPayouts call.
   - **Conclusion:** sDGNRS is truly ineligible for BAF, not just skipping the guard. No further BAF guard analysis needed for sDGNRS path.

4. **Reentrancy Analysis** -- Can an attacker use reentrancy to bypass the BAF guard?
   - Check: Does `_claimCoinflipsInternal` make external calls before the BAF guard check?
   - Check: Are the claim functions protected by reentrancy guards (nonReentrant modifier or equivalent)?
   - Check: Can state manipulation via reentrancy change any of the 7 conditions between check and effect?

5. **Timing Attack Analysis** -- Can an attacker exploit the timing window?
   - When exactly does rngLockedFlag transition from true to false? (AdvanceModule._unlockRng, called after processCoinflipPayouts)
   - Can an attacker submit a claim transaction that gets mined in the same block as advanceGame but after _unlockRng? (Answer: No, because _unlockRng happens at the end of advanceGame, after payout processing)
   - Document the exact ordering within advanceGame: rngGate -> processCoinflipPayouts -> ... -> _unlockRng

6. **Deposit Lock Layer** -- Document `_coinflipLockedDuringTransition()` (BurnieCoinflip line ~985-998) as a separate protection layer that blocks deposits during BAF resolution levels. This prevents auto-claim triggering via deposits.

7. **Attacker Model Evaluation** -- For each attacker model:
   - MEV-aware: Can they see VRF word in mempool and front-run with a claim to manipulate BAF outcomes? Evaluate whether claim timing relative to BAF resolution creates any advantage.
   - Compromised VRF: Can they delay/manipulate the VRF word delivery to create BAF bypass windows? Evaluate rngLockedFlag lifecycle against this scenario.

8. **RNG-02 Verdict** -- SAFE or UNSAFE. Expected: SAFE -- BAF guard covers the exact window where claims could affect BAF outcomes, sDGNRS is truly ineligible, no bypass via timing or reentrancy.
