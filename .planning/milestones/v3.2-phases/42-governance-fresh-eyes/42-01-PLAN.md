**GOV-02 section: Timing Attack Analysis and Post-v2.1 Change Evaluation**

Structure with these mandatory subsections:

1. **Post-v2.1 Code Change Analysis** -- For each of the 5 post-v2.1 changes, document:

   **Change 1: Death Clock Pause Removal (commit 73c50cb3)**
   - What was removed: `anyProposalActive()` function and death clock interaction
   - Security impact: Simplification -- removes an entire class of liveness manipulation attacks
   - New attack surfaces: NONE -- removing functionality only reduces attack surface
   - Verdict: SAFE (simplification)

   **Change 2: activeProposalCount -> activeProposalId (commit df1e9f78)**
   - What was removed: `uint8 activeProposalCount` state variable with increment/decrement
   - What replaced it: Per-address `activeProposalId` mapping with 1-per-address limit (verified in Task 1 as part of VOTE-03 fix confirmation)
   - Security impact: Eliminates uint8 overflow at 256 proposals (VOTE-03). Limits spam to 1 active per address.
   - New attack surfaces: Attacker uses multiple addresses to spam proposals. Each address needs DGVE majority or 0.5% sDGNRS. For DGVE path: only one address can hold >50.1% DGVE. For sDGNRS path: 0.5% per address, max 200 addresses theoretically, but sDGNRS is soulbound so attacker cannot redistribute. Practical limit: attacker's number of sDGNRS-holding addresses with >= 0.5% each.
   - Verdict: SAFE (improvement over uint8 counter)

   **Change 3: _executeSwap CEI Fix (commit df1e9f78)**
   - What changed: `_voidAllActive` now runs at line 568, BEFORE external calls starting at line 578
   - Security impact: Fixes GOV-07 (already confirmed in Task 1)
   - New attack surfaces: NONE -- CEI ordering eliminates reentrancy concerns
   - Verdict: SAFE (fix confirmed)

   **Change 4: Threshold Decay -- 60% -> 50% (commit fd9dbad1)**
   - What changed: Initial threshold from 5000 BPS (50%), previously 6000 BPS (60%). First tier now 48h (was 24h).
   - Security impact: Lower barrier reduces community coordination requirement. Reduces from 60% to 50% the sDGNRS needed to pass a proposal in the first 48 hours.
   - New attack surfaces: Slightly easier for a cartel to pass proposals early. But 50% of circulating sDGNRS is still a massive amount. During a stall, sDGNRS is frozen so this cannot be acquired.
   - Impact on WAR-02: The day-6 threshold of 5% (500 BPS) is UNCHANGED. The change only affects early tiers (0-48h: 50% instead of 60%, 48-72h: 40% instead of 50%). WAR-02's core scenario (day 6, 5% threshold) is unaffected.
   - Verify the full decay schedule from `threshold()` function:
     ```
     0-48h:   50% (5000 BPS)    // was 60%
     48-72h:  40% (4000 BPS)    // was 50%
     72-96h:  30% (3000 BPS)    // was 40%
     96-120h: 20% (2000 BPS)    // was 30%
     120-144h: 10% (1000 BPS)   // was 20%
     144-168h: 5% (500 BPS)     // unchanged
     168h+:   expired           // unchanged
     ```
   - Verdict: SAFE (WAR-02 unaffected, early tiers still require substantial majority)

   **Change 5: voidedUpTo Watermark (commit df1e9f78)**
   - What changed: `_voidAllActive` scans from `voidedUpTo + 1` instead of from proposal 1
   - Security impact: Gas optimization for WAR-06 scenario -- reduces iteration cost when many proposals exist
   - New attack surfaces: Verify no off-by-one: watermark is set to `count` after voiding (verify exact line in `_voidAllActive`). Scanning starts from `voidedUpTo + 1`. If a proposal is created between the void loop and watermark update, it could be missed. Verify atomicity: `_voidAllActive` is a simple loop + assignment with no external calls between them.
   - Verdict: SAFE (gas optimization, no new attack surface, atomicity holds)

2. **Timing Window Analysis** -- For each critical timing boundary, evaluate attack scenarios:

   **Window 1: 5h unwrapTo Block vs 20h Governance Threshold**
   - unwrapTo blocked at 5h stall (`DegenerusStonk.sol` line 153)
   - Admin governance eligible at 20h stall (`DegenerusAdmin.sol` line 424)
   - Community governance eligible at 7 days (168h) stall
   - Gap analysis: 15h between unwrapTo block and admin governance. During this gap, sDGNRS is already frozen but governance is not yet possible. This is the correct ordering -- sDGNRS must be frozen BEFORE governance becomes available.
   - Attack scenario: Creator tries to unwrap DGNRS to sDGNRS at 4h59m to gain vote weight, then votes at 20h01m. Defense: unwrapTo blocks at 5h, so the unwrap fails. Even if unwrap succeeds at 4h59m, the resulting sDGNRS balance was already possible before the stall.
   - Verdict: SAFE

   **Window 2: Proposal Lifetime (168h) vs Threshold Decay**
   - Proposal expires at exactly 168h from creation
   - Threshold reaches minimum 5% (500 BPS) at 144h
   - Final window: 144h-168h, threshold = 5%
   - Attack scenario: Cartel with 5% sDGNRS waits until 144h, votes to approve. If no opposition, proposal passes. This IS WAR-02.
   - Defense: Community must monitor proposals and vote to reject. Reject requires `reject > approve AND reject >= threshold%`. At 5% threshold, 5% circulating sDGNRS is enough to kill if reject > approve.
   - Verdict: KNOWN RISK (WAR-02)

   **Window 3: Auto-Invalidation Race**
   - VRF recovers at exactly 20h boundary
   - vote() at block with timestamp showing stall < 20h -> reverts NotStalled
   - vote() at block with timestamp showing stall >= 20h -> succeeds
   - Block timestamp granularity: ~12 seconds on XRPL EVM sidechain
   - Attack scenario: Attacker sees VRF recovery in mempool, tries to vote before recovery tx lands. On XRPL EVM, block ordering is determined by validators, not MEV auctions. No practical timing exploit.
   - Verdict: SAFE

   **Window 4: Stall Re-Check in vote()**
   - Every vote() call re-checks `lastVrfProcessed()` against 20h threshold (verify exact line in vote(), should be lines 454-456)
   - If VRF has recovered between proposal creation and vote, vote reverts
   - This means Active proposals are effectively dead after VRF recovery even though they are not explicitly cancelled
   - They eventually expire at 168h from creation
   - Attack scenario: VRF briefly recovers then stalls again. Old proposal is still Active but vote() now passes the stall check again. Is the old proposal valid? Check: proposal stores `circulatingSnapshot` from creation time. If significant time passed, circulating supply may have changed (burns). The snapshot is from creation time, not current, so threshold calculation uses stale denominator. During a stall, sDGNRS is frozen, so snapshot remains accurate.
   - Verdict: SAFE (snapshot accuracy preserved because sDGNRS is frozen during stalls)

3. **Open Question Analysis** -- Evaluate the 3 open questions from research:

   **OQ-1: lastVrfProcessedTimestamp not reset on governance swap**
   - Read `updateVrfCoordinatorAndSub` in AdvanceModule and verify `lastVrfProcessedTimestamp` is NOT among the 7 reset variables
   - Implication: After a successful swap, the stall timer continues from the ORIGINAL stall until the new coordinator processes a request
   - Is this a finding? If the new coordinator takes time to fund/configure, governance could trigger again immediately
   - Evaluate: This is likely intentional -- if new coordinator also fails, governance should be able to swap again. The alternative (resetting the timer) would create a window where a malicious coordinator is installed and cannot be replaced for 20h.
   - Classification: INFO (intentional design, not a vulnerability) OR note as design observation

   **OQ-2: createSubscription not wrapped in try/catch**
   - Verify: `cancelSubscription` at lines 578-585 uses try/catch, `createSubscription` at lines 590-591 does NOT
   - Implication: If proposed coordinator's createSubscription reverts, entire tx reverts, all state changes revert (including `p.state = Executed` and `_voidAllActive`)
   - This is CORRECT behavior: a non-functional coordinator should not be installed
   - Classification: SAFE (correct design)

   **OQ-3: circulatingSupply changes between propose and execute**
   - `circulatingSnapshot` set once at proposal creation (verify in `propose()`)
   - During stall: sDGNRS burns reduce totalSupply but circulatingSnapshot stays fixed
   - If 10% of sDGNRS is burned, circulatingSnapshot is 10% too high -> threshold harder to meet (denominator unchanged, fewer voting tokens exist)
   - This HURTS attackers (need same % of a now-smaller supply but threshold uses old larger supply)
   - Burning one's own sDGNRS to avoid opposition: loses voting power while keeping denominator the same
   - Classification: SAFE (burns during stall make governance harder, not easier)

4. **GOV-02 Verdict** -- Overall verdict for timing attack analysis. Expected: No new findings. All 5 post-v2.1 changes are improvements (simplification, bug fixes, or neutral gas optimization). Timing windows are correctly ordered (5h < 20h < 168h). Open questions resolved as intentional design choices.
