  <action>
For every production contract listed in read_first (NOT mocks, NOT test/, NOT libraries/, NOT ContractAddresses.sol), perform two systematic scans:

**Scan A -- Forwarding Wrappers (SCAN-01):**
For each external/public non-view function, inspect the function body. A function is a "forwarding wrapper" if its ENTIRE body is a single call to another contract's function with identical (or trivially mapped) parameters and the return value is passed through unchanged. The classic example: `BurnieCoin.creditFlip(addr, amt)` whose body is just `burnieCoinflip.creditFlip(addr, amt)` -- callers could call BurnieCoinflip directly.

For each forwarding wrapper found, record:
- Function name and signature
- Source contract (where the wrapper lives)
- Target contract and function (what it delegates to)
- Whether the wrapper adds ANY logic (access control, event emission, state change) beyond the forwarded call -- if it does, it is NOT a candidate
- Removal rationale (e.g., "callers can call TargetContract.fn() directly")
- Risk notes (e.g., "check if any external caller relies on this contract address as entry point")

**Scan B -- Unused View/Pure Functions (SCAN-02):**
For each view/pure function, use grep across ALL .sol files in contracts/ (including modules/, interfaces/, libraries/, storage/) to determine if ANY other contract calls it on-chain (cross-contract). A function is an "unused view" candidate if:
1. No other .sol file contains a cross-contract call to it (e.g., `contractInstance.functionName(`)
2. It is not part of a standard interface (ERC-20, ERC-721) that external tooling expects
3. It is not an override of a parent contract's virtual function that other contracts depend on

For the grep, search for patterns like `contractVarName.functionName(` or `InterfaceName(addr).functionName(` across all .sol files. Be thorough -- check both the function name and reasonable variable names that reference the contract.

Important: view/pure functions called by OTHER contracts on-chain are NOT candidates. Only functions with zero on-chain callers are candidates. Functions that exist purely for off-chain/UI consumption but add no unique data (just forwarding a view from another contract) ARE candidates. Functions that provide genuinely useful aggregated/computed data for off-chain use should be noted but marked with lower confidence.
