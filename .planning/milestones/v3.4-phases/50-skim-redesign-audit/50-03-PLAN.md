**ECON-01: Overshoot Surcharge Accelerates Futurepool Growth During Fast Levels**

Economic mechanism trace:
1. When a level fills fast, nextPool grows large relative to lastPool. The ratio `rBps = (nextPoolBefore * 10000) / lastPool` exceeds 12500 (1.25x threshold)
2. The surcharge `(excess * 4000) / (excess + 10000)` adds bps on top of the base U-curve bps
3. More bps means larger `take = (nextPoolBefore * bps) / 10000`
4. Larger take moves more ETH from nextPool to futurePool, accelerating future prize growth

Numeric walkthrough -- R=3.0 scenario (nextPool = 3x lastPool):
- rBps = 30000
- excess = 30000 - 12500 = 17500
- surcharge = (17500 * 4000) / (17500 + 10000) = 70000000 / 27500 = 2545 bps
- If base bps = 3000 (fast fill, no bonus), total bps = 3000 + 2545 = 5545
- take = nextPool * 5545 / 10000 = 55.45% of nextPool -- substantially higher than the 30% base

Numeric walkthrough -- R=1.5 scenario (modest overshoot):
- rBps = 15000
- excess = 15000 - 12500 = 2500
- surcharge = (2500 * 4000) / (2500 + 10000) = 10000000 / 12500 = 800 bps
- If base bps = 3000, total bps = 3800
- take = 38% of nextPool -- moderate acceleration

Numeric walkthrough -- R=1.24 (just below threshold):
- rBps = 12400 < 12500
- Overshoot block does not execute
- take stays at base bps only -- no acceleration

Cross-reference: `test_B_fastOvershoot_R3` (line 135) confirms R=3.0 produces higher take than baseline. SKIM-01 verdict confirms monotonicity -- larger R always produces larger surcharge.

Verdict: SAFE -- Overshoot surcharge correctly adds bps proportional to growth rate. Monotonicity (from SKIM-01) ensures faster levels always get higher surcharge. The hyperbolic shape provides gentle ramp-up near threshold and asymptotic cap at 3500 bps for extreme growth.

**ECON-02: Stall Escalation Functions Without Growth Adjustment**

The stall escalation path in `_nextToFutureBps` (lines 976-980):
```
} else {
    // elapsed > 28 days
    bps = NEXT_TO_FUTURE_BPS_FAST + lvlBonus +
        ((elapsed - 28 days) / 1 weeks) * NEXT_TO_FUTURE_BPS_WEEK_STEP;
}
```

This formula is:
- `bps = 3000 + lvlBonus + weeksStalled * 100`
- Where `lvlBonus = (lvl % 100 / 10) * 100` ranges from 0 to 900
- And `weeksStalled = (elapsed - 28 days) / 1 weeks`

Key observation: This formula has NO reference to any growth adjustment variable. The old growth adjustment was removed in commit b06d80a8 and replaced with the overshoot surcharge. The stall path is purely a function of `elapsed` and `lvl` -- both of which are unchanged by the redesign.

Regression check:
- The stall path starts at elapsed > 28 days, producing bps = 3000 + lvlBonus (same as fast fill)
- Each additional week adds 100 bps (1% more skim per week)
- At 60 days stall: weeksStalled = (60-28)/7 = 4, bps = 3000 + lvlBonus + 400
- Return value capped at 10000 by line 983: `return uint16(bps > 10_000 ? 10_000 : bps)`
- Time to saturation (bps=10000) with lvlBonus=0: (10000-3000)/100 = 70 weeks beyond 28d = ~98 weeks total

Cross-reference: `test_D_stall_60day` (line 180) confirms stall at 60 days produces non-zero take with escalated bps. The test was written for the redesigned code and passes.

Verdict: SAFE -- Stall escalation is completely independent of the removed growth adjustment. The formula `FAST + lvlBonus + weeks*100` is self-contained and produces monotonically increasing bps with stall duration, correctly pressuring stalled pools.

**ECON-03: Level 1 (lastPool=0) is Safe**

This requirement has two distinct scenarios that must be analyzed separately:

**Scenario A: The `lastPool=0` guard**
- Line 1012: `if (lastPool != 0)` guards the entire overshoot block
- When lastPool=0, the overshoot surcharge is skipped entirely -- no division, no surcharge computation
- This is correct: if there is no previous pool to compare against, there is no meaningful growth ratio
- The test `test_level1_overshootDormant` (line 455) validates this by passing lastPool=0 and asserting no surcharge is applied

**Scenario B: Production level 1 reality**
- In the constructor (DegenerusGame.sol line 252): `levelPrizePool[0] = BOOTSTRAP_PRIZE_POOL` where `BOOTSTRAP_PRIZE_POOL = 50 ether` (DegenerusGameStorage.sol line 137-138)
- At level 1, `lastPool = levelPrizePool[lvl - 1] = levelPrizePool[0] = 50 ether`
- This means lastPool is NOT zero at level 1 in production -- it is 50 ether
- The overshoot threshold fires when `rBps = (nextPoolBefore * 10000) / lastPool > 12500`, i.e., when nextPoolBefore > lastPool * 1.25 = 62.5 ether
- If level 1 fills to >62.5 ether, the overshoot surcharge WILL activate

**Impact assessment of overshoot at level 1:**
- At R=1.5 (nextPool=75 ether, lastPool=50 ether): surcharge = (2500*4000)/(2500+10000) = 800 bps. This is a moderate acceleration of futurepool growth -- not a problem, it is the intended behavior
- The surcharge increases the skim from nextPool to futurePool, which means more ETH is saved for future prizes. This is economically desirable at level 1 where the game is establishing its prize pool structure
- There is no exploit: overshoot surcharge only moves ETH WITHIN the system (next -> future), it does not create or destroy ETH (proven by SKIM-06)

**Test gap:**
- The existing test `test_level1_overshootDormant` uses lastPool=0, which is unreachable in production. The test proves the guard works but does not test the production scenario
- Recommend: Add a test with lastPool=50 ether and nextPool=75 ether to verify overshoot fires correctly at level 1 with bootstrap pool. This is a test coverage gap, not a code bug

Verdict: SAFE -- The `if (lastPool != 0)` guard correctly prevents division-by-zero for the theoretical lastPool=0 case. In production, level 1 has lastPool=50 ether and overshoot CAN fire -- this is acceptable behavior as it moves ETH within the system per SKIM-06 conservation. INFO: Test `test_level1_overshootDormant` uses unreachable lastPool=0; recommend adding a test with production bootstrap value.
