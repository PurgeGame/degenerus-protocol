**SKIM-04: Triangular Variance Cannot Underflow Take**
- Analyze lines 1033-1044 of DegenerusGameAdvanceModule.sol
- Step through the bounds chain:
  1. `halfWidth = (take * NEXT_SKIM_VARIANCE_BPS) / 10000` = take * 2500 / 10000 = take/4. Since take > 0 (guard at line 1030), halfWidth >= 0
  2. `minWidth = (nextPoolBefore * NEXT_SKIM_VARIANCE_MIN_BPS) / 10000` = nextPoolBefore * 1000 / 10000 = nextPoolBefore/10. Floor for halfWidth
  3. `if (halfWidth < minWidth) halfWidth = minWidth` -- raises halfWidth if too small
  4. `if (halfWidth > take) halfWidth = take` -- THIS IS THE CRITICAL CLAMP. halfWidth <= take
  5. `range = halfWidth * 2 + 1` -- range in [1, take*2+1]
  6. `roll1 = (rngWord >> 64) % range` -- roll1 in [0, range-1] = [0, halfWidth*2]
  7. `roll2 = (rngWord >> 192) % range` -- roll2 in [0, halfWidth*2]
  8. `combined = (roll1 + roll2) / 2` -- combined in [0, halfWidth*2] (because (max+max)/2 = max). More precisely combined in [0, halfWidth*2] but since integer division, combined in [0, halfWidth*2]
  9. Subtraction path (combined < halfWidth): `take -= halfWidth - combined`. Since halfWidth <= take (from clamp at step 4), and (halfWidth - combined) <= halfWidth <= take, the subtraction is safe
  10. Addition path (combined >= halfWidth): `take += combined - halfWidth`. Since combined <= halfWidth*2, addition adds at most halfWidth. No overflow concern for uint256
- Note the minWidth floor (step 3) can push halfWidth above take/4, but the clamp at step 4 ensures halfWidth <= take regardless
- Cross-reference: `testFuzz_G2_takeCapped` at line 262 runs 1000 fuzz iterations and never reverts (implicit underflow safety). Also `testFuzz_conservation` at line 404 proves no ETH is lost/created
- Verdict: SAFE -- the `halfWidth > take` clamp at line 1034 guarantees the subtraction path cannot underflow. Formal bound: subtraction amount = halfWidth - combined <= halfWidth <= take
