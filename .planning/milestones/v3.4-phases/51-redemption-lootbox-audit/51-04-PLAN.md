## REDM-06: Lootbox Reclassification -- No ETH Transfer

Audit the internal accounting in DegenerusGame.sol resolveRedemptionLootbox():

1. **Debit side -- Lines 1808-1813**:
   ```
   uint256 claimable = claimableWinnings[ContractAddresses.SDGNRS];
   unchecked {
       claimableWinnings[ContractAddresses.SDGNRS] = claimable - amount;
   }
   claimablePool -= amount;
   ```

   a. **Unchecked subtraction (Open Question 2)**: Can `amount > claimable`?
      - Trace the source of `amount`: it comes from `lootboxEth` in sDGNRS.claimRedemption (line 624)
      - `lootboxEth = totalRolledEth - ethDirect` where `totalRolledEth = (ethValueOwed * roll) / 100`
      - The original `ethValueOwed` was segregated from sDGNRS's claimable winnings (via _claimableWinnings on line 723)
      - But the segregation was at base value (100%), and the rolled value can be up to 175% of base
      - When roll > 100: `totalRolledEth > ethValueOwed`. The lootbox portion is `totalRolledEth / 2` which could exceed the original base.
      - CRITICAL: Verify that `claimableWinnings[SDGNRS]` is updated during period resolution (advanceGame) to account for the rolled amount, not just the base amount
      - If claimableWinnings is only credited with base amounts but debited with rolled amounts, this underflows

   b. **Checked subtraction on claimablePool** (line 1813): `claimablePool -= amount`
      - This is a CHECKED subtraction (not in unchecked block). If amount > claimablePool, this reverts.
      - This provides a secondary guard -- but claimablePool is the GLOBAL pool, not per-address. Can claimablePool be >= amount while claimableWinnings[SDGNRS] < amount? If so, the unchecked underflow on line 1811 silently creates a massive claimableWinnings[SDGNRS] value.
      - Trace: what credits claimableWinnings[SDGNRS] and claimablePool? The advanceGame resolution should credit both.

   c. **No ETH transfer**: Verify that between lines 1808-1813, there is no `.call`, `.send`, `.transfer`, or `payable` interaction. This is pure storage manipulation. Confirmed by reading the code -- no external calls in this section.

2. **Credit side -- Lines 1815-1822**:
   ```
   if (prizePoolFrozen) {
       (uint128 pNext, uint128 pFuture) = _getPendingPools();
       _setPendingPools(pNext, pFuture + uint128(amount));
   } else {
       (uint128 next, uint128 future) = _getPrizePools();
       _setPrizePools(next, future + uint128(amount));
   }
   ```

   a. **uint128 cast (Open Question 3)**: Can `amount > uint128.max`?
      - uint128.max = 3.4e38. Maximum lootbox ETH per claim: totalRolledEth / 2 = (160 ETH * 175 / 100) / 2 = 140 ETH
      - 140 ETH = 1.4e20 wei. 1.4e20 << 3.4e38. SAFE from truncation.
      - Verify no other caller of resolveRedemptionLootbox can pass a larger amount (access control ensures only sDGNRS calls this)

   b. **No ETH transfer**: The _setPrizePools and _setPendingPools functions are pure storage writes (slot packing). Verify by reading these helper functions -- they should contain no external calls.

   c. **Freeze state handling**: Verify both branches (frozen and unfrozen) credit the correct pool. When prizePoolFrozen is true, credits go to pending pools (accumulated for next unfreeze). When false, credits go directly to live pools.

3. **Delegatecall to LootboxModule -- Lines 1824-1844**:
   ```
   uint256 remaining = amount;
   while (remaining != 0) {
       uint256 box = remaining > 5 ether ? 5 ether : remaining;
       (bool ok, bytes memory data) = ContractAddresses.GAME_LOOTBOX_MODULE.delegatecall(...);
       if (!ok) _revertDelegate(data);
       remaining -= box;
       rngWord = uint256(keccak256(abi.encode(rngWord)));
   }
   ```

   a. **5 ETH chunking**: Verify `remaining -= box` is safe (box <= remaining by construction)
   b. **Entropy rotation**: Verify each chunk gets different entropy via keccak256 re-hash
   c. **No ETH transfer in LootboxModule**: Since this is delegatecall, the LootboxModule runs in Game's storage context. Verify that _resolveLootboxCommon does NOT execute any `.call{value:...}` or `.transfer` -- it only awards tickets and BURNIE tokens (internal accounting).
   d. Read _resolveLootboxCommon (lines 849-1025) and scan for any ETH transfer instruction. Expected: only `emit` events, ticket increments, and BURNIE token minting/crediting.

4. **Conservation check**: Verify that the debit (claimableWinnings[SDGNRS] decrease) exactly equals the credit (futurePrizePool increase). The `amount` variable is used in both operations, so they must be equal. No ETH is created or destroyed.

## REDM-07: Cross-Contract Access Control

Audit the access control at each hop of the call chain:

1. **Hop 1: sDGNRS -> Game** (line 624 in StakedDegenerusStonk.sol):
   ```
   game.resolveRedemptionLootbox(player, lootboxEth, entropy, actScore);
   ```
   - Caller: StakedDegenerusStonk contract
   - Callee: DegenerusGame contract

2. **Hop 1 gate: Game checks caller** (line 1805 in DegenerusGame.sol):
   ```
   if (msg.sender != ContractAddresses.SDGNRS) revert E();
   ```
   - Verify ContractAddresses.SDGNRS is the correct sDGNRS address
   - Verify this gate is the FIRST check in the function (before any state changes)
   - Verify: can any other contract impersonate sDGNRS? Only if deployed at the same address -- ContractAddresses uses deterministic CREATE2 or hardcoded addresses. Read ContractAddresses.sol to confirm.

3. **Hop 2: Game -> LootboxModule** (lines 1828-1840 in DegenerusGame.sol):
   ```
   ContractAddresses.GAME_LOOTBOX_MODULE.delegatecall(...)
   ```
   - This is a DELEGATECALL: the LootboxModule code runs in Game's storage context
   - The LootboxModule function is called with Game's msg.sender (which is sDGNRS)
   - But since it's delegatecall, the LootboxModule does NOT need its own access control -- it inherits Game's context
   - Verify: the LootboxModule's `resolveRedemptionLootbox` function (line 724) does NOT have a separate access control check. If it does, verify the check would pass in the delegatecall context.

4. **Attack surface analysis**:
   a. Can an EOA call Game.resolveRedemptionLootbox directly? No -- msg.sender would not be SDGNRS address
   b. Can a malicious contract call Game.resolveRedemptionLootbox? No -- same reason
   c. Can sDGNRS be tricked into calling resolveRedemptionLootbox with attacker-controlled parameters? Only through claimRedemption, which reads from the player's own PendingRedemption storage
   d. Can a reentrancy in the lootbox resolution corrupt state? Check: does Game.resolveRedemptionLootbox modify state BEFORE the delegatecall loop? Yes -- lines 1810-1822 debit/credit BEFORE the delegatecall loop on lines 1824-1844. This is the checks-effects-interactions pattern. SAFE.

5. **LootboxModule internal access**: Read the LootboxModule resolveRedemptionLootbox function header (line 724). It is `external` -- verify it can only be reached via delegatecall from Game (not directly). If it has no access control of its own, an EOA could call it directly on the LootboxModule contract. But since the LootboxModule's storage is separate from Game's storage, a direct call would operate on wrong storage. Verify: is there any risk from direct calls to the LootboxModule's own storage?
