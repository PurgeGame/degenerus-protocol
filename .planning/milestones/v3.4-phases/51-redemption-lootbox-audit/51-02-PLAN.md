## REDM-03: 160 ETH Daily Cap Enforcement

Audit these specific code points in StakedDegenerusStonk.sol _submitGamblingClaimFrom():

1. **Cap check -- Line 753**: `if (claim.ethValueOwed + ethValueOwed > MAX_DAILY_REDEMPTION_EV) revert ExceedsDailyRedemptionCap();`
   - Verify MAX_DAILY_REDEMPTION_EV = 160 ether (line 227) = 1.6e20 wei
   - Verify this is a CUMULATIVE check (claim.ethValueOwed is the existing accumulated value + new ethValueOwed)
   - Verify claim.ethValueOwed is uint96 (read from storage), ethValueOwed is uint256 (computed on line 725)
   - Verify the addition cannot overflow in uint256 context (uint96.max + uint256 value fits in uint256)
   - Verify the check happens BEFORE the uint96 cast on line 755

2. **Period gating -- Lines 747-750**:
   ```
   if (claim.periodIndex != 0 && claim.periodIndex != currentPeriod) {
       revert UnresolvedClaim();
   }
   ```
   - Verify this prevents cross-period stacking: if a player has an unresolved claim from a previous period, they cannot burn again
   - Verify periodIndex == 0 means "no active claim" (fresh state after delete in claimRedemption)
   - Verify periodIndex == currentPeriod means "same period, can stack up to cap"

3. **Multiple calls within same period**:
   - Trace: first burn sets claim.ethValueOwed = X, second burn in same period adds Y
   - Line 755: `claim.ethValueOwed += uint96(ethValueOwed)` -- cumulative addition
   - The cap check on line 753 uses `claim.ethValueOwed + ethValueOwed` which is the FULL accumulated value
   - Verify: after two burns of 80 ETH each, claim.ethValueOwed = 160 ETH, third burn of any amount reverts

4. **Cross-day boundary analysis (Pitfall 4)**:
   - currentPeriod comes from `game.currentDayView()` (line 709) which uses GameTimeLib
   - Read GameTimeLib to verify: day resets at 22:57 UTC (not midnight)
   - Scenario: burn 160 ETH at 22:56:59, advanceGame resolves the period, burn 160 ETH at 22:57:01
   - This gives 320 ETH in 2 seconds. Is this a vulnerability?
   - Verify: the UnresolvedClaim check on line 748 PREVENTS this unless the first period is resolved first
   - The first period resolution requires advanceGame (which requires RNG), so the window is not instant
   - Document this as by-design: one cap per period, period boundaries create new windows

5. **50% supply cap per period -- Lines 708-716**:
   - This is a separate cap (not REDM-03 specifically) but interacts with the daily cap
   - Verify it cannot be used to bypass the 160 ETH cap (it restricts token amount, not ETH value)

## REDM-05: PendingRedemption Slot Packing

Audit the struct definition and all cast sites:

1. **Struct definition -- Lines 182-187**:
   ```
   struct PendingRedemption {
       uint96  ethValueOwed;   // bits [0:95]
       uint96  burnieOwed;     // bits [96:191]
       uint48  periodIndex;    // bits [192:239]
       uint16  activityScore;  // bits [240:255]
   }
   ```
   - Verify: 96 + 96 + 48 + 16 = 256 bits exactly
   - Verify: Solidity compiler packs struct fields in order for storage
   - Verify: no padding needed (all fields fill the slot completely)
   - Verify: no bit overlap (each field occupies a distinct range)

2. **uint96 cast for ethValueOwed -- Line 755**: `claim.ethValueOwed += uint96(ethValueOwed)`
   - ethValueOwed is computed on line 725: `(totalMoney * amount) / supplyBefore`
   - The cap check on line 753 ensures `claim.ethValueOwed + ethValueOwed <= 160 ether`
   - 160 ether = 1.6e20. uint96.max = 7.922e28. 1.6e20 << 7.922e28. SAFE from truncation.

3. **uint96 cast for burnieOwed -- Line 756**: `claim.burnieOwed += uint96(burnieOwed)`
   - burnieOwed is computed on line 731: `(totalBurnie * amount) / supplyBefore`
   - **CRITICAL (Open Question 1)**: burnieOwed has NO explicit cap
   - Compute worst case: initial BURNIE supply ~1 trillion (1e30 in 18-decimal notation = 1e48 raw units? Check BURNIE decimals). If totalBurnie = 1e30 and amount/supplyBefore is close to 1 (burning nearly all supply), burnieOwed could approach 1e30
   - uint96.max = 7.922e28. If burnieOwed > 7.922e28, truncation occurs silently
   - DETERMINE: Is BURNIE an 18-decimal token? What is the realistic max totalBurnie?
   - Read the BURNIE token contract or check its decimals to determine if overflow is possible
   - If overflow IS possible, this is a FINDING (severity depends on likelihood)

4. **uint48 cast for periodIndex -- Line 757**: `claim.periodIndex = currentPeriod`
   - currentPeriod is returned by game.currentDayView() as uint48
   - No cast needed (already uint48). Verify the return type.

5. **uint16 cast for activityScore -- Line 761**: `claim.activityScore = uint16(game.playerActivityScore(beneficiary)) + 1`
   - Verify playerActivityScore returns a value that fits in uint16 minus 1 (max 65534 before +1)
   - If playerActivityScore can return >= 65535, the +1 overflows uint16 to 0
   - Check playerActivityScore return type and range
