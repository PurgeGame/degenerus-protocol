## REDM-01: 50/50 Split Routing

Audit these specific code points in StakedDegenerusStonk.sol claimRedemption():

1. **Line 584**: `uint256 totalRolledEth = (claim.ethValueOwed * roll) / 100;`
   - Verify roll range is [25, 175] (from RedemptionPeriod.roll, set during advanceGame resolution)
   - Verify claim.ethValueOwed is uint96 (max 7.9e28), roll is uint16 (max 65535), product fits uint256
   - Verify no overflow: uint96.max * 175 = 1.38e31, well within uint256

2. **Lines 592-594**: The split logic:
   ```
   ethDirect = totalRolledEth / 2;
   lootboxEth = totalRolledEth - ethDirect;
   ```
   - Prove conservation: ethDirect + lootboxEth == totalRolledEth for ALL values
   - Arithmetic proof: (x/2) + (x - x/2) = x (integer division: x/2 truncates, x - x/2 captures remainder)
   - Document: on odd totalRolledEth, lootboxEth gets the extra 1 wei (e.g., totalRolledEth=101: ethDirect=50, lootboxEth=51)

3. **Lines 619-624**: Lootbox resolution call:
   - Verify lootboxEth is only non-zero when isGameOver is false (line 594 only reached in else branch)
   - Verify the guard `if (lootboxEth != 0)` on line 620 prevents spurious calls
   - Verify actScore computation on line 621: `claimActivityScore > 0 ? claimActivityScore - 1 : 0` -- correctly reverses the +1 encoding from storage

4. **Line 609**: `pendingRedemptionEthValue -= totalRolledEth;`
   - Verify this releases the FULL segregated amount (both direct and lootbox portions)
   - Verify this cannot underflow: totalRolledEth = (ethValueOwed * roll) / 100, and ethValueOwed was originally added to pendingRedemptionEthValue at face value. When roll > 100, totalRolledEth > ethValueOwed. Check if aggregate pendingRedemptionEthValue across ALL claims can support this.
   - IMPORTANT: This is a critical check. If roll=175 and ethValueOwed=160 ETH, totalRolledEth=280 ETH. But pendingRedemptionEthValue only had 160 ETH added. The resolution in advanceGame must adjust pendingRedemptionEthValue by the roll. Find and verify this adjustment.

## REDM-02: gameOver Bypass

Audit these specific code points:

1. **Line 590**: `if (isGameOver) { ethDirect = totalRolledEth; }`
   - Verify lootboxEth remains 0 (default uint256 value, never assigned in this branch)
   - Verify resolveRedemptionLootbox is never called (guard on line 620: `if (lootboxEth != 0)`)

2. **_deterministicBurnFrom (lines 479-521)**:
   - Verify this path is only entered when `game.gameOver()` is true (check burn() at line 448 and burnWrapped() at line 462)
   - Verify no BURNIE payout: line 519 comment "No BURNIE payout for gameOver burns", line 520 `emit Burn(beneficiary, amount, ethOut, stethOut, 0)` -- the 0 is burnieOut
   - Verify no call to _submitGamblingClaimFrom or resolveRedemptionLootbox
   - Verify pendingRedemptionEthValue is properly deducted from totalMoney on line 487

3. **gameOver transition during active claims (Pitfall 6)**:
   - Analyze: player submits gambling burn during active game (enters _submitGamblingClaimFrom), then gameOver triggers before they claim
   - In claimRedemption, game.gameOver() returns true -> ethDirect = totalRolledEth (100% direct, no lootbox)
   - Verify this is safe: player gets full ETH value, no lootbox resolution, BURNIE still handled by coinflip
   - Document this as an intentional design decision (player benefits from 100% direct ETH)
