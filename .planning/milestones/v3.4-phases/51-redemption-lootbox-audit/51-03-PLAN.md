## REDM-04: Activity Score Snapshot Immutability

### Phase 1: Snapshot (Write Path)

1. **Lines 759-762** in _submitGamblingClaimFrom:
   ```
   if (claim.activityScore == 0) {
       claim.activityScore = uint16(game.playerActivityScore(beneficiary)) + 1;
   }
   ```
   - Verify the guard condition: `activityScore == 0` means "not yet set" (fresh struct or after delete)
   - Verify the +1 encoding: score is stored as `score + 1` so that 0 unambiguously means "not set"
   - Verify this is called only ONCE per period: on the first burn, activityScore is 0 -> set. On subsequent burns in the same period, activityScore is non-zero -> guard prevents overwrite
   - Verify: what if playerActivityScore returns 0? Then claim.activityScore = 0 + 1 = 1 (non-zero, guard works)
   - Verify: what if playerActivityScore returns 65535 (uint16 max - 1)? Then claim.activityScore = 65536, which wraps to 0 in uint16. This would make the guard fail and allow re-snapshotting on the next burn. Determine if playerActivityScore can ever return >= 65535.
   - Check game.playerActivityScore() return type and max value

2. **No mutation paths**: Search the contract for any other write to `claim.activityScore` or `pendingRedemptions[...].activityScore` outside of line 761. The only other modification should be `delete pendingRedemptions[player]` on line 613 (which zeros everything).

### Phase 2: Consumption (Read Path)

3. **Line 581** in claimRedemption:
   ```
   uint16 claimActivityScore = claim.activityScore;
   ```
   - Verify this reads the snapshotted value into a LOCAL variable BEFORE any struct modification
   - Verify line 581 occurs before line 613 (`delete pendingRedemptions[player]`) and line 616 (`claim.ethValueOwed = 0`)
   - The local variable `claimActivityScore` is immune to storage changes after this point

4. **Line 621** in claimRedemption:
   ```
   uint16 actScore = claimActivityScore > 0 ? claimActivityScore - 1 : 0;
   ```
   - Verify the +1 encoding is correctly reversed: if claimActivityScore was stored as score+1, subtracting 1 recovers the original score
   - Edge case: claimActivityScore == 0 (should never happen because line 574 `if (claim.periodIndex == 0) revert NoClaim()` ensures a claim exists, and if a claim exists, activityScore was set on first burn). But the ternary handles it safely with a fallback to 0.
   - Edge case: claimActivityScore == 1 (original score was 0). actScore = 0. This is correct.

5. **Line 624** in claimRedemption:
   ```
   game.resolveRedemptionLootbox(player, lootboxEth, entropy, actScore);
   ```
   - Verify actScore is passed as the 4th parameter to Game.resolveRedemptionLootbox

### Phase 3: Cross-Contract Flow

6. **DegenerusGame.sol line 1798**: `uint16 activityScore` parameter received
   - Verify this is the same value passed from sDGNRS (no transformation in Game)
   - Verify it's forwarded to LootboxModule delegatecall at line 1838

7. **LootboxModule line 724**: `uint16 activityScore` parameter received
   - Verify line 732: `_lootboxEvMultiplierFromScore(uint256(activityScore))` -- casts uint16 to uint256 for the multiplier function
   - Verify the multiplier function at lines 476-501 uses the score correctly:
     - Score 0 -> minimum EV (LOOTBOX_EV_MIN_BPS = 8000 = 80%)
     - Score at neutral threshold -> neutral EV (10000 = 100%)
     - Score at max -> maximum EV (LOOTBOX_EV_MAX_BPS = 13500 = 135%)
   - Verify no path reads the LIVE activity score instead of the snapshotted one

8. **LootboxModule line 733**: `_applyEvMultiplierWithCap(player, currentLevel, amount, evMultiplierBps)`
   - Verify the 10 ETH per-account-per-level cap is applied correctly
   - Verify this cap cannot be bypassed by manipulating the activity score

### Phase 4: Partial Claim Interaction

9. **Lines 611-617** in claimRedemption (partial claim when coinflip unresolved):
   ```
   if (flipResolved) {
       delete pendingRedemptions[player]; // Full delete: zeros activityScore
   } else {
       claim.ethValueOwed = 0; // Partial: keeps activityScore intact
   }
   ```
   - Verify: on partial claim, activityScore is NOT zeroed (only ethValueOwed is zeroed)
   - Verify: on the second claim (when coinflip resolves), the player does NOT re-enter claimRedemption for lootbox purposes (ETH portion already claimed, only BURNIE remains)
   - Trace the second claim path: claim.ethValueOwed == 0 -> totalRolledEth == 0 -> lootboxEth == 0 -> resolveRedemptionLootbox NOT called. This means activityScore is effectively consumed only once even in the split-claim case.
