## Tier definitions (D-226-04, inherited from D-225-04) — verbatim

**Tier A (always flag): Outright wrong.** A claim in a comment factually contradicts the column declaration.
- Example: comment says "UNIX seconds" but column type is `timestamp` (stores ISO datetime).
- Example: comment says "FK to `players`" but no `.references(...)` and no `foreignKey(...)` exists.
- Example: comment says "nullable" but column has `.notNull()`.

**Tier B (flag when material): Stale/incomplete.** Comment documents the column but omits a material property a caller would rely on.
- Example: comment says "FK to players" but omits that it's nullable with DEFAULT NULL.
- Example: comment describes purpose but omits the unit (ms vs seconds vs blocks).
- Example: comment describes two error conditions but code has a third.

**Tier C (DO NOT flag per-column; count as context only):** column has no comment whatsoever.
**Tier D (DO NOT flag anywhere):** cosmetic drift, typos, stale parameter names after rename.

## Comment-extraction scope (226-RESEARCH.md §Comment Extraction for SCHEMA-02)

Scout finding: the 3 sampled schema files contain **no column-level JSDoc blocks** and **no `.comment(...)` runtime calls**. Comments live at:
1. **File-header JSDoc** (e.g. `trait-burn-tickets.ts:1-12`) — often describes multiple columns of a table in prose. Treat EACH claim about a specific column within these blocks as a Tier A/B candidate.
2. **Inline `//` comments** co-located on or directly above a column declaration line.
3. **Table-level JSDoc** (block comment immediately preceding a `pgTable(...)`) — same treatment: each column-claim inside is a candidate.

If the plan 226-03 executor finds `.comment(...)` calls or actual column-level `/** */` blocks in schema files not sampled during scout, treat them per the same Tier threshold — do not skip.

Scope is expected to be SMALL (A1 in 226-RESEARCH.md Assumptions Log): 30 tables × 2-5 column-claims-per-table = 60-150 rows max. Full enumeration, not sampling.

## Severity rules

- Default: **INFO** (most comment drift is documentation-only).
- Promote to **LOW** only when a caller relying on the comment would produce wrong data (e.g. comment says "seconds" but value is milliseconds → client may schedule events 1000× wrong; comment says "nullable" but `.notNull()` → inserts with NULL rejected at runtime).
