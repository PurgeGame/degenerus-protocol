**NEW pattern:**
```solidity
// ... in per-winner loop:
    emit JackpotBurnieWin(winner, lvl, traitIds[traitIdx], amount, bucketTicketIndexes[i]);
    coinflip.creditFlip(winner, amount);
```

**Proof of equivalence:**

1. **Same set of (player, amount) pairs:** Both old and new iterate the same `bucketWinners` array from `_randTraitTicket` with the same `baseAmount + extra` calculation. The winner selection, amount computation, and `winner != address(0) && amount != 0` guard are identical.

2. **Credit effect:** OLD: `creditFlipBatch([w1,w2,w3], [a1,a2,a3])` which internally calls `creditFlip` for each non-zero entry. NEW: `creditFlip(winner, amount)` called directly per-winner. Net effect: identical credits to identical addresses.

3. **Gas tradeoff:** OLD batched 3 at a time to amortize call overhead. NEW uses per-winner calls (simpler code, slightly more gas per call but no batch bookkeeping). This is a gas optimization choice, not a behavioral change.

4. **Event change:** OLD emitted `JackpotTicketWinner(..., AWARD_BURNIE)`. NEW emits `JackpotBurnieWin(...)`. Covered in Section 5.

**Verdict: EQUIVALENT** -- Same credits to same addresses with same amounts. Batch-of-3 replaced by per-winner calls with identical net effect.

### 4.9: payDailyJackpot Coin Jackpot Guard Removal (Item 36)

**OLD:**
```solidity
uint24 targetLevel = _selectDailyCoinTargetLevel(lvl, winningTraitsPacked, coinEntropy);
if (targetLevel != 0) {
    _awardDailyCoinToTraitWinners(targetLevel, winningTraitsPacked, nearBudget, entropy);
}
```

**NEW:**
```solidity
uint24 targetLevel = _selectDailyCoinTargetLevel(lvl, entropy);
{
    _awardDailyCoinToTraitWinners(targetLevel, winningTraitsPacked, nearBudget, entropy);
}
```

**Context:** This guard removal is a CONSEQUENCE of `_selectDailyCoinTargetLevel` simplification (Item 25, INTENTIONAL, covered in Plan 02). OLD `_selectDailyCoinTargetLevel` could return 0 (skip) when no trait tickets existed. NEW always returns `lvl + uint24(entropy % 5)` (never 0).

**Why the guard removal is safe as a REFACTOR classification:**

1. When `_selectDailyCoinTargetLevel` returns a non-zero level but the target level has no tickets: `_awardDailyCoinToTraitWinners` calls `_computeBucketCounts` which returns `activeCount == 0`, causing immediate return. No winners selected, no credits, no events.

2. When OLD returned 0 (skip): the `if (targetLevel != 0)` guard prevented the call entirely.

3. NEW always calls `_awardDailyCoinToTraitWinners`, but when no tickets exist, `_computeBucketCounts` returns `activeCount == 0` and the function returns immediately.

4. **Net effect:** Identical outcomes. The guard was redundant because `_awardDailyCoinToTraitWinners` naturally handles empty buckets. The only difference is a slightly longer gas path (entering the function, computing bucket counts, then returning) vs. not entering at all.

**Note:** The full correctness proof for the `_selectDailyCoinTargetLevel` simplification itself is INTENTIONAL and covered in Plan 02. This section only proves the guard removal does not change behavior.

**Verdict: EQUIVALENT** -- Guard was redundant. `_awardDailyCoinToTraitWinners` handles empty buckets gracefully.
