# Phase 160.1: Purchase Path Correctness - Context

**Gathered:** 2026-04-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix three correctness issues in the MintModule purchase path discovered during Phase 161 SLOAD dedup work:
1. `purchaseLevel` doesn't account for jackpot phase (always `level + 1`, should be `level` during jackpot)
2. Lootbox level references inconsistently use `purchaseLevel` vs `level` — lootbox baseline should always be `level + 1` regardless of jackpot phase
3. MintModule reads the `price` storage variable when `PriceLookupLib.priceForLevel` already exists and is used by every other module — price should be computed from the target level, not read from stale storage

</domain>

<decisions>
## Implementation Decisions

### D-01: purchaseLevel definition
`purchaseLevel` must reflect the actual ticket target level:
- Purchase phase: `level + 1` (tickets go to next level)
- Jackpot phase: `level` (tickets go to current level)

Change L624 from `uint24 purchaseLevel = level + 1` to `uint24 purchaseLevel = jackpotPhaseFlag ? level : level + 1`.

### D-02: Lootbox level baseline
All lootbox level references use `level + 1` regardless of jackpot phase. Lootbox tickets always route to the next level. Specifically:
- L694 `lootboxBaseLevelPacked`: currently `level + 2` — **BUG**, should be `level + 1`
- L709 `lootboxEth` packing: currently `purchaseLevel` — must be `level + 1` (after D-01 fix, `purchaseLevel` would be `level` during jackpot)
- L728 presale check: `level == 0` — correct as-is (raw game state check)
- L761 emit: `level` — correct as-is (raw level for indexing)
- L811, L821 affiliate `payAffiliate`: currently `purchaseLevel` — must be `level + 1`
- L829 `_awardEarlybirdDgnrs`: currently `purchaseLevel` — must be `level + 1`

### D-03: Replace price storage with PriceLookupLib.priceForLevel
`PriceLookupLib.priceForLevel(uint24)` already exists at `contracts/libraries/PriceLookupLib.sol` and is a `pure` function (no SLOAD). Every other module already uses it. MintModule is the only holdout.

Replace all `price` storage reads in MintModule:
- L625 `_purchaseFor`: `PriceLookupLib.priceForLevel(purchaseLevel)` — price matches ticket target level
- L882 `_callTicketPurchase`: receives price as parameter (Phase 161 will handle this) OR computes `PriceLookupLib.priceForLevel(cachedLevel)` internally
- L1051 `_purchaseBurnieLootboxFor`: `PriceLookupLib.priceForLevel(level + 1)` — BURNIE lootbox always uses next-level price

Also replace all other `price` storage reads across the codebase:
- `DegenerusGame.sol:2155`
- `DegenerusGameJackpotModule.sol:2151, 2453`
- `DegenerusGameAdvanceModule.sol:740`
- `DegenerusGameLootboxModule.sol:620, 1118`

Each call site needs the correct target level passed to `priceForLevel`.

### D-04: Remove price storage variable and writes
After all reads are replaced:
- Remove `uint128 internal price = ...` from `DegenerusGameStorage.sol:337`
- Remove all `price = uint128(...)` writes from `DegenerusGameAdvanceModule.sol:1382-1408`
- Add `import` for `PriceLookupLib` in any module that doesn't already have it

### D-05: Quest price split — daily uses purchaseLevel, level quests use level+1
`handlePurchase` receives TWO prices:
- `mintPrice` = `PriceLookupLib.priceForLevel(purchaseLevel)` — used for daily quest targets
- `levelQuestPrice` = `PriceLookupLib.priceForLevel(level + 1)` — used for level quest targets

During normal purchase phase both are identical. During jackpot phase, daily quests use the current level's (lower) price while level quests maintain the next-level (higher) price target.

Requires adding `uint256 levelQuestPrice` parameter to `handlePurchase` in DegenerusQuests.sol and threading it through `_questHandleProgressSlot` → `_handleLevelQuestProgress` → `_levelQuestTargetValue`.

### D-06: Level can only change via advanceGame
`level` is only written in `AdvanceModule.sol:1373` during `advanceGame`. It never changes during a purchase transaction. This means within `_purchaseFor`, all `level` reads return the same value — they are safe to cache as locals for Phase 161 SLOAD dedup.

</decisions>
