### 3. Packed Struct Investigation (per D-03, D-04 -- addresses SCORE-01 packing analysis)

#### 3a. Score-Only Packing: Can deityPassCount be merged into mintPacked_?

Current mintPacked_ bit layout (from BitPackingLib):
- Bits 0-23: lastLevel (uint24)
- Bits 24-47: levelCount (uint24)
- Bits 48-71: levelStreak (uint24)
- Bits 72-103: lastMintDay (uint32)
- Bits 104-127: unitsLevel (uint24)
- Bits 128-151: frozenUntilLevel (uint24)
- Bits 152-153: whaleBundleType (uint2)
- Bits 228-243: levelUnits (uint16)
- Available: bits 154-227 = 74 unused bits

deityPassCount is uint16 (16 bits). It fits in the 74 unused bits of mintPacked_.

Savings: 1 mapping SLOAD (2,100 gas cold) per score computation eliminated.
Cost: Add BitPackingLib shift/mask constants, modify all deityPassCount read/write sites.
Recommendation: Include specific bit assignment (e.g., bits 154-169 for deityPassCount) and list every read/write site that must change.

#### 3b. Combined Score+Quest Packing

Quest streak lives in DegenerusQuests (separate contract, not delegatecall). Co-location options:
1. **Duplicate write** -- DegenerusQuests writes streak to its own storage AND writes a copy into Game storage. Cost: extra SSTORE (2,900-5,000 gas) on every quest completion. Benefit: eliminates 6,800-8,900 gas STATICCALL on score computation. Net: breakeven or worse.
2. **Parameter forwarding** -- handleMint/handleLootBox already return streak. Capture it and pass to score function. Cost: zero storage overhead. Benefit: eliminates the STATICCALL.

**Recommendation:** Parameter forwarding is strictly better for quest streak. Combined packing NOT recommended -- the cross-contract boundary makes co-location more expensive than forwarding. Score-only packing (deityPassCount into mintPacked_) IS recommended.

#### 5a. Quest Streak Elimination (RECOMMENDED)

Current: `questView.playerQuestStates(player)` STATICCALL at L2324. Cost: 6,800-8,900 gas.
Proposed: Capture `streak` return value from `handleMint`/`handleLootBox` (already returned, currently discarded). Forward as parameter to `_playerActivityScore(player, questStreak)`.

Changes required:
1. _questMint (MintModule:1114): Change `(uint256 reward, uint8 questType,, bool completed)` to capture streak: `(uint256 reward, uint8 questType, uint32 questStreak, bool completed)`
2. handleLootBox call (MintModule:810): Change `(uint256 lbReward,,, bool lbCompleted)` to capture streak
3. _playerActivityScore: Add `uint32 questStreak` parameter, remove `questView.playerQuestStates(player)` call, use parameter directly
4. External `playerActivityScore(address)` view: Keep as backward-compatible wrapper that internally calls `questView.playerQuestStates` and forwards streak

#### 5b. Affiliate Bonus Elimination (NOT RECOMMENDED -- per research finding)

Current: `affiliate.affiliateBonusPointsBest(currLevel, player)` STATICCALL. Cost: 5,200-13,100 gas.
Reason to keep: The affiliate address is warm on the purchase path (prior payAffiliate calls). The STATICCALL base cost is only 100 gas (warm). The internal 5 mapping SLOADs (~10,500 gas) are the real cost. Co-locating affiliate data would add SSTORE overhead on every payAffiliate call that exceeds the savings. Parameter forwarding from payAffiliate is problematic because payAffiliate writes to affiliateCoinEarned before score computation reads it -- ordering matters.

Decision: ACCEPT the affiliate STATICCALL. Document as conscious architectural decision. Potential future optimization if affiliate storage is restructured for other reasons.

### 7. DegeneretteModule Duplicate Elimination (per D-11 -- addresses SCORE-01 architecture)
