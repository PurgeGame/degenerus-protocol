# Plan 118-02: Integration Attack Analysis

Perform the Mad Genius integration attack analysis -- examining cross-contract composition risks that individual unit audits could not catch. This is NOT a function-by-function re-audit. It attacks interaction patterns across contract boundaries.

## Tasks

### Task 1: Integration Attack Analysis
type="auto"

**Action -- Analyze these 7 cross-contract attack surfaces:**

1. **Delegatecall Storage Coherence (All 10 Module Boundaries)**
   - For each module pair that shares a top-level transaction path, verify no module caches storage that another module writes
   - Special focus: advanceGame() -> rngGate() chain which touches JackpotModule, EndgameModule, DecimatorModule
   - Special focus: nested delegatecall (LootboxModule -> BoonModule, DegeneretteModule -> LootboxModule)

2. **ETH Conservation Analysis**
   - Trace every msg.value entry through the protocol
   - Verify every ETH exit (claimWinnings, vault burns, game-over drain)
   - Prove: total ETH in = total ETH distributed + total ETH held
   - Check: can any code path create or destroy ETH (e.g., rounding that favors extractors)?

3. **Token Supply Invariant Verification**
   - BURNIE: Sum all mint paths vs burn paths. Can supply grow without corresponding game action?
   - DGNRS: No runtime mint. Verify supply only decreases.
   - sDGNRS: pool accounting (Whale + Affiliate + Claims) + reserves = totalMoney. Verify across all state transitions.
   - WWXRP: Intentionally undercollateralized. Verify mintPrize authorization chain is complete.

4. **Cross-Contract Reentrancy via ETH Sends**
   - Every `.call{value}` send site is a potential re-entry. For each: what state has been written? What state could be read by a re-entrant call?
   - Focus: Game.claimWinnings sends ETH to player. Can player re-enter through another Game function?
   - Focus: sDGNRS.claimRedemption sends ETH. Can recipient call back into sDGNRS?
   - Focus: Vault.burnEth sends ETH. Can recipient call back into Vault?

5. **State Machine Consistency**
   - Can gameOver + VRF timeout leave the protocol in a state where advanceGame() always reverts?
   - Can rngLocked get permanently stuck (VRF never responds AND timeout doesn't fire)?
   - Can prizePoolFrozen persist across transactions?
   - Can jackpotPhaseFlag and currentDay become inconsistent?

6. **decBucketOffsetPacked Collision (Escalation from Unit 7)**
   - Trace the exact call chain at GAMEOVER level where both regular and terminal decimator fire
   - Verify: does the EndgameModule call runDecimatorJackpot BEFORE GameOverModule calls runTerminalDecimatorJackpot?
   - Quantify: at which levels does this collision actually occur?
   - Assess: what is the economic impact on affected claimants?

7. **Auto-Rebuy BAF Pattern (Cross-Module)**
   - VAULT can enable auto-rebuy (Unit 3 F-01 correction)
   - Trace the yield surplus path where vault auto-rebuy modifies futurePrizePool after snapshot
   - Verify the 8% buffer is sufficient to absorb the maximum possible staleness

**For each attack surface, provide verdict:** SAFE / VULNERABLE / INVESTIGATE with evidence.
