4. **Budget cap**: Verify `budget = totalFunds - claimablePool`. This ensures deity refunds cannot consume funds already owed to claimable winners. Verify:
   - No underflow if claimablePool > totalFunds (should use safe subtraction or check)
   - Budget correctly limits total refunds
   - Each iteration: `if (budget < refund) break;` or equivalent

5. **claimablePool mutation**: `claimablePool += totalRefunded` at line ~105. Verify:
   - totalRefunded <= budget <= totalFunds - claimablePool (before update)
   - After update: claimablePool <= totalFunds (invariant preserved)

6. **unchecked arithmetic** (Research Open Question 3): Lines ~92-96 use `unchecked { claimableWinnings[owner] += refund; totalRefunded += refund; budget -= refund; }`. Verify:
   - `claimableWinnings[owner]` cannot overflow uint256 (owner would need ~5.7e57 ETH already claimable)
   - `totalRefunded` cannot overflow (bounded by budget which is bounded by totalFunds)
   - `budget -= refund` cannot underflow (loop checks budget >= refund before subtraction)

7. **Refund claimability**: Refunds go to `claimableWinnings[owner]`. Verify these are claimable via the normal `claimWinnings` path at DegenerusGame:1415-1447. Is there any path where deity refunds become unclaimable?
