**System Card 1: PAY-01 Purchase-Phase Daily Jackpot**

Info table fields:
- Trigger: `Daily, during purchase phase when daysSince > 0 and lvl > 1`
- Source Pool: `futurePrizePool (1% daily drip via poolBps = 100)`
- Calculation: `ethDaySlice = (_getFuturePrizePool() * poolBps) / 10_000` then split `lootboxBudget = (ethPool * PURCHASE_REWARD_JACKPOT_LOOTBOX_BPS) / 10_000` where `PURCHASE_REWARD_JACKPOT_LOOTBOX_BPS = 7500`
- Recipients: `VRF-selected ticket holders across 4 trait buckets (Rare/Epic/Legendary/Unique)`
- Claim Mechanism: `claimWinnings() via claimableWinnings[player] mapping`
- Currency: `75% lootbox tickets, 25% ETH`

Formula block (exact from audit report):
```
// JackpotModule.sol:635-639
uint256 poolBps = 100; // 1% daily drip
ethDaySlice = (_getFuturePrizePool() * poolBps) / 10_000;
_setFuturePrizePool(_getFuturePrizePool() - ethDaySlice);

// JackpotModule.sol:644-651 — 75/25 split
lootboxBudget = (ethPool * PURCHASE_REWARD_JACKPOT_LOOTBOX_BPS) / 10_000;
// PURCHASE_REWARD_JACKPOT_LOOTBOX_BPS = 7500 (75%)
// Remaining 25% distributed as ETH to winners
```

File references: `JackpotModule.sol:619-673`, `PayoutUtils.sol:30-74`

SVG flow diagram (viewBox="0 0 800 350"):
- futurePrizePool (pool, left) --"1% drip"--> _distributeJackpotEth (contract, center)
- _distributeJackpotEth --"75%"--> Lootbox Tickets (recipient, top-right)
- _distributeJackpotEth --"25% ETH"--> _addClaimableEth (contract, bottom-center)

- Diamond --"Yes"--> Ticket Conversion (contract)
- Diamond --"No"--> claimablePool (pool, bottom-right)
- claimablePool ---> Players (recipient)

Edge cases:
- Empty pool: If `futurePrizePool == 0`, `ethDaySlice = 0`, no distribution occurs
- No winners at trait bucket: Unfilled slots refund to yield surplus (not returned to pool)
- Auto-rebuy dust: Unconverted remainder dropped; max loss = `ticketPrice - 1` wei per payout

**System Card 2: PAY-02 Jackpot-Phase 5-Day Draws**

Info table fields:
- Trigger: `During jackpot phase (5-day sequence), once per day`
- Source Pool: `currentPrizePool (6-14% BPS days 1-4, 100% day 5)`
- Calculation: `dailyBudget = (currentPrizePool * randomBps) / BPS_DENOMINATOR` where `randomBps` is random in `[600, 1400]` for days 1-4, and `10000` for day 5. Share split: `JACKPOT_SHARES_PACKED = 60/13/13/13` weighted (6000/1300/1300/1300/100 BPS)
- Recipients: `4 winners per draw via VRF, weighted by share class (1st: 60%, 2nd-4th: 13% each, remainder: 1%)`
- Claim Mechanism: `claimWinnings() via claimableWinnings[player] mapping`
- Currency: `ETH (with auto-rebuy option)`

Formula block (exact from audit report):
```
// JackpotModule.sol:336-613
// Days 1-4: random BPS in [600, 1400] range (6-14%)
// Day 5: 100% of currentPrizePool
dailyBudget = (currentPrizePool * randomBps) / BPS_DENOMINATOR;

// Share split: JACKPOT_SHARES_PACKED
// 1st place: 6000 BPS (60%)
// 2nd place: 1300 BPS (13%)
// 3rd place: 1300 BPS (13%)
// 4th place: 1300 BPS (13%)
// Remainder: 100 BPS (1%)

// Compressed mode (< threshold): single winner gets 100%
// Turbo mode: instant resolution without VRF wait
```

File references: `JackpotModule.sol:336-613`, `JackpotModule.sol:2723-2736`

SVG flow diagram (viewBox="0 0 800 400"):
- currentPrizePool (pool, left) --"6-14% / 100%"--> Daily Draw (contract, center)
- Daily Draw --"60%"--> Winner 1 (recipient, top-right)
- Daily Draw --"13% each"--> Winners 2-4 (recipient, middle-right)
- Daily Draw --"1%"--> Remainder (recipient, bottom-right)
- Each winner arrow leads through _addClaimableEth -> auto-rebuy decision -> claimablePool

Edge cases:
- Compressed mode: When prize pool is below threshold, single winner gets 100% (no 4-winner split)
- Turbo mode: Instant resolution without VRF wait for small pools
- Day 5 depletes 100% of currentPrizePool -- no carry to next jackpot phase
- randomBps range [600, 1400] ensures minimum 6% and maximum 14% per day for days 1-4

- Recipients: `Ticket holders (their tickets become redeemable against currentPrizePool)`
- Claim Mechanism: `Indirect -- ticket value realized through jackpot draw participation`
- Currency: `ETH (pool lifecycle management, not direct payout)`

Formula block (exact from audit report):

// JackpotModule.sol:1078-1118 — _budgetToTicketUnits
// 2x over-collateralization: each ticket backed by 2x pool ETH

// futurePrizePool -> nextPrizePool (at level N-1 completion)
// nextPrizePool -> currentPrizePool (at level N start)

File references: `JackpotModule.sol:1078-1118`, `JackpotModule.sol:886-924`
