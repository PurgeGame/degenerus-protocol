Append the INV-02 section to audit/v3.0-cross-cutting-invariants-pool.md.

**Invariant statement:** All pool additions and subtractions balance across all paths. No ETH is created or destroyed by pool accounting -- every increment has a corresponding source, every decrement has a corresponding destination.

**Pool transition chain to verify:**
```
ETH in (recordMint) -> futurePrizePool
                    -> nextPrizePool (via pool transition)
                    -> currentPrizePool (via day advance)
                    -> claimablePool (via distribution events)
                    -> ETH out (claimWinnings)
```

**Methodology:**
1. **Entry points:** Enumerate every function that adds to any pool variable. For each:
   - What is the source? (msg.value, balance transfer, internal reallocation)
   - Is the source properly accounted for?

2. **Transition points:** Enumerate every function that moves value between pool variables:
   - futurePrizePool -> nextPrizePool (when?)
   - nextPrizePool -> currentPrizePool (when?)
   - currentPrizePool -> claimablePool (distribution events)
   - baseFuturePool (snapshot) vs futurePoolLocal (running)

3. **Exit points:** Enumerate every function that subtracts from any pool variable:
   - Distribution to claimablePool (covered in INV-01)
   - Direct pool draws (BAF, decimator, yield distribution)
   - GAMEOVER zeroing

4. **Conservation proof per pool variable:**
   For each pool variable (futurePrizePool, nextPrizePool, currentPrizePool, claimablePool):
   - List ALL increment sites (file:line, amount, source)
   - List ALL decrement sites (file:line, amount, destination)
   - Prove: sum(increments from external ETH) = sum(decrements to external ETH)
   - Prove: internal transfers are zero-sum (one pool's loss = another's gain)

5. **Cross-path interactions:**
   - Can advanceGame pool transitions interleave with distribution events?
   - Can GAMEOVER pool zeroing race with normal distributions?
   - Document the locking/ordering that prevents double-counting

6. **baseFuturePool vs futurePoolLocal distinction:**
   - baseFuturePool is a snapshot taken at level start (used by BAF + x00 decimator)
   - futurePoolLocal is running total (used by normal decimator)
   - Verify neither can be drawn from twice for the same event

**Output format:**
```
### Pool: [variable name]
**Increment sites:**
| # | File:Line | Amount | Source |
**Decrement sites:**
| # | File:Line | Amount | Destination |
**Conservation proof:**
  [Algebraic proof that increments - decrements = 0 for external flows]
**Cross-path safety:**
  [Ordering/locking analysis]
**Verdict:** PASS | FINDING
```

Conclude with INV-02 overall verdict.
