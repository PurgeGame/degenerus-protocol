## Overview

v33.0 replaces the open `propose(address)` / approve-reject vote flow in `GNRUS.sol` with a vault-owner-curated allowlist that voters select from directly — eliminating the collusion path where any sDGNRS coalition could route the per-level GNRUS distribution to an address they control.

Five phases delivered the design end-to-end:

1. **Phase 254** demolished v32-shape governance and laid the new storage skeleton (`currentSlate[20]` + `pendingEdit` mapping + bitmap helpers + 5 view helpers + `setCharity(uint8, address)` admin entry point).
2. **Phase 255** rewrote `vote(uint8 slot)` (approve-only, full sDGNRS weight, 4-path revert order) and `pickCharity(uint24 level)` (atomic flush + strict-`>` winner loop 0..19 + 3 LevelSkipped paths + 2%-of-pool distribution).
3. **Phase 256** built the Hardhat test surface (49 governance it-blocks + integration conservation evidence + gas guardrail).
4. **Phase 257** published `audit/FINDINGS-v33.0.md` as the milestone-closure deliverable (9 sections, 8/8 §4 adversarial surfaces SAFE at the time of initial emission, zero F-33-NN findings).
5. **Phase 258** (post-closure patch, inserted after Phase 257 independent re-run surfaced an ordering bug): reordered `pickCharity` so queued-edit flush executes AFTER the distribution payout (FIX-01) + added `lastWinningRecipient` storage + `PreviousWinnerNotVotable()` revert in `vote()` (FIX-02). Re-audited at the patched HEAD; closure signal re-emitted superseding `MILESTONE_V33_AT_HEAD_dcb70941`.

The most significant event of the milestone was the Phase 257 independent adversarial re-run (executed by fresh-context `/contract-auditor` and `/zero-day-hunter` agents loaded with skill specs after the original Task 7 SPAWN_FAILED forced an executor-manual fallback). That re-run surfaced the queue-branch vote-redirect mechanism — a real ordering bug where queued edits applied during level L would take effect in time for level L's vote tallying. Phase 258 closed it structurally via FIX-01 (flush-after-payout reorder) and added FIX-02 as a defense-in-depth block on consecutive recipient capture.
