Create `audit/event-correctness-game.md` with partial report. Methodology per D-02: three passes per function.

**Step 1: Inventory event declarations.** Read DegenerusGameStorage.sol for all event declarations used by the game system. List every `event` declaration with its parameters and which are `indexed`.

**Step 2: Audit DegenerusGame.sol router.** For every `external`/`public` function that changes state:
- Does it emit an event (directly or via delegatecall to a module)?
- If it delegatecalls a module, trace into the module to verify the event is emitted there.
- For each emit: do the parameter values match the ACTUAL post-state? Watch for:
  - Stale locals: variable emitted before it was updated
  - Pre-update snapshots: old value emitted instead of new value
  - Incorrect parameter ordering

**Step 3: Audit smaller modules** (GameOverModule 235 lines, EndgameModule 566 lines, BoonModule 327 lines, PayoutUtils 92 lines, MintStreakUtils 62 lines, WhaleModule 822 lines). Same three-pass audit per external/public state-changing function.

**Step 4: Check indexer-critical transitions.** For the game system, these are:
- Level changes (player advances/regresses)
- Game over events
- Jackpot win notifications
- Whale purchase events
- Any state transition an off-chain indexer needs for game state reconstruction

For each indexer-critical event, verify: sufficient data for reconstruction, key fields indexed per D-04 (only on indexer-critical events, not cosmetic).

```markdown

```

**Important:**
- Per D-03: Default disposition is DOCUMENT, not fix. No contract code changes.
- Per D-04: Only evaluate indexed fields on indexer-critical events.
- Note delegatecall context: events emitted in modules execute in DegenerusGame's context. This is correct behavior.
- Flag any function that changes state but emits NO event, with assessment of whether it NEEDS one.
- For functions that are pure view/read-only, skip (they don't need events).
