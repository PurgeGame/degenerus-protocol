Perform a systematic ERC-20 compliance audit of all 4 token contracts. For each token, trace every ERC-20 function and evaluate against the EIP-20 specification.

**DGNRS (DegenerusStonk.sol) — Full ERC-20 audit (per D-02, D-04):**

Verify these ERC-20 required functions exist and behave correctly:
- `name()` — returns "Degenerus Stonk" (constant)
- `symbol()` — returns "DGNRS" (constant)
- `decimals()` — returns 18 (constant)
- `totalSupply` — public state variable, decremented on burn
- `balanceOf` — public mapping
- `transfer(to, amount)` — calls _transfer(msg.sender, to, amount), returns true
- `transferFrom(from, to, amount)` — checks/decrements allowance (max-uint skip), calls _transfer, returns true
- `approve(spender, amount)` — sets allowance, emits Approval, returns true
- `allowance` — public mapping

Deep edge cases to verify:
1. **Zero-amount transfer**: Does `transfer(to, 0)` succeed? Trace _transfer: `amount > bal` check — if bal >= 0 and amount == 0, passes. Yes, succeeds. EIP-20 compliant.
2. **Self-transfer**: `transfer(msg.sender, amount)` — _transfer subtracts then adds to same address via unchecked block. Verify no double-counting or loss. The unchecked subtraction followed by addition preserves balance. Compliant.
3. **Max-uint approval**: `approve(spender, type(uint256).max)` — sets allowance. In transferFrom, `allowed != type(uint256).max` skips deduction. Correct infinite approval pattern.
4. **Approve race condition**: EIP-20 known issue. DGNRS has no increaseAllowance/decreaseAllowance. This is standard behavior — document as known ERC-20 limitation, not a bug.
5. **EIP-2612 (permit)**: Not implemented. Document as intentional omission — not required by ERC-20.
6. **Transfer to zero address**: _transfer reverts with ZeroAddress(). EIP-20 says "SHOULD throw" — compliant.
7. **Transfer to contract address(this)**: _transfer reverts with Unauthorized() if `to == address(this)`. This is a DEVIATION — EIP-20 does not mandate this restriction. Document.
8. **Transfer event on zero-amount**: Verify Transfer event is emitted even for zero-amount transfers. Trace: _transfer always emits Transfer. Compliant.
9. **Burn rejects zero amount**: _burn reverts if `amount == 0`. This means `burn(0)` reverts — not an ERC-20 function but worth noting.
10. **No Approval event on transferFrom allowance change**: transferFrom does NOT emit Approval when decrementing allowance. This is a DEVIATION from EIP-20 which says transferFrom SHOULD emit Approval. Document.
11. **Reentrancy via receiver**: _transfer has no external calls — pure state manipulation + event. No reentrancy surface in ERC-20 functions.

**BURNIE (BurnieCoin.sol) — Full ERC-20 audit (per D-02, D-04):**

Verify ERC-20 required functions:
- `name()` — returns "Burnies" (constant)
- `symbol()` — returns "BURNIE" (constant)
- `decimals()` — returns 18 (constant)
- `totalSupply()` — view function reading _supply.totalSupply (packed struct). Excludes vault allowance.
- `balanceOf` — public mapping
- `transfer(to, amount)` — calls _claimCoinflipShortfall then _transfer, returns true
- `transferFrom(from, to, amount)` — game bypass + allowance check, claims shortfall, _transfer, returns true
- `approve(spender, amount)` — sets allowance (with optimization: skips write if unchanged), emits Approval, returns true
- `allowance` — public mapping

Deep edge cases:
1. **Zero-amount transfer**: transfer calls _claimCoinflipShortfall(msg.sender, 0) which returns immediately (amount == 0 check). Then _transfer: `balanceOf[from] -= 0` succeeds. Compliant.
2. **Self-transfer**: _transfer subtracts then adds. Same address: `balanceOf[from] -= amount; balanceOf[to] += amount` — if from == to, net zero. Compliant.
3. **Max-uint approval**: transferFrom checks `allowed != type(uint256).max && amount != 0` — infinite approval preserved. Compliant.
4. **Approve race condition**: Same as DGNRS — no increaseAllowance/decreaseAllowance. Known ERC-20 limitation.
5. **Transfer to zero address**: _transfer reverts with ZeroAddress() if from or to is address(0). Compliant.
6. **Game contract bypass**: transferFrom skips allowance check when caller is GAME contract. This is a DEVIATION — game contract has implicit infinite approval over ALL addresses. Document as intentional design (trusted contract pattern).
7. **transferFrom emits extra Approval event**: When allowance is decremented (non-game, non-max, non-zero), BurnieCoin emits `Approval(from, msg.sender, newAllowance)`. This is actually MORE compliant than DGNRS (EIP-20 says SHOULD).
8. **Coinflip auto-claim on transfer**: transfer/transferFrom call _claimCoinflipShortfall which may mint BURNIE to the sender before the transfer. This is a DEVIATION — ERC-20 transfers should not have side effects that mint tokens. Document as intentional design.
9. **Transfer to VAULT redirects to burn**: _transfer special-cases `to == ContractAddresses.VAULT` — burns tokens and adds to vault allowance instead of transferring. Emits Transfer(from, address(0)). This is a DEVIATION — sending to VAULT burns instead of transferring. Document.
10. **EIP-2612 (permit)**: Not implemented. Document as intentional omission.
11. **approve optimization**: BurnieCoin skips SSTORE if new value == current value. This is gas optimization, not a deviation. Approval event always emitted regardless.
12. **Reentrancy via _claimCoinflipShortfall**: transfer/transferFrom make external call to coinflip contract before state changes. Potential reentrancy vector. However the coinflip contract is trusted (compile-time constant). Document as accepted trust model.

Confirm soulbound is airtight: GNRUS can only be distributed by the game via pickCharity (governance) or created at deploy. It can only leave an address via burn(). No transfer path exists.
