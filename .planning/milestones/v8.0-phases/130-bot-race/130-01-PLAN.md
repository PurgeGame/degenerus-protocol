  <action>
Parse slither-raw.json and create a structured triage document. For EVERY finding:

1. Read the raw JSON finding (detector name, confidence, impact, description, source locations)
2. Read the actual contract code at the flagged location to understand context
3. Classify as one of:
   - **FIX** — Real issue that should be addressed in Phase 134
   - **DOCUMENT** — Real behavior but intentional/acceptable, add to known-issues triage
   - **FALSE-POSITIVE** — Detector is wrong about this code
4. Write reasoning for the classification (1-3 sentences explaining WHY)

Per D-05: Default disposition is DOCUMENT, not FIX. Do not touch contract code.
Per D-06: Batch all findings -- do not escalate individual findings.
Per D-07: Use judgment on grouping vs itemizing FPs (group by category if many similar FPs from same detector, itemize if unique contexts).

Expected FP patterns from CONTEXT.md code_context:
- Heavy inline assembly (Yul) will trigger false positives
- Custom storage packing may trigger "uninitialized variable" FPs
- Delegatecall pattern (DegenerusGame -> modules) will trigger reentrancy detectors
