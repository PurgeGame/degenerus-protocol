MED · CONFIRMED sDGNRS auto-rebuy carry excluded from the redemption BURNIE backing BURNIE-04

Convergent — gemini council PRIME-01 + the Claude net (the surface-map FA-1 prime lead); second-sourced by codex in the council-on-refuted re-run. Every sDGNRS BURNIE-base read funnels through previewClaimCoinflips(SDGNRS) = _viewClaimableCoin + claimableStored (BurnieCoinflip.sol:971-975), which provably omits autoRebuyCarry. Once sDGNRS arms perpetual 0-take-profit auto-rebuy (post day-20), winnings never route to claimableStored — they roll into the carry. So the redemption BURNIE base is structurally near-zero and the carry — the bulk of sDGNRS's BURNIE house value — is invisible to redeemBurnieShare with no sDGNRS-reachable liquidation path. → redeemers progressively under-credited.
Skeptic gate (396-SKEPTIC-GATE.md section 2):
CONSERVATIVE: base <= burnieBal + claimableBurnie  —  no over-credit, no insolvency
OFF the ETH / claimablePool spine (BURNIE-06 attested)  —  no ETH-solvency consequence
attacker profit: NONE (under-credit of redeemers, not an extraction)
verdict: CONFIRMED-MED — below HIGH (no pump/supply-break/insolvency), above INFO
USER ruling: REAL GAP (under-implementation of the intended design — redemption BURNIE value = OWNED + flip CARRY, carry slice flip-contingent on the next coinflip). Routed to a gated fix — NOT applied. A separate post-audit, USER-hand-reviewed, batched contract change (392-BURNIE-04-FIX-DESIGN.md). Adjudicated direction: widen the redemption base to include the carry via a new sDGNRS-only settle-then-read; remove the carry slice from the shared autoRebuyCarry at submit (the conservation fix); resolve the contingency on day D+1's coinflip (unknown at submit, not the grindable day-D result) keyed off an absolute coinflipDayWon(D+1) view; win-path = a pure deferred mint of the segregated slice; uint96-whole-token DayPending re-pack (raw-wei uint64 rejected — overflows ~10,842x at genesis).
