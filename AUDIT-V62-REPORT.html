Actionable findings — all reproduced on the real contracts

HIGH advanceGame gas brick: subscriber chunk + gap-backfill compose V62-02

Convergent — both council models independently. The new-day path runs the afking subscriber stage then falls through to rngGate → _backfillGapDays with no intervening stage-break (AdvanceModule:336→341). On a 121+ day VRF-stall recovery, a saturated all-evict chunk and the 120-day backfill run in one advanceGame tx. The v60 decouple (6d2c8d0c) only separates the backfill from the downstream jackpot — not the upstream subscriber stage.
test/repro/V62GasBrickCompose.t.sol — composed single tx (COLD):
subscribers_evicted_in_final_chunk: 497   gap_days_backfilled: 119
composed_advanceGame_tx_gas: 20,255,533 > cap 16,777,216  →  EIP-7825 BRICK
boundary control: full-budget chunk breaks & does NOT compose (proves worst case)
Candidate fix (gated, not applied): insert a stage-break between subscriber-stage completion and the gap-backfill — mirror the v60 decouple on the upstream side so the two heavy stages never share a tx.

HIGH sDGNRS redemption reentrancy → stETH double-counted as backing V62-03

Solvency spine. claimRedemption decrements pendingRedemptionEthValue and deletes the claim before payout; _payEth's mixed branch sends ETH via player.call before transferring the stETH remainder, with no reentrancy guard. In the ETH hook the attacker reentrantly calls burn(); the reentrant backing calc still counts the in-flight stETH (not yet sent, no longer reserved) as free backing → over-reserves a new redemption (+175%) → breaks SOLVENCY-01.
test/repro/V62RedemptionReentrancy.t.sol:
BEFORE: pending 17.50 ETH = ETH+stETH held 17.50  (identity holds)
reentrant burn() landed during ETH hook: 28×
AFTER: pending 0.791 ETH vs ETH+stETH held 0  →  unbacked deficit
Candidate fix (gated, not applied): transfer stETH before the untrusted ETH .call — exactly as DegenerusVault.burnEth already does (direct evidence this is an sDGNRS-local ordering bug) — or add a reentrancy guard.

MED–HIGH Permissionless lootbox auto-open is structurally dead V62-01

Surfaced by the FUZZ-06 council review. Boxes enqueue at boxPlayers[LR_INDEX], but VRF words land at LR_INDEX − 1 (the request pre-increments). _openHumanBoxes / boxesPending read the active LR_INDEX, so the just-finalized box is never auto-opened — human & presale boxes silently degrade to manual-only openLootBox. The afking-cover leg is immune (keys off rngWordByDay) — the differential tell. This re-opens the WHALE-01 anti-timing vector for the mainline box classes.
test/repro/C1BoxAutoOpen.t.sol:
word lands at N=2, LR_INDEX=3 → openBoxes(50) opened 0, base unchanged (×2 + further advance)
control: manual openLootBox(actor,2) → base zeroed (box was ready & owed)
Candidate fix (gated, not applied): point the openBoxes/boxesPending index reads at LR_INDEX − 1; include the presale-box leg.
