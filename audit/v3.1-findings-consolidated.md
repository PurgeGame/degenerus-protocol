## DegenerusAdmin.sol

**Source:** Phase 31 | **Lines reviewed:** 801 | **Findings:** 2 CMT, 2 DRIFT

### CMT-001: Stale threshold decay starting value in header block comment

- **What:** Header block comment states `Approval voting with decaying threshold (60% -> 5% over 7 days)` but the `threshold()` function returns 5000 (50%) as the starting value, not 6000 (60%). The 24h/6000 tier was explicitly removed in commit fd9dbad1.
- **Where:** DegenerusAdmin.sol:38
- **Why:** A warden reading the architecture header would form the incorrect assumption that the initial governance threshold is 60%. The actual initial threshold is 50% (line 539: `return 5000; // 50%`). This discrepancy between the header and code would likely prompt a warden to file a QA or informational finding about stale documentation.
- **Suggestion:** Change line 38 to: `*   - Approval voting with decaying threshold (50% -> 5% over 7 days)`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### CMT-002: Stale death clock pause reference in header block comment

- **What:** Header block comment states `Death clock pauses while any proposal is active` but no death clock pause logic exists in the contract. There is no `anyProposalActive()` function, no `activeProposalCount` variable, and no interaction with any game liveness timer. The death clock pause feature was entirely removed in commit 73c50cb3.
- **Where:** DegenerusAdmin.sol:41
- **Why:** A warden reading the header would believe that active governance proposals affect the game's liveness timer (death clock). This is entirely false in the current code. A warden could spend significant time looking for this interaction and, when not finding it, might file a finding about missing functionality or broken governance integration.
- **Suggestion:** Delete line 41 entirely. The remaining governance bullet points (lines 36-40) fully describe the current governance behavior.
- **Category:** comment-inaccuracy
- **Severity:** INFO

### DRIFT-001: Vestigial `jackpotPhase()` in IDegenerusGameAdmin interface

- **What:** The `IDegenerusGameAdmin` interface declares `jackpotPhase() external view returns (bool)` but this function is never called anywhere in DegenerusAdmin.sol. It was likely used by the removed death clock pause feature (which needed to know if the game was in jackpot phase to decide whether to pause the liveness timer).
- **Where:** DegenerusAdmin.sol:70
- **Why:** A warden auditing the interface might wonder why `jackpotPhase()` is declared but never used, potentially concluding that the contract has incomplete functionality or a missing code path. The interface comment at line 67 (`Game contract admin interface (VRF + liquidity + liveness)`) further suggests liveness interaction that no longer exists.
- **Suggestion:** Remove `jackpotPhase()` from the interface declaration. Optionally update the interface `@dev` comment on line 67 to remove "liveness" (e.g., `Game contract admin interface (VRF + liquidity management)`). The liveness concept was specific to the removed death clock pause.
- **Category:** intent-drift
- **Severity:** INFO

### DRIFT-002: propose() NatSpec omits 1-per-address active proposal limit

- **What:** The `propose()` function's `@dev` NatSpec (lines 393-395) documents the two proposal paths (Admin and Community) and their requirements, but does not mention the 1-per-address active proposal limit added in commit df1e9f78. The limit is only documented as an inline comment on line 408 (`// 1-per-address active proposal limit`). A user calling `propose()` while they already have an active proposal will receive an `AlreadyHasActiveProposal` revert with no NatSpec-level documentation explaining this behavior.
- **Where:** DegenerusAdmin.sol:392-398
- **Why:** NatSpec is the primary documentation surface for wardens and integrators. The 1-per-address limit is a significant behavioral restriction that could affect governance strategy (e.g., a large sDGNRS holder cannot propose multiple alternative coordinator swaps). Omitting it from NatSpec means wardens must read the implementation to discover it, reducing auditability.
- **Suggestion:** Add to the `@dev` block: `///      Each address may have at most one active proposal at a time (AlreadyHasActiveProposal).`
- **Category:** intent-drift
- **Severity:** LOW

### CMT-006: Jackpot section header omits compression tiers and oversimplifies "day 5"

- **What:** The jackpot payout block comment (lines 1794-1798) states `Daily jackpot - Paid each day to burn ticket holders (day 5 = full pool payout)`. This description assumes all jackpot phases are 5 physical days. However, jackpot compression tiers (added in commit b3da98e9) allow turbo mode (all 5 logical days in 1 physical day, `compressedJackpotFlag=2`) and compressed mode (5 logical days in 3 physical days, `compressedJackpotFlag=1`). The code at JackpotModule lines 349-361 implements counterStep=5 for turbo and counterStep=2 for compressed. A warden reading "day 5 = full pool payout" would assume jackpot phase always takes exactly 5 calendar days.
- **Where:** DegenerusGame.sol:1795
- **Why:** Jackpot compression directly affects time-sensitive mechanics: how long funds are locked, how many advanceGame() calls are needed, and when the next level starts. A warden analyzing timing attacks or fund lockup would use "5 days" as a constant, missing that turbo levels resolve in a single day. This was also flagged by Phase 29 (DOC-02) as stale after the compression tier addition.
- **Suggestion:** Update to: `Daily jackpot - Paid each day (5 logical days; compressed to 3 or 1 physical days based on compression tier)`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### CMT-007: Jackpot section header misrepresents decimator scope and percentage

- **What:** The jackpot payout block comment at line 1796 states `Decimator - Special 100-level milestone jackpot (30% of pool)`. This is incorrect in two ways: (1) Decimator jackpots fire at x5 levels (5, 15, 25...85) at 10% of future pool, AND at x00 levels at 30% of future pool (EndgameModule lines 193-202 and 208-209). The comment implies decimator only fires at 100-level milestones. (2) The 30% percentage only applies to x00 levels; x5 levels use 10%. Phase 29 also flagged this as stale ("incomplete decimator scope").
- **Where:** DegenerusGame.sol:1796
- **Why:** A warden analyzing fund flow would believe decimator only distributes 30% of the future pool every 100 levels. They would miss the x5-level distributions entirely (10% of future pool every 10 levels except x95), which represent the majority of decimator events. This significantly understates the frequency of future pool drawdowns and overstates the amount held in reserve between x00 milestones.
- **Suggestion:** Update to: `Decimator - Every x5 level (10% of future pool) + x00 milestone (30% of future pool)`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### CMT-008: Jackpot section header misrepresents BAF scope

This severely understates the frequency of future pool drawdowns from BAF and could lead to incorrect conclusions about fund sustainability.
- **Suggestion:** Update to: `BAF - Big-ass-flip jackpot (every 10 levels: 10% of future pool; 20% at x00 and L=50)`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### CMT-009: futurePrizePoolTotalView() NatSpec says "aggregate" but returns same value as futurePrizePoolView()
