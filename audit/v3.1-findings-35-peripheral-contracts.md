## DegenerusQuests.sol

**Review scope:** 1,598 lines, 249 NatSpec tags, ~558 comment lines, 35 functions.
**Post-Phase-29 changes:** None -- contract unchanged since Phase 29 (commit 9238faf2).

### CMT-059: Contract header says "called by the Degenerus ContractAddresses.COIN contract" but COINFLIP is also an authorized caller

- **What:** The @dev Architecture Overview (line 16) states the contract is "called by the Degenerus ContractAddresses.COIN contract." The `onlyCoin` modifier (lines 284-288) allows both `ContractAddresses.COIN` and `ContractAddresses.COINFLIP`. BurnieCoinflip.sol is an active caller of handler functions.
- **Where:** `DegenerusQuests.sol:16`
- **Why:** A warden reading the header would assume only BurnieCoin calls this contract. The omission of COINFLIP as a caller is misleading for access-control review.
- **Suggestion:** Change to "called by the Degenerus ContractAddresses.COIN and ContractAddresses.COINFLIP contracts."
- **Category:** CMT (comment inaccuracy)
- **Severity:** INFO

### CMT-060: Security Model references nonexistent `onlyCoinOrGame` modifier

- **What:** The Security Model block (line 24) states "Quest normalization allows ContractAddresses.COIN OR game to trigger via `onlyCoinOrGame`." No `onlyCoinOrGame` modifier exists in the contract. The only modifiers are `onlyCoin` (lines 284-288) and `onlyGame` (lines 290-293). No function uses a combined coin-or-game guard.
- **Where:** `DegenerusQuests.sol:24`
- **Why:** A warden searching for the `onlyCoinOrGame` modifier to understand access control would find nothing. The concept of "quest normalization" is also absent from the contract -- there is no `normalizeQuestProgress` function or equivalent. The entire NatSpec bullet point describes non-existent functionality.
- **Suggestion:** Remove the bullet point or replace with an accurate description of actual access control. The `onlyCoin` modifier gates all handler functions; the `onlyGame` modifier gates only `awardQuestStreakBonus`.
- **Category:** CMT (comment inaccuracy)
- **Severity:** LOW -- describes absent functionality, could mislead access control review

### CMT-061: Security Model says "COIN-gated via `onlyCoin` modifier" without clarifying COINFLIP is also allowed

- **What:** Line 23 states "All player-action handlers are ContractAddresses.COIN-gated via `onlyCoin` modifier." While the `onlyCoin` modifier name suggests COIN-only access, the modifier implementation (line 286) checks `sender != ContractAddresses.COIN && sender != ContractAddresses.COINFLIP`, allowing both contracts. Saying "COIN-gated" is technically incomplete.
- **Where:** `DegenerusQuests.sol:23`
- **Why:** Same pattern as CMT-059. A warden performing access-control analysis would miss that COINFLIP can also call handlers.
- **Suggestion:** Change to "All player-action handlers are gated via `onlyCoin` modifier (accepts COIN or COINFLIP)."
- **Category:** CMT (comment inaccuracy)
- **Severity:** INFO -- the modifier NatSpec at line 283 correctly says "COIN or COINFLIP"

### CMT-062: rollDailyQuest @dev says "Slot 0 uses `entropy` directly for type selection" but Slot 0 type is fixed

- **What:** The @dev block for `rollDailyQuest` (line 303) states "Slot 0 uses `entropy` directly for type selection." In reality, Slot 0 is hardcoded to `QUEST_TYPE_MINT_ETH` (line 376: `uint8 primaryType = QUEST_TYPE_MINT_ETH`). Entropy is not used for Slot 0 type selection at all. The `_seedQuestType` call at line 383 uses the fixed `primaryType`, not any entropy-derived value.
- **Where:** `DegenerusQuests.sol:303`
- **Why:** A warden analyzing quest randomness would incorrectly believe Slot 0 has entropy-dependent type selection, potentially investigating a non-existent attack surface around VRF manipulation.
- **Suggestion:** Replace "Slot 0 uses `entropy` directly for type selection" with "Slot 0 is fixed to QUEST_TYPE_MINT_ETH (no entropy needed)" or simply remove the Entropy Usage sub-block for Slot 0.
- **Category:** CMT (comment inaccuracy)
- **Severity:** LOW -- misleads randomness analysis

### CMT-063: `onlyCoin` modifier name suggests COIN-only access but permits COINFLIP

- **What:** The modifier is named `onlyCoin` (line 284) but the implementation (line 286) checks both `ContractAddresses.COIN` and `ContractAddresses.COINFLIP`. The OnlyCoin error name (line 55) also implies COIN-only. The error's @notice (line 54) correctly says "COIN or COINFLIP" and the modifier's @dev (line 283) correctly says "COIN or COINFLIP contract." The naming mismatch is between the identifier (`onlyCoin`) and its actual semantics (COIN or COINFLIP).
- **Where:** `DegenerusQuests.sol:284` (modifier name), `DegenerusQuests.sol:55` (error name)
- **Why:** Identifier names are the first thing a warden reads. The NatSpec correctly documents dual access, but a warden scanning function signatures would see `onlyCoin` and assume only COIN is allowed until they read the modifier body. This is a naming issue that makes the NatSpec feel inconsistent with the code surface.
- **Suggestion:** Add an inline comment at the modifier declaration: `// Accepts both COIN and COINFLIP despite the name` or rename to `onlyCoinOrCoinflip`. The existing NatSpec is correct, so this is a low-priority naming clarity issue.
- **Category:** CMT (comment/naming inconsistency)
- **Severity:** INFO -- NatSpec is correct; only the identifier is misleading

### CMT-064: `_questComplete` NatSpec says "credits streak when all slots finish" but streak credits on first slot completion

- **What:** The @dev for `_questComplete` (line 1370) states "Completes a quest slot, credits streak when all slots finish, and returns rewards." However, the code (lines 1415-1422) credits streak on the first quest completion of the day (using the `QUEST_STATE_STREAK_CREDITED` flag), not when all slots are complete. The @dev Streak Logic sub-block (line 1373) correctly says "Streak increments on the first quest completion of the day" -- contradicting the function's own summary line.
- **Where:** `DegenerusQuests.sol:1370`
- **Why:** The summary line directly contradicts the detailed documentation below it. A warden reading only the summary line would misunderstand streak crediting logic.
- **Suggestion:** Change "credits streak when all slots finish" to "credits streak on first slot completion of the day."
- **Category:** CMT (comment inaccuracy)
- **Severity:** INFO -- the detailed @dev sub-block is correct

### DRIFT-004: QUEST_TYPE_RESERVED = 4 is vestigial from a removed quest type but serves as a defensive skip guard

- **What:** The constant `QUEST_TYPE_RESERVED = 4` (line 153) is declared with @dev "Retired quest type id kept reserved for compatibility." It is referenced in exactly one location: `_bonusQuestType` at line 1309, where it is explicitly skipped during weighted random quest selection (preventing quest type 4 from being rolled). The constant is never matched in any handler function and no progress can ever be recorded against type 4.
- **Where:** `DegenerusQuests.sol:153` (declaration), `DegenerusQuests.sol:1309` (skip guard)
- **Why:** The @dev says "kept reserved for compatibility" but does not explain what quest type 4 originally was or why it was retired. A warden would need to understand that the skip at line 1309 is intentional defensive code (preventing ID 4 from entering the quest pool) rather than dead code. The constant is not purely vestigial -- it actively prevents the retired type from being rolled. However, the @dev could better explain the defensive purpose.
- **Suggestion:** Expand the @dev to: "Retired quest type id (originally [purpose]). Kept reserved so `_bonusQuestType` skips it, preventing ID 4 from re-entering the quest pool. Removal would break the QUEST_TYPE_COUNT enumeration gap."
- **Category:** DRIFT (intent drift -- vestigial constant with minimal but real defensive function)
- **Severity:** INFO -- the constant serves a purpose but its provenance is undocumented

### CMT-070: @param lootboxActivityScore documents wrong taper start and wrong floor percentage

- **What:** The `payAffiliate` function's `@param lootboxActivityScore` at line 383 says `"0 = no taper; 15000+ triggers linear taper to 50% floor at 25500"`. The code uses `LOOTBOX_TAPER_START_SCORE = 10_000` (taper starts at 10,000, not 15,000) and `LOOTBOX_TAPER_MIN_BPS = 2_500` (25% payout floor, not 50%). Both values are wrong.
- **Where:** `DegenerusAffiliate.sol:383`
- **Why:** A warden reading the @param would believe taper begins at activity score 15,000 with a 50% floor, when the actual implementation starts tapering at 10,000 with a 25% floor. The block comment in the same function (lines 372-376) correctly documents these values, creating an internal contradiction within the same NatSpec block. A warden cross-referencing the REWARD FLOW block comment against the @param would flag the discrepancy themselves.
- **Suggestion:** Replace `"15000+ triggers linear taper to 50% floor at 25500"` with `"10000+ triggers linear taper to 25% floor at 25500"` to match the code and the block comment above.
- **Category:** CMT (comment inaccuracy -- two numeric values wrong in @param)
- **Severity:** LOW -- the @param contradicts both the code and its own function's block comment; a warden reviewing taper behavior would discover the mismatch.

### CMT-071: Inline comment says "batch for gas efficiency" but mechanism is weighted random winner selection

- **What:** The inline comment at line 546 says `"Batch recipients for gas efficiency."` in the DISTRIBUTION section. The actual mechanism (lines 597-618) does not batch-pay all recipients -- when multiple recipients exist, it rolls a single weighted random winner who receives the combined total. The gas efficiency comes from the single transfer instead of 2-3, but the comment "batch recipients" implies a batch distribution.
- **Where:** `DegenerusAffiliate.sol:546`
- **Why:** A warden reading "Batch recipients for gas efficiency" would expect all 3 potential recipients (direct affiliate, upline1, upline2) to receive their amounts in a single multicall. The actual behavior is winner-takes-all per transaction (with EV preserved across many transactions). The later inline comment at lines 593-596 correctly describes the weighted winner mechanism, but the introductory comment at line 546 sets a misleading expectation.
- **Suggestion:** Replace `"Batch recipients for gas efficiency."` with `"Collect recipients for weighted winner selection."` or `"Build recipient array for single-winner payout (see distribution below)."` to set correct expectations.
- **Category:** CMT (comment inaccuracy -- describes wrong distribution model)
- **Severity:** INFO -- the full weighted winner mechanism is correctly documented at lines 593-596; the issue is the introductory summary at line 546.
