## DegenerusQuests.sol

**Review scope:** 1,598 lines, 249 NatSpec tags, ~558 comment lines, 35 functions.
**Post-Phase-29 changes:** None -- contract unchanged since Phase 29 (commit 9238faf2).

### CMT-059: Contract header says "called by the Degenerus ContractAddresses.COIN contract" but COINFLIP is also an authorized caller

- **What:** The @dev Architecture Overview (line 16) states the contract is "called by the Degenerus ContractAddresses.COIN contract." The `onlyCoin` modifier (lines 284-288) allows both `ContractAddresses.COIN` and `ContractAddresses.COINFLIP`. BurnieCoinflip.sol is an active caller of handler functions.
- **Where:** `DegenerusQuests.sol:16`
- **Why:** A warden reading the header would assume only BurnieCoin calls this contract. The omission of COINFLIP as a caller is misleading for access-control review.
- **Suggestion:** Change to "called by the Degenerus ContractAddresses.COIN and ContractAddresses.COINFLIP contracts."
- **Category:** CMT (comment inaccuracy)
- **Severity:** INFO

### CMT-060: Security Model references nonexistent `onlyCoinOrGame` modifier

- **What:** The Security Model block (line 24) states "Quest normalization allows ContractAddresses.COIN OR game to trigger via `onlyCoinOrGame`." No `onlyCoinOrGame` modifier exists in the contract. The only modifiers are `onlyCoin` (lines 284-288) and `onlyGame` (lines 290-293). No function uses a combined coin-or-game guard.
- **Where:** `DegenerusQuests.sol:24`
- **Why:** A warden searching for the `onlyCoinOrGame` modifier to understand access control would find nothing. The concept of "quest normalization" is also absent from the contract -- there is no `normalizeQuestProgress` function or equivalent. The entire NatSpec bullet point describes non-existent functionality.
- **Suggestion:** Remove the bullet point or replace with an accurate description of actual access control. The `onlyCoin` modifier gates all handler functions; the `onlyGame` modifier gates only `awardQuestStreakBonus`.
- **Category:** CMT (comment inaccuracy)
- **Severity:** LOW -- describes absent functionality, could mislead access control review

### CMT-061: Security Model says "COIN-gated via `onlyCoin` modifier" without clarifying COINFLIP is also allowed

### CMT-063: `onlyCoin` modifier name suggests COIN-only access but permits COINFLIP

### DRIFT-004: QUEST_TYPE_RESERVED = 4 is vestigial from a removed quest type but serves as a defensive skip guard

- **What:** The constant `QUEST_TYPE_RESERVED = 4` (line 153) is declared with @dev "Retired quest type id kept reserved for compatibility." - **Where:** `DegenerusQuests.sol:153` (declaration), `DegenerusQuests.sol:1309` (skip guard)

A warden would need to understand that the skip at line 1309 is intentional defensive code (preventing ID 4 from entering the quest pool) rather than dead code. The constant is not purely vestigial -- it actively prevents the retired type from being rolled. However, the @dev could better explain the defensive purpose.

- **Category:** DRIFT (intent drift -- vestigial constant with minimal but real defensive function)
- **Severity:** INFO -- the constant serves a purpose but its provenance is undocumented

### CMT-071: Inline comment says "batch for gas efficiency" but mechanism is weighted random winner selection

- **What:** The inline comment at line 546 says `"Batch recipients for gas efficiency."` in the DISTRIBUTION section. The actual mechanism (lines 597-618) does not batch-pay all recipients -- when multiple recipients exist, it rolls a single weighted random winner who receives the combined total. The gas efficiency comes from the single transfer instead of 2-3, but the comment "batch recipients" implies a batch distribution.
- **Where:** `DegenerusAffiliate.sol:546`
- **Why:** A warden reading "Batch recipients for gas efficiency" would expect all 3 potential recipients (direct affiliate, upline1, upline2) to receive their amounts in a single multicall. - **Suggestion:** Replace `"Batch recipients for gas efficiency."` with `"Collect recipients for weighted winner selection."` or `"Build recipient array for single-winner payout (see distribution below)."` to set correct expectations.
- **Category:** CMT (comment inaccuracy -- describes wrong distribution model)
- **Severity:** INFO -- the full weighted winner mechanism is correctly documented at lines 593-596; the issue is the introductory summary at line 546.
