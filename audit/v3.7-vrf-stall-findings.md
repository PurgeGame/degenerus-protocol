## Gap Backfill Entropy Derivation (STALL-01)

### Objective

Verify that `_backfillGapDays` produces unique, nonzero, VRF-derived words for each gap day via `keccak256(abi.encodePacked(vrfWord, gapDay))`.

### Source Code (AdvanceModule lines 1436-1450)

```solidity
function _backfillGapDays(
    uint256 vrfWord,
    uint48 startDay,
    uint48 endDay,
    bool bonusFlip
) private {
    for (uint48 gapDay = startDay; gapDay < endDay;) {
        uint256 derivedWord = uint256(keccak256(abi.encodePacked(vrfWord, gapDay)));
        if (derivedWord == 0) derivedWord = 1;
        rngWordByDay[gapDay] = derivedWord;
        coinflip.processCoinflipPayouts(bonusFlip, derivedWord, gapDay);
        emit DailyRngApplied(gapDay, derivedWord, 0, derivedWord);
        unchecked { ++gapDay; }
    }
}
```

### Preimage Uniqueness Analysis

- **Input:** `keccak256(abi.encodePacked(uint256 vrfWord, uint48 gapDay))` produces 38 bytes (32 + 6)
- **Fixed vrfWord, distinct gapDay:** Each distinct `gapDay` value produces a distinct 38-byte preimage, therefore a distinct derived word (keccak256 collision resistance, birthday bound 2^128)
- **Distinct vrfWord values (different stall events):** Different `vrfWord` inputs produce different preimages regardless of `gapDay`
- **No ambiguity:** `abi.encodePacked(uint256, uint48)` is unambiguous because the first element is fixed-width (32 bytes)
- **Zero guard:** `if (derivedWord == 0) derivedWord = 1` prevents EntropyLib xorshift fixed point

### Critical Property: Zero Nudges for Gap Days

Gap days receive zero nudges (`totalFlipReversals` is not consumed). NatSpec at line 1428 documents this. Players who purchased nudges before the stall get their nudges applied to the current (post-gap) day only.

### Test Evidence

- `test_gapBackfillEntropyUnique_fuzz(uint256)` -- Fuzz: 5-day gap produces 5 unique nonzero derived words (1000 runs)
- `test_gapBackfillZeroGuard()` -- Confirms derived words are nonzero for all gap days
- `test_gapBackfillSingleDayGap()` -- Boundary: 1-day gap produces exactly 1 derived word

**Verdict: STALL-01 VERIFIED** -- `keccak256(vrfWord, gapDay)` produces unique nonzero per-day words. Zero guard prevents EntropyLib fixed point. Preimage uniqueness holds for fixed and varying VRF words.

## Manipulation Window Analysis (STALL-02)

### Objective

Analyze the time window between VRF callback storing `rngWordCurrent` and `advanceGame` consuming it for gap backfill, and assess whether gap backfill introduces additional attack surface.

### Window Sequence

1. Coordinator swap (`updateVrfCoordinatorAndSub`) clears all VRF state
2. `advanceGame` calls `rngGate`, enters "Need fresh RNG" at line 841
3. `_requestRng` fires, sets `rngLockedFlag = true`, `lootboxRngIndex++`
4. VRF callback arrives (separate transaction), stores word to `rngWordCurrent`
5. `advanceGame` calls `rngGate`, sees `currentWord != 0 && rngRequestTime != 0`
6. Gap detection: `day > idx + 1` triggers `_backfillGapDays(currentWord, ...)`
7. Same call: `_backfillOrphanedLootboxIndices(currentWord)` fills orphaned lootbox indices

### Analysis

Between step 4 (VRF callback) and step 5 (advanceGame consumption), the VRF word is visible on-chain via `rngWordCurrent`. This is the **standard VRF callback-to-consumption window** that exists for ALL daily VRF usage, not specific to gap backfill.

**Coinflip positions pre-committed:** Players placed coinflip positions BEFORE the stall. They cannot add new coinflip bets for past gap days. An attacker who sees the VRF word could compute all gap day derived words and predict coinflip outcomes, but cannot change their positions.

**Lootbox purchases possible but safe during stall:** `purchase`/`purchaseBurnieLootbox` have no `rngLockedFlag` gate, so players can buy lootboxes during a stall. However, `lootboxRngIndex` was already incremented at the pre-stall RNG request — stall-period purchases target index N+1 while the pending VRF word resolves index N via `_finalizeLootboxRng`. Stall-period purchases are resolved by a future unknown VRF word.

### Test Evidence

- `test_manipulationWindowIdenticalToDaily()` -- Proves gap backfill uses the same `rngWordCurrent` as normal daily processing
- `test_gapDayPositionsPreCommitted()` -- Verifies coinflip positions exist before stall and cannot be modified post-VRF-callback

**Verdict: STALL-02 VERIFIED** -- Manipulation window is standard VRF (INFO). No new surface introduced by gap backfill.

## Zero-Seed Edge Case (STALL-05)

### Objective

Prove that `lastLootboxRngWord == 0` at coordinator swap time cannot produce degenerate entropy in `processTicketBatch`.

### Consumption Trace

**`processTicketBatch` (JackpotModule line 1916):** `uint256 entropy = lastLootboxRngWord;`
- Used as entropy seed for ticket trait assignment
- If 0: `EntropyLib.entropyStep(0) == 0` -- all subsequent steps return 0, producing deterministic (exploitable) trait assignments

### When Can lastLootboxRngWord == 0?

1. **At game start:** Before any VRF fulfillment. Initialized to 0 by default.
2. **After coordinator swap:** The swap does NOT reset `lastLootboxRngWord`. It retains its pre-stall value.

**Post-swap sequence analysis:**
1. Swap resets VRF state but preserves `lastLootboxRngWord`
2. `advanceGame` fires `_requestRng` (new request to new coordinator)
3. VRF callback stores word to `rngWordCurrent`
4. `advanceGame` calls `rngGate` -> `_applyDailyRng` -> `_finalizeLootboxRng(currentWord)` which sets `lastLootboxRngWord = currentWord` (line 849)
5. Any ticket processing AFTER this point uses the new (nonzero) `lastLootboxRngWord`

**Can processTicketBatch run before step 4?**

Ticket processing requires `advanceGame` to enter the mid-day path (line 162: `day == dailyIdx`). After a coordinator swap, `dailyIdx` is the last completed day but `day` (wall-clock) has advanced. So `day > dailyIdx` and the mid-day path is NOT entered. Instead, the gap backfill and daily processing path runs first, which calls `_finalizeLootboxRng` setting `lastLootboxRngWord` to a nonzero value.

**Edge case (same wall-clock day):** If the swap happens within the same day as the last completed day, `day == dailyIdx` and the mid-day path COULD run. But: (a) `midDayTicketRngPending` was cleared by the swap, so line 166's check fails, and (b) `rngWordByDay[day] != 0` (already completed), so `rngGate` returns early at line 776.

**Game start edge case:** At level 0, before any VRF fulfillment, `lastLootboxRngWord == 0`. But tickets cannot exist at level 0 -- ticket purchases require level > 0, which requires at least one completed day. By the first day completion, `_finalizeLootboxRng` sets `lastLootboxRngWord` to a nonzero value.

### Test Evidence

- `test_zeroSeedUnreachableAfterSwap()` -- Proves `lastLootboxRngWord` is nonzero after post-swap advanceGame completes
- `test_zeroSeedAtGameStart()` -- Proves `lastLootboxRngWord` is set to nonzero by first day completion, before any ticket processing

**Verdict: STALL-05 VERIFIED** -- Zero-seed window is unreachable. Post-swap `_finalizeLootboxRng` sets `lastLootboxRngWord` before any ticket processing can run. At game start, no tickets exist before the first day completion.

## Gameover Fallback Entropy (STALL-06)

### Objective

Assess `_getHistoricalRngFallback` entropy quality and verify `_tryRequestRng` guard branches.

### Three Gameover Entropy Paths

**`_gameOverEntropy` (AdvanceModule lines 858-931):**

| Path | Condition | Source | Proven In |
|------|-----------|--------|-----------|
| 1. Normal VRF | `rngWordCurrent != 0 && rngRequestTime != 0` | `_applyDailyRng` + `_finalizeLootboxRng` | Phase 63-64 (VRFC-01, LBOX-02) |
| 2. Historical fallback | `elapsed >= GAMEOVER_RNG_FALLBACK_DELAY` (3 days) | `_getHistoricalRngFallback(day)` | Phase 65 (this document) |
| 3. VRF request attempt | Otherwise | `_tryRequestRng` (try/catch) | Phase 65 |

### _getHistoricalRngFallback Analysis (lines 944-963)

```solidity
function _getHistoricalRngFallback(uint48 currentDay) private view returns (uint256 word) {
    uint256 found;
    uint256 combined;
    uint48 searchLimit = currentDay > 30 ? 30 : currentDay;
    for (uint48 searchDay = 1; searchDay < searchLimit; ) {
        uint256 w = rngWordByDay[searchDay];
        if (w != 0) {
            combined = uint256(keccak256(abi.encodePacked(combined, w)));
            unchecked { ++found; }
            if (found == 5) break;
        }
        unchecked { ++searchDay; }
    }
    return uint256(keccak256(abi.encodePacked(combined, currentDay, block.prevrandao)));
}
```

**Entropy quality assessment:**

- **Historical VRF words:** Committed Chainlink VRF outputs, immutable on-chain. Cannot be manipulated retroactively. If the game has 5+ completed days, `combined` incorporates 5 VRF words (1280 bits of committed entropy).
- **`currentDay`:** Deterministic, not manipulable.
- **`block.prevrandao`:** On Base L2, the sequencer controls `prevrandao` entirely. A validator/sequencer can choose between two prevrandao values, giving them a 2x advantage on a binary outcome. They cannot deterministically control the keccak256 output.

### Finding: V37-006 (Prevrandao 1-Bit Bias)

**Trigger condition:** VRF must be dead for 3+ days AFTER gameover is reached. This is an edge-of-edge case.

**Impact if manipulated:** The fallback word determines (a) coinflip outcome for the gameover day, (b) lootbox RNG word for the gameover day. On Base L2, the sequencer could bias binary outcomes by 2x.

**Mitigation:** 5 committed VRF words provide overwhelming entropy. The sequencer would need to precompute the keccak256 output for both prevrandao choices and pick the favorable one -- this gives a 2x advantage on a binary outcome, not deterministic control.

See Master Findings Table for V37-006.

### Finding: V37-007 (Level-0 Prevrandao-Only Path)

If NO historical VRF words exist (game at level 0, zero completed days), `combined == 0` and the output is `keccak256(abi.encodePacked(0, currentDay, block.prevrandao))`. This is prevrandao-only entropy.

**Impact:** At level 0, there are no player positions to manipulate -- no coinflips placed, no lootboxes purchased. The NatSpec at lines 939-941 documents this case explicitly.

See Master Findings Table for V37-007.

### Test Evidence

- `test_tryRequestRngGuardBranches()` -- Verifies `_tryRequestRng` returns false (no revert) when VRF is unconfigured
- `test_historicalRngFallbackNonzero()` -- Proves `_getHistoricalRngFallback` produces nonzero word with 5 historical VRF words

**Verdict: STALL-06 VERIFIED** -- Three gameover entropy paths audited. Prevrandao 1-bit bias documented as INFO (V37-006, V37-007). NatSpec accurately documents trade-offs.

**sDGNRS (StakedDegenerusStonk):**

### resolveRedemptionPeriod Timing Analysis

`flipDay = day + 1` where `day` is the dailyIdx-aligned game day (AdvanceModule lines 818, 881, 910). This is correct because:
- Coinflip results are stored by game day (`coinflipDayResult[day]`)
- The redemption's `flipDay` references the NEXT game day's coinflip result
- During a stall, gap days do NOT call `resolveRedemptionPeriod` (NatSpec lines 1429-1431)
- Resolution occurs only when `advanceGame` processes the current day after backfill

### Gambling Burn Period Tracking

Players submit gambling burns via `_submitGamblingClaimFrom` (sDGNRS line 710): `uint48 currentPeriod = game.currentDayView()`. This uses wall-clock time, meaning players CAN submit burns during a VRF stall. Resolution is deferred until `advanceGame` processes the current day.

This is correct behavior: the redemption period is indexed by wall-clock time (when submitted), but resolution is gated by game progression (when `advanceGame` runs). The delay prevents gaming the resolution timing.

- `test_wallClockDayAdvancesDuringStall()` -- Proves `_simulatedDayIndex()` advances while `dailyIdx` stays frozen

**Verdict: STALL-07 VERIFIED** -- All `block.timestamp` usages are correct for their context. No `block.timestamp` is used where `dailyIdx` is expected. Redemption timing is consistent with documented design.
