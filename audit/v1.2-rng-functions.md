## Section 3: Guard Condition Analysis (RFN-04)

### 3a. `rngLockedFlag` Guard

The `rngLockedFlag` prevents state mutations during the window between VRF request and daily jackpot resolution. It is set `true` in `_finalizeRngRequest` (AdvanceModule:1208) and cleared in `_unlockRng` (AdvanceModule:1287) or `updateVrfCoordinatorAndSub` (AdvanceModule:1275).

**State machine (updated from v1.0 audit):**

```
UNLOCKED (rngLockedFlag=false)
  |
  +-- advanceGame() -> rngGate() -> _requestRng() -> _finalizeRngRequest()
  |   SET true at AdvanceModule:1208
  v
LOCKED (rngLockedFlag=true, rngRequestTime!=0)
  |
  +-- VRF callback: rawFulfillRandomWords() stores rngWordCurrent
  |   Flag stays true (word awaits advanceGame processing)
  |
  +-- advanceGame() -> rngGate() -> _applyDailyRng() -> ... -> _unlockRng()
  |   CLEAR false at AdvanceModule:1287
  |
  +-- Emergency: updateVrfCoordinatorAndSub() (admin, 3-day gap)
      CLEAR false at AdvanceModule:1275
```

**All rngLockedFlag check sites:**

### 3b. `prizePoolFrozen` Guard

The `prizePoolFrozen` flag protects prize pool integrity during jackpot resolution. It is set `true` in `_swapAndFreeze` (Storage:744-745) and cleared in `_unfreezePool` (Storage:758).

**Freeze lifecycle:**
1. **Set true:** `_swapAndFreeze()` called from `advanceGame()` when daily RNG is requested (AdvanceModule, via rngGate returning 1)
2. **Stays true:** Through entire jackpot phase (days 1-4 of jackpot resolution)
3. **Set false:** `_unfreezePool()` called when jackpot/transition processing completes in advanceGame
