**Verdict:** PASS
**Files:** BurnieCoinflip.sol:224-627, BurnieCoin.sol:518-529, BurnieCoin.sol:556-566
**Summary:** Complete coinflip lifecycle verified correct. Both claim paths route to identical internal logic. Auto-rebuy carry calculations bounded by recycling caps and take-profit extraction.

### 1. Deposit Path

**Entry:** `depositCoinflip(address player, uint256 amount)` (BurnieCoinflip.sol:225-239)

The deposit function resolves the caller identity (direct vs operator-approved), then calls `_depositCoinflip`.

**`_depositCoinflip` (BurnieCoinflip.sol:242-316):**

2. **BAF lock check:** Deposits blocked during BAF resolution at every 10th level to prevent leaderboard front-running (line 252)
3. **Auto-claim:** Calls `_claimCoinflipsInternal(caller, false)` to settle pending winnings before depositing (line 255)
4. **CEI burn:** `burnie.burnForCoinflip(caller, amount)` burns BURNIE BEFORE any downstream state changes (line 266). Comment explicitly notes "CEI PATTERN: Burn first so reentrancy into downstream module calls cannot spend the same balance twice."
5. **Quest reward:** Quest module processes the deposit and may return bonus flip credit (lines 269-282)
6. **Recycling bonus:** Applied to the rebet portion only (lines 287-306)
7. **Daily flip storage:** `_addDailyFlip` stores the credited amount as a pending wager for the next day (line 308-315)

**Access control on `burnForCoinflip`** (BurnieCoin.sol:518-521):

Only the BurnieCoinflip contract can call this. Verified correct.

The `onlyFlipCreditors` modifier restricts callers to `degenerusGame` or `burnie` (lines 196-201). BurnieCoin.sol:556-566 wraps creditFlip with its own `onlyFlipCreditors` modifier restricting to GAME and AFFILIATE. Verified correct.

**creditFlipBatch access control** (BurnieCoinflip.sol:903-917): Same `onlyFlipCreditors` guard. Fixed-size array of 3 (no dynamic allocation DoS). Verified correct.

### 2. VRF Resolution

**Entry:** `processCoinflipPayouts(bool bonusFlip, uint256 rngWord, uint48 epoch)` (BurnieCoinflip.sol:803-887)

**Access:** `onlyDegenerusGameContract` modifier (line 807) -- only the game contract.

**Win/loss determination** (line 835):
```solidity
bool win = (rngWord & 1) == 1;
```
Pure 50/50 coin flip based on VRF random word's least significant bit. The win/loss outcome uses the raw `rngWord`, while the multiplier uses a mixed seed `keccak256(abi.encodePacked(rngWord, epoch))` to decorrelate outcome from multiplier (line 809). Verified correct.

**Variable multiplier** (lines 813-825):
- `roll = seedWord % 20`
- `roll == 0`: rewardPercent = 50% (1.5x total payout) -- 5% probability
- `roll == 1`: rewardPercent = 150% (2.5x total payout) -- 5% probability
- `roll == 2-19`: rewardPercent = `seedWord % 38 + 78` = [78%, 115%] (1.78x to 2.15x) -- 90% probability

**Note:** The normal-range uses `seedWord % COINFLIP_EXTRA_RANGE` which reuses `seedWord`, not a second modular reduction on a different value. Since `seedWord` is already hashed from VRF, this is fine for distribution purposes though there is a minor correlation with the roll value (both derived from same word). This is standard practice and not a finding.

**Presale bonus** (lines 827-832): Adds +6pp to rewardPercent during presale. Maximum total: 156% (2.56x). Verified correct.

### 3. Win Path

**Win payout calculation** (BurnieCoinflip.sol:528-530, inside `_claimCoinflipsInternal`):

Where `stake` = stored stake + carry (if auto-rebuy active). The payout is `stake * (1 + rewardPercent/100)`.

**Minting:** Via `burnie.mintForCoinflip(player, toClaim)` at multiple points:
- `_claimCoinflipsAmount` line 419: mints if `mintTokens` is true
- `_claimCoinflipsTakeProfit` line 394: mints take-profit multiples
- `_setCoinflipAutoRebuy` line 771: mints when disabling auto-rebuy

**`mintForCoinflip`** (BurnieCoin.sol:526-529):
```solidity

    if (msg.sender != coinflipContract) revert OnlyGame();

```
Only BurnieCoinflip can call. Verified correct.

### 4. Loss Path

On loss (BurnieCoinflip.sol:562-568):
```solidity
} else {
    unchecked { ++lossCount; }
```

- Principal is NOT returned (it was burned on deposit)

### 7. Auto-Rebuy Carry

**Carry forward** (BurnieCoinflip.sol:538-558, inside `_claimCoinflipsInternal` win branch):

When auto-rebuy is active:
1. Win payout calculated: `payout = stake + (stake * rewardPercent) / 100`
2. Take-profit extraction: `reserved = (payout / takeProfit) * takeProfit` -- integer multiples of stop amount are extracted as mintable
3. Carry = `payout - reserved`
4. Recycling bonus applied to carry: `carry += _recyclingBonus(carry)` or `_afKingRecyclingBonus(carry, ...)`

**Extraction analysis:** The carry forward is bounded because:
- On loss (line 567): `carry = 0` -- all accumulated carry is destroyed
- On win: carry = payout after take-profit extraction, plus recycling bonus (max 3.1%)
- The recycling bonus is applied ONCE per win-day, not compounded within a single claim
- Maximum single-day amplification: `carry_new = carry_old * (1 + rewardPercent/100) * (1 + recycling_rate)`
- At max values: `carry * 2.5 * 1.031 = carry * 2.5775` -- this is correct; the player wagered and won
- The 50/50 loss risk means expected carry per day is `carry * (1 + rewardPercent/100) * (1 + recycling_rate) * 0.5`

**Auto-rebuy carry does not amplify extraction beyond the protocol-designed recycling bonuses.** The `_calcAutoRebuy` interaction (referenced in plan as CP-03) is handled through the carry mechanism within `_claimCoinflipsInternal`, not a separate function. The carry is the running total of un-extracted winnings that are re-wagered on subsequent days. Verified: non-extractive.

**RNG lock on all claim paths:** Both `claimCoinflips`, `claimCoinflipsFromBurnie`, `consumeCoinflipsForBurn`, and `claimCoinflipsTakeProfit` check `degenerusGame.rngLocked()` and revert if locked. This prevents extracting carry when VRF outcome is already known. Auto-rebuy toggle (`_setCoinflipAutoRebuy`) also checks RNG lock (line 731). Verified: no carry extraction vulnerability.

**Token Flow:**
```
Deposit:  Player --burn--> BurnieCoin._burn(player, amount) [supply decreases]
Win:      BurnieCoin._mint(player, payout)                  [supply increases]
Loss:     No mint                                            [supply unchanged; deposit was already burned]
```

**Supply Impact:** Net deflationary on expectation. E[payout|win] ~= 1.9685x, but only 50% of flips win. EV = 0.98425x per flip before recycling. Recycling bonuses partially offset the house edge for active players.

**CEI Status:** Correct. Burns occur before state updates in deposit path (line 266). Claims update storage (`claimableStored`, `lastClaim`, `autoRebuyCarry`) before minting. The mint at line 419/394/771 is the final external call.

**Double-Claim Guard:** `coinflipBalance[cursor][player] = 0` clears stake on first processing (line 522). `state.lastClaim = processed` advances the cursor (line 614). No double-processing possible.

**Auto-Rebuy Interaction:** Carry forward verified non-extractive. Bounded by recycling caps and 50/50 loss risk. RNG lock prevents known-outcome extraction.

**Recommendation:** None.

## PAY-08: Coinflip Bounty System

**Verdict:** PASS
**Files:** BurnieCoinflip.sol:160-164 (storage), 634-692 (arming), 843-882 (resolution), DegenerusGame.sol:197-199 (DGNRS constants), 444-467 (DGNRS payout)
**Summary:** Bounty accumulates 1000 BURNIE/day. Arming requires all-time biggest flip record. Resolution credits half as flip stake (not direct mint). DGNRS gating verified at 50k bet + 20k pool.

### 1. Bounty Accumulation

**Accumulation** (BurnieCoinflip.sol:870-873):
```solidity
unchecked {
    currentBounty = currentBounty_ + uint128(PRICE_COIN_UNIT);
}
// PRICE_COIN_UNIT = 1000 ether  (BurnieCoinflip.sol:129)
```

Every time `processCoinflipPayouts` resolves a day, 1000 BURNIE is added to the bounty pool. This uses unchecked arithmetic -- at uint128 max (~3.4e38), overflow would reset the bounty. This is practically unreachable (would require ~3.4e35 years of accumulation). Verified correct.

**Initial value:** `currentBounty = 1_000 ether` (BurnieCoinflip.sol:161). Starts at 1000 BURNIE. Verified matches v1.1 spec.

### 2. Arming Conditions

**Arming** (BurnieCoinflip.sol:668-691, inside `_addDailyFlip`):

Arming requires ALL of:
1. `canArmBounty == true` -- only set for direct deposits via `depositCoinflip` (line 311: `directDeposit ? amount : 0`). Credited flips and bounty payouts pass `canArmBounty=false`.
2. `bountyEligible == true` -- same condition as `canArmBounty` (both `directDeposit`).
3. `recordAmount != 0` -- the raw deposit amount before bonuses/recycling.
4. `recordAmount > biggestFlipEver` -- must exceed all-time record.
5. `!game.rngLocked()` -- cannot arm during VRF pending.
6. If bounty already armed (`bountyOwedTo != address(0)`): must exceed current record by at least 1% (minimum +1 wei if 1% rounds to zero) (lines 680-685).

**DGNRS gating** is NOT in the arming logic -- it's in the resolution payout. The arming itself only requires setting a new all-time flip record. The DGNRS bounty is a separate reward triggered at resolution time.

### 3. Resolution

**Resolution** (BurnieCoinflip.sol:848-863):

When `bountyOwner != address(0) && currentBounty_ > 0`:
1. `slice = currentBounty_ >> 1` -- half of the bounty pool
2. `currentBounty_ -= uint128(slice)` -- remove half from pool
3. **If WIN:** Credit half as flip stake via `_addDailyFlip(to, slice, 0, false, false)` -- note `canArmBounty=false` and `bountyEligible=false`, so bounty payout cannot arm a new bounty. Also calls `game.payCoinflipBountyDgnrs(to, slice, currentBounty_)`.
4. **Regardless of win/loss:** `bountyOwedTo = address(0)` -- bounty is always disarmed after resolution.

**Critically:** On LOSS, half the pool is removed but not credited to anyone -- it is effectively destroyed (removed from `currentBounty_` but never minted or credited). This is a supply sink. Verified correct per v1.1-burnie-coinflip.md Section 5c.

### 4. DGNRS Payout (Bounty Resolution)

**`payCoinflipBountyDgnrs`** (DegenerusGame.sol:444-467):

Access: Only COIN or COINFLIP contracts (line 450-452).

Gating thresholds:
- `winningBet < COINFLIP_BOUNTY_DGNRS_MIN_BET` (50,000 ether = 50,000 BURNIE) -> early return (line 454)
- `bountyPool < COINFLIP_BOUNTY_DGNRS_MIN_POOL` (20,000 ether = 20,000 BURNIE) -> early return (line 455)

Constants verified (DegenerusGame.sol:197-199):
```solidity
uint16 private constant COINFLIP_BOUNTY_DGNRS_BPS = 20;          // 0.2%
uint256 private constant COINFLIP_BOUNTY_DGNRS_MIN_BET = 50_000 ether;  // 50k BURNIE
uint256 private constant COINFLIP_BOUNTY_DGNRS_MIN_POOL = 20_000 ether; // 20k BURNIE
```

These match the v1.1-parameter-reference.md exactly. If thresholds are met, payout = `poolBalance * 20 / 10_000 = 0.2%` of the DGNRS reward pool. Verified correct.

**Note on "DGNRS gating":** The plan refers to "50k DGNRS bet, 20k DGNRS in pool". More precisely, both thresholds are denominated in **BURNIE** (not DGNRS): the winning bet and bounty pool are BURNIE values. The DGNRS payout is `0.2%` of the sDGNRS reward pool balance, gated by those BURNIE thresholds. Clarified for accuracy.

**Token Flow:**
```
Accumulation:  +1000 BURNIE/day added to currentBounty (virtual, not minted)
Win resolution: Half credited as flip stake via _addDailyFlip (becomes next-day wager)
                DGNRS: 0.2% of reward pool transferred if bet >= 50k BURNIE and pool >= 20k BURNIE
Loss resolution: Half removed from pool (destroyed)
Both: bountyOwedTo = address(0), pool += 1000 BURNIE
```

**Supply Impact:** The bounty is a virtual accounting pool -- BURNIE in `currentBounty` is never actually minted. On win, the payout is credited as a flip stake (which the winner must then win to realize). On loss, the removed half is simply deleted from the counter. Net: slightly deflationary since loss resolution destroys virtual bounty value.

**CEI Status:** Correct. The bounty resolution writes to `currentBounty_`, `bountyOwedTo`, and credits via `_addDailyFlip` before the external `payCoinflipBountyDgnrs` call. The `_addDailyFlip` call only modifies internal state (coinflipBalance mapping and event emission). The final external call to `game.payCoinflipBountyDgnrs` is a cross-contract call but follows CEI since all local state is committed.

**Double-Claim Guard:** `bountyOwedTo = address(0)` at line 863 clears the bounty owner after every resolution. A bounty can only be claimed once per arming cycle.

**Auto-Rebuy Interaction:** Bounty payout is credited via `_addDailyFlip(to, slice, 0, false, false)` -- `recordAmount=0` means it cannot arm a new bounty and does not participate in the biggest-flip record. Verified correct.

**Recommendation:** None.

## PAY-19: Coinflip Recycling and Boons

**Verdict:** PASS
**Files:** BurnieCoinflip.sol:1040-1065 (recycling), 119-134 (constants), 287-306 (deposit recycling), 549-558 (carry recycling), 642-653 (boon application)
**Summary:** Normal recycling 1% capped at 1000 BURNIE. AfKing 1.6% base + up to 1.5% deity (3.1% max). Deity bonus capped at 1M BURNIE threshold. Boons are single-use, 2-day expiry, 100k BURNIE cap. No extraction amplification path.

### 1. Recycling Mechanics

**Normal recycling** (BurnieCoinflip.sol:1041-1048):
```solidity
function _recyclingBonus(uint256 amount) private pure returns (uint256 bonus) {
    if (amount == 0) return 0;
    bonus = amount / 100;              // 1% of rebet amount
    uint256 bonusCap = 1000 ether;     // Cap at 1000 BURNIE
    if (bonus > bonusCap) bonus = bonusCap;
}
```

**Rate:** 1% of rebet amount. **Cap:** 1000 BURNIE per event. Cap reached at 100,000 BURNIE rebet. Verified matches v1.1-burnie-coinflip.md Section 6a.

**AfKing recycling** (BurnieCoinflip.sol:1052-1065):
```solidity
function _afKingRecyclingBonus(uint256 amount, uint16 deityBonusHalfBps) private pure returns (uint256 bonus) {
    if (amount == 0) return 0;
    uint256 baseHalfBps = uint256(AFKING_RECYCLE_BONUS_BPS) * 2;  // 160 * 2 = 320
    if (deityBonusHalfBps == 0 || amount <= DEITY_RECYCLE_CAP) {
        uint256 totalHalfBps = baseHalfBps + uint256(deityBonusHalfBps);
        return (amount * totalHalfBps) / (uint256(BPS_DENOMINATOR) * 2);  // / 20000
    }
    uint256 fullHalfBps = baseHalfBps + uint256(deityBonusHalfBps);
    return (DEITY_RECYCLE_CAP * fullHalfBps + (amount - DEITY_RECYCLE_CAP) * baseHalfBps)
        / (uint256(BPS_DENOMINATOR) * 2);
}
```

**Base afKing rate:** `320 / 20000 = 1.60%`. Verified matches `AFKING_RECYCLE_BONUS_BPS = 160` (line 124).

**Deity bonus** (BurnieCoinflip.sol:1068-1082):
- `+2 halfBps per level` since activation (`AFKING_DEITY_BONUS_PER_LEVEL_HALF_BPS = 2`, line 125)
- Max: `300 halfBps` (`AFKING_DEITY_BONUS_MAX_HALF_BPS = 300`, line 126)
- Maximum deity rate: `300 / 20000 = 1.50%`
- **Total afKing + max deity: `(320 + 300) / 20000 = 3.10%`**
- Levels to cap: `300 / 2 = 150 levels`

**Deity cap split** (lines 1058-1064): When `amount > DEITY_RECYCLE_CAP` (1,000,000 BURNIE, line 127), deity bonus applies only to the first 1M. Remainder gets base-only 1.60%. This prevents whale amplification. Verified correct per v1.1-burnie-coinflip.md Section 6c.

**Note on afKing vs normal cap:** Normal recycling has a hard cap of 1000 BURNIE. AfKing recycling has NO hard cap -- it's purely percentage-based with the deity split at 1M. At 3.1% rate on 1M BURNIE, the bonus would be 31,000 BURNIE. This is by design -- afKing mode is a premium feature requiring sustained engagement.

### 5. Extraction Analysis

**Can recycling + boons create more BURNIE than the original stake?**

Consider a worst-case scenario: Player deposits 100k BURNIE with a 25% boon (max) and afKing + max deity recycling (3.1%).

1. Original deposit: 100,000 BURNIE (burned)
2. Quest reward: Variable, but separate from recycling/boon analysis
3. Boon boost: `min(100k, 100k) * 2500 / 10000 = 25,000 BURNIE`
4. `creditedFlip = 100,000 + questReward + 25,000 = 125,000` (ignoring quest)
5. Recycling on rebet portion: Only if prior winnings exist. For a fresh deposit with no carry, `rollAmount = 0`, so `rebetAmount = 0`, and no recycling applies.

**With prior carry (auto-rebuy):** If the player had 100k carry from prior wins and deposits 100k fresh:
- `creditedFlip = 100,000 + 25,000 = 125,000` (boon on fresh deposit)
- `rollAmount = autoRebuyCarry = 100,000` (prior carry)
- `rebetAmount = min(125,000, 100,000) = 100,000`
- Recycling: `100,000 * 3.1% = 3,100 BURNIE`
- Total `creditedFlip = 125,000 + 3,100 = 128,100 BURNIE` stored as next-day wager

**This is 128,100 credited from 100,000 fresh burn + 100,000 carry.** The carry was already earned through a previous win. The 28,100 "extra" BURNIE (25k boon + 3.1k recycling) comes from protocol-designed bonuses, not an accounting error.

**Key bound:** Recycling percentage plus boon amplification is bounded by:
- Recycling: max 3.1% on rebet portion (and only 1% capped at 1k for non-afKing)
- Boon: max 25% on 100k cap = 25k absolute maximum, single-use
- Combined: Cannot compound -- boon is one-time, recycling is per-event

**No extraction amplification path exists.** The boon provides a one-time boost and is consumed. Recycling adds a bounded percentage. Neither compounds unboundedly. The 50/50 loss risk on the subsequent flip ensures the expected value remains bounded.

**Token Flow:**
```
Recycling:  Prior winnings carried -> recycling bonus added to stake -> credited as next-day wager
            No mint occurs -- bonus is added to the wager amount stored in coinflipBalance
Boon:       Deity lootbox -> boon stored -> consumed on next manual deposit -> boost added to stake
            No mint occurs -- boost is added to the wager amount stored in coinflipBalance
```

**Supply Impact:** Recycling and boons increase the player's wager size but do not directly mint BURNIE. The increased wager means a larger potential win mint if the flip succeeds. At max recycling (3.1%), the house edge of ~1.575% is partially offset, resulting in ~0.985x to ~1.015x EV depending on mode. This is bounded and by design.

**CEI Status:** N/A for recycling -- it's a pure arithmetic bonus applied before storage write. Boon consumption is via `game.consumeCoinflipBoon(player)` which is a cross-contract state read+clear; no reentrancy concern since the result is used for arithmetic only.

**Recommendation:** None.

## BURNIE Supply Impact Summary

| Source | Direction | Magnitude |
|--------|-----------|-----------|
| Coinflip deposits (burn) | Deflationary | All deposited BURNIE is burned |
| Coinflip wins (mint) | Inflationary | ~1.97x principal minted on win |
| Coinflip losses | Deflationary | Principal burned, no mint |
| Net per flip | Deflationary | ~1.575% house edge |
| Recycling bonus | Inflationary (indirect) | 1-3.1% larger wager -> larger win mint |
| Boon boost | Inflationary (indirect) | One-time 5-25% larger wager |
| Bounty win | Inflationary (indirect) | Half of virtual pool becomes flip stake |
| Bounty loss | Deflationary | Half of virtual pool destroyed |
| Claim expiry | Deflationary | Unclaimed wins permanently lost |

**Net assessment:** The coinflip economy is structurally deflationary on expectation. The house edge (~1.575%) dominates recycling bonuses (max 3.1%) for most players. Only afKing players with max deity bonus and 100% re-wager can achieve slightly positive EV (~1.015x), and this is bounded by the deity cap at 1M BURNIE and the 50/50 loss risk. Claim expiry and bounty loss destruction provide additional supply sinks.
