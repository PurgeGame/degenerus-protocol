## 1. Scope Statement

**Requirements:** RNG-01, RNG-02

**RNG-01:** No permissionless external call can mutate ticketQueue read-slot entries while VRF is pending.
**RNG-02:** No permissionless external call can mutate traitBurnTicket entries used for jackpot winner selection during VRF pending window.

This proof covers the unified boundary (`targetLevel > level + 5`) introduced in v3.9, enumerating ALL permissionless external callers that can write to ticketQueue or traitBurnTicket. It extends:

- **v3.8** (Phases 68-72): General commitment window audit -- 55 variables, 87 permissionless paths, 51/51 SAFE. Established the double-buffer mechanism and rngLocked guard as the primary protection layers.
- **v3.9** (Phase 79): Far-future coin jackpot proof -- proved `_awardFarFutureCoinJackpot` is immune to manipulation. Covered 8 paths under the `> level + 6` boundary (since updated).

**What changed since v3.9:** The near/far boundary moved from `> level + 6` to `> level + 5` (consolidated in the v3.9 code changes). This TIGHTENS the far-future space -- one fewer level is classified as far-future, meaning more purchases route to the write buffer (which is structurally isolated from reads). The rngLocked guard and double-buffer architecture are unchanged. This proof re-enumerates all 9 permissionless paths under the new boundary to confirm safety.

## 2. Architecture Summary

### Double-Buffer (Write Key vs Read Key)

The ticket queue uses a double-buffer indexed by `ticketWriteSlot` (uint8 at slot 1, offset 23):

- **Write key:** `_tqWriteKey(lvl)` = `ticketWriteSlot != 0 ? lvl | TICKET_SLOT_BIT : lvl` (GS:686-688)
- **Read key:** `_tqReadKey(lvl)` = `ticketWriteSlot == 0 ? lvl | TICKET_SLOT_BIT : lvl` (GS:691-692)
- **TICKET_SLOT_BIT** = `1 << 23` -- bit 23 separates the two buffer sides

Write and read keys occupy different bit-23 spaces by construction. All purchases route to the write key. Jackpot resolution reads from the read key. No overlap is possible.

At VRF request time, `_swapAndFreeze(purchaseLevel)` (GS:709-714) flips `ticketWriteSlot ^= 1`, making the current write buffer become the read buffer. New purchases after the swap write to the NEW write buffer, which is invisible to jackpot resolution.

### Far-Future Key Space (Bit 22)

- **FF key:** `_tqFarFutureKey(lvl)` = `lvl | TICKET_FAR_FUTURE_BIT` (GS:699-701)
- **TICKET_FAR_FUTURE_BIT** = `1 << 22` -- separate from both buffer sides
- FF tickets are NOT double-buffered; they persist until drained by `processFutureTicketBatch`

### rngLocked Guard

All three queue functions contain the same guard:

```solidity
bool isFarFuture = targetLevel > level + 5;
if (isFarFuture && rngLockedFlag && !phaseTransitionActive) revert RngLocked();
```

- `_queueTickets` (GS:544-545)
- `_queueTicketsScaled` (GS:572-573)
- `_queueTicketRange` (GS:621-622)

During the VRF pending window:
- `rngLockedFlag = true` (set at AM:1328 by `_finalizeRngRequest`)
- `phaseTransitionActive = false` for all external callers (only set at AM:493 within `_endPhase`, cleared at AM:260)

Therefore, any external call that targets a far-future level (`> level + 5`) reverts with `RngLocked()`.

### Near-Future Routing

Near-future targets (`targetLevel <= level + 5`) route to `_tqWriteKey(targetLevel)` -- the write buffer. Since the write buffer is structurally separated from the read buffer (bit-23 space), near-future writes are invisible to jackpot resolution regardless of rngLocked state.

### Non-Permissionless Paths (NOT in enumeration)

**advanceGame():** The game resolver (`advanceGame`) is NOT permissionless -- it is triggered by the advanceGame resolver and executes atomically with jackpot resolution. Its internal paths include:

- **_processPhaseTransition** (AM:1240-1249): Writes vault perpetual tickets at `purchaseLevel + 99` (= `level + 100`). Uses `phaseTransitionActive = true` (AM:493) to bypass the `rngLockedFlag` guard. These writes are deterministic (fixed addresses: `ContractAddresses.SDGNRS`, `ContractAddresses.VAULT`; fixed quantity: `VAULT_PERPETUAL_TICKETS = 16`; computed level) and atomic with jackpot draws.
- **Auto-rebuy** within advanceGame: Targets `level + 1` to `level + 4` (near-future), routes to write buffer.
- **processFutureTicketBatch / processTicketBatch:** Drain (delete) entries from read and FF queues during advanceGame. These reduce population, not increase it, and execute atomically before jackpot draws.

**Constructor** (DG:249-254): `_queueTickets` called for levels 1-100 during deployment. Not callable post-deployment.

### RNG-01 Conclusion

**No permissionless external call can mutate `ticketQueue[readKey]` entries.** Write-buffer isolation is structural: all 9 permissionless callers write to `_tqWriteKey`, which occupies a different bit-23 space than `_tqReadKey`. Far-future writes (`targetLevel > level + 5`) are blocked by `rngLockedFlag` in all three queue functions (GS:544-545, GS:572-573, GS:621-622). The combined invariant: read-slot population is frozen at VRF request time via `_swapAndFreeze` and cannot be modified by any external actor during the commitment window.

### RNG-02 Conclusion

**traitBurnTicket is written exclusively by advanceGame-internal paths.** Both `_raritySymbolBatch` implementations (MintModule and JackpotModule) are reachable only through private functions called via delegatecall from the AdvanceModule during `advanceGame`. No permissionless external function (purchase, lootbox, whale bundle, endgame claim, decimator claim) can reach `_raritySymbolBatch` or write to `traitBurnTicket`. The 0/0 permissionless writer count makes this structurally safe regardless of rngLocked state.

## 5. Delta from v3.8 / v3.9

### Boundary Change: `> level + 6` to `> level + 5`

The v3.9 code changes unified the near/far boundary. The prior boundary was `targetLevel > level + 6` (one extra level of near-future padding). The new boundary is `targetLevel > level + 5`.

**Impact analysis:**
- **Tighter FF space:** Level `level + 6` was previously near-future (routed to write key); it is now far-future (routed to FF key). This means more levels are subject to `rngLockedFlag` protection, not fewer. The protection surface EXPANDED.
- **More write-buffer routing:** Level `level + 5` was and remains near-future. The boundary change only affects level `level + 6`, moving it from write key to FF key. Since FF has the additional `rngLockedFlag` guard, this is strictly more protective.
- **rngLocked guard unchanged:** The guard logic (`isFarFuture && rngLockedFlag && !phaseTransitionActive`) is identical across all three queue functions. Only the `isFarFuture` classification shifted by one level.
- **Double-buffer unchanged:** `_tqWriteKey`, `_tqReadKey`, `_swapAndFreeze`, and `ticketWriteSlot` are unmodified.

### v3.8 Verdicts

All 51/51 SAFE verdicts from v3.8 remain valid. The v3.8 audit analyzed the general commitment window for all 55 variables. The ticket queue double-buffer and rngLocked guard were key components of those verdicts. Since the boundary change only tightens the FF space (more protection), no v3.8 verdict is weakened.

### v3.9 Verdict

The v3.9 far-future coin jackpot proof remains valid. That proof analyzed `_awardFarFutureCoinJackpot` specifically, which reads from `_tqReadKey` (frozen by swap) and `_tqFarFutureKey` (protected by rngLocked). The boundary change does not affect these read paths. The v3.9 proof covered 8 paths; this proof covers 9 (adding `claimDecimatorJackpot` auto-rebuy as the 9th distinct permissionless path).

### Three Independent Protection Layers

1. **Double-buffer swap** (Layer 1): `_swapAndFreeze` (AM:233, GS:709-714) flips `ticketWriteSlot` at VRF request time. All subsequent external writes go to the NEW write buffer. Jackpot resolution reads from `_tqReadKey`, which points to the OLD (frozen) buffer.

2. **rngLockedFlag guard** (Layer 2): `rngLockedFlag = true` during commitment window (AM:1328). Every permissionless call to `_queueTickets` (GS:544-545), `_queueTicketsScaled` (GS:572-573), or `_queueTicketRange` (GS:621-622) that targets far-future levels reverts with `RngLocked()`.

3. **phaseTransitionActive exemption** (Layer 3): The only code that writes FF tickets while `rngLockedFlag = true` requires `phaseTransitionActive = true` (GS:544-545: `!phaseTransitionActive`). This flag is set exclusively within `advanceGame` at AM:493 and cleared at AM:260. Writes under this exemption are deterministic and atomic with jackpot draws.

**No code changes required.**
