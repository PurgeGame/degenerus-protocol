#### 4. _addClaimableEth Path

**_addClaimableEth (JackpotModule:973-994):**
```solidity
function _addClaimableEth(address beneficiary, uint256 weiAmount, uint256 entropy)
    private returns (uint256 claimableDelta) {
    if (weiAmount == 0) return 0;
    if (!gameOver) {
        AutoRebuyState memory state = autoRebuyState[beneficiary];
        if (state.autoRebuyEnabled) {

        }
    }

    return weiAmount;
}
```

- Returns `claimableDelta` representing only the ETH actually credited to claimableWinnings (not ticket-converted ETH).
- Normal path: calls `_creditClaimable(beneficiary, weiAmount)` which does `claimableWinnings[beneficiary] += weiAmount` (unchecked, safe because uint256 max is ~10^77 wei).
- Auto-rebuy path: `_processAutoRebuy` returns `calc.reserved` (the take-profit portion), which is the only ETH credited to claimableWinnings. The ticket-converted ETH goes to futurePrizePool or nextPrizePool, not claimablePool.

**_creditClaimable (PayoutUtils:30-36):**
```solidity
function _creditClaimable(address beneficiary, uint256 weiAmount) internal {
    if (weiAmount == 0) return;
    unchecked { claimableWinnings[beneficiary] += weiAmount; }
    emit PlayerCredited(beneficiary, beneficiary, weiAmount);
}
```
- Unchecked addition is safe: uint256 overflow at ~1.15 * 10^59 ETH, physically impossible.

#### 5. claimablePool Accounting

The claimablePool update for this path follows the batched liability pattern:

1. `_resolveTraitWinners` (JackpotModule:1630-1757) calls `_addClaimableEth` per winner, accumulating `totalLiability` (line 1742).
2. `_processOneBucket` (JackpotModule:1579-1606) propagates `bucketLiability` to `ctx.liabilityDelta` (line 1605).
3. `_distributeJackpotEth` (JackpotModule:1537-1576) applies the batched update: `claimablePool += ctx.liabilityDelta` (line 1572-1573).

**Invariant check:** The liability delta represents only ETH credited to claimableWinnings (via `_creditClaimable`), not ETH converted to tickets by auto-rebuy. The auto-rebuy path returns only `calc.reserved` from `_processAutoRebuy` (JackpotModule:1043), which is the portion actually credited. Therefore: `claimablePool increment == sum of all _creditClaimable calls` -- **invariant holds**.

**balance >= claimablePool:** The ETH for this distribution was deducted from futurePrizePool (stays in contract balance). ETH credited via `_creditClaimable` increments claimablePool by the same amount. Auto-rebuy routes ETH to futurePrizePool/nextPrizePool (still in contract). No ETH leaves the contract until `claimWinnings()` is called. Therefore `address(this).balance >= claimablePool` is maintained.

#### 5. _distributeJackpotEth

**Function (JackpotModule:1537-1576):**
```solidity
function _distributeJackpotEth(
    uint24 lvl, uint256 ethPool, uint256 entropy,
    uint8[4] memory traitIds, uint16[4] memory shareBps, uint16[4] memory bucketCounts
) private returns (uint256 totalPaidEth) {
    JackpotEthCtx memory ctx;
    ctx.entropyState = entropy;
    ctx.lvl = lvl;

    uint256 unit = PriceLookupLib.priceForLevel(lvl + 1) >> 2;
    uint8 remainderIdx = JackpotBucketLib.soloBucketIndex(entropy);
    uint256[4] memory shares = JackpotBucketLib.bucketShares(
        ethPool, shareBps, bucketCounts, remainderIdx, unit
    );

    for (uint8 traitIdx; traitIdx < 4; ) {
        _processOneBucket(ctx, traitIdx, traitIds, shares, bucketCounts);
        unchecked { ++traitIdx; }
    }

    if (ctx.liabilityDelta != 0) {
        claimablePool += ctx.liabilityDelta;
    }
    return ctx.totalPaidEth;
}
```

**Verification:**
- `bucketShares` computes per-bucket ETH allocations. The `remainderIdx` (solo bucket) receives any rounding remainder, ensuring `sum(shares) == ethPool` -- **no dust loss**.
- `_processOneBucket` iterates all 4 buckets, calling `_resolveTraitWinners` which returns `(ethDelta, bucketLiability, ticketSpent)`.
- `ctx.liabilityDelta` accumulates all claimable credits across all buckets.
- `claimablePool += ctx.liabilityDelta` is a single batched SSTORE -- **correct and gas-efficient**.
- `totalPaidEth` includes both ETH credited and whale pass conversions -- used by caller for pool accounting.

**claimablePool delta correctness:** The liability delta is the sum of all `_addClaimableEth` return values. When auto-rebuy converts winnings, `_processAutoRebuy` returns only `calc.reserved` (the take-profit portion credited to claimableWinnings). The ticket-converted ETH is routed to futurePrizePool/nextPrizePool and does NOT appear in the liability delta. Therefore `claimablePool` accurately reflects only the ETH available for player claims.

1. **Checks:** `isEthDay` condition verified (JackpotModule:634)

## Shared Infrastructure

This section documents shared payout infrastructure used by all jackpot distribution paths.

### _addClaimableEth (JackpotModule:973-994)

Central ETH crediting function for all jackpot paths. Returns `claimableDelta` representing ETH actually added to claimableWinnings.

**Normal path:** Credits full amount via `_creditClaimable` -> returns `weiAmount`.

**Auto-rebuy path:** If `autoRebuyState[beneficiary].autoRebuyEnabled`:
1. Calls `_calcAutoRebuy` (PayoutUtils:38-74) to compute ticket conversion.
2. `_calcAutoRebuy` determines:
   - `takeProfit`: portion reserved as claimable ETH (based on player's `takeProfit` setting).
   - `rebuyAmount`: `weiAmount - reserved` (portion to convert to tickets).
   - `levelOffset`: random 1-4 levels ahead (VRF-derived). +1 = nextPool (25% chance), +2/+3/+4 = futurePool (75% chance).
   - `ticketPrice`: `PriceLookupLib.priceForLevel(targetLevel) >> 2` (quarter of face price, i.e., 4 tickets per level price).
   - `baseTickets`: `rebuyAmount / ticketPrice`.
   - `bonusTickets`: `baseTickets * bonusBps / 10_000` where bonusBps = 13000 (130% normal) or 14500 (145% afKing).
   - `ethSpent`: `baseTickets * ticketPrice` (no rounding exploitation: integer division then multiply back).
3. `_processAutoRebuy` (JackpotModule:1004-1043):
   - Queues `calc.ticketCount` tickets at `calc.targetLevel`.
   - Routes `calc.ethSpent` to futurePrizePool (75%) or nextPrizePool (25%) based on `calc.toFuture`.
   - Credits `calc.reserved` (take-profit) to claimableWinnings.
   - Returns `calc.reserved` as claimableDelta.

**Key invariant:** `calc.reserved + calc.ethSpent <= weiAmount`. Any dust (from `rebuyAmount - ethSpent` when ethSpent < rebuyAmount and ticketCount > 0) is dropped unconditionally (not credited, not pooled). This is at most `ticketPrice - 1` wei per payout (subpenny amounts), documented as intentional.

### _creditClaimable (PayoutUtils:30-36)

Minimal function that increments `claimableWinnings[beneficiary]` (unchecked) and emits `PlayerCredited`.

**Safety of unchecked:** uint256 max is ~1.15 * 10^77. Total ETH in existence is ~120M * 10^18 = ~1.2 * 10^26. Overflow is physically impossible.

### _calcAutoRebuy (PayoutUtils:38-74)

Pure function computing auto-rebuy parameters.

- **bonusBps=13000 (130%):** Player receives 30% bonus tickets beyond what their ETH buys. These are unfunded by the player's ETH but are backed by the protocol's incentive pool design (the pool has over-collateralization from lootbox mechanics).
- **bonusBps=14500 (145% for afKing):** Enhanced bonus for afKing-active players. 45% extra tickets.
- **ethSpent = baseTickets * ticketPrice:** Not `bonusTickets * ticketPrice`. The bonus tickets are "free" -- they don't consume additional ETH from the payout. This is **correct and by design** -- the auto-rebuy bonus incentivizes participation without extracting more value from the pool than the original payout.

### _queueWhalePassClaimCore (PayoutUtils:77-93)

Used by solo bucket (PAY-02 day-5 primarily) for large payouts. Splits amount into whale passes (2.25 ETH each) and claimable remainder.

- `fullHalfPasses = amount / HALF_WHALE_PASS_PRICE` (integer division)
- `remainder = amount - (fullHalfPasses * HALF_WHALE_PASS_PRICE)`
- Remainder credited to claimableWinnings AND claimablePool (PayoutUtils:88-91) -- **correct**.
- Whale pass cost routed to futurePrizePool (not claimablePool) via solo bucket handler.

**Invariant verification:**
- Every `+increment` to claimablePool corresponds to an equal `+increment` to some player's `claimableWinnings[player]` (via `_creditClaimable` or direct assignment in `_queueWhalePassClaimCore`).
- Every `-decrement` to claimablePool corresponds to zeroing `claimableWinnings[player]` to sentinel (1) and transferring `payout = amount - 1` to the player.
- The sentinel value means claimablePool carries ~1 wei per player who has ever claimed, which is negligible and does not affect solvency.
- Auto-rebuy reduces both the claimableWinnings increment and the claimablePool delta by the same amount (the ticket-converted portion), maintaining the 1:1 relationship.

**Balance >= claimablePool invariant:**
- All pool sources (futurePrizePool, currentPrizePool) are backed by contract ETH/stETH balance.
- ETH credited to claimablePool remains in the contract until claimed.
- Auto-rebuy moves ETH between pool variables (not out of contract).
- The only way ETH exits is via `_payoutWithStethFallback` / `_payoutWithEthFallback` in `_claimWinningsInternal`, which first decrements claimablePool.
