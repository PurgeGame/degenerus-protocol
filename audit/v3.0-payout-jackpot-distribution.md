#### 4. _addClaimableEth Path

**_addClaimableEth (JackpotModule:973-994):**


#### 5. claimablePool Accounting

1. `_resolveTraitWinners` (JackpotModule:1630-1757) calls `_addClaimableEth` per winner, accumulating `totalLiability` (line 1742).

The auto-rebuy path returns only `calc.reserved` from `_processAutoRebuy` (JackpotModule:1043), which is the portion actually credited. Therefore: `claimablePool increment == sum of all _creditClaimable calls` -- **invariant holds**.

**balance >= claimablePool:** The ETH for this distribution was deducted from futurePrizePool (stays in contract balance). ETH credited via `_creditClaimable` increments claimablePool by the same amount. Auto-rebuy routes ETH to futurePrizePool/nextPrizePool (still in contract). No ETH leaves the contract until `claimWinnings()` is called. Therefore `address(this).balance >= claimablePool` is maintained.

#### 5. _distributeJackpotEth

**Function (JackpotModule:1537-1576):**
```solidity
function _distributeJackpotEth(
    uint24 lvl, uint256 ethPool, uint256 entropy,
    uint8[4] memory traitIds, uint16[4] memory shareBps, uint16[4] memory bucketCounts
) private returns (uint256 totalPaidEth) {
    JackpotEthCtx memory ctx;
    ctx.entropyState = entropy;
    ctx.lvl = lvl;

    uint256 unit = PriceLookupLib.priceForLevel(lvl + 1) >> 2;
    uint8 remainderIdx = JackpotBucketLib.soloBucketIndex(entropy);
    uint256[4] memory shares = JackpotBucketLib.bucketShares(
        ethPool, shareBps, bucketCounts, remainderIdx, unit
    );

    for (uint8 traitIdx; traitIdx < 4; ) {
        _processOneBucket(ctx, traitIdx, traitIds, shares, bucketCounts);
        unchecked { ++traitIdx; }
    }

    if (ctx.liabilityDelta != 0) {
        claimablePool += ctx.liabilityDelta;
    }
    return ctx.totalPaidEth;
}
```

**Verification:**
- `bucketShares` computes per-bucket ETH allocations. The `remainderIdx` (solo bucket) receives any rounding remainder, ensuring `sum(shares) == ethPool` -- **no dust loss**.
- `_processOneBucket` iterates all 4 buckets, calling `_resolveTraitWinners` which returns `(ethDelta, bucketLiability, ticketSpent)`.
- `ctx.liabilityDelta` accumulates all claimable credits across all buckets.
- `claimablePool += ctx.liabilityDelta` is a single batched SSTORE -- **correct and gas-efficient**.
- `totalPaidEth` includes both ETH credited and whale pass conversions -- used by caller for pool accounting.

**claimablePool delta correctness:** The liability delta is the sum of all `_addClaimableEth` return values. When auto-rebuy converts winnings, `_processAutoRebuy` returns only `calc.reserved` (the take-profit portion credited to claimableWinnings). The ticket-converted ETH is routed to futurePrizePool/nextPrizePool and does NOT appear in the liability delta. Therefore `claimablePool` accurately reflects only the ETH available for player claims.

## Shared Infrastructure

This section documents shared payout infrastructure used by all jackpot distribution paths.

### _addClaimableEth (JackpotModule:973-994)

Central ETH crediting function for all jackpot paths. Returns `claimableDelta` representing ETH actually added to claimableWinnings.

**Normal path:** Credits full amount via `_creditClaimable` -> returns `weiAmount`.

**Auto-rebuy path:** If `autoRebuyState[beneficiary].autoRebuyEnabled`:
1. Calls `_calcAutoRebuy` (PayoutUtils:38-74) to compute ticket conversion.
2. `_calcAutoRebuy` determines:
   - `takeProfit`: portion reserved as claimable ETH (based on player's `takeProfit` setting).
   - `rebuyAmount`: `weiAmount - reserved` (portion to convert to tickets).
   - `levelOffset`: random 1-4 levels ahead (VRF-derived). +1 = nextPool (25% chance), +2/+3/+4 = futurePool (75% chance).
   - `ticketPrice`: `PriceLookupLib.priceForLevel(targetLevel) >> 2` (quarter of face price, i.e., 4 tickets per level price).
   - `baseTickets`: `rebuyAmount / ticketPrice`.
   - `bonusTickets`: `baseTickets * bonusBps / 10_000` where bonusBps = 13000 (130% normal) or 14500 (145% afKing).
   - `ethSpent`: `baseTickets * ticketPrice` (no rounding exploitation: integer division then multiply back).
3. `_processAutoRebuy` (JackpotModule:1004-1043):
   - Queues `calc.ticketCount` tickets at `calc.targetLevel`.
   - Routes `calc.ethSpent` to futurePrizePool (75%) or nextPrizePool (25%) based on `calc.toFuture`.
   - Credits `calc.reserved` (take-profit) to claimableWinnings.
   - Returns `calc.reserved` as claimableDelta.

**Key invariant:** `calc.reserved + calc.ethSpent <= weiAmount`. Any dust (from `rebuyAmount - ethSpent` when ethSpent < rebuyAmount and ticketCount > 0) is dropped unconditionally (not credited, not pooled). This is at most `ticketPrice - 1` wei per payout (subpenny amounts), documented as intentional.

### _creditClaimable (PayoutUtils:30-36)

Minimal function that increments `claimableWinnings[beneficiary]` (unchecked) and emits `PlayerCredited`.

**Safety of unchecked:** uint256 max is ~1.15 * 10^77. Total ETH in existence is ~120M * 10^18 = ~1.2 * 10^26. Overflow is physically impossible.

### _calcAutoRebuy (PayoutUtils:38-74)

Pure function computing auto-rebuy parameters.

- **bonusBps=13000 (130%):** Player receives 30% bonus tickets beyond what their ETH buys. These are unfunded by the player's ETH but are backed by the protocol's incentive pool design (the pool has over-collateralization from lootbox mechanics).
- **bonusBps=14500 (145% for afKing):** Enhanced bonus for afKing-active players. 45% extra tickets.
- **ethSpent = baseTickets * ticketPrice:** Not `bonusTickets * ticketPrice`. The bonus tickets are "free" -- they don't consume additional ETH from the payout. This is **correct and by design** -- the auto-rebuy bonus incentivizes participation without extracting more value from the pool than the original payout.

### _queueWhalePassClaimCore (PayoutUtils:77-93)

Used by solo bucket (PAY-02 day-5 primarily) for large payouts. Splits amount into whale passes (2.25 ETH each) and claimable remainder.

- `fullHalfPasses = amount / HALF_WHALE_PASS_PRICE` (integer division)
- `remainder = amount - (fullHalfPasses * HALF_WHALE_PASS_PRICE)`

### Detailed Analysis

#### 1. _budgetToTicketUnits (JackpotModule:1078-1086)

```solidity
function _budgetToTicketUnits(uint256 budget, uint24 lvl) private pure returns (uint256) {
    if (budget == 0) return 0;
    uint256 ticketPrice = PriceLookupLib.priceForLevel(lvl);
    return ticketPrice == 0 ? 0 : (budget << 2) / ticketPrice;
}
```

**Verification:**
- `budget << 2` = `budget * 4` (multiply by 4, since each level price covers 4 tickets).
- `ticketPrice = PriceLookupLib.priceForLevel(lvl)` -- full level price.
- Result: `(budget * 4) / priceForLevel(lvl)` = number of ticket units affordable.
- Each ticket costs `priceForLevel(lvl) / 4` = `priceForLevel(lvl) >> 2`.

**Over-collateralization analysis:**

When called from `_distributeLootboxAndTickets` (JackpotModule:1107):
```solidity
uint256 ticketBasis = (lootboxBudget * ticketConversionBps) / 10_000;
uint256 ticketUnits = _budgetToTicketUnits(ticketBasis, lvl + 1);
```

For purchase-phase drip (PAY-01): `ticketConversionBps = 5000` (50%).
- ticketBasis = lootboxBudget / 2
- ticketUnits = (lootboxBudget / 2 * 4) / priceForLevel(lvl + 1)
- = (lootboxBudget * 2) / priceForLevel(lvl + 1)

But the FULL lootboxBudget goes to nextPrizePool (line 1103). So:
- **ETH backing per ticket** = lootboxBudget / ticketUnits
- = lootboxBudget / ((lootboxBudget * 2) / priceForLevel(lvl + 1))
- = priceForLevel(lvl + 1) / 2

- **Ticket face value** = priceForLevel(lvl + 1) / 4

- **Backing ratio** = (priceForLevel(lvl+1) / 2) / (priceForLevel(lvl+1) / 4) = **2x**

This confirms the 2x over-collateralization: each ticket issued is backed by twice its face value in pool ETH.

For jackpot-phase daily (PAY-02), the lootbox budget from the daily jackpot uses `_budgetToTicketUnits(dailyLootboxBudget, lvl + 1)` directly (line 403-404), with no `ticketConversionBps` reduction. However, the full `dailyLootboxBudget` is moved to nextPrizePool (line 410), so the backing ratio is:
- ticketUnits = (dailyLootboxBudget * 4) / priceForLevel(lvl + 1)
- Backing per ticket = dailyLootboxBudget / ticketUnits = priceForLevel(lvl+1) / 4 = **1x** (face value)

This is different from the purchase-phase 2x over-collateralization. In the jackpot phase, tickets are backed at 1x face value. This is by design -- the jackpot phase is distributing currentPrizePool which will be fully consumed by end of day 5, so 1x backing is appropriate.

**Rounding exploitation check:**
- `ticketUnits = (budget << 2) / ticketPrice` uses integer division (truncates).
- `ethSpent = ticketUnits * (ticketPrice >> 2)` would be less than or equal to budget.
- No overflow: `budget << 2` could overflow for budget > 2^254, but pool sizes are bounded by total ETH supply (~10^26 wei), well below this.
- The ticket count is always the floor, meaning the pool retains slightly more ETH per ticket than the formula suggests -- **conservative, correct**.

**Verification:**

- Only the reduced `ticketBasis` determines ticket count -- over-collateralization.
- Tickets priced at `lvl + 1` (next level), not current level -- **matches spec** (v1.1-purchase-phase-distribution.md Section 2f).
- `_distributeTicketJackpot` distributes ticket units to trait-matched winners at current level, with up to LOOTBOX_MAX_WINNERS (100) winners.
- If `ticketUnits == 0` (e.g., budget too small for any tickets), the ETH still goes to nextPrizePool -- **no fund loss**.

#### 3. Pool Transition Chain

The 4-pool chain: `futurePrizePool -> nextPrizePool -> currentPrizePool -> claimablePool`

**futurePrizePool -> nextPrizePool:**
- Purchase-phase ETH drip lootbox: 75% of drip -> nextPrizePool via `_distributeLootboxAndTickets` (JackpotModule:1103).
- Jackpot-phase carryover lootbox: 50% of 1% reserve -> nextPrizePool (JackpotModule:451-453).
- Auto-rebuy: 75% chance routes to futurePrizePool (JackpotModule:1027), 25% to nextPrizePool (JackpotModule:1029).
- Day 1 early-bird: 3% of futurePrizePool -> nextPrizePool (JackpotModule:884).

**nextPrizePool -> currentPrizePool:**
- At level transition via `consolidatePrizePools` (JackpotModule:886-924):
  ```solidity
  currentPrizePool += _getNextPrizePool();
  _setNextPrizePool(0);
  ```
  This is a complete merge -- all of nextPrizePool moves to currentPrizePool at each level transition.

**currentPrizePool -> claimablePool:**
- Via jackpot draws: ETH credited to winners via `_addClaimableEth` -> `_creditClaimable` -> claimableWinnings, then claimablePool += liabilityDelta.
- Daily lootbox budget moved to nextPrizePool (JackpotModule:409-410).

**Verification of no fund loss or duplication:**
1. **futurePrizePool drip:** Deducted upfront (JackpotModule:639), then distributed. If no winners, ETH stays in contract as yield surplus. No duplication.
2. **nextPrizePool accumulation:** Only grows via explicit `_setNextPrizePool(current + delta)`. Zeroed on merge. No duplication.
3. **currentPrizePool consumption:** Decremented for lootbox (line 409), decremented for ETH payouts (line 528). Day 5 uses 10000 BPS (100%), emptying the pool. Remaining dust (from per-winner rounding) stays in currentPrizePool and becomes yield surplus after jackpot phase.
4. **claimablePool:** Only incremented by liability delta from actual `_creditClaimable` calls. Only decremented by actual player claims. Sentinel mechanism prevents double-claim.

**No fund loss path:** Every wei deducted from one pool is either added to another pool or credited to a player's claimableWinnings. The only "leakage" is rounding dust (at most a few wei per distribution), which remains in the contract balance.

**No fund duplication path:** Each deduction is a single `pool -= amount` followed by a single `otherPool += amount` or `claimableWinnings[player] += amount`. There are no code paths where the same ETH is counted in two pools simultaneously.

#### 4. Ticket Pricing

**`PriceLookupLib.priceForLevel(level)`** (PriceLookupLib.sol:21-46):

| Level Range | Price (ETH) | Ticket Price (ETH) |
|-------------|-------------|---------------------|
| 0-4 | 0.01 | 0.0025 |
| 5-9 | 0.02 | 0.005 |
| 10-29 | 0.04 | 0.01 |
| 30-59 | 0.08 | 0.02 |
| 60-89 | 0.12 | 0.03 |
| 90-99 | 0.16 | 0.04 |
| x00 (100, 200, ...) | 0.24 | 0.06 |
| x01-x29 | 0.04 | 0.01 |
| x30-x59 | 0.08 | 0.02 |
| x60-x89 | 0.12 | 0.03 |
| x90-x99 | 0.16 | 0.04 |

Ticket price = priceForLevel / 4 (since `_budgetToTicketUnits` does `budget << 2 / ticketPrice`, and tickets are individually `ticketPrice >> 2`).

**Over-collateralization at level transitions:** When the game transitions from level N to level N+1, tickets issued at level N+1's price are backed by ETH in nextPrizePool. At consolidation, nextPrizePool merges into currentPrizePool. The 2x over-collateralization (from purchase-phase lootbox conversion) means the pool grows faster than ticket obligations. Since tickets at level N+1 cost `priceForLevel(N+1) / 4` but are backed by `priceForLevel(N+1) / 2` worth of ETH in the pool, there is always surplus ETH per ticket.

**Edge case: price tier boundary transitions.** When a level transitions across a price tier boundary (e.g., level 29 -> 30, where price jumps from 0.04 to 0.08), tickets issued for level 30 at 0.08/4 = 0.02 ETH each are backed by pool ETH accumulated at the 0.04 tier. However, the 2x over-collateralization means the pool had 2 * 0.04/4 = 0.02 ETH per ticket -- exactly matching the new tier's ticket price. The over-collateralization absorbs the price doubling. For the 0.08 -> 0.12 transition (1.5x increase), the 2x over-collateralization provides 2 * 0.08/4 = 0.04 > 0.12/4 = 0.03 -- still solvent.

#### 5. Auto-Rebuy Carry Interaction

**_calcAutoRebuy (PayoutUtils:38-74):**

```solidity
uint256 ticketPrice = PriceLookupLib.priceForLevel(c.targetLevel) >> 2;
uint256 baseTickets = c.rebuyAmount / ticketPrice;
c.ethSpent = baseTickets * ticketPrice;
uint256 bonusTickets = (baseTickets * (state.afKingMode ? bonusBpsAfKing : bonusBps)) / 10_000;
c.ticketCount = uint32(bonusTickets);
```

**Pricing verification:**
- `c.targetLevel = currentLevel + levelOffset` where levelOffset is 1-4 (VRF-derived).
- Ticket price uses `PriceLookupLib.priceForLevel(targetLevel) >> 2` -- priced at the target level (1-4 levels ahead), not the current level.
- This means auto-rebuy tickets are priced at a future level's price, which is >= current level's price (prices are non-decreasing within a cycle, with the x00 milestone being the highest at 0.24 ETH).

**Over-collateralization maintenance:**
- `ethSpent = baseTickets * ticketPrice` -- ETH routed to futurePrizePool or nextPrizePool.
- `ticketCount = bonusTickets` where `bonusTickets = baseTickets * 13000/10000 = baseTickets * 1.3` (or 1.45 for afKing).
- The bonus tickets are NOT backed by additional ETH from the player's payout. They are "free" bonus tickets.
- **Is this an extraction risk?** No. The bonus tickets participate in future jackpot draws. The ETH that funds those draws comes from the pool (which received the `ethSpent` amount). The bonus tickets represent a protocol incentive: the game deliberately issues 30-45% more tickets than the ETH covers to encourage auto-rebuy behavior. The structural over-collateralization (2x from lootbox mechanics) absorbs this bonus. Net effective backing: 2x / 1.3x = ~1.54x for normal, 2x / 1.45x = ~1.38x for afKing -- still over-collateralized.

**Carry mechanism:**
- `calc.reserved = (weiAmount / state.takeProfit) * state.takeProfit` -- take-profit portion, credited to claimableWinnings.
- `calc.rebuyAmount = weiAmount - calc.reserved` -- remainder converted to tickets.
- Dust from `rebuyAmount - ethSpent` (when ethSpent < rebuyAmount and baseTickets > 0) is dropped unconditionally. This is at most `ticketPrice - 1` wei (~0.05 ETH worst case at x00 level milestone).

#### 6. Extraction Analysis

**Can a player extract more ETH than the pool contains?**

1. **Ticket conversion cannot create more value than ETH input:**
   - `_budgetToTicketUnits` uses integer division (floor). `ticketUnits * (ticketPrice / 4) <= budget`. No over-minting possible.
   - The full lootbox budget goes to nextPrizePool, but ticket face value is based on a reduced budget (50% for purchase-phase, 100% for jackpot-phase). So pool receives >= ticket face value.

2. **Over-collateralization prevents insolvency at level transitions:**
   - Purchase-phase: 2x backing ratio means pool grows 2x faster than ticket obligations.
   - Jackpot-phase: 1x backing (direct conversion) but this is currentPrizePool being consumed by design.
   - Consolidation merges nextPrizePool into currentPrizePool atomically -- no window for inconsistency.

3. **Auto-rebuy bonus is funded by protocol design, not accounting error:**
   - 130% / 145% bonus means player receives 30-45% extra tickets.
   - ETH spent equals `baseTickets * ticketPrice` (no bonus on ETH).
   - Extra tickets are issued against the pool's structural over-collateralization.
   - Net backing after bonus: ~1.38x-1.54x (still above 1x).
   - If every player used auto-rebuy with afKing, the net backing would be 2x / 1.45x = 1.38x, which is still solvent. In practice, not all players use auto-rebuy, so effective backing is higher.

4. **prizePoolFrozen prevents concurrent mutation:**
   - During jackpot phase, `prizePoolFrozen = true` (DegenerusGameStorage:728-729).
   - All inflows during freeze go to `prizePoolPendingPacked` accumulators (DegenerusGame.sol:405-408, WhaleModule:294, MintModule:752).
   - Accumulators are applied atomically at unfreeze (DegenerusGameStorage:736-743).
   - Decimator claims revert during freeze (DecimatorModule:416).
   - Degenerette ETH payouts revert during freeze (DegeneretteModule:685).
   - This ensures no concurrent reads/writes corrupt the pool snapshot that jackpot distribution operates on.

**Conclusion: No extraction path beyond intended amounts exists.** The 2x over-collateralization from lootbox mechanics, combined with prizePoolFrozen guards and correct integer arithmetic, prevents insolvency.
