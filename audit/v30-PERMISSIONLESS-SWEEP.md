## Methodology — Two-Pass Sweep (Pass 1 + Pass 2)

Per D-11 two-pass methodology:

### Pass 1 — Mechanical Grep Discovery (D-11 Pass 1)

**Input:** `contracts/` tree at HEAD `7ab515fe` (excluding `contracts/mocks/` per D-09).

**Procedure:**

1. Mechanical `grep` over `contracts/` (excluding mocks) for `function` declarations with `external` or `public` visibility.
2. Filter out `view` / `pure` declarations (no state mutation per D-09).
3. Filter out interface-style forward declarations (declarations ending in `;` rather than `{` — not implementations; abstract interface declarations embedded in `.sol` files).
4. Filter out functions carrying explicit admin-gate modifiers (`onlyAdmin`, `onlyOwner`, `onlyGovernance`, `onlyVaultOwner`) per D-09.
5. Filter out functions carrying game-internal modifiers (`onlyGame`, `onlyCoinflip`, `onlyCoin`, `onlyVault`, `onlyBurnieCoin`, `onlyDegenerusGameContract`, `onlyFlipCreditors`) per D-09.
6. Filter out functions whose body opens with inline caller-restriction checks equivalent to the above modifier set (e.g., `if (msg.sender != ContractAddresses.GAME) revert`, `if (!vault.isVaultOwner(msg.sender)) revert`, `if (msg.sender != address(this)) revert`) — these are inline-expressed same-class gates per D-09.
7. Filter out functions in `contracts/modules/` that are exclusively delegatecall-invoked targets (module deployment is the delegatecall implementation tree; direct calls into a module operate on the module's own empty storage, making them no-ops without state-mutation effect on protocol; per D-09 "delegatecall-only-invoked").
8. Filter out pure delegatecall wrappers on `DegenerusGame.sol` whose wrapped module-level implementation carries an inherited admin/game-internal gate (msg.sender preserved through delegatecall; the inherited gate applies).

**Output:** Pass 1 candidate universe — the functions that are simultaneously external/public + mutating + player-EOA-callable without any of the above caller restrictions. Each candidate enters Pass 2 for semantic classification.

### Pass 2 — Semantic Classification (D-11 Pass 2)

**Input:** Pass 1 candidate list (61 functions at HEAD `7ab515fe`).

**Procedure (per function):**

1. **Enumerate storage write set.** From the function body (including private helpers invoked synchronously and module-delegatecall callees whose storage writes land in the Game contract's slot space), list every storage slot SSTORE'd on any reachable path.
2. **Touch-check against Phase 237 RNG-consumer state space.** For each write slot, check whether any `INV-237-NNN` consumer reads that slot at fulfillment/consumption time (per `audit/v30-CONSUMER-INVENTORY.md` Per-Consumer Call Graphs).
3. **Classify per D-08:**
   - If the function's execution path reverts via `rngLockedFlag` (direct or inherited through helper/delegatecall) when RNG is locked AND its write set would otherwise touch RNG-consumer state, classify `respects-rngLocked`. Cite the exact `if (rngLockedFlag)` / `rngLocked()` revert site + inherited-gate chain if relevant.
   - If the function operates under a non-flag freeze mechanism proven equivalent to rngLocked for the specific RNG-consumer subset it touches (canonical member: lootbox-index-advance subset — the mid-day lootbox consumer family), classify `respects-equivalent-isolation`. Cite `audit/v30-ASYMMETRY-RE-JUSTIFICATION.md § Asymmetry A` (Plan 239-03 deliverable; forward-cite per D-15).
   - If the function's write set is proven disjoint from the RNG-consumer read set (`WRITE_SET ∩ RNG_CONSUMER_READ_SET = ∅`), classify `proven-orthogonal`. Cite the write-set enumeration + the specific INV-237-NNN consumer read sets checked.
   - Otherwise classify `CANDIDATE_FINDING` with observation routing to Phase 242 FIND-01 per D-23.
4. **Assign Verdict:** `CLASSIFIED_CLEAN` iff Classification ∈ 3-class taxonomy; `CANDIDATE_FINDING` otherwise.

**Output:** Permissionless Sweep Table — one row per Pass 1 candidate with Classification + Verdict per D-08 + D-10.
