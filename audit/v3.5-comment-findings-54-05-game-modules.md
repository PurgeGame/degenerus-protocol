## Prior Finding Re-verification

### CMT-V32-001 (LOW) -- JackpotModule `@return ticketSpent` -- FIXED

**v3.2 finding:** `_resolveTraitWinners` `@return ticketSpent` said "auto-rebuy tickets to next/futurePool" but actual return is whale pass ETH to futurePrizePool only (line 1605).

**v3.5 status:** FIXED. Line 1605 now reads: `/// @return ticketSpent Whale pass ETH routed to futurePrizePool.`

This accurately describes the return value. The `ticketSpent` variable tracks ETH converted via `_processSoloBucketWinner` into whale pass claims added to `futurePrizePool` (line 1788).

### CMT-V32-002 (INFO) -- JackpotModule "BURNIE only" inline -- FIXED

**v3.2 finding:** Inline comment at line 609 said "BURNIE only, no ETH bonuses" but early-burn path distributes ETH on non-day-1 levels.

**v3.5 status:** FIXED. Line 609 now reads: `// Non-daily (early-burn) path - BURNIE and ETH bonuses on non-day-1 levels`

This accurately describes the path behavior. The code at lines 613-634 computes `isEthDay` based on whether `daysSince > 0 && lvl > 1`, and when true, takes a 1% daily drip from `futurePrizePool` (line 630). The comment correctly reflects that ETH bonuses occur on non-day-1 levels.

### CMT-V32-005 (INFO) -- MintModule `@return writesUsed` -- FIXED

**v3.2 finding:** `processFutureTicketBatch` `@return writesUsed` said "queue entries" but returns write-budget units.

**v3.5 status:** FIXED. Line 297 now reads: `/// @return writesUsed Write-budget units consumed (each storage write or skip costs one unit).`

This accurately describes the return semantics. The variable tracks storage write cost (including base overhead and per-ticket write costs), not queue entries processed.

### CMT-V32-006 (INFO) -- DecimatorModule `@return "or expired"` -- FIXED

**v3.2 finding:** `decClaimable` `@return` still referenced "expired" after expiry removal (line 437).

**v3.5 status:** FIXED. Line 437 now reads: `/// @return amountWei Claimable amount (0 if not winner or already claimed).`

The "or expired" text has been removed. The implementation at lines 443-448 only checks `poolWei == 0` (inactive round) and `e.claimed != 0` (already claimed) -- no expiry concept exists.

### DRIFT-V32-001 (LOW) -- GameOverModule `_sendToVault` hard-revert -- FIXED

**v3.2 finding:** `_sendToVault` hard-revert consequence undocumented: stETH failure blocks game-over processing because terminal state flags roll back on revert.

**v3.5 status:** FIXED. Lines 190-196 now read:
```
/// @dev Send funds to vault (50%) and DGNRS (50%), prioritizing stETH transfers over ETH.
///      IMPORTANT: Hard-reverts on stETH/ETH transfer failure. Because game-over
///      sets terminal state flags that roll back on revert, a stuck stETH transfer
///      would block game-over processing until the transfer succeeds.
```

This explicitly documents the hard-revert consequence. Cross-referenced with `audit/KNOWN-ISSUES.md` entry: "_sendToVault uses hard reverts (GO-05-F01)" which documents that vault and sDGNRS have unconditional `receive()` functions and Lido stETH has never paused transfers.

### CMT-104 (INFO) -- DegenerusJackpots.sol `OnlyCoin` -- DEFERRED

**v3.2 finding:** `OnlyCoin` error @notice says "restricted to the coinflip contract" but modifier accepts both COIN and COINFLIP.

**v3.5 note:** DegenerusJackpots.sol is NOT a game module contract -- it is a standalone core contract. This finding will be re-verified in Plan 54-06 (peripheral/remaining contracts).
