## New Findings

### CMT-V32-001: _resolveTraitWinners @return ticketSpent describes wrong conversion type and wrong destination pool

- **What:** The `@return ticketSpent` NatSpec at line 1605 states `ETH converted into auto-rebuy tickets (added to next/futurePool)`. The actual return value maps to `totalWhalePassSpent` (line 1730), which is the ETH cost of whale pass conversions performed by `_processSoloBucketWinner` (solo bucket path only). This ETH is added exclusively to `futurePrizePool` (line 1788: `_setFuturePrizePool(_getFuturePrizePool() + whalePassCost)`), not to `nextPrizePool`. Auto-rebuy ETH in the normal bucket path is routed through `_addClaimableEth` -> `_processAutoRebuy` and is NOT captured in `ticketSpent` at all. The v3.1 fix (CMT-029) replaced the original "loot box awards" text but substituted a different inaccuracy ("auto-rebuy tickets") rather than using the correct suggestion from the v3.1 findings ("whale pass claims").
- **Where:** DegenerusGameJackpotModule.sol:1605
- **Why:** A warden tracing fund flow through `_resolveTraitWinners` would model `ticketSpent` as auto-rebuy ETH split across next/futurePool. The actual flow is whale pass conversion ETH going exclusively to futurePrizePool. This causes incorrect fund flow diagrams: the warden would attribute pool inflows to the wrong mechanism (auto-rebuy vs. whale passes) and the wrong destinations (next+future vs. future only). The caller `_processOneBucket` at line 1580 adds `ticketSpent` to `ctx.totalPaidEth`, so correct understanding of what this value represents is critical for auditing solvency.
- **Suggestion:** Change line 1605 to: `/// @return ticketSpent ETH converted into whale pass claims (added to futurePrizePool).`
- **Category:** comment-inaccuracy
- **Severity:** LOW

### CMT-V32-002: Inline comment says "BURNIE only, no ETH bonuses" but early-burn path distributes ETH on non-day-1 levels

- **What:** The inline section header at line 609 states `// Non-daily (early-burn) path - BURNIE only, no ETH bonuses`. The code immediately below (lines 613-654) checks `isEthDay` and, for levels after day 1, takes a 1% drip from `futurePrizePool` (line 629-633), splits it into a lootbox budget (75% ticket conversion, line 640) and an ETH pool, then runs `_executeJackpot` which distributes actual ETH to winners via `_distributeJackpotEth`. The "BURNIE only" characterization is only accurate for day 1 of each level; on subsequent days, the early-burn path distributes both ETH (from the future pool drip) and triggers BURNIE distribution via the separate `payDailyCoinJackpot` flow. The NatSpec block above (lines 312-318) correctly describes both behaviors, making this inline comment inconsistent with its own function documentation.
- **Where:** DegenerusGameJackpotModule.sol:609
- **Why:** A warden reading through the function body sequentially would see "BURNIE only, no ETH bonuses" and might skip over the ETH distribution logic below, assuming it is dead code or only applies to a different path. The NatSpec above correctly describes the 1% drip, but inline comments are often relied upon for quick code comprehension during audits. The contradiction could cause a warden to file a false positive about undocumented ETH distribution in the early-burn path.
- **Suggestion:** Change line 609 to: `// Non-daily (early-burn) path — ETH from future pool drip (except day 1) + jackpot distribution`
- **Category:** comment-inaccuracy
- **Severity:** INFO
