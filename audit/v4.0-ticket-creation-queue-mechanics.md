## Section 2: Three Key Spaces and Double-Buffer Mechanics (TKT-05)

### Constants

```solidity
// Source: contracts/storage/DegenerusGameStorage.sol
uint24 internal constant TICKET_SLOT_BIT     = 1 << 23;  // GS:154 = 0x800000
uint24 internal constant TICKET_FAR_FUTURE_BIT = 1 << 22; // GS:162 = 0x400000
```

### Key Encoding Functions

#### _tqWriteKey (GS:686-688)

```solidity
function _tqWriteKey(uint24 lvl) internal view returns (uint24) {
    return ticketWriteSlot != 0 ? lvl | TICKET_SLOT_BIT : lvl;
}
```

- **When `ticketWriteSlot = 0`:** Write key = raw level `lvl` (range `0x000000` to `0x3FFFFF`)
- **When `ticketWriteSlot = 1`:** Write key = `lvl | 0x800000` (range `0x800000` to `0xBFFFFF`)
- **Storage dependency:** Reads `ticketWriteSlot` (1 SLOAD)

#### _tqReadKey (GS:691-693)

```solidity
function _tqReadKey(uint24 lvl) internal view returns (uint24) {
    return ticketWriteSlot == 0 ? lvl | TICKET_SLOT_BIT : lvl;
}
```

- **Opposite of write key.** When write = Slot 0, read = Slot 1, and vice versa.
- **When `ticketWriteSlot = 0`:** Read key = `lvl | 0x800000` (Slot 1)
- **When `ticketWriteSlot = 1`:** Read key = raw `lvl` (Slot 0)

#### _tqFarFutureKey (GS:699-701)

```solidity
function _tqFarFutureKey(uint24 lvl) internal pure returns (uint24) {
    return lvl | TICKET_FAR_FUTURE_BIT;
}
```

- **Pure function** -- no storage dependency, no SLOAD
- **Key = `lvl | 0x400000`** (range `0x400000` to `0x7FFFFF`)
- **Independent of ticketWriteSlot** -- FF key space is not double-buffered

### Key Space Collision Analysis

| Key Space | Bit Pattern | Range | Determined By |
|-----------|-------------|-------|---------------|
| Slot 0 | `0b0_0_LLLLLLLLLLLLLLLLLLLLLL` | `0x000000` - `0x3FFFFF` | Raw level (no bits set) |
| Far Future | `0b0_1_LLLLLLLLLLLLLLLLLLLLLL` | `0x400000` - `0x7FFFFF` | Bit 22 set |
| Slot 1 | `0b1_0_LLLLLLLLLLLLLLLLLLLLLL` | `0x800000` - `0xBFFFFF` | Bit 23 set |

**Collision proof:** Bits 22 and 23 are mutually exclusive across the three ranges. For any valid level L (0 to 2^22-1):
- Slot 0: bits [23:22] = `00`
- Far Future: bits [23:22] = `01`
- Slot 1: bits [23:22] = `10`

No two key spaces can produce the same uint24 key for the same level. The pattern `11` (both bits set) is never generated by any key function.

### Double-Buffer Swap Mechanics

#### _swapTicketSlot (GS:709-714)

```solidity
function _swapTicketSlot(uint24 purchaseLevel) internal {
    uint24 rk = _tqReadKey(purchaseLevel);
    if (ticketQueue[rk].length != 0) revert E();
    ticketWriteSlot ^= 1;
    ticketsFullyProcessed = false;
}
```

- **Guard:** Reverts if read-side queue for `purchaseLevel` is not empty (drain must complete first)
- **Effect:** XOR-toggle `ticketWriteSlot` (0->1 or 1->0)
- **Side effect:** Resets `ticketsFullyProcessed = false` (new read slot needs processing)

#### _swapAndFreeze (GS:719-725)

```solidity
function _swapAndFreeze(uint24 purchaseLevel) internal {
    _swapTicketSlot(purchaseLevel);
    if (!prizePoolFrozen) {
        prizePoolFrozen = true;
        prizePoolPendingPacked = 0;
    }
}
```

- **Combines:** Queue swap + prize pool freeze
- **No-op on re-freeze:** If already frozen (jackpot phase), pending accumulators keep growing

#### _unfreezePool (GS:729-736)

```solidity
function _unfreezePool() internal {
    if (!prizePoolFrozen) return;
    (uint128 pNext, uint128 pFuture) = _getPendingPools();
    (uint128 next, uint128 future) = _getPrizePools();
    _setPrizePools(next + pNext, future + pFuture);
    prizePoolPendingPacked = 0;
    prizePoolFrozen = false;
}
```

- **Effect:** Merges pending pools into live pools, clears freeze flag
- **No-op if not frozen**

### Swap Trigger Conditions

| Trigger | Location | Condition | Effect |
|---------|----------|-----------|--------|
| Daily RNG request | `AM:233` | `rngGate()` returns VRF request ID (daily advance) | `_swapAndFreeze(purchaseLevel)` -- swap buffer AND freeze pools |
| Mid-day lootbox RNG | `AM:720` | `ticketQueue[wk].length > 0 && ticketsFullyProcessed` in `requestLootboxRng()` | `_swapTicketSlot(purchaseLevel_)` -- swap buffer only (no pool freeze). Sets `midDayTicketRngPending = true`. |

**Lifecycle:**
1. Level N starts: write buffer active, tickets accumulate
2. Daily advance triggers: `_swapAndFreeze` swaps (old write becomes new read)
3. Read-side drains during advanceGame processing
4. Optional: mid-day lootbox RNG can trigger another swap (`_swapTicketSlot`) if write buffer has new tickets AND read-side is fully processed
5. Next daily advance repeats the cycle

## Section 3: rngLockedFlag and prizePoolFrozen Behavior (TKT-03)

### rngLockedFlag Guard Analysis

The rngLockedFlag guard exists in all three queue helper functions:

```solidity
// GS:545 (_queueTickets):
if (isFarFuture && rngLockedFlag && !phaseTransitionActive) revert RngLocked();

// GS:573 (_queueTicketsScaled):
if (isFarFuture && rngLockedFlag && !phaseTransitionActive) revert RngLocked();

// GS:622 (_queueTicketRange):
if (isFarFuture && rngLockedFlag && !phaseTransitionActive) revert RngLocked();
```

**Three conditions must ALL be true for revert:**
1. `isFarFuture` -- target level > current level + 6
2. `rngLockedFlag` -- VRF commitment window is active
3. `!phaseTransitionActive` -- NOT in advanceGame phase transition

**This means:**
- Near-future writes (target <= level+6) are NEVER blocked by rngLockedFlag
- Far-future writes during VRF window are BLOCKED unless phaseTransitionActive
- Vault perpetual tickets (path 13) are EXEMPTED because they execute during phase transition

### prizePoolFrozen Behavior (TKT-03 supplement)

**prizePoolFrozen does NOT affect ticket creation.** It only affects ETH pool routing:

**Key insight:** `_swapAndFreeze` at `AM:233` does two independent things:
1. Swaps the ticket double-buffer (`_swapTicketSlot`)
2. Freezes the prize pool (`prizePoolFrozen = true`)

These are independent effects. The pool freeze redirects future ETH contributions to pending pools, but does not prevent ticket creation.

### [DISCREPANCY] DSC-01: v3.9 RNG Commitment Window Proof Describes Combined Pool

**File:** `audit/v3.9-rng-commitment-window-proof.md`
**Severity:** INFO (documentation accuracy, not security)

**What the proof says (selected excerpts):**

- Lines 41-42: "`readQueue[idx] OR ffQueue[idx - readLen]`" -- implies dual-source selection

**What the current code does (JM:2543):**

```solidity
address[] storage queue = ticketQueue[_tqFarFutureKey(candidate)];
```

The function reads ONLY from `_tqFarFutureKey`. There is no `_tqReadKey` reference in `_awardFarFutureCoinJackpot`.

**Explanation:** Phase 77 implemented the combined pool approach. Commit `2bf830a2` ("fix: gas cleanup + BURNIE lootbox terminal protection + jackpot FF-only read") reverted the combined pool to FF-only read. The v3.9 RNG commitment window proof (Phase 79) was written after Phase 77 but before the revert, so it documents the combined pool behavior that no longer exists in code.

**Impact:**
- The proof's backward trace and variable inventory are STALE for the combined pool claims
- However, the FF-only approach is a SUBSET of the combined pool (fewer data sources, simpler)
- All protection layers (double-buffer swap, rngLockedFlag guard) described in the proof are still correct for the FF-only case
- The proof is OVERLY CONSERVATIVE (proves safety for more data sources than actually used)
- No security vulnerability from this discrepancy

**Recommendation:** Update the proof to reflect FF-only read behavior. The proof remains valid (it proves a superset of the actual code) but should accurately describe current code to avoid confusion.

### [NEW FINDING] DSC-02: sampleFarFutureTickets Uses _tqWriteKey

**File:** `contracts/DegenerusGame.sol:2669-2705`
**Severity:** INFO (view function correctness -- no state mutation)

**Current code at DG:2681:**

```solidity
address[] storage queue = ticketQueue[_tqWriteKey(candidate)];
```

**Problem:** After v3.9, far-future tickets (targeting > level+6) are routed to `_tqFarFutureKey` key space. The `sampleFarFutureTickets` view function still reads from `_tqWriteKey`, which returns the current write buffer. For candidate levels > level+6, the write buffer will be EMPTY (tickets went to FF key), so this function returns incorrect (empty) results for its intended purpose.

**What it should read:** `_tqFarFutureKey(candidate)` to sample tickets from the far-future key space.

**Impact:**
- View function only -- no on-chain state affected
- Off-chain consumers or UI calling this function will get incorrect results (empty arrays where tickets exist)
- Not exploitable for any security benefit

**Recommendation:** Update `DG:2681` from `_tqWriteKey(candidate)` to `_tqFarFutureKey(candidate)`.

### [POTENTIAL DISCREPANCY] DSC-03: NatSpec Claims Cap, Code Uses Unchecked

**File:** `contracts/storage/DegenerusGameStorage.sol:533` (NatSpec) vs `GS:554` (code)

**NatSpec at GS:533:**
```
/// @dev Queues whole tickets for a buyer at a target level.
///      If buyer has no existing tickets at that level, adds them to the queue.
///      Caps at uint32 max to prevent overflow.
```

**Code at GS:553-555:**
```solidity
unchecked {
    owed += quantity;
}
```

The NatSpec says "Caps at uint32 max to prevent overflow" but the code uses `unchecked` arithmetic with no explicit cap. In practice, uint32 max = 4,294,967,295 tickets. At the minimum ticket price (~0.001 ETH), accumulating this many tickets would require ~4.3M ETH, which exceeds the entire ETH supply. The overflow is infeasible in practice, but the comment incorrectly implies an explicit cap exists.

**Severity:** Potential INFO (comment correctness issue)
**Recommendation:** Update NatSpec to: "Overflow infeasible: uint32 max requires more ETH than exists" or add explicit `if (owed > type(uint32).max - quantity) owed = type(uint32).max;` check to match the comment.

## Appendix: processFutureTicketBatch Dual-Queue Drain (Reference)

The `processFutureTicketBatch` function at `MM:298-339` implements the v3.9 dual-queue drain:

1. **Read-side first:** Drains `ticketQueue[_tqReadKey(lvl)]`
2. **Transition to FF:** When read-side is empty, checks `ticketQueue[_tqFarFutureKey(lvl)]`
3. **FF drain:** Sets `ticketLevel = lvl | TICKET_FAR_FUTURE_BIT` to track cursor state
4. **Completion:** Returns `finished = true` only when both queues are empty

The FF-bit encoding in `ticketLevel` (using `TICKET_FAR_FUTURE_BIT`) provides unambiguous cursor state tracking. When `ticketLevel` has the FF bit set, the function knows to use `_tqFarFutureKey` as the queue key.
