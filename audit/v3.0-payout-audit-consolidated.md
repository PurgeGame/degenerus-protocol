## Research Open Questions Resolution

All 5 open questions from 27-RESEARCH.md have been resolved across the five partial audit reports.

### Q1: claimAffiliateDgnrs Implementation vs v1.1 Doc

**Resolution: Code uses fixed allocation (proportional), not sequential depletion. Classified as FINDING-INFO (stale documentation).**
**Resolved in:** v3.0-payout-lootbox-quest-affiliate.md (PAY-11 Section 4)

The v1.1 affiliate system doc describes sequential pool depletion: "5% of affiliate pool per level (not reserved -- sequential depletion)." The actual code at DegenerusGame:1458-1501 uses `levelDgnrsAllocation[currLevel]` (snapshot at level transition) with `totalAffiliateScore[currLevel]` as denominator. This eliminates first-mover advantage entirely. The storage natspec explicitly confirms: "eliminating first-mover advantage."

### Q2: _addClaimableEth Auto-Rebuy Variants Consistency

**Resolution: All implementations use the same _calcAutoRebuy from PayoutUtils. Consistent across all modules.**
**Resolved in:** v3.0-payout-jackpot-distribution.md (Shared Infrastructure section), v3.0-payout-scatter-decimator.md (PAY-03 Auto-Rebuy Interaction)

Four modules have `_addClaimableEth` implementations: JackpotModule:973, EndgameModule:241, DecimatorModule:510, DegeneretteModule:1153. All implementations:
- Check `gameOver` flag to skip auto-rebuy during GAMEOVER
- Call `_calcAutoRebuy` from PayoutUtils with bonusBps=13000 (130%) and bonusBpsAfKing=14500 (145%)
- Return `calc.reserved` (take-profit portion) as the claimable delta when auto-rebuy is active
- Fall through to `_creditClaimable(beneficiary, weiAmount)` when auto-rebuy is disabled

The EndgameModule variant (line 275) additionally handles `calc.reserved` inline with `claimablePool += calc.reserved`, which is correct because BAF does not use the batched liability pattern -- it tracks `claimableDelta` separately. All variants produce consistent claimablePool accounting.

### Q3: Lootbox Module Complexity

**Resolution: All 5 lootbox reward types mapped with correct pool accounting. Only whale pass remainder mutates claimablePool.**
**Resolved in:** v3.0-payout-lootbox-quest-affiliate.md (PAY-09 Sections 1-7)

The 1,778-line LootboxModule distributes 5 reward types: whale passes (claimablePool remainder), lazy pass discount boons (storage flags), deity pass discount boons (storage flags), future tickets (ticket queue), and BURNIE (coinflip credit). Only the whale pass remainder path touches claimablePool. All other paths operate in token domains with no ETH pool impact. The 2x over-collateralization for ticket conversion is inherited from the jackpot distribution system (PAY-16).

### Q4: claimCoinflips vs claimCoinflipsFromBurnie Routing

**Resolution: Both paths route to identical _claimCoinflipsInternal with only access control differing.**
**Resolved in:** v3.0-payout-coinflip-economy.md (PAY-07 Section 5)

`claimCoinflips(player, amount)` uses `_resolvePlayer` for caller resolution. `claimCoinflipsFromBurnie(player, amount)` uses `onlyBurnieCoin` modifier and passes `player` directly. Both call `_claimCoinflipsAmount(player, amount, true)` with `mintTokens=true`, routing to the same `_claimCoinflipsInternal`. A third path, `consumeCoinflipsForBurn`, uses `mintTokens=false` for burn operations. All paths have identical RNG lock protection.

### Q5: Yield Distribution Trigger Timing

**Resolution: Yield surplus formula is rate-independent. Fires exactly once per level transition.**
**Resolved in:** v3.0-payout-yield-burns.md (PAY-12 Sections 12.1, 12.4)

The surplus formula `yieldSurplus = (ETH + stETH) - (currentPrizePool + nextPrizePool + claimablePool + futurePrizePool + yieldAccumulator)` depends only on how much stETH has appreciated since the last distribution, not on when the check occurs. The `poolConsolidationDone` flag ensures it fires exactly once per level transition. No external function calls `_distributeYieldSurplus` directly -- it is private to JackpotModule and only reachable through the delegatecall from `_consolidatePrizePools`.

### 2. Auto-Rebuy During GAMEOVER vs Normal Play

**Phase 27 finding:** During normal play, `_addClaimableEth` routes to `_processAutoRebuy` when `autoRebuyState[beneficiary].autoRebuyEnabled` is true, correctly splitting between ticket conversion and take-profit (PAY-01, PAY-02, PAY-03, PAY-04, PAY-05, PAY-06, PAY-12).

**Result: CONSISTENT.** Auto-rebuy suppression during GAMEOVER and activation during normal play are correctly controlled by the same flag in the same function.
