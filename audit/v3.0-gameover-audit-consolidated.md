### claimablePool Invariant: Full handleGameOverDrain Trace

This trace consolidates the step-by-step proof from core-distribution GO-01 Section 7, verified against safety-properties GO-06 Steps 1-10:

| Step | Action | claimablePool State | Balance State | Invariant |
|------|--------|-------------------|---------------|-----------|
| Start | Entry to handleGameOverDrain | CP_0 (from normal gameplay) | totalFunds >= CP_0 | HOLDS |
| 1 | Deity refunds (lines 78-107) | CP_0 + totalRefunded | Unchanged | HOLDS (totalRefunded <= totalFunds - CP_0) |
| 2 | Terminal state (lines 112-132) | Unchanged | Unchanged | HOLDS |
| 3 | Terminal decimator (lines 137-147) | CP_0 + totalRefunded + decSpend | Unchanged | HOLDS (decSpend <= available) |
| 4 | Terminal jackpot (lines 149-155) | CP_0 + totalRefunded + decSpend + liabilityDelta | Unchanged | HOLDS (liabilityDelta <= termPaid <= remaining) |
| 5 | Vault sweep (lines 157-159) | Unchanged | Decreases by (remaining - termPaid) | HOLDS (proven algebraically in core-distribution GO-01 S7) |
| 6 | sDGNRS burn (line 163) | Unchanged | Unchanged (no ETH/stETH transferred) | HOLDS |

**Algebraic proof (Step 5):** After Step 4, `claimablePool = CP_0 + totalRefunded + decSpend + liabilityDelta`. The vault sweep removes `(remaining - termPaid)` from the balance. We need: `balance_after >= claimablePool_after`, i.e., `totalFunds - (remaining - termPaid) >= CP_0 + totalRefunded + decSpend + liabilityDelta`. Since `remaining = available - decSpend` and `available = totalFunds - CP_0 - totalRefunded`, substituting: `totalFunds - ((totalFunds - CP_0 - totalRefunded - decSpend) - termPaid) >= CP_0 + totalRefunded + decSpend + liabilityDelta`, which simplifies to `CP_0 + totalRefunded + decSpend + termPaid >= CP_0 + totalRefunded + decSpend + liabilityDelta`, i.e., `termPaid >= liabilityDelta`. This is guaranteed because `liabilityDelta` is the net claimable credits from `_distributeJackpotEth` and `termPaid` is the total ETH distributed (including auto-rebuy conversions, which are zero during GAMEOVER since `gameOver = true` skips auto-rebuy). QED.

## Open Questions Resolution Summary

All 5 open questions from 26-RESEARCH.md have been resolved across the three partial audit reports.

### Q1: decBucketOffsetPacked Collision Between Normal and Terminal Decimator

**Resolution: No collision possible -- PASS**
**Resolved in:** core-distribution (GO-08 Section 1, "decBucketOffsetPacked write")

Both `runDecimatorJackpot` (DecimatorModule:341) and `runTerminalDecimatorJackpot` (DecimatorModule:914) write to `decBucketOffsetPacked[lvl]`. Collision is impossible because:
- Normal decimator fires during level completion milestones (level DID complete)
- Terminal decimator fires during GAMEOVER (level did NOT complete -- death clock expired)
- These are mutually exclusive states for the same level number
- If normal decimator resolved level X, the game advanced past X, so GAMEOVER fires at level X+n (n >= 1)

### Q2: stBal Staleness in handleGameOverDrain

**Resolution: Not stale -- PASS**
**Resolved in:** core-distribution (GO-01 Section 5, "stBal staleness concern")

`stBal` is read at GameOverModule:75 and used at line 158. Between these points, two delegatecall modules execute (terminal decimator at line 140, terminal jackpot at lines 152-153). Neither module transfers stETH -- they only update storage variables. The only stETH transfers in the codebase are in `_sendToVault` itself. Lido rebase delta during a single transaction is negligible (~1 wei per ETH per hour), and `_sendToVault` handles insufficient stETH by supplementing with ETH.

### Q3: Unchecked Arithmetic in Deity Refund Loop

**Resolution: Cannot overflow -- PASS**
**Resolved in:** ancillary-paths (GO-07 Section 6), core-distribution (GO-01 Section 2, Q3 resolution)

Both reports independently verify the unchecked block at GameOverModule:92-96:
- `claimableWinnings[owner] += refund`: Overflow requires ~5.7e57 ETH (physically impossible)
- `totalRefunded += refund`: Bounded by `budget_initial <= totalFunds`
- `budget -= refund`: `refund <= budget` enforced by cap at lines 88-89

### Q4: Terminal Decimator Claim Expiry / lastTerminalDecClaimRound Overwrite

**Resolution: Cannot be overwritten after GAMEOVER -- PASS**
**Resolved in:** core-distribution (GO-08 Section 3, "lastTerminalDecClaimRound overwrite protection")

`lastTerminalDecClaimRound` is only written by `runTerminalDecimatorJackpot` (line 917-919), which is only callable from `handleGameOverDrain` (line 140). The `gameOverFinalJackpotPaid` latch (set at line 128) prevents any subsequent call from reaching the terminal decimator dispatch. Defense-in-depth: the double-resolution guard at DecimatorModule:888 would also return early. Terminal decimator claims remain valid for the entire 30-day window.

### Q5: _processAutoRebuy During Terminal Decimator Claims

**Resolution: Correctly skipped -- PASS**
**Resolved in:** core-distribution (GO-08 Section 3, "_addClaimableEth routing")

`claimTerminalDecimatorJackpot()` calls `_addClaimableEth` (DecimatorModule:510), which calls `_processAutoRebuy` (DecimatorModule:458), which checks `if (gameOver) return false;` (line 463). Since `gameOver = true` post-GAMEOVER, auto-rebuy is skipped and the flow falls through to `_creditClaimable`. Solidity resolves the `private` `_addClaimableEth` within DecimatorModule's bytecode -- the EndgameModule version (line 241) is not called.

## Findings Summary

### FINDING-01: _sendToVault Hard Reverts Can Block Terminal Distribution

- **ID:** GO-05-F01
- **Severity:** Medium
- **Affected Requirement:** GO-05 (Revert Safety)
- **Description:** `_sendToVault` (GameOverModule:195-231) uses `revert E()` on any ETH or stETH transfer failure. If the vault or sDGNRS contract cannot receive funds, both `handleGameOverDrain` and `handleFinalSweep` revert permanently, stranding all terminal distribution.
- **Files:** GameOverModule.sol:201, 205, 210-211, 218, 219, 228-229 (7 dangerous revert sites)
- **Mitigation:** Vault and sDGNRS are immutable protocol-owned contracts with simple, unconditional `receive()` functions. sDGNRS's `receive()` has an `onlyGame` modifier permanently satisfied. Lido stETH has never paused transfers.
- **Recommendation:** Consider wrapping `_sendToVault` in try/catch or using a pull-based pattern for vault sweep. Alternatively, accept as known risk given immutable recipients.

### FINDING-INFO-01: Stale Test Comments

- **ID:** GO-03-I01
- **Severity:** Informational
- **Affected Requirement:** GO-03 (Death Clock Triggers)
- **Description:** Test file `test/edge/GameOver.test.js` references 912-day timeout and 365-day inactivity thresholds in comments and constants, but code uses 365 days (level 0) and 120 days (level 1+). Tests pass because they overshoot the actual thresholds.
- **Files:** test/edge/GameOver.test.js (lines 29, 30, 46, 104, 182-185)

## Key Design Decisions Documented

These are intentional design behaviors that C4A wardens should be aware of (not bugs):

### 1. Level Aliasing at Level 0 (BY DESIGN)

**Location:** GameOverModule.sol:72
**Behavior:** When GAMEOVER fires at level 0, `lvl` is aliased from 0 to 1, causing the terminal jackpot at line 153 to target `lvl + 1 = 2` (ticket holders for level 2, not level 1).
**Rationale:** At level 0, no tickets exist for level 1, so the terminal jackpot targets level 2 where pre-sale or early-bird tickets may have been purchased.
**Confirmed in:** core-distribution (GO-01 Section 1), economics specification.

### 2. No-Latch Retry on Missing RNG (BY DESIGN)

**Location:** GameOverModule.sol:126
**Behavior:** `if (rngWord == 0) return;` does NOT set `gameOverFinalJackpotPaid`, allowing `handleGameOverDrain` to be called again on the next `advanceGame` when RNG becomes available.
**Rationale:** Defensive mechanism ensuring GAMEOVER can complete once RNG arrives. In current architecture, this path is unreachable (dead code) because `_handleGameOverPath` ensures `rngWordByDay[day]` is non-zero before calling `handleGameOverDrain`.
**Confirmed in:** safety-properties (GO-09 "No-Latch Retry" section).

### 3. Terminal Decimator Time Multiplier Discontinuity (BY DESIGN)

**Location:** DecimatorModule.sol:1000-1006
**Behavior:** At `daysRemaining = 11` the multiplier is 2.75x (27,500 BPS). At `daysRemaining = 10` it drops to 2.0x (20,000 BPS). This is an intentional regime-change signal.
**Rationale:** Documented in code comments and PLAN-TERMINAL-DECIMATOR.md. The two-regime system incentivizes early burns (>10 days: linear with day count) while providing a flatter multiplier curve in the final 10 days.
**Confirmed in:** core-distribution (GO-08 Section 4, boundary value table).

### 4. 30-Day Claim Window Forfeiture (BY DESIGN)

**Location:** GameOverModule.sol:173, 177
**Behavior:** After 30 days post-GAMEOVER, `handleFinalSweep` sets `claimablePool = 0` and sweeps all remaining balance to vault/sDGNRS. Players who did not claim within 30 days permanently lose their winnings.
**Rationale:** Intentional forfeiture policy applied uniformly to all claimable winnings including deity refunds, terminal decimator claims, and terminal jackpot credits.
**Confirmed in:** ancillary-paths (GO-02 Section 3, Section 6).

### 5. Stale Test Comments (INFORMATIONAL, DEFER)

**Location:** test/edge/GameOver.test.js
**Behavior:** Comments reference 912-day timeout (level 0) and 365-day inactivity (level 1+), but actual code uses 365 days and 120 days respectively. Test assertions pass correctly by overshooting.
**Recommendation:** Fix in Phase 29 (Comment/Documentation Correctness). Tests are functionally correct despite misleading comments.
**Confirmed in:** ancillary-paths (GO-03 Section 5, "Stale Test Comments -- FINDING-INFO").
