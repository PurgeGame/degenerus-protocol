## Section 1: Ticket Creation End-to-End Trace

This section traces the complete lifecycle of a ticket from purchase through trait assignment, identifying the entropy source (or lack thereof) at every step.

### 1a: Purchase to Write Buffer

**Entry points.** Tickets enter the system through purchase functions that ultimately call `_queueTickets` or `_queueTicketsScaled`:

| Entry Point | Module | Queuing Function Called |
|---|---|---|
| `purchase` / `purchaseCoin` | AdvanceModule | `_queueTickets` (whole) or `_queueTicketsScaled` (fractional) |
| `purchaseWhaleBundle` | WhaleModule | `_queueTicketRange` (multi-level) |
| `purchaseLazyPass` | WhaleModule | `_queueTicketRange` (multi-level) |
| `purchaseDeityPass` | WhaleModule | `_queueTickets` (whole) |
| `purchaseBurnieLootbox` | AdvanceModule | `_queueTicketsScaled` (fractional) |

**`_queueTickets` (Storage:545-571).** The core queuing function:

1. Computes the write buffer key: `uint24 wk = _tqWriteKey(targetLevel)` (Storage:552).
2. `_tqWriteKey` returns `level | TICKET_SLOT_BIT` when `ticketWriteSlot != 0`, else `level` (Storage:717-718). `TICKET_SLOT_BIT = 1 << 23` (Storage:154).
3. Reads the player's existing packed entry: `uint40 packed = ticketsOwedPacked[wk][buyer]` (Storage:553).
4. If the player has no existing entry (`owed == 0 && rem == 0`), appends the player address to `ticketQueue[wk]` (Storage:556-558).
5. Adds `quantity` to the existing `owed` count, capping at `type(uint32).max` to prevent overflow (Storage:559-565).
6. Writes the updated packed value: `(uint40(uint32(newOwed)) << 8) | uint40(rem)` (Storage:567-569).

**Packing format:** `uint40 = (uint32 owed << 8) | (uint8 remainder)`. The upper 32 bits store whole tickets owed; the lower 8 bits store fractional remainder (used by `_queueTicketsScaled`).

**`_queueTicketsScaled` (Storage:578-627).** Handles fractional ticket quantities (scaled by `TICKET_SCALE = 100`, Storage:129). Separates the input into whole tickets and fractional remainder. When accumulated remainder reaches `TICKET_SCALE`, promotes one whole ticket. Same write-buffer mechanics as `_queueTickets`.

**Entropy source at this step: NONE.** Ticket queuing is entirely deterministic. The write buffer key is derived from level and `ticketWriteSlot` (a boolean toggle). No random values are read or consumed. The queue is an append-only array of player addresses with packed ticket counts.

### 1b: Buffer Swap -- Write to Read

The double-buffer mechanism uses two slots (toggled by `ticketWriteSlot`) so that purchases can continue while previously queued tickets are being processed.

**`_swapTicketSlot` (Storage:732-737):**

```
function _swapTicketSlot(uint24 purchaseLevel) internal {
    uint24 rk = _tqReadKey(purchaseLevel);
    if (ticketQueue[rk].length != 0) revert E();  // Read buffer must be fully drained
    ticketWriteSlot ^= 1;                          // Toggle active write slot
    ticketsFullyProcessed = false;                 // Mark new read buffer as unprocessed
}
```

The assert at Storage:734 enforces that the read buffer must be completely drained before a swap can occur. This prevents interleaving of two batches.

**`_swapAndFreeze` (Storage:742-748):**

```
function _swapAndFreeze(uint24 purchaseLevel) internal {
    _swapTicketSlot(purchaseLevel);
    if (!prizePoolFrozen) {
        prizePoolFrozen = true;
        prizePoolPendingPacked = 0;
    }
}
```

Called during daily `advanceGame` to simultaneously swap the ticket buffer and freeze the prize pool for jackpot draws.

**Two swap triggers:**

1. **Daily path:** `_swapAndFreeze` called within `advanceGame` when advancing to a new day. Freezes the pool and swaps the buffer in a single atomic operation.
2. **Mid-day path:** `_swapTicketSlot` called within `requestLootboxRng` (AdvanceModule:708-715) when the write buffer has pending tickets and the read buffer has been fully processed. Sets `midDayTicketRngPending = true` to signal that a VRF callback is needed before processing can begin.

The mid-day swap trigger at AdvanceModule:713-715:
```
if (ticketQueue[wk].length > 0 && ticketsFullyProcessed) {
    _swapTicketSlot(purchaseLevel_);
    midDayTicketRngPending = true;
}
```

**Entropy source at this step: NONE.** The buffer swap is a mechanical toggle (`ticketWriteSlot ^= 1`). No random values are involved. After the swap, the former write buffer becomes the read buffer and is frozen -- no further modifications are possible.

### 1c: Read Buffer to Trait Assignment

After the buffer swap, `processTicketBatch` (JackpotModule:1949-2010) processes the frozen read buffer to assign traits to each queued ticket.

**Step 1: Read entropy.** At JackpotModule:1975:
```
uint256 entropy = lastLootboxRngWord;
```
This is the single entropy source for the entire batch. `lastLootboxRngWord` is a VRF-derived value set either via the mid-day drain path (AdvanceModule:166) or the piggyback path (`_finalizeLootboxRng` at AdvanceModule:789). See Phase 12 `v1.2-rng-data-flow.md` Section 3 for the complete mid-day ticket RNG lifecycle.

**Step 2: Iterate queue.** The function reads `ticketQueue[readKey]` (JackpotModule:1951) and iterates through players using a cursor (`ticketCursor`, JackpotModule:1960) with a gas budget (`WRITES_BUDGET_SAFE = 550`, JackpotModule:1969). Multiple `processTicketBatch` calls may be needed to drain the full queue.

**Step 3: Per-player processing.** For each player, `_processOneTicketEntry` (JackpotModule:2044-2107) constructs a per-player salt:
```
uint256 rollSalt = (uint256(lvl) << 224) |
    (queueIdx << 192) |
    (uint256(uint160(player)) << 32);
```
This salt encodes the level (bits 255-224), queue index (bits 223-192), and player address (bits 191-32). It ensures each player at each queue position gets unique entropy derivation even though they share the same `lastLootboxRngWord`.

**Step 4: Trait generation.** `_generateTicketBatch` (JackpotModule:2110-2130) constructs the baseKey:
```
uint256 baseKey = (uint256(lvl) << 224) |
    (queueIdx << 192) |
    (uint256(uint160(player)) << 32);
```
Then calls `_raritySymbolBatch` (JackpotModule:2187-2281).

**Step 5: LCG trait derivation.** `_raritySymbolBatch` processes tickets in groups of 16. For each group:

1. **Seed derivation:** `seed = (baseKey + groupIdx) ^ entropyWord` (JackpotModule:2211), where `groupIdx = i >> 4` (groups of 16 tickets).
2. **LCG initialization:** `s = uint64(seed) | 1` (JackpotModule:2213) -- ensures odd for full-period LCG.
3. **Offset warmup:** `s = s * (TICKET_LCG_MULT + uint64(offset)) + uint64(offset)` (JackpotModule:2216) -- fast-forwards the LCG to the correct starting position within a group.
4. **LCG stepping:** `s = s * TICKET_LCG_MULT + 1` (JackpotModule:2221) -- standard Knuth LCG with multiplier `TICKET_LCG_MULT = 0x5851F42D4C957F2D` (= 6364136223846793005, JackpotModule:176).
5. **Trait derivation:** `DegenerusTraitUtils.traitFromWord(s)` (JackpotModule:2224) splits the 64-bit LCG state into two 32-bit halves for weighted category and sub-bucket selection (TraitUtils:143-149), producing a 6-bit trait ID (bits 5-3: category, bits 2-0: sub-bucket).
6. **Quadrant offset:** `+ (uint8(i & 3) << 6)` (JackpotModule:2225) adds a 2-bit quadrant to the trait ID, producing a final 8-bit `traitId` that indexes into 256 trait buckets.

The generated traits are counted per-traitId in memory, then batch-written to `traitBurnTicket[lvl][traitId]` storage via inline assembly (JackpotModule:2254-2281).

**Entropy source at this step: `lastLootboxRngWord` (VRF-derived).** This is the ONLY point in the entire ticket lifecycle where randomness enters the system. The LCG expands this single 256-bit VRF word into per-ticket trait assignments via deterministic derivation keyed by `(level, queueIndex, playerAddress, groupIndex)`.

### 1d: Entropy Source Summary Table

| Step | Function(s) | Contract Location | Entropy Source | Notes |
|---|---|---|---|---|
| Purchase | `buyTickets`, `purchaseDeityPass`, etc. | Various entry points | **None** | Deterministic ticket count based on payment |
| Queue | `_queueTickets` / `_queueTicketsScaled` | Storage:545-627 | **None** | Append-only write to write buffer; packed uint40 accounting |
| Buffer Swap | `_swapTicketSlot` / `_swapAndFreeze` | Storage:732-748 | **None** | Mechanical toggle of `ticketWriteSlot ^= 1`; freezes read buffer |
| Batch Init | `processTicketBatch` | JackpotModule:1949-1975 | **`lastLootboxRngWord`** | Single VRF-derived word read once; shared across entire batch |
| Trait Assign | `_raritySymbolBatch` | JackpotModule:2187-2281 | **LCG from `lastLootboxRngWord`** | Per-ticket derivation via `(baseKey + groupIdx) ^ entropyWord` seeded LCG |

**Key finding:** Entropy enters the ticket lifecycle at exactly one point -- when `processTicketBatch` reads `lastLootboxRngWord` at JackpotModule:1975. All upstream steps (purchase, queuing, buffer swap) are fully deterministic. All downstream steps (LCG expansion, trait derivation) are deterministic transformations of this single entropy source.

Phase 14 L4 analysis confirms this consumption point is **BLOCKED** from manipulation: the read buffer is frozen by the double-buffer swap, entropy is VRF-derived, and processing occurs atomically within `advanceGame` (see `v1.2-manipulation-windows.md` Section 1b, L4 verdict).

## Section 2: Mid-Day RNG Flow -- Trigger, Callback, Drain, and Manipulation Resistance

This section traces the complete mid-day RNG flow from `requestLootboxRng` through VRF callback to `processTicketBatch` consumption, then provides explicit manipulation resistance reasoning.

### 2a: requestLootboxRng Trigger and Buffer Swap

**`requestLootboxRng` (AdvanceModule:673-735)** is an externally callable function (callable by anyone) that initiates a mid-day VRF request for ticket trait assignment. Its preconditions and flow:

**Preconditions (AdvanceModule:674-706):**

1. `rngLockedFlag` must be false (AdvanceModule:674) -- no daily VRF in-flight.
2. Not within 15 minutes of day reset (AdvanceModule:680) -- avoids competing with the daily jackpot RNG flow.
3. Today's daily RNG word must already be recorded: `rngWordByDay[currentDay] != 0` (AdvanceModule:682).
4. No other VRF request in-flight: `rngRequestTime != 0` reverts (AdvanceModule:684).
5. Sufficient LINK balance: `linkBal < MIN_LINK_FOR_LOOTBOX_RNG` reverts (AdvanceModule:688).
6. Pending lootbox purchases exist: `pendingEth` or `pendingBurnie` must be nonzero (AdvanceModule:693), with an ETH-equivalent threshold check against `lootboxRngThreshold` when `pendingBurnie < BURNIE_RNG_TRIGGER` (AdvanceModule:694-706).

**Atomic buffer swap (AdvanceModule:708-717):**

```
uint24 purchaseLevel_ = level + 1;
uint24 wk = _tqWriteKey(purchaseLevel_);
if (ticketQueue[wk].length > 0 && ticketsFullyProcessed) {
    _swapTicketSlot(purchaseLevel_);
    midDayTicketRngPending = true;
}
```

The swap fires when two conditions hold simultaneously:
1. The write buffer has pending tickets: `ticketQueue[wk].length > 0` (AdvanceModule:713).
2. The read buffer has been fully drained: `ticketsFullyProcessed == true` (AdvanceModule:713).

`_swapTicketSlot` (Storage:732-737) atomically toggles `ticketWriteSlot ^= 1` and sets `ticketsFullyProcessed = false`. After this toggle, any tickets purchased subsequent to this call are written to the NEW write buffer (the former read slot), which will NOT be resolved by the incoming VRF word.

**VRF request (AdvanceModule:719-729):**

After the buffer swap, the function sends a VRF request to the Chainlink coordinator with `VRF_MIDDAY_CONFIRMATIONS = 3` (AdvanceModule:724). It then calls `_reserveLootboxRngIndex(id)` (AdvanceModule:731) to allocate a lootbox index for the response, sets `vrfRequestId = id` (AdvanceModule:732), clears `rngWordCurrent = 0` (AdvanceModule:733), and records `rngRequestTime` (AdvanceModule:734).

**Key structural point:** The buffer swap at AdvanceModule:713-715 occurs BEFORE the VRF request at AdvanceModule:720-729. This ordering is critical -- it means the set of tickets that will be resolved by the incoming VRF word is frozen before the entropy request is even sent. No actor can add, remove, or reorder tickets in the read buffer after `_swapTicketSlot` executes.

### 2b: VRF Callback Routing (Lootbox Path)

**`rawFulfillRandomWords` (AdvanceModule:1326-1347)** is the VRF callback invoked by the Chainlink coordinator.

**Access control:** Only the VRF coordinator can call this function: `if (msg.sender != address(vrfCoordinator)) revert E()` (AdvanceModule:1330). Additionally, the callback validates that `requestId == vrfRequestId` and `rngWordCurrent == 0` (AdvanceModule:1331); mismatched or duplicate callbacks are silently dropped.

**Path routing (AdvanceModule:1336-1346):** The callback distinguishes daily vs lootbox paths using `rngLockedFlag`:

| Path | Condition | Action |
|---|---|---|
| Daily RNG | `rngLockedFlag == true` | Stores word to `rngWordCurrent` for later `advanceGame` processing (AdvanceModule:1338) |
| Lootbox (mid-day) RNG | `rngLockedFlag == false` | Looks up `lootboxRngRequestIndexById[requestId]` to get the index (AdvanceModule:1341), writes `lootboxRngWordByIndex[index] = word` (AdvanceModule:1342), emits `LootboxRngApplied`, and clears `vrfRequestId = 0` and `rngRequestTime = 0` (AdvanceModule:1344-1345) |

**Zero-word guard:** If the VRF returns 0, it is replaced with 1: `if (word == 0) word = 1` (AdvanceModule:1334). This prevents a zero word from being confused with "no word received" in downstream logic.

**Critical observation:** The lootbox path callback does NOT immediately trigger `processTicketBatch`. It only writes the word to `lootboxRngWordByIndex[index]` and clears VRF request state. The word sits in storage until `advanceGame` reads it. This means the callback block's builder/proposer cannot influence trait assignment by manipulating callback transaction ordering -- trait assignment happens in a separate, future `advanceGame` transaction.

### 2c: advanceGame Mid-Day Drain

**`advanceGame` mid-day path (AdvanceModule:158-184)** handles same-day ticket drain when `day == dailyIdx` (AdvanceModule:158).

**Step-by-step flow:**

1. **Check drain needed:** `!ticketsFullyProcessed` (AdvanceModule:160) -- if tickets remain in the read buffer.

2. **Check VRF pending:** `midDayTicketRngPending` (AdvanceModule:162) -- if a mid-day VRF was requested.

3. **Read VRF word:** `uint256 word = lootboxRngWordByIndex[lootboxRngIndex - 1]` (AdvanceModule:163). If `word == 0`, the VRF callback has not yet delivered -- reverts with `NotTimeYet()` (AdvanceModule:164). This is the blocking wait: no processing until VRF arrives.

4. **Write entropy:** `lastLootboxRngWord = word` (AdvanceModule:166). This copies the VRF word from the indexed storage to the consumption variable that `processTicketBatch` reads.

5. **Process batch:** `_runProcessTicketBatch(purchaseLevel)` (AdvanceModule:171) invokes `processTicketBatch` which reads `lastLootboxRngWord` (JackpotModule:1975) and processes tickets from the frozen read buffer. This may require multiple `advanceGame` calls if the queue is large (gas budget: `WRITES_BUDGET_SAFE = 550` per call).

6. **Mark complete:** When the read buffer is fully drained, `ticketsFullyProcessed = true` (AdvanceModule:178).

7. **Clear pending flag:** `midDayTicketRngPending = false` (AdvanceModule:183) -- signals that the mid-day VRF flow is complete. This allows a subsequent `requestLootboxRng` call to initiate a new cycle.

**Piggyback path (AdvanceModule:785-791):** During daily `advanceGame` processing (new-day path), `_finalizeLootboxRng(rngWord)` writes the daily VRF word directly to both `lootboxRngWordByIndex[index]` and `lastLootboxRngWord` (AdvanceModule:789). This handles the case where tickets were swapped during the daily path via `_swapAndFreeze` and need a VRF word without a separate mid-day request.

### 2d: Manipulation Resistance Analysis

The mid-day RNG flow provides manipulation resistance through five structural properties:

**1. Buffer swap is atomic with VRF request -- no window to modify tickets after swap but before VRF.**

`_swapTicketSlot` at AdvanceModule:713-715 executes within the same `requestLootboxRng` transaction as the VRF request at AdvanceModule:720-729. After the swap, `ticketWriteSlot` is toggled (Storage:735), so all subsequent `_queueTickets` calls write to the new write buffer. No code path exists to modify the now-frozen read buffer. The swap-to-VRF-request sequence is atomic within a single transaction -- there is no inter-block gap where an attacker could observe the swap and inject tickets before the VRF request is sent.

**2. VRF word is cryptographically unpredictable at swap time.**

The VRF request is sent with `VRF_MIDDAY_CONFIRMATIONS = 3` (AdvanceModule:724). The Chainlink VRF v2.5 proof is verified on-chain by the coordinator (`msg.sender != address(vrfCoordinator)` check at AdvanceModule:1330). At the time of the buffer swap, the VRF output is unknown -- it depends on the block hash of a future block (the request block + confirmation blocks). No actor can predict or influence the VRF output at the time tickets are committed to the read buffer.

**3. VRF callback only writes to storage -- no trait assignment in callback.**

The lootbox callback path (AdvanceModule:1340-1345) writes `lootboxRngWordByIndex[index] = word`, emits an event, and clears request state. It does NOT call `processTicketBatch` or any trait assignment logic. Therefore, the block builder who includes the VRF callback transaction cannot influence trait outcomes by manipulating transaction ordering within the callback block -- the actual trait assignment happens in a separate `advanceGame` call (AdvanceModule:171).

**4. processTicketBatch reads from frozen read buffer -- immutable between VRF arrival and consumption.**

`processTicketBatch` (JackpotModule:1949-2010) reads `ticketQueue[readKey]` which is the frozen read buffer. After `_swapTicketSlot` toggles `ticketWriteSlot`, the read buffer cannot be modified:
- No code writes to the read buffer key after swap (all `_queueTickets` calls use `_tqWriteKey` which returns the opposite slot).
- No removal, reorder, or modification functions exist for `ticketQueue` arrays.
- The only mutation is `processTicketBatch` itself consuming entries sequentially (JackpotModule:2003-2004 deletes the array when fully drained).

**5. Phase 14 L4 BLOCKED verdict provides supporting evidence.**

Phase 14 `v1.2-manipulation-windows.md` Section 1b analyzed the `processTicketBatch` manipulation window (L4) and issued a **BLOCKED** verdict: the double-buffer mechanism structurally prevents any actor from modifying the inputs (queue composition or entropy word) to `processTicketBatch` after the buffer swap. The mid-day flow uses the same `processTicketBatch` function and the same frozen read buffer -- the L4 verdict applies equally to mid-day processing.

**Verdict: SAFE.**

The mid-day RNG flow provides manipulation resistance equivalent to the daily flow through:
- **Structural isolation:** Buffer swap before VRF request ensures ticket set is committed before entropy is known.
- **VRF unpredictability:** Chainlink VRF v2.5 proof verification guarantees cryptographic randomness.
- **Separation of concerns:** VRF callback writes storage only; trait assignment occurs in a separate transaction.
- **Read buffer immutability:** Frozen queue cannot be modified between VRF arrival and consumption by `processTicketBatch`.

No manipulation vector exists in the mid-day flow. An attacker would need to either predict the VRF output before the buffer swap (cryptographically infeasible) or modify the frozen read buffer after the swap (structurally impossible).

## Section 3: lastLootboxRngWord Observability Analysis

This section examines whether the public readability of `lastLootboxRngWord` creates an exploitation vector for trait manipulation.

### 3a: Who Can Read lastLootboxRngWord and When

`lastLootboxRngWord` is declared as an `internal` Solidity state variable, meaning no public getter function exists. However, all contract storage is publicly readable on-chain via `eth_getStorageAt`. Any observer can read the current value of `lastLootboxRngWord` at any time by querying the appropriate storage slot.

**Two write paths:**

| Write Path | Location | Trigger |
|---|---|---|
| Mid-day drain | AdvanceModule:166 | `advanceGame` on same day as VRF delivery, when `midDayTicketRngPending == true`. Reads `lootboxRngWordByIndex[lootboxRngIndex - 1]` and writes to `lastLootboxRngWord`. |
| Piggyback | AdvanceModule:789 via `_finalizeLootboxRng` | Daily `advanceGame` path. When `_finalizeLootboxRng` fires during daily RNG processing, writes the daily VRF word to both `lootboxRngWordByIndex[index]` and `lastLootboxRngWord`. |

**Timing.** After either write path executes, the value of `lastLootboxRngWord` is observable on-chain. If there is any delay before the next `processTicketBatch` call (e.g., multi-block ticket processing requiring multiple `advanceGame` calls to drain the queue), an observer has a window during which they know the exact entropy word that will be used for trait assignment.

In practice, both writes occur within `advanceGame` transactions, and `processTicketBatch` is called within the same `advanceGame` call chain (AdvanceModule:171 for mid-day, AdvanceModule:1080 for daily). The word and its consumption are typically in the same transaction. However, if the ticket queue is large enough to require multiple `advanceGame` calls to fully drain, subsequent calls reuse the same `lastLootboxRngWord` value that is already visible on-chain.

### 3b: What Information It Reveals

Knowing `lastLootboxRngWord` reveals the **exact trait assignment** for any given queue composition. The LCG seed derivation is fully deterministic:

```
seed = (baseKey + groupIdx) ^ entropyWord
```

Where:
- `baseKey = (uint256(lvl) << 224) | (queueIdx << 192) | (uint256(uint160(player)) << 32)` -- all publicly knowable. Level is public game state. Queue index is derivable by reading `ticketQueue[readKey]` from storage. Player address is public.
- `groupIdx = i >> 4` -- deterministic from ticket position within the player's owed count.
- `entropyWord = lastLootboxRngWord` -- the observable value.

Given these inputs, an observer can reconstruct the entire LCG sequence:
1. `s = uint64(seed) | 1`
2. `s = s * (TICKET_LCG_MULT + uint64(offset)) + uint64(offset)` (warmup)
3. `s = s * TICKET_LCG_MULT + 1` (per-ticket step)
4. `traitId = DegenerusTraitUtils.traitFromWord(s) + (uint8(i & 3) << 6)` (trait derivation)

This means an observer CAN predict which traits each ticket in the read buffer will receive BEFORE `processTicketBatch` executes. This determinism is **by design** -- it enables on-chain verification that trait assignment was performed correctly.

### 3c: Why the Read Buffer Prevents Exploitation

The critical defense is the double-buffer mechanism. Even though trait outcomes are predictable given the entropy word and queue composition, the queue composition in the **read buffer is frozen at swap time**. An attacker who observes `lastLootboxRngWord` cannot:

1. **Add tickets to the read buffer.** All new purchases write to the write buffer via `_tqWriteKey` (Storage:717-718). The write key and read key are always opposite slots (`ticketWriteSlot ^= 1` at swap, Storage:735). There is no code path that writes to the read buffer after a swap.

2. **Remove tickets from the read buffer.** No removal function exists for `ticketQueue`. The only operation that modifies the read buffer is `processTicketBatch` itself, which processes entries sequentially and deletes the array only when fully drained (JackpotModule:2003-2004).

3. **Reorder tickets in the read buffer.** The `ticketQueue` array is append-only (Storage:557, 590). There is no swap, sort, or reorder operation. Queue position is fixed at the time of the original `push`.

4. **Change the entropy word.** `lastLootboxRngWord` is written only by the two paths documented in 3a. Both are internal functions callable only within `advanceGame`. No external entry point can overwrite this value.

The only theoretical "attack" would be for a player to decide NOT to purchase future tickets based on predicted trait outcomes for tickets already in the read buffer. But:
- Their own tickets are already committed in the read buffer (frozen).
- Future purchases go to the write buffer, which will use a different (unknown) `lastLootboxRngWord` after the next swap.
- Not purchasing is a personal economic choice, not a protocol exploit.

### 3d: Verdict

**SAFE.**

The observability of `lastLootboxRngWord` does not create a manipulation vector because the read buffer is immutable after the swap. The double-buffer mechanism serves as a **structural commit-reveal**:

1. **Commit phase:** Tickets are committed to the write buffer during purchases. Players cannot know what `lastLootboxRngWord` will be when their tickets are eventually processed -- the VRF word is requested after the buffer swap.
2. **Reveal phase:** When the VRF callback delivers the word and `processTicketBatch` executes, the queue composition is frozen. The entropy determines traits, but no actor can change what it indexes into.

This is a stronger guarantee than entropy secrecy. Even if `lastLootboxRngWord` were exposed via a public getter function, the protocol would remain secure because manipulation requires changing the queue composition or the entropy, neither of which is possible after the swap.

**VRF confirmation asymmetry (design observation, not vulnerability):** The mid-day lootbox VRF path uses `VRF_MIDDAY_CONFIRMATIONS = 3` (AdvanceModule:96, 724), compared to the daily path's higher confirmation count. This is a design tradeoff for faster mid-day ticket processing. The security guarantee is unaffected because:
- The VRF proof is verified on-chain by the Chainlink coordinator regardless of confirmation count.
- Confirmation count affects reorg risk (whether the requesting block could be orphaned), not the cryptographic quality of the VRF output.
- Even with a reorg that replays the VRF request, the read buffer remains frozen -- a different word would produce different traits, but no actor controls which word the VRF produces.

## Section 4: Coinflip Lock Timing -- Gap Analysis and Alignment Verdict

This section analyzes whether `_coinflipLockedDuringTransition` adequately protects coinflip operations during RNG-sensitive periods, identifying any gaps and assessing their safety.

### 4a: _coinflipLockedDuringTransition Condition Analysis

**`_coinflipLockedDuringTransition` (BurnieCoinflip:1032-1045)** is a private view function that returns `true` when ALL five conditions hold simultaneously (5-way AND):

```
locked = (!inJackpotPhase) && !degenerusGame.gameOver() && lastPurchaseDay_ && rngLocked_ && (purchaseLevel_ % 10 == 0);
```

| # | Condition | Meaning | When True |
|---|---|---|---|
| 1 | `!inJackpotPhase` | Not currently in a 5-day jackpot draw sequence | During purchase phase (normal gameplay) |
| 2 | `!degenerusGame.gameOver()` | Game has not ended | Before final decimator completes |
| 3 | `lastPurchaseDay_` | Current day is the last purchase day of the current level | The final day before level transition triggers |
| 4 | `rngLocked_` | Daily VRF request is in-flight | Between `_requestRng` and `rawFulfillRandomWords` callback (rngLockedFlag = true) |
| 5 | `purchaseLevel_ % 10 == 0` | Current purchase level is a multiple of 10 | BAF-eligible transition levels (10, 20, 30, ...) |

**Scope of protection:** This guard is checked ONLY in `_depositCoinflip` (BurnieCoinflip:258). It does NOT apply to the four claim functions (`claimCoinflips` at BurnieCoinflip:336, `claimCoinflipsTakeProfit` at BurnieCoinflip:347, `claimCoinflipsFromBurnie` at BurnieCoinflip:357, `consumeCoinflipsForBurn` at BurnieCoinflip:367) -- those have their own independent `rngLocked()` guard (see 4c below).

**Design intent:** The 5-way AND is intentionally narrow. It targets the highest-risk scenario: a BAF-eligible level transition day with daily VRF in-flight, where a coinflip deposit would trigger `_claimCoinflipsInternal` (BurnieCoinflip:261) which could update the BAF leaderboard mid-resolution. The guard description in the source confirms this: "Block deposits during BAF jackpot resolution to prevent auto-claim from updating the BAF leaderboard mid-resolution" (BurnieCoinflip:256-257).

### 4b: RNG-Sensitive Period Enumeration

All periods when RNG state is sensitive (VRF in-flight or entropy pending consumption):

| # | Period | Duration | State Indicators |
|---|---|---|---|
| 1 | **Daily VRF in-flight** | From `_requestRng` (sets `rngLockedFlag = true`) to `rawFulfillRandomWords` callback (stores word) | `rngLockedFlag == true`, `rngWordCurrent == 0` |
| 2 | **Mid-day VRF in-flight** | From `requestLootboxRng` VRF request (AdvanceModule:720) to `rawFulfillRandomWords` callback writing `lootboxRngWordByIndex[index]` (AdvanceModule:1342) | `rngRequestTime != 0`, `midDayTicketRngPending == true` (if buffer was swapped), `rngLockedFlag == false` |
| 3 | **Post-callback pre-consumption (mid-day)** | From VRF callback writing `lootboxRngWordByIndex[index]` to `advanceGame` reading it into `lastLootboxRngWord` (AdvanceModule:166) and running `processTicketBatch` | `lootboxRngWordByIndex[lootboxRngIndex - 1] != 0`, `lastLootboxRngWord` not yet updated, `midDayTicketRngPending == true` |
| 4 | **Jackpot phase inter-block gaps** | Between consecutive `advanceGame` calls during the 5-day jackpot draw sequence (BAF auction, daily draws, decimator) | `inJackpotPhase == true`, various jackpot sub-states |

### 4c: Gap Analysis

Three gaps exist where `_coinflipLockedDuringTransition` does NOT fire but RNG is in a sensitive period.

**Gap 1: Non-x10 levels with rngLocked=true on lastPurchaseDay**

**Scenario:** `purchaseLevel % 10 != 0` (e.g., level 13), `lastPurchaseDay == true`, `rngLocked == true`. The `purchaseLevel_ % 10 == 0` condition fails, so `_coinflipLockedDuringTransition` returns `false`. Coinflip DEPOSITS are allowed.

**Deposit path analysis:** `_depositCoinflip` (BurnieCoinflip:248-258) passes the guard, then calls `_claimCoinflipsInternal(caller, false)` (BurnieCoinflip:261) for auto-claim. This auto-claim reads `coinflipDayResult[epoch]` -- results that are ALREADY DETERMINED from a prior day's VRF word. The results for past epochs are immutable storage values. The deposit itself writes to the player's state for FUTURE epochs (incrementing their stake for the next coinflip resolution).

**Coinflip CLAIM functions:** All four claim functions independently check `degenerusGame.rngLocked()` (BurnieCoinflip:336, 347, 357, 367) and revert if true. Claims are blocked regardless of what `_coinflipLockedDuringTransition` returns.

**Verdict: SAFE.** Deposits at non-x10 levels during `rngLocked` cannot influence any RNG outcome. Auto-claim reads already-determined results. The deposit writes future state. Claims are independently blocked by `rngLocked()`.

---

**Gap 2: During jackpot phase**

**Scenario:** `inJackpotPhase == true`. The `!inJackpotPhase` condition fails, so `_coinflipLockedDuringTransition` returns `false`. Coinflip DEPOSITS are allowed during the entire jackpot phase.

**Deposit path analysis:** Same as Gap 1. `_depositCoinflip` triggers `_claimCoinflipsInternal` auto-claim, which reads already-determined `coinflipDayResult[epoch]` values. The deposit writes future epoch state. No interaction with jackpot draw VRF words, pool allocations, or winner selection.

**Claim guard:** During jackpot phase, daily VRF requests fire each day for jackpot draws. When `rngLockedFlag == true` (VRF in-flight for that day's draw), all four claim functions revert via `rngLocked()`. Between jackpot draws (inter-block gaps where `rngLockedFlag == false`), claims ARE allowed -- but this is safe because the coinflip results being claimed were determined by a prior VRF word, not the pending jackpot VRF.

**Cross-reference:** Phase 13-03 decision: "Coinflip deposits during jackpot phase gap safe -- BURNIE-only, no pool/RNG interaction."

**Verdict: SAFE.** Coinflip deposits during jackpot phase do not interact with jackpot RNG, pool state, or winner selection. Claims are guarded independently by `rngLocked()` during active VRF requests.

---

**Gap 3: Mid-day VRF in-flight on non-lastPurchaseDay**

**Scenario:** `requestLootboxRng` has been called, mid-day VRF is in-flight (`rngRequestTime != 0`), but `lastPurchaseDay == false`. The `lastPurchaseDay_` condition fails, so `_coinflipLockedDuringTransition` returns `false`. Coinflip DEPOSITS are allowed.

**RNG independence analysis:** The mid-day VRF (lootbox path) produces a word stored in `lootboxRngWordByIndex[index]`, which is consumed ONLY by `processTicketBatch` for ticket trait assignment. Coinflip results are determined by the DAILY VRF word stored in `rngWordByDay[dayIndex]`, which is a completely separate VRF request/callback/storage path. The two RNG streams share no state:

| Property | Daily VRF (coinflip) | Lootbox VRF (tickets) |
|---|---|---|
| Request function | `_requestRng` (internal, within `advanceGame`) | `requestLootboxRng` (external) |
| Callback routing | `rngLockedFlag == true` path (AdvanceModule:1336-1338) | `rngLockedFlag == false` path (AdvanceModule:1339-1345) |
| Word storage | `rngWordCurrent` -> `rngWordByDay[day]` | `lootboxRngWordByIndex[index]` -> `lastLootboxRngWord` |
| Consumer | Coinflip resolution, jackpot draws | `processTicketBatch` ticket traits |

**State flag check:** During mid-day VRF in-flight, `rngLockedFlag == false` (the mid-day path explicitly requires `!rngLockedFlag` at AdvanceModule:674). Therefore, the `rngLocked()` check on claim functions also does NOT block. However, this is correct: no coinflip-relevant RNG is in-flight. The mid-day VRF only affects ticket traits.

**Verdict: SAFE.** Mid-day lootbox VRF and daily coinflip VRF are independent RNG streams with no shared state. Coinflip deposits and claims during mid-day VRF in-flight cannot influence or be influenced by the lootbox VRF outcome.

### 4d: Alignment Verdict

**ALIGNED WITH ACCEPTABLE GAPS.**

The `_coinflipLockedDuringTransition` guard (BurnieCoinflip:1032-1045) protects the highest-risk window: a BAF-eligible transition day (x10 level) with daily VRF in-flight, where coinflip auto-claim during deposit could update the BAF leaderboard mid-resolution. The three identified gaps are all safe:

1. **Gap 1 (non-x10 levels):** No BAF leaderboard interaction. Deposits write future state; auto-claim reads determined results. Claims independently blocked by `rngLocked()`.
2. **Gap 2 (jackpot phase):** Coinflip operations are BURNIE-only with no pool/RNG interaction. Claims independently blocked by `rngLocked()` during active jackpot VRF requests.
3. **Gap 3 (mid-day VRF):** Lootbox VRF and daily VRF are independent RNG streams. No shared state, no cross-contamination.

The 5-condition AND is deliberately narrow to avoid unnecessary UX friction. Broader locking (e.g., blocking all deposits whenever any VRF is in-flight) would prevent legitimate coinflip activity during periods where no security risk exists. The current design correctly identifies the single dangerous scenario (BAF leaderboard corruption during transition) and blocks precisely that case, while relying on independent `rngLocked()` guards on claim functions (BurnieCoinflip:336, 347, 357, 367) to prevent claim-based exploitation during daily VRF sensitivity.

**Deposit vs. claim lock distinction:**
- **Deposits:** Guarded by `_coinflipLockedDuringTransition` (5-condition AND, BurnieCoinflip:258) -- blocks only during BAF-eligible transition with daily VRF in-flight.
- **Claims:** Guarded by `degenerusGame.rngLocked()` (BurnieCoinflip:336, 347, 357, 367) -- blocks whenever daily VRF is in-flight, regardless of level or phase. This is a broader and simpler guard, appropriate because claims can extract value based on known VRF outcomes.

This two-tier guard design provides comprehensive protection: deposits are locked when they could corrupt transition-sensitive state, and claims are locked whenever outcome knowledge could enable extraction.
