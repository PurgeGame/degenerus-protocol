## Section 1: Ticket Creation End-to-End Trace

This section traces the complete lifecycle of a ticket from purchase through trait assignment, identifying the entropy source (or lack thereof) at every step.

### 1a: Purchase to Write Buffer

**Entry points.** Tickets enter the system through purchase functions that ultimately call `_queueTickets` or `_queueTicketsScaled`:

| Entry Point | Module | Queuing Function Called |
|---|---|---|
| `purchase` / `purchaseCoin` | AdvanceModule | `_queueTickets` (whole) or `_queueTicketsScaled` (fractional) |
| `purchaseWhaleBundle` | WhaleModule | `_queueTicketRange` (multi-level) |
| `purchaseLazyPass` | WhaleModule | `_queueTicketRange` (multi-level) |
| `purchaseDeityPass` | WhaleModule | `_queueTickets` (whole) |
| `purchaseBurnieLootbox` | AdvanceModule | `_queueTicketsScaled` (fractional) |

**`_queueTickets` (Storage:545-571).** The core queuing function:

1. Computes the write buffer key: `uint24 wk = _tqWriteKey(targetLevel)` (Storage:552).
2. `_tqWriteKey` returns `level | TICKET_SLOT_BIT` when `ticketWriteSlot != 0`, else `level` (Storage:717-718). `TICKET_SLOT_BIT = 1 << 23` (Storage:154).
3. Reads the player's existing packed entry: `uint40 packed = ticketsOwedPacked[wk][buyer]` (Storage:553).
4. If the player has no existing entry (`owed == 0 && rem == 0`), appends the player address to `ticketQueue[wk]` (Storage:556-558).
5. Adds `quantity` to the existing `owed` count, capping at `type(uint32).max` to prevent overflow (Storage:559-565).
6. Writes the updated packed value: `(uint40(uint32(newOwed)) << 8) | uint40(rem)` (Storage:567-569).

**Packing format:** `uint40 = (uint32 owed << 8) | (uint8 remainder)`. The upper 32 bits store whole tickets owed; the lower 8 bits store fractional remainder (used by `_queueTicketsScaled`).

**`_queueTicketsScaled` (Storage:578-627).** Handles fractional ticket quantities (scaled by `TICKET_SCALE = 100`, Storage:129). Separates the input into whole tickets and fractional remainder. When accumulated remainder reaches `TICKET_SCALE`, promotes one whole ticket. Same write-buffer mechanics as `_queueTickets`.

**Entropy source at this step: NONE.** Ticket queuing is entirely deterministic. The write buffer key is derived from level and `ticketWriteSlot` (a boolean toggle). No random values are read or consumed. The queue is an append-only array of player addresses with packed ticket counts.

### 1b: Buffer Swap -- Write to Read

The double-buffer mechanism uses two slots (toggled by `ticketWriteSlot`) so that purchases can continue while previously queued tickets are being processed.

**`_swapTicketSlot` (Storage:732-737):**

```
function _swapTicketSlot(uint24 purchaseLevel) internal {
    uint24 rk = _tqReadKey(purchaseLevel);
    if (ticketQueue[rk].length != 0) revert E();  // Read buffer must be fully drained
    ticketWriteSlot ^= 1;                          // Toggle active write slot
    ticketsFullyProcessed = false;                 // Mark new read buffer as unprocessed
}
```

The assert at Storage:734 enforces that the read buffer must be completely drained before a swap can occur. This prevents interleaving of two batches.

**`_swapAndFreeze` (Storage:742-748):**

```
function _swapAndFreeze(uint24 purchaseLevel) internal {
    _swapTicketSlot(purchaseLevel);
    if (!prizePoolFrozen) {
        prizePoolFrozen = true;
        prizePoolPendingPacked = 0;
    }
}
```

Called during daily `advanceGame` to simultaneously swap the ticket buffer and freeze the prize pool for jackpot draws.

**Two swap triggers:**

1. **Daily path:** `_swapAndFreeze` called within `advanceGame` when advancing to a new day. Freezes the pool and swaps the buffer in a single atomic operation.
2. **Mid-day path:** `_swapTicketSlot` called within `requestLootboxRng` (AdvanceModule:708-715) when the write buffer has pending tickets and the read buffer has been fully processed. Sets `midDayTicketRngPending = true` to signal that a VRF callback is needed before processing can begin.

The mid-day swap trigger at AdvanceModule:713-715:
```
if (ticketQueue[wk].length > 0 && ticketsFullyProcessed) {
    _swapTicketSlot(purchaseLevel_);
    midDayTicketRngPending = true;
}
```

**Entropy source at this step: NONE.** The buffer swap is a mechanical toggle (`ticketWriteSlot ^= 1`). No random values are involved. After the swap, the former write buffer becomes the read buffer and is frozen -- no further modifications are possible.

### 1c: Read Buffer to Trait Assignment

After the buffer swap, `processTicketBatch` (JackpotModule:1949-2010) processes the frozen read buffer to assign traits to each queued ticket.

**Step 1: Read entropy.** At JackpotModule:1975:
```
uint256 entropy = lastLootboxRngWord;
```
This is the single entropy source for the entire batch. `lastLootboxRngWord` is a VRF-derived value set either via the mid-day drain path (AdvanceModule:166) or the piggyback path (`_finalizeLootboxRng` at AdvanceModule:789). See Phase 12 `v1.2-rng-data-flow.md` Section 3 for the complete mid-day ticket RNG lifecycle.

**Step 2: Iterate queue.** The function reads `ticketQueue[readKey]` (JackpotModule:1951) and iterates through players using a cursor (`ticketCursor`, JackpotModule:1960) with a gas budget (`WRITES_BUDGET_SAFE = 550`, JackpotModule:1969). Multiple `processTicketBatch` calls may be needed to drain the full queue.

**Step 3: Per-player processing.** For each player, `_processOneTicketEntry` (JackpotModule:2044-2107) constructs a per-player salt:
```
uint256 rollSalt = (uint256(lvl) << 224) |
    (queueIdx << 192) |
    (uint256(uint160(player)) << 32);
```
This salt encodes the level (bits 255-224), queue index (bits 223-192), and player address (bits 191-32). It ensures each player at each queue position gets unique entropy derivation even though they share the same `lastLootboxRngWord`.

**Step 4: Trait generation.** `_generateTicketBatch` (JackpotModule:2110-2130) constructs the baseKey:
```
uint256 baseKey = (uint256(lvl) << 224) |
    (queueIdx << 192) |
    (uint256(uint160(player)) << 32);
```
Then calls `_raritySymbolBatch` (JackpotModule:2187-2281).

**Step 5: LCG trait derivation.** `_raritySymbolBatch` processes tickets in groups of 16. For each group:

1. **Seed derivation:** `seed = (baseKey + groupIdx) ^ entropyWord` (JackpotModule:2211), where `groupIdx = i >> 4` (groups of 16 tickets).
2. **LCG initialization:** `s = uint64(seed) | 1` (JackpotModule:2213) -- ensures odd for full-period LCG.
3. **Offset warmup:** `s = s * (TICKET_LCG_MULT + uint64(offset)) + uint64(offset)` (JackpotModule:2216) -- fast-forwards the LCG to the correct starting position within a group.
4. **LCG stepping:** `s = s * TICKET_LCG_MULT + 1` (JackpotModule:2221) -- standard Knuth LCG with multiplier `TICKET_LCG_MULT = 0x5851F42D4C957F2D` (= 6364136223846793005, JackpotModule:176).
5. **Trait derivation:** `DegenerusTraitUtils.traitFromWord(s)` (JackpotModule:2224) splits the 64-bit LCG state into two 32-bit halves for weighted category and sub-bucket selection (TraitUtils:143-149), producing a 6-bit trait ID (bits 5-3: category, bits 2-0: sub-bucket).
6. **Quadrant offset:** `+ (uint8(i & 3) << 6)` (JackpotModule:2225) adds a 2-bit quadrant to the trait ID, producing a final 8-bit `traitId` that indexes into 256 trait buckets.

The generated traits are counted per-traitId in memory, then batch-written to `traitBurnTicket[lvl][traitId]` storage via inline assembly (JackpotModule:2254-2281).

**Entropy source at this step: `lastLootboxRngWord` (VRF-derived).** This is the ONLY point in the entire ticket lifecycle where randomness enters the system. The LCG expands this single 256-bit VRF word into per-ticket trait assignments via deterministic derivation keyed by `(level, queueIndex, playerAddress, groupIndex)`.

### 1d: Entropy Source Summary Table

| Step | Function(s) | Contract Location | Entropy Source | Notes |
|---|---|---|---|---|
| Purchase | `buyTickets`, `purchaseDeityPass`, etc. | Various entry points | **None** | Deterministic ticket count based on payment |
| Queue | `_queueTickets` / `_queueTicketsScaled` | Storage:545-627 | **None** | Append-only write to write buffer; packed uint40 accounting |
| Buffer Swap | `_swapTicketSlot` / `_swapAndFreeze` | Storage:732-748 | **None** | Mechanical toggle of `ticketWriteSlot ^= 1`; freezes read buffer |
| Batch Init | `processTicketBatch` | JackpotModule:1949-1975 | **`lastLootboxRngWord`** | Single VRF-derived word read once; shared across entire batch |
| Trait Assign | `_raritySymbolBatch` | JackpotModule:2187-2281 | **LCG from `lastLootboxRngWord`** | Per-ticket derivation via `(baseKey + groupIdx) ^ entropyWord` seeded LCG |

**Key finding:** Entropy enters the ticket lifecycle at exactly one point -- when `processTicketBatch` reads `lastLootboxRngWord` at JackpotModule:1975. All upstream steps (purchase, queuing, buffer swap) are fully deterministic. All downstream steps (LCG expansion, trait derivation) are deterministic transformations of this single entropy source.

Phase 14 L4 analysis confirms this consumption point is **BLOCKED** from manipulation: the read buffer is frozen by the double-buffer swap, entropy is VRF-derived, and processing occurs atomically within `advanceGame` (see `v1.2-manipulation-windows.md` Section 1b, L4 verdict).
