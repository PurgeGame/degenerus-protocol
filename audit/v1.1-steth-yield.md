## 1. Overview

The game contract holds both ETH and stETH (Lido rebasing staked ETH). stETH enters the system through two paths: admin staking of excess ETH via Lido, and automatic staking during level transitions. Once held, stETH balance increases over time through Lido staking rewards (rebasing), creating a yield surplus.

Key characteristics:
- **Segregated yield accumulator:** 46% of yield surplus routes to a dedicated accumulator each level transition. Additionally, 1% of nextPool is skimmed into the accumulator at every level transition (`INSURANCE_SKIM_BPS = 100`). At x00 milestones, 50% of the accumulator amplifies the current prize pool; 50% is retained as growing terminal insurance.
- **Admin-controlled staking:** ADMIN can manually stake excess ETH via Lido, constrained by player-claim ETH reserve (claimablePool minus vault/sDGNRS claimable, since those accept stETH natively)
- **Automatic staking:** The AdvanceModule auto-stakes all ETH above claimablePool during level transitions
- **Dual payout strategies:** Player claims use ETH-first ordering; vault/sDGNRS claims use stETH-first ordering
- **Value-preserving:** All admin operations are value-neutral; no fund extraction is possible

**Primary source:** `contracts/DegenerusGame.sol` lines 1775--1823 (admin functions), lines 2138--2154 (yieldPoolView, yieldAccumulatorView)

### 2b. Admin Staking: adminStakeEthForStEth

**Entry point:** `adminStakeEthForStEth(uint256 amount)`
**Source:** `DegenerusGame.sol` lines 1803--1823
**Access:** ADMIN only

Stakes game-held ETH into stETH via Lido `submit()`. The critical constraint: must retain enough ETH to cover player claims. Vault and sDGNRS claimable amounts are excluded from the ETH reserve because those addresses accept stETH payouts natively via `claimWinningsStethFirst`.

```solidity
// DegenerusGame.sol:1803-1823
function adminStakeEthForStEth(uint256 amount) external {
    if (msg.sender != ContractAddresses.ADMIN) revert E();
    if (amount == 0) revert E();

    uint256 ethBal = address(this).balance;
    if (ethBal < amount) revert E();
    // Vault and sDGNRS claimable can be settled in stETH, so exclude from ETH reserve
    uint256 stethSettleable = claimableWinnings[ContractAddresses.VAULT]
        + claimableWinnings[ContractAddresses.SDGNRS];
    uint256 reserve = claimablePool > stethSettleable
        ? claimablePool - stethSettleable
        : 0;
    if (ethBal <= reserve) revert E();
    uint256 stakeable = ethBal - reserve;
    if (amount > stakeable) revert E();

    // stETH return value intentionally ignored: Lido mints 1:1 for ETH, validated by input checks
    try steth.submit{value: amount}(address(0)) returns (uint256) {} catch {
        revert E();
    }
}
```

**Staking constraint formula:**

```
stethSettleable = claimableWinnings[VAULT] + claimableWinnings[SDGNRS]
reserve = claimablePool - stethSettleable   (clamped to 0)
stakeable = address(this).balance - reserve
require: amount <= stakeable
```

The reserve represents ETH owed to *player* claims only. Vault and sDGNRS claims are excluded because those contracts use `claimWinningsStethFirst` and can receive stETH directly.

### 2c. Admin Rebalancing: adminSwapEthForStEth

**Entry point:** `adminSwapEthForStEth(address recipient, uint256 amount)`
**Source:** `DegenerusGame.sol` lines 1783--1794
**Access:** ADMIN only

Value-neutral swap: admin sends ETH to the game contract and receives an equal amount of stETH back. Used to rebalance when stETH yield should be converted to ETH.

```solidity
// DegenerusGame.sol:1783-1794
function adminSwapEthForStEth(
    address recipient,
    uint256 amount
) external payable {
    if (msg.sender != ContractAddresses.ADMIN) revert E();
    if (recipient == address(0)) revert E();
    if (amount == 0 || msg.value != amount) revert E();

    uint256 stBal = steth.balanceOf(address(this));
    if (stBal < amount) revert E();
    if (!steth.transfer(recipient, amount)) revert E();
}
```

**Security invariant:** `msg.value == amount` ensures 1:1 value exchange. The game receives ETH and sends stETH. No net value extraction is possible.

### 2d. Automatic Staking During Level Transitions

**Entry point:** `_autoStakeExcessEth()`
**Source:** `DegenerusGameAdvanceModule.sol` lines 1081--1089

During level transitions, the AdvanceModule automatically stakes all ETH above claimablePool into stETH. This is non-critical: uses try/catch so the game continues even if Lido is paused.

```solidity
// DegenerusGameAdvanceModule.sol:1081-1089
function _autoStakeExcessEth() private {
    uint256 ethBal = address(this).balance;
    uint256 reserve = claimablePool;
    if (ethBal <= reserve) return;
    uint256 stakeable = ethBal - reserve;
    try steth.submit{value: stakeable}(address(0)) returns (uint256) {} catch {
        emit StEthStakeFailed(stakeable);
    }
}
```

**Key difference from admin staking:** Auto-staking is non-reverting (silent failure via catch). Admin staking reverts on failure.

## 3. Yield Surplus Formula (STETH-02)

### 3a. yieldPoolView

**Entry point:** `yieldPoolView()`
**Source:** `DegenerusGame.sol` lines 2138--2148

The yield surplus is the difference between total holdings (ETH + stETH) and total obligations (all prize pools + yield accumulator):

```solidity
// DegenerusGame.sol:2138-2154
function yieldPoolView() external view returns (uint256) {
    uint256 totalBalance = address(this).balance +
        steth.balanceOf(address(this));
    uint256 obligations = currentPrizePool +
        _getNextPrizePool() +
        claimablePool +
        _getFuturePrizePool() +
        yieldAccumulator;
    if (totalBalance <= obligations) return 0;
    return totalBalance - obligations;
}

function yieldAccumulatorView() external view returns (uint256) {
    return yieldAccumulator;
}
```

**Formula:**

```
yieldSurplus = (ETH balance + stETH balance) - (currentPrizePool + nextPrizePool + claimablePool + futurePrizePool + yieldAccumulator)

If totalBalance <= obligations: yieldSurplus = 0 (clamped, never negative)
```

### 3b. How Yield Accrues

stETH is a rebasing token. The game contract's stETH balance increases daily as Lido distributes staking rewards. This means:

1. Admin stakes 100 ETH via `adminStakeEthForStEth` -- game receives ~100 stETH
2. Over time, Lido rebases increase stETH balance (e.g., to 100.01 stETH after one day)
3. The 0.01 stETH increase is reflected in `steth.balanceOf(address(this))`
4. Since obligations have not changed, `yieldPoolView()` reports a higher surplus

> **Yield Distribution Mechanism.** At each level transition, `_distributeYieldSurplus` extracts yield above obligations and splits it: 23% sDGNRS claimable, 23% vault claimable, 46% to a **segregated yield accumulator**, ~8% buffer retained. Additionally, 1% of `nextPrizePool` is skimmed into the accumulator at every level transition (`INSURANCE_SKIM_BPS = 100` in `_applyTimeBasedFutureTake`), providing a second funding source beyond stETH yield. At **x00 milestone levels**, 50% of the accumulator flows into `futurePrizePool` (entering before the keep-roll, so only the kept fraction reaches `currentPrizePool`), while 50% is retained as growing terminal insurance. The accumulator is zeroed at GAMEOVER. Every level survived makes the terminal incentive stronger — the accumulator grows monotonically with time and contract size.

## 4. Payout Ordering (STETH-02)

The game implements two payout strategies depending on the claimant type. Both strategies deliver the same total value but differ in asset preference.

### 4a. Player Claims: ETH-First (claimWinnings)

**Entry point:** `claimWinnings(address player)`
**Source:** `DegenerusGame.sol` lines 1376--1378, 1392--1408
**Payout helper:** `_payoutWithStethFallback` (lines 1948--1975)

Players call `claimWinnings` to collect accrued jackpot/prize winnings. ETH is sent first; stETH covers any remainder.

```solidity
// DegenerusGame.sol:1376-1378
function claimWinnings(address player) external {
    player = _resolvePlayer(player);
    _claimWinningsInternal(player, false);
}
```

```solidity
// DegenerusGame.sol:1392-1408
function _claimWinningsInternal(address player, bool stethFirst) private {
    if (finalSwept) revert E();
    uint256 amount = claimableWinnings[player];
    if (amount <= 1) revert E();
    uint256 payout;
    unchecked {
        claimableWinnings[player] = 1; // Leave sentinel
        payout = amount - 1;
    }
    claimablePool -= payout; // CEI: update state before external call
    emit WinningsClaimed(player, msg.sender, payout);
    if (stethFirst) {
        _payoutWithEthFallback(player, payout);
    } else {
        _payoutWithStethFallback(player, payout);
    }
}
```

**ETH-first payout logic** (`_payoutWithStethFallback`):

```solidity
// DegenerusGame.sol:1948-1975
function _payoutWithStethFallback(address to, uint256 amount) private {
    if (amount == 0) return;

    // Try ETH first (preferred for player claims)
    uint256 ethBal = address(this).balance;
    uint256 ethSend = amount <= ethBal ? amount : ethBal;
    if (ethSend != 0) {
        (bool okEth, ) = payable(to).call{value: ethSend}("");
        if (!okEth) revert E();
    }
    uint256 remaining = amount - ethSend;
    if (remaining == 0) return;

    // Fall back to stETH for remainder
    uint256 stBal = steth.balanceOf(address(this));
    uint256 stSend = remaining <= stBal ? remaining : stBal;
    _transferSteth(to, stSend);

    // Retry ETH for any remaining (handles edge cases)
    uint256 leftover = remaining - stSend;
    if (leftover != 0) {
        uint256 ethRetry = address(this).balance;
        if (ethRetry < leftover) revert E();
        (bool ok, ) = payable(to).call{value: leftover}("");
        if (!ok) revert E();
    }
}
```

**Payout sequence:** ETH -> stETH -> ETH retry (edge case fallback)

### 4b. Vault/sDGNRS Claims: stETH-First (claimWinningsStethFirst)

**Entry point:** `claimWinningsStethFirst()`
**Source:** `DegenerusGame.sol` lines 1383--1389
**Payout helper:** `_payoutWithEthFallback` (lines 1981--1995)

Restricted to VAULT and sDGNRS contracts only. stETH is sent first; ETH covers any remainder.

```solidity
// DegenerusGame.sol:1383-1389
function claimWinningsStethFirst() external {
    address player = msg.sender;
    if (
        player != ContractAddresses.VAULT &&
        player != ContractAddresses.SDGNRS
    ) revert E();
    _claimWinningsInternal(player, true);
}
```

**stETH-first payout logic** (`_payoutWithEthFallback`):

```solidity
// DegenerusGame.sol:1981-1995
function _payoutWithEthFallback(address to, uint256 amount) private {
    if (amount == 0) return;

    uint256 stBal = steth.balanceOf(address(this));
    uint256 stSend = amount <= stBal ? amount : stBal;
    _transferSteth(to, stSend);

    uint256 remaining = amount - stSend;
    if (remaining == 0) return;

    uint256 ethBal = address(this).balance;
    if (ethBal < remaining) revert E();
    (bool ok, ) = payable(to).call{value: remaining}("");
    if (!ok) revert E();
}
```

**Payout sequence:** stETH -> ETH fallback (no retry)

### 4c. Payout Ordering Summary

| Claimant | Function | Asset Priority | Fallback | Access |
|----------|----------|---------------|----------|--------|
| Player | `claimWinnings` | ETH first | stETH, then ETH retry | Any address |
| Vault | `claimWinningsStethFirst` | stETH first | ETH | VAULT only |
| sDGNRS | `claimWinningsStethFirst` | stETH first | ETH | SDGNRS only |

**Why the split?** Players prefer ETH (directly usable, no rebasing complexity). The VAULT and sDGNRS contracts prefer stETH because they hold it as backing for their own token mechanics.

### 4d. stETH Transfer Helper

**Source:** `DegenerusGame.sol` lines 1933--1941

The `_transferSteth` helper has special routing for sDGNRS deposits:

```solidity
// DegenerusGame.sol:1933-1941
function _transferSteth(address to, uint256 amount) private {
    if (amount == 0) return;
    if (to == ContractAddresses.SDGNRS) {
        if (!steth.approve(ContractAddresses.SDGNRS, amount)) revert E();
        sdgnrs.depositSteth(amount);
        return;
    }
    if (!steth.transfer(to, amount)) revert E();
}
```

When paying the sDGNRS contract, the game approves stETH then calls `depositSteth` (pull pattern) rather than direct transfer. All other recipients receive a direct `steth.transfer`.

## 5. stETH in sDGNRS Burns (STETH-02)

### 5a. stETH Deposits to sDGNRS

The game deposits stETH into the sDGNRS contract via the approve-then-pull pattern:

```solidity
// StakedDegenerusStonk.sol:296-298
function depositSteth(uint256 amount) external onlyGame {
    if (!steth.transferFrom(msg.sender, address(this), amount)) revert TransferFailed();
    emit Deposit(msg.sender, 0, amount, 0);
}
```

This adds stETH to the sDGNRS contract's backing reserves without minting new tokens.

### 5b. Burn Payout Composition

When a holder burns sDGNRS, they receive proportional shares of three assets: ETH, stETH, and BURNIE.

**Source:** `StakedDegenerusStonk.sol` lines 404--462

```solidity
// StakedDegenerusStonk.sol:412-416 -- Value calculation
uint256 ethBal = address(this).balance;
uint256 stethBal = steth.balanceOf(address(this));
uint256 claimableEth = _claimableWinnings();
uint256 totalMoney = ethBal + stethBal + claimableEth;
uint256 totalValueOwed = (totalMoney * amount) / supplyBefore;
```

**Payout ordering within burn (ETH-preferred):**

```solidity
// StakedDegenerusStonk.sol:431-437
if (totalValueOwed <= ethBal) {
    ethOut = totalValueOwed;
} else {
    ethOut = ethBal;
    stethOut = totalValueOwed - ethOut;
    if (stethOut > stethBal) revert Insufficient();
}
```

**Key points for agents:**
1. `totalMoney` includes ETH + stETH + claimable winnings from the game contract
2. If ETH is insufficient, the contract first claims any pending winnings from the game (`game.claimWinnings`)
3. ETH is preferred: burn pays ETH first, stETH only for the remainder
4. BURNIE payout is calculated separately: `burnieOut = (totalBurnie * amount) / supplyBefore`

### 5c. Agent Formula: sDGNRS Burn Value

```
totalMoney = sDGNRS.balance (ETH) + steth.balanceOf(sDGNRS) + claimableWinnings
totalBurnie = burnie.balanceOf(sDGNRS) + coinflip.previewClaimCoinflips(sDGNRS)

Per-token value = totalMoney / totalSupply
Per-token BURNIE = totalBurnie / totalSupply

For burning `amount` tokens:
  ethOut + stethOut = (totalMoney * amount) / totalSupply
  burnieOut = (totalBurnie * amount) / totalSupply
```

## 6. stETH Rounding Pitfall

Lido stETH uses an internal shares-to-balance conversion. This introduces known 1-2 wei rounding errors on transfers:

- A transfer of 1.000000000000000000 stETH may deliver 0.999999999999999999 stETH
- This is a known Lido property, not a bug in the game contract

**How the game handles it:**
- `_payoutWithStethFallback` has a three-phase retry: ETH -> stETH -> ETH retry. If stETH is short by 1-2 wei, the final ETH retry covers the gap.
- `_payoutWithEthFallback` sends stETH first then ETH for remainder. Any stETH shortfall falls to the ETH path.
- The sDGNRS burn uses exact balance checks but the `Insufficient()` revert is a hard safety check, not expected to trigger under normal rounding.

**Agent guidance:** Do not expect exact wei precision in stETH amounts. When modeling stETH payouts, treat 1-2 wei as noise within the contract's fallback tolerance.
