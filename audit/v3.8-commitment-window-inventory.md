## Forward-Trace Catalog (CW-01)

This section catalogs every storage variable read or written during VRF fulfillment and all downstream processing. The trace starts at `rawFulfillRandomWords` (AdvanceModule line 1436) and follows every code path through `advanceGame -> rngGate` and all delegatecall modules.

Slot numbers are from `forge inspect` output (authoritative). All DegenerusGameStorage variables are accessed via delegatecall modules operating on the game contract's storage.

### Branch 1: Daily RNG Path (rngLockedFlag == true)

**Entry:** `rawFulfillRandomWords` (AdvanceModule:1436) stores `rngWordCurrent = word` when `rngLockedFlag` is true.

**Processing:** The stored word is consumed by the next `advanceGame()` call, which enters `rngGate` (AdvanceModule:765) and triggers all downstream consumers.

## Mutation Surface Catalog (CW-03)

This section catalogs every external/public function that can write to each variable from the Forward-Trace (CW-01) and Backward-Trace (CW-02) catalogs. For each mutation path, the call-graph depth and access control are recorded.

**Methodology:**

For each cataloged variable:
1. All contracts are searched for assignments (not just the module that reads it), because all delegatecall modules share DegenerusGameStorage.
2. Each write is traced backward to the nearest external/public function entry point.
3. Call-graph depth is recorded:
   - **D0 (direct):** External function directly writes the variable
   - **D1 (one hop):** External -> internal that writes
   - **D2 (two hops):** External -> internal -> internal that writes
   - **D3+ (deep):** Three or more hops
4. Access control is noted: **permissionless** (any caller), **admin-only** (DegenerusAdmin owner), **game-only** (only DegenerusGame contract), **VRF-only** (only VRF coordinator).

### Ticket Queue Double-Buffer Commitment Boundary

**CRITICAL: The `_swapAndFreeze` / `_swapTicketSlot` commitment boundary.**

The ticket queue uses a double-buffer controlled by `ticketWriteSlot` (slot 1, offset 23). The TICKET_SLOT_BIT (1 << 23) is XORed into the uint24 level key to create separate read and write keys:
- **Write key:** `level ^ (ticketWriteSlot == 1 ? TICKET_SLOT_BIT : 0)` -- where new purchases write
- **Read key:** `level ^ (ticketWriteSlot == 0 ? TICKET_SLOT_BIT : 0)` -- where processTicketBatch reads

**When `_swapAndFreeze` is called** (at daily RNG request time in advanceGame, AdvanceModule:230):
1. `_swapTicketSlot(purchaseLevel)` is called (DegenerusGameStorage:713-718)
2. Guard: the read slot must be empty (`ticketQueue[readKey].length != 0` reverts)
3. `ticketWriteSlot ^= 1` -- flips the write/read buffers
4. `ticketsFullyProcessed = false` -- marks the new read slot as not yet processed
5. If not already frozen: `prizePoolFrozen = true`, `prizePoolPendingPacked = 0`

**When `_swapTicketSlot` is called** (at mid-day lootbox RNG request in requestLootboxRng, AdvanceModule:717):
1. Same buffer swap as above, but WITHOUT prize pool freeze
2. `midDayTicketRngPending = true` is set (AdvanceModule:718)

**What happens if `purchase()` is called between swap and fulfillment:**
- New purchases write to the NEW write slot (post-swap), which is separate from the read slot being processed
- The read slot (old write slot) contains tickets committed before the swap
- **The double-buffer protects `ticketQueue` reads:** processTicketBatch reads from the old buffer, new purchases go to the new buffer
- **BUT:** purchases still write to `currentPrizePool` (or `prizePoolPendingPacked` if frozen) and other shared variables. The prize pool freeze mitigates current pool mutation during jackpot, but `prizePoolPendingPacked` accumulation continues.

#### reverseFlip() During Commitment Window

**Function:** `reverseFlip()` (DegenerusGame:1984 -> AdvanceModule:1419-1427)
**Access control:** Permissionless (any player with BURNIE)
**Guard:** `if (rngLockedFlag) revert RngLocked()` (AdvanceModule:1420)
**Variable written:** `totalFlipReversals` (slot 6) -- incremented by 1

**Analysis:** reverseFlip is SAFE during the daily RNG commitment window because `rngLockedFlag` is set true BEFORE the daily VRF request (at `_swapAndFreeze` time in advanceGame). The guard blocks all calls during the window.

For the mid-day lootbox RNG path: `rngLockedFlag` is NOT set by requestLootboxRng. However, totalFlipReversals is not consumed by the mid-day path -- `rawFulfillRandomWords` directly stores `lootboxRngWordByIndex[index]` without applying nudges. totalFlipReversals is only consumed by `_applyDailyRng` during the next daily RNG gate. So reverseFlip during a mid-day VRF request is harmless because the mid-day RNG result is independent of totalFlipReversals.

#### requestLootboxRng() Side Effects

**Function:** `requestLootboxRng()` (DegenerusGame:1967 -> AdvanceModule:674-741)
**Access control:** Permissionless
**State changes beyond VRF request:**

1. `ticketWriteSlot ^= 1` (if write slot non-empty and read slot empty) -- buffer swap via `_swapTicketSlot`

#### purchase() / purchaseCoin() / purchaseLazyPass() During Commitment Window

**Functions:** All purchase functions (MintModule, WhaleModule)
**Access control:** Permissionless
**Variables written during commitment window:**

1. **ticketQueue[writeKey]** -- writes to WRITE slot. Protected by double-buffer: processTicketBatch reads from READ slot, which was frozen at swap time.

## Mutation Verdicts (CW-04, MUT-01, MUT-02, MUT-03)

**Date:** 2026-03-22
**Scope:** Phase 69 -- Per-variable binary verdicts for all 51 cataloged variables
**Mode:** Audit-only (no code changes)

This section produces a binary SAFE or VULNERABLE verdict for every variable inventoried in Phase 68. The analysis cross-references the Forward-Trace Catalog (CW-01), Backward-Trace Catalog (CW-02), and Mutation Surface Catalog (CW-03) to prove that no player-controllable action between VRF request and fulfillment can influence outcomes determined by that RNG word.

### Verdict Methodology

Each verdict is a three-column proof:

1. **Permissionless mutation paths** -- sourced from the CW-03 Mutation Surface Catalog. Only permissionless paths matter for commitment window safety because admin-only, governance-only, and VRF-coordinator-only paths cannot be exploited by players during normal operation. For each permissionless writer, the call-graph depth (D0/D1/D2/D3+) is recorded per MUT-03.

2. **Guard analysis** -- evaluated for BOTH commitment windows:
   - **Daily window:** `rngLockedFlag` is TRUE and `prizePoolFrozen` is TRUE. This window opens when `_swapAndFreeze` executes during `advanceGame` (AdvanceModule:230) and closes when `_unlockRng` completes (AdvanceModule:1406-1412). The VRF request is sent during this window; the VRF callback (`rawFulfillRandomWords`) arrives in a separate later transaction. During this window, the `rngLockedFlag` guard blocks `reverseFlip()`, `requestLootboxRng()`, `burn()`/`burnWrapped()`, `setCoinflipAutoRebuy()`, and `purchaseDeityPass()`. The `prizePoolFrozen` flag redirects purchase revenue from `currentPrizePool` to `prizePoolPendingPacked`.
   - **Mid-day window:** `rngLockedFlag` is FALSE and `prizePoolFrozen` is FALSE. This window opens when `requestLootboxRng()` sends a VRF request (AdvanceModule:674-741) and closes when `rawFulfillRandomWords` callback stores `lootboxRngWordByIndex[index]` (AdvanceModule:1449-1456). The mid-day VRF fulfillment ONLY writes `lootboxRngWordByIndex[index]` -- it does NOT trigger advanceGame, rngGate, or any outcome computation. The word sits in storage until a player opens a lootbox or resolves a Degenerette bet.

3. **Outcome influence** -- which of the 7 backward-trace categories (from CW-02) consume this variable as an INPUT to outcome computation:
   - Cat 1: Coinflip Win/Loss
   - Cat 2: Jackpot ETH Winner Selection
   - Cat 3: Jackpot BURNIE (Coin) Winner
   - Cat 4: Lootbox Outcome
   - Cat 5: Redemption Roll
   - Cat 6: Prize Pool Consolidation and Future Take
   - Cat 7: Degenerette (Roulette) Outcome

   If no category consumes the variable as an INPUT, the variable is SAFE regardless of mutability -- mutations cannot influence any VRF-dependent outcome.

**Verdict decision tree:**

1. No permissionless mutation paths exist (VRF-only / admin-only / game-internal-only writers) --> **SAFE** (no player can mutate during any window)
2. Permissionless mutation paths exist, but ALL are guarded by `rngLockedFlag` --> check mid-day window separately:
   - Mid-day VRF fulfillment does not consume this variable --> **SAFE**
   - Mid-day VRF fulfillment consumes this variable --> requires deeper analysis
3. Permissionless, unguarded, but outcome-irrelevant (variable is not consumed as an INPUT by any of the 7 backward-trace categories) --> **SAFE** (mutable but cannot influence outcomes)
4. Permissionless, unguarded, outcome-relevant, but separated by index/day keying (writes target a different key than the one consumed by the pending VRF path) --> **SAFE** (temporal/index separation prevents influence)
5. Permissionless, unguarded, outcome-relevant, no separation --> **VULNERABLE** (requires fix recommendation per MUT-02)

**Category explanations:**

**VRF-only / game-internal write (17 variables):** These variables are ONLY written by the VRF coordinator callback, by advanceGame's own internal processing pipeline, or by game-only callers (e.g., `processCoinflipPayouts` is restricted to `onlyDegenerusGameContract`). A player calling advanceGame between VRF request and fulfillment triggers a DIFFERENT code path -- rngGate checks `rngWordCurrent != 0` (the VRF word must be present) and reverts if no word has arrived. advanceGame-internal writes are part of the VRF processing itself, not external mutations (per Research Pitfall 4). Variables like `rngWordCurrent` and `vrfRequestId` are written by `requestLootboxRng()` (permissionless), but these writes are VRF request setup operations (clearing to 0 / storing a new request ID) that do not influence the VRF word the coordinator will return. The `rngRequestTime` variable is only written by advanceGame (via `_applyDailyRng`) and by admin-only `configureVrf()`.

**rngLockedFlag guarded (11 variables):** The `rngLockedFlag` guard blocks all permissionless writes during the daily RNG commitment window. For `totalFlipReversals`: `reverseFlip()` checks `rngLockedFlag` at AdvanceModule:1420. For StakedDegenerusStonk burn variables (7 vars): `burn()`/`burnWrapped()` check `game.rngLocked()` at StakedDegenerusStonk:447/465. For `deityBySymbol`: `purchaseDeityPass()` checks `rngLockedFlag` at WhaleModule:467. For `ticketWriteSlot` and `ticketsFullyProcessed`: only `requestLootboxRng()` writes them (outside advanceGame), and `requestLootboxRng()` checks `rngLockedFlag` at AdvanceModule:675. During mid-day window (rngLockedFlag FALSE): these variables CAN be mutated, but mid-day VRF fulfillment (`rawFulfillRandomWords` when `!rngLockedFlag`) does NOT read any of them -- it only stores `lootboxRngWordByIndex[index]` and clears VRF state. The mutations are consumed only during the NEXT daily advanceGame cycle.

**Double-buffer protected (2 variables):** The `_swapTicketSlot` mechanism (DegenerusGameStorage:713-718) ensures that `ticketQueue` and `ticketsOwedPacked` writes from purchases go to the WRITE slot, while `processTicketBatch` reads from the READ slot. The swap is atomic (`ticketWriteSlot ^= 1`) and occurs before any VRF request. Post-swap purchases cannot contaminate the pre-swap read buffer.

**Index-keyed separation (7 variables):** All 7 lootbox per-player data variables are keyed by `[lootboxRngIndex][player]`. At VRF request time, the index is incremented BEFORE the request is sent. New purchases use the NEW index value; VRF fulfillment stores the word at `index - 1`. Per-player data written at the new index cannot influence the VRF word being stored at the old index, and the old per-player data (committed before the request) is immutable because only `purchase()` writes and each purchase creates a new entry at the current index.

**Freeze-gated / outcome-irrelevant in mid-day (3 variables):** During the daily window, `prizePoolFrozen = true` redirects purchase revenue to `prizePoolPendingPacked`, protecting `currentPrizePool` from direct mutation. During the mid-day window (freeze is off), `purchase()` CAN write directly to `currentPrizePool` and `prizePoolsPacked`. However, mid-day VRF fulfillment does NOT read these variables -- it only stores `lootboxRngWordByIndex[index]`. These variables are consumed later by `payDailyJackpot` (daily path), `_consolidatePrizePools` (daily path), and `_resolveFullTicketBet` (player-initiated resolution after fulfillment). The VRF word itself is not influenced by prize pool mutations. See per-variable verdicts for detailed analysis per Research Pitfall 5.

**Permissionless, unguarded, outcome-irrelevant (6 variables):** These variables are freely mutable during both windows but do not appear as INPUTS to any of the 7 outcome categories in the backward trace. `claimableWinnings` and `claimablePool` are WRITE destinations (outputs) in outcome paths -- jackpots credit winners by writing to them, but winner selection does not READ them as inputs. `autoRebuyState` is a player preference that converts winnings to tickets AFTER winner selection. `lootboxRngPendingEth`/`lootboxRngPendingBurnie` are threshold accumulators cleared to 0 at RNG request time. `playerState` (BurnieCoinflip) tracks claim cursors and auto-rebuy preferences -- it is written by `depositCoinflip()`, `claimCoinflips()`, and `setCoinflipAutoRebuy()`, but it does not feed into any VRF-dependent outcome computation (processCoinflipPayouts reads `coinflipBalance` for payout amounts, not `playerState`; playerState is updated as a side-effect during auto-claim).

**Permissionless, special analysis required (5 variables):** These variables have unguarded permissionless writes and appear in at least one backward-trace outcome category. Each requires individual deep analysis -- see Per-Variable Verdicts below:
- `coinflipBalance`: written by `depositCoinflip()` (no rngLockedFlag guard), appears in Cat 1 backward trace. Requires `_targetFlipDay()` temporal separation analysis.
- `bountyOwedTo`: written by `depositCoinflip()` record-flip path (with `!game.rngLocked()` guard at BurnieCoinflip:645), appears in Cat 1 backward trace. Requires outcome-influence analysis.
- `degeneretteBets`: written by `placeFullTicketBets()` (no rngLockedFlag guard), appears in Cat 7 backward trace. Requires index-keyed analysis.
- `degeneretteBetNonce`: written by `placeFullTicketBets()` (no rngLockedFlag guard), appears in Cat 7 backward trace. Requires same analysis.
- `midDayTicketRngPending`: written by `requestLootboxRng()`, not consumed by VRF outcome categories, but requires confirmation.

#### coinflipBalance[epoch][player] (slot 0 CF) -- SAFE

**Permissionless writers:** depositCoinflip() D2 (_depositCoinflip -> _addDailyFlip -> coinflipBalance[day][player] += amount). Game-only: creditFlip() D1, creditFlipBatch() D1.

### Cross-Reference Proof (CW-04)

**Claim:** No external function callable by a non-admin actor can mutate any committed input between VRF request and fulfillment in a way that influences VRF-dependent outcomes.

**Definitions:**

- **Committed input** = a variable that feeds into one of the 7 VRF-dependent outcome categories (Cat 1-7 from CW-02 backward trace) as a computational INPUT, not merely a write destination.
- **Between VRF request and fulfillment** = either (a) between `_finalizeRngRequest` setting `rngLockedFlag=true` and `_unlockRng` clearing it (daily window -- single advanceGame transaction followed by separate VRF callback transaction), or (b) between `requestLootboxRng` VRF request and `rawFulfillRandomWords` callback (mid-day window -- spans multiple blocks, no rngLockedFlag/prizePoolFrozen).
- **Non-admin actor** = any address except the DegenerusAdmin owner, governance (sDGNRS vote), or the VRF coordinator.

**Proof method:** Exhaustive enumeration of all 87 permissionless mutation paths from CW-03 Mutation Surface Catalog, organized by protection mechanism. For each path, we state which variable it writes, whether that variable is a committed input, and what guard or property blocks the mutation from influencing VRF-dependent outcomes during each window.

Per memory-mandated RNG audit discipline, tracing BACKWARD from each consumer to verify the word was unknown at input commitment time.

#### 1.4 Temporal Separation Property

The `_targetFlipDay()` function ensures deposits and resolution operate on different day indices, making key overlap impossible.

**Call chain:**
- `_targetFlipDay()` returns `degenerusGame.currentDayView() + 1` (BurnieCoinflip:1060-1062)
- `currentDayView()` returns `_simulatedDayIndex()` (DegenerusGame:506-507)
- `_simulatedDayIndex()` returns `GameTimeLib.currentDayIndex()` (DegenerusGameStorage:1262-1264)

**Proof of separation:**
- Deposits target day N+1 (via `_targetFlipDay()` at BurnieCoinflip:631)
- Resolution processes day N (the `day` parameter to `rngGate` / `processCoinflipPayouts`)
- No key overlap is possible because `currentDayView()` is time-derived (not a writable storage variable)
- During daily processing, `dailyIdx` is updated only at `_unlockRng(day)` (AdvanceModule:1409-1415) which runs AFTER `rngGate` (after all coinflip processing is complete)
- During mid-day window, `dailyIdx` already reflects today (prerequisite: daily RNG must have been processed)

**Edge case -- VRF stall:** During a VRF stall, `currentDayView()` continues advancing in real time. If a player deposits on stall day S, `_targetFlipDay()` returns `S + 1`. When the stall resolves, `_backfillGapDays` processes historical gap days (which are strictly earlier than S + 1). The deposit remains safe because the target day is ahead of all backfilled days.

#### 1.5 Nudge (reverseFlip) Impact

**Mechanism:**
- `reverseFlip()` increments `totalFlipReversals` (AdvanceModule:1428): `totalFlipReversals = reversals + 1`
- `_applyDailyRng` adds `totalFlipReversals` to `rawWord`: `finalWord = rawWord + nudges` (AdvanceModule:1524-1528)
- After application, `totalFlipReversals` is reset to 0 (AdvanceModule:1530)

**Effect on coinflip outcome:**
- `win = (rngWord & 1) == 1` (BurnieCoinflip:810)
- Each +1 to the word flips bit 0, toggling win/loss for ALL coinflip participants on that day
- Odd total nudges: outcome flipped from VRF base
- Even total nudges: outcome preserved (net zero effect)
- This is a collective mechanism -- a single nudge affects every player's coinflip for that day equally

**Guard:**
- `reverseFlip()` checks `if (rngLockedFlag) revert RngLocked()` (AdvanceModule:1423)
- This prevents nudging after VRF request is in-flight (when the base word becomes knowable upon fulfillment)
- Cost escalation: base 100 BURNIE, +50% compound per queued nudge (AdvanceModule:1547-1553)

**Assessment:** The reverseFlip mechanism is a deliberate design feature ("reverse flip") that provides collective outcome influence, not per-player manipulation. It is protected from commitment window exploitation by `rngLockedFlag` -- players cannot nudge after seeing the VRF word. The escalating cost prevents spam. The all-or-nothing collective nature means no individual can selectively benefit without affecting all participants.

- When `gameOver == true`, the function returns `false` (lock is NOT active), so deposits are NOT blocked by this guard
- There is NO separate game-over guard on `depositCoinflip` or `_depositCoinflip` or `_addDailyFlip`

**Result:** Deposits ARE allowed during game-over.

**Impact analysis:**
- If a player deposits during game-over, `_targetFlipDay()` returns `currentDayView() + 1` (BurnieCoinflip:1060-1062)
- `currentDayView()` is time-derived and continues advancing during game-over
- The game-over resolution processes the current game-over day (not future days)
- `_gameOverEntropy` resolves the day using either VRF (Path 3) or fallback (Path 4)
- A deposit during game-over targets day+1 from the current real-time day, which is AHEAD of the game-over resolution day

**Case 1: VRF available (Path 3):** The resolution word is from Chainlink VRF. The deposit targets a future day that will never be resolved (game is over, no more `advanceGame` calls). The player's stake for that future day is unresolvable -- the BURNIE was already burned on deposit (BurnieCoinflip:266) and cannot be recovered. This is a UX concern (lost funds) but NOT a security vulnerability -- no manipulation is possible.

**Case 2: VRF stalled, fallback (Path 4):** Same analysis. The fallback word resolves the current game-over day. Any deposit targets day+1, which is never resolved. The player loses their deposit.

**Case 3: Can a deposit target the SAME day being resolved?** No. `_targetFlipDay()` always returns `currentDayView() + 1`, and `currentDayView()` returns the time-based day index. The game-over resolution processes the current `day` parameter from `advanceGame`, which equals `dailyIdx` or `dailyIdx + 1`. Since `_targetFlipDay()` returns `currentDayView() + 1` and `currentDayView() >= dailyIdx`, the target is always `>= dailyIdx + 1`, meaning at minimum it equals the day being resolved (if `currentDayView() == dailyIdx`) or is ahead of it. However, if `currentDayView() == dailyIdx`, then `_targetFlipDay() == dailyIdx + 1` which IS the day being resolved in the next `advanceGame` call. But this deposit must occur BEFORE that `advanceGame` call, and the VRF word for that call is not yet known (it arrives via `rawFulfillRandomWords`). So the deposit is placed before the outcome is knowable.

**Verdict:** SAFE. No security vulnerability. Post-game-over deposits target future days that will never be resolved, resulting in lost deposits (UX/economic concern, not security). For deposits targeting the game-over resolution day itself, the VRF word is not yet known at deposit time, maintaining commitment window integrity.

**Severity:** INFO -- documented behavior. Players depositing after game-over lose their BURNIE. No manipulation vector exists because the temporal separation property holds even during game-over.

#### Attack 1: Deposit-During-Lock

**Attack Name:** Deposit-During-Lock (front-run VRF fulfillment with a deposit)

**Attacker Goal:** Place a bet on a coinflip whose outcome the attacker already knows (or can predict) by observing the VRF request transaction in the mempool and depositing before fulfillment.

**Preconditions:**
- Attacker monitors the mempool for VRF request transactions (`advanceGame()` calls that trigger `_requestRng`)
- Attacker has BURNIE tokens to deposit
- A daily VRF cycle is in progress (`rngLockedFlag == true`)

**Attack Steps:**
1. Attacker sees VRF request tx in mempool (or monitors `rngLockedFlag` on-chain)
2. Attacker calls `depositCoinflip(attacker, amount)` (BurnieCoinflip:225)
3. `_depositCoinflip` executes: burns BURNIE (line 266), processes quests (line 275), calls `_addDailyFlip` (line 308)
4. `_addDailyFlip` writes to `coinflipBalance[_targetFlipDay()][attacker]` (line 637)
5. `_targetFlipDay()` returns `degenerusGame.currentDayView() + 1` (BurnieCoinflip:1060-1062)
6. VRF is fulfilled via `rawFulfillRandomWords` (AdvanceModule:1439) -- stores `rngWordCurrent`
7. Next `advanceGame()` call processes `rngGate` (AdvanceModule:768) which calls `processCoinflipPayouts(bonusFlip, currentWord, day)` (AdvanceModule:799) for TODAY's day index
8. Attacker's deposit targets day+1 (tomorrow), NOT today's resolution day

**Target State:** `coinflipBalance[day+1][attacker]`

**Defense Mechanism:** `_targetFlipDay()` = `currentDayView() + 1` (BurnieCoinflip:1060-1062). Deposits ALWAYS target TOMORROW. Resolution processes TODAY via the `day` parameter to `processCoinflipPayouts` (AdvanceModule:799). These are strictly disjoint: `_targetFlipDay() >= dailyIdx + 1` while resolution processes `day == dailyIdx` (or gap days that are strictly earlier). The VRF word that resolves the attacker's deposit has not yet been requested at deposit time.

**Gap-day edge case:** During VRF stall, `_backfillGapDays` (AdvanceModule:1473-1491) processes historical gap days (startDay to endDay-1). `_targetFlipDay()` uses real-time `currentDayView()` which is always ahead of any backfilled gap day. A deposit during a stall targets a day beyond all gap days, maintaining temporal separation.

**Verdict:** SAFE
**Feasibility:** Not feasible
**C4A Severity:** N/A

#### Attack 2: Auto-Rebuy Toggle Extraction

**Attacker Goal:** After seeing the daily VRF word on-chain (via `rawFulfillRandomWords` event or reading `rngWordCurrent`), toggle auto-rebuy OFF to extract accumulated `autoRebuyCarry` if today's flip is a loss, avoiding the carry being zeroed.

**Preconditions:**
- Attacker has `autoRebuyEnabled == true` with accumulated `autoRebuyCarry > 0`

**Both functions guarded:**
- `setCoinflipAutoRebuy`: `if (degenerusGame.rngLocked()) revert RngLocked()` (BurnieCoinflip:706)

#### Attack 3: Bounty-Arm During Mid-Day Window

**Attack Name:** Bounty-Arm During Mid-Day Window (become bounty recipient via record flip)

**Attacker Goal:** During mid-day (when `rngLockedFlag == false`), make a record-breaking deposit to set `bountyOwedTo` to the attacker's address, ensuring the attacker receives the bounty payout if the next daily resolution is a win.

**Preconditions:**
- `rngLockedFlag == false` (mid-day window, between daily processing completion and next VRF request)
- Attacker has enough BURNIE to exceed `biggestFlipEver`
- `!game.rngLocked()` is true (required by BurnieCoinflip:645)

**Attack Steps:**
1. Attacker calls `depositCoinflip(attacker, largeAmount)` (BurnieCoinflip:225) during mid-day

#### Attack 7: Game-Over Predictable Fallback

**Attack Name:** Game-Over Predictable Fallback (bet on coinflip with known fallback word)

**Attacker Goal:** After game-over is triggered and the VRF stalls for 3+ days, the fallback word derivation is deterministic from on-chain data. The attacker deposits on the game-over day to bet on a coinflip whose outcome can be predicted from the fallback word.

**Preconditions:**
- `gameOver == true` (game has ended)
- VRF request was sent but has stalled for 3+ days (`elapsed >= GAMEOVER_RNG_FALLBACK_DELAY`, where `GAMEOVER_RNG_FALLBACK_DELAY = 3 days` at AdvanceModule:91)
- `rngWordCurrent == 0` (VRF word not yet delivered)
- `rngRequestTime != 0` (request is pending)
- Attacker can compute the fallback word from on-chain data

**Attack Steps:**
1. Game-over is triggered; daily VRF request is sent (`_requestRng` sets `rngLockedFlag = true`)
2. VRF stalls for 3+ days; `elapsed >= GAMEOVER_RNG_FALLBACK_DELAY` (AdvanceModule:902)
3. Attacker reads on-chain state: `rngWordByDay[1]` through `rngWordByDay[5]` (earliest historical words)
4. Attacker computes fallback word using `_getHistoricalRngFallback(day)` algorithm (AdvanceModule:962-986):
   - Combines up to 5 earliest `rngWordByDay` entries via chained `keccak256(combined, w)` (AdvanceModule:971)
   - Final word: `keccak256(combined, currentDay, block.prevrandao)` (AdvanceModule:983)
5. **Key dependency:** The final hash includes `block.prevrandao` (AdvanceModule:983), which is NOT known until the block is produced
6. Attacker attempts to deposit: calls `depositCoinflip(attacker, amount)` (BurnieCoinflip:225)
7. `_depositCoinflip` checks `_coinflipLockedDuringTransition()` (BurnieCoinflip:252)
8. `_coinflipLockedDuringTransition()` returns (BurnieCoinflip:1012): `(!inJackpotPhase) && !degenerusGame.gameOver() && lastPurchaseDay_ && rngLocked_ && (purchaseLevel_ % 10 == 0)`
9. Since `gameOver == true`, the `!degenerusGame.gameOver()` term is `false`, so the function returns `false` (NOT locked) -- **deposit is allowed**
10. `_targetFlipDay()` returns `currentDayView() + 1` (day+1), NOT the game-over resolution day
11. `_gameOverEntropy` processes the fallback: `_applyDailyRng(day, fallbackWord)` (AdvanceModule:905) then `coinflip.processCoinflipPayouts(isTicketJackpotDay, fallbackWord, day)` (AdvanceModule:907-911) -- resolving the game-over day
12. Attacker's deposit targets day+1, which will NEVER be resolved (no more `advanceGame` calls after game-over resolution)

**Target State:** `coinflipBalance[day+1][attacker]` -- but this day is never resolved

**Defense Mechanism:** Multiple layers:

1. **Temporal separation (primary):** `_targetFlipDay()` returns `currentDayView() + 1` (BurnieCoinflip:1060-1062). Post-game-over deposits target a future day that will never be resolved by `_gameOverEntropy` (which resolves the current `day` parameter, not day+1). The attacker's BURNIE is burned (BurnieCoinflip:266) and the coinflip stake for the target day is permanently stranded.

2. **`block.prevrandao` in fallback (secondary):** Even if the temporal separation did not exist, the fallback word includes `block.prevrandao` in its final derivation (AdvanceModule:983). This value is determined by the block's proposer and is NOT predictable before the block is produced. While `block.prevrandao` has limited entropy (1 bit bias for a proposer controlling a single validator), the attacker cannot pre-compute the exact fallback word before the resolution transaction executes. A proposer-attacker could bias 1 bit but cannot fully control the outcome.

3. **No same-day deposit (tertiary):** Even if an attacker could predict the fallback word AND somehow target the game-over day, `_targetFlipDay()` always adds +1 to `currentDayView()`. There is no mechanism to deposit for a specific past or current day. The attacker cannot target the exact day being resolved.

**Impact of post-game-over deposits:** The attacker loses their BURNIE (burned on deposit at BurnieCoinflip:266). The coinflip stake sits in `coinflipBalance[day+1]` forever, as no future resolution will process it. This is a self-inflicted loss, not an exploitation vector.

**Detailed feasibility analysis for deposit-blocking:**
- `_coinflipLockedDuringTransition()` does NOT block during game-over (returns `false` when `gameOver == true`)
- There is no separate `gameOver` guard on `depositCoinflip`, `_depositCoinflip`, or `_addDailyFlip`
- Deposits ARE allowed during game-over, but they target a future day that is never resolved
- The depositor loses their BURNIE -- the protocol does not refund stranded coinflip stakes
- This is documented as INFO-level (UX concern: user loses funds if they deposit after game-over)

**Verdict:** SAFE
**Feasibility:** Feasible but harmless (attacker can deposit but loses their BURNIE; cannot bet on a known outcome due to day+1 keying)
**C4A Severity:** Informational (post-game-over deposits result in lost BURNIE; no manipulation vector exists)

**During the daily commitment window:**

- `currentDayView()` returns the current real-time day index. Call this N.
- `dailyIdx` has NOT yet been updated (it updates at `_unlockRng` after all processing completes, at AdvanceModule:1410).
- `_targetFlipDay()` therefore returns N + 1.
- `processCoinflipPayouts` resolves day N (the day parameter passed from `rngGate`).
- Deposits during the window write to `coinflipBalance[N+1][player]` (BurnieCoinflip:637).
- Day N's `coinflipBalance[N][player]` was committed in prior transactions (before the window opened).

**Conclusion:** Deposits during the daily commitment window target day N+1. `processCoinflipPayouts` resolves day N. No deposit can modify the balance being resolved. The temporal separation property (`_targetFlipDay = currentDayView + 1`) provides complete protection. This is consistent with Phase 69 verdict #43 ("Day-keyed temporal separation") and Phase 70 Section 1.4.

#### Part 4: Open Question Resolution -- _requestRng vs _swapAndFreeze Ordering

**Question (from Research Open Question 2):** Is `_swapAndFreeze` always called before `_requestRng`, or the reverse?

**Answer: `_requestRng` fires FIRST, `_swapAndFreeze` fires SECOND.** Documented with exact line citations:

1. `advanceGame()` enters the RNG path at AdvanceModule:222-236
2. `rngGate(ts, day, purchaseLevel, lastPurchase, bonusFlip)` is called at line 225
3. Inside `rngGate`, when no VRF word is available and no pending request exists (branch (d)):
   - `_requestRng(isTicketJackpotDay, lvl)` is called at line 839
   - `_requestRng` calls `vrfCoordinator.requestRandomWords(...)` at line 1263
   - Then calls `_finalizeRngRequest(isTicketJackpotDay, lvl, id)` at line 1273
   - `_finalizeRngRequest` sets `rngLockedFlag = true` at line 1325
   - `rngGate` returns `1` (sentinel) at line 840
4. Back in `advanceGame()`, `rngWord == 1` is detected at line 232
5. `_swapAndFreeze(purchaseLevel)` is called at line 233
6. `_swapAndFreeze` swaps the ticket buffer and sets `prizePoolFrozen = true` (GameStorage:723-729)

**The ordering is definitively:** `_requestRng` (sets `rngLockedFlag = true`) -> return -> `_swapAndFreeze` (swaps ticket buffer, sets `prizePoolFrozen = true`).

**Why the gap between them is not exploitable:** Both `_requestRng` and `_swapAndFreeze` execute within the SAME transaction. The EVM does not allow external calls to interleave during a single transaction's execution (no reentrancy point between these two internal calls). The only external call that occurs is `vrfCoordinator.requestRandomWords()` at line 1263, which is inside `_requestRng` and returns BEFORE `_finalizeRngRequest` runs. After `_finalizeRngRequest` sets `rngLockedFlag = true`, the only code that runs before `_swapAndFreeze` is `rngGate` returning the sentinel `1` and `advanceGame` checking the `if (rngWord == 1)` condition -- all internal, no external calls.

**Timeout retry path (branch (c)):** When `rngRequestTime != 0` and `elapsed >= 12 hours`, `_requestRng` is called at line 832. This is a retry -- the ticket buffer was ALREADY swapped in the original `advanceGame` call. `_finalizeRngRequest` detects the retry (`isRetry = true` at line 1310-1312) and skips `lootboxRngIndex++`. The buffer swap is NOT repeated. This is correct because the original swap is still in effect.

#### Part 5: Dual Sub-Window Summary Verdict

| Sub-Window | Duration | Guards Active | Key Invariant | Verdict |
|-----------|----------|---------------|---------------|---------|
| Period A: VRF in-flight | `_requestRng` to `rawFulfillRandomWords` (typically 2-4 blocks; max 12h before retry) | rngLockedFlag=true, prizePoolFrozen=true, double-buffer active, index-keying active, day+1 keying active | VRF word unknown; no on-chain state can predict or influence the word | **SAFE** |
| Period B: Word stored, unprocessed | `rawFulfillRandomWords` to next `advanceGame` (variable, incentivized by escalating bounty) | Same as Period A (rngLockedFlag still true, all guards remain) | VRF word committed; all state read during Period C processing is frozen or keyed | **SAFE** |
| Period C: Processing | Single `advanceGame` tx (rngGate through _unlockRng) | N/A (processing is atomic within one tx; no external mutation possible mid-tx) | All consumers fire sequentially within a single atomic transaction | **SAFE** |

**Overall DAYRNG-02 verdict: The daily VRF commitment window is SAFE.** All permissionless actions during the window are either:
1. **Blocked by guards** (reverseFlip, burn/burnWrapped, setDecimatorAutoRebuy -- revert with RngLocked/BurnsBlockedDuringRng)
2. **Write to outcome-irrelevant state** (setAutoRebuy, claimWinnings -- affect payout routing, not winner selection)
3. **Separated by temporal/index keying** (purchase ticket buffer writes to opposite slot; purchase prize contributions route to pending accumulators; coinflip deposits target day+1; degenerette bets key to next lootbox index)

This is consistent with Phase 69's 51/51 SAFE finding but narrows the proof to the advanceGame-specific temporal context, adding the dual sub-window analysis (Period A: VRF in-flight + Period B: word-stored-but-unprocessed) that Phase 69's per-variable approach did not explicitly address.
