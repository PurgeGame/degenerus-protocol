### NatSpec Discrepancies

**D1: Misplaced payDailyJackpot NatSpec block (lines 258-287)**

The comprehensive `@notice`/`@dev` documentation for `payDailyJackpot` (describing both daily and early-burn paths) appears in lines 258-278, but is positioned above `runTerminalJackpot` (line 288), not above `payDailyJackpot` (line 329). The Solidity compiler associates the entire `///` comment block (lines 258-287) with `runTerminalJackpot`, leaving `payDailyJackpot` with only `@param` tags (lines 326-328).

Content accuracy of the misplaced block: The descriptions are **factually correct** when verified against PAY-01 and PAY-02:
- Daily path: "6%-14% slice of remaining currentPrizePool" -- matches `DAILY_CURRENT_BPS_MIN=600, DAILY_CURRENT_BPS_MAX=1400`
- "Day 5: 100% of remaining currentPrizePool" -- matches `dailyBps = 10_000` at line 374
- "1% from the unified future pool" -- matches `reserveSlice = _getFuturePrizePool() / 100` at line 433
- Early-burn path: "1% futurePrizePool ETH slice with 75% converted to lootbox tickets" -- matches `PURCHASE_REWARD_JACKPOT_LOOTBOX_BPS = 7500`

**Severity:** INFORMATIONAL -- NatSpec is accurate but misplaced. No security impact. Tools parsing NatSpec (Etherscan, documentation generators) would show the wrong description for each function.

**D2: runTerminalJackpot inherits unrelated documentation**

As a consequence of D1, `runTerminalJackpot`'s compiled NatSpec would include the full payDailyJackpot description (lines 258-278) prepended to its own description (lines 280-287). The actual `@notice`/`@dev`/`@param`/`@return` tags for `runTerminalJackpot` (lines 280-287) are correct:
- "Terminal jackpot for x00 levels: Day-5-style bucket distribution" -- matches code using `FINAL_DAY_SHARES_PACKED` (60/13/13/13)
- "Called via IDegenerusGame(address(this)) from EndgameModule and GameOverModule" -- matches `msg.sender != ContractAddresses.GAME` guard

### Delegatecall Context Assessment

- Line 19: "Delegate-called module that hosts the jackpot distribution logic for DegenerusGame"
- Lines 22-23: "This contract is ONLY meant to be invoked via delegatecall from the main game contract. Storage layout inherits from DegenerusGameStorage"

### PAY-03-I01 winnerMask Status

The PAY-03-I01 finding (unused winnerMask variable) pertains to `DegenerusJackpots.sol` and `DegenerusGameEndgameModule.sol`, **not** DecimatorModule.sol. The winnerMask is returned by `DegenerusJackpots.runBafJackpot()` but unused by the caller in EndgameModule. Within DecimatorModule.sol, there are **zero references** to winnerMask. The NatSpec in `DegenerusJackpots.sol` (lines 105, 110, 214, 219, 226) correctly describes winnerMask's purpose for scatter ticket routing, and the `@return winnerMask` tag at line 226 is accurate to what the function returns. The finding is that the caller (EndgameModule) discards the value -- this is a dead-code observation classified as INFO with no NatSpec inaccuracy.

**Status: PAY-03-I01 NatSpec is accurate in the source contract (DegenerusJackpots.sol). The dead-code issue is in the caller, not in the NatSpec.**

### baseFuturePool vs futurePoolLocal Distinction

The plan specifically asked to verify this distinction in DecimatorModule. Analysis:

- DecimatorModule does NOT use `baseFuturePool` or `futurePoolLocal` directly. These are variables in `EndgameModule.runRewardJackpots()`.
- DecimatorModule operates on `poolWei` (passed as parameter), `futurePrizePool` (via `_getFuturePrizePool()`/`_setFuturePrizePool()`), and `claimablePool`.
- The pool source distinction is correctly documented in the audit ground truth (`audit/v3.0-payout-scatter-decimator.md`): normal decimator gets 10% of `futurePoolLocal`, x00 decimator gets 30% of `baseFuturePool`. These are computed in EndgameModule and passed as `poolWei` to DecimatorModule.
- DecimatorModule's NatSpec does not reference these internal EndgameModule variables, which is correct -- it would be misleading for DecimatorModule to document pool source details that belong to its caller.

**No comment inaccuracy regarding pool sources.**
