Pool Architecture Overview

All ETH entering the protocol flows through a 4-pool lifecycle. futurePrizePool accumulates incoming ETH, transitions to currentPrizePool at each level, which is drawn down by jackpot distributions into claimablePool. Players claim from claimablePool via claimWinnings(). The yieldAccumulator collects stETH rebasing surplus and releases 50% to futurePrizePool at x00 milestones.

1. Jackpot Distribution

PAY-01: Purchase-Phase Daily Jackpot

TriggerDaily, during purchase phase when daysSince > 0 and lvl > 1
Source PoolfuturePrizePool (1% daily drip via poolBps = 100)
CalculationethDaySlice = (_getFuturePrizePool() * poolBps) / 10_000 then split: lootboxBudget = (ethPool * PURCHASE_REWARD_JACKPOT_LOOTBOX_BPS) / 10_000 where PURCHASE_REWARD_JACKPOT_LOOTBOX_BPS = 7500

// JackpotModule.sol:644-651 -- 75/25 split

// Remaining 25% distributed as ETH to winners

Edge Cases

  • Empty pool: If futurePrizePool == 0, ethDaySlice = 0, no distribution occurs
  • No winners at trait bucket: Unfilled slots refund to yield surplus (not returned to pool)
  • Auto-rebuy dust: Unconverted remainder dropped; max loss = ticketPrice - 1 wei per payout

ETH PAY-02

Source PoolcurrentPrizePool (6-14% BPS days 1-4, 100% day 5)

Edge Cases

  • Compressed mode: When prize pool is below threshold, counterStep = 2 doubles logical days per physical day. Day 1 is never compressed (preserves early-bird lootbox)
  • Turbo mode: Instant resolution without VRF wait for small pools; counterStep = JACKPOT_LEVEL_CAP processes all 5 days in one physical day
  • Day 5 depletes 100%: dailyBps = 10_000 (100%) of currentPrizePool -- no carry to next jackpot phase
  • randomBps range: [600, 1400] ensures minimum 6% and maximum 14% per day for days 1-4

Edge Cases

  • Auto-rebuy bonus absorption: 130% (normal) / 145% (enhanced) ticket bonus covered by structural 2x over-collateralization, yielding net 1.38x-1.54x coverage
  • prizePoolFrozen guard: Blocks decimator claims during jackpot phase days 1-4 to prevent pool depletion mid-draw
  • Atomic pool transition: Pool transition is atomic per level -- no partial transitions possible
  • Empty pool at transition: If futurePrizePool is 0 at transition, nextPrizePool receives 0 (no error, just empty jackpot phase)

2. Scatter / Decimator

Pool Source Distinction: BAF scatter and x00 decimator use baseFuturePool (a snapshot taken at the start of level processing). Normal decimator uses futurePoolLocal (the running total after BAF deduction). This means the normal decimator's 10% is calculated AFTER the BAF scatter has already deducted its share from the pool.

BAF Normal Scatter (PAY-03)

Fires at levels 10, 20, 30, etc. (every lvl % 10 == 0). Uses 10% of baseFuturePool, except at level 50 where the percentage is elevated to 20%. The pool allocation is distributed across 7 prize categories via the _runBafJackpot function in DegenerusJackpots.

BAF Century Scatter (PAY-04)

Fires at levels 100, 200, 300, etc. (every prevMod100 == 0). Uses 20% of baseFuturePool. Century scatter uses a distinct sampling pattern: 4 rounds at lvl+1, 4 rounds at lvl+2, 4 rounds at lvl+3, then 38 rounds randomly selected from the past 99 levels. Both share the same 7-category distribution mechanics via _runBafJackpot.

Formula

// EndgameModule.sol:168-172
uint256 bafPct = prevMod100 == 0 ? 20 : (lvl == 50 ? 20 : 10);
uint256 bafPoolWei = (baseFuturePool * bafPct) / 100;
futurePoolLocal -= bafPoolWei;

// 7-category split -- DegenerusJackpots.sol:229-530
// A:  10% -- Active player reward (ETH)
// A2:  5% -- Secondary active player reward (ETH)
// B:   5% -- Bonus category (lootbox)
// D:   5% -- DGNRS holder reward
// D2:  5% -- Secondary DGNRS holder reward
// E1: 45% -- Primary endgame reward (ETH)
// E2: 25% -- Secondary endgame reward (lootbox)

Code: EndgameModule.sol:138-408, DegenerusJackpots.sol:229-530   Commit: 3fa32f51

Edge Cases

  • No ticket holders at target level: Prize returns to futurePoolLocal via toReturn (DegenerusJackpots.sol:259, 290). No funds are lost.
  • Single player with large win: 50/50 ETH/lootbox split; whale pass queueing via _queueWhalePassClaimCore if payout exceeds 5 ETH.
  • Level 50 special case: Gets 20% instead of standard 10% (lvl == 50 ? 20 : 10).
  • Century scatter distinct sampling: 4+4+4+38 rounds (vs normal scatter pattern). Both use the same distribution mechanics.

Decimator Normal Claims (PAY-05)

Uses 10% of futurePoolLocal (the running total AFTER BAF scatter has deducted its share). Pool allocation at EndgameModule.sol:210. Claims are pro-rata by the amount of tokens each player burned in the decimator round. The 50/50 split between ETH and lootbox tickets is enforced in the claim path.

Decimator x00 Claims (PAY-06)

Uses 30% of baseFuturePool (the snapshot taken at the start of level processing, before any BAF deduction). Pool allocation at EndgameModule.sol:194. At x00 levels where both BAF century and x00 decimator fire, the combined draw is up to 50% of baseFuturePool (20% BAF + 30% decimator). Shares the same runDecimatorJackpot resolution and claimDecimatorJackpot claim path as normal decimator.

Formula

// Pool allocation -- EndgameModule.sol:210
uint256 decPoolWei = (futurePoolLocal * 10) / 100; // Normal: 10% of futurePoolLocal

// Pool allocation -- EndgameModule.sol:194
uint256 decPoolWei = (baseFuturePool * 30) / 100;  // x00: 30% of baseFuturePool

// Claim formula -- DecimatorModule.sol:297-547
payout = (poolWei * playerBurn) / totalBurn;  // Pro-rata by burn
ethPortion = payout / 2;                       // 50% ETH
lootboxPortion = payout - ethPortion;          // 50% lootbox

Code: DecimatorModule.sol:297-547, EndgameModule.sol:175-210, DegenerusGame.sol:1293-1305   Commit: 3fa32f51

Edge Cases

  • totalBurn == 0: No winners; full pool returned as decRefund. In normal gameplay, recycled to the next round. In GAMEOVER, returned to terminal jackpot.
  • lastDecClaimRound overwrite: By-design per v1.1 spec (EDGE-04). Previous round's unclaimed funds are recycled to the next round. Only one decimator round can be active at a time.
  • claimed[player] flag: Prevents double-claim within a round. Per-level per-player flag.
  • prizePoolFrozen guard: Blocks claims during jackpot phase days 1-4 to prevent interference with jackpot distribution.

3. Coinflip Economy

BURNIE Isolation: The coinflip economy operates entirely in BURNIE tokens via a burn-and-mint model. It is fully isolated from the ETH claimablePool -- no coinflip operation can affect protocol ETH solvency. Net house edge is ~1.575% per flip (structurally deflationary).

PAY-07: Coinflip Deposit / Win / Loss

BURNIE PAY-07
TriggerPlayer deposits BURNIE into coinflip; resolved via VRF the following day
Source PoolN/A -- BURNIE burn-and-mint model (no pool, virtual stakes)
Calculationwin = (rngWord & 1) == 1 (50/50). Variable multiplier: roll 0 = 1.5x, roll 1 = 2.5x, rolls 2-19 = 1.78x-2.15x. Payout = stake + (stake * uint256(rewardPercent)) / 100
RecipientsWinning player (BURNIE minted) or house (BURNIE burned on loss)
Claim MechanismclaimCoinflips() or claimCoinflipsFromBurnie() -- both route to _claimCoinflipsInternal
CurrencyBURNIE

Formula

// BurnieCoinflip.sol:813-825 -- variable multiplier

// roll == 0:    rewardPercent = 50  (1.5x payout)
// roll == 1:    rewardPercent = 150 (2.5x payout)
// roll 2-19:    rewardPercent = seedWord % 38 + 78 (1.78x-2.15x)

// Payout:
uint256 payout = stake + (stake * uint256(rewardPercent)) / 100;

// Net house edge: ~1.575% (players pay via expected burn > expected mint)

Code: BurnieCoinflip.sol:224-627, BurnieCoin.sol:518-529   Commit: 3fa32f51

Edge Cases

  • Zero stake: coinflipBalance cleared to 0 on first processing; no double-payout path exists.
  • RNG locked: rngLockedFlag blocks claims during VRF resolution on all 6 external claim/toggle paths.
  • Claim window asymmetry: 30 days first-time, 90 days returning (by-design; PAY-07-I01 INFORMATIONAL). First-time players who wait 31+ days lose winnings.
  • Auto-rebuy carry zeroed on loss: carry = 0 on loss branch -- all accumulated carry is destroyed.
  • Fully isolated from ETH claimablePool: No coinflip path mutates ETH pool accounting.

Edge Cases

  • Bounty not armed: Requires new all-time record daily volume (not just high volume). Re-arming requires exceeding current record by at least 1%.
  • DGNRS gating: If bet < 50k BURNIE or pool < 20k BURNIE, DGNRS bounty not awarded even on record volume.
  • Half-pool payout: Ensures bounty is never fully depleted in one trigger. On loss, half is destroyed (supply sink).
  • Bounty payout cannot re-arm: _addDailyFlip called with canArmBounty=false, preventing self-triggering.

Edge Cases

  • WWXRP mint authority is compile-time restricted: Only GAME, COIN, COINFLIP addresses can call mintPrize(). Cannot be changed by admin -- permanent restriction via ContractAddresses library.
  • Recycling rates are fixed per deity tier: Normal 1% (capped 1000 BURNIE), afKing 1.6% base, max deity 3.1%. Deity bonus applies only to first 1M BURNIE (DEITY_RECYCLE_CAP).
  • Boons expire after 2 days: COINFLIP_BOON_EXPIRY_DAYS = 2. Cap at 100k BURNIE per boon. Tiers: 5%, 10%, 25%.
  • No extraction amplification through recycling: Net deflationary. Combined recycling + boon is bounded and non-compounding. 50/50 loss risk prevents unbounded extraction.
