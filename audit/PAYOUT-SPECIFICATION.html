Pool Architecture Overview

All ETH entering the protocol flows through a 4-pool lifecycle. futurePrizePool accumulates incoming ETH, transitions to currentPrizePool at each level, which is drawn down by jackpot distributions into claimablePool. Players claim from claimablePool via claimWinnings(). The yieldAccumulator collects stETH rebasing surplus and releases 50% to futurePrizePool at x00 milestones.

1. Jackpot Distribution

PAY-01: Purchase-Phase Daily Jackpot

ETH PAY-01
TriggerDaily, during purchase phase when daysSince > 0 and lvl > 1
Source PoolfuturePrizePool (1% daily drip via poolBps = 100)
CalculationethDaySlice = (_getFuturePrizePool() * poolBps) / 10_000 then split: lootboxBudget = (ethPool * PURCHASE_REWARD_JACKPOT_LOOTBOX_BPS) / 10_000 where PURCHASE_REWARD_JACKPOT_LOOTBOX_BPS = 7500
RecipientsVRF-selected ticket holders across 4 trait buckets (Rare / Epic / Legendary / Unique)
Claim MechanismclaimWinnings() via claimableWinnings[player] mapping
Currency75% lootbox tickets, 25% ETH

Formula

// JackpotModule.sol:635-639
uint256 poolBps = 100; // 1% daily drip from futurePool
ethDaySlice = (_getFuturePrizePool() * poolBps) / 10_000;
_setFuturePrizePool(_getFuturePrizePool() - ethDaySlice);

// JackpotModule.sol:644-651 -- 75/25 split
lootboxBudget = (ethPool * PURCHASE_REWARD_JACKPOT_LOOTBOX_BPS) / 10_000;
// PURCHASE_REWARD_JACKPOT_LOOTBOX_BPS = 7500 (75%)
// Remaining 25% distributed as ETH to winners

Code: JackpotModule.sol:619-673, PayoutUtils.sol:30-74   Commit: 3fa32f51

Edge Cases

  • Empty pool: If futurePrizePool == 0, ethDaySlice = 0, no distribution occurs
  • No winners at trait bucket: Unfilled slots refund to yield surplus (not returned to pool)
  • Auto-rebuy dust: Unconverted remainder dropped; max loss = ticketPrice - 1 wei per payout

PAY-02: Jackpot-Phase 5-Day Draws

ETH PAY-02
TriggerDuring jackpot phase (5-day sequence), once per day
Source PoolcurrentPrizePool (6-14% BPS days 1-4, 100% day 5)
CalculationdailyBudget = (currentPrizePool * randomBps) / BPS_DENOMINATOR where randomBps is random in [600, 1400] for days 1-4, and 10000 for day 5. Share split: JACKPOT_SHARES_PACKED = 60/13/13/13 weighted (6000/1300/1300/1300/100 BPS)
Recipients4 winners per draw via VRF, weighted by share class (1st: 60%, 2nd-4th: 13% each, remainder: 1%)
Claim MechanismclaimWinnings() via claimableWinnings[player] mapping
CurrencyETH (with auto-rebuy option)

Formula

// JackpotModule.sol:336-613
// Days 1-4: random BPS in [600, 1400] range (6-14%)
// Day 5: 100% of currentPrizePool
dailyBudget = (currentPrizePool * randomBps) / BPS_DENOMINATOR;

// Share split: JACKPOT_SHARES_PACKED
// 1st place: 6000 BPS (60%)
// 2nd place: 1300 BPS (13%)
// 3rd place: 1300 BPS (13%)
// 4th place: 1300 BPS (13%)
// Remainder: 100 BPS (1%)

// Compressed mode (< threshold): single winner gets 100%
// Turbo mode: instant resolution without VRF wait

Code: JackpotModule.sol:336-613, JackpotModule.sol:2723-2736   Commit: 3fa32f51

Edge Cases

  • Compressed mode: When prize pool is below threshold, counterStep = 2 doubles logical days per physical day. Day 1 is never compressed (preserves early-bird lootbox)
  • Turbo mode: Instant resolution without VRF wait for small pools; counterStep = JACKPOT_LEVEL_CAP processes all 5 days in one physical day
  • Day 5 depletes 100%: dailyBps = 10_000 (100%) of currentPrizePool -- no carry to next jackpot phase
  • randomBps range: [600, 1400] ensures minimum 6% and maximum 14% per day for days 1-4

PAY-16: Ticket Conversion and Futurepool Mechanics

ETH PAY-16
TriggerLevel transition (when level advances)
Source PoolfuturePrizePool transitions to currentPrizePool; ticket value backed by 2x over-collateralization
Calculation_budgetToTicketUnits provides 2x over-collateralization -- each ticket unit backed by 2x its face value in pool ETH. Pool transition: futurePrizePool → nextPrizePool → currentPrizePool
RecipientsTicket holders (their tickets become redeemable against currentPrizePool)
Claim MechanismIndirect -- ticket value realized through jackpot draw participation
CurrencyETH (pool lifecycle management, not direct payout)

Formula

// JackpotModule.sol:1078-1118 -- _budgetToTicketUnits
// 2x over-collateralization: each ticket backed by 2x pool ETH
// ticketUnits = (budget << 2) / priceForLevel(lvl)
// = (budget * 4) / priceForLevel(lvl)
// Over-collateralization absorbs auto-rebuy 130%/145% bonus
// Net coverage: 1.38x-1.54x after bonus absorption

// JackpotModule.sol:886-924 -- Pool transition chain
// futurePrizePool -> nextPrizePool (at level N-1 completion)
// nextPrizePool -> currentPrizePool (at level N start)
// prizePoolFrozen guard blocks claims during days 1-4

Code: JackpotModule.sol:1078-1118, JackpotModule.sol:886-924   Commit: 3fa32f51

Edge Cases

  • Auto-rebuy bonus absorption: 130% (normal) / 145% (enhanced) ticket bonus covered by structural 2x over-collateralization, yielding net 1.38x-1.54x coverage
  • prizePoolFrozen guard: Blocks decimator claims during jackpot phase days 1-4 to prevent pool depletion mid-draw
  • Atomic pool transition: Pool transition is atomic per level -- no partial transitions possible
  • Empty pool at transition: If futurePrizePool is 0 at transition, nextPrizePool receives 0 (no error, just empty jackpot phase)
