## 1. Overview

During the purchase phase of each level, two independent daily distributions fire via `advanceGame()`:

1. **ETH drip** -- 1% of `futurePrizePool` is sliced off daily and distributed as ETH (25%) and lootbox tickets (75%) to trait-matched ticket holders at the current level.
2. **BURNIE jackpot** -- 0.5% of the previous level's prize pool target, denominated in BURNIE, is distributed to near-future (75%) and far-future (25%) ticket holders as coinflip credits.

Both flows are independent and run in separate transactions each day. The ETH drip runs inside `payDailyJackpot(isDaily=false)` and the BURNIE jackpot runs inside `payDailyCoinJackpot()`.

**Pool context:** The `futurePrizePool` that funds the ETH drip is the long-term reserve described in [Pool Architecture](06-pool-architecture.md). During purchase phase, `futurePrizePool` is drawn down by 1% daily, with the drip ETH either paid to winners or recycled into `nextPrizePool` as lootbox backing.

**Orchestration:**

```
advanceGame() [daily during purchase phase]
  -> payDailyJackpot(isDaily=false)        // ETH drip (JackpotModule:630-688)
  -> _payDailyCoinJackpot(lvl, randWord)   // BURNIE jackpot (AdvanceModule:581)
  -> check nextPrizePool >= target          // level advancement trigger
```

## 2. Daily ETH Drip from Future Pool

**Entry point:** `payDailyJackpot(isDaily=false)`
**Source:** `contracts/modules/DegenerusGameJackpotModule.sol` lines 630--688

### 2a. Trigger Conditions

The ETH drip fires only when both conditions hold:

```solidity
// JackpotModule:637-645
uint48 startDayBoundary = uint48((startTime - JACKPOT_RESET_TIME) / 1 days);
uint48 startDay = startDayBoundary - ContractAddresses.DEPLOY_DAY_BOUNDARY + 1;
uint48 daysSince = questDay - startDay;
isEthDay = daysSince > 0 && lvl > 1;
// JACKPOT_RESET_TIME = 82620 (22:57 UTC)       (JackpotModule:105)
```

- `daysSince > 0` -- at least one full day has passed since the level's purchase phase started (no drip on the first day of a level)
- `lvl > 1` -- no drip during level 0 or level 1

**Result:** The earliest an ETH drip can fire is the second day of level 2's purchase phase.

### 2b. Pool Slice Formula

```solidity
// JackpotModule:650-654
uint256 poolBps = 100;  // 1% daily drip from futurePool
ethDaySlice = (_getFuturePrizePool() * poolBps) / 10_000;
_setFuturePrizePool(_getFuturePrizePool() - ethDaySlice);  // upfront deduction
```

The slice is deducted from `futurePrizePool` immediately, before any distribution occurs. If distribution produces no winners, the ETH remains deducted (it is not refunded).

**Daily drip = 1% of futurePrizePool at time of call.**

### 2c. Lootbox Conversion (75%)

Of the ETH drip (`ethDaySlice`), 75% is allocated to lootbox tickets:

```solidity
// JackpotModule:660-664
lootboxBudget = _validateTicketBudget(
    (ethPool * PURCHASE_REWARD_JACKPOT_LOOTBOX_BPS) / 10_000,
    lvl,
    winningTraitsPacked
);
// PURCHASE_REWARD_JACKPOT_LOOTBOX_BPS = 7500 (75%)   (JackpotModule:185)
if (lootboxBudget != 0) ethPool -= lootboxBudget;
```

`_validateTicketBudget` returns 0 (no lootbox) if no trait tickets exist at the current level for the winning traits. Otherwise, it returns the full lootbox budget.

**After lootbox deduction:**
- `ethPool` = 25% of the drip (distributed as ETH to trait-matched winners)
- `lootboxBudget` = 75% of the drip (converted to tickets and pool backing)

### 2d. ETH Distribution (25%)

The remaining 25% is distributed to trait-matched ticket holders at the current level:

```solidity
// JackpotModule:667-674
_executeJackpot(JackpotParams({
    lvl: lvl,
    ethPool: ethPool,                                   // 25% of drip
    entropy: randWord ^ (uint256(lvl) << 192),
    winningTraitsPacked: winningTraitsPacked,
    traitShareBpsPacked: DAILY_JACKPOT_SHARES_PACKED    // 2000/2000/2000/2000
}));
// DAILY_JACKPOT_SHARES_PACKED = 2000 each (20% x 4)   (JackpotModule:124-125)
```

The ETH is split across 4 trait buckets at 20% each, with the entropy-selected solo bucket receiving any remainder. Winners are selected from `traitBurnTicket[lvl]` for each trait bucket.

### 2e. Winning Trait Selection (Purchase Phase)

```solidity
// JackpotModule:631
winningTraitsPacked = _rollWinningTraits(lvl, randWord, false);
// false = non-burn-weighted selection (purchase phase)
```

During purchase phase, `_rollWinningTraits` uses `false` for the burn-weighted parameter, meaning trait selection is NOT weighted by burn amount. This differs from jackpot phase where burn-weighted selection is used.

### 2f. Lootbox Ticket Conversion Flow

The lootbox budget follows a two-step process:

```solidity
// JackpotModule:679-685
_distributeLootboxAndTickets(lvl, winningTraitsPacked, lootboxBudget, randWord, 5_000);
// ticketConversionBps = 5000 (50%)
```

Inside `_distributeLootboxAndTickets` (JackpotModule:1112-1135):

```solidity
// Step 1: Full lootbox budget -> nextPrizePool
_setNextPrizePool(_getNextPrizePool() + lootboxBudget);

// Step 2: Ticket count based on HALF the budget (over-collateralization)
uint256 ticketBasis = (lootboxBudget * ticketConversionBps) / 10_000;
// ticketConversionBps = 5000 -> ticketBasis = lootboxBudget / 2
uint256 ticketUnits = _budgetToTicketUnits(ticketBasis, lvl + 1);
```

Where `_budgetToTicketUnits` (JackpotModule:1096-1103):

```solidity
function _budgetToTicketUnits(uint256 budget, uint24 lvl) private pure returns (uint256) {
    uint256 ticketPrice = PriceLookupLib.priceForLevel(lvl);
    return ticketPrice == 0 ? 0 : (budget << 2) / ticketPrice;
    // << 2 = multiply by 4 (4 tickets per level)
}
```

**Over-collateralization:** The full lootbox budget (75% of drip) goes to `nextPrizePool`, but only 50% of that budget determines the ticket count. This means each ticket issued is backed by 2x its face value in pool ETH.

**Complete ticket count formula:**

```
ticketUnits = ((lootboxBudget * 5000 / 10000) * 4) / priceForLevel(lvl + 1)
            = (lootboxBudget * 2) / priceForLevel(lvl + 1)
```

Note: tickets are priced at `lvl + 1` (the next level), not the current level.

### 2g. Worked Example

Given: `futurePrizePool = 100 ETH`, `lvl = 10`, `priceForLevel(11) = 0.04 ETH`

1. **Drip:** `ethDaySlice = (100 ETH * 100) / 10000 = 1 ETH`
2. **Lootbox budget:** `lootboxBudget = (1 ETH * 7500) / 10000 = 0.75 ETH`
3. **ETH to winners:** `ethPool = 1 ETH - 0.75 ETH = 0.25 ETH`
4. **Pool addition:** `nextPrizePool += 0.75 ETH`
5. **Ticket basis:** `ticketBasis = (0.75 ETH * 5000) / 10000 = 0.375 ETH`
6. **Ticket units:** `(0.375 ETH * 4) / 0.04 ETH = 37 tickets`
7. **Result:** 0.25 ETH distributed as ETH to trait winners, 37 tickets distributed to trait winners, 0.75 ETH added to nextPrizePool

## 3. Daily BURNIE Jackpot

**Entry point:** `payDailyCoinJackpot(lvl, randWord)`
**Source:** `contracts/modules/DegenerusGameJackpotModule.sol` lines 2420--2461
**Wrapper:** `_payDailyCoinJackpot(lvl, randWord)` in AdvanceModule line 581

### 3a. Budget Formula

```solidity
// JackpotModule:2714-2717
function _calcDailyCoinBudget(uint24 lvl) private view returns (uint256) {
    uint256 priceWei = price;
    if (priceWei == 0) return 0;
    return (levelPrizePool[lvl - 1] * PRICE_COIN_UNIT) / (priceWei * 200);
    // PRICE_COIN_UNIT = 1000 ether              (DegenerusGameStorage.sol:125)
}
```

**Interpretation:** 0.5% of the previous level's prize pool target (`levelPrizePool[lvl - 1]`), converted to BURNIE at the current ETH/BURNIE exchange rate.

**Expanded:**

```
coinBudget = levelPrizePool[lvl - 1] * 1000e18 / (priceWei * 200)
           = levelPrizePool[lvl - 1] / (priceWei * 0.2)
           = (levelPrizePool[lvl - 1] / priceWei) * 5
```

This equals 0.5% of `levelPrizePool[lvl-1]` denominated in BURNIE, because:
- `levelPrizePool[lvl-1] / priceWei` = number of "1000 BURNIE units" the pool target is worth
- Multiply by `PRICE_COIN_UNIT / 200` = `1000 ether / 200` = `5 ether` per unit
- So: `(poolTarget * 1000e18) / (price * 200)` = `poolTarget * 5e18 / price` = 0.5% of pool target in BURNIE

### 3b. Budget Split

```solidity
// JackpotModule:2425-2426
uint256 farBudget = (coinBudget * FAR_FUTURE_COIN_BPS) / 10_000;
// FAR_FUTURE_COIN_BPS = 2500 (25%)                    (JackpotModule:212)
uint256 nearBudget = coinBudget - farBudget;
// nearBudget = 75% of coinBudget
```

- **25% far-future** -- ticketQueue-based, levels `[lvl+5, lvl+99]`
- **75% near-future** -- trait-matched, levels `[lvl, lvl+4]`

### 3c. Near-Future Distribution (75%)

**Trait reuse:** Before rolling new winning traits, `payDailyCoinJackpot` attempts to load previously-rolled traits from `lastDailyJackpotWinningTraits` via `_loadDailyWinningTraits(lvl, questDay)` (JackpotModule:2435-2443). If the cached traits match the current level and quest day, they are reused. New traits are only rolled when no valid cache exists, ensuring the ETH drip and BURNIE jackpot use the same winning traits when both fire on the same day.

**Target level selection:** `_selectDailyCoinTargetLevel()` (JackpotModule:2465-2481)
- Selects a random target level in `[lvl, lvl+4]` that has winning-trait tickets
- Falls back to 0 (skip) if none of the five levels has eligible tickets

**Winner selection:** `_awardDailyCoinToTraitWinners()` (JackpotModule:2485)
- Up to `DAILY_COIN_MAX_WINNERS` (50) winners from trait buckets at the target level. However, if `coinBudget < 50`, the cap is reduced to `coinBudget` (JackpotModule:2493: `if (cap > coinBudget) cap = uint16(coinBudget)`), preventing per-winner amounts from rounding to zero
- Winners selected via `_randTraitTicketWithIndices()` from `traitBurnTicket[targetLevel]`
- Budget divided evenly among all trait bucket winners (`baseAmount = coinBudget / cap`)
- Remainder distributed round-robin (`extra = coinBudget % cap`)

**Payment:** `coin.creditFlipBatch(batchPlayers, batchAmounts)` (JackpotModule:2555)
- Awards are coinflip credits, NOT direct BURNIE token transfers
- Recipients can use credits in the BurnieCoinflip game

### 3d. Far-Future Distribution (25%)

**Function:** `_awardFarFutureCoinJackpot()` (JackpotModule:2588)

```solidity
// JackpotModule:2604-2627
for (uint8 s; s < FAR_FUTURE_COIN_SAMPLES; ) {        // FAR_FUTURE_COIN_SAMPLES = 10
    entropy = EntropyLib.entropyStep(entropy ^ uint256(s));
    uint24 candidate = lvl + 5 + uint24(entropy % 95); // random level in [lvl+5, lvl+99]
    address[] storage queue = ticketQueue[_tqWriteKey(candidate)];
    uint256 len = queue.length;
    if (len != 0) {
        uint256 idx = (entropy >> 32) % len;
        address winner = queue[idx];
        if (winner != address(0)) {
            winners[found] = winner;
            unchecked { ++found; }
        }
    }
}
```

- Samples up to `FAR_FUTURE_COIN_SAMPLES` (10) random levels from `[lvl+5, lvl+99]`
- 1 winner per level from `ticketQueue` (pre-trait assignment queue)
- Even split among found winners: `perWinner = farBudget / found` (JackpotModule:2632)
- Payment via `coin.creditFlipBatch()` (same as near-future)
- Levels with no ticketQueue entries are skipped (no re-roll)

**Note:** `ticketQueue` holds addresses of players who have purchased tickets for future levels but whose traits have not yet been assigned. This is a separate data structure from `traitBurnTicket` (which holds trait-assigned tickets).

### 3e. Worked Example

Given: `levelPrizePool[9] = 80 ETH` (level 9 target), `price = 0.04 ETH`, `lvl = 10`

1. **Budget:** `coinBudget = (80e18 * 1000e18) / (0.04e18 * 200) = 10,000,000 BURNIE`
2. **Far-future (25%):** `farBudget = (10,000,000 * 2500) / 10000 = 2,500,000 BURNIE`
3. **Near-future (75%):** `nearBudget = 10,000,000 - 2,500,000 = 7,500,000 BURNIE`
4. **If 5 far-future winners found:** `perWinner = 2,500,000 / 5 = 500,000 BURNIE each`
5. **If 30 near-future winners across trait buckets:** `baseAmount = 7,500,000 / 30 = 250,000 BURNIE each`

## 4. Lootbox Conversion Reference

This section consolidates the purchase-phase-specific lootbox conversion mechanics. Jackpot-phase ratios differ and will be detailed in the jackpot-phase distribution document.

### 4a. Purchase-Phase ETH Drip Lootbox Conversion

| Parameter | Value | Source |
|-----------|-------|--------|
| Lootbox share of drip | 75% | `PURCHASE_REWARD_JACKPOT_LOOTBOX_BPS = 7500` (JackpotModule:185) |
| Ticket conversion basis | 50% of lootbox budget | `ticketConversionBps = 5000` passed to `_distributeLootboxAndTickets` (JackpotModule:684) |
| Pool destination | 100% of lootbox budget to nextPrizePool | `_setNextPrizePool(_getNextPrizePool() + lootboxBudget)` (JackpotModule:1120) |
| Ticket price level | `lvl + 1` (next level) | `_budgetToTicketUnits(ticketBasis, lvl + 1)` (JackpotModule:1124) |

### 4b. Flow Diagram

```
ethDaySlice (1% of futurePrizePool)
  |
  +-- 75% -> lootboxBudget
  |           |
  |           +-- 100% ETH -> nextPrizePool (pool backing)
  |           |
  |           +-- 50% of budget -> ticket count calculation
  |                |
  |                +-- tickets distributed to trait winners at current level
  |
  +-- 25% -> ethPool
              |
              +-- distributed as ETH to trait-matched winners
                  (4 buckets @ 20% each, solo bucket gets remainder)
```

### 4c. Over-Collateralization Explained

The full lootbox budget (75% of drip) is added to `nextPrizePool`, but tickets are only minted based on 50% of that budget. This means:

- **Pool ETH per ticket** = `lootboxBudget / ticketUnits`
- **Face value per ticket** = `priceForLevel(lvl + 1) / 4`
- **Backing ratio** = `lootboxBudget / (ticketUnits * ticketPrice / 4)` = approximately 2x

This over-collateralization ensures the pool grows faster than the ticket obligations it creates, maintaining long-term solvency.

### 4d. Comparison with Jackpot-Phase Ratios

| Context | Lootbox BPS | Ticket Conversion BPS | Effect |
|---------|-------------|----------------------|--------|
| Purchase-phase daily drip | 7500 (75%) | 5000 (50%) | Heavy lootbox conversion, 2x backing |
| Jackpot-phase daily carryover | 5000 (50%) | 5000 (50%) | Equal ETH/lootbox split, 2x backing |
| Jackpot-phase daily budget | `budget / 5` (20%) | varies | 80% ETH, 20% lootbox |

Purchase-phase drip converts more aggressively to lootbox (75% vs 50% for carryover), seeding the next level's pool with ETH while distributing tickets to incentivize continued participation.

## Common Pitfalls

> **Pitfall 1: ETH Drip Is Not Refunded If No Winners Found.**
> The 1% slice is deducted from `futurePrizePool` upfront (JackpotModule:654). If `_executeJackpot` finds no trait-matched tickets, the 25% ETH portion remains in the contract but is no longer tracked by any pool variable. It is not "lost" in the sense of leaving the contract -- it stays on-chain and contributes to the yield surplus (the difference between contract balance and total tracked obligations), which is periodically redistributed via `_distributeYieldSurplus`. The 75% lootbox portion still goes to `nextPrizePool` regardless.

> **Pitfall 2: Lootbox Budget vs Ticket Count Are Independent.**
> The full lootbox budget (75% of drip) goes to `nextPrizePool`, but only 50% of that budget determines ticket count. Do NOT assume `ticketUnits * ticketPrice / 4 = lootboxBudget`. The pool receives 2x the ETH that the tickets represent. See Section 4c.

> **Pitfall 3: BURNIE Jackpot Budget Uses Previous Level's Target.**
> `_calcDailyCoinBudget(lvl)` reads `levelPrizePool[lvl - 1]`, not the current level's target. This is the snapshot of `nextPrizePool` at the previous level transition. If the current `nextPrizePool` has grown significantly, the BURNIE budget does NOT increase until the next level starts.

> **Pitfall 4: BURNIE Awards Are Coinflip Credits, Not Token Transfers.**
> `coin.creditFlipBatch()` adds to the player's coinflip credit balance, usable in BurnieCoinflip games. It does NOT transfer BURNIE tokens to the player's wallet. Winners cannot directly sell or transfer these credits.

> **Pitfall 5: No ETH Drip on First Day or Levels 0-1.**
> The `isEthDay` check (JackpotModule:645) requires `daysSince > 0 && lvl > 1`. The first day of any level and all of levels 0 and 1 receive BURNIE jackpots only, no ETH drip.
