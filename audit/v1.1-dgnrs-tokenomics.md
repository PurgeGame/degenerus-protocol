## 1. Overview

DGNRS ("Degenerus Stonk") is a dual-contract token system with a fixed initial supply of **1 trillion tokens** (1,000,000,000,000 * 10^18). The entire supply is minted at deployment across two contracts with distinct transfer properties.

### Dual-Contract Architecture

**StakedDegenerusStonk (sDGNRS) -- soulbound backing token:**
- Holds all ETH, stETH, and BURNIE reserves
- Manages the five reward pools (Whale, Affiliate, Lootbox, Reward, Earlybird)
- Distributes sDGNRS to game players by drawing down pool balances
- `burn(amount)` redeems sDGNRS for proportional backing assets
- `burnRemainingPools()` (game-only) burns all undistributed pool tokens at game over
- **No transfer function.** sDGNRS balances held by players cannot be moved to any other address. The only movement mechanism is `wrapperTransferTo`, callable exclusively by the DGNRS wrapper contract.

**DegenerusStonk (DGNRS) -- transferable ERC20 wrapper:**
- Receives the creator's 20% allocation as its initial supply (minted by sDGNRS to the DGNRS contract address, then distributed to `CREATOR` by the DGNRS constructor)
- Full ERC20: `transfer`, `transferFrom`, `approve`, `allowance` -- freely tradeable
- `burn(amount)` delegates to `sDGNRS.burn(amount)` and forwards ETH/stETH/BURNIE to the caller
- `unwrapTo(recipient, amount)` (creator-only) burns DGNRS and sends the underlying sDGNRS to a specific recipient as soulbound

**Primary sources:** `contracts/StakedDegenerusStonk.sol`, `contracts/DegenerusStonk.sol`

## 2. Initial Supply Distribution (DGNR-01)

### 2a. Pool Allocations

All allocations are computed as `(INITIAL_SUPPLY * BPS) / 10_000` in the constructor:

| Pool | BPS | Percentage | Token Amount | Enum Index |
|------|-----|-----------|-------------|------------|
| Creator | 2000 | 20.00% | 200,000,000,000 * 10^18 | N/A (minted to DGNRS wrapper) |
| Whale | 1000 | 10.00% | 100,000,000,000 * 10^18 | 0 |
| Affiliate | 3500 | 35.00% | 350,000,000,000 * 10^18 | 1 |
| Lootbox | 2000 + dust | 20.00%+ | 200,000,000,000 * 10^18 + dust | 2 |
| Reward | 500 | 5.00% | 50,000,000,000 * 10^18 | 3 |
| Earlybird | 1000 | 10.00% | 100,000,000,000 * 10^18 | 4 |
| **Total** | **10000** | **100%** | **1,000,000,000,000 * 10^18** | |

**BPS sum verification:** 2000 + 1000 + 3500 + 2000 + 500 + 1000 = 10000 (100%)

### 2b. Dust Handling

Integer division of `(INITIAL_SUPPLY * BPS) / 10_000` can produce rounding shortfall. The constructor computes the total of all six allocations and adds any dust (shortfall) to the Lootbox pool:

```solidity
// StakedDegenerusStonk.sol:201-208
uint256 totalAllocated = creatorAmount + whaleAmount + earlybirdAmount
    + affiliateAmount + lootboxAmount + rewardAmount;
if (totalAllocated < INITIAL_SUPPLY) {
    uint256 dust;
    unchecked {
        dust = INITIAL_SUPPLY - totalAllocated;
    }
    lootboxAmount += dust;
}
```

### 2c. Minting Structure

The creator allocation is minted by sDGNRS directly to the DGNRS wrapper contract address (`ContractAddresses.DGNRS`). The DGNRS constructor then reads that balance and assigns it entirely to `ContractAddresses.CREATOR`. All five pool allocations are minted by sDGNRS to `address(this)` (the sDGNRS contract itself), which holds the tokens and tracks per-pool balances in a `uint256[5] private poolBalances` array:

```solidity
// StakedDegenerusStonk.sol:212-219
_mint(ContractAddresses.DGNRS, creatorAmount);
_mint(address(this), poolTotal);

poolBalances[uint8(Pool.Whale)] = whaleAmount;
poolBalances[uint8(Pool.Affiliate)] = affiliateAmount;
poolBalances[uint8(Pool.Lootbox)] = lootboxAmount;
poolBalances[uint8(Pool.Reward)] = rewardAmount;
poolBalances[uint8(Pool.Earlybird)] = earlybirdAmount;
```

Pool balances live in the sDGNRS contract. The DGNRS wrapper holds no pool state.

### 2d. Pool Enum Order

```solidity
// StakedDegenerusStonk.sol:133-139
enum Pool {
    Whale,      // 0
    Affiliate,  // 1
    Lootbox,    // 2
    Reward,     // 3
    Earlybird   // 4
}
```

## 3. Earlybird Reward Curve (DGNR-02)

### 3a. Activation Conditions

Earlybird DGNRS rewards are distributed to any buyer making an ETH purchase during levels < `EARLYBIRD_END_LEVEL` (3). Eligible purchase types: tickets, lootbox, whale bundle, lazy pass, deity pass -- any call that invokes `_awardEarlybirdDgnrs`.

**Source:** `DegenerusGameStorage.sol:1062-1138`

### 3b. Quadratic Decreasing Emission Formula

The earlybird curve awards fewer DGNRS per ETH as cumulative ETH grows, using a quadratic function:

```solidity
// DegenerusGameStorage.sol:1115-1128
uint256 totalEth = EARLYBIRD_TARGET_ETH;          // 1,000 ETH
uint256 ethIn = earlybirdEthIn;                    // cumulative ETH so far
if (ethIn >= totalEth) return;

uint256 remaining = totalEth - ethIn;
uint256 delta = purchaseWei > remaining ? remaining : purchaseWei;
if (delta == 0) return;

uint256 nextEthIn = ethIn + delta;
uint256 denom = totalEth * totalEth;               // 1000^2 = 1,000,000 ETH^2
uint256 totalEth2 = totalEth * 2;                  // 2000 ETH
uint256 d1 = (ethIn * totalEth2) - (ethIn * ethIn);
uint256 d2 = (nextEthIn * totalEth2) - (nextEthIn * nextEthIn);
uint256 payout = (poolStart * (d2 - d1)) / denom;
```

**Variables:**
- `poolStart` = initial earlybird pool balance, snapshot on first payout (`earlybirdDgnrsPoolStart`)
- `ethIn` = cumulative ETH deposited toward earlybird so far (`earlybirdEthIn`)
- `totalEth` = `EARLYBIRD_TARGET_ETH` = 1,000 ETH
- `purchaseWei` = ETH spent on this purchase
- `delta` = min(purchaseWei, remaining capacity)
- `d1`, `d2` = quadratic accumulators at old and new cumulative ETH
- `payout` = DGNRS tokens awarded to buyer

### 3c. Pool Start Snapshot

On the first earlybird payout, the contract snapshots the current earlybird pool balance:

```solidity
// DegenerusGameStorage.sol:1106-1113
uint256 poolStart = earlybirdDgnrsPoolStart;
if (poolStart == 0) {
    uint256 poolBalance = IStakedDegenerusStonk(ContractAddresses.SDGNRS)
        .poolBalance(IStakedDegenerusStonk.Pool.Earlybird);
    if (poolBalance == 0) return;
    poolStart = poolBalance;
    earlybirdDgnrsPoolStart = poolBalance;
}
```

This means the formula uses the full initial earlybird allocation as `poolStart` -- approximately 100B DGNRS.

### 3d. Level 3+ Cutoff: Remainder Transfer to Lootbox Pool

When `currentLevel >= EARLYBIRD_END_LEVEL` (3), a one-shot transfer moves all remaining earlybird pool tokens to the Lootbox pool. This is guarded by setting `earlybirdDgnrsPoolStart = type(uint256).max`:

```solidity
// DegenerusGameStorage.sol:1085-1103
if (currentLevel >= EARLYBIRD_END_LEVEL) {
    if (earlybirdDgnrsPoolStart != type(uint256).max) {
        earlybirdDgnrsPoolStart = type(uint256).max;
        IStakedDegenerusStonk dgnrsContract = IStakedDegenerusStonk(ContractAddresses.SDGNRS);
        uint256 earlybirdRemaining = dgnrsContract.poolBalance(
            IStakedDegenerusStonk.Pool.Earlybird
        );
        if (earlybirdRemaining != 0) {
            dgnrsContract.transferBetweenPools(
                IStakedDegenerusStonk.Pool.Earlybird,
                IStakedDegenerusStonk.Pool.Lootbox,
                earlybirdRemaining
            );
        }
    }
    return;
}
```

Any earlybird DGNRS not distributed during levels 0-2 rolls into the Lootbox pool, increasing the Lootbox pool beyond its initial 200B+ allocation.

### 3e. Worked Example

**Scenario:** Player deposits 10 ETH when cumulative `earlybirdEthIn` = 500 ETH.

Given:
- `poolStart` = 100,000,000,000 * 10^18 (initial earlybird pool, ~100B DGNRS)
- `totalEth` = 1,000 ETH = 1000 * 10^18
- `ethIn` = 500 * 10^18
- `purchaseWei` = 10 * 10^18
- `delta` = 10 * 10^18 (within remaining capacity of 500 ETH)
- `nextEthIn` = 510 * 10^18

Computation (in ETH units for clarity, scaling cancels):
- `denom` = 1000^2 = 1,000,000
- `totalEth2` = 2000
- `d1` = (500 * 2000) - (500 * 500) = 1,000,000 - 250,000 = 750,000
- `d2` = (510 * 2000) - (510 * 510) = 1,020,000 - 260,100 = 759,900
- `d2 - d1` = 9,900
- `payout` = 100,000,000,000 * 9,900 / 1,000,000 = **990,000,000,000 DGNRS** (approximately 990B raw units, i.e., ~990 DGNRS)

For comparison, the first 10 ETH (ethIn=0):
- `d1` = 0
- `d2` = (10 * 2000) - (10 * 10) = 20,000 - 100 = 19,900
- `payout` = 100,000,000,000 * 19,900 / 1,000,000 = **1,990,000,000,000 DGNRS** (~1,990 DGNRS)

The first depositor gets roughly **2x** the DGNRS per ETH compared to depositing at the halfway point.

## 4. Per-Purchase DGNRS Rewards (DGNR-03)

### 4a. Whale Bundle DGNRS Rewards

Each whale bundle purchase triggers `_rewardWhaleBundleDgnrs`, which distributes from both the Whale and Affiliate pools. Called once per unit quantity (so buying 5 bundles calls this 5 times):

**Source:** `DegenerusGameWhaleModule.sol:628-680`

| Recipient | Pool | Rate | Unit System | Computation |
|-----------|------|------|-------------|-------------|
| Buyer (minter) | Whale | 1% | 10,000 PPM | `(whaleReserve * 10_000) / 1_000_000` |
| Affiliate (direct) | Affiliate | 0.1% | 1,000 PPM | `(affiliateReserve * 1_000) / 1_000_000` |
| Affiliate (upline) | Affiliate | 0.02% | 200 PPM | `(affiliateReserve * 200) / 1_000_000` |
| Affiliate (upline2) | Affiliate | 0.01% | 100 PPM | `uplineShare / 2` (computed, not a named constant) |

> **CRITICAL PPM vs BPS PITFALL:** Whale bundle rewards use **parts-per-million** (1,000,000 scale = `DGNRS_WHALE_REWARD_PPM_SCALE`). The PPM constants divide by 1,000,000, NOT 10,000. Do not confuse with BPS constants used elsewhere.

> **Note on upline2:** The "100 PPM" for upline2 is not a named constant in the contract. It is computed as `uplineShare / 2` (where `uplineShare` uses `DGNRS_AFFILIATE_UPLINE_WHALE_PPM = 200`), yielding an effective 100 PPM. See WhaleModule.sol:672.

```solidity
// DegenerusGameWhaleModule.sol:88-97
uint32 private constant DGNRS_WHALE_REWARD_PPM_SCALE = 1_000_000;
uint32 private constant DGNRS_WHALE_MINTER_PPM = 10_000;         // 1%
uint32 private constant DGNRS_AFFILIATE_DIRECT_WHALE_PPM = 1_000; // 0.1%
uint32 private constant DGNRS_AFFILIATE_UPLINE_WHALE_PPM = 200;   // 0.02%
```

```solidity
// DegenerusGameWhaleModule.sol:634-637
uint256 whaleReserve = dgnrs.poolBalance(IStakedDegenerusStonk.Pool.Whale);
if (whaleReserve != 0) {
    uint256 minterShare = (whaleReserve * DGNRS_WHALE_MINTER_PPM) /
        DGNRS_WHALE_REWARD_PPM_SCALE;
```

### 4b. Deity Pass DGNRS Rewards

Each deity pass purchase triggers `_rewardDeityPassDgnrs` (called once per pass):

**Source:** `DegenerusGameWhaleModule.sol:688-743`

| Recipient | Pool | Rate | Unit System | Computation |
|-----------|------|------|-------------|-------------|
| Buyer | Whale | 5% | 500 BPS | `(whaleReserve * 500) / 10_000` |
| Affiliate (direct) | Affiliate | 0.5% | 5,000 PPM | `(affiliateReserve * 5_000) / 1_000_000` |
| Affiliate (upline) | Affiliate | 0.1% | 1,000 PPM | `(affiliateReserve * 1_000) / 1_000_000` |
| Affiliate (upline2) | Affiliate | 0.05% | 500 PPM | `uplineShare / 2` (computed, not a named constant) |

> **CRITICAL PPM vs BPS PITFALL:** Deity pass buyer reward uses **BPS** (10,000 scale) for the whale pool, but **PPM** (1,000,000 scale) for affiliate pool rewards. Both unit systems coexist in the same function.

> **Note on upline2:** The "500 PPM" for upline2 is not a named constant in the contract. It is computed as `uplineShare / 2` (where `uplineShare` uses `DGNRS_AFFILIATE_UPLINE_DEITY_PPM = 1_000`), yielding an effective 500 PPM. See WhaleModule.sol:734.


```solidity
// DegenerusGameWhaleModule.sol:694-696
uint256 whaleReserve = dgnrs.poolBalance(IStakedDegenerusStonk.Pool.Whale);
if (whaleReserve != 0) {
    uint256 totalReward = (whaleReserve * DEITY_WHALE_POOL_BPS) / 10_000;
```

### 4c. Pool Balance Decay -- Worked Example

> **Pool Balance Decay Pitfall:** All percentages are of **remaining** pool balance at time of claim. Sequential claims compound the depletion. The 100th whale bundle buyer gets 1% of a much smaller pool than the 1st buyer.

**Whale Pool depletion over 10 whale bundle purchases (buyer reward only):**

Starting whale pool: 100,000,000,000 DGNRS (100B)

| Purchase # | Pool Before (B DGNRS) | 1% Payout (B DGNRS) | Pool After (B DGNRS) |
|------------|----------------------|---------------------|---------------------|
| 1 | 100.000 | 1.000 | 99.000 |
| 2 | 99.000 | 0.990 | 98.010 |
| 3 | 98.010 | 0.980 | 97.030 |
| 5 | 96.060 | 0.961 | 95.099 |
| 10 | 91.352 | 0.914 | 90.438 |
| 50 | 60.501 | 0.605 | 59.896 |
| 100 | 36.604 | 0.366 | 36.238 |

Formula: `pool_after_n = pool_start * (0.99)^n`

After 100 whale bundles, the whale pool retains ~36.6% of its original balance. The 100th buyer receives only 37% as much DGNRS as the 1st buyer.

**Deity pass impact:** A single deity pass takes 5% of the whale pool. After one deity pass, the pool drops to 95% -- equivalent to approximately 5 whale bundle purchases.

## 5. Affiliate DGNRS Claim Per Level (DGNR-03 continued)

### 5a. claimAffiliateDgnrs Mechanics

Affiliates can claim DGNRS from the Affiliate pool once per level, based on their referral volume.

**Source:** `DegenerusGame.sol:1410-1461`

**Eligibility requirements:**
1. `level > 1` (claims are for the previous level: `prevLevel = level - 1`)
2. Player has not already claimed for `prevLevel` (`affiliateDgnrsClaimedBy[prevLevel][player]` is false)
3. Either:
   - `affiliateScore(prevLevel, player) >= AFFILIATE_DGNRS_MIN_SCORE` (10 ETH), OR
   - Player holds a deity pass (`deityPassCount[player] != 0`)

### 5b. Reward Formula

```solidity
// DegenerusGame.sol:1429-1438
uint256 denominator = levelPrizePool[prevLevel];
if (denominator == 0) {
    denominator = BOOTSTRAP_PRIZE_POOL;     // 50 ETH fallback
}

uint256 poolBalance = dgnrs.poolBalance(IStakedDegenerusStonk.Pool.Affiliate);
uint256 levelShare = (poolBalance * AFFILIATE_DGNRS_LEVEL_BPS) / 10_000;  // 5% of pool
if (levelShare == 0) revert E();
uint256 reward = (levelShare * score) / denominator;
if (reward == 0) revert E();
```

**Formula:** `reward = (affiliatePoolBalance * 500 / 10_000) * score / levelPrizePool[prevLevel]`

Where:
- `affiliatePoolBalance` = remaining Affiliate pool DGNRS at time of claim
- `AFFILIATE_DGNRS_LEVEL_BPS` = 500 (5% of pool available per level)
- `score` = player's affiliate score for prevLevel (ETH volume referred)
- `denominator` = `levelPrizePool[prevLevel]` or `BOOTSTRAP_PRIZE_POOL` (50 ETH) if zero

### 5c. Deity Pass Bonus

Deity pass holders receive an additional BURNIE FLIP credit bonus on top of their DGNRS claim:

```solidity
// DegenerusGame.sol:1447-1456
if (hasDeityPass && score != 0) {
    uint256 bonus = (score * AFFILIATE_DGNRS_DEITY_BONUS_BPS) / 10_000;  // 20% of score
    uint256 cap = (AFFILIATE_DGNRS_DEITY_BONUS_CAP_ETH * PRICE_COIN_UNIT) / price;
    if (bonus > cap) {
        bonus = cap;
    }
    if (bonus != 0) {
        coin.creditFlip(player, bonus);
    }
}
```

- Bonus = min(score * 20%, 5 ETH worth of BURNIE at current price)
- Paid as FLIP credit (not direct BURNIE mint) via `coin.creditFlip`
- Only applies when the player has a deity pass AND score > 0

### 5d. Worked Example

**Scenario:** Level is 5, claiming for prevLevel 4. Affiliate pool has 300B DGNRS remaining. Player's affiliate score for level 4 is 25 ETH. `levelPrizePool[4]` = 200 ETH.

- `levelShare` = (300B * 500) / 10,000 = 15B DGNRS (5% of pool)
- `reward` = (15B * 25 ETH) / 200 ETH = **1.875B DGNRS**

If a second affiliate claims at the same level with 50 ETH score:
- Pool is now 300B - 1.875B = 298.125B (pool depleted by first claim)
- `levelShare` = (298.125B * 500) / 10,000 = 14.906B
- `reward` = (14.906B * 50) / 200 = **3.727B DGNRS**

> **Pitfall:** The 5% "per level" share is not reserved. Multiple claimants at the same level each take 5% of the **current** pool balance, sequentially depleting it.

## 6. Soulbound Mechanics and Transfer Restrictions (DGNR-04)

### 6a. sDGNRS is Soulbound -- No Transfer Function

sDGNRS (`StakedDegenerusStonk`) has no `transfer` function at all. This is a stricter constraint than a transfer-restricted token -- there is simply no public or external transfer entry point for token holders. The contract's `_transfer`-equivalent logic is entirely internal and is only exercised by:

1. `_mint` -- mints tokens into a balance during construction
2. `transferFromPool` -- game-only, moves tokens from pool balance to a recipient
3. `wrapperTransferTo` -- DGNRS-wrapper-only, moves tokens from the DGNRS address to a recipient when the creator calls `unwrapTo`

Any sDGNRS held in a player's balance is permanently locked there. There is no mechanism to send sDGNRS to another address once it has been distributed from a pool.

### 6b. DGNRS is a Full ERC20

The DGNRS wrapper (`DegenerusStonk`) is an unrestricted ERC20. It implements:
- `transfer(address to, uint256 amount)` -- callable by any holder
- `transferFrom(address from, address to, uint256 amount)` -- respects `allowance` mapping
- `approve(address spender, uint256 amount)` -- sets allowance
- `allowance(address owner, address spender)` -- public mapping

DGNRS can be listed on DEXes, transferred peer-to-peer, held by contracts, and traded freely. There are no transfer restrictions.

### 6c. Who Holds What

| Holder | Token | Transferable | Burn Path |
|--------|-------|-------------|-----------|
| Creator (`ContractAddresses.CREATOR`) | DGNRS | Yes -- full ERC20 | `DGNRS.burn()` forwards to sDGNRS, pays backing to caller |
| Game players (pool distributions) | sDGNRS | No -- permanently locked | `sDGNRS.burn()` directly, pays backing to msg.sender |
| Any DGNRS holder | DGNRS | Yes | `DGNRS.burn()` forwards to sDGNRS |

The creator receives DGNRS (transferable) and can distribute it to team members, sell it, or lock it in a vesting contract. Game players receive sDGNRS (soulbound) and can only extract value by burning for backing.

### 6d. unwrapTo -- Creator Converts DGNRS to Soulbound sDGNRS

The creator can call `DGNRS.unwrapTo(recipient, amount)` to burn DGNRS and send the underlying sDGNRS to a specific recipient as soulbound. This is a one-way conversion: the DGNRS supply decreases and the recipient receives locked sDGNRS that cannot be transferred further.

```solidity
// DegenerusStonk.sol:111-117
function unwrapTo(address recipient, uint256 amount) external {
    if (msg.sender != ContractAddresses.CREATOR) revert Unauthorized();
    if (recipient == address(0)) revert ZeroAddress();
    _burn(msg.sender, amount);
    stonk.wrapperTransferTo(recipient, amount);
    emit UnwrapTo(recipient, amount);
}
```

## 7. Burn-for-Backing (DGNR-04 continued)

### 7a. Two Burn Paths

There are two entry points to burn tokens and claim backing:

**Path 1 -- Direct sDGNRS burn:**
`sDGNRS.burn(amount)` called by a player holding sDGNRS. Burns from `msg.sender`'s balance and pays ETH/stETH/BURNIE directly to `msg.sender`.

**Path 2 -- DGNRS wrapper burn (delegates to sDGNRS):**
`DGNRS.burn(amount)` burns DGNRS from `msg.sender`, then calls `sDGNRS.burn(amount)` to liquidate the underlying backing. The sDGNRS contract pays ETH/stETH/BURNIE to the DGNRS contract address, which the DGNRS contract forwards to `msg.sender`.

Both paths result in the same proportional payout; the difference is which token is burned. A holder of DGNRS must use Path 2. A holder of sDGNRS must use Path 1.

**Entry points:**
- `sDGNRS.burn(uint256 amount)` -- `StakedDegenerusStonk.sol:379-441`
- `DGNRS.burn(uint256 amount)` -- `DegenerusStonk.sol:125-142` (delegates to sDGNRS)

### 7b. Proportional Payout Formula

The payout math lives entirely in `StakedDegenerusStonk.sol`:

```solidity
// StakedDegenerusStonk.sol:385-396
uint256 supplyBefore = totalSupply;

uint256 ethBal = address(this).balance;
uint256 stethBal = steth.balanceOf(address(this));
uint256 claimableEth = _claimableWinnings();
uint256 totalMoney = ethBal + stethBal + claimableEth;
uint256 totalValueOwed = (totalMoney * amount) / supplyBefore;

uint256 burnieBal = coin.balanceOf(address(this));
uint256 claimableBurnie = coinflip.previewClaimCoinflips(address(this));
uint256 totalBurnie = burnieBal + claimableBurnie;
burnieOut = (totalBurnie * amount) / supplyBefore;
```

**Formula:**
- ETH+stETH value: `(totalMoney * burnAmount) / totalSupply`
- BURNIE value: `(totalBurnie * burnAmount) / totalSupply`

Where `totalMoney` includes:
1. ETH balance of sDGNRS contract
2. stETH balance of sDGNRS contract
3. Claimable game winnings (minus 1 wei dust)

And `totalBurnie` includes:
1. BURNIE balance of sDGNRS contract
2. Claimable coinflip withdrawals

### 7c. ETH-Preferential Payout Ordering

The contract pays ETH first, falling back to stETH only when ETH is insufficient:

```solidity
// StakedDegenerusStonk.sol:410-416
if (totalValueOwed <= ethBal) {
    ethOut = totalValueOwed;
} else {
    ethOut = ethBal;
    stethOut = totalValueOwed - ethOut;
    if (stethOut > stethBal) revert Insufficient();
}
```

If the full value owed exceeds on-hand ETH, claimable game winnings are claimed first (`game.claimWinnings`), then ETH is re-checked. Only the residual shortfall is paid as stETH.

### 7d. BURNIE Payout with Coinflip Claim

BURNIE is paid from on-hand balance first, with any shortfall withdrawn from coinflip claimables on demand:

```solidity
// StakedDegenerusStonk.sol:418-429
uint256 remainingBurnie = burnieOut;
if (remainingBurnie != 0) {
    uint256 payBal = remainingBurnie <= burnieBal ? remainingBurnie : burnieBal;
    if (payBal != 0) {
        remainingBurnie -= payBal;
        if (!coin.transfer(player, payBal)) revert TransferFailed();
    }
    if (remainingBurnie != 0) {
        coinflip.claimCoinflips(address(0), remainingBurnie);
        if (!coin.transfer(player, remainingBurnie)) revert TransferFailed();
    }
}
```

### 7e. burnRemainingPools -- Game-Over Pool Burn

`burnRemainingPools` is game-only and burns all undistributed pool tokens at game over. This burns the entire `address(this)` balance of sDGNRS (all tokens still sitting in pools that were never distributed to players), reducing `totalSupply` without paying any backing. The effect concentrates backing per remaining token for all existing sDGNRS and DGNRS holders.


## 9. Agent Pitfalls

> **Pitfall 1 -- PPM vs BPS unit confusion:** Whale bundle rewards use PPM (1,000,000 scale). Deity pass buyer reward uses BPS (10,000 scale). Deity pass affiliate rewards use PPM. Both unit systems coexist in `_rewardDeityPassDgnrs`. Always check whether the denominator is 10,000 or 1,000,000 before computing percentages.

> **Pitfall 2 -- Pool balance decay:** All pool reward percentages are of the **remaining** pool balance at the instant of the claim. The pool is not snapshotted per level or per day. A model that assumes fixed pool balance will overestimate rewards for later claimants. Use geometric decay: `pool_after_n_claims = pool_start * ((scale - rate) / scale)^n`.

> **Pitfall 3 -- Earlybird level 3 cutoff:** Earlybird rewards end abruptly at level 3 (not gradually). Any remaining earlybird pool tokens are dumped into the Lootbox pool via a one-shot `transferBetweenPools`. After level 3, the Earlybird pool is permanently empty and the Lootbox pool is larger.

> **Pitfall 4 -- Earlybird ETH cap:** Even within levels 0-2, earlybird rewards stop once cumulative ETH reaches `EARLYBIRD_TARGET_ETH` (1,000 ETH). The level and ETH cap are independent cutoffs -- whichever is hit first terminates earlybird rewards.

> **Pitfall 5 -- burnRemainingPools concentrates backing:** At game over, `burnRemainingPools` burns all undistributed pool tokens (the entire `address(this)` balance in sDGNRS) without paying any reserves. This reduces `totalSupply` significantly, increasing the backing per remaining token for all existing holders. Models must account for this terminal supply shock.

> **Pitfall 6 -- claimAffiliateDgnrs "5% per level" is not reserved:** The 5% is computed from current pool balance at claim time. If 10 affiliates each claim at the same level, the first gets 5% of the full pool, the second gets 5% of the depleted pool, etc. Total drain per level depends on claim order and count.

> **Pitfall 7 -- Whale bundle quantity multiplier:** `_rewardWhaleBundleDgnrs` is called once per unit in the quantity loop (`for i in 0..quantity`). Each call reads the current pool balance. Buying 5 bundles in one transaction does NOT give 5x the first payout -- it gives a geometrically decreasing series of 1% claims.

> **Pitfall 8 -- DGNRS vs sDGNRS confusion:** These are two separate contracts with fundamentally different properties. DGNRS (`DegenerusStonk`) is the liquid ERC20 wrapper -- transferable, tradeable, approvable. sDGNRS (`StakedDegenerusStonk`) is the soulbound backing token -- no transfer function, permanently locked once distributed. The game distributes sDGNRS to players. The creator holds DGNRS. All reserve assets (ETH, stETH, BURNIE) and pool balances live in the sDGNRS contract. Burn mechanics (proportional payout formula, ETH-preferential ordering, BURNIE coinflip claims) all execute in sDGNRS. The DGNRS `burn()` function is a thin delegation layer that forwards to sDGNRS. Never assume a player's DGNRS balance -- players receive sDGNRS from pools, not DGNRS.
