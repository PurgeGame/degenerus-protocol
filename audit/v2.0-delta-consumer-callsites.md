## Executive Summary

This report covers the consumer side of the sDGNRS/DGNRS split -- every game module function that calls into StakedDegenerusStonk for pool transfers, deposits, or terminal burns. The scope includes 30 callsites across 8 consumer contracts: DegenerusGame (4 callsites), DegenerusGameStorage (3), DegenerusGameWhaleModule (10), DegenerusGameLootboxModule (2), DegenerusGameEndgameModule (3), DegenerusGameJackpotModule (2), DegenerusGameDegeneretteModule (2), and DegenerusGameGameOverModule (4).

**Overall Assessment: SOUND.** All 30 callsites use correct Pool enum values, all `dgnrs`/`sdgnrs` variable references resolve to `ContractAddresses.SDGNRS`, return values from `transferFromPool` are handled safely (either used for accounting or safely ignored where pool depletion is gracefully handled), and all callers execute via delegatecall from the game contract, satisfying the `onlyGame` modifier on sDGNRS. One Informational finding was identified: a stale comment in DegenerusGameStorage.sol that says "reward pool" but the code correctly dumps Earlybird into Lootbox. All BPS/PPM constants verified against source declarations with correct denominators.

### Return Value Handling Analysis

The `transferFromPool` function returns `uint256 transferred` -- the actual amount transferred, which may be less than requested if the pool is depleted. The following return value handling patterns were observed:

**Pattern A: Return value used for accounting (5 callsites)**
- Callsite 3 (DegenerusGame.sol:1460): Stores `paid`, checks `if (paid == 0) revert E()`, uses `paid` for `levelDgnrsClaimed` accounting. **Correct.**
- Callsite 16 (WhaleModule.sol:698): Stores `paid`, uses for `buyerDgnrs` return value (capped to uint96). **Correct.**
- Callsite 22 (LootboxModule.sol:1710): Returns `paid` to caller for lootbox accounting. **Correct.**
- Callsite 24 (EndgameModule.sol:125): Stores `paid`, emits in `AffiliateDgnrsReward` event. **Correct.**
- Callsite 30d (GameOverModule.sol:227): Checks `(bool ok, )` on ETH send, reverts on failure. **Correct.**

**Pattern B: Return value ignored (remaining 14 transferFromPool callsites)**

These callsites ignore the return value. This is safe because:
1. All callers check `poolBalance > 0` before calling `transferFromPool`, so a zero-transfer scenario means the pool was depleted between the check and the call (a race with a concurrent transaction).
2. The callers treat the transfer as "best effort" -- they do not assume a specific amount was transferred for subsequent calculations. The transfers are terminal (rewards to players), not intermediate values feeding into further arithmetic.
3. The only exception is the pool-balance-then-transfer pattern in the whale/deity functions, where `affiliateReserve` is read once and used for multiple transfers. If the pool drains between reads, subsequent transfers return less, but this is handled gracefully by the cap-to-available logic inside `transferFromPool` itself.

**Pattern C: depositSteth (3 callsites -- no return value)**
- `depositSteth` has no return value. The function either succeeds (transfers stETH to sDGNRS) or reverts on transferFrom failure. All callers pre-approve the stETH amount before calling. **Correct.**

**Return value handling verdict: PASS.** No caller assumes more was transferred than actually occurred.

## Degenerette Reward Math (DELTA-06)

### Formula Verification

**Location:** DegenerusGameDegeneretteModule.sol, lines 1166-1180

**Step 1: BPS tier selection (lines 1167-1170)**
```solidity
uint256 bps;
if (matchCount == 6) bps = DEGEN_DGNRS_6_BPS;       // 400
else if (matchCount == 7) bps = DEGEN_DGNRS_7_BPS;   // 800
else bps = DEGEN_DGNRS_8_BPS;                         // 1500
```

The function is called at line 660:
```solidity
if (currency == CURRENCY_ETH && matches >= 6) {
    _awardDegeneretteDgnrs(player, amountPerTicket, matches);
}
```

`matches` comes from `_countMatches()` (line 857), which checks 4 quads of (color + symbol), producing 0-8 matches. The `matches >= 6` guard ensures `matchCount` is 6, 7, or 8. The `else` branch (1500 BPS) is only reachable for matchCount == 8. **Branching is correct and exhaustive.**

**Step 2: Pool balance check (lines 1172-1173)**
```solidity
uint256 poolBalance = sdgnrs.poolBalance(IStakedDegenerusStonk.Pool.Reward);
if (poolBalance == 0) return;
```
- Uses `sdgnrs` variable (DegeneretteModule line 234) which is `IStakedDegenerusStonk(ContractAddresses.SDGNRS)`. **CORRECT.**
- Pool.Reward (enum 3). **CORRECT.**

**Step 3: Bet cap (line 1175)**
```solidity
uint256 cappedBet = betWei > 1 ether ? 1 ether : betWei;
```
- Caps the bet at 1 ETH (1e18 wei). Bets above 1 ETH get the same reward as a 1 ETH bet.
- This prevents high-rollers from draining the Reward pool disproportionately.
- **CORRECT.**

**Step 4: Reward calculation (line 1176)**
```solidity
uint256 reward = (poolBalance * bps * cappedBet) / (10_000 * 1 ether);
```

Algebraically:
```
reward = poolBalance * (bps / 10_000) * (cappedBet / 1 ether)
       = poolBalance * tier_percentage * bet_fraction_of_1_ETH
```

This means:
- For a 1 ETH bet with 6 matches: `reward = poolBalance * 4%` = 4% of the pool
- For a 0.5 ETH bet with 7 matches: `reward = poolBalance * 8% * 0.5` = 4% of the pool
- For a 1 ETH bet with 8 matches: `reward = poolBalance * 15%` = 15% of the pool

The formula scales linearly with bet size (up to the 1 ETH cap) and with the tier percentage. **Semantically correct.**

**Step 5: Zero reward guard (line 1177)**
```solidity
if (reward == 0) return;
```
- Prevents zero-amount transfer. **CORRECT.**

**Step 6: Transfer (line 1179)**
```solidity
sdgnrs.transferFromPool(IStakedDegenerusStonk.Pool.Reward, player, reward);
```
- Pool: Reward (3). **CORRECT.**
- Recipient: `player`. **CORRECT.**
- Return value: Ignored. **Safe** -- terminal reward.

### Overflow Analysis

**Worst-case inputs:**
- `poolBalance`: Maximum Reward pool = 50,000,000,000 * 1e18 = 5e28 (50B tokens with 18 decimals)
- `bps`: Maximum = 1500
- `cappedBet`: Maximum = 1e18 (1 ETH)

**Calculation:**
```
numerator = poolBalance * bps * cappedBet
          = 5e28 * 1500 * 1e18
          = 7.5e49

denominator = 10_000 * 1e18 = 1e22

result = 7.5e49 / 1e22 = 7.5e27
```

`uint256.max` = ~1.16e77. The numerator (7.5e49) is well within bounds. **No overflow possible.**

**Minimum meaningful reward:**
- `poolBalance` = 1e18 (1 token), `bps` = 400, `cappedBet` = 5e15 (0.005 ETH min bet)
- `reward = (1e18 * 400 * 5e15) / (1e22) = 2e36 / 1e22 = 2e14` (0.0002 tokens)
- Non-zero for any pool balance >= 1 token with minimum ETH bet. **Correct.**

**Degenerette reward math verdict: PASS.** Formula verified, BPS constants confirmed, overflow safe, tier branching exhaustive.
