### Change 2: _backfillGapDays function (lines 1448-1473)

```solidity
/// @dev Backfill rngWordByDay and process coinflip payouts for gap days
///      caused by VRF stall. Derives deterministic words from the first
///      post-gap VRF word via keccak256(vrfWord, gapDay).
///      NOTE: Gap days get zero nudges (totalFlipReversals not consumed).
///      NOTE: resolveRedemptionPeriod is NOT called for backfilled gap days --
///      the redemption timer continued ticking in real time during the stall;
///      it resolves only on the current day via the normal rngGate path.
/// @param vrfWord The first post-gap VRF random word.
/// @param startDay First gap day (dailyIdx + 1).
/// @param endDay Current day (exclusive -- not backfilled, handled by normal path).
/// @param bonusFlip Whether presale bonus applies to coinflip resolution.
function _backfillGapDays(
    uint256 vrfWord,
    uint48 startDay,
    uint48 endDay,
    bool bonusFlip
) private {
    for (uint48 gapDay = startDay; gapDay < endDay;) {
        uint256 derivedWord = uint256(keccak256(abi.encodePacked(vrfWord, gapDay)));
        if (derivedWord == 0) derivedWord = 1;
        rngWordByDay[gapDay] = derivedWord;
        coinflip.processCoinflipPayouts(bonusFlip, derivedWord, gapDay);
        emit DailyRngApplied(gapDay, derivedWord, 0, derivedWord);
        unchecked { ++gapDay; }
    }
}
```

**Purpose:** For each gap day, derive a deterministic RNG word from `keccak256(vrfWord, gapDay)`, store it in `rngWordByDay`, process coinflip payouts, and emit the `DailyRngApplied` event.

**Source phase:** 59-01

## 2. Per-Change Correctness Analysis

### Change 1: rngGate gap detection (lines 791-795)

**1. Correctness:** The code correctly identifies gap days by comparing `day` (current day from `_simulatedDayIndexAt(block.timestamp)`) with `dailyIdx` (last processed day index). When `day > idx + 1`, there is at least one unprocessed gap day. The call to `_backfillGapDays(currentWord, idx + 1, day, bonusFlip)` passes the correct range: `[idx+1, day)` (exclusive upper bound, since `day` itself is processed by the normal path at line 798).

**2. State invariants:** No storage writes in this code -- it only reads `dailyIdx` and delegates to `_backfillGapDays`. `dailyIdx` is updated later by `_unlockRng` (line 1385: `dailyIdx = day`), which runs after both backfill and current-day processing complete.

**3. Access control:** This code runs inside `rngGate()` which is `internal`. The only caller is `advanceGame()` which is `external` but permissionless (anyone can advance the game). The gap detection itself introduces no new access control concern -- it merely extends the existing `rngGate` path.

**4. Reentrancy:** No external calls here. The external call to `coinflip.processCoinflipPayouts` happens inside `_backfillGapDays` (analyzed below).

**5. Arithmetic safety:** `idx + 1` -- `idx` is `uint48`, and `idx + 1` could overflow at `type(uint48).max`, but `dailyIdx` represents a day index starting from game launch. Reaching `2^48 - 1` days (~770 billion years) is operationally impossible. Solidity 0.8.34 would revert on overflow regardless.

**6. Edge cases:**
- **0 gap days** (`day == idx + 1`): The `if` condition is false, `_backfillGapDays` is not called. Correct -- no backfill needed.
- **1 gap day** (`day == idx + 2`): `_backfillGapDays(currentWord, idx+1, idx+2, bonusFlip)` processes exactly one gap day. Correct.
- **Large gap** (`day >> idx + 1`): Analyzed in Gas Ceiling Assessment (Section 6).

### Change 2: _backfillGapDays function (lines 1448-1473)

**1. Correctness:**
- `keccak256(abi.encodePacked(vrfWord, gapDay))` produces a unique 256-bit hash per gap day. Each gap day uses the same VRF word but a different `gapDay` counter, producing distinct derived words.
- `derivedWord == 0` guard ensures `rngWordByDay[gapDay]` is never set to 0. This matches `rawFulfillRandomWords` (line 1433: `if (word == 0) word = 1`). Since `rngWordByDay[day] != 0` is used as the "already processed" guard at line 773, storing 0 would make the day appear unprocessed -- the zero-guard prevents this.
- `coinflip.processCoinflipPayouts` is called with the derived word and the correct gap day epoch, resolving all coinflip stakes for that day.
- `DailyRngApplied` is emitted with `nudges=0` (third parameter), correctly distinguishing backfilled days from normally processed days where nudges may be non-zero.
- The event's `rawWord` and `finalWord` parameters are both `derivedWord` (no nudge modification), which is correct since backfilled days do not consume nudges.

**2. State invariants:**
- `rngWordByDay[gapDay]` is written from 0 to a non-zero value. This is consistent: the day was previously unprocessed (word=0), and after backfill it has a valid word.
- No other storage variables are modified directly in this function (coinflip state is modified via external call).

**3. Access control:** `private` function, callable only from within DegenerusGameAdvanceModule. The only call site is line 794 inside `rngGate()`. No unauthorized caller path.

**4. Reentrancy:**
- External call: `coinflip.processCoinflipPayouts(bonusFlip, derivedWord, gapDay)`.
- The coinflip contract is a trusted protocol contract at a fixed address (`ContractAddresses.COINFLIP`). It has the `onlyDegenerusGameContract` modifier (line 782).
- CEI analysis: `rngWordByDay[gapDay]` is written BEFORE the external call (line 1468 before 1469). Even if `processCoinflipPayouts` re-entered, it could not cause a double-write because `rngWordByDay[gapDay]` is already non-zero, and there is no re-entrant path back to `_backfillGapDays` (the `rngWordByDay[day] != 0` guard at line 773 would return early).
- **VERDICT: No CEI violation.** Storage write happens before external call, and re-entry is further blocked by the non-zero guard.

**5. Arithmetic safety:**
- `keccak256` output is 256 bits, cast to `uint256` -- no truncation.
- `unchecked { ++gapDay; }` -- `gapDay` is `uint48` and starts at `startDay`. The loop terminates when `gapDay >= endDay`. Since `endDay` is bounded by current day (realistic max ~3650 for 10 years of game operation), overflow of `uint48` is impossible.
- `abi.encodePacked(vrfWord, gapDay)` -- `vrfWord` is `uint256` (32 bytes), `gapDay` is `uint48` (6 bytes). Total 38 bytes, no ambiguity in packed encoding since both types have fixed sizes.

**6. Edge cases:**
- **startDay == endDay:** Loop body never executes. This cannot happen at the call site (guarded by `day > idx + 1`, so `startDay = idx+1 < day = endDay`).
- **derivedWord == 0:** Handled by the zero-guard (`derivedWord = 1`). The probability of `keccak256` producing exactly 0 is `1/2^256`, but the guard ensures correctness regardless.
- **No coinflip stakes for a gap day:** `processCoinflipPayouts` handles this gracefully -- it writes the coinflip day result regardless of whether stakes exist (line 842: `flipsClaimableDay = epoch`).

### Change 3: Orphaned lootbox index recovery (lines 1346-1360)

**1. Correctness:**
- `outgoingRequestId = vrfRequestId` captures the stalled request ID before it is cleared at line 1364 (`vrfRequestId = 0`). Correct ordering.
- `orphanedIndex = lootboxRngRequestIndexById[outgoingRequestId]` looks up which lootbox index was reserved by the stalled request. If no index was reserved (e.g., no lootbox purchases were pending when the VRF request was made), `orphanedIndex == 0`.
- Double guard: `orphanedIndex != 0 && lootboxRngWordByIndex[orphanedIndex] == 0` ensures we only backfill if (a) an index was actually reserved and (b) it does not already have a word (prevents overwrite if the VRF somehow fulfilled before the swap).
- `keccak256(abi.encodePacked(lastLootboxRngWord, orphanedIndex))` derives a deterministic fallback word from the entropy chain.
- `lastLootboxRngWord = fallbackWord` updates the entropy chain so future derivations have fresh entropy.

**2. State invariants:**
- `lootboxRngWordByIndex[orphanedIndex]` transitions from 0 to non-zero. Consistent with the invariant that non-zero means "RNG available for this index."
- `lastLootboxRngWord` is updated, maintaining the lootbox entropy chain. This is the same pattern as `_finalizeLootboxRng` (line 837).

**3. Access control:** Runs inside `updateVrfCoordinatorAndSub` which requires `msg.sender == ContractAddresses.ADMIN` (line 1339). `ContractAddresses.ADMIN` is the DegenerusAdmin governance contract. Only governance can trigger this code.

**4. Reentrancy:** No external calls in this block. Pure storage writes and event emission. No CEI concern.

**5. Arithmetic safety:**
- `keccak256` output cast to `uint256` -- no truncation.
- `abi.encodePacked(lastLootboxRngWord, orphanedIndex)` -- `uint256` (32 bytes) + `uint48` (6 bytes) = 38 bytes. Fixed-size types, no packed encoding ambiguity.
- `fallbackWord == 0` guard prevents storing 0, matching the pattern elsewhere.

**6. Edge cases:**
- **vrfRequestId == 0:** `outgoingRequestId` is 0, the `if` block is skipped. Correct -- no stalled request means nothing to backfill.
- **orphanedIndex == 0:** The `if` guard at line 1351 prevents the backfill. This handles the case where `lootboxRngRequestIndexById[requestId]` was never set (request was for daily RNG only, not lootbox).
- **lootboxRngWordByIndex[orphanedIndex] != 0:** The second guard prevents overwrite. This handles the edge case where VRF fulfilled between the swap decision and execution (possible with governance delay).
- **lastLootboxRngWord == 0:** If no lootbox RNG has ever been processed, `lastLootboxRngWord` is 0. Then `keccak256(abi.encodePacked(0, orphanedIndex))` still produces a valid hash. This is not a weakness -- the derivation is deterministic but unpredictable to an attacker who does not know when governance will execute the swap.

## 3. Attack Surface Verdicts

### Surface 1: keccak256 derivation predictability

**Concern:** Can a player predict the derived words used for gap day coinflip resolution or orphaned lootbox outcomes before they are committed?

**Gap day derivation** (line 1466):
```solidity
uint256 derivedWord = uint256(keccak256(abi.encodePacked(vrfWord, gapDay)));
```

**Orphaned lootbox derivation** (lines 1352-1354):
```solidity
uint256 fallbackWord = uint256(keccak256(abi.encodePacked(
    lastLootboxRngWord, orphanedIndex
)));
```

**Analysis:**

For gap day words: The derivation uses `vrfWord` which is the raw VRF random word delivered by Chainlink. This word is unknown until the new coordinator's VRF response arrives in `rawFulfillRandomWords` (line 1437: `rngWordCurrent = word`). The backfill runs atomically inside `rngGate()` during `advanceGame()` -- by the time `_backfillGapDays` executes, the VRF word is already stored in `rngWordCurrent` and the derived words are immediately committed to `rngWordByDay`. There is no transaction boundary between VRF delivery and backfill where a player could observe `vrfWord` and front-run. The VRF fulfillment and `advanceGame` are separate transactions, but `vrfWord` is only stored in `rngWordCurrent` (not in `rngWordByDay` for gap days) during fulfillment. The backfill happens in the next `advanceGame` call. A miner/validator who sees the VRF fulfillment transaction could theoretically observe `rngWordCurrent` before calling `advanceGame`, but this is the standard VRF security model -- the VRF word is committed on-chain before game processing, and Chainlink VRF v2.5 prevents miners from withholding fulfillments.

For orphaned lootbox words: `lastLootboxRngWord` is a public storage variable (anyone can read it). However, the derivation runs inside `updateVrfCoordinatorAndSub` which is admin-only (governance-gated). Players cannot call this function or front-run it with strategic lootbox purchases because the orphaned index is already reserved (fixed before the swap). The deterministic nature of the fallback word is acceptable because the lootbox outcome is determined by the word + player address + index, and the player's lootbox position was already committed before the stall.

**VERDICT: SAFE.** Gap day words depend on unpredictable VRF entropy. Orphaned lootbox words run in an admin-only context with pre-committed player positions. No manipulation window exists in either path.

### Surface 2: Gap day coinflip manipulation

**Concern:** Can a player place strategic coinflip stakes during the VRF stall, knowing they will be resolved by backfill with derived words?

**Relevant code** (line 1469):
```solidity
coinflip.processCoinflipPayouts(bonusFlip, derivedWord, gapDay);
```

**Analysis:**

During a VRF stall, `currentDayView()` keeps advancing based on `block.timestamp`. Players can purchase tickets that stake into future days via `_targetFlipDay = currentDayView() + 1`. So a player CAN place stakes targeting gap days during the stall.

However, the player does NOT know:
1. The VRF word that will resolve those stakes (it comes from the new coordinator, which has not been deployed yet during the stall).
2. The derived word for each gap day (`keccak256(vrfWord, gapDay)` requires the unknown VRF word).
3. Whether the coordinator swap will happen at all, or when.

The coinflip outcome for each day is determined by `keccak256(abi.encodePacked(rngWord, epoch)) % 20` (line 784 of BurnieCoinflip.sol). This mixing makes each day's outcome dependent on both the derived word and the day epoch, preventing cross-day correlation attacks.

The attacker's best strategy during a stall is the same as normal play: stake and hope for a favorable outcome. There is no informational advantage from knowing that backfill will occur.

**VERDICT: SAFE.** Stakes placed during stall are resolved with unpredictable derived words. No strategic advantage from knowing backfill will occur.

### Surface 3: Unbounded backfill loop gas

**Concern:** Can an attacker force a stall lasting long enough that `_backfillGapDays` exceeds the block gas limit?

**Relevant code** (lines 1465-1472):
```solidity
for (uint48 gapDay = startDay; gapDay < endDay;) {
    uint256 derivedWord = uint256(keccak256(abi.encodePacked(vrfWord, gapDay)));
    if (derivedWord == 0) derivedWord = 1;
    rngWordByDay[gapDay] = derivedWord;
    coinflip.processCoinflipPayouts(bonusFlip, derivedWord, gapDay);
    emit DailyRngApplied(gapDay, derivedWord, 0, derivedWord);
    unchecked { ++gapDay; }
}
```

**Analysis:** See Section 6 (Gas Ceiling Assessment) for detailed per-iteration cost breakdown.

Summary: Each iteration costs ~55,000-75,000 gas (1 cold SSTORE + 1 external call to `processCoinflipPayouts` + 1 event emit). At a 14M gas ceiling, approximately 186-254 gap days can be processed before breach. This requires a VRF stall lasting 6+ months without governance intervention.

Mitigating factors:
1. VRF stalls require Chainlink coordinator failure -- not an attacker-controlled condition.
2. Governance can swap coordinators after a 20h+ stall via propose/vote/execute.
3. The 120-day inactivity timeout exists as an independent game-ending recovery path.
4. A 6-month VRF outage without governance action is operationally implausible.

**VERDICT: SAFE.** The theoretical gas ceiling breach requires an unrealistic 6+ month VRF stall without any governance intervention. The attack requires Chainlink failure (not attacker-controlled) and governance inaction (documented in KNOWN-ISSUES.md as WAR-01 equivalent trust assumption).

### Surface 4: Orphaned lootbox index backfill in admin function

**Concern:** Can governance exploit the orphaned index backfill to manipulate lootbox outcomes?

**Relevant code** (lines 1346-1360):
```solidity

    uint48 orphanedIndex = lootboxRngRequestIndexById[outgoingRequestId];
    if (orphanedIndex != 0 && lootboxRngWordByIndex[orphanedIndex] == 0) {
```

**Analysis:**

The admin (`ContractAddresses.ADMIN`) is the DegenerusAdmin governance contract, not an EOA. A coordinator swap requires:
1. Proposal by any sDGNRS holder
2. 3-day voting period
3. Threshold approval by sDGNRS holders
4. 12h stall duration requirement

The fallback word is deterministic: `keccak256(lastLootboxRngWord, orphanedIndex)`. Both inputs are fixed before the swap:
- `lastLootboxRngWord` is the last successfully processed lootbox RNG word (historical, public on-chain).
- `orphanedIndex` is the index reserved by the stalled VRF request (fixed when the request was made).

Governance cannot choose the fallback word. The only governance "attack" would be:
- **Refusing to swap:** Harms all players equally. Documented trust assumption.
- **Swapping to a malicious coordinator:** The new coordinator address is visible in the proposal. Community can reject.

At most one lootbox index can be orphaned because `requestLootboxRng` (line 690) reverts if `rngRequestTime != 0`, and `rngRequestTime` is only set during VRF request flow.

**VERDICT: SAFE.** Fallback word is deterministic from public on-chain values. Governance cannot choose the outcome. Single-orphan guarantee prevents amplification.

### Surface 5: midDayTicketRngPending clearing side effects

**Concern:** Does clearing `midDayTicketRngPending = false` during coordinator swap discard pending lootbox funds?

**Relevant code** (line 1371):
```solidity
midDayTicketRngPending = false;
```

**Analysis:**

`midDayTicketRngPending` is a boolean flag that controls whether `advanceGame` checks for a pending mid-day lootbox RNG fulfillment (line 166-171). When true, `advanceGame` reads `lootboxRngWordByIndex[lootboxRngIndex - 1]` and reverts with `NotTimeYet` if the word is still 0.

Clearing this flag does NOT affect:
- `lootboxRngPendingEth` -- tracks ETH accumulated for the next lootbox RNG trigger. NOT reset during swap.
- `lootboxRngPendingBurnie` -- tracks BURNIE accumulated for the next lootbox RNG trigger. NOT reset during swap.
- `lootboxRngWordByIndex[orphanedIndex]` -- handled by Change 3 (orphaned index backfill).
- `lootboxEth[index][player]` / `lootboxBurnie[index][player]` -- player-specific lootbox balances, untouched.

The flag only prevents the `NotTimeYet` revert. Since the orphaned index has already been backfilled (Change 3 runs before this line), the lootbox word exists. The flag clearing is safe because the condition it guards (`word == 0` for the pending index) is no longer true after orphan backfill.

**VERDICT: SAFE.** The flag controls a revert guard, not fund flows. All fund-tracking variables (`lootboxRngPendingEth`, `lootboxRngPendingBurnie`, per-player balances) are preserved. The orphaned index backfill ensures the lootbox word exists.

### Surface 6: Interaction with redemption periods (skipping resolveRedemptionPeriod)

**Concern:** Is it correct that `_backfillGapDays` does NOT call `resolveRedemptionPeriod` for gap days?

**Relevant code context -- normal path** (lines 801-808, inside rngGate after backfill):
```solidity
// Resolve gambling burn period if pending
{
    IStakedDegenerusStonk sdgnrs = IStakedDegenerusStonk(ContractAddresses.SDGNRS);
    if (sdgnrs.hasPendingRedemptions()) {
```

`resolveRedemptionPeriod` is called only for the current day (the day after all gap days), not for each backfilled gap day.

**Analysis:**

Redemption periods operate on wall-clock time (`block.timestamp`), not game-day time. During a VRF stall:
1. The redemption timer continues ticking because `block.timestamp` advances.
2. A 7-day redemption period that started before the stall may expire during the stall.
3. When `advanceGame` finally runs post-swap, `resolveRedemptionPeriod` is called once for the current day, which checks `block.timestamp` against the period's expiry.

If `resolveRedemptionPeriod` were called for each gap day, it would:
- Process the SAME pending redemption N times (once per gap day).
- The RNG roll for redemption resolution uses `(currentWord >> 8) % 151 + 25` (line 748). Each gap day has a different `currentWord`, so each call would produce a different roll, potentially resolving the redemption differently each time.
- This would be a critical bug: a single redemption period could be resolved multiple times with different outcomes.

By calling it only once (for the current day), the redemption is resolved exactly once with the current day's word. This is the correct behavior.

**VERDICT: SAFE.** Calling `resolveRedemptionPeriod` for gap days would cause multi-processing (a critical bug). The current approach of resolving only on the current day is correct because the redemption timer is wall-clock-based, not game-day-based.

### Surface 7: totalFlipReversals carry-over

**Concern:** Could carry-over nudges be used to manipulate backfilled gap day outcomes?

**Relevant code -- backfill path** (line 1466):
```solidity
uint256 derivedWord = uint256(keccak256(abi.encodePacked(vrfWord, gapDay)));
```

**Relevant code -- normal day path** (lines 1480-1486):
```solidity
uint256 nudges = totalFlipReversals;
finalWord = rawWord;
if (nudges != 0) {
    unchecked {
        finalWord += nudges;
    }
    totalFlipReversals = 0;
}
```

**Analysis:**

Gap days use `keccak256(vrfWord, gapDay)` directly. They do NOT read or consume `totalFlipReversals`. The nudge modification only happens in `_applyDailyRng` (line 798), which runs for the current day only (after backfill completes).

Flow:
1. `_backfillGapDays` processes all gap days with raw keccak-derived words (no nudges).
2. `_applyDailyRng` processes the current day, reading and consuming `totalFlipReversals`.

Nudges purchased before/during the stall accumulate in `totalFlipReversals` and apply ONLY to the current day's word. They cannot affect gap day outcomes because the backfill function does not read `totalFlipReversals`.

This is the correct design: nudges represent a player's attempt to influence the NEXT day's outcome. During a stall, "the next day" becomes "the first post-swap day" (the current day when advanceGame runs). Applying them to historical gap days would violate the temporal expectation.

**VERDICT: SAFE.** Gap days use raw derived words without nudges. `totalFlipReversals` is consumed only by `_applyDailyRng` for the current day. No cross-contamination.

### Surface 8: flipsClaimableDay monotonicity

**Concern:** Does `processCoinflipPayouts` maintain `flipsClaimableDay` monotonicity during backfill?

**Relevant code -- processCoinflipPayouts** (BurnieCoinflip.sol line 842):
```solidity
flipsClaimableDay = epoch;
```

**Relevant code -- backfill loop** (lines 1465-1469):
```solidity
for (uint48 gapDay = startDay; gapDay < endDay;) {
    ...
    coinflip.processCoinflipPayouts(bonusFlip, derivedWord, gapDay);
    ...
    unchecked { ++gapDay; }
}
```

**Analysis:**

The backfill loop iterates `gapDay` from `startDay` to `endDay - 1` in strictly ascending order (`++gapDay` in each iteration). Each call to `processCoinflipPayouts` sets `flipsClaimableDay = epoch` where `epoch` is `gapDay`.

After backfill: `flipsClaimableDay = endDay - 1` (last gap day processed).

Then the current day processing at line 799:
```solidity
coinflip.processCoinflipPayouts(bonusFlip, currentWord, day);
```
Sets `flipsClaimableDay = day`. Since `day == endDay > endDay - 1`, this is monotonically increasing.

Consumers of `flipsClaimableDay`:
- `claimCoinflips` (BurnieCoinflip.sol line 407): checks `latest = flipsClaimableDay` and ensures `latestClaimedDay < latest`.
- `bulkClaimCoinflips` (line 935): similar check with `lastDay = flipsClaimableDay`.
- Both only require monotonicity to prevent claiming unresolved days. The ascending loop guarantees this.

**VERDICT: SAFE.** The ascending loop order guarantees `flipsClaimableDay` increases monotonically through gap days and into the current day. No user can claim an unresolved day.

## 5. Flow Interaction Analysis

### Flow 1: Coordinator swap (updateVrfCoordinatorAndSub)

**Entry:** `msg.sender == ContractAddresses.ADMIN` (governance)

**Step 1 -- State capture:**
```
outgoingRequestId = vrfRequestId   // Save before clearing
```

**Step 2 -- Orphaned lootbox backfill (if applicable):**
```
IF outgoingRequestId != 0 AND orphanedIndex != 0 AND lootboxRngWordByIndex[orphanedIndex] == 0:
    fallbackWord = keccak256(lastLootboxRngWord, orphanedIndex)
    lootboxRngWordByIndex[orphanedIndex] = fallbackWord
    lastLootboxRngWord = fallbackWord
    emit LootboxRngApplied(orphanedIndex, fallbackWord, outgoingRequestId)
```

**Step 3 -- Coordinator rotation:**
```
vrfCoordinator = newCoordinator
vrfSubscriptionId = newSubId
vrfKeyHash = newKeyHash
```

**Step 4 -- VRF state reset:**
```
rngLockedFlag = false
vrfRequestId = 0
rngRequestTime = 0
rngWordCurrent = 0
```

**Step 5 -- Mid-day flag clearing:**
```
midDayTicketRngPending = false
```

**Step 6 -- Event:**
```
emit VrfCoordinatorUpdated(current, newCoordinator)
```

**State after swap:**
- New coordinator is wired.
- Orphaned lootbox index (if any) has a valid word.
- `rngLockedFlag = false` -- `advanceGame` can proceed.
- `vrfRequestId = 0` -- new VRF request can be made.
- `rngRequestTime = 0` -- no stale timeout.
- `rngWordCurrent = 0` -- forces new VRF request (line 773 `rngWordByDay[day] != 0` check returns 0, falls through to line 828 `_requestRng`).
- `midDayTicketRngPending = false` -- no `NotTimeYet` revert.
- `totalFlipReversals` preserved -- nudges apply on next day.

**Consistency check:** All state variables are in a clean state that allows immediate `advanceGame` execution. No dangling references.

### Flow 2: First post-swap advanceGame (rngGate)

**Entry:** Any caller invokes `advanceGame()` which calls `rngGate(ts, day, lvl, isTicketJackpotDay, bonusFlip)`.

**Step 1 -- Day check (line 773):**
```
rngWordByDay[day] == 0  // True -- current day not yet processed
```

**Step 2 -- VRF word check (line 775-778):**
```
currentWord = rngWordCurrent  // 0 after swap
currentWord == 0              // True -- no VRF word available
```
Falls through to line 828: `_requestRng(isTicketJackpotDay, lvl)`. A new VRF request is sent to the new coordinator. Returns 1.

**Step 3 -- VRF fulfillment (separate tx):**
New coordinator calls `rawFulfillRandomWords(requestId, randomWords)`.
```
msg.sender == address(vrfCoordinator)  // New coordinator
requestId == vrfRequestId              // Matches
rngWordCurrent = 0                     // Not yet set
word = randomWords[0]
rngLockedFlag == true                  // Set by _requestRng
rngWordCurrent = word                  // Store for advanceGame
```

**Step 4 -- Second advanceGame call (rngGate, lines 773-799):**
```
rngWordByDay[day] == 0     // Still 0 -- day not yet processed
currentWord = rngWordCurrent  // Non-zero VRF word
currentWord != 0 && rngRequestTime != 0  // Both true
requestDay = _simulatedDayIndexAt(rngRequestTime)  // Current day
requestDay == day  // Request was made on current day (not stale)
```

**Step 5 -- Gap detection (lines 791-795):**
```
idx = dailyIdx  // Last processed day (before stall)
day > idx + 1   // True if gap days exist
_backfillGapDays(currentWord, idx + 1, day, bonusFlip)
```

**Step 6 -- Backfill (per gap day):**
For each gap day `g` in `[idx+1, day)`:
```
derivedWord = keccak256(currentWord, g)
if derivedWord == 0: derivedWord = 1
rngWordByDay[g] = derivedWord
coinflip.processCoinflipPayouts(bonusFlip, derivedWord, g)
emit DailyRngApplied(g, derivedWord, 0, derivedWord)
```

**Step 7 -- Current day processing (lines 797-799):**
```
currentWord = _applyDailyRng(day, currentWord)  // Applies nudges, stores word
coinflip.processCoinflipPayouts(bonusFlip, currentWord, day)
```

**Step 8 -- Post-processing (lines 801+):**
Redemption period resolution, lootbox finalization, jackpots, unlock.

**State after full flow:**
- All gap days have non-zero `rngWordByDay[g]` entries.
- All gap day coinflips resolved.
- Current day processed normally with nudges.
- `dailyIdx = day` (set by `_unlockRng`).
- `flipsClaimableDay = day` (monotonically increasing through gap days then current day).

### Flow consistency verification:

1. **Ordering dependency:** The swap MUST happen before the first `advanceGame`. This is guaranteed because `advanceGame` cannot proceed without a VRF word (it requests one from the new coordinator), and the new coordinator is set during the swap.

2. **Double-processing:** Could backfill + normal path process the same day? No. Backfill range is `[idx+1, day)` (exclusive of `day`). Normal path processes `day`. No overlap.

3. **Missing days:** Could any day be skipped? The range `[idx+1, day)` covers all gap days. The normal path covers `day`. Together, they cover `[idx+1, day]`, which is every day from the last processed day to the current day. No gaps.

4. **State leakage:** Does the swap leave any state that could cause the post-swap advanceGame to misbehave? All VRF state is reset. The only preserved state (`totalFlipReversals`) is intentional and correct.

|-----------|---------------------|

| `emit DailyRngApplied(gapDay, derivedWord, 0, derivedWord)` | ~375 (base) + 4 * ~256 (per topic/data) = ~1,399 |
| `unchecked { ++gapDay; }` | ~3 |
| Loop overhead (comparison, jump) | ~10 |

**Minimum per-iteration (no coinflip stakes):** ~26,557 gas
**Maximum per-iteration (with coinflip stakes and bounty):** ~56,557 gas

### Maximum gap days before gas ceiling breach

| Scenario | Per-iteration gas | Max gap days at 14M gas | Calendar equivalent |
|----------|-------------------|------------------------|---------------------|

| Heavy stakes | ~56,557 | ~247 days | ~8 months |

**Note:** The `advanceGame` function has overhead beyond just the backfill loop (VRF processing, current day processing, jackpots, etc.). Accounting for ~2M gas of non-backfill overhead, the effective ceiling is:

| Scenario | Effective max gap days |
|----------|----------------------|
| No stakes | ~452 days |
| Heavy stakes | ~212 days |
| Worst case | ~160 days |

### Risk assessment

A 160+ day VRF stall without governance intervention is operationally implausible:
1. Governance can propose a coordinator swap after 20h of stall.
2. The voting period is 3 days.
3. Even with governance delays, a swap within 2 weeks is realistic.
4. The 120-day inactivity timeout provides a hard backstop (game ends).
5. A 6-month continuous Chainlink VRF failure has never occurred on any supported chain.

**Assessment: No realistic gas ceiling concern.** The theoretical maximum is far beyond any plausible operational scenario.
