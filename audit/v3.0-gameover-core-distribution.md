### 1. runTerminalDecimatorJackpot (DecimatorModule.sol:880-922)

**Double-resolution guard (line 888):**
```solidity
if (lastTerminalDecClaimRound.lvl == lvl) {
    return poolWei;
}
```
Verified correct. If `handleGameOverDrain` were somehow called twice for the same level (which is already prevented by `gameOverFinalJackpotPaid` at GameOverModule:69), this guard returns the full pool amount back to the caller, preventing any state mutation. The guard uses `lastTerminalDecClaimRound` (separate from `lastDecClaimRound`), so it does not interfere with normal decimator operations.

**Return value flow (lines 909-921):**
- When `totalWinnerBurn == 0` (no qualifying burns): returns `poolWei` (full refund). This flows back to `handleGameOverDrain` as `decRefund`, where `decSpend = decPool - decRefund = 0`, so `claimablePool` is not incremented, and `remaining += decRefund` returns the full 10% to the terminal jackpot pool. Correct.
- When winners exist: returns `0`. `decRefund = 0`, `decSpend = decPool`, `claimablePool += decPool`. Correct.

**decBucketOffsetPacked write (line 914):**
```solidity
decBucketOffsetPacked[lvl] = packedOffsets;
```
This writes to the same `mapping(uint24 => uint64)` used by the normal decimator at line 341. Collision analysis (resolves Research Q1):
- Normal decimator: called by `_handleEndgame` (EndgameModule:193-218) during level completion milestones (x5 and x00). The `lvl` parameter is the level that just completed.
- Terminal decimator: called by `handleGameOverDrain` (GameOverModule:140) during GAMEOVER. The `lvl` parameter is the current level (aliased to 1 if 0).
- **Collision is impossible:** GAMEOVER fires because the current level did NOT complete (death clock expired). Normal decimator fires because the current level DID complete. These are mutually exclusive states for the same level number. If normal decimator resolved level X, the game advanced past X, so GAMEOVER fires at level X+n (n >= 1), targeting a different key in the mapping.

**Weighted burn calculation (lines 892-906):**
The resolution loop iterates `denom = 2..12` (matching `DECIMATOR_MAX_DENOM`), selects winning subbuckets via `_decWinningSubbucket(decSeed, denom)`, and accumulates `totalWinnerBurn` from `terminalDecBucketBurnTotal[bucketKey]`. The `bucketKey = keccak256(abi.encode(lvl, denom, winningSub))` correctly scopes to both level and winning subbucket. The separate `terminalDecBucketBurnTotal` mapping (vs `decBucketBurnTotal` for normal decimator) prevents cross-contamination.

**Snapshot storage (lines 917-919):**
```solidity
lastTerminalDecClaimRound.lvl = lvl;
lastTerminalDecClaimRound.poolWei = uint96(poolWei);
lastTerminalDecClaimRound.totalBurn = uint128(totalWinnerBurn);
```
The `uint96` cast on `poolWei` is safe: this is 10% of the remaining available funds, which would need to exceed 7.9e28 wei (~79 billion ETH) to overflow `uint96`. The `uint128` cast on `totalWinnerBurn` is safe: each player's `weightedBurn` is capped at `uint88` max (~3e26), and the total across all winning subbuckets across all 11 denominators would need an astronomically large number of players to overflow `uint128` (max ~3.4e38).

### 4. _terminalDecMultiplierBps (DecimatorModule.sol:1000-1006)


**Boundary value verification:**
| daysRemaining | Branch | Calculation | Result (BPS) | Multiplier |
|---------------|--------|-------------|--------------|------------|
| 120 | > 10 | 120 * 2500 | 300,000 | 30x |
| 40 | > 10 | 40 * 2500 | 100,000 | 10x |
| 11 | > 10 | 11 * 2500 | 27,500 | 2.75x |

### 5. claimablePool Accounting Through Terminal Decimator Path

**Trace of claimablePool mutations in the terminal decimator sub-path:**

**Step 1: runTerminalDecimatorJackpot called (GameOverModule:140)**
At this point, `claimablePool` has already been incremented by deity refunds (if any) at line 105. The `remaining` variable tracks unallocated funds.
- `decPool = remaining / 10` -- 10% of available
- `decRefund = runTerminalDecimatorJackpot(decPool, lvl, rngWord)` -- returns 0 (winners exist) or poolWei (no winners)
- **No claimablePool mutation inside runTerminalDecimatorJackpot itself.** The function only snapshots claim round data; actual claimable credits happen when players call `claimTerminalDecimatorJackpot` later.

**Step 2: Post-decimator accounting (GameOverModule:141-146)**
```solidity
uint256 decSpend = decPool - decRefund;
if (decSpend != 0) {
    claimablePool += decSpend;
}
remaining -= decPool;
remaining += decRefund;
```
- **Winners exist (decRefund = 0):** `decSpend = decPool`, `claimablePool += decPool`. This reserves the full 10% for terminal decimator claims. `remaining -= decPool` removes it from unallocated.
- **No winners (decRefund = decPool):** `decSpend = 0`, no claimablePool change. `remaining -= decPool; remaining += decRefund` nets to zero change -- the 10% flows back to `remaining` for the terminal jackpot.

**Step 3: Player claims (post-GAMEOVER, DecimatorModule:930-936)**
```solidity
uint256 amountWei = _consumeTerminalDecClaim(msg.sender);
_addClaimableEth(msg.sender, amountWei, 0);
```
`_addClaimableEth` -> `_processAutoRebuy` returns false (gameOver) -> `_creditClaimable(msg.sender, amountWei)` -> `claimableWinnings[msg.sender] += amountWei`.

Note: `_creditClaimable` increments `claimableWinnings` but does NOT increment `claimablePool`. The pool was already reserved in Step 2. This is correct -- `claimablePool` was pre-reserved when `decSpend` was added, and individual claim credits are against that pre-reserved amount.

**Step 4: Player withdrawal (DegenerusGame.sol:1431-1441)**
```solidity
claimableWinnings[player] = 1; // sentinel
claimablePool -= payout;
```
The payout reduces `claimablePool`, returning accounting to equilibrium.

**Invariant verification: `address(this).balance + stETH >= claimablePool`**
- After Step 2: No ETH left the contract. `claimablePool` increased by `decSpend`. Since `decSpend <= remaining <= available = totalFunds - claimablePool_before`, we have `claimablePool_after = claimablePool_before + decSpend <= totalFunds`. Balance unchanged, invariant holds.
- After Step 3: No ETH left the contract. `claimableWinnings` updated but `claimablePool` unchanged. Invariant trivially holds.
- After Step 4: `claimablePool -= payout`, and ETH/stETH is transferred out by `payout` amount. `balance_after = balance_before - payout`, `claimablePool_after = claimablePool_before - payout`. If invariant held before, it holds after (same amount subtracted from both sides).

**PASS:** The `balance >= claimablePool` invariant is maintained at every mutation point in the terminal decimator path.

## Open Questions Resolved

### Research Q1: decBucketOffsetPacked Collision Between Normal and Terminal Decimator

**Resolution: No collision possible -- PASS**

Both `runDecimatorJackpot` (DecimatorModule:341) and `runTerminalDecimatorJackpot` (DecimatorModule:914) write to `decBucketOffsetPacked[lvl]`. However, they can never target the same level:

- **Normal decimator** fires during `_handleEndgame` (EndgameModule:193-218) at milestone level completions (levels ending in x5 or x00). The `lvl` parameter is the level that just completed. After the milestone, the game advances past level X.
- **Terminal decimator** fires during `handleGameOverDrain` (GameOverModule:140) at GAMEOVER. The `lvl` parameter is the current level where GAMEOVER triggered (death clock expired).

GAMEOVER fires because the current level did NOT complete (inactivity timeout). Normal decimator fires because a level DID complete. These are mutually exclusive states for the same level number. If normal decimator resolved level X, the game has already advanced past X, so GAMEOVER fires at level X+n (n >= 1), targeting a different key in the mapping.

Additionally, even in the hypothetical edge case where both used the same level (impossible per above), the separate claim round structs (`lastDecClaimRound` vs `lastTerminalDecClaimRound`) would prevent cross-contamination of claim validation, though the packed offsets would be corrupted.

### Research Q4: Terminal Decimator Claim Expiry / lastTerminalDecClaimRound Overwrite

**Resolution: Cannot be overwritten after GAMEOVER -- PASS**

`lastTerminalDecClaimRound` is only written by `runTerminalDecimatorJackpot` (DecimatorModule:917-919). The write path is:
1. `handleGameOverDrain` (GameOverModule:140) calls `IDegenerusGame(address(this)).runTerminalDecimatorJackpot`
2. `handleGameOverDrain` is guarded by `if (gameOverFinalJackpotPaid) return;` (line 69)
3. `gameOverFinalJackpotPaid = true` is set at line 128 before any delegatecalls
4. The double-resolution guard at DecimatorModule:888 provides defense-in-depth

After GAMEOVER, `gameOverFinalJackpotPaid` prevents any subsequent call from reaching the terminal decimator dispatch. Terminal decimator claims remain valid for the entire 30-day claim window (until `handleFinalSweep` zeroes `claimablePool` and sets `finalSwept = true`).

### Research Q5: _processAutoRebuy During Terminal Decimator Claims

**Resolution: Correctly skipped -- PASS**

`claimTerminalDecimatorJackpot()` (DecimatorModule:930) calls `_addClaimableEth(msg.sender, amountWei, 0)` (line 936). The call chain is:

1. `_addClaimableEth` (DecimatorModule:510-520) -- private function resolved within DecimatorModule's bytecode
2. Calls `_processAutoRebuy` (DecimatorModule:458-503)
3. `_processAutoRebuy` checks `if (gameOver) return false;` (line 463)
4. Since `gameOver = true` post-GAMEOVER, auto-rebuy is skipped
5. Falls through to `_creditClaimable(beneficiary, weiAmount)` (PayoutUtils.sol:30)
6. `_creditClaimable` does `claimableWinnings[beneficiary] += weiAmount` (unchecked)

There is no risk of the EndgameModule's `_addClaimableEth` (EndgameModule:241) being called instead. Solidity resolves `private` functions within the contract where they are defined, and `delegatecall` preserves this resolution. DecimatorModule's bytecode contains its own `_addClaimableEth` at line 510.

## GO-01: handleGameOverDrain Distribution

**Verdict:** PASS
**Files:** `DegenerusGameGameOverModule.sol:68-164`, `DegenerusGameJackpotModule.sol:288-324,1537-1576`
**Summary:** The handleGameOverDrain distribution sequence correctly processes deity refunds, allocates 10% to terminal decimator, 90% + refund to terminal jackpot, and sweeps remainder to vault. CEI ordering is correct. The `claimablePool` invariant is maintained through every mutation step. Level aliasing at level 0 is intentional by design.

### 1. Level Aliasing (GameOverModule.sol:71-72)

```solidity
uint24 currentLevel = level;
uint24 lvl = currentLevel == 0 ? 1 : currentLevel;
```

When GAMEOVER fires at level 0 (365-day deploy timeout), `lvl` is aliased to 1. This causes the terminal jackpot at line 153 to target `lvl + 1 = 2` (i.e., ticket holders for level 2). This is **by design** per the economics specification (Research Pitfall 4): at level 0, no tickets exist for level 1, so the terminal jackpot targets level 2 where pre-sale or early-bird tickets may have been purchased. Documented as known behavior.

If `currentLevel >= 1`, `lvl = currentLevel` and the terminal jackpot targets `lvl + 1` (next level's ticket holders), which is the standard behavior.

### 2. Deity Refund Section (GameOverModule.sol:78-107)

**Budget calculation (line 81):**
```solidity
uint256 budget = totalFunds > claimablePool ? totalFunds - claimablePool : 0;
```
Safe subtraction via ternary. `totalFunds = ethBal + stBal` (line 76) where `ethBal = address(this).balance` and `stBal = steth.balanceOf(address(this))`. The ternary ensures no underflow: if `claimablePool > totalFunds` (shouldn't happen under normal invariants but defensive), budget is 0 and no refunds are issued.

**Refund per pass (lines 79, 87):** Fixed at `DEITY_PASS_EARLY_GAMEOVER_REFUND = 20 ether`. Each owner's refund is `refundPerPass * uint256(purchasedCount)`. With `purchasedCount` being `uint16` (max 65535), the maximum refund per owner is `20e18 * 65535 = 1.31e24`, well within `uint256`.

**FIFO ordering (lines 83-103):** Iterates `deityPassOwners` array in order (FIFO by purchase time). Budget-capped: if `refund > budget` then `refund = budget` (lines 88-90). When `budget == 0`, loop breaks early (line 98).

**Budget cap prevents over-allocation (lines 88-90, 95-98):** Each refund is clamped to remaining `budget`. `budget -= refund` in unchecked block (line 95) is safe because `refund <= budget` is enforced by the cap. `totalRefunded += refund` is safe because `totalRefunded <= initial_budget <= totalFunds` which is bounded by contract balance. `claimableWinnings[owner] += refund` (line 93) is safe because the sum of all claimable winnings cannot exceed `totalFunds` (see Q3 resolution below for detailed analysis).

**claimablePool update (lines 104-106):**
```solidity
if (totalRefunded != 0) {
    claimablePool += totalRefunded;
}
```
After deity refunds, `claimablePool` increases by the total amount credited to deity pass owners. Since `totalRefunded <= budget = totalFunds - claimablePool_before`, we have `claimablePool_after = claimablePool_before + totalRefunded <= totalFunds`. Balance unchanged (no ETH transferred), invariant holds.

### 5. Terminal Jackpot Dispatch (GameOverModule.sol:149-160)

```solidity
if (remaining != 0) {
```

**runTerminalJackpot (JackpotModule.sol:288-324):**
Called with `(remaining, lvl + 1, rngWord)`. This function:
1. Rolls winning traits via `_rollWinningTraits(targetLvl, rngWord, true)` -- the `true` parameter indicates terminal jackpot (vs daily)
2. Computes bucket counts and share BPS using `JackpotBucketLib`
3. Calls `_distributeJackpotEth` (JackpotModule:1537-1576) which:
   - Iterates 4 trait buckets, selects winners from ticket pools, credits via `_addClaimableEth`
   - Accumulates `ctx.liabilityDelta` (total credited to players)
   - At line 1572-1573: `if (ctx.liabilityDelta != 0) { claimablePool += ctx.liabilityDelta; }`
   - Returns `ctx.totalPaidEth` (total distributed, including auto-rebuy ticket conversions)

**claimablePool updated INSIDE _distributeJackpotEth (line 1573):**
The `liabilityDelta` represents the net increase in player claimable balances. During GAMEOVER, `gameOver = true` means auto-rebuy is skipped (JackpotModule:982), so all winnings are credited as claimable ETH. `liabilityDelta` equals the total ETH credited to all winners.

This amount is at most `remaining` (the pool being distributed). Since `remaining <= available - decPool = totalFunds - claimablePool_before - decPool`, and `claimablePool` was already incremented by `decSpend` (which equals `decPool` when winners exist), we have:
- `claimablePool_after = claimablePool_before + decSpend + liabilityDelta`
- `liabilityDelta <= remaining <= totalFunds - claimablePool_before - decSpend`
