### CMT-011: Orphaned NatSpec from removed lootbox resolution function

- **What:** Lines 1140-1146 contain 6 NatSpec lines (`@notice Resolve a lootbox directly (decimator claims) using provided RNG.`, `@dev Access: ContractAddresses.JACKPOTS contract only...`, and 3 `@param` tags) with no corresponding function body. Lines 1147-1192 are empty whitespace. The function was removed but its NatSpec documentation was left behind. A NatSpec parser would associate these tags with the contract's closing brace `}` at line 1193.
- **Where:** DegenerusGameMintModule.sol:1140-1146
- **Why:** A warden reading the source or using a NatSpec extraction tool would see documentation for a `resolveLootboxDirect(address player, uint256 amount, uint256 rngWord)` function that does not exist. This could lead a warden to investigate why the function is documented but absent -- potentially filing a finding about missing functionality or incomplete implementation. The orphaned NatSpec also inflates the perceived API surface of the module.
- **Suggestion:** Delete lines 1140-1146 and the trailing blank lines (1147-1192).
- **Category:** comment-inaccuracy
- **Severity:** INFO

### CMT-012: Missing NatSpec on external function processFutureTicketBatch

- **What:** `processFutureTicketBatch(uint24 lvl)` at line 294 is a significant external function with zero NatSpec -- no `@notice`, `@dev`, `@param`, or `@return` tags. This function processes the ticket activation queue (rolling fractional remainders, generating trait batches via `_raritySymbolBatch`, managing write budgets for gas control) and is callable externally. It returns three values (`worked`, `finished`, `writesUsed`) with no documentation of their meaning.
- **Where:** DegenerusGameMintModule.sol:294-296
- **Why:** A warden auditing the module's external API would encounter a 127-line function with complex gas budgeting, LCG-based trait generation, and packed storage manipulation -- all without any NatSpec guidance. The return values `worked`, `finished`, and `writesUsed` have non-obvious semantics (e.g., `worked` means "at least one write was performed," `finished` means "queue fully processed," `writesUsed` is the gas budget consumed). Without NatSpec, a warden must reverse-engineer the function's purpose, access control expectations, and return value semantics from code alone.
- **Suggestion:** Add NatSpec documenting: the function's purpose (process ticket activation queue for a given level), access control (called via delegatecall from DegenerusGame during `advanceGame`), parameters (`@param lvl` -- the level whose ticket queue to process), return values (`@return worked` -- true if any queue entries were processed, `@return finished` -- true if the entire queue was drained, `@return writesUsed` -- storage writes consumed from the gas budget), and the write budget system (`@dev` explaining WRITES_BUDGET_SAFE and cold storage scaling).
- **Category:** comment-inaccuracy
- **Severity:** LOW

### CMT-013: purchaseCoin NatSpec claims RNG gating that does not exist

- **What:** The `@dev` NatSpec for `purchaseCoin` at line 564 states `BURNIE ticket and loot box purchases are allowed whenever RNG is unlocked`, implying that RNG lock status gates the function. However, neither the module's `_purchaseCoinFor` implementation nor the `DegenerusGame.purchaseCoin` entry point checks `rngLockedFlag`. The only gating conditions are `gameOver` (line 592) and the time-based `CoinPurchaseCutoff` for tickets (line 598). BURNIE purchases proceed regardless of VRF state.
- **Where:** DegenerusGameMintModule.sol:564
- **Why:** A warden reading this NatSpec would assume BURNIE purchases revert when a VRF request is pending (`rngLockedFlag == true`). This would affect their analysis of the purchase flow's availability during jackpot resolution, potentially leading to incorrect conclusions about player optionality during VRF-locked periods. The warden might also file a finding about inconsistent RNG gating if they observe that ETH `purchase()` similarly lacks an RNG check despite its own "Blocked when RNG is locked" NatSpec (DegenerusGame.sol:537).
- **Suggestion:** Remove the RNG reference. Change to: `@dev BURNIE ticket purchases are gated by a time-based cutoff before the liveness guard fires. Loot box purchases require lootboxRngIndex > 0.`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### CMT-014: recordMintData NatSpec lists "milestones" as tracked field but no milestone logic exists

- **What:** The `recordMintData` function's `@dev` NatSpec at line 169 states `mintPacked_[player] updated with level count, whale bonuses, milestones`. The function updates level count (levelCount field in mintPacked_) and interacts with whale bundle frozen state (clearing frozenUntilLevel and whaleBundleType when the player reaches the frozen level). However, there is no "milestones" concept in this function -- no milestone tracking, no milestone field in mintPacked_, and the word "milestones" appears nowhere else in the contract.
- **Where:** DegenerusGameMintModule.sol:169
- **Why:** A warden reading the NatSpec would look for milestone-related logic in the function and, not finding any, might suspect that milestone tracking was removed without updating the documentation -- potentially filing a finding about missing functionality. The NatSpec inflates the perceived scope of `recordMintData` beyond its actual behavior.
- **Suggestion:** Change line 169 to: `- \`mintPacked_[player]\` updated with level count, level units, and whale bundle frozen state`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### CMT-015: Inline comment claims "+10pp" affiliate bonus but code applies variable multiplier

- **What:** Line 942 states `// Final jackpot day affiliate bonus: +10pp on fresh ETH`. The code at lines 947-949 applies two different multipliers depending on the target level: `(freshBurnie * 7) / 5` for `targetLevel <= 3` (a 40% increase) and `(freshBurnie * 3) / 2` for `targetLevel > 3` (a 50% increase). The "+10pp" (percentage points) characterization is incorrect for both cases -- the actual effect depends on the affiliate rate. For a 20% affiliate rate, the 50% multiplier yields +10pp (to 30%), but the 40% multiplier yields only +8pp (to 28%). For any other affiliate rate, neither multiplier produces a +10pp increase.
- **Where:** DegenerusGameMintModule.sol:942
- **Why:** A warden analyzing the affiliate reward system would use "+10pp" as a fixed quantity when calculating effective affiliate rates during jackpot resolution. The actual bonus is a 40% or 50% multiplier on the BURNIE base (not a fixed percentage point increase), and the two tiers are not distinguished in the comment. A warden would miss both the tier split at targetLevel 3 and the rate-dependent nature of the actual percentage point increase.
- **Suggestion:** Change to: `// Final jackpot day affiliate bonus: 50% boost on fresh ETH BURNIE base (40% for levels 0-3)`
- **Category:** comment-inaccuracy
- **Severity:** INFO
