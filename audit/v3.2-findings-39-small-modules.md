| DRIFT-003 | _sendToVault hard-revert risk absent from handleGameOverDrain/handleFinalSweep NatSpec | **NOT FIXED** | No changes applied to this file. ### New Findings

**CMT-V32-001: DRIFT-003 re-report -- _sendToVault hard-revert consequence undocumented in NatSpec**

- **What:** `_sendToVault` (line 194) hard-reverts via `if (!steth.transfer(...)) revert E()` and `if (!ok) revert E()`. When called from `handleGameOverDrain` (line 157) or `handleFinalSweep` (line 187), a revert rolls back all state changes including terminal flags (`gameOver`, `gameOverFinalJackpotPaid`, `finalSwept`). The `@custom:reverts` tags document that the functions CAN revert, but not the CONSEQUENCE: if stETH (Lido) is paused or reverts, game-over drain and final sweep are blocked because no state transition persists past the revert. `handleGameOverDrain` can be retried but will hit the same revert. This contrasts with AdvanceModule's `_autoStakeExcessEth` which uses try/catch to make stETH non-blocking.
- **Where:** DegenerusGameGameOverModule.sol:67, 169, 190-193
- **Why:** A warden analyzing the game-over path would see `@custom:reverts E When stETH transfer fails` and understand the function can revert, but would not recognize the systemic consequence without tracing the entire state machine. A warden familiar with the Lido pause risk would flag this as a centralization/external-dependency finding. The NatSpec provides no hint that this risk exists or was considered.
- **Suggestion:** Add `@dev` to `_sendToVault`: "WARNING: Hard reverts on stETH failure. If Lido is paused, handleGameOverDrain and handleFinalSweep are blocked until stETH recovers. No try/catch fallback -- by design, funds should not be distributed if stETH is non-functional."
- **Category:** intent-drift
- **Severity:** LOW
