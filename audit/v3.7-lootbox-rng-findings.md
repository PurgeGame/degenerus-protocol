## Lootbox RNG Index Mutation Audit (LBOX-01)

### Objective

Map every code path that mutates `lootboxRngIndex` and verify: increment on fresh daily/mid-day requests, no increment on timeout retry, no increment on coordinator swap.

### Mutation Site Table

| Site | Function | Line | When | Increments? | Status |
|------|----------|------|------|-------------|--------|
| 1 | `_finalizeRngRequest` | 1277 | Fresh daily/gameover request (`isRetry == false`) | YES | CORRECT |
| 2 | `requestLootboxRng` | 738 | Mid-day standalone request (guards prevent double-call) | YES | CORRECT |
| 3 | `_finalizeRngRequest` | 1275-1280 | Timeout retry after 12h (`isRetry == true`) | NO | CORRECT |
| 4 | `updateVrfCoordinatorAndSub` | 1338-1367 | Coordinator swap (VRF state reset, game state preserved) | NO | CORRECT |

### Analysis

**Site 1 -- Fresh daily request (AdvanceModule line 1277):**
`_finalizeRngRequest` computes `isRetry = vrfRequestId != 0 && rngRequestTime != 0 && rngWordCurrent == 0`. On a fresh request (first of the day), `vrfRequestId == 0` (cleared by previous fulfillment), so `isRetry` is false. The `lootboxRngIndex++` fires, advancing the index for new purchases. `lootboxRngPendingEth` and `lootboxRngPendingBurnie` are zeroed. This is the primary daily increment path.

**Site 2 -- Mid-day request (AdvanceModule line 738):**
`requestLootboxRng` unconditionally increments `lootboxRngIndex` after the VRF request. Guards (`rngLockedFlag == false`, `rngRequestTime == 0`, daily RNG already recorded, not in 15-min pre-reset window, LINK balance sufficient, pending value above threshold) prevent double-call. The increment separates "purchases awaiting this word" from "new purchases targeting the next word."

**Site 3 -- Retry (no increment):**
When `isRetry == true`, the `lootboxRngIndex++` line at 1277 is inside the `if (!isRetry)` block and is skipped. The retry reuses the same VRF slot (`lootboxRngIndex - 1`). This is correct: the original fresh request already advanced the index, and the retry fills the same slot.

**Site 4 -- Coordinator swap (no increment):**
`updateVrfCoordinatorAndSub` clears `vrfRequestId`, `rngRequestTime`, `rngWordCurrent`, and `rngLockedFlag`, but does NOT touch `lootboxRngIndex`. Any orphaned index (word never delivered) is handled by `_backfillOrphanedLootboxIndices` on the next successful VRF fulfillment.

### Test Evidence

- `test_indexIncrementsOnFreshDaily` -- Verifies index increments by 1 on daily advanceGame
- `test_indexIncrementsOnMidDay` -- Verifies index increments by 1 on mid-day requestLootboxRng
- `test_indexNoIncrementOnRetry(uint256)` -- Fuzz: 12h timeout retry does not increment index (1000 runs)
- `test_indexNoIncrementOnCoordinatorSwap` -- Coordinator swap preserves index value
- `test_indexSequentialAcrossMultipleDays(uint8)` -- Fuzz: N consecutive days produce sequential indices (1000 runs)

**Verdict: LBOX-01 VERIFIED** -- All 4 mutation/non-mutation sites behave correctly. Index advances exactly once per fresh VRF request (daily or mid-day), never on retry or swap.

## Word-to-Index Write Site Audit (LBOX-02)

### Objective

Map every code path that writes to `lootboxRngWordByIndex[index]` and verify the index used is correct (matches the index reserved during the corresponding VRF request).

### Write Site Table

| Site | Function | Line | Path | Index Used | Zero Guard |
|------|----------|------|------|-----------|------------|
| 1 | `_finalizeLootboxRng` (via `rngGate` daily) | 848 | Daily VRF fulfillment -> advanceGame | `lootboxRngIndex - 1` | `rawFulfillRandomWords` line 1410 |
| 2 | `rawFulfillRandomWords` mid-day branch | 1418 | Mid-day VRF callback direct write | `lootboxRngIndex - 1` | line 1410 |
| 3 | `_finalizeLootboxRng` (via stale redirect) | 788, 848 | Stale cross-day word redirected to lootbox | `lootboxRngIndex - 1` | `rawFulfillRandomWords` line 1410 |
| 4 | `_backfillOrphanedLootboxIndices` | 1466 | Gap backfill after coordinator swap/stall | Scans backwards from `lootboxRngIndex - 1` | line 1465 |
| 5 | `_finalizeLootboxRng` (via `_gameOverEntropy`) | 888, 917, 848 | Gameover normal VRF / fallback path | `lootboxRngIndex - 1` | line 1410 / V37-003 |

### Analysis

**Site 1 -- Daily path:**
`rawFulfillRandomWords` receives the VRF word, applies `if (word == 0) word = 1`, stores to `rngWordCurrent`. On `advanceGame`, `rngGate` reads `rngWordCurrent`, calls `_applyDailyRng(day, currentWord)` (adds nudges), then `_finalizeLootboxRng(currentWord)`. The function writes to `lootboxRngWordByIndex[lootboxRngIndex - 1]`. Since `_finalizeRngRequest` incremented `lootboxRngIndex` at request time, `lootboxRngIndex - 1` points to the slot reserved for purchases made before the request. Correct.

**Site 2 -- Mid-day path:**
`rawFulfillRandomWords` with `rngLockedFlag == false` writes directly to `lootboxRngWordByIndex[lootboxRngIndex - 1]`. Since `requestLootboxRng` incremented the index at request time, `lootboxRngIndex - 1` points to the correct slot. Correct.

**Site 3 -- Stale redirect:**
When `requestDay < day` in `rngGate`, the stale word is used for lootboxes only via `_finalizeLootboxRng(currentWord)`. The index is the same `lootboxRngIndex - 1` -- the slot reserved by the (now stale) daily request. A fresh daily request is then made. Correct.

**Site 4 -- Backfill:**
`_backfillOrphanedLootboxIndices(vrfWord)` scans backwards from `lootboxRngIndex - 1`, filling empty slots with `keccak256(abi.encodePacked(vrfWord, i))`. Each index `i` gets a deterministic but VRF-derived fallback word. Zero guard: `if (fallbackWord == 0) fallbackWord = 1`. Correct.

**Site 5 -- Gameover:**
Both the normal VRF path (line 888) and the fallback path (line 917) call `_finalizeLootboxRng`, which writes to `lootboxRngIndex - 1`. The gameover fallback uses `_getHistoricalRngFallback` -> `_applyDailyRng` -> `_finalizeLootboxRng`. See V37-003 for the missing zero guard on `_getHistoricalRngFallback`.

**Idempotency guard:** `_finalizeLootboxRng` checks `if (lootboxRngWordByIndex[index] != 0) return` before writing. This prevents double-write if multiple paths attempt to fill the same index.

### Test Evidence

- `test_wordWriteDaily(uint256)` -- Fuzz: daily fulfillment writes correct word at `index - 1` (1000 runs)
- `test_wordWriteMidDay(uint256)` -- Fuzz: mid-day fulfillment writes correct word at `index - 1` (1000 runs)
- `test_wordWriteStaleRedirect(uint256)` -- Fuzz: stale cross-day word correctly stored via `_finalizeLootboxRng` (1000 runs)
- `test_wordWriteBackfill` -- Backfill after coordinator swap fills orphaned indices with keccak256-derived words
- `test_wordWriteIdempotent(uint256)` -- Fuzz: second fulfillment does not overwrite existing word (1000 runs)

**Verdict: LBOX-02 VERIFIED** -- All 5 write sites store the correct word at the correct index. The `lootboxRngIndex - 1` pattern is consistent across all paths. Idempotency guard prevents double-write.

## Zero-State Guard Inventory (LBOX-03)

### Objective

Verify that every VRF word injection point into lootbox storage has a `if (word == 0) word = 1` guard. EntropyLib's xorshift PRNG has a fixed point at 0 -- if `entropyStep(0)` is called, all outputs are 0, producing deterministic and exploitable lootbox outcomes.

### Guard Inventory

| Site | Location | Guard | Status |
|------|----------|-------|--------|
| `rawFulfillRandomWords` | AdvanceModule line 1410 | `if (word == 0) word = 1` | GUARDED |
| `_backfillGapDays` | AdvanceModule line 1444 | `if (derivedWord == 0) derivedWord = 1` | GUARDED |
| `_backfillOrphanedLootboxIndices` | AdvanceModule line 1465 | `if (fallbackWord == 0) fallbackWord = 1` | GUARDED |
| `_getHistoricalRngFallback` | AdvanceModule line 962 | **NONE** | UNGUARDED (V37-003) |
| `_applyDailyRng` | AdvanceModule lines 1478-1488 | Adds nudges to input; no explicit zero guard | DEPENDS ON INPUT |

### Analysis

**`rawFulfillRandomWords` (line 1410):** The primary VRF entry point. All words arriving from Chainlink pass through `if (word == 0) word = 1` before any storage write. This guard protects Write Sites 1, 2, and 3 (daily, mid-day, stale redirect).

**`_backfillGapDays` (line 1444):** Gap day backfill computes `keccak256(abi.encodePacked(vrfWord, gapDay))` and guards: `if (derivedWord == 0) derivedWord = 1`. Protected.

**`_backfillOrphanedLootboxIndices` (line 1465):** Orphaned lootbox backfill computes `keccak256(abi.encodePacked(vrfWord, i))` and guards: `if (fallbackWord == 0) fallbackWord = 1`. Protected.

**`_getHistoricalRngFallback` (lines 944-962):** Returns `keccak256(abi.encodePacked(combined, currentDay, block.prevrandao))` with NO explicit zero guard. The result flows to `_applyDailyRng` (which adds nudges but does not guard zero if nudges are 0), then to `_finalizeLootboxRng`. If the keccak256 output were 0 (probability 2^-256) with zero nudges, the stored word would be 0, causing permanent `RngNotReady` for that lootbox index. See **V37-003**.

**`_applyDailyRng` (lines 1478-1488):** Not a VRF injection point itself -- it receives an already-guarded word from `rawFulfillRandomWords` on the daily path. On the gameover fallback path, it receives the unguarded `_getHistoricalRngFallback` output. The function adds nudges (`rawWord + nudges`), but if nudges happen to be 0, a zero input passes through unchanged.

**`openLootBox` (LootboxModule line 550):** `if (rngWord == 0) revert RngNotReady()` -- this is the consumption guard, not a write-time guard. It prevents entropy derivation from a zero word but does not fix the stored state. A zero word at this point means permanent revert for that index.

### Test Evidence

- `test_zeroGuardRawFulfill` -- Confirms `rawFulfillRandomWords(word=0)` stores 1 (not 0) at lootbox index
- `test_zeroGuardBackfill` -- Confirms `_backfillOrphanedLootboxIndices` produces nonzero fallback words
- `test_zeroGuardMidDay` -- Confirms mid-day `rawFulfillRandomWords(word=0)` stores 1

**Verdict: LBOX-03 VERIFIED** -- 4 of 5 VRF word injection points are explicitly guarded. The single unguarded path (`_getHistoricalRngFallback`) has negligible probability of triggering (2^-256) and is documented as V37-003 (INFO).

## Entropy Derivation Analysis (LBOX-04)

### Objective

Prove that lootbox opening entropy is unique per purchase: no two distinct `(player, day, amount)` tuples can produce the same entropy, given a shared `rngWord`.

### ETH Lootbox Entropy (LootboxModule line 570)

```solidity
uint256 entropy = uint256(keccak256(abi.encode(rngWord, player, day, amount)));
```

### BURNIE Lootbox Entropy (LootboxModule line 644)

```solidity
uint256 entropy = uint256(keccak256(abi.encode(rngWord, player, day, amountEth)));
```

### Uniqueness Proof

For `keccak256(abi.encode(rngWord, player, day, amount))` to produce a collision, two distinct input tuples must hash to the same 256-bit output. Given that `abi.encode` produces fixed-width 32-byte fields with no ambiguity, distinct tuples produce distinct preimages. The collision resistance of keccak256 (2^128 birthday bound) makes this infeasible.

**Same player, same day, different amounts:** `amount` differs -> distinct preimage -> distinct entropy.

**Same player, different days:** `day` differs -> distinct preimage -> distinct entropy. Note: the cross-day guard at MintModule line 700 (`if (storedDay != day) revert E()`) prevents accumulation across day boundaries for the same index, but different indices on different days produce different `rngWord` values as well.

**Different players, same day:** `player` differs -> distinct preimage -> distinct entropy.

**Same player, same day, same amount (theoretical):** This would require the same accumulated `lootboxEth` for the same player at the same index. Since `lootboxEth[index][player]` accumulates (MintModule line 715: `newAmount = existingAmount + boostedAmount`), the `amount` is the total for that player at that index. Multiple purchases within the same day produce a single increasing `amount`. Same-player same-day same-amount would require zero additional purchase, meaning only one opening -- no collision.

### Degenerette Entropy (DegeneretteModule lines 597-618)

```solidity
uint256 resultSeed = spinIdx == 0
    ? uint256(keccak256(abi.encodePacked(rngWord, index, QUICK_PLAY_SALT)))
    : uint256(keccak256(abi.encodePacked(rngWord, index, spinIdx, QUICK_PLAY_SALT)));
```

Different `spinIdx` values produce different preimages. `QUICK_PLAY_SALT` prevents cross-function collision with lootbox entropy (which uses `abi.encode` not `abi.encodePacked`).

### Redemption Lootbox Entropy (Game.sol line 1841)

```solidity
rngWord = uint256(keccak256(abi.encode(rngWord)));
```

Each 5 ETH chunk uses a chained keccak hash of the previous rngWord, ensuring distinct entropy per chunk. The chain is deterministic given the initial word, so the same redemption sequence always produces the same outcomes.

### Test Evidence

- `test_entropyUniqueDifferentPlayers(uint256)` -- Fuzz: 3 different players produce 3 distinct entropy values (1000 runs)
- `test_entropyUniqueDifferentAmounts(uint256)` -- Fuzz: same player, different purchase amounts produce distinct entropy (1000 runs)
- `test_entropyUniqueDifferentDays(uint256)` -- Fuzz: same player, different days produce distinct entropy (1000 runs)
- `test_entropyAccumulationSamePlayer` -- Same player's accumulated amount changes preimage on each additional purchase

**Verdict: LBOX-04 VERIFIED** -- Entropy is unique per `(rngWord, player, day, amount)` tuple. Amount accumulation prevents same-player same-day collision. Cross-function salt prevents lootbox/degenerette collision.

## Full Lifecycle Trace (LBOX-05)

### Objective

Trace the complete lootbox RNG lifecycle from purchase to prize determination, verifying that each step correctly links to the next.

### Step 1: Purchase

Player calls `purchaseCoin` / `purchaseWhaleBundle` / `purchaseBurnieLootbox`, which delegatecalls to MintModule.

- **MintModule line 686:** `uint48 index = lootboxRngIndex` -- reads current index
- **MintModule line 689:** `lootboxEth[index][buyer]` -- checks/accumulates purchase amount
- **MintModule line 694:** `lootboxDay[index][buyer] = day` -- records purchase day (first purchase only)
- **MintModule line 695:** `lootboxBaseLevelPacked[index][buyer]` -- records base level
- **MintModule line 696-697:** `lootboxEvScorePacked[index][buyer]` -- records activity score
- **MintModule line 715-716:** `lootboxEth[index][buyer] = (uint256(purchaseLevel) << 232) | newAmount` -- accumulates
- **MintModule line 700:** `if (storedDay != day) revert E()` -- cross-day guard prevents accumulation across day boundaries

### Step 2: VRF Request

Either daily (`advanceGame` -> `rngGate` -> `_requestRng` -> `_finalizeRngRequest`) or mid-day (`requestLootboxRng`):

- `lootboxRngIndex++` (fresh request only, not retry)
- `lootboxRngPendingEth = 0; lootboxRngPendingBurnie = 0`
- New purchases now target `lootboxRngIndex` (the incremented value)
- Old purchases at `lootboxRngIndex - 1` await VRF word

### Step 3: VRF Fulfillment

Chainlink calls `rawFulfillRandomWords` (AdvanceModule line 1402):

- Validates `requestId == vrfRequestId && rngWordCurrent == 0` (line 1407)
- Zero guard: `if (word == 0) word = 1` (line 1410)
- **Daily path** (`rngLockedFlag == true`): `rngWordCurrent = word` (line 1414) -- stored for `advanceGame`
- **Mid-day path** (`rngLockedFlag == false`): `lootboxRngWordByIndex[lootboxRngIndex - 1] = word` (line 1418) -- direct write

### Step 4: Daily Processing (daily path only)

`advanceGame` -> `rngGate`:

- `_applyDailyRng(day, currentWord)` -- adds nudges, stores to `rngWordByDay[day]`
- `_finalizeLootboxRng(currentWord)` (line 826/848) -- stores to `lootboxRngWordByIndex[lootboxRngIndex - 1]`
- Sets `lastLootboxRngWord = rngWord` (line 849) for ticket processing entropy

### Step 5: RngNotReady Guard (opening)

Player calls `openLootBox(player, lootboxIndex)`:

- **LootboxModule line 549:** `uint256 rngWord = lootboxRngWordByIndex[index]` -- loads word for this index
- **LootboxModule line 550:** `if (rngWord == 0) revert RngNotReady()` -- prevents premature opening before VRF fulfillment

### Step 6: Prize Determination

- **LootboxModule line 570:** `uint256 entropy = uint256(keccak256(abi.encode(rngWord, player, day, amount)))` -- derives per-player entropy
- **LootboxModule line 571:** `_rollTargetLevel(baseLevel, entropy)` -- determines target level
- **LootboxModule line 599:** `_resolveLootboxCommon(...)` -- resolves lootbox outcome using `EntropyLib.entropyStep` chain:
  - 55% chance: ticket roll (count with variance via entropy chain)
  - 10% chance: DGNRS reward (amount via entropy)
  - 10% chance: WWXRP reward (fixed 1 token)
  - 25% chance: large BURNIE (variance via entropy)
  - Boon draw: additional entropy chain if allowed

### Lifecycle Integrity

The purchase at index `N` -> VRF request increments to `N+1` -> VRF word writes to slot `N+1 - 1 = N` -> `openLootBox(player, N)` reads word at slot `N` -> entropy derived from that word. The chain is unbroken. The `RngNotReady` guard prevents any opening before the word is available.

### Test Evidence

- `test_fullLifecycleDailyPath` -- Complete daily purchase -> VRF -> advanceGame -> open lifecycle
- `test_fullLifecycleMidDayPath` -- Complete mid-day purchase -> requestLootboxRng -> VRF callback -> open lifecycle
- `test_fullLifecycleRngNotReady` -- Confirms `openLootBox` reverts with `RngNotReady` before VRF fulfillment
- `test_fullLifecycleMultipleIndices` -- Multiple indices with correct respective words across consecutive days

**Verdict: LBOX-05 VERIFIED** -- The 6-step lifecycle is traced end-to-end. The index-to-word linkage is correct. The `RngNotReady` guard prevents premature opening. Prize determination uses per-player unique entropy.

## Cross-Cutting Observations

1. **Idempotency of `_finalizeLootboxRng` is safe against double-call.** The function checks `if (lootboxRngWordByIndex[index] != 0) return` before writing. This means multiple code paths can call `_finalizeLootboxRng` for the same index (e.g., stale redirect followed by daily processing) without corrupting the stored word. The first write wins.

2. **DegeneretteModule piggybacks on lootbox RNG.** Degenerette bets record `lootboxRngIndex` at bet placement time (DegeneretteModule line 473) and resolve using `lootboxRngWordByIndex[index]` (line 597). The same index-to-word guarantees verified in LBOX-02 apply to degenerette bet resolution. The `QUICK_PLAY_SALT` prevents entropy collision between lootbox and degenerette outcomes.

3. **Coordinator swap + backfill is the most complex path but correctly handles orphaned indices.** After a coordinator swap, `_backfillOrphanedLootboxIndices` scans backwards from `lootboxRngIndex - 1` and fills empty slots with keccak256-derived words. The zero guard at line 1465 ensures all backfilled words are nonzero. This path was verified in `test_wordWriteBackfill` and `test_zeroGuardBackfill`.

4. **Amount accumulation in `lootboxEth` prevents same-player same-day entropy collision.** Because `lootboxEth[index][player]` accumulates on each purchase (MintModule line 715: `newAmount = existingAmount + boostedAmount`), the entropy input `amount` increases monotonically for a player within a single index. This makes the keccak256 preimage `(rngWord, player, day, amount)` unique even across multiple purchases.

5. **The `RngNotReady` guard is both a safety mechanism and a UX signal.** `openLootBox` reverts with `RngNotReady` when `lootboxRngWordByIndex[index] == 0`. This prevents entropy derivation from a zero word (which would produce all-zero xorshift outputs) and signals to the frontend that the VRF fulfillment has not yet arrived. The guard is the last line of defense for the zero-state invariant.
