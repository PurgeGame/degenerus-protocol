## Cross-System Interaction Summary

### Interaction 1: GAMEOVER + Active Coinflip Claims

**Question:** What happens to pending coinflip claims when GAMEOVER fires?

**Resolution:** Coinflip economics operate entirely in BURNIE (burn-and-mint). GAMEOVER affects ETH claimablePool, not BURNIE supply. When GAMEOVER fires:
- `rngLockedFlag` blocks new claim attempts during VRF resolution
- After GAMEOVER, coinflip winnings that are still pending (virtual stakes in `claimableStored`) remain accessible via `claimCoinflips()` -- there is no `if (gameOver) revert` guard in BurnieCoinflip claim paths
- Physical BURNIE for coinflip wins is minted from BurnieCoin's supply (not from claimablePool)
- The 30-day GAMEOVER final sweep applies only to ETH `claimablePool` and ETH `claimableWinnings` -- it does not touch BURNIE virtual stakes
- **Verdict:** GAMEOVER does not disrupt pending coinflip claims. The two economies (ETH pool vs BURNIE mint) are cleanly isolated per Phase 27 PAY-07 finding.

### Interaction 2: sDGNRS Burn + Pending Decimator Claims

**Question:** What happens when sDGNRS holders burn tokens while decimator claims are pending?

**Resolution:** sDGNRS burn (`StakedDegenerusStonk.burn()`) calls `game.claimWinnings()` which draws from `claimablePool` and debits `claimableWinnings[sDGNRS]`. The sDGNRS contract does not hold individual decimator entitlements -- decimator claims go to `claimableWinnings[player]` (the individual who burned BURNIE). sDGNRS burn does not interfere with individual player decimator claims. The `prizePoolFrozen` guard (set during jackpot resolution) prevents concurrent pool mutations. INV-01 proves claimablePool solvency at all 15 mutation sites including both sDGNRS-related sites (N8: claimWinnings outflow, G6: claimWinnings outflow). **Verdict:** Clean isolation. sDGNRS burn and decimator claims target different `claimableWinnings` entries.

### Interaction 3: advanceGame + Concurrent claimWinnings

**Question:** What prevents a player from calling claimWinnings while advanceGame is mid-execution?

**Resolution:** `claimWinnings` does not have an `rngLocked` guard. However, CEI ordering in both functions prevents cross-contamination:
- `advanceGame` modifies `claimablePool` and `claimableWinnings[player]` in atomic SSTORE operations before any external call
- `claimWinnings` reads `claimableWinnings[player]`, sets it to 1 (sentinel), decrements `claimablePool`, then transfers ETH
- Since Solidity has no concurrent execution, these two call paths cannot interleave within the same transaction
- The `rngLockedFlag` prevents coinflip claim operations during VRF resolution but not ETH claims -- this is safe because `advanceGame` completes all pool accounting writes before returning
- **Verdict:** No reentrancy or concurrent execution path exists. Solidity's serial execution model prevents the interaction. CEI is correct in both paths.

### Interaction 4: Pool Transition + Distribution Event in Same Block

**Question:** Can a pool transition (nextPrizePool -> currentPrizePool) and a BAF scatter distribution happen in the same block?

**Resolution:** `advanceGame` is the sole entry point for both pool transitions and BAF distributions. Within a single `advanceGame` call, the batch-and-resume pattern ensures only one type of work is done per call (`ticketsFullyProcessed` gate, `_processPhaseTransition`, `_prepareFutureTickets`). The pool transition and the scatter event cannot both fire in the same `advanceGame` invocation. They happen in sequential `advanceGame` calls across different blocks. Pool transition at level end -> scatter fires at the start of the new level's advance. `baseFuturePool` (snapshot) is committed atomically at level transition before any scatter calculation. **Verdict:** No same-block collision is possible by protocol design. The batch state machine prevents concurrent actions within a single advance tick.

### FINDING-LOW-EDGE03-01: advanceGame Queue Inflation DOS

**Severity:** Low
**Requirement:** EDGE-03
**Files:** DegenerusGameAdvanceModule.sol:156-400, DegenerusGameJackpotModule.sol (processTicketBatch)
**Description:** An attacker can purchase large numbers of tickets at any level, inflating the ticket queue and requiring many sequential `advanceGame` calls before the day resolves. No single call can exceed the block gas limit (batch mechanism bounds per-call work), but sustained high-volume ticket purchasing delays daily jackpot resolution by hours or days.
**Recommendation:** Document as accepted design tradeoff. The advance bounty escalation (1x -> 2x after 1h -> 3x after 2h) economically incentivizes external callers to clear the queue. No code change required unless adversarial griefing is observed in practice. Consider a maximum queue depth per day or higher per-address ticket rate limit as a future mitigation.
