## GOV-02: propose() Access Control

**Verdict:** PASS
**Severity:** N/A

### Evidence

**Code:** `DegenerusAdmin.sol` lines 394-431, function `propose(address newCoordinator, bytes32 newKeyHash)`

**Constants:**
- `ADMIN_STALL_THRESHOLD = 20 hours` (72000 seconds) -- line 297
- `COMMUNITY_STALL_THRESHOLD = 7 days` (168 hours = 604800 seconds) -- line 300
- `COMMUNITY_PROPOSE_BPS = 50` (0.5%) -- line 303
- `BPS = 10000` -- line 309

**Tests:**
- `test/unit/VRFGovernance.test.js` "propose" describe block: 8 tests covering admin path success at 21h stall, NotStalled revert at 19h stall, community NotStalled revert before 7d, community InsufficientStake revert without sDGNRS, ZeroAddress reverts for zero coordinator/keyHash, proposalCount increment, circulatingSnapshot storage
- `test/unit/GovernanceGating.test.js` ADMIN-01 describe block: 6 tests verifying DGVE-based ownership including deployer (100% DGVE) passes, alice (0% DGVE) fails, deployer fails after transferring >49.9%, alice passes after receiving >50.1%, CREATOR (0% DGVE) fails, multiple owner-gated functions all check DGVE majority
- `test/unit/GovernanceGating.test.js` "isVaultOwner boundary: exactly 50.1% fails, 50.1% + 1 passes" -- precise boundary test

### Analysis

#### Admin Path (lines 407-409)

**Step 1: `vault.isVaultOwner(msg.sender)` verification**

The `isVaultOwner()` function in DegenerusVault.sol (line 425) delegates to `_isVaultOwner()` (line 416):

```solidity
function _isVaultOwner(address account) private view returns (bool) {
    uint256 supply = ethShare.totalSupply();
    uint256 balance = ethShare.balanceOf(account);
    return balance * 1000 > supply * 501;
}
```

This checks `balance * 1000 > supply * 501`, which means `balance / supply > 501 / 1000 = 50.1%`. The threshold is STRICTLY greater than 50.1% -- exactly 50.1% does NOT pass. This is confirmed by GovernanceGating.test.js "isVaultOwner boundary" test.

Overflow analysis: `balance * 1000` where max balance = totalSupply = 1e30 (1 trillion tokens * 1e18 decimals). Max product: 1e33. Well within uint256 (1.15e77). Similarly `supply * 501` max: ~5e32. No overflow.

**Step 2: `ADMIN_STALL_THRESHOLD` = 20 hours (72000 seconds)**

Confirmed at line 297: `uint256 private constant ADMIN_STALL_THRESHOLD = 20 hours;`. Solidity compiles `20 hours` to `72000` (20 * 3600 = 72000 seconds).

**Step 3: Stall check `stall < ADMIN_STALL_THRESHOLD` (line 408)**

```solidity
uint48 lastVrf = gameAdmin.lastVrfProcessed();
uint256 stall = block.timestamp - uint256(lastVrf);
```

`lastVrf` is `uint48` upcast to `uint256`. `block.timestamp` is always >= `lastVrf` (timestamp is set at block creation, `lastVrfProcessedTimestamp` is set to `uint48(block.timestamp)` in previous blocks). No underflow possible. The subtraction yields the stall duration in seconds.

If `stall < 72000` (less than 20 hours), the function reverts with `NotStalled()`. Only when `stall >= 72000` does the admin path proceed. Confirmed by VRFGovernance.test.js "reverts with NotStalled if VRF stall < 20h for admin" (19h stall).

#### Community Path (lines 410-416)

**Step 1: `COMMUNITY_STALL_THRESHOLD` = 7 days**

Confirmed at line 300: `uint256 private constant COMMUNITY_STALL_THRESHOLD = 7 days;`. Solidity compiles `7 days` to `604800` (7 * 86400 = 604800 seconds).

The community path runs in the `else` branch (line 410) -- i.e., only if `vault.isVaultOwner(msg.sender)` is FALSE. Non-DGVE-holders are routed here. Confirmed by VRFGovernance.test.js "community path: reverts with NotStalled before 7d" (21h stall still reverts for non-DGVE holder).

**Step 2: Stake check mathematical derivation**

```solidity
uint256 circ = circulatingSupply();
if (circ == 0 || sDGNRS.balanceOf(msg.sender) * BPS < circ * COMMUNITY_PROPOSE_BPS)
    revert InsufficientStake();
```

Substituting constants: `balance * 10000 < circ * 50` causes a revert. The propose succeeds when `balance * 10000 >= circ * 50`, i.e., `balance >= circ * 50 / 10000 = circ * 0.005`. This means balance must be >= 0.5% of circulating supply.

Boundary example: if `circ = 10000`:
- Need `balance * 10000 >= 10000 * 50 = 500000`
- `balance >= 50` (0.5% of 10000)
- At `balance = 49`: `49 * 10000 = 490000 < 500000` -- reverts. Correct.
- At `balance = 50`: `50 * 10000 = 500000 >= 500000` -- passes. Correct.

**Step 3: `circ == 0` guard**

If circulating supply is zero, the function reverts immediately. This prevents meaningless governance (nobody holds sDGNRS) and prevents division-by-zero-equivalent scenarios in the stake check. Correct.

#### General Checks

**Zero address validation (lines 400-401):** Both `newCoordinator == address(0)` and `newKeyHash == bytes32(0)` cause revert with `ZeroAddress()`. Confirmed by VRFGovernance.test.js "reverts with ZeroAddress for zero coordinator" and "reverts with ZeroAddress for zero keyHash".

**NotWired check (line 398):** `subscriptionId == 0` reverts with `NotWired()`. Prevents proposals before VRF is configured. Since the constructor atomically creates a subscription and sets `subscriptionId`, this check would only trigger if the subscription was previously shutdown (via `shutdownVrf()`). Correct guard.

**GameOver check (line 399):** `gameAdmin.gameOver()` reverts with `GameOver()`. Prevents proposals after the game ends. This is an external call to the Game contract's view function. No state mutation risk.

**Proposal ID and storage (lines 418-428):** `proposalId = ++proposalCount` (pre-increment, so IDs start at 1). Proposal struct fields are set correctly. `p.state` defaults to `ProposalState.Active` (enum value 0, which is the default for uninitialized storage). `unchecked { activeProposalCount++; }` -- uint8 overflow analyzed in VOTE-03.

#### `circulatingSupply()` Double-Call Analysis (lines 412 and 424)

`circulatingSupply()` is called TWICE in `propose()`:
1. **Line 412:** For the community stake check (only in the community path)
2. **Line 424:** For the proposal's `circulatingSnapshot`

Between these calls, the following operations occur (lines 416-423):
- Line 418: `proposalId = ++proposalCount` (internal storage write)
- Lines 419-425: Proposal struct field writes (internal storage writes)

**No external calls occur between lines 412 and 424.** The `circulatingSupply()` function reads `sDGNRS.totalSupply()`, `sDGNRS.balanceOf(SDGNRS)`, and `sDGNRS.balanceOf(DGNRS)` -- all view calls. Since there is no opportunity for sDGNRS state to change between the two calls within the same transaction, both calls return identical values. No state inconsistency possible.

Note: For the admin path, `circulatingSupply()` is called only once (line 424) because the admin path does not need the stake check. No double-call concern on the admin path.

### Adversarial Check

**Attack 1: Can an admin propose with less than 50.1% DGVE?**
- The `vault.isVaultOwner(msg.sender)` check calls `_isVaultOwner()` which requires `balance * 1000 > supply * 501`. With supply of 1 trillion tokens (1e30 wei), an attacker with exactly 50.1% (5.01e29 wei) would compute `5.01e29 * 1000 = 5.01e32`. The threshold is `1e30 * 501 = 5.01e32`. Since `5.01e32 > 5.01e32` is FALSE, exactly 50.1% FAILS. The attacker needs at least 50.1% + 1 wei. This is confirmed by GovernanceGating.test.js boundary test.
- An attacker with 50.0% DGVE: `5e29 * 1000 = 5e32` vs `1e30 * 501 = 5.01e32`. `5e32 > 5.01e32` is FALSE. Correctly reverts.

**Attack 2: Can an admin propose when stall < 20h?**
- `stall < ADMIN_STALL_THRESHOLD` at line 408 reverts with `NotStalled()`. At exactly 19h59m59s (71999 seconds): `71999 < 72000` is TRUE. Reverts. Correct.
- At exactly 20h (72000 seconds): `72000 < 72000` is FALSE. Propose succeeds. Correct -- the threshold is "at least 20 hours."
- Confirmed by VRFGovernance.test.js "reverts with NotStalled if VRF stall < 20h for admin" (19h stall).

**Attack 3: Can someone with 0.49% sDGNRS propose via community path?**
- If `circ = 100000` and `balance = 490` (0.49%): `490 * 10000 = 4900000`. `100000 * 50 = 5000000`. `4900000 < 5000000` is TRUE. Reverts with `InsufficientStake`. Correct.
- At exactly 0.5% (`balance = 500`): `500 * 10000 = 5000000 >= 5000000`. Passes. Correct.

**Attack 4: Community propose with zero circulating supply**
- `circ == 0` at line 413 reverts. Even if an attacker somehow has sDGNRS (which would be impossible with zero circulating supply), the guard prevents proposals. Correct.

**Attack 5: Stall time manipulation via `lastVrfProcessed` spoofing**
- `lastVrf` is read via `gameAdmin.lastVrfProcessed()` which returns the `lastVrfProcessedTimestamp` storage variable. This variable is only written by `_applyDailyRng()` (when VRF word is successfully processed) and `wireVrf()` (admin-only, one-time setup). An attacker cannot spoof the timestamp without controlling the VRF fulfillment flow or being the Admin contract's constructor. Not exploitable.

**Attack 6: DGVE holder proposes, then transfers DGVE before vote resolves**
- The `vault.isVaultOwner` check is only enforced at `propose()` time. After proposal creation, the proposer could transfer away their DGVE. The proposal remains Active with its snapshot. This is by design -- ownership at proposal time determines the path, not ongoing ownership. The proposal's legitimacy is decided by voters (sDGNRS holders), not by the proposer maintaining DGVE. Not a vulnerability.

## GOV-03: vote() Arithmetic

**Verdict:** PASS (conditional on VOTE-01 -- sDGNRS frozen invariant)
**Severity:** N/A (governance correctness depends on VOTE-01 supply-frozen invariant)

### Evidence

**Code:** `DegenerusAdmin.sol` lines 438-505, function `vote(uint256 proposalId, bool approve)`

**Key state variables:**
- `votes[proposalId][msg.sender]` -- enum `Vote { None, Approve, Reject }` (per-voter vote direction)
- `voteWeight[proposalId][msg.sender]` -- `uint256` (per-voter recorded weight at time of vote)
- `p.approveWeight` -- `uint256` (aggregate approve weight for proposal)
- `p.rejectWeight` -- `uint256` (aggregate reject weight for proposal)

**Tests:**
- `test/unit/VRFGovernance.test.js` "vote" describe block: 3 tests covering NotStalled revert on VRF recovery, ProposalNotActive revert for non-existent proposal, InsufficientStake revert for 0 sDGNRS voter
- `test/unit/VRFGovernance.test.js` "proposal expiry" describe block: 1 test verifying expired proposal triggers ProposalExpired revert
- `test/unit/VRFGovernance.test.js` "VRF recovery invalidation" describe block: 1 test (stall re-check)

### Analysis

#### Vote Change State Machine Trace

**Initial state:** `votes[id][voter] = None`, `voteWeight[id][voter] = 0`, `p.approveWeight = 0`, `p.rejectWeight = 0`

**Sequence 1: First vote (Approve, balance=100)**
1. `weight = sDGNRS.balanceOf(voter)` = 100 (line 457)
2. `currentVote = votes[id][voter]` = `Vote.None` (line 461)
3. `currentVote != Vote.None` is FALSE -- skip subtraction block (lines 462-468)
4. `newVote = Vote.Approve` (line 472)
5. `votes[id][voter] = Approve` (line 473)
6. `voteWeight[id][voter] = 100` (line 474)
7. `p.approveWeight += 100` (line 477)
8. **Final state:** `approveWeight=100, rejectWeight=0, votes=Approve, voteWeight=100`

**Sequence 2: Change to Reject (balance=100, unchanged)**
1. `weight = sDGNRS.balanceOf(voter)` = 100
2. `currentVote = votes[id][voter]` = `Vote.Approve`
3. `currentVote != Vote.None` is TRUE -- enter subtraction block
4. `oldWeight = voteWeight[id][voter]` = 100
5. `currentVote == Vote.Approve` is TRUE -- `p.approveWeight -= 100` (now 0)
6. `newVote = Vote.Reject` (line 472)
7. `votes[id][voter] = Reject` (line 473)
8. `voteWeight[id][voter] = 100` (line 474)
9. `p.rejectWeight += 100` (line 479)
10. **Final state:** `approveWeight=0, rejectWeight=100, votes=Reject, voteWeight=100`
11. **Correct:** Old weight fully subtracted before new weight added. No leakage.

**Sequence 3: Change back to Approve (balance=100)**
1. `weight = 100`, `currentVote = Vote.Reject`
2. `oldWeight = 100`, `p.rejectWeight -= 100` (now 0)
3. `votes = Approve`, `voteWeight = 100`, `p.approveWeight += 100`
4. **Final state:** `approveWeight=100, rejectWeight=0`
5. **Correct:** Full round-trip, weights properly tracked.

#### Multi-Voter Correctness

**Two voters, A (balance=100) and B (balance=200):**
1. A votes Approve: `approveWeight=100, rejectWeight=0`
2. B votes Reject: `approveWeight=100, rejectWeight=200`
3. A changes to Reject: `approveWeight -= 100` (now 0), `rejectWeight += 100` (now 300)
4. B changes to Approve: `rejectWeight -= 200` (now 100), `approveWeight += 200` (now 200)
5. **Final state:** `approveWeight=200, rejectWeight=100`
6. **Correct:** Each voter's old weight properly subtracted from their old direction, new weight added to new direction.

#### Write Path Exhaustiveness for `approveWeight` and `rejectWeight`

All write sites for `approveWeight` (confirmed via grep):
- Line 465: `p.approveWeight -= oldWeight` (subtraction during vote change from Approve)
- Line 477: `p.approveWeight += weight` (addition when voting Approve)
- Lines 489-490: read-only (execute condition comparison)

All write sites for `rejectWeight` (confirmed via grep):
- Line 467: `p.rejectWeight -= oldWeight` (subtraction during vote change from Reject)
- Line 479: `p.rejectWeight += weight` (addition when voting Reject)
- Lines 498-499: read-only (kill condition comparison)

**No other code path modifies `approveWeight` or `rejectWeight`.** These values are only modified inside `vote()`. The `_executeSwap()` and `_voidAllActive()` functions do NOT touch these values. This means the weight accounting is entirely contained within `vote()`.

#### VOTE-01 Dependency: sDGNRS Frozen Invariant

The `vote()` function uses **live sDGNRS balances** at line 457: `uint256 weight = sDGNRS.balanceOf(msg.sender)`. This means the weight recorded is the voter's balance at the moment of the `vote()` call, NOT a snapshotted value.

**If sDGNRS supply is NOT frozen during VRF stall, the following attack vector exists:**

1. Voter holds 100 sDGNRS, votes Approve on proposal (weight=100, approveWeight=100)
2. Voter transfers 100 sDGNRS to accomplice address B
3. B votes Approve (weight=100, approveWeight=200)
4. The same 100 sDGNRS has been counted TWICE

**Why this is blocked in practice:** The comment at line 456 states "Safe: VRF dead = supply frozen (no advances, unwrapTo blocked)." During VRF stall:
- `advanceGame()` cannot process (VRF fulfillment is broken), so no sDGNRS distribution occurs
- `unwrapTo()` in DegenerusStonk.sol (line 147-155) reverts when `block.timestamp - lastVrfProcessed() > 20 hours` (the stall guard)
- sDGNRS is soulbound (non-transferable by design -- StakedDegenerusStonk has no `transfer()` or `transferFrom()` function)

**The soulbound property of sDGNRS is the CRITICAL invariant.** Since sDGNRS cannot be transferred between addresses at any time (not just during stalls), the vote-splitting attack described above is impossible. The correctness of GOV-03 is CONDITIONAL on VOTE-01 confirming that sDGNRS cannot be transferred between addresses during governance.

**This dependency is explicitly documented and must be verified in VOTE-01.**

### Adversarial Check

**Attack 1: Weight leakage via balance change between votes**

If voter has 100 sDGNRS, votes Approve (weight=100). Suppose their balance later increases to 150 (e.g., via some distribution mechanism). Voter changes to Reject:
- `oldWeight = 100` (stored from previous vote), `p.approveWeight -= 100` (correct subtraction)
- `weight = 150` (live balance), `p.rejectWeight += 150`
- Net: approveWeight lost 100, rejectWeight gained 150. Delta of +50.

This is NOT a bug -- it is correct behavior. The voter's influence should reflect their current stake. The "extra" 50 came from a legitimate balance increase. As long as sDGNRS supply is frozen (VOTE-01), no new sDGNRS enters circulation during governance, so this scenario is impossible during a stall. If VOTE-01 holds, balance changes cannot occur, and oldWeight always equals current weight.

**Attack 2: Double-counting via same-direction re-vote**

Voter votes Approve (weight=100). Balance increases to 150. Voter votes Approve again:
- `currentVote = Approve, oldWeight = 100, p.approveWeight -= 100`
- `newVote = Approve, weight = 150, p.approveWeight += 150`
- Net: `approveWeight = original - 100 + 150 = original + 50`

Correct. The subtract-before-add pattern ensures no double-counting. The voter's recorded weight is updated to their current balance. There is exactly one `weight` entry per voter at all times.

**Attack 3: Underflow in `p.approveWeight -= oldWeight`**

Could `p.approveWeight < oldWeight` ever occur? This would cause a revert (Solidity 0.8.34 checked arithmetic -- the subtraction is NOT in an `unchecked` block).

For `p.approveWeight < oldWeight` to be true, the aggregate approve weight would need to be less than a single voter's recorded old weight. Since `approveWeight` is the sum of all current approve votes, and `oldWeight` is one voter's contribution from a previous `+= weight` operation on this same `approveWeight`, the only way `approveWeight < oldWeight` could happen is if:
1. Another code path decreased `approveWeight` without going through the vote change logic -- grep confirms no other write paths exist.
2. An overflow in a previous `approveWeight += weight` that wrapped the value to something smaller -- but `+=` on uint256 reverts on overflow in Solidity 0.8.34.

**Underflow is impossible under normal operation.** The invariant `approveWeight >= sum of all current approve voters' voteWeight` is maintained by the vote change logic.

**Attack 4: Zero weight vote after balance drops to 0**

Line 458: `if (weight == 0) revert InsufficientStake();` prevents zero-weight votes.

If a voter's sDGNRS balance drops to 0 after previously voting, they cannot call `vote()` again (zero balance revert). Their old vote weight remains counted in `approveWeight` or `rejectWeight`. This is correct: the voter contributed weight at the time of their vote, and losing their balance later does not retroactively remove their vote. Under VOTE-01 (frozen supply), balance cannot drop to 0 during governance anyway.

**Attack 5: First vote with `currentVote == Vote.None` skips subtraction**

When `currentVote == Vote.None` (first vote), the subtraction block (lines 462-468) is skipped entirely. The function proceeds directly to recording the new vote and adding weight. This is correct -- there is no old weight to subtract on a first vote. `voteWeight[id][voter]` defaults to 0 for unmapped keys, so even if the subtraction ran, it would subtract 0. But skipping it is a gas optimization and semantically correct.

**Attack 6: Stall re-check prevents post-recovery voting (lines 440-442)**

The stall re-check at the top of `vote()` requires `block.timestamp - uint256(lastVrf) >= ADMIN_STALL_THRESHOLD` (20 hours). If VRF recovers (a successful `_applyDailyRng()` updates `lastVrfProcessedTimestamp` to the current block time), the stall duration drops to near-zero, and all subsequent `vote()` calls revert with `NotStalled()`. This is the auto-invalidation mechanism: VRF recovery makes all governance proposals unvotable. Proposals remain Active but cannot accumulate more votes or be executed. They will eventually expire after 168h. Correct behavior.

## GOV-07: `_executeSwap()` CEI Compliance and Reentrancy Analysis

**Verdict:** KNOWN-ISSUE (Theoretical reentrancy via sibling proposal, not practically exploitable)
**Severity:** Low (defense-in-depth improvement recommended, not a critical vulnerability)

### CEI Trace

#### State Changes (SSTOREs)

| Line | Variable | Value | Phase |
|------|----------|-------|-------|
| 560 | `p.state` | `ProposalState.Executed` | BEFORE all external calls (CEI-compliant guard) |
| 581 | `coordinator` | `newCoordinator` (user-supplied) | BETWEEN external calls 1 and 2 (CEI violation) |
| 584 | `subscriptionId` | `newSubId` (return value from call #2) | AFTER external call 2 |
| 585 | `vrfKeyHash` | `newKeyHash` (user-supplied) | AFTER external call 2 |
| 618 via `_voidAllActive` | `proposals[i].state` | `ProposalState.Killed` for each Active sibling | AFTER all 5 external calls |
| 618 via `_voidAllActive` | `activeProposalCount` | `0` (hard-set) | AFTER all 5 external calls |

#### External Calls

| # | Line | Target | Function | Trusted? | Wrapping |
|---|------|--------|----------|----------|----------|
| 1 | 570-577 | `oldCoord` (current coordinator) | `cancelSubscription(oldSub, address(this))` | YES -- current Chainlink coordinator | try/catch |
| 2 | 582-583 | `newCoordinator` (user-supplied) | `createSubscription()` | **NO -- untrusted** | none |
| 3 | 590-593 | `newCoordinator` (user-supplied) | `addConsumer(newSubId, GAME)` | **NO -- untrusted** | none |
| 4 | 597-601 | `gameAdmin` (`ContractAddresses.GAME`) | `updateVrfCoordinatorAndSub(...)` | YES -- hardcoded Game address | none |
| 5 | 607-612 | `newCoordinator` (via `linkToken.transferAndCall`) | ERC-677 callback | MIXED -- linkToken is trusted, but sends to untrusted `newCoordinator` | try/catch |

### CEI Compliance Assessment

**`p.state = ProposalState.Executed` at line 560 is the key state guard.** It is set BEFORE any external call executes. This prevents the SAME proposal from being re-executed via reentrancy (the `p.state != ProposalState.Active` check at line 445 in `vote()` would revert).

**CEI violation exists between lines 581 and 618:** Three storage variables (`coordinator`, `subscriptionId`, `vrfKeyHash`) are written BETWEEN external calls rather than before them. The `_voidAllActive` cleanup runs AFTER all 5 external calls. This is a partial CEI violation -- the critical state guard (line 560) is correctly positioned, but secondary state updates interleave with external calls.

### Reentrancy Path Analysis

**Attack surface:** External call #2 (`createSubscription()` on line 582-583) is the primary reentrancy vector. The `newCoordinator` address is user-supplied via `propose()`. A malicious coordinator contract could execute arbitrary code when `createSubscription()` is called.

**Path A: Reenter `vote()` to execute sibling proposal**

Setup: Attacker creates proposal 1 (malicious coordinator M) and proposal 2 (legitimate coordinator L). Both accumulate enough approve weight.

1. A voter calls `vote(1, true)` which triggers `_executeSwap(1)`
2. At line 560: `proposals[1].state = Executed` (guard set)
3. At line 570-577: `cancelSubscription` on old coordinator (succeeds or try/catch catches)
4. At line 581: `coordinator = M` (SSTORE to malicious address)
5. At line 582: `M.createSubscription()` is called -- **M reenters** `vote(2, true)`
6. Inside reentrant `vote(2, true)`:
   - Stall check passes (stall is still active -- no VRF recovery)
   - `proposals[2].state == Active` -- check passes (only proposal 1 was set to Executed)
   - Expiry check passes (within lifetime)
   - Vote weight recorded, approve threshold met
   - `_executeSwap(2)` is triggered:
     - `proposals[2].state = Executed` (line 560)
     - `oldSub = subscriptionId` -- this is still the OLD subscription ID (line 584 hasn't executed in the outer call yet)
     - `oldCoord = coordinator` -- this is NOW `M` (set at line 581 of outer call)
     - Tries to `cancelSubscription` on `M` with `oldSub` -- likely fails (M's try/catch catches it)
     - `coordinator = L` (proposal 2's coordinator)
     - `L.createSubscription()` -- returns `newSubId2`
     - `subscriptionId = newSubId2`
     - `vrfKeyHash = proposal2.keyHash`
     - `L.addConsumer(newSubId2, GAME)` -- configures consumer on L
     - `gameAdmin.updateVrfCoordinatorAndSub(L, newSubId2, proposal2.keyHash)` -- Game now points to L
     - LINK transfer to L
     - `_voidAllActive(2)` -- kills all Active proposals (proposal 1 is already Executed, skipped). Sets `activeProposalCount = 0`.
7. Reentrant call completes. Control returns to outer `M.createSubscription()` call.
8. Outer call resumes at line 583: `newSubId` = return value from M (whatever M chose to return)
9. At line 584: `subscriptionId = newSubId` -- **OVERWRITES** the valid `newSubId2` set by inner call
10. At line 585: `vrfKeyHash = proposal1.keyHash` -- **OVERWRITES** the valid keyHash from proposal 2
11. At line 590: `M.addConsumer(newSubId, GAME)` -- adds Game as consumer on malicious coordinator M
12. At line 597: `gameAdmin.updateVrfCoordinatorAndSub(M, newSubId, proposal1.keyHash)` -- **Game now points to M**, overwriting the legitimate L configuration
13. LINK transferred to M
14. `_voidAllActive(1)` runs -- no Active proposals remain (already killed by inner call). `activeProposalCount = 0` (idempotent).

**Result of Path A:** The outer call's state updates OVERWRITE the inner call's legitimate coordinator setup. The Game contract ends up pointing to the malicious coordinator M. DegenerusAdmin's `coordinator`, `subscriptionId`, and `vrfKeyHash` all point to M's configuration. The attacker has successfully installed their malicious coordinator despite a legitimate coordinator also being executed.

**Path B: Reenter `propose()` during execution**

A malicious coordinator could call `propose()` during `createSubscription()`. This would increment `proposalCount` and `activeProposalCount`. The subsequent `_voidAllActive` in the outer call would kill the newly created proposal and set `activeProposalCount = 0`. Net effect: the new proposal is immediately voided. No lasting impact beyond wasted gas.

**Path C: Reenter `shutdownVrf()`**

Requires `msg.sender == ContractAddresses.GAME`. The malicious coordinator cannot satisfy this constraint. Not exploitable.

### Practical Exploitability Assessment

**The reentrancy in Path A is theoretically possible but faces severe practical barriers:**

1. **Attacker must control sufficient sDGNRS vote weight.** To execute proposal 1 (malicious coordinator), enough sDGNRS holders must vote to approve it. The threshold starts at 60% and decays to 5% over 7 days. Even at 5%, the attacker must control that share of circulating sDGNRS.

2. **The malicious coordinator must pass community review.** In the governance flow, proposals are visible on-chain. A coordinator that reenters DegenerusAdmin would be identifiable by code inspection. Voters who approve such a coordinator are complicit.

3. **If the attacker controls enough votes to execute proposal 1, they already control governance.** The reentrancy does not grant privilege escalation -- the attacker could achieve the same outcome (installing malicious coordinator) by simply having their proposal be the only one that executes. The sibling proposal dual-execution is redundant with the attacker's existing governance control.

4. **The attack requires TWO proposals with sufficient votes simultaneously.** Both must be Active, both must meet the threshold. This is an unusual scenario -- typically governance converges on a single proposal.

**However:** The theoretical path exists and violates defense-in-depth. If a legitimate coordinator swap is in progress and a separate malicious proposal somehow accumulates enough weight (e.g., through a flash-loan-like sDGNRS accumulation if the frozen invariant were broken -- see VOTE-01), the reentrancy could corrupt VRF state.

### Recommended Mitigation

Add a reentrancy guard or reorder state changes to be fully CEI-compliant:

**Option A (minimal):** Move `_voidAllActive(proposalId)` to immediately after line 560 (before any external calls). This kills all sibling proposals before the malicious coordinator receives any call, preventing Path A entirely.

**Option B (comprehensive):** Add OpenZeppelin `ReentrancyGuard` to `vote()` and `_executeSwap()`. This is belt-and-suspenders but adds ~2600 gas per `vote()` call.

**Option C (current state):** Accept as known issue. The practical exploitation requires governance control that already subsumes the attack. Document in the audit report.

### Adversarial Check

**Attack attempted:** Construct a reentrancy scenario where a malicious VRF coordinator, approved through governance, reenters `vote()` during `_executeSwap()` to trigger dual execution of a sibling proposal, corrupting the final VRF configuration.

**Result:** The attack path exists in theory (Path A above). The `p.state = ProposalState.Executed` guard at line 560 prevents re-execution of the SAME proposal but does NOT prevent execution of a SIBLING proposal. State variables `coordinator`, `subscriptionId`, and `vrfKeyHash` are written between external calls, enabling the outer call to overwrite the inner call's legitimate configuration.

**Practical barrier:** The attacker must control enough sDGNRS voting power to pass two proposals simultaneously, which means they already control governance. The reentrancy is not a privilege escalation vector.

**Verdict rationale:** KNOWN-ISSUE rather than FAIL because (1) the primary state guard `p.state = Executed` is correctly positioned, (2) practical exploitation requires pre-existing governance control, and (3) the severity is bounded by the governance trust model. However, the CEI violation is real and a defense-in-depth fix (moving `_voidAllActive` before external calls) is recommended.

## GOV-04: Threshold Decay Verification

**Verdict:** PASS
**Severity:** N/A

### Evidence

**Code:** `DegenerusAdmin.sol` lines 524-534, function `threshold(uint256 proposalId)`

```solidity
function threshold(uint256 proposalId) public view returns (uint16) {
    uint256 elapsed = block.timestamp - uint256(proposals[proposalId].createdAt);
    if (elapsed >= 168 hours) return 0;
    if (elapsed >= 144 hours) return 500;   // 5%
    if (elapsed >= 120 hours) return 1000;  // 10%
    if (elapsed >= 96 hours)  return 2000;  // 20%
    if (elapsed >= 72 hours)  return 3000;  // 30%
    if (elapsed >= 48 hours)  return 4000;  // 40%
    if (elapsed >= 24 hours)  return 5000;  // 50%
    return 6000; // 60%
}
```

**Tests:** `test/unit/VRFGovernance.test.js` -- "threshold decay" describe block with 8 tests covering every step: 6000 at creation, 5000 after 24h, 4000 after 48h, 3000 after 72h, 2000 after 96h, 1000 after 120h, 500 after 144h, 0 after 168h.

### Analysis

**Spec compliance:** The decay schedule matches the specification exactly:

| Time Elapsed | Threshold (bps) | Threshold (%) | Spec Match |
|-------------|-----------------|---------------|------------|
| 0-24h | 6000 | 60% | YES |
| 24h-48h | 5000 | 50% | YES |
| 48h-72h | 4000 | 40% | YES |
| 72h-96h | 3000 | 30% | YES |
| 96h-120h | 2000 | 20% | YES |
| 120h-144h | 1000 | 10% | YES |
| 144h-168h | 500 | 5% | YES |
| 168h+ | 0 | 0% | YES |

**If-else ordering:** The function checks from the largest elapsed time (168h) downward to the smallest (24h), with the default return of 6000 for anything below 24h. This ordering is critical -- if reversed, a 168h-old proposal would match `>= 24 hours` first and return 5000 instead of 0. Current order is correct.

**Return type:** `uint16` can hold max 65535. Maximum returned value is 6000. No truncation risk.

**Arithmetic:** `block.timestamp - uint256(proposals[proposalId].createdAt)` -- `createdAt` is `uint48`, upcast to `uint256`. `block.timestamp` is always >= `createdAt` (set at proposal creation in same or earlier block). No underflow possible.

### Adversarial Check

**Attack 1: Exact boundary at 24h (86400 seconds)**
- At exactly 24h: `elapsed >= 24 hours` is TRUE. The prior checks (`>= 168h`, `>= 144h`, etc.) are all FALSE. Function falls through to `>= 24 hours` and returns 5000. Correct.
- At 23h59m59s (86399 seconds): `elapsed >= 24 hours` is FALSE. Falls through to default, returns 6000. Correct.
- No off-by-one: `>=` includes the exact boundary.

**Attack 2: Exact boundary at 168h (threshold 0 meets expiry)**
- At exactly 168h: `elapsed >= 168 hours` is TRUE, returns 0. But the vote function (line 449) checks `block.timestamp - uint256(p.createdAt) >= PROPOSAL_LIFETIME` where `PROPOSAL_LIFETIME = 168 hours`. This means a 168h-old proposal's vote would revert with `ProposalExpired` before `threshold()` is ever called. The 0 threshold return at 168h is unreachable via `vote()`.
- `canExecute()` (line 542) also checks `block.timestamp - uint256(p.createdAt) >= PROPOSAL_LIFETIME` and returns false. The 0 threshold is unreachable via `canExecute()` as well.
- At 167h59m59s: `elapsed >= 168 hours` is FALSE. Falls to `>= 144 hours` (TRUE), returns 500. This is the last valid voting window with a 5% threshold.
- Conclusion: The 0 return value at 168h is dead code in practice -- both `vote()` and `canExecute()` reject expired proposals before calling `threshold()`. This is not a bug; the 0 return is a safe fallback that could only be reached via a direct external `threshold()` view call.

**Attack 3: Proposal with `createdAt == 0` (never created)**
- If `proposals[proposalId].createdAt == 0` (non-existent proposal), `elapsed = block.timestamp - 0 = block.timestamp` (a very large number). The function returns 0. But `vote()` checks `p.createdAt == 0` on line 445 and reverts. `canExecute()` checks `p.createdAt == 0` on line 541 and returns false. The threshold for a non-existent proposal is unreachable via governance flows.

## GOV-05: Execute Condition Verification

**Verdict:** PASS
**Severity:** N/A

### Evidence

**Code:** `DegenerusAdmin.sol` lines 487-494

```solidity
// Execute: approve% >= threshold AND approve > reject
if (
    p.approveWeight * BPS >= uint256(t) * p.circulatingSnapshot &&
    p.approveWeight > p.rejectWeight
) {
    _executeSwap(proposalId);
    return;
}
```

**Constants:** `BPS = 10000` (uint256, line 309). `threshold()` returns uint16 with max value 6000.

**Tests:** `test/unit/VRFGovernance.test.js` -- "canExecute" view tests (returns false for non-existent proposals and when no approve weight). Execute-with-weight test added in this audit plan (Task 2).

### Analysis

**Overflow analysis -- left side: `p.approveWeight * BPS`**
- `approveWeight` is `uint256`. Maximum value = total sDGNRS supply. sDGNRS is an ERC20 with 18 decimals. Realistic max totalSupply: 1 billion tokens = 1e9 * 1e18 = 1e27.
- `BPS` is `uint256(10000)` = 1e4.
- Maximum product: `1e27 * 1e4 = 1e31`.
- `uint256` max: ~1.15e77.
- `1e31 << 1.15e77`. **NO OVERFLOW.** Margin: 46 orders of magnitude.

**Overflow analysis -- right side: `uint256(t) * p.circulatingSnapshot`**
- `t` is `uint16` upcast to `uint256`. Maximum value: 6000.
- `circulatingSnapshot` is `uint256`. Maximum value = sDGNRS totalSupply = 1e27 (same as above).
- Maximum product: `6000 * 1e27 = 6e30`.
- `6e30 << 1.15e77`. **NO OVERFLOW.** Margin: 47 orders of magnitude.

**Truncation analysis:** Both sides are pure multiplications -- no division involved. No truncation or rounding error is possible.

**Semantic correctness:** The condition checks `(approveWeight / circulatingSnapshot) >= (threshold / BPS)` but expressed as cross-multiplication to avoid division (and thus truncation). This is the correct way to compare ratios in integer arithmetic.

**Strict greater-than for approve > reject:** Line 490 uses `>` (not `>=`). Ties go to neither side -- a 50/50 split does NOT execute. This prevents deadlock and is correct behavior: the status quo (no swap) prevails on ties.

### Adversarial Check

**Attack 1: Threshold = 0 (day 7+)**
- `approveWeight * 10000 >= 0 * circulatingSnapshot` simplifies to `approveWeight * 10000 >= 0`, which is TRUE for any `approveWeight >= 0` (uint256 is always >= 0).
- Combined with `approveWeight > rejectWeight`: any single approve vote with no reject votes would execute.
- This is intentional: at 0% threshold after 7 days, the minimum possible approval wins. This is the decay's purpose -- ensure proposals eventually resolve.

**Attack 2: `circulatingSnapshot == 0`**
- If `circulatingSnapshot` were 0, the right side becomes `uint256(t) * 0 = 0`. Any non-zero `approveWeight` meets the threshold.
- How could `circulatingSnapshot == 0`? The snapshot is captured at `propose()` time (line 424) via `circulatingSupply()`. `circulatingSupply()` returns `sDGNRS.totalSupply() - sDGNRS.balanceOf(SDGNRS) - sDGNRS.balanceOf(DGNRS)`.
- Community path: `propose()` checks `circ == 0` at line 413 and reverts. A community proposal cannot have a zero snapshot.
- Admin path: No circ check. An admin could propose when circulating supply is 0. The snapshot would be 0, and any approve weight meets the threshold.
- Practical impact: If circ is 0, nobody holds sDGNRS, so nobody can vote (line 458 reverts with `InsufficientStake` on zero balance). The proposal would sit Active until expiry. An admin with zero-circ proposal cannot unilaterally execute because `vote()` requires `weight > 0`.
- **Edge case documented but not exploitable.** Zero circulating supply prevents voting, which prevents execution.

**Attack 3: `approveWeight == 0` with `threshold == 0`**
- `0 * 10000 >= 0 * circ` is `0 >= 0`, which is TRUE.
- But `0 > rejectWeight` requires `rejectWeight == 0` (no reject votes either).
- Combined: if nobody votes and threshold reaches 0, the condition `0 >= 0 AND 0 > 0` evaluates to `TRUE AND FALSE = FALSE`. No execution without at least one approve vote.
- If exactly 1 approve vote and 0 reject votes at threshold 0: `weight * 10000 >= 0 AND weight > 0` = `TRUE AND TRUE`. Executes. This is correct -- one voter with approval at 0% threshold should suffice.

## GOV-06: Kill Condition Verification

**Verdict:** PASS
**Severity:** N/A

### Evidence

**Code:** `DegenerusAdmin.sol` lines 496-504

```solidity
// Kill: reject > approve AND reject% >= threshold
if (
    p.rejectWeight > p.approveWeight &&
    p.rejectWeight * BPS >= uint256(t) * p.circulatingSnapshot
) {
    p.state = ProposalState.Killed;
    unchecked { activeProposalCount--; }
    emit ProposalKilled(proposalId);
}
```

**Tests:** Kill condition test added in this audit plan (Task 2). Tests verify state transition to Killed, `activeProposalCount` decrement, and `ProposalKilled` event emission. Tie condition test confirms neither execute nor kill triggers on equal weights.

### Analysis

**Symmetry with execute condition:**
- Execute: `approveWeight * BPS >= threshold * circ AND approveWeight > rejectWeight`
- Kill: `rejectWeight > approveWeight AND rejectWeight * BPS >= threshold * circ`

The conditions are symmetric -- kill mirrors execute with approve/reject swapped. The conjunction (AND) is the same. The operand order within each sub-condition differs (kill checks `reject > approve` first, execute checks the threshold first) but since both are AND conditions, evaluation order does not affect correctness.

**Overflow analysis:** Identical to GOV-05. `rejectWeight * BPS` has the same max value as `approveWeight * BPS` (both capped by total sDGNRS supply). No overflow. The right side `uint256(t) * circulatingSnapshot` is identical. **NO OVERFLOW.**

**Mutual exclusion -- can both execute and kill trigger in the same `vote()` call?**
- Execute requires `approveWeight > rejectWeight`.
- Kill requires `rejectWeight > approveWeight`.
- These are strict inequalities. `a > b AND b > a` is a contradiction for any values of a, b. **Mutual exclusion is guaranteed.**
- A single `vote()` call can trigger at most one of: execute, kill, or neither.

**Neither condition met (tie):**
- If `approveWeight == rejectWeight`, neither `approveWeight > rejectWeight` nor `rejectWeight > approveWeight` is true. Neither execute nor kill triggers. The proposal remains Active. This is correct: ties preserve the status quo (no swap occurs).

**Kill state transition:** The kill path sets `p.state = ProposalState.Killed` (line 501). Subsequent votes on this proposal would revert at line 445 (`p.state != ProposalState.Active`). No double-kill possible.

**`activeProposalCount` decrement:** `unchecked { activeProposalCount--; }` (line 502). This is the same uint8 underflow concern flagged in VOTE-03 (separate requirement). If `activeProposalCount` is already 0 (e.g., due to overflow from VOTE-03), the decrement wraps to 255. This is an existing concern tracked under VOTE-03 and does not represent a separate vulnerability in the kill condition itself.

**Kill does NOT call `_executeSwap`:** Correct. Kill only updates state and decrements the counter. No external calls are made, so no reentrancy surface exists in the kill path. This is cleaner than the execute path.

### Adversarial Check

**Attack 1: Kill at threshold = 0**
- `rejectWeight * 10000 >= 0 * circ` is always TRUE (for any reject weight including 0).
- Combined with `rejectWeight > approveWeight`: any single reject vote with no approve votes would kill the proposal at threshold 0.
- Symmetric with execute behavior at threshold 0. Intentional.

**Attack 2: Simultaneous execute and kill via reentrancy**
- Could a malicious actor trigger execute on proposal A, then during `_executeSwap`'s external calls reenter `vote()` to kill proposal B?
- Yes, this is theoretically possible (the reentering `vote()` could trigger the kill path for a different proposal). But the kill path has no external calls -- it just writes storage and emits an event. No cascading reentrancy from the kill path. The kill during reentrancy would correctly set proposal B's state to Killed and decrement `activeProposalCount`. This is safe because `_voidAllActive()` (called at the end of `_executeSwap`) would then skip proposal B (already Killed, not Active). The final `activeProposalCount = 0` in `_voidAllActive` overrides any intermediate decrements.

**Attack 3: Kill a proposal that was already executed**
- `_executeSwap` sets `p.state = ProposalState.Executed` at line 560, before `_voidAllActive`. If a reentering call tries to kill the same proposal, `p.state != ProposalState.Active` (line 445) would revert. Cannot kill an executed proposal.

**Attack 4: Race between kill and expiry**
- Expiry check (lines 449-453) runs before the kill condition check (lines 497-504). If the proposal has expired, the vote reverts with `ProposalExpired` before the kill condition is evaluated. Expiry takes priority. Correct ordering.

## GOV-09: Proposal Expiry

**Verdict:** PASS -- with INFORMATIONAL note on lazy expiry
**Severity:** INFORMATIONAL

### Evidence

**Expiry logic at DegenerusAdmin.sol lines 449-453:**

```solidity
// Check expiry
if (block.timestamp - uint256(p.createdAt) >= PROPOSAL_LIFETIME) {
    p.state = ProposalState.Expired;
    unchecked { activeProposalCount--; }
    revert ProposalExpired();
}
```

`PROPOSAL_LIFETIME = 168 hours` (7 days).

**Expiry in `canExecute()` at line 542:**

```solidity
if (block.timestamp - uint256(p.createdAt) >= PROPOSAL_LIFETIME) return false;
```

**Expiry in `threshold()` at line 526:**

```solidity
if (elapsed >= 168 hours) return 0;
```

### Analysis

**1. State transition correctness -- revert rollback behavior:**

Expiry is triggered inside `vote()` before the voter weight check (line 449 runs before line 457). When a proposal has expired, the code:
1. Sets `p.state = ProposalState.Expired` (storage write)
2. Decrements `activeProposalCount` via `unchecked`
3. Executes `revert ProposalExpired()`

**Because the function reverts, all state changes (the `p.state` write and the `activeProposalCount` decrement) are rolled back.** The proposal remains `Active` in storage permanently, but every future `vote()` call on it will trigger the same revert sequence. The proposal is functionally dead but not cleaned up in storage.

**2. Lazy (passive) expiry -- `activeProposalCount` inflation:**

Since the revert rolls back the decrement, `activeProposalCount` remains inflated after a proposal expires. The count is only reduced when:
- Another proposal is executed, triggering `_voidAllActive()` which resets `activeProposalCount = 0`
- A non-expired proposal is killed via the reject threshold in `vote()`

There is no dedicated cleanup function for expired proposals.

**Impact on `anyProposalActive()`:**
- `anyProposalActive()` returns `activeProposalCount > 0`
- After all proposals expire, `activeProposalCount` remains non-zero
- The death clock in `_handleGameOverPath()` (DegenerusGameAdvanceModule line 432) pauses when `anyProposalActive()` returns true
- The death clock stays paused even though governance is effectively dead (all proposals expired)

**Net effect:** The death clock pauses longer than necessary. This is **protective behavior** -- it extends the window during a VRF stall, giving more time for recovery or new proposals. An attacker cannot exploit this to harm the protocol; at worst it delays game-over during an ongoing VRF failure (which is the desired behavior during a stall).

**3. Threshold consistency at expiry boundary:**

- `threshold()` returns `0` when `elapsed >= 168 hours`
- `vote()` reverts with `ProposalExpired()` when `elapsed >= PROPOSAL_LIFETIME` (168 hours -- same condition)
- The vote revert at line 449 runs BEFORE the threshold check at line 485
- Therefore `threshold=0` is unreachable via any path that modifies state. Consistent.
- `canExecute()` returns `false` at `elapsed >= PROPOSAL_LIFETIME` (line 542). Consistent.

**4. Existing test coverage:**

VRFGovernance.test.js "proposal expiry" section (lines 276-293): votes on an expired proposal after advancing 168h+1s and confirms the revert.

### Adversarial Check

**Attack 1: Grief via permanent activeProposalCount inflation**
- **Attempted:** Create proposals that expire without cleanup, permanently inflating `activeProposalCount` to keep the death clock paused indefinitely.
- **Result:** Creating proposals requires either DGVE >50.1% (admin path) or 0.5% sDGNRS (community path) plus a genuine VRF stall (20h or 7d). The inflated count only pauses the death clock, which is protective. When any proposal IS executed, `_voidAllActive()` resets `activeProposalCount = 0`. The only harmful scenario is a uint8 overflow (analyzed separately in VOTE-03).

**Attack 2: Exploit the revert rollback to keep a proposal "alive" indefinitely**
- **Attempted:** Since the Expired state is never persisted (rolled back by revert), can the proposal somehow be made executable again?
- **Result:** No. The `block.timestamp - createdAt >= PROPOSAL_LIFETIME` check is deterministic and monotonically increasing. Once true, it is always true for all future blocks. Every `vote()` call reverts. `canExecute()` returns false. The proposal is permanently inaccessible despite being technically Active in storage.

**Attack 3: Race condition at expiry boundary (168h - 1 second)**
- **Attempted:** Vote at the last possible second before expiry when threshold is at its lowest (5% at 144h+).
- **Result:** At elapsed = 167h 59m 59s, the proposal is still Active and threshold is 500 (5%). A vote at this point would succeed and check execution conditions normally. At elapsed = 168h exactly, `vote()` reverts. There is no gap between "threshold=0 allows trivial execution" and "expiry blocks execution" because the threshold never reaches 0 while the proposal is votable (threshold returns 500 at 144h-167h59m59s).

- **`sDGNRS.totalSupply()`** -- All minted sDGNRS (1 trillion initial supply). Reduced only by `StakedDegenerusStonk.burn()` (public) and `StakedDegenerusStonk.burnRemainingPools()` (onlyGame, game-over only).
- **`sDGNRS.balanceOf(ContractAddresses.SDGNRS)`** -- sDGNRS held by the StakedDegenerusStonk contract itself. This represents undistributed reward pools (Whale, Affiliate, Lootbox, Reward, Earlybird). These tokens have never been distributed to players and should not count toward voting power.
- **`sDGNRS.balanceOf(ContractAddresses.DGNRS)`** -- sDGNRS held by the DegenerusStonk wrapper contract. This is the backing for the transferable DGNRS token. DGNRS holders cannot vote in governance; only sDGNRS holders can.

**Result = sDGNRS in individual voter wallets** (addresses that received sDGNRS via pool distributions or via `unwrapTo`).

**2. Completeness of exclusions:**

| Address | sDGNRS Balance | Excluded? | Reason |
|---------|---------------|-----------|--------|
| StakedDegenerusStonk (self) | Undistributed pools | YES | Not distributed to voters |
| DegenerusStonk (wrapper) | DGNRS backing | YES | Represents DGNRS, not direct sDGNRS holders |
| DegenerusAdmin | 0 | N/A | Admin never holds sDGNRS |
| DegenerusGame | 0 | N/A | Game distributes from pools, never holds sDGNRS itself |
| Individual voters | Distributed sDGNRS | NOT excluded (correct) | These are the governance participants |

sDGNRS is **soulbound** -- StakedDegenerusStonk has no `transfer()` or `transferFrom()` function. The only paths that move sDGNRS to a new address:
- `transferFromPool()` (Game -> player, reduces `balanceOf(SDGNRS)`, increases `balanceOf(player)`)
- `wrapperTransferTo()` (DGNRS wrapper -> recipient, reduces `balanceOf(DGNRS)`, increases `balanceOf(recipient)`)

No other contract can receive sDGNRS because peer-to-peer transfers are impossible.

**3. Underflow analysis:**

Subtraction `totalSupply - balanceOf(SDGNRS) - balanceOf(DGNRS)` in Solidity 0.8.34 is checked arithmetic. Underflow would revert with a panic.

**Can `balanceOf(SDGNRS) + balanceOf(DGNRS) > totalSupply`?**
- `totalSupply` always equals the sum of ALL `balanceOf` entries (invariant of `_mint` and `burn` in StakedDegenerusStonk).
- `balanceOf(SDGNRS) + balanceOf(DGNRS) <= totalSupply` because other addresses may also hold positive balances.
- The only operation that reduces `totalSupply` is `burn()`, which decrements `totalSupply` and `balanceOf[player]` by the same amount. If `player` is neither SDGNRS nor DGNRS, the excluded balances stay unchanged while `totalSupply` decreases. But `totalSupply` still >= `balanceOf(SDGNRS) + balanceOf(DGNRS)` because the burned amount was in `balanceOf[player]`, not in the excluded addresses.

**Underflow is impossible under correct accounting.** If all voter sDGNRS were burned, `circulatingSupply()` would return 0.

**4. Existing test coverage:**

VRFGovernance.test.js "circulatingSupply" section (lines 320-333): verifies the formula against independently computed `total - sdgnrsBal - dgnrsBal`.

### Adversarial Check

**Attack 1: Manipulate circulatingSupply before proposing (via DGNRS burn)**
- **Attempted:** Burn DGNRS tokens to reduce `sDGNRS.balanceOf(DGNRS)`, hoping to inflate `circulatingSupply` and lower the effective threshold.
- **Result:** Burning DGNRS calls `DegenerusStonk._burn()` (reduces DGNRS.totalSupply and DGNRS.balanceOf[caller]) then calls `StakedDegenerusStonk.burn(amount)` with `msg.sender = DGNRS address`. Inside sDGNRS.burn: `totalSupply -= amount` and `balanceOf[DGNRS] -= amount`. Both decrease by the same `amount`. In the formula:
  - `circulatingSupply = (totalSupply - amount) - balanceOf(SDGNRS) - (balanceOf(DGNRS) - amount)`
  - `= totalSupply - balanceOf(SDGNRS) - balanceOf(DGNRS)`

**Attack 2: Inflate circulatingSupply via pool distributions**
- **Attempted:** Trigger game distributions to move sDGNRS from pools (SDGNRS) to voter wallets.

- **Result:** `burnRemainingPools()` is `onlyGame` and only callable during game-over. `propose()` checks `gameAdmin.gameOver()` and reverts with `GameOver()`. Governance is inaccessible when this function is callable. Not exploitable.

## GOV-08: `_voidAllActive()` Correctness

**Verdict:** PASS
**Severity:** N/A

### Evidence

**Code:** `DegenerusAdmin.sol` lines 622-632

```solidity
function _voidAllActive(uint256 exceptId) internal {
    uint256 count = proposalCount;
    for (uint256 i = 1; i <= count; i++) {
        if (i == exceptId) continue;
        if (proposals[i].state == ProposalState.Active) {
            proposals[i].state = ProposalState.Killed;
            emit ProposalKilled(i);
        }
    }
    activeProposalCount = 0;
}
```

**Tests:** `test/unit/VRFGovernance.test.js` -- "_voidAllActive via execute with multiple proposals" describe block with 2 tests:
1. "executing one proposal voids all other active proposals" -- Creates 3 proposals, executes proposal 2 via approve vote, verifies proposals 1 and 3 are Killed, proposal 2 is Executed, `activeProposalCount == 0`, `ProposalKilled` events emitted for 1 and 3 but not 2.
2. "_voidAllActive skips non-Active proposals (expired/killed)" -- Creates 3 proposals, kills proposal 1 via reject vote, then executes proposal 2. Verifies proposal 1 remains Killed (not re-emitted), proposal 3 is Killed by `_voidAllActive`, and `activeProposalCount == 0`.

### Analysis

**1. Loop boundary correctness (off-by-one analysis):**

- Loop starts at `i = 1`: Proposals are 1-indexed (created via `proposalId = ++proposalCount` at line 418). The first proposal has ID 1. Starting at `i = 1` is correct.
- Loop condition `i <= count` where `count = proposalCount`: The last proposal has ID equal to `proposalCount`. Using `<=` ensures the last proposal is included. Correct.
- If `proposalCount == 0` (no proposals ever created): `1 <= 0` is false. Loop body never executes. `activeProposalCount = 0` is still set (already 0 since no proposals exist). Correct and harmless.

**2. `exceptId` skip logic:**

- The executed proposal (passed as `exceptId`) is skipped via `if (i == exceptId) continue`. Its state was already set to `ProposalState.Executed` at line 560 of `_executeSwap`, before `_voidAllActive` is called at line 618.
- Even without the `continue`, the `if (proposals[i].state == ProposalState.Active)` check would skip it because its state is `Executed`, not `Active`. The `exceptId` skip is a belt-and-suspenders guard.

**3. Non-Active proposal handling:**

- Proposals with state `Expired`, `Killed`, or `Executed` fail the `proposals[i].state == ProposalState.Active` check and are skipped. Only `Active` proposals are transitioned to `Killed`.
- No `ProposalKilled` event is emitted for proposals that were already non-Active. No duplicate events.

**4. `activeProposalCount = 0` (hard-set vs decrement):**

- The hard-set to `0` is correct because after `_voidAllActive` completes, zero proposals should be Active:
  - The executed proposal: `Executed` (not Active)
  - All other Active proposals: now `Killed`
  - Any previously non-Active proposals: unchanged
- A decremental approach (`activeProposalCount -= killedCount`) would be fragile:
  - Requires accurate tracking of how many were Active before the loop
  - Susceptible to bugs if `activeProposalCount` is already inconsistent (e.g., due to VOTE-03 uint8 overflow)
- The hard-set is both simpler and more robust. It serves as a "state reset" that corrects any prior counting errors.

**5. Gas analysis:**

- Each loop iteration performs one cold SLOAD (`proposals[i].state`, 2100 gas first access per slot) plus potentially one warm SSTORE (state change, 5000 gas for Killed) and one event emission (~375 gas base + indexed topic).
- Worst case: all proposals are Active. With N proposals: N * (2100 + 5000 + 375) = ~7475 * N gas.
- At N = 255 (uint8 max of `activeProposalCount`): ~1.9M gas. Well within block gas limit (30M).
- At N = 1000 (if someone creates many proposals): ~7.5M gas. Still within limits but significant. This is WAR-06 territory (spam-propose griefing).

**6. Idempotency under reentrancy:**

- If `_voidAllActive` is called twice (due to reentrancy described in GOV-07):
  - First call: sets all Active proposals to Killed, sets `activeProposalCount = 0`.
  - Second call: loop runs again, but all proposals are now Killed or Executed. The `state == ProposalState.Active` check fails for every proposal. No state changes occur. `activeProposalCount = 0` is written again (already 0). No harm.
- The function is idempotent. Double invocation is safe.

### Adversarial Check

**Attack 1: Off-by-one -- proposal at boundary**
- **Attempted:** Create proposal at exactly `proposalCount` and verify it is included in the void loop.
- **Result:** The loop condition `i <= count` includes `i == count`. Since `proposalCount` was read into `count` before the loop, and `proposalCount` is only modified by `propose()` (which is not called during `_voidAllActive`), the snapshot is consistent. The last proposal is always included.

**Attack 2: Reentrancy creates new proposal during loop**
- **Attempted:** If a malicious coordinator reenters `propose()` during `_executeSwap` (before `_voidAllActive`), a new proposal is created with `proposalId = proposalCount + 1`. When `_voidAllActive` runs, `count = proposalCount` (now incremented). The new proposal is included in the loop and voided. Correct behavior -- proposals created during execution are immediately invalidated.
- **Follow-up:** What if `propose()` is called DURING `_voidAllActive`? This is impossible because `_voidAllActive` makes no external calls. It only reads/writes storage and emits events. No reentrancy surface exists within the loop itself.

**Attack 3: `activeProposalCount = 0` when proposals are still Active**
- **Attempted:** If the loop somehow skips an Active proposal, `activeProposalCount` would be incorrectly set to 0 while a proposal remains Active.
- **Result:** The only way to skip an Active proposal is if `i == exceptId` AND the proposal at `exceptId` is still Active. But `exceptId` is the just-executed proposal, whose state was set to `Executed` at line 560. The `exceptId` skip is redundant but not incorrect. No Active proposal can be skipped unless it is the executed one (which is no longer Active).

**Attack 4: `activeProposalCount` mismatch after partial execution**
- **Attempted:** If `_executeSwap` partially fails (revert after some external calls but before `_voidAllActive`), the executed proposal has `state = Executed` but `activeProposalCount` was never adjusted.
- **Result:** If `_executeSwap` reverts at any point, ALL state changes are rolled back, including `p.state = Executed`. The proposal returns to Active, `activeProposalCount` is unchanged. The revert atomicity of the EVM ensures consistency. However, note that external calls wrapped in `try/catch` (lines 570-577, 607-612) catch reverts and continue. Only an uncaught revert between the try/catch blocks would trigger this scenario -- specifically, if `createSubscription()`, `addConsumer()`, or `updateVrfCoordinatorAndSub()` reverts (these are NOT try/catch wrapped). In that case, the entire `vote()` transaction reverts atomically. No partial state corruption.

### Evidence

### Analysis

**1. During VRF stall (>20 hours), individual voter sDGNRS balances are frozen because:**

- **No minting:** `_mint` is `private` and only called in the constructor. No post-deployment minting is possible.
- **No pool distributions:** `transferFromPool` requires `advanceGame()` to execute, which requires a VRF word. No VRF word during stall = no distributions.
- **No unwrapTo:** `DegenerusStonk.unwrapTo()` (line 151) checks `block.timestamp - lastVrfProcessed() > 20 hours` and reverts with `Unauthorized`. When stall > 20h, unwrap is BLOCKED.
- **No peer-to-peer transfers:** sDGNRS is soulbound. There is no `transfer()` function. Period.
- **Burn only affects the caller's OWN balance:** A voter can burn their own sDGNRS, which reduces their `balanceOf` and `totalSupply`. This does NOT increase any other voter's balance.

**2. Impact of `burn` on vote arithmetic during stall:**

When a voter burns sDGNRS during a stall:
- Their `balanceOf` decreases (voter has less voting power for future votes)
- `totalSupply` decreases
- `circulatingSupply()` changes: both `totalSupply` and voter's balance decrease, but `balanceOf(SDGNRS)` and `balanceOf(DGNRS)` are unchanged. So `circulatingSupply = (totalSupply - X) - balanceOf(SDGNRS) - balanceOf(DGNRS)` where X is the burned amount. **circulatingSupply DECREASES by X.**
- But the `circulatingSnapshot` was captured at proposal creation and is immutable (verified in VOTE-02). The threshold comparison uses the SNAPSHOT, not the live circulating supply.
- The voter's previously recorded `voteWeight[proposalId][voter]` is NOT retroactively updated. Their old vote weight persists.
- If the voter tries to re-vote, their new weight will reflect their reduced balance. The subtract-old-add-new pattern in `vote()` correctly handles this.

**Burn during stall is safe for vote integrity.** It cannot inflate any voter's balance or allow double-counting.

**3. Edge case: unwrapTo at exactly 20 hours:**

`DegenerusStonk.unwrapTo()` line 151: `block.timestamp - lastVrfProcessed() > 20 hours`

At exactly 20 hours (72000 seconds): `72000 > 72000` is FALSE. unwrapTo is ALLOWED.
At 20h + 1 second (72001 seconds): `72001 > 72000` is TRUE. unwrapTo is BLOCKED.

This means there is a 1-block window at exactly 20h where unwrapTo is permitted, allowing the DGNRS creator to unwrap DGNRS to sDGNRS for a specific recipient. This is the WAR-04 edge case (already documented in the research). The `propose()` admin stall threshold uses `< 20 hours` (strict less-than), so proposing is allowed when stall >= 20h. At exactly 20h, both proposing AND unwrapTo are permitted simultaneously.

**Impact:** The creator could unwrap DGNRS to sDGNRS for themselves or an accomplice at exactly 20h, then propose and vote in the same block. This moves sDGNRS from the DGNRS wrapper to a voter wallet, increasing that voter's balance. However:
- This is a single-block window (12 seconds on Ethereum mainnet)
- The creator already controls the DGNRS allocation (20% of total supply) and could have unwrapped before the stall
- The amount moved is bounded by the DGNRS contract's sDGNRS balance
- After this block, unwrapTo is blocked for the remainder of the stall

**Severity:** INFORMATIONAL. Documented as WAR-04 in the research. The 1-block window does not meaningfully undermine vote integrity.

**4. Existing test coverage:**

VRFGovernance.test.js "unwrapTo VRF stall guard" section (lines 358-377): confirms unwrapTo works normally without stall and reverts during >20h stall. The exact-boundary behavior at 20h is not explicitly tested but is analyzed here.

### Adversarial Check

**Attack 1: Vote-splitting via sDGNRS transfer**
- **Attempted:** Voter with 100 sDGNRS votes Approve, then transfers 100 sDGNRS to accomplice B, who also votes Approve. The same 100 sDGNRS is counted twice.
- **Result:** IMPOSSIBLE. StakedDegenerusStonk has no `transfer()` function. sDGNRS is soulbound. There is no mechanism to move sDGNRS between arbitrary addresses. Attack vector does not exist.

## VOTE-03: uint8 activeProposalCount Overflow

**Verdict:** KNOWN-ISSUE
**Severity:** LOW

### Evidence

**Declaration at DegenerusAdmin.sol line 274:**

```solidity
uint8 public activeProposalCount;
```

**All mutation sites:**

| Line | Function | Operation | Context |
|------|----------|-----------|---------|
| 428 | `propose()` | `unchecked { activeProposalCount++; }` | Increment on new proposal |
| 451 | `vote()` (expiry) | `unchecked { activeProposalCount--; }` | Decrement on expiry (rolled back by revert -- see GOV-09) |
| 502 | `vote()` (kill) | `unchecked { activeProposalCount--; }` | Decrement when reject threshold met |
| 631 | `_voidAllActive()` | `activeProposalCount = 0;` | Hard reset after execution |

**All read sites:**

| Line | Function | Usage |
|------|----------|-------|
| 510 | `anyProposalActive()` | `return activeProposalCount > 0;` |

### Analysis

**1. Overflow mechanics:**

`uint8` has range 0-255. `unchecked { activeProposalCount++; }` at line 428 wraps from 255 to 0.

If 256 proposals are created without any being killed or executed, `activeProposalCount` overflows to 0.

**2. Impact of overflow to 0:**

- `anyProposalActive()` returns `activeProposalCount > 0` = `0 > 0` = FALSE
- The death clock in `_handleGameOverPath()` (AdvanceModule line 432) calls `anyProposalActive()`. If FALSE, the death clock is NOT paused.
- **Result:** With 256+ active proposals, the system thinks no proposals are active. The death clock resumes countdown. If the death clock reaches zero, the game enters game-over WHILE governance is actively trying to swap VRF coordinators.
- This defeats the purpose of the death clock pause (M-02 mitigation).

**3. Exploitability analysis:**

**Admin path (20h stall):**
- Each `propose()` requires `vault.isVaultOwner(msg.sender)` (DGVE >50.1%) and stall >= 20h
- Cost per proposal: ~100k-150k gas (storage writes + external calls)
- 256 proposals: ~25.6M-38.4M gas total
- At 3M gas per block, this requires ~9-13 transactions
- **FEASIBLE** if the admin (DGVE majority holder) is the attacker

**Community path (7d stall):**
- Each `propose()` requires 0.5% sDGNRS and stall >= 7d
- A single community member with 0.5% sDGNRS can call `propose()` 256 times
- Each proposal needs a different `newCoordinator` address and `newKeyHash` (both are stored in the proposal struct and do not need to be valid contracts)
- **FEASIBLE** -- attacker can use any 256 different addresses

**VRF stall duration requirement:**
- Admin path: 20h minimum. Realistic for a Chainlink outage.
- Community path: 7d minimum. Extended outage, less likely but possible.

**Gas cost computation (admin path, worst case):**
- `propose()` gas: ~130k (4 SSTORE + 2 external calls + event)
- 256 calls: 256 * 130k = 33.28M gas
- At 30 gwei gas price: 33.28M * 30 gwei = ~1 ETH
- At $3000/ETH: ~$3000 total cost
- **Economically feasible** for a motivated attacker

**4. Subsequent underflow after overflow:**

After overflow to 0, the next `propose()` increments to 1 (correct). But if a kill or expiry-cleanup triggers `unchecked { activeProposalCount--; }` while the count is 0, it wraps to 255. This creates oscillating behavior where the count bounces between 0 and 255.

However, per GOV-09, the expiry decrement is rolled back by the revert. The kill decrement at line 502 IS persisted. So if a proposal is killed when `activeProposalCount == 0`, the count wraps to 255 and `anyProposalActive()` returns TRUE (255 > 0). This is "accidentally correct" -- there are still active proposals. But the count is now 255 instead of the true count (256+ minus the killed one).

**5. `_voidAllActive` reset:**

When any proposal IS executed, `_voidAllActive()` sets `activeProposalCount = 0`. This resets the overflow. But if no proposal is executed (e.g., all proposals expire during a prolonged stall), the overflow persists indefinitely.

### Mitigation Recommendation

**Option A (minimal):** Add a checked increment:
```solidity
require(activeProposalCount < 255, "TooManyProposals");
unchecked { activeProposalCount++; }
```
Cost: ~100 gas per `propose()` call (one SLOAD for the require).

**Option B (type change):** Change `uint8` to `uint16` (max 65535). This makes overflow practically impossible (65536 proposals during a single VRF stall is unrealistic). However, this changes storage layout and requires re-verification of slot packing. Currently `activeProposalCount` (uint8) shares slot 7 with `linkEthPriceFeed` (address, 20 bytes). Changing to uint16 still fits (2 + 20 = 22 bytes < 32 bytes), but the offset of `linkEthPriceFeed` shifts.

**Option C (keep as-is):** Document as known issue. The attack requires sustained VRF failure AND a motivated attacker with sufficient DGVE or sDGNRS. ## XCON-02: Death Clock Pause via anyProposalActive()

**Verdict:** PASS
**Severity:** N/A

### Evidence

**Code:** `DegenerusGameAdvanceModule._handleGameOverPath()` lines 417-436

```solidity

    uint48 ts, uint48 day, uint48 lst, uint24 lvl,
    bool lastPurchase, uint48 /* _dailyIdx */

            if (active) livenessTriggered = false;
        } catch {}

```

**`anyProposalActive()` at DegenerusAdmin.sol lines 509-511:**

```solidity
function anyProposalActive() external view returns (bool) {
    return activeProposalCount > 0;
}
```

### Analysis

**1. Liveness trigger conditions:**

- **Level 0 (pre-game):** `ts - lst > DEPLOY_IDLE_TIMEOUT_DAYS * 1 days` -- triggers if the game has been idle for the deploy timeout period since the last state change.
- **Level non-zero (active game):** `ts - 120 days > lst` -- triggers if 120 days have passed since the last significant state change.

When liveness is triggered, the game would enter game-over mode. The governance pause check intercepts this.

**2. Governance pause mechanism:**

If `livenessTriggered` is true, the module calls `IDegenerusAdmin(ADMIN).anyProposalActive()`. This is an external call to the DegenerusAdmin contract, which returns `activeProposalCount > 0`.

If any proposal is active (`activeProposalCount > 0`), `livenessTriggered` is set to false, preventing game-over. The death clock is effectively paused.

**3. try/catch analysis:**

The call is wrapped in `try ... catch {}`. Failure modes:

| Scenario | Behavior | livenessTriggered |
|----------|----------|-------------------|
| ADMIN contract returns `true` | `active = true`, `livenessTriggered = false` | FALSE (paused) |
| ADMIN contract returns `false` | `active = false`, `livenessTriggered` unchanged | TRUE (not paused) |
| ADMIN contract reverts (any reason) | Catch block executes (empty), `livenessTriggered` unchanged | TRUE (not paused) |
| ADMIN contract selfdestructed | External call returns 0 bytes, try block receives default `false` | TRUE (not paused) |
| Out-of-gas in external call | Catch block, `livenessTriggered` unchanged | TRUE (not paused) |

**Revert analysis for `anyProposalActive()`:** The function body is `return activeProposalCount > 0`. This performs:
1. One `SLOAD` (read `activeProposalCount` from storage slot 7, offset 0)
2. One comparison (`> 0`)
3. Return the boolean result

No external calls, no math that could overflow, no array indexing, no require/revert statements. The ONLY possible revert source is out-of-gas during the SLOAD (2100 gas for cold access). Since `advanceGame()` is called with a full block's gas limit and the SLOAD is the only operation, this is practically impossible.

The `catch {}` block is defensive coding. If the ADMIN contract is broken, destroyed, or upgraded to something unexpected, the death clock proceeds normally. This is the safe default: without governance, the game's liveness mechanism should function as designed (trigger game-over if abandoned).

**4. Correctness of `activeProposalCount > 0` as proxy for "governance is active":**

`activeProposalCount` is incremented by `propose()` and decremented by `vote()` (kill path, line 502) or hard-set to 0 by `_voidAllActive()` (execution path, line 631).

However, per GOV-09 (lazy expiry), expired proposals do NOT decrement `activeProposalCount` because the decrement is rolled back by the revert. This means `activeProposalCount` can be inflated -- returning TRUE when all proposals have actually expired.

**Impact:** The death clock stays paused longer than strictly necessary. This is **protective behavior**:
- If all proposals expired, the VRF stall is ongoing (proposals only expire during stalls)
- Keeping the death clock paused gives more time for new proposals
- An attacker cannot exploit this to harm the protocol -- at worst, game-over is delayed during a genuine VRF failure

**5. VOTE-03 interaction (uint8 overflow):**

If `activeProposalCount` overflows to 0 (256 proposals created without execution or cleanup), `anyProposalActive()` returns FALSE despite active proposals existing. This unpauses the death clock during active governance.

This is the same VOTE-03 finding (KNOWN-ISSUE, Low severity). Cross-reference: VOTE-03 recommends `require(activeProposalCount < 255)` in `propose()`.

**6. Timing between check and execution:**

`anyProposalActive()` is called during `advanceGame()`. Between this call and the next block, a proposal could expire or be killed. But the check is per-block: if active when checked, the death clock is paused for that block. On the next `advanceGame()` call (next block), the check runs again with the latest state. This is correct -- the death clock is re-evaluated every block. No stale-data risk.

**Cross-contract trace diagram:**

```
advanceGame() caller
  └──> DegenerusGame.advanceGame()
         └──> AdvanceModule._handleGameOverPath()  [delegatecall]
                ├── Compute livenessTriggered (local bool)
                ├── if livenessTriggered:
                │     └──> DegenerusAdmin.anyProposalActive()  [external STATICCALL]

```

### Adversarial Check

**Attack 1: Grief death clock by creating proposals to keep it paused forever**
- **Attempted:** Create proposals repeatedly to keep `activeProposalCount > 0`, preventing the death clock from ever triggering game-over.
- **Result:** Creating proposals requires DGVE >50.1% (admin path, 20h stall) or 0.5% sDGNRS (community path, 7d stall). The stall must be genuine. If the VRF stall persists, keeping the death clock paused is the DESIRED behavior -- it prevents game-over while governance tries to fix VRF. If VRF recovers, `vote()` re-checks the stall and reverts with `NotStalled`, making governance inaccessible. New proposals cannot be created. `activeProposalCount` remains inflated (lazy expiry) but no new proposals are added. Eventually the stall clears, no more proposals, but lazy expiry keeps count inflated. This is protective, not exploitable. The only harm is delayed game-over, which is the intended behavior during VRF failure.

No external manipulation possible beyond VOTE-03.

**Attack 3: Make the try/catch fail to suppress the pause**
- **Attempted:** Cause `anyProposalActive()` to revert so the catch block runs and `livenessTriggered` stays true.

**Impact assessment:**

- **Creator's action:** `unwrapTo` burns DGNRS from the creator's balance and transfers the underlying sDGNRS to a specified recipient (not the creator).
- **sDGNRS is soulbound:** The recipient cannot transfer the sDGNRS to another address. They can only use it for voting weight.

  1. Creator must time the transaction to the exact second (possible with MEV but non-trivial)
  2. The recipient must be pre-positioned to vote in the same block
  3. The amount is bounded by the creator's DGNRS balance
  4. The recipient's sDGNRS weight is still proportional to circulating supply -- even with the increase, the creator cannot single-handedly reach threshold (60% initial, decaying to 5% over 7 days)
  5. The creator LOSES their DGNRS in this process (permanently burned)

**Severity justification:** Informational. The 1-second window is a mathematical consequence of using `>` vs `<` operators with the same threshold. It is not practically exploitable for meaningful vote manipulation because:
- The window is 1 second wide
- sDGNRS is soulbound (no double-voting via transfer)
- The creator permanently loses the DGNRS tokens
- The recipient's weight increase is bounded and visible on-chain

The stall duration check is NOT in this function -- it is enforced upstream by `DegenerusAdmin.propose()`, `DegenerusAdmin.vote()`, and the execute threshold system. By the time `updateVrfCoordinatorAndSub` is called (from `_executeSwap` at DegenerusAdmin line 597), the governance process has already verified:

- A VRF stall of at least 20h (admin) or 7d (community) existed when proposed
- The stall persisted through voting (re-checked on every vote, line 441)
- The proposal accumulated enough sDGNRS-weighted votes to meet the decay threshold

**3. Direct call bypass analysis:**

Can someone call `DegenerusGame.updateVrfCoordinatorAndSub()` directly, bypassing Admin governance?

The function at AdvanceModule line 1277: `if (msg.sender != ContractAddresses.ADMIN) revert E();`

DegenerusGame routes to AdvanceModule via delegatecall. In the delegatecall context, `msg.sender` is the original external caller. For the check to pass, the external caller must be `ContractAddresses.ADMIN` -- the DegenerusAdmin contract.

DegenerusAdmin calls this function only from `_executeSwap` (line 597):
```solidity
gameAdmin.updateVrfCoordinatorAndSub(
    newCoordinator,
    newSubId,
    p.keyHash
);
```

No other function in DegenerusAdmin calls `updateVrfCoordinatorAndSub`. No other contract has `ContractAddresses.ADMIN` as its address. **Direct bypass is impossible.**

**4. Retained monitoring use:**

`_threeDayRngGap` is retained in `DegenerusGame.sol` as a private helper for the `rngStalledForThreeDays()` public view function (line 2199-2200). This is a monitoring/external use function with NatSpec:

```solidity
/// @dev Retained for monitoring/external use. Governance uses lastVrfProcessed() instead.
function rngStalledForThreeDays() external view returns (bool) {
    return _threeDayRngGap(_simulatedDayIndex());
}
```

This is explicitly documented as a monitoring function, not a governance function. It has no write effects and no callers in governance paths.

### Adversarial Check

**Attack 1: Call updateVrfCoordinatorAndSub directly to bypass governance**
- **Attempted:** External caller tries to call `DegenerusGame.updateVrfCoordinatorAndSub(maliciousCoord, 0, 0x0)` directly.
- **Result:** `msg.sender != ContractAddresses.ADMIN` check reverts with `E()`. Only the Admin contract can call this. IMPOSSIBLE.

**Attack 2: Deploy a contract at the ADMIN address to call updateVrfCoordinatorAndSub**
- **Attempted:** Use CREATE2 to deploy a malicious contract at the ADMIN precomputed address.
- **Result:** The ADMIN address is determined by the deployer's nonce or CREATE2 salt. Without the deployer's private key and the exact salt/nonce, deploying at the same address is computationally infeasible. IMPOSSIBLE in practice.

**Attack 3: Exploit lack of _threeDayRngGap check to call with no stall**
- **Attempted:** Since `updateVrfCoordinatorAndSub` does not check for VRF stall, could Admin execute a coordinator swap without a genuine stall?
- **Result:** The Admin contract enforces the stall check at `propose()` (line 403-408 for admin, line 411 for community) and `vote()` (line 440-442). A proposal cannot be created without a genuine stall. Votes revert if the stall resolves. The execute condition (`_executeSwap`) checks stall at `canExecute()` (line 544-546). **All stall enforcement is upstream of `updateVrfCoordinatorAndSub`.** The function correctly omits the check because it would be redundant.

### Tests

- `test/edge/RngStall.test.js` "rngStalledForThreeDays" tests (lines 435-648): 5+ tests confirming the monitoring function works correctly (returns false after deployment, false with normal VRF, true after 3+ days of missing words, false after retry cycle).
- `test/unit/VRFGovernance.test.js` "propose" tests (lines 46-97): Confirm `NotStalled()` revert without VRF stall, verifying the stall enforcement is in `DegenerusAdmin.propose()`, not in `updateVrfCoordinatorAndSub`.

## XCON-05: VRF Retry Timeout Change (18h to 12h)

**Verdict:** PASS
**Severity:** N/A

### Evidence

**Code:** `DegenerusGameAdvanceModule.rngGate()` lines 783-791

```solidity
// Waiting for VRF - check for timeout retry
if (rngRequestTime != 0) {
    uint48 elapsed = ts - rngRequestTime;
    if (elapsed >= 12 hours) {
        _requestRng(isTicketJackpotDay, lvl);
        return 1;
    }
    revert RngNotReady();
}
```

**The timeout is 12 hours** (43200 seconds). The old value was 18 hours (64800 seconds).

### Downstream Impact Analysis

**1. Governance timing interaction:**

| Threshold | Duration | Source |
|-----------|----------|--------|
| VRF retry timeout | 12 hours | AdvanceModule.rngGate() line 786 |
| Admin propose stall | 20 hours | DegenerusAdmin.ADMIN_STALL_THRESHOLD line 297 |
| Community propose stall | 7 days | DegenerusAdmin.COMMUNITY_STALL_THRESHOLD line 300 |
| Death clock (level 0) | DEPLOY_IDLE_TIMEOUT_DAYS * 1 days | AdvanceModule._handleGameOverPath() line 427 |
| Death clock (active game) | 120 days | AdvanceModule._handleGameOverPath() line 428 |

**Timeline during VRF stall:**

```
t=0h:   VRF request sent, no fulfillment
t=12h:  rngGate() retries VRF request (new requestId)
t=20h:  Admin can propose (governance activates)
t=24h:  Second VRF retry (if first retry also unfulfilled)
t=36h:  Third VRF retry
...
t=168h: Community can propose (7-day stall)
```

The 12h retry gives the VRF coordinator TWO chances to fulfill before governance activates at 20h. With the old 18h timeout, there was only ONE retry opportunity before the 20h governance threshold (18h retry, then only 2h for fulfillment before governance activates). The change to 12h is a **net improvement** -- it provides more recovery opportunities before governance intervention is needed.

**2. No conflict with governance timing:**

- 12h retry < 20h governance threshold: The retry occurs before governance can activate. No collision.
- Multiple retries before 20h: At t=12h (first retry) and t=24h (second retry). Only the first occurs before governance activates. This is correct -- the system tries to self-heal before escalating to governance.
- No governance function depends on the retry timeout value. `DegenerusAdmin` uses `lastVrfProcessedTimestamp` (which is NOT updated by retry requests -- only by successful `_applyDailyRng` processing). The retry does not reset the stall timer.

**3. `lastVrfProcessedTimestamp` NOT updated by retry:**

The retry calls `_requestRng()` (line 787), which sends a new VRF request but does NOT update `lastVrfProcessedTimestamp`. The timestamp is only updated when `_applyDailyRng()` processes a fulfilled word (XCON-01 verdict). This means VRF retries do not interfere with governance stall detection. The stall timer keeps running regardless of retries.

**4. Boundary behavior at 12h:**

- At elapsed = 43199 (11h 59m 59s): `43199 >= 43200` is FALSE. `revert RngNotReady()`. Correct -- no retry yet.
- At elapsed = 43200 (exactly 12h): `43200 >= 43200` is TRUE. Retry triggers. Correct.

### Adversarial Check

**Attack 1: DoS via repeated retry requests**
- **Attempted:** An attacker triggers retries at 12h intervals to waste gas or create subscription cost issues.
- **Result:** `advanceGame()` is permissionless -- anyone can call it. The retry sends a VRF request (costs LINK from the subscription). However, each retry replaces the previous pending request (new `vrfRequestId`). Only one pending request exists at a time. The subscription LINK cost is bounded by the VRF coordinator's pricing. No amplification attack -- one request per 12h at most. The LINK cost is borne by the subscription, not the caller. NOT EXPLOITABLE for meaningful DoS.

**Attack 2: Exploit 12h retry to reset rngRequestTime and delay governance**
- **Attempted:** After retry, `rngRequestTime` is updated to the retry time. Could this delay governance activation?
- **Result:** Governance uses `lastVrfProcessedTimestamp` (not `rngRequestTime`). The retry updates `rngRequestTime` (which controls the next retry interval) but does NOT update `lastVrfProcessedTimestamp`. Governance timing is completely independent of retry timing. NOT EXPLOITABLE.

**Attack 3: Race between VRF fulfillment and governance execution**
- **Attempted:** VRF fulfills at t=19h (after retry at t=12h). A governance proposal was created at t=20h (in a previous stall). Does the fulfillment invalidate the proposal?
- **Result:** If VRF fulfills at t=19h, `_applyDailyRng()` runs and updates `lastVrfProcessedTimestamp` to the current time. The stall duration drops to ~0. All subsequent `vote()` calls revert with `NotStalled()`. The proposal remains Active but unvotable. This is the intended auto-invalidation mechanism (confirmed in GOV-03). If a proposal was already at the execute threshold, `canExecute()` (line 544-546) also checks the stall -- it returns FALSE when VRF recovers. The 12h retry timeout makes this recovery MORE likely (more retry chances), which is beneficial. NOT EXPLOITABLE.

### Tests

- `test/edge/RngStall.test.js` "advanceGame before 18-hour timeout reverts" describe block (lines 151-213):
  - Line 175: 6h elapsed reverts with `RngNotReady` (under 12h threshold)
  - Line 190: 11h 59m elapsed reverts with `RngNotReady` (just under 12h)
  - Line 198: 12h+ triggers retry (no revert) -- **confirms 12h timeout is active in code**
- `test/edge/RngStall.test.js` "18-hour timeout and retry" describe block (lines 66-149): Tests use 18h+1s in the old test name but the code correctly retries at 12h. The describe block name is a stale reference from the old 18h timeout (test name not updated, but the logic tests correctly).

Note: The describe block title "18-hour timeout and retry" is a stale test name. The actual code uses 12 hours. The contract passes `elapsed >= 12 hours` (line 786). The test at line 80 advances by `18 * 3600 + 1` which is greater than 12h, so the retry correctly triggers. The tests at lines 175-213 are the precise boundary tests that confirm the 12h threshold.

## WAR-01: Compromised Admin Key

**Verdict:** KNOWN-ISSUE
**Severity:** Medium (informational with governance mitigation)
**Exploit Feasibility:** Medium

### Attacker Profile

Holds >50.1% DGVE (vault governance token). Capabilities: can call any `onlyOwner` function on DegenerusAdmin, including `propose()` via the admin path. Goal: swap the VRF coordinator to a malicious one that produces predictable RNG, enabling manipulation of game outcomes (the original M-02 vector).

### Attack Path

1. VRF stalls (attacker causes coordinator to stop responding, or waits for natural outage)
2. At `t = lastVrfProcessedTimestamp + 72000` (20h stall): governance activates. Stall check in `propose()` line 408: `stall < ADMIN_STALL_THRESHOLD` is FALSE. Admin path is open.
3. Attacker calls `propose(maliciousCoordinator, maliciousKeyHash)`. Admin path requires only DGVE ownership (line 407: `vault.isVaultOwner(msg.sender)`), no sDGNRS stake needed to propose.
4. Proposal is created with `circulatingSnapshot = circulatingSupply()` (line 424). `activeProposalCount` increments (line 428).
5. **Critical distinction:** The admin path grants PROPOSE access via DGVE, but VOTE power requires sDGNRS (line 457-458: `uint256 weight = sDGNRS.balanceOf(msg.sender); if (weight == 0) revert InsufficientStake()`). DGVE and sDGNRS are completely separate tokens. Holding DGVE does NOT grant vote weight.
6. Attacker must acquire sDGNRS to self-approve. sDGNRS is soulbound (non-transferable), so it cannot be purchased on secondary markets. The attacker must have received sDGNRS through legitimate protocol participation (deity quest rewards, century bonuses, etc.).
7. If attacker holds ANY sDGNRS and waits for threshold decay:
   - Day 0-1: threshold = 6000 BPS (60%). Need 60% of circulatingSnapshot.
   - Day 5 (120h): threshold = 1000 BPS (10%).
   - Day 6 (144h): threshold = 500 BPS (5%).
   - Day 7 (168h): threshold = 0 BPS. ANY single approve vote with `approveWeight > rejectWeight` triggers execution.
8. At threshold 0 (day 7+), if no one has voted Reject, attacker votes Approve with minimal sDGNRS. Execute condition (line 488-490): `approveWeight * BPS >= 0 * circulatingSnapshot` is trivially TRUE. `approveWeight > rejectWeight` (where rejectWeight = 0) is TRUE. `_executeSwap` triggers.

### Defense Analysis

1. **DGVE/sDGNRS separation (GOV-02):** The admin can propose but cannot vote without sDGNRS. This is a critical design defense -- propose access and vote power are decoupled.
2. **7-day decay window:** The threshold starts at 60% and takes 7 full days to reach 0%. This gives the community a 7-day window to detect the malicious proposal and vote Reject.
3. **Community reject defense:** A single large sDGNRS holder voting Reject can block execution at any threshold level, as `approveWeight > rejectWeight` must be strict (line 490). Even a 1-wei reject advantage blocks.
4. **Auto-invalidation on VRF recovery (GOV-03):** If VRF resumes during the 7-day window, all `vote()` calls revert with `NotStalled()` (line 441). The proposal becomes unvotable and expires harmlessly after 168h.
5. **Death clock pause (XCON-02):** While proposals are Active, `anyProposalActive()` returns TRUE, pausing the death clock. The game is protected from liveness failure during governance.
6. **Soulbound sDGNRS (VOTE-01):** Attacker cannot buy sDGNRS on open market. Must have earned it through protocol participation. Limits attacker's ability to accumulate vote weight.

### Assessment

The attack is feasible but requires a convergence of conditions: (1) DGVE key compromise, (2) attacker holds some sDGNRS, (3) community is inattentive for 7 full days during a VRF stall. The 7-day decay window is the primary defense -- it provides ample time for community detection and response. The worst case (threshold 0 at day 7) requires sustained community absence. Rated Medium because a successful attack grants RNG control (the original M-02 impact), but the governance system transforms a single-key risk into a time-gated community-override-able process. Cross-references: GOV-02 (propose access control), GOV-04 (threshold decay), VOTE-01 (sDGNRS frozen invariant), XCON-02 (death clock pause).

## WAR-02: Colluding Voter Cartel at Low Threshold

**Verdict:** KNOWN-ISSUE
**Severity:** Medium
**Exploit Feasibility:** Medium (depends on sDGNRS distribution)

### Attacker Profile

Small group of colluding sDGNRS holders controlling >= 5% of circulating sDGNRS supply. Goal: approve a malicious VRF coordinator swap at a low threshold day to gain RNG control. May or may not have DGVE (admin access).

### Attack Path

1. VRF stalls. Either naturally or induced by a colluding VRF coordinator operator.
2. **Proposal creation:**
   - If cartel includes admin (DGVE holder): propose at t=20h (admin path, GOV-02).
   - If community-only: wait until t=168h (7-day community stall threshold). Any cartel member with >= 0.5% sDGNRS can propose (line 412-414: `sDGNRS.balanceOf(msg.sender) * BPS >= circ * COMMUNITY_PROPOSE_BPS`).
3. Wait for threshold to decay. Target: day 6 (144h elapsed from proposal creation).
4. At day 6: threshold = 500 BPS (5%). Execute condition requires:
   - `approveWeight * 10000 >= 500 * circulatingSnapshot` => `approveWeight >= 5% of circulatingSnapshot` (GOV-05).
   - `approveWeight > rejectWeight` (strict inequality, GOV-06).
5. Cartel coordinates off-chain. All members vote Approve in rapid succession. Combined weight >= 5% of snapshot.
6. If no Reject votes exist, the last vote that meets the threshold triggers `_executeSwap`.

### Defense Analysis

1. **5% threshold at day 6 (GOV-04):** The 500 BPS threshold at 144h requires meaningful stake concentration. In a protocol with wide sDGNRS distribution, 5% may require many colluding parties.
2. **Single-voter Reject block:** Any single sDGNRS holder voting Reject with weight >= cartel's approve weight blocks execution. The `approveWeight > rejectWeight` strict inequality means Reject ties block execution.
3. **Snapshot anchoring (VOTE-02):** `circulatingSnapshot` is set at proposal creation (line 424) and is immutable. Cartel cannot inflate their proportion by manipulating supply post-proposal.
4. **sDGNRS is soulbound (VOTE-01):** Cannot accumulate vote weight by purchasing on secondary markets. All sDGNRS was earned through protocol participation. This limits artificial stake concentration.
5. **6-day delay:** The cartel must wait 6 days from proposal creation before the 5% threshold becomes available. This is 6 days of community detection time.
6. **Multi-proposal defense:** Honest community members can create competing proposals. If multiple proposals exist, votes are split across them, but `_voidAllActive` kills all others when any one executes (GOV-08).

### Assessment

Exploitability depends on sDGNRS distribution. In a protocol with concentrated holdings (a few whales), 5% is easily achievable. In widely distributed protocols, collusion at 5% requires coordinating many participants. The attack requires: (1) VRF stall lasting at least 6+ days from proposal creation, (2) cartel holds 5%+ sDGNRS, (3) no honest voter with sufficient weight votes Reject within the 6-day window. The 60% initial threshold provides strong early defense; the risk concentrates at late-stage decay (days 6-7). This is an inherent tension in decaying-threshold governance: lower thresholds enable recovery when turnout is low, but also enable minority capture. Severity is Medium because successful exploitation grants RNG control, but requires multi-day stall + stake concentration + voter apathy. Cross-references: GOV-04 (threshold decay schedule), GOV-05 (execute condition), GOV-06 (kill condition), VOTE-01 (sDGNRS frozen), VOTE-02 (snapshot immutability).

## WAR-03: VRF Oscillation Attack

**Verdict:** PASS
**Severity:** Low
**Exploit Feasibility:** Low-Medium

### Attacker Profile

Controls the current VRF coordinator (or can cause intermittent VRF failures). Goal: create permanent governance DoS by cycling stall -> governance activation -> VRF recovery -> auto-invalidation -> repeat, preventing any proposal from completing.

### Attack Path

1. **Cycle 1 start:** Attacker stops VRF coordinator response. `lastVrfProcessedTimestamp` is frozen.
2. At t=20h: governance activates. Admin (or community at t=168h) creates Proposal A.
3. `activeProposalCount` increments. Death clock is paused (XCON-02: `anyProposalActive()` returns TRUE).
4. Community begins voting on Proposal A.
5. **Oscillation trigger:** Attacker resumes VRF coordinator. `_applyDailyRng()` runs via `rawFulfillRandomWords`, updating `lastVrfProcessedTimestamp = uint48(block.timestamp)` (line 1374 of AdvanceModule).
6. **Auto-invalidation kicks in:** All subsequent `vote()` calls hit the stall re-check (line 441): `block.timestamp - lastVrfProcessedTimestamp < ADMIN_STALL_THRESHOLD` is now TRUE. Votes revert with `NotStalled()`. This is the intended auto-invalidation mechanism -- VRF recovery invalidates governance because the emergency is over.
7. **Proposal A remains Active:** It is not killed by VRF recovery. It cannot be voted on (votes revert), but its state is still `Active`. It will remain Active until it expires at `createdAt + 168h` (lazy expiry via vote(), GOV-09).
8. `activeProposalCount` remains > 0 because Proposal A is still Active. Death clock stays paused.
9. **Cycle 2 start:** Attacker stops VRF again. After another 20h, governance reactivates. New proposals can be created. Old Proposal A is still Active and can be voted on again (if stall re-check passes).
10. Repeat. Each cycle: ~20h stall + brief recovery + stall again. Minimum cycle duration ~24h.

### Defense Analysis

1. **Proposals expire at 168h regardless (GOV-09):** Each proposal has a hard lifetime of 168 hours. After expiry, `vote()` triggers lazy cleanup (lines 449-453): state set to Expired, `activeProposalCount--`. Oscillation delays governance but cannot prevent expiry.
2. **New proposals on each stall:** Each time VRF stalls for 20h+, new proposals can be created. The attacker must maintain oscillation indefinitely to prevent any proposal from completing.
3. **Voting window within stall:** During each stall period (before VRF recovery), votes CAN be cast. If the community acts quickly enough within a stall window (before the attacker resumes VRF), a proposal can reach the execute threshold and trigger `_executeSwap` in a single voting session.
4. **Death clock protection (XCON-02):** While proposals are Active (even during oscillation), `anyProposalActive()` returns TRUE, pausing the death clock. The game is protected from liveness failure during oscillation.
5. **Circular dependency:** The attacker needs VRF coordinator control to oscillate, but governance exists precisely to replace the coordinator. If the attacker keeps oscillating, they are confirming the coordinator is compromised -- which is exactly why governance should succeed.
6. **Cost to attacker:** Each VRF fulfillment (recovery) costs LINK from the subscription. The attacker must selectively fulfill to cause oscillation, which may be detectable by the community.

### Assessment

The VRF oscillation attack degrades governance throughput but cannot permanently prevent governance from completing. Key insight: the attacker must resume VRF to trigger auto-invalidation, but each stall period provides a voting window. If the community coordinates rapid voting during a stall window, a proposal can execute within a single stall cycle. The death clock pause via `anyProposalActive()` ensures the game survives oscillation. The attack has a circular dependency: the attacker controls the VRF coordinator (which is what governance is trying to replace). Severity is Low because governance is degraded but not defeated, and the death clock protection preserves game integrity. Cross-references: XCON-01 (`lastVrfProcessedTimestamp` write paths), XCON-02 (death clock pause), GOV-09 (proposal expiry), GOV-03 (auto-invalidation via stall re-check).
