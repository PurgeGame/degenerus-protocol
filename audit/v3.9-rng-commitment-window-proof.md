## 1. Scope Statement

**Requirement:** RNG-01 -- No permissionless action during the VRF commitment window can influence which player wins a far-future coin jackpot draw.

**Scope:** This proof covers the `_awardFarFutureCoinJackpot` function (JM:2522-2614) which selects winners from a combined pool of read-buffer and far-future key ticket queues. It extends the v3.8 commitment window audit (CW-03 Category 3: Jackpot BURNIE Winner) with the new FF key data source introduced in Phases 74-77.

**Function under analysis:** `_awardFarFutureCoinJackpot(uint24 lvl, uint256 farBudget, uint256 rngWord)` in `DegenerusGameJackpotModule.sol` (JM:2522-2614).

**Callers (exhaustive):**

| Caller | Location | Call Site |
|--------|----------|-----------|
| `payDailyJackpotCoinAndTickets` | DegenerusGameJackpotModule.sol | JM:707 |
| `payDailyCoinJackpot` | DegenerusGameJackpotModule.sol | JM:2370 |

Both callers are invoked exclusively via the daily `advanceGame` flow:
- `payDailyJackpotCoinAndTickets` is called from `advanceGame` at AM:363 when `dailyJackpotCoinTicketsPending` is true (jackpot phase coin+ticket distribution).
- `payDailyCoinJackpot` is called via `_payDailyCoinJackpot` delegatecall at AM:283 during purchase-phase daily processing.

Both callers execute after `rngGate` (AM:225) has consumed the VRF word, at which point `rngLockedFlag` is true (set at AM:1328 by `_finalizeRngRequest`).

**Mid-day lootbox path (Pitfall 1):** The mid-day `requestLootboxRng` path (AM:677-744) does NOT call `_awardFarFutureCoinJackpot`. That function is only reachable via the daily advanceGame path where `rngLockedFlag` IS true. The mid-day path requests its own VRF word for lootbox resolution only and does not trigger any jackpot draw. Therefore the mid-day VRF commitment window is irrelevant to this proof.

## 2. Backward Trace

Tracing backward from winner selection to every input that influences the outcome:

```
Winner address
  <- readQueue[idx] OR ffQueue[idx - readLen]              [JM:2553-2555]
    <- idx = (entropy >> 32) % combinedLen                  [JM:2552]
      <- entropy = entropyStep(prevEntropy ^ s)             [JM:2539]
        <- initial entropy = rngWord ^ (lvl << 192) ^ FAR_FUTURE_COIN_TAG   [JM:2529-2531]
          <- rngWord: VRF word from advanceGame (parameter)
          <- lvl: current level (parameter, set at AM:1340)
          <- FAR_FUTURE_COIN_TAG: compile-time constant
      <- combinedLen = readLen + ffLen                      [JM:2549]
        <- readLen = ticketQueue[_tqReadKey(candidate)].length    [JM:2545-2546]
          <- _tqReadKey depends on ticketWriteSlot               [GS:721-722]
        <- ffLen = ticketQueue[_tqFarFutureKey(candidate)].length [JM:2547-2548]
          <- _tqFarFutureKey is pure (bit 22 OR, no storage)     [GS:729-730]
    <- candidate = lvl + 5 + (entropy % 95)                 [JM:2542]
      <- lvl (current level, parameter)
      <- entropy (PRNG chain seeded from VRF word)

  readQueue = ticketQueue[_tqReadKey(candidate)]             [JM:2545]
    <- _tqReadKey: ticketWriteSlot == 0 ? lvl | TICKET_SLOT_BIT : lvl   [GS:721-722]
    <- population committed by _queueTickets callers to WRITE buffer before swap

  ffQueue = ticketQueue[_tqFarFutureKey(candidate)]          [JM:2547]
    <- _tqFarFutureKey: lvl | TICKET_FAR_FUTURE_BIT                     [GS:729-730]
    <- population committed by _queueTickets callers to FF key space
```

Every node has been verified against source. The backward trace shows 7 input variables that influence winner selection.

## 4. Mutation Surface

### Path 1: purchase() -> _purchaseFor -> _queueTickets

**External function:** `purchase()` in MintModule
**Call chain:** `purchase` -> `_purchaseFor` -> `_queueTickets` (GS:537-565)
**Target variable:** `ticketQueue[_tqWriteKey(targetLevel)]` (near-future) or `ticketQueue[_tqFarFutureKey(targetLevel)]` (far-future)
**Protection:**
  1. If `targetLevel <= level + 6` -> routes to `_tqWriteKey(targetLevel)` (GS:546) -> writes to WRITE buffer, not READ buffer -> **IRRELEVANT** to `_awardFarFutureCoinJackpot` which reads from `_tqReadKey` (JM:2545)

     - `phaseTransitionActive = false` for external callers (only set within advanceGame at AM:239)
     - Result: **REVERTS** -- cannot write to FF key during commitment window
**Verdict: SAFE** -- Near-future writes go to write buffer (invisible to jackpot). Far-future writes revert with `RngLocked()` during commitment window.
**Evidence:** GS:544-546, AM:1328

### Path 8: advanceGame() (auto-rebuy + phase transition) -- SAME TRANSACTION

This path requires special treatment because `advanceGame` writes to `ticketQueue` AND consumes the VRF word in the same transaction. There are three sub-paths:

#### Path 8a: _processAutoRebuy -> _queueTickets

**Call chain:** `advanceGame` -> `payDailyJackpot` -> `_addClaimableEth` (JM:957-978) -> `_processAutoRebuy` (JM:988-1028) -> `_queueTickets` (JM:1008)
**Target variable:** `ticketQueue[_tqWriteKey(calc.targetLevel)]` or `ticketQueue[_tqFarFutureKey(calc.targetLevel)]`
**Auto-rebuy target level:** `currentLevel + ((entropy & 3) + 1)` = `level + 1` to `level + 4` (PayoutUtils:54-58). Since `level + 4 <= level + 6`, auto-rebuy targets are always near-future and route to `_tqWriteKey` (GS:546). Even if edge conditions pushed the target beyond `level + 6` (which they cannot given the `(entropy & 3) + 1` range of 1-4), the write would be:
  - Guarded by `rngLockedFlag` at GS:544-545, but exempted because execution is within `advanceGame` where `phaseTransitionActive` may be true
  - **Atomic with the jackpot draw** -- auto-rebuy and `_awardFarFutureCoinJackpot` execute in the same EVM transaction. No external actor can insert a transaction between auto-rebuy's writes and the jackpot's reads.

**Pitfall 4 addressed:** Auto-rebuy targets `level + 1` to `level + 4` (PayoutUtils:54-58), which are all near-future (`<= level + 6`). They route to `_tqWriteKey`, not `_tqFarFutureKey`. The write buffer is invisible to `_awardFarFutureCoinJackpot` which reads from `_tqReadKey` (JM:2545).

**Verdict: SAFE** -- Auto-rebuy writes target near-future levels (level+1 to level+4) which route to the write buffer. Even if they targeted FF levels, they would be atomic with the jackpot draw (same transaction). No external actor can exploit this.
**Evidence:** PayoutUtils:54-58, JM:1008, GS:544-546

#### Path 8b: _processPhaseTransition -> _queueTickets

**Call chain:** `advanceGame` -> `_processPhaseTransition` (AM:1222-1236) -> `_queueTickets` (AM:1227-1235)
**Target variable:** `ticketQueue[_tqFarFutureKey(purchaseLevel + 99)]`
**Target level:** `purchaseLevel + 99` (AM:1226). Since `purchaseLevel = level + 1`, the target is `level + 100`, which is always far-future (`> level + 6`).
**Protection:**
  - `phaseTransitionActive = true` when `_processPhaseTransition` runs (set at AM:239, cleared at AM:244)
  - The `rngLockedFlag` guard at GS:544-545 checks `!phaseTransitionActive` -- when `phaseTransitionActive = true`, the guard does NOT revert, allowing the write.
  - However, this write is:
    1. **Deterministic:** Fixed addresses (`ContractAddresses.SDGNRS`, `ContractAddresses.VAULT`), fixed quantity (`VAULT_PERPETUAL_TICKETS` = 16), computed target level (`purchaseLevel + 99`). No player-controllable inputs.
    2. **Same transaction:** Executes atomically within `advanceGame`. No external actor can influence the write or intervene between the write and the jackpot draw.
    3. **Different target level than jackpot candidates:** The jackpot samples from `[lvl+5, lvl+99]` (JM:2542). The phase transition writes to `purchaseLevel + 99 = level + 100`, which is OUTSIDE the candidate range. These tickets are not eligible for the current jackpot draw.

**Verdict: SAFE** -- Phase transition writes are deterministic (fixed addresses, fixed quantities), atomic (same transaction), and target `level + 100` which is outside the jackpot's candidate range `[lvl+5, lvl+99]`.
**Evidence:** AM:1222-1236, AM:239-244, GS:544-545, JM:2542

#### Path 8c: processTicketBatch -> ticketQueue drain

**Call chain:** `advanceGame` -> `_runProcessTicketBatch` (AM:269) -> `processTicketBatch` (JM:1890-1951) via delegatecall
**Target variable:** `ticketQueue[_tqReadKey(lvl)]` (READ slot, current level)
**Effect:** Drains (deletes) entries from `ticketQueue[rk]` where `rk = _tqReadKey(lvl)` and `lvl` is the current purchase level (AM:270: `purchaseLevel`).

**Pitfall 5 addressed:** `processTicketBatch` drains the current purchase level's read-side queue only. It is called with `purchaseLevel` (= `level + 1`). The candidate levels sampled by `_awardFarFutureCoinJackpot` are `[lvl+5, lvl+99]` (JM:2542), which does not overlap with `purchaseLevel` (= `lvl + 1` = `level + 1`). Therefore `processTicketBatch` does not modify any queue that `_awardFarFutureCoinJackpot` reads from.

**Verdict: SAFE** -- `processTicketBatch` drains level `purchaseLevel` (= `level + 1`). The jackpot reads from levels `[level+5, level+99]`. No overlap.
**Evidence:** JM:1890-1891, AM:269-271, JM:2542

### Path 8d: processFutureTicketBatch -> ticketQueue drain

**Call chain:** `advanceGame` -> `_prepareFutureTickets` (AM:1156-1184) -> `_processFutureTicketBatch` (AM:1134-1148) -> `processFutureTicketBatch` (MintModule:298-339) via delegatecall
**Target variable:** `ticketQueue[_tqReadKey(target)]` and `ticketQueue[_tqFarFutureKey(target)]` for levels `[lvl+2, lvl+6]`
**Effect:** Drains read-side queue then FF key entries for each level in `[lvl+2, lvl+6]` where `lvl` is the current level. This moves tickets from queues into trait assignments.

**Open Question 1 resolution:** The advanceGame call order is:
1. `_prepareFutureTickets(lvl)` at AM:262 -- processes levels `[lvl+2, lvl+6]` (AM:1157-1158)
2. `_runProcessTicketBatch(purchaseLevel)` at AM:269 -- processes current purchase level
3. `payDailyJackpot` / `_payDailyCoinJackpot` at AM:282-283 -- calls `_awardFarFutureCoinJackpot`

`processFutureTicketBatch` runs BEFORE the jackpot functions. It processes levels `[lvl+2, lvl+6]`. The jackpot reads from levels `[lvl+5, lvl+99]`. There is partial overlap at `[lvl+5, lvl+6]`.

However, this is NOT a vulnerability because:
1. `processFutureTicketBatch` drains entries from `_tqReadKey(target)` AND `_tqFarFutureKey(target)` into trait assignments. After draining, those queues are empty.
2. If levels `lvl+5` or `lvl+6` are drained by `processFutureTicketBatch`, then `_awardFarFutureCoinJackpot` will find `combinedLen == 0` for those levels (JM:2551) and skip them. This means fewer candidate levels have tickets, but the outcome is still deterministic based on the VRF word.
3. The drain happens BEFORE the jackpot draw, in the same transaction. The population at jackpot time is the post-drain population, which is the deterministic result of the advanceGame flow. No external actor can influence whether a level has been drained.

**Verdict: SAFE** -- `processFutureTicketBatch` drains levels `[lvl+2, lvl+6]` before the jackpot samples `[lvl+5, lvl+99]`. The overlap at `[lvl+5, lvl+6]` is handled: drained levels yield `combinedLen == 0` and are skipped. The drain is deterministic and atomic within the same transaction.
**Evidence:** AM:262, AM:1157-1158, MintModule:298-339, JM:2549-2551

At the level transition boundary, `processFutureTicketBatch` is also called with `nextLevel = purchaseLevel + 1` (AM:300-305). This processes a single level that is `level + 2`. Since the jackpot at level transition uses the current level (not nextLevel), the candidate range `[lvl+5, lvl+99]` starts at `level + 5`, which does not include `level + 2`. No overlap.

### Additional path: requestLootboxRng (mid-day buffer swap)

**External function:** `requestLootboxRng()` in AdvanceModule (AM:677-744)
**Potential concern (Pitfall 3):** Could a mid-day `requestLootboxRng` call swap the ticket buffer (`_swapTicketSlot` at AM:720), changing which buffer `_tqReadKey` points to during the daily commitment window?

**Protection:** `requestLootboxRng` checks `rngLockedFlag` at its entry point (AM:678):
```
if (rngLockedFlag) revert RngLocked();
```

During the daily commitment window, `rngLockedFlag = true` (set at AM:1328). Therefore `requestLootboxRng` reverts immediately. No mid-day buffer swap can occur while the daily RNG is locked. `ticketWriteSlot` is stable throughout the daily commitment window.

**Verdict: SAFE** -- `requestLootboxRng` is blocked by `rngLockedFlag` during the daily commitment window. No buffer swap can occur.
**Evidence:** AM:678, AM:1328

## 5. Combined Pool Length Invariant

**Claim:** `combinedLen = readLen + ffLen` (JM:2549) at the jackpot draw cannot differ from what it was at VRF request time for any candidate level, when considering only external (non-advanceGame) actors.

### readLen stability

`readLen = ticketQueue[_tqReadKey(candidate)].length` (JM:2546)

- `_tqReadKey` depends on `ticketWriteSlot` (GS:721-722), which is swapped by `_swapAndFreeze` at RNG request time (AM:233) and not modified again until `_unlockRng` (AM:1412-1414).
- No external function writes to the read key. All `_queueTickets`/`_queueTicketsScaled`/`_queueTicketRange` calls route near-future writes to `_tqWriteKey` (the opposite buffer). External callers cannot push to `ticketQueue[_tqReadKey(candidate)]`.
- `processTicketBatch` (JM:1890-1951) drains `ticketQueue[_tqReadKey(lvl)]` for the current level (`purchaseLevel`). The candidate levels `[lvl+5, lvl+99]` do not include `purchaseLevel` (= `level + 1`). No overlap.
- `processFutureTicketBatch` (MintModule:298-339) drains `ticketQueue[_tqReadKey(target)]` for levels `[lvl+2, lvl+6]`. The overlap with candidate range at `[lvl+5, lvl+6]` is handled: drains occur BEFORE the jackpot draw, atomically in the same transaction (see Path 8d above). The resulting readLen at jackpot time is deterministic.

**Conclusion:** `readLen` is stable between VRF request and consumption for all external actors.

### ffLen stability

`ffLen = ticketQueue[_tqFarFutureKey(candidate)].length` (JM:2548)

- `_tqFarFutureKey` is a pure function (GS:729-730) with no storage dependency. It always returns `lvl | TICKET_FAR_FUTURE_BIT`.
- `rngLockedFlag` guard (GS:544-545, GS:579-580, GS:641-642) blocks ALL permissionless far-future ticket writes during the commitment window. Every path through `_queueTickets`, `_queueTicketsScaled`, and `_queueTicketRange` checks `isFarFuture && rngLockedFlag && !phaseTransitionActive` before writing.
- The `phaseTransitionActive` exemption:
  - `_processPhaseTransition` writes to `purchaseLevel + 99 = level + 100`, which is outside the candidate range `[lvl+5, lvl+99]`. Does not affect `ffLen` for any sampled candidate level.
  - Auto-rebuy targets `level + 1` to `level + 4` (PayoutUtils:54-58), which are near-future and route to `_tqWriteKey`, not `_tqFarFutureKey`. Does not affect `ffLen`.
- `processFutureTicketBatch` drains `ticketQueue[_tqFarFutureKey(target)]` for levels `[lvl+2, lvl+6]`. The overlap with candidate range at `[lvl+5, lvl+6]` is handled: drains are deterministic and atomic (same transaction).

**Conclusion:** `ffLen` is stable between VRF request and consumption for all external actors.

### Open Question 2 resolution: auto-rebuy target level range

Auto-rebuy computes `targetLevel = currentLevel + ((entropy & 3) + 1)` (PayoutUtils:54-58). The range is `level + 1` to `level + 4`. Since `level + 4 <= level + 6`, auto-rebuy targets are always classified as near-future (`targetLevel <= level + 6`), routing to `_tqWriteKey`. Auto-rebuy never writes to the FF key space. Even if it did, the write would be atomic with the jackpot draw (same advanceGame transaction), making exploitation impossible.

### Combined invariant

```
combinedLen = readLen + ffLen
  readLen: stable (double-buffer swap freezes read buffer; no external writes)
  ffLen: stable (rngLockedFlag blocks all external FF writes; exempted writes target
         level+100 which is outside candidate range [lvl+5, lvl+99])
  -> combinedLen is stable between VRF request and consumption for all external actors
```

Intra-transaction modifications by `processFutureTicketBatch` and `_processPhaseTransition` are deterministic and cannot be influenced by an external attacker.

## 7. Summary Verdict

**RNG-01: SAFE**

No permissionless action during the VRF commitment window can influence which player wins a far-future coin jackpot draw.

**Three independent protection layers:**

1. **Double-buffer swap** (Layer 1): `_swapAndFreeze` (AM:233) flips `ticketWriteSlot` at RNG request time. All subsequent `purchase()`/`purchaseCoin()` writes go to the NEW write buffer. `_awardFarFutureCoinJackpot` reads from `_tqReadKey(candidate)` (JM:2545) which points to the OLD (frozen) buffer. The read-side population was committed before the VRF request.

2. **rngLockedFlag guard** (Layer 2): `rngLockedFlag = true` is set by `_finalizeRngRequest` (AM:1328) and cleared by `_unlockRng` (AM:1414).
