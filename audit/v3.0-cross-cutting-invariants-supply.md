## INV-03: sDGNRS Supply Conservation

### Formal Invariant

```
For all states S reachable from the constructor:
  sDGNRS.totalSupply == SUM(sDGNRS.balanceOf[addr]) for all addr in ADDRESS_SPACE
```

### Re-Verification of NOVEL-05 Proof

The NOVEL-05 formal proof in `audit/novel-03-invariants-privilege.md` (Invariant 1) proves this invariant by exhaustive enumeration of six modification paths:

1. **`_mint(to, amount)`** (constructor only, line 510-517): `totalSupply += amount`, `balanceOf[to] += amount`. Net effect = 0.
2. **`burn(amount)`** (lines 379-441 in original; lines 373-435 in current): `balanceOf[player] -= amount`, `totalSupply -= amount`. Net effect = 0.
3. **`burnRemainingPools()`** (lines 359-367 in original; lines 352-361 in current): `balanceOf[address(this)] = 0`, `totalSupply -= bal`. Net effect = 0.
4. **`wrapperTransferTo(to, amount)`** (lines 242-252 in current): `balanceOf[DGNRS] -= amount`, `balanceOf[to] += amount`. `totalSupply` unchanged. Net effect on SUM = 0.
5. **`transferFromPool(pool, to, amount)`** (lines 308-325 in current): `balanceOf[address(this)] -= amount`, `balanceOf[to] += amount`. `totalSupply` unchanged. Net effect on SUM = 0.
6. **`transferBetweenPools(from, to, amount)`** (lines 333-348 in current): Only `poolBalances[]` entries modified. No `balanceOf` or `totalSupply` change. Trivially preserved.

The NOVEL-05 proof is formally sound. All paths either preserve `totalSupply` and zero-sum transfer balances, or decrease both `totalSupply` and `SUM(balanceOf)` by the same amount.

### INV-03 Formal Proof Summary

**For each path, proof that `totalSupply_after - totalSupply_before == SUM(balanceOf_after) - SUM(balanceOf_before)`:**

- **P1 (_mint):** Both sides increase by `amount`. `amount - amount = 0`.
- **P2 (burn):** Both sides decrease by `amount`. `(-amount) - (-amount) = 0`.
- **P3 (burnRemainingPools):** `totalSupply` decreases by `bal`. `SUM(balanceOf)` decreases by `bal` (via `balanceOf[address(this)] = 0`). `(-bal) - (-bal) = 0`.
- **P4 (wrapperTransferTo):** `totalSupply` unchanged. `SUM(balanceOf)` unchanged (DGNRS loses `amount`, `to` gains `amount`, net 0). `0 - 0 = 0`.
- **P5 (transferFromPool):** `totalSupply` unchanged. `SUM(balanceOf)` unchanged (`address(this)` loses, `to` gains). `0 - 0 = 0`.
- **P6 (transferBetweenPools):** Neither `totalSupply` nor any `balanceOf` changes. `0 - 0 = 0`.

**Observation on `_mint` call scope:** `_mint` is `private` (line 504) and called in exactly two places: the constructor at lines 212 and 213. Post-construction minting is not possible by any address. This is confirmed by the absence of any public or external mint function.

## INV-04: BURNIE Mint/Burn Lifecycle

### Formal Invariant

```
BURNIE token operations are internally consistent:
  (a) No path creates unbacked BURNIE (minting without a corresponding game event or claim)
  (b) No path destroys BURNIE that was tracked in a player's virtual stake balance
  (c) Physical BURNIE totalSupply = initial_supply (0) + SUM(all mints) - SUM(all burns)
  (d) Virtual stakes are accounted separately; they affect physical supply only at claim time
```

### BURNIE Supply Architecture

`BurnieCoin.sol` uses a packed `Supply` struct:
```solidity
Supply private _supply = Supply({totalSupply: 0, vaultAllowance: uint128(2_000_000 ether)});
```

The **critical invariant documented in BurnieCoin.sol** (line 17 of contract header):
```
totalSupply + vaultAllowance = supplyIncUncirculated
```

Initial state: `totalSupply = 0`, `vaultAllowance = 2,000,000 BURNIE`, `supplyIncUncirculated = 2,000,000 BURNIE`.

### Enumeration: ALL BURNIE Mint Paths

**Mint paths that increase `_supply.totalSupply`:**

|---|----------|------|-------|--------|--------|---------|
| M1 | `mintForCoinflip(to, amount)` | BurnieCoin.sol | 527-530 | COINFLIP contract only | Claimed coinflip winnings | Player wins flip day and claims |
| M2 | `mintForGame(to, amount)` | BurnieCoin.sol | 532-540 | GAME contract only | Game-determined amount | Degenerette wins / game payouts |
| M3 | `creditCoin(player, amount)` | BurnieCoin.sol | 546-549 | GAME or AFFILIATE (onlyFlipCreditors) | As specified | Game/affiliate rewards |
| M4 | `vaultMintTo(to, amount)` | BurnieCoin.sol | 695-707 | VAULT only | From vaultAllowance | Vault distributes to players |

**Vault allowance paths (virtual, not yet in totalSupply):**

**Key insight about vault paths:** `V3` and `V4` do NOT increase physical supply; they shift from circulating to non-circulating reserve. The `supplyIncUncirculated = totalSupply + vaultAllowance` invariant is preserved because `V3` decreases one and increases the other by the same amount.

**Access control for all mint paths:**

- `mintForGame` (M2): `msg.sender != ContractAddresses.GAME` check (BurnieCoin.sol:537). Only GAME.

- `vaultMintTo` (M4): `onlyVault` modifier -- VAULT only (line 655-658).
- `creditLinkReward` -- calls `creditFlip` (virtual, not mint), not a direct mint.

**No admin mint path exists.** The ADMIN can only call `creditLinkReward` which routes to `creditFlip` (virtual stake credit in `BurnieCoinflip`, no physical token minting).

- Otherwise: reduces `balanceOf[from]` and `_supply.totalSupply` (standard ERC20 burn).

**Vault special case for burns:** When VAULT burns its allowance, `totalSupply` is NOT decreased (since it was never circulating). This preserves the `supplyIncUncirculated` invariant.

### Virtual Stake Ledger (BurnieCoinflip)

**The virtual stake system is the bridge between burns and mints:**

```
Player deposits BURNIE ->
  BurnieCoinflip.depositCoinflip() ->
    burnie.burnForCoinflip(caller, amount) [B1 -- physical BURNIE burned]
    coinflipBalance[nextDay][caller] += creditedFlip [virtual stake recorded]
```

```
VRF resolves day ->
  BurnieCoinflip.processCoinflipPayouts() ->
    coinflipDayResult[day] = {rewardPercent, win/loss}

Player claims ->
  BurnieCoinflip._claimCoinflipsInternal(player) ->
    reads coinflipBalance[day][player] [virtual stake retrieved]
    if win: mintable += principal + (principal * rewardPercent/100)
    if loss: coinflipBalance[day][player] = 0 [stake forfeited, no mint]

  BurnieCoinflip._claimCoinflipsAmount(player, amount, mintTokens=true) ->
    burnie.mintForCoinflip(player, toClaim) [M1 -- physical BURNIE minted for wins]
```

**Lifecycle consistency proof:**

1. **No double-claim:** `coinflipBalance[cursor][player] = 0` is set at line 522 of `BurnieCoinflip.sol` in `_claimCoinflipsInternal` when processing a day, **whether win or loss**. This happens inside `_claimCoinflipsInternal` which is called before any mint. The stake is cleared before minting. A second call to `_claimCoinflipsInternal` for the same day returns 0 stake, no re-mint.

2. **Claim window enforcement:** `minClaimableDay` is calculated at BurnieCoinflip.sol:466-483 based on `latest - windowDays`. Days before this window are skipped. Combined with `state.lastClaim` tracking (line 614), each day is processed at most once per player.

3. **`claimableStored` accounting:** After `_claimCoinflipsInternal` runs, unclaimed mintable is stored in `state.claimableStored` (line 257-258 of BurnieCoinflip.sol). When `_claimCoinflipsAmount` runs, it deducts from `stored` before minting (line 414). The deduction and the mint are both bounded by `stored`, preventing mint of more than what was earned.

4. **Auto-rebuy carry accounting:** `state.autoRebuyCarry` tracks the roll-over balance. On loss days (`if (win) {} else { carry = 0 }` at BurnieCoinflip.sol:567), carry is zeroed -- the principal (original physical burn + carried roll) is forfeited with no mint. On win days, carry is grown by winnings. When claim occurs, `mintable += reserved` where `reserved = (payout / takeProfit) * takeProfit`, bounded by `payout`. No over-mint is possible.

### BURNIE Lifecycle Closure Proof

**Net flows:**

| Event | Physical Supply Change | Virtual Stake Change |
|-------|----------------------|---------------------|
| Player deposits to flip | -amount (B1) | +creditedFlip (virtual) |
| Win day resolved | 0 | win: payout computed in state |
| Loss day resolved | 0 | loss: carry = 0, stake cleared |
| Player claims win | +toClaim (M1) | -toClaim from stored |
| Player forfeits (window expired) | 0 | stake pruned (minClaimableDay skip) |
| Decimator burn | -amount (B3) | 0 |
| Game burn | -amount (B2) | 0 |
| Vault escrow transfer | -amount from totalSupply, +amount to vaultAllowance | 0 |
| Vault mints to player | +amount from vaultAllowance to totalSupply | 0 |

**House edge mechanism:** The coinflip house edge is ~1.575% (verified in Phase 27, PAY-07). On average, `SUM(win claims) < SUM(deposits)`. This means physical supply decreases net across the coinflip economy, making BURNIE **deflationary** via the coinflip system.

**"Credits from nothing" check:**
- `creditFlip` (via `coin.creditFlip -> IBurnieCoinflip.creditFlip`) does NOT mint physical tokens. It only calls `_addDailyFlip` (BurnieCoinflip.sol:632+) which writes to `coinflipBalance[nextDay][player]`. Physical minting only happens when the player later claims. If they never claim, no physical tokens are created.
- `creditCoin` (M3) does mint physical BURNIE directly. This is gated to GAME and AFFILIATE. Every `creditCoin` call in the codebase is for a specific game event (affiliate reward delivery). No free-credit path exists.

**Verification that `mintForCoinflip` is only called from claim paths:**

`mintForCoinflip` is only callable by the COINFLIP contract (BurnieCoin.sol:519). Inside `BurnieCoinflip`, `burnie.mintForCoinflip(player, toClaim)` is called in exactly two places:
1. `_claimCoinflipsAmount` (line 419): only when `toClaim != 0` and `mintTokens == true`, after deducting from `stored`.
2. `_claimCoinflipsTakeProfit` (line 394): only after computing `toClaim = claimMultiples * takeProfit`, bounded by `stored`.

Both paths deduct from `state.claimableStored` before minting, preventing re-mint.

### INV-04 Verdict

**PASS**

The BURNIE mint/burn lifecycle is internally consistent across all paths:

1. **Physical BURNIE supply** is correctly maintained: `totalSupply = SUM(balanceOf)` (standard ERC20 invariant). Every mint increases both sides equally, every burn decreases both sides equally, every transfer is zero-sum.

2. **Virtual stake ledger** is self-consistent: stakes are cleared on processing (win or loss), claim window prevents re-processing of expired days, `claimableStored` accounting prevents minting more than earned, no double-claim path exists.

3. **Lifecycle is closed:** Every BURNIE mint corresponds to either (a) a claimed coinflip win, (b) a game event via `mintForGame`, (c) a direct GAME/AFFILIATE credit via `creditCoin`, or (d) a vault distribution from the pre-seeded allowance. No free-mint path exists.

4. **No path destroys tracked virtual stakes:** Loss day processing clears `coinflipBalance[day][player] = 0` (the principal is forfeited as intended by the coinflip house edge), but the player's `claimableStored` is preserved -- only days that were pending virtual claims lose value on loss days. This is the intended economic design.

5. **net BURNIE effect of coinflip:** Deflationary. `SUM(B1 burns for deposits) > SUM(M1 mints for wins)` by the house edge fraction (~1.575%). The virtual stake system has zero direct effect on physical supply until claims materialize.

**Files verified:** `contracts/BurnieCoin.sol` (~860 lines), `contracts/BurnieCoinflip.sol` (1154 lines).

## INV-05: No Permanently Unclaimable Funds

### Formal Invariant

```
No mechanism exists that credits value to a user but provides no path to claim that value,
EXCEPT for intentional expiry mechanisms explicitly documented in KNOWN-ISSUES.md or the
v1.1 specification.
```

### Methodology

For every mechanism that credits value to a user, the claim path is enumerated and classified:
- **(A) PERMANENT:** No expiry -- claimable indefinitely until claimed or GAMEOVER.
- **(B) EXPIRING-INTENTIONAL:** Expiry exists, documented in KNOWN-ISSUES.md or v1.1 spec.
- **(C) EXPIRING-UNDOCUMENTED:** Expiry exists but NOT documented as intentional -- FINDING.
- **(D) UNCLAIMABLE:** No claim path exists -- FINDING.

### Analysis of Each Expiring Mechanism

**#5/#6/#7 -- Decimator `lastDecClaimRound` overwrite:**

`lastDecClaimRound` is stored per player in `DecimatorModule`. When a new decimator round starts, any player who hasn't claimed from the prior round loses their entitlement. The unclaimed ETH remains in `claimablePool` -- it is NOT locked. Other claimants (players who did claim) received their shares; the unclaimed portions stay as surplus in the pool available to future distributions. This is not a fund-locking mechanism.

**Destination of forfeited decimator rewards:** The ETH was pre-reserved in `claimablePool` via `_addClaimableEth`. When a player fails to claim, their `claimableWinnings[player]` entry simply is never zeroed (it expires logically, not technically -- the contract does not track this separately). The ETH backing the unclaimed entry remains in `claimablePool`. The global `claimablePool` counter does not include unclaimed amounts that have been logically forfeited (the pre-reserve was for the entire decimator pool, not per-player). Verified from Phase 27 PAY-05: the decimator pre-reserves the entire pool amount in `claimablePool` before distribution, so forfeited amounts effectively remain as overcollateralization.

**FINDING NOTE:** The decimator expiry mechanism does NOT lock funds -- unclaimed decimator rewards reduce `claimableWinnings[player]` to zero (via the `lastDecClaimRound` overwrite logic) but leave the underlying ETH in the contract. The contract's `claimablePool` was credited when the decimator resolved; the player's individual `claimableWinnings` entry is what expires, not the pool itself. Surplus ETH backing expired claims provides overcollateralization for remaining claimants. This is by-design per KNOWN-ISSUES.md.

**#8/#9/#14/#15/#20 -- Coinflip claim window asymmetry:**

- First-time claimants: 30-day window (`COIN_CLAIM_FIRST_DAYS = 30` at BurnieCoinflip.sol:131).
- Returning claimants: 90-day window (`COIN_CLAIM_DAYS = 90` at BurnieCoinflip.sol:130).

When a player's `lastClaim == 0` (never claimed), `minClaimableDay = latest - 30`. Days older than 30 are unreachable. When `lastClaim > 0`, `minClaimableDay = latest - 90`. Days in expired windows are pruned: `coinflipBalance[day][player]` is not zeroed (no write), but they become unprocessable in `_claimCoinflipsInternal`.

**Destination of forfeited coinflip winnings:** The BURNIE for expired winning flips was NEVER minted (physical minting only occurs at claim time). The "forfeit" here means virtual stakes are simply abandoned. No physical BURNIE was created for them. No funds are locked -- they never entered circulation.

This is documented in KNOWN-ISSUES.md: "Coinflip claim window asymmetry (30d first-time vs 90d returning) is by-design."

**#24 -- Deity pass refund (GAMEOVER):**

ETH is credited to `claimableWinnings[player]` via `claimablePool += totalRefunded` at GameOverModule:105. The player claims via `claimWinnings(player)` on DegenerusGame.sol. The 30-day GAMEOVER window applies. After `handleFinalSweep`, `claimablePool = 0` and all unclaimed winnings (including deity refunds) are swept to vault/sDGNRS.

**Destination after GAMEOVER sweep:** All remaining ETH + stETH goes to vault (50%) and sDGNRS (50%) via `_sendToVault`. The sDGNRS recipients (sDGNRS token holders) indirectly receive value through increased backing per token.
