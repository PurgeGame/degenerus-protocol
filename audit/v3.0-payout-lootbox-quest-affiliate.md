## PAY-09: Lootbox Rewards

**Verdict:** PASS
**Files:** DegenerusGameLootboxModule.sol:1-1778, DegenerusGameEndgameModule.sol:515-534, DegenerusGamePayoutUtils.sol:77-93, DegenerusGameStorage.sol:529-669
**Summary:** The lootbox system distributes 5 reward types (future tickets, BURNIE, DGNRS, WWXRP, boons including whale/lazy/deity passes) through probability-weighted rolls with VRF entropy. All claimablePool mutations are correctly scoped to whale pass queueing remainders. CEI ordering is maintained throughout.
**Reward Types Audited:** whale pass, lazy pass, deity pass, future tickets, BURNIE
**Pool Source:** Lootbox rewards are resolved from the lootbox's own ETH-equivalent value (pre-funded at purchase time); DGNRS comes from the sDGNRS Lootbox pool; WWXRP is minted; BURNIE is credited via coinflip system
**claimablePool Impact:** Only whale pass remainder path mutates claimablePool (PayoutUtils.sol:90)
**CEI Status:** Correct -- storage zeroed before resolution in all entry points
**Double-Claim Guard:** lootboxEth[index][player] zeroed at line 602; lootboxBurnie[index][player] zeroed at line 640; whalePassClaims[player] zeroed at EndgameModule:521 before ticket award
**Recommendation:** None

### Detailed Analysis

#### 1. Whale Passes (via `_queueWhalePassClaimCore` and `claimWhalePass`)

**Queueing path (PayoutUtils.sol:77-93):**
- Entry: When jackpot ticket award amount > 5 ETH (`LOOTBOX_CLAIM_THRESHOLD`, Storage.sol:132), `_awardJackpotTickets` (EndgameModule:430-431) defers to `_queueWhalePassClaimCore`.
- Division: `fullHalfPasses = amount / HALF_WHALE_PASS_PRICE` (2.25 ETH per half-pass, LootboxModule:310-311).
- Accumulation: `whalePassClaims[winner] += fullHalfPasses` (line 84). This is an additive accumulation with **no expiry** -- half-passes persist until claimed (CP-07 confirmed).
- Remainder: `remainder = amount - (fullHalfPasses * HALF_WHALE_PASS_PRICE)`. Any sub-2.25 ETH remainder is credited to `claimableWinnings[winner]` and `claimablePool` (lines 88-91). This is the **only** claimablePool mutation in the entire lootbox reward path.

**Claim path (EndgameModule:515-534):**
- Guard: `if (gameOver) revert E()` -- whale pass claims disabled after GAMEOVER.
- Double-claim prevention: `whalePassClaims[player] = 0` at line 521, **before** any ticket award. CEI ordering is correct.
- Ticket award: `_queueTicketRange(player, startLevel, 100, uint32(halfPasses))` at line 533 queues `halfPasses` tickets per level across 100 levels. Each half-pass equates to 1 ticket/level (not 100 tickets per half-pass; the naming is "half of a full whale pass which gives 2 tickets/level").
- Stats: `_applyWhalePassStats(player, startLevel)` at line 531 applies mint streak and activity bonuses.

**Verdict:** PASS. Whale pass queueing and claiming correctly manage claimablePool. No double-claim path exists.

#### 2. Lazy Passes (Boon Type 29-31)

**Award path (LootboxModule:1502-1513):**
- Lazy pass rewards from lootbox are implemented as **discount boons** (types BOON_LAZY_PASS_10/25/50), not direct lazy pass grants.
- When a lazy pass boon is rolled during `_rollLootboxBoons`, it sets `lazyPassBoonDiscountBps[player]` to 10%, 25%, or 50% discount.
- Eligibility: `allowLazyPass` flag is true for ETH lootboxes (line 620), false for BURNIE lootboxes (line 668).
- Upgrade semantics: From non-deity source, only upgrades if new BPS > current (line 1507). Deity boons always overwrite.
- The boon has a day-based expiry tracked by `lazyPassBoonDay[player]`.

**Lazy pass value calculation (LootboxModule:1720-1732):**
- `_lazyPassPriceForLevel` sums 10 levels of ticket prices via `PriceLookupLib.priceForLevel`. This matches the lazy pass product definition.

**Note:** Lootbox does NOT grant a direct lazy pass activation. It grants a **discount boon** that reduces the cost of a future lazy pass purchase. The lazy pass itself must still be purchased by the player. This is correct behavior -- lootbox boons are temporary buffs, not direct product grants.

**Verdict:** PASS. Lazy pass discount boons correctly tracked with day expiry and upgrade-only semantics.

#### 3. Deity Passes (Boon Types 25-27, plus Deity Boon System)

**Deity pass discount boons (LootboxModule:1476-1491):**
- Types BOON_DEITY_PASS_10/25/50 grant 10%/25%/50% discount on future deity pass purchase.
- Stored as `deityPassBoonTier[player]` (tiers 1/2/3 for 10%/25%/50%).
- Eligibility: Only rolled when `deityEligible = (deityPassCount[player] == 0 && deityPassOwners.length < 32)` (line 1026-1027). Player must not own a deity pass and passes must still be available (max 32).
- Day-based expiry via `deityPassBoonDay[player]`.

**Deity boon system (LootboxModule:720-781):**
- `deityBoonSlots`: View function returning 3 deterministic boon types per deity per day.
- `issueDeityBoon`: Deity holders (with `deityPassPurchasedCount > 0`) can issue 1 boon per slot per day (3 slots/day), 1 recipient per day.
- Guards: zero address check, self-issue check, slot bounds check, pass ownership check, RNG availability check, recipient-already-received check, slot-already-used check.
- Uses `deityBoonUsedMask` bitfield (line 770-772) for per-slot tracking.
- `_deityBoonForSlot` uses deterministic seed from `_deityDailySeed(day)` + deity address + slot for reproducible boon type selection.

**Verdict:** PASS. Deity pass boons correctly gated by eligibility. Deity boon issuance has comprehensive guards against double-use.

#### 4. Future Tickets (Primary Lootbox Reward)

**Resolution path (LootboxModule:1530-1616):**
- `_resolveLootboxRoll` determines reward type with entropy-based roll (line 1551): `roll = nextEntropy % 20`.
- 55% tickets (roll < 11): Budget = `amount * LOOTBOX_TICKET_ROLL_BPS / 10000` (161% base multiplier). Tickets counted via `_lootboxTicketCount` with 5-tier variance system.
- 10% DGNRS (roll 11-12): From sDGNRS Lootbox pool via `_creditDgnrsReward` -> `dgnrs.transferFromPool`.
- 10% WWXRP (roll 13-14): Minted via `wwxrp.mintPrize(player, 1 ether)` -- fixed 1 WWXRP per hit.
- 25% BURNIE (roll 15-19): Large BURNIE reward with 80%/20% low/high variance paths (58%-590% of value).

**Ticket queueing (LootboxModule:952-964):**
- Tickets queued via `_queueTicketsScaled(player, targetLevel, futureTickets)` at line 963.
- `_queueTicketsScaled` (Storage.sol:562-578) handles fractional tickets with scaled arithmetic (TICKET_SCALE = 100).
- Distress bonus: 25% extra tickets on the fraction of lootbox ETH bought during distress mode (lines 954-962).
- Target level: Rolled via `_rollTargetLevel` (lines 793-811): 95% chance 0-5 levels ahead, 5% chance 5-50 levels ahead.

**Pool accounting:** Future tickets do NOT mutate claimablePool. They go into the ticket queue (`ticketsOwedPacked`) for the target level's future pool. The ETH backing these tickets was already allocated to pools at purchase time.

**Verdict:** PASS. Future ticket distribution correctly uses scaled arithmetic with variance tiers. No claimablePool impact.

#### 5. BURNIE Payouts (via `coin.creditFlip`)

**BURNIE credit path (LootboxModule:973-975):**
- All BURNIE rewards (from ticket-path fallback, large BURNIE roll, and presale bonus) are aggregated into `burnieAmount` (line 966).
- Credited via `coin.creditFlip(player, burnieAmount)` at line 974. This routes to the coinflip system as a **next-day coinflip wager**, NOT a direct BURNIE transfer. Player can claim it through the coinflip claim path.
- Presale bonus: If `presale` flag is true and `burniePresale > 0`, an additional 62% bonus is added (line 969: `LOOTBOX_PRESALE_BURNIE_BONUS_BPS = 6200`).

**BURNIE amount calculation (large path, LootboxModule:1598-1614):**
- Low path (80%, rolls 0-15): 58.08% to 130.43% of ETH-equivalent value. Formula: `LOOTBOX_LARGE_BURNIE_LOW_BASE_BPS + roll * LOOTBOX_LARGE_BURNIE_LOW_STEP_BPS` = 5808 + roll * 477 BPS.
- High path (20%, rolls 16-19): 307.05% to 590.95% of value. Formula: `LOOTBOX_LARGE_BURNIE_HIGH_BASE_BPS + (roll-16) * LOOTBOX_LARGE_BURNIE_HIGH_STEP_BPS` = 30705 + (roll-16) * 9430 BPS.
- Conversion: `burnieOut = (burnieBudget * PRICE_COIN_UNIT) / targetPrice` (line 1613), using the target level's price for ETH-to-BURNIE conversion.

**Verdict:** PASS. BURNIE rewards correctly credited via `creditFlip` (not direct transfer). Amount calculations use correct BPS with variance tiers.

#### 6. Tier System (Probability-Weighted Selection)

**Reward type selection (LootboxModule:1551):**
- Single roll: `roll = nextEntropy % 20` gives 5% granularity per outcome.
- Distribution: 55% tickets (11/20), 10% DGNRS (2/20), 10% WWXRP (2/20), 25% BURNIE (5/20).

**Ticket variance tiers (LootboxModule:1618-1670):**
- 5-tier system with `varianceRoll = nextEntropy % 10000`:
  - Tier 1 (1%): 4.6x multiplier (46000 BPS)
  - Tier 2 (4%): 2.3x multiplier (23000 BPS)
  - Tier 3 (20%): 1.1x multiplier (11000 BPS)
  - Tier 4 (45%): 0.651x multiplier (6510 BPS)
  - Tier 5 (30%): 0.45x multiplier (4500 BPS)
- Applied to base budget of 161% (`LOOTBOX_TICKET_ROLL_BPS = 16100`).
- Weighted average: (0.01 * 46000 + 0.04 * 23000 + 0.20 * 11000 + 0.45 * 6510 + 0.30 * 4500) = 460 + 920 + 2200 + 2929.5 + 1350 = 7859.5 BPS = ~78.6% of the 161% base = ~126.5% effective EV before activity scaling.

**Target level selection (LootboxModule:793-811):**
- VRF-derived entropy via `EntropyLib.entropyStep(entropy)`.
- 95% near (0-5 levels ahead), 5% far (5-50 levels ahead).

**2x over-collateralization:** This was verified in PAY-16 (audit/v3.0-payout-jackpot-distribution.md). Lootbox ticket budget uses `_budgetToTicketUnits` which pre-reserves 2x the ticket face value from pools, ensuring tickets are always backed. The lootbox module itself does not manage pool pre-reservation -- it inherits the structural over-collateralization from the jackpot distribution system.

**Verdict:** PASS. Probability-weighted tier system uses VRF entropy correctly. Distribution percentages match constants.

### Lootbox EV Multiplier System

The activity-score-based EV multiplier (LootboxModule:466-544) scales lootbox value from 80% (zero activity) to 135% (max activity), with a per-account per-level cap of 10 ETH benefit (`LOOTBOX_EV_BENEFIT_CAP`). This prevents whales from extracting unlimited above-neutral EV. The cap is tracked in `lootboxEvBenefitUsedByLevel[player][lvl]` and correctly applied with partial neutral fallback when cap is partially exhausted.

### Lootbox Split Mechanics

Lootboxes above 0.5 ETH (`LOOTBOX_SPLIT_THRESHOLD`, LootboxModule:313) are split into two independent rolls (line 869-872). Each half gets its own `_resolveLootboxRoll`. This increases reward variance while maintaining the same expected value.

## PAY-10: Quest Rewards and Streak Bonuses

**Verdict:** PASS
**Files:** DegenerusQuests.sol:1-1598, BurnieCoin.sol:556-558 (creditFlip routing), DegenerusGame.sol:2478-2484 (activity score contribution)
**Summary:** Quest system correctly delivers 100 BURNIE (slot 0) and 200 BURNIE (slot 1) via `creditFlip` routing. Streak tracks up to 100 consecutive days contributing 10000 BPS to activity score. Version-gated progress prevents stale carry-over or re-triggering.
**Reward Amounts:** Slot 0 = 100 BURNIE (`QUEST_SLOT0_REWARD`), Slot 1 = 200 BURNIE (`QUEST_RANDOM_REWARD`)
**Delivery Method:** `coin.creditFlip(player, amount)` -- quest rewards become next-day coinflip wagers
**Streak Range:** 0 to 100 days (capped at uint24 max, but activity contribution capped at 100)
**Activity Contribution:** `questStreak * 100` BPS, max 10000 BPS (DegenerusGame.sol:2481-2484)
**Double-Claim Guard:** Per-slot completion mask (`completionMask` bits 0-1), version-gated progress reset
**Recommendation:** None

### Detailed Analysis

#### 1. Quest Reward Amounts

**Slot 0 (DegenerusQuests.sol:135):**
- `QUEST_SLOT0_REWARD = 100 ether` = 100 BURNIE (18 decimals).
- Always MINT_ETH type (line 376: `primaryType = QUEST_TYPE_MINT_ETH`).
- Target: 1x current ticket price, capped at 0.5 ETH (`QUEST_ETH_TARGET_CAP`).

**Slot 1 (DegenerusQuests.sol:138):**
- `QUEST_RANDOM_REWARD = 200 ether` = 200 BURNIE (18 decimals).
- Randomly selected from 8 eligible types (weighted, excluding RESERVED type 4).
- Target varies by type: 1 ticket (MINT_BURNIE), 2000 BURNIE (FLIP/DEC/AFFILIATE/DEGENERETTE_BURNIE), 2x mint price capped at 0.5 ETH (LOOTBOX/DEGENERETTE_ETH).

**Reward selection (DegenerusQuests.sol:1425):**
```
uint256 rewardShare = slot == 1 ? QUEST_RANDOM_REWARD : QUEST_SLOT0_REWARD;
```
This is deterministic based on slot index -- no ambiguity.

**Verification:** Constants cross-referenced against v1.1-quest-rewards.md Section 3a. Matches: slot 0 = 100 BURNIE, slot 1 = 200 BURNIE.

#### 2. Reward Delivery (creditFlip Routing)

Quest rewards follow this path:
1. Player action triggers `handle*()` on DegenerusQuests.sol (COIN-gated).
2. On completion, `_questComplete` returns `rewardShare` (100 or 200 BURNIE).
3. BurnieCoin's `_questApplyReward` (BurnieCoin.sol:1007-1022) emits event and returns reward.
4. Caller routes reward via `IBurnieCoinflip(coinflipContract).creditFlip(player, questReward)`.

**Confirmed:** Quest rewards are credited as coinflip stakes, NOT direct BURNIE transfers. The player receives the value as a pending coinflip wager, consistent with v1.1-quest-rewards.md Section 3b.

#### 3. Streak Mechanics

**Streak increment (DegenerusQuests.sol:1414-1422):**
- Streak increments on the **first quest completion of the day** (either slot).
- `QUEST_STATE_STREAK_CREDITED` (bit 7 of `completionMask`) prevents double-increment per day.
- Streak stored as `uint24` (max ~16.7 million, effectively unbounded for practical purposes).

**Streak reset (DegenerusQuests.sol:1111-1144, `_questSyncState`):**
- On each handler call, checks if `currentDay > anchorDay + 1` (gap detected).
- If gap exists and shields are insufficient: `state.streak = 0`.
- Shield consumption: 1 shield per missed day, consumed in order.

**Streak contribution to activity score (DegenerusGame.sol:2478-2484):**
```solidity
(uint32 questStreakRaw, , , ) = questView.playerQuestStates(player);
uint256 questStreak = questStreakRaw > 100 ? 100 : uint256(questStreakRaw);
bonusBps += questStreak * 100;
```
- **100 BPS per streak day, capped at 100 days = 10000 BPS maximum.**
- This is the **single largest activity score component** (out of max ~30500 BPS total).

**Verification:** Matches v1.1-quest-rewards.md Section 7: "1% per quest streak, max 100%" = 10000 BPS.

#### 4. Trigger Conditions

Each quest type has a dedicated `handle*` function called by the COIN contract:
- `handleMint` (DegenerusQuests.sol:440-524): Triggered by ticket purchases (ETH or BURNIE).
- `handleFlip` (DegenerusQuests.sol:538-579): Triggered by coinflip stakes.
- `handleDecimator` (DegenerusQuests.sol:593-631): Triggered by decimator burns.
- `handleAffiliate` (DegenerusQuests.sol:644-682): Triggered by affiliate earnings.
- `handleLootBox` (DegenerusQuests.sol:697-736): Triggered by lootbox purchases.
- `handleDegenerette` (DegenerusQuests.sol:750-789): Triggered by degenerette bets.

All handlers follow the same pattern:
1. Check `currentDay != 0` (quest rolled for today).
2. `_questSyncState` (reset streak if missed days).
3. Find matching quest slot via `_currentDayQuestOfType`.
4. Sync progress (`_questSyncProgress` resets if day/version mismatch).
5. Accumulate progress, check target.
6. **Slot 1 prerequisite:** `if (slotIndex == 1 && (state.completionMask & 1) == 0) return` -- slot 1 cannot complete until slot 0 is done.
7. On completion, `_questCompleteWithPair` checks if the other slot can also auto-complete.

#### 5. Double-Claim Guard

**Per-slot completion mask (DegenerusQuests.sol:1397-1406):**
- `completionMask` uses bit 0 for slot 0, bit 1 for slot 1.
- On each `_questComplete` call: `if ((mask & slotMask) != 0) return (0, ...)` -- already completed.
- Mask set: `mask |= slotMask` (line 1406).

**Per-day reset (DegenerusQuests.sol:1139-1143):**
- `_questSyncState` resets `completionMask = 0` when `lastSyncDay != currentDay24`.
- This means each day starts fresh -- no carry-over from previous days.

**Version gating (DegenerusQuests.sol:1146-1165):**
- `_questSyncProgress` resets `state.progress[slot] = 0` when `lastProgressDay[slot] != currentDay24` or `lastQuestVersion[slot] != quest.version`.
- This prevents stale progress from completing a new quest.

**Verdict:** No double-claim path exists. Completion is tracked per-slot per-day with version validation.

## PAY-11: Affiliate Commissions

**Verdict:** PASS
**Files:** DegenerusAffiliate.sol:386-623 (payAffiliate), DegenerusAffiliate.sol:803-811 (lootbox taper), DegenerusGame.sol:1458-1501 (claimAffiliateDgnrs), DegenerusGameEndgameModule.sol:107-137 (rewardTopAffiliate / levelDgnrsAllocation)
**Summary:** 3-tier affiliate system correctly distributes BURNIE commissions via weighted random lottery. DGNRS claims use fixed per-level allocation with proportional distribution, eliminating first-mover advantage described in v1.1 docs. Per-level claim guard (`affiliateDgnrsClaimedBy`) prevents double-claim.
**Tier System:** Direct (base-kickback), Upline1 (20% of scaled), Upline2 (4% of scaled)
**ETH Commission Rates:** Fresh L0-3: 25% (2500 BPS), Fresh L4+: 20% (2000 BPS), Recycled: 5% (500 BPS)
**Per-Sender Cap:** 0.5 ETH BURNIE per sender per level (`MAX_COMMISSION_PER_REFERRER_PER_LEVEL`)
**DGNRS Mechanism:** Fixed allocation per level (not sequential depletion)
**claimablePool Impact:** None -- affiliate commissions operate entirely in BURNIE domain via creditFlip
**Recommendation:** None

### Detailed Analysis

#### 1. 3-Tier System

**Tier computation (DegenerusAffiliate.sol:543-591):**

| Tier | Recipient | Share | Source |
|------|-----------|-------|--------|
| Direct | Code owner | `scaledAmount - kickback + questReward` | Line 552-556 |
| Upline 1 | Affiliate's referrer | `scaledAmount / 5` (20%) + quest reward | Line 566-568 |
| Upline 2 | Upline 1's referrer | `scaledAmount / 25` (4%) + quest reward | Line 581-583 |

**Commission rate verification (DegenerusAffiliate.sol:476-485):**
- `REWARD_SCALE_FRESH_L1_3_BPS = 2500` (25% for levels 0-3).
- `REWARD_SCALE_FRESH_L4P_BPS = 2000` (20% for levels 4+).
- `REWARD_SCALE_RECYCLED_BPS = 500` (5% for recycled ETH).
- Matches v1.1-affiliate-system.md Section 2.

#### 2. ETH Commission Distribution

**Weighted random lottery (DegenerusAffiliate.sol:597-618):**
- When multiple recipients exist (2 or 3 tiers), all amounts are pooled and a **single winner** is selected via `_rollWeightedAffiliateWinner` (line 609-616).
- Winner receives combined total. Each recipient's probability = their share / total.
- This preserves expected value (EV-neutral) but introduces high variance per payout.
- Entropy derived from: `keccak256(AFFILIATE_ROLL_TAG, currentDay, sender, storedCode, totalAmount)` (line 824-828).

**Reward routing (DegenerusAffiliate.sol:761-767):**
- `_routeAffiliateReward(player, amount)` calls `coin.creditFlip(player, amount)`.
- All affiliate commissions are credited as coinflip stakes, NOT direct ETH transfers.
- No claimablePool mutation occurs.

#### 3. Lootbox Taper

**Taper function (DegenerusAffiliate.sol:803-811):**
```solidity
function _applyLootboxTaper(uint256 amt, uint16 score) private pure returns (uint256) {
    if (score >= LOOTBOX_TAPER_END_SCORE) {
        return (amt * LOOTBOX_TAPER_MIN_BPS) / BPS_DENOMINATOR;
    }
    uint256 excess = uint256(score) - LOOTBOX_TAPER_START_SCORE;
    uint256 range = uint256(LOOTBOX_TAPER_END_SCORE) - LOOTBOX_TAPER_START_SCORE;
    uint256 reductionBps = (BPS_DENOMINATOR - LOOTBOX_TAPER_MIN_BPS) * excess / range;
    return (amt * (BPS_DENOMINATOR - reductionBps)) / BPS_DENOMINATOR;
}
```

**Taper schedule:**
- `LOOTBOX_TAPER_START_SCORE = 10000` (activity score 100%).
- `LOOTBOX_TAPER_END_SCORE = 25500` (activity score 255%).
- `LOOTBOX_TAPER_MIN_BPS = 2500` (25% floor).
- Linear interpolation: 100% payout at score 10000, tapering to 25% at score 25500.
- Applied **after** leaderboard recording (line 510-513 records full untapered amount, line 526-528 applies taper for payout).
- Matches v1.1-affiliate-system.md Section 5.

#### 4. DGNRS Claim Path (claimAffiliateDgnrs)

**Function (DegenerusGame.sol:1458-1501):**

```solidity
function claimAffiliateDgnrs(address player) external {
    player = _resolvePlayer(player);
    uint24 currLevel = level;
    if (currLevel == 0) revert E();
    if (affiliateDgnrsClaimedBy[currLevel][player]) revert E();  // per-level guard
    uint256 score = affiliate.affiliateScore(currLevel, player);
    bool hasDeityPass = deityPassCount[player] != 0;
    if (!hasDeityPass && score < AFFILIATE_DGNRS_MIN_SCORE) revert E();  // 10 ETH min
    uint256 denominator = affiliate.totalAffiliateScore(currLevel);
    if (denominator == 0) revert E();
    uint256 allocation = levelDgnrsAllocation[currLevel];  // fixed per-level
    if (allocation == 0) revert E();
    uint256 reward = (allocation * score) / denominator;  // proportional
    // ... transferFromPool, mark claimed
    affiliateDgnrsClaimedBy[currLevel][player] = true;
}
```

**CRITICAL VERIFICATION -- Fixed Allocation (not Sequential Depletion):**

The v1.1 affiliate system doc (Section 9) describes DGNRS claims as sequential depletion: "5% of affiliate pool per level (not reserved -- sequential depletion)." The economics primer (Section 8) repeats this: "5% of affiliate pool per level (not reserved -- sequential depletion)."

**The actual code uses a fundamentally different mechanism:**

1. At level transition, `rewardTopAffiliate` (EndgameModule:118-137) snapshots 5% of the current affiliate pool balance: `levelDgnrsAllocation[lvl] = (remainingPool * AFFILIATE_DGNRS_LEVEL_BPS) / 10_000`.
2. This allocation is a **fixed denominator** for that level -- it does not decrease as players claim.
3. Each player's reward = `(allocation * score) / denominator` where `denominator = totalAffiliateScore[currLevel]`.
4. The `transferFromPool` call draws from the live pool balance, but the reward amount is calculated from the snapshot.

**This means:**
- Early claimants do NOT get more than late claimants.
- All claimants at the same level with the same score get the same DGNRS reward.
- The system is proportional, not first-come-first-served.
- If total claims exceed the snapshot (theoretically possible if the pool was depleted by other levels), `transferFromPool` returns less than requested, and the `if (paid == 0) revert E()` guard catches zero returns.

**Storage evidence (DegenerusGameStorage.sol:1005-1011):**
```solidity
/// @dev Segregated DGNRS allocation per level (5% of affiliate pool at transition time).
///      Set during level transition in rewardTopAffiliate. Claims draw against this
///      fixed amount instead of the live pool balance, eliminating first-mover advantage.
mapping(uint24 => uint256) internal levelDgnrsAllocation;
```

The natspec explicitly confirms "eliminating first-mover advantage" -- this is intentional design.

**v1.1 Documentation Discrepancy (FINDING-INFO):** The v1.1 docs describe the old sequential depletion mechanism. The code implements a superior proportional mechanism. This is a documentation staleness issue, not a code bug. Classified as INFO per Research Open Question 1 / CP-06.

- One claim per level per player. Boolean flag -- no bypass path.
- Checked BEFORE reward calculation, set AFTER transfer. CEI ordering is correct (check -> interact -> effect for the flag, but the transfer is also an effect, so technically C-I-E where I and E are both state changes; the key is the revert-on-true check happens first).
- No proxy or alternate claim path exists.

**Additional eligibility requirements:**
- `currLevel > 0` (no claims at level 0).
- `score >= AFFILIATE_DGNRS_MIN_SCORE` (10 ETH BURNIE, DegenerusGame.sol:208) unless player holds a deity pass.
- `totalAffiliateScore[currLevel] > 0` (at least one affiliate earned something).
- `levelDgnrsAllocation[currLevel] > 0` (allocation was set during transition).

#### 6. First-Mover Advantage Analysis

Using the actual code (fixed allocation with `totalAffiliateScore` denominator):

- Player A claims at time T1: `reward_A = (allocation * score_A) / totalScore`.
- Player B claims at time T2 (later): `reward_B = (allocation * score_B) / totalScore`.
- If `score_A == score_B`, then `reward_A == reward_B`. **No first-mover advantage.**

The only risk is pool exhaustion: if many levels accumulate claims faster than the pool can support, later claimants might receive less from `transferFromPool` due to insufficient balance. However:
- Each level only allocates 5% of the pool snapshot at transition time.
- Unclaimed allocations from past levels remain in the pool for future snapshots.
- The 1% top-affiliate reward (EndgameModule:123) is a small initial deduction.

**Verdict:** PASS. No first-mover advantage in the DGNRS claim mechanism.
