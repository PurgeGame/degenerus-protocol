### CMT-025: Architecture header JACKPOT FLOW OVERVIEW has numbering gap (1, 2, 4)

- **What:** The JACKPOT FLOW OVERVIEW in the contract-level `@dev` block lists steps 1, 2, and 4, skipping step 3. Step 3 was likely a separate flow item (possibly related to the removed future dump feature or a prior daily coin jackpot step) that was removed without renumbering the remaining items.
- **Where:** DegenerusGameJackpotModule.sol:29-31
- **Why:** A warden reading the architecture overview would notice the gap and wonder what step 3 was, potentially suspecting incomplete feature removal or missing documentation. The gap also gives the impression that the overview was not maintained as the contract evolved. A warden comparing the JACKPOT FLOW OVERVIEW against the actual function list might file a QA finding about the inconsistency.
- **Suggestion:** Renumber step 4 to step 3: `3. \`processTicketBatch\` -- Batched airdrop processing with gas budgeting to stay block-safe.`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### CMT-026: payDailyJackpot NatSpec physically attached to runTerminalJackpot

- **What:** Lines 252-272 contain extensive `@notice` and `@dev` NatSpec describing `payDailyJackpot`'s two execution paths (DAILY PATH and EARLY-BURN PATH). This is immediately followed by lines 274-281 containing `runTerminalJackpot`'s NatSpec. Since `runTerminalJackpot` is the first function declaration (line 282), a Solidity NatSpec parser associates the entire block (lines 252-281) with `runTerminalJackpot`. Meanwhile `payDailyJackpot` (line 323) has only `@param` tags (lines 320-322) -- no `@notice` or `@dev`.
- **Where:** DegenerusGameJackpotModule.sol:252-282
- **Why:** A warden using generated documentation or a NatSpec extraction tool would see `runTerminalJackpot` documented with two `@notice` tags, one describing daily/early-burn jackpot flows that are entirely irrelevant to its terminal jackpot behavior. Meanwhile `payDailyJackpot` -- the most complex function in the module with 350+ lines and two distinct execution paths -- would appear undocumented except for parameter descriptions. A warden manually reading the source would likely not notice the misattribution, but any tooling-based analysis would be significantly confused.
- **Suggestion:** Move the `payDailyJackpot` NatSpec block (lines 252-272) to immediately above the function at line 323, before the existing `@param` tags at lines 320-322. This correctly associates each function with its own documentation.
- **Category:** comment-inaccuracy
- **Severity:** LOW

### CMT-027: _processAutoRebuy NatSpec says "next level or next+1 (50/50)" but code uses 1-4 levels (25% each)

- **What:** The `@dev` NatSpec for `_processAutoRebuy` at line 980 states `Converts winnings to tickets for next level or next+1 (50/50)`. The actual implementation calls `_calcAutoRebuy` (PayoutUtils line 54-58) which computes `levelOffset = (entropy & 3) + 1`, producing values 1 through 4 with 25% probability each. Level offset 1 targets `currentLevel + 1` (next level, 25% chance), while offsets 2-4 target `currentLevel + 2/3/4` (75% chance, routed to futurePrizePool). The "50/50" split and "next or next+1" range are both wrong.
- **Where:** DegenerusGameJackpotModule.sol:980
- **Why:** A warden modeling auto-rebuy fund flow would assume a 50/50 split between the next two levels, significantly underestimating the spread across 4 levels and the 75% probability of funds going to futurePrizePool vs. 25% to nextPrizePool. This matters for prize pool sustainability analysis: the actual design channels 75% of auto-rebuy ETH into the future pool (long-term reserve) vs. only 25% into the immediate next level, which is a deliberately conservative approach that the NatSpec fails to communicate.
- **Suggestion:** Change to: `Converts winnings to tickets 1-4 levels ahead (25% each; +1 goes to nextPrizePool, +2/+3/+4 go to futurePrizePool).`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### CMT-028: _resolveTraitWinners NatSpec says "loot box conversion" but code performs auto-rebuy

- **What:** The `_resolveTraitWinners` NatSpec flow step 3 at line 1593 states `Credit ETH payouts to claimableWinnings (with optional loot box conversion)`. The function calls `_addClaimableEth` (line 1705) which, for players with auto-rebuy enabled, calls `_processAutoRebuy` to convert ETH winnings into tickets -- not loot boxes. Loot box distribution happens at a higher level via `_distributeLootboxAndTickets` (called from `payDailyJackpot`), not within `_resolveTraitWinners`.
- **Where:** DegenerusGameJackpotModule.sol:1593
- **Why:** A warden reading the NatSpec would expect to find loot box logic within `_resolveTraitWinners` or its callees. When they find auto-rebuy ticket conversion instead, they might question whether the loot box path is missing or broken. The distinction matters: loot boxes are awarded to winners based on jackpot budgets at the daily/early-burn level, while auto-rebuy is a per-player opt-in that converts ETH winnings into queued tickets with a bonus multiplier.
- **Suggestion:** Change to: `Credit ETH payouts to claimableWinnings (with optional auto-rebuy ticket conversion).`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### CMT-029: _resolveTraitWinners @return ticketSpent describes wrong pool and wrong conversion type

- **What:** The `@return ticketSpent` NatSpec at line 1605 states `ETH converted into loot box awards (added to nextPrizePool)`. The actual return value maps to `totalWhalePassSpent` (line 1730), which is the ETH cost of whale pass conversions performed by `_processSoloBucketWinner`. This ETH is added to `futurePrizePool` (line 1788: `_setFuturePrizePool(_getFuturePrizePool() + whalePassCost)`), not `nextPrizePool`. Both the conversion type (whale passes, not loot boxes) and the destination pool (future, not next) are wrong.
- **Where:** DegenerusGameJackpotModule.sol:1605
- **Why:** A warden tracing fund flow through `_resolveTraitWinners` would model the `ticketSpent` return value as ETH flowing into `nextPrizePool` for loot box backing. The actual flow is whale pass conversion ETH flowing into `futurePrizePool`, which is the long-term reserve. This mismatch would cause incorrect fund flow diagrams: the warden would overstate `nextPrizePool` inflows and understate `futurePrizePool` inflows during jackpot distribution. The caller (`_processOneBucket` at line 1580) adds `ticketSpent` to `ctx.totalPaidEth`, confirming it's tracked as "paid out" regardless of destination -- but the NatSpec misleads about *where* the ETH goes.
- **Suggestion:** Change to: `@return ticketSpent ETH converted into whale pass claims (added to futurePrizePool).`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### CMT-030: Orphaned NatSpec from removed loot box distribution function

- **What:** Lines 1797-1798 contain two orphaned NatSpec lines: `/// @dev Distributes jackpot loot box rewards to winners based on trait buckets.` and `///      Awards tickets only (no BURNIE) using jackpot loot box mechanics.` These lines describe a function that no longer exists -- the loot box distribution logic was refactored into `_distributeLootboxAndTickets` (line 1079) and `_distributeTicketJackpot` (line 1104). The orphaned lines sit between `_processSoloBucketWinner` (ending line 1795) and the NatSpec for `_getWinningTraits` (starting line 1800). A NatSpec parser would merge lines 1797-1798 with lines 1800-1806 and associate the combined block with `_getWinningTraits`, making it appear that `_getWinningTraits` distributes loot box rewards.
- **Where:** DegenerusGameJackpotModule.sol:1797-1798
- **Why:** A warden using generated documentation or NatSpec extraction tools would see `_getWinningTraits` described as distributing loot box rewards and tickets, which is entirely unrelated to its actual purpose (deriving daily winning traits from VRF entropy with hero override). This is the fourth instance of the orphaned NatSpec pattern across the codebase (after CMT-010 in DegenerusGame.sol, CMT-011 in MintModule, and CMT-020 in DegeneretteModule).
- **Suggestion:** Delete lines 1797-1798.
- **Category:** comment-inaccuracy
- **Severity:** INFO
