#### DSC-01: v3.9 RNG Commitment Window Proof Is Stale [INFO]

**Source:** Phase 81 Plans 01 + 02
**Documents:** `v4.0-ticket-creation-queue-mechanics.md` Section 4, `v4.0-ticket-queue-double-buffer.md` Section 11
**Location:** `audit/v3.9-rng-commitment-window-proof.md` (compared against `JM:2543` current code)

The v3.9 RNG commitment window proof describes a combined pool approach — reading from both `_tqReadKey` and `_tqFarFutureKey` in `_awardFarFutureCoinJackpot`. This was implemented in Phase 77 but reverted in commit `2bf830a2`. Current code at JM:2543 reads ONLY from `_tqFarFutureKey`.

**Stale claims in the proof (5 items):**

| Claim # | v3.9 Reference | What It Says | Current Code |
|---------|----------------|--------------|-------------|
| 8 | Line 14 | Combined pool in _awardFarFutureCoinJackpot | FF-only at JM:2543 |
| 9 | Lines 41-63 | Backward trace through readLen + ffLen | No readLen/ffLen — just `queue.length` at JM:2544 |
| 10 | Lines 76-77 | combinedLen = readLen + ffLen | combinedLen doesn't exist |
| 11 | Lines 258-296 | Read buffer accessed by jackpot | Read buffer not accessed |
| 12 | Line 313 | Combined pool length invariant | Invariant is trivially satisfied (FF-only) |

**Security impact:** None. The proof proved a superset of the actual behavior — FF-only is strictly simpler than combined pool. RNG-01 (SAFE) conclusion remains valid, but the proof document needs rewriting before C4A audit to avoid warden confusion.

**Recommendation:** Rewrite `audit/v3.9-rng-commitment-window-proof.md` to describe current FF-only code. Defer to a future phase.

#### DSC-02: sampleFarFutureTickets Uses Wrong Key Space [INFO]

**Source:** Phase 81 Plans 01 + 02
**Documents:** `v4.0-ticket-creation-queue-mechanics.md` Section 4, `v4.0-ticket-queue-double-buffer.md` Section 5
**Location:** `contracts/DegenerusGame.sol:2669-2705` (specifically DG:2681)

```solidity
// DG:2681 — current code
ticketQueue[_tqWriteKey(candidate)]
```

Since v3.9 Phase 74, far-future tickets (targetLevel > currentLevel + 6) route to `_tqFarFutureKey`, not `_tqWriteKey`. The `sampleFarFutureTickets` view function still reads from `_tqWriteKey(candidate)` for candidates ranging `currentLvl + 5` to `currentLvl + 99`, returning empty results for levels where tickets actually live in the FF key space.

**Impact:** View function only — no on-chain state mutation. Off-chain consumers (UI, indexer, dashboard) calling this function receive empty/incorrect results for far-future ticket holders.

**Fix:** Replace `_tqWriteKey(candidate)` with `_tqFarFutureKey(candidate)` at DG:2681 for candidates beyond level + 6. Near-future candidates (level + 5 to level + 6) should continue using `_tqWriteKey`.

#### DSC-03: NatSpec Claims Cap But Code Uses Unchecked [INFO]

**Source:** Phase 81 Plan 01
**Document:** `v4.0-ticket-creation-queue-mechanics.md` Section 4
**Location:** `contracts/storage/DegenerusGameStorage.sol:533` (NatSpec) vs `GS:553-555` (code)

NatSpec at GS:533 claims "Caps at uint32 max to prevent overflow" but code at GS:553-555 uses `unchecked { owed += quantity; }` with no explicit cap check. Overflow would require accumulating > 4.3 billion tickets (> 4.3M ETH in purchases), which exceeds total ETH supply.

**Impact:** Comment correctness issue only. Overflow is physically infeasible. No security concern.

**Fix:** Update NatSpec to say "Overflow infeasible (requires > total ETH supply)" or add an explicit cap check to match the comment.
