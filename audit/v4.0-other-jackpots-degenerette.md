## 1. Degenerette Overview

The Degenerette is a player-initiated symbol-match betting game that operates **independently** from the `advanceGame` state machine. It is not triggered by daily jackpot processing or level transitions. Players place bets, choosing a 4-quadrant ticket (8 attributes total: 4 colors + 4 symbols), and resolve them once a lootbox VRF word becomes available.

**Call chain:**
```
User -> DegenerusGame.placeFullTicketBets (DG:747-771) -> delegatecall -> DDM.placeFullTicketBets (DDM:388-404)
User -> DegenerusGame.resolveDegeneretteBets (DG:776-790) -> delegatecall -> DDM.resolveBets (DDM:411-423)
```

The delegatecall is routed via `ContractAddresses.GAME_DEGENERETTE_MODULE` (DG:756). All storage reads and writes operate on the main `DegenerusGame` contract's storage through the shared `DegenerusGameStorage` layout.

The module inherits from both `DegenerusGamePayoutUtils` and `DegenerusGameMintStreakUtils` (DDM:55).

## 2. Bet Placement

Entry point: `placeFullTicketBets` at DDM:388-404. This resolves the player address via `_resolvePlayer` (DDM:160-166), supporting operator approvals, then calls the internal `_placeFullTicketBets` (DDM:430-460).

### 2.1 Currency Types

Three currencies are supported, defined as constants at DDM:242-248:

| Constant | Value | Token |
|----------|-------|-------|
| `CURRENCY_ETH` | 0 | Native ETH |
| `CURRENCY_BURNIE` | 1 | BURNIE ERC20 |
| `CURRENCY_WWXRP` | 3 | WWXRP ERC20 |

Value 2 is explicitly unsupported -- `_validateMinBet` reverts with `UnsupportedCurrency()` at DDM:536.

### 2.2 Minimum Bets

Enforced by `_validateMinBet` (DDM:528-538):

| Currency | Constant | Value | Line |
|----------|----------|-------|------|
| ETH | `MIN_BET_ETH` | 0.005 ETH | DDM:251 |
| BURNIE | `MIN_BET_BURNIE` | 100 BURNIE | DDM:254 |
| WWXRP | `MIN_BET_WWXRP` | 1 WWXRP | DDM:257 |

### 2.3 Max Spins Per Bet

`MAX_SPINS_PER_BET = 10` (DDM:260). Validated at DDM:470: `if (ticketCount == 0 || ticketCount > MAX_SPINS_PER_BET) revert InvalidBet()`.

### 2.4 RNG Binding (lootboxRngIndex)

At DDM:473-475, the current `lootboxRngIndex` (GS:1195) is captured. Two critical checks:
1. `if (index == 0) revert E()` -- index must be initialized (DDM:474).
2. `if (lootboxRngWordByIndex[index] != 0) revert RngNotReady()` -- the word must NOT yet be fulfilled (DDM:475). This ensures the bet is placed BEFORE the randomness is known.

The `index` is stored in the packed bet at bits [172..219] (DDM:324, DDM:351). At resolution time, the word is retrieved via `lootboxRngWordByIndex[index]` (DDM:597) and must be non-zero (DDM:598).

**RNG commitment model:** The bet is bound to a specific VRF request index. The word is unknown at bet placement time (DDM:475 enforces `word == 0`). At resolution time, the word must have been fulfilled (DDM:598 enforces `word != 0`). This is a valid commitment scheme -- the player cannot know the randomness when choosing their ticket.

### 2.5 Packed Bet Storage

Layout documented at DDM:316-341, stored in `degeneretteBets[player][nonce]` (GS:1274):

| Bits | Field | Type | Description |
|------|-------|------|-------------|
| [0] | mode | 1 bit | Always 1 (MODE_FULL_TICKET, DDM:344) |
| [1] | isRandom | 1 bit | Always 0 (no random tickets) |
| [2..33] | customTicket | 32 bits | Packed traits [D:24-31][C:16-23][B:8-15][A:0-7] |
| [34..41] | ticketCount | 8 bits | Spin count (1..10) |
| [42..43] | currency | 2 bits | 0=ETH, 1=BURNIE, 3=WWXRP |
| [44..171] | amountPerTicket | 128 bits | Bet amount per spin |
| [172..219] | index | 48 bits | lootboxRngIndex at bet time |
| [220..235] | activityScore | 16 bits | Player activity score (BPS) |
| [236] | hasCustom | 1 bit | Always 1 |
| [237..239] | hero | 3 bits | [0]=enabled, [1..2]=quadrant (0-3) |

Packing implemented in `_packFullTicketBet` (DDM:764-786). Shift constants: `FT_TICKET_SHIFT=2` (DDM:347), `FT_COUNT_SHIFT=34` (DDM:348), `FT_CURRENCY_SHIFT=42` (DDM:349), `FT_AMOUNT_SHIFT=44` (DDM:350), `FT_INDEX_SHIFT=172` (DDM:351), `FT_ACTIVITY_SHIFT=220` (DDM:352), `FT_HAS_CUSTOM_SHIFT=236` (DDM:353), `FT_HERO_SHIFT=237` (DDM:354).

Hero quadrant encoding at DDM:783-785: if `heroQuadrant < 4`, stores `(1 | (heroQuadrant << 1))` at FT_HERO_SHIFT, giving enabled=1 and the quadrant index in bits [1..2].

Nonce management at DDM:489-493: `degeneretteBetNonce[player]` (GS:1277) is incremented per bet.

### 2.6 ETH Fund Flow

`_collectBetFunds` (DDM:541-574) handles funds per currency:

**ETH (DDM:547-567):**
1. Validates `ethPaid <= totalBet` (DDM:549) -- no overpayment.
2. If `ethPaid < totalBet`, the shortfall is pulled from `claimableWinnings[player]` (DDM:551-554). **[NEW FINDING] Off-by-one in claimable check (Low):** At DDM:552, the check is `claimableWinnings[player] <= fromClaimable`. This uses `<=` (less-than-or-equal), meaning it reverts even when the player has EXACTLY enough claimable balance. The correct check should be `<` (strict less-than). For example, if a player has exactly 1 ETH claimable and needs 1 ETH from claimable, `1 <= 1` is true and the transaction reverts. This forces players to always maintain a claimable balance strictly greater than the shortfall, which wastes 1 wei of claimable balance.
3. ETH goes to prize pools: if `prizePoolFrozen`, it routes to pending accumulators (DDM:559-560); otherwise to live `futurePrizePool` (DDM:562-563).
4. `lootboxRngPendingEth += totalBet` (DDM:565) -- tracks pending ETH toward next RNG request.

**BURNIE (DDM:568-570):** Burns tokens via `coin.burnCoin(player, totalBet)`. Updates `lootboxRngPendingBurnie`.

**WWXRP (DDM:571-573):** Burns tokens via `wwxrp.burnForGame(player, totalBet)`. No pending tracking.

### 2.7 Per-Level Tracking

ETH bets only (DDM:497-524):

**Daily hero wagers (DDM:499-512):** If hero quadrant is set (`< 4`), extracts the hero symbol from the ticket (DDM:501), scales the wager to 1e12 units (DDM:502), and packs into `dailyHeroWagers[day][heroQuadrant]` (GS:1343) at DDM:504-510. Each quadrant's uint256 packs 8 x 32-bit amounts (one per symbol).

**Per-player per-level ETH wagered (DDM:514-524):** Accumulates in `playerDegeneretteEthWagered[player][lvl]` (GS:1351) at DDM:516-517. Updates `topDegeneretteByLevel[lvl]` (GS:1355) if the player's scaled total exceeds the current top (DDM:518-523). The top player packing is `(playerScaled << 160) | uint160(player)` at DDM:522.

## 3. Bet Resolution

Entry: `resolveBets` (DDM:411-423) iterates over bet IDs, calling `_resolveBet` (DDM:577-582) which reads the packed bet and delegates to `_resolveFullTicketBet` (DDM:585-672).

### 3.1 RNG Requirement

At DDM:597-598: `uint256 rngWord = lootboxRngWordByIndex[index]` followed by `if (rngWord == 0) revert RngNotReady()`. The VRF word must have been fulfilled before resolution can proceed. The bet is deleted immediately after this check (DDM:600).

### 3.2 Per-Spin Result Derivation

The resolution loop iterates `ticketCount` spins (DDM:613-664). Each spin derives a result deterministically:

**Spin 0 (legacy, DDM:616-617):**
```solidity
resultSeed = keccak256(abi.encodePacked(rngWord, index, QUICK_PLAY_SALT))
```
where `QUICK_PLAY_SALT = 0x51` ('Q') at DDM:275.

**Spin N > 0 (DDM:618):**
```solidity
resultSeed = keccak256(abi.encodePacked(rngWord, index, spinIdx, QUICK_PLAY_SALT))
```

The `resultSeed` is converted to a packed trait ticket via `DegenerusTraitUtils.packedTraitsFromSeed(resultSeed)` (DDM:619). Each 64-bit chunk of the seed maps to one quadrant, producing a 32-bit packed ticket with format `[D:24-31][C:16-23][B:8-15][A:0-7]`, where each byte is `[QQ][CCC][SSS]` (quadrant, color category, symbol sub-bucket).

### 3.3 Match Counting (8 Attributes)

`_countMatches` (DDM:852-876) compares player and result tickets across 4 quadrants (DDM:856). For each quadrant:
- **Color match:** bits [5:3] of each quadrant byte, masked as `(pQuad >> 3) & 7` (DDM:861-862). If equal, `++matches` (DDM:864).
- **Symbol match:** bits [2:0] of each quadrant byte, masked as `pQuad & 7` (DDM:868-869). If equal, `++matches` (DDM:871).

Total possible matches: 4 quadrants x 2 attributes = 8 matches maximum.

### 3.4 Payout Calculation

`_fullTicketPayout` (DDM:912-961) computes the payout in three stages:

**Stage 1: Base payout multiplier (DDM:923)**

`_getBasePayoutBps` (DDM:990-993) returns the base multiplier in centi-x from `QUICK_PLAY_BASE_PAYOUTS_PACKED` (DDM:282-290):

| Matches | Centi-x | Multiplier | Line |
|---------|---------|------------|------|
| 0 | 0 | 0x | DDM:283 |
| 1 | 0 | 0x | DDM:284 |
| 2 | 190 | 1.90x | DDM:285 |
| 3 | 475 | 4.75x | DDM:286 |
| 4 | 1500 | 15x | DDM:287 |
| 5 | 4250 | 42.5x | DDM:288 |
| 6 | 19500 | 195x | DDM:289 |
| 7 | 100000 | 1,000x | DDM:290 |
| 8 | 10,000,000 | 100,000x | DDM:293 |

**Stage 2: ROI and bonus scaling (DDM:926-948)**

The effective ROI starts from `roiBps` (activity-score-based, DDM:926). For 5+ match buckets:
- **WWXRP:** If `wwxrpHighRoi > roiBps`, adds per-bucket bonus via `_wwxrpBonusRoiForBucket` (DDM:927-933). Factors at DDM:307-310.
- **ETH:** Always adds ETH bonus (`ETH_ROI_BONUS_BPS = 500` at DDM:213) redistributed into 5+ buckets via the same WWXRP bonus factor system (DDM:934-943).

Final payout formula (DDM:948):
```
payout = betAmount * basePayoutBps * effectiveRoi / 1,000,000
```

**Stage 3: EV normalization (DDM:952-953)**

`_evNormalizationRatio` (DDM:803-846) computes a per-outcome probability ratio across 4 quadrants. For each quadrant, weights are derived from trait rarity (DDM:822-823: bucket 0-3 = weight 10, bucket 4-6 = weight 9, bucket 7 = weight 8). Three cases per quadrant:
- Both match: `num *= 100; den *= wC * wS` (DDM:830-831)
- One match: `num *= 1300; den *= 75*(wC+wS) - 2*wC*wS` (DDM:835-836)
- No match: `num *= 4225; den *= (75-wC)*(75-wS)` (DDM:839-840)

The product `payout * evNum / evDen` (DDM:953) normalizes for trait rarity, ensuring equal expected value regardless of ticket selection.

**Stage 4: Hero quadrant boost/penalty (DDM:958-960)**

Applied for matches 2-7 when hero is enabled (DDM:958). `_applyHeroMultiplier` (DDM:966-985):
- If hero quadrant's color AND symbol both match: boost from `HERO_BOOST_PACKED` (DDM:361, indexed by `matches-2`, 16 bits each)
- Otherwise: penalty = `HERO_PENALTY = 9500` (DDM:362) = 95%
- Applied: `payout * multiplier / HERO_SCALE` (DDM:984) where `HERO_SCALE = 10_000` (DDM:363)

Hero boost values (from DDM:360): M=2: 23500 (2.35x), M=3: 14166, M=4: 11833, M=5: 10900, M=6: 10433, M=7: 10166. These are EV-neutral per match count.

## 4. ETH Payout Distribution

`_distributePayout` (DDM:680-715) handles per-spin payout delivery.

### 4.1 prizePoolFrozen Guard

At DDM:685: `if (prizePoolFrozen) revert E()`. ETH payouts are completely blocked during prize pool freeze (jackpot phase). This prevents corruption of the snapshot that `advanceGame`/`runRewardJackpots` operates on. BURNIE and WWXRP payouts are NOT blocked by this guard.

### 4.2 25/75 Split

At DDM:690-691:
```solidity
uint256 ethPortion = payout / 4;          // 25% as ETH
uint256 lootboxPortion = payout - ethPortion;  // 75% as lootbox
```

Note: integer division means for odd payouts, the lootbox portion gets the rounding remainder (up to 3 wei extra).

### 4.3 10% Pool Cap

At DDM:695-700:
```solidity
uint256 maxEth = (pool * ETH_WIN_CAP_BPS) / 10_000;
```

Where `ETH_WIN_CAP_BPS = 1_000` (DDM:224) = 10%. If `ethPortion > maxEth`, the excess is redirected to `lootboxPortion` (DDM:697) and `ethPortion` is capped at `maxEth` (DDM:698). `PayoutCapped` event emitted (DDM:699).

### 4.4 ETH Credit

At DDM:702-704:
```solidity
unchecked { pool -= ethPortion; }
_setFuturePrizePool(pool);
_addClaimableEth(player, ethPortion);
```

The `futurePrizePool` is reduced by the ETH portion. The unchecked subtraction is safe because `ethPortion <= maxEth = pool * 10%`, so `ethPortion < pool` always holds.

### 4.5 Lootbox Conversion

At DDM:707-709: If `lootboxPortion > 0`, delegates to `_resolveLootboxDirect` (DDM:741-757) which performs a delegatecall to `GAME_LOOTBOX_MODULE.resolveLootboxDirect`. This converts the ETH value into lootbox rewards with activity-score EV multiplier applied.

### 4.6 Pool Update

For non-ETH currencies:
- **BURNIE** (DDM:711): `coin.mintForGame(player, payout)` -- mints fresh BURNIE.
- **WWXRP** (DDM:713): `wwxrp.mintPrize(player, payout)` -- mints fresh WWXRP.

No pool deductions for non-ETH currencies. The bet amounts were burned at placement time, and winnings are minted fresh.

## 5. _addClaimableEth Comparison

**[CRITICAL DIFFERENCE]** The degenerette module has its own `_addClaimableEth` at DDM:1153-1159 that is fundamentally different from the version used in JackpotModule (JM:957-978), EndgameModule (EM:256-276), and DecimatorModule (DM:414-424).

### Degenerette version (DDM:1153-1159) -- NO auto-rebuy:

```solidity
function _addClaimableEth(address beneficiary, uint256 weiAmount) private {
    if (weiAmount == 0) return;
    claimablePool += weiAmount;
    _creditClaimable(beneficiary, weiAmount);
}
```

- Signature: `(address, uint256)` -- **two parameters, no RNG entropy**.
- Behavior: Always credits directly to `claimableWinnings[beneficiary]` and increments `claimablePool`.
- **Does NOT check `autoRebuyState[beneficiary].autoRebuyEnabled`.**
- **Does NOT process auto-rebuy ticket conversion.**

### JackpotModule version (JM:957-978) -- WITH auto-rebuy:

```solidity
function _addClaimableEth(address beneficiary, uint256 weiAmount, uint256 entropy)
    private returns (uint256 claimableDelta)
{
    if (weiAmount == 0) return 0;
    if (!gameOver) {
        AutoRebuyState memory state = autoRebuyState[beneficiary];
        if (state.autoRebuyEnabled) {
            return _processAutoRebuy(beneficiary, weiAmount, entropy, state);
        }
    }
    _creditClaimable(beneficiary, weiAmount);
    return weiAmount;
}
```

- Signature: `(address, uint256, uint256)` -- **three parameters, includes RNG entropy**.
- Behavior: Checks auto-rebuy state. If enabled, converts winnings to tickets via `_processAutoRebuy`. Otherwise credits claimable.
- Returns `claimableDelta` for batched `claimablePool` updates.

### EndgameModule version (EM:256-276) -- WITH auto-rebuy:

Same pattern as JM: three parameters, checks `autoRebuyState`, calls `_calcAutoRebuy` for ticket conversion.

### DecimatorModule version (DM:414-424) -- WITH auto-rebuy:

Same pattern: checks `_processAutoRebuy`, routes to tickets if enabled.

### Impact Assessment

[DISCREPANCY] **Degenerette ETH winnings bypass auto-rebuy entirely.** Players who have enabled auto-rebuy (which converts jackpot ETH winnings into tickets with a 30-45% bonus) will NOT have their degenerette winnings auto-converted. The ETH goes directly to `claimableWinnings` regardless of auto-rebuy settings.

**Severity: Informational / Design Choice.** This appears intentional -- the degenerette version lacks the `entropy` parameter needed for auto-rebuy level selection randomness. Since degenerette payouts use a separate lootbox-based distribution path (75% lootbox, 25% ETH), and the ETH portion is already capped at 10% of pool, the auto-rebuy omission may be a deliberate simplification. However, it is **undocumented** and creates inconsistent behavior for players who expect auto-rebuy to apply to all ETH winnings.

## 6. sDGNRS Rewards

`_awardDegeneretteDgnrs` at DDM:1164-1178.

### 6.1 Eligibility

Called at DDM:657-659: requires `currency == CURRENCY_ETH && matches >= 6`. Only ETH bets with 6 or more matches qualify. BURNIE and WWXRP bets never receive sDGNRS rewards.

### 6.2 BPS Rates

Defined at DDM:237-239:

| Match Count | Constant | BPS | Percentage |
|-------------|----------|-----|------------|
| 6 | `DEGEN_DGNRS_6_BPS` | 400 | 4% per ETH |
| 7 | `DEGEN_DGNRS_7_BPS` | 800 | 8% per ETH |
| 8 | `DEGEN_DGNRS_8_BPS` | 1500 | 15% per ETH |

Selected at DDM:1166-1168.

### 6.3 1 ETH Cap

At DDM:1173: `uint256 cappedBet = betWei > 1 ether ? 1 ether : betWei`. The reward calculation uses at most 1 ETH of the bet amount, regardless of actual bet size.

### 6.4 Pool Source

At DDM:1170-1171: Reads `sdgnrs.poolBalance(IStakedDegenerusStonk.Pool.Reward)`. If pool is zero, returns early. The `sdgnrs` reference is at DDM:234.

### 6.5 Transfer

Reward formula at DDM:1174:
```solidity
reward = (poolBalance * bps * cappedBet) / (10_000 * 1 ether)
```

This means for a 1 ETH bet with 6 matches: `reward = poolBalance * 400 / 10_000 = 4%` of the Reward pool balance.

Transfer at DDM:1177: `sdgnrs.transferFromPool(IStakedDegenerusStonk.Pool.Reward, player, reward)`.

**Note:** This is called per-spin (inside the `for` loop at DDM:613). A 10-spin bet with multiple 6+ match results could trigger multiple transfers, each reading the (decreasing) pool balance.

## 8. topDegeneretteByLevel Usage Analysis

### Write Path

**Single write location:** DDM:518-523 in `_placeFullTicketBetsCore`. Written only for ETH bets (`currency == CURRENCY_ETH`). Updates when `playerScaled > topAmount` (DDM:521). Format: `(playerScaled << 160) | uint160(player)` (DDM:522).

Storage declaration at GS:1355: `mapping(uint24 => uint256) internal topDegeneretteByLevel`.

### Read Paths

**Exhaustive cross-contract search results (3 files total):**

1. **DDM:518** -- Read in the same write path to compare current top against new wager (`topPacked = topDegeneretteByLevel[lvl]`).
2. **DG:2817** -- Public view function `getTopDegenerette(uint24 lvl)` at DG:2814-2820. Unpacks to return `(address topPlayer, uint256 amountUnits)`.

No other contracts read `topDegeneretteByLevel`. Specifically:
- JackpotModule: does NOT read it.
- EndgameModule: does NOT read it.
- AdvanceModule: does NOT read it.
- DecimatorModule: does NOT read it.
- GameOverModule: does NOT read it.

### Verdict

**VIEW-ONLY.** `topDegeneretteByLevel` is written by the degenerette module and exposed through a public view function but is **not consumed by any jackpot logic, payout logic, or game state transitions**. It exists purely for UI display / off-chain indexing. It does not affect prize distribution, winner selection, or any on-chain economic outcomes.

### DGN-01 Detail: Off-by-one in Claimable Balance Check

**Location:** DDM:552
**Code:** `if (claimableWinnings[player] <= fromClaimable) revert InvalidBet();`
**Expected:** `if (claimableWinnings[player] < fromClaimable) revert InvalidBet();`

The `<=` check means that when `claimableWinnings[player]` equals `fromClaimable` exactly, the transaction reverts. The subsequent line `claimableWinnings[player] -= fromClaimable` (DDM:553) would produce 0, which is a perfectly valid balance. This forces players to maintain at least 1 wei more claimable balance than the shortfall amount.

**Impact:** Low. Players can work around this by sending 1 extra wei of ETH with the transaction or by slightly increasing their claimable balance. The failure mode is a reverted transaction, not a loss of funds.
