## Executive Summary

This audit covers the two core contracts in the sDGNRS/DGNRS split -- the largest code delta since v1.1. StakedDegenerusStonk (sDGNRS) is a 520-line soulbound token that holds ETH, stETH, and BURNIE reserves, distributes pre-minted supply via five reward pools, and allows holders to burn tokens for proportional redemption. DegenerusStonk (DGNRS) is a 177-line transferable ERC20 wrapper over the creator's 20% allocation, providing standard transfer/approve functionality plus burn-through (delegates to sDGNRS.burn()) and unwrapTo (creator-only, converts DGNRS back to soulbound sDGNRS for specific recipients).

**Overall Assessment: SOUND.** Both contracts demonstrate correct CEI (Checks-Effects-Interactions) patterns throughout all external call sites. The burn function's five external calls are all preceded by state changes (balance and supply deductions at lines 398-401). Access control is correctly applied: all game-facing pool functions are `onlyGame`, the wrapper function checks `msg.sender == DGNRS`, and burn() is correctly public. The cross-contract supply invariant `sDGNRS.balanceOf[DGNRS_ADDRESS] >= DGNRS.totalSupply` holds across all modification paths. One informational finding was identified regarding stale pool balance accounting after burnRemainingPools, which is not exploitable due to the game-over terminal state guard. One low-severity finding was identified regarding DGNRS.transfer() permitting transfers to the DGNRS contract itself, which could create orphaned tokens.

## StakedDegenerusStonk.sol (DELTA-01)

### Reentrancy Analysis

The `burn()` function (lines 379-441) is the critical reentrancy surface. It makes five categories of external calls after state changes. The CEI pattern is analyzed for each.

**State changes (lines 398-401) -- executed BEFORE all external calls:**
```solidity
unchecked {
    balanceOf[player] = bal - amount;   // line 399
    totalSupply -= amount;              // line 400
}
emit Transfer(player, address(0), amount);  // line 402
```

These state changes are committed before any external call. If any callee re-enters `burn()`, the re-entrant call sees the already-reduced balance, preventing double-burn.

**External Call 1: `game.claimWinnings(address(0))` (line 405)**

- **When called:** Only if `totalValueOwed > ethBal && claimableEth != 0` (line 404)
- **Call path:** sDGNRS -> DegenerusGame.claimWinnings(address(0)) -> _resolvePlayer(address(0)) returns msg.sender (= sDGNRS address) -> _claimWinningsInternal(sDGNRS, false)
- **Game's CEI:** Sets `claimableWinnings[sDGNRS] = 1` (sentinel, line 1419) and decrements `claimablePool` (line 1422) BEFORE external ETH send
- **ETH destination:** Game sends ETH to sDGNRS via `payable(to).call{value: ethSend}("")` (line 1974 in _payoutWithStethFallback)
- **sDGNRS receive():** Has `onlyGame` modifier (line 282). Since msg.sender is the GAME contract during this call, the check passes. ETH is accepted and increases `address(this).balance`.
- **Reentrancy risk:** The game's `_claimWinningsInternal` sends ETH to sDGNRS, triggering `receive()`. This function only emits an event (line 283) -- no state mutation that could create a reentrancy vector. The game has already set the sentinel value, so re-entering claimWinnings would revert (`amount <= 1` at line 1416).
- **Verdict: SAFE.** State changes committed before call. Game's own CEI prevents double-claim. sDGNRS receive() is minimal.

**External Call 2: `coinflip.claimCoinflips(address(0), remainingBurnie)` (line 426)**

- **When called:** Only if `remainingBurnie != 0` after paying from existing BURNIE balance (lines 418-428)
- **Call path:** sDGNRS -> BurnieCoinflip.claimCoinflips(address(0), remainingBurnie) -> _resolvePlayer(address(0)) returns sDGNRS -> _claimCoinflipsAmount(sDGNRS, remainingBurnie, true)
- **Internal flow:** `_claimCoinflipsAmount` (BurnieCoinflip.sol lines 406-430) calls `_claimCoinflipsInternal` which iterates over daily flip data (storage reads + game view calls only), then calls `burnie.mintForCoinflip(sDGNRS, toClaim)` which mints BURNIE to the sDGNRS address.
- **Reentrancy risk:** `mintForCoinflip` is a standard ERC20 mint -- it updates balances and emits Transfer. BurnieCoin does NOT use ERC-777 hooks or any callback mechanism. No callback into sDGNRS.
- **Verdict: SAFE.** No callback to sDGNRS. State changes already committed.

**External Call 3: `coin.transfer(player, payBal)` (line 423) and `coin.transfer(player, remainingBurnie)` (line 427)**

- **When called:** BURNIE payout to player
- **Call path:** sDGNRS -> BurnieCoin.transfer(player, amount) -- standard ERC20 transfer
- **Reentrancy risk:** BurnieCoin is a standard ERC20 (no ERC-777 hooks). Transfer updates balances and emits Transfer. No callback to sDGNRS.
- **Verdict: SAFE.** Standard ERC20 transfer, no callbacks.

**External Call 4: `steth.transfer(player, stethOut)` (line 432)**

- **When called:** Only if `stethOut > 0` (line 431)
- **Call path:** sDGNRS -> Lido stETH.transfer(player, stethOut) -- standard ERC20 transfer
- **Reentrancy risk:** Lido stETH is standard ERC-20 (not ERC-777, as confirmed in FINAL-FINDINGS-REPORT.md finding I-20 / XCON-06). No recipient callbacks.
- **Verdict: SAFE.** stETH is standard ERC-20, no callbacks.

**External Call 5: `player.call{value: ethOut}("")` (line 436)**

- **When called:** Only if `ethOut > 0` (line 435). This is the LAST external call -- correct CEI ordering.
- **Reentrancy risk:** This sends ETH to the player. If the player is a contract with a malicious `receive()`, it could re-enter sDGNRS.burn(). However, the player's balance and totalSupply have already been updated (lines 399-400). A re-entrant burn() would:
  - Read the already-reduced `balanceOf[player]` (line 383)
  - Calculate proportional shares based on already-reduced `totalSupply` (line 385)
  - This is safe -- the re-entrant call gets a smaller share proportional to remaining balance
- **When DGNRS calls burn-through:** The "player" is the DGNRS contract, which has `receive() external payable {}` (line 79) -- a no-op that cannot re-enter.
- **Verdict: SAFE.** CEI correctly prevents double-spend. Re-entrant burn gets proportionally correct share.

**Overall reentrancy verdict: PASS.** All five external call categories are safe due to correct CEI ordering (state changes at lines 398-401 before all calls).

**Constructor pool BPS allocation (lines 195-219):**

| Pool | Constant | BPS | Calculated Amount |
|------|----------|-----|-------------------|

| Whale | WHALE_POOL_BPS | 1000 | 100,000,000,000 * 1e18 |
| Affiliate | AFFILIATE_POOL_BPS | 3500 | 350,000,000,000 * 1e18 |

| Earlybird | EARLYBIRD_POOL_BPS | 1000 | 100,000,000,000 * 1e18 |

- BPS sum: 2000 + 1000 + 3500 + 2000 + 500 + 1000 = **10,000** (exactly 100%). With INITIAL_SUPPLY = 1T * 1e18, each BPS calculation is exact (no remainder), so `totalAllocated == INITIAL_SUPPLY` and dust = 0.
- Verification: `(1_000_000_000_000e18 * 2000) / 10_000 = 200_000_000_000e18`. Each multiplication is exact because INITIAL_SUPPLY * any BPS constant < uint256.max (1e30 * 3500 = 3.5e33, well within uint256).
- Dust handling (lines 202-208): If `totalAllocated < INITIAL_SUPPLY`, dust goes to lootboxAmount. In practice, dust = 0 because BPS sum to exactly 10,000. The `unchecked { dust = INITIAL_SUPPLY - totalAllocated; }` is safe because the `if` guard ensures no underflow.
- **Verdict: CORRECT.** Pool allocation sums to exactly INITIAL_SUPPLY. Dust path is safe but never triggered.

**Minting (lines 212-213):**

- `_mint(address(this), poolTotal)` -- mints 800B sDGNRS to the sDGNRS contract itself (for pool distributions)

**Pool balance tracking in `transferFromPool` (lines 315-332):**

- `amount = min(amount, available)` (lines 321-322) -- caps to available, preventing over-spend

- `balanceOf[address(this)] -= amount` (line 326) -- decrements contract's token balance
- `balanceOf[to] += amount` (line 327) -- credits recipient
- **Invariant:** poolBalances[idx] decrement == balanceOf[address(this)] decrement == balanceOf[to] increment. This is a zero-sum transfer from the contract's balance to the recipient.

**Pool balance tracking in `transferBetweenPools` (lines 340-355):**
- Decrements `poolBalances[fromIdx]` (line 350), increments `poolBalances[toIdx]` (line 352)
- No modification to `balanceOf[address(this)]` -- correct, because tokens stay in the contract

**`burnRemainingPools` (lines 359-367):**
- Burns `balanceOf[address(this)]` -- the contract's entire token balance
- `totalSupply -= bal` -- decrements supply
- **Note:** `poolBalances[]` array is NOT zeroed. After this call, individual poolBalances entries contain stale nonzero values. However, this is not exploitable because:
  1. `burnRemainingPools` is called during game over (DegenerusGameGameOverModule.sol line 163)
  2. After `gameOver = true`, all game entry points that call `transferFromPool` revert (whale, mint, endgame, etc. all check `if (gameOver) revert E()`)
  3. Therefore, no `transferFromPool` call can occur after `burnRemainingPools`, preventing the stale `poolBalances[]` from causing an underflow in the unchecked `balanceOf[address(this)] -= amount` at line 326
- **See Finding DELTA-I-01** for details. This is informational only.

### Burn-Through Flow

The burn-through path (DGNRS.burn -> sDGNRS.burn -> forward assets) is traced step by step.

**Step 1: DGNRS._burn(msg.sender, amount) (line 126)**
```solidity
function _burn(address from, uint256 amount) private {
    uint256 bal = balanceOf[from];
    if (amount == 0 || amount > bal) revert Insufficient();  // line 170
    unchecked {
        balanceOf[from] = bal - amount;  // line 172
        totalSupply -= amount;           // line 173
    }
    emit Transfer(from, address(0), amount);  // line 175
}
```
- DGNRS balance and totalSupply are reduced BEFORE any external call. CEI correct.
- After this line: `DGNRS.balanceOf[msg.sender]` reduced, `DGNRS.totalSupply` reduced.

**Step 2: `stonk.burn(amount)` (line 128)**
- Calls sDGNRS.burn(amount) where msg.sender = DGNRS contract
- sDGNRS sees `player = msg.sender = DGNRS_ADDRESS`
- sDGNRS deducts `balanceOf[DGNRS_ADDRESS] -= amount` and `totalSupply -= amount` (lines 399-400)
- sDGNRS sends ETH/stETH/BURNIE to DGNRS contract (the "player")
- DGNRS's `receive()` accepts ETH silently (line 79)

**Step 3: DGNRS forwards BURNIE (lines 130-132)**
```solidity
if (burnieOut != 0) {
    if (!burnie.transfer(msg.sender, burnieOut)) revert TransferFailed();
}
```
- Forwards BURNIE from DGNRS contract to the actual user (msg.sender of DGNRS.burn)

**Step 4: DGNRS forwards stETH (lines 133-135)**
```solidity
if (stethOut != 0) {
    if (!steth.transfer(msg.sender, stethOut)) revert TransferFailed();
}
```
- Forwards stETH from DGNRS contract to the actual user

**Step 5: DGNRS forwards ETH (lines 136-139)**
```solidity
if (ethOut != 0) {
    (bool success, ) = msg.sender.call{value: ethOut}("");
    if (!success) revert TransferFailed();
}
```
- Forwards ETH from DGNRS contract to the actual user
- This is the LAST external call -- correct CEI ordering (ETH send is last)

**Reentrancy analysis of burn-through:**
- After Step 1: DGNRS balance and supply already reduced. Re-entering DGNRS.burn() would fail if `amount > remaining balance`.
- After Step 2: sDGNRS balance of DGNRS contract already reduced. Re-entering stonk.burn() would calculate proportional share based on reduced balance.
- Steps 3-5: Asset forwarding. DGNRS has no mutable state that could be exploited via reentrancy at this point -- both `_burn()` effects are committed.
- **Worst case reentrancy via Step 5:** User's receive() calls DGNRS.burn() again. This would re-enter Step 1 with the already-reduced balance. If the user has remaining DGNRS, they get a proportionally correct second burn. No double-spend.

**CEI verdict: CORRECT.** Both DGNRS._burn() and sDGNRS state changes precede all asset transfers.

### unwrapTo Authorization

**`unwrapTo(recipient, amount)` (lines 111-117):**

function unwrapTo(address recipient, uint256 amount) external {
    if (msg.sender != ContractAddresses.CREATOR) revert Unauthorized();  // line 112

### Constructor Correctness

**DGNRS Constructor (lines 71-77):**

### receive() Analysis

**`receive() external payable {}` (line 79)**

- **Purpose:** Accepts ETH from sDGNRS during the burn-through flow. When sDGNRS.burn() sends ETH to the DGNRS contract (the "player"), this function receives it silently.
- **Stray ETH:** Anyone can send ETH to the DGNRS contract. This ETH is not extractable except through the burn-through mechanism. When a DGNRS holder burns, sDGNRS calculates their share based on sDGNRS reserves, NOT DGNRS contract balance. The sDGNRS burn sends ETH proportional to sDGNRS reserves, then DGNRS forwards exactly that amount (`ethOut`).
- **Impact of stray ETH:** Stray ETH sent directly to DGNRS sits in the contract balance. It would only be forwarded if sDGNRS.burn() sends MORE ETH to DGNRS than the previous forward consumed. In practice, each burn-through is atomic: sDGNRS sends ethOut to DGNRS, DGNRS forwards ethOut to the user. No accumulation between calls.
- **Edge case:** If ETH is sent to DGNRS between Step 2 (sDGNRS sends to DGNRS) and Step 5 (DGNRS forwards to user), the forwarding only sends `ethOut` (the exact amount returned by stonk.burn), not the full balance. Stray ETH remains stuck.
- **Recovery:** There is no sweep function. Stray ETH is permanently locked in the DGNRS contract.

**Verdict: INFORMATIONAL.** Stray ETH is locked but does not create an exploit. No tokens at risk. See Finding DELTA-I-02.

## Cross-Contract Supply Invariant (DELTA-03)

### Invariant Statement

```
sDGNRS.balanceOf[DGNRS_ADDRESS] >= DGNRS.totalSupply   (always)
```

The gap represents sDGNRS that has been "unwrapped" -- moved from the DGNRS contract's sDGNRS balance to individual recipients via `unwrapTo()`, without corresponding DGNRS tokens existing.

### Proof

**Initial state (constructors):**
- sDGNRS constructor mints `creatorAmount` to `ContractAddresses.DGNRS` (line 212): `sDGNRS.balanceOf[DGNRS] = creatorAmount`
- DGNRS constructor reads this: `deposited = stonk.balanceOf(address(this)) = creatorAmount` (line 72)
- Sets `DGNRS.totalSupply = deposited = creatorAmount` (line 74)
- **Initial state: `sDGNRS.balanceOf[DGNRS] == DGNRS.totalSupply == creatorAmount`**
- Invariant holds: equality (gap = 0).

**Modification path 1: DGNRS.burn(amount) (burn-through)**
1. `DGNRS._burn(msg.sender, amount)`: `DGNRS.totalSupply -= amount` (line 173)
2. `stonk.burn(amount)`: sDGNRS reads `player = msg.sender = DGNRS_ADDRESS`, then `balanceOf[DGNRS_ADDRESS] -= amount` (line 399)
- **Both sides decrease by exactly `amount`.**
- If gap was G before: new gap = G - amount + amount = G. Gap unchanged.
- **Invariant preserved.**

**Modification path 2: DGNRS.unwrapTo(recipient, amount)**
1. `DGNRS._burn(msg.sender, amount)`: `DGNRS.totalSupply -= amount` (line 173) [burns from CREATOR's DGNRS balance]
2. `stonk.wrapperTransferTo(recipient, amount)`: `sDGNRS.balanceOf[DGNRS] -= amount` (line 248)
- **Both sides decrease by exactly `amount`.**
- Gap unchanged. Invariant preserved.

**Modification path 3: DGNRS.transfer / DGNRS.transferFrom**
- These modify `DGNRS.balanceOf` mappings but NOT `DGNRS.totalSupply`.
- They do NOT modify `sDGNRS.balanceOf[DGNRS]`.
- **Neither side of the invariant changes. Preserved.**

**Modification path 4: sDGNRS.transferFromPool (game distributes sDGNRS)**
- Transfers from `balanceOf[address(this)]` (the sDGNRS contract's own pools) to recipients.
- Does NOT modify `balanceOf[DGNRS_ADDRESS]` (the DGNRS contract's sDGNRS balance).
- Does NOT modify `DGNRS.totalSupply`.
- **Neither side of the invariant changes. Preserved.**

**Modification path 5: sDGNRS.burnRemainingPools**
- Burns `balanceOf[address(this)]` (sDGNRS contract's pool tokens).
- Does NOT modify `balanceOf[DGNRS_ADDRESS]`.
- Does NOT modify `DGNRS.totalSupply`.
- **Neither side of the invariant changes. Preserved.**

**Modification path 6: sDGNRS.burn called by a direct sDGNRS holder (not via DGNRS)**
- Burns from `balanceOf[holder]` where holder != DGNRS_ADDRESS.
- Does NOT modify `balanceOf[DGNRS_ADDRESS]`.
- Does NOT modify `DGNRS.totalSupply`.
- **Neither side of the invariant changes. Preserved.**

**Can DGNRS.totalSupply increase?**
- DGNRS has no `_mint` function (only `_burn`). There is no code path that increases `DGNRS.totalSupply` after construction.
- **DGNRS.totalSupply is monotonically non-increasing.**

**Can sDGNRS.balanceOf[DGNRS] increase?**
- The only paths that increase any `balanceOf` in sDGNRS are:
  - `_mint` (constructor only, private function)
  - `transferFromPool` (credits `balanceOf[to]` -- could credit DGNRS if game sends to DGNRS address)
  - `wrapperTransferTo` (credits `balanceOf[to]` -- but deducts from DGNRS, net negative for DGNRS balance)
- In practice, `transferFromPool` sends to player addresses, not the DGNRS contract. Even if it did send to DGNRS, it would increase `sDGNRS.balanceOf[DGNRS]` while `DGNRS.totalSupply` stays the same, widening the gap (strengthening the invariant).

### Conclusion

**The invariant `sDGNRS.balanceOf[DGNRS_ADDRESS] >= DGNRS.totalSupply` HOLDS at all times.**

- At construction: equality holds (gap = 0).
- burn() and unwrapTo() decrease both sides by equal amounts: gap unchanged.
- DGNRS transfers don't affect either side.
- sDGNRS pool operations don't affect either side.
- DGNRS.totalSupply can only decrease (no mint function).
- sDGNRS.balanceOf[DGNRS] can only decrease via burn() or wrapperTransferTo(), both of which reduce DGNRS.totalSupply by the same amount.

The gap `sDGNRS.balanceOf[DGNRS] - DGNRS.totalSupply` is always >= 0 and equals the cumulative amount unwrapped via `unwrapTo()`. This gap represents sDGNRS that is now soulbound in individual recipient balances, no longer backed by DGNRS tokens.

### Finding Details

**DELTA-L-01: DGNRS transfer-to-self-contract token lock**

- **Severity:** Low
- **Contract:** DegenerusStonk.sol
- **Location:** `_transfer()` (lines 156-166)
- **Description:** The `_transfer` function checks `to != address(0)` (line 157) but does not check `to != address(this)`. A user could call `transfer(DGNRS_ADDRESS, amount)` or `transferFrom(from, DGNRS_ADDRESS, amount)`, sending DGNRS tokens to the contract itself. Since DGNRS has no sweep function and no mechanism to spend tokens in its own balance, these tokens are permanently lost.
- **Impact:** Low -- requires explicit user action (sending tokens to the contract address). No protocol funds at risk. This is a common ERC20 pattern (most tokens don't guard against this).
- **Mitigation:** Consider adding `if (to == address(this)) revert ZeroAddress();` to `_transfer`. Alternatively, document as known behavior.

**DELTA-I-01: Stale poolBalances after burnRemainingPools**

- **Severity:** Informational
- **Contract:** StakedDegenerusStonk.sol
- **Location:** `burnRemainingPools()` (lines 359-367)
- **Description:** After `burnRemainingPools()` sets `balanceOf[address(this)] = 0` and decrements `totalSupply`, the individual `poolBalances[]` entries still hold their pre-burn values. If `transferFromPool` were called after this point, the unchecked `balanceOf[address(this)] -= amount` at line 326 would underflow (0 - positive = wrap-around to ~2^256).
- **Why not exploitable:** `burnRemainingPools` is only called during game over (DegenerusGameGameOverModule.sol line 163). After `gameOver = true` (line 112 in same module), all game entry points that call `transferFromPool` check `if (gameOver) revert E()` and revert. No path exists to call `transferFromPool` after `burnRemainingPools`.
- **Recommendation:** Consider adding `delete poolBalances;` or zeroing the array after the burn for defensive coding. Not required for correctness.

**DELTA-I-02: Stray ETH locked in DGNRS contract**

- **Severity:** Informational
- **Contract:** DegenerusStonk.sol
- **Location:** `receive()` (line 79)
- **Description:** The DGNRS contract's `receive() external payable {}` accepts ETH from any sender. ETH sent outside the burn-through flow is permanently locked. There is no sweep or recovery function.
- **Impact:** None on protocol security. Stray ETH does not affect burn calculations (sDGNRS.burn calculates payouts from sDGNRS reserves, not DGNRS balance). The locked ETH does not benefit or harm any party.
- **Recommendation:** Consider adding access control to `receive()` to only accept from sDGNRS address. However, this would add gas cost to every burn-through for minimal benefit.

## Open Questions Resolution

The 19-RESEARCH.md document identified four open questions. Each is resolved below.

### 1. sDGNRS.burn() reentrancy via claimWinnings

**Question:** Could `game.claimWinnings` re-enter sDGNRS?

**Resolution: NO.** The call path is: sDGNRS -> game.claimWinnings(address(0)) -> _claimWinningsInternal(sDGNRS, false) -> _payoutWithStethFallback(sDGNRS, payout). The game sends ETH to sDGNRS via `payable(to).call{value}`. sDGNRS's `receive()` (line 282) has `onlyGame` which passes (msg.sender is GAME). The `receive()` function only emits `Deposit` event (line 283) -- no state mutation, no callback. The game's own CEI (`claimableWinnings[sDGNRS] = 1` sentinel at line 1419 before ETH send) prevents double-claim. **SAFE.**

### 2. coinflip.claimCoinflips re-entry risk during burn

**Question:** Does claimCoinflips have any callback that could re-enter sDGNRS?

**Resolution: NO.** The call path is: sDGNRS -> BurnieCoinflip.claimCoinflips(address(0), remainingBurnie) -> _claimCoinflipsAmount(sDGNRS, remainingBurnie, true) -> _claimCoinflipsInternal (storage reads + game view calls) -> burnie.mintForCoinflip(sDGNRS, toClaim). BurnieCoin.mintForCoinflip is a standard ERC20 mint operation -- it updates balance/supply and emits Transfer. No ERC-777 hooks, no callback into sDGNRS. **SAFE.**

### 3. DGNRS receive() safety

**Question:** Could anyone send ETH to DGNRS outside of the burn flow, permanently locking it?

**Resolution: YES, but it's harmless.** DGNRS's `receive() external payable {}` (line 79) has no access restriction -- anyone can send ETH. This ETH is permanently locked because DGNRS only forwards the exact `ethOut` returned by `stonk.burn()`, not its full balance. The stray ETH does not affect sDGNRS reserve calculations and does not benefit or harm any party. See Finding DELTA-I-02.

### 4. Pool balance consistency after burnRemainingPools

**Question:** Could poolBalances[] entries get out of sync with the actual balanceOf[address(this)]?

**Resolution: YES, after burnRemainingPools, but not exploitable.** After `burnRemainingPools()`, `balanceOf[address(this)] = 0` but `poolBalances[]` entries retain stale values. The `transferFromPool` function would underflow in unchecked arithmetic if called after this point. However, this path is unreachable because all game entry points that call `transferFromPool` check `if (gameOver) revert E()`, and `burnRemainingPools` is only called during game over. See Finding DELTA-I-01.

**Proof that poolBalances and balanceOf[this] are in sync BEFORE burnRemainingPools:**
- Constructor: `balanceOf[address(this)] = poolTotal = sum(all pool amounts)`. Each `poolBalances[i]` is set to the corresponding amount. Invariant: `balanceOf[address(this)] == sum(poolBalances[i])`.
- `transferFromPool`: decrements both `poolBalances[idx]` and `balanceOf[address(this)]` by the same `amount` (lines 325-326). Invariant preserved.
- `transferBetweenPools`: modifies `poolBalances` entries only (zero-sum rebalance). Does NOT touch `balanceOf[address(this)]`. Invariant preserved.
- No other function modifies `poolBalances` or `balanceOf[address(this)]` (except `burnRemainingPools` itself).
- **Conclusion:** Before `burnRemainingPools`, the invariant `balanceOf[address(this)] == sum(poolBalances[i])` always holds. Therefore `burnRemainingPools` correctly burns the sum of all pool tokens.
