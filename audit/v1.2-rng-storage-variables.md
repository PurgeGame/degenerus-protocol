### 2b. Ticket Queue and Double-Buffer State

The double-buffer uses readKey/writeKey encoding: `_tqWriteKey(level)` returns the write buffer key, `_tqReadKey(level)` returns the read buffer key. The writeKey uses `ticketWriteSlot` to select buffer 0 or 1 (bit 23 toggle). New tickets go to writeKey; jackpot processing reads from readKey. `_swapTicketSlot` toggles the active buffer when the read slot is empty.

## Section 3: Detailed Lifecycle Traces

### 3.1 `lastLootboxRngWord` Lifecycle

**Declaration:** `contracts/storage/DegenerusGameStorage.sol:1397` (slot 76, `uint256`)

**Purpose:** Caches the most recently resolved lootbox VRF word. Used as entropy source for trait assignment during `processTicketBatch` in JackpotModule. This avoids re-reading the lootbox mapping and ensures trait entropy is deterministic based on the latest available VRF word at processing time.

#### Write Sites

| # | Location | Value Written | Condition |
|---|----------|---------------|-----------|
| W1 | AdvanceModule:166 | `lootboxRngWordByIndex[lootboxRngIndex - 1]` | During `advanceGame` when `midDayTicketRngPending == true` and tickets have been drained. Copies the most recent lootbox word into this cache. |
| W2 | AdvanceModule:789 | `rngWord` (from VRF callback) | Inside `_handleLootboxRngCallback`, called from both `rngGate` (stale daily word routed to lootbox) and `rawFulfillRandomWords` (direct lootbox callback). Always the raw VRF word. |

#### Read Sites

| # | Location | Value Read For | Decision Driven |
|---|----------|----------------|-----------------|
| R1 | JackpotModule:1975 | Trait assignment entropy | `processTicketBatch` uses this as the base entropy for generating traits via `keccak256(lastLootboxRngWord, player, queueIdx, ...)`. Determines which of 256 trait buckets each ticket enters. |

#### Clear/Overwrite Behavior

`lastLootboxRngWord` is **never zeroed**. It is overwritten each time a lootbox RNG resolves (W2) or when mid-day ticket processing consumes a lootbox word (W1). After the very first daily advance that triggers a lootbox RNG, this variable always holds a valid VRF-derived value.

**Initial value:** 0 (before first lootbox RNG resolution). JackpotModule:1975 handles this: if `entropy == 0`, the function uses a fallback path.

#### State Machine Diagram

```
            [0 / uninitialized]
                    |
                    | First lootbox RNG resolves (W2)
                    v
    +-------------------------------+
    |   VALID (holds VRF word)      |<-----------+
    |                               |             |
    |  Value = latest resolved      |   W2: _handleLootboxRngCallback
    |  lootbox VRF word             |   (stale daily routed to lootbox
    |                               |    OR direct lootbox callback)
    +-------------------------------+             |
            |                 |                   |
            |                 +-------------------+
            |
            | W1: advanceGame mid-day path
            | (copies lootboxRngWordByIndex[idx-1])
            v
    +-------------------------------+
    |   VALID (updated cache)       |
    |   Value = lootboxRngWordByIndex|
    |   [lootboxRngIndex - 1]       |
    +-------------------------------+
            |
            | R1: processTicketBatch reads
            | (used as trait entropy seed)
            v
        [No state change on read]
```

**Key invariant:** After the first lootbox RNG resolution, `lastLootboxRngWord` always holds a valid VRF-derived value. It is never zeroed or reset. Each new lootbox resolution overwrites the previous value.

### 3.2 `midDayTicketRngPending` Lifecycle

**Declaration:** `contracts/storage/DegenerusGameStorage.sol:1401` (slot 77, `bool`)

**Purpose:** Signals that `requestLootboxRng` performed a mid-day ticket buffer swap and there are unprocessed tickets in the read buffer that need draining before the next swap can occur. This flag bridges the gap between the lootbox RNG request (which swaps buffers) and the next `advanceGame` call (which drains the read buffer).

#### Write Sites

| # | Location | Value Written | Condition |
|---|----------|---------------|-----------|
| W1 | AdvanceModule:715 | `true` | Inside `requestLootboxRng`, after the condition `ticketQueue[wk].length > 0 && ticketsFullyProcessed` is met. The function calls `_swapAndFreeze(purchaseLevel)` at line 714, then sets this flag. This means: (a) write queue had tickets, (b) read queue was already drained, (c) buffers have been swapped, (d) new read queue needs draining. |
| W2 | AdvanceModule:183 | `false` | Inside `advanceGame`, after the mid-day ticket drain loop completes. Condition: `midDayTicketRngPending` was true (checked at line 182) and `ticketsFullyProcessed` is now true (set at line 178 when read queue is empty). |

#### Read Sites

| # | Location | Value Read For | Decision Driven |
|---|----------|----------------|-----------------|
| R1 | AdvanceModule:162 | Mid-day drain entry | If `midDayTicketRngPending == true`, `advanceGame` enters the mid-day ticket processing path instead of normal daily logic. Processes tickets from the newly-swapped read buffer. |
| R2 | AdvanceModule:182 | Drain completion check | After ticket processing iteration, if `midDayTicketRngPending` is still true and `ticketsFullyProcessed` is true, clears the flag (W2) and updates `lastLootboxRngWord`. |

#### Clear/Overwrite Behavior

- **Set to true** only by `requestLootboxRng` (W1), and only when there were tickets in the write queue and the read queue was already drained.
- **Set to false** only by `advanceGame` (W2), after it finishes draining the mid-day read buffer.
- **Never read outside AdvanceModule.** This is purely internal sequencing state.

#### State Machine Diagram

```
    +---------------------------+
    |  FALSE (idle)             |
    |  Normal advanceGame flow  |
    +---------------------------+
            |
            | W1: requestLootboxRng()
            | Conditions:
            |   1. ticketQueue[writeKey].length > 0
            |   2. ticketsFullyProcessed == true
            |   3. _swapAndFreeze() executed
            |
            v
    +---------------------------+
    |  TRUE (pending drain)     |
    |  Buffers swapped;         |
    |  read queue has tickets   |
    +---------------------------+
            |
            | R1: advanceGame() enters
            |     mid-day drain path
            |     (line 162)
            |
            v
    +---------------------------+
    |  TRUE (draining)          |
    |  processTicketBatch()     |
    |  iterating read queue     |
    +---------------------------+
            |
            | Tickets drained:
            |   ticketsFullyProcessed = true
            |   (AdvanceModule:178)
            |
            | R2 + W2: Check flag (182),
            |   clear to false (183),
            |   update lastLootboxRngWord (166)
            |
            v
    +---------------------------+
    |  FALSE (idle)             |
    |  Resume normal flow       |
    +---------------------------+
```

**Key invariant:** `midDayTicketRngPending` is true if and only if a mid-day buffer swap has occurred and the new read buffer has not yet been fully drained by `advanceGame`. It is a single-producer (`requestLootboxRng`) / single-consumer (`advanceGame`) flag.

**Security note:** While `midDayTicketRngPending` is true, `requestLootboxRng` cannot fire again because `ticketsFullyProcessed` is false (reset by `_swapTicketSlot` at line 736). This prevents double-swaps and ensures exactly one pending drain at a time.
