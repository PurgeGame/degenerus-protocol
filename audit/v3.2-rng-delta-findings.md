| 2 | `claimCoinflipsFromBurnie()` (line 339) | Removed `if (degenerusGame.rngLocked()) revert RngLocked()` guard before `_claimCoinflipsAmount` call | Was at function entry, before line 343 |

### Carry Isolation Trace

The safety argument for removing `rngLocked` from claim paths rests on **carry isolation**: the `autoRebuyCarry` field (rolling bankroll for future flips) and `claimableStored` field (take-profit amounts available for withdrawal) are separate storage fields with separate write paths. Claims only expose `claimableStored + mintable`, and `mintable` never contains carry while auto-rebuy is enabled.

#### Storage Layout (PlayerCoinflipState struct)

```solidity
struct PlayerCoinflipState {
    uint128 claimableStored;     // Take-profit: withdrawable at any time
    uint48 lastClaim;
    uint48 autoRebuyStartDay;
    bool autoRebuyEnabled;
    uint128 autoRebuyStop;
    uint128 autoRebuyCarry;      // Rolling bankroll: bets on future flips
}
```

#### Write Paths: `autoRebuyCarry`

| # | Location | Function | Value Written | Conditions | Can Carry Cross to Claimable? |
|---|----------|----------|---------------|------------|-------------------------------|
| 1 | Line 578 | `_claimCoinflipsInternal()` | `uint128(carry)` | `rebuyActive == true` AND `oldCarry != carry` (carry changed during day processing) | **NO** -- carry is written back to its own storage field. The local `carry` variable holds payout minus take-profit portions. `mintable` only receives `reserved` amounts (line 503). |
| 2 | Line 420 | `_claimCoinflipsInternal()` | `0` (zeroed) | `rebuyActive == false` AND `oldCarry != 0` (leftover carry from previously-enabled auto-rebuy) | **YES, but safely** -- carry is added to `mintable` at line 419, but this path only executes when `rebuyActive == false`, meaning auto-rebuy is already disabled. Disabling auto-rebuy requires `rngLocked == false` (line 691). |
| 3 | Line 722 | `_setCoinflipAutoRebuy()` | `0` (zeroed) | Called when disabling auto-rebuy (`enabled == false`). Carry is added to `mintable` at line 721. | **YES, but guarded** -- this function has its own `rngLocked` guard at line 691. Carry can only enter `mintable` via this path when RNG is unlocked. |

**Key observation:** Path 1 is the hot path during normal auto-rebuy operation. Carry stays in `autoRebuyCarry` storage. Paths 2 and 3 are the only paths where carry enters `mintable`, and both require auto-rebuy to be disabled (which itself requires `rngLocked == false`).

#### Write Paths: `claimableStored`

| # | Location | Function | Value Written | Conditions | Source of Value |
|---|----------|----------|---------------|------------|-----------------|
| 1 | Line 219 | `settleFlipModeChange()` | `state.claimableStored + mintable` (addition) | Called by game contract before afKing mode changes. `mintable` comes from `_claimCoinflipsInternal(player, false)`. | `mintable` = take-profit reserved amounts (when `rebuyActive`) or full payout (when `!rebuyActive`). Never carry while rebuy is active. |
| 2 | Line 260 | `_depositCoinflip()` | `state.claimableStored + mintable` (addition) | Called during new deposits. `mintable` comes from `_claimCoinflipsInternal(caller, false)`. | Same as path 1. `mintable` from `_claimCoinflipsInternal` never contains carry while rebuy is active. |
| 3 | Line 374 | `_claimCoinflipsAmount()` | `uint128(stored - toClaim)` (deduction) | Called from `claimCoinflips`, `claimCoinflipsFromBurnie`, `consumeCoinflipsForBurn`. `stored = state.claimableStored + mintable` (line 366). | Deduction from existing stored balance. `mintable` from `_claimCoinflipsInternal` never contains carry while rebuy is active. |

**Key observation:** All three `claimableStored` write paths source their `mintable` value from `_claimCoinflipsInternal()`. The carry isolation invariant therefore depends entirely on proving that `_claimCoinflipsInternal()` never puts carry into `mintable` while auto-rebuy is enabled.

### Formal Invariant

```
INVARIANT: While state.autoRebuyEnabled == true, no execution path from
claimCoinflips(), claimCoinflipsFromBurnie(), or consumeCoinflipsForBurn()
can cause state.autoRebuyCarry to be added to mintable, claimableStored,
or any value returned to msg.sender.
```

**Proof from code:**

1. All three claim functions call `_claimCoinflipsAmount()` (lines 332, 343, 353).

2. `_claimCoinflipsAmount()` calls `_claimCoinflipsInternal(player, false)` at line 365. The second argument `deepAutoRebuy` is always `false` for claim paths. This does not affect carry isolation (it only controls the claim window size).

3. Inside `_claimCoinflipsInternal()`, the local variable `rebuyActive` is set from `state.autoRebuyEnabled` at line 396:
   ```solidity
   bool rebuyActive = state.autoRebuyEnabled;
   ```

4. **When `rebuyActive == true`:** The carry-to-mintable transfer path (lines 418-421) is gated by `!rebuyActive`:
   ```solidity
   if (rebuyActive) {
       carry = oldCarry;            // line 417: carry stays local
   } else if (oldCarry != 0) {
       mintable += oldCarry;        // line 419: carry -> mintable (ONLY when !rebuyActive)
       state.autoRebuyCarry = 0;    // line 420: zero storage
   }
   ```
   Since `rebuyActive == true`, the `else if` branch is unreachable. Carry stays in the local `carry` variable.

5. **During day processing (win path, lines 498-508):** When `rebuyActive == true`:
   ```solidity
   if (rebuyActive) {
       if (takeProfit != 0) {
           uint256 reserved = (payout / takeProfit) * takeProfit;
           mintable += reserved;     // line 503: ONLY take-profit goes to mintable
           carry = payout - reserved; // line 505: remainder stays in carry
       } else {
           carry = payout;           // line 507: ALL goes to carry
       }
   }
   ```
   Only the take-profit `reserved` portion enters `mintable`. The remainder stays in the local `carry` variable.

6. **After the loop (line 577-578):** Carry is written back to storage, not to mintable:
   ```solidity
   if (rebuyActive && oldCarry != carry) {
       state.autoRebuyCarry = uint128(carry);  // line 578
   }
   ```

7. **Return value:** `mintable` is returned at line 585. It contains only take-profit reserved amounts (never carry) when `rebuyActive == true`.

8. **Back in `_claimCoinflipsAmount()`** (line 366):
   ```solidity
   uint256 stored = state.claimableStored + mintable;
   ```
   Since `mintable` never contains carry (per step 7), `stored` never contains carry. The value returned to the player (minted at line 379) comes from `toClaim <= stored`, which is carry-free.

**QED:** The carry isolation invariant holds by construction. The `rngLocked` guard on claim paths was defense-in-depth but never the mechanism preventing carry leakage. The mechanism is the `rebuyActive` branch at lines 416-421 and 498-508.

### Attacker Model Evaluation

#### Attacker Model 1: MEV-Aware Attacker (1000 ETH budget)

**Scenario:** Attacker observes the VRF random word in the mempool (via `rawFulfillRandomWords` calldata) before it is processed by `advanceGame`. They know the coinflip outcome (win or loss) for the current day before it is recorded.

**Attack vector:** Can the attacker front-run `advanceGame` with a claim transaction to extract carry before a known loss?

**Analysis:**
- The attacker calls `claimCoinflips()` which calls `_claimCoinflipsAmount()` -> `_claimCoinflipsInternal()`.
- `_claimCoinflipsInternal()` processes days up to `flipsClaimableDay` (line 393). The current day's result is NOT yet recorded (that happens in `processCoinflipPayouts` during `advanceGame`).
- Therefore, the claim only processes already-resolved days. The pending day (which the attacker knows will be a loss) is not claimable yet.
- Carry isolation holds regardless: even if claims processed the pending day, `mintable` would not contain carry (per the invariant proof above).

**Verdict:** No advantage. The attacker gains nothing from claiming during RNG lock because (a) the pending day is not yet claimable, and (b) carry isolation holds independently of RNG state.

#### Attacker Model 2: Compromised VRF Operator (1000 ETH budget)

**Scenario:** A compromised Chainlink VRF operator knows the random word before delivering it. They can choose when to deliver the word and can potentially delay delivery.

**Attack vector:** Can the operator manipulate carry by choosing favorable delivery timing combined with claim transactions?

**Analysis:**
- The operator knows the word but cannot change it (VRF proof verification prevents this).
- The operator can delay delivery, but this only extends the RNG lock window. It does not change which days are claimable (still limited to `flipsClaimableDay`).
- Even with advance knowledge, the operator cannot cause carry to enter `mintable` while auto-rebuy is enabled. The isolation is structural (code branching), not temporal (timing-dependent).
- The operator could theoretically toggle auto-rebuy off to extract carry, but `_setCoinflipAutoRebuy()` has its own `rngLocked` guard at line 691, which prevents toggling during RNG lock regardless of VRF operator behavior.

**Verdict:** No advantage. Carry isolation is a structural property of the code, not a timing property. The compromised VRF operator cannot bypass it.

### RNG-01 Verdict

**SAFE** -- Carry isolation holds by construction: the `rebuyActive` branch (lines 416-421, 498-508) prevents carry from entering `mintable` while auto-rebuy is enabled, regardless of RNG lock state. The removed `rngLocked` guards were defense-in-depth, not the mechanism preventing carry leakage. The remaining `rngLocked` guard on `_setCoinflipAutoRebuy()` (line 691) prevents the only carry-extraction path (toggling auto-rebuy off during RNG lock).

## RNG-02: BAF Epoch-Based Guard Sufficiency

### BAF Guard Location and Conditions

The BAF guard is located in `_claimCoinflipsInternal()` at BurnieCoinflip.sol lines 542-563. It is the sole remaining RNG-sensitive protection on coinflip claim paths after the four `rngLocked` removals.

```solidity
// BurnieCoinflip.sol lines 542-563
if (winningBafCredit != 0 && player != ContractAddresses.SDGNRS) {  // line 542
    if (!levelCached) {                                              // line 543
        cachedLevel = game.level();                                  // line 544
        levelCached = true;                                          // line 545
    }                                                                // line 546
    (                                                                // line 547
        uint24 purchaseLevel_,                                       // line 548
        bool inJackpotPhase,                                         // line 549
        bool lastPurchaseDay_,                                       // line 550
        bool rngLocked_,                                             // line 551
                                                                     // line 552
    ) = game.purchaseInfo();                                         // line 553
    bool over = game.gameOver();                                     // line 554
    if (                                                             // line 555
        !inJackpotPhase &&                                           // line 556
        !over &&                                                     // line 557
        lastPurchaseDay_ &&                                          // line 558
        rngLocked_ &&                                                // line 559
        (purchaseLevel_ % 10 == 0)                                   // line 560
    ) {                                                              // line 561
        revert RngLocked();                                          // line 562
    }                                                                // line 563
    // ... recordBafFlip follows at line 569 ...
}
```

**All 7 conditions that must be true for the guard to revert:**

| # | Condition | Code | Meaning |
|---|-----------|------|---------|
| 1 | `winningBafCredit != 0` | line 542 | Player has winning BAF flips to record for this claim batch |
| 2 | `player != ContractAddresses.SDGNRS` | line 542 | Player is NOT the sDGNRS staking contract |
| 3 | `!inJackpotPhase` | line 556 | Game is in purchase phase (not jackpot phase) |
| 4 | `!over` | line 557 | Game is not over |
| 5 | `lastPurchaseDay_` | line 558 | Purchase target met, level is transitioning |
| 6 | `rngLocked_` | line 559 | VRF request is currently in flight |
| 7 | `purchaseLevel_ % 10 == 0` | line 560 | Current purchase level is a BAF resolution level (multiple of 10) |

If ANY condition is false, the guard does NOT revert and the claim proceeds normally (with BAF credit being recorded via `recordBafFlip` at line 569).

### BAF Guard Bypass Enumeration

For each of the 7 conditions, this section documents: what makes it false (legitimate bypass), whether an attacker can force it false, and whether bypassing creates a security issue.

#### Condition 1: `winningBafCredit != 0`

**What makes it false:** Player has no winning flips to record in this claim batch. This happens when the player had no coinflip stakes during the BAF-eligible days, or all their flips were losses.

**Can attacker force it false?** No. `winningBafCredit` is accumulated from actual coinflip results recorded on-chain (line 496: `winningBafCredit += payout` only on `cursor > bafResolvedDay` for winning days). An attacker cannot retroactively change flip results.

**Security impact of bypass:** None. If there are no winning BAF credits, there is nothing to record. The `recordBafFlip` call that follows the guard would be a no-op anyway (zero amount).

#### Condition 2: `player != ContractAddresses.SDGNRS`

**What makes it false:** Player IS the sDGNRS staking contract. This is a hardcoded constant address comparison.

**Can attacker force it false?** No. The sDGNRS address is a constant (`ContractAddresses.SDGNRS`). An attacker cannot impersonate it.

**Security impact of bypass:** None -- see "sDGNRS BAF Ineligibility Proof" section below. sDGNRS is truly ineligible for BAF at both caller and callee layers.

#### Condition 3: `!inJackpotPhase`

**What makes it false:** Game IS in jackpot phase. During jackpot phase, BAF has already been resolved for this level (BAF jackpot fires at the purchase-to-jackpot transition, not during jackpot phase).

**Can attacker force it false?** No. Phase transitions are controlled by `advanceGame()` which requires VRF words and specific game state conditions. An attacker cannot force a phase transition.

**Security impact of bypass:** None. During jackpot phase, BAF leaderboard data has already been consumed by `runBafJackpot()`. Any credits recorded during jackpot phase will be accumulated for the NEXT BAF resolution (next multiple-of-10 level), which is the correct behavior.

#### Condition 4: `!over`

**What makes it false:** Game IS over (`gameOver == true`). This is a terminal state.

**Can attacker force it false?** No. Game over is set by `_handleGameOverPath` in AdvanceModule and requires liveness guard timeout (120 days inactivity). Irreversible.

**Security impact of bypass:** None. When the game is over, no further BAF resolutions will occur. BAF credits are meaningless.

#### Condition 5: `lastPurchaseDay_`

**What makes it false:** Purchase target is NOT yet met. The game is in normal purchase phase, still accumulating prize pool toward the level target.

**Can attacker force it false?** No. `lastPurchaseDay` is set when `_getNextPrizePool() >= levelPrizePool[lvl]` (AdvanceModule lines 131-134). An attacker cannot reduce the prize pool below the target to unset this flag, and the flag is only set to true by the game contract itself.

**Security impact of bypass:** None. If `lastPurchaseDay` is false, the level is not transitioning, so no BAF resolution is imminent. Recording BAF credit during normal purchase phase is correct behavior -- it builds the leaderboard for the eventual BAF resolution.

#### Condition 6: `rngLocked_`

**What makes it false:** RNG is NOT locked. No VRF request is in flight.

**Can attacker force it false?** No. `rngLockedFlag` is set by `_finalizeRngRequest()` at AdvanceModule line 1207 and cleared by `_unlockRng()` at AdvanceModule line 1283. Both are internal functions callable only within the `advanceGame()` execution flow. An attacker cannot clear the RNG lock.

**Security impact of bypass:** None -- this is the key insight. When RNG is not locked, there is no pending VRF word to front-run. BAF credit recording during RNG-unlocked periods is completely safe because the BAF jackpot resolution (which consumes the leaderboard) only happens during `advanceGame` after `rngGate` processes a VRF word.

#### Condition 7: `purchaseLevel_ % 10 == 0`

**What makes it false:** Current purchase level is NOT a multiple of 10 (e.g., level 13, 27, etc.). BAF jackpots only fire at levels 10, 20, 30, etc.

**Can attacker force it false?** No. The purchase level is derived from `level + 1` (or `level` when lastPurchaseDay and rngLocked). Level increments are controlled by `_finalizeRngRequest()` at AdvanceModule line 1219 and only occur on fresh RNG requests during lastPurchaseDay.

**Security impact of bypass:** None. Non-BAF levels (not multiples of 10) have no BAF resolution. Credits recorded at non-BAF levels accumulate toward the next BAF level bracket (handled by `_bafBracketLevel()` at BurnieCoinflip line 1091).

### sDGNRS BAF Ineligibility Proof

sDGNRS (the staked DegenerusStonk contract) is proven BAF-ineligible at two independent layers:

#### Caller-side: BurnieCoinflip.sol (line 542)

```solidity
if (winningBafCredit != 0 && player != ContractAddresses.SDGNRS) {
    // ... entire BAF section skipped for sDGNRS ...
}
```

When `player == ContractAddresses.SDGNRS`, the condition `player != ContractAddresses.SDGNRS` evaluates to `false`, and the entire BAF block (lines 542-570) is skipped. No `purchaseInfo()` call, no guard check, no `recordBafFlip` call.

#### Callee-side: DegenerusJackpots.sol (line 175)

```solidity
function recordBafFlip(address player, uint24 lvl, uint256 amount) external override onlyCoin {
    if (player == ContractAddresses.VAULT || player == ContractAddresses.SDGNRS) return;
    // ... BAF recording logic ...
}
```

Even if the caller-side skip were somehow bypassed, `recordBafFlip` returns immediately for `ContractAddresses.SDGNRS` at line 175. No BAF totals are accumulated, no leaderboard position is updated.

#### processCoinflipPayouts Path (line 846)

The critical sDGNRS claim path runs during `advanceGame`:

```solidity
// BurnieCoinflip.sol line 846 (in processCoinflipPayouts)
_claimCoinflipsInternal(ContractAddresses.SDGNRS, false);
```

This call occurs during active RNG lock (called from `advanceGame` -> `rngGate` -> `processCoinflipPayouts` at AdvanceModule line 764). The call chain is:

1. `advanceGame()` calls `rngGate()` (AdvanceModule line 228)
2. `rngGate()` calls `coinflip.processCoinflipPayouts()` (AdvanceModule line 764)
3. `processCoinflipPayouts()` calls `_claimCoinflipsInternal(ContractAddresses.SDGNRS, false)` (BurnieCoinflip line 846)
4. Inside `_claimCoinflipsInternal()`, the sDGNRS skip at line 542 bypasses the entire BAF section
5. The BAF guard (lines 555-562) is never reached for sDGNRS

**Without the sDGNRS skip:** If the BAF guard applied to sDGNRS, `processCoinflipPayouts` would always revert during RNG lock because conditions 5 and 6 (`lastPurchaseDay_` and `rngLocked_`) would typically be true during the transition path. This would break the game's daily advance flow.

#### sDGNRS Ineligibility Conclusion

sDGNRS is truly BAF-ineligible, not just "skipping the guard." The exclusion operates at both the caller-side (BurnieCoinflip line 542 skips the entire BAF section) and the callee-side (DegenerusJackpots line 175 returns early). This dual-layer exclusion ensures sDGNRS can never accumulate BAF credit or appear on BAF leaderboards, making BAF guard analysis irrelevant for the sDGNRS path.

### Reentrancy Analysis

**Question:** Can an attacker use reentrancy to bypass the BAF guard?

**Check 1: External calls before BAF guard**

Inside `_claimCoinflipsInternal()`, before the BAF guard at line 542:
- Line 392: `game.syncAfKingLazyPassFromCoin(player)` -- external call to DegenerusGame
- Line 492: `jackpots.getLastBafResolvedDay()` -- external view call to DegenerusJackpots

Both calls occur before the BAF guard check. However:
- `syncAfKingLazyPassFromCoin` is a state-sync function on DegenerusGame. Even if an attacker could reenter during this call, they cannot change `purchaseInfo()` return values (which depend on game-level state variables set by `advanceGame`).
- `getLastBafResolvedDay` is a `view` function -- no state changes possible.

**Check 2: Reentrancy protection**

The claim functions (`claimCoinflips`, `claimCoinflipsFromBurnie`, `consumeCoinflipsForBurn`) do NOT have explicit `nonReentrant` modifiers. However:
- `claimCoinflipsFromBurnie` and `consumeCoinflipsForBurn` are restricted to `onlyBurnieCoin` (BurnieCoin contract only).
- BurnieCoin's `_spendAllowance` / `transferFrom` paths that call `claimCoinflipsFromBurnie` use CEI pattern (checks-effects first, external call last).
- `claimCoinflips` can be called by anyone (via `_resolvePlayer`), but the mint at line 379 (`burnie.mintForCoinflip`) happens after state updates (line 374: `state.claimableStored = ...`), following CEI.

**Check 3: State manipulation via reentrancy**

Even if an attacker achieved reentrancy into `_claimCoinflipsInternal` during the `mintForCoinflip` call:
- `state.lastClaim` has already been updated (line 574), so the reentrant call would process 0 days.
- `state.claimableStored` has already been updated (line 374 in the outer `_claimCoinflipsAmount`).
- The 7 BAF guard conditions read from `game.purchaseInfo()` and `game.gameOver()` -- game-level state that cannot be changed by coinflip reentrancy.

**Reentrancy Verdict:** No reentrancy-based BAF guard bypass is possible. The guard conditions depend on game-level state (not coinflip state), and coinflip state updates follow CEI pattern.

**Can an attacker submit a claim that executes after _unlockRng but during the same advanceGame call?**

The `emit Advance(stage, lvl)` and `coin.creditFlip(caller, advanceBounty)` at lines 375-376 happen after unlockRng, but:
- `emit` is not an external call
- `coin.creditFlip` calls BurnieCoinflip which calls `_addDailyFlip` (not a claim path)

**Can an attacker submit a separate transaction that gets included after _unlockRng in the same block?**

No. `_unlockRng` executes within the `advanceGame` transaction. Once the transaction completes, RNG is unlocked. ### Deposit Lock Layer

A separate protection layer exists for deposits during BAF resolution windows.

**Location:** BurnieCoinflip.sol, `_coinflipLockedDuringTransition()` at lines 985-998:

```solidity
function _coinflipLockedDuringTransition()
    private
    view
    returns (bool locked)
{
    (
        uint24 purchaseLevel_,
        bool inJackpotPhase,
        bool lastPurchaseDay_,
        bool rngLocked_,

    ) = degenerusGame.purchaseInfo();
    locked = (!inJackpotPhase) && !degenerusGame.gameOver() && lastPurchaseDay_ && rngLocked_ && (purchaseLevel_ % 10 == 0);
}
```

This function is called at `_depositCoinflip()` line 255:
```solidity
if (_coinflipLockedDuringTransition()) revert CoinflipLocked();
```

**Purpose:** Deposits trigger `_claimCoinflipsInternal()` (line 258), which would record BAF credit. By blocking deposits during BAF resolution levels, this prevents:
1. Players from front-running BAF resolution by depositing large amounts that would boost their BAF credit via auto-claim
2. Auto-claim from updating the BAF leaderboard mid-resolution

**Relationship to BAF guard:** This is a complementary protection layer. The BAF guard in `_claimCoinflipsInternal` reverts if claims would record BAF credit during resolution. The deposit lock prevents the deposit -> auto-claim -> BAF credit path entirely. Together, they form a two-layer defense:
- **Layer 1 (deposit lock):** Blocks deposits that would trigger auto-claim during BAF resolution
- **Layer 2 (BAF guard):** Reverts if direct claims would record BAF credit during BAF resolution

Both layers use the same 5 conditions (`!inJackpotPhase`, `!gameOver`, `lastPurchaseDay_`, `rngLocked_`, `purchaseLevel_ % 10 == 0`), ensuring consistent protection.

### Attacker Model Evaluation

#### Attacker Model 1: MEV-Aware (1000 ETH budget)

**Scenario:** Attacker observes the VRF word in the mempool (via `rawFulfillRandomWords` calldata) and knows the BAF resolution outcome before `advanceGame` processes it.

**Attack vector 1: Front-run with claims to manipulate BAF leaderboard.**
The attacker calls `claimCoinflips()` before `advanceGame` processes the VRF word. They have winning BAF credit to record.

**Analysis:** The BAF guard catches this exactly. During the BAF resolution window (lastPurchaseDay, rngLocked, BAF level), all 7 conditions are true, and the claim reverts with `RngLocked()`. The attacker cannot record BAF credit before the resolution.

**Attack vector 2: Back-run advanceGame to record BAF credit for the NEXT level.**
After `advanceGame` processes and unlocks RNG, the attacker claims immediately to record BAF credit for the next BAF bracket.

**Analysis:** This is legitimate behavior. After `_unlockRng()`, the current level has incremented, and BAF resolution for the previous level is complete. Recording BAF credit for the next level bracket is correct -- it builds the leaderboard for the next resolution. No advantage is gained because the attacker's credit is proportional to their actual coinflip stakes.

**MEV-Aware Verdict:** No advantage. The BAF guard prevents credit recording during the exact window where it could affect the current resolution, and post-unlock recording is legitimate.

#### Attacker Model 2: Compromised VRF Operator (1000 ETH budget)

**Scenario:** A compromised Chainlink VRF operator knows the random word before delivering it and can choose delivery timing.

**Attack vector 1: Delay VRF delivery to extend the BAF resolution window.**
The operator delays `rawFulfillRandomWords` to keep `rngLocked_ == true` for longer, preventing other players from recording BAF credit.

**Analysis:** This extends the window where the BAF guard reverts, but it does NOT help the attacker. The attacker's own claims would also be blocked by the same guard. The operator could delay indefinitely, but after 12 hours, `advanceGame` will retry the VRF request (AdvanceModule lines 772-774), generating a new request with a new word. The operator would need to compromise the next word too.

**Attack vector 2: Deliver VRF at a specific time to create a BAF bypass window.**
The operator delivers the word exactly when the attacker's front-run claim transaction is in the mempool.

**Analysis:** The VRF word delivery (`rawFulfillRandomWords`) does NOT unlock RNG. It only stores the word (`rngWordCurrent = word` at line 1334). RNG remains locked until `advanceGame` calls `_unlockRng()`. There is no window between word delivery and unlock where claims succeed but BAF credit recording is still meaningful -- the BAF guard checks `rngLocked_` which remains true until `_unlockRng()`.

**Compromised VRF Verdict:** No advantage. The separation of word delivery (`rawFulfillRandomWords`) from word processing and unlock (`advanceGame` -> `_unlockRng()`) ensures no exploitable window exists. The `rngLockedFlag` lifecycle is: set at request, maintained through delivery, cleared only after all processing completes.

### RNG-02 Verdict

**SAFE** -- The BAF epoch-based guard is sufficient as the sole claim protection during BAF resolution windows. The guard covers the exact window where claims could affect BAF outcomes (all 7 conditions precisely define the BAF resolution window). sDGNRS is truly BAF-ineligible at both the caller layer (BurnieCoinflip line 542) and callee layer (DegenerusJackpots line 175), with no guard bypass needed for the processCoinflipPayouts path. No bypass is possible via reentrancy (guard conditions depend on game-level state), timing attacks (`_unlockRng` executes after all payout processing within a single transaction), or attacker models (MEV-aware and compromised VRF scenarios evaluated). The deposit lock layer (`_coinflipLockedDuringTransition` at line 985-998) provides complementary protection against deposit-triggered auto-claim during BAF resolution.
