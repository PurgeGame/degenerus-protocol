## 1. Overview

BURNIE is an ERC20 token (18 decimals, symbol `BURNIE`, name "Burnies") with no hard supply cap. Supply is governed by three mechanisms:

- **Earning paths (inflows):** Functions that bring BURNIE into circulation via minting or credit it as virtual coinflip stakes
- **Burn sinks (outflows):** Functions that permanently destroy circulating BURNIE
- **Vault reserve invariant:** A constraint linking circulating supply to a virtual reserve

**Supply flow diagram:**

```
                        mintForCoinflip / mintForGame
                              |
                              v
  +-------------------+   vaultMintTo   +--------------------+    burnForCoinflip
  |   Vault Reserve   | -------------> |    Circulation      | ----------------->  BURNED
  |  (vaultAllowance) | <------------- |    (totalSupply)    | ----------------->  BURNED
  +-------------------+  transfer(VAULT)+--------------------+    decimatorBurn
         ^                                     |                   burnCoin
         |                                     |
    vaultEscrow                          creditFlip (virtual only,
    (increases reserve)                   NO supply change)
```

**Key distinction:** `mintForCoinflip` and `mintForGame` create supply *outside* the vault constraint. The vault's 2M initial allowance is a separate reserve that can be drawn down (via `vaultMintTo`) or grown (via `vaultEscrow`).

```solidity
        uint128 allowanceVault = _supply.vaultAllowance;

        unchecked { _supply.vaultAllowance = allowanceVault - amount128; }  // line 495

    balanceOf[from] -= amount;          // line 445

        unchecked {
            _supply.totalSupply -= amount128;    // line 451
            _supply.vaultAllowance += amount128; // line 452
        }
        return;
    }
    balanceOf[to] += amount;            // line 460
}

// vaultEscrow — BurnieCoin.sol:677-688
function vaultEscrow(uint256 amount) external {
    // Access: GAME or VAULT only
    uint128 amount128 = _toUint128(amount);
    unchecked { _supply.vaultAllowance += amount128; }  // line 685
}

// vaultMintTo — BurnieCoin.sol:694-706
function vaultMintTo(address to, uint256 amount) external onlyVault {
    uint128 amount128 = _toUint128(amount);
    uint128 allowanceVault = _supply.vaultAllowance;
    if (amount128 > allowanceVault) revert Insufficient();
    unchecked {
        _supply.vaultAllowance = allowanceVault - amount128;  // line 700
        _supply.totalSupply += amount128;                      // line 701
        balanceOf[to] += amount;                               // line 702
    }
}
```

### 2d. Critical Notes

1. **`supplyIncUncirculated` is NOT constant.** It changes with every `_mint`, `_burn`, `mintForCoinflip`, `burnForCoinflip`, and `vaultEscrow` call. Only `transfer(user, VAULT)` and `vaultMintTo` preserve it (they move between the two buckets).

2. **No hard supply cap.** `mintForCoinflip` and `mintForGame` create new supply by calling `_mint(user, amount)`, which increases `totalSupply` without decreasing `vaultAllowance`. There is no ceiling enforced.

### 3h. Earning Paths Summary

| Path | Delivery Method | Creates New Supply? | Supply Impact |
|------|----------------|-------------------|---------------|
| Coinflip win | `mintForCoinflip` (wallet) | Yes | totalSupply + X, supplyIncUncirculated + X |
| Lootbox BURNIE | `creditFlip` (coinflip stake) | No -- virtual only | None until claimed |
| Quest reward | `creditFlip` (coinflip stake) | No -- virtual only | None until claimed |
| Bounty payout | `_addDailyFlip` (coinflip stake) | No -- virtual only | None until claimed |
| LINK donation | `creditFlip` (coinflip stake) | No -- virtual only | None until claimed |
| Vault mint | `vaultMintTo` (wallet) | Yes (from allowance) | totalSupply + X, vaultAllowance - X |
| Game mint | `mintForGame` (wallet) | Yes | totalSupply + X, supplyIncUncirculated + X |

**CRITICAL:** `creditFlip` adds to `coinflipBalance` (a virtual ledger in BurnieCoinflip) without minting. BURNIE only enters circulation when a player *claims* winning flips via `mintForCoinflip`. Agents must NOT count `creditFlip` amounts as circulating supply.

## 4. Burn Sinks (BURN-03)

### 4a. Coinflip Losses (permanent burn)

**Entry point:** `BurnieCoin.burnForCoinflip` (line 517), called by `BurnieCoinflip.depositCoinflip`
**Permanent:** Yes -- burned on deposit; never returned on loss
**Condition:** Player deposits BURNIE into coinflip
**Minimum:** 100 BURNIE (`MIN = 100 ether` in BurnieCoinflip.sol:119)

```solidity
// BurnieCoin.sol:517-519
function burnForCoinflip(address from, uint256 amount) external {
    if (msg.sender != coinflipContract) revert OnlyGame();
    _burn(from, amount);  // totalSupply -= amount, supplyIncUncirculated -= amount
}
```

On deposit, 100% of the staked BURNIE is burned immediately. On a win, the payout (stake + bonus) is *minted* fresh via `mintForCoinflip`. On a loss, nothing is returned -- the burned amount is permanently destroyed.

**Net effect:** 50% of deposited BURNIE is permanently destroyed (in expectation), since outcomes are 50/50.

```solidity
function _adjustDecimatorBucket(uint256 bonusBps, uint8 minBucket) private pure returns (uint8) {
    adjustedBucket = DECIMATOR_BUCKET_BASE;  // 12 (worst)
    if (bonusBps == 0) return adjustedBucket;
    if (bonusBps > DECIMATOR_ACTIVITY_CAP_BPS) bonusBps = DECIMATOR_ACTIVITY_CAP_BPS;
```

## 5. Vault Claim Mechanics (BURN-04 detail)

### 5a. DGVB Share Redemption

DGVB (Degenerus Vault Burnie) is a share token representing proportional claim on the vault's BURNIE reserve. Players burn DGVB shares to redeem BURNIE.

**Entry point:** `DegenerusVault.burnCoin` (line 760)

```solidity

    DegenerusVaultShare share = coinShare;

    uint256 coinBal = _syncCoinReserves();                              // line 777: vaultMintAllowance
    uint256 supplyBefore = share.totalSupply();                         // line 778
    uint256 vaultBal = coinToken.balanceOf(address(this));              // line 779
    uint256 claimable = coinflipPlayer.previewClaimCoinflips(address(this)); // line 780
    if (vaultBal != 0 || claimable != 0) {
        coinBal += vaultBal + claimable;                                // line 782

    coinOut = (coinBal * amount) / supplyBefore;                        // line 784

    share.vaultBurn(player, amount);                                    // line 786

    // Source priority for fulfilling coinOut:
    // 1. Vault's BURNIE wallet balance (transfer)
    // 2. Vault's claimable coinflips (claim then transfer)
    // 3. vaultMintTo remainder (mint from allowance)

                remaining -= claimed;
                coinToken.transfer(player, claimed);                     // line 804
            }
        }
        if (remaining != 0) {
            coinTracked -= remaining;
            coinToken.vaultMintTo(player, remaining);                    // line 810
        }
    }
```

### 5b. DGVB Reserve Calculation

The total DGVB reserve includes three components:

```
DGVB reserve = vaultMintAllowance + vault BURNIE balance + vault claimable coinflips
```

```solidity
// DegenerusVault.sol:996-1008
function _coinReservesView() private view returns (uint256 mainReserve) {
    uint256 allowance = coinToken.vaultMintAllowance();   // line 999
    mainReserve = allowance;
    uint256 vaultBal = coinToken.balanceOf(address(this));
    uint256 claimable = coinflipPlayer.previewClaimCoinflips(address(this));
    if (vaultBal != 0 || claimable != 0) {
        unchecked { mainReserve += vaultBal + claimable; }
    }
}
```

### 5c. Redemption Formula

```
coinOut = (DGVB_reserve * sharesBurned) / totalDGVBSupply
```

This is the primary mechanism for BURNIE to flow from the vault reserve into circulation. The supply impact depends on which source fulfills the redemption:

| Source | totalSupply | vaultAllowance | supplyIncUncirculated |
|--------|------------|----------------|----------------------|
| Vault balance (transfer) | unchanged | unchanged | unchanged |
| Claimable coinflips (claim+transfer) | +coinOut (minted to vault, then transferred) | unchanged | +coinOut |
| `vaultMintTo` remainder | +remaining | -remaining | unchanged |

## 6. Supply Dynamics Summary for Agents

### 6a. Key Observations

1. **Coinflip is net-negative on supply** -- ~1.5% expected loss per cycle. 50% of deposits are burned permanently (loss outcome). Wins return ~1.97x average, so expected return per flip = 0.5 * 0 + 0.5 * 1.97 = 0.985. Net loss: 1.5% of staked amount per cycle. Coinflip acts as a gradual BURNIE sink.

2. **Vault allowance grows via game deposits** -- every time the game contract calls `vaultEscrow(amount)`, the vault's BURNIE allowance increases, expanding potential future supply available through DGVB redemptions.

3. **Claim window expiry is an additional supply sink** -- unclaimed coinflip winnings beyond the 30-day (first claim) or 90-day (subsequent) window are permanently lost. These never get minted.

4. **Practical supply constraint** -- while there is no hard cap, the negative-EV coinflip combined with permanent decimator burns creates deflationary pressure that counterbalances minting from wins and rewards.

### 6b. Simplified Supply Flow Equation

For agent modeling over a time period T:

```
deltaCirculating =
    + coinflipWinsMinted          (mintForCoinflip: new supply)
    + gameMints                   (mintForGame: new supply)
    + vaultMints                  (vaultMintTo: from allowance)
    - coinflipDeposits            (burnForCoinflip: permanent burn)
    - decimatorBurns              (decimatorBurn: permanent burn)
    - burnieTicketBurns           (burnCoin: permanent burn)
    - transfersToVault            (transfer to VAULT: vault return)

deltaVaultAllowance =
    + vaultEscrow                 (game deposits)
    + transfersToVault            (user transfers to VAULT)
    - vaultMints                  (vaultMintTo: drawn from allowance)

deltaSupplyIncUncirculated =
    + coinflipWinsMinted
    + gameMints
    + vaultEscrow
    - coinflipDeposits
    - decimatorBurns
    - burnieTicketBurns
```

Note: `creditFlip` amounts do NOT appear in these equations. They only affect the virtual coinflip ledger and have zero supply impact until claimed (at which point they become `coinflipWinsMinted`).

## 7. Worked Example

Trace a complete BURNIE lifecycle tracking all three supply variables.

**Starting state:** totalSupply = 100,000 BURNIE, vaultAllowance = 2,100,000 BURNIE, supplyIncUncirculated = 2,200,000 BURNIE.

Operation: `burnForCoinflip(player, 1000)` calls `_burn(player, 1000)`

| vaultAllowance | 2,100,000 | 0 | 2,100,000 |

### Step 2: Player wins coinflip with 96% bonus -- minted 1,960 BURNIE

Operation: `mintForCoinflip(player, 1960)` calls `_mint(player, 1960)`
Payout: 1000 + (1000 * 96 / 100) = 1,960

| Variable | Before | Change | After |
|----------|--------|--------|-------|

| vaultAllowance | 2,100,000 | 0 | 2,100,000 |

Net from coinflip cycle: totalSupply +960, supplyIncUncirculated +960.

### Step 3: Player burns 1,000 BURNIE for decimator

Operation: `decimatorBurn(player, 1000)` calls `_burn(player, 1000)`

| Variable | Before | Change | After |
|----------|--------|--------|-------|
| totalSupply | 100,960 | -1,000 | 99,960 |
| vaultAllowance | 2,100,000 | 0 | 2,100,000 |
| supplyIncUncirculated | 2,200,960 | -1,000 | 2,199,960 |

### Step 4: Lootbox credits 500 BURNIE via creditFlip

Operation: `coin.creditFlip(player, 500)` -- adds to coinflipBalance virtual ledger

| Variable | Before | Change | After |
|----------|--------|--------|-------|

**No supply change.** The 500 BURNIE exists only in the coinflip balance ledger. It will affect supply only when the player's next coinflip resolves and they claim winnings (via `mintForCoinflip`).

### Step 5: Game calls vaultEscrow(5,000) from purchase proceeds

Operation: `vaultEscrow(5000)` increases vaultAllowance

| Variable | Before | Change | After |
