## 1. Overview

The jackpot phase is a 5-day (logical) draw sequence that distributes `currentPrizePool` to players holding trait-matched burn tickets. It fires after the purchase phase target is met and pools are consolidated (see `06-pool-architecture.md` for how `currentPrizePool` gets funded at consolidation via `consolidatePrizePools()`).

**Entry point:** `payDailyJackpot(isDaily=true, lvl, randWord)` -- JackpotModule line 336

Each physical day, `advanceGame()` triggers one daily jackpot call. The jackpot executes in two gas-separated phases:
- **Phase 0:** ETH distribution from `currentPrizePool` (and carryover from `futurePrizePool`)
- **Phase 1 (coin+tickets):** BURNIE jackpot and ticket distribution via `payDailyJackpotCoinAndTickets()` (JackpotModule line 702)

### High-Level Daily Flow

```
Day 1 (counter=0):
  [1] Roll winning traits (burn-weighted)
  [2] Compute dailyBps = random 6-14% of currentPrizePool
  [3] Budget split: 20% lootbox tickets, 80% ETH distribution
  [4] Run early-bird lootbox jackpot (3% of futurePrizePool -> tickets)
  [5] Distribute 80% ETH to 4 trait buckets (equal 20% shares)
  [6] BURNIE jackpot: 25% far-future + 75% near-future
  [7] Distribute daily + early-bird lootbox tickets
  [8] Increment jackpotCounter

Days 2-4 (counter=1,2,3):
  [1] Roll winning traits (burn-weighted)
  [2] Compute dailyBps = random 6-14% of currentPrizePool
  [3] Budget split: 20% lootbox tickets, 80% ETH distribution
  [4] Carryover: 1% of futurePrizePool -> 50% lootbox, 50% ETH to future-level winners
  [5] Distribute 80% ETH to 4 trait buckets (equal 20% shares)
  [6] Distribute carryover ETH to future-level trait buckets
  [7] BURNIE jackpot: 25% far-future + 75% near-future
  [8] Distribute daily + carryover lootbox tickets
  [9] Increment jackpotCounter

Day 5 (counter=4, i.e. counter >= JACKPOT_LEVEL_CAP-1):
  [1] Roll winning traits (burn-weighted)
  [2] dailyBps = 100% (10000 BPS) -- all remaining currentPrizePool
  [3] Budget split: 20% lootbox tickets, 80% ETH distribution
  [4] No carryover on day 5
  [5] Distribute 80% ETH to 4 trait buckets (weighted 60/13/13/13 shares)
  [6] BURNIE jackpot: 25% far-future + 75% near-future
  [7] Distribute daily lootbox tickets
  [8] Award 1% of DGNRS reward pool to solo bucket winner
  [9] Run transition jackpots (BAF/Decimator -- see 07-03)
  [10] End phase, reset counters, begin new level
```

## 2. Daily Pool Slice Formula (JACK-03)

**Function:** `_dailyCurrentPoolBps(counter, randWord)` -- JackpotModule line 2723

### Formula

```solidity
function _dailyCurrentPoolBps(uint8 counter, uint256 randWord) private pure returns (uint16 bps) {
    if (counter >= JACKPOT_LEVEL_CAP - 1) return 10_000;  // Day 5: 100%

    uint16 range = DAILY_CURRENT_BPS_MAX - DAILY_CURRENT_BPS_MIN + 1;  // 1400 - 600 + 1 = 801
    uint256 seed = uint256(
        keccak256(abi.encodePacked(randWord, DAILY_CURRENT_BPS_TAG, counter))
    );
    return uint16(DAILY_CURRENT_BPS_MIN + (seed % range));  // 600-1400 (6%-14%)
}
```
**Source:** JackpotModule lines 2723-2736

**Day-by-day BPS:**

| Logical Day | Counter | BPS Range | Meaning |
|-------------|---------|-----------|---------|
| 1 | 0 | 600-1400 | Random 6%-14% of currentPrizePool |
| 2 | 1 | 600-1400 | Random 6%-14% of currentPrizePool |
| 3 | 2 | 600-1400 | Random 6%-14% of currentPrizePool |
| 4 | 3 | 600-1400 | Random 6%-14% of currentPrizePool |
| 5 | 4 (>= JACKPOT_LEVEL_CAP-1) | 10000 | 100% of remaining currentPrizePool |

**Expected value:** Days 1-4 average 10% each (midpoint of 6-14%). After 4 draws at ~10%, roughly 34% has been distributed, leaving ~66% for day 5.

### Budget Split per Day

```solidity
uint256 budget = (currentPrizePool * dailyBps) / 10_000;                // JackpotModule line 385
uint256 dailyLootboxBudget = _validateTicketBudget(budget / 5, ...);    // JackpotModule line 394
if (dailyLootboxBudget != 0) {
    budget -= dailyLootboxBudget;                                        // JackpotModule line 400
}
```
**Source:** JackpotModule lines 385-401

- **20%** (`budget / 5`) goes to lootbox tickets (moved from `currentPrizePool` to `nextPrizePool`)
- **80%** remaining goes to ETH distribution across 4 trait buckets

The lootbox budget is validated via `_validateTicketBudget()` which returns 0 if no valid ticket level exists, in which case the full budget goes to ETH distribution.

## 3. Trait Bucket Distribution (JACK-04)

### Trait ID Encoding

```
traitId = (quadrant << 6) | (color << 3) | symbol
```
- 4 quadrants (Q0-Q3): 0, 64, 128, 192
- 8 colors (0-7) per quadrant
- 8 symbols (0-7) per quadrant
- Total: 256 possible trait IDs

**Source:** JackpotBucketLib lines 263-283, JackpotModule lines 1866-1900

### Winning Trait Derivation

**Function:** `_getWinningTraits(randomWord)` -- JackpotModule line 1866

| Quadrant | Base Derivation | Symbol | Color |
|----------|----------------|--------|-------|
| Q0 | `(color << 3) \| 0` | Fixed at 0 | `randomWord & 7` |
| Q1 | `64 + (color << 3) \| 0` | Fixed at 0 | `(randomWord >> 3) & 7` |
| Q2 | `128 + (color << 3) \| 0` | Fixed at 0 | `(randomWord >> 6) & 7` |
| Q3 | `192 + (randomWord >> 9) & 63` | Random (0-7) | Random (0-7) |

**Hero override:** If a top daily hero symbol exists (highest `dailyHeroWagers` total), that symbol auto-wins its own quadrant with a random color, replacing only that quadrant's trait derivation.

**Source:** JackpotModule lines 1871-1900 (trait derivation), lines 1904-1932 (hero symbol lookup)

### Share Distribution

**Days 1-4 shares:**
```solidity
uint64 private constant DAILY_JACKPOT_SHARES_PACKED =
    uint64(2000) * 0x0001000100010001;  // [2000, 2000, 2000, 2000]
```
**Source:** JackpotModule lines 124-125

Each bucket nominally receives 20% (2000 BPS). The solo bucket (entropy-selected) receives any remainder after rounding, so in practice it gets slightly more than 20%.

**Day 5 shares:**
```solidity
uint64 private constant FINAL_DAY_SHARES_PACKED =
    (uint64(6000)) |
        (uint64(1333) << 16) |
        (uint64(1333) << 32) |
        (uint64(1334) << 48);  // [6000, 1333, 1333, 1334]
```
**Source:** JackpotModule lines 116-120

The 60% share is assigned to the solo (1-winner) bucket. The other 3 buckets split the remaining 40% at roughly 13.33% each.

**Share selection logic:**
```solidity
uint64 sharesPacked = (jackpotCounter == JACKPOT_LEVEL_CAP - 1)
    ? FINAL_DAY_SHARES_PACKED
    : DAILY_JACKPOT_SHARES_PACKED;
uint16[4] memory shareBps = JackpotBucketLib.shareBpsByBucket(
    sharesPacked, uint8(entropy & 3)
);
```
**Source:** JackpotModule lines 510-517

The `shareBpsByBucket` function rotates shares based on `entropy & 3` so the 60% share is not always assigned to the same bucket index.

**Solo bucket index:** `soloBucketIndex(entropy) = (3 - (entropy & 3)) & 3`
**Source:** JackpotBucketLib line 240-242

The solo bucket always receives the remainder (pool minus all other bucket shares), guaranteeing no dust is lost.

### Base Winner Counts per Bucket

```solidity
function traitBucketCounts(uint256 entropy) internal pure returns (uint16[4] memory counts) {
    uint16[4] memory base;
    base[0] = 25;  // Large bucket
    base[1] = 15;  // Mid bucket
    base[2] = 8;   // Small bucket
    base[3] = 1;   // Solo bucket
    // Rotated by entropy & 3 for fairness
}
```
**Source:** JackpotBucketLib lines 36-51

Base total: 49 winners (25 + 15 + 8 + 1). Rotation ensures the solo (1 winner) position cycles across buckets.

### Bucket Winner Count Scaling

Winner counts scale with pool size to distribute gas and reward breadth:

```
Pool < 10 ETH:    1x (base counts)
Pool 10-50 ETH:   Linear interpolation 1x -> 2x
Pool 50-200 ETH:  Linear interpolation 2x -> maxScale
Pool >= 200 ETH:  maxScale (capped)
```

**Scale constants (applied to non-solo buckets only):**

| Context | Constant | Value | Max Scale |
|---------|----------|-------|-----------|
| Non-daily jackpot | JACKPOT_SCALE_MAX_BPS | 40,000 | 4x at 200+ ETH |
| Daily jackpot | DAILY_JACKPOT_SCALE_MAX_BPS | 66,667 | 6.67x at 200+ ETH |

**Source:** JackpotModule lines 228-231, JackpotBucketLib lines 54-95

**Max winners hard cap:** Total daily winners (daily + carryover) capped at DAILY_ETH_MAX_WINNERS = 321.
**Source:** JackpotModule line 199

The solo bucket always stays at 1 winner. Only buckets with `count > 1` are scaled.

### Winner Selection: `_randTraitTicket()`

**Function:** `_randTraitTicket(traitBurnTicket_, randomWord, trait, numWinners, salt)` -- JackpotModule line 2297

```solidity
address[] storage holders = traitBurnTicket_[trait];
uint256 len = holders.length;

// Virtual deity entries: floor(2% of bucket tickets), minimum 2
uint8 fullSymId = (trait >> 6) * 8 + (trait & 0x07);
if (fullSymId < 32) {
    deity = deityBySymbol[fullSymId];
    if (deity != address(0)) {
        virtualCount = len / 50;       // 2% of real tickets
        if (virtualCount < 2) virtualCount = 2;
    }
}

uint256 effectiveLen = len + virtualCount;
// Winner: idx = slice % effectiveLen
// If idx < len: real ticket holder. If idx >= len: deity.
// Duplicates allowed -- more tickets = more chances.
```
**Source:** JackpotModule lines 2297-2340

Key properties:
- **Duplicates allowed:** A player with multiple tickets can win multiple times in the same draw
- **Virtual deity entries:** If a deity exists for the trait's symbol, `floor(2% of bucket, min 2)` virtual entries are added, giving the deity a chance to win
- **Entropy rotation:** Each subsequent winner uses `slice = (slice >> 16) | (slice << 240)` for different indices

### Solo Bucket Special Handling

**Function:** `_processSoloBucketWinner(winner, perWinner, entropy)` -- JackpotModule line 1821

Solo bucket (1 winner) gets special 75/25 ETH/whale-pass treatment:

```solidity
uint256 quarterAmount = perWinner >> 2;                      // perWinner / 4
uint256 whalePassCount = quarterAmount / HALF_WHALE_PASS_PRICE;  // 2.25 ETH per pass

if (whalePassCount != 0) {
    // 75% ETH + 25% as whale passes
    uint256 whalePassCost = whalePassCount * HALF_WHALE_PASS_PRICE;
    uint256 ethAmount = perWinner - whalePassCost;
    // ethAmount credited to claimableWinnings
    // whalePassCost moved to futurePrizePool
} else {
    // 25% too small for a whale pass -- pay 100% as ETH
    _creditJackpot(false, winner, perWinner, entropy);
}
```
**Source:** JackpotModule lines 1821-1854

**Threshold:** The 25% portion must cover at least one HALF_WHALE_PASS_PRICE (2.25 ETH). This means the solo bucket payout must be at least 9.0 ETH (4 x 2.25) for the 75/25 split to activate. Below that, the winner receives 100% ETH.

### DGNRS Day-5 Reward

## 4. Day 1: Early-Bird Lootbox Jackpot

**Function:** `_runEarlyBirdLootboxJackpot(lvl + 1, randWord)` -- JackpotModule line 822

Day 1 (counter == 0) runs the early-bird lootbox jackpot **instead of** the normal carryover flow.

### Budget

```solidity
uint256 reserveContribution = (_getFuturePrizePool() * 300) / 10_000;  // 3% of futurePrizePool
uint256 totalBudget = reserveContribution;
_setFuturePrizePool(_getFuturePrizePool() - reserveContribution);       // Deducted upfront
```
**Source:** JackpotModule lines 824-828

### Winner Selection

```solidity
uint256 maxWinners = 100;                           // LOOTBOX_MAX_WINNERS equivalent
uint256 perWinnerEth = totalBudget / maxWinners;    // Even split across 100 slots
```
**Source:** JackpotModule lines 835-836

For each of the 100 winner slots:
1. Derive random trait ID: `traitId = uint8(entropy)` -- uniform, not burn-weighted
2. Select 1 winner from `traitBurnTicket[lvl][traitId]` via `_randTraitTicket()`
3. Roll random level offset 0-4 (20% chance each): `levelOffset = uint24(entropy % 5)`
4. Convert `perWinnerEth` into tickets at that level's price: `ticketCount = perWinnerEth / levelPrices[levelOffset]`
5. Queue tickets for the winner at `baseLevel + levelOffset`

**Source:** JackpotModule lines 850-880

### Pool Flow

The full `totalBudget` is added to `nextPrizePool`:
```solidity
_setNextPrizePool(_getNextPrizePool() + totalBudget);
```
**Source:** JackpotModule line 884

This means the 3% taken from `futurePrizePool` enters `nextPrizePool`, backing the tickets issued. It will be merged into `currentPrizePool` at the next level's consolidation.

## 5. Days 2-4: Carryover Mechanics (JACK-05)

**Location:** JackpotModule lines 416-476 (setup), lines 574-627 (distribution)

On days 2-4 (counter > 0, counter < JACKPOT_LEVEL_CAP-1), a carryover distribution sends ETH and lootbox tickets to winners at a future level's trait pool.

### Carryover Budget

```solidity
uint256 reserveSlice = _getFuturePrizePool() / 100;  // 1% of futurePrizePool
_setFuturePrizePool(_getFuturePrizePool() - reserveSlice);  // Deducted upfront
```
**Source:** JackpotModule lines 436-439

**Condition:** Carryover only runs if `initCarryoverSourceOffset != 0` (i.e., at least one future level has trait tickets). If no eligible levels exist, `reserveSlice` stays 0 and no carryover occurs.

### Carryover Lootbox Split

```solidity
uint256 futureEthPool = reserveSlice;
uint16 carryoverLootboxBps = DAILY_REWARD_JACKPOT_LOOTBOX_BPS;  // 5000 (50%)
uint256 carryoverLootboxBudget = _validateTicketBudget(
    (futureEthPool * carryoverLootboxBps) / 10_000,  // 50% of reserveSlice
    carryoverSourceLevel, winningTraitsPacked
);
if (carryoverLootboxBudget != 0) {
    futureEthPool -= carryoverLootboxBudget;
    _setNextPrizePool(_getNextPrizePool() + carryoverLootboxBudget);
}
```
**Source:** JackpotModule lines 442-457

- **50%** of the 1% slice goes to lootbox tickets (converted and added to `nextPrizePool`)
- **50%** remaining goes to ETH distribution at the carryover source level

### Carryover Source Level Selection

**Function:** `_selectCarryoverSourceOffset(lvl, winningTraitsPacked, randWord, counter)` -- JackpotModule line 2775

1. Find highest eligible offset in [1..DAILY_CARRYOVER_MAX_OFFSET] (up to +5) with actual winning-trait tickets
2. Random start point within [1..highestEligible], probe all offsets circularly
3. Return first offset with `_hasActualTraitTickets() == true`
4. Return 0 if no eligible offsets

```solidity
uint8 startOffset = uint8(
    (keccak256(abi.encodePacked(randWord, DAILY_CARRYOVER_SOURCE_TAG, counter))
    % highestEligible) + 1
);
```
**Source:** JackpotModule lines 2775-2818

The carryover ETH distribution uses the same bucket system and winner selection but targets the carryover source level instead of the current level. **On the final physical day** (when `isFinalPhysicalDay_` is true), carryover uses `FINAL_DAY_SHARES_PACKED` (60/13/13/13) instead of the usual `DAILY_JACKPOT_SHARES_PACKED` (20/20/20/20), matching the daily distribution's weighted shares for that day (JackpotModule lines 596-598).

### Carryover Winner Cap

```solidity
if (totalDailyWinners >= DAILY_ETH_MAX_WINNERS) {
    dailyCarryoverWinnerCap = 0;  // No room for carryover winners
} else {
    uint16 remaining = uint16(DAILY_ETH_MAX_WINNERS - totalDailyWinners);
    dailyCarryoverWinnerCap = remaining < DAILY_CARRYOVER_MIN_WINNERS
        ? DAILY_CARRYOVER_MIN_WINNERS
        : remaining;
}
```
**Source:** JackpotModule lines 539-549

The carryover winner cap is the remaining budget after daily distribution, with a floor of DAILY_CARRYOVER_MIN_WINNERS (20). If daily distribution used all 321 winner slots, carryover is skipped.

### Day 5: No Carryover

Day 5 (`counter >= JACKPOT_LEVEL_CAP - 1`) distributes 100% of remaining `currentPrizePool`. No carryover from `futurePrizePool` occurs on this day.

## 6. Compressed & Turbo Jackpot (JACK-05)

### Trigger

Two compression levels exist, controlled by `compressedJackpotFlag`:

**Turbo mode (flag=2):** Set when the target is met within 1 purchase day (AdvanceModule line 134):
```solidity
// AdvanceModule:131-134
if (purchaseDays <= 1 && _getNextPrizePool() >= levelPrizePool[lvl]) {
    lastPurchaseDay = true;
    compressedJackpotFlag = 2;
}
```

**Compressed mode (flag=1):** Set when the purchase phase lasted 2 days or fewer (AdvanceModule line 256):
```solidity
// AdvanceModule:255-257
if (day - purchaseStartDay <= 2) {
    compressedJackpotFlag = 1;
}
```

This flag is set in `advanceGame()` when `lastPurchaseDay` becomes true -- i.e., when `nextPrizePool >= levelPrizePool[lvl - 1]`.

The flag is cleared at phase transition: `compressedJackpotFlag = 0` (AdvanceModule line 445).

### Effect on Daily Jackpot

```solidity
uint8 counterStep = 1;
// Turbo (flag=2): all 5 logical days in 1 physical day.
// Compressed (flag=1): 5 logical days in 3 physical days.
if (compressedJackpotFlag == 2 && counter == 0) {
    counterStep = JACKPOT_LEVEL_CAP;    // = 5, all days at once
} else if (
    compressedJackpotFlag == 1 &&
    counter > 0 &&                       // Day 1 (counter=0) is NOT doubled
    counter < JACKPOT_LEVEL_CAP - 1
) {
    counterStep = 2;                     // Advance counter by 2 per physical day
}
if (counterStep == 2) {
    dailyBps *= 2;                       // Double BPS to combine two logical days
}
```
**Source:** JackpotModule lines 359-383

### Physical vs Logical Day Mapping (Compressed Mode, flag=1)

| Physical Day | Counter Before | Counter After | Logical Days Covered | BPS |
|-------------|---------------|---------------|---------------------|-----|
| 1 | 0 | 1 | Day 1 only | 1x random (6-14%) |
| 2 | 1 | 3 | Day 2 + Day 3 | 2x random (12-28%) |
| 3 | 3 | 5 (>= cap) | Day 4 + Day 5 | 100% (final physical day) |

**Result:** 5 logical days complete in 3 physical days. Day 1 is NOT doubled because the `counter > 0` guard prevents it -- the early-bird lootbox jackpot runs on day 1 and should not be combined with a carryover day.

### Physical vs Logical Day Mapping (Turbo Mode, flag=2)

| Physical Day | Counter Before | Counter After | Logical Days Covered | BPS |
|-------------|---------------|---------------|---------------------|-----|
| 1 | 0 | 5 (>= cap) | Days 1-5 | 100% (final physical day) |

**Result:** All 5 logical days execute in 1 physical day. The `counterStep = JACKPOT_LEVEL_CAP` (= 5) causes `isFinalPhysicalDay` to be true immediately, so `dailyBps = 10_000` (100%). This triggers when the level target was met so quickly that full compression is warranted.

**Important:** Day 5 is never doubled because `counterStep` stays 1 when `counter >= JACKPOT_LEVEL_CAP - 1`. The `counterStep` is also stored in `dailyTicketBudgetsPacked` and used by `payDailyJackpotCoinAndTickets()` to correctly increment `jackpotCounter`.

## 7. Lootbox Conversion Reference (JACK-06)

All lootbox conversions during the jackpot phase, consolidated for agent reference.

| Source | Budget Formula | Lootbox % | ETH % | Destination Pool |
|--------|---------------|-----------|-------|-----------------|
| Daily budget (days 1-5) | `budget / 5` (20% of daily slice) | 100% of this 20% | 0% | currentPrizePool -> nextPrizePool |
| Carryover (days 2-4) | `reserveSlice * 5000 / 10000` (50% of 1% future) | 100% of this 50% | 0% | futurePrizePool -> nextPrizePool |
| Early-bird (day 1) | `futurePrizePool * 300 / 10000` (3% of future) | 100% (all tickets) | 0% | futurePrizePool -> nextPrizePool |

**Cross-reference with purchase phase (from 07-01):**

| Context | Constant | Value |
|---------|----------|-------|
| Purchase-phase daily drip | PURCHASE_REWARD_JACKPOT_LOOTBOX_BPS | 7500 (75% to lootbox) |
| Jackpot-phase carryover | DAILY_REWARD_JACKPOT_LOOTBOX_BPS | 5000 (50% to lootbox) |
| Jackpot-phase daily budget | `budget / 5` | 20% of daily budget to lootbox |

**Ticket conversion mechanics:**

Daily lootbox budget is converted to ticket units via `_budgetToTicketUnits(budget, targetLevel)`. Tickets are distributed in Phase 2 (`payDailyJackpotCoinAndTickets`) to trait-matched winners at the relevant level, with up to LOOTBOX_MAX_WINNERS (100) winners per distribution.

## 8. BURNIE Parallel Distribution (JACK-07)

**Function:** `payDailyJackpotCoinAndTickets(randWord)` -- JackpotModule line 702

BURNIE (coinflip credits) are distributed every day during the jackpot phase alongside the ETH jackpot. The BURNIE distribution runs in Phase 2, after Phase 0/1 (ETH) completes.

### Budget

```solidity
function _calcDailyCoinBudget(uint24 lvl) private view returns (uint256) {
    uint256 priceWei = price;
    if (priceWei == 0) return 0;
    return (levelPrizePool[lvl - 1] * PRICE_COIN_UNIT) / (priceWei * 200);
    // = 0.5% of prize pool target, denominated in BURNIE
    // PRICE_COIN_UNIT = 1000 ether (1000 BURNIE per unit)
}
```
**Source:** JackpotModule lines 2714-2718

This is the same formula used during the purchase phase (JACK-02). The budget is denominated in BURNIE tokens, not ETH.

### Split

```solidity
uint256 farBudget = (coinBudget * FAR_FUTURE_COIN_BPS) / 10_000;  // 25%
uint256 nearBudget = coinBudget - farBudget;                        // 75%
```
**Source:** JackpotModule lines 727-730

| Portion | Share | Target Levels | Max Winners | Selection Method |
|---------|-------|--------------|-------------|-----------------|
| Far-future | 25% (FAR_FUTURE_COIN_BPS=2500) | lvl+5 to lvl+99 | FAR_FUTURE_COIN_SAMPLES=10 | ticketQueue (traits not yet assigned) |
| Near-future | 75% | lvl to lvl+4 | DAILY_COIN_MAX_WINNERS=50 | traitBurnTicket (trait-matched) |

### Far-Future Distribution

**Function:** `_awardFarFutureCoinJackpot(lvl, farBudget, rngWord)` -- JackpotModule line 2588

```solidity
for (uint8 s; s < FAR_FUTURE_COIN_SAMPLES; ) {          // Sample 10 levels
    entropy = EntropyLib.entropyStep(entropy ^ uint256(s));
    uint24 candidate = lvl + 5 + uint24(entropy % 95);    // Random level in [lvl+5, lvl+99]

    address[] storage queue = ticketQueue[_tqWriteKey(candidate)];
    uint256 len = queue.length;
    if (len != 0) {
        uint256 idx = (entropy >> 32) % len;
        address winner = queue[idx];                       // 1 winner per level from ticketQueue
        // Track winner
    }
}

uint256 perWinner = farBudget / found;                     // Even split among found winners
```
**Source:** JackpotModule lines 2588-2663

Properties:
- Samples up to 10 random levels in [lvl+5, lvl+99]
- 1 winner per level from `ticketQueue` (not trait-matched, since traits aren't assigned yet at these levels)
- Even split of the 25% budget among all found winners
- Paid via `coin.creditFlipBatch()` (batches of 3 for gas efficiency)

### Near-Future Distribution

**Function:** `_awardDailyCoinToTraitWinners(lvl, winningTraitsPacked, nearBudget, entropy)` -- JackpotModule line 2485

1. Select random target level from [lvl, lvl+4] that has winning-trait tickets: `_selectDailyCoinTargetLevel()` (line 2465)
2. Compute bucket counts up to DAILY_COIN_MAX_WINNERS (50)
3. Even split of budget across all winners: `baseAmount = coinBudget / cap`
4. Select winners from trait buckets via `_randTraitTicketWithIndices()`
5. Pay via `coin.creditFlipBatch()`

**Source:** JackpotModule lines 2485-2583

## Appendix: Agent Simulation Pseudocode

For a game theory agent to simulate a single jackpot phase:

```
INPUT: currentPrizePool, futurePrizePool, vrf_words[5], level
OUTPUT: per-day ETH distributed, carryover amounts, BURNIE budgets

for day in 1..5:
    counter = day - 1  // 0-indexed
    randWord = vrf_words[day-1]

    // Step 1: Compute daily BPS
    if counter >= 4:
        dailyBps = 10000
    else:
        seed = keccak256(randWord, "daily-current-bps", counter)
        dailyBps = 600 + (seed % 801)

    // Step 2: Apply compressed/turbo mode (if active)
    if turbo (flag=2) and counter == 0:
        counterStep = 5  // all days at once
    elif compressed (flag=1) and counter > 0 and counter < 4:
        counterStep = 2
        dailyBps *= 2

    // Step 3: Compute budget
    budget = currentPrizePool * dailyBps / 10000
    lootboxBudget = budget / 5           // 20% to lootbox
    ethBudget = budget - lootboxBudget   // 80% to ETH

    // Step 4: Deduct from pool
    currentPrizePool -= budget

    // Step 5: Day-specific extra
    if day == 1:
        earlyBird = futurePrizePool * 300 / 10000  // 3%
        futurePrizePool -= earlyBird
    elif day in [2,3,4]:
        carryover = futurePrizePool / 100           // 1%
        futurePrizePool -= carryover
        carryoverLootbox = carryover * 5000 / 10000 // 50%
        carryoverEth = carryover - carryoverLootbox  // 50%

    // Step 6: BURNIE budget (same every day)
    burnieBudget = levelPrizePool[level-1] * 1000e18 / (price * 200)

    // Step 7: Distribute ethBudget across 4 trait buckets
    if day == 5:
        shares = [6000, 1333, 1333, 1334]  // rotated by entropy
    else:
        shares = [2000, 2000, 2000, 2000]  // rotated, remainder to solo
```
