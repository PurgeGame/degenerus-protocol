## 1. BAF Jackpot Overview

The BAF (Big Ass Flip) jackpot is a leaderboard-based distribution mechanism that fires every 10 levels. It is the only jackpot subsystem that spans two contracts: `DegenerusJackpots.sol` (standalone external contract for winner selection) and `DegenerusGameEndgameModule.sol` (delegatecall module for payout processing). All other jackpots execute entirely within delegatecall modules sharing game storage.

### 1.1 Trigger Path

```
advanceGame (AM)
  -> payDailyJackpotCoinAndTickets
     -> jackpotCounter >= JACKPOT_LEVEL_CAP (5)                  [AM:364]
        -> _awardFinalDayDgnrsReward(lvl, rngWord)               [AM:365]
        -> _rewardTopAffiliate(lvl)                               [AM:366]
        -> _runRewardJackpots(lvl, rngWord)                       [AM:367]
           -> delegatecall EndgameModule.runRewardJackpots        [AM:516-527]
              -> prevMod10 == 0 condition                         [EM:179]
              -> bafPoolWei = (baseFuturePool * bafPct) / 100     [EM:181]
              -> _runBafJackpot(bafPoolWei, lvl, rngWord)         [EM:189]
                 -> jackpots.runBafJackpot(poolWei, lvl, rngWord) [EM:363]
                    (external call to DegenerusJackpots contract)
```

The BAF jackpot triggers when `lvl % 10 == 0` at EM:179. The `_runRewardJackpots` wrapper at AM:516 performs a delegatecall to `EndgameModule.runRewardJackpots` (AM:521), which computes the pool allocation and calls `_runBafJackpot` (EM:189). That private function then makes an external call to the standalone `DegenerusJackpots` contract at EM:363.

**Pool allocation logic (EM:180):**

| Level Pattern | Pool % | Condition |
|--------------|--------|-----------|
| Level 50 | 20% | `lvl == 50` |
| Century (x00) | 20% | `prevMod100 == 0` |
| Other x0 (10,20,30,40,60,70,80,90) | 10% | default |

The percentage is computed as `bafPct = prevMod100 == 0 ? 20 : (lvl == 50 ? 20 : 10)` at EM:180, applied against `baseFuturePool` at EM:181.

**Pool deduction:** The full BAF pool is pulled from `futurePoolLocal` before resolution at EM:184. Any refund (unawarded prizes) is returned at EM:192-194, and lootbox ETH is recycled back at EM:196-198. The net effect on `futurePrizePool` is `-(bafPoolWei - refund - lootboxToFuture)`.

### 1.2 BAF State Tracking

BAF state is maintained in `DegenerusJackpots.sol` (standalone contract, NOT shared game storage):

| Variable | Type | Location | Purpose |
|----------|------|----------|---------|
| `bafTotals` | `mapping(uint24 => mapping(address => uint256))` | DJ:124 | Accumulated coinflip stake per player per level |
| `bafTop` | `mapping(uint24 => PlayerScore[4])` | DJ:127 | Top-4 leaderboard per level (sorted descending) |
| `bafTopLen` | `mapping(uint24 => uint8)` | DJ:130 | Current length of bafTop for each level (0-4) |
| `bafEpoch` | `mapping(uint24 => uint256)` | DJ:133 | Epoch counter per level, incremented on resolution |
| `bafPlayerEpoch` | `mapping(uint24 => mapping(address => uint256))` | DJ:136 | Player's last-known epoch (lazy reset mechanism) |
| `lastBafResolvedDay` | `uint48` | DJ:139 | Day index of most recent BAF resolution |

**PlayerScore struct (DJ:78-84):** Packed into a single 256-bit slot: `address player` (160 bits) + `uint96 score` (96 bits). The `score` is the total stake in whole tokens (not wei), capped at `type(uint96).max` via `_score96` at DJ:580-586.

### 1.3 recordBafFlip Leaderboard Recording

`recordBafFlip(address player, uint24 lvl, uint256 amount)` at DJ:174 is called by the coin/coinflip contract (guarded by `onlyCoin` modifier at DJ:149-152) on every manual coinflip.

**Execution flow:**

1. **Vault/sDGNRS filter (DJ:175):** Silently returns if `player == VAULT || player == SDGNRS`. This prevents protocol-owned addresses from contaminating the leaderboard.

2. **Lazy epoch reset (DJ:177-181):** If `bafPlayerEpoch[lvl][player] != bafEpoch[lvl]`, the player's epoch is updated and their `bafTotals` are zeroed. This implements a zero-cost lazy reset: instead of clearing all player data at jackpot resolution, each player's data is lazily invalidated by incrementing the epoch.

3. **Accumulation (DJ:183-185):** `total += amount` using `unchecked` arithmetic. The amount is the raw coinflip stake in wei. The player's total is the cumulative stake across all flips in this epoch.

4. **Leaderboard update (DJ:187):** Calls `_updateBafTop(lvl, player, total)` to maintain the sorted top-4 board.

5. **Event emission (DJ:188):** `BafFlipRecorded(player, lvl, amount, total)`.

### 1.4 _updateBafTop Insertion Sort

`_updateBafTop(uint24 lvl, address player, uint256 stake)` at DJ:594 maintains the top-4 leaderboard as a sorted array (highest score first).

**Score conversion:** `_score96(stake)` at DJ:595 converts raw wei to whole tokens via `s / 1 ether` and caps at `type(uint96).max` (DJ:580-586).

**Three cases:**

1. **Player already on board (DJ:612-626):** If score improved, update the entry and bubble up via swap. If not improved, return immediately (DJ:613).

2. **Board not full, len < 4 (DJ:629-640):** Insert in sorted position by shifting entries right. Increment `bafTopLen[lvl]`.

3. **Board full, len == 4 (DJ:643-651):** If `score > board[3].score`, replace bottom entry and shift to maintain sort order. Otherwise ignore.

## 2. Prize Distribution Slices

The `runBafJackpot` function at DJ:229 distributes the pool `P` across 7 prize slices. The function allocates to pre-sized buffers (`tmpW[107]`, `tmpA[107]` at DJ:241-242) and tracks the write index `n`.

### 2.1 Slice A -- Top BAF Bettor (10%)

```
DJ:252: uint256 topPrize = P / 10;
DJ:253: (address w, ) = _bafTop(lvl, 0);
DJ:254: if (_creditOrRefund(w, topPrize, tmpW, tmpA, n)) { ++n; }
DJ:259: else { toReturn += topPrize; }
```

**Recipient:** The #1 BAF bettor for this level. Retrieved via `_bafTop(lvl, 0)` which reads `bafTop[lvl][0]` if `0 < bafTopLen[lvl]`, else returns `address(0)` (DJ:659-663).

**Amount:** `P / 10` = 10% of pool.

**Refund:** If no top bettor exists (empty leaderboard), the full 10% is added to `toReturn`.

### 2.2 Slice A2 -- Top Coinflip Bettor (5%)

```
DJ:265: uint256 topPrize = P / 20;
DJ:266: (address w, ) = coin.coinflipTopLastDay();
DJ:267: if (_creditOrRefund(w, topPrize, tmpW, tmpA, n)) { ++n; }
DJ:272: else { toReturn += topPrize; }
```

**Recipient:** The top coinflip bettor from the last 24-hour window, retrieved via an external view call to the coin contract at DJ:266. This is independent of the BAF leaderboard -- it queries `IDegenerusCoinJackpotView.coinflipTopLastDay()` defined at DJ:25.

**Amount:** `P / 20` = 5% of pool.

### 2.3 Slice B -- Random 3rd/4th BAF (5%)

```
DJ:280: entropy = uint256(keccak256(abi.encodePacked(entropy, salt)));
DJ:281: uint256 prize = P / 20;
DJ:282: uint8 pick = 2 + uint8(entropy & 1);
DJ:283: (address w, ) = _bafTop(lvl, pick);
```

**Recipient:** Either the 3rd or 4th BAF leaderboard slot, selected pseudo-randomly. `entropy & 1` produces 0 or 1, so `pick` is either 2 or 3 (indices into the top-4 board). This provides a random tie-break between the 3rd and 4th positions.

**Amount:** `P / 20` = 5% of pool.

**Entropy chain:** Fresh entropy derived via `keccak256(abi.encodePacked(entropy, salt))` at DJ:280 with `salt` incremented at DJ:278.

### 2.4 Slice D -- Far-Future Ticket Holders, 1st Draw (5%)

```
DJ:297: entropy = uint256(keccak256(abi.encodePacked(entropy, salt)));
DJ:298: address[] memory farTickets = degenerusGame.sampleFarFutureTickets(entropy);
DJ:300: uint256 farFirst = (P * 3) / 100;
DJ:301: uint256 farSecond = P / 50;
```

**Sampling:** External view call to `DegenerusGame.sampleFarFutureTickets(entropy)` at DJ:298, which samples up to 4 far-future ticket holders from levels [current+5, current+99] (DG:2669-2705). The function tries up to 10 random levels and returns however many non-zero holders are found (max 4).

**Top-2 selection by BAF score (DJ:309-322):** The sampled ticket holders are ranked by their `_bafScore(cand, lvl)` at DJ:311. The scoring comparison at DJ:312 uses `score > bestScore || best == address(0)`, which handles the zero-score case: a candidate with score 0 can still become `best` if `best` is still `address(0)`. This allows non-BAF-bettors to win if they hold far-future tickets.

**Distribution within slice:**

| Sub-slice | Percentage | Amount |
|-----------|-----------|--------|
| 1st by BAF score | 3% of pool | `(P * 3) / 100` at DJ:300 |
| 2nd by BAF score | 2% of pool | `P / 50` at DJ:301 |
| **Total Slice D** | **5% of pool** | |

### 2.5 Slice D2 -- Far-Future Ticket Holders, 2nd Draw (5%)

```
DJ:339: entropy = uint256(keccak256(abi.encodePacked(entropy, salt)));
DJ:340: address[] memory farTickets = degenerusGame.sampleFarFutureTickets(entropy);
DJ:342: uint256 farFirst = (P * 3) / 100;
DJ:343: uint256 farSecond = P / 50;
```

Identical logic to Slice D but with a fresh entropy seed. This is an independent draw: the same player can win in both D and D2. The top-2 selection at DJ:351-363 uses the same `_bafScore` ranking with the same `|| best == address(0)` fallback pattern (DJ:354).

**Distribution:** Same 3%/2% split as Slice D. Total: 5% of pool.

### 2.6 Slice E-1st -- Scatter 1st Place (45%)

```
DJ:382: uint256 scatterTop = (P * 45) / 100;
DJ:393: for (uint8 round = 0; round < BAF_SCATTER_ROUNDS; ) {
```

**Scatter mechanics:** 50 rounds of trait-based ticket sampling (`BAF_SCATTER_ROUNDS = 50`, DJ:114). Each round samples up to 4 trait burn tickets from a target level via `degenerusGame.sampleTraitTicketsAtLevel(targetLvl, entropy)` at DJ:418.

**Level targeting (DJ:399-416):**

| BAF Type | Rounds 0-3 | Rounds 4-7 | Rounds 8-11 | Rounds 12-19 | Rounds 20-29 | Rounds 30-39 | Rounds 40-49 |
|----------|-----------|-----------|------------|-------------|-------------|-------------|-------------|
| Non-century | lvl+1 | lvl+1 | lvl+1 | lvl+1 | lvl+2 | lvl+3 | lvl+4 |
| Century (x00) | lvl+1 | lvl+2 | lvl+3 | random past 99 | random past 99 | random past 99 | random past 99 |

For non-century: 20 rounds at lvl+1, 10 at lvl+2, 10 at lvl+3, 10 at lvl+4 (DJ:412-415).
For century: 4 rounds at lvl+1, 4 at lvl+2, 4 at lvl+3, 38 rounds sampling from the past 99 levels (DJ:403-409).

**Century random past level formula (DJ:408-409):**
```solidity
uint24 maxBack = lvl > 99 ? 99 : lvl - 1;
targetLvl = maxBack > 0 ? lvl - 1 - uint24(entropy % maxBack) : lvl;
```
Range: `[lvl-99, lvl-1]` when `lvl > 99`; `[1, lvl-1]` when `lvl <= 99`. Edge case: if `maxBack == 0` (only possible if `lvl == 1`, which cannot trigger century), falls back to `lvl`.

**Per-round top-2 selection (DJ:429-444):** Each round selects up to 4 tickets, ranks them by `_bafScore(cand, lvl)` at DJ:431, and takes the top-2. The scatter comparison at DJ:432 uses `score > bestScore` (strict greater-than, no `address(0)` fallback), unlike the far-future comparison at DJ:312 which uses `score > bestScore || best == address(0)`. This means scatter candidates with zero BAF score (non-bettors) will NOT beat the initial `bestScore = 0` default. Only the first non-zero-address candidate will be promoted via the `best != address(0)` check at DJ:447.

[NEW FINDING] **BAF-01: Inconsistent zero-score handling between far-future and scatter selection.** The far-future selection (DJ:312) explicitly allows zero-score candidates to win via `|| best == address(0)`. The scatter selection (DJ:432) does not have this fallback, meaning a single ticket holder with zero BAF score will still be selected as `best` (via the `best != address(0)` check at DJ:447) but a second zero-score candidate cannot become `second` (stuck at 0 > 0 = false at DJ:437). This is functionally correct but inconsistent -- in far-future, two zero-score candidates can both win (1st and 2nd), while in scatter, only one zero-score candidate wins per round. **Severity: INFO** -- no economic impact since scatter prizes are split evenly among all round winners, and the second-place scatter prize goes to a different accumulator (`secondWinners`). A zero-score-only round just yields 1 first-place winner instead of 1 first + 1 second, causing slightly higher per-winner payout for first-place and the second-place prize accruing to fewer winners.

**Winner accumulation (DJ:447-458):** First-place and second-place winners from each round are bucketed into `firstWinners[50]` and `secondWinners[50]` fixed-size arrays with separate counts.

**Prize distribution (DJ:465-481, DJ:483-498):** After all 50 rounds:

- If `firstCount == 0`: full scatterTop goes to `toReturn` (DJ:466).
- Otherwise: `per = scatterTop / firstCount`, with remainder `scatterTop - per * firstCount` going to `toReturn` (DJ:468-470). Each first-place winner receives `per` wei.
- Same logic for secondCount with `scatterSecond` (DJ:483-498).

### 2.7 Slice E-2nd -- Scatter 2nd Place (25%)

```
DJ:383: uint256 scatterSecond = P / 4;
```

The second-place scatter pool is `P / 4 = 25%` of total pool. Distribution follows the same logic described in 2.6 but uses the `secondWinners` array.

### 2.8 Return Values

```
DJ:516-521: winners = tmpW; amounts = tmpA;
            assembly ("memory-safe") { mstore(winners, n); mstore(amounts, n) }
DJ:523:     winnerMask = mask;
DJ:526-528: _clearBafTop(lvl); ++bafEpoch[lvl]; lastBafResolvedDay = currentDayView();
DJ:529:     return (winners, amounts, winnerMask, toReturn);
```

The function truncates the pre-allocated 107-element arrays to actual size `n` using inline assembly at DJ:518-521. This is safe because `n <= 107` is bounded by the max possible winners (1 + 1 + 1 + 2 + 2 + 50 + 50 = 107, matching the allocation at DJ:240).

**Return values:**
- `winners`: Address array of length `n` (actual winner count)
- `amounts`: Corresponding prize amounts
- `winnerMask`: Bitmask with scatter winner flags in upper bits (see Section 4)
- `returnAmountWei`: Cumulative `toReturn` from all unfilled slices (returned to caller as refund)

| Slice | Percentage | Expression | Verification |
|-------|-----------|------------|--------------|
| A | 10% | `P / 10` | DJ:252 |
| A2 | 5% | `P / 20` | DJ:265 |
| B | 5% | `P / 20` | DJ:281 |
| D (1st) | 3% | `(P * 3) / 100` | DJ:300 |

## 3. DSC-02 Impact on BAF

### 3.1 The Issue

`sampleFarFutureTickets` at DG:2669-2705 is the view function used by BAF Slices D and D2 to sample far-future ticket holders. The known finding DSC-02 (from Phase 81 audit, `v4.0-ticket-creation-queue-mechanics.md` Section 4) documents that DG:2681 reads from `_tqWriteKey(candidate)` instead of `_tqFarFutureKey(candidate)`.

**DG:2681:**
```solidity
address[] storage queue = ticketQueue[_tqWriteKey(candidate)];
```

**Expected:**
```solidity
address[] storage queue = ticketQueue[_tqFarFutureKey(candidate)];
```

### 3.2 Impact on BAF Fairness

For candidate levels in the range [currentLvl+5, currentLvl+99] (DG:2679), the far-future key space (`_tqFarFutureKey`, which sets bit 22, GS:699-700) contains the actual ticket holders for these distant levels. The write key space (`_tqWriteKey`, which uses bit 23 based on `ticketWriteSlot`, GS:686-688) is the near-future buffer for levels within +6.

**Consequence for BAF:**
- For levels in [currentLvl+5, currentLvl+6]: The write key buffer MAY contain entries (these levels are within the near-future range). So `sampleFarFutureTickets` might return results for these 2 levels, but reads from the write buffer instead of the far-future buffer.
- For levels in [currentLvl+7, currentLvl+99]: The write key buffer is almost certainly empty for these distant levels (tickets were routed to far-future key space). So `sampleFarFutureTickets` returns empty results.

**Net effect:** Slices D and D2 will almost always fail to find any ticket holders, causing both 5% slices (10% total) to flow into `toReturn` and be refunded to the pool. This means the BAF jackpot effectively distributes only 90% of its pool to winners, with 10% recycled back to `futurePrizePool`.

**Severity: INFO** -- No funds are lost (they return to the prize pool). But the far-future ticket holder reward mechanism is effectively disabled for BAF. This is consistent with the Phase 81 assessment: "view function correctness issue, no state mutation." The economic impact is that far-future ticket purchases do not provide the intended BAF eligibility benefit.

## 4. winnerMask Usage Analysis

### 4.1 Construction in DegenerusJackpots

The `winnerMask` is constructed in the scatter section at DJ:501-513:

```solidity
uint256 scatterCount = n - scatterStart;                      // DJ:501
if (scatterCount != 0) {
    uint256 targetSpecialCount = scatterCount < BAF_SCATTER_TICKET_WINNERS
        ? scatterCount : BAF_SCATTER_TICKET_WINNERS;           // DJ:503-505
    for (uint256 i; i < targetSpecialCount; ) {
        uint256 idx = (scatterStart + scatterCount - 1) - i;  // DJ:507
        mask |= (uint256(1) << (BAF_SCATTER_MASK_OFFSET + idx)); // DJ:508
        unchecked { ++i; }
    }
}
```

**BAF_SCATTER_MASK_OFFSET = 128** (DJ:107): Scatter flags are placed in bits 128+ of the mask.
**BAF_SCATTER_TICKET_WINNERS = 40** (DJ:111): Up to 40 scatter winners get flagged.

The flags are set for the LAST `targetSpecialCount` scatter winners (counting backward from `n-1`). The index formula `(scatterStart + scatterCount - 1) - i` walks backward through the scatter entries in the winners array.

### 4.2 Consumption in EndgameModule

[NEW FINDING] **BAF-02: winnerMask return value is discarded by EndgameModule.**

At EM:358-363:
```solidity
(
    address[] memory winnersArr,
    uint256[] memory amountsArr,
    ,                              // <-- winnerMask discarded here
    uint256 refund
) = jackpots.runBafJackpot(poolWei, lvl, rngWord);
```

The third return value (winnerMask) is explicitly discarded using the blank `,` syntax at EM:361. The EndgameModule's `_runBafJackpot` function at EM:345-423 processes all winners uniformly using the large/small threshold split (EM:373-416) with NO reference to `winnerMask`.

**Tracing all consumers:** A search for `winnerMask` across the entire codebase shows:
- DJ:105-107 (constant declaration)
- DJ:110-111 (constant declaration)
- DJ:214, DJ:219, DJ:226, DJ:237 (NatSpec and return type declaration)
- DJ:245, DJ:379, DJ:508, DJ:523, DJ:529 (construction and return in runBafJackpot)
- IDegenerusJackpots.sol:15, :23 (interface declaration)
- EM:361 (discarded on receipt)

**No code in EndgameModule or any other module reads the winnerMask value.** It is constructed, returned across the external call boundary, and immediately discarded.

**Assessment:** The `winnerMask` and its supporting constants (`BAF_SCATTER_MASK_OFFSET`, `BAF_SCATTER_TICKET_WINNERS`) represent dead code within the current BAF system. The NatSpec at DJ:214 states "winnerMask flags scatter winners for ticket routing" and DJ:379 comments "Game applies special ticket handling for the last BAF_SCATTER_TICKET_WINNERS scatter winners via winnerMask", but the game does not implement this routing. The 40-winner scatter ticket flag construction at DJ:501-513 burns gas for no functional effect.

**Severity: INFO** (dead code / gas waste, no security impact). The wasted gas is bounded: the winnerMask construction loop runs at most 40 iterations of simple bitwise operations.

## 5. BAF Payout Processing (EndgameModule)

### 5.1 Large/Small Winner Threshold

```solidity
// EM:373
uint256 largeWinnerThreshold = poolWei / 20; // 5% of total BAF pool
```

Winners are classified as "large" if their individual prize `amount >= largeWinnerThreshold` (EM:381). At the 5% threshold, this captures:
- Slice A (10% of pool): Always large
- Slice A2 (5% of pool): Exactly at threshold (5% = 5%), classified as large
- Slice B (5% of pool): Exactly at threshold, classified as large
- Slice D/D2 first-place (3% each): Below threshold (3% < 5%), classified as small
- Slice D/D2 second-place (2% each): Below threshold, classified as small
- Scatter per-winner: Depends on winner count; 45%/firstCount or 25%/secondCount

### 5.2 Large Winner Payout

For large winners (`amount >= largeWinnerThreshold`), EM:381-401:

1. **50% ETH (EM:382-386):** `ethPortion = amount / 2`, credited via `_addClaimableEth(winner, ethPortion, rngWord)` at EM:386.

2. **50% Lootbox (EM:383, EM:389-400):** `lootboxPortion = amount - ethPortion` (handles odd amounts by giving the extra wei to lootbox).
   - If `lootboxPortion <= LOOTBOX_CLAIM_THRESHOLD` (5 ETH, GS:132-133): Award immediately via `_awardJackpotTickets(winner, lootboxPortion, lvl, rngWord)` at EM:391-396.
   - If `lootboxPortion > LOOTBOX_CLAIM_THRESHOLD`: Defer via `_queueWhalePassClaimCore(winner, lootboxPortion)` at EM:399.

3. **Lootbox tracking (EM:401):** `lootboxTotal += lootboxPortion`.

### 5.3 Small Winner Payout

For small winners (`amount < largeWinnerThreshold`), EM:403-411:

- **Even index (i % 2 == 0, EM:404-406):** 100% ETH via `_addClaimableEth(winner, amount, rngWord)`.
- **Odd index (i % 2 == 1, EM:407-411):** 100% lootbox via `_awardJackpotTickets(winner, amount, lvl, rngWord)`, with `lootboxTotal += amount`.

This alternating pattern is a gas optimization: instead of splitting every small winner 50/50 (which would double the number of operations), entire prizes alternate between ETH and lootbox. Over many small winners, the expected split converges toward 50/50.

### 5.4 Pool Accounting

**Within _runBafJackpot (EM:345-423):**

```solidity
lootboxToFuture = lootboxTotal;              // EM:419
netSpend = poolWei - refund;                 // EM:421
return (netSpend, claimableDelta, lootboxToFuture);  // EM:422
```

**In runRewardJackpots (EM:179-199), the caller:**

```solidity
futurePoolLocal -= bafPoolWei;               // EM:184 (deduct full pool)
(netSpend, claimed, lootboxToFuture) = _runBafJackpot(...);  // EM:189
claimableDelta += claimed;                   // EM:190
if (netSpend != bafPoolWei) {
    futurePoolLocal += (bafPoolWei - netSpend);  // EM:193 (refund unspent)
}
if (lootboxToFuture != 0) {
    futurePoolLocal += lootboxToFuture;      // EM:197 (recycle lootbox ETH)
}
```

**Net pool impact:** `futurePrizePool -= (netSpend - lootboxToFuture)`. The ETH that goes to claimable balances leaves the future pool permanently; the ETH that becomes lootbox tickets stays in the future pool (since those tickets are backed by pool ETH).

**claimablePool update (EM:237-239):**
```solidity
if (claimableDelta != 0) {
    claimablePool += claimableDelta;          // EM:238
}
```

### 5.5 _addClaimableEth Auto-Rebuy

`_addClaimableEth(address beneficiary, uint256 weiAmount, uint256 entropy)` at EM:256-305 includes auto-rebuy functionality. If the player has `autoRebuyState[beneficiary].autoRebuyEnabled == true` (EM:265):

1. **Compute rebuy (EM:266-274):** `_calcAutoRebuy` determines how many tickets to buy and at what level, with BPS bonuses (13,000 and 14,500 at EM:272-273).

2. **If tickets can be purchased (EM:275-299):**
   - Route ETH to `futurePrizePool` or `nextPrizePool` depending on `calc.toFuture` (EM:280-284).
   - Queue tickets via `_queueTickets(beneficiary, calc.targetLevel, calc.ticketCount)` at EM:286.
   - Credit any reserved (take-profit) portion to claimable at EM:288-291.
   - Return `0` (no claimable delta for the auto-rebuy portion).

3. **If no tickets (EM:275-278):** Fall through to normal `_creditClaimable(beneficiary, weiAmount)` at EM:276, return `weiAmount`.

4. **Non-auto-rebuy (EM:302-304):** Straight `_creditClaimable` + return `weiAmount`.

This auto-rebuy mechanism is present in the EndgameModule version of `_addClaimableEth`. The research notes that the DegeneretteModule's `_addClaimableEth` does NOT include auto-rebuy. This is consistent with the EndgameModule serving jackpot-level payouts where auto-rebuy is a meaningful feature.

### 5.6 _awardJackpotTickets

`_awardJackpotTickets(address winner, uint256 amount, uint24 minTargetLevel, uint256 entropy)` at EM:438-476 is the unified jackpot ticket award function.

### 5.7 _queueWhalePassClaimCore

`_queueWhalePassClaimCore(address winner, uint256 amount)` at `DegenerusGamePayoutUtils.sol:75` handles large lootbox payouts:

```solidity
function _queueWhalePassClaimCore(address winner, uint256 amount) internal {
    if (winner == address(0) || amount == 0) return;           // PU:76
    uint256 fullHalfPasses = amount / HALF_WHALE_PASS_PRICE;   // PU:78
    uint256 remainder = amount - (fullHalfPasses * HALF_WHALE_PASS_PRICE);  // PU:79
    if (fullHalfPasses != 0) {
        whalePassClaims[winner] += fullHalfPasses;             // PU:82
    }
    if (remainder != 0) {
        unchecked { claimableWinnings[winner] += remainder; }  // PU:86
        claimablePool += remainder;                            // PU:88
        emit PlayerCredited(winner, winner, remainder);        // PU:89
    }
}
```

**HALF_WHALE_PASS_PRICE = 2.25 ether** (DegenerusGamePayoutUtils.sol:17-18).

The function converts ETH amount into half-whale-pass units (each = 2.25 ETH = 1 ticket/level for 100 levels). The `whalePassClaims[winner]` mapping at GS:808 accumulates half-passes across multiple jackpot awards.

**Claiming (EM:530-549):** `claimWhalePass(address player)` reads `whalePassClaims[player]` at EM:532, zeroes it at EM:536 (before awarding, preventing reentrancy), applies stats via `_applyWhalePassStats(player, startLevel)` at EM:546 (GS:1076), and queues 100 levels of tickets via `_queueTicketRange(player, startLevel, 100, uint32(halfPasses))` at EM:548. Start level is always `level + 1` (EM:544).

## 6. BAF Cleanup and State Reset

After constructing all return values, `runBafJackpot` performs cleanup at DJ:525-528:

### 6.1 _clearBafTop

`_clearBafTop(uint24 lvl)` at DJ:668-679:

```solidity
function _clearBafTop(uint24 lvl) private {
    uint8 len = bafTopLen[lvl];
    if (len != 0) {
        delete bafTopLen[lvl];                // DJ:671 (zero the length)
    }
    for (uint8 i; i < len; ) {
        delete bafTop[lvl][i];               // DJ:674 (zero each PlayerScore slot)
        unchecked { ++i; }
    }
}
```

This explicitly zeroes both the length counter and each PlayerScore entry. The loop runs up to 4 iterations (max board size). After clearing, `bafTopLen[lvl] == 0` and all `bafTop[lvl][i]` are zeroed.

### 6.2 Epoch Increment

`++bafEpoch[lvl]` at DJ:527 increments the epoch counter for this level. Any player whose `bafPlayerEpoch[lvl][player]` still references the old epoch will have their `bafTotals` lazily cleared on their next `recordBafFlip` call (DJ:178-181). This avoids iterating over all players.

### 6.3 lastBafResolvedDay

`lastBafResolvedDay = degenerusGame.currentDayView()` at DJ:528 records the current day index. This is an external view call back to the game contract. The value is exposed via `getLastBafResolvedDay()` at DJ:686-688 and may be consumed by off-chain indexers.
