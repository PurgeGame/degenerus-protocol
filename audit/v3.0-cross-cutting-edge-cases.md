## EDGE-01: GAMEOVER at Level 0, Level 1, Level 100 Boundaries

**Verdict:** PASS
**Files:** `DegenerusGameGameOverModule.sol:68-164`, `DegenerusGameAdvanceModule.sol:412-462`, `DegenerusGameDecimatorModule.sol:880-921`
**Summary:** All three boundary levels execute correctly through `handleGameOverDrain`. Level 0 aliases to 1 for distribution purposes, safety valve is correctly skipped for level 0 and respected for level 1, terminal jackpot handles the case where no tickets exist at the target level by returning 0 paidWei.

---

### EDGE-01a: GAMEOVER at Level 0

**Preconditions:**
```
state:
  level             = 0
  levelStartTime    = T₀ (≥365 days ago)
  address(this).balance = 45 ETH
  stETH.balanceOf(this) = 5 ETH
  claimablePool     = 10 ETH
  deityPassOwners   = [Alice, Bob]
  deityPassPurchasedCount[Alice] = 1
  deityPassPurchasedCount[Bob]   = 1
  rngWordByDay[day] = 0xABCD... (non-zero)
```

**Trigger check (AdvanceModule:421-423):**
```
liveness = (lvl == 0 && ts - levelStartTime > 365 * 1 days)
         = (0 == 0 && elapsed > 31,536,000s) = TRUE
```

**Safety valve check (AdvanceModule:442-445):**
```solidity
if (lvl != 0 && _getNextPrizePool() >= levelPrizePool[lvl]) {
    levelStartTime = ts;
    return false;
}
```
`lvl == 0`, so this check is SKIPPED. GAMEOVER proceeds unconditionally at level 0.

**handleGameOverDrain execution trace (GameOverModule:68-164):**

Step 1 -- Level aliasing:
```
currentLevel = 0
lvl = (currentLevel == 0) ? 1 : currentLevel = 1
```

Step 2 -- Fund snapshot:
```
ethBal  = 45 ETH
stBal   = 5 ETH
totalFunds = 50 ETH
```

Step 3 -- Deity refunds (currentLevel < 10 branch fires):
```
refundPerPass = 20 ETH (DEITY_PASS_EARLY_GAMEOVER_REFUND)
ownerCount    = 2 (Alice, Bob)
budget        = totalFunds - claimablePool = 50 - 10 = 40 ETH

Alice: purchasedCount = 1 -> refund = 20 ETH; budget = 40 - 20 = 20 ETH
       claimableWinnings[Alice] += 20 ETH; totalRefunded = 20 ETH
Bob:   purchasedCount = 1 -> refund = 20 ETH; budget = 20 - 20 = 0 ETH
       claimableWinnings[Bob]   += 20 ETH; totalRefunded = 40 ETH
budget = 0 -> loop breaks

totalRefunded = 40 ETH
claimablePool += 40 -> claimablePool = 50 ETH
```

Step 4 -- Available funds calculation:
```
available = totalFunds - claimablePool = 50 - 50 = 0 ETH
```

Step 5 -- Early return path (available == 0):
```solidity
if (available == 0) {
    gameOverFinalJackpotPaid = true;
    // pool cleanup
    return;
}
```

**Result:** When deity refunds consume all available funds, GAMEOVER skips the terminal decimator and jackpot entirely. The entire balance becomes claimable by the deity pass holders. No division or distribution happens on empty amounts.

**Invariant at level 0 GAMEOVER:**
- `address(this).balance + stETH = 50 ETH`
- `claimablePool = 50 ETH`
- Invariant: `50 >= 50` -- HOLDS

**Variant: More players, smaller deity refunds**

If `deityPassPurchasedCount[Alice] = 0` and `deityPassPurchasedCount[Bob] = 0`:
- `totalRefunded = 0`; `claimablePool = 10 ETH`
- `available = 50 - 10 = 40 ETH`
- `decPool = 40 / 10 = 4 ETH`
- Terminal decimator runs with lvl=1 as target
- Terminal jackpot runs with lvl=2 as target (lvl+1 = 1+1 = 2)
- If no tickets at level 2: `runTerminalJackpot` returns `paidWei=0`
- `remaining -= 0`; `_sendToVault(40, stBal)` sweeps remainder

**Critical finding check:** What if `decPool = 0` (no available funds)? The `if (decPool != 0)` guard at GameOverModule:139 prevents calling `runTerminalDecimatorJackpot` with 0. What if no tickets at terminal jackpot level? `_distributeJackpotEth` at JackpotModule:1537 can return 0 if all winner buckets are empty (JackpotModule:1653 early return on `totalCount == 0`). Confirmed: no division by zero or revert.

### EDGE-01b: GAMEOVER at Level 1

**Preconditions:**
```
state:
  level             = 1
  levelStartTime    = T₀ (≥120 days ago)
  address(this).balance = 100 ETH
  stETH.balanceOf(this) = 10 ETH
  claimablePool     = 20 ETH
  deityPassOwners   = [Alice]
  deityPassPurchasedCount[Alice] = 1
  nextPrizePool     = 5 ETH (below levelPrizePool[1])
```

**Trigger check:**
```
liveness = (lvl != 0 && ts - levelStartTime > 120 days) = TRUE
```

**Safety valve check (AdvanceModule:442-445):**
```
lvl != 0 -> check fires
_getNextPrizePool() = 5 ETH
levelPrizePool[1] = target (e.g., 100 ETH)
5 < 100 -> safety valve DOES NOT reset timer -> GAMEOVER proceeds
```

If instead `nextPrizePool >= levelPrizePool[1]` (pool target met), the safety valve fires:
```
levelStartTime = ts  (reset timer)
return false          (no GAMEOVER)
```
This is the correct behavior -- level 1+ allows indefinite deferral if the pool target is met.

**Execution when GAMEOVER fires at level 1:**
```
currentLevel = 1; lvl = 1 (no aliasing)
totalFunds = 110 ETH

Deity refunds (level < 10):
  budget = 110 - 20 = 90 ETH
  Alice: 20 ETH; totalRefunded = 20; budget = 70
  claimablePool = 40 ETH

available = 110 - 40 = 70 ETH
decPool = 70 / 10 = 7 ETH
  -> runTerminalDecimatorJackpot(7 ETH, lvl=1, rng)
  -> If no terminal dec burns at level 1: returns 7 ETH (refund)
  -> decSpend = 7 - 7 = 0 ETH
  -> claimablePool unchanged from terminal dec
remaining = 70 - 7 + 7 = 70 ETH (all returned)

Terminal jackpot: runTerminalJackpot(70 ETH, targetLvl=2, rng)
  -> Targets level 2 ticket pool
  -> If no tickets at level 2: all 70 ETH swept to vault via _sendToVault
```

**Terminal jackpot with no level 2 tickets:** `runTerminalJackpot` calls `_distributeJackpotEth` which calls `_resolveTraitWinners`. With no tickets at level 2, `winners.length == 0` (JackpotModule:1672), returning `(0,0,0,0)`. Total paidWei = 0. GameOverModule:157: `_sendToVault(70, stBal)` handles the sweep.

### EDGE-01c: GAMEOVER at Level 100

**Preconditions:**
```
state:
  level             = 100
  levelStartTime    = T₀ (≥120 days ago)
  address(this).balance = 500 ETH
  stETH.balanceOf(this) = 50 ETH
  claimablePool     = 100 ETH
  deityPassOwners   = [10 addresses, 1 pass each]
  nextPrizePool     = 10 ETH (well below target)
  ticketPool at level 101 = 1000 tickets (active game)
```

**Trigger check:** `lvl != 0 && elapsed > 120 days` -- GAMEOVER fires (safety valve: nextPrizePool < levelPrizePool[100]).

**Execution:**
```
currentLevel = 100; lvl = 100
totalFunds = 550 ETH

Deity refunds (currentLevel < 10 is FALSE): NO deity refunds at level 100.
claimablePool = 100 ETH (unchanged)
available = 550 - 100 = 450 ETH

decPool = 450 / 10 = 45 ETH
  -> runTerminalDecimatorJackpot(45 ETH, lvl=100, rng)
  -> Terminal dec uses terminalDecEntries (not normal decBurn)
  -> If winners: claimablePool += 45 (decSpend); remaining = 405
  -> If no winners: decRefund = 45; remaining = 405 (same result)

Terminal jackpot: runTerminalJackpot(405 ETH, targetLvl=101, rng)
  -> Level 101 has 1000 tickets
  -> bucketCounts populated from JackpotBucketLib.bucketCountsForPoolCap
  -> _distributeJackpotEth processes up to DAILY_ETH_MAX_WINNERS per bucket
  -> Winners credited proportionally
```

**Century level boundary -- special x00 behavior in `decWindowOpen`:**
At level 100, `mod100 == 0`. In `_endPhase` (AdvanceModule:470): `if (lvl % 100 == 0) { levelPrizePool[lvl] = _getFuturePrizePool() / 3; }` -- this is the x00 prize pool sizing, which fires at level transition, not at GAMEOVER. The GAMEOVER path does NOT invoke `_endPhase`. No x00-specific behavior in `handleGameOverDrain` -- only the `lvl` value (100) is used for terminal dec target.

**Large ticket pool handling:** `_distributeJackpotEth` uses `JackpotBucketLib.bucketCountsForPoolCap` to compute how many winners are selected per bucket. `MAX_BUCKET_WINNERS` caps winners per bucket (confirmed at JackpotModule:1656: `if (totalCount > MAX_BUCKET_WINNERS) totalCount = MAX_BUCKET_WINNERS`). With 1000 tickets the bucketing mechanism fires but the cap prevents unbounded iteration. No per-winner gas explosion at level 100.

**Verdict: PASS.** All three boundary levels (0, 1, 100) execute without division by zero, overflow, or stuck state. Level 0 is cleanly handled by the lvl-alias and the `currentLevel < 10` refund branch. Level 1 correctly subjects to safety valve. Level 100 fires the same code path as any other level with no special x00 complications in the GAMEOVER drain.

## EDGE-02: Single-Player GAMEOVER

**Verdict:** PASS
**Files:** `DegenerusGameGameOverModule.sol:68-164`, `DegenerusGameJackpotModule.sol:288-324`, `DegenerusGameDecimatorModule.sol:880-921`
**Summary:** All distribution paths handle pool size 1 correctly. No division by zero or empty-winner reverts occur with a single player.

---

**Preconditions:**
```
state:
  level             = 5
  address(this).balance = 200 ETH
  stETH.balanceOf(this) = 20 ETH
  claimablePool     = 30 ETH
  totalFunds        = 220 ETH
  deityPassOwners   = [Alice]
  deityPassPurchasedCount[Alice] = 1
  traitBurnTicket[6] = [[Alice_ticket]] (level 6 has 1 ticket for Alice)
  terminalDecEntries[Alice].bucket != 0 (Alice has a terminal dec burn)
```

**Execution trace (handleGameOverDrain):**

**Step 1 -- Deity refund (level < 10):**
```
refundPerPass = 20 ETH
ownerCount = 1
budget = 220 - 30 = 190 ETH

Alice: purchasedCount = 1 -> refund = 20 ETH; budget = 170
claimableWinnings[Alice] += 20 ETH
totalRefunded = 20 ETH
claimablePool = 30 + 20 = 50 ETH
```

**Step 2 -- Available:**
```
available = 220 - 50 = 170 ETH
```

**Step 3 -- Terminal decimator (10% = 17 ETH):**
```
runTerminalDecimatorJackpot(17 ETH, lvl=5, rng)
  - 11 denominators (2..12) checked; Alice's bucket/subBucket wins one
  - totalWinnerBurn = Alice's weighted burn (non-zero)
  - lastTerminalDecClaimRound.lvl = 5; poolWei = 17 ETH; totalBurn = Alice's burn
  - returns 0 (funds held for claim)
decSpend = 17 - 0 = 17 ETH
claimablePool += 17 -> 67 ETH
remaining = 170 - 17 = 153 ETH
remaining += 0 (no refund) -> 153 ETH
```

**Step 4 -- Terminal jackpot (90% = 153 ETH):**
```
runTerminalJackpot(153 ETH, targetLvl=6, rng)
```

Inside `runTerminalJackpot`:
```
_rollWinningTraits(6, rng, true) -> 4 trait IDs
bucketCounts[i] = [1, 1, 1, 1] (or similar small counts from bucketCountsForPoolCap)
shareBps = [6000, 1333, 1333, 1334]  (FINAL_DAY_SHARES_PACKED rotated)
perWinner[0] = 153 * 6000 / 10000 / 1 = 91.8 ETH
```

Each `_resolveTraitWinners` call:
- `winnerCount = 1` (or 0 if no tickets for that trait at level 6)
- If winnerCount > 0: `winners = [Alice]` (only 1 ticket)
  - `perWinner = traitShare / 1 = traitShare`
  - `_processOneBucket` runs; `isSoloBucket = true` (winnerCount==1)
  - `_processSoloBucketWinner(Alice, perWinner, entropy)` called
  - Alice credited via `_addClaimableEth`
- If winnerCount == 0 for a bucket: `return (entropyState, 0, 0, 0)` -- safe empty return

**Division-by-1 analysis:**
- `perWinner = traitShare / totalCount` where `totalCount = 1` -- no issue
- `_decClaimableFromEntry`: `amountWei = (poolWei * entryBurn) / totalBurn` -- totalBurn is Alice's burn (> 0 since she participated)
- Pro-rata with 1 player: Alice gets 100% of the pool (fraction = 1/1 = 1.0)

**Division-by-zero checks:**
- `_distributeJackpotEth`: `unit = PriceLookupLib.priceForLevel(7) >> 2` -- priceForLevel is a lookup, non-zero for valid levels
- `JackpotBucketLib.bucketCountsForPoolCap`: uses poolWei and entropy, no division by player count

**Empty winner set analysis:**
- If Alice has no tickets at level 6 (she only has burns, not ETH tickets): all 4 trait buckets return 0 winners
- `_distributeJackpotEth` returns `totalPaidEth = 0`
- `remaining -= 0` -> remaining = 153 ETH
- `_sendToVault(153, stBal)` sweeps to vault -- no stuck funds

**Terminal decimator claim (Alice calls `claimTerminalDecimatorJackpot()`):**
```
_consumeTerminalDecClaim(Alice):
  lvl = lastTerminalDecClaimRound.lvl = 5
  e.burnLevel = 5, e.weightedBurn != 0
  packedOffsets = decBucketOffsetPacked[5]
  winningSub = _unpackDecWinningSubbucket(packed, Alice.bucket)
  if e.subBucket == winningSub:
    amountWei = (17 ETH * e.weightedBurn) / totalBurn
    (Alice is the only participant: totalBurn == e.weightedBurn)
    amountWei = 17 ETH (100%)
  e.weightedBurn = 0 (claimed marker)

_addClaimableEth(Alice, 17 ETH, 0):
  gameOver = true -> auto-rebuy skipped
  _creditClaimable(Alice, 17 ETH) -> claimableWinnings[Alice] += 17
```

**handleFinalSweep (30 days later):**
```
claimablePool = 0 (zeroed)
Sweeps remaining balance 50/50 to vault/sDGNRS
```

**Summary for single-player path:**
- Deity refund: `20 ETH / 1 = 20 ETH` per pass -- Alice gets full refund. No division by player count.
- Terminal decimator: pro-rata with 1 participant; `amountWei = poolWei * burn / burn = poolWei` -- Alice gets 100%. No division by zero.
- Terminal jackpot: 1 ticket holder gets per-bucket share. Empty buckets handled with 0 return.
- `handleFinalSweep`: single-call zero-out with vault sweep.

**No loops that iterate zero times** on this path. No percentage splits that produce 0 wei (unless `perWinner = traitShare / bucketCounts` where bucketCounts > traitShare in wei, but then the 0 check at JackpotModule:1675 exits cleanly without crediting anyone).

**Verdict: PASS.** Single-player GAMEOVER is fully handled. All distribution paths have guards for empty sets, zero amounts, and 1-participant scenarios. No division by zero exists.

## EDGE-03: advanceGame Gas Griefing

**Verdict:** FINDING-LOW
**Files:** `DegenerusGameAdvanceModule.sol:156-400`, `DegenerusGameJackpotModule.sol:processTicketBatch`
**Summary:** advanceGame is designed as a resumable multi-transaction state machine. No single call can exceed the block gas limit because ticket batching is bounded per call. However, a sustained attack of buying many tickets can extend the number of advanceGame calls required before a day resolves, adding time delay but not permanent stuck state. Classified as FINDING-LOW: DOS potential exists but is bounded in time by natural protocol economics.

---

### Loop Inventory in advanceGame

| Loop | Location | Iterations per call | Bounded? |
|------|----------|---------------------|---------|
| Ticket queue draining (`_runProcessTicketBatch`) | AdvanceModule:266-272 | 1 batch per call (returns early after batch) | YES -- resumes next call |
| Future ticket activation (`_prepareFutureTickets`) | AdvanceModule:1039-1067 | 5 levels (lvl+2..+6), 1 batch per level | YES -- returns early after each batch |
| Phase transition (`_processPhaseTransition`) | AdvanceModule:1099-1117 | Fixed: 2 queues + 1 stETH call | YES -- O(1) per call |
| Decimator burn loop in `runDecimatorJackpot` | DecimatorModule:315-333 | 11 denominators (2..12) | YES -- fixed loop, 11 iterations |
| GAMEOVER deity refunds | GameOverModule:83-103 | `deityPassOwners.length` | YES -- see below |

### Gas Cost Estimates per Loop

**Ticket batch processing (`processTicketBatch` in JackpotModule):**
From Phase 27 audit (PAY-16), the batch size is explicitly controlled: tickets are processed in fixed-size batches with a gas budget check. Each processed ticket involves: 1 SLOAD (ticket data, ~2100 gas), 1 SSTORE (mark processed, ~20000 gas), 1 SLOAD per winner candidate. Approximate gas per ticket: 25,000 gas. With a batch cap of approximately 50 tickets per call, gas per call ≈ 1.25M gas -- well within 30M block limit.

**Future ticket activation (5 levels x batch):**
5 delegatecall invocations of `_processFutureTicketBatch`. Each delegatecall overhead ≈ 2,600 gas. With empty queues: ~15,000 gas total. With work to do: delegates to MintModule batch (similar 50-ticket cap assumed).

**Total advanceGame gas (one full non-working call):**
- Safety checks: ~5,000 gas
- `_handleGameOverPath`: ~3,000 gas (no trigger)
- `_enforceDailyMintGate`: ~5,000 gas
- `rngGate`: ~2,000 gas (word already present)
- `_prepareFutureTickets`: ~15,000-50,000 gas
- `payDailyJackpot` (jackpot phase): ~100,000-300,000 gas (winner selection for 4 buckets)
- Total normal call: ~200,000-500,000 gas

**Maximum iterations analysis -- can an attacker exceed 30M gas?**

No single advanceGame call processes all tickets in the queue at once. The batch mechanism (`ticketsFullyProcessed` flag + cursor pattern) ensures that each `advanceGame` call returns early after one batch worth of work. Even with 100,000 tickets in the queue, each call does one batch (~50 tickets, ~1.25M gas) and returns. The caller must call advanceGame ~2000 times to drain 100,000 tickets.

This means:
- **Single call: Cannot exceed 30M gas limit.** The batching design prevents this.
- **Sustained griefing: Can delay day resolution.** An attacker purchasing many tickets forces 100-2000 calls before the daily jackpot fires. At current gas prices, this imposes ongoing BURNIE cost on the game's progression.

**Attacker cost to delay:**
- BURNIE ticket cost: minimum `price` per ticket (varies by level)
- Tickets inflate queue; each added batch requires one more `advanceGame` call
- The advance bounty (0.01 ETH in BURNIE, escalating 2x/3x after 1-2 hours) incentivizes others to clear the queue
- Net effect: attacker pays ticket prices; advance bounty is paid from BURNIE creditFlip to the caller

**Can the game get permanently stuck?**
No. The batch cursor tracks progress. Calling `advanceGame` repeatedly always makes progress. There is no deadlock condition. The stuck-ticket failure mode would require an infinite-length queue, which is bounded by the ticket purchase mechanism (each purchase is rate-limited by BURNIE price and block inclusion).

**Key mitigation:** The advance bounty escalation (1x → 2x after 1h → 3x after 2h) makes it increasingly profitable for external callers to clear ticket queues, providing a natural incentive to keep the game moving.

**FINDING-LOW-EDGE03-01: advanceGame Queue Inflation DOS**
- **Severity:** Low
- **Description:** An attacker can purchase large numbers of tickets at any level, inflating the ticket queue and requiring many sequential `advanceGame` calls before the day resolves. This delays reward distribution but does not permanently block it.
- **Impact:** Daily jackpots can be delayed by hours/days if an attacker sustains high-volume ticket purchasing. The advance bounty partially compensates external callers.
- **Mitigation:** The batch mechanism and advance bounty already mitigate this. Consider adding a maximum queue depth per day or a higher per-address ticket purchase rate limit.
- **Recommendation:** Document this as an accepted design tradeoff; the advance bounty incentivizes clearing. No code change required unless griefing is observed in practice.

## EDGE-04: Decimator lastDecClaimRound Overwrite Timing

**Verdict:** PASS (by-design, confirmed)
**Files:** `DegenerusGameDecimatorModule.sol:297-387`
**Source decision:** Phase 27 STATE.md: "lastDecClaimRound overwrite expiry classified as by-design per v1.1 spec Section 8"
**Summary:** The overwrite mechanism is intentional. Claims expire when the next decimator fires. The ETH that was reserved for unclaimed claims is returned to `futurePoolLocal` (never escrow) -- the pool was pre-reserved but the pro-rata reserve is structural (claimablePool accounting), not per-claim. No attacker can profitably brick victim claims. Timing is irrelevant since the overwrite is atomic with the new resolution.

---

### Mechanism Analysis

**How lastDecClaimRound is set (DecimatorModule:341-349):**
```solidity
// Snapshot last claim round (overwrites previous - old claims expire)
lastDecClaimRound.lvl = lvl;
lastDecClaimRound.poolWei = poolWei;
lastDecClaimRound.totalBurn = uint232(totalBurn);
lastDecClaimRound.rngWord = rngWord;
```

**When is it overwritten?** `runDecimatorJackpot` is called once per level-x5 or x00 decimator event (from EndgameModule via `advanceGame`). The overwrite happens exactly once per decimator event during the level transition.

**Can Alice still claim Round N winnings after Round N+1 fires?**
No. The claim validation (DecimatorModule:369):
```solidity
if (lastDecClaimRound.lvl != lvl) revert DecClaimInactive();
```
After Round N+1, `lastDecClaimRound.lvl = N+1`. Any attempt to claim for `lvl=N` reverts with `DecClaimInactive`.

**Concrete timing scenario:**
```
State: lastDecClaimRound.lvl = N (Alice has 10 ETH claim)
       game advances to next x5 or x00 level
       runDecimatorJackpot fires -> lastDecClaimRound.lvl = N+1

Alice calls claimDecimatorJackpot(N):
  -> DecClaimInactive revert
```

**What happens to unclaimed ETH?**
The decimator pool (e.g., 10% of `futurePoolLocal`) was allocated from `futurePoolLocal` when `runDecimatorJackpot` was called. The poolWei sits in the conceptual "held for claims" accounting. When the next decimator fires:
- `runDecimatorJackpot` is called with a new poolWei
- The previous poolWei is **overwritten** in `lastDecClaimRound.poolWei`
- The previously held ETH was included in `claimablePool` when `runDecimatorJackpot` returned 0 (i.e., funds were considered held, not returned)
- If the old pool was held but unclaimed, the overwrite means the new pool replaces it

From Phase 27 PAY-05 analysis: the claimablePool accounting for decimator is "pre-reserve then deduct." When `runDecimatorJackpot` returns 0, the caller (EndgameModule) does NOT add poolWei to claimablePool -- it trusts the claimable path. The poolWei in lastDecClaimRound is the claimable amount. On overwrite, the old poolWei becomes inaccessible and the ETH it represented is effectively retained in the game's free balance (since claimablePool was not incremented for the old round's remaining unclaimed portion).

This is the by-design expiry mechanism: unclaimed decimator ETH is implicitly recycled back into the game's pool when the next decimator fires. This matches v1.1 spec Section 8.

**Can attacker profit from bricking others' claims?**
To trigger Round N+1, the attacker must:
1. Advance the game to the next decimator level (x5 or x00 from the current level)
2. This requires burning enough BURNIE to move the game forward naturally -- there is no shortcut

The cost to advance one game level is the levelPrizePool target in ETH. No attacker can economically advance the game for free to brick others' decimator claims. The cost far exceeds any potential extraction.

**Time window analysis:**
The window between Round N resolution and Round N+1 overwrite is:
- Minimum: approximately 5 game levels * days-per-level (weeks to months)
- The window is not a "race" -- players have weeks to claim between decimator events

**Verdict: PASS (by-design).** The lastDecClaimRound overwrite is intentional expiry per v1.1 spec Section 8. No attacker can profitably trigger premature overwrite. The claim window is weeks-long by design. No ETH is permanently lost -- unclaimed ETH is recycled into the pool.

**Timing diagram:**
```
Level N (x5/x00):  runDecimatorJackpot(poolWei=D₁, lvl=N) -> claims open
                   |
                   | <- Alice has time to claim (weeks to months)
                   |
Level M (x5/x00):  runDecimatorJackpot(poolWei=D₂, lvl=M) -> overwrites N
                   Alice's Round N claims -> DecClaimInactive
                   D₁ unclaimed portion implicitly recycled to game pool
```

## EDGE-05: Coinflip Auto-Rebuy During Known-RNG Windows

**Verdict:** PASS
**Files:** `BurnieCoinflip.sol:325-362, 700-735`, `DegenerusGameAdvanceModule.sol:737-783`
**Summary:** The protocol has three independent defenses against auto-rebuy carry extraction during known-RNG windows: (1) rngLocked flag blocks claims/deposits during VRF pending, (2) `_targetFlipDay()` targets day+1, placing stakes beyond the known window, (3) auto-rebuy carry is applied on claim, not on deposit. Together these make selective auto-rebuy frontrunning impossible.

---

### RNG Timing Analysis

**VRF request lifecycle:**
```
T₀: advanceGame triggers _requestRng -> vrfCoordinator.requestRandomWords
    rngLockedFlag = true (set in _finalizeRngRequest)
    rngWordCurrent = 0 (cleared)
    rngRequestTime = timestamp

T₁: VRF callback -> fulfillRandomWords
    rngWordCurrent = received_word
    (rngLockedFlag remains true until rngGate processes the word)

T₂: advanceGame called again
    rngGate: currentWord != 0 -> _applyDailyRng(day, currentWord)
    coinflip.processCoinflipPayouts(bonusFlip, currentWord, day) fires
    rngLockedFlag released (_unlockRng(day))
    flipsClaimableDay = epoch (day becomes claimable)
```

**Between T₁ and T₂ (VRF word is in storage, day not yet resolved):**
- `rngWordCurrent` holds the next day's VRF word
- An observer can read `rngWordCurrent` from contract storage
- `win = (rngWordCurrent & 1) == 1` -- the win/loss outcome is knowable

**But: rngLockedFlag is still TRUE between T₁ and T₂:**
```solidity

```

**Auto-rebuy toggle also blocked (BurnieCoinflip:731):**

**Deposit path analysis:**
```solidity

```

A player can make a new deposit while RNG is locked, but the deposit targets `day+1` (the NEXT day after the one whose outcome is known). The known outcome is for `day` (current), not `day+1`.

**Auto-rebuy carry -- can it be selectively applied?**
Auto-rebuy carry is accumulated inside `_claimCoinflipsInternal` when processing win days:
```solidity

    // ...
    if (rebuyActive) {
        // carry += payout (or partial after take-profit)
    }
}
```
The carry is updated in storage as `state.autoRebuyCarry`. For this to be a frontrunning vector, Alice would need to:
1. See that day N is a win (read rngWordCurrent)
2. Enable auto-rebuy before day N resolves to capture the carry
3. Disable auto-rebuy after day N resolves to extract the carry

But: **Step 2 and 3 are both blocked by `rngLocked`**. Auto-rebuy cannot be toggled while the VRF word is pending.

**Explicit timeline:**
```
T₀: VRF requested -> rngLocked = true
T₁: VRF callback -> rngWordCurrent = W (outcome knowable from storage)
    [Alice sees W, determines day N is a win]
    [Alice tries to enableAutoRebuy -> RngLocked revert]
    [Alice tries to disableAutoRebuy -> RngLocked revert]
    [Alice tries to claimCoinflips -> RngLocked revert]
T₂: advanceGame processes W -> day N resolved, rngLocked = false
    [Too late: day N already resolved, carry is set]
    [Auto-rebuy can now be configured for day N+1 outcomes]
```

**Can Alice deposit into day N between T₁ and T₂?**
Yes, if `_depositCoinflip` does not check rngLocked. Looking at the code:
```solidity
function _depositCoinflip(address caller, uint256 amount, bool directDeposit) private {
    // ...
    if (amount != 0) {
        if (amount < MIN) revert AmountLTMin();
        if (_coinflipLockedDuringTransition()) revert CoinflipLocked();
    }
```
The lock check is `_coinflipLockedDuringTransition()`, which gates deposits during BAF jackpot resolution -- NOT during VRF RNG lock. So deposits are possible between T₁ and T₂.

**However, deposits always target day N+1 (not day N):**
```solidity

```
At T₁, the current day is still `day N` (advanceGame hasn't fired yet). `_targetFlipDay()` returns `day N + 1`. The deposit goes into `coinflipBalance[N+1][player]`, which will be resolved the following day.

**Verdict: PASS.** The known-RNG window between VRF callback (T₁) and advanceGame processing (T₂) cannot be exploited for coinflip extraction because: (1) deposits target day+1 (the day after the known outcome), (2) auto-rebuy toggle and carry claims are all blocked by rngLocked, (3) the carry accumulation happens automatically during `_claimCoinflipsInternal` which only fires when the outcome is already set. Alice cannot selectively route winnings to a known-winning day.

## EDGE-06: Affiliate Self-Referral Loops

**Verdict:** PASS
**Files:** `DegenerusAffiliate.sol:386-623`
**Summary:** Direct self-referral is explicitly blocked. Loop formation via intermediaries cannot generate infinite commission. The 0.5 ETH cap per sender per level bounds any cross-account extraction to a maximum of ~20% net loss rate.

---

### Self-Referral Defense Map

**Vector 1: Direct self-referral (Alice refers herself)**
```solidity
// DegenerusAffiliate.sol:423-429
AffiliateCodeInfo storage candidate = affiliateCode[code];
if (
    candidate.owner == address(0) ||
    candidate.owner == sender        // <-- Self-referral check

    _setReferralCode(sender, REF_CODE_LOCKED);

```
**BLOCKED.** `candidate.owner == sender` explicitly catches and redirects to VAULT. Code reference: DegenerusAffiliate.sol:426.

**Vector 2: Proxy loop (Alice -> Bob -> Alice)**
- Alice registers Bob's affiliate code at first purchase
- Bob registers Alice's affiliate code at his first purchase
- At Alice's second purchase, her code is permanently stored as Bob's code

Commission flow on Alice's purchase:
```
affiliateAddr = Bob
upline = _referrerAddress(Bob) = Alice (via Bob's stored code)
upline2 = _referrerAddress(Alice) = Bob  ← loop

Tier 0 (Bob): 20% of scaledAmount
Tier 1 (Alice): 20% of scaledAmount / 5 = 4%
Tier 2 (Bob): 20% of scaledAmount / 25 = 0.8%  ← Bob appears again
```

**Wait -- is Tier 2 actually calculated?**
```solidity
// DegenerusAffiliate.sol:578-590
address upline2 = _referrerAddress(upline);  // upline = Alice; upline2 = Alice's referrer = Bob
if (upline2 != address(0)) {
    uint256 bonus2 = scaledAmount / 25; // 4% of 20% = 0.8%
    ...
    players[cursor] = upline2;  // Bob added as third recipient
```

Bob appears in both Tier 0 (100% of scaledAmount) and Tier 2 (0.8%). However, the distribution is done via a **weighted lottery, not direct payment**:
```solidity
// DegenerusAffiliate.sol:608-618
address winner = _rollWeightedAffiliateWinner(
    players, amounts, cursor, totalAmount, sender, storedCode
);
_routeAffiliateReward(winner, totalAmount);
```
The lottery picks ONE winner from [Bob, Alice, Bob] weighted by amounts. If Bob wins, he gets the full totalAmount (100% + 4% + 0.8% = ~104.8% of scaledAmount). But this is in BURNIE, not ETH. The ETH is not taken from the protocol -- affiliate rewards are minted as BURNIE credit via `creditFlip`.

**Maximum extraction from Alice -> Bob -> Alice loop:**
```
Alice's purchase: 1 ETH
rewardScaleBps = 2000 (20%, for levels 4+)
scaledAmount = 0.2 ETH equivalent in BURNIE

Cap check: affiliateCommissionFromSender[lvl][Bob][Alice] <= 0.5 ETH
  After 2.5 purchases at 0.2 ETH each: cap reached -> no more commission from Alice to Bob

Total commission to Bob over the loop: max 0.5 ETH BURNIE per level
```

**Vector 3: Three-hop loop (Alice -> Bob -> Charlie -> Alice)**
Tier 0: Bob gets scaledAmount
Tier 1: Charlie gets 20% of scaledAmount (Bob's upline)
Tier 2: Alice gets 4% of scaledAmount (Charlie's upline)

Alice receives BURNIE worth 4% of her purchase's scaledAmount (= 4% of 20% = 0.8% of purchase).
Alice paid 1 ETH in, received 0.008 ETH equivalent in BURNIE back. Net: -99.2% for Alice.
Alice cannot profit from this arrangement.

**Vector 4: Multi-account (Alice refers herself via alternate address Bob_fake)**
Alice uses a second wallet as "affiliate":
- Alice buys 1 ETH using Bob_fake's affiliate code
- Bob_fake (Alice's alt) gets 20% commission in BURNIE
- Net: Alice spent 1 ETH, got ~0.2 ETH in BURNIE back

**Cap: 0.5 ETH BURNIE per sender per level (DegenerusAffiliate.sol:498).**
This prevents Bob_fake (alt) from earning more than 0.5 ETH from Alice (main) per level.

**Maximum extraction via multi-account per level:**
```
At level 4+: scaledAmount = 20% of purchase amount
Cap = 0.5 ETH BURNIE
Activation purchase required: 0.5 / 0.20 = 2.5 ETH to hit cap

Alice spends 2.5 ETH; alt gets 0.5 ETH BURNIE back
Net: 2.5 ETH out, 0.5 ETH (BURNIE) back = 20% "kickback" max
```

This is the intended affiliate commission rate -- it IS the 20% commission. There is no extraction beyond the designed rate.

**Can the attacker game across multiple levels?**
The cap `MAX_COMMISSION_PER_REFERRER_PER_LEVEL` resets per level:
```solidity
mapping(address => uint256) storage earned = affiliateCoinEarned[lvl];
```
After each level transition, the Alice -> alt commission cap resets. Maximum extraction per level: 0.5 ETH BURNIE per referred address per affiliate per level. With N levels, max is N * 0.5 ETH BURNIE per referral pair.

**Maximum extractable over full game:** N levels * 0.5 ETH BURNIE. But each extraction requires 2.5 ETH of purchases (at 20% rate). This is not profitable extraction -- it is the designed affiliate system operating as intended.

**Verdict: PASS.** Direct self-referral is blocked (code:426). Loop formation cannot generate infinite commission -- each loop tier pays the next person in the chain from BURNIE minting, not from ETH, and the weighted lottery distributes a total of scaledAmount regardless of loop length. The 0.5 ETH cap per sender per affiliate per level bounds maximum commission extraction to the intended 20% affiliate rate.

### Worst-Case Accumulation Analysis

**Scenario: 1000 purchase transactions at 0.1 ETH each over 100 days**

Per transaction BPS divisions triggered:
- Affiliate reward: 1 site, max 9999 wei rounding loss
- Ticket price calculation: 1 site, max 9999 wei
- Auto-rebuy calculation (if enabled): 2 sites, max 19998 wei

Per transaction total truncation: max ~40,000 wei (~0.00004 ETH)

Over 1000 transactions: 1000 * 40,000 = 40,000,000 wei = 0.04 ETH

**Scenario: 100 days of advanceGame with pool operations**

Per advanceGame BPS divisions:
- `_applyTimeBasedFutureTake`: 3 BPS divisions (bps calc, variance, insurance) -- max 29,997 wei
- `_drawDownFuturePrizePool`: 1 percent division -- max 9999 wei
- `payDailyJackpot` bucket shares: 4 x `bucketShares` divisions -- max 39,996 wei

Per advanceGame: max ~80,000 wei rounding loss

Over 100 days: 100 * 80,000 = 8,000,000 wei = 0.008 ETH

**Combined worst-case over realistic game (100 levels, 5 days/level, 1000 purchases/level):**
```
Purchase rounding: 1000 purchases * 100 levels * 40,000 wei = 4,000,000,000 wei = 4 ETH
Daily rounding:    100 levels * 5 days * 80,000 wei = 40,000,000 wei = 0.04 ETH
Total: ~4 ETH rounding loss over full game lifecycle
```

### Where Does the "Lost" Rounding Go?

Rounding truncation means the **distributed amount is always less than or equal to the pool**. The unspent wei stays in:
- `address(this).balance` or `stETH.balanceOf(this)` (for ETH-based divisions)
- Pool variables (`futurePrizePool`, `nextPrizePool`, `claimablePool`) are only incremented by exact amounts

Since all distributions use floor division, the protocol always distributes slightly less than allocated. The remainder stays in the game's balance, effectively acting as a very small "dust reserve."

### Impact on INV-01 (claimablePool Solvency)

`claimablePool` is only incremented by amounts already calculated with floor division. The invariant `balance + stETH >= claimablePool` is:
- For each `claimablePool += delta`: delta comes from floor-divided pool allocation (so balance >= delta, and we only ADD to claimablePool)
- For each `claimablePool -= payout`: payout is the exact amount from `claimableWinnings`, which was credited via the same floor-division path

**Rounding NEVER violates INV-01** because:
1. Rounding always reduces distributions (truncation), so claimablePool is always under-credited relative to gross pool amounts
2. Under-crediting leaves more in the game's balance, strengthening the invariant
3. No rounding path can OVER-credit claimablePool

**Rounding always favors the protocol (not the user):** Solidity integer division truncates toward zero. All reward calculations use `a * b / c` (floor), meaning the distributed amount is `floor(true_amount)`. The remainder (up to `c-1` wei) stays in the protocol.

**Cross-reference with INV-01:** Rounding accumulation is benign for solvency. The 4 ETH lifetime rounding across a full game represents funds retained in the game rather than distributed, not lost. These will be swept to vault/sDGNRS during `handleFinalSweep`.

**Verdict: PASS.** Rounding is systematic truncation (floor division), always in favor of the protocol. Worst-case lifetime accumulation is approximately 4 ETH over a full game lifecycle. This does not threaten claimablePool solvency (INV-01), as under-distribution strengthens the invariant. No rounding accumulation path exists that could over-credit any player or drain claimablePool.
