### DGS Slot Shift Summary

27 of 42 DGS variables have shifted slot numbers compared to v3.8. All shifts are caused by the v3.8 Phase 73 boon packing, which replaced 29 individual boon mappings with a single `boonPacked` mapping (GS:1470), collapsing approximately 28 storage slots from the layout. Variables in packed slots 0-1 and sequential slots 2-16 are unaffected. Variables at slot 17+ are shifted by varying amounts depending on their position relative to the removed boon mappings.

**These slot shifts are INFO-level observations only.** Solidity storage slot assignments are compile-time deterministic and do not affect runtime behavior. The v3.8 audit's functional conclusions (guard analysis, outcome influence analysis) are not affected by slot renumbering. The slot numbers matter only for storage layout documentation accuracy and `forge inspect` cross-referencing.

## 4. Delta Assessment (RDV-03)

### Delta 1: ticketQueue (slot 15) -- Three Key Spaces

**What v3.8 said (v3.8-commitment-window-inventory.md, line 1889):**
> Protection: "Double-buffer" -- ticketQueue writes go to `_tqWriteKey(targetLevel)` (using TICKET_SLOT_BIT at bit 23), processTicketBatch reads from `_tqReadKey(targetLevel)`. Buffer swap at `_swapAndFreeze` is atomic.

**What current code does:**
- TICKET_FAR_FUTURE_BIT (1 << 22) added at GS:162 (v3.9 Phase 74)
- `_tqFarFutureKey(lvl)` pure helper at GS:699-701 returns `lvl | TICKET_FAR_FUTURE_BIT`
- Three disjoint key spaces: Slot 0 `[0x000000-0x3FFFFF]`, Far-Future `[0x400000-0x7FFFFF]`, Slot 1 `[0x800000-0xBFFFFF]` (collision-freedom proven by Phase 74 fuzz tests)
- `_queueTickets` (GS:537-558): line 544 computes `isFarFuture = targetLevel > level + 6`; line 545 guards `if (isFarFuture && rngLockedFlag && !phaseTransitionActive) revert RngLocked()`; line 546 routes to `_tqFarFutureKey(targetLevel)` or `_tqWriteKey(targetLevel)`
- `_queueTicketsScaled` (GS:565-603): same guard pattern at GS:572-573
- `_queueTicketRange` (GS:611-641): same guard pattern at GS:621-622
- `_awardFarFutureCoinJackpot` (JM:2521-2580): reads ONLY from `ticketQueue[_tqFarFutureKey(candidate)]` at JM:2543 (FF-only read after commit 2bf830a2 reverted the combined pool approach)
- `processFutureTicketBatch` in AdvanceModule: new function draining FF keys during ticket processing, reading from `ticketQueue[_tqFarFutureKey(lvl)]`

**SAFE verdict affected:** No. The v3.9 changes EXPANDED protection:
1. Double-buffer separation still holds for near-future tickets (Slot 0/Slot 1 key spaces)
2. FF key space adds a THIRD isolation layer with its own rngLockedFlag guard
3. The rngLockedFlag guard on FF writes (GS:544-545) blocks permissionless far-future ticket writes during the VRF commitment window
4. The phaseTransitionActive exemption allows advanceGame-internal writes (vault/sDGNRS perpetual tickets) since advanceGame holds the lock

**Cross-reference:** DSC-01 (v3.9 RNG proof is stale regarding combined pool) is already documented in audit/v4.0-findings-consolidated.md. Not re-flagged here.

### Delta 2: ticketsOwedPacked (slot 16) -- Same FF Key Routing

**What v3.8 said (v3.8-commitment-window-inventory.md, line 1890):**
> Protection: "Double-buffer" -- same double-buffer mechanism as ticketQueue.

**What current code does:**
- ticketsOwedPacked is keyed by the same `wk` variable computed in `_queueTickets` (GS:546-557), `_queueTicketsScaled` (GS:574-602), and `_queueTicketRange` (GS:623-634)
- Same three key spaces as ticketQueue (Slot 0, FF, Slot 1)
- Same rngLockedFlag guard protects FF writes

**SAFE verdict affected:** No. Same reasoning as ticketQueue -- protection expanded, not reduced.

### Delta 3: _awardFarFutureCoinJackpot -- FF-Only Read (Post-Revert)

**What v3.9 Phase 79 proof described:**
> Combined pool approach: `readLen + ffLen` with index routing across both queues.

**What current code does (JM:2521-2580):**
> FF-only: reads only from `ticketQueue[_tqFarFutureKey(candidate)]` at JM:2543. No `readLen`, no `combinedLen`, no dual-queue indexing.

This is simpler and strictly safer than the combined pool approach described in the v3.9 proof. The backward trace is trivially satisfied: the only data source is the FF key queue, which is frozen during VRF commitment window by the rngLockedFlag guard on all three queue helpers.

**Cross-reference:** DSC-01 already documents this stale proof. Not re-flagged.

### Delta 4: boonPacked -- Storage Layout Collapse

**What v3.8 assumed:**
> 29 individual boon mappings occupied slots between the deity/earlybird section and the VRF configuration section.

**What current code does:**
> `boonPacked` mapping (GS:1470) replaced 29 individual boon mappings with a 2-slot packed struct per player. Old mapping slots remain as placeholders per GS:1468-1469 comment. This collapsed ~28 slots from the sequential layout, causing all subsequent variable slot numbers to shift.

**SAFE verdict affected:** No. boonPacked is NOT in the VRF path. Boon variables were never in the v3.8 CW-04 inventory (they have no permissionless mutation paths that influence VRF-dependent outcomes). The slot shifts affect only documentation accuracy, not runtime security.

### sDGNRS rngLockedFlag Guard Verification

The rngLockedFlag guard for all 7 sDGNRS verdict variables is applied at two entry points:

1. **`burn(uint256 amount)`** at sDGNRS:443-451:
   - Line 448: `if (game.rngLocked()) revert BurnsBlockedDuringRng();`
   - `game.rngLocked()` calls DegenerusGame's `rngLocked()` public view function, which returns the `rngLockedFlag` boolean from DGS slot 0, offset 26.
   - After the guard passes, calls `_submitGamblingClaim(msg.sender, amount)` which writes to pendingRedemption* variables.

2. **`burnWrapped(uint256 amount)`** at sDGNRS:461-469:
   - Line 467: `if (game.rngLocked()) revert BurnsBlockedDuringRng();`
   - Same guard pattern. After passing, calls `_submitGamblingClaimFrom(msg.sender, ContractAddresses.DGNRS, amount)`.

Both guards confirmed present in current code. The `rngLocked()` view function on the game contract (confirmed at interface sDGNRS:20) returns the correct `rngLockedFlag` value.

**Post-gameOver bypass:** Both `burn` and `burnWrapped` have an early-return path for `game.gameOver()` (sDGNRS:444-446, sDGNRS:463-465) that takes the deterministic burn path without the rngLockedFlag check. This is correct because after gameOver, rngLockedFlag is irrelevant (no more VRF requests) and deterministic burns do not write to any of the 7 verdict variables.
