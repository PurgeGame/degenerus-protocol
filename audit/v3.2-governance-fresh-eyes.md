## GOV-01: Attack Surface Catalogue

### 1. Governance State Machine

The VRF swap governance system has 6 state transitions covering the full proposal lifecycle:

#### Transition 1: IDLE -> PROPOSE

- **Function:** `propose()` (DegenerusAdmin.sol lines 398-445)
- **Preconditions:**
  - `subscriptionId != 0` (line 402, revert `NotWired`)
  - `!gameAdmin.gameOver()` (line 403, revert `GameOver`)
  - `newCoordinator != address(0) && newKeyHash != bytes32(0)` (lines 404-405, revert `ZeroAddress`)
  - 1-per-address active proposal limit: `activeProposalId[msg.sender]` must be 0 or reference a non-active/expired proposal (lines 408-415, revert `AlreadyHasActiveProposal`)
  - **Admin path:** `vault.isVaultOwner(msg.sender)` at line 421, stall >= 20h (`ADMIN_STALL_THRESHOLD`, line 300) checked at line 422
  - **Community path:** stall >= 7 days (`COMMUNITY_STALL_THRESHOLD`, line 303) checked at line 425, plus `sDGNRS.balanceOf(msg.sender) * BPS >= circ * COMMUNITY_PROPOSE_BPS` (0.5%) at lines 427-428
- **State changes:** Creates Proposal struct at `proposals[++proposalCount]` with `state = Active` (default 0), records `circulatingSnapshot` (line 438), sets `activeProposalId[msg.sender]` (line 442)

#### Transition 2: PROPOSE -> VOTE

- **Function:** `vote()` (DegenerusAdmin.sol lines 452-517)
- **Trigger:** Immediate -- any sDGNRS holder can vote on an Active proposal
- **Preconditions:**
  - Stall re-check: `block.timestamp - uint256(lastVrf) >= ADMIN_STALL_THRESHOLD` (lines 454-456, revert `NotStalled`)
  - Proposal must be Active with non-zero `createdAt` (lines 459-460, revert `ProposalNotActive`)
  - Not expired: `block.timestamp - uint256(p.createdAt) < PROPOSAL_LIFETIME` (lines 463-466, sets state to Expired and reverts `ProposalExpired`)
  - Voter weight > 0: `sDGNRS.balanceOf(msg.sender) > 0` (lines 470-471, revert `InsufficientStake`)

#### Transition 3: VOTE -> EXECUTED

- **Function:** `_executeSwap()` called from `vote()` (lines 501-506)
- **Condition:** `p.approveWeight * BPS >= uint256(t) * p.circulatingSnapshot && p.approveWeight > p.rejectWeight` (lines 502-503)
- **Where `t`:** `threshold(proposalId)` at line 498, decaying from 5000 BPS (50%) to 500 BPS (5%) over 168h (lines 530-539)

#### Transition 4: VOTE -> KILLED

- **Function:** State set directly in `vote()` (lines 510-516)
- **Condition:** `p.rejectWeight > p.approveWeight && p.rejectWeight * BPS >= uint256(t) * p.circulatingSnapshot` (lines 511-512)
- **State change:** `p.state = ProposalState.Killed` (line 514)

#### Transition 5: VOTE -> EXPIRED

- **Function:** Checked at the start of `vote()` (lines 463-466)
- **Condition:** `block.timestamp - uint256(p.createdAt) >= PROPOSAL_LIFETIME` where `PROPOSAL_LIFETIME = 168 hours` (line 309)
- **State change:** `p.state = ProposalState.Expired` (line 464), then reverts with `ProposalExpired`

#### Transition 6: VOTE -> AUTO-INVALIDATED

- **Function:** Stall re-check at the top of `vote()` (lines 454-456)
- **Condition:** VRF recovers, causing `block.timestamp - uint256(lastVrf) < ADMIN_STALL_THRESHOLD` (20h)
- **Effect:** `vote()` reverts with `NotStalled()`. No state change -- the proposal remains Active but is effectively dead because no votes can be cast. Eventually expires at 168h via Transition 5.

### 3. Attack Surface Enumeration

#### Propose Phase

**AS-01: Stall Fabrication -- Can attacker cause VRF stall?**

- **Scenario:** Attacker attempts to force a VRF stall to enable governance proposals.
- **Code references:** `lastVrfProcessed()` called at DegenerusAdmin.sol line 417 (propose) and line 454 (vote). The value is set in DegenerusGameAdvanceModule.sol at line 1360 (`_applyDailyRng`) and line 402 (`wireVrf`).
- **Defense:** VRF stalls are caused by Chainlink coordinator failure -- an external service. The attacker cannot control Chainlink uptime. `lastVrfProcessedTimestamp` is only writable by `_applyDailyRng` (called during successful VRF processing) and `wireVrf` (deploy-only, admin-gated). No external setter exists.
- **Verdict:** **SAFE** -- VRF stalls are externally caused, not attacker-controlled.

**AS-02: `lastVrfProcessed()` Manipulation -- Can stall measurement be spoofed?**

- **Scenario:** Attacker attempts to manipulate the stall timer to make governance appear eligible when VRF is healthy.
- **Code references:** `lastVrfProcessedTimestamp` in DegenerusGameStorage.sol line 1591. Set ONLY in: (1) `_applyDailyRng` at AdvanceModule line 1360, (2) `wireVrf` at AdvanceModule line 402. Both are internal/admin-gated.
- **Defense:** `_applyDailyRng` is a private function only reachable through the VRF fulfillment path. `wireVrf` requires `msg.sender == ContractAddresses.ADMIN`. No public setter exists. The `lastVrfProcessed()` view function at DegenerusGame.sol line 2227 simply returns `lastVrfProcessedTimestamp`.
- **Verdict:** **SAFE** -- No external manipulation path exists.

**AS-03: Community Proposal with Zero Circulating Supply**

- **Scenario:** All sDGNRS is undistributed (held by sDGNRS contract itself or DGNRS wrapper), causing `circulatingSupply()` to return 0. Attacker tries to propose with zero circulating supply.
- **Code references:** DegenerusAdmin.sol line 427: `if (circ == 0 || sDGNRS.balanceOf(msg.sender) * BPS < circ * COMMUNITY_PROPOSE_BPS) revert InsufficientStake()`.
- **Defense:** The `circ == 0` check explicitly handles this case -- reverts with `InsufficientStake`. No division by zero is possible.
- **Verdict:** **SAFE** -- Explicit zero-supply guard prevents the attack.

**AS-04: Malicious Coordinator Address**

- **Scenario:** Proposer provides an arbitrary address as the new VRF coordinator. No on-chain validation that the address is a real VRF coordinator.
- **Code references:** DegenerusAdmin.sol lines 404-405 only check for `address(0)` and `bytes32(0)`. No interface validation.
- **Defense:** By-design. The governance mechanism assumes sDGNRS voters will vet the proposed coordinator address before approving. If voters approve a malicious coordinator, the system swaps to it. This is the intended trust model -- governance = community responsibility.
- **Verdict:** **SAFE** (by-design -- governance assumes voter vetting, documented as WAR-01 accepted risk).

#### Vote Phase

**AS-05: Vote Weight Inflation via sDGNRS Acquisition During Stall**

- **Scenario:** Attacker attempts to acquire more sDGNRS during a VRF stall to inflate their vote weight before voting.
- **Code references:** Vote weight from `sDGNRS.balanceOf(msg.sender)` at DegenerusAdmin.sol line 470 (live balance, not snapshot).
- **Defense:** During a VRF stall, sDGNRS balances are effectively frozen:
  1. **Game pool distributions** (`transferFromPool` at StakedDegenerusStonk.sol line 308): `onlyGame` modifier, requires `advanceGame` which needs a VRF word -- blocked during stall.
  2. **DGNRS.unwrapTo()** (DegenerusStonk.sol lines 149-158): Blocked when stall > 5 hours (line 153), which is before the 20h governance threshold. Creator cannot convert DGNRS to sDGNRS once governance becomes eligible.
  3. **Burning sDGNRS** (StakedDegenerusStonk.sol line 373): Reduces supply, does not help attacker.
  4. **Pool operations** (`burnRemainingPools`, `transferBetweenPools`): `onlyGame` modifier -- blocked during stall.
  5. **No transfer function**: sDGNRS is soulbound -- StakedDegenerusStonk.sol has no `transfer()` or `transferFrom()` public functions.
- **Verdict:** **SAFE** -- sDGNRS is frozen during VRF stalls; no acquisition path exists.

**AS-06: Double-Voting via Vote Change -- Arithmetic Safety**

- **Scenario:** Voter changes their vote direction (approve -> reject or vice versa). Old weight is subtracted, new weight is added. Can arithmetic underflow occur?
- **Code references:** DegenerusAdmin.sol lines 474-481 for old weight subtraction, lines 489-493 for new weight addition.
- **Defense:** Solidity 0.8.34 uses checked arithmetic by default. `p.approveWeight -= oldWeight` (line 478) and `p.rejectWeight -= oldWeight` (line 480) will revert on underflow. The `oldWeight` is read from `voteWeight[proposalId][msg.sender]` (line 476), which was stored at the time of the previous vote. If accounting is correct (weight added on vote, same weight subtracted on change), underflow is impossible. The weight stored at vote time equals the weight that was added, so subtraction is safe.
- **Verdict:** **SAFE** -- Checked arithmetic in Solidity 0.8.34 prevents underflow. Accounting is correct.

**AS-07: Front-Running Execute with Reject Vote**

- **Scenario:** Attacker sees that approval threshold is about to be met, front-runs with a large reject vote to prevent execution.
- **Code references:** DegenerusAdmin.sol lines 501-507 (execute condition), lines 510-516 (kill condition). Execute requires `approveWeight > rejectWeight` (line 503). Kill requires `rejectWeight > approveWeight` (line 511).
- **Defense:** This is a feature, not a bug. Reject voters CAN block execution by ensuring `rejectWeight >= approveWeight`. This is the intended governance design -- community opposition can prevent coordinator swaps.
- **Verdict:** **SAFE** -- By-design feature enabling community veto.

**AS-08: Flash Loan sDGNRS for Vote Weight**

- **Scenario:** Attacker flash-loans sDGNRS to temporarily gain vote weight, votes, then returns the tokens.
- **Code references:** Vote weight from `sDGNRS.balanceOf(msg.sender)` at DegenerusAdmin.sol line 470.
- **Defense:** sDGNRS is **soulbound** -- StakedDegenerusStonk.sol has no `transfer()` or `transferFrom()` public functions. The only "transfer"-like function is `wrapperTransferTo` (line 242) restricted to `msg.sender == ContractAddresses.DGNRS`, and pool functions restricted to `onlyGame`. There is no way to lend, borrow, or flash-loan sDGNRS. No transfer function means no DEX listing, no lending protocol integration, and no flash loan surface.
- **Verdict:** **SAFE** -- Soulbound token with no transfer function eliminates flash loan attacks entirely.

#### Execute Phase

**AS-09: Reentrancy During Execution**

- **Scenario:** During `_executeSwap`, an external call re-enters DegenerusAdmin to manipulate governance state.
- **Code references:** DegenerusAdmin.sol `_executeSwap` function (lines 563-624). CEI ordering: (1) `p.state = ProposalState.Executed` at line 565, (2) `_voidAllActive(proposalId)` at line 568, (3) external calls start at line 578+.
- **Defense:**
  - State changes complete before external calls (CEI ordering confirmed -- GOV-07 fix verified).
  - External calls: `cancelSubscription` (try/catch, line 578-585), `createSubscription` (line 590-591), `addConsumer` (line 598-601), `updateVrfCoordinatorAndSub` (line 605-609), `transferAndCall` (try/catch, line 614-620).
  - `updateVrfCoordinatorAndSub` is called on `gameAdmin` which is the Game contract. Game executes this via delegatecall to AdvanceModule. The AdvanceModule function checks `msg.sender == ContractAddresses.ADMIN` (AdvanceModule line 1263). Since the caller IS Admin, this succeeds. No callback path back to Admin exists.
  - VRF coordinator operations (`createSubscription`, `addConsumer`) call the proposed coordinator. Even if malicious, the proposal state is already `Executed` and all other proposals are already `Killed` via `_voidAllActive`. Re-entering `vote()` or `propose()` cannot undo these state changes.
- **Verdict:** **SAFE** -- CEI ordering ensures all state changes complete before external calls. No re-entrant path back to Admin.

**AS-10: Partial Execution Failure**

- **Scenario:** `createSubscription` on the proposed coordinator reverts, causing the entire `_executeSwap` transaction to revert.
- **Code references:** DegenerusAdmin.sol lines 590-591 -- `createSubscription` is NOT wrapped in try/catch. Lines 578-585 -- `cancelSubscription` IS wrapped in try/catch. Lines 614-620 -- `transferAndCall` IS wrapped in try/catch.
- **Defense:** If `createSubscription` reverts, the ENTIRE transaction reverts. This means `p.state = Executed` (line 565) and `_voidAllActive` (line 568) also revert -- no state changes persist. The proposal remains Active and can be voted on again. This is correct behavior: a non-functional coordinator should not be installed, and the governance state should remain unchanged.
- **Verdict:** **SAFE** -- Full revert on non-functional coordinator is the correct design. State is preserved atomically.

**AS-11: State Consistency After Swap**

- **Scenario:** After a successful swap, stale VRF state persists, potentially allowing the old coordinator to interfere.
- **Code references:** `updateVrfCoordinatorAndSub` in DegenerusGameAdvanceModule.sol lines 1258-1276 resets 7 variables:
  1. `vrfCoordinator` = new coordinator (line 1266)
  2. `vrfSubscriptionId` = new sub ID (line 1267)
  3. `vrfKeyHash` = new key hash (line 1268)
  4. `rngLockedFlag` = false (line 1271)
  5. `vrfRequestId` = 0 (line 1272)
  6. `rngRequestTime` = 0 (line 1273)
  7. `rngWordCurrent` = 0 (line 1274)
- **NOT reset:** `lastVrfProcessedTimestamp` -- the stall timer continues from the original stall until the new coordinator processes a VRF request. (See GOV-02 OQ-1 for analysis.)
- **Verdict:** **SAFE** -- All 7 VRF operational variables are reset. `lastVrfProcessedTimestamp` intentionally not reset (see OQ-1 analysis in GOV-02). Full trace deferred to GOV-03.

#### Timing

**AS-12: Threshold Decay Exploitation**

- **Scenario:** Attacker creates a proposal, waits for the threshold to decay to 5% (day 6, 144-168h), then votes with a small sDGNRS holding (>= 5% circulating).
- **Code references:** `threshold()` function at DegenerusAdmin.sol lines 530-539. At 144+ hours: `return 500` (5%, line 533).
- **Defense:** This IS WAR-02 (Colluding Voter Cartel at Day-6 Threshold). Requires: (1) genuine VRF stall persisting for 144+ hours, (2) attacker holding >= 5% circulating sDGNRS, (3) no opposition votes (reject must not exceed approve). See WAR-02 re-verification below.
- **Verdict:** **KNOWN RISK** -- Cross-reference WAR-02.

**AS-13: Proposal Expiry Race at 168h Boundary**

- **Scenario:** Voter calls `vote()` at timestamp very close to 168h from proposal creation. Depending on block timestamp, vote may execute or proposal may expire.
- **Code references:** DegenerusAdmin.sol lines 463-466: `if (block.timestamp - uint256(p.createdAt) >= PROPOSAL_LIFETIME)` sets Expired.
- **Defense:** Block timestamp granularity is ~12 seconds on XRPL EVM sidechain. The window of uncertainty is one block (~12 seconds). This is standard for any time-based check and is not exploitable beyond normal block timing variance. Whether a vote lands at 167h59m48s or 168h00m12s is determined by block inclusion, which is validator-controlled (not MEV-auctioned on XRPL EVM).
- **Verdict:** **SAFE** -- Standard block timestamp granularity, not practically exploitable.

**AS-14: Stall Recovery Race at 20h Boundary**

- **Scenario:** VRF recovers at exactly the 20h mark. A voter tries to vote in the same block that VRF processes a request.
- **Code references:** DegenerusAdmin.sol lines 454-456: `if (block.timestamp - uint256(lastVrf) < ADMIN_STALL_THRESHOLD) revert NotStalled()`.
- **Defense:** If VRF processes a request in the same block, `lastVrfProcessedTimestamp` is updated to `block.timestamp`. The stall check `block.timestamp - lastVrf` becomes 0, which is < 20h, so the vote reverts with `NotStalled`. This is correct behavior -- VRF recovery invalidates governance. On XRPL EVM, there is no MEV auction to manipulate transaction ordering.
- **Verdict:** **SAFE** -- Correct behavior; VRF recovery causes governance reverts.

### 4. Known Issue Re-Verification

#### WAR-01: Compromised Admin + 7-Day Inattention (MEDIUM, Accepted Risk)

Re-verification against current code (DegenerusAdmin.sol at HEAD):

1. **Admin path requires 20h stall AND DGVE majority:** CONFIRMED. `vault.isVaultOwner(msg.sender)` at line 421, stall check `stall < ADMIN_STALL_THRESHOLD` at line 422 where `ADMIN_STALL_THRESHOLD = 20 hours` (line 300).

2. **Community can kill proposals:** CONFIRMED. Kill condition at lines 510-516: `p.rejectWeight > p.approveWeight && p.rejectWeight * BPS >= uint256(t) * p.circulatingSnapshot`. Community with sufficient sDGNRS weight can reject and kill any proposal.

3. **No new path bypasses vote requirement:** CONFIRMED. `_executeSwap` (line 563) is `internal` -- it can only be called from `vote()` at line 505. There is no external `execute()` function. The execute condition is checked inside `vote()` (lines 501-506). No way to execute a proposal without going through the vote flow.

4. **Threshold decay: 50% -> 5% over 168h:** CONFIRMED. `threshold()` function at lines 530-539:
   - 0-48h: 5000 BPS (50%) -- line 538 `return 5000`
   - 48-72h: 4000 BPS (40%) -- line 537
   - 72-96h: 3000 BPS (30%) -- line 536
   - 96-120h: 2000 BPS (20%) -- line 535
   - 120-144h: 1000 BPS (10%) -- line 534
   - 144-168h: 500 BPS (5%) -- line 533
   - 168h+: 0 (expired) -- line 532

5. **Proposal lifetime still 168h:** CONFIRMED. `PROPOSAL_LIFETIME = 168 hours` at line 309. Used in expiry check at line 463.

**WAR-01 Status: CONFIRMED -- all 5 conditions hold in current code. Severity and accepted-risk classification remain accurate.**

#### WAR-02: Colluding Voter Cartel at Day-6 Threshold (MEDIUM, Accepted Risk)

Re-verification against current code:

1. **Threshold at 144-168h is 500 BPS (5%):** CONFIRMED. `threshold()` at line 533: `if (elapsed >= 144 hours) return 500`. This is the minimum non-zero threshold before expiry.

2. **approve > reject requirement exists:** CONFIRMED. Execute condition at line 503: `p.approveWeight > p.rejectWeight` (strict inequality). A cartel must have strictly more approve weight than reject weight. Even a single opposition voter prevents execution if their weight matches.

3. **circulatingSnapshot from proposal creation time:** CONFIRMED. `p.circulatingSnapshot = circulatingSupply()` set once in `propose()` at line 438. Never updated after creation. Used in execute condition at line 502 and kill condition at line 512. Cannot be manipulated post-creation.

4. **sDGNRS frozen during stall:** CONFIRMED.
   - `unwrapTo` blocked at 5h stall: DegenerusStonk.sol line 153, `block.timestamp - IDegenerusGame(ContractAddresses.GAME).lastVrfProcessed() > 5 hours` reverts with `Unauthorized`. Since 5h < 20h governance threshold, sDGNRS supply is frozen before governance can activate.
   - Game distributions require VRF: `advanceGame` needs a VRF word to process, so `transferFromPool` (onlyGame) cannot fire during a stall.
   - Burns reduce supply: Burning sDGNRS removes voting power from the burner while `circulatingSnapshot` stays fixed. This hurts attackers, not helps them.

**WAR-02 Status: CONFIRMED -- all 4 conditions hold in current code. Severity and accepted-risk classification remain accurate.**

#### WAR-06: Admin Spam-Propose Gas Griefing (LOW, Known Issue)

Re-verification against current code:

1. **`voidedUpTo` watermark exists and is used:** CONFIRMED. `voidedUpTo` declared at line 277 (private). `_voidAllActive` function at lines 628-640 starts scanning from `voidedUpTo + 1` (line 629: `uint256 start = voidedUpTo + 1`), not from 1. After voiding, watermark updated: `voidedUpTo = count` at line 639.

2. **1-per-address limit constrains spam per address:** CONFIRMED. `activeProposalId` mapping at line 273. Check in `propose()` at lines 408-415: if existing proposal is Active and not expired, reverts with `AlreadyHasActiveProposal`. Attacker can only have 1 active proposal per address.

3. **Gas cost bounded by `proposalCount`:** CONFIRMED. `_voidAllActive` loop at lines 631-637: `for (uint256 i = start; i <= count; i++)` where `count = proposalCount` (line 630). After optimization via `voidedUpTo`, the effective loop range is `(voidedUpTo + 1)` to `proposalCount`.

4. **No new mitigation beyond watermark exists:** CONFIRMED. The watermark and 1-per-address limit are the only mitigations. No batch deletion, no per-address gas accounting, no proposal deposit/bond mechanism. The gas impact is real but bounded and mitigated.

**WAR-06 Status: CONFIRMED -- all 4 conditions hold in current code. Severity and accepted-risk classification remain accurate.**
