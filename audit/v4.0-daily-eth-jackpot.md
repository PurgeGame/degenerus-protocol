### 1.3 Advance Flow Ordering

The jackpot phase branch in `advanceGame` executes in this priority order (AM:345-381):

1. **Resume check** (AM:350-354): If any daily ETH state is non-zero, resume via Call Site #2
2. **Coin+ticket pending** (AM:362): If `dailyJackpotCoinTicketsPending`, execute Phase 2 via Call Site #3
3. **Fresh start** (AM:379): Otherwise, start fresh daily jackpot via Call Site #4

The purchase phase branch always executes Call Site #1 on non-final purchase days (AM:279-282).

Additionally, before any jackpot processing, the advance module checks at AM:255-261 whether fresh daily tickets need processing:

This ensures future ticket queues are processed ONLY when no daily ETH distribution is in progress.

## 3. BPS Allocation Table (DETH-01)

### 3.1 _dailyCurrentPoolBps Function

```solidity
// JM:2656-2668
function _dailyCurrentPoolBps(
    uint8 counter,
    uint256 randWord
) private pure returns (uint16 bps) {
    if (counter >= JACKPOT_LEVEL_CAP - 1) return 10_000;

    uint16 range = DAILY_CURRENT_BPS_MAX - DAILY_CURRENT_BPS_MIN + 1;
    uint256 seed = uint256(
        keccak256(
            abi.encodePacked(randWord, DAILY_CURRENT_BPS_TAG, counter)
        )
    );
    return uint16(DAILY_CURRENT_BPS_MIN + (seed % range));
}
```

- **Tag:** `DAILY_CURRENT_BPS_TAG = keccak256("daily-current-bps")` (JM:135-136)
- **Modulo range:** `DAILY_CURRENT_BPS_MAX - DAILY_CURRENT_BPS_MIN + 1 = 1400 - 600 + 1 = 801`
- **Output for counter < 4:** `600 + (seed % 801)` = [600, 1400] BPS (6%-14%)
- **Output for counter >= 4:** `10000` BPS (100%) -- the `JACKPOT_LEVEL_CAP - 1 = 4` check at JM:2660

Note: The day 5 override is also enforced at the caller level in `payDailyJackpot`:
```solidity
// JM:362-368
bool isFinalPhysicalDay = (counter + counterStep >= JACKPOT_LEVEL_CAP);
// ...
if (isFinalPhysicalDay) {
    dailyBps = 10_000; // Final physical day: 100% of remaining pool
} else {
    dailyBps = _dailyCurrentPoolBps(counter, randWord);
```
So `_dailyCurrentPoolBps` is only called for non-final days. The function's own `counter >= 4` guard is a defense-in-depth redundancy.

### 3.2 Compressed Mode (JM:351-374)

```solidity
// JM:353-361
if (compressedJackpotFlag == 2 && counter == 0) {
    counterStep = JACKPOT_LEVEL_CAP;  // = 5
} else if (
    compressedJackpotFlag == 1 &&
    counter > 0 &&
    counter < JACKPOT_LEVEL_CAP - 1
) {
    counterStep = 2;
}
```

- **Normal mode (flag=0):** `counterStep = 1`, five physical days
- **Compressed mode (flag=1):** `counterStep = 2` on days 2-4 (when `counter > 0 && counter < 4`). Day 1 still has counterStep=1. This collapses 5 logical days into 3 physical days.
- **Turbo mode (flag=2):** `counterStep = 5` on day 1 only (when `counter == 0`). All 5 logical days in 1 physical day.

BPS doubling for compressed mode:
```solidity
// JM:372-374
if (counterStep == 2) {
    dailyBps *= 2;
}
```
When `counterStep == 2`, the random BPS is doubled: e.g., if `_dailyCurrentPoolBps` returns 1000 BPS, the actual dailyBps becomes 2000 BPS. This combines two days' payouts into one physical day.

### 3.3 Counter Step Packing

The counter step is packed into `dailyTicketBudgetsPacked` at JM:459-464 via `_packDailyTicketBudgets`:
```solidity
// JM:2753-2763
function _packDailyTicketBudgets(
    uint8 counterStep,
    uint256 dailyTicketUnits,
    uint256 carryoverTicketUnits,
    uint8 carryoverSourceOffset
) private pure returns (uint256) {
    return
        uint256(counterStep) |
        (dailyTicketUnits << 8) |
        (carryoverTicketUnits << 72) |
        (uint256(carryoverSourceOffset) << 136);
}
```

Layout: `[counterStep (8 bits @ 0)] [dailyTicketUnits (64 bits @ 8)] [carryoverTicketUnits (64 bits @ 72)] [carryoverSourceOffset (8 bits @ 136)]`

### 3.4 Allocation Table

| Day | Counter | Normal Mode (flag=0) | Compressed Mode (flag=1) | Turbo Mode (flag=2) | Share Packing | Source |
|-----|---------|---------------------|-------------------------|---------------------|---------------|--------|
| Day 1 | 0 | 6-14% (600-1400 BPS), step=1 | 6-14%, step=1 (counter=0, not > 0) | 100% (10000 BPS), step=5 (isFinalPhysicalDay=true) | DAILY_JACKPOT (turbo: FINAL_DAY) | JM:370, JM:353-354, JM:362-368 |
| Day 2 | 1 | 6-14%, step=1 | 12-28% (doubled), step=2 | N/A (all done in day 1) | DAILY_JACKPOT | JM:370, JM:356-360, JM:372-374 |
| Day 3 | 2 | 6-14%, step=1 | 12-28% (doubled), step=2 | N/A | DAILY_JACKPOT | JM:370, JM:356-360, JM:372-374 |
| Day 4 | 3 | 6-14%, step=1 | 100% (isFinalPhysicalDay), step=2 | N/A | FINAL_DAY | JM:370, JM:362-368 |
| Day 5 | 4 | 100% (10000 BPS), step=1 | N/A (already done) | N/A | FINAL_DAY | JM:367-368 |

**Notes on compressed day 3 (counter=3):** With counter=3 and counterStep=2, `counter + counterStep = 5 >= JACKPOT_LEVEL_CAP`, so `isFinalPhysicalDay = true` and `dailyBps = 10000`. This is the final physical day in compressed mode, using FINAL_DAY_SHARES_PACKED.

**Notes on compressed day 1:** At counter=0, `compressedJackpotFlag == 1 && counter > 0` is false, so counterStep remains 1. Compressed mode only doubles on days 2 and 3.

**Notes on turbo:** At counter=0, `compressedJackpotFlag == 2 && counter == 0` sets counterStep=5. Then `counter + counterStep = 5 >= 5`, so `isFinalPhysicalDay = true` and `dailyBps = 10000`. All five logical days happen in one physical day with 100% of pool.

## 4. Budget Split Logic (DETH-01)

### 4.1 Fresh-Start Budget Calculation (JM:365-405)

```solidity
// JM:365-376
uint256 poolSnapshot = currentPrizePool;
uint16 dailyBps;
if (isFinalPhysicalDay) {
    dailyBps = 10_000; // Final physical day: 100% of remaining pool
} else {
    dailyBps = _dailyCurrentPoolBps(counter, randWord);
    // Double BPS on compressed days to combine two days' payouts.
    if (counterStep == 2) {
        dailyBps *= 2;
    }
}
uint256 budget = (poolSnapshot * dailyBps) / 10_000;
```

Step 1: `poolSnapshot = currentPrizePool` (JM:365) -- snapshot before any deductions.

Step 2: `dailyBps` determined by day/mode (JM:367-374).

Step 3: `budget = (poolSnapshot * dailyBps) / 10_000` (JM:376).

### 4.2 Early-Bird Lootbox on Day 1 (JM:378-382)

```solidity
// JM:380-382
if (isEarlyBirdDay) {
    _runEarlyBirdLootboxJackpot(lvl + 1, randWord);

```

On day 1 only (`counter == 0`), runs `_runEarlyBirdLootboxJackpot` (JM:801-864):
- Takes 3% from `futurePrizePool`: `reserveContribution = (_getFuturePrizePool() * 300) / 10_000` (JM:803)

### 10.1 traitBucketCounts (JBL:36-51)

```solidity
// JBL:36-51
function traitBucketCounts(uint256 entropy) internal pure returns (uint16[4] memory counts) {
    uint16[4] memory base;
    base[0] = 25; // Large bucket
    base[1] = 15; // Mid bucket
    base[2] = 8;  // Small bucket
    base[3] = 1;  // Solo bucket (receives the 60% share via rotation)

    // Rotate bucket assignments based on entropy for fairness across traits.
    uint8 offset = uint8(entropy & 3);
    for (uint8 i; i < 4; ) {
        counts[i] = base[(i + offset) & 3];
        unchecked { ++i; }
    }
}
```

**Base counts:** `[25, 15, 8, 1]` = 49 total winners before scaling.

**Rotation:** `offset = entropy & 3` (bottom 2 bits of entropy). The assignment `counts[i] = base[(i + offset) & 3]` rotates which trait index receives which bucket size. For example:
- `offset=0`: trait 0 gets 25, trait 1 gets 15, trait 2 gets 8, trait 3 gets 1
- `offset=1`: trait 0 gets 15, trait 1 gets 8, trait 2 gets 1, trait 3 gets 25
- `offset=2`: trait 0 gets 8, trait 1 gets 1, trait 2 gets 25, trait 3 gets 15
- `offset=3`: trait 0 gets 1, trait 1 gets 25, trait 2 gets 15, trait 3 gets 8

This ensures every trait has an equal chance of being the solo bucket (1 winner, highest per-winner payout) or the large bucket (25 winners, lower per-winner payout).

### 10.2 scaleTraitBucketCountsWithCap (JBL:55-95)

```solidity
// JBL:55-95
function scaleTraitBucketCountsWithCap(
    uint16[4] memory baseCounts,
    uint256 ethPool,
    uint256 entropy,
    uint16 maxTotal,
    uint32 maxScaleBps
) internal pure returns (uint16[4] memory counts) {
    counts = baseCounts;
    if (ethPool < JACKPOT_SCALE_MIN_WEI) return counts;   // JBL:64

    uint256 scaleBps;
    if (ethPool < JACKPOT_SCALE_FIRST_WEI) {               // JBL:67
        // Tier 1: linear from 1x to 2x
        uint256 range = JACKPOT_SCALE_FIRST_WEI - JACKPOT_SCALE_MIN_WEI;    // 50-10 = 40 ETH
        uint256 progress = ethPool - JACKPOT_SCALE_MIN_WEI;
        scaleBps = JACKPOT_SCALE_BASE_BPS + (progress * (JACKPOT_SCALE_FIRST_BPS - JACKPOT_SCALE_BASE_BPS)) / range;
    } else if (ethPool < JACKPOT_SCALE_SECOND_WEI) {       // JBL:71
        // Tier 2: linear from 2x to maxScaleBps
        uint256 range = JACKPOT_SCALE_SECOND_WEI - JACKPOT_SCALE_FIRST_WEI; // 200-50 = 150 ETH
        uint256 progress = ethPool - JACKPOT_SCALE_FIRST_WEI;
        scaleBps = JACKPOT_SCALE_FIRST_BPS + (progress * (uint256(maxScaleBps) - JACKPOT_SCALE_FIRST_BPS)) / range;
    } else {                                                 // JBL:75-77
        scaleBps = maxScaleBps;
    }
    // Apply scale to non-solo buckets only (buckets with count > 1)
    // JBL:79-92
    if (scaleBps != JACKPOT_SCALE_BASE_BPS) {
        for (uint8 i; i < 4; ) {
            uint16 baseCount = counts[i];
            if (baseCount > 1) {
                uint256 scaled = (uint256(baseCount) * scaleBps) / 10_000;
                if (scaled < baseCount) scaled = baseCount;   // Floor: never reduce
                if (scaled > type(uint16).max) scaled = type(uint16).max; // Overflow guard
                counts[i] = uint16(scaled);
            }
            unchecked { ++i; }
        }
    }
    return capBucketCounts(counts, maxTotal, entropy);       // JBL:94
}
```

**Piecewise linear scaling tiers:**

| Pool Size | Scale Factor | ScaleBps | Formula |
|-----------|-------------|----------|---------|
| < 10 ETH | 1x (no scaling) | 10000 | Early return at JBL:64 |
| 10-50 ETH | 1x to 2x (linear) | 10000-20000 | `10000 + (progress * 10000) / 40e18` at JBL:67-70 |
| 50-200 ETH | 2x to maxScaleBps (linear) | 20000-maxScaleBps | `20000 + (progress * (maxScaleBps - 20000)) / 150e18` at JBL:71-74 |
| >= 200 ETH | maxScaleBps (capped) | maxScaleBps | `scaleBps = maxScaleBps` at JBL:76 |

**maxScaleBps parameter values:**
- Daily path: `DAILY_JACKPOT_SCALE_MAX_BPS = 66667` (JM:223) = 6.6667x. At 200+ ETH, base counts scale to `[25*6.67, 15*6.67, 8*6.67, 1]` = `[166, 100, 53, 1]` = 320, which is under the 321 cap.
- Early-burn path: `JACKPOT_SCALE_MAX_BPS = 40000` (JM:220) = 4x. At 200+ ETH, base counts scale to `[100, 60, 32, 1]` = 193, under the 300 cap.

**Solo bucket preservation (JBL:82):** The condition `if (baseCount > 1)` ensures solo buckets (count=1) are never scaled. This guarantees exactly one winner in the solo bucket regardless of pool size.

**Floor guard (JBL:84):** `if (scaled < baseCount) scaled = baseCount` -- prevents rounding from reducing counts below their base values. Since `scaleBps >= JACKPOT_SCALE_BASE_BPS (10000)`, this guard triggers only if BPS arithmetic overflows, which is prevented by the `type(uint16).max` check at JBL:85.

### 10.4 capBucketCounts (JBL:115-203)

```solidity
// JBL:115-203
function capBucketCounts(
    uint16[4] memory counts,
    uint16 maxTotal,
    uint256 entropy
) internal pure returns (uint16[4] memory capped) {
    capped = counts;
```

**Edge cases handled first:**
- `maxTotal == 0` (JBL:121-127): all buckets zeroed -- no winners
- `total == 0` (JBL:129-136): all buckets zeroed -- no tickets exist
- `maxTotal == 1` (JBL:137-144): only the solo bucket gets 1 winner (via `soloBucketIndex(entropy)` at JBL:142). All other buckets zeroed.
- `total <= maxTotal` (JBL:145): no capping needed, return as-is

**Proportional reduction algorithm (JBL:147-200):**

1. Reserve 1 winner for solo bucket: `nonSoloCap = maxTotal - 1` (JBL:147)
2. Compute non-solo total: `nonSoloTotal = total - 1` (JBL:148)
3. Scale each non-solo bucket proportionally: `scaled = (bucketCount * nonSoloCap) / nonSoloTotal` (JBL:154)
4. Minimum guarantee: `if (scaled == 0) scaled = 1` (JBL:155) -- every non-solo bucket with winners gets at least 1

**Overflow trimming (JBL:167-183):** When `nonSoloCap` is very small (e.g., 1-2), the minimum-1 guarantee can cause `scaledTotal > nonSoloCap`. The algorithm trims excess by zeroing out buckets that were rounded up to 1. The trim order is entropy-based: `trimOff = (entropy >> 24) & 3` determines starting offset, and it iterates backwards `(trimOff + 3 - i) & 3` to zero out the smallest buckets first. Only buckets where `capped[idx] == 1 && counts[idx] > 1` (i.e., buckets that were rounded up from a larger count) are eligible for trimming.

**Remainder distribution (JBL:185-200):** When `scaledTotal < nonSoloCap`, remaining capacity is distributed to non-solo buckets in entropy-based order: `offset = (entropy >> 24) & 3`, then round-robin `(offset + i) & 3`. Only buckets with `capped[idx] > 1` (already-populated non-solo buckets) receive extras.

### 10.5 bucketShares (JBL:211-237)

```solidity
// JBL:211-237
function bucketShares(
    uint256 pool,
    uint16[4] memory shareBps,
    uint16[4] memory bucketCounts,
    uint8 remainderIdx,
    uint256 unit
) internal pure returns (uint256[4] memory shares) {
    uint256 distributed;
    for (uint8 i; i < 4; ) {
        if (i != remainderIdx) {                              // JBL:220
            uint16 count = bucketCounts[i];
            uint256 share = (pool * shareBps[i]) / 10_000;    // JBL:222
            if (count != 0) {
                if (unit != 0) {
                    uint256 unitBucket = unit * count;         // JBL:225
                    share = (share / unitBucket) * unitBucket; // JBL:226 -- round down to unit
                }
                shares[i] = share;                             // JBL:228
            }
            distributed += share;                              // JBL:230
        }
        unchecked { ++i; }
    }
    shares[remainderIdx] = pool - distributed;                 // JBL:236
}
```

**Share calculation per bucket:**
1. Base share: `(pool * shareBps[i]) / 10000` (JBL:222) -- standard BPS application
2. Unit rounding: `share = (share / (unit * count)) * (unit * count)` (JBL:225-226) -- rounds down to ensure each winner gets exactly `share / count` with no sub-unit fractions. The `unit` is `PriceLookupLib.priceForLevel(lvl + 1) >> 2` (JM:1400).
3. Empty bucket: if `count == 0`, share is not assigned to this bucket but is still counted in `distributed` (JBL:230 executes even when `shares[i]` is not set). This means empty-bucket share flows to the remainder bucket.

**Remainder bucket (JBL:236):** `shares[remainderIdx] = pool - distributed`. This captures:
- All BPS rounding dust from non-remainder buckets
- All unit-rounding dust
- Empty bucket shares
- The remainder bucket's own BPS allocation (skipped in the loop via `i != remainderIdx`)

The `remainderIdx` is always the solo bucket index: `JackpotBucketLib.soloBucketIndex(entropy)` (JM:1401).

### 10.6 soloBucketIndex (JBL:240-242)

```solidity
// JBL:240-242
function soloBucketIndex(uint256 entropy) internal pure returns (uint8) {
    return uint8((uint256(3) - (entropy & 3)) & 3);
}
```

**Formula:** `(3 - (entropy & 3)) & 3`

**Inverse rotation relationship with traitBucketCounts:** In `traitBucketCounts`, `offset = entropy & 3`, and the solo bucket (base[3] = 1 winner) is mapped to trait index `i` where `base[(i + offset) & 3] == 1`, which solves to `(i + offset) & 3 == 3`, giving `i = (3 - offset) & 3`. This is exactly what `soloBucketIndex` computes. So `soloBucketIndex(entropy)` returns the trait index that holds the solo bucket (1 winner) after rotation.

**Verification of inverse:**

| entropy & 3 | traitBucketCounts assigns solo (count=1) to trait index | soloBucketIndex returns |
|-------------|--------------------------------------------------------|----------------------|
| 0 | trait 3 (base[(3+0)&3] = base[3] = 1) | (3-0)&3 = 3 |
| 1 | trait 2 (base[(2+1)&3] = base[3] = 1) | (3-1)&3 = 2 |
| 2 | trait 1 (base[(1+2)&3] = base[3] = 1) | (3-2)&3 = 1 |
| 3 | trait 0 (base[(0+3)&3] = base[3] = 1) | (3-3)&3 = 0 |

Confirmed: `soloBucketIndex` correctly identifies which trait index holds the 1-winner solo bucket.

### 10.8 bucketOrderLargestFirst (JBL:290-306)

```solidity
// JBL:290-306
function bucketOrderLargestFirst(uint16[4] memory counts) internal pure returns (uint8[4] memory order) {
    uint8 largestIdx;
    uint16 largestCount = counts[0];
    for (uint8 i = 1; i < 4; ++i) {                          // JBL:293-298
        if (counts[i] > largestCount) {
            largestCount = counts[i];
            largestIdx = i;
        }
    }
    order[0] = largestIdx;                                     // JBL:299
    uint8 k = 1;
    for (uint8 i; i < 4; ++i) {                              // JBL:301-305
        if (i != largestIdx) {
            order[k++] = i;
        }
    }
}
```

**Algorithm:** Single-pass largest-first extraction. Finds the bucket with the highest count (JBL:293-298), places it at `order[0]`, then fills remaining positions in natural order (0,1,2,3) excluding the largest.

**Returns indices, not sorted counts.** The output `order[i]` is a trait index (0-3), not a count. This is used by `_processDailyEthChunk` at JM:1427 to iterate buckets largest-first.

**Purpose:** Processing the largest bucket first maximizes gas utilization per `advanceGame` call. If gas runs out mid-distribution, the largest bucket (most winners) is already completed, meaning fewer resume cycles are needed. This is a gas optimization for the chunked distribution pattern.

**Tie-breaking:** When counts are equal, the bucket with the lower index wins (`>` strict comparison at JBL:294, so first-found wins). The remaining buckets are ordered by their natural index (0, 1, 2, 3).

**Note:** Only the single largest bucket is pulled to the front. The remaining three buckets are in natural index order, not sorted. This is a deliberate simplification -- full sorting of 4 elements provides negligible gas benefit over the single-extraction approach.

## 12. Resume Logic and Entropy Determinism (DETH-03)

### 12.1 _skipEntropyToBucket (JM:1367-1384)

```solidity
// JM:1367-1384
function _skipEntropyToBucket(
    uint256 entropy,
    uint8[4] memory order,
    uint256[4] memory shares,
    uint16[4] memory bucketCounts,
    uint8 startOrderIdx
) private pure returns (uint256 entropyState) {
    entropyState = entropy;
    for (uint8 j; j < startOrderIdx; ++j) {                   // JBL:1375
        uint8 traitIdx = order[j];
        uint16 count = bucketCounts[traitIdx];
        uint256 share = shares[traitIdx];
        if (count == 0 || share == 0) continue;                // JBL:1379 -- skip empty buckets
        entropyState = EntropyLib.entropyStep(
            entropyState ^ (uint256(traitIdx) << 64) ^ share   // JBL:1380-1382
        );
    }
}
```

**Purpose:** Fast-forward the entropy state to match the position saved by `dailyEthBucketCursor`. Each processed bucket performs one `entropyStep` (at JM:1436-1438). To resume mid-stream, the entropy must be replayed through exactly the same sequence of steps for all already-completed buckets.

**How it works:**
1. Starts from the base entropy (same `randWord ^ (uint256(lvl) << 192)` as the original call)
2. Iterates through `order[0..startOrderIdx-1]` (buckets already completed)
3. For each non-empty bucket, applies the same `entropyStep` with the same XOR mixing (`traitIdx << 64 | share`)
4. Skips empty buckets (`count == 0 || share == 0`) -- matching the `continue` at JM:1431-1434 in the main loop
5. Returns the entropy state positioned at the start of bucket `order[startOrderIdx]`

**Why this produces identical entropy:** All inputs to `_skipEntropyToBucket` are derived from the stored parameters (see Section 12.3), which are identical on resume. The `entropyStep` function is deterministic (LCG), so replaying the same sequence of inputs produces the same output.

### 12.2 Winner Cursor Restore

```solidity
// JM:1413-1414
uint8 startOrderIdx = dailyEthBucketCursor;
uint16 startWinnerIdx = dailyEthWinnerCursor;
```

On resume:
- `startOrderIdx` = saved bucket cursor -> outer loop starts at this bucket
- `startWinnerIdx` = saved winner cursor -> inner loop starts at this winner within the bucket
- For the resumed bucket, `_randTraitTicketWithIndices` is called again with the same parameters, producing the same `winners[]` array. Winners at indices `[0..startWinnerIdx-1]` were already credited in the previous chunk and are skipped.

### 12.3 Determinism Proof

All inputs to `_processDailyEthChunk` on resume must be identical to the original call for the distribution to be deterministic. Verification:

| Input | Storage Source | Set At | Read At | Immutable During Distribution? |
|-------|---------------|--------|---------|-------------------------------|
| `ethPool` | `dailyEthPoolBudget` | JM:394 (fresh-start) | JM:489 (resume) | Yes -- only cleared at `_clearDailyEthState` (JM:2789) which runs after ALL phases complete |
| `lvl` | `lastDailyJackpotLevel` | JM:2633 via `_syncDailyWinningTraits` (fresh-start) | JM:341 (resume) or AM:356 argument | Yes -- level does not change during jackpot phase |
| `randWord` | VRF word passed from AdvanceModule | AM:356 passes the same `rngWord` | Same `rngWord` from AdvanceModule | Yes -- VRF word is the same for all calls within the same day (from `rngWordCurrent` at AM:64) |
| `winningTraitsPacked` | `lastDailyJackpotWinningTraits` | JM:2632 via `_syncDailyWinningTraits` (fresh-start) | JM:340 (resume) | Yes -- only written once per fresh-start |
| `sharesPacked` | Derived from `isFinalPhysicalDay_` and `jackpotCounter` | JM:501-503 | Same derivation on resume | Yes -- `jackpotCounter` is only incremented in `payDailyJackpotCoinAndTickets` (JM:757), which runs AFTER ETH distribution completes |
| `maxScaleBps` | `DAILY_JACKPOT_SCALE_MAX_BPS` | Constant (JM:223) | Same constant | Yes -- immutable |
| `maxWinners` | `DAILY_ETH_MAX_WINNERS` (Phase 0) or `dailyCarryoverWinnerCap` (Phase 1) | JM:193 / JM:531-542 | Same value | Yes -- winner cap does not change during distribution |

**Therefore:** All derived values -- `bucketCounts`, `shares`, `order`, `soloBucketIndex`, `shareBps` -- are identical on resume. Combined with cursor restore and entropy fast-forward via `_skipEntropyToBucket`, the distribution is fully deterministic across resume boundaries. Winners are never double-credited (cursor skips already-processed winners) and no winners are missed (cursor resumes exactly where the previous chunk stopped).

### 12.4 traitBurnTicket Stability

**Concern:** If `traitBurnTicket` content changes between chunks, `_randTraitTicketWithIndices` would produce different winners on resume.

**Analysis:** `traitBurnTicket[lvl]` arrays are written ONLY during ticket processing in `processTicketBatch` (JM:1880+) and `processFutureTicketBatch` (MM:296+). Both functions run during the `_prepareFutureTickets` phase of `advanceGame` (AM:255-261), which is gated by the condition:

```solidity
// AM:255-261
if (
    !dailyJackpotCoinTicketsPending &&
    dailyEthPoolBudget == 0 &&
    dailyEthPhase == 0 &&
    dailyEthBucketCursor == 0 &&
    dailyEthWinnerCursor == 0
) {
    if (!_prepareFutureTickets(lvl)) { ... }
}
```

During ETH distribution, at least one of `dailyEthPoolBudget`, `dailyEthPhase`, `dailyEthBucketCursor`, or `dailyEthWinnerCursor` is non-zero, so `_prepareFutureTickets` is NEVER called. This guarantees `traitBurnTicket` arrays are stable throughout the entire daily ETH distribution (both Phase 0 and Phase 1, across all resume cycles).

**Conclusion:** The chunked distribution is provably deterministic. All inputs are immutable during distribution, entropy is replayed identically, and the ticket pool is frozen by the advance flow ordering guard.
