### 1.3 Advance Flow Ordering

The jackpot phase branch in `advanceGame` executes in this priority order (AM:345-381):

1. **Resume check** (AM:350-354): If any daily ETH state is non-zero, resume via Call Site #2
2. **Coin+ticket pending** (AM:362): If `dailyJackpotCoinTicketsPending`, execute Phase 2 via Call Site #3
3. **Fresh start** (AM:379): Otherwise, start fresh daily jackpot via Call Site #4

The purchase phase branch always executes Call Site #1 on non-final purchase days (AM:279-282).

Additionally, before any jackpot processing, the advance module checks at AM:255-261 whether fresh daily tickets need processing:

```solidity
    dailyEthWinnerCursor == 0

```
This ensures future ticket queues are processed ONLY when no daily ETH distribution is in progress.

## 3. BPS Allocation Table (DETH-01)

### 3.1 _dailyCurrentPoolBps Function

```solidity
// JM:2656-2668
function _dailyCurrentPoolBps(
    uint8 counter,
    uint256 randWord
) private pure returns (uint16 bps) {
    if (counter >= JACKPOT_LEVEL_CAP - 1) return 10_000;

    uint16 range = DAILY_CURRENT_BPS_MAX - DAILY_CURRENT_BPS_MIN + 1;
    uint256 seed = uint256(
        keccak256(
            abi.encodePacked(randWord, DAILY_CURRENT_BPS_TAG, counter)
        )
    );
    return uint16(DAILY_CURRENT_BPS_MIN + (seed % range));
}
```

- **Tag:** `DAILY_CURRENT_BPS_TAG = keccak256("daily-current-bps")` (JM:135-136)
- **Modulo range:** `DAILY_CURRENT_BPS_MAX - DAILY_CURRENT_BPS_MIN + 1 = 1400 - 600 + 1 = 801`
- **Output for counter < 4:** `600 + (seed % 801)` = [600, 1400] BPS (6%-14%)
- **Output for counter >= 4:** `10000` BPS (100%) -- the `JACKPOT_LEVEL_CAP - 1 = 4` check at JM:2660

Note: The day 5 override is also enforced at the caller level in `payDailyJackpot`:
```solidity
// JM:362-368
bool isFinalPhysicalDay = (counter + counterStep >= JACKPOT_LEVEL_CAP);
// ...
if (isFinalPhysicalDay) {
    dailyBps = 10_000; // Final physical day: 100% of remaining pool
} else {
    dailyBps = _dailyCurrentPoolBps(counter, randWord);
```
So `_dailyCurrentPoolBps` is only called for non-final days. The function's own `counter >= 4` guard is a defense-in-depth redundancy.

### 3.2 Compressed Mode (JM:351-374)

```solidity
// JM:353-361
if (compressedJackpotFlag == 2 && counter == 0) {
    counterStep = JACKPOT_LEVEL_CAP;  // = 5
} else if (
    compressedJackpotFlag == 1 &&
    counter > 0 &&
    counter < JACKPOT_LEVEL_CAP - 1
) {
    counterStep = 2;
}
```

- **Normal mode (flag=0):** `counterStep = 1`, five physical days
- **Compressed mode (flag=1):** `counterStep = 2` on days 2-4 (when `counter > 0 && counter < 4`). Day 1 still has counterStep=1. This collapses 5 logical days into 3 physical days.
- **Turbo mode (flag=2):** `counterStep = 5` on day 1 only (when `counter == 0`). All 5 logical days in 1 physical day.

BPS doubling for compressed mode:
```solidity
// JM:372-374
if (counterStep == 2) {
    dailyBps *= 2;
}
```
When `counterStep == 2`, the random BPS is doubled: e.g., if `_dailyCurrentPoolBps` returns 1000 BPS, the actual dailyBps becomes 2000 BPS. This combines two days' payouts into one physical day.

### 3.3 Counter Step Packing

The counter step is packed into `dailyTicketBudgetsPacked` at JM:459-464 via `_packDailyTicketBudgets`:
```solidity
// JM:2753-2763
function _packDailyTicketBudgets(
    uint8 counterStep,
    uint256 dailyTicketUnits,
    uint256 carryoverTicketUnits,
    uint8 carryoverSourceOffset
) private pure returns (uint256) {
    return
        uint256(counterStep) |
        (dailyTicketUnits << 8) |
        (carryoverTicketUnits << 72) |
        (uint256(carryoverSourceOffset) << 136);
}
```

Layout: `[counterStep (8 bits @ 0)] [dailyTicketUnits (64 bits @ 8)] [carryoverTicketUnits (64 bits @ 72)] [carryoverSourceOffset (8 bits @ 136)]`

### 3.4 Allocation Table

| Day | Counter | Normal Mode (flag=0) | Compressed Mode (flag=1) | Turbo Mode (flag=2) | Share Packing | Source |
|-----|---------|---------------------|-------------------------|---------------------|---------------|--------|
| Day 1 | 0 | 6-14% (600-1400 BPS), step=1 | 6-14%, step=1 (counter=0, not > 0) | 100% (10000 BPS), step=5 (isFinalPhysicalDay=true) | DAILY_JACKPOT (turbo: FINAL_DAY) | JM:370, JM:353-354, JM:362-368 |
| Day 2 | 1 | 6-14%, step=1 | 12-28% (doubled), step=2 | N/A (all done in day 1) | DAILY_JACKPOT | JM:370, JM:356-360, JM:372-374 |
| Day 3 | 2 | 6-14%, step=1 | 12-28% (doubled), step=2 | N/A | DAILY_JACKPOT | JM:370, JM:356-360, JM:372-374 |
| Day 4 | 3 | 6-14%, step=1 | 100% (isFinalPhysicalDay), step=2 | N/A | FINAL_DAY | JM:370, JM:362-368 |
| Day 5 | 4 | 100% (10000 BPS), step=1 | N/A (already done) | N/A | FINAL_DAY | JM:367-368 |

**Notes on compressed day 3 (counter=3):** With counter=3 and counterStep=2, `counter + counterStep = 5 >= JACKPOT_LEVEL_CAP`, so `isFinalPhysicalDay = true` and `dailyBps = 10000`. This is the final physical day in compressed mode, using FINAL_DAY_SHARES_PACKED.

**Notes on compressed day 1:** At counter=0, `compressedJackpotFlag == 1 && counter > 0` is false, so counterStep remains 1. Compressed mode only doubles on days 2 and 3.

**Notes on turbo:** At counter=0, `compressedJackpotFlag == 2 && counter == 0` sets counterStep=5. Then `counter + counterStep = 5 >= 5`, so `isFinalPhysicalDay = true` and `dailyBps = 10000`. All five logical days happen in one physical day with 100% of pool.

## 4. Budget Split Logic (DETH-01)

### 4.1 Fresh-Start Budget Calculation (JM:365-405)

```solidity
// JM:365-376
uint256 poolSnapshot = currentPrizePool;
uint16 dailyBps;
if (isFinalPhysicalDay) {
    dailyBps = 10_000; // Final physical day: 100% of remaining pool
} else {
    dailyBps = _dailyCurrentPoolBps(counter, randWord);
    // Double BPS on compressed days to combine two days' payouts.
    if (counterStep == 2) {
        dailyBps *= 2;
    }
}
uint256 budget = (poolSnapshot * dailyBps) / 10_000;
```

Step 1: `poolSnapshot = currentPrizePool` (JM:365) -- snapshot before any deductions.

Step 2: `dailyBps` determined by day/mode (JM:367-374).

Step 3: `budget = (poolSnapshot * dailyBps) / 10_000` (JM:376).

### 4.2 Early-Bird Lootbox on Day 1 (JM:378-382)

```solidity
// JM:380-382
if (isEarlyBirdDay) {
    _runEarlyBirdLootboxJackpot(lvl + 1, randWord);
}
```

On day 1 only (`counter == 0`), runs `_runEarlyBirdLootboxJackpot` (JM:801-864):
- Takes 3% from `futurePrizePool`: `reserveContribution = (_getFuturePrizePool() * 300) / 10_000` (JM:803)

- **This replaces the normal carryover flow on day 1** -- the `isEarlyBirdDay` guard at JM:409 prevents carryover calculation on day 1

- `JACKPOT_SCALE_MAX_BPS = 40000` (JM:220) = 4x max scaling vs daily path's `DAILY_JACKPOT_SCALE_MAX_BPS = 66667` (JM:223) = 6.67x

### 9.6 Non-Chunked Distribution

`_distributeJackpotEth` (JM:1512-1551) processes all 4 buckets in a single loop with no gas budgeting or cursor resume:

```solidity
// JM:1534-1545
for (uint8 traitIdx; traitIdx < 4; ) {
    _processOneBucket(ctx, traitIdx, traitIds, shares, bucketCounts);
    unchecked { ++traitIdx; }
}
```

There is no `unitsBudget` check, no `dailyEthBucketCursor` save, and no early return. This is safe because the early-burn path has a smaller pool (1% of futurePrizePool) and fewer max winners (300), keeping gas costs manageable.

### 9.7 Lootbox Distribution (JM:656-665)

```solidity
// JM:657-664
if (lootboxBudget != 0) {
    _distributeLootboxAndTickets(
        lvl,
        winningTraitsPacked,
        lootboxBudget,
        randWord,
        5_000 // 50% ticket conversion
    );
}
```

`_distributeLootboxAndTickets` (JM:1079-1102) adds the full `lootboxBudget` to `nextPrizePool` (JM:1087), then distributes 50% (`ticketConversionBps = 5000`) as actual ticket entries via `_distributeTicketJackpot`.
