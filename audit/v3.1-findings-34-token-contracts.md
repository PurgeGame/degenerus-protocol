### CMT-041: Orphaned DATA TYPES section documents structs that live in BurnieCoinflip

- **What:** The DATA TYPES section header (lines 213-218) promises "Packed structs for gas-efficient storage" but the section body (lines 220-226) contains NatSpec for two structs -- a "Leaderboard entry" (lines 220-222) and an "Outcome record" (lines 224-226) -- with no corresponding struct declarations. The only struct in BurnieCoin.sol is `Supply` at line 196, which lives in the ERC20 STATE section. The documented leaderboard and outcome structs were moved to BurnieCoinflip.sol during the coinflip split (confirmed: BurnieCoinflip.sol declares `CoinflipDayResult` at line 139, `PlayerCoinflipState` at line 145, and `PlayerScore` at line 169).
- **Where:** BurnieCoin.sol:213-227
- **Why:** A warden reading the DATA TYPES section would expect struct definitions to follow the NatSpec. Finding none, they would investigate whether the structs were removed without removing the documentation, potentially filing a QA finding about phantom type documentation. The section inflates the perceived data model of BurnieCoin beyond its actual storage layout.
- **Suggestion:** Remove the entire DATA TYPES section (lines 213-227). BurnieCoinflip.sol has its own struct documentation. The `Supply` struct at line 196 is already documented in the ERC20 STATE section where it belongs.
- **Category:** comment-inaccuracy
- **Severity:** INFO

### CMT-042: Orphaned BOUNTY STATE section documents variables that live in BurnieCoinflip

- **What:** The BOUNTY STATE section (lines 333-361) contains: (1) a section header describing bounty pool mechanics (accumulation, arming, payout), (2) a STORAGE LAYOUT table referencing slots 17-18 for `currentBounty`, `biggestFlipEver`, and `bountyOwedTo`, and (3) NatSpec for three state variables (lines 352-361). None of these variables are declared in BurnieCoin.sol -- they are declared in BurnieCoinflip.sol at lines 161-163. Verified: `grep -n 'currentBounty\|biggestFlipEver\|bountyOwedTo' contracts/BurnieCoin.sol` returns only comment references in the section header.
- **Where:** BurnieCoin.sol:333-361
- **Why:** A warden reading the BOUNTY STATE section would expect to find `currentBounty`, `biggestFlipEver`, and `bountyOwedTo` as state variables in BurnieCoin.sol. Their absence, combined with a STORAGE LAYOUT table referencing slots 17-18 that do not correspond to BurnieCoin's actual storage, would cause significant confusion about the contract's state model. A warden might suspect storage collisions or a missing inheritance chain. The slot numbers are incorrect for BurnieCoin.sol (slot 0 is `_supply`, slots 1-2 are `balanceOf`/`allowance` mappings).
- **Suggestion:** Remove the entire BOUNTY STATE section (lines 333-361). BurnieCoinflip.sol already documents its own bounty state variables.
- **Category:** comment-inaccuracy
- **Severity:** LOW

### CMT-043: onlyAdmin modifier reverts with OnlyGame() error without acknowledgment comment

- **What:** The `onlyAdmin` modifier at line 662-665 reverts with `OnlyGame()` when the caller is not ADMIN. The coinflip guard functions `burnForCoinflip` (line 519) and `mintForCoinflip` (line 528) also revert with `OnlyGame()` for non-coinflip callers, but both include the inline comment `// Reusing error for simplicity`. The `onlyAdmin` modifier has no such acknowledgment.
- **Where:** BurnieCoin.sol:663
- **Why:** A warden seeing `revert OnlyGame()` inside a modifier named `onlyAdmin` would reasonably suspect incorrect access control -- either the wrong error is used or the modifier checks the wrong address. The coinflip functions demonstrate that error reuse is intentional (via their inline comment), but the inconsistent documentation makes `onlyAdmin` look like a copy-paste bug. A warden comparing the three reuse sites would flag the undocumented instance.
- **Suggestion:** Add inline comment matching the established pattern: `if (msg.sender != ContractAddresses.ADMIN) revert OnlyGame(); // Reusing error for simplicity`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### CMT-044: burnCoin @dev lists "DegenerusGame, game, or affiliate" -- redundant naming

- **What:** The `burnCoin` function's `@dev` NatSpec at line 854 says `Access: DegenerusGame, game, or affiliate.` "DegenerusGame" and "game" are the same contract. The function uses the `onlyTrustedContracts` modifier which checks `ContractAddresses.GAME` and `ContractAddresses.AFFILIATE` -- exactly two callers, not three.
- **Where:** BurnieCoin.sol:854
- **Why:** A warden reading "DegenerusGame, game, or affiliate" could interpret the comma-separated list as three distinct authorized callers, leading them to investigate what "game" is if not DegenerusGame. This creates unnecessary confusion about the access control surface.
- **Suggestion:** Change to: `@dev Access: DegenerusGame or affiliate (onlyTrustedContracts).`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### CMT-045: EVENTS section header claims events track "coinflip results" and "bounty state"

- **What:** The EVENTS section header (line 45) states events are "the primary mechanism for UIs to track coinflip results, quest completions, and bounty state." BurnieCoin.sol's actual events are: Transfer, Approval, DecimatorBurn, TerminalDecimatorBurn, DailyQuestRolled, QuestCompleted, LinkCreditRecorded, VaultEscrowRecorded, VaultAllowanceSpent. None of these track coinflip results or bounty state. The coinflip result events (CoinflipDeposit, CoinflipDayResolved, CoinflipStakeUpdated, etc.) and bounty events (BountyOwed, BountyPaid, BiggestFlipUpdated) all live in BurnieCoinflip.sol.
- **Where:** BurnieCoin.sol:45
- **Why:** A warden looking for coinflip result or bounty events in BurnieCoin.sol -- based on the section header's promise -- would not find them. They might suspect the events were removed without the header being updated, or that the events are missing entirely. This is a vestigial reference from when BurnieCoin.sol contained the coinflip logic before the split.
- **Suggestion:** Change line 45 to: `track decimator burns, quest completions, and vault escrow state.` -- reflecting the actual events in this contract.
- **Category:** comment-inaccuracy
- **Severity:** INFO

### CMT-049: "QUEST INTEGRATION HELPERS" section contains decimator utility functions

- **What:** The "QUEST INTEGRATION HELPERS" section header (lines 1012-1014) is followed by three private functions: `_adjustDecimatorBucket` (line 1018), `_decimatorBurnMultiplier` (line 1037), and `_questApplyReward` (line 1049). The first two functions are decimator bucket/multiplier utilities with no quest-related logic. Only `_questApplyReward` is a quest helper.
- **Where:** BurnieCoin.sol:1012-1014
- **Why:** A warden looking for decimator calculation logic would search the DECIMATOR section (lines 867-997) and not find the bucket adjustment or burn multiplier functions, because they are filed under "QUEST INTEGRATION HELPERS." Conversely, a warden reading the quest helpers section would encounter unexplained decimator math. The mislabeling adds friction to code navigation.
- **Suggestion:** Rename section to "INTERNAL HELPERS" or split into "DECIMATOR HELPERS" and "QUEST HELPERS" subsections.
- **Category:** comment-inaccuracy
- **Severity:** INFO

### CMT-050: vaultEscrow @dev omits VAULT as an allowed caller

- **What:** The `vaultEscrow` function NatSpec at line 676 says `Called by game contract and modules to credit virtual BURNIE to the vault.` The inline access check (lines 680-683) allows both `ContractAddresses.GAME` and `ContractAddresses.VAULT`. The NatSpec omits VAULT as an explicit caller and says "modules" which, while technically true via delegatecall, does not match the access check pattern.
- **Where:** BurnieCoin.sol:676
- **Why:** A warden auditing the vault escrow flow would check who can increase the mint allowance. The NatSpec says only "game contract and modules" can call, but the code also allows VAULT. A warden might flag the VAULT access as an undocumented privilege escalation. In practice, "modules" are executed via delegatecall as the GAME address, so they are already covered by "game contract" -- making "modules" redundant while "VAULT" is genuinely missing.
- **Suggestion:** Change to: `Called by game contract (including via modules) and vault contract to credit virtual BURNIE to the vault reserve.`
- **Category:** comment-inaccuracy
- **Severity:** INFO
