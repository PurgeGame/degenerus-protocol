## Section 1: lastLootboxRngWord Attack Surface

### Variable Profile

- **What it stores:** The most recently resolved lootbox VRF random word, cached for use as trait entropy during ticket processing.
- **Type:** `uint256` at storage slot 76 (`DegenerusGameStorage.sol:1397`)
- **Who writes it:**
  - W1: `DegenerusGameAdvanceModule.sol:166` -- mid-day `advanceGame` path copies `lootboxRngWordByIndex[lootboxRngIndex - 1]` when `midDayTicketRngPending` is true
  - W2: `DegenerusGameAdvanceModule.sol:789` -- `_finalizeLootboxRng()` stores the raw VRF word on every lootbox finalization (called from `rngGate` at lines 766 and 815)
- **Who reads it:**
  - R1: `DegenerusGameJackpotModule.sol:1975` -- `processTicketBatch` uses it as the base entropy for trait bucket assignment via `keccak256(lastLootboxRngWord, player, queueIdx, ...)`
- **Visibility:** Fully observable on-chain. Any address can read storage slot 76. Once written, the value is publicly known before the next `processTicketBatch` call consumes it.
- **Important note on rawFulfillRandomWords lootbox path:** The VRF callback lootbox branch (`AdvanceModule:1341-1343`) writes directly to `lootboxRngWordByIndex[index]` but does NOT update `lastLootboxRngWord`. The cache is only updated through `_finalizeLootboxRng` (called from `rngGate` during daily processing) or the mid-day path (line 166). This means `lastLootboxRngWord` may lag behind the latest lootbox word by one resolution cycle.

#### Attack Vector: Known-Word Exploitation

**Hypothesis:** An attacker who reads `lastLootboxRngWord` from a prior block can predict which trait bucket their tickets will enter and manipulate ticket purchase timing to gain favorable bucket placement.

**Mechanism:** The attacker reads slot 76 on-chain, computes `keccak256(lastLootboxRngWord, attackerAddress, queueIdx, ...)` for various `queueIdx` values, determines which 256-bucket the result maps to, and times purchases to land in favorable (less populated) buckets.

**Analysis:**

1. `lastLootboxRngWord` is publicly readable on-chain (all storage is public on EVM) -- **attacker CAN know the value**
2. The entropy derivation at `JackpotModule:1975` is `keccak256(lastLootboxRngWord, player, queueIdx, ...)` -- the attacker knows their own address (`player`) and can estimate `queueIdx` from queue state
3. However, the attacker's `queueIdx` depends on when their ticket entry is processed by `processTicketBatch`, which is determined by queue ordering from prior `_queueTickets` calls. Queue position depends on all other purchasers' ordering -- the attacker controls only their own position.
4. The attacker cannot change `lastLootboxRngWord` -- it is written only by VRF-derived values (Chainlink callback). They cannot influence the VRF output.
5. The attacker CAN choose WHEN to purchase (which `advanceGame` cycle), but `lastLootboxRngWord` changes unpredictably with each lootbox RNG resolution. The attacker would need to: (a) observe the current word, (b) purchase tickets, (c) hope no new lootbox RNG fires before their tickets are processed. If a new lootbox RNG fires, the word changes and their prediction is invalid.
6. Even if the attacker correctly predicts their bucket, the jackpot winner is selected by the DAILY VRF word (`rngWordByDay`), not by `lastLootboxRngWord`. Bucket placement affects probability (smaller buckets = higher win chance per entry), but the daily word determines the actual winner.
7. The economic cost of ticket purchases far exceeds the expected value gain from marginally favorable bucket placement, especially given the unpredictability of whether `lastLootboxRngWord` will change before processing.

**Verdict:** BLOCKED / SAFE BY DESIGN
**Evidence:** `DegenerusGameJackpotModule.sol:1975` (entropy derivation), `DegenerusGameAdvanceModule.sol:789` (VRF-only writes), `DegenerusGameAdvanceModule.sol:166` (VRF-derived copy). Winner selection uses independent daily VRF word (`rngWordByDay`), not `lastLootboxRngWord`. Bucket manipulation requires predicting both the entropy value AND queue ordering of all other participants.

#### Attack Vector: Stale-Word Recycling

**Hypothesis:** The Phase 12 data flow shows stale daily VRF words are routed to lootbox via `_finalizeLootboxRng`. If `lastLootboxRngWord` creates a secondary recycling path, an attacker could exploit deterministic entropy reuse.

**Mechanism:** The attacker observes a stale daily VRF word being recycled to lootbox, reads the resulting `lastLootboxRngWord`, and exploits the fact that this value is derived from an already-known daily word.

**Analysis:**

1. Stale word routing occurs in `rngGate` at `AdvanceModule:754-759`: when `requestDay < day`, the stale word is passed to `_finalizeLootboxRng(currentWord)` at line 757. This writes `lastLootboxRngWord = rngWord` at line 789.
2. The stale word was originally a Chainlink VRF word -- it is NOT attacker-controllable. The word is "stale" only in the sense that it arrived for a previous day, but it is still VRF-derived entropy.
3. After `_finalizeLootboxRng` stores the stale word, `rngWordCurrent` is zeroed (line 758) and a fresh `_requestRng` is made (line 759). The fresh VRF callback will overwrite `rngWordCurrent` with new entropy.
4. `lastLootboxRngWord` retains the stale word until the next lootbox resolution. During this window, `processTicketBatch` would use the stale word for trait entropy. However, the stale word is still VRF-derived and was NOT known to any party before the VRF callback delivered it.
5. The concern would be if the same VRF word appeared as BOTH the daily word (for winner selection) AND `lastLootboxRngWord` (for bucket assignment). In the stale routing path, the word goes to lootbox only and `rngWordByDay[day]` gets a DIFFERENT word from the fresh VRF request (line 759 triggers new request, line 763 stores the fresh word later). The two entropy sources diverge.
6. No secondary recycling path exists -- `lastLootboxRngWord` is a terminal consumer (read-only by `processTicketBatch`). It does not feed back into any VRF request or daily word path.

**Verdict:** SAFE BY DESIGN
**Evidence:** `DegenerusGameAdvanceModule.sol:754-759` (stale routing zeroes `rngWordCurrent` and requests fresh VRF), `DegenerusGameAdvanceModule.sol:789` (stale word stored in lootbox cache only), `DegenerusGameAdvanceModule.sol:763` (fresh word recorded separately to `rngWordByDay`). No recycling loop exists -- `lastLootboxRngWord` is a leaf node in the data flow graph.

---

### Section 1 Verdict

**lastLootboxRngWord: SAFE**

Both attack vectors are blocked. The variable stores VRF-derived entropy that cannot be attacker-influenced. Its use as trait entropy for bucket assignment is independent of the daily VRF word used for winner selection. The stale-word recycling path is unidirectional (no feedback loop) and produces new entropy via fresh VRF requests for the daily word.

## Section 2: midDayTicketRngPending Attack Surface

### Variable Profile

- **What it stores:** Boolean flag indicating a mid-day lootbox RNG request triggered a ticket buffer swap, and the new read buffer has unprocessed tickets awaiting drain by `advanceGame`.
- **Type:** `bool` at storage slot 77 (`DegenerusGameStorage.sol:1401`)
- **Who writes it:**
  - W1 (set true): `DegenerusGameAdvanceModule.sol:715` -- inside `requestLootboxRng`, after `_swapTicketSlot` is called at line 714, when write queue had tickets and read queue was already drained
  - W2 (set false): `DegenerusGameAdvanceModule.sol:183` -- inside `advanceGame`, after mid-day ticket drain completes
- **Who reads it:**
  - R1: `DegenerusGameAdvanceModule.sol:162` -- `advanceGame` checks if mid-day drain path should execute
  - R2: `DegenerusGameAdvanceModule.sol:182` -- `advanceGame` checks if flag should be cleared after drain
- **Scope:** Entirely internal to `DegenerusGameAdvanceModule.sol` -- no external reads

#### Attack Vector: State Desync

**Hypothesis:** An attacker manipulates transaction timing to create a state where `midDayTicketRngPending` is true but no VRF request is actually in-flight, or where the VRF callback arrives in an unexpected flag state.

**Mechanism:** The attacker calls `requestLootboxRng` to set the flag, then somehow causes the VRF callback to fail or be redirected, leaving the flag stuck in the `true` state with no pending VRF word.

**Analysis:**

1. `midDayTicketRngPending` is set to `true` at `AdvanceModule:715`, AFTER `_swapTicketSlot` (line 714). At this point, the VRF request has NOT yet been made -- it happens at lines 720-729 (the `vrfCoordinator.requestRandomWords` call).
2. If the VRF request at line 720 reverts, the entire `requestLootboxRng` transaction reverts, including the flag set at line 715. **State desync from failed VRF request is impossible** -- it's atomic.
3. If VRF callback never arrives (VRF failure), `midDayTicketRngPending` stays true. The next `advanceGame` call checks `midDayTicketRngPending` at line 162, enters the mid-day path, and reads `lootboxRngWordByIndex[lootboxRngIndex - 1]` at line 163. If the VRF word is still 0 (unfulfilled), it reverts with `NotTimeYet()` at line 164. This is a liveness issue (blocks ticket processing until VRF arrives or admin intervenes) but NOT an exploitable attack -- the revert prevents any bad state.
4. If VRF callback arrives normally, it writes to `lootboxRngWordByIndex[index]` at `AdvanceModule:1342` (lootbox path, since `rngLockedFlag` is false during mid-day requests). The word becomes available for the check at line 163.
5. The flag and the VRF word are on orthogonal write paths: the flag is written by `requestLootboxRng`, the word is written by `rawFulfillRandomWords`. They cannot interfere with each other because they are in separate transactions with separate senders.

**Verdict:** BLOCKED
**Evidence:** `DegenerusGameAdvanceModule.sol:715` (flag set), `DegenerusGameAdvanceModule.sol:720-729` (VRF request in same tx -- atomic), `DegenerusGameAdvanceModule.sol:163-164` (revert guard on missing word). State desync is prevented by atomicity of `requestLootboxRng` and the `NotTimeYet()` revert guard.

5. `requestLootboxRng` cannot fire again while `midDayTicketRngPending` is true because `ticketsFullyProcessed` is false (reset by `_swapTicketSlot` at `Storage:736`), and the condition at line 713 (`ticketsFullyProcessed`) blocks re-entry into the swap logic.    At `AdvanceModule:730-735`: Inside `_requestRng`, it sets `vrfRequestId`, `rngWordCurrent = 0`, `rngRequestTime`, `rngLockedFlag = true`. The gate sequence is: flag set (715) -> VRF requested (720) -> VRF callback writes word (1342) -> advanceGame reads word (163) -> updates lastLootboxRngWord (166) -> processes tickets -> sets ticketsFullyProcessed (178) -> clears flag (183). Each step requires the previous step to complete. No bypass exists.

**Evidence:** `DegenerusGameAdvanceModule.sol:163-164` (revert on missing VRF word), `DegenerusGameAdvanceModule.sol:166` (fresh entropy update before processing), `DegenerusGameAdvanceModule.sol:178` (drain completion gate), `DegenerusGameAdvanceModule.sol:183` (flag clear only after drain). Sequential gate enforcement prevents bypass.

#### Attack Vector: Interaction with rngLockedFlag

**Hypothesis:** `rngLockedFlag=false` but `midDayTicketRngPending=true` creates a contradictory state that enables an exploit -- e.g., allowing operations that should be blocked during pending ticket processing.

**Mechanism:** After `requestLootboxRng` fires (sets `midDayTicketRngPending=true`), `rngLockedFlag` remains false (lootbox RNG does NOT set `rngLockedFlag` -- only daily RNG via `_requestRng` sets it at line 1208). The attacker exploits operations that check `rngLockedFlag` but not `midDayTicketRngPending`.

**Analysis:**

1. `requestLootboxRng` guards against execution when `rngLockedFlag` is true (line 674: `if (rngLockedFlag) revert RngLocked()`). This ensures mid-day lootbox RNG ONLY fires when daily RNG is NOT in progress.
2. After `requestLootboxRng` succeeds, `rngLockedFlag` is false and `midDayTicketRngPending` is true. This is the INTENDED state -- it means "daily RNG is done for today, but we have a mid-day ticket drain pending."
3. In this state, operations gated by `rngLockedFlag` remain unblocked:
   - `reverseFlip` (line 1299): Can execute. This is correct -- nudges affect the NEXT day's daily VRF word, not the current lootbox resolution.
   - Deity pass purchase (WhaleModule:468): Can execute. Correct -- no daily RNG conflict.
   - Lootbox opening (LootboxModule, lock removed in v1.0): Can execute. Correct -- uses independent `lootboxRngWordByIndex` per player.
   - New `requestLootboxRng` call: Blocked by `rngRequestTime != 0` (line 684), NOT by `rngLockedFlag`. Correct -- prevents concurrent VRF requests.
4. Operations gated by `prizePoolFrozen` ARE affected: `_swapAndFreeze` at line 714 sets `prizePoolFrozen = true` (via `_swapAndFreeze` which calls `Storage:745`). So during `midDayTicketRngPending=true`, pools are frozen and revenue routes to pending accumulators. This is the intended behavior.
5. The two flags serve different purposes: `rngLockedFlag` gates daily VRF lifecycle (set by `_requestRng`, cleared by `_unlockRng`). `midDayTicketRngPending` gates mid-day ticket drain (set by `requestLootboxRng`, cleared by `advanceGame`). They are orthogonal by design.
6. The only potentially dangerous interaction: if a new day boundary is crossed while `midDayTicketRngPending=true`. In this case, `advanceGame` would be called for a new day (`day != dailyIdx`), but the code at line 158 checks `day == dailyIdx` for the mid-day path. A new-day call would take the normal daily path (line 158 is false), which requests daily VRF via `_requestRng` and sets `rngLockedFlag=true`. The `midDayTicketRngPending` flag would be handled on the next same-day `advanceGame` call. However, the mid-day path at line 162 checks `midDayTicketRngPending` only WITHIN the `day == dailyIdx` block -- on a new day, the flag is simply ignored until a same-day call occurs. This means the pending drain carries over to the next day, which is safe because the drain operates on the read buffer independently of daily RNG.

**Verdict:** SAFE BY DESIGN
**Evidence:** `DegenerusGameAdvanceModule.sol:674` (requestLootboxRng requires rngLockedFlag=false), `DegenerusGameAdvanceModule.sol:1208` (only _requestRng sets rngLockedFlag), `DegenerusGameAdvanceModule.sol:158` (mid-day path scoped to same-day calls), `DegenerusGameStorage.sol:745` (_swapAndFreeze freezes pools during pending drain). The two flags are orthogonal by design and serve non-overlapping purposes.

---

### Section 2 Verdict

**midDayTicketRngPending: SAFE**

All three attack vectors are blocked. The flag is a single-producer (`requestLootboxRng`) / single-consumer (`advanceGame`) coordination mechanism with atomic set semantics, a `NotTimeYet()` revert guard preventing premature processing, and orthogonal lifecycle to `rngLockedFlag`. No state desync, gate bypass, or cross-flag contradiction is possible.

## Section 3: Coinflip Lock Changes Attack Surface

### Mechanism Profile

The v1.0 changes removed `rngLockedFlag` guards from 6 purchase paths (LOCK-01 through LOCK-06, with LOCK-06 re-added per the impact assessment). The coinflip system (`BurnieCoinflip.sol`) has its own lock mechanism:

- **BurnieCoinflip reads game state:** `_coinflipLockedDuringTransition()` at `BurnieCoinflip.sol:1032-1045` calls `degenerusGame.purchaseInfo()` which returns `rngLocked_` (from `rngLockedFlag`), `inJackpotPhase`, `lastPurchaseDay_`, and `purchaseLevel_`.
- **Coinflip lock condition:** `(!inJackpotPhase) && !gameOver && lastPurchaseDay_ && rngLocked_ && (purchaseLevel_ % 10 == 0)`. This blocks coinflip deposits ONLY during BAF resolution levels (every 10th level) when: not in jackpot phase, on last purchase day, AND rngLockedFlag is true.
- **`_coinflipLockedDuringTransition` is separate from `rngLockedFlag`:** It is a derived condition that combines multiple game state variables. It provides a narrower block than `rngLockedFlag` alone -- only at BAF levels.
- **Coinflip entropy:** `processCoinflipPayouts` at `BurnieCoinflip.sol:807-813` receives the daily VRF word as a parameter from `rngGate` and derives entropy via `keccak256(rngWord, epoch)`. Coinflip does NOT use `lastLootboxRngWord` or any lootbox entropy.

#### Attack Vector: Coinflip During Daily RNG

**Hypothesis:** With lock removals from purchase paths, a player can execute a coinflip while daily RNG is in-flight, potentially exploiting the VRF word that will determine their coinflip outcome.

**Mechanism:** Daily `advanceGame` triggers `_requestRng` (sets `rngLockedFlag=true`). Player observes the mempool for VRF callback, reads the random word, and deposits to coinflip before the callback is included, hoping to influence their payout.

**Analysis:**

1. Coinflip DEPOSITS are gated by `_coinflipLockedDuringTransition` at `BurnieCoinflip.sol:258`. This gate checks `rngLocked_` (among other conditions). When `rngLockedFlag=true` AND at a BAF resolution level AND `lastPurchaseDay`, deposits are blocked.
2. Coinflip PAYOUTS are processed by `processCoinflipPayouts` which is called from `rngGate` at `AdvanceModule:765/809/826` -- this happens INSIDE `advanceGame`, not as a separate user-callable function. The payout function is `onlyDegenerusGameContract` (`BurnieCoinflip.sol:811`).
3. Coinflip payout entropy is derived from `keccak256(rngWord, epoch)` where `rngWord` is the daily VRF word passed from `rngGate`. The player does NOT choose when payouts are processed -- they are batch-processed during `advanceGame`.
4. A player depositing BURNIE during daily RNG in-flight does NOT influence their payout. The payout is determined by the daily VRF word at the time `advanceGame` processes it, which is a FUTURE event. The deposit only affects how much BURNIE is at risk.
5. The lock removals (LOCK-01 through LOCK-05) affect ETH/ticket purchase paths, NOT the coinflip deposit path. The coinflip deposit path has its own independent `_coinflipLockedDuringTransition` guard.
6. Even if a player could front-run the VRF callback, they cannot change their coinflip deposit retroactively -- deposits are burned immediately (`burnie.burnForCoinflip` at `BurnieCoinflip.sol:272`). The payout percentage is determined later by `processCoinflipPayouts`.

**Verdict:** BLOCKED
**Evidence:** `BurnieCoinflip.sol:258` (deposit lock at BAF levels), `BurnieCoinflip.sol:811` (`onlyDegenerusGameContract` on payout processing), `BurnieCoinflip.sol:813` (entropy from daily VRF word parameter, not player-controllable), `DegenerusGameAdvanceModule.sol:765` (payout processed inside advanceGame, not user-callable).

#### Attack Vector: Coinflip Timing Around Lootbox RNG

**Hypothesis:** A player times coinflip execution around lootbox VRF callback to gain advantage -- either by knowing the lootbox word or by exploiting the interaction between lootbox and coinflip entropy.

**Mechanism:** Player observes `requestLootboxRng` transaction, deposits to coinflip, then the lootbox VRF callback arrives. The player expects the lootbox word to influence their coinflip outcome.

**Analysis:**

1. Coinflip payouts use DAILY VRF entropy (`rngWord` from `rngGate`), NOT lootbox entropy. The `processCoinflipPayouts` function at `BurnieCoinflip.sol:807-813` receives `rngWord` and `epoch` as parameters -- these come from `rngGate`'s daily word processing.
2. `requestLootboxRng` occurs mid-day (after daily VRF has been consumed). The daily word is already recorded in `rngWordByDay[day]` before any lootbox request can fire (guard at `AdvanceModule:682`: `if (rngWordByDay[currentDay] == 0) revert E()`).
3. Coinflip payouts for the current day are ALREADY processed by the time `requestLootboxRng` can fire -- `processCoinflipPayouts` is called during `rngGate` (which runs during `advanceGame`), and `requestLootboxRng` requires `rngWordByDay[currentDay] != 0` (meaning `advanceGame` already ran for today).
4. The lootbox VRF callback writes to `lootboxRngWordByIndex` and `lastLootboxRngWord` -- neither of which feeds into coinflip payout calculation.
5. No timing attack is possible because the coinflip and lootbox entropy paths are fully independent.

**Verdict:** SAFE BY DESIGN
**Evidence:** `BurnieCoinflip.sol:813` (coinflip uses daily VRF word, not lootbox word), `DegenerusGameAdvanceModule.sol:682` (lootbox RNG requires daily word already recorded), `DegenerusGameAdvanceModule.sol:765` (coinflip payouts processed during daily rngGate). Complete entropy independence between coinflip and lootbox paths.

#### Attack Vector: Lock Gap Between Jackpot Days

**Hypothesis:** During the 5-day jackpot phase, `rngLockedFlag` clears between daily `advanceGame()` calls but `prizePoolFrozen` stays true. Coinflip operations during this gap could affect jackpot outcomes.

**Mechanism:** Between jackpot day 1 and day 2, `rngLockedFlag` is false (cleared by `_unlockRng` at the end of day 1 processing) but `prizePoolFrozen` is true (stays frozen throughout jackpot phase). A player deposits to coinflip during this gap, potentially altering the pool state or jackpot composition.

**Analysis:**

1. `_coinflipLockedDuringTransition` at `BurnieCoinflip.sol:1044` checks `(!inJackpotPhase)` as the FIRST condition. During jackpot phase, `inJackpotPhase=true`, so `(!inJackpotPhase)=false`, and the entire expression is false. **Coinflip deposits are NOT locked during jackpot phase** -- the lock only applies at BAF resolution levels during purchase phase.
2. This means coinflip deposits ARE allowed during the jackpot phase gap. Is this a problem?
3. Coinflip deposits burn BURNIE tokens (`burnForCoinflip` at `BurnieCoinflip.sol:272`) and record deposit state in `BurnieCoinflip` contract storage. They do NOT write to `DegenerusGame` storage directly.
4. Coinflip deposits trigger auto-claim (`_claimCoinflipsInternal` at `BurnieCoinflip.sol:261`) which may mint BURNIE. Auto-claim reads from BurnieCoinflip's own storage and calls `burnie.mint()` -- this does not touch prize pools, ticket queues, or any RNG state.
5. `prizePoolFrozen=true` during jackpot phase means any ETH flowing through purchase paths goes to pending accumulators. Coinflip deposits are BURNIE-denominated (no ETH involved in the deposit), so pool freeze is irrelevant to coinflip.
6. Coinflip payouts for jackpot-phase days are processed by `processCoinflipPayouts` during `_rngGateJackpot` at `AdvanceModule:809/826`. These use the daily VRF word for that jackpot day. Deposits made during the gap do NOT affect the payout entropy or the jackpot draw outcomes.
7. The BAF leaderboard concern (mentioned in `_coinflipLockedDuringTransition` comments at `BurnieCoinflip.sol:1028-1031`) is specific to BAF resolution levels. BAF does not fire during jackpot phase -- it fires at level transitions (every 10th level during purchase phase). So the leaderboard front-running concern does not apply during jackpot phase.

**Verdict:** SAFE BY DESIGN
**Evidence:** `BurnieCoinflip.sol:1044` (`!inJackpotPhase` means lock is inactive during jackpot phase), `BurnieCoinflip.sol:272` (deposits are BURNIE burns, no ETH/pool interaction), `BurnieCoinflip.sol:261` (auto-claim touches only BurnieCoinflip storage), `DegenerusGameAdvanceModule.sol:809` (jackpot-day coinflip payouts use independent daily VRF word). Coinflip deposits during jackpot phase gap are harmless -- they touch only BURNIE token supply and BurnieCoinflip internal state.

---

### Section 3 Verdict

**Coinflip lock changes: SAFE**

All three attack vectors are blocked or safe by design. Coinflip deposits have their own independent gate (`_coinflipLockedDuringTransition`) that is narrower but correctly scoped to BAF resolution levels. Coinflip entropy is fully independent of lootbox entropy. Deposits during jackpot phase gaps are harmless because they involve only BURNIE burns and do not interact with prize pools, ticket queues, or RNG state.

## Section 4: Cross-Variable Interaction Analysis

### lastLootboxRngWord + midDayTicketRngPending Combined Attack

**Question:** Can the combination of `lastLootboxRngWord` + `midDayTicketRngPending` create a combined attack not possible with either alone?

**Analysis:** These two variables are tightly coupled by design -- `midDayTicketRngPending` is the coordination flag that ensures `lastLootboxRngWord` is updated with fresh VRF entropy before `processTicketBatch` consumes it. The interaction is:

1. `requestLootboxRng` sets `midDayTicketRngPending=true` (line 715)
2. VRF callback writes `lootboxRngWordByIndex[index]` (line 1342)
3. `advanceGame` reads the word (line 163), updates `lastLootboxRngWord` (line 166), then processes tickets
4. Tickets use `lastLootboxRngWord` for trait entropy (JackpotModule:1975)

A combined attack would require manipulating BOTH variables simultaneously. But `midDayTicketRngPending` is only writable by `requestLootboxRng` (which requires passing multiple guards including threshold checks and `rngRequestTime == 0`), and `lastLootboxRngWord` is only writable by VRF callback results. No user transaction can directly set either variable.

The only theoretical combined concern: if `midDayTicketRngPending` is true but `lastLootboxRngWord` holds a stale word from a previous resolution. The code at line 163-166 prevents this by reading the FRESH word from `lootboxRngWordByIndex` and updating `lastLootboxRngWord` BEFORE processing.

**Verdict:** SAFE -- the sequential gate (flag check -> word read -> entropy update -> ticket processing) prevents any combined exploitation.

### Coinflip Lock Changes + New Variables Interaction

**Question:** Do coinflip lock changes interact with `lastLootboxRngWord` or `midDayTicketRngPending`?

**Analysis:** No interaction exists. The coinflip system (`BurnieCoinflip.sol`) is an external contract that:
- Reads game state via `purchaseInfo()` view function (returns `rngLockedFlag`, `inJackpotPhase`, etc.)
- Does NOT read `lastLootboxRngWord` or `midDayTicketRngPending`
- Does NOT write to any DegenerusGame storage
- Uses its own BURNIE-denominated deposit/payout system
- Receives daily VRF entropy as a parameter from `advanceGame` (via `processCoinflipPayouts`)

The lock changes (removing `rngLockedFlag` from purchase paths) do not affect coinflip behavior because coinflip has its own independent lock (`_coinflipLockedDuringTransition`). The new variables are internal to the mid-day ticket processing system, which operates independently of coinflip.

**Verdict:** SAFE -- no interaction path exists between coinflip and the two new variables.
