## GO-07: Deity Pass Refunds on Early GAMEOVER -- PASS

**Verdict:** PASS
**Files:** `GameOverModule.sol:68-107`, `DegenerusGameStorage.sol:1025-1034`, `DegenerusGame.sol:1431-1447`
**Summary:** Deity pass refunds on early GAMEOVER (levels 0-9) are correctly implemented with FIFO ordering, budget cap, safe unchecked arithmetic, and full claimability via the normal claimWinnings path.

### Detailed Trace

#### 1. Eligibility Gate (GameOverModule.sol:78)

```solidity
if (currentLevel < 10) {
```

The refund block only executes when `currentLevel < 10` (levels 0 through 9). At level 10 exactly, `currentLevel < 10` is `false`, so no refunds are issued. This is correct -- deity refunds are designed as early-termination compensation only.

**Verdict:** Correct. Level 10+ receives no deity refund.

#### 2. Refund Amount (GameOverModule.sol:38-39, 79, 87)

```solidity
uint256 private constant DEITY_PASS_EARLY_GAMEOVER_REFUND = 20 ether;  // line 38-39
// ...
uint256 refundPerPass = DEITY_PASS_EARLY_GAMEOVER_REFUND;              // line 79
uint256 refund = refundPerPass * uint256(purchasedCount);              // line 87
```

Fixed 20 ETH per deity pass. Cross-reference against `audit/v1.1-parameter-reference.md` line 274: `DEITY_PASS_EARLY_GAMEOVER_REFUND | 20 ether | 20 ETH`. The constant matches the documented value.

The refund is multiplied by `purchasedCount` (GameOverModule.sol:85), which uses `deityPassPurchasedCount[owner]` (Storage.sol:1028). This is the count of passes the owner has purchased (not transferred), which is a `uint16`. Maximum possible refund per owner: `20 ETH * 65535 = 1,310,700 ETH`. In practice, each address can hold at most 1 deity pass (`deityPassCount[buyer] != 0` reverts on purchase -- see WhaleModule), so `purchasedCount` is effectively 1 per address. However, a transferred-away deity pass does not decrement `purchasedCount`, so an address that bought and transferred could have `purchasedCount > deityPassCount`. This means refunds go to original purchasers by design, not current holders.

**Verdict:** Correct. 20 ETH per purchased pass, matching parameter reference.

#### 3. FIFO Ordering (GameOverModule.sol:80, 83-84)

```solidity
uint256 ownerCount = deityPassOwners.length;     // line 80
for (uint256 i; i < ownerCount; ) {              // line 83
    address owner = deityPassOwners[i];           // line 84
```

Refunds iterate through `deityPassOwners` array (Storage.sol:1034) in index order. The `deityPassOwners` array is an append-only array: new buyers are pushed via `deityPassOwners.push(buyer)` in the WhaleModule during deity pass purchase. Therefore, the iteration order is FIFO by purchase order. Earlier purchasers get refunded first, and the budget-cap mechanism (below) means later purchasers may receive reduced or zero refunds if funds run out.

**Verdict:** Correct. True FIFO by purchase order via array append semantics.

#### 4. Budget Cap (GameOverModule.sol:81, 88-89, 98)

```solidity
uint256 budget = totalFunds > claimablePool ? totalFunds - claimablePool : 0;  // line 81
// ...
if (refund > budget) {                                                         // line 88
    refund = budget;                                                           // line 89
}
// ...
if (budget == 0) break;                                                        // line 98
```

**Underflow protection:** Line 81 uses a ternary check: if `claimablePool > totalFunds`, budget is set to 0 rather than underflowing. This is safe.

**Budget limiting:** Each refund is capped to the remaining budget (line 88-89). If the budget is insufficient for a full refund, the owner receives whatever remains. Once budget reaches 0, the loop breaks (line 98).

**Invariant:** `totalRefunded <= budget <= totalFunds - claimablePool` (before the claimablePool update at line 105). After the update: `claimablePool_new = claimablePool_old + totalRefunded <= claimablePool_old + (totalFunds - claimablePool_old) = totalFunds`. So `claimablePool <= totalFunds` is preserved.

**Verdict:** Correct. Budget cap prevents deity refunds from consuming funds owed to existing claimable winners.

#### 5. claimablePool Mutation (GameOverModule.sol:104-106)

```solidity
if (totalRefunded != 0) {
    claimablePool += totalRefunded;
}
```

This is a checked addition (outside the unchecked block). Since we proved `totalRefunded <= totalFunds - claimablePool`, the sum `claimablePool + totalRefunded <= totalFunds`, which is well within uint256 range. No overflow possible.

**Verdict:** Correct. claimablePool invariant preserved.

#### 6. Unchecked Arithmetic -- Research Open Question 3 (GameOverModule.sol:92-96)

```solidity
unchecked {
    claimableWinnings[owner] += refund;    // line 93
    totalRefunded += refund;               // line 94
    budget -= refund;                      // line 95
}
```

**`claimableWinnings[owner] += refund`:** Can this overflow uint256? The maximum theoretical `claimableWinnings[owner]` before this addition would need to be close to `type(uint256).max`. Since `claimableWinnings` is denominated in wei and the total ETH supply is ~120M ETH = ~1.2e26 wei, and `type(uint256).max` = ~1.15e77, overflow requires ~5.7e57 ETH already claimable. This is physically impossible. **Safe.**

**`totalRefunded += refund`:** `totalRefunded` starts at 0 and accumulates individual refunds. Each refund is at most `budget` (line 88-89), and `budget` starts at `totalFunds - claimablePool` (at most `totalFunds`). Sum of all refunds cannot exceed `budget_initial` since `budget -= refund` is applied in each iteration and the loop breaks when `budget == 0`. So `totalRefunded <= budget_initial <= totalFunds`, well within uint256. **Safe.**

**`budget -= refund`:** The check at line 88 ensures `refund <= budget` before this line executes. Therefore `budget - refund >= 0`. No underflow. **Safe.**

**Resolution of Research Open Question 3:** The unchecked block at GameOverModule.sol:92-96 is safe. All three operations are bounded by physical constraints (total ETH supply for claimableWinnings), loop invariants (totalRefunded bounded by initial budget), and explicit guards (refund capped to budget before subtraction).

#### 7. Refund Claimability (GameOverModule.sol:93, DegenerusGame.sol:1431-1447)

Deity refunds are credited via `claimableWinnings[owner] += refund` (GameOverModule.sol:93). These are claimable through the normal `claimWinnings` path:

```solidity
// DegenerusGame.sol:1431-1447
function _claimWinningsInternal(address player, bool stethFirst) private {
    if (finalSwept) revert E();                      // line 1432
    uint256 amount = claimableWinnings[player];       // line 1433
    if (amount <= 1) revert E();                     // line 1434
    uint256 payout;
    unchecked {
        claimableWinnings[player] = 1;               // line 1437 (sentinel)
        payout = amount - 1;                         // line 1438
    }
    claimablePool -= payout;                         // line 1440 (CEI)
    emit WinningsClaimed(player, msg.sender, payout);
    // ... external payout call
}
```

**Is there any path where deity refunds become unclaimable?**

1. **Before finalSwept:** `claimWinnings` works normally. Players can call `claimWinnings(player)` to withdraw deity refunds during the 30-day window after GAMEOVER.

2. **After finalSwept:** `_claimWinningsInternal` reverts at line 1432 (`if (finalSwept) revert E()`). At this point, `claimablePool` has been zeroed by `handleFinalSweep`, and remaining funds have been sent to vault/sDGNRS. Deity refund holders who did not claim within 30 days lose their refund. This is by design -- the 30-day claim window applies to ALL claimable winnings, including deity refunds.

3. **Sentinel value:** The sentinel pattern (`claimableWinnings[player] = 1`) means a deity refund of exactly 1 wei would result in `amount = 1`, which fails the `amount <= 1` check, making 1 wei effectively unclaimable. However, deity refunds are `20 ether * purchasedCount`, which is always a large value. Not a practical concern.

**Verdict:** Correct. Deity refunds are fully claimable via `claimWinnings` during the 30-day post-GAMEOVER window. Forfeiture after 30 days is intentional and applies uniformly to all claimable winnings.

## GO-02: handleFinalSweep -- 30-Day Claim Window -- PASS

**Verdict:** PASS
**Files:** `GameOverModule.sol:171-189`, `DegenerusGame.sol:1431-1447`
**Summary:** The 30-day claim window is correctly enforced, the finalSwept latch prevents double execution, claimablePool zeroing is intentional forfeiture of unclaimed funds, and remaining balance distribution goes 50/50 to vault/sDGNRS via `_sendToVault`.

### Detailed Trace

#### 1. 30-Day Claim Window Guard (GameOverModule.sol:172-173)

```solidity
if (gameOverTime == 0) return;                                  // line 172
if (block.timestamp < uint256(gameOverTime) + 30 days) return;  // line 173
```

**Line 172:** If `gameOverTime` is 0, the game has not entered GAMEOVER yet (`gameOverTime` is set to `block.timestamp` at GameOverModule.sol:113 inside `handleGameOverDrain`). Silent return -- correct.

**Line 173:** The 30-day window check. `gameOverTime` is a `uint48`, cast to `uint256` for the addition. `30 days` = 2,592,000 seconds. The comparison ensures `block.timestamp >= gameOverTime + 30 days` before proceeding. This gives players exactly 30 days from the moment `handleGameOverDrain` sets `gameOverTime` to claim their winnings.

**Overflow consideration:** `uint48` max = ~281 trillion seconds (~8.9 million years). Adding 30 days (2.6M seconds) cannot overflow `uint256`. Safe.

**Verdict:** Correct. 30-day window enforced with the expected constant.

#### 2. finalSwept Latch (GameOverModule.sol:174, 176)

```solidity
if (finalSwept) return;     // line 174 -- guard
// ...
finalSwept = true;           // line 176 -- set BEFORE external calls
```

The `finalSwept` boolean (Storage.sol:809) is checked at the top of the function and set to `true` before any external calls (`admin.shutdownVrf()` at line 180, balance reads at lines 182-183, `_sendToVault` at line 188). This follows the Checks-Effects-Interactions (CEI) pattern correctly.

**Double-execution prevention:** Once `finalSwept = true`, any subsequent call returns immediately at line 174. The latch is set BEFORE external calls, so even if a reentrant call occurred, it would be rejected.

**Verdict:** Correct. Double-sweep impossible. CEI ordering maintained.

#### 3. claimablePool Zeroing (GameOverModule.sol:177)

```solidity
claimablePool = 0;           // line 177
```

This is set BEFORE the balance check at lines 182-184 and before `_sendToVault` at line 188. The zeroing means:

1. Any subsequent call to `claimWinnings` will fail because `claimablePool -= payout` at DegenerusGame.sol:1440 would underflow (since `claimablePool` is 0 and `payout > 0`).
2. Actually, `claimWinnings` is also blocked by the `finalSwept` check at DegenerusGame.sol:1432.

The zeroing is a belt-and-suspenders approach: even if someone bypassed the `finalSwept` check, the zeroed `claimablePool` would prevent fund extraction.

**Is this intentional?** Yes. The natspec at line 167 says "Forfeits all unclaimed winnings and sweeps entire balance." Players had 30 days to call `claimWinnings`. After that, unclaimed funds are forfeited by design.

**Verdict:** Correct. claimablePool zeroing is intentional forfeiture. Players had the full 30-day window.

#### 4. Remaining Balance Distribution (GameOverModule.sol:182-188)

```solidity
uint256 ethBal = address(this).balance;          // line 182
uint256 stBal = steth.balanceOf(address(this));  // line 183
uint256 totalFunds = ethBal + stBal;             // line 184

if (totalFunds == 0) return;                     // line 186

_sendToVault(totalFunds, stBal);                 // line 188
```

After `claimablePool = 0`, all remaining funds in the contract are treated as forfeited. `_sendToVault` (GameOverModule.sol:195-232) splits funds 50/50 between the vault and sDGNRS:

```solidity
uint256 dgnrsAmount = amount / 2;                // 50% to sDGNRS
uint256 vaultAmount = amount - dgnrsAmount;      // 50% to vault (gets the rounding wei)
```

The `_sendToVault` function prioritizes stETH transfers over ETH transfers. For each recipient (vault first, then sDGNRS), it sends stETH up to the available balance, then sends remaining ETH. Both paths revert on failure (`revert E()`).

**Forfeited funds destination:** All forfeited funds go to vault (50%) and sDGNRS (50%). No attacker can intercept these funds -- the vault and sDGNRS addresses are hardcoded constants in ContractAddresses.sol.

**Verdict:** Correct. Forfeited funds go 50/50 to protocol-controlled addresses.

#### 5. admin.shutdownVrf() Try/Catch (GameOverModule.sol:180)

```solidity
try admin.shutdownVrf() {} catch {}             // line 180
```

The VRF shutdown is wrapped in a try/catch that silently absorbs any failure. This is correct -- a VRF shutdown failure must not block the final sweep. The empty catch block means any revert, out-of-gas, or other error in `shutdownVrf()` is ignored, and execution continues to the balance sweep.

**Verdict:** Correct. Fire-and-forget pattern for non-critical cleanup.

#### 6. Unclaimed Forfeiture Logic

**Players had the full 30-day window to call claimWinnings:**

- During the 30-day window (between `gameOverTime` and `gameOverTime + 30 days`), `handleFinalSweep` returns early at line 173.
- The `finalSwept` flag is still `false`, so `claimWinnings` works normally (DegenerusGame.sol:1432 does not revert).
- `claimablePool` still holds the correct liability, so `claimablePool -= payout` at DegenerusGame.sol:1440 succeeds for valid claims.

**No path exists where claimable funds become unclaimable BEFORE the 30-day window expires:**

- `gameOver = true` is set in `handleGameOverDrain` (GameOverModule.sol:112), but `claimWinnings` has no `gameOver` guard. Players can claim during and after GAMEOVER.
- `finalSwept` is only set in `handleFinalSweep`, which cannot execute until 30 days have passed (line 173).
- No other function modifies `finalSwept` or zeroes `claimablePool` prematurely.

**After 30 days:**

- `handleFinalSweep` executes: `finalSwept = true`, `claimablePool = 0`.
- All remaining contract balance goes to vault/sDGNRS.
- Any player who did not claim loses their winnings permanently.

**Verdict:** Correct. The 30-day claim window is complete and uninterruptible. Forfeiture is intentional and well-documented.

## Open Questions Resolved

### Research Q3: Unchecked Arithmetic in Deity Refund Loop

**Status:** RESOLVED -- SAFE

The unchecked block at GameOverModule.sol:92-96 contains three operations:

1. **`claimableWinnings[owner] += refund`** -- Cannot overflow. Would require ~5.7e57 ETH already claimable, exceeding the total ETH supply by ~47 orders of magnitude.

2. **`totalRefunded += refund`** -- Cannot overflow. Bounded by `budget_initial` which is bounded by `totalFunds` (a uint256 representing actual contract balance).

3. **`budget -= refund`** -- Cannot underflow. Line 88 ensures `refund <= budget` before subtraction.

All three operations are provably safe. The unchecked block is a legitimate gas optimization.
