This report covers the four scatter and decimator distribution requirements: PAY-03 (BAF normal scatter), PAY-04 (BAF century scatter), PAY-05 (decimator normal claims), and PAY-06 (decimator x00 claims). Together these govern the level-transition reward mechanisms that fire periodically as the game advances through levels.

**Overall assessment: SOUND.**

- **PAY-03 (BAF Normal Scatter Payout): PASS.** The BAF fires every 10 levels using `baseFuturePool` (snapshot at entry) with correct percentage (10% standard, 20% at level 50). The 7-category prize split across leaderboard, coinflip top, random pick, far-future tickets, and scatter winners matches the v1.1 specification. Large/small winner classification at 5% threshold is correct with proper 50/50 ETH/lootbox split for large winners and alternating 100% ETH/lootbox for small winners. Whale pass queueing via `_queueWhalePassClaimCore` is correct with no expiry on `whalePassClaims`. claimablePool accounting is verified correct through the entire BAF path. CEI ordering is maintained (all operations run inside delegatecall context with no external ETH transfers).
- **PAY-04 (BAF Century Scatter Payout): PASS.** Century BAF (x00 levels) correctly uses 20% of `baseFuturePool`. The different scatter sampling pattern (4 rounds each at lvl+1/+2/+3, then 38 rounds randomly from past 99 levels) matches the v1.1 specification. The century BAF shares the same 7-category distribution mechanics and winner processing as normal BAF, with only the pool percentage (20% vs 10%) and scatter level targeting differing. No separate code path exists for century BAF distribution -- it flows through the same `_runBafJackpot` function.
- **PAY-05 (Decimator Normal Claims): PASS.** The pro-rata formula `(poolWei * playerBurn) / totalBurn` is correct with truncation (Solidity default) favoring the protocol. The 50/50 ETH/lootbox split is verified. `lastDecClaimRound` overwrite expiry is confirmed as by-design per v1.1-transition-jackpots.md Section 8. The `claimed` flag is per-level per-player, preventing double-claims within a round. The `prizePoolFrozen` guard correctly blocks claims during jackpot phase days 1-4. claimablePool accounting traces correctly through `_creditDecJackpotClaimCore` with the lootbox portion correctly deducted from claimablePool before being routed to futurePrizePool.
- **PAY-06 (Decimator x00 Claims): PASS.** The x00 decimator correctly uses 30% of `baseFuturePool` (snapshot before any BAF deduction). It shares the same `runDecimatorJackpot` resolution function and `claimDecimatorJackpot` claim path as normal decimator. The `lastDecClaimRound` struct is shared between normal and x00 decimator -- only one can be active at a time, with the most recent overwriting the previous. At x00 levels where both BAF and decimator fire, the combined draw is up to 50% of `baseFuturePool` (20% BAF + 30% decimator), which is within the design budget.

No findings above INFORMATIONAL severity.

#### 6. CEI Ordering

BAF runs inside `runRewardJackpots()` which is called via delegatecall from DegenerusGame during the `advanceGame` flow. All operations execute within the game contract's context:

- **External call:** `jackpots.runBafJackpot(poolWei, lvl, rngWord)` at EndgameModule.sol:348. This is a regular call to the DegenerusJackpots contract, which has `onlyGame` modifier. The Jackpots contract writes to its own storage (leaderboard clearing, epoch increment) and returns results. It does not call back into the game.
- **State updates after external call:** Winner processing (lines 361-401) and claimablePool update (line 226) happen after the external call. However, since the Jackpots contract cannot re-enter the game (it doesn't hold ETH and doesn't make callbacks), this is safe.
- **No ETH transfers:** All BAF "payouts" are credits to `claimableWinnings` mappings. Actual ETH transfer only happens when the player calls `claimWinnings()`, which has its own CEI guard (sentinel + claimablePool decrement before external call).

**CEI assessment: Correct. No reentrancy vector.**

## Shared Infrastructure Notes

### Lootbox Flow Summary

All lootbox ETH from BAF (both normal and century) stays in `futurePrizePool`:
- `lootboxToFuture = lootboxTotal` at EndgameModule.sol:404.
- Added back to `futurePoolLocal` at EndgameModule.sol:185: `futurePoolLocal += lootboxToFuture`.
- Final `futurePoolLocal` is committed via `_setFuturePrizePool(futurePoolLocal)` at EndgameModule.sol:223.

This means lootbox tickets are "minted" against the future pool without moving ETH out. The ETH backing comes from the pool's existing balance. This is consistent with the over-collateralization model verified in PAY-16 (v3.0-payout-jackpot-distribution.md).

### winnerMask Ticket Routing

The `winnerMask` returned by `jackpots.runBafJackpot()` is used to flag the last 40 scatter winners for special ticket routing treatment. However, the current `_runBafJackpot` implementation in EndgameModule.sol does NOT use `winnerMask` (it is returned by the Jackpots contract but the third return value is discarded at EndgameModule.sol:346 via `,`). This means all scatter winners are processed uniformly through the same large/small classification. This is a minor spec deviation but does not affect correctness or security -- it simplifies the payout logic without economic impact.
