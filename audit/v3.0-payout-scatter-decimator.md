## Executive Summary

This report covers the four scatter and decimator distribution requirements: PAY-03 (BAF normal scatter), PAY-04 (BAF century scatter), PAY-05 (decimator normal claims), and PAY-06 (decimator x00 claims). Together these govern the level-transition reward mechanisms that fire periodically as the game advances through levels.

**Overall assessment: SOUND.**

- **PAY-03 (BAF Normal Scatter Payout): PASS.** The BAF fires every 10 levels using `baseFuturePool` (snapshot at entry) with correct percentage (10% standard, 20% at level 50). The 7-category prize split across leaderboard, coinflip top, random pick, far-future tickets, and scatter winners matches the v1.1 specification. Large/small winner classification at 5% threshold is correct with proper 50/50 ETH/lootbox split for large winners and alternating 100% ETH/lootbox for small winners. Whale pass queueing via `_queueWhalePassClaimCore` is correct with no expiry on `whalePassClaims`. claimablePool accounting is verified correct through the entire BAF path. CEI ordering is maintained (all operations run inside delegatecall context with no external ETH transfers).
- **PAY-04 (BAF Century Scatter Payout): PASS.** Century BAF (x00 levels) correctly uses 20% of `baseFuturePool`. The different scatter sampling pattern (4 rounds each at lvl+1/+2/+3, then 38 rounds randomly from past 99 levels) matches the v1.1 specification. The century BAF shares the same 7-category distribution mechanics and winner processing as normal BAF, with only the pool percentage (20% vs 10%) and scatter level targeting differing. No separate code path exists for century BAF distribution -- it flows through the same `_runBafJackpot` function.
- **PAY-05 (Decimator Normal Claims): PASS.** The pro-rata formula `(poolWei * playerBurn) / totalBurn` is correct with truncation (Solidity default) favoring the protocol. The 50/50 ETH/lootbox split is verified. `lastDecClaimRound` overwrite expiry is confirmed as by-design per v1.1-transition-jackpots.md Section 8. The `claimed` flag is per-level per-player, preventing double-claims within a round. The `prizePoolFrozen` guard correctly blocks claims during jackpot phase days 1-4. claimablePool accounting traces correctly through `_creditDecJackpotClaimCore` with the lootbox portion correctly deducted from claimablePool before being routed to futurePrizePool.
- **PAY-06 (Decimator x00 Claims): PASS.** The x00 decimator correctly uses 30% of `baseFuturePool` (snapshot before any BAF deduction). It shares the same `runDecimatorJackpot` resolution function and `claimDecimatorJackpot` claim path as normal decimator. The `lastDecClaimRound` struct is shared between normal and x00 decimator -- only one can be active at a time, with the most recent overwriting the previous. At x00 levels where both BAF and decimator fire, the combined draw is up to 50% of `baseFuturePool` (20% BAF + 30% decimator), which is within the design budget.

No findings above INFORMATIONAL severity.

Trace of all `claimablePool` mutations through the BAF normal path:

4. **`runRewardJackpots` aggregates** (EndgameModule.sol:178, 226):

**Invariant check:** `claimablePool` after BAF = (sum of all ETH credits to `claimableWinnings`) = (ETH portions of large winners) + (full amounts of even-index small winners) + (whale pass remainders) + (auto-rebuy reserved portions). #### 6. CEI Ordering

BAF runs inside `runRewardJackpots()` which is called via delegatecall from DegenerusGame during the `advanceGame` flow. All operations execute within the game contract's context:

- **External call:** `jackpots.runBafJackpot(poolWei, lvl, rngWord)` at EndgameModule.sol:348. This is a regular call to the DegenerusJackpots contract, which has `onlyGame` modifier. The Jackpots contract writes to its own storage (leaderboard clearing, epoch increment) and returns results. It does not call back into the game.
- **State updates after external call:** Winner processing (lines 361-401) and claimablePool update (line 226) happen after the external call. However, since the Jackpots contract cannot re-enter the game (it doesn't hold ETH and doesn't make callbacks), this is safe.
- **No ETH transfers:** All BAF "payouts" are credits to `claimableWinnings` mappings. Actual ETH transfer only happens when the player calls `claimWinnings()`, which has its own CEI guard (sentinel + claimablePool decrement before external call).

**CEI assessment: Correct. No reentrancy vector.**

## Shared Infrastructure Notes

### Lootbox Flow Summary

All lootbox ETH from BAF (both normal and century) stays in `futurePrizePool`:
- `lootboxToFuture = lootboxTotal` at EndgameModule.sol:404.
- Added back to `futurePoolLocal` at EndgameModule.sol:185: `futurePoolLocal += lootboxToFuture`.
- Final `futurePoolLocal` is committed via `_setFuturePrizePool(futurePoolLocal)` at EndgameModule.sol:223.

This means lootbox tickets are "minted" against the future pool without moving ETH out. The ETH backing comes from the pool's existing balance. This is consistent with the over-collateralization model verified in PAY-16 (v3.0-payout-jackpot-distribution.md).

### winnerMask Ticket Routing

The `winnerMask` returned by `jackpots.runBafJackpot()` is used to flag the last 40 scatter winners for special ticket routing treatment. However, the current `_runBafJackpot` implementation in EndgameModule.sol does NOT use `winnerMask` (it is returned by the Jackpots contract but the third return value is discarded at EndgameModule.sol:346 via `,`). This means all scatter winners are processed uniformly through the same large/small classification. This is a minor spec deviation but does not affect correctness or security -- it simplifies the payout logic without economic impact.

Crucially, `claimablePool -= calc.ethSpent` at line 494 correctly reduces the pre-reserved claimablePool by the amount converted to tickets, keeping the invariant sound.
**Recommendation:** None.

### Detailed Analysis

#### 1. Pro-Rata Formula

**Code (DecimatorModule.sol:618-638):**
```solidity
function _decClaimableFromEntry(
    uint256 poolWei,
    uint256 totalBurn,
    DecEntry storage e,
    uint64 packedOffsets
) private view returns (uint256 amountWei) {
    if (totalBurn == 0) return 0;
    uint8 denom = e.bucket;
    uint8 sub = e.subBucket;
    uint192 entryBurn = e.burn;
    if (denom == 0 || entryBurn == 0) return 0;
    uint8 winningSub = _unpackDecWinningSubbucket(packedOffsets, denom);
    if (sub != winningSub) return 0;
    amountWei = (poolWei * uint256(entryBurn)) / totalBurn;
}
```

**Verification:**
- Formula: `amountWei = (poolWei * playerBurn) / totalBurn` -- **matches v1.1-transition-jackpots.md Section 8**.
- **Rounding direction:** Solidity integer division truncates toward zero. This means `sum(all_player_amounts) <= poolWei` -- the protocol keeps any dust from rounding. **Favors protocol. Correct.**
- **Overflow safety:** `poolWei` is at most 30% of `baseFuturePool` (x00 case). `entryBurn` is `uint192` (max ~6.3e57). The multiplication `poolWei * entryBurn` could overflow `uint256` only if both are astronomically large. In practice, `poolWei` is bounded by total ETH in the protocol (well under `uint128`), and `entryBurn` is bounded by the multiplier cap (200k BURNIE tokens = ~2e23 base units). The product fits within `uint256` with enormous margin. **Safe.**
- **Winner eligibility:** Only players whose `subBucket` matches the VRF-selected `winningSub` for their chosen `denom` can claim. Non-winners get `amountWei = 0` and the `revert DecNotWinner()` at DecimatorModule.sol:383. **Correct.**

#### 2. 50/50 ETH/Lootbox Split

**Code (DecimatorModule.sol:529-543):**
```solidity
function _creditDecJackpotClaimCore(
    address account,
    uint256 amount,
    uint256 rngWord
) private returns (uint256 lootboxPortion) {
    uint256 ethPortion = amount >> 1;        // 50% ETH
    lootboxPortion = amount - ethPortion;    // 50% lootbox (gets odd-wei remainder)
    _addClaimableEth(account, ethPortion, rngWord);
    claimablePool -= lootboxPortion;         // remove lootbox from reserved
    _awardDecimatorLootbox(account, lootboxPortion, rngWord);
}
```

**Verification against v1.1-transition-jackpots.md Section 8:**
- "50% ETH, 50% lootbox" -- **matches** (`amount >> 1` is `amount / 2`, truncation gives ETH the smaller half on odd amounts).
- The caller (`claimDecimatorJackpot`, DecimatorModule.sol:425-432) adds `lootboxPortion` back to futurePrizePool: `_setFuturePrizePool(_getFuturePrizePool() + lootboxPortion)`. This is correct: the lootbox ETH stays in the pool as ticket backing, not extracted.
- **Game Over mode** (DecimatorModule.sol:420-423): `if (gameOver) { _addClaimableEth(msg.sender, amountWei, lastDecClaimRound.rngWord); return; }` -- 100% as ETH. This is the GAMEOVER-mode behavior verified in Phase 26 (v3.0-gameover-core-distribution.md). **Correctly distinct from normal play.**

#### 3. lastDecClaimRound Overwrite Expiry

**Code (DecimatorModule.sol:343-349):**
```solidity
// Snapshot last claim round (overwrites previous - old claims expire)
lastDecClaimRound.lvl = lvl;
lastDecClaimRound.poolWei = poolWei;
lastDecClaimRound.totalBurn = uint232(totalBurn);
lastDecClaimRound.rngWord = rngWord;
return 0; // All funds held for claims
```

**Verification against v1.1-transition-jackpots.md Section 8:**
- "Claims expire when the next decimator resolves. The `lastDecClaimRound` struct is overwritten with new data, making previous round claims permanently inaccessible." -- **matches exactly**.
- The claim validation at DecimatorModule.sol:369 enforces this: `if (lastDecClaimRound.lvl != lvl) revert DecClaimInactive()`. Once a new round overwrites `lastDecClaimRound`, all claims for the previous round's `lvl` will fail this check.
- **By-design classification (CP-02 from research):** This is intentional game design -- players are expected to claim between decimator resolutions. The claim window is bounded by game progression (typically 10 levels apart), giving adequate time. **Classified as by-design, not a finding.**

**Unclaimed pool treatment:** When `lastDecClaimRound` is overwritten, any unclaimed ETH from the previous round remains in `claimablePool` as an overshoot. Over time, this creates a surplus in `claimablePool` relative to `sum(claimableWinnings)`. This surplus is harmless -- it means the protocol holds more ETH than strictly needed, improving solvency. The surplus naturally resolves when the game reaches GAMEOVER and remaining balances are distributed. **No finding.**

#### 4. Double-Claim Guard

**Code (DecimatorModule.sol:364-387):**

```solidity
    if (lastDecClaimRound.lvl != lvl) revert DecClaimInactive();
```

**Verification:**
- **`claimed` flag scope:** `decBurn[lvl][player].claimed` is scoped by both level and player. Each player has a separate `DecEntry` per level. Setting `claimed = 1` prevents re-entry for the same (level, player) pair. **Correct per-round guard.**
- **Flag persistence:** The `claimed` flag persists in storage. Even if `lastDecClaimRound` is overwritten (making the round inactive), the flag remains. This is harmless -- the `lastDecClaimRound.lvl != lvl` check (line 369) already blocks claims for expired rounds. The persistent flag provides defense-in-depth. **Correct.**
- **Access paths:** Two paths reach `_consumeDecClaim`:
  1. `claimDecimatorJackpot(uint24 lvl)` at DecimatorModule.sol:411 -- public, uses `msg.sender`.
  2. `consumeDecClaim(address player, uint24 lvl)` at DecimatorModule.sol:395 -- restricted to `ContractAddresses.GAME` via `onlyGame` modifier. Used for game-initiated claims.
  Both paths converge at `_consumeDecClaim`. **No bypass vector.**

#### 5. prizePoolFrozen Guard

**Code (DecimatorModule.sol:416):**
```solidity
function claimDecimatorJackpot(uint24 lvl) external {
    if (prizePoolFrozen) revert E();
    // ...
}
```

**Verification:**
- `prizePoolFrozen` is set to `true` during jackpot phase days 1-4 and cleared when day 5 resolves (AdvanceModule controls this lifecycle).
- **Why needed:** The decimator claim path writes to `futurePrizePool` (lootbox portion) and potentially `nextPrizePool` (auto-rebuy). - **Edge case:** A player who wins a decimator at level 5 must wait until after the jackpot phase (prizePoolFrozen cleared) to claim. This is typically a few hours to days. #### 6. claimablePool Accounting Through Decimator Claim Path

Full trace of claimablePool mutations:

**At resolution time (EndgameModule.sol:208-218, normal decimator):**
```solidity
uint256 decPoolWei = (futurePoolLocal * 10) / 100;
if (decPoolWei != 0) {
    uint256 returnWei = IDegenerusGame(address(this))
        .runDecimatorJackpot(decPoolWei, lvl, rngWord);
    uint256 spend = decPoolWei - returnWei;
    futurePoolLocal -= spend;
    claimableDelta += spend;  // pre-reserve full pool for claims
}
```
- `runDecimatorJackpot` returns 0 if winners exist (all funds held), or `poolWei` if no winners.
- When winners exist: `spend = decPoolWei`, `claimableDelta += decPoolWei`. The full pool is pre-reserved in `claimablePool`.
- When no winners: `spend = 0`, no claimablePool change. Funds stay in `futurePoolLocal`.

**At claim time (DecimatorModule.sol:411-432):**
1. `_consumeDecClaim(msg.sender, lvl)` validates and returns `amountWei` (pro-rata share).
2. If NOT gameOver: `_creditDecJackpotClaimCore(msg.sender, amountWei, rngWord)`:
   - ETH portion (`amountWei >> 1`): `_addClaimableEth` -> if not auto-rebuy: `_creditClaimable` adds to `claimableWinnings[player]`. claimablePool is already pre-reserved.
   - Lootbox portion: `claimablePool -= lootboxPortion` (line 541). The lootbox ETH is no longer owed as claimable ETH -- it's converted to tickets backed by futurePrizePool.
   - `_setFuturePrizePool(_getFuturePrizePool() + lootboxPortion)` at line 431 -- lootbox ETH added to futurePool.
3. If auto-rebuy active on ETH portion: `_processAutoRebuy` handles ticket conversion. `claimablePool -= calc.ethSpent` at line 494. ETH goes to future/nextPrizePool. Only `calc.reserved` (take-profit) stays in claimablePool.

**Net claimablePool change per claim:**
- Pre-reserved: `+amountWei` (at resolution time, as part of total pool)
- Lootbox deduction: `-lootboxPortion` (line 541)
- Auto-rebuy deduction (if active): `-calc.ethSpent` (line 494)
- Remaining: ETH actually owed to player via `claimableWinnings`
- **Invariant: claimablePool >= sum(claimableWinnings). Verified correct.**

**Unclaimed remainder:** If a player never claims, their pro-rata share stays permanently in claimablePool as surplus. This is safe -- the protocol holds the ETH. When `lastDecClaimRound` is overwritten, the new round's pool is ALSO added to claimablePool. Over many rounds with unclaimed amounts, claimablePool grows monotonically as a solvency buffer. **Not a leak -- surplus favors protocol solvency.**

## PAY-06: Decimator x00 Claims

**Verdict:** PASS
**Files:** `DegenerusGameEndgameModule.sol:193-201`, `DegenerusGameDecimatorModule.sol:297-350` (shared resolution)
**Summary:** The x00 decimator correctly uses 30% of `baseFuturePool` (snapshot before any deductions). It shares the same `runDecimatorJackpot` and `claimDecimatorJackpot` paths as normal decimator. The `lastDecClaimRound` overwrite means only the most recent decimator (whether normal or x00) is claimable at any time.
**Pool Source:** `baseFuturePool` (snapshot, EndgameModule.sol:194) at 30%
**claimablePool Impact:** Same as PAY-05 -- pre-reserved at resolution time (EndgameModule.sol:200: `claimableDelta += spend`), deducted at claim time for lootbox/auto-rebuy portions.
**CEI Status:** Correct. Same delegatecall path as PAY-05.
**Double-Claim Guard:** Same as PAY-05 -- `decBurn[lvl][player].claimed` flag + `lastDecClaimRound.lvl` check.
**Auto-Rebuy Interaction:** Verified. Same `_processAutoRebuy` path as PAY-05.
**Recommendation:** None.

### Detailed Analysis

#### 1. Pool Source

**Code (EndgameModule.sol:193-201):**
```solidity
if (prevMod100 == 0) {
    uint256 decPoolWei = (baseFuturePool * 30) / 100;  // 30% of baseFuturePool
    if (decPoolWei != 0) {
        uint256 returnWei = IDegenerusGame(address(this))
            .runDecimatorJackpot(decPoolWei, lvl, rngWord);
        uint256 spend = decPoolWei - returnWei;
        futurePoolLocal -= spend;
        claimableDelta += spend;
    }
}
```

**Verification against v1.1-transition-jackpots.md Section 5:**
- "x00 Decimator: 30% of `baseFuturePool` (BEFORE any deduction)" -- **matches**.
- `baseFuturePool` is the snapshot taken at entry (line 158), before BAF deduction. At x00 levels, BAF runs first (20% of `baseFuturePool`), modifying `futurePoolLocal`. The x00 decimator then uses `baseFuturePool` (unmodified), not `futurePoolLocal`. **Correct pool source.**
- Combined draw at x00: BAF 20% + Decimator 30% = 50% of `baseFuturePool`. This is within design budget per v1.1-transition-jackpots.md Section 5: "Combined: up to 50% of the future pool snapshot." **Matches spec.**

#### 2. Separate Resolution vs Normal Decimator

**Code flow:**
- x00 decimator: `EndgameModule.sol:193-201` calls `runDecimatorJackpot(decPoolWei, lvl, rngWord)`.
- Normal decimator: `EndgameModule.sol:208-218` calls `runDecimatorJackpot(decPoolWei, lvl, rngWord)`.

Both call the same `runDecimatorJackpot` function. The function uses `lastDecClaimRound` (a single struct) to track the most recent resolution. **The x00 decimator overwrites `lastDecClaimRound` just as normal decimator does.**

**Key observation:** At x00 levels, the `prevMod10 == 5` condition (line 208) is FALSE (`lvl % 10 == 0` at x00 levels), so the normal decimator block does NOT fire. Only the x00 decimator fires at x00 levels. Similarly, at x5 levels (where normal decimator fires), the `prevMod100 == 0` condition is FALSE, so x00 decimator does NOT fire. **They are mutually exclusive for the same level.** At x00, the `lastDecClaimRound` is overwritten by the x00 decimator. At x5, it's overwritten by the normal decimator.

**Double-snapshot guard:** `runDecimatorJackpot` at DecimatorModule.sol:306: `if (lastDecClaimRound.lvl == lvl) return poolWei`. This prevents the same level from being resolved twice (defensive guard against hypothetical bugs in the calling code). **Correct.**

#### 3. Enhanced Payout (30% vs 10%)

The x00 decimator allocates 30% of `baseFuturePool` vs 10% of `futurePoolLocal` for normal. The larger allocation reflects the rarer occurrence (x00 vs every 10 levels) and the x00 milestone's special significance. This creates a larger prize pool for the same number of decimator participants.

**Math example at level 100 with 100 ETH baseFuturePool:**
- BAF: 20% of 100 = 20 ETH drawn, `futurePoolLocal` drops to 80 ETH (post-BAF)
- x00 Decimator: 30% of 100 = 30 ETH drawn from `baseFuturePool` (not 80 ETH)
- `futurePoolLocal` after both: 80 - 30 = 50 ETH (50% consumed)
- Total in `claimablePool`: up to 20 + 30 = 50 ETH (before refunds/lootbox recycling)

**Verification:** The v1.1-transition-jackpots.md Section 5 documents this exact behavior. **Matches spec.**

#### 4. Eligibility and Claim Mechanism

The x00 decimator uses the same claim mechanism as normal decimator:
- Claims go through `claimDecimatorJackpot(uint24 lvl)` at DecimatorModule.sol:411.
- The `lvl` parameter must match `lastDecClaimRound.lvl` (the x00 level).
- All the same guards apply: `prizePoolFrozen`, `claimed` flag, winner validation.
- The pro-rata formula is identical: `(poolWei * playerBurn) / totalBurn`.
- The 50/50 ETH/lootbox split is identical.

**Level check:** There is no explicit check in `claimDecimatorJackpot` that verifies the level is a x00 level. The eligibility is implicitly enforced by the fact that:
1. `lastDecClaimRound` is only set during `runDecimatorJackpot` calls.
2. `runDecimatorJackpot` is only called at appropriate levels (x5 for normal, x00 for x00).
3. The `lvl` parameter in the claim must match `lastDecClaimRound.lvl`.

This means a player claiming "level 100" will succeed only if the most recent decimator resolution was for level 100. **Correct implicit enforcement.**

#### 5. Interaction with Terminal Decimator (Phase 26 Reference)

The terminal decimator (audited in Phase 26, GO-08 PASS) uses a separate `lastTerminalDecClaimRound` struct and `terminalDecBucketBurnTotal` mapping. It operates during GAMEOVER only. The normal/x00 decimator uses `lastDecClaimRound` and `decBucketBurnTotal`. These are completely independent storage paths.

The `decBucketOffsetPacked[lvl]` mapping IS shared between normal and terminal decimator, but as verified in Phase 26 (v3.0-gameover-core-distribution.md), collision is impossible because GAMEOVER and normal level completion are mutually exclusive for the same level number.

**No interference between normal/x00 decimator and terminal decimator. Verified consistent with Phase 26 findings.**

## Informational Notes

### INFO-01: Decimator Claim Expiry is By-Design

The `lastDecClaimRound` overwrite pattern means players must claim between decimator resolutions. At standard game pace, this window is 10 levels (approximately 50 jackpot-phase days). The v1.1-transition-jackpots.md Section 8 explicitly documents this: "Claims expire when the next decimator resolves." No natspec documentation exists on the expiry behavior in DecimatorModule.sol, but the code comments at line 343 state "overwrites previous - old claims expire." **Classified as INFORMATIONAL: documented by-design behavior, not a vulnerability.**

### INFO-02: winnerMask Unused in EndgameModule

The `winnerMask` returned by `jackpots.runBafJackpot()` (third return value) is discarded in `_runBafJackpot` (EndgameModule.sol:346). The DegenerusJackpots contract computes this mask to flag the last 40 scatter winners for special treatment, but the EndgameModule processes all winners uniformly through the large/small classification. This is a minor dead-code observation with no security or economic impact -- the contract pays slightly more gas to compute the mask than necessary. **No action required.**
