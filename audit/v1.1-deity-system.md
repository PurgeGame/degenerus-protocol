## 1. Overview

Deity passes are premium NFT-like positions in the Degenerus game. Each pass is bound to one of 32 symbols (symbolId 0--31), and each symbol can have at most one deity owner. Deity pass owners receive three primary benefits:

- **Boon issuance:** 3 boon grants per day to other players (deterministic type per slot)
- **Activity score bonus:** +80% flat bonus plus floor values on streak/count components
- **Jackpot virtual entries:** ~2% of bucket tickets as virtual entries in every jackpot draw for their symbol

Pricing escalates quadratically via the triangular number formula, starting at 24 ETH for the first pass and reaching 520 ETH for the 32nd. Passes are transferable but incur a 5 ETH BURNIE burn cost and nuke the sender's progression stats.

**Primary source:** `contracts/modules/DegenerusGameWhaleModule.sol`

## 2. Deity Pass Pricing Curve (DEIT-01)

### 2a. Base Price Formula

The price of the k-th deity pass (where k = number of passes already sold, starting from 0) follows a quadratic escalation using the triangular number T(k) = k*(k+1)/2:

```
basePrice = DEITY_PASS_BASE + T(k) = 24 + k*(k+1)/2   (in ETH)
```

**Source:** `DegenerusGameWhaleModule.sol` lines 474--475

```solidity
// DegenerusGameWhaleModule.sol:474-475
uint256 k = deityPassOwners.length;
uint256 basePrice = DEITY_PASS_BASE + (k * (k + 1) * 1 ether) / 2;
```

Maximum 32 passes total. Each pass is bound to a unique symbolId (0--31). A buyer can only hold one deity pass at a time (`deityPassCount[buyer] != 0` reverts).

### 2c. Discount Boons

Deity pass discount boons reduce the purchase price. Three tiers:

| Tier | Discount | BPS | Discounted Formula |
|------|----------|-----|--------------------|
| 1 | 10% | 1000 | basePrice * 9000 / 10000 |
| 2 | 25% | 2500 | basePrice * 7500 / 10000 |
| 3 | 50% | 5000 | basePrice * 5000 / 10000 |

**Source:** `DegenerusGameWhaleModule.sol` lines 491--493

```solidity
// DegenerusGameWhaleModule.sol:491-493
uint16 discountBps = boonTier == 3 ? uint16(5000) : (boonTier == 2 ? uint16(2500) : uint16(1000));
totalPrice = (basePrice * (10_000 - discountBps)) / 10_000;
```

The discount boon is consumed on purchase attempt regardless of whether it has expired. Expiry check: deity-sourced boons expire when `_simulatedDayIndex() > deityDay`; lootbox-sourced boons expire when `_simulatedDayIndex() > stampDay + DEITY_PASS_BOON_EXPIRY_DAYS` (4 days).

|------------|-----------|-------------|
| Pre-game (level 0) | 30% (3000 BPS) | 70% (7000 BPS) |
| Post-game (level 1+) | 5% (500 BPS) | 95% (9500 BPS) |

If the prize pool is frozen (during RNG resolution), amounts are added to pending pools instead.

## 3. Deity Pass Transfer (DEIT-01)

Deity passes are transferable via the ERC721 contract, which triggers `handleDeityPassTransfer` on the game via delegatecall.

### 3a. Transfer Cost

The sender burns 5 ETH worth of BURNIE:

```solidity
// DegenerusGameWhaleModule.sol:157, 577-578
uint256 private constant DEITY_TRANSFER_ETH_COST = 5 ether;
// ...
uint256 burnAmount = (DEITY_TRANSFER_ETH_COST * PRICE_COIN_UNIT) / price;
IDegenerusCoin(ContractAddresses.COIN).burnCoin(from, burnAmount);
```

Where `PRICE_COIN_UNIT = 1000 ether`. So: `burnAmount = (5 ether * 1000 ether) / price`.

### 3b. Sender Stat Nuke

Transfer nukes the sender's progression stats:

```solidity
// DegenerusGameWhaleModule.sol:885-896
function _nukePassHolderStats(address player) private {
    uint256 data = mintPacked_[player];
    // Zero: LEVEL_COUNT, LEVEL_STREAK, LAST_LEVEL, MINT_STREAK_LAST_COMPLETED
    data = BitPackingLib.setPacked(data, BitPackingLib.LEVEL_COUNT_SHIFT, BitPackingLib.MASK_24, 0);
    data = BitPackingLib.setPacked(data, BitPackingLib.LEVEL_STREAK_SHIFT, BitPackingLib.MASK_24, 0);
    data = BitPackingLib.setPacked(data, BitPackingLib.LAST_LEVEL_SHIFT, BitPackingLib.MASK_24, 0);
    data = BitPackingLib.setPacked(data, MINT_STREAK_LAST_COMPLETED_SHIFT, BitPackingLib.MASK_24, 0);
    mintPacked_[player] = data;

    // Reset quest streak via external call to quests contract
    IDegenerusQuestsReset(ContractAddresses.QUESTS).resetQuestStreak(player);
}
```

**Fields zeroed:**
- `levelCount` -- total mints across all levels (used in activity score mint count component)
- `levelStreak` -- consecutive levels minted (used in activity score streak component)
- `lastLevel` -- last level the player minted at
- `mintStreakLastCompleted` -- last level where mint streak was recorded
- Quest streak (via external call to `resetQuestStreak`)

### 3c. Transfer Restrictions

- Not available at level 0 (`level == 0` reverts)
- Sender must hold a deity pass (`deityPassCount[from] == 0` reverts)
- Recipient must not already hold a deity pass (`deityPassCount[to] != 0` reverts)

### 4b. Conditional Total Weights

The total weight changes based on game state:

| Scenario | Total Weight | Condition |
|----------|-------------|-----------|
| All available | 1298 | `decWindowOpen == true` AND `deityPassOwners.length < 32` |
| No decimator | 1248 | `decWindowOpen == false` AND `deityPassOwners.length < 32` |
| No decimator + no deity pass boons | 1208 | `decWindowOpen == false` AND `deityPassOwners.length >= 32` |

**Weight breakdown:**
- Always-available categories: Coinflip (248) + Lootbox (238) + Purchase (496) + Whale (40) + Activity (138) + Whale Pass (8) + Lazy Pass (40) = **1208**
- Decimator (conditional): 40 + 8 + 2 = **50**
- Deity Pass (conditional): 28 + 10 + 2 = **40** (= BOON_WEIGHT_DEITY_PASS_ALL)

Verification: 1208 + 50 + 40 = 1298 = BOON_WEIGHT_TOTAL. 1298 - 50 = 1248 = BOON_WEIGHT_TOTAL_NO_DECIMATOR. 1248 - 40 = 1208.

**Deity pass boon availability cutoff:** When `deityPassOwners.length >= 32` (all 32 passes sold), deity pass discount boons are removed from the draw pool. Both `_deityBoonForSlot` (LootboxModule.sol:756) and `_boonPoolStats` (LootboxModule.sol:1196) gate deity pass boons on `deityPassOwners.length < DEITY_PASS_MAX_TOTAL` (32). For lootbox draws, `_boonPoolStats` additionally requires `deityPassCount[player] == 0`. Note: the `deityBoonData` view function in DegenerusGame.sol:917 uses a hardcoded `< 32` literal rather than the named constant, but the threshold is the same.

**Source:** `DegenerusGameLootboxModule.sol` lines 456--460, 1766--1779

```solidity
// DegenerusGameLootboxModule.sol:1766-1779
function _deityBoonForSlot(
    address deity, uint48 day, uint8 slot,
    bool decimatorAllowed, bool deityPassAvailable
) private view returns (uint8 boonType) {
    uint256 seed = uint256(keccak256(abi.encode(_deityDailySeed(day), deity, day, slot)));
    uint256 total = decimatorAllowed
        ? BOON_WEIGHT_TOTAL
        : BOON_WEIGHT_TOTAL_NO_DECIMATOR;
    if (!deityPassAvailable) total -= BOON_WEIGHT_DEITY_PASS_ALL;
    uint256 roll = seed % total;
    return _boonFromRoll(roll, decimatorAllowed, deityPassAvailable, true, true);
}
```

## 5. Boon Expiry Rules (DEIT-02)

Boons have two distinct lifetimes depending on their source (deity-granted vs lootbox-rolled). Each category has its own expiry constant.

### 5a. Expiry Constants

| Category | Expiry (lootbox-sourced) | Expiry (deity-sourced) | Constant | Source |
|----------|-------------------------|------------------------|----------|--------|
| Coinflip | 2 days | Current day only | COINFLIP_BOON_EXPIRY_DAYS = 2 | BoonModule:23 |
| Lootbox boost | 2 days | Current day only | LOOTBOX_BOOST_EXPIRY_DAYS = 2 | BoonModule:24 |
| Purchase boost | 4 days | Current day only | PURCHASE_BOOST_EXPIRY_DAYS = 4 | BoonModule:25 |
| Decimator boost | N/A (no lootbox constant) | Current day only | (no explicit constant) | BoonModule:222-228 |
| Whale discount | 4 days (hardcoded) | Current day only | (hardcoded in BoonModule) | BoonModule:231-238 |
| Deity pass discount | 4 days | Current day only | DEITY_PASS_BOON_EXPIRY_DAYS = 4 | BoonModule:26 |
| Activity | 2 days | Current day only | COINFLIP_BOON_EXPIRY_DAYS = 2 | BoonModule:284-285 |
| Lazy pass discount | 4 days (hardcoded) | Current day only | (hardcoded `> lazyDay + 4`) | BoonModule:247 |

### 5b. Expiry Logic Pattern

Every boon category follows the same two-phase expiry check:

```solidity
// DegenerusGameBoonModule.sol:40-46 (coinflip example)
uint48 deityDay = deityCoinflipBoonDay[player];
if (deityDay != 0 && deityDay != currentDay) {
    // DEITY-SOURCED: expired (not same day anymore)
    coinflipBoonBps[player] = 0;
    // ... clear all tracking fields
    return 0;
}
uint48 stampDay = coinflipBoonDay[player];
if (stampDay > 0 && currentDay > stampDay + COINFLIP_BOON_EXPIRY_DAYS) {
    // LOOTBOX-SOURCED: expired (more than N days since stamp)
    coinflipBoonBps[player] = 0;
    // ... clear all tracking fields
    return 0;
}
```

**Key distinction:**
- **Deity-sourced** (`deityDay != 0`): Expires when `currentDay != deityDay`. Same-day use only.
- **Lootbox-sourced** (`deityDay == 0`): Expires when `currentDay > stampDay + expiryDays`. Multi-day window.

### 5c. Overwrite vs Upgrade Semantics

| Source | Behavior | Rule |
|--------|----------|------|
| Deity-granted | Always overwrite | Replaces existing boon regardless of tier |
| Lootbox-rolled | Upgrade only | Applied only if new tier > current tier |

**Source:** `DegenerusGameLootboxModule.sol` lines 1349--1358

```solidity
// DegenerusGameLootboxModule.sol:1365-1366
if (isDeity || bps > coinflipBoonBps[player]) {
    coinflipBoonBps[player] = bps;
}
```

> **PITFALL: Deity boon overwrite can be a downgrade.** A deity granting a +5% coinflip boon will overwrite an existing +25% lootbox-sourced coinflip boon. The deity boon replaces unconditionally AND limits lifetime to the current day. This makes deity boon issuance a strategic decision -- issuing to a player who already has a higher-tier lootbox boon would actively harm them.

## 6. Deity Boon Issuance Rules (DEIT-02)

### 6a. Slot Mechanics

Each deity pass holder gets 3 boon issuance slots per day. Each slot's boon type is deterministic, derived from the daily seed, deity address, day index, and slot number.

**Source:** `DegenerusGameLootboxModule.sol` lines 755--781

```solidity
// DegenerusGameLootboxModule.sol:755-781
function issueDeityBoon(address deity, address recipient, uint8 slot) external {
    if (deity == address(0) || recipient == address(0)) revert E();
    if (deity == recipient) revert E();
    if (slot >= DEITY_DAILY_BOON_COUNT) revert E();
    if (deityPassPurchasedCount[deity] == 0) revert E();

    uint48 day = _simulatedDayIndex();
    if (rngWordByDay[day] == 0 && rngWordCurrent == 0) revert E();
    if (deityBoonDay[deity] != day) {
        deityBoonDay[deity] = day;
        deityBoonUsedMask[deity] = 0;
    }
    if (deityBoonRecipientDay[recipient] == day) revert E();

    uint8 mask = deityBoonUsedMask[deity];
    uint8 slotMask = uint8(1) << slot;
    if ((mask & slotMask) != 0) revert E();
    deityBoonUsedMask[deity] = mask | slotMask;
    deityBoonRecipientDay[recipient] = day;

    bool decimatorAllowed = _isDecimatorWindow();
    bool deityPassAvailable = deityPassOwners.length < DEITY_PASS_MAX_TOTAL;
    uint8 boonType = _deityBoonForSlot(deity, day, slot, decimatorAllowed, deityPassAvailable);
    _applyBoon(recipient, boonType, day, day, 0, true);

    emit DeityBoonIssued(deity, recipient, day, slot, boonType);
}
```

### 6b. Issuance Constraints

| Constraint | Rule | Check |
|------------|------|-------|
| Slots per day | 3 (DEITY_DAILY_BOON_COUNT) | `slot >= 3` reverts |
| Self-issue | Prohibited | `deity == recipient` reverts |
| Recipient limit | 1 boon per recipient per day | `deityBoonRecipientDay[recipient] == day` reverts |
| RNG requirement | Valid RNG word must exist for the day | `rngWordByDay[day] == 0 && rngWordCurrent == 0` reverts |
| Pass ownership | Must have purchased (not just transferred) | `deityPassPurchasedCount[deity] == 0` reverts |
| Slot reuse | Each slot used at most once per day | Bitmask tracking via `deityBoonUsedMask` |

### 6c. Boon Type Determination

Each slot's boon type is computed deterministically:

```
seed = keccak256(abi.encode(_deityDailySeed(day), deity, day, slot))
roll = seed % totalWeight
boonType = _boonFromRoll(roll, decimatorAllowed, deityPassAvailable, true, true)
```

The `_deityDailySeed` derives from the day's RNG word. Since the seed is deterministic, the deity can preview all 3 slot types before deciding which slot to issue to which recipient. The `deityBoonSlots` view function exposes this.

### 7a. Deity Pass Bonus

### 7b. Flat Assignment vs Floor Clamping

Deity pass holders and whale/lazy pass holders follow **different code paths** for the streak and mint count components:

**Deity pass path** -- flat assignment (DegenerusGame.sol:2425-2427): `bonusBps` is directly SET to `50*100 + 25*100 = 7500`, bypassing all streak/count computation. Actual participation is irrelevant.

```solidity
// DegenerusGame.sol:2425-2427
if (hasDeityPass) {
    bonusBps = 50 * 100;   // streak = 50 -> 50% (max 50%)
    bonusBps += 25 * 100;  // count = 25 -> 25% (max 25%)
}
```

**Whale/lazy pass path** -- floor clamping (DegenerusGame.sol:2437-2443): actual streak and mint count are computed first, then clamped upward to `PASS_STREAK_FLOOR_POINTS` (50) and `PASS_MINT_COUNT_FLOOR_POINTS` (25) if the player's values are lower. This means a pass holder with streak > 50 keeps their higher value, while a pass holder with streak < 50 is raised to the floor.

This means deity pass holders always get exactly the maximum streak (50%) and count (25%) components as a flat assignment, while whale/lazy pass holders get `max(actual, floor)` via clamping.

## 8. Deity Pass Jackpot Virtual Entries (DEIT-03)

### 8a. Virtual Entry Formula

When a jackpot bucket draw occurs, the deity who owns the symbol matching the bucket's trait receives virtual entries. These virtual entries effectively give the deity a chance to win the jackpot without holding any actual tickets.

**Source:** `DegenerusGameJackpotModule.sol` lines 2274--2306

```solidity
// DegenerusGameJackpotModule.sol:2274-2306
// Virtual deity entries: floor(2% of bucket tickets), minimum 2, if a deity exists for this symbol.
// traitId layout: (quadrant << 6) | (color << 3) | symIdx
// fullSymId = quadrant * 8 + symIdx
uint8 fullSymId = (trait >> 6) * 8 + (trait & 0x07);
address deity;
uint256 virtualCount;
if (fullSymId < 32) {
    deity = deityBySymbol[fullSymId];
    if (deity != address(0)) {
        virtualCount = len / 50;
        if (virtualCount < 2) virtualCount = 2;
    }
}

uint256 effectiveLen = len + virtualCount;
// ...
for (uint256 i; i < numWinners; ) {
    uint256 idx = slice % effectiveLen;
    winners[i] = idx < len ? holders[idx] : deity;
    // ...
}
```

### 8b. Formula Breakdown

1. **Symbol mapping:** `fullSymId = (trait >> 6) * 8 + (trait & 0x07)` extracts the symbol index from the trait byte layout (quadrant * 8 + symIdx)
2. **Eligibility:** `fullSymId < 32` AND `deityBySymbol[fullSymId] != address(0)` -- a deity must exist for this symbol
3. **Virtual count:** `virtualCount = floor(len / 50)` -- floor of 2% of actual bucket ticket count
4. **Minimum floor:** If `virtualCount < 2`, set `virtualCount = 2`
5. **Effective length:** `effectiveLen = len + virtualCount`
6. **Winner selection:** Random index in `[0, effectiveLen)`. If `idx < len`, an actual ticket holder wins. If `idx >= len`, the deity address wins.

### 8c. Deity Win Probability

For a bucket with `len` tickets where a deity exists:

```
virtualCount = max(floor(len / 50), 2)
deityWinProbability = virtualCount / (len + virtualCount)
```

| Bucket Size (len) | virtualCount | effectiveLen | Deity Win P(%) | Note |
|-------------------|--------------|--------------|----------------|------|
| 1 | 2 | 3 | 66.67% | Minimum floor dominates |
| 5 | 2 | 7 | 28.57% | Minimum floor dominates |
| 10 | 2 | 12 | 16.67% | Minimum floor dominates |
| 20 | 2 | 22 | 9.09% | Minimum floor dominates |
| 50 | 2 | 52 | 3.85% | floor(50/50)=1, min=2 |
| 100 | 2 | 102 | 1.96% | floor(100/50)=2 |
| 150 | 3 | 153 | 1.96% | floor(150/50)=3 |
| 200 | 4 | 204 | 1.96% | floor(200/50)=4 |
| 500 | 10 | 510 | 1.96% | floor(500/50)=10 |
| 1000 | 20 | 1020 | 1.96% | floor(1000/50)=20 |
| 5000 | 100 | 5100 | 1.96% | Converges to ~1.96% |

> **PITFALL: Virtual entries scale with bucket size.** The floor(2%) formula means deity advantage converges to ~1.96% for large buckets (since virtualCount/len converges to 1/50, and 1/50 / (1 + 1/50) = 1/51 = 1.96%). But the minimum-2 floor creates a disproportionate advantage in small buckets: 10 tickets yields 2/12 = 16.67% deity win rate. For very small buckets (1 ticket), the deity has a 66.67% chance of winning.

### 8d. Worked Examples

**Example 1: Large bucket (200 tickets)**
- `len = 200`, deity exists for this symbol
- `virtualCount = floor(200 / 50) = 4`
- `effectiveLen = 200 + 4 = 204`
- Deity win probability = 4/204 = **1.96%**
- Any individual ticket holder win probability = 1/204 = **0.49%**

**Example 2: Small bucket (50 tickets)**
- `len = 50`, deity exists for this symbol
- `virtualCount = floor(50 / 50) = 1`, but minimum is 2, so `virtualCount = 2`
- `effectiveLen = 50 + 2 = 52`
- Deity win probability = 2/52 = **3.85%**
- Any individual ticket holder win probability = 1/52 = **1.92%**

**Example 3: Tiny bucket (10 tickets)**
- `len = 10`, deity exists for this symbol
- `virtualCount = floor(10 / 50) = 0`, but minimum is 2, so `virtualCount = 2`
- `effectiveLen = 10 + 2 = 12`
- Deity win probability = 2/12 = **16.67%**
- Any individual ticket holder win probability = 1/12 = **8.33%**

### 8e. Multi-Winner Draws

When `numWinners > 1`, each winner is drawn independently from the same `effectiveLen` pool with rotated entropy (`slice = (slice >> 16) | (slice << 240)`). The deity can win multiple slots in the same draw. Each draw is with replacement from the perspective of the virtual entries (the deity address is always eligible in the virtual range).

## 10. Agent Simulation Notes

### 10a. Deity Pass EV Analysis

```python
def deity_pass_cost(k, discount_tier=0):
    """Cost of the k-th deity pass (k = passes already sold, 0-indexed)."""
    base = 24 + k * (k + 1) / 2
    discount_bps = {0: 0, 1: 1000, 2: 2500, 3: 5000}[discount_tier]
    return base * (10000 - discount_bps) / 10000

def deity_jackpot_win_probability(bucket_size):
    """Probability deity wins a single jackpot draw given bucket_size tickets."""
    if bucket_size == 0:
        return 0.0
    virtual = max(bucket_size // 50, 2)
    return virtual / (bucket_size + virtual)

def deity_activity_score_advantage():
    """Additional activity score BPS from deity pass vs max non-deity."""
    # Deity: 50% streak + 25% count + quest + affiliate + 80% deity = 305% max
    # Non-deity with 100-lvl whale: 50% streak + 25% count + quest + affiliate + 40% whale = 265% max
    # Net advantage: +40% (4000 BPS) from deity bonus replacing whale bonus
    # Plus: deity guarantees 50% streak and 25% count floors regardless of participation
    return 4000  # BPS net advantage at max

def boon_probability(weight, scenario="all"):
    """Probability of drawing a boon type given its weight."""
    totals = {"all": 1298, "no_dec": 1248, "no_dec_no_deity": 1208}
    return weight / totals[scenario]

def deity_daily_boon_ev(boon_values, scenario="all"):
    """
    Expected value of 3 daily boon slots.
    boon_values: dict mapping boon_type_id -> ETH value of receiving that boon
    """
    total_ev = 0
    totals = {"all": 1298, "no_dec": 1248, "no_dec_no_deity": 1208}
    total_weight = totals[scenario]
    # Each slot is an independent draw from the weight table
    # 3 slots per day, each deterministic but treated as random for EV
    for boon_id, weight in BOON_WEIGHTS.items():
        p = weight / total_weight
        value = boon_values.get(boon_id, 0)
        total_ev += 3 * p * value  # 3 slots, EV per slot
    return total_ev
```

### 10b. Key Agent Considerations

1. **Boon issuance is social:** Deity cannot issue to self. The 3 daily boons must be given to other players. Value accrues to the ecosystem, not directly to the deity. Social coordination (e.g., reciprocal boon exchanges between deities) is the primary mechanism for deity boon value capture.

2. **Slot preview:** Deities can preview all 3 slot boon types via `deityBoonSlots` before choosing which recipient gets which slot. Strategic matching of boon types to recipients is possible.

3. **Small bucket jackpot advantage:** The minimum-2 virtual entry floor creates outsized deity win probability in small buckets. Agents should model bucket size distributions to estimate jackpot EV accurately.

4. **Transfer as option:** Deity passes are transferable (at 5 ETH BURNIE cost + stat nuke). This creates a secondary market floor and makes deity pass ownership a time-bounded option rather than a permanent commitment.

5. **Activity score guarantee:** Deity pass ensures maximum streak (50%) and count (25%) components regardless of actual minting behavior. This eliminates participation risk for activity-dependent rewards.
