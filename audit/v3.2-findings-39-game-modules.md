## Per-Contract Findings

### DegenerusGameJackpotModule.sol

#### CMT-V32-001: _resolveTraitWinners @return ticketSpent describes wrong conversion type and wrong destination pool (LOW)

- **What:** The `@return ticketSpent` NatSpec at line 1605 states `ETH converted into auto-rebuy tickets (added to next/futurePool)`. The actual return value maps to `totalWhalePassSpent` (line 1730), which is the ETH cost of whale pass conversions performed by `_processSoloBucketWinner` (solo bucket path only). This ETH is added exclusively to `futurePrizePool` (line 1788: `_setFuturePrizePool(_getFuturePrizePool() + whalePassCost)`), not to `nextPrizePool`. Auto-rebuy ETH in the normal bucket path is routed through `_addClaimableEth` -> `_processAutoRebuy` and is NOT captured in `ticketSpent` at all. The v3.1 fix (CMT-029) replaced the original "loot box awards" text but substituted a different inaccuracy ("auto-rebuy tickets") rather than using the correct suggestion from the v3.1 findings ("whale pass claims").
- **Where:** DegenerusGameJackpotModule.sol:1605
- **Why:** A warden tracing fund flow through `_resolveTraitWinners` would model `ticketSpent` as auto-rebuy ETH split across next/futurePool. The actual flow is whale pass conversion ETH going exclusively to futurePrizePool. The caller `_processOneBucket` at line 1580 adds `ticketSpent` to `ctx.totalPaidEth`, so correct understanding of what this value represents is critical for auditing solvency.
- **Suggestion:** Change line 1605 to: `/// @return ticketSpent ETH converted into whale pass claims (added to futurePrizePool).`
- **Category:** comment-inaccuracy
- **Severity:** LOW

#### CMT-V32-002: Inline comment says "BURNIE only, no ETH bonuses" but early-burn path distributes ETH on non-day-1 levels (INFO)

- **What:** The inline section header at line 609 states `// Non-daily (early-burn) path - BURNIE only, no ETH bonuses`. The code below (lines 613-654) checks `isEthDay` and, for levels after day 1, takes a 1% drip from `futurePrizePool`, splits it into a lootbox budget (75% ticket conversion) and an ETH pool, then runs `_executeJackpot` which distributes actual ETH to winners. The "BURNIE only" characterization is only accurate for day 1 of each level.
- **Where:** DegenerusGameJackpotModule.sol:609
- **Why:** A warden reading the function body sequentially would see "BURNIE only" and might skip over the ETH distribution logic below. The NatSpec block above correctly describes both behaviors, making this inline comment inconsistent with its own function documentation.
- **Suggestion:** Change line 609 to: `// Non-daily (early-burn) path -- ETH from future pool drip (except day 1) + jackpot distribution`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### DegenerusGameLootboxModule.sol

#### CMT-V32-003: PlayerCredited event missing @param recipient NatSpec tag (INFO)

- **What:** The `PlayerCredited` event at line 53 has three parameters (`address indexed player, address indexed recipient, uint256 amount`) but only two `@param` tags are documented (lines 51-52): `@param player` and `@param amount`. The `@param recipient` tag is missing.
- **Where:** DegenerusGameLootboxModule.sol:50-53
- **Why:** A warden reading the event NatSpec would not know what the `recipient` parameter represents or how it differs from `player`. In PayoutUtils (the canonical emitter), both are set to the same value, but the two-field pattern suggests they can diverge.
- **Suggestion:** Add `/// @param recipient The recipient address (always equal to player in current implementation).`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### DegenerusGameAdvanceModule.sol

### DegenerusGameMintModule.sol

#### CMT-V32-005: processFutureTicketBatch @return writesUsed describes queue entries but returns write-budget units (INFO)

- **What:** The `@return writesUsed` NatSpec at line 301 states `Number of queue entries processed in this batch.` The actual return value at line 421 (`writesUsed = used`) is the accumulated write-budget cost, not a count of queue entries. A single queue entry with many owed tickets may consume hundreds of budget units (line 391: `writesThis = (take <= 256) ? (take * 2) : (take + 256)`), and a skipped/zero entry still costs 1 unit.
- **Where:** DegenerusGameMintModule.sol:301
- **Why:** A caller interpreting `writesUsed` as a simple entry count would grossly overestimate progress through the queue. The advanceGame caller uses this return value for gas budget decisions.
- **Suggestion:** Change to: `@return writesUsed Write-budget units consumed in this batch (not a simple entry count; each entry costs 2+ units depending on ticket count).`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### DegenerusGameDecimatorModule.sol

#### CMT-V32-006: decClaimable @return still references "expired" after expiry removal (INFO)

- **What:** The `@return amountWei` NatSpec for `decClaimable` at line 437 states `Claimable amount (0 if not winner, already claimed, or expired).` Since commit 19f5bc60 removed the decimator claim expiry, claims are now persistent and never expire. The "or expired" clause is stale.

- **Suggestion:** Change to: `@return amountWei Claimable amount (0 if not winner or already claimed).`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### DegenerusGameGameOverModule.sol

#### DRIFT-V32-001: _sendToVault hard-revert consequence undocumented in NatSpec (LOW)

- **What:** `_sendToVault` (line 194) hard-reverts via `if (!steth.transfer(...)) revert E()` and `if (!ok) revert E()`. When called from `handleGameOverDrain` (line 157) or `handleFinalSweep` (line 187), a revert rolls back all state changes including terminal flags (`gameOver`, `gameOverFinalJackpotPaid`, `finalSwept`). The `@custom:reverts` tags document that the functions CAN revert, but not the CONSEQUENCE: if stETH (Lido) is paused or reverts, game-over drain and final sweep are blocked because no state transition persists past the revert. This contrasts with AdvanceModule's `_autoStakeExcessEth` which uses try/catch to make stETH non-blocking.
- **Where:** DegenerusGameGameOverModule.sol:67, 169, 190-193
- **Why:** A warden analyzing the game-over path would see `@custom:reverts E When stETH transfer fails` and understand the function can revert, but would not recognize the systemic consequence without tracing the entire state machine. A warden familiar with the Lido pause risk would flag this as a centralization/external-dependency finding.
- **Suggestion:** Add `@dev` to `_sendToVault`: "WARNING: Hard reverts on stETH failure. If Lido is paused, handleGameOverDrain and handleFinalSweep are blocked until stETH recovers. No try/catch fallback -- by design, funds should not be distributed if stETH is non-functional."
- **Category:** intent-drift
- **Severity:** LOW
- **Origin:** v3.1 DRIFT-003 (GameOverModule), not fixed -- re-reported with unified ID.

## Cross-Cutting Patterns

1. **Missing @param tags on partially-documented functions (3 instances):** CMT-V32-003 (LootboxModule: PlayerCredited missing @param recipient), CMT-V32-004 (AdvanceModule: wireVrf missing @param keyHash_), CMT-V32-005 (MintModule: processFutureTicketBatch @return writesUsed inaccurate). Functions with partial NatSpec are more dangerous than functions with no NatSpec -- a warden assumes the existing tags are complete and may not verify undocumented parameters.

2. **v3.1 fix text substitution errors (2 instances):** CMT-029 was applied with "auto-rebuy tickets" instead of the suggested "whale pass claims" (creating CMT-V32-001). CMT-012 was applied with "queue entries" for writesUsed instead of "write-budget units" (creating CMT-V32-005). Both cases where the fix addressed the original problem but introduced a new, different inaccuracy.

3. **Stale reference from code change not propagated to comments (1 instance):** CMT-V32-006 (DecimatorModule: "expired" still present after expiry removal). When a code change removes behavior, all NatSpec referencing that behavior must be updated.
