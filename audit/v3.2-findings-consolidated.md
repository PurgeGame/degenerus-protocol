### Pattern 1: Stale NatSpec after code removal (7 findings)

When code is removed or refactored, NatSpec referencing the old behavior is left behind.

| Finding ID | Contract | Description |
|-----------|----------|-------------|
| LOW-01 | IBurnieCoinflip.sol | `@custom:reverts RngLocked` left on claimCoinflips after rngLocked guard removal |

**Recommendation:** Any code change that removes a feature should include a grep for all references to that feature across NatSpec, headers, and error declarations.

### Pattern 2: Incomplete NatSpec on partially-documented functions (5 findings)

Functions with some @param/@return tags but missing others are more dangerous than undocumented functions because wardens assume completeness.

| Finding ID | Contract | Description |
|-----------|----------|-------------|
| CMT-V32-003 | LootboxModule | PlayerCredited event missing `@param recipient` |
| CMT-V32-004 | AdvanceModule | wireVrf missing `@param keyHash_` |
| CMT-V32-005 | MintModule | processFutureTicketBatch `@return writesUsed` inaccurate |
| CMT-059 | BurnieCoin.sol | `_burn` @dev lists only 2 of 4 callers |
| NEW-002 | DegenerusGame.sol | DELEGATE MODULE HELPERS header lists 5 of 9 modules |

**Recommendation:** NatSpec should either document ALL parameters/callers or document none (with a @dev note explaining the omission). Partial documentation is worse than no documentation.
