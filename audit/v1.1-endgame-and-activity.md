## 1. Overview

Three interconnected systems govern the endgame lifecycle of Degenerus:

- **Activity scores** modify reward curves: higher engagement yields better lootbox EV multipliers and degenerette ROI. Scores are computed per-player in basis points (BPS) from mint streaks, participation rate, quest streaks, affiliate bonuses, and pass/deity status.

- **Death clock** creates terminal game conditions: each level has a liveness timeout (120 days for level > 0, 365 days for level 0). As timeout approaches, the game escalates through "game over imminent" (decimator burns allowed) and "distress mode" (lootbox 100% to nextPool, 25% ticket bonus) before triggering terminal game over.

- **Terminal distribution** pays out all remaining funds when game over fires: 10% to decimator jackpot, 90% to next-level ticketholders, with deity pass refunds on early game over (levels 0-9). After 30 days, a final sweep forfeits unclaimed winnings and splits the remainder 50/50 between vault and DGNRS contract.

## 2. Activity Score System

**Requirement:** LEVL-04
**Entry point:** `_playerActivityScore(address player)`
**Source:** `DegenerusGame.sol` lines 2387--2463

The activity score is a single BPS value computed from additive components. Three player categories have different computation paths.

### 2a. Base Components (Non-Deity, Non-Pass Players)

| Component | Formula | Max BPS | Notes |
|-----------|---------|---------|-------|
| Mint streak | `min(streak, 50) * 100` | 5,000 | 1% per consecutive level minted, capped at 50 |
| Mint count | `min((mintCount * 25) / currLevel, 25) * 100` | 2,500 | Participation rate scaled to 25 points max |
| Quest streak | `min(questStreakRaw, 100) * 100` | 10,000 | 1% per quest streak, capped at 100 |
| Affiliate bonus | `affiliateBonusPointsBest * 100` | variable | Best affiliate bonus points for current level |
| Whale bundle (10-level, type 1) | `+1,000` | 1,000 | Only while `frozenUntilLevel > currLevel` |
| Whale bundle (100-level, type 3) | `+4,000` | 4,000 | Only while `frozenUntilLevel > currLevel` |

**Theoretical maximum (non-deity, non-pass):** 5,000 + 2,500 + 10,000 + affiliate + 4,000 = 21,500+ BPS

### 2b. Active Pass Holder Floors

**Condition:** `frozenUntilLevel > currentLevel` AND `bundleType` is 1 (10-level) or 3 (100-level)

When a player has an active whale/lazy pass, two floor values override lower computed scores:

| Component | Floor Constant | Floor Value (BPS) | Source |
|-----------|---------------|-------------------|--------|
| Streak | `PASS_STREAK_FLOOR_POINTS = 50` | 5,000 | DegenerusGame.sol:219 |
| Mint count | `PASS_MINT_COUNT_FLOOR_POINTS = 25` | 2,500 | DegenerusGame.sol:221 |

```solidity
// DegenerusGame.sol:2424-2431
if (passActive) {
    if (streakPoints < PASS_STREAK_FLOOR_POINTS) {
        streakPoints = PASS_STREAK_FLOOR_POINTS;
    }
    if (mintCountPoints < PASS_MINT_COUNT_FLOOR_POINTS) {
        mintCountPoints = PASS_MINT_COUNT_FLOOR_POINTS;
    }
}
```

**Active pass holder guaranteed minimum:** 5,000 + 2,500 + bundle bonus (1,000 or 4,000) = 8,500 or 11,500 BPS before quest/affiliate.

### 2c. Deity Pass Holders

**Condition:** `deityPassCount[player] != 0`

Deity pass holders bypass the streak/count computation entirely with fixed values:

```solidity
// DegenerusGame.sol:2412-2414
if (hasDeityPass) {
    bonusBps = 50 * 100;    // 5,000 BPS (replaces streak)
    bonusBps += 25 * 100;   // 2,500 BPS (replaces mint count)
}
```

Then adds:
- `DEITY_PASS_ACTIVITY_BONUS_BPS = 8,000` (DegenerusGame.sol:217)
- Quest streak (same formula as base: `min(questStreakRaw, 100) * 100`)
- Affiliate bonus (same formula as base: `affiliateBonusPointsBest * 100`)

**Deity minimum:** 5,000 + 2,500 + 8,000 = **15,500 BPS** before quest/affiliate.

**Deity theoretical maximum:** 15,500 + 10,000 (quest) + affiliate = **25,500+ BPS**

> **Pitfall: Deity Pass Replaces, Does Not Stack With, Bundle Bonuses.**
> When `hasDeityPass` is true, the code enters the deity branch (line 2412) which does NOT add bundle bonuses (1,000/4,000 BPS). The deity bonus of 8,000 BPS replaces the bundle bonus, not stacks with it (DegenerusGame.sol:2450-2458).

### 2d. Full Activity Score Solidity

```solidity
// DegenerusGame.sol:2387-2463
function _playerActivityScore(
    address player
) internal view returns (uint256 scoreBps) {
    if (player == address(0)) return 0;

    bool hasDeityPass = deityPassCount[player] != 0;
    uint256 packed = mintPacked_[player];
    uint24 levelCount = uint24(
        (packed >> BitPackingLib.LEVEL_COUNT_SHIFT) & BitPackingLib.MASK_24
    );
    uint24 streak = _mintStreakEffective(player, _activeTicketLevel());
    uint24 currLevel = level;
    uint24 frozenUntilLevel = uint24(
        (packed >> BitPackingLib.FROZEN_UNTIL_LEVEL_SHIFT) &
            BitPackingLib.MASK_24
    );
    uint8 bundleType = uint8(
        (packed >> BitPackingLib.WHALE_BUNDLE_TYPE_SHIFT) & 3
    );
    bool passActive = frozenUntilLevel > currLevel &&
        (bundleType == 1 || bundleType == 3);

    uint256 bonusBps;

    unchecked {
        if (hasDeityPass) {
            bonusBps = 50 * 100;
            bonusBps += 25 * 100;
        } else {
            uint256 streakPoints = streak > 50 ? 50 : uint256(streak);
            uint256 mintCountPoints = _mintCountBonusPoints(
                levelCount,
                currLevel
            );
            if (passActive) {
                if (streakPoints < PASS_STREAK_FLOOR_POINTS) {
                    streakPoints = PASS_STREAK_FLOOR_POINTS;
                }
                if (mintCountPoints < PASS_MINT_COUNT_FLOOR_POINTS) {
                    mintCountPoints = PASS_MINT_COUNT_FLOOR_POINTS;
                }
            }
            bonusBps = streakPoints * 100;
            bonusBps += mintCountPoints * 100;
        }

        (uint32 questStreakRaw, , , ) = questView.playerQuestStates(player);
        uint256 questStreak = questStreakRaw > 100
            ? 100
            : uint256(questStreakRaw);
        bonusBps += questStreak * 100;

        bonusBps +=
            affiliate.affiliateBonusPointsBest(currLevel, player) *
            100;

        if (hasDeityPass) {
            bonusBps += DEITY_PASS_ACTIVITY_BONUS_BPS;
        } else if (frozenUntilLevel > currLevel) {
            if (bundleType == 1) {
                bonusBps += 1000;
            } else if (bundleType == 3) {
                bonusBps += 4000;
            }
        }
    }

    scoreBps = bonusBps;
}
```

### 2e. Mint Count Bonus Points Helper

```solidity
// DegenerusGame.sol:2474-2486
function _mintCountBonusPoints(
    uint24 mintCount,
    uint24 currLevel
) private pure returns (uint256) {
    if (currLevel == 0) return 0;
    if (mintCount >= currLevel) return 25;
    return (uint256(mintCount) * 25) / uint256(currLevel);
}
```

**Perfect participation** (mintCount >= currLevel) always yields 25 points (2,500 BPS). Partial participation is proportional with integer truncation.

| Level | Mints | Calculation | Points | BPS |
|-------|-------|-------------|--------|-----|
| 0 | any | returns 0 | 0 | 0 |
| 2 | 2 | 2 >= 2 | 25 | 2,500 |
| 10 | 7 | (7 * 25) / 10 = 17 | 17 | 1,700 |
| 10 | 10 | 10 >= 10 | 25 | 2,500 |
| 30 | 15 | (15 * 25) / 30 = 12 | 12 | 1,200 |

## 3. Mint Streak Mechanics

**Requirement:** LEVL-04 (continued)
**Source:** `DegenerusGameMintStreakUtils.sol` (62 lines)

### 3a. Recording (`_recordMintStreakForLevel`)

```solidity
// DegenerusGameMintStreakUtils.sol:17-46
function _recordMintStreakForLevel(address player, uint24 mintLevel) internal {
    if (player == address(0)) return;
    uint256 mintData = mintPacked_[player];
    uint24 lastCompleted = uint24(
        (mintData >> MINT_STREAK_LAST_COMPLETED_SHIFT) & BitPackingLib.MASK_24
    );
    if (lastCompleted == mintLevel) return;  // idempotent per level

    uint24 newStreak;
    if (lastCompleted != 0 && lastCompleted + 1 == mintLevel) {
        // Consecutive: increment streak
        uint24 streak = uint24(
            (mintData >> BitPackingLib.LEVEL_STREAK_SHIFT) &
                BitPackingLib.MASK_24
        );
        if (streak < type(uint24).max) {
            unchecked {
                newStreak = streak + 1;
            }
        } else {
            newStreak = streak;
        }
    } else {
        newStreak = 1;  // Gap detected: reset streak to 1
    }

    uint256 updated = (mintData & ~MINT_STREAK_FIELDS_MASK) |
        (uint256(mintLevel) << MINT_STREAK_LAST_COMPLETED_SHIFT) |
        (uint256(newStreak) << BitPackingLib.LEVEL_STREAK_SHIFT);
    mintPacked_[player] = updated;
}
```

**Key behaviors:**
- **Idempotent:** calling twice for the same `mintLevel` is a no-op (`lastCompleted == mintLevel` guard)
- **Consecutive detection:** streak increments ONLY when `lastCompleted + 1 == mintLevel`
- **Gap reset:** any gap (including starting from zero when `lastCompleted == 0`) resets streak to 1
- **Overflow safe:** capped at `type(uint24).max` (16,777,215)

### 3b. Evaluation (`_mintStreakEffective`)

```solidity
// DegenerusGameMintStreakUtils.sol:49-61
function _mintStreakEffective(
    address player,
    uint24 currentMintLevel
) internal view returns (uint24 streak) {
    uint256 packed = mintPacked_[player];
    uint256 lastCompleted = (packed >> MINT_STREAK_LAST_COMPLETED_SHIFT) &
        BitPackingLib.MASK_24;
    if (lastCompleted == 0) return 0;
    if (uint256(currentMintLevel) > lastCompleted + 1) return 0;  // gap detection
    streak = uint24(
        (packed >> BitPackingLib.LEVEL_STREAK_SHIFT) & BitPackingLib.MASK_24
    );
}
```

**Gap detection at read time:** if `currentMintLevel > lastCompleted + 1`, the player missed a level and their effective streak returns 0 -- even though the stored streak value hasn't been reset yet. The stored value only updates on the next `_recordMintStreakForLevel` call.

### 3c. Storage Layout

Streak data is packed into `mintPacked_[player]`:

| Bit Range | Field | Shift Constant |
|-----------|-------|----------------|
| [24:47] | levelCount (total mints) | `LEVEL_COUNT_SHIFT = 24` |
| [48:71] | streak (consecutive levels) | `LEVEL_STREAK_SHIFT = 48` |
| [128:151] | frozenUntilLevel | `FROZEN_UNTIL_LEVEL_SHIFT = 128` |
| [152:153] | bundleType (0/1/3) | `WHALE_BUNDLE_TYPE_SHIFT = 152` |
| [160:183] | lastCompleted (last streak level) | `MINT_STREAK_LAST_COMPLETED_SHIFT = 160` |

Source: `BitPackingLib.sol` lines 45--60, `DegenerusGameMintStreakUtils.sol` line 10.

## 4. Activity Score Consumers

The activity score BPS value feeds into two reward systems with distinct threshold curves.

### 4a. Lootbox Future Ticket EV Multiplier

**Source:** `DegenerusGameLootboxModule.sol` lines 319--499

| Constant | Value | Source Line | Meaning |
|----------|-------|-------------|---------|
| `ACTIVITY_SCORE_NEUTRAL_BPS` | 6,000 | LootboxModule:321 | 60% activity = neutral (100%) EV |
| `ACTIVITY_SCORE_MAX_BPS` | 25,500 | LootboxModule:323 | 255% activity = maximum (135%) EV |
| `LOOTBOX_EV_MIN_BPS` | 8,000 | LootboxModule:325 | 0% activity = 80% EV |
| `LOOTBOX_EV_NEUTRAL_BPS` | 10,000 | LootboxModule:327 | 60% activity = 100% EV (breakeven) |
| `LOOTBOX_EV_MAX_BPS` | 13,500 | LootboxModule:329 | 255%+ activity = 135% EV |

**Piecewise linear curve:**

```
Score <= 6,000 BPS:
  EV = 8,000 + (score * (10,000 - 8,000)) / 6,000
     = 8,000 + score * 2,000 / 6,000

6,000 < Score < 25,500 BPS:
  excess = score - 6,000
  EV = 10,000 + (excess * (13,500 - 10,000)) / (25,500 - 6,000)
     = 10,000 + excess * 3,500 / 19,500

Score >= 25,500 BPS:
  EV = 13,500 (capped)
```

| Activity Score | EV Multiplier | Description |
|---------------|---------------|-------------|
| 0 BPS | 80% (0.80x) | Inactive player |
| 3,000 BPS | 90% (0.90x) | Low engagement |
| 6,000 BPS | 100% (1.00x) | Neutral (breakeven) |
| 15,750 BPS | ~117.5% (~1.175x) | High engagement |
| 25,500+ BPS | 135% (1.35x) | Maximum (deity-level) |

### 4b. Degenerette ROI Curve

**Source:** `DegenerusGameDegeneretteModule.sol` lines 191--211, 1086--1112

| Constant | Value | Source Line | Meaning |
|----------|-------|-------------|---------|
| `ACTIVITY_SCORE_MID_BPS` | 7,500 | DegeneretteModule:191 | 75% activity = mid-tier |
| `ACTIVITY_SCORE_HIGH_BPS` | 25,500 | DegeneretteModule:194 | 255% activity = high-tier |
| `ACTIVITY_SCORE_MAX_BPS` | 30,500 | DegeneretteModule:197 | 305% activity = max (deity theoretical) |
| `ROI_MIN_BPS` | 9,000 | DegeneretteModule:200 | 0% activity = 90% ROI |
| `ROI_MID_BPS` | 9,500 | DegeneretteModule:203 | 75% activity = 95% ROI |
| `ROI_HIGH_BPS` | 9,950 | DegeneretteModule:206 | 255% activity = 99.5% ROI |
| `ROI_MAX_BPS` | 9,990 | DegeneretteModule:209 | 305% activity = 99.9% ROI |

**Three-segment piecewise curve:**

```
Segment 1 (0 to 7,500 BPS) -- Quadratic:
  x = score / 7,500
  ROI = 9,000 + 1,000*x - 500*x^2
  (Concave curve: fast initial gains, slowing as x approaches 1)

Segment 2 (7,500 to 25,500 BPS) -- Linear:
  delta = score - 7,500
  span = 25,500 - 7,500 = 18,000
  ROI = 9,500 + (delta * (9,950 - 9,500)) / 18,000
      = 9,500 + delta * 450 / 18,000

Segment 3 (25,500 to 30,500 BPS) -- Linear:
  delta = score - 25,500
  span = 30,500 - 25,500 = 5,000
  ROI = 9,950 + (delta * (9,990 - 9,950)) / 5,000
      = 9,950 + delta * 40 / 5,000
```

| Activity Score | ROI (BPS) | ROI % | Segment |
|---------------|-----------|-------|---------|
| 0 BPS | 9,000 | 90.0% | Quadratic |
| 3,750 BPS | 9,406 | 94.06% | Quadratic |
| 7,500 BPS | 9,500 | 95.0% | Quadratic/Linear boundary |
| 15,000 BPS | 9,688 | 96.88% | Linear |
| 25,500 BPS | 9,950 | 99.5% | Linear/Linear boundary |
| 30,500 BPS | 9,990 | 99.9% | Linear (cap) |

**Source:** `DegenerusGameDegeneretteModule.sol` lines 1121--1132

A separate linear function for WWXRP currency degenerettes:

| Constant | Value | Source Line |

| `WWXRP_HIGH_ROI_BASE_BPS` | 9,000 | DegeneretteModule:217 |
| `WWXRP_HIGH_ROI_MAX_BPS` | 10,990 | DegeneretteModule:220 |

```
WWXRP ROI = 9,000 + (score * (10,990 - 9,000)) / 30,500
```

## 5. Death Clock: Three-Stage Escalation

**Requirement:** END-01

The death clock is a liveness guard that prevents the game from being permanently locked if player activity drops to zero. It escalates through three stages before triggering terminal game over.

### 5a. Base Timeouts

| Level | Timeout Duration | Source |
|-------|-----------------|--------|
| Level 0 (deploy) | 365 days from `levelStartTime` | `DEPLOY_IDLE_TIMEOUT_DAYS = 365` (AdvanceModule:91) |
| Level > 0 | 120 days from `levelStartTime` | Hardcoded in AdvanceModule:382 |

`levelStartTime` is set to `block.timestamp` at each jackpot phase entry (AdvanceModule:289).

### 5b. Safety Valve

**Source:** `DegenerusGameAdvanceModule.sol` lines 400--403

If the liveness timeout fires but the level's purchase target has already been met, the game does NOT terminate:

```solidity
// DegenerusGameAdvanceModule.sol:400-403
if (lvl != 0 && _getNextPrizePool() >= levelPrizePool[lvl]) {
    levelStartTime = ts;
    return false;
}
```

This resets `levelStartTime` and allows the level to continue. The safety valve only applies to levels > 0 (level 0 has no purchase target to check against).

### 5c. Escalation Timeline

The three stages are measured as countdowns before the base timeout:

```
Timeline for level > 0 (120-day timeout):

|--- normal gameplay ---|-- imminent (5d) --|-- distress (6h) --|-- GAME OVER
0 days                  115 days            119d 18h              120 days
                        ^                   ^                    ^
                        decimator burns     100% nextPool        timeout fires
                        allowed             + 25% ticket bonus
```

```
Timeline for level 0 (365-day timeout):

|--- normal gameplay ---|-- imminent (10d) --|-- distress (6h) --|-- GAME OVER
0 days                  355 days             364d 18h             365 days
```

### 5d. Stage 1: Game Over Imminent

**Source:** `DegenerusGame.sol` lines 2227--2237

```solidity
// DegenerusGame.sol:2227-2237
function _isGameoverImminent() private view returns (bool) {
    if (gameOver) return false;
    uint48 lst = levelStartTime;
    uint48 ts = uint48(block.timestamp);

    if (level == 0) {
        return
            uint256(ts) + 10 days >
            uint256(lst) + uint256(DEPLOY_IDLE_TIMEOUT_DAYS) * 1 days;
    }
    return uint256(ts) + 5 days > uint256(lst) + 120 days;
}
```

| Level | Window | Effect |
|-------|--------|--------|
| Level 0 | Last 10 days before timeout | Decimator burns allowed |
| Level > 0 | Last 5 days before timeout | Decimator burns allowed |

**Consumer:** `decWindow()` (DegenerusGame.sol:2211) returns `true` when `decWindowOpen || _isGameoverImminent()`, allowing decimator entries near timeout.

### 5e. Stage 2: Distress Mode

**Source:** `DegenerusGameStorage.sol` lines 169--178

```solidity
// DegenerusGameStorage.sol:169-178
function _isDistressMode() internal view returns (bool) {
    if (gameOver) return false;
    uint48 lst = levelStartTime;
    uint48 ts = uint48(block.timestamp);
    if (level == 0) {
        return uint256(ts) + uint256(DISTRESS_MODE_HOURS) * 1 hours >
            uint256(lst) + uint256(_DEPLOY_IDLE_TIMEOUT_DAYS) * 1 days;
    }
    return uint256(ts) + uint256(DISTRESS_MODE_HOURS) * 1 hours > uint256(lst) + 120 days;
}
```

**Trigger:** 6 hours before liveness timeout (`DISTRESS_MODE_HOURS = 6`, Storage:161).

**Effects on lootbox purchases (DegenerusGameMintModule.sol:724-736):**

1. **Pool routing override:** 100% of lootbox ETH routes to nextPool (normally split between next/future/vault)
2. **Ticket bonus:** 25% extra future tickets on the distress-bought portion

```solidity
// DegenerusGameLootboxModule.sol:953-960
// Distress-mode ticket bonus: 25% extra on the fraction bought during distress
if (distressEth != 0 && totalPackedEth != 0) {
    uint256 bonus = (uint256(futureTickets) * distressEth * DISTRESS_TICKET_BONUS_BPS)
        / (totalPackedEth * 10_000);
    // DISTRESS_TICKET_BONUS_BPS = 2500          (LootboxModule:317)
}
```

The bonus is proportional: if 40% of a player's accumulated lootbox ETH was bought during distress, they get `futureTickets * 0.40 * 0.25 = 10%` bonus tickets on top of the base award.

### 5f. Stage 3: Game Over

**Source:** `DegenerusGameAdvanceModule.sol` lines 379--382

```solidity
// DegenerusGameAdvanceModule.sol:379-382
bool livenessTriggered = (lvl == 0 &&
    ts - lst > uint256(DEPLOY_IDLE_TIMEOUT_DAYS) * 1 days) ||
    (lvl != 0 && ts - 120 days > lst);
```

When `livenessTriggered` is true AND the safety valve does not apply (nextPool < target or level 0):
1. RNG is acquired (VRF or fallback -- see Section 6)
2. `handleGameOverDrain` is called via delegatecall to GameOverModule
3. The game enters terminal state

## 6. RNG Fallback for Game Over

**Requirement:** END-01 (continued)
**Source:** `DegenerusGameAdvanceModule.sol` lines 744--825

Game over requires an RNG word for jackpot distribution. The system has a fallback mechanism to ensure the game can always terminate, even if Chainlink VRF stalls.

### 6a. Normal Path

1. Attempt VRF request via `_tryRequestRng`
2. If VRF responds before timeout, use the word directly
3. `_gameOverEntropy` returns the word, game over proceeds

### 6b. Fallback Path (3-Day Delay)

```
GAMEOVER_RNG_FALLBACK_DELAY = 3 days    (AdvanceModule:92)
```

If VRF has not responded after 3 days:

```solidity
// DegenerusGameAdvanceModule.sol:768-780
if (elapsed >= GAMEOVER_RNG_FALLBACK_DELAY) {
    uint256 fallbackWord = _getHistoricalRngFallback(day);
    fallbackWord = _applyDailyRng(day, fallbackWord);
    // ... process coinflip and lootbox RNG, return fallbackWord
}
```

### 6c. Historical VRF Fallback Construction

```solidity
// DegenerusGameAdvanceModule.sol:806-825
function _getHistoricalRngFallback(
    uint48 currentDay
) private view returns (uint256 word) {
    uint256 found;
    uint256 combined;
    uint48 searchLimit = currentDay > 30 ? 30 : currentDay;
    for (uint48 searchDay = 1; searchDay < searchLimit; ) {
        uint256 w = rngWordByDay[searchDay];
        if (w != 0) {
            combined = uint256(keccak256(abi.encodePacked(combined, w)));
            unchecked { ++found; }
            if (found == 5) break;
        }
        unchecked {
            ++searchDay;
        }
    }

    return uint256(keccak256(abi.encodePacked(combined, currentDay, block.prevrandao)));
}
```

**Construction:** Hashes up to 5 early historical VRF words (from the first 30 days of game history) together with `currentDay` and `block.prevrandao`.

**Security properties:**
- Historical VRF words are committed on-chain (non-manipulable)
- `block.prevrandao` adds unpredictability with 1-bit validator manipulation (propose or skip)
- At level 0 with zero VRF history, falls through to prevrandao-only entropy (acceptable since zero history means zero completed advances)

### 6d. Failure Modes

| Scenario | Return Value | Effect |
|----------|-------------|--------|
| VRF word already available | word | Proceeds immediately |
| VRF pending, responded | word | Proceeds with VRF response |
| VRF pending, < 3 days | 0 | Waits, retries on next call |
| VRF pending, >= 3 days | fallback word | Proceeds with historical fallback |
| VRF request failed | sets rngRequestTime, returns 0 | Starts 3-day fallback timer |

### 7a. Full Distribution Flow

**Step 1: Early Game Over Deity Refund (levels 0-9 only)**

```solidity
        uint16 purchasedCount = deityPassPurchasedCount[owner];
        if (purchasedCount != 0) {
            uint256 refund = refundPerPass * uint256(purchasedCount);
            if (refund > budget) {
                refund = budget;
            }
            if (refund != 0) {
                unchecked {
                    claimableWinnings[owner] += refund;
                    totalRefunded += refund;
                    budget -= refund;
                }
            }
            if (budget == 0) break;
        }
        unchecked {
            ++i;
        }
    }
    if (totalRefunded != 0) {
        claimablePool += totalRefunded;
    }
}
```

- **Refund amount:** 20 ETH per deity pass (`DEITY_PASS_EARLY_GAMEOVER_REFUND = 20 ether`, GameOverModule:38-39)
- **Order:** FIFO by `deityPassOwners` array (purchase order)
- **Budget cap:** `totalFunds - claimablePool` (cannot dip into already-claimed winnings)
- **Multiple passes:** `deityPassPurchasedCount[owner]` allows refund for multiple passes per owner

Players who only participated in the current level receive nothing from the terminal jackpot unless they also hold next-level tickets.

Any funds not distributed by the terminal jackpot (e.g., rounding dust, no next-level ticketholders) are sent to vault via `_sendToVault`.

### 10a. Computing Activity Score

```
function computeActivityScore(player):
    if player.hasDeityPass:
        bonusBps = 5000 + 2500                          // fixed deity
        bonusBps += DEITY_PASS_ACTIVITY_BONUS_BPS       // +8000
        bonusBps += min(player.questStreak, 100) * 100
        bonusBps += player.affiliateBonusPointsBest * 100

    streak = mintStreakEffective(player, activeTicketLevel)
    streakPoints = min(streak, 50)
    mintCountPoints = mintCountBonusPoints(player.levelCount, currentLevel)

    passActive = player.frozenUntilLevel > currentLevel
                 AND player.bundleType in {1, 3}

    if passActive:
        streakPoints = max(streakPoints, 50)
        mintCountPoints = max(mintCountPoints, 25)

    bonusBps = streakPoints * 100 + mintCountPoints * 100
    bonusBps += min(player.questStreak, 100) * 100
    bonusBps += player.affiliateBonusPointsBest * 100

    if passActive:
        if player.bundleType == 1: bonusBps += 1000
        if player.bundleType == 3: bonusBps += 4000
```

### 10c. Computing Terminal Distribution Shares

```
function terminalDistribution(totalFunds, claimablePool, currentLevel, deityPassOwners):
    // Step 1: Deity refund (levels 0-9)
    available = totalFunds - claimablePool
    deityRefunded = 0
    if currentLevel < 10:
        budget = available
        for owner in deityPassOwners:  // FIFO order
            refund = min(20 ETH * owner.purchasedCount, budget)
            owner.claimable += refund
            deityRefunded += refund
            budget -= refund
            if budget == 0: break
        claimablePool += deityRefunded

    // Step 2: Recalculate available after refunds
    available = totalFunds - claimablePool

    // Step 3: Decimator 10%
    decPool = available / 10
    decRefund = runDecimatorJackpot(decPool)  // unclaimed returns
    remaining = available - decPool + decRefund

    // Step 4: Terminal jackpot to NEXT-level ticketholders
    termPaid = runTerminalJackpot(remaining, currentLevel + 1)
    vaultSweep = remaining - termPaid

    // Step 5: Any remainder to vault 50/50 (vault + DGNRS)

    return {
        deityRefund: deityRefunded,
        decimator: decPool - decRefund,
        terminalJackpot: termPaid,
        vaultSweep: vaultSweep
    }
```

## Common Pitfalls

> **Pitfall 1: Terminal Jackpot Uses lvl + 1, Not Current Level.**
> `runTerminalJackpot(remaining, lvl + 1, rngWord)` pays next-level ticketholders. Players with only current-level participation receive nothing from the 90% terminal pool. Only whale bundle holders, lazy pass holders, and lootbox future ticket holders with next-level tickets benefit (DegenerusGameGameOverModule.sol:151).

> **Pitfall 2: Deity Pass Does Not Stack Bundle Bonus.**
> When `hasDeityPass` is true, the code follows the deity branch (DegenerusGame.sol:2412) which gives a fixed 7,500 BPS for streak+count and adds 8,000 BPS deity bonus. It does NOT also add the 1,000/4,000 BPS bundle bonus, even if the player has an active whale pass (DegenerusGame.sol:2450-2458).

> **Pitfall 3: Pass Floors Are Conditional on Bundle Type.**
> The `passActive` check requires BOTH `frozenUntilLevel > currentLevel` AND `bundleType in {1, 3}`. A player with `bundleType == 0` (no bundle) or `bundleType == 2` does not receive floor guarantees even if `frozenUntilLevel > currentLevel` (DegenerusGame.sol:2406-2407).

> **Pitfall 4: Mint Streak Returns 0 on Gap Even If Stored Value Is Nonzero.**
> `_mintStreakEffective` returns 0 when `currentMintLevel > lastCompleted + 1`, regardless of the stored streak value. The actual stored streak only resets on the next `_recordMintStreakForLevel` call. This means a player's displayed streak may differ from their effective streak (DegenerusGameMintStreakUtils.sol:57).

> **Pitfall 5: Safety Valve Resets levelStartTime, Extending Death Clock.**
> If the liveness timeout fires but `nextPool >= levelTarget` for level > 0, the game does NOT enter game over. Instead `levelStartTime` resets to current timestamp, giving another full 120 days. This can repeat indefinitely as long as the target is met (DegenerusGameAdvanceModule.sol:401-403).

> **Pitfall 6: Distress Ticket Bonus Is Proportional, Not Flat.**
> The 25% ticket bonus applies proportionally to the fraction of lootbox ETH bought during distress mode, not as a flat 25% to all tickets. A player who bought 80% of their lootbox value during normal mode and 20% during distress gets only `tickets * 0.20 * 0.25 = 5%` bonus tickets (DegenerusGameLootboxModule.sol:955-956).

> **Pitfall 7: Final Sweep Forfeits ALL Unclaimed Winnings.**
> `handleFinalSweep` sets `claimablePool = 0`, forfeiting all unclaimed balances -- including terminal jackpot winnings, decimator prizes, and any pre-game-over claimable ETH not yet withdrawn. The 30-day window is absolute (DegenerusGameGameOverModule.sol:178).
