## Executive Summary

This audit covers the complete sDGNRS/DGNRS split -- the largest code delta since v1.1. The split refactored the single DegenerusStonk soulbound token into a dual-contract architecture: StakedDegenerusStonk (sDGNRS, 520 lines, soulbound, holds ETH/stETH/BURNIE reserves with 5 reward pools) and DegenerusStonk (DGNRS, 177 lines, transferable ERC20 wrapper for the 20% creator allocation). The audit scope included the two core contracts, their interface, and all 30 game module callsites across 8 consumer contracts that interact with sDGNRS pool functions.

**Overall Assessment: SOUND.** The sDGNRS/DGNRS architecture is correctly implemented. All 5 external call paths in the burn function follow CEI (Checks-Effects-Interactions) patterns. The cross-contract supply invariant `sDGNRS.balanceOf[DGNRS] >= DGNRS.totalSupply` was formally proven across all 6 modification paths. All 30 game-to-sDGNRS callsites use correct Pool enum values, correct addresses, and handle return values safely. All BPS/PPM constants across 7 consumer contracts are verified with correct denominators. No Critical, High, or Medium findings were identified. One Low finding (DGNRS transfer-to-self token lock) and 4 Informational findings were documented.

The sDGNRS/DGNRS split does not invalidate any findings from the prior v1.0-v1.2 audit. The pre-existing M-02 finding (Admin + VRF failure) remains valid and unchanged. All 56 prior requirements remain PASS.

## All Findings

### DELTA-L-01: DGNRS transfer-to-self-contract token lock

**Severity:** Low
**Contract:** DegenerusStonk.sol
**Requirement:** DELTA-02
**Line(s):** 156-166 (`_transfer` function)

**Description:** The `_transfer` function checks `to != address(0)` (line 157) but does not check `to != address(this)`. A user could call `transfer(DGNRS_ADDRESS, amount)` or `transferFrom(from, DGNRS_ADDRESS, amount)`, sending DGNRS tokens to the contract itself. Since DGNRS has no sweep function and no mechanism to spend tokens in its own balance, these tokens are permanently lost.

**Impact:** Requires explicit user action (sending tokens to the contract address). No protocol funds at risk. This is a common ERC20 pattern -- most tokens do not guard against this.

**Status:** Acknowledged

---

### DELTA-I-01: Stale poolBalances after burnRemainingPools

**Severity:** Informational
**Contract:** StakedDegenerusStonk.sol
**Requirement:** DELTA-01
**Line(s):** 359-367 (`burnRemainingPools`)

**Description:** After `burnRemainingPools()` sets `balanceOf[address(this)] = 0` and decrements `totalSupply`, the individual `poolBalances[]` entries retain stale nonzero values. If `transferFromPool` were called after this point, the unchecked `balanceOf[address(this)] -= amount` at line 326 would underflow.

**Impact:** Not exploitable. `burnRemainingPools` is only called during game over (DegenerusGameGameOverModule.sol line 163). After `gameOver = true` (line 112), all game entry points that call `transferFromPool` check `if (gameOver) revert E()` and revert. No path exists to call `transferFromPool` after `burnRemainingPools`.

**Status:** Acknowledged (defensive zeroing recommended but not required)

### DELTA-I-02: Stray ETH locked in DGNRS contract

**Severity:** Informational
**Contract:** DegenerusStonk.sol
**Requirement:** DELTA-02
**Line(s):** 79 (`receive() external payable {}`)

**Description:** The DGNRS contract's `receive()` function accepts ETH from any sender. ETH sent outside the burn-through flow is permanently locked. There is no sweep or recovery function.

**Impact:** None on protocol security. Stray ETH does not affect burn calculations (sDGNRS calculates payouts from its own reserves, not DGNRS balance). The locked ETH does not benefit or harm any party.

**Status:** Acknowledged

---

### DELTA-I-03: previewBurn may show slightly different results than actual burn

**Severity:** Informational
**Contract:** StakedDegenerusStonk.sol
**Requirement:** DELTA-01
**Line(s):** 379-441 (`burn`), 454-481 (`previewBurn`)

**Description:** `burn()` calculates `totalValueOwed` using `totalMoney` which includes `claimableEth`, but the ETH-preferential payout logic only considers the post-claim `ethBal`. If `claimWinnings` is not triggered (because `totalValueOwed <= ethBal`), claimable ETH is not included in the actual payout. This means `previewBurn` may show slightly different ETH/stETH split than actual burn when claimableEth is nonzero and totalValueOwed is within the available ETH balance.

**Impact:** By design. The claimable ETH remains as reserves for future burners. No fund loss or accounting error.

**Status:** By Design

---

### DELTA-I-04: Stale comment in earlybird dump logic

**Severity:** Informational
**Contract:** DegenerusGameStorage.sol
**Requirement:** DELTA-07
**Line(s):** 1086

**Description:** Comment at line 1086 says "dump remaining earlybird pool into reward pool" but the code at line 1098 correctly transfers to `Pool.Lootbox`. The comment was not updated when the destination was changed from Reward to Lootbox.

**Impact:** None. The code is correct; only the comment is misleading. No functional impact.

**Status:** Acknowledged

## Open Questions Resolved

All 4 open questions from 19-RESEARCH.md were resolved in the core contracts audit (v2.0-delta-core-contracts.md):

1. **sDGNRS.burn() reentrancy via claimWinnings** -- **RESOLVED: NO reentrancy risk.** Call path: sDGNRS -> game.claimWinnings(address(0)) -> _claimWinningsInternal(sDGNRS, false) -> _payoutWithStethFallback -> ETH sent to sDGNRS `receive()`. The `receive()` function (line 282) only emits a `Deposit` event -- no state mutation. The game's CEI (sentinel `claimableWinnings[sDGNRS] = 1` before ETH send) prevents double-claim. State changes in sDGNRS are committed at lines 398-401 before the call. **SAFE.**

2. **coinflip.claimCoinflips re-entry risk during burn** -- **RESOLVED: NO re-entry risk.** Call path: sDGNRS -> BurnieCoinflip.claimCoinflips -> _claimCoinflipsAmount -> _claimCoinflipsInternal (storage reads + game view calls) -> burnie.mintForCoinflip(sDGNRS, toClaim). BurnieCoin.mintForCoinflip is a standard ERC20 mint -- no callbacks, no ERC-777 hooks. **SAFE.**

3. **DGNRS receive() safety** -- **RESOLVED: Stray ETH is harmless.** Anyone can send ETH to DGNRS via its `receive() external payable {}`. This ETH is permanently locked -- DGNRS only forwards the exact `ethOut` returned by `stonk.burn()`, not its full balance. Stray ETH does not affect sDGNRS reserve calculations. See DELTA-I-02.

4. **Pool balance consistency after burnRemainingPools** -- **RESOLVED: Stale values are not exploitable.** After `burnRemainingPools()`, `poolBalances[]` entries retain stale values while `balanceOf[address(this)] = 0`. The `transferFromPool` function would underflow if called after this point. However, all game entry points check `if (gameOver) revert E()`, making the stale values unreachable. The invariant `balanceOf[address(this)] == sum(poolBalances[i])` holds at all times before `burnRemainingPools`. See DELTA-I-01.
