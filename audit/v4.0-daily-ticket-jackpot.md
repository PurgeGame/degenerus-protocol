## Section 2: _distributeTicketJackpot (JM:1105) -- Core Function

**Source:** `contracts/modules/DegenerusGameJackpotModule.sol:1105-1138`

### Signature

```solidity
function _distributeTicketJackpot(
    uint24 lvl,                    // Source level for trait ticket pool selection
    uint32 winningTraitsPacked,    // 4 trait IDs packed by JackpotBucketLib
    uint256 ticketUnits,           // Total ticket units to distribute
    uint256 entropy,               // VRF-derived entropy for winner selection
    uint16 maxWinners,             // Maximum total winners across all buckets
    uint8 saltBase                 // Per-call entropy salt to differentiate calls
) private
```

### Execution Flow

| Step | Line | Action |
|------|------|--------|
| 1 | JM:1113 | **Early exit:** `if (ticketUnits == 0) return` -- zero-budget distributions are silently skipped |
| 2 | JM:1115-1117 | **Trait unpacking:** `uint8[4] memory traitIds = JackpotBucketLib.unpackWinningTraits(winningTraitsPacked)` -- extracts 4 trait IDs from packed uint32 |
| 3 | JM:1118-1119 | **Cap computation:** `cap = maxWinners; if (ticketUnits < cap) cap = uint16(ticketUnits)` -- winners capped at min(maxWinners, ticketUnits) to prevent fractional distribution |
| 4 | JM:1121-1126 | **Bucket sizing:** `(uint16[4] memory counts, uint8 activeCount) = _computeBucketCounts(lvl, traitIds, cap, entropy)` -- determines how many winners per trait bucket |
| 5 | JM:1127 | **Early exit:** `if (activeCount == 0) return` -- no active buckets means no distribution possible |
| 6 | JM:1129-1137 | **Distribution:** `_distributeTicketsToBuckets(lvl, traitIds, counts, ticketUnits, entropy, cap, saltBase)` -- distributes to individual buckets |

### Observations

- Two early exits ensure no-op for zero budgets or empty trait pools
- `cap` prevents more winners than ticket units (each winner gets at least 1 unit when `extra` rounds correctly)
- `maxWinners` comes from caller: always `LOOTBOX_MAX_WINNERS` (100) for all 3 current callers

## Section 3: _computeBucketCounts (JM:1222) -- Bucket Sizing

**Source:** `contracts/modules/DegenerusGameJackpotModule.sol:1222-1274`

### Signature


### Phase 1: Active Bucket Detection (JM:1228-1247)

For each of 4 traits (JM:1229-1247):

1. **Direct check** (JM:1231): `traitBurnTicket[lvl][trait].length != 0` -- are there real ticket holders?
2. **Deity fallback** (JM:1232-1237): If no direct entries, check `fullSymId < 32 && deityBySymbol[fullSymId] != address(0)` where `fullSymId = (trait >> 6) * 8 + (trait & 0x07)` (JM:1233)
3. **Mark active** (JM:1238-1242): `activeMask |= uint8(1 << i)` and `++activeCount`

### Phase 2: Count Distribution (JM:1248-1273)

| Step | Line | Action |
|------|------|--------|
| 1 | JM:1248 | Early exit if `activeCount == 0` |
| 2 | JM:1250 | Base count: `baseCount = maxWinners / activeCount` |
| 3 | JM:1251 | Remainder: `remainder = maxWinners - baseCount * activeCount` |
| 4 | JM:1253-1260 | Assign `baseCount` to each active bucket |
| 5 | JM:1262-1273 | Distribute remainder round-robin starting at entropy-derived index: `idx = uint8(entropy & 3)` (JM:1263) |

### Observations

- Remainder distribution uses entropy for starting index to prevent first-bucket bias
- Round-robin wraps with `idx = uint8((idx + 1) & 3)` (JM:1271) -- cycles through all 4 positions
- Non-active buckets receive 0 count and are skipped during round-robin remainder assignment

## Section 4: _distributeTicketsToBuckets (JM:1141) -- Bucket Loop

**Source:** `contracts/modules/DegenerusGameJackpotModule.sol:1141-1175`

### Signature

```solidity
function _distributeTicketsToBuckets(
    uint24 lvl,
    uint8[4] memory traitIds,
    uint16[4] memory counts,
    uint256 ticketUnits,
    uint256 entropy,
    uint16 cap,
    uint8 saltBase
) private
```

### Execution Flow

| Step | Line | Action |
|------|------|--------|
| 1 | JM:1150 | **Base units:** `baseUnits = ticketUnits / cap` -- floor division, every winner gets at least this many units |
| 2 | JM:1151 | **distParams packing:** `(ticketUnits % cap) | ((entropy % cap) << 128)` -- low 128 bits = remainder count (`extra`); high bits = rotation offset for fairness |
| 3 | JM:1152 | **globalIdx:** initialized to 0, tracks position across all buckets for round-robin extra distribution |
| 4 | JM:1154-1174 | **Loop over 4 traits:** for each non-zero `counts[traitIdx]`, generates per-trait entropy via `EntropyLib.entropyStep(entropy ^ (uint256(traitIdx) << 64) ^ ticketUnits)` (JM:1156-1157), then calls `_distributeTicketsToBucket` with `salt = saltBase + traitIdx` (JM:1164) |

### Unit Distribution Design

The `distParams` packing is a compact representation of the round-robin remainder system:

- `extra = ticketUnits % cap`: number of winners who get +1 unit (the remainder from integer division)
- `offset = entropy % cap`: rotation starting point so the +1 extras are distributed fairly across multiple calls
- `globalIdx` tracks position across buckets, ensuring remainder distribution is continuous and fair

## Section 5: _distributeTicketsToBucket (JM:1178) -- Per-Bucket Distribution

**Source:** `contracts/modules/DegenerusGameJackpotModule.sol:1178-1219`

### Signature

```solidity
function _distributeTicketsToBucket(
    uint24 lvl,
    uint8 traitId,
    uint16 count,
    uint256 entropy,
    uint8 salt,
    uint256 baseUnits,
    uint256 distParams,
    uint16 cap,
    uint256 startIdx
) private returns (uint256 endIdx)
```

### Execution Flow

| Step | Line | Action |
|------|------|--------|
| 1 | JM:1189 | **Winner cap:** `if (count > MAX_BUCKET_WINNERS) count = MAX_BUCKET_WINNERS` where `MAX_BUCKET_WINNERS = 250` (JM:183). This caps per-bucket winners even if _computeBucketCounts assigned more. |
| 2 | JM:1190-1196 | **Winner selection:** `winners = _randTraitTicket(traitBurnTicket[lvl], entropy, traitId, uint8(count), salt)` -- selects `count` winners from the trait-level pool |
| 3 | JM:1198-1199 | **Unpack distParams:** `extra = distParams & type(uint128).max` and `offset = distParams >> 128` |
| 4 | JM:1201 | **Cursor init:** `cursor = (startIdx + offset) % cap` -- position in the round-robin for +1 distribution |
| 5 | JM:1202-1217 | **Distribution loop:** for each winner: |
| 5a | JM:1204 | `units = baseUnits` -- every winner gets at least the base amount |
| 5b | JM:1205-1206 | `if (extra != 0 && cursor < extra) units += 1` -- winners at cursor positions below `extra` get +1 |
| 5c | JM:1208-1209 | **CRITICAL: Null skip + ticket queuing:** `if (winner != address(0) && units != 0) _queueTickets(winner, lvl + 1, uint32(units))` |
| 5d | JM:1212-1213 | **Cursor advance:** `++cursor; if (cursor == cap) cursor = 0` -- wraps around |
| 5e | JM:1214 | **Global index advance:** `++startIdx` -- tracks position for next bucket |

### Critical: Target Level is lvl + 1

At JM:1209, ticket winners receive tickets for `lvl + 1`, NOT `lvl`. This means:

- Winners are selected from `traitBurnTicket[lvl]` (the source level's trait ticket holders)
- But receive tickets for the NEXT level (`lvl + 1`)
- This is consistent with the protocol's forward-looking ticket design: ticket holders at level N win tickets for level N+1

### Null Winner Handling

The `if (winner != address(0) && units != 0)` guard at JM:1208 means:

- `address(0)` entries in `traitBurnTicket` storage arrays are silently skipped
- Zero-unit distributions are also skipped (can occur if `baseUnits == 0` and cursor is past `extra`)
- Skipped winners still consume cursor/index positions, so the round-robin remains fair

## Section 6: _randTraitTicket (JM:2237) -- Winner Selection Mechanism

**Source:** `contracts/modules/DegenerusGameJackpotModule.sol:2237-2280`

### Signature

```solidity
function _randTraitTicket(
    address[][256] storage traitBurnTicket_,
    uint256 randomWord,
    uint8 trait,
    uint8 numWinners,
    uint8 salt
) private view returns (address[] memory winners)
```

### Execution Flow

| Step | Line | Action |
|------|------|--------|
| 1 | JM:2244 | **Source array:** `holders = traitBurnTicket_[trait]` -- storage reference to trait-level ticket holders |
| 2 | JM:2245 | **Array length:** `len = holders.length` |
| 3 | JM:2250 | **Deity virtual entry computation:** `fullSymId = (trait >> 6) * 8 + (trait & 0x07)` -- maps trait ID to deity symbol index |
| 4 | JM:2253-2258 | **Deity eligibility:** if `fullSymId < 32` AND `deityBySymbol[fullSymId] != address(0)`: `virtualCount = max(len / 50, 2)` (JM:2256-2257). This gives the deity a 2% weight floor (min 2 virtual entries). |
| 5 | JM:2261 | **Effective length:** `effectiveLen = len + virtualCount` |
| 6 | JM:2262-2264 | **Empty check:** if `effectiveLen == 0 || numWinners == 0` return empty array |
| 7 | JM:2268-2270 | **Entropy derivation:** `slice = randomWord ^ (uint256(trait) << 128) ^ (uint256(salt) << 192)` -- unique per trait+salt combination |
| 8 | JM:2272 | **Index selection:** `idx = slice % effectiveLen` |
| 9 | JM:2273 | **Winner resolution:** `winners[i] = idx < len ? holders[idx] : deity` -- if index falls in virtual range, deity wins |
| 10 | JM:2277 | **Entropy rotation:** `slice = (slice >> 16) | (slice << 240)` -- bit rotation for subsequent winners |

### Deity Virtual Entry Mechanism

The deity pass system injects virtual entries into the selection pool:

- **Eligibility:** Only symbols with `fullSymId < 32` (first 32 of the symbol space) can have deities
- **Virtual count:** `max(len / 50, 2)` -- at least 2 entries, scaling to 2% of pool at large sizes
- **Selection:** If the random index lands in `[len, effectiveLen)`, the deity address wins
- **Implication:** For a pool with 100 holders, `virtualCount = 2`, so deity has 2/102 = ~1.96% chance per slot
- **Implication:** For a pool with 200 holders, `virtualCount = 4`, so deity has 4/204 = ~1.96% chance per slot (stable 2%)
- **Edge case:** For an empty pool (`len == 0`) with a deity, `virtualCount = 2`, `effectiveLen = 2` -- deity wins 100% of selections

### Duplicate Winner Behavior [NF-01]

`_randTraitTicket` does NOT deduplicate winners. The same address can appear multiple times in the returned array because:

1. Bit rotation (`slice >> 16 | slice << 240`) at JM:2277 produces different indices but can map to the same holder
2. No `seen` tracking is implemented
3. **Design rationale:** This is gas-efficient and proportionally fair -- addresses appearing multiple times in `traitBurnTicket` (from multiple ticket purchases) have proportionally higher win probability, and duplicate wins simply mean more ticket units queued for that address

## Section 8: Budget Computation Chain

### 8.1 _budgetToTicketUnits (JM:1063-1070)

**Source:** `contracts/modules/DegenerusGameJackpotModule.sol:1063-1070`

```solidity
function _budgetToTicketUnits(
    uint256 budget,
    uint24 lvl
) private pure returns (uint256) {
    if (budget == 0) return 0;
    uint256 ticketPrice = PriceLookupLib.priceForLevel(lvl);
    return ticketPrice == 0 ? 0 : (budget << 2) / ticketPrice;
}
```

**Analysis:**
- `ticketPrice = PriceLookupLib.priceForLevel(lvl)` at JM:1068 -- pure function lookup from `PriceLookupLib.sol:21`
- `(budget << 2) / ticketPrice` = `(budget * 4) / ticketPrice` -- tickets cost 1/4 of the standard ticket price for jackpot rewards
- Returns 0 if either `budget == 0` (JM:1067) or `ticketPrice == 0` (JM:1069) -- both are safe zero-division guards
- The `<< 2` multiplier effectively means jackpot ticket rewards are 4x more generous than purchase-based tickets per ETH unit

### 8.2 _packDailyTicketBudgets (JM:2753-2764)

```solidity
function _packDailyTicketBudgets(
    uint8 counterStep,
    uint256 dailyTicketUnits,
    uint256 carryoverTicketUnits,
```

## Section 10: Findings

### [NF-01] INFO: Duplicate Winners from _randTraitTicket

**Found during:** Section 6 analysis of _randTraitTicket (JM:2237)

**Issue:** `_randTraitTicket` does not deduplicate selected winners. The bit-rotation entropy at JM:2277 (`slice = (slice >> 16) | (slice << 240)`) produces different raw values but `% effectiveLen` can map to the same index, resulting in the same address appearing multiple times in the winners array.

**Impact:** A single address may receive tickets from multiple winner slots in the same bucket. This is proportionally fair since addresses with more entries in `traitBurnTicket` have proportionally higher selection probability. The duplicate-win scenario simply means more tickets queued via separate `_queueTickets` calls.

**Assessment:** Intentional design trade-off -- deduplication would require O(n) storage writes or O(n^2) comparisons, both costly in gas. The current approach is gas-efficient and statistically fair.

**Severity:** INFO -- no correctness or security concern.
