### 3.1 Guard-evaluation walk (the formal closure)

The ternary at L185 returns 0 iff `lastPurchase = T ∧ rngLockedFlag = T ∧ lvl = 0`. We prove this conjunction is unreachable at any advanceGame entry point.

**Step 1 — `lvl = 0` requires bootstrap pre-state per INV-PLV-C-01:**

- Sole-writer grep: `git grep -nE '\blevel\s*=\|\blevel\s*+=\|\blevel\+\+\|\+\+level' acd88512 -- 'contracts/storage/*.sol' 'contracts/modules/*.sol' 'contracts/*.sol'` returns:

  - All other matches are read-only comparisons (`level == 0`), local-shadow declarations (GNRUS:367/416), or unrelated identifiers (`++levelQuestVersion`).
- Therefore `lvl = level = 0` holds ONLY before the first call to `_finalizeRngRequest` where `isTicketJackpotDay && !isRetry` was true. Once L1616 fires with `lvl ≥ 1`, the storage `level ≥ 1` sticks (no decrementing writer exists).

**Step 2 — `rngLockedFlag = T` at advanceGame entry requires `_finalizeRngRequest` has fired at least once per INV-PLV-A-01:**

- Sole T-writer of `rngLockedFlag`: AdvanceModule:1607 inside `_finalizeRngRequest` (`rngLockedFlag = true;`).
- F-writers of `rngLockedFlag`: AdvanceModule:1663 (inside `updateVrfCoordinatorAndSub` — admin path) + AdvanceModule:1704 (inside `_unlockRng` — paired with `dailyIdx = day` advance at L1703).
- Therefore `rngLockedFlag = T` is observable at advanceGame entry ONLY AFTER `_finalizeRngRequest` has fired and BEFORE `_unlockRng` (or admin override) has cleared it.

**Step 3 — Combine Step 1 and Step 2:**

- `lvl = 0` requires `_finalizeRngRequest` has NEVER fired with `isTicketJackpotDay && !isRetry = true` (the gate around L1616).
- `rngLockedFlag = T` requires `_finalizeRngRequest` has fired (the L1607 write is unconditional within the function body — runs whether `isTicketJackpotDay` is true or false).
- It IS possible for `_finalizeRngRequest` to have fired with `isTicketJackpotDay = false` (so L1607 wrote rngLocked=T but L1616 was skipped). In that case `rngLocked=T ∧ lvl=0` co-occur. **However**, that is the only path that produces this combination, and it does so with `lastPurchaseDay = false` at the prior advanceGame entry (since `_finalizeRngRequest` is called from `_requestRng` which is invoked through `rngGate` only after the daily processing path; the `isTicketJackpotDay` parameter to `_finalizeRngRequest` reflects the caller's `lastPurchase` value at the time of request, so `isTicketJackpotDay = false` ⇒ `lastPurchase` was false at the request).
- Therefore the only path to `rngLocked=T ∧ lvl=0` leaves `lastPurchaseDay = false` at the moment rngLocked is set. The remaining question: can the next advanceGame entry then transition `lastPurchaseDay` to true while `rngLocked=T ∧ lvl=0` still hold? That requires Step 4.

**Step 4 — The L173 turbo guard prevents the only path that flips `lastPurchaseDay` to T while `rngLocked=T`:**

- The two T-writers of `lastPurchaseDay` are:
  - AdvanceModule:178 inside the turbo block at L167-181 (the SOLE writer that runs in a fresh advanceGame entry where `lastPurchaseDay` was previously false).
  - AdvanceModule:399 inside the daily-jackpot region L370-407, gated by `if (!lastPurchaseDay)` at L371 AND `if (targetMet)` at L398. The `!lastPurchaseDay` gate at L371 means this writer can only fire when `lastPurchaseDay` is currently false; AND L404 `_unlockRng(day);` runs unconditionally inside the same call BEFORE control exits the daily-jackpot region (L406 `break;`), so by the time the next advanceGame entry observes `lastPurchaseDay = T`, `_unlockRng` has already cleared `rngLockedFlag = false` (at L1704). Therefore the L399 writer cannot leave `(lastPurchaseDay=T ∧ rngLockedFlag=T)` observable at any subsequent advanceGame entry.
- The L178 writer is gated by L173 `if (!inJackpot && !lastPurchaseDay && !rngLockedFlag)`. The third conjunct `!rngLockedFlag` is the WIP fix: it short-circuits the turbo block when rngLocked=T, preventing the `lastPurchaseDay = true` write at L178 from firing while rngLocked=T.
- Therefore there is NO writer of `lastPurchaseDay = true` that can produce `(lastPurchaseDay = T ∧ rngLockedFlag = T)` observable at an advanceGame entry once the L173 guard is in place.

**Conclusion:** The ternary at L185 cannot return 0 at any reachable advanceGame entry. The conjunction `(lastPurchase = T ∧ rngLockedFlag = T ∧ lvl = 0)` is unreachable: Step 4 closes the only observable path that could co-locate `lastPurchase=T` with `rngLocked=T`, and Steps 1-3 establish that even if Step 4 were violated by a future code change, the underlying `lvl=0 ∧ rngLocked=T` co-occurrence requires a specific `_finalizeRngRequest` calling pattern with `isTicketJackpotDay = false`, which itself implies `lastPurchase` was false at request time.
