### New Findings

### CMT-V32-001: PlayerCredited event missing @param recipient NatSpec tag

- **What:** The `PlayerCredited` event at line 53 has three parameters (`address indexed player, address indexed recipient, uint256 amount`) but only two `@param` tags are documented (lines 51-52): `@param player` and `@param amount`. The `@param recipient` tag is missing. A NatSpec-compliant documentation parser would flag this as an incomplete parameter list.
- **Where:** DegenerusGameLootboxModule.sol:50-53
- **Why:** A warden reading the event NatSpec would not know what the `recipient` parameter represents or how it differs from `player`. In PayoutUtils (the canonical emitter), both `player` and `recipient` are set to the same value (`beneficiary`), but the two-field pattern suggests they can diverge. Without documenting `recipient`, a warden might misinterpret the event schema when building payout analytics.
- **Suggestion:** Add `/// @param recipient The address receiving the credited ETH (may differ from player for delegated claims).` between the existing @param lines, or if player and recipient are always identical in practice, document that: `/// @param recipient The recipient address (always equal to player in current implementation).`
- **Category:** comment-inaccuracy
- **Severity:** INFO

### New Findings

### CMT-V32-002: wireVrf function missing @param keyHash_ NatSpec tag

- **What:** The `wireVrf` function at line 391 has three parameters (`address coordinator_`, `uint256 subId`, `bytes32 keyHash_`) but only two `@param` tags are documented (lines 389-390): `@param coordinator_` and `@param subId`. The `@param keyHash_` tag is missing despite keyHash_ being a critical VRF configuration value stored as `vrfKeyHash` (line 401).
- **Where:** DegenerusGameAdvanceModule.sol:386-394
- **Why:** A warden auditing VRF configuration would see only 2 documented parameters for a 3-parameter function. Without documentation on `keyHash_`, a warden might not immediately recognize that this single function sets the full VRF configuration triplet (coordinator, subscription, gas lane). The key hash determines which gas lane is used for VRF requests, which affects confirmation times and costs.
- **Suggestion:** Add `/// @param keyHash_ VRF key hash for the gas lane (determines confirmation timing and cost).` after the existing @param lines.
- **Category:** comment-inaccuracy
- **Severity:** INFO
