## 1. Constants

### TICKET_SLOT_BIT (GS:154)

```solidity
// Source: contracts/storage/DegenerusGameStorage.sol:154
uint24 internal constant TICKET_SLOT_BIT = 1 << 23;
```

- **Value:** `0x800000` (8,388,608 decimal)
- **Purpose:** Distinguishes write/read buffer slots in the double-buffer system. When set (bit 23 = 1), the key targets Slot 1. When clear (bit 23 = 0), the key targets Slot 0.
- **NatSpec (GS:151-153):** "Bit mask for ticket queue double-buffer key encoding. Set bit 23 of the uint24 level key to distinguish write/read slots. Max real level: 2^22 - 1 = 4,194,303 (game would take millennia)."

### TICKET_FAR_FUTURE_BIT (GS:162)

```solidity
// Source: contracts/storage/DegenerusGameStorage.sol:162
uint24 internal constant TICKET_FAR_FUTURE_BIT = 1 << 22;
```

- **Value:** `0x400000` (4,194,304 decimal)
- **Purpose:** Creates a third key space for tickets targeting > currentLevel + 6. Independent of the double-buffer toggle. Introduced in v3.9 Phase 74.
- **NatSpec (GS:156-161):** "Bit mask for far-future ticket key encoding. Set bit 22 of the uint24 level key to create a third key space disjoint from both double-buffer slots (bit 23). Far-future = tickets targeting > currentLevel + 6. Three key spaces: Slot0 [0x000000-0x3FFFFF], FF [0x400000-0x7FFFFF], Slot1 [0x800000-0xBFFFFF]. Disjoint for all lvl < 2^22."

## 2. Key Encoding Formulas

### _tqWriteKey (GS:686-688)

```solidity
// Source: contracts/storage/DegenerusGameStorage.sol:684-688
/// @dev Compute the ticket queue key for the write slot.
///      Slot 0 uses raw level, slot 1 sets bit 23.
function _tqWriteKey(uint24 lvl) internal view returns (uint24) {
    return ticketWriteSlot != 0 ? lvl | TICKET_SLOT_BIT : lvl;
}
```

- **Visibility:** `internal view` (reads storage: `ticketWriteSlot`)
- **When ticketWriteSlot = 0:** returns `lvl` (raw level, Slot 0)
- **When ticketWriteSlot = 1:** returns `lvl | 0x800000` (Slot 1)
- **Purpose:** Directs new ticket writes to the active write buffer.

### _tqReadKey (GS:691-693)

```solidity
// Source: contracts/storage/DegenerusGameStorage.sol:690-693
/// @dev Compute the ticket queue key for the read slot (opposite of write).
function _tqReadKey(uint24 lvl) internal view returns (uint24) {
    return ticketWriteSlot == 0 ? lvl | TICKET_SLOT_BIT : lvl;
}
```

- **Visibility:** `internal view` (reads storage: `ticketWriteSlot`)
- **When ticketWriteSlot = 0:** returns `lvl | 0x800000` (Slot 1 -- opposite of write)
- **When ticketWriteSlot = 1:** returns `lvl` (Slot 0 -- opposite of write)
- **Purpose:** Directs ticket reads/processing to the buffer opposite the write buffer.

### _tqFarFutureKey (GS:699-701)

```solidity
// Source: contracts/storage/DegenerusGameStorage.sol:695-701
/// @dev Compute the ticket queue key for the far-future key space.
///      Always sets bit 22, independent of ticketWriteSlot.
///      Far-future tickets are not double-buffered; they persist until
///      drained by processFutureTicketBatch.
function _tqFarFutureKey(uint24 lvl) internal pure returns (uint24) {
    return lvl | TICKET_FAR_FUTURE_BIT;
}
```

- **Visibility:** `internal pure` (NO storage read -- independent of `ticketWriteSlot`)
- **Returns:** `lvl | 0x400000` always
- **Purpose:** Routes far-future tickets (targetLevel > currentLevel + 6) to a stable third key space. Not affected by double-buffer swaps.

## 3. Three Disjoint Key Spaces

### Key Space Ranges

For any valid level value `lvl` where `lvl < 2^22` (4,194,303):

```
Key Space 0 (Slot 0): [0x000000, 0x3FFFFF]  -- raw lvl, bits 22-23 both 0
Key Space 1 (FF):     [0x400000, 0x7FFFFF]  -- lvl | TICKET_FAR_FUTURE_BIT, bit 22 = 1, bit 23 = 0
Key Space 2 (Slot 1): [0x800000, 0xBFFFFF]  -- lvl | TICKET_SLOT_BIT, bit 22 = 0, bit 23 = 1
```

### Disjointness Proof

For any `lvl < 2^22`:
1. **Slot 0 key** = `lvl` -- bit 22 = 0, bit 23 = 0
2. **FF key** = `lvl | (1 << 22)` -- bit 22 = 1, bit 23 = 0
3. **Slot 1 key** = `lvl | (1 << 23)` -- bit 22 = 0, bit 23 = 1

The two discriminator bits (22 and 23) create four possible combinations:

| Bit 23 | Bit 22 | Key Space | Used? |
|--------|--------|-----------|-------|
| 0 | 0 | Slot 0 | Yes |
| 0 | 1 | Far-Future (FF) | Yes |
| 1 | 0 | Slot 1 | Yes |
| 1 | 1 | (unused: 0xC00000-0xFFFFFF) | No |

Since the raw level `lvl` has bit 22 = 0 (given `lvl < 2^22`), the OR operation sets exactly one bit without interference. **No two key spaces can produce the same uint24 value for any valid level.**

### Maximum Valid Level

- `2^22 - 1 = 4,194,303`
- Practical game maximum: ~1000 levels (game would take millennia to reach 4M levels)
- The restriction from 2^23 - 1 (8,388,607) to 2^22 - 1 was introduced in v3.9 Phase 74 to accommodate the FF bit

## 4. ticketWriteSlot State Machine

### State A: ticketWriteSlot = 0

```
_tqWriteKey(lvl) = lvl                  (writes go to Slot 0)
_tqReadKey(lvl)  = lvl | 0x800000       (reads come from Slot 1)
_tqFarFutureKey(lvl) = lvl | 0x400000   (unchanged)
```

### State B: ticketWriteSlot = 1

```
_tqWriteKey(lvl) = lvl | 0x800000       (writes go to Slot 1)
_tqReadKey(lvl)  = lvl                  (reads come from Slot 0)
_tqFarFutureKey(lvl) = lvl | 0x400000   (unchanged)
```

### State Machine Invariant

At any given time:
- One slot accumulates new tickets (write buffer)
- The other slot is being processed by jackpot draws and ticket batch processing (read buffer)
- The far-future key space is **independent** of this toggle -- it stores tickets for levels > currentLevel + 6, which persist until explicitly drained by `processFutureTicketBatch`

### Transition

`ticketWriteSlot` is modified in exactly one location:

```solidity
// Source: contracts/storage/DegenerusGameStorage.sol:712
ticketWriteSlot ^= 1;
```

This is inside `_swapTicketSlot` (GS:709-714). The XOR toggle switches between State A (0) and State B (1). No other code modifies `ticketWriteSlot`.

## 6. _swapTicketSlot Function (TKT-06)

```solidity
// Source: contracts/storage/DegenerusGameStorage.sol:707-714
/// @dev Swap the active ticket queue buffer. Reverts if read slot is not drained.
///      Resets ticketsFullyProcessed to false for the new read slot.
function _swapTicketSlot(uint24 purchaseLevel) internal {
    uint24 rk = _tqReadKey(purchaseLevel);       // GS:710
    if (ticketQueue[rk].length != 0) revert E(); // GS:711
    ticketWriteSlot ^= 1;                        // GS:712
    ticketsFullyProcessed = false;                // GS:713
}
```

### Line-by-Line Analysis

1. **GS:710** `uint24 rk = _tqReadKey(purchaseLevel)` -- Compute the read key for the specified purchase level. This determines which buffer is currently the "read" buffer.

2. **GS:711** `if (ticketQueue[rk].length != 0) revert E()` -- **Precondition: the read buffer must be fully drained.** If any entries remain in the read-side queue, the swap is rejected with error `E()`. This prevents swapping before ticket processing is complete.

3. **GS:712** `ticketWriteSlot ^= 1` -- Flip the buffer toggle. If `ticketWriteSlot` was 0, it becomes 1; if 1, it becomes 0. This atomically swaps which slot is "write" and which is "read".

4. **GS:713** `ticketsFullyProcessed = false` -- Reset the processing flag. The newly-promoted read buffer (old write buffer) has unprocessed tickets, so the flag must be cleared to gate subsequent processing.

### Semantics

After swap:
- The old write buffer (containing tickets accumulated during the previous period) becomes the new read buffer
- The old read buffer (now empty, verified at GS:711) becomes the new write buffer
- New ticket purchases will now route to the freshly-emptied buffer
- Ticket processing will read from the buffer that was just being written to

## 7. _swapAndFreeze Function (TKT-06)

```solidity
// Source: contracts/storage/DegenerusGameStorage.sol:716-725
/// @dev Swap queue buffer AND activate prize pool freeze (daily RNG path only).
///      If not already frozen, zeros pending accumulators.
///      If already frozen (jackpot phase), accumulators keep growing.
function _swapAndFreeze(uint24 purchaseLevel) internal {
    _swapTicketSlot(purchaseLevel);       // GS:720
    if (!prizePoolFrozen) {               // GS:721
        prizePoolFrozen = true;           // GS:722
        prizePoolPendingPacked = 0;       // GS:723
    }
}
```

### Line-by-Line Analysis

1. **GS:720** `_swapTicketSlot(purchaseLevel)` -- Perform the buffer swap. All preconditions from _swapTicketSlot apply (read buffer must be drained).

2. **GS:721** `if (!prizePoolFrozen)` -- Check if pools are already frozen. This gate is critical for multi-day jackpot scenarios.

3. **GS:722** `prizePoolFrozen = true` -- Activate freeze. While frozen, purchase revenue is redirected from `prizePoolsPacked` to `prizePoolPendingPacked`.

4. **GS:723** `prizePoolPendingPacked = 0` -- Zero the pending accumulator. This ensures a clean slate for the new frozen period.

### Freeze Behavior

- **First freeze (fresh daily RNG):** prizePoolFrozen goes false->true. Pending accumulators zeroed. Live pools are now snapshot values for jackpot computation.
- **Re-freeze (multi-day jackpot):** prizePoolFrozen is already true, so the `if` block is skipped. Pending accumulators KEEP GROWING across jackpot days. This prevents loss of purchase revenue during multi-day jackpot phases (days 1-5).
- **Unfreeze:** Happens in `_unfreezePool` (GS:729-736) which merges pending into live and clears the flag.

## 10. Relationship to processTicketBatch (TKT-06)

**Note:** Full `processTicketBatch` trace is deferred to Phase 82 (TKT-08). This section documents interface points only.

### processTicketBatch (JM:1889-1950)

- Reads from `ticketQueue[_tqReadKey(lvl)]` at JM:1890
- Processes entries (mints traits) via `_processOneTicketEntry` loop
- When all entries processed (idx >= total): deletes the queue array (`delete ticketQueue[rk]`) at JM:1903/1944
- Returns `finished = true` when queue is fully drained

### processFutureTicketBatch (MM:298-452)

- Reads from `ticketQueue[_tqReadKey(lvl)]` (when not in FF phase) or `ticketQueue[_tqFarFutureKey(lvl)]` (when in FF phase) at MM:303
- After draining read-side queue, transitions to FF phase by setting `ticketLevel = lvl | TICKET_FAR_FUTURE_BIT` at MM:310/331/441
- Returns `finished = true` only when BOTH read-side and FF queues are fully drained for a level

### Interface Chain

```
processTicketBatch/processFutureTicketBatch drain read buffer
  -> ticketQueue[_tqReadKey(lvl)].length becomes 0
  -> ticketsFullyProcessed set to true (by advanceGame callers at AM:173/218/276)
  -> enables swap (_swapTicketSlot checks ticketQueue[rk].length at GS:711)
  -> _swapAndFreeze or _swapTicketSlot called
  -> ticketWriteSlot flips, ticketsFullyProcessed reset to false
  -> cycle repeats
```
