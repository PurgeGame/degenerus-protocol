## 1. Regular Decimator

### 1.1 Trigger Path

The regular decimator fires during level transitions at specific level milestones. The call chain:

```
advanceGame (AM)
  -> payDailyJackpotCoinAndTickets (when jackpotCounter >= JACKPOT_LEVEL_CAP=5)
    -> _awardFinalDayDgnrsReward(lvl, rngWord)     [AM:365]
    -> _rewardTopAffiliate(lvl)                     [AM:366]
    -> _runRewardJackpots(lvl, rngWord)             [AM:367]
       -> delegatecall to EndgameModule.runRewardJackpots [AM:516-527]
          -> IDegenerusGame(address(this)).runDecimatorJackpot(poolWei, lvl, rngWord)
             -> DecimatorModule.runDecimatorJackpot [DM:205]
```

`runRewardJackpots` at EM:168 dispatches the decimator at two distinct level patterns with different pool allocations:

**Levels x00 (century levels):** EM:205-214

```solidity
if (prevMod100 == 0) {                              // EM:205
    uint256 decPoolWei = (baseFuturePool * 30) / 100; // EM:206 -- 30% of ORIGINAL futurePrizePool
    if (decPoolWei != 0) {
        uint256 returnWei = IDegenerusGame(address(this))
            .runDecimatorJackpot(decPoolWei, lvl, rngWord);  // EM:208-209
        uint256 spend = decPoolWei - returnWei;              // EM:210
        futurePoolLocal -= spend;                            // EM:211
        claimableDelta += spend;                             // EM:212
    }
}
```

Key: uses `baseFuturePool` (captured at EM:170 before any deductions), so the 30% allocation is computed on the pre-BAF value.

**Levels x5 (not x95):** EM:220-231

```solidity
if (prevMod10 == 5 && prevMod100 != 95) {            // EM:220
    uint256 decPoolWei = (futurePoolLocal * 10) / 100; // EM:222 -- 10% of CURRENT futurePrizePool
    if (decPoolWei != 0) {
        uint256 returnWei = IDegenerusGame(address(this))
            .runDecimatorJackpot(decPoolWei, lvl, rngWord);  // EM:224-225
        uint256 spend = decPoolWei - returnWei;              // EM:226
        futurePoolLocal -= spend;                            // EM:227
        claimableDelta += spend;                             // EM:229
    }
}
```

Key: uses `futurePoolLocal` (the running balance, already reduced by BAF on x0 levels). The exclusion of x95 levels prevents double-dipping when the century-level special and regular decimator would overlap.

**Firing schedule:**
- Levels 5, 15, 25, 35, 45, 55, 65, 75, 85: 10% of current futurePrizePool
- NOT level 95 (excluded by `prevMod100 != 95`)
- Levels 100, 200, 300...: 30% of original (pre-BAF) baseFuturePool

**Level 100 double-fire analysis:** At level 100, `prevMod100 == 0` and `prevMod10 == 0`. The x5 branch requires `prevMod10 == 5`, so it does NOT fire. Only the 30% century allocation fires. At level 105, `prevMod10 == 5` and `prevMod100 == 5 != 95`, so the 10% branch fires normally. Correct behavior.

**Self-call pattern:** The dispatch uses `IDegenerusGame(address(this)).runDecimatorJackpot(...)` because `runRewardJackpots` runs via delegatecall in EndgameModule's context. The self-call routes through DegenerusGame's fallback, which delegates to DecimatorModule. This ensures `msg.sender == ContractAddresses.GAME` for the `OnlyGame()` check at DM:210.

**Pool accounting:** The return value from `runDecimatorJackpot` determines how much ETH is committed vs returned:
- Returns 0 when winners exist: full `decPoolWei` is committed to claims, `spend = decPoolWei`, `futurePoolLocal -= spend`, `claimableDelta += spend` (EM:210-212, EM:226-229).
- Returns `poolWei` when no winners or already snapshotted: `spend = 0`, no pool changes.
- The `claimableDelta` accumulator feeds into `claimablePool += claimableDelta` at EM:238, reserving funds for later claims.

### 1.2 Burn Tracking: recordDecBurn

**Entry:** `recordDecBurn(address player, uint24 lvl, uint8 bucket, uint256 baseAmount, uint256 multBps)` at DM:129-188

**Access control:** `msg.sender != ContractAddresses.COIN` reverts with `OnlyCoin()` at DM:136. Only the BURNIE coin contract can record burns.

**Flow (line by line):**

1. Load player entry: `DecEntry storage e = decBurn[lvl][player]` at DM:138. Copy to memory `m` at DM:139. Save `prevBurn = m.burn` at DM:140.

2. **First burn detection** (DM:143): If `m.bucket == 0`, this is the player's first burn at this level:
   - Set bucket (denominator 2-12): `m.bucket = bucket` at DM:144
   - Deterministic subbucket: `m.subBucket = _decSubbucketFor(player, lvl, bucket)` at DM:145

3. **Bucket migration** (DM:146): If `bucket != 0 && bucket < m.bucket` (strictly better = lower denominator):
   - Remove old aggregate: `_decRemoveSubbucket(lvl, m.bucket, m.subBucket, prevBurn)` at DM:148
   - Update bucket and subbucket: DM:149-150
   - Seed new subbucket with carried-over burn: `_decUpdateSubbucket(lvl, m.bucket, m.subBucket, prevBurn)` at DM:153-154
   - Guard: only seeds if `prevBurn != 0` (DM:152)

4. **Bucket used:** `bucketUsed = m.bucket` at DM:157. This is the actual bucket after first-burn or migration logic, not necessarily the requested `bucket`.

5. **Effective amount calculation:** `_decEffectiveAmount(uint256(prevBurn), baseAmount, multBps)` at DM:159-163

6. **Burn accumulation with uint192 saturation** (DM:166-168):
   ```solidity
   uint256 updated = uint256(prevBurn) + effectiveAmount;   // DM:166
   if (updated > type(uint192).max) updated = type(uint192).max; // DM:167
   uint192 newBurn = uint192(updated);                      // DM:168
   ```

7. **Storage write** (DM:169-171): `e.burn = newBurn; e.bucket = m.bucket; e.subBucket = m.subBucket;`

8. **Subbucket aggregate update** (DM:174-176):
   ```solidity
   uint192 delta = newBurn - prevBurn;                       // DM:174
   if (delta != 0) {
       _decUpdateSubbucket(lvl, bucketUsed, m.subBucket, delta); // DM:176
   }
   ```

9. **Event** (DM:177-184): `DecBurnRecorded(player, lvl, bucketUsed, m.subBucket, delta, newBurn)` emitted only when `delta != 0`.

**Return:** `bucketUsed` (the actual bucket, which may be the existing one if the new one was not better).

**Edge cases:**
- If `bucket == 0` is passed and `m.bucket == 0`, first-burn sets bucket to 0 and subbucket to 0. Subsequent burns will keep bucket 0 (the migration check `bucket != 0` fails). This player effectively has no bucket assignment, and `denom == 0` causes `_decClaimableFromEntry` to return 0 at DM:535 -- they cannot win. This is by design: bucket 0 is invalid.
- If `bucket > m.bucket` (worse bucket), the migration check fails and the existing bucket is kept. Correct behavior: only upgrades allowed.
- uint192 saturation at ~6.2e57: unreachable in practice given BURNIE supply constraints, but prevents overflow.

### 1.3 Subbucket Assignment: _decSubbucketFor

**Entry:** `_decSubbucketFor(address player, uint24 lvl, uint8 bucket)` at DM:610-621

```solidity
function _decSubbucketFor(address player, uint24 lvl, uint8 bucket)
    private pure returns (uint8)
{
    if (bucket == 0) return 0;                              // DM:615
    return uint8(
        uint256(keccak256(abi.encodePacked(player, lvl, bucket))) % bucket
    );                                                      // DM:617-620
}
```

**Properties:**
- Deterministic: same (player, lvl, bucket) always produces the same subbucket
- Range: [0, bucket-1] -- uniform distribution modulo bucket
- Player cannot choose subbucket (prevents concentrated burning)
- Changes with level (prevents cross-level subbucket prediction)
- Changes with bucket (migration to a better bucket gets a new subbucket)

**Modular bias analysis:** `keccak256` output is 256 bits. Modulo by small denominators (2-12) introduces negligible bias (2^256 mod 12 = 4, so bias is 4/2^256, effectively zero).

### 1.4 Effective Amount: _decEffectiveAmount

**Entry:** `_decEffectiveAmount(uint256 prevBurn, uint256 baseAmount, uint256 multBps)` at DM:454-473

```solidity
function _decEffectiveAmount(uint256 prevBurn, uint256 baseAmount, uint256 multBps)
    private pure returns (uint256 effectiveAmount)
{
    if (baseAmount == 0) return 0;                           // DM:459
    if (multBps <= BPS_DENOMINATOR || prevBurn >= DECIMATOR_MULTIPLIER_CAP) {
        return baseAmount;                                   // DM:463 -- 1x or already capped
    }

    uint256 remaining = DECIMATOR_MULTIPLIER_CAP - prevBurn; // DM:466
    uint256 fullEffective = (baseAmount * multBps) / BPS_DENOMINATOR; // DM:467
    if (fullEffective <= remaining) return fullEffective;    // DM:468

    // Partial: multiplied portion up to cap + 1x for remainder
    uint256 maxMultBase = (remaining * BPS_DENOMINATOR) / multBps; // DM:470
    uint256 multiplied = (maxMultBase * multBps) / BPS_DENOMINATOR; // DM:471
    effectiveAmount = multiplied + (baseAmount - maxMultBase);      // DM:472
}
```

**Constants:**
- `DECIMATOR_MULTIPLIER_CAP = 200 * PRICE_COIN_UNIT` at DM:101
- `PRICE_COIN_UNIT = 1000 ether` (GS:125)
- Cap = 200,000 BURNIE (200 mints worth)
- `BPS_DENOMINATOR = 10_000` at DM:98

**Multiplier logic:**
- `multBps` is the player's bonus multiplier in basis points (10000 = 1x)
- Multiplier applies only to the first `DECIMATOR_MULTIPLIER_CAP` worth of burn
- Once accumulated burn exceeds cap, all subsequent burns are counted at 1x
- Partial application: if the current burn would cross the cap, the portion up to the cap gets the multiplier, the remainder gets 1x

**Rounding analysis:** The partial-cap path at DM:470-472 computes `maxMultBase` by dividing `remaining * BPS_DENOMINATOR` by `multBps`. This rounds down, meaning `multiplied` at DM:471 may be slightly less than `remaining`. The difference is at most 1 unit of base amount -- negligible dust.

### 1.5 Subbucket Aggregate Functions

**_decUpdateSubbucket** at DM:577-585:
```solidity
function _decUpdateSubbucket(uint24 lvl, uint8 denom, uint8 sub, uint192 delta) internal {
    if (delta == 0 || denom == 0) return;              // DM:583
    decBucketBurnTotal[lvl][denom][sub] += uint256(delta); // DM:584
}
```

**_decRemoveSubbucket** at DM:592-601:
```solidity
function _decRemoveSubbucket(uint24 lvl, uint8 denom, uint8 sub, uint192 delta) internal {
    if (delta == 0 || denom == 0) return;              // DM:598
    uint256 slotTotal = decBucketBurnTotal[lvl][denom][sub]; // DM:599
    if (slotTotal < uint256(delta)) revert E();        // DM:600
    decBucketBurnTotal[lvl][denom][sub] = slotTotal - uint256(delta); // DM:601
}
```

**Underflow protection:** The removal function reverts if the aggregate would go negative (DM:600). This catches inconsistency between per-player burns and the aggregate -- a defensive invariant.

It is stored as `uint232` at DM:252. uint232 max is ~6.9e69. Each burn entry is uint192 max (~6.2e57). With 11 subbuckets, worst case is 11 * 6.2e57 = 6.8e58, well within uint232. ### 1.7 Packed Offset Storage: decBucketOffsetPacked

**Storage:** `mapping(uint24 => uint64) internal decBucketOffsetPacked` at GS:1332.

**Layout:** 4 bits per denominator, starting at denom 2. Bits [0:3] = denom 2, bits [4:7] = denom 3, ..., bits [40:43] = denom 12. Total: 44 bits used out of 64.

**Pack** at DM:493-501:
```solidity
function _packDecWinningSubbucket(uint64 packed, uint8 denom, uint8 sub)
    private pure returns (uint64)
{
    uint8 shift = (denom - 2) << 2;               // DM:498 -- 4 bits per denom
    uint64 mask = uint64(0xF) << shift;            // DM:499
    return (packed & ~mask) | ((uint64(sub) & 0xF) << shift); // DM:500
}
```

**Unpack** at DM:507-514:
```solidity
function _unpackDecWinningSubbucket(uint64 packed, uint8 denom)
    private pure returns (uint8)
{
    if (denom < 2) return 0;                       // DM:511
    uint8 shift = (denom - 2) << 2;               // DM:512
    return uint8((packed >> shift) & 0xF);          // DM:513
}
```

**Max subbucket per denom:** denom 12 has subbuckets 0-11. Value 11 = 0xB, fits in 4 bits (max 0xF = 15). All valid subbucket values for denoms 2-12 fit in 4 bits.

**Unused bits:** Bits [44:63] (20 bits) are always zero. The pack function uses clear-then-set (`packed & ~mask | value`), so these bits are never accidentally set.

## 2. Terminal Decimator (Death Bet)

### 2.1 Trigger Path

The terminal decimator is resolved at GAMEOVER via `handleGameOverDrain`:

```
handleGameOverDrain(day) [GOVM:68]
  -> gameOver = true                                    [GOVM:111]
  -> remaining = totalFunds - claimablePool             [GOVM:109]
  -> decPool = remaining / 10                           [GOVM:137] -- 10% of available funds
  -> IDegenerusGame(address(this))
       .runTerminalDecimatorJackpot(decPool, lvl, rngWord)  [GOVM:139]
  -> decRefund = decPool - decSpend                     [GOVM:140]
  -> claimablePool += decSpend                          [GOVM:142]
  -> remaining -= decPool; remaining += decRefund       [GOVM:144-145]
  -> remaining goes to terminal jackpot (90% + refund)  [GOVM:150-152]
```

**GAMEOVER triggers:** The death clock is defined by two constants (DM:687-688):
- `TERMINAL_DEC_IDLE_TIMEOUT_DAYS = 365` -- used when `level == 0` (game never started)
- `TERMINAL_DEC_DEATH_CLOCK = 120 days` -- used when `level > 0`

The deadline is `levelStartTime + timeout` (DM:926). When `block.timestamp >= deadline`, the game is over and anyone can call `handleGameOverDrain`.

**RNG source:** `rngWord = rngWordByDay[day]` at GOVM:124. Uses historical VRF word, which includes a fallback mechanism for stalled Chainlink VRF (after a 3-day wait period per the GOVM:64 NatSpec).

**Retry mechanism:** If `rngWord == 0` (VRF not yet available), the function returns without latching `gameOverFinalJackpotPaid` (GOVM:125). This allows retry with a different `day` parameter once VRF is available.

**Fund flow:** The terminal decimator gets 10% of available funds (after deducting `claimablePool`). If the decimator has no winners, it returns the full pool, which flows into the terminal jackpot (90% base + 10% refund = 100% to terminal jackpot). If the decimator has winners, the spent portion is added to `claimablePool` for deferred claims (GOVM:142), and the remaining 90% goes to the terminal jackpot.

### 2.2 Burn Tracking: recordTerminalDecBurn

**Entry:** `recordTerminalDecBurn(address player, uint24 lvl, uint256 baseAmount)` at DM:707-770

**Access control:** `msg.sender != ContractAddresses.COIN` reverts `OnlyCoin()` at DM:712.

**Days remaining calculation** (`_terminalDecDaysRemaining` at DM:922-929):

```solidity
function _terminalDecDaysRemaining() private view returns (uint256) {
```


- `TERMINAL_DEC_ACTIVITY_CAP_BPS = 23_500` (DM:697) -- caps activity score at 235%
- Bucket computed from activity score via `_terminalDecBucket` (DM:912-918):
  ```solidity
  function _terminalDecBucket(uint256 bonusBps) private pure returns (uint8) {
      if (bonusBps == 0) return TERMINAL_DEC_BUCKET_BASE;     // DM:913 -- 12
      uint256 range = uint256(TERMINAL_DEC_BUCKET_BASE) - uint256(TERMINAL_DEC_MIN_BUCKET); // DM:914 -- 10
      uint256 reduction = (range * bonusBps + (TERMINAL_DEC_ACTIVITY_CAP_BPS / 2)) / TERMINAL_DEC_ACTIVITY_CAP_BPS; // DM:915
      uint256 b = uint256(TERMINAL_DEC_BUCKET_BASE) - reduction; // DM:916
      if (b < TERMINAL_DEC_MIN_BUCKET) b = TERMINAL_DEC_MIN_BUCKET; // DM:917
      return uint8(b);                                         // DM:918
  }
  ```
  - `TERMINAL_DEC_BUCKET_BASE = 12`, `TERMINAL_DEC_MIN_BUCKET = 2` (DM:695-696)
  - Range is 10 (12 - 2). At max activity (23500 BPS): reduction = (10 * 23500 + 11750) / 23500 = 10, bucket = 12 - 10 = 2
  - At zero activity: bucket = 12 (worst odds, 1/12 chance per subbucket)
  - Rounding: `TERMINAL_DEC_ACTIVITY_CAP_BPS / 2` (11750) added for round-half-up behavior

- Multiplier: `bonusBps == 0 ? 10000 : 10000 + bonusBps/3` (DM:721)
  - Zero activity: 1x multiplier
  - Max activity (23500 BPS): `10000 + 7833 = 17833` BPS = 1.7833x multiplier
  - Note: integer division of `bonusBps / 3` truncates. At 23500: `23500 / 3 = 7833` (remainder 1 lost)

**Lazy reset** (DM:726-732):
```solidity
if (e.burnLevel != uint48(lvl)) {
    e.totalBurn = 0;
    e.weightedBurn = 0;
    e.bucket = 0;
    e.subBucket = 0;
    e.burnLevel = uint48(lvl);
}
```
If the player's stored entry is from a previous level, all fields are zeroed and the new level is recorded. This is a gas-efficient alternative to clearing storage on level transition.

**First burn at this level** (DM:735-738):
```solidity
if (e.bucket == 0) {
    e.bucket = bucket;
    e.subBucket = _decSubbucketFor(player, lvl, bucket);
}
```
Note: unlike regular decimator, there is no bucket migration for terminal decimator. The bucket is set once per level based on activity score at time of first burn. Subsequent burns at the same level keep the original bucket even if the player's activity score changes.

**Effective amount with cap enforcement** (DM:741-746):
```solidity

    uint256(e.totalBurn), baseAmount, multBps
);
if (effectiveAmount == 0) revert TerminalDecCapped(); // DM:746
```
Reuses the same `_decEffectiveAmount` function (DM:454-473) with the same `DECIMATOR_MULTIPLIER_CAP` of 200,000 BURNIE. The revert on zero effective amount means further burns after hitting the cap are rejected rather than silently ignored.

**Pre-time-multiplier total** (DM:749-751):
```solidity
uint256 newTotal = uint256(e.totalBurn) + effectiveAmount; // DM:749
if (newTotal > type(uint80).max) newTotal = type(uint80).max; // DM:750
e.totalBurn = uint80(newTotal);                              // DM:751
```
uint80 max = ~1.2e24. The `DECIMATOR_MULTIPLIER_CAP` is 200,000e18 = 2e23. At max multBps (17833), effective amount can reach `200000e18 * 17833 / 10000 = ~3.57e23`. With saturation at 1.2e24, this is safe for several burns worth of accumulation before saturation. Since the effective amount function caps total at 200k BURNIE equivalent, the pre-multiplier total realistically caps at `DECIMATOR_MULTIPLIER_CAP` which is 2e23, well within uint80.

**Time multiplier** (DM:754-755):
```solidity
uint256 timeMultBps = _terminalDecMultiplierBps(daysRemaining); // DM:754
uint256 weightedAmount = (effectiveAmount * timeMultBps) / BPS_DENOMINATOR; // DM:755
```

**_terminalDecMultiplierBps** at DM:903-909:
```solidity
function _terminalDecMultiplierBps(uint256 daysRemaining) private pure returns (uint256) {
    if (daysRemaining > 10) {
        return daysRemaining * 2500;                   // DM:905
    }
    // Linear: 2x at day 10, 1x at day 2 (day 1 blocked by caller)
    return 10000 + ((daysRemaining - 2) * 10000) / 8; // DM:908
}
```

**Time multiplier schedule:**

| Days Remaining | BPS | Multiplier | Regime |
|----------------|-----|------------|--------|
| 120 | 300,000 | 30x | High (`days * 2500`) |
| 60 | 150,000 | 15x | High |
| 30 | 75,000 | 7.5x | High |
| 11 | 27,500 | 2.75x | High |
| 10 | 20,000 | 2x | Low (linear) |
| 6 | 15,000 | 1.5x | Low |
| 2 | 10,000 | 1x | Low (minimum) |
| 1 | blocked | -- | Burns reverted |

**[DISCREPANCY] Day-10 discontinuity:** At day 11, the multiplier is `11 * 2500 = 27500` BPS (2.75x). At day 10, it switches to `10000 + (8 * 10000) / 8 = 20000` BPS (2x). This is a 27.3% drop from 2.75x to 2x between consecutive days. The NatSpec at DM:902 explicitly documents this as "Intentional discontinuity at day 10 (2.75x -> 2x regime change)." This is by design -- it creates urgency to burn before the regime change while ensuring late burners still get some reward.

**Post-time-multiplier total (weighted burn)** (DM:758-760):
```solidity
uint256 newWeighted = uint256(e.weightedBurn) + weightedAmount; // DM:758
if (newWeighted > type(uint88).max) newWeighted = type(uint88).max; // DM:759
e.weightedBurn = uint88(newWeighted);                              // DM:760
```
uint88 max = ~3.1e26. Worst case: if player burns 200k BURNIE at full multiplier (30x), weighted = 200000e18 * 30 = 6e24, well within uint88.

**Bucket aggregate update** (DM:763-764):
```solidity
bytes32 bucketKey = keccak256(abi.encode(lvl, e.bucket, e.subBucket)); // DM:763
terminalDecBucketBurnTotal[bucketKey] += weightedAmount;                // DM:764
```
Note: the key includes `lvl`, so entries from previous levels (before lazy reset) are in separate keys. The lazy reset at DM:726-732 ensures players' entries are fresh per level, but old bucket aggregate values remain in storage -- they are simply never read during resolution at the current level because different keys are used.

**Event** (DM:766-769): `TerminalDecBurnRecorded(player, lvl, e.bucket, e.subBucket, effectiveAmount, weightedAmount, timeMultBps)`.

### 2.3 Resolution: runTerminalDecimatorJackpot

**Entry:** `runTerminalDecimatorJackpot(uint256 poolWei, uint24 lvl, uint256 rngWord)` at DM:783-825

**Access control:** `msg.sender != ContractAddresses.GAME` reverts `OnlyGame()` at DM:788.

**Double-resolution guard** (DM:791-793):
```solidity
if (lastTerminalDecClaimRound.lvl == lvl) {  // DM:791
    return poolWei;                           // DM:792

```
Unlike the regular decimator's `poolWei != 0` check, the terminal decimator checks if the stored level matches. This means if `lastTerminalDecClaimRound.lvl` is 0 (never resolved) and `lvl` is also 0, it would incorrectly return early. However, in practice `lvl == 0` means the game never advanced past level 0, and the GAMEOVER path at GOVM:139 passes `lvl` which is `level` (the current level). Level 0 GAMEOVER is handled by the early refund path at GOVM:77-106 before reaching the decimator dispatch -- so this edge case is unreachable.

**Winning subbucket selection** (DM:798-810):
```solidity
uint256 decSeed = rngWord;                               // DM:799
for (uint8 denom = 2; denom <= DECIMATOR_MAX_DENOM; ) {  // DM:800
    uint8 winningSub = _decWinningSubbucket(decSeed, denom); // DM:801
    packedOffsets = _packDecWinningSubbucket(packedOffsets, denom, winningSub); // DM:802
    bytes32 bucketKey = keccak256(abi.encode(lvl, denom, winningSub)); // DM:804
```

Uses the same `_decWinningSubbucket` and `_packDecWinningSubbucket` functions as the regular decimator. The difference is the aggregate lookup: uses `terminalDecBucketBurnTotal[keccak256(lvl, denom, winningSub)]` instead of `decBucketBurnTotal[lvl][denom][winningSub]`.

**No winners:** If `totalWinnerBurn == 0` (DM:812), returns `poolWei`. This refund flows back to `remaining` in `handleGameOverDrain` (GOVM:145), which then feeds into the terminal jackpot.

**Storage writes** (DM:817-822):
```solidity

lastTerminalDecClaimRound.lvl = lvl;                      // DM:820
lastTerminalDecClaimRound.poolWei = uint96(poolWei);     // DM:821
```

**[NEW FINDING] (INFO) decBucketOffsetPacked write at DM:817:** This writes to the same `decBucketOffsetPacked[lvl]` mapping that the regular decimator uses. See Section 3 for full collision analysis.

**No rngWord stored:** The terminal claim round does not store `rngWord` because GAMEOVER claims are 100% ETH with auto-rebuy skipped (no lootbox entropy needed). The NatSpec at DM:819 confirms: "single slot -- no rngWord needed, GAMEOVER skips auto-rebuy."

**Return:** 0 (all funds held for claims).

### 2.4 Claims: claimTerminalDecimatorJackpot

**Entry:** `claimTerminalDecimatorJackpot()` at DM:833-840

```solidity
function claimTerminalDecimatorJackpot() external {
    if (prizePoolFrozen) revert E();                       // DM:834
    uint256 amountWei = _consumeTerminalDecClaim(msg.sender); // DM:836
    _addClaimableEth(msg.sender, amountWei, 0);            // DM:839
}
```

**No level parameter:** Unlike the regular decimator claim, the terminal claim does not take a `lvl` parameter. The level is read from `lastTerminalDecClaimRound.lvl` (DM:872). This is correct because there is only one terminal decimator resolution per game.

**100% ETH:** All terminal claims are ETH-only (DM:838-839). The `entropy` parameter is 0 -- auto-rebuy would need entropy for level selection, but `_processAutoRebuy` returns false when `gameOver == true` (DM:367), so the entropy is unused.

**Claim validation** (`_consumeTerminalDecClaim` at DM:869-893):

```solidity
function _consumeTerminalDecClaim(address player) private returns (uint256 amountWei) {
    uint24 lvl = lastTerminalDecClaimRound.lvl;             // DM:872
    if (lvl == 0) revert TerminalDecNotActive();            // DM:873

    TerminalDecEntry storage e = terminalDecEntries[player]; // DM:875
    if (e.burnLevel != uint48(lvl) || e.weightedBurn == 0) revert TerminalDecNotWinner(); // DM:876

    uint88 weight = e.weightedBurn;                          // DM:879

    uint64 packedOffsets = decBucketOffsetPacked[lvl];       // DM:881
    uint8 winningSub = _unpackDecWinningSubbucket(packedOffsets, e.bucket); // DM:882
    if (e.subBucket != winningSub) revert TerminalDecNotWinner(); // DM:883

    uint256 totalBurn = uint256(lastTerminalDecClaimRound.totalBurn); // DM:885
    if (totalBurn == 0) revert TerminalDecNotWinner();       // DM:886

    amountWei = (uint256(lastTerminalDecClaimRound.poolWei) * uint256(weight)) / totalBurn; // DM:888
    if (amountWei == 0) revert TerminalDecNotWinner();       // DM:889

    e.weightedBurn = 0;                                       // DM:892
}
```

**Claim flag mechanism:** The terminal decimator uses `e.weightedBurn = 0` (DM:892) as the claimed flag, unlike the regular decimator's `e.claimed = 1` (DM:292). The check at DM:876 (`e.weightedBurn == 0`) catches already-claimed entries. This means `weightedBurn` serves a dual purpose: claim share weight AND claim flag.

**Level validation:** `e.burnLevel != uint48(lvl)` at DM:876 ensures the player's burn is from the same level as the resolution. The lazy reset in `recordTerminalDecBurn` (DM:726-732) clears entries on level change, but if a player burned at a previous level and never burned at the current level, their entry has `burnLevel` of the old level and they correctly get rejected.

**Pro-rata calculation:** `(poolWei * weight) / totalBurn` at DM:888, using the player's `weightedBurn` (post-time-multiplier) against the total weighted burn from winning subbuckets.

**View function:** `terminalDecClaimable(address player)` at DM:846-866. Performs the same logic as the claim path but without state changes. Returns `(amountWei, winner)`.

## 3. decBucketOffsetPacked Collision Analysis

### 3.1 The Shared Write

Both regular decimator resolution and terminal decimator resolution write to `decBucketOffsetPacked[lvl]`:

- **Regular:** `decBucketOffsetPacked[lvl] = packedOffsets` at DM:248
- **Terminal:** `decBucketOffsetPacked[lvl] = packedOffsets` at DM:817

If both resolve at the same level, the second write overwrites the first.

### 3.2 Scenario Analysis

**Can regular and terminal decimator fire at the same level?**

The regular decimator fires during `runRewardJackpots` (EM:168) at levels x5 and x00 during level transitions. The terminal decimator fires during `handleGameOverDrain` (GOVM:68) at GAMEOVER.

GAMEOVER occurs when the death clock expires (120 days after last level start). At that point, `lvl = level` (the current level, GOVM:71). The last regular decimator fires at the most recent level matching x5 or x00.

**Case 1: GAMEOVER at level N where N was the last regular decimator level.**
For example, if the game stalls at level 15 (which is an x5 level), the regular decimator fires at level 15 during the transition from 14 to 15. Then the game stalls and GAMEOVER fires also at level 15. Both write to `decBucketOffsetPacked[15]`.

**Case 2: GAMEOVER at level N where N was NOT a regular decimator level.**
For example, GAMEOVER at level 17. The last regular decimator was at level 15. Regular writes `decBucketOffsetPacked[15]`, terminal writes `decBucketOffsetPacked[17]`. No collision.

### 3.3 Impact Assessment

**Does the terminal overwrite break regular decimator claims?**

The regular decimator claim path reads `decBucketOffsetPacked[lvl]` at DM:281 to determine winning subbuckets:
```solidity
uint64 packedOffsets = decBucketOffsetPacked[lvl];     // DM:281
```

The regular decimator claim round (`decClaimRounds[lvl]`) stores `poolWei`, `totalBurn`, and `rngWord` but does NOT store `packedOffsets`. The packed offsets are read fresh from storage on every claim.

**If terminal overwrites `decBucketOffsetPacked[lvl]` after regular resolution:**

1. The regular decimator's stored `rngWord` (in `decClaimRounds[lvl].rngWord`, DM:253) could be used to recompute the correct packed offsets. However, the claim path does NOT recompute from rngWord -- it reads directly from `decBucketOffsetPacked[lvl]`.

2. Both regular and terminal use the same `_decWinningSubbucket(rngWord, denom)` function. If both use the same `rngWord`, the packed offsets would be identical and the overwrite would be benign. However, the rngWords are DIFFERENT:
   - Regular decimator gets `rngWord` from `advanceGame` -> VRF during level transition
   - Terminal decimator gets `rngWord` from `rngWordByDay[day]` during GAMEOVER (GOVM:124)
   - These are different VRF words from different Chainlink responses

3. Since the rngWords differ, `_decWinningSubbucket(rngWord, denom)` produces different winning subbuckets. The terminal's packed offsets overwrite the regular's, meaning:
   - Players who were winners under the regular decimator's VRF word may no longer be winners
   - Players who were NOT winners may now incorrectly appear as winners
   - The `totalBurn` in `decClaimRounds[lvl]` was computed from the regular VRF's winning subbuckets, so the pro-rata denominator is wrong for the terminal's winning subbuckets

**[NEW FINDING] (MEDIUM) decBucketOffsetPacked collision between regular and terminal decimator.**

**Location:** DM:248 (regular write), DM:817 (terminal write), DM:281 (regular claim read).

**Trigger condition:** GAMEOVER occurs at the same level as the last regular decimator (levels ending in 5 or levels ending in 00). Given the 120-day death clock, the game must stall immediately after a decimator level transition.

**Impact:**
- Regular decimator winners may lose their claim (subbucket no longer matches)
- Non-winners may incorrectly pass the subbucket check but get wrong pro-rata amounts
- The `totalBurn` stored in `decClaimRounds[lvl]` (DM:252) does NOT match the terminal's winning subbuckets, so pro-rata calculations would be computed against a mismatched denominator

**Severity reasoning -- MEDIUM not HIGH:**
1. GAMEOVER must occur at exactly a decimator level (x5 or x00). If the game stalls at level 17, no collision.
2. The timing window requires the game to stall immediately after a level transition that triggers the decimator.
3. If GAMEOVER occurs at a non-decimator level, both regular and terminal claims work correctly (different `lvl` keys).
4. Regular decimator claims can still work if claimed BEFORE GAMEOVER triggers the terminal resolution.
5. Post-GAMEOVER, regular decimator claims switch to 100% ETH mode (DM:325-328) with `gameOver == true`, which still reads from the overwritten `decBucketOffsetPacked` -- so the corruption affects the claim validation.

**Possible mitigations (not implemented):**
- Store packed offsets inside `DecClaimRound` instead of a separate mapping
- Use a different storage slot for terminal offsets (e.g., `terminalDecBucketOffsetPacked[lvl]`)
- Recompute packed offsets from stored `rngWord` during regular claims

### 3.4 Reverse Collision: Does Regular Overwrite Affect Terminal?

The terminal claim reads `decBucketOffsetPacked[lvl]` at DM:881. The terminal resolution writes at DM:817. Since `handleGameOverDrain` calls the terminal decimator AFTER `gameOver = true` is set (GOVM:111), and regular decimator fires during `advanceGame` which cannot run after `gameOver` (the advance path checks game state), the execution order is guaranteed:

1. Regular decimator resolves at level N (writes `decBucketOffsetPacked[N]`)
2. Time passes, game stalls
3. GAMEOVER triggers, terminal resolves at level N (overwrites `decBucketOffsetPacked[N]`)

The terminal resolution always runs AFTER the regular resolution. So the terminal's write is the final value, and terminal claims read the correct offsets. The collision only damages regular claims, not terminal claims.
