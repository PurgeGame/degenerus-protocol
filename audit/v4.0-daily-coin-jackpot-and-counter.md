## Executive Summary

Exhaustive trace of both daily BURNIE (coin) jackpot winner selection entry points and the `jackpotCounter` lifecycle across all four contract files. The coin jackpot awards 0.5% of the previous level's prize pool target in BURNIE each jackpot day, split 25% to far-future ticketQueue holders and 75% to near-future trait-matched winners. Two entry points exist: `payDailyCoinJackpot` (purchase phase) and `payDailyJackpotCoinAndTickets` (jackpot phase). The `jackpotCounter` governs how many daily jackpots run per level, with normal (1-step), compressed (2-step), and turbo (5-step) modes.

## 2. Entry Point 1: payDailyCoinJackpot (JM:2360)

### Caller Context

Called during the **purchase phase** when `!inJackpot && !lastPurchaseDay`:
- `advanceGame()` at AM:128 enters the main do-while loop
- At AM:283: `_payDailyCoinJackpot(purchaseLevel, rngWord)` -- a delegatecall wrapper at AM:616-627
- The wrapper delegatecalls into `IDegenerusGameJackpotModule.payDailyCoinJackpot.selector` targeting `ContractAddresses.GAME_JACKPOT_MODULE` (AM:618-622)

### Function Flow (JM:2360-2401)

```
payDailyCoinJackpot(uint24 lvl, uint256 randWord) external        [JM:2360]
  |-- coinBudget = _calcDailyCoinBudget(lvl)                       [JM:2361 -> JM:2647]
  |   |-- return (levelPrizePool[lvl-1] * PRICE_COIN_UNIT) / (priceWei * 200)  [JM:2650]
  |   |   Budget = 0.5% of previous level target in BURNIE units
  |-- if coinBudget == 0: return                                   [JM:2362]
  |
  |-- farBudget = (coinBudget * FAR_FUTURE_COIN_BPS) / 10_000     [JM:2365]
  |   FAR_FUTURE_COIN_BPS = 2500 (25%)                            [JM:206]
  |-- nearBudget = coinBudget - farBudget                          [JM:2366]
  |   (75% of coinBudget)
  |
  |-- FAR-FUTURE (25%):
  |   _awardFarFutureCoinJackpot(lvl, farBudget, randWord)        [JM:2369 -> JM:2521]
  |
  |-- NEAR-FUTURE (75%):
  |   if nearBudget == 0: return                                   [JM:2372]
  |   questDay = _calculateDayIndex()                              [JM:2374]
  |   (winningTraitsPacked, valid) = _loadDailyWinningTraits(lvl, questDay)  [JM:2375-2378]
  |   if !valid:                                                   [JM:2379]
  |     useBurn = jackpotPhaseFlag                                 [JM:2380]
  |     winningTraitsPacked = _rollWinningTraits(lvl, randWord, useBurn)  [JM:2381]
  |     _syncDailyWinningTraits(lvl, winningTraitsPacked, questDay)       [JM:2382]
  |
  |   entropy = randWord ^ (uint256(lvl) << 192) ^ uint256(COIN_JACKPOT_TAG)  [JM:2385-2387]
  |     COIN_JACKPOT_TAG = keccak256("coin-jackpot")               [JM:132]
  |
  |   targetLevel = _selectDailyCoinTargetLevel(lvl, winningTraitsPacked, entropy)  [JM:2388-2392 -> JM:2405]
  |   if targetLevel == 0: return                                  [JM:2393]
  |
  |   _awardDailyCoinToTraitWinners(targetLevel, winningTraitsPacked, nearBudget, entropy)  [JM:2395-2400 -> JM:2418]
```

### Key Observations -- Entry Point 1

1. **Fresh trait roll**: If winning traits not already stored for this `questDay`, a fresh `_rollWinningTraits` is called (JM:2381). In purchase phase, `jackpotPhaseFlag` is false (AM:327 sets it true only at jackpot phase entry), so `useBurn=false` and random traits are used (not burn-weighted).
2. **Entropy domain separation**: `COIN_JACKPOT_TAG = keccak256("coin-jackpot")` (JM:132) XORed into entropy ensures coin jackpot entropy is distinct from ETH jackpot entropy.
3. **Silent skip on empty level**: If `_selectDailyCoinTargetLevel` returns 0 (no trait tickets at chosen level), the entire near-future budget is silently skipped (JM:2393). This is documented as a deliberate design choice per the NatSpec at JM:2403-2404.

## 4. Shared Subroutine: _awardFarFutureCoinJackpot (JM:2521)

### Purpose
Awards 25% of the BURNIE coin budget to random ticket holders on far-future levels ([lvl+5, lvl+99]). Selects up to `FAR_FUTURE_COIN_SAMPLES` (10, JM:209) winners, one per sampled level.

### Flow (JM:2521-2606)

```
_awardFarFutureCoinJackpot(uint24 lvl, uint256 farBudget, uint256 rngWord) private  [JM:2521]
  |-- if farBudget == 0: return                                    [JM:2526]
  |
  |-- ENTROPY DERIVATION:
  |   entropy = rngWord ^ (uint256(lvl) << 192) ^ uint256(FAR_FUTURE_COIN_TAG)  [JM:2528-2530]
  |     FAR_FUTURE_COIN_TAG = keccak256("far-future-coin")         [JM:213]
  |
  |-- WINNER SEARCH (up to 10 samples):
  |   for s = 0..FAR_FUTURE_COIN_SAMPLES-1:                       [JM:2537]
  |     entropy = EntropyLib.entropyStep(entropy ^ uint256(s))     [JM:2538]
  |     candidate = lvl + 5 + uint24(entropy % 95)                 [JM:2541]
  |       Range: [lvl+5, lvl+99] (95 values)
  |
  |     queue = ticketQueue[_tqFarFutureKey(candidate)]            [JM:2543]  *** FAR-FUTURE KEY ***
  |     len = queue.length                                         [JM:2544]
  |     if len != 0:                                               [JM:2546]
  |       winner = queue[(entropy >> 32) % len]                    [JM:2547]
  |       if winner != address(0):                                 [JM:2548]
  |         winners[found] = winner                                [JM:2549]
  |         winnerLevels[found] = candidate                        [JM:2550]
  |         ++found                                                [JM:2552]
  |
  |-- if found == 0: return                                        [JM:2562]
  |
  |-- DISTRIBUTE:
  |   perWinner = farBudget / found                                [JM:2565]
  |   if perWinner == 0: return                                    [JM:2566]
  |   Batch via coin.creditFlipBatch(batchPlayers, batchAmounts) in batches of 3  [JM:2587, 2604]
  |   Emit FarFutureCoinJackpotWinner per winner                   [JM:2573-2578]
```

### Critical Details

1. **Key space**: Uses `_tqFarFutureKey(candidate)` (JM:2543) -- the dedicated far-future key space with bit 22 set, introduced in v3.9 Phase 74. This is **NOT** `_tqReadKey` or `_tqWriteKey`.
2. **Level range**: `lvl + 5 + uint24(entropy % 95)` gives [lvl+5, lvl+99] -- 95 distinct possible levels.
3. **Winner index**: `(entropy >> 32) % len` (JM:2547) uses bits 32+ of entropy to avoid correlation with level selection (bits 0-6 used for `% 95`).
4. **No trait matching**: Far-future winners are selected purely by position in `ticketQueue`, independent of traits. This is correct since tickets at far-future levels haven't been processed into `traitBurnTicket` yet.
5. **Even split**: All found winners receive `farBudget / found` (JM:2565). Remainder (up to `found-1` BURNIE units) is dust-lost.
6. **Batch ceiling**: Maximum 10 winners, always fits in fixed-size arrays (JM:2533-2534).

## 5. Shared Subroutine: _awardDailyCoinToTraitWinners (JM:2418) -- Near-Future

### Target Level Selection: _selectDailyCoinTargetLevel (JM:2405)

```
_selectDailyCoinTargetLevel(uint24 lvl, uint32 winningTraitsPacked, uint256 entropy) private view  [JM:2405]
  |-- candidate = lvl + uint24(entropy % 5)                        [JM:2410]
  |   Range: [lvl, lvl+4] -- 5 possible levels
  |-- if _hasTraitTickets(candidate, winningTraitsPacked):         [JM:2411]
  |     return candidate                                           [JM:2412]
  |-- return 0 (skip near-future entirely)                         [JM:2414]
```

**_hasTraitTickets** (JM:1031-1050): Checks whether any of the 4 winning traits have non-empty `traitBurnTicket[lvl][trait]` arrays OR have a virtual deity entry (`deityBySymbol[fullSymId] != address(0)`).

**Silent skip behavior**: If the randomly chosen level has no eligible trait tickets (no real tickets and no deity for any of the 4 winning traits), `targetLevel=0` and the **entire 75% near-future coin budget is lost for that day**. There is no fallback to try other levels. See [DCJ-03] in Section 7.

### Winner Selection and Distribution: _awardDailyCoinToTraitWinners (JM:2418)

```
_awardDailyCoinToTraitWinners(uint24 lvl, uint32 winningTraitsPacked, uint256 coinBudget, uint256 entropy) private  [JM:2418]
  |-- if coinBudget == 0: return                                   [JM:2424]
  |-- cap = DAILY_COIN_MAX_WINNERS (50)                            [JM:2425, constant at JM:200]
  |   if cap > coinBudget: cap = uint16(coinBudget)                [JM:2426]
  |     (prevents more winners than BURNIE units)
  |
  |-- traitIds = JackpotBucketLib.unpackWinningTraits(winningTraitsPacked)  [JM:2428-2430]
  |-- (counts, activeCount) = _computeBucketCounts(lvl, traitIds, cap, entropy)  [JM:2431-2436 -> JM:1222]
  |   if activeCount == 0: return                                  [JM:2437]
  |
  |-- PAYOUT MATH:
  |   baseAmount = coinBudget / cap                                [JM:2439]
  |   extra = coinBudget % cap                                     [JM:2440]
  |   cursor = entropy % cap                                       [JM:2441]
  |   (cursor distributes the `extra` remainder BURNIE to the first `extra` winners in cursor order)
  |
  |-- FOR EACH TRAIT BUCKET (4 buckets, traitIdx 0..3):            [JM:2447]
  |   if count != 0:                                               [JM:2449]
  |     entropy = EntropyLib.entropyStep(entropy ^ (traitIdx << 64) ^ coinBudget)  [JM:2450-2452]
  |     if count > MAX_BUCKET_WINNERS (250): count = 250           [JM:2454]
  |
  |     (bucketWinners, bucketTicketIndexes) = _randTraitTicketWithIndices(
  |       traitBurnTicket[lvl], entropy, traitIds[traitIdx], uint8(count),
  |       uint8(DAILY_COIN_SALT_BASE + traitIdx)                   [JM:2458-2464]
  |     )
  |     DAILY_COIN_SALT_BASE = 252                                 [JM:203]
  |     So salt values: 252, 253, 254, 255 for traitIdx 0-3
  |
  |     for each winner:
  |       amount = baseAmount + (1 if cursor < extra)              [JM:2468-2471]
  |       if winner != address(0) && amount != 0:                  [JM:2474]
  |         emit JackpotTicketWinner(winner, lvl, traitId, amount, ticketIndex)  [JM:2475-2481]
  |         batch via coin.creditFlipBatch in batches of 3         [JM:2487-2490]
  |       cursor = (cursor + 1) % cap                              [JM:2494-2495]
```

### _randTraitTicketWithIndices (JM:2283)

This is the core winner selection function for near-future coin jackpot:

```
_randTraitTicketWithIndices(
  address[][256] storage traitBurnTicket_,                         [JM:2284]
  uint256 randomWord, uint8 trait, uint8 numWinners, uint8 salt
) private view -> (address[] winners, uint256[] ticketIndexes)     [JM:2283-2337]
  |-- holders = traitBurnTicket_[trait]                            [JM:2294]
  |-- len = holders.length                                         [JM:2295]
  |
  |-- VIRTUAL DEITY ENTRIES:                                       [JM:2297-2308]
  |   fullSymId = (trait >> 6) * 8 + (trait & 0x07)                [JM:2300]
  |   deity = deityBySymbol[fullSymId]                             [JM:2304]
  |   if deity != address(0):
  |     virtualCount = max(len / 50, 2)                            [JM:2306-2307]
  |     (2% of bucket, minimum 2)
  |
  |-- effectiveLen = len + virtualCount                            [JM:2311]
  |-- if effectiveLen == 0 || numWinners == 0: return empty        [JM:2312-2313]
  |
  |-- SELECTION LOOP:
  |   slice = randomWord ^ (trait << 128) ^ (salt << 192)         [JM:2319-2321]
  |   for i = 0..numWinners-1:                                    [JM:2322]
  |     idx = slice % effectiveLen                                 [JM:2323]
  |     if idx < len: winners[i] = holders[idx]                   [JM:2324-2326]
  |     else: winners[i] = deity                                  [JM:2328]
  |     ticketIndexes[i] = idx < len ? idx : type(uint256).max    [JM:2326,2329]
  |     slice = (slice >> 16) | (slice << 240)                    [JM:2334]
  |       (bit rotation for different indices per winner)
```

**Note on _randTraitTicket (JM:2237)**: Nearly identical to `_randTraitTicketWithIndices` but without the `ticketIndexes` return. Used by `_distributeTicketJackpot` (ticket distribution), not coin jackpot. Same deity virtual entry logic and bit-rotation selection.

### [NEW FINDING - INFO] DCJ-03: Near-Future Coin Budget Silently Skipped When Target Level Empty

**Found during:** Section 5 analysis of `_selectDailyCoinTargetLevel`
**Location:** JM:2405-2414 (`_selectDailyCoinTargetLevel`) and JM:2393 (caller check)
**Issue:** When the randomly selected target level (1 of 5 levels in [lvl, lvl+4]) has no eligible trait tickets for any of the 4 winning traits (and no deity), the function returns 0 and the entire 75% near-future coin budget is silently discarded for that day. There is no fallback to try other levels.
**Analysis:** This appears to be a deliberate design choice:
- The NatSpec at JM:2403-2404 explicitly states: "Returns 0 (skip) if the chosen level has no eligible trait tickets"
- At early game levels where ticket density is low, this could mean some near-future budgets are lost
- The probability scales with ticket density: more tickets = more likely a non-empty level is chosen
- At level N with 5 candidate levels [N, N+4], the probability of a completely empty level depends on the trait distribution across levels
**Severity:** INFO -- intentional design that reduces average payouts at low ticket density but avoids the gas cost of multi-level fallback searches.

## 8. jackpotCounter Lifecycle (DCOIN-03)

### 8.1 Storage Declaration

- **Location:** `uint8 internal jackpotCounter` at GS:245
- **EVM slot:** 0, byte offset 22 (uint8)
- **NatSpec (GS:240-244):** "Count of jackpots processed within the current level. Capped at 5 (JACKPOT_LEVEL_CAP in JackpotModule); triggers level advancement when reached. Reset at level start."
- **SECURITY:** uint8 sufficient (max 255, only need 0-5). Max possible value: 0 + 5 (turbo step) = 5, fits comfortably.

### 8.2 Reset (WRITE)

- **Location:** `_endPhase()` at AM:475
- **Reset:** `jackpotCounter = 0` at AM:481
- **Also resets:** `compressedJackpotFlag = 0` at AM:482
- **Trigger:** Called when `jackpotCounter >= JACKPOT_LEVEL_CAP` (AM:364) after `payDailyJackpotCoinAndTickets` returns, followed by `_awardFinalDayDgnrsReward`, `_rewardTopAffiliate`, `_runRewardJackpots`, then `_endPhase()` at AM:368
- **Phase transition:** `_endPhase()` also sets `phaseTransitionActive = true` (AM:477) and conditionally seeds the next level's prize pool at AM:478-479

### 8.3 Increment (WRITE)

- **Location:** `payDailyJackpotCoinAndTickets` at JM:757: `jackpotCounter += counterStep` (inside `unchecked` block at JM:756-758)
- **counterStep values:**
  - 1 (normal): default when `compressedJackpotFlag` is 0 or conditions don't match
  - 2 (compressed): when `compressedJackpotFlag == 1 && counter > 0 && counter < JACKPOT_LEVEL_CAP - 1`
  - 5 (turbo): when `compressedJackpotFlag == 2 && counter == 0` (JACKPOT_LEVEL_CAP = 5)
- **counterStep computation:** Happens in `payDailyJackpot` at JM:349-363, packed into `dailyTicketBudgetsPacked` at JM:459 (via `_packDailyTicketBudgets`), unpacked in `payDailyJackpotCoinAndTickets` at JM:686 (via `_unpackDailyTicketBudgets`)
- **Unchecked safety:** Max counterStep = 5, max counter before increment = 4 (for step=1 case, since step=2 requires counter < 4, and step=5 requires counter == 0). Max result = 5 (turbo: 0+5) or 5 (compressed: 3+2) or 5 (normal: 4+1). All fit uint8.

### 8.4 Reads (READ)

1. **AM:224 -- bonusFlip computation:**
   ```solidity

   ```
   Determines whether to request a new VRF word (`bonusFlip=true` means a new flip is requested even if the VRF word hasn't been used yet). When `jackpotCounter == 0` and in jackpot phase, this ensures a fresh VRF word for the first jackpot day.

2. **AM:364 -- JACKPOT_LEVEL_CAP check:**
   ```solidity

   ```
   After `payDailyJackpotCoinAndTickets` returns, this check determines whether to end the level phase. If true, triggers `_awardFinalDayDgnrsReward`, `_rewardTopAffiliate`, `_runRewardJackpots`, `_endPhase()`, `_unfreezePool()`.

3. **JM:349 -- counterStep computation:**
   ```solidity

   ```
   Read at the start of fresh daily jackpot initialization (not resuming). Used to compute `counterStep` based on `compressedJackpotFlag` and current counter value.

4. **JM:484 -- isFinalPhysicalDay_ check:**
   ```solidity
   bool isFinalPhysicalDay_ = (jackpotCounter + counterStep_ >= JACKPOT_LEVEL_CAP);
   ```
   Read during ETH distribution phase of `payDailyJackpot` (after `dailyTicketBudgetsPacked` is unpacked). Determines Phase 0/1 ETH distribution behavior -- final physical day uses 100% of remaining pool.

5. **MM:971 -- Affiliate bonus timing:**

### 8.5 counterStep Computation Detail (JM:349-363)

| compressedJackpotFlag | counter | counterStep | Logical Days | Effect |
|----------------------|---------|-------------|-------------|--------|
| 2 (turbo) | 0 | 5 (= JACKPOT_LEVEL_CAP) | All 5 in 1 physical day | Instant level completion |
| 1 (compressed) | 1-3 | 2 | 5 in 3 physical days | 2 logical days per physical day |
| 1 (compressed) | 0 | 1 | Normal first day | First day always step=1 |
| 1 (compressed) | 4 | 1 | Normal last day | counter >= CAP-1 excluded from step=2 |
| 0 (normal) | any | 1 | 1 per physical day | Standard: 5 physical days |

Exact code at JM:349-361:
```solidity
uint8 counter = jackpotCounter;                     // JM:349
uint8 counterStep = 1;                              // JM:350
// Turbo (flag=2): all 5 logical days in 1 physical day.
// Compressed (flag=1): 5 logical days in 3 physical days.
if (compressedJackpotFlag == 2 && counter == 0) {   // JM:353
    counterStep = JACKPOT_LEVEL_CAP;                // JM:354 (= 5)
} else if (
    compressedJackpotFlag == 1 &&                   // JM:356
    counter > 0 &&                                  // JM:357
    counter < JACKPOT_LEVEL_CAP - 1                 // JM:358 (< 4)
) {
    counterStep = 2;                                // JM:360
}
```

### 8.6 compressedJackpotFlag Relationship

**Set to 2 (turbo):** AM:140 -- when `!inJackpot && !lastPurchaseDay` and `purchaseDays <= 1` and prize pool target already met (AM:134-141). Meaning: target reached within the first 1-2 purchase days.

**Set to 1 (compressed):** AM:289 -- when `!inJackpot && !lastPurchaseDay` and `day - purchaseStartDay <= 3` and target just now met (AM:284-291). Meaning: target reached within first 3 purchase days.

**Reset to 0:** AM:482 -- in `_endPhase()`, alongside `jackpotCounter = 0` at AM:481.

**Note:** The turbo check (AM:134-141) runs **before** the main loop at AM:222, while the compressed check (AM:288-289) runs **inside** the purchase phase branch. The turbo flag=2 at AM:140 is set when `_getNextPrizePool() >= levelPrizePool[lvl]` is already true before daily processing, while flag=1 at AM:289 is set when it becomes true during daily processing.

### 8.8 Security Assessment

1. **Write access:** Only `advanceGame` flow writes `jackpotCounter`:
   - Reset at AM:481 via `_endPhase()` (called from advanceGame jackpot phase branch)
   - Increment at JM:757 via `payDailyJackpotCoinAndTickets` (delegatecall from advanceGame)
   - No external/permissionless function can directly write `jackpotCounter`

2. **Overflow safety:** Unchecked arithmetic at JM:757 is safe:
   - Max counterStep = 5 (turbo, JM:354)
   - Max counter before increment: 0 (turbo step=5), 3 (compressed step=2), 4 (normal step=1)
   - Max result: 5 (all paths). Fits comfortably in uint8 (max 255).

3. **v3.8 claim verification:** "Not writable by any external/permissionless function" (v3.8 line 3486): **[CONFIRMED]** -- both write locations are in private functions called only through the advanceGame delegatecall chain.

4. **MintModule read (MM:971-973):** The affiliate bonus calculation mirrors the JackpotModule's counterStep logic but only considers compressed (step=2) and normal (step=1) modes, explicitly excluding turbo (`compressedJackpotFlag != 2` guard at MM:969). This is correct: turbo mode completes the entire jackpot phase in one call, so there is no "day before final draw" for affiliates to benefit from.
