### Detailed Latch Analysis

#### 1. `gameOverFinalJackpotPaid` Latch (GameOverModule:69, 128)

- **Set at:** Line 128, AFTER `gameOver = true` (line 112) but BEFORE any delegatecall to decimator (line 140) or jackpot (line 152).
- **Guard at:** Line 69, first statement of `handleGameOverDrain`. Returns immediately if already set.
- **Re-entry vector analysis:** The self-calls at lines 140 and 152 dispatch through `DegenerusGame.runTerminalDecimatorJackpot()` and `DegenerusGame.runTerminalJackpot()`. Both have `msg.sender != address(this)` guards. These are delegatecalls that execute DecimatorModule and JackpotModule code in the game's storage context. Neither module calls back to `handleGameOverDrain`. Even if they could, `gameOverFinalJackpotPaid` is already `true`.
- **Verdict:** Airtight. No path allows double execution.

#### 2. `gameOver = true` (GameOverModule:112)

- **Set at:** Line 112, BEFORE all delegatecalls.
- **Purpose:** Terminal flag. Once set, normal gameplay functions (purchase, lootbox, etc.) revert or behave differently.
- **Key downstream effect:** `_addClaimableEth` in JackpotModule (line 982) checks `if (!gameOver)` before auto-rebuy. Since `gameOver = true` was set in the same delegatecall context before `runTerminalJackpot`, auto-rebuy is correctly skipped during terminal jackpot distribution.
- **Verdict:** Correctly ordered.

#### 3. `finalSwept` Latch (GameOverModule:174, 176)

- **Set at:** Line 176, immediately after the guard check at line 174.
- **Guard at:** Line 174: `if (finalSwept) return`.
- **Re-entry vector analysis:** `_sendToVault` at line 188 makes external ETH/stETH transfers. If vault re-enters via `advanceGame -> _handleGameOverPath`, it hits the `finalSwept` check in `handleFinalSweep` and returns. The `advanceGame` path through `_handleGameOverPath` will delegatecall `handleFinalSweep` when `gameOver == true`, which returns immediately.
- **Verdict:** Airtight. No double-sweep possible.

#### 4. Delegatecall Context Verification

The calls from `handleGameOverDrain` to `runTerminalDecimatorJackpot` and `runTerminalJackpot` use the pattern:
```solidity
IDegenerusGame(address(this)).runTerminalDecimatorJackpot(decPool, lvl, rngWord);
```

This is a **self-call** (external call to `address(this)`), which enters `DegenerusGame.runTerminalDecimatorJackpot()`. That function then performs:
```solidity
ContractAddresses.GAME_DECIMATOR_MODULE.delegatecall(...)
```

The delegatecall executes `DecimatorModule.runTerminalDecimatorJackpot()` in the game contract's storage context. Storage writes (e.g., `lastTerminalDecClaimRound`, `decBucketOffsetPacked`) happen to the game's storage slots. This is correct behavior -- not reentrancy -- but the intermediate self-call does create a call-stack entry. The `msg.sender != address(this)` guard at DegenerusGame:1215 ensures only the game contract itself can invoke these dispatch functions.

**State mutation ordering in delegatecall context:** The `runTerminalDecimatorJackpot` delegatecall writes to `lastTerminalDecClaimRound` and `decBucketOffsetPacked[lvl]` at DecimatorModule:914-919. These writes occur in the game's storage during the delegatecall, but `gameOverFinalJackpotPaid` is already `true` (set at GameOverModule:128 before the self-call), so re-entry is blocked regardless of ordering within the delegatecall.

#### 5. `claimWinnings` Sentinel Pattern (DegenerusGame:1431-1447)

```
Line 1432: if (finalSwept) revert E();        // Checks
Line 1433: amount = claimableWinnings[player]; // Checks
Line 1434: if (amount <= 1) revert E();        // Checks
Line 1437: claimableWinnings[player] = 1;      // Effects (sentinel)
Line 1440: claimablePool -= payout;            // Effects
Line 1442-1446: external ETH/stETH transfer    // Interactions
```

**CEI verification:** All state mutations (claimableWinnings zeroed to sentinel, claimablePool decremented) happen BEFORE the external call. If the recipient contract re-enters `claimWinnings`, it finds `amount <= 1` and reverts. This is a textbook CEI pattern.

#### 6. `_sendToVault` External Calls (GameOverModule:195-231)

`_sendToVault` makes up to 4 external calls:
1. `steth.transfer(VAULT, vaultAmount)` -- ERC20 transfer to vault
2. `payable(VAULT).call{value: ethAmount}("")` -- ETH transfer to vault
3. `steth.approve(SDGNRS, dgnrsAmount)` + `dgnrs.depositSteth(dgnrsAmount)` -- stETH deposit to sDGNRS
4. `payable(SDGNRS).call{value: ethAmount}("")` -- ETH transfer to sDGNRS

**Can vault re-enter?** The vault contract's `receive()` has no external calls beyond emitting events. sDGNRS's `receive()` is `onlyGame` (only accepts ETH from game contract) and emits an event. Neither performs callbacks. However, even if they did re-enter:
- `handleGameOverDrain`: blocked by `gameOverFinalJackpotPaid = true`
- `handleFinalSweep`: blocked by `finalSwept = true`
- `claimWinnings`: would require the vault to have claimable winnings, and the sentinel pattern prevents double-claim

**Verdict:** No exploitable reentrancy vector exists.

#### 7. `burnRemainingPools` External Call (GameOverModule:163)

`dgnrs.burnRemainingPools()` (StakedDegenerusStonk:352-361) modifies only sDGNRS state:
```solidity
balanceOf[address(this)] = 0;
totalSupply -= bal;
delete poolBalances;
```

No external calls, no callbacks, no ETH transfers. The `onlyGame` modifier ensures only the game contract can call it. Re-entry impossible.

### Finding: _sendToVault Hard Reverts Can Block Terminal Distribution

**Severity:** Medium
**Location:** GameOverModule.sol:195-231
**Impact:** If vault or sDGNRS cannot receive ETH or stETH, `handleGameOverDrain` reverts entirely. Since `gameOverFinalJackpotPaid` is set at line 128 (BEFORE `_sendToVault` at line 158), and `_sendToVault` only runs on the `remaining` amount after decimator and jackpot distribution, the revert at `_sendToVault` would roll back the ENTIRE transaction including the latch set. This means `handleGameOverDrain` can be retried, but it will keep reverting on `_sendToVault`.

**Attack scenario:** Not externally exploitable (vault and sDGNRS are protocol-owned immutable contracts), but operational risk exists:
1. Lido pauses stETH transfers (has never happened, but is technically possible)
2. Protocol redeploys vault/sDGNRS without maintaining `receive()` compatibility

**Why not Critical:** The vault and sDGNRS are immutable protocol-owned contracts with simple, unconditional `receive()` functions. sDGNRS's `receive()` has an `onlyGame` modifier that is permanently satisfied. Lido stETH has never paused transfers. The risk is theoretical.

**Recommendation:** Consider wrapping `_sendToVault` in a try/catch or using a pull-based pattern for the vault sweep. Alternatively, accept as known risk given immutable recipients.

### AdvanceModule Safety Valve Analysis (Lines 441-444)

```solidity
if (lvl != 0 && _getNextPrizePool() >= levelPrizePool[lvl]) {
    levelStartTime = ts;
    return false;
}
```

This valve only fires when `lvl != 0` AND the next prize pool already meets the level-up requirement. In this case, the game should naturally level up on the next purchase cycle, so triggering GAMEOVER would be incorrect. The valve correctly resets the death clock (`levelStartTime = ts`) to give the game time to process the level-up.

**Can this incorrectly abort a valid GAMEOVER?** No. If `nextPrizePool >= levelPrizePool[lvl]`, the game is not actually stalled -- it has sufficient funds to level up but simply hasn't had an `advanceGame` call to process it. The valve prevents a false GAMEOVER during this transient state.

**Edge case:** If `levelPrizePool[lvl]` is set to 0 or a very low value, the valve would perpetually reset the death clock. However, `levelPrizePool` is set during level transitions based on economic parameters and is always >> 0 for active levels.

### Summary of Dangerous Reverts

All dangerous reverts (#11-16, #21) are concentrated in `_sendToVault`, which handles the vault and sDGNRS fund transfers. The function uses a fail-hard pattern (`revert E()`) with no fallback or try/catch protection.

**Mitigating factors:**
- Vault and sDGNRS are protocol-owned, immutable contracts
- Vault's `receive()` is unconditional (no logic, no revert path beyond OOG)
- sDGNRS's `receive()` has `onlyGame` modifier, permanently satisfied by the game contract
- stETH (Lido) has never paused transfers and has strong operational guarantees

**Residual risk:** A Lido stETH pause or operational error on protocol infrastructure could permanently block terminal distribution. This is an accepted design trade-off -- the alternative (swallowing transfer failures) would silently lose funds.

## GO-09: No-RNG-Available GAMEOVER Fallback Path -- PASS

**Verdict:** PASS
**Files:** AdvanceModule.sol:91, 797-878, GameOverModule.sol:126, DegenerusGameStorage.sol:210, 354, 429
**Summary:** The `_gameOverEntropy` function (AdvanceModule:797-846) implements a robust 4-branch RNG acquisition path that guarantees GAMEOVER can eventually fire even when Chainlink VRF is permanently unavailable. The 3-day fallback timer starts on first VRF failure, cannot reset, and uses historical VRF words combined with prevrandao for secure fallback entropy. The fallback word cannot be zero (keccak256 output), so it cannot trigger the no-latch retry at GameOverModule:126. No path permanently prevents GAMEOVER from firing.

### `_gameOverEntropy` -- 4-Branch Trace (AdvanceModule:797-846)

```
_gameOverEntropy(ts, day, lvl, isTicketJackpotDay) -> word
```

#### Branch A: Word already exists for today (line 803)

```solidity
if (rngWordByDay[day] != 0) return rngWordByDay[day];
```

- **Trigger:** A previous call already set `rngWordByDay[day]` for this day index.
- **Return:** The stored word (non-zero by definition).
- **State changes:** None.
- **Analysis:** This is the fast path for repeated `advanceGame` calls on the same day. The word was already committed by a previous Branch B or C execution. PASS -- no security concern.

#### Branch B: VRF fulfilled (lines 805-817)

```solidity
uint256 currentWord = rngWordCurrent;
if (currentWord != 0 && rngRequestTime != 0) {
    currentWord = _applyDailyRng(day, currentWord);
    if (lvl != 0) {
        coinflip.processCoinflipPayouts(isTicketJackpotDay, currentWord, day);
    }
    _finalizeLootboxRng(currentWord);
    return currentWord;
}
```

- **Trigger:** `rngWordCurrent != 0` (VRF word delivered) AND `rngRequestTime != 0` (VRF request was pending).
- **Return:** The VRF-derived word after applying nudges via `_applyDailyRng`.
- **State changes:** `_applyDailyRng` sets `rngWordByDay[day] = finalWord`, `rngWordCurrent = finalWord`, `lastVrfProcessedTimestamp = block.timestamp`. Also processes coinflip payouts and lootbox RNG.
- **Analysis:** Normal VRF path. Word is cryptographically secure (Chainlink VRF). PASS.

#### Branch C: Fallback timer expired (lines 819-836)

```solidity
if (rngRequestTime != 0) {
    uint48 elapsed = ts - rngRequestTime;
    if (elapsed >= GAMEOVER_RNG_FALLBACK_DELAY) {  // 3 days
        uint256 fallbackWord = _getHistoricalRngFallback(day);
        fallbackWord = _applyDailyRng(day, fallbackWord);
        if (lvl != 0) {
            coinflip.processCoinflipPayouts(isTicketJackpotDay, fallbackWord, day);
        }
        _finalizeLootboxRng(fallbackWord);
        return fallbackWord;
    }
    return 0;  // Still waiting, not yet 3 days
}
```

- **Trigger:** `rngRequestTime != 0` (a VRF request or fallback timer is active) AND `rngWordCurrent == 0` (VRF not fulfilled). If `elapsed >= 3 days`, use fallback.
- **Return (fallback):** Historical VRF-derived word via `_getHistoricalRngFallback`. Non-zero (see analysis below).
- **Return (waiting):** 0 -- signals "not ready yet, retry next advanceGame".
- **State changes (fallback):** Same as Branch B -- `_applyDailyRng` commits the word to `rngWordByDay[day]`.
- **State changes (waiting):** None.
- **Analysis:** The `return 0` causes `_handleGameOverPath` (line 450) to `return true`, which exits `advanceGame` without calling `handleGameOverDrain`. This is the correct behavior during the wait period. PASS.

#### Branch D: No VRF request yet (lines 838-845)

```solidity
if (_tryRequestRng(isTicketJackpotDay, lvl)) {
    return 1;
}

// VRF request failed; start fallback timer
rngWordCurrent = 0;
rngRequestTime = ts;
return 0;
```

- **Trigger:** `rngRequestTime == 0` (no VRF request in flight, no fallback timer started).
- **Sub-path D1:** `_tryRequestRng` succeeds -- returns 1 (VRF request submitted). `_handleGameOverPath` line 450 catches `rngWord == 1` and returns true (wait for VRF callback).
- **Sub-path D2:** `_tryRequestRng` fails (VRF coordinator unavailable, no subscription, etc.) -- sets `rngWordCurrent = 0`, `rngRequestTime = ts` to start the 3-day fallback timer, returns 0.
- **State changes (D2):** `rngRequestTime` set to current timestamp. This is the **timer start**.
- **Analysis:** `_tryRequestRng` (AdvanceModule:1151-1178) uses try/catch on `vrfCoordinator.requestRandomWords()`. If VRF is misconfigured (zero address, zero keyHash, zero subscription), it returns false immediately without attempting the call. If the call reverts, the catch absorbs it. In both cases, the fallback timer starts. PASS.

### `_getHistoricalRngFallback` Security Assessment (AdvanceModule:859-878)

```solidity
function _getHistoricalRngFallback(uint48 currentDay) private view returns (uint256 word) {
    uint256 found;
    uint256 combined;
    uint48 searchLimit = currentDay > 30 ? 30 : currentDay;
    for (uint48 searchDay = 1; searchDay < searchLimit; ) {
        uint256 w = rngWordByDay[searchDay];
        if (w != 0) {
            combined = uint256(keccak256(abi.encodePacked(combined, w)));
            unchecked { ++found; }
            if (found == 5) break;
        }
        unchecked { ++searchDay; }
    }
    return uint256(keccak256(abi.encodePacked(combined, currentDay, block.prevrandao)));
}
```

**Entropy sources:**
1. **Historical VRF words** (up to 5): Collected from `rngWordByDay[1..30]`. These are committed Chainlink VRF words from previous days -- cryptographically secure and non-manipulable after commitment.
2. **currentDay**: Deterministic, known to all parties. Adds uniqueness per day but no entropy.
3. **block.prevrandao**: The RANDAO mix from the previous block. Subject to 1-bit validator manipulation (a validator can choose to propose or skip their block).

**Can the fallback word be zero?**
The function returns `keccak256(abi.encodePacked(combined, currentDay, block.prevrandao))`. The probability of keccak256 producing exactly 0 is 1/2^256, which is effectively impossible. The fallback word is guaranteed non-zero for all practical purposes.

This is critical because `handleGameOverDrain` at GameOverModule:126 checks `if (rngWord == 0) return`. If the fallback could produce 0, it would create an infinite retry loop. Since keccak256 cannot practically produce 0, this path is safe.

**Validator manipulation assessment:**
- A validator can manipulate `prevrandao` by 1 bit (propose their block or skip it).
- With 5 historical VRF words contributing to `combined`, the manipulation capability is negligible: the validator would need to predict the keccak256 output to determine whether proposing or skipping produces a more favorable GAMEOVER outcome.
- In the GAMEOVER context, the RNG word selects terminal decimator winning subbuckets and terminal jackpot winners. A validator would need to be an active player with burns in a specific subbucket to benefit from manipulation, AND would need to predict the keccak256 output chain -- computationally infeasible.
- **Verdict:** 1-bit validator bias is acceptable for a gameover-only fallback. The primary entropy source (historical VRF) is non-manipulable.

**Level 0 edge case (no VRF history):**
If `currentDay <= 1` or no historical VRF words exist (all `rngWordByDay[1..30]` are 0), `combined` remains 0, and the fallback is purely:
```solidity
keccak256(abi.encodePacked(0, currentDay, block.prevrandao))
```
This is prevrandao-only entropy with 1-bit validator bias. However, this can only occur at level 0 (no VRF history means no successful `advanceGame` calls ever occurred). At level 0, the GAMEOVER is a timeout on an abandoned game -- the outcome distribution is irrelevant (all funds go to deity refunds and level-1 ticketholders, if any exist). The 1-bit bias is inconsequential.

### No-Latch Retry on Missing RNG (GameOverModule:126)

```solidity
// GameOverModule.sol, line 126 (inside handleGameOverDrain)
if (rngWord == 0) return;
```

**Analysis of state before line 126:**

When `handleGameOverDrain` is entered and reaches line 126, the following state changes have already occurred:
- Lines 92-96: `claimableWinnings[owner] += refund` for deity pass holders (if level < 10)
- Line 105: `claimablePool += totalRefunded`
- Line 112: `gameOver = true`
- Line 113: `gameOverTime = block.timestamp`

If `rngWord == 0` triggers the return at line 126, then `gameOver` is `true` and `gameOverTime` is set, but `gameOverFinalJackpotPaid` is NOT set.

**Can this scenario actually occur?**

No, in practice. `handleGameOverDrain` is only called from `_handleGameOverPath` (AdvanceModule:454-460), which is only reached when:
1. `rngWordByDay[day] == 0` was false (word already set), OR
2. `_gameOverEntropy` returned a non-zero, non-one word (word was set by `_applyDailyRng`)

In both cases, `rngWordByDay[day]` is non-zero when `handleGameOverDrain` is called, so line 125 (`rngWord = rngWordByDay[day]`) will produce a non-zero value, and line 126 will not trigger.

**Why the check exists:** It is defensive dead code in the current architecture. If a future code change allowed `handleGameOverDrain` to be called without ensuring `rngWordByDay[day]` is set, the check prevents an uncontrolled GAMEOVER with no RNG (which would mean `keccak256(0)` used for all selections).

**State corruption if it DID trigger:** If line 126 returned with `gameOver = true` but `gameOverFinalJackpotPaid = false`, subsequent `advanceGame` calls would enter `_handleGameOverPath`, see `gameOver == true` at line 430, and delegatecall `handleFinalSweep`. However, `handleFinalSweep` would silently return at line 173 (`block.timestamp < gameOverTime + 30 days`) until 30 days pass. After 30 days, `handleFinalSweep` would execute: `finalSwept = true`, `claimablePool = 0`, sweep all to vault. This would skip the terminal decimator and jackpot distribution entirely. This is a hypothetical concern -- it cannot occur with the current call path.

### Timer Start and Progression Analysis

**Timer start (Branch D2):**
```solidity
rngWordCurrent = 0;
rngRequestTime = ts;  // Timer starts NOW
```

The timer starts when `_tryRequestRng` fails AND `rngRequestTime` was previously 0. `rngRequestTime` is set to the current timestamp.

**Can the timer never start?** No. The only path where `rngRequestTime` remains 0 is Branch D1 (`_tryRequestRng` succeeds). But if VRF succeeds, we get a word via Branch B on the next call -- no fallback needed. If VRF is permanently broken, `_tryRequestRng` will fail, and Branch D2 starts the timer.

**Can the timer reset (extending the wait)?**

Once `rngRequestTime` is set to a non-zero value:
1. Branch A: Requires `rngWordByDay[day] != 0`. If this is true, `handleGameOverDrain` proceeds directly -- no timer check needed.
2. Branch B: Requires `rngWordCurrent != 0`. If VRF delivers a word, the timer is irrelevant -- we have a word.
3. Branch C: Checks `rngRequestTime != 0` (true) and `elapsed >= 3 days`. Either waits or uses fallback.
4. Branch D: Requires `rngRequestTime == 0`. Since it is non-zero, this branch is unreachable.

**The timer cannot reset.** Once `rngRequestTime` is set in Branch D2, it stays set until either:
- Branch B consumes it (VRF word arrives), OR
- Branch C uses the fallback (3 days elapsed)

Neither Branch C nor Branch D modifies `rngRequestTime` during the wait period. The `return 0` at line 835 (waiting, < 3 days) preserves the timer value.

**After fallback use:** `_applyDailyRng` sets `rngWordByDay[day]`. Then `_handleGameOverPath` calls `_unlockRng(day)` which resets `rngRequestTime = 0`. But this happens AFTER the word is committed -- the timer's purpose is complete.

**Verdict:** Timer progression is monotonic. No path resets or extends it.

### Deterministic Selection with Fallback Word

Once the fallback word is generated by `_getHistoricalRngFallback` and applied via `_applyDailyRng`, it flows identically to a normal VRF word:

1. Stored in `rngWordByDay[day]` by `_applyDailyRng` (AdvanceModule:1360)
2. Read by `handleGameOverDrain` at GameOverModule:125
3. Passed to `runTerminalDecimatorJackpot(decPool, lvl, rngWord)` at line 140
4. Passed to `runTerminalJackpot(remaining, lvl + 1, rngWord)` at line 152-153

Neither `runTerminalDecimatorJackpot` nor `runTerminalJackpot` validate the source of the RNG word. They accept any `uint256` value. The fallback word is used identically to a VRF word for:
- Decimator winning subbucket selection (`_decWinningSubbucket(decSeed, denom)` at DecimatorModule:898)
- Jackpot trait rolling (`_rollWinningTraits` at JackpotModule:295)
- Winner selection entropy (`_resolveTraitWinners` at JackpotModule:1630)

**Verdict:** Fallback word is accepted without additional validation. Distribution logic is source-agnostic. PASS.

### Fund Safety During Wait Period

During the 3-day wait for the fallback timer:

1. **Can normal gameplay advance?** No. `_handleGameOverPath` returns `true` at AdvanceModule:450 when `_gameOverEntropy` returns 0, which causes `advanceGame` to exit early. The game is effectively paused in the GAMEOVER pending state.

2. **Can players withdraw?** No pending withdrawals are affected. `claimWinnings` is independent of `advanceGame` and continues to work normally. Players can claim any existing `claimableWinnings` during the wait.

3. **Does the liveness guard remain triggered?** Yes. The liveness check at lines 421-423 evaluates `ts - lst > threshold`. Since no new purchases can occur (advanceGame exits early, and the liveness condition means no recent activity), the liveness guard remains triggered on every subsequent `advanceGame` call.

4. **Can fund movement corrupt the eventual GAMEOVER?** No. During the wait:
   - `gameOver` is false (not yet set -- it's set inside `handleGameOverDrain` which hasn't been called)
   - `claimablePool` is unchanged (no distributions in progress)
   - Contract balance may change slightly due to stETH rebasing, but `handleGameOverDrain` re-reads balances at lines 74-76
   - No new purchases can occur (liveness guard blocks `advanceGame` from reaching purchase processing)

5. **Can deposits or withdrawals during the wait cause issues?** The game's `receive()` function (DegenerusGame:2846) adds ETH to `futurePrizePool`. If someone sends ETH during the wait, it increases the contract balance and `futurePrizePool`. When `handleGameOverDrain` eventually fires, it reads `address(this).balance + steth.balanceOf(address(this))` as `totalFunds` -- this would include the new deposit. Since `handleGameOverDrain` zeroes all prize pools (lines 129-132), any deposited ETH during the wait becomes part of the terminal distribution. This is correct behavior -- the funds are not lost.

Wait -- actually, `receive()` at DegenerusGame:2846 checks `if (gameOver) revert E()`. Since `gameOver` is false during the wait period (it's only set inside `handleGameOverDrain`), deposits are still accepted. This is technically fine -- the depositor is adding to the terminal distribution pool.

**Verdict:** Fund safety is maintained during the 3-day wait period. PASS.

### Summary

The VRF fallback path is well-designed:

1. **Guaranteed termination:** After 3 days without VRF, `_getHistoricalRngFallback` provides a non-zero word that cannot trigger the retry loop.
2. **Monotonic timer:** Once started, the fallback timer cannot reset or extend.
3. **Secure entropy:** Historical VRF words are non-manipulable; prevrandao adds only 1-bit validator bias, acceptable for GAMEOVER-only use.
4. **Source-agnostic consumption:** Downstream functions accept the fallback word identically to a VRF word.
5. **Fund safety:** No state corruption or fund movement during the wait period.
6. **Defensive dead code:** The `if (rngWord == 0) return` at GameOverModule:126 is unreachable in the current architecture but provides a safety net against future refactoring.
