### Detailed Latch Analysis

#### 1. `gameOverFinalJackpotPaid` Latch (GameOverModule:69, 128)

- **Set at:** Line 128, AFTER `gameOver = true` (line 112) but BEFORE any delegatecall to decimator (line 140) or jackpot (line 152).
- **Guard at:** Line 69, first statement of `handleGameOverDrain`. Returns immediately if already set.
- **Re-entry vector analysis:** The self-calls at lines 140 and 152 dispatch through `DegenerusGame.runTerminalDecimatorJackpot()` and `DegenerusGame.runTerminalJackpot()`. Both have `msg.sender != address(this)` guards. These are delegatecalls that execute DecimatorModule and JackpotModule code in the game's storage context. Neither module calls back to `handleGameOverDrain`. Even if they could, `gameOverFinalJackpotPaid` is already `true`.
- **Verdict:** Airtight. No path allows double execution.

#### 2. `gameOver = true` (GameOverModule:112)

- **Set at:** Line 112, BEFORE all delegatecalls.
- **Purpose:** Terminal flag. Once set, normal gameplay functions (purchase, lootbox, etc.) revert or behave differently.
- **Key downstream effect:** `_addClaimableEth` in JackpotModule (line 982) checks `if (!gameOver)` before auto-rebuy. Since `gameOver = true` was set in the same delegatecall context before `runTerminalJackpot`, auto-rebuy is correctly skipped during terminal jackpot distribution.
- **Verdict:** Correctly ordered.

#### 3. `finalSwept` Latch (GameOverModule:174, 176)

- **Set at:** Line 176, immediately after the guard check at line 174.
- **Guard at:** Line 174: `if (finalSwept) return`.
- **Re-entry vector analysis:** `_sendToVault` at line 188 makes external ETH/stETH transfers. If vault re-enters via `advanceGame -> _handleGameOverPath`, it hits the `finalSwept` check in `handleFinalSweep` and returns. The `advanceGame` path through `_handleGameOverPath` will delegatecall `handleFinalSweep` when `gameOver == true`, which returns immediately.
- **Verdict:** Airtight. No double-sweep possible.

#### 4. Delegatecall Context Verification

The calls from `handleGameOverDrain` to `runTerminalDecimatorJackpot` and `runTerminalJackpot` use the pattern:
```solidity
IDegenerusGame(address(this)).runTerminalDecimatorJackpot(decPool, lvl, rngWord);
```

This is a **self-call** (external call to `address(this)`), which enters `DegenerusGame.runTerminalDecimatorJackpot()`. That function then performs:
```solidity
ContractAddresses.GAME_DECIMATOR_MODULE.delegatecall(...)
```

The delegatecall executes `DecimatorModule.runTerminalDecimatorJackpot()` in the game contract's storage context. Storage writes (e.g., `lastTerminalDecClaimRound`, `decBucketOffsetPacked`) happen to the game's storage slots. This is correct behavior -- not reentrancy -- but the intermediate self-call does create a call-stack entry. The `msg.sender != address(this)` guard at DegenerusGame:1215 ensures only the game contract itself can invoke these dispatch functions.

**State mutation ordering in delegatecall context:** The `runTerminalDecimatorJackpot` delegatecall writes to `lastTerminalDecClaimRound` and `decBucketOffsetPacked[lvl]` at DecimatorModule:914-919. These writes occur in the game's storage during the delegatecall, but `gameOverFinalJackpotPaid` is already `true` (set at GameOverModule:128 before the self-call), so re-entry is blocked regardless of ordering within the delegatecall.

#### 5. `claimWinnings` Sentinel Pattern (DegenerusGame:1431-1447)

```
Line 1432: if (finalSwept) revert E();        // Checks
Line 1433: amount = claimableWinnings[player]; // Checks
Line 1434: if (amount <= 1) revert E();        // Checks
Line 1437: claimableWinnings[player] = 1;      // Effects (sentinel)
Line 1440: claimablePool -= payout;            // Effects
Line 1442-1446: external ETH/stETH transfer    // Interactions
```

**CEI verification:** All state mutations (claimableWinnings zeroed to sentinel, claimablePool decremented) happen BEFORE the external call. If the recipient contract re-enters `claimWinnings`, it finds `amount <= 1` and reverts. This is a textbook CEI pattern.

#### 6. `_sendToVault` External Calls (GameOverModule:195-231)

`_sendToVault` makes up to 4 external calls:
1. `steth.transfer(VAULT, vaultAmount)` -- ERC20 transfer to vault
2. `payable(VAULT).call{value: ethAmount}("")` -- ETH transfer to vault
3. `steth.approve(SDGNRS, dgnrsAmount)` + `dgnrs.depositSteth(dgnrsAmount)` -- stETH deposit to sDGNRS
4. `payable(SDGNRS).call{value: ethAmount}("")` -- ETH transfer to sDGNRS

**Can vault re-enter?** The vault contract's `receive()` has no external calls beyond emitting events. sDGNRS's `receive()` is `onlyGame` (only accepts ETH from game contract) and emits an event. Neither performs callbacks. However, even if they did re-enter:
- `handleGameOverDrain`: blocked by `gameOverFinalJackpotPaid = true`
- `handleFinalSweep`: blocked by `finalSwept = true`
- `claimWinnings`: would require the vault to have claimable winnings, and the sentinel pattern prevents double-claim

**Verdict:** No exploitable reentrancy vector exists.

#### 7. `burnRemainingPools` External Call (GameOverModule:163)

`dgnrs.burnRemainingPools()` (StakedDegenerusStonk:352-361) modifies only sDGNRS state:
```solidity
balanceOf[address(this)] = 0;
totalSupply -= bal;
delete poolBalances;
```

No external calls, no callbacks, no ETH transfers. The `onlyGame` modifier ensures only the game contract can call it. Re-entry impossible.

### Finding: _sendToVault Hard Reverts Can Block Terminal Distribution

**Severity:** Medium
**Location:** GameOverModule.sol:195-231
**Impact:** If vault or sDGNRS cannot receive ETH or stETH, `handleGameOverDrain` reverts entirely. Since `gameOverFinalJackpotPaid` is set at line 128 (BEFORE `_sendToVault` at line 158), and `_sendToVault` only runs on the `remaining` amount after decimator and jackpot distribution, the revert at `_sendToVault` would roll back the ENTIRE transaction including the latch set. This means `handleGameOverDrain` can be retried, but it will keep reverting on `_sendToVault`.

**Attack scenario:** Not externally exploitable (vault and sDGNRS are protocol-owned immutable contracts), but operational risk exists:
1. Lido pauses stETH transfers (has never happened, but is technically possible)
2. Protocol redeploys vault/sDGNRS without maintaining `receive()` compatibility

**Why not Critical:** The vault and sDGNRS are immutable protocol-owned contracts with simple, unconditional `receive()` functions. sDGNRS's `receive()` has an `onlyGame` modifier that is permanently satisfied. Lido stETH has never paused transfers. The risk is theoretical.

**Recommendation:** Consider wrapping `_sendToVault` in a try/catch or using a pull-based pattern for the vault sweep. Alternatively, accept as known risk given immutable recipients.

### AdvanceModule Safety Valve Analysis (Lines 441-444)

```solidity
if (lvl != 0 && _getNextPrizePool() >= levelPrizePool[lvl]) {
    levelStartTime = ts;
    return false;
}
```

This valve only fires when `lvl != 0` AND the next prize pool already meets the level-up requirement. In this case, the game should naturally level up on the next purchase cycle, so triggering GAMEOVER would be incorrect. The valve correctly resets the death clock (`levelStartTime = ts`) to give the game time to process the level-up.

**Can this incorrectly abort a valid GAMEOVER?** No. If `nextPrizePool >= levelPrizePool[lvl]`, the game is not actually stalled -- it has sufficient funds to level up but simply hasn't had an `advanceGame` call to process it. The valve prevents a false GAMEOVER during this transient state.

**Edge case:** If `levelPrizePool[lvl]` is set to 0 or a very low value, the valve would perpetually reset the death clock. However, `levelPrizePool` is set during level transitions based on economic parameters and is always >> 0 for active levels.

### Summary of Dangerous Reverts

All dangerous reverts (#11-16, #21) are concentrated in `_sendToVault`, which handles the vault and sDGNRS fund transfers. The function uses a fail-hard pattern (`revert E()`) with no fallback or try/catch protection.

**Mitigating factors:**
- Vault and sDGNRS are protocol-owned, immutable contracts
- Vault's `receive()` is unconditional (no logic, no revert path beyond OOG)
- sDGNRS's `receive()` has `onlyGame` modifier, permanently satisfied by the game contract
- stETH (Lido) has never paused transfers and has strong operational guarantees

**Residual risk:** A Lido stETH pause or operational error on protocol infrastructure could permanently block terminal distribution. This is an accepted design trade-off -- the alternative (swallowing transfer failures) would silently lose funds.
