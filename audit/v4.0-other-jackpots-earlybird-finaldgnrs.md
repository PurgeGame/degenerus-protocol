## 1. Early-Bird Lootbox Jackpot

### 1.1 Trigger Path

The early-bird lootbox fires on the first daily jackpot of each level (jackpot Day 1).

**Complete call chain:**

1. `advanceGame` in AdvanceModule enters the JACKPOT PHASE block (AM:345).
2. On fresh daily jackpot, delegates to `payDailyJackpot(true, lvl, rngWord)` (AM:379).
3. `payDailyJackpot` reads `jackpotCounter` (JM:349). `isEarlyBirdDay` is true when `counter == 0` (JM:364).
4. The early-bird call: `_runEarlyBirdLootboxJackpot(lvl + 1, randWord)` (JM:381).

The `lvl + 1` argument means the early-bird operates on the NEXT level's trait bucket (`traitBurnTicket[lvl + 1]`), which is the level being purchased into during the current purchase phase. This is intentional: the trait bucket for `lvl + 1` has already been populated by purchases and ticket processing before the jackpot phase began.

On Day 1 specifically, the normal carryover flow is skipped: `if (!isEarlyBirdDay)` guards the carryover source selection (JM:409) and the 1% reserve slice (JM:426). Early-bird replaces carryover on Day 1.

### 1.2 Prize Pool Source

**Budget computation (JM:801-807):**

```
reserveContribution = (_getFuturePrizePool() * 300) / 10_000  // 3% of futurePrizePool
totalBudget = reserveContribution
_setFuturePrizePool(_getFuturePrizePool() - reserveContribution)
```

- **Source:** `futurePrizePool` -- the unified reserve backing future jackpots. Read via `_getFuturePrizePool()` (GS:755-757), which unpacks the high 128 bits of `prizePoolsPacked` (GS:356, GS:664-667).
- **Allocation:** Hardcoded 3% (300 bps / 10,000). There is no named constant like `EARLYBIRD_LOOTBOX_BPS` -- the value is inline at JM:803.
- **Deduction:** Immediately subtracted from `futurePrizePool` at JM:807.
- **Early exit:** If `totalBudget == 0`, the function returns immediately (JM:809-811).

**Budget recycling (JM:861-863):**

After all 100 winner iterations, the ENTIRE `totalBudget` is added to `nextPrizePool`:

```
_setNextPrizePool(_getNextPrizePool() + totalBudget)
```

This is critical: the ETH is moved from `futurePrizePool` to `nextPrizePool`, not consumed. The newly-queued tickets are backed by this ETH, which will merge into `currentPrizePool` at the next level's consolidation phase. The flow is: futurePrizePool -> nextPrizePool (via early-bird) -> currentPrizePool (via consolidatePrizePools at JM:866+).

`_getNextPrizePool()` reads the low 128 bits of `prizePoolsPacked` (GS:743-745). `_setNextPrizePool()` writes via `_setPrizePools(uint128(val), future)` (GS:749-751, GS:660-661).

### 1.3 Winner Selection Loop (100 Iterations)

**Loop structure (JM:814-858):**

```
uint256 maxWinners = 100;
uint256 perWinnerEth = totalBudget / maxWinners;  // JM:815
uint256 entropy = rngWord;                         // JM:816
```

**Pre-computed price table (JM:818-826):** Five level prices are cached in a `uint256[5]` array for gas efficiency, covering `baseLevel + 0` through `baseLevel + 4`.

**Per-iteration flow (JM:829-858):**

1. **Entropy advance:** `entropy = EntropyLib.entropyStep(entropy)` (JM:830). The xorshift PRNG (EntropyLib.sol:16-23) applies three shifts: `state ^= state << 7; state ^= state >> 9; state ^= state << 8`.

2. **Trait selection:** `uint8 traitId = uint8(entropy)` (JM:831). Takes the low 8 bits, producing values 0-255 uniformly.

3. **Winner lookup:** `_randTraitTicket(traitBurnTicket[lvl], entropy, traitId, 1, uint8(i))` (JM:832-838). The `salt` parameter is `uint8(i)`, which wraps to 0-255 (safe since `i < 100`).

4. **Winner check:** `address winner = winners.length != 0 ? winners[0] : address(0)` (JM:839). If `winner != address(0)`, proceed with payout.

5. **Level offset roll:** `entropy = EntropyLib.entropyStep(entropy); uint24 levelOffset = uint24(entropy % 5)` (JM:842-843). Produces 0-4 with near-uniform distribution (negligible bias since 2^256 mod 5 ~ 0).

6. **Ticket count:** `uint32 ticketCount = uint32(perWinnerEth / ticketPrice)` (JM:846). Integer division -- any remainder is lost (per-winner dust).

7. **Queue tickets:** `_queueTickets(winner, baseLevel + levelOffset, ticketCount)` (JM:848-852). Only if `ticketPrice != 0` and `ticketCount != 0`.

**Critical: traitId range analysis (JM:831)**

`uint8(entropy)` produces values 0-255, but the trait system has 256 possible trait IDs organized as `(quadrant << 6) | (color << 3) | symIdx` where quadrant is 0-3, color is 0-7, and symIdx is 0-7. All 256 positions in the `traitBurnTicket[lvl]` array are addressable.

However, `_randTraitTicket` (JM:2237-2280) does the following:
- Reads `holders = traitBurnTicket_[trait]` at JM:2244 -- the array at this trait index.
- Computes `fullSymId = (trait >> 6) * 8 + (trait & 0x07)` at JM:2250. This produces values 0-31 for valid traits but can produce values 32-71 for certain trait IDs.
- Checks `if (fullSymId < 32)` at JM:2253 for deity virtual entries.
- If `effectiveLen == 0`, returns empty array at JM:2262-2263.

For a trait ID where `traitBurnTicket[lvl][traitId]` is empty AND the deity check fails (fullSymId >= 32 or no deity), `effectiveLen == 0` and the function returns an empty array. The caller gets `winners.length == 0`, the `winner` is set to `address(0)`, and the iteration is skipped (no tickets queued).

**Effective hit rate:** All 256 trait IDs are valid array indices, but only trait IDs that have had tickets burned into them at this level will have non-empty arrays. The practical hit rate depends entirely on how many of the 256 trait positions have been populated by ticket burns at `traitBurnTicket[lvl]`. In a mature game with active trading, many positions will be populated; in early levels, fewer will be. The raw 256-slot space is fully utilized by the trait system (4 quadrants x 8 colors x 8 symbols = 256 combinations), so the miss rate depends on burn distribution, not wasted address space.

**Note on prior research:** The research document (87-RESEARCH.md:120) stated "traitId >= 32" causes ~87.5% miss rate, but this analysis was based on the incorrect assumption that only 32 trait positions exist. The trait system uses all 256 positions (4 quadrants x 8 colors x 8 symbols). The miss rate is determined by how many of the 256 positions have burn tickets at the given level, which is a function of game state, not a fixed structural limitation. [DISCREPANCY] Research overcounted the miss rate by confusing `fullSymId` (used only for deity virtual entries) with the trait bucket index.

### 1.5 RNG Dependency

**VRF word source:** The `rngWord` parameter originates from the AdvanceModule's `advanceGame` function, which reads `rngWordCurrent` (GS:362) -- the latest VRF-delivered random word. This word is passed through `payDailyJackpot` at JM:326 and forwarded to `_runEarlyBirdLootboxJackpot` at JM:381.

**RNG state at call time:** The early-bird runs during jackpot phase, when `jackpotPhaseFlag = true` (set at AM:327). The `rngLockedFlag` state depends on whether a new VRF request is pending.

**_queueTickets routing (GS:537-558):** The `isFarFuture` check at GS:544 compares `targetLevel > level + 6`. For early-bird tickets:
- `baseLevel = lvl` (which was called with `lvl + 1` from JM:381)
- `levelOffset` ranges 0-4
- Target levels are `baseLevel + 0` through `baseLevel + 4`
- Since `baseLevel` is the NEXT level (lvl + 1 relative to the current game level), all targets are at most `level + 5`, which is always <= `level + 6`
- Therefore: all early-bird tickets route to `_tqWriteKey(targetLevel)`, NEVER to `_tqFarFutureKey`

This means the `rngLockedFlag` guard at GS:545 (`if (isFarFuture && rngLockedFlag && !phaseTransitionActive) revert RngLocked()`) is never triggered for early-bird tickets. This is correct behavior -- near-future writes are safe during jackpot phase.

### 1.6 Audit Findings

**[INFO] EB-01: Integer division dust in perWinnerEth**
- **Location:** JM:815
- **Description:** `perWinnerEth = totalBudget / 100`. Integer division truncates up to 99 wei. However, the entire `totalBudget` (including the un-distributed dust) is added to `nextPrizePool` at JM:863. The dust is not lost -- it is recycled into the prize pool system. Individual winners may receive slightly fewer tickets than their "fair share" of the 3% allocation, but the total ETH is conserved.
- **Severity:** INFO -- no fund loss, negligible per-winner impact.

**[INFO] EB-02: Per-winner ticket count truncation**
- **Location:** JM:846
- **Description:** `ticketCount = uint32(perWinnerEth / ticketPrice)`. For levels with high ticket prices (late game), `perWinnerEth` may be less than `ticketPrice`, producing `ticketCount == 0`. In this case, the winner receives nothing despite being selected. The ETH backing is still recycled to `nextPrizePool`, but this specific winner gets no tickets.
- **Severity:** INFO -- by-design for high-price levels; winner selection is "best effort".

**[INFO] EB-03: Entropy step is not cryptographically independent**
- **Location:** JM:830, EntropyLib.sol:16-23
- **Description:** `EntropyLib.entropyStep` uses a simple xorshift PRNG seeded from VRF. Each step is deterministic and cheap, but consecutive outputs are correlated (xorshift is a linear function). For winner selection purposes this is adequate since the seed is a VRF word and no on-chain actor can predict or influence the step outputs. The trait selection at JM:831 and level offset roll at JM:843 use different entropy steps, providing sufficient independence for this purpose.
- **Severity:** INFO -- acceptable for game mechanics where VRF seed is the trust anchor.

**[INFO] EB-04: Double read of futurePrizePool**
- **Location:** JM:803, JM:807
- **Description:** `_getFuturePrizePool()` is called twice: once to compute `reserveContribution` and once inside `_setFuturePrizePool(_getFuturePrizePool() - reserveContribution)`. Both calls unpack `prizePoolsPacked` (GS:664-667). This is a minor gas inefficiency (double SLOAD followed by SSTORE), not a correctness issue. The second read returns the same value as the first since no write occurred between them.
- **Severity:** INFO -- gas optimization opportunity only.

## 2. Final-Day DGNRS Distribution

### 2.1 Trigger Path

**Complete call chain:**

1. `advanceGame` in AdvanceModule processes the JACKPOT PHASE (AM:345+).
2. `payDailyJackpotCoinAndTickets(rngWord)` completes coin and ticket distribution (AM:363). This function increments `jackpotCounter += counterStep` (JM:757).
3. After return, the AdvanceModule checks: `if (jackpotCounter >= JACKPOT_LEVEL_CAP)` (AM:364). `JACKPOT_LEVEL_CAP = 5` (AM:92).
4. On match (all 5 daily jackpots complete), the level transition sequence fires:
   - `_awardFinalDayDgnrsReward(lvl, rngWord)` (AM:365)
   - `_rewardTopAffiliate(lvl)` (AM:366)
   - `_runRewardJackpots(lvl, rngWord)` (AM:367)
   - `_endPhase()` (AM:368)
   - `_unfreezePool()` (AM:369)

**Delegatecall wrapper (AM:555-568):**

```solidity
function _awardFinalDayDgnrsReward(uint24 lvl, uint256 rngWord) private {
    (bool ok, bytes memory data) = ContractAddresses.GAME_JACKPOT_MODULE.delegatecall(
        abi.encodeWithSelector(
            IDegenerusGameJackpotModule.awardFinalDayDgnrsReward.selector,
            lvl, rngWord
        )
    );
    if (!ok) _revertDelegate(data);
}
```

This is a standard delegatecall wrapper in the AdvanceModule. The actual logic resides in JackpotModule.

### 2.2 Prize Pool Source

**Entry point:** `awardFinalDayDgnrsReward(uint24 lvl, uint256 rngWord)` (JM:773).

**Pool query (JM:774):**
```solidity
uint256 dgnrsPool = dgnrs.poolBalance(IStakedDegenerusStonk.Pool.Reward);
```

This reads the current balance of the sDGNRS Reward pool via an external call to the staked DGNRS contract (IStakedDegenerusStonk interface at `ContractAddresses.SDGNRS`). The `poolBalance` function is defined at IStakedDegenerusStonk.sol:26.

**Reward calculation (JM:775):**
```solidity
uint256 reward = (dgnrsPool * FINAL_DAY_DGNRS_BPS) / 10_000;
```

`FINAL_DAY_DGNRS_BPS = 100` (JM:173), which is 1% of the Reward pool balance. This is a sDGNRS (staked DGNRS token) distribution, not an ETH distribution.

**Early exit (JM:776):** `if (reward == 0) return;` -- handles the case where the Reward pool is empty or rounds to zero.

### 2.3 Winner Selection

**Entropy derivation (JM:779):**
```solidity
uint256 entropy = rngWord ^ (uint256(lvl) << 192);
```

The level is mixed into the entropy by XOR at bit position 192, providing domain separation per level. This is the same pattern used in `payDailyJackpotCoinAndTickets` at JM:695.

**Solo bucket index (JM:780):**
```solidity
uint8 soloIdx = JackpotBucketLib.soloBucketIndex(entropy);
```

`soloBucketIndex` (JackpotBucketLib.sol:240-242) computes:
```solidity
return uint8((uint256(3) - (entropy & 3)) & 3);
```

This maps the low 2 bits of entropy to a bucket index 0-3, with the mapping: `entropy & 3 == 0 -> 3`, `1 -> 2`, `2 -> 1`, `3 -> 0`. The result selects which of the 4 winning trait quadrants is the "solo" (1-winner) bucket.

**Winning traits unpacking (JM:781-782):**
```solidity
uint32 packed = lastDailyJackpotWinningTraits;
uint8[4] memory traitIds = JackpotBucketLib.unpackWinningTraits(packed);
```

`unpackWinningTraits` (JackpotBucketLib.sol:269-273) extracts 4 trait IDs from the packed uint32:
- `traits[0] = uint8(packed)` -- quadrant 0
- `traits[1] = uint8(packed >> 8)` -- quadrant 1
- `traits[2] = uint8(packed >> 16)` -- quadrant 2
- `traits[3] = uint8(packed >> 24)` -- quadrant 3

**Winner selection via _randTraitTicket (JM:784-790):**
```solidity
address[] memory winners = _randTraitTicket(
    traitBurnTicket[lvl],
    entropy,
    traitIds[soloIdx],
    1,        // numWinners = 1
    254       // salt = 254
);
```

Parameters:
- `traitBurnTicket[lvl]` (GS:417) -- the trait ticket array for the current level
- `entropy` -- derived from `rngWord ^ (lvl << 192)`
- `traitIds[soloIdx]` -- the winning trait for the solo bucket
- `numWinners = 1` -- exactly one winner
- `salt = 254` -- unique salt to differentiate from other `_randTraitTicket` calls in the same transaction

Inside `_randTraitTicket` (JM:2237-2280):
- Reads `holders = traitBurnTicket_[trait]` (JM:2244)
- Computes deity virtual entries if `fullSymId < 32` (JM:2253)
- If `effectiveLen == 0`, returns empty array (JM:2262-2263)
- Otherwise: `slice = randomWord ^ (uint256(trait) << 128) ^ (uint256(salt) << 192)` (JM:2268-2270)
- `idx = slice % effectiveLen` (JM:2272)
- Returns `holders[idx]` or `deity` if `idx >= len` (JM:2273)

### 2.4 Payout

**Winner validation and transfer (JM:791-797):**
```solidity
if (winners.length > 0 && winners[0] != address(0)) {
    dgnrs.transferFromPool(
        IStakedDegenerusStonk.Pool.Reward,
        winners[0],
        reward
    );
}
```

`transferFromPool` (IStakedDegenerusStonk.sol:34) transfers sDGNRS tokens directly from the Reward pool to the winner. This is NOT an ETH transfer -- it is a token transfer within the staked DGNRS contract.

**No-winner path:** If `_randTraitTicket` returns an empty array (no holders for the selected trait) or `address(0)`, the reward is NOT distributed. The sDGNRS remains in the Reward pool. There is no fallback winner selection or retry logic.

### 2.5 Key Dependency: lastDailyJackpotWinningTraits

**Write path:** `lastDailyJackpotWinningTraits` is written by `_syncDailyWinningTraits` (JM:2627-2634):
```solidity
lastDailyJackpotWinningTraits = packed;  // JM:2632
lastDailyJackpotLevel = lvl;              // JM:2633
lastDailyJackpotDay = questDay;           // JM:2634
```

This is called from `payDailyJackpot` at JM:345 on fresh (non-resuming) daily jackpots:
```solidity
winningTraitsPacked = _rollWinningTraits(lvl, randWord, true);  // JM:344
_syncDailyWinningTraits(lvl, winningTraitsPacked, questDay);     // JM:345
```

**Ordering verification:**

The critical question: is `lastDailyJackpotWinningTraits` populated BEFORE `awardFinalDayDgnrsReward` reads it?

1. Day 1-5 `payDailyJackpot` calls write `lastDailyJackpotWinningTraits` at JM:345 (fresh start) or read it at JM:340 (resume).
2. Each Day's `payDailyJackpotCoinAndTickets` reads `lastDailyJackpotWinningTraits` at JM:694 and increments `jackpotCounter` at JM:757.
3. After Day 5's `payDailyJackpotCoinAndTickets` returns, `jackpotCounter >= 5` triggers `_awardFinalDayDgnrsReward` at AM:365.
4. `awardFinalDayDgnrsReward` reads `lastDailyJackpotWinningTraits` at JM:781.

**Conclusion:** The traits are ALWAYS set before the final-day reward runs. Specifically, Day 5's `payDailyJackpot` wrote the traits (JM:345), then `payDailyJackpotCoinAndTickets` incremented the counter (JM:757), and only then does the AdvanceModule check the counter and call `awardFinalDayDgnrsReward`. The traits reflect the LAST daily jackpot's winning traits (Day 5), not a stale value.

**Compressed/Turbo edge case:** When `compressedJackpotFlag == 2` (turbo mode), `counterStep = JACKPOT_LEVEL_CAP = 5` (JM:354), so a single `payDailyJackpotCoinAndTickets` call jumps the counter from 0 to 5. The traits are still written by that same Day 1's `payDailyJackpot` call. In this case, `lastDailyJackpotWinningTraits` reflects Day 1's winning traits, not Day 5's (since Days 2-5 are skipped). This is by design -- turbo mode collapses all 5 days into one physical day.

### 2.7 Audit Findings

**[INFO] FD-01: No-winner path silently skips reward**
- **Location:** JM:791
- **Description:** If `_randTraitTicket` returns an empty array (the selected trait bucket has no holders and no deity), the reward is silently skipped. There is no event emission, no retry, and no fallback. The 1% sDGNRS remains in the Reward pool indefinitely. This is by design (the pool accumulates for future distributions), but the lack of an event makes it invisible to off-chain monitoring.
- **Severity:** INFO -- no fund loss, but lack of observability.

**[INFO] FD-02: Solo bucket index uses only 2 bits of entropy**
- **Location:** JM:780, JackpotBucketLib.sol:241
- **Description:** `soloBucketIndex` uses `entropy & 3` (2 bits). This is adequate for selecting among 4 buckets, but means the bucket selection is highly predictable given the VRF word and level. Since the VRF word is unknown before fulfillment and the level is known, this is not exploitable -- the VRF provides the unpredictability.
- **Severity:** INFO -- correct usage of entropy for 4-way selection.

**[INFO] FD-03: DSC-02 non-applicability confirmed**
- **Description:** The Phase 81 finding DSC-02 (`sampleFarFutureTickets` reads `_tqWriteKey` instead of `_tqFarFutureKey`) does NOT affect `awardFinalDayDgnrsReward`. The final-day DGNRS reward uses `_randTraitTicket` to select from `traitBurnTicket[lvl]` (realized trait arrays from processed tickets), not from `ticketQueue` or far-future buffers. DSC-02 is specific to BAF jackpot's far-future sampling.
- **Severity:** INFO -- explicit non-applicability confirmation.

**[INFO] FD-04: Turbo mode uses Day 1 traits for final-day reward**
- **Location:** JM:354, AM:365
- **Description:** In turbo mode (`compressedJackpotFlag == 2`), all 5 logical jackpot days are collapsed into 1 physical day with `counterStep = 5`. The `lastDailyJackpotWinningTraits` written by that single day's `payDailyJackpot` are the traits used for the final-day DGNRS reward. This means the "Day 5 winning traits" are actually Day 1 winning traits. No correctness issue -- this is intentional behavior for turbo mode -- but it differs from the normal flow where Day 5's traits are used.
- **Severity:** INFO -- behavioral note for turbo mode.

## 3. Cross-Cutting Analysis

### 3.1 Shared Patterns

**Both subsystems use `_randTraitTicket` (JM:2237-2280):**

- **Early-bird:** `_randTraitTicket(traitBurnTicket[lvl], entropy, traitId, 1, uint8(i))` (JM:832-838) -- called up to 100 times with varying `traitId` and `salt`.
- **Final-day DGNRS:** `_randTraitTicket(traitBurnTicket[lvl], entropy, traitIds[soloIdx], 1, 254)` (JM:784-790) -- called once with a specific trait and fixed salt.

**Both depend on `traitBurnTicket[lvl]` population:**

The `traitBurnTicket` mapping (GS:405-417) is populated by ticket processing (`processTicketBatch` at JM) during the purchase phase. If a level has few burns, both jackpot types will find fewer winners. The early-bird is more resilient due to its 100 iterations (more chances to hit a populated trait bucket), while the final-day DGNRS makes a single attempt on a single trait.

**Both operate during jackpot phase:**

Early-bird fires at the start of Day 1's jackpot (JM:381). Final-day fires after Day 5's coin+ticket distribution completes (AM:365). Both have access to the same VRF word passed through `advanceGame`.

**Deity virtual entries affect both:**

Inside `_randTraitTicket`, deity pass holders get virtual entries: `virtualCount = max(len / 50, 2)` (JM:2256-2257) when `fullSymId < 32` and `deityBySymbol[fullSymId] != address(0)`. This means a deity holder has at least 2 virtual tickets in their symbol's trait bucket, even if the bucket is otherwise empty. This applies to both early-bird and final-day winner selection.
