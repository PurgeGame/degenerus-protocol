## Section 1a: Daily Consumption Points (D1-D9)

### Shared Temporal Window

All 9 daily consumption points share a single temporal window:

1. **Block N:** `advanceGame()` calls `_requestRng()`, sets `rngLockedFlag=true`, fires VRF request to Chainlink.
2. **Block N+K:** `rawFulfillRandomWords()` stores raw VRF word in `rngWordCurrent` (AdvanceModule:1338). `rngLockedFlag` remains true.
3. **Block N+K+M:** `advanceGame()` calls `rngGate()`, which reads `rngWordCurrent`, calls `_applyDailyRng()` (applies nudge offset from `totalFlipReversals`), then executes all 9 daily consumers atomically within a single transaction.

**Critical property:** Between Block N (rngLockedFlag set true) and Block N+K+M (advanceGame consumes word), 12 entry points are blocked by rngLockedFlag revert guards:
- `requestLootboxRng` (AdvanceModule:674)
- `reverseFlip` (AdvanceModule:1299)
- `purchaseDeityPass` (WhaleModule:468)
- `setAutoRebuy` (DegenerusGame:1526)
- `setDecimatorAutoRebuy` (DegenerusGame:1505)
- `setAutoRebuyTakeProfit` (DegenerusGame:1541)
- `setAfKingMode` (DegenerusGame:1606)
- `claimCoinflips` (BurnieCoinflip:336)
- `claimCoinflipsTakeProfit` (BurnieCoinflip:347)
- `claimCoinflipsFromBurnie` (BurnieCoinflip:357)
- `consumeCoinflipsForBurn` (BurnieCoinflip:367)
- BAF credit gate (BurnieCoinflip:603, at x10 levels)

**Entry points still callable during lock (13):**
- `advanceGame`, `rawFulfillRandomWords` -- system functions
- `purchase`, `purchaseCoin`, `purchaseBurnieLootbox`, `purchaseWhaleBundle`, `purchaseLazyPass` -- ticket purchases (write to write-buffer, not read-buffer)
- `openLootBox`, `openBurnieLootBox` -- independent lootbox RNG
- `placeFullTicketBets` -- assigned to current lootbox index
- `resolveDegeneretteBets` -- uses lootbox RNG word
- `issueDeityBoon` -- read-only RNG access
- `setLootboxRngThreshold` -- admin config only

**Atomicity guarantee:** Once `advanceGame` enters `rngGate` and `_applyDailyRng` returns the finalized word, all 9 consumption points execute within the same transaction. No user transaction can interleave between D1-D9. The word is consumed and `_unlockRng` clears the lock before the transaction completes.

## Section 1b: Lootbox Consumption Points (L1-L8)

### Shared Temporal Window

Lootbox consumption points have a fundamentally different temporal model from daily points:

1. **Block N:** `requestLootboxRng()` fires VRF request. `rngLockedFlag=false` (mid-day path). `rngRequestTime!=0` blocks concurrent requests.
2. **Block N+K:** `rawFulfillRandomWords()` finalizes word directly to `lootboxRngWordByIndex[index]` (AdvanceModule:1342). This is a **direct finalize** -- the word is immediately available for consumption. `rngLockedFlag` remains false.
3. **Block N+K+M (variable, per-user):** Users call `openLootBox`/`openBurnieLootBox`/`resolveDegeneretteBets` in separate transactions, potentially many blocks later.

**Critical difference from daily:** The window between word finalization (Block N+K) and consumption (Block N+K+M) is OPEN -- `rngLockedFlag` is false, and all entry points are callable. However, lootbox outcomes use **per-player entropy derivation**: `keccak256(word, player, day, amount)` or `keccak256(word, index, player, spinIdx)`, making outcomes deterministic per-player once the word is set.

**Entry points callable during lootbox window (all 25+ external entry points):** Since `rngLockedFlag=false`, nothing is blocked by the lock. Since `prizePoolFrozen` may or may not be true (depends on whether jackpot phase is active), pool-dependent operations may be blocked by freeze but not by lootbox RNG state.

### Piggyback Pattern Analysis

When `_finalizeLootboxRng` is called during daily `rngGate` processing (AdvanceModule:766), the daily finalized word is written to the pending lootbox index:
- `lootboxRngWordByIndex[index] = rngWord` (AdvanceModule:788)
- `lastLootboxRngWord = rngWord` (AdvanceModule:789)

This piggyback write occurs **within the same transaction as rngGate**, meaning:
1. The word written to the lootbox index was committed in the VRF callback (Block N+K) and consumed by `_applyDailyRng` (Block N+K+M, same tx).
2. The lootbox word write happens atomically with the daily word write -- there is no window between the daily word being known and the lootbox word being set.
3. The same VRF word (with nudge applied) is used for both daily draws AND the piggybacked lootbox index. This is documented behavior -- the daily word also resolves the batch reserved during `_requestRng`.
4. **No cross-path window exists for the piggyback write** because both writes occur in the same atomic transaction.

## Section 2: Adversarial Timeline

### Block Builder Adversarial Model

Capabilities assumed for a block builder (proposer):
- **Read access to mempool:** Can see pending transactions including VRF callback payloads before block inclusion
- **Ordering control:** Can arrange transaction ordering within a block they propose
- **Inclusion/exclusion control:** Can include or exclude specific transactions from their block
- **Multi-block persistence:** Can collude across multiple consecutive blocks (rare but modeled)
- **No VRF control:** Cannot predict or influence Chainlink VRF random words before the callback is broadcast
- **No protocol bypass:** Cannot call internal/private functions or bypass Solidity access controls

### Section 2a: Daily VRF Path Timeline

**Block builder capabilities at Block N+K+M:**
- **CAN sandwich `advanceGame`** with other transactions before and after it in the same block.
- **CANNOT interleave** with the 9 consumption steps (D1-D9) because they execute atomically within a single `advanceGame` call. Solidity execution is non-preemptive within a transaction.
- **Pre-advanceGame sandwich:** Builder includes their own txs before advanceGame. But all RNG-influencing entry points are still locked (rngLockedFlag=true). Purchases go to write buffer. No effect on D1-D9 outcomes.
- **Post-advanceGame sandwich:** Builder includes txs after advanceGame. By now rngLockedFlag is false, but all 9 daily consumption points have already executed. The draw is complete and results are immutable.

**Net effect:** Block builder can observe the VRF word for ~K blocks before consumption but cannot act on it. Consumption is atomic and uninterruptible.

- **CAN order user transactions before/after the callback** within the block.

**Critical analysis -- self-front-running:**
1. Builder sees word in mempool, computes their own lootbox outcome.
2. If outcome is favorable: include callback, then include their own `openLootBox` tx after it.
3. If outcome is unfavorable: exclude callback from this block (delay).

**But this provides NO advantage because:**
- The builder's lootbox deposit (`lootboxEth[index][builder]`) was made BEFORE the VRF request (purchases go to current index, VRF resolves previous index). The deposit amount is immutable.
- Delaying the callback does not change the VRF word -- the same word will arrive in the next block.
- The builder cannot modify their deposit amount, player address, or day after the fact.
- The ONLY possible action is choosing to open or NOT open their lootbox, which is equivalent to having the option to not claim a lottery ticket after seeing the result. This is not an attack -- the outcome is the same whether they open now or later.

**Critical analysis -- other-player manipulation:**
- Per-player entropy: `keccak256(word, player, day, amount)` means the builder CANNOT predict other players' outcomes without knowing their exact addresses and deposit amounts.
- Even if the builder knows another player's address and deposit, they cannot change that player's outcome. The outcome is deterministic.
- Including/excluding the callback does not change any player's eventual outcome -- the same word will arrive.

- **No sandwich opportunity:** The outcome depends only on immutable inputs (word, player, day, amount). No state can be sandwiched to change the result.
- **No backrunning opportunity:** Opening a lootbox does not create a price impact or arbitrage opportunity for the builder.

3. **Cross-path piggyback is atomic:** When daily VRF finalization also writes to a pending lootbox index (via `_finalizeLootboxRng`), both writes occur in the same transaction. No window exists for the piggyback write path.

4. **No consumption point relies solely on VRF unpredictability:** Every consumption point has structural protections (rngLockedFlag, double-buffer, per-player entropy, prizePoolFrozen, commit-reveal) independent of whether the VRF word is known. This satisfies the requirement to avoid "SAFE because VRF is unpredictable" reasoning -- each point is protected by architectural guarantees.

#### RESOLVED: Open Question 1 -- Deity Pass Purchase During Jackpot Inter-Block Gap

**Code trace:** `purchaseDeityPass` (WhaleModule:463) -> `_purchaseDeityPass` (WhaleModule:467)

**Guard analysis:**
```solidity
function _purchaseDeityPass(address buyer, uint8 symbolId) private {
    if (rngLockedFlag) revert RngLocked();     // WhaleModule:468
    if (gameOver) revert E();                   // WhaleModule:469
    if (symbolId >= 32) revert E();             // WhaleModule:470
    if (deityBySymbol[symbolId] != address(0)) revert E();  // WhaleModule:471
    if (deityPassCount[buyer] != 0) revert E(); // WhaleModule:472
    // ... no jackpotPhaseFlag check, no prizePoolFrozen check
```

**Finding:** `purchaseDeityPass` is guarded ONLY by `rngLockedFlag` (WhaleModule:468). There is NO `jackpotPhaseFlag` guard and NO `prizePoolFrozen` guard on the deity pass purchase path. During inter-block gaps when `rngLockedFlag=false`, deity pass purchases ARE callable.

**Co-state impact:**
- `deityPassOwners.push(buyer)` (WhaleModule:509) -- adds buyer to the deity pass holder list
- `deityBySymbol[symbolId] = buyer` (WhaleModule:511) -- binds a symbol to the buyer
- `deityPassSymbol[buyer] = symbolId` (WhaleModule:510) -- records the buyer's chosen symbol

**Jackpot draw co-state usage:** The jackpot draw in `payDailyJackpot` uses `_rollWinningTraits` (JackpotModule:353) to select winning traits from the VRF word, then iterates trait buckets to find ticket holders matching those traits. Deity pass holders get virtual trait-targeted tickets queued at WhaleModule:529-535. These tickets are queued to FUTURE levels (up to 100 levels ahead), not the current jackpot level.

**Critical analysis:** Can a deity pass purchase between Day N and Day N+1 change who wins Day N+1's jackpot?

1. **Deity tickets are queued to future levels:** The tickets queued by `_queueTickets` at WhaleModule:529-535 target `ticketStartLevel` through `ticketStartLevel + 99`, where `ticketStartLevel = passLevel <= 4 ? 1 : ((passLevel+1)/50)*50 + 1`. For a mid-game jackpot phase, these future-level tickets are NOT in the current read buffer being processed by the active jackpot draws. They go to the write buffer.

2. **The VRF word for Day N+1 is unknown:** During the inter-block gap, no VRF request has been made for Day N+1 yet. The attacker cannot predict which traits will win, which buckets will be selected, or which ticket holder index will be chosen.

3. **deityBySymbol changes do not affect current-level jackpot draws:** The deity virtual entries in jackpot buckets are processed during `processTicketBatch` for the tickets already in the read buffer. New deity pass purchases queue tickets to the write buffer for future levels.

4. **ETH routing:** `prizePoolFrozen=true` routes the deity pass purchase ETH to pending accumulators (WhaleModule:544-550), not the live prize pool being distributed.

**Verdict: SAFE BY DESIGN** -- Deity pass purchases during jackpot inter-block gaps do not influence current jackpot outcomes because (a) the next day's VRF word is unknown and unpredictable, (b) deity-queued tickets go to future levels in the write buffer (not the current read buffer), (c) prize pool ETH is routed to pending accumulators by prizePoolFrozen, and (d) existing read-buffer bucket composition is frozen by the double-buffer swap.

```solidity
}
```

**Finding:** `reverseFlip` is callable during inter-block gaps (`rngLockedFlag=false`). Each call increments `totalFlipReversals` by 1 at a compounding cost of 100 * 1.5^n BURNIE for the nth nudge.

**Impact on Day N+1:** When `advanceGame` calls `_applyDailyRng` for Day N+1 (AdvanceModule:1350-1365):
```solidity
function _applyDailyRng(uint48 day, uint256 rawWord) private returns (uint256 finalWord) {
    uint256 nudges = totalFlipReversals;
    finalWord = rawWord;
    if (nudges != 0) {
        unchecked { finalWord += nudges; }
        totalFlipReversals = 0;
    }
    rngWordCurrent = finalWord;
    rngWordByDay[day] = finalWord;
}
```

The nudge count is added to the raw VRF word (uint256 addition with unchecked overflow). This shifts the final word by exactly `nudges` integer positions.

**Economic analysis:**

The attacker's goal: shift the VRF word such that a specific modular outcome changes (e.g., `finalWord % bucketCount` selects a different bucket, or `finalWord % effectiveLen` selects a different ticket index).

- **Target shift for outcome change:** For `word % N` to select a different outcome, the word must shift by at least 1 modular position relative to N. With a 256-bit word space, the word is essentially uniformly distributed. Adding `nudges` to a uniformly random uint256 produces another uniformly random uint256 (modular arithmetic). The probability that `(word + nudges) % N` equals a specific target is exactly `1/N`, regardless of nudges.

- **Key insight:** Since the raw VRF word is unknown during the inter-block gap (the next day's VRF request has not yet been made), adding any deterministic offset `nudges` to an unknown random base produces an outcome with identical distribution. The attacker is blindly shifting a uniformly random value -- the probability distribution of outcomes is unchanged.

- **Cost of nudges:**
  - 1 nudge: 100 BURNIE
  - 2 nudges: 100 + 150 = 250 BURNIE
  - 5 nudges: 100 + 150 + 225 + 337.5 + 506.25 = 1,318.75 BURNIE
  - 10 nudges: ~7,593 BURNIE
  - 20 nudges: ~346,564 BURNIE
  - 50 nudges: ~6.37 billion BURNIE

- **Probability analysis:** Even if the attacker could somehow predict the raw VRF word (they cannot -- VRF is requested AFTER the inter-block gap ends), shifting a uint256 by `k` positions changes `word % N` only when the original word falls in specific modular residue classes. For `N` typical bucket sizes (10-1000), the probability of a targeted outcome is `1/N` regardless of the nudge offset. Nudges do not improve the probability of any specific outcome.

**Verdict: SAFE BY DESIGN** -- Nudges during jackpot inter-block gaps cannot influence jackpot outcomes because: (a) the next day's raw VRF word is unknown (VRF request not yet made), (b) adding a deterministic offset to an unknown random value does not change the outcome distribution, (c) even with hypothetical VRF knowledge, the compounding cost (100 * 1.5^n BURNIE per nudge) makes targeted shifting economically infeasible at ~10^-72 probability improvement per million nudges against 2^256 word space, and (d) `totalFlipReversals` is zeroed after application, so nudges do not accumulate across days.

#### RESOLVED: Open Question 3 -- processTicketBatch Entropy During Jackpot Phase

**Code trace:** `processTicketBatch` is called via `_runProcessTicketBatch` (AdvanceModule:1080) during `advanceGame`. The entropy source is `lastLootboxRngWord` (JackpotModule:1975 per L4 analysis in Section 1b).

**Which value of `lastLootboxRngWord` is used during jackpot-phase `advanceGame` calls?**

There are two paths that update `lastLootboxRngWord`:

1. **Piggyback path (AdvanceModule:785-789):** `_finalizeLootboxRng(rngWord)` is called during daily `rngGate` processing. It writes `lastLootboxRngWord = rngWord` where `rngWord` is the daily finalized word (post-nudge). This occurs within the same `advanceGame` call that processes the jackpot day.

2. **Mid-day lootbox path (AdvanceModule:166):** During same-day mid-day processing, if `midDayTicketRngPending=true`, the VRF word from `lootboxRngWordByIndex[lootboxRngIndex - 1]` is assigned to `lastLootboxRngWord`.

**During jackpot-phase days:** Each day's `advanceGame` call follows this sequence:
1. `rngGate` returns the daily VRF word
2. `_finalizeLootboxRng(finalWord)` writes the piggybacked daily word to `lastLootboxRngWord` (AdvanceModule:789)
3. `processTicketBatch` (via `_runProcessTicketBatch`) reads `lastLootboxRngWord` for ticket trait assignment

**Therefore:** During jackpot-phase days, `lastLootboxRngWord` holds the piggybacked daily VRF word (post-nudge), set by `_finalizeLootboxRng` within the same `advanceGame` transaction. This is the same VRF word used for jackpot draws (D2-D9), written atomically in the same call.

**Manipulation assessment:** Does this create a manipulation path?

- **No.** The `lastLootboxRngWord` value used for ticket trait assignment is the same daily VRF word that drives the jackpot draws. It is set within the same atomic `advanceGame` transaction. No external transaction can modify it between the piggyback write and the `processTicketBatch` consumption.
- **Trait assignment is independent of winner selection.** `processTicketBatch` assigns trait-bucket membership to tickets. Winner selection within those buckets happens separately during the jackpot draw using the same VRF word. Even if an attacker could influence trait assignment (they cannot), it would not change which ticket index is selected as the winner within each bucket -- that selection uses the jackpot VRF word directly.

**Verdict: SAFE BY DESIGN** -- `processTicketBatch` during jackpot phase uses the piggybacked daily VRF word, set atomically within the same `advanceGame` call. No manipulation window exists between the write and consumption of this value. Trait assignment and winner selection are independent lottery mechanics.

### Section 3c: Cross-Day State Mutation Analysis

For each category of state change possible during inter-block gaps, this subsection consolidates the manipulation assessment:

**1. New Ticket Purchases (Write Buffer)**
- **State change:** Tickets queued to write buffer via `_queueTickets`
- **Guard:** Double-buffer isolation (`ticketWriteSlot` toggled at `_swapTicketSlot`)
- **Jackpot impact:** None. Write buffer tickets are not processed during current jackpot draws. They target the read buffer of the NEXT level after phase transition.
- **Verdict:** SAFE BY DESIGN

**2. Lootbox Opens**
- **State change:** Per-player lootbox outcomes resolved
- **Guard:** Per-player entropy derivation `keccak256(word, player, day, amount)`
- **Jackpot impact:** None. Lootbox outcomes are completely independent of jackpot draw mechanics. Each player's outcome is deterministic from the prior committed lootbox VRF word.
- **Verdict:** SAFE BY DESIGN

**3. Coinflip Deposits and Claims**
- **State change:** BURNIE burns/credits via `claimCoinflips` variants
- **Guard:** Coinflip operations are BURNIE-denominated, no pool/RNG interaction
- **Jackpot impact:** None. Coinflip state is independent of jackpot draws. Phase 13 confirmed all coinflip surfaces SAFE (`v1.2-delta-new-attack-surfaces.md`).
- **Verdict:** SAFE BY DESIGN

**4. Deity Pass Purchases**
- **State change:** `deityPassOwners`, `deityBySymbol`, future-level tickets queued
- **Guard:** Next day's VRF word unknown; tickets queued to future levels in write buffer
- **Jackpot impact:** None for current level draws. See Open Question 1 resolution above.
- **Verdict:** SAFE BY DESIGN

**5. Nudges (reverseFlip)**
- **State change:** `totalFlipReversals` incremented
- **Guard:** Raw VRF word unknown; deterministic offset does not change outcome distribution
- **Jackpot impact:** None (statistical). See Open Question 2 resolution above.
- **Verdict:** SAFE BY DESIGN

**6. Auto-Rebuy / Settings Changes**
- **State change:** Per-player configuration flags
- **Guard:** Settings only affect future purchase routing, not current draw mechanics

- **State change:** `lootboxRngIndex` incremented, VRF request fired
- **Guard:** Lootbox VRF is independent from daily VRF; `requestLootboxRng` gates on `rngWordByDay[today] != 0` (AdvanceModule:682)
- **Jackpot impact:** None. Lootbox VRF words are separate from daily jackpot VRF words.
- **Verdict:** SAFE BY DESIGN
